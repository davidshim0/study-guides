Accelerating PIM with reconfigurable logic, not bigger cores

PIM fixes the memory wall—then hits a compute wall. PIM-CCA offloads hot code to a tiny Configurable Compute Accelerator, achieving 1.55× speedup with only 0.036% area overhead.

📄 Kim et al. · MICRO '25 🎓 Assumes basic PIM, pipeline, and ISA familiarity ⏱ ~25 min read
⚙️

Once PIM eliminates the memory bottleneck, the compute bottleneck emerges—but scaling PIM cores is constrained by wiring and power. Instead, offload hot code patterns to a lightweight, reconfigurable accelerator.

Before you begin

Eight concepts PIM-CCA builds on

Each card unpacks one prerequisite. Read in order the first time, or jump to refresh any topic.

1What PIM is and why it exists▾

Processing-in-Memory (PIM) integrates compute logic near or within memory chips. This eliminates the long data path between CPU and DRAM, leveraging high internal memory bandwidth to alleviate the memory wall. Commercial examples include UPMEM (128 DPUs per DIMM) and HBM-PIM.

PIM units are usually simple, pipelined processors—trading clock speed and complexity for proximity to data. Memory-intensive workloads (GEMV, histogram) benefit enormously, but once memory bottlenecks are mitigated, compute bottlenecks emerge.

Now you're ready: PIM-CCA targets workloads that become compute-bound inside PIM—offloading hot code to a lightweight accelerator without scaling up the main processor.
2UPMEM architecture: DPU, pipeline, tasklets▾

UPMEM provides a 14-stage pipeline DPU (Dispatch…Fetch1-3…ALU1-4…Merge1-2), but only the final 3 stages overlap with dispatch—so 11 concurrent tasklets (software threads) can hide latency. Each DPU has 64MB MRAM, 64KB WRAM (scratchpad), and 24KB IRAM. The paper uses this as the baseline.

Multiplication on UPMEM is expensive: a 32-bit multiply expands into 32 mul_step instructions (32 cycles). This is the primary compute bottleneck PIM-CCA addresses.

Now you're ready: PIM-CCA integrates a CCA into each UPMEM DPU, offloading mul_step sequences to reconfigurable logic and cutting execution cycles by up to 75%.
3Configurable Compute Accelerators (CCA)▾

CCAs are modular hardware units that extend a processor's ISA with custom instructions. The compiler identifies frequently-executed dataflow subgraphs (e.g., multiply-and-add), and the CCA offloads them via Custom Functional Units (CFUs). Unlike fixed-function accelerators, CCAs are reconfigurable—one design supports multiple patterns via a lookup table.

Originally proposed for embedded systems, CCAs balance area/power constraints with flexible acceleration—ideal for PIM's limited silicon budget.

Now you're ready: PIM-CCA's CCA is a 7-stage pipelined unit (shifters, adders, subtractors, accumulator) that fits in 0.036% of the total UPMEM chip area and supports three core operations (4-step multiply, accumulation, max).
4Hot code and why it matters in PIM▾

Hot code = performance-critical, frequently-executed regions (often loops). In PIM, hot codes converge on a small set of operations: multiply-and-add (GEMV, GEMM, MLP), accumulation (reduction), or complex indexing (transpose). For example, GEMV spends 42% of cycles on multiply-and-add; GEMM 83% on multiply-and-add plus reduction.

Because PIM workloads shift from memory-bound to compute-bound inside PIM, these hot codes become the new bottleneck—and targeting them with a lightweight accelerator delivers wide coverage with minimal hardware.

Now you're ready: PIM-CCA's compiler identifies hot codes via dataflow graph analysis and offloads them to the CCA, accelerating the few operations that dominate execution.
5Memory vs compute intensity inside PIM▾

The paper measures intensity by observing how many tasklets are active each cycle. High tasklet utilization (11–16 active) = compute-intensive; low (0–3) = memory-intensive. Workloads like GEMV, MLP, GEMM move to compute-intensive inside PIM—high bandwidth alleviates memory, exposing compute as the bottleneck.

Others (e.g., SCAN-RSS, UNI) remain memory-bound even in PIM. PIM-CCA benefits the first group by offloading compute; the second group benefits from tasklet tuning (reducing scheduling overhead).

Now you're ready: PIM-CCA uses compute intensity to guide strategy—CCA for compute-bound workloads, fewer tasklets for memory-bound workloads.
6Why scaling PIM cores is hard▾

PIM is memory-centric: fabrication uses only ~3 metal layers (vs 10+ in CPUs), limiting wiring for high-speed interconnects. Integrating complex processing units (higher frequency, wider SIMD, multi-core) consumes area meant for memory, increases power, and raises thermal risk to DRAM cells. Fixed-function accelerators (FP16 MAC units) exist but are constrained to simple, low-precision ops.

Thus, enhancing compute capability requires lightweight, modular solutions that respect wiring, area, and power constraints.

Now you're ready: CCA's modular, compiler-guided design fits these constraints—0.036% area (4464µm² after 25nm scaling), 1.28mW power, nothing on the critical path.
7Tasklet parallelism and pipeline utilization▾

UPMEM's 11-stage pipeline can support 11 tasklets to hide latency. But more tasklets = higher scheduling overhead. For memory-intensive code, performance saturates before 11 tasklets because memory stalls prevent full utilization—using fewer tasklets reduces control overhead without sacrificing throughput.

In CCA-enabled PIM, hot codes are offloaded, so compute bottlenecks are alleviated—allowing high performance with fewer tasklets (e.g., 8 instead of 16) and cutting scheduling/synchronization costs.

Now you're ready: PIM-CCA tunes tasklet count based on workload characteristics—fewer tasklets for near-memory-intensive workloads (e.g., SCAN-SSA: 8 tasklets = 54% overhead cut, 8% performance loss), more for compute-intensive (TS saturates at 16).
8Quick glossary of PIM-CCA terms▾

DPU = DRAM Processing Unit (UPMEM's in-memory processor). Tasklet = software thread (up to 11 concurrent in UPMEM). mul_step = single-bit multiplication step (32 for 32-bit multiply). CFU = Custom Functional Unit (the building block of CCA logic). CCA code = 7-bit index into the CCA lookup table. cca / cca_move = ISA extensions to offload to CCA and transfer operands.

Pattern string/graph = compiler representation of hot code (e.g., "o24 = i24 + i25 + i26 + i27 + i28" for accumulation). Logic palette = intermediate representation mapping pattern graphs to CCA hardware stages.

Now you're ready: These terms reappear throughout—refer back as needed. The core flow is: compiler identifies hot codes → maps to CCA codes → hardware offloads via CFUs.
The whole story in one place

PIM-CCA in a nutshell

The problem, the solution, and how they connect—starting from workload characteristics.

PIM fixes the memory wall by processing near data, but once communication bottlenecks are alleviated, compute bottlenecks emerge. Workloads like GEMV, GEMM, MLP shift to compute-intensive behavior inside PIM—spending 65–95% of cycles on multiply-and-add or accumulation. Yet scaling PIM processors is constrained by memory-centric fabrication (3 metal layers, limited wiring, shared power budget with DRAM).

PIM-CCA solves this by integrating a Configurable Compute Accelerator (CCA)—a modular, compiler-guided unit that offloads hot code patterns to reconfigurable logic. The CCA is a 7-stage pipelined datapath (4 right-shifters, 4 adders, 4 subtractors, 4 left-shifters, 2-level accumulator) controlled by a 7-bit CCA code that indexes a lookup table. The compiler identifies hot codes via instruction-level dataflow graph analysis, replaces them with cca instructions, and the CCA executes them in parallel with the main DPU pipeline.

To preserve compatibility, PIM-CCA extends the 48-bit UPMEM ISA with two instructions: cca (7-bit code + 35-bit operands) and cca_move (transfers up to 6 operands from DPU registers to CCA registers). The CCA branches at FETCH1 and returns results by MERGE2, matching the baseline pipeline's 9-cycle latency—so no bubbles are introduced.

Furthermore, the paper analyzes tasklet management: CCA-enabled PIM reduces compute intensity (e.g., GEMV: 11.62% drop in 11–16 tasklet cycles), allowing fewer tasklets without performance loss. For memory-intensive workloads, reducing tasklets (e.g., SCAN-SSA: 16→8) cuts control overhead by ~54% with only 8% performance degradation.

The payoff: PIM-CCA achieves up to 1.55× (avg 1.26×) speedup on 14 compute-intensive benchmarks (including GEMM), with only 0.036% area overhead (4464µm² at 25nm, 1.28mW power). Hardware overhead is off the critical path, and the design is modular—applicable to other PIM architectures beyond UPMEM.

⚙️ Key takeaways

  • PIM alleviates memory bottlenecks → workloads shift to compute-intensive inside PIM → new bottleneck emerges.
  • Hot codes (multiply-and-add, accumulation, indexing) dominate 42–95% of cycles in compute-bound PIM workloads.
  • Scaling PIM processors is constrained by 3-layer wiring, area, power—CCA provides a lightweight, reconfigurable alternative.
  • CCA = 7-stage pipeline (shifters, adders, subtractors, accumulator), controlled by 7-bit codes.
  • Compiler: dataflow graph analysis → pattern matching → offload to cca instructions → logic palette maps to hardware.
  • ISA extension: cca (offload) + cca_move (operand transfer) fit 48-bit UPMEM format, branch at FETCH1, return by MERGE2.
  • Tasklet tuning: CCA reduces compute intensity → fewer tasklets achieve high performance with lower overhead (e.g., SCAN-SSA: 54% overhead cut, 8% perf loss).
  • Result: 1.55× (avg 1.26×) speedup, 0.036% area, 1.28mW power—modular design applicable to other PIM architectures.
Six everyday pictures

How to think about PIM-CCA

Each picture stands alone—read them in any order. Every card ends with where the analogy breaks down.

The core problem · Compute bottleneck

Assembly line with a slow station

An assembly line (pipeline) runs smoothly until one station (multiplication) takes 32 steps while others take 1. That station becomes the bottleneck. You can't make the station faster (clock is fixed), so you offload it to a parallel specialist (CCA) who handles 4 steps at once—cutting station time to 8 cycles. The line moves 4× faster through that task.

Where it breaks: Real assembly lines can't share workers mid-task; PIM-CCA's pipeline continues with other tasklets while CCA executes, achieving true parallelism via interleaving.
Hot code · Targeting high-cycle regions

The Pareto principle in code

In TS, 95% of execution time is spent in one loop: multiply-and-add. Instead of optimizing the entire program, you identify that hot region and accelerate it with a specialist tool (CCA). The rest of the code still runs on the main processor—but it consumes only a small fraction of total cycles, so it doesn't matter. Focus where it counts.

Where it breaks: Unlike business metrics, hot code percentages are measured in cycles, not lines—a short loop can dominate if it iterates millions of times.
CCA · Reconfigurable logic

A Swiss Army knife vs dedicated tools

A fixed-function accelerator is like a specialized drill—fast at one task, useless for others. CCA is a Swiss Army knife: one device with multiple attachments (shifters, adders, subtractors). The compiler chooses which blade to deploy (CCA code 0x0 for mul_step, 0x1 for accumulation). Same hardware, multiple functions—flexibility within tight area budgets.

Where it breaks: Swiss Army knives are manually reconfigured; CCA reconfigures electronically via MUX settings in the lookup table—instant, no mechanical delay.
Compiler · Pattern matching

Recognizing faces in a crowd

The compiler builds a dataflow graph (nodes = operations, edges = data dependencies) and searches for known patterns—like recognizing a friend's face in a crowd. It matches "i24 + i25 + i26 + i27 + i28" (accumulation pattern) against the instruction stream, replaces it with a cca instruction, and the hardware executes it. Same result, fewer cycles.

Where it breaks: Face recognition is probabilistic; pattern matching is exact—every operator and operand must align, or the match fails and no offload occurs.
Tasklet tuning · Fewer threads

Overstaffing vs right-sizing a team

A team of 16 people (tasklets) on a project can be overkill—meetings, coordination overhead, people waiting for resources. If the workload allows, a team of 8 can deliver the same output with half the management cost. CCA accelerates compute tasks, so the pipeline doesn't need 16 tasklets to stay busy—8 suffice, cutting scheduling cycles by ~54% (SCAN-SSA) with only 8% performance loss.

Where it breaks: Teams adapt roles; tasklets are static—once launched, they all run the same code. The optimization is in choosing the count, not dynamically reassigning mid-execution.
Area overhead · Tiny footprint

Sticky note on a whiteboard

A whiteboard (PIM chip) is covered with notes (memory arrays, main processor). Adding a small sticky note (CCA: 0.036% area) doesn't obscure the board. The note holds critical reminders (CCA lookup table) without taking meaningful space. Similarly, CCA's 4464µm² footprint is negligible on a chip measured in mm²—lightweight acceleration without sacrificing memory capacity.

Where it breaks: Sticky notes are 2D; chips are 3D (metal layers). CCA uses wiring budget, not just area—but the paper's 3-layer P&R confirms it fits memory-centric constraints.
The full explanation

How it actually works

Section by section, with key diagrams and the paper's real numbers.

1 · The computational bottleneck inside PIM

PIM workloads enter memory-bound (high communication, low compute). Inside PIM, high bandwidth alleviates communication → workloads shift to compute-bound. Figure 3 of the paper measures this by counting active tasklets per cycle: 0–10 active = memory-bound; 11–16 active = compute-bound. Workloads like GEMV (42%), GEMM (83%), MLP (65%), TS (95%), TRNS (80%) spend most cycles compute-bound.

The bottleneck: UPMEM's 32-bit multiply = 32 mul_step instructions (32 cycles). GEMV, GEMM, MLP all saturate on multiply-and-add. TRNS saturates on complex indexing (multiply for address calculation). Thus, accelerating multiply-and-add plus accumulation covers the majority of hot codes.

🌱
Concretely: TS spends 95% of cycles on multiply-and-add. CCA offloads 4 mul_steps → 1 CCA cycle (75% instruction reduction). Result: TS achieves 1.23× speedup; GEMM 1.55×.

2 · Hot code detection via dataflow graph analysis

The compiler uses LLVM's front-end to generate IR-level dataflow graphs. It identifies common patterns in innermost loops: multiply-and-add (GEMV, GEMM, MLP), accumulation (reduction), max selection (originally NW, not evaluated due to simulator limits). The paper verifies patterns via cycle-accurate simulation (uPIMulator) and checks they meet hardware constraints (e.g., offloading full 32-bit multiply would be too large; instead, offload 4 mul_steps to fit 9-stage pipeline depth).

Pattern coverage: GEMV (42% cycles), GEMM (83%), MLP (65%), TS (95%), TRNS (80%). These five workloads justify CCA investment—wide coverage, high cycle percentage.

3 · CCA logic design: 7 stages, three core operations

The CCA is a 7-stage pipelined datapath designed to fit the 9-cycle window (CCA Decode → CCA Execution 0–6 → CCA Return) between FETCH1 and MERGE2. Each stage contains functional units:

  • Stage 0: 4 right-shifters (left), 4 adders (right)
  • Stage 1: 4 bitwise-AND units (left), 4 left-shifters (right)
  • Stage 2: 4 subtractors (left, generate predicates for conditional routing)
  • Stage 3: conditional routing network (left), passthrough (right)
  • Stage 4: 2 adders (first level of accumulator)
  • Stage 5: 2 adders (second level of accumulator)
  • Stage 6: 1 final adder

The logic supports 3 core operations (Table 1): 0x0 = 4 mul_steps (right-shift, AND, left-shift, conditional add); 0x1 = accumulation (5-operand add via 2-level tree); 0x2 = max of 3 values (subtract, compare, conditional select). These map to the CCA via MUX configuration in the lookup table.

⚙️
Hardware cost: 4 right-shifters + 4 left-shifters + 4 adders + 4 subtractors + 4 ANDs + MUX network = 4464µm² (25nm), 1.28mW. The 2-level accumulator enables reduction-style parallelism (4 partial sums → 2 → 1 in 3 cycles).

4 · ISA extension: cca and cca_move

UPMEM uses 48-bit instructions: 6-bit opcode, 7-bit metadata, 35-bit operands. PIM-CCA adds two instructions:

// cca: offload to CCA
cca code       // 6-bit opcode + 7-bit CCA code + 35-bit unused

// cca_move: transfer operands (up to 6 registers or immediates)
cca_move r24, r25, r26, r27, r28, r29  // or mix with immediates
cca_move r24, r25, r26, imm            // 3 regs + 1 immediate

The CCA code (7 bits) indexes the lookup table. cca_move precedes cca to preload CCA registers—executed in the inter-tasklet pipelining region. The CCA branches at FETCH1 (early decode detects cca in upper 6 bits) and returns by MERGE2—exactly 9 cycles, matching the baseline's ALU4→MERGE1→MERGE2 latency. However, cca_move overhead can be non-negligible and may erode CCA gains in memory-intensive workloads where the data-movement cost is high.

💡
Why 9 stages? UPMEM's pipeline has 8–9 remaining stages after FETCH1. CCA must complete by MERGE2 to maintain compatibility. 7 execution stages + 1 decode + 1 return = 9 total—perfect match.

5 · Compiler: pattern string → pattern graph → logic palette → code generation

The PIM-CCA compiler operates in four phases:

  1. Pattern string: human-readable spec (e.g., "o24 = i24 + i25 + i26 + i27 + i28" for accumulation). Parsed via Lex/Yacc into a pattern graph.
  2. Pattern graph: operator nodes (add, mul_step, subtract) + register nodes (i24, o24). Algorithm 1 matches the graph against IR instructions—bidirectional matching for commutative ops, recursive for nested subgraphs.
  3. Logic palette: intermediate representation mapping pattern graphs to hardware stages. Algorithm 2 constructs the palette by: (a) reusing existing computing units (best), (b) adding new units (higher cost), or (c) inserting new stages (highest cost). Goal: minimize hardware cost via maximal overlap.
  4. Code generation: matched instructions replaced with cca_move + cca. Register-value map (rvmap) tracks operand correspondence (pattern register → actual value).
⚙️
Example: IR contains %36 = mul %a, %b. Compiler detects multiply, matches 4 mul_steps to pattern graph for 0x0, emits cca_move r24, r25, imm then cca 0x0. Hardware executes in 9 cycles (vs 32 for baseline mul_steps).

6 · Tasklet tuning: balancing parallelism and overhead

UPMEM's 11-stage pipeline can support 11 tasklets to hide latency. But more tasklets = higher scheduling overhead (tasklet scheduler searches for next executable tasklet when many are stalled). For memory-intensive workloads, performance saturates before 11 tasklets because memory stalls prevent full utilization. For compute-intensive workloads, CCA offloads hot codes → compute intensity drops → fewer tasklets achieve high performance.

The paper measures control overhead (scheduler search cycles) and relative performance for varying tasklet counts. Result: workloads like SCAN-SSA (low memory-intensity in Figure 3) achieve 0.92× performance at 8 tasklets vs 16, but reduce control overhead by ~54%. Workloads like TS (high compute-intensity) still benefit from 16 tasklets (1.23× speedup with CCA).

💡
Why this matters: Scaling from thousands of DPUs, even small per-DPU overhead (1–2 cycles/tasklet) aggregates to significant energy waste. Tasklet tuning enables cost-effective execution without sacrificing performance.
Follow the numbers, step by step

Worked examples

Four traces, each carried through with the paper's real data.

① Offloading 4 mul_steps to CCA (pattern 0x0)

A 32-bit multiply on baseline UPMEM: 32 mul_step instructions, each 1 cycle = 32 cycles. CCA offloads 4 mul_steps per cca instruction. How many cycles for the full multiply?

  1. Baseline: 32 mul_steps × 1 cycle = 32 cycles.
  2. PIM-CCA: 32 ÷ 4 = 8 CCA calls. Each cca = 9 cycles (7 execution + 1 decode + 1 return). But pipeline overlaps: only the first cca pays decode/return overhead; subsequent ones pipeline. Effective: ~9 + (7 × 7) = 58 cycles for 8 calls (or ~7.25 cycles/call amortized).
  3. Speedup: In practice, GEMV (which is multiply-dominated) sees 1.46× overall speedup—confirming the mul_step offload is the primary gain.
⚙️
Insight: CCA doesn't eliminate multiply entirely—it reduces the per-step cost via parallelism (4 steps → 1 stage). The 9-stage pipeline ensures no bubbles, so throughput scales linearly with loop iterations.

② Accumulation pattern (0x1) in GEMM reduction

GEMM's reduction phase accumulates 5 partial sums. Baseline: 4 sequential add instructions (r0 + r1 → t0; t0 + r2 → t1; t1 + r3 → t2; t2 + r4 → result) = 4 cycles + operand fetch overhead. CCA pattern 0x1: how many cycles?

  1. Baseline: 4 dependent adds. Each add = 1 cycle (ALU latency) + register fetch (~2–3 cycles). Total ~10–12 cycles.
  2. PIM-CCA: cca_move r24, r25, r26, r27, r28 (1 cycle, inter-tasklet region), then cca 0x1 (9 cycles). Total = 10 cycles.
  3. Benefit: For tight loops (GEMM's reduction), avoiding sequential dependencies via 2-level accumulator tree saves ~2–3 cycles/iteration. Over millions of iterations, this compounds to the observed 1.55× speedup.
💡
The 2-level accumulator (4 partial sums → 2 → 1) exploits parallelism: stage 4 adds pairs in parallel, stage 5 adds results. Baseline sequential adds cannot parallelize—each depends on the previous result.

③ Compute intensity reduction (GEMV case study)

Baseline GEMV (16 tasklets): 42% of cycles have 11–16 tasklets active (compute-intensive). PIM-CCA GEMV: compute intensity drops to ~31%. Why, and what does this enable?

  1. Baseline: Multiply-and-add dominates. To keep pipeline full, 16 tasklets needed → moderate compute intensity (42%).
  2. PIM-CCA: CCA offloads mul_steps (75% instruction reduction). Fewer instructions → pipeline drains faster → lower compute intensity (~31%, −11.62%).
  3. Implication: Pipeline no longer needs 16 tasklets to stay busy. Fewer tasklets (e.g., 11–13) achieve same performance with lower scheduling overhead.
  4. Result: GEMV at 13 tasklets (PIM-CCA) ≈ GEMV at 16 tasklets (baseline), but with ~15% lower control overhead. For energy-efficient mode, this is the sweet spot.
⚙️
Insight: CCA doesn't just speed up code—it changes the architecture's resource utilization profile, enabling smarter thread management. This is why the paper emphasizes tasklet tuning as a co-optimization.

④ Area overhead calculation (25nm scaling)

Synthesized at 28nm: CCA = 5234µm² area. Scale to 25nm (UPMEM's node). What is the final area overhead as % of UPMEM chip?

  1. Scaling formula: Assuming constant voltage, area scales quadratically with feature size: Area₂₅ = Area₂₈ × (25/28)².
  2. Calculate: 5234µm² × (25/28)² = 5234 × 0.7965 ≈ 4167µm². (Paper reports 4464µm² after P&R with 4 metal layers, which includes routing overhead not captured in synthesis-only estimate.)
  3. UPMEM chip area: ~12.4mm² (1 chip, not full DIMM). CCA = 4464µm² = 0.004464mm².
  4. Overhead: (0.004464 / 12.4) × 100% = 0.036%. For dual-core (2 CCAs): 0.072%; quad-core: 0.144%—all under 1%.
⚙️
Insight: 0.036% is negligible—equivalent to a few rows of DRAM cells. The real constraint is wiring (3 metal layers), not area. The paper's P&R confirmation shows CCA fits within PIM's metal budget.

Practice problems

Try each before opening the answer. Every problem ends with the misconception it checks.

Problem 1 · easy
Why does PIM-CCA offload only 4 mul_steps per CCA call instead of all 32?
Show answer
Pipeline depth constraint. UPMEM's 11-stage pipeline can support up to 11 concurrent operations. CCA must complete by MERGE2 (9 stages: decode + 7 execution + return). 4 mul_steps fit in 7 execution stages (each mul_step = shift + AND + add); more than 4 mul_steps would exceed the allowable pipeline depth and cause stalls due to control complexity. The paper's Figure 6 shows saturation at 4 mul_steps (2.39× throughput vs baseline).Misconception: that CCA size is unlimited. It's constrained by the existing pipeline depth to maintain compatibility and avoid bubbles.
Problem 2 · medium
GEMV achieves 1.46× speedup, GEMM 1.55×, but MLP only 1.06×. All three use multiply-and-add as hot code and benefit from CCA pattern 0x0. Why does MLP lag?
Show answer
Input data size. MLP's dataset is small (matrices fit in WRAM), so the baseline completes the multiply phase quickly—hot code executes fewer iterations, CCA benefits don't amortize. GEMV/GEMM use larger datasets (GEMV default: vector × matrix; GEMM: (32,128) × (128,128)), so hot code loops millions of times, and CCA's per-iteration savings compound. MLP's 1.06× speedup confirms CCA overhead (cca_move insertion) slightly exceeds gains for small workloads.Misconception: that CCA always helps multiply-bound workloads. It helps workloads where hot code *iterates frequently*—small datasets see minimal benefit.
Problem 3 · medium
The paper reports CCA reduces GEMV's compute intensity from 42% to ~31% (−11.62% in 11–16 tasklet cycles). If CCA makes code faster, why does compute intensity *decrease* instead of increase?
Show answer
Compute intensity = % cycles with high tasklet utilization (11–16 active). CCA offloads hot codes → fewer instructions → pipeline drains faster → fewer cycles need all tasklets active. The remaining cycles shift to lower tasklet counts (more memory/control, less compute). High compute intensity means the pipeline is saturated with compute—CCA alleviates saturation, so intensity *drops*. This is desirable: it means fewer tasklets are needed to achieve the same throughput, reducing scheduling overhead.Misconception: that faster code = higher compute intensity. Intensity measures *how saturated* the pipeline is, not raw speed. CCA reduces saturation by accelerating bottlenecks.
Problem 4 · hard
Algorithm 2 (logic palette construction) minimizes cost by reusing computing units across pattern graphs. Construct a short example where reuse is impossible and a new stage must be added.
Show answer
Example: Pattern A requires "right-shift → add → left-shift" (3 stages). Pattern B requires "subtract → XOR → right-shift" (3 stages, but XOR in stage 2 conflicts with Pattern A's add in stage 2). If both patterns must execute, and stage 2 already has an adder reserved for Pattern A, adding Pattern B's XOR requires either: (a) a new computing unit in stage 2 (higher cost), or (b) inserting a new stage 3 for XOR, shifting subsequent stages (highest cost, increases pipeline depth). The heuristic prefers (a) unless pipeline depth is violated, then falls back to (b). The paper's 7-stage design accommodates 3 core patterns (0x0, 0x1, 0x2) by carefully overlapping compatible operations (e.g., shifters/adders in stage 0).Misconception: that all patterns can share all stages. Reuse is limited by functional unit type and data dependencies—incompatible ops require new units or stages.
Problem 5 · medium
Why does PIM-CCA introduce cca_move as a separate instruction instead of embedding operand transfer inside the cca instruction?
Show answer
Pipeline interleaving. cca_move executes in the inter-tasklet pipelining region (between tasklet dispatches), so it doesn't stall the main pipeline. If operand transfer were embedded in cca, it would occur during CCA Decode (stage 1), forcing the CCA to wait for DPU register fetch—adding latency. By separating cca_move, operands are pre-loaded into CCA registers before cca executes, so CCA Decode reads from CCA's dedicated register file (no DPU dependency). This preserves the 9-cycle CCA latency and avoids bubbles. However, in some cases (e.g., memory-intensive workloads with high data movement), cca_move overhead can still be non-negligible.Misconception: that combining instructions reduces overhead. Separation enables pipelining—cca_move overlaps with other work, though its cost may not always be fully hidden.
Problem 6 · hard
The paper reports SCAN-SSA achieves 0.92× performance at 8 tasklets vs 16, but cuts control overhead by 54%. Under what conditions is 8 tasklets preferable to 16, and when is it not?
Show answer
Preferable when: (1) workload is memory-intensive (frequent stalls), so 16 tasklets over-saturate → scheduling overhead dominates; (2) energy efficiency matters—54% overhead cut translates to ~1.5× energy savings per DPU, which scales to significant savings across thousands of DPUs; (3) 8% performance loss is acceptable (e.g., batch workloads with soft deadlines). Not preferable when: (1) workload is compute-intensive (e.g., TS: performance still improves 16→12 tasklets, so overhead cost < compute benefit); (2) latency-critical (interactive workloads where 8% matters); (3) single-DPU scenario (overhead savings negligible). The paper suggests compile-time mode selection (energy-efficient vs max-performance) to choose tasklet count per workload.Misconception: that fewer tasklets always improve efficiency. It depends on memory vs compute intensity—the paper's Figure 15 shows varying saturation points (RED/VA at 12, SCAN-RSS/SCAN-SSA at 8, SEL/UNI between).
Problem 7 · easy
CCA's 7-bit code supports three core operations evaluated in the paper. Why only three, and what happens if a workload needs a fourth operation?
Show answer
Three core operations = the subset of dataflow subgraphs identified across PrIM benchmarks + GEMM that the paper evaluates: 4-step multiply, accumulation, and max. The lookup table could store additional MUX configurations for more operations. If a new workload needs a fourth operation: (1) check if it can be decomposed into existing operations (e.g., 2 CCA calls); (2) if not, add the operation to the lookup table (compile-time extension, no hardware change—CCA logic is reconfigurable); (3) if the operation requires new functional units (e.g., FP multiply), hardware must be redesigned (area/power cost). The paper's logic palette construction (Algorithm 2) is designed to accommodate new operations within existing hardware by reusing computing units.Misconception: that CCA is fixed-function. It's reconfigurable via the lookup table—new operations can be added at compile time as long as they map to existing functional units.
Problem 8 · medium
Why does the paper scale synthesis results from 28nm to 25nm instead of directly synthesizing at 25nm?
Show answer
Technology library availability. The paper uses Synopsys Design Compiler with a 28nm CMOS library (standard academic/commercial availability). UPMEM's actual fabrication node is 25nm, so scaling is necessary for fair comparison. The scaling model (Dennard scaling: area ∝ feature²; power ∝ feature, assuming constant voltage) is industry-standard and conservative—real 25nm may perform better due to process improvements. Direct 25nm synthesis would require a 25nm technology library (often proprietary/NDA-restricted), which the paper doesn't have access to. The paper's P&R results at 4 metal layers (matching PIM constraints) validate the scaled estimates are realistic.Misconception: that scaling introduces significant error. For small feature size differences (28→25nm), Dennard scaling is accurate within ~5%. The paper's 0.036% area overhead is robust to this error.
Quick answers

Frequently asked questions

Twelve questions newcomers ask most. Tap any to expand.

Q1What is a CCA, and how is it different from a fixed-function accelerator?▾
A CCA (Configurable Compute Accelerator) is a modular hardware unit composed of generic functional units (shifters, adders, subtractors) connected via MUXes. It's *reconfigurable*: the compiler selects which datapath to activate via a lookup table (CCA code). Fixed-function accelerators (e.g., an FP16 MAC unit) are hardwired for one operation—fast but inflexible. CCA trades peak throughput for versatility: one design supports multiple operations (mul_step, accumulation, max), making it ideal for PIM's diverse hot codes.
Q2Why integrate CCA into PIM instead of using an external accelerator (GPU, FPGA)?▾
Proximity to data. External accelerators require off-chip communication (PCIe, network), reintroducing the memory wall PIM solves. CCA sits *inside* each DPU, accessing WRAM/MRAM directly—no host CPU, no off-chip traffic. Furthermore, external accelerators are overkill for PIM's hot codes (simple multiply-and-add), and their power/area budgets exceed PIM's constraints (3 metal layers, shared power with DRAM). CCA is lightweight (0.036% area, 1.28mW), fits PIM's silicon budget, and offloads only the hot regions that dominate cycles, leaving the rest on the simple DPU. It's *co-located* acceleration, not external.
Q3Can PIM-CCA be applied to other PIM architectures (HBM-PIM, GDDR6-AiM)?▾
Yes—the paper emphasizes this. CCA is modular and compiler-guided, so it integrates with any pipelined PIM processor. HBM-PIM uses FP16 MAC units (fixed-function); CCA could complement them by handling non-MAC hot codes (e.g., indexing, max selection). GDDR6-AiM has compute units per bank; CCA could offload their bottlenecks (e.g., accumulation). Command-driven PIMs (e.g., RowHammer-based) benefit less (no programmable pipeline to offload from), but CCA could still expand their operation set. The core insight—*hot code offload to reconfigurable logic*—is architecture-agnostic. Only the ISA extension and pipeline integration details change.
Q4Why doesn't PIM-CCA support floating-point operations?▾
Area, power, and thermal constraints. FP units (especially FP32) are 5–10× larger and more power-hungry than INT units. Integrating FP into CCA would violate PIM's 0.036% area budget and risk thermal damage to adjacent DRAM cells (FP units generate significant heat). HBM-PIM and GDDR6-AiM support FP16/BF16 via dedicated, heavily-optimized fixed-function units—but even these are limited to simple ops (MAC, accumulation). UPMEM uses software FP emulation (slow); PIM-CCA accelerates the *INT-level primitives* (mul_step, add) that underpin emulation, achieving partial speedup. Future PIM technologies with better thermal management could add FP to CCA, but current memory-centric constraints preclude it.
Q5How does the compiler decide which code regions to offload to CCA?▾
Three-step process: (1) Hot code detection—identify loops/regions with high cycle % (via profiling on uPIMulator or static analysis of IR). (2) Pattern matching—check if the hot code matches a known pattern graph (Algorithm 1); if yes, it's a candidate. (3) Gain prediction—estimate speedup vs overhead (cca_move insertion); offload only if net gain > threshold (e.g., > 5% speedup). The paper uses cycle-accurate simulation to verify gains. For example, TS's multiply-and-add is 95% of cycles and matches pattern 0x0 → high-priority offload. RED's accumulation is frequent but small gains (due to cca_move overhead) → low-priority. The compiler balances coverage (wide patterns) vs efficiency (minimize overhead).
Q6What is the "logic palette," and why is it needed?▾
An intermediate representation between pattern graphs and hardware. Pattern graphs are abstract (nodes = operations), hardware is concrete (stages, MUXes). The logic palette bridges them: it's a staged dataflow diagram showing which computing units execute each operation. Algorithm 2 constructs the palette by mapping multiple pattern graphs to the same hardware, maximizing reuse (e.g., mul_step and accumulation both use adders in stage 0—palette shows them sharing the same 4 adders). Without the palette, each pattern would require separate hardware → many patterns = proportionally higher area. The palette enables *reconfigurability*: one set of functional units, multiple patterns via MUX selection. It's the key to CCA's efficiency.
Q7Why does PIM-CCA use only 4 metal layers in P&R, and what are the implications?▾
PIM fabrication uses 3–4 metal layers (vs 10+ in CPUs) to prioritize memory density. Metal layers enable vertical wiring; fewer layers = limited routing bandwidth, higher resistance. The paper's P&R (Place-and-Route) uses 4 layers to match realistic PIM constraints and prove CCA fits within them. Implications: (1) CCA's datapath must be compact (short wires, local connections); the 7-stage pipeline minimizes long-range routing. (2) The 0.036% area overhead includes routing congestion—synthesis-only estimates are lower, P&R confirms it's realistic. (3) More than 4 mul_steps would exceed the pipeline depth allowed without introducing stalls and control complexity, not primarily a metal-layer limitation. It's a *pipeline-compatibility* design, addressing depth constraints.
Q8How does PIM-CCA handle data dependencies in hot code (e.g., loop-carried dependencies)?▾
CCA executes in pipeline order—dependencies are preserved. Example: accumulation (pattern 0x1) has no loop-carried dependency (each iteration is independent), so CCA offloads trivially. Multiply-and-add (pattern 0x0) has intra-operation dependencies (mul_step n depends on result of mul_step n-1), but CCA's 7-stage pipeline respects them (stage 0 shifts, stage 1 ANDs, stage 2 adds—sequential within the CCA). Cross-iteration dependencies (e.g., result of loop i used in loop i+1) are *not* offloaded—CCA only accelerates independent or intra-iteration ops. The compiler's pattern matching (Algorithm 1) checks for dependencies and skips patterns that violate them. This is why CCA handles hot codes (tight, independent kernels) but not arbitrary control flow.
Q9Why does reducing tasklet count improve energy efficiency but sometimes hurt performance?▾
Trade-off between parallelism and overhead. Tasklets hide latency (while one stalls, others execute), so more tasklets = higher throughput. But each tasklet incurs scheduling cost (tasklet scheduler searches for next executable when many stall) + synchronization cost (barriers, locks). For memory-intensive workloads (e.g., SCAN-SSA), many tasklets stall waiting for memory → scheduler wastes cycles searching → overhead dominates. Reducing tasklets (16→8) cuts overhead (54% in SCAN-SSA) with minimal throughput loss (8%, because memory stalls persist regardless). For compute-intensive workloads (e.g., TS), fewer tasklets under-utilize the pipeline → performance drops. CCA shifts this balance: by accelerating compute, it reduces the need for high tasklet counts, enabling energy-efficient operation (fewer tasklets) without sacrificing throughput. The paper's compile-time mode selection chooses the optimal count per workload.
Q10What happens if a workload's hot code doesn't match any of the CCA patterns evaluated in the paper?▾
Three scenarios: (1) Hot code is decomposable—e.g., 8-operand accumulation = 2× pattern 0x1 (accumulate 5, then accumulate remaining 3). Compiler emits multiple CCA calls. (2) Hot code requires new pattern but reuses existing functional units—e.g., a variant of mul_step with different shift amounts. Compiler adds a new pattern to the lookup table via logic palette construction (Algorithm 2), extending the CCA without hardware change. (3) Hot code requires new functional units—e.g., FP multiply, XOR chains. CCA cannot offload; code runs on main DPU. The paper's three core operations (4-step multiply, accumulation, max) cover compute-intensive benchmarks (GEMV, GEMM, MLP, TS, TRNS) + memory-intensive (for tasklet tuning), demonstrating wide applicability. New workloads typically fall into (1) or (2)—(3) is rare for PIM's memory-centric domains.
Q11Why does the paper use UPMEM as the baseline instead of HBM-PIM or other architectures?▾
Programmability and tool availability. UPMEM provides an open-source SDK (compiler, runtime, simulator) and a standard SPMD programming model, enabling academic research. HBM-PIM and GDDR6-AiM are commercial products with limited public specs and proprietary toolchains. uPIMulator (the cycle-accurate simulator) is UPMEM-based, allowing accurate performance modeling. Furthermore, UPMEM's compute bottleneck (mul_step = 32 cycles) is severe and well-documented, making it an ideal testbed for CCA. The paper's insights (hot code offload, tasklet tuning, reconfigurable logic) are architecture-agnostic—Section 5 (Discussion) explicitly addresses applicability to HBM-PIM, GDDR6-AiM, and command-driven PIMs. UPMEM is the *reference platform*, not the only target.
Q12What are the key limitations of PIM-CCA, and what does future work need to address?▾
Three limitations: (1) Fixed CCA size—4 mul_steps fits UPMEM's 11-stage pipeline; more mul_steps would exceed the allowable pipeline depth and introduce stalls/control complexity, not primarily metal-layer limits. Future work could explore variable-latency CCA or async execution. (2) Limited pattern coverage—three core operations suffice for PrIM benchmarks, but emerging workloads (e.g., transformers, sparse ops) may need new operations. Extending the lookup table is easy, but some operations (e.g., FP, complex control flow) require hardware redesign. (3) Tasklet tuning is compile-time—the paper chooses a fixed tasklet count per workload. Dynamic tuning (adjusting tasklets per phase) could improve efficiency for workloads with shifting characteristics. The paper flags these as future directions, particularly FP support (awaiting better thermal management in next-gen PIM) and multi-threaded workloads (shared data complicates offload).

Core terms
PIM-CCAThe paper's architecture: UPMEM DPU + Configurable Compute Accelerator (CCA) to offload hot code and tune tasklets.
CCA (Configurable Compute Accelerator)A modular, reconfigurable hardware unit (7-stage pipeline: shifters, adders, subtractors, accumulator) controlled by a 7-bit code.
Hot codePerformance-critical, frequently-executed code regions (often loops). In PIM: multiply-and-add (TS 95%), accumulation (GEMM 83%), indexing (TRNS 80%).
DPU (DRAM Processing Unit)UPMEM's in-memory processor: 14-stage pipeline, 11 concurrent tasklets, 64MB MRAM, 64KB WRAM, 24KB IRAM. Baseline for PIM-CCA.
mul_stepUPMEM's single-bit multiplication instruction. 32 mul_steps = 32-bit multiply (32 cycles). CCA offloads 4 mul_steps → 1 CCA cycle (75% reduction).
TaskletUPMEM's software thread (up to 11 concurrent). More tasklets = higher parallelism but higher scheduling overhead. PIM-CCA tunes this trade-off.
Compute intensity% of cycles with 11–16 tasklets active (pipeline saturated with compute). High intensity = compute-bound; low = memory-bound. CCA reduces intensity by offloading hot codes.
CFU (Custom Functional Unit)A building block of CCA logic (e.g., a 4-bit adder). Multiple CFUs + MUXes = reconfigurable datapath supporting multiple operations.
Supporting terms
CCA code7-bit index into CCA lookup table (0x0 = 4 mul_steps, 0x1 = accumulation, 0x2 = max, etc.). Compiler emits this in cca instructions.
cca / cca_moveISA extensions. cca: offload to CCA (6-bit opcode + 7-bit code + 35-bit unused). cca_move: transfer up to 6 operands (regs/immediates) to CCA registers.
Pattern stringHuman-readable spec of a hot code pattern (e.g., "o24 = i24 + i25 + i26 + i27 + i28" for accumulation). Parsed via Lex/Yacc into pattern graph.
Pattern graphCompiler IR for hot code: operator nodes (add, mul_step) + register nodes (i24, o24). Algorithm 1 matches this against IR instructions to identify offload candidates.
Logic paletteIntermediate representation mapping pattern graphs to CCA hardware stages. Algorithm 2 constructs it by reusing computing units across patterns (minimize area).
UPMEMCommercial PIM DIMM: 128 DPUs, 64MB MRAM/DPU, 350MHz, 14-stage pipeline. First widely-available PIM, used as PIM-CCA baseline.
uPIMulatorCycle-accurate UPMEM simulator. Used to profile hot codes, measure compute intensity, and validate PIM-CCA's performance gains.
PrIM benchmarkSuite of 14 memory-intensive workloads for PIM evaluation (GEMV, GEMM, histogram, scan, etc.). PIM-CCA evaluates 14 total benchmarks, including GEMM.
