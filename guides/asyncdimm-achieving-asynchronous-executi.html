HPCA 2025 · A Field Guide

Let host and memory accelerators work at once — without stepping on each other

Your memory ranks sit idle 37–76% of the time while the host runs. Near-memory accelerators could fill that gap — if only you could safely issue commands from both sides. This is the story of AsyncDIMM, a mechanism to offload, schedule, and return memory accesses so host and NMAs run concurrently.

📄 Chen et al. · SJTU · HPCA'25 🎓 Assumes DDR + DIMM basics ⏱ ~25 min read
🔀

Instead of coarse-grained switching that idles either the host or the NMA, offload host commands to the NMA-side MC, schedule them together, and return results — all on existing buses.

Eight concepts the paper assumes

Each card unpacks one building block. Read top to bottom or jump to what's unfamiliar.

1DIMM-NMP architecture▾

A DIMM (Dual In-line Memory Module) holds multiple ranks, each a set of DRAM chips accessed in parallel. DIMM-NMP (near-memory processing) integrates accelerators on the buffer chip of each rank, giving them high-bandwidth local access to DRAM. The host MC sits on the CPU side; NMA MCs sit rank-side.

Now you're ready: AsyncDIMM targets rank-level NMP — one NMA per rank, with its own local MC — and ensures both host and NMA can issue commands without bus collisions or DRAM state errors.
2Memory controller (MC) basics▾

An MC turns read/write requests into DDR commands: ACT (activate a row into the row buffer), RD/WR (read/write columns), PRE (precharge to close the row). It tracks bank states (open/closed), enforces timing constraints (tRCD, tRAS, etc.), and schedules commands to maximize row-buffer hits.

Now you're ready: AsyncDIMM modifies both host MC and NMA MC — the host offloads commands instead of issuing them directly; the NMA MC schedules host commands alongside its own, maintaining correct DRAM states.
3The memory wall for NMP▾

Ideally, compute-intensive tasks run on the host (high ALU throughput), memory-intensive tasks offload to NMAs (high bandwidth). But if both issue commands independently, they may hit the same bank concurrently (violating DRAM's single-row-open rule) or collide on the C/A or data bus. Protocol violations = data corruption.

Now you're ready: AsyncDIMM's goal is to let both sides run at once without protocol violations — achieving "asynchronous execution" that exploits wasted bandwidth instead of coarse-grained switching that idles one side.
4Synchronous execution (the baseline problem)▾

Most existing DIMM-NMP designs (Chameleon, etc.) switch between host and NMA modes in coarse-grained epochs: host executes, then NMA executes, never concurrently. During host mode, NMA ranks sit idle; during NMA mode, host may be starved for bandwidth. The paper shows rank idle time can be 37–76% (2–8 ranks), wasting bandwidth.

Now you're ready: asynchronous execution (concurrent mode) aims to fill those idle cycles with NMA work, raising total throughput. AsyncDIMM's contribution is making it correct and efficient on existing DDR buses.
5Existing async methods (and their limits)▾

Shielding (UPMEM): segregate ranks — host never touches NMA ranks during NMA execution, limiting parallelism. Opportunistic (Chopim): host MC tracks NMA accesses in real time, requires predetermined workload patterns, hurts row locality. Packet-based (HMC-Link): replace DDR bus with packets/NoC, needs new interface pins, costly. All have generality or performance issues.

Now you're ready: AsyncDIMM reuses the existing DDR C/A and data buses, introducing new commands (not a new interface), to achieve concurrency with minimal hardware overhead.
6Explicit and implicit synchronization▾

Explicit synchronization (host-to-NMA, H2N): in host mode, every command the host issues is also forwarded to the NMA MC's FSMs so they track the same DRAM state. Implicit synchronization (NMA-to-host, N2H): before switching back to host mode, the NMA MC issues recovery commands to restore the bank state the host FSMs expect.

Now you're ready: these two syncs enable seamless mode switches (host ↔ NMA ↔ concurrent) without the host having to poll or re-track states — the NMA MC always knows what the host thinks the state is.
7Offload-schedule-return (OSR) mechanism▾

Offload: host MC generates commands as usual but sends offload versions (ACTO, RDO, WRO, PREO) to the NMA MC instead of DRAM. Schedule: NMA MC places them in per-bank FIFOs, arbitrates between host commands and NMA requests (switch-recovery method), issues DDR commands. Return: for offloaded reads, NMA MC notifies host (interrupt), host sends RT command to retrieve data batch.

Now you're ready: OSR is the core mechanism. It turns concurrent access into a producer-consumer problem: host produces offloaded commands, NMA consumes them alongside local requests, and returns results on demand.
8Timing constraints and row locality▾

DDR commands obey strict timing: tRCD (ACT→RD), tBL (burst length), tRAS (row active time), etc. Violating them causes errors. A well-scheduled MC maximizes row-buffer hits (consecutive accesses to the same open row) to avoid costly ACT+PRE overhead. Offloaded commands are pre-scheduled by host MC for row locality; NMA MC preserves it in FIFOs.

Now you're ready: AsyncDIMM optimizes offload timing (omit tRCD for ACTO, use read insertion to avoid write-read/read-write latencies during offloading) and schedules with a switch-recovery method that maximizes row locality between host and NMA streams, keeping bandwidth high.

The whole story

AsyncDIMM in a nutshell

The problem, the mechanism, the performance gain, and how it connects to your tiering/CXL work.

When host and NMAs share a DIMM channel, they both need to issue DDR commands to the same DRAM banks. If both MCs operate independently, they can collide on the bus or issue conflicting commands to the same bank (e.g., host opens row A, NMA opens row B in the same bank simultaneously — DRAM doesn't support it). So most designs adopt synchronous execution: host mode or NMA mode, coarse-grained epochs, mutually exclusive.

But this underutilizes both sides. The paper measures 37–76% rank idle time during host execution (SPEC CPU workloads, 2–8 ranks/channel). Those idle cycles could be filled with NMA work — if only you could coordinate the two MCs safely.

AsyncDIMM's solution: OSR mechanism — offload host commands to the NMA MC instead of issuing them directly, let the NMA MC schedule them alongside its own local requests (maximizing row locality), and return read data on demand via a new RT command. To ensure correctness across mode switches, use H2N explicit sync (every host command updates NMA FSMs in real time) and N2H implicit sync (before switching back, NMA MC restores the bank state the host expects). This requires only four new offload commands (ACTO/RDO/WRO/PREO) and one return command (RT) — reusing the existing DDR bus, no packet controller, no interface changes.

Key optimizations: 1) Omit timing constraints for offloaded commands that don't actually change bank state (e.g., ACTO doesn't open a row, so no tRCD). 2) Switch-recovery arbitration — switch between host-command FIFO and NMA-request FIFO only when no row locality remains (FIFO empty or PRE issued), and issue recovery commands to align DRAM state before actually switching. 3) Adaptive batch with time-division-multiplexed interrupt — instead of polling all ranks when the alert_n signal fires, use cycle timing to identify which rank, and batch-return multiple completed reads in one RT.

The payoff: on 16 host-NMA combinations (8 host workload sets × 2 NMA workload groups: BK with 6 kernels and CK with 5 kernels), AsyncDIMM achieves average speedups of 1.92× over Chameleon, 1.25× over UPMEM, 1.26× over Chopim, and 1.19× over HMC-Link (geometric mean 1.41×), with up to 2.25× NMA bandwidth utilization and 47% host latency reduction. Hardware overhead: host MC area +3.7%, NMA MC area +7.7%, total MC overhead +5.6% (for 1 host MC + 4 NMA MCs = 0.316 mm² / 5.639 mm²); as a fraction of a typical 100 mm² DRAM chip, <1% — all off the critical path.

🍯 Key takeaways

  • Synchronous execution wastes 37–76% rank bandwidth by idling either host or NMA.
  • AsyncDIMM enables concurrent mode: host offloads commands to NMA MC, which schedules them with local requests.
  • H2N explicit sync keeps NMA MC tracking host-issued commands; N2H implicit sync restores bank states before mode switch.
  • OSR = offload (new commands ACTO/RDO/WRO/PREO) + schedule (switch-recovery) + return (RT with TDM interrupt).
  • Timing optimizations: omit tRCD for ACTO (doesn't open row yet); use read insertion to avoid write-read/read-write latency during offload.
  • Switch-recovery: arbitrate FIFOs only when row locality exhausted; issue recovery commands to align DRAM state.
  • Adaptive batch: interrupt identifies completed reads, RT retrieves multiple at once, reducing latency.
  • Result: 1.19×–1.92× speedup (geo. mean 1.41×), 2.25× bandwidth uplift, 47% latency cut, MC area overhead +5.6% (for 1 host + 4 NMA MCs), <1% of DRAM chip area.
💡
Note on generality: The OSR pattern (offload commands to a secondary MC, schedule them, return results) could potentially generalize to other memory-side actors like tier managers or CXL.mem devices. The switch-recovery method's state-alignment approach might apply to scenarios requiring coordination between host and memory-side logic. However, the paper does not evaluate AsyncDIMM for CXL.mem, tiering workloads, or page-granularity operations — such applications would be future work.
Six everyday pictures

How to think about AsyncDIMM

Each analogy stands alone. Every card ends with an honest "where it breaks."

The core idea · Concurrent access

Two cooks, one kitchen

One chef (host) and one assistant (NMA) share a kitchen (DRAM). If both reach for the same pot at once, they collide. Synchronous execution = take turns, one works while the other waits. Asynchronous = assistant takes orders from chef (offload), schedules them with their own tasks (switch-recovery), and passes finished dishes back (return). Both work at once, no collisions.

Where it breaks: a kitchen has physical space for both; DRAM banks allow only one open row at a time. AsyncDIMM's trick is the NMA MC acting as a scheduler/arbiter, not both MCs issuing commands directly.
OSR offload · Command queueing

A dual-lane drive-thru

Instead of taking turns at the window (synchronous), both lanes (host MC, NMA MC) order at once. The host lane's orders go into a separate queue; the kitchen (NMA MC) merges them with its own queue, cooks everything, and passes finished orders back to the host lane's pickup window. Both lanes stay active, no idle time.

Where it breaks: a drive-thru has two physical lanes; AsyncDIMM has one DDR bus. The "queueing" happens in the NMA MC's per-bank FIFOs, not on the bus itself — offloaded commands travel the bus once, get stored, then scheduled.
SR scheduling · Row locality

Batching laundry

You have two piles: host clothes and NMA clothes. Switching piles mid-load wastes time (load/unload dryer = ACT/PRE overhead). The switch-recovery method finishes one pile's load before switching — analogous to draining a FIFO until it hits a PRE (row closed) or empties, then switching to the other FIFO. Maximizes throughput, minimal switching cost.

Where it breaks: laundry is batch-oriented; memory access is request-by-request. AsyncDIMM doesn't batch commands into epochs — it switches per bank FIFO when row locality is exhausted, which can be every few commands. The key is the switch is cheap (only when PRE already issued).
H2N sync · Explicit tracking

A shared clock

Host and NMA keep their own clocks (FSMs tracking bank state). If they drift, they issue conflicting commands. H2N explicit sync = every time the host ticks its clock (issues a command), it also ticks the NMA's clock (forwards the command to NMA FSMs). They stay synchronized continuously, no drift, no conflict.

Where it breaks: a clock ticks whether you look at it or not; FSMs only update on commands. If the host issues no commands (NMA mode), the NMA MC's FSMs diverge from host. N2H implicit sync fixes that by resetting the NMA clock to match the host before switching back (recovery commands).
Return mechanism · RT command

A pickup window

Host orders food (offloaded reads), NMA kitchen cooks it. When ready, NMA rings a bell (interrupt). Host goes to the pickup window (issues RT command) and collects the batch of completed orders. Adaptive batch = if multiple orders are done, return them all at once, minimizing round trips.

Where it breaks: a pickup window is physical; RT is a DDR command on the same bus as all other commands. The NMA MC can't send data at will (would collide with host's bus ownership) — it notifies, host explicitly requests, then data travels. TDM interrupt eliminates polling overhead by encoding which rank in the cycle timing.
Timing optimization · Offload latency

Skipping the commute

Normally ACT takes tRCD cycles to open a row. But ACTO doesn't actually open the row (NMA MC will schedule a real ACT later), so you can skip the tRCD wait and immediately send the next offload command. For writes, interleaving RDO commands avoids write-read/read-write latency. The bus is free sooner, more commands can be offloaded, higher effective bandwidth.

Where it breaks: skipping timing constraints is safe only because the offloaded commands don't touch DRAM yet — they're queued in the NMA MC. When the NMA MC issues real ACT/RD/WR commands, it obeys full timing. Write offload (WRO) cannot simply omit timing because write data occupies the data bus; read insertion optimizes scheduling, not placeholder status.
The full explanation

How it actually works

Section by section, with the paper's key mechanisms rebuilt as step-by-step visuals.

1 · The problem — independent MCs violate DRAM protocol

In DIMM-NMP, the host MC sits on the CPU side, NMA MC sits on the rank side. If both issue commands independently (e.g., host: "ACT bank 0 row A", NMA: "ACT bank 0 row B"), DRAM can't open two rows in the same bank simultaneously — physical violation. Even if they target different banks, bus contention on the C/A or data bus causes signal errors. Packet-based access (HMC-Link) avoids this but requires new interface pins, costly NoC, and complex packet controllers.

🌱
Concretely: with 8 ranks/channel, the host can only access one at a time. The other 7 sit idle 76% of the time (SPEC CPU workloads). Synchronous execution fills that gap by coarse-grained switching, but then either host or NMA is idle for the epoch. AsyncDIMM fills it with concurrent access, exploiting wasted bandwidth.

2 · OSR mechanism overview

AsyncDIMM introduces three phases: offload (host MC generates offload commands, sends them to NMA MC), schedule (NMA MC places them in per-bank FIFOs, arbitrates with local requests, issues real DDR commands), return (for reads, NMA MC notifies host when done, host sends RT command to retrieve data batch). This producer-consumer pattern reuses the existing DDR C/A and data buses — no new interface.

# OSR phases (concurrent mode)
offload: host_MC → ACTO/RDO/WRO/PREO → NMA_MC FIFOs
schedule: NMA_MC arbitrates (host FIFO vs NMA FIFO), issues ACT/RD/WR/PRE to DRAM
return: NMA_MC → interrupt → host_MC → RT → data batch back to host
⚠
Critical requirement: the NMA MC must track the DRAM state the host MC thinks is current (explicit sync) and restore it before switching back to host mode (implicit sync). Otherwise, host-issued commands after the switch would be based on stale state, causing errors.

3 · Explicit synchronization (H2N) — keeping NMA MC in sync

In host mode, every command the host issues (ACT, RD, WR, PRE) is also forwarded to the NMA MC's FSMs. The NMA MC updates its internal bank-state records as if it had issued those commands. When the mode switches to concurrent or NMA, the NMA MC's FSMs already reflect the current DRAM state — no need for host to "hand off" state manually.

4 · Offload — new commands, optimized timing

Instead of issuing ACT/RD/WR/PRE directly to DRAM, the host MC issues ACTO/RDO/WRO/PREO (offload versions) to the NMA MC. These commands are encoded using reserved bits in the DDR command space (e.g., C2-C0 all High for offload). The host MC's FSMs update as if they had issued normal commands, but the commands actually go to the NMA MC's per-bank FIFOs.

Timing optimization: offloaded commands don't actually change DRAM state yet (they're queued), so certain timing constraints can be omitted. For example, ACTO doesn't open a row, so no tRCD wait; RDO doesn't fetch data, so no tRL wait. For writes, WRO commands can use read insertion (interleaving RDO commands) to avoid write-read/read-write latency. This lets the host issue offload commands faster, filling the NMA FIFOs more quickly, raising effective bandwidth.

The paper shows that without timing optimization, offload introduces ~34 cycles latency. With optimization, <4 cycles — negligible, even gains as more ranks are added (higher parallelism).

5 · Schedule — switch-recovery method

The NMA MC maintains two FIFOs per bank: CMD FIFO (host offloaded commands) and REQ FIFO (NMA local requests, which generate commands on the fly). The switch-recovery (SR) unit arbitrates between them. Key rules:

  1. Drain current FIFO until row locality exhausted: switch only when the FIFO empties or a PRE is issued (bank closed). This maximizes row-buffer hits within each stream.
  2. Record last BSC command from each FIFO: the SR unit tracks the last bank-state-change command (ACT or PRE) issued from host and NMA sides.
  3. Recovery before switch: when switching from REQ FIFO to CMD FIFO, check if the DRAM bank state matches the expectation of the last host BSC command. If not, issue recovery commands (e.g., PRE if host expects closed, ACT if host expects open) to align state.
  4. Issue commands: once state is aligned, drain the new FIFO until next switch trigger.
💡
Why this works: host commands are generated based on potentially stale state (host FSMs), but the NMA MC knows the actual state. Recovery commands fix the mismatch, so when host commands execute, they're correct. Meanwhile, the FIFO structure preserves row locality — host MC pre-scheduled its commands for row hits, and the FIFO doesn't reorder them.

6 · Return — RT command with adaptive batch and TDM interrupt

When an offloaded read completes (NMA MC fetched data from DRAM into its buffer), the NMA MC can't send data immediately — the host MC controls bus ownership. Instead, the NMA MC asserts the alert_n interrupt signal. The host MC sees the interrupt, knows which rank via time-division multiplexing (TDM) — each rank's interrupt is valid only in a specific cycle window (e.g., rank 0: cycle 0–7, rank 1: cycle 8–15), so host can identify source without polling all ranks.

The host MC then issues an RT (Return) command to that rank, specifying the batch size (number of completed reads). The NMA MC sends the data batch on the data bus. Adaptive batch: if multiple offloaded reads completed before the host issued RT, the NMA MC returns them all at once, minimizing latency. The batch size is encoded in the RT command (7 bits: BG, BA, C2-C0).

# Return flow
NMA_MC: offloaded RD done → insert data into buffer → assert alert_n
Host_MC: detect alert_n at cycle T → infer rank from T → issue RT(batch=N)
NMA_MC: send N×burst_length data on DQ bus
🍯
TDM interrupt eliminates polling overhead (50–80 cycles to check all ranks). Adaptive batch reduces round trips — if 4 reads are done, one RT retrieves all 4, not 4 separate RTs. Combined, these cut return latency by ~47% in the paper's experiments.

7 · Implicit synchronization (N2H) — restoring state before host mode

When switching from concurrent or NMA mode back to host mode, the host MC's FSMs may have stale bank states (they were updated by offloaded commands that the NMA MC didn't issue yet, or the NMA MC issued different commands). The NMA MC performs N2H implicit sync: for each bank, check if the actual DRAM state matches the expectation of the last host BSC command recorded in the SR unit. If not, issue recovery commands (PRE or ACT) to restore the state. Once all banks are aligned, the mode switches.

The paper shows that on average, <2 banks need recovery per switch, and >50% of switches need no recovery at all (the last host BSC command already matches the current state). So the synchronization overhead is low, often overlapping with the completion of remaining offloaded commands.

Follow the numbers

Worked examples

Four traces from the paper, carried all the way through — offload timing, SR scheduling, return batching, and hardware overhead.

① Offload timing optimization — RDO command

Host MC issues ACTO (bank 0, row 5), then RDO (bank 0, columns 0–7). Without optimization, how much latency? With optimization?

  1. Normal ACT → RD timing: ACT takes tRCD=22 cycles before RD can issue. Then RD takes tRL (CL + tBL) ≈ 22+4=26 cycles to data arrival.
  2. Without optimization: ACTO still waits tRCD=22, RDO waits tRL=26. If NMA-0 gets the commands and NMA-1 is idle, you're wasting NMA-1's bus cycles (rank-to-rank switch tRTRs=4). Total for 2 NMAs: 22+26+4=52 cycles.
  3. With optimization: ACTO doesn't open the row yet (NMA MC will issue real ACT later), so no tRCD wait. RDO doesn't fetch data yet, so no tRL wait. Host issues ACTO, then immediately RDO to NMA-0, then another RDO to NMA-1 (tRTRs=4 only). Total: 4 cycles for offload, then NMA MC schedules the real ACT+RD with full timing.
  4. Bandwidth gain: offloading faster means more commands queued per unit time, higher effective BW. The paper shows ~2.25× BW uplift in some workloads.
🍯
The optimization is safe because the offloaded commands are placeholders — they don't touch DRAM yet. The NMA MC issues real commands with full timing after scheduling. (Paper Fig. 4b)

② SR scheduling — host FIFO vs NMA FIFO

Bank 3 has host CMD FIFO (ACT R1, RD×3, PRE) and NMA REQ FIFO (REQ R0 ×3). Last host BSC = PRE (bank closed), last NMA BSC = ACT R0 (bank open). Walk through the SR scheduling.

  1. Initial state: NMA MC is draining REQ FIFO (NMA requests to row 0). Bank is open with row 0.
  2. REQ FIFO drains: NMA MC issues ACT R0 (already open, so hit), RD×3, then decides to switch (FIFO empty or PRE issued in this case).
  3. Switch to CMD FIFO: SR unit checks: last host BSC = PRE (host expects bank closed), but actual state = open (row 0 from NMA). Mismatch → issue recovery PRE to close bank.
  4. Now state aligned: bank closed, matching host expectation. SR unit issues the host commands: ACT R1 (open row 1), RD×3, PRE (close bank).
  5. Result: host commands execute correctly, no DRAM error. Row locality within each FIFO is preserved (host had 3 RDs to R1, NMA had 3 RDs to R0 — all hits).
💡
The switch happens when row locality is exhausted (FIFO empty or PRE). Recovery commands are cheap (1 PRE = ~22 cycles), and often not needed (50% of switches). This maximizes row-buffer hit rate (~80% in the paper's experiments).

③ Return batching — RT with multiple reads

Host offloaded 4 RDO commands to NMA-0. NMA MC completes them out-of-order (RDO #2 done first, then #1, #3, #4). How does return work?

  1. Completion order: RDO #2 finishes first, data in NMA buffer. NMA MC can't issue interrupt yet — AsyncDIMM enforces offload order for interrupts (oldest first). Wait for RDO #1.
  2. RDO #1 finishes: now the oldest (RDO #1) is done. NMA MC asserts alert_n in its TDM window (e.g., cycle 0 for NMA-0).
  3. Host detects interrupt: cycle 0 → rank 0 (NMA-0). Host checks its Return Unit (RU) table. Host issues RT(batch=1) to NMA-0.
  4. NMA-0 sends data for RDO #1. Host receives it. Meanwhile, RDO #2, #3, #4 also finish. NMA MC issues 3 more interrupts (cycle 0 each time). Host detects all 3, issues RT(batch=3). NMA-0 sends 3×burst data in one go.
  5. Adaptive batch: if host had waited longer before issuing the first RT, it could have collected all 4 interrupts and issued RT(batch=4) once, further reducing latency.
🍯
Batching reduces round trips. TDM interrupt (cycle-based rank ID) eliminates polling. Combined, the paper shows up to 47% latency reduction vs. non-batched return.

④ Hardware overhead — synthesized area and power

Calculate the overhead for a quad-core system (4 NMAs, 1 host MC).

  1. Host MC: conventional host MC area = 2.991 mm², power = 0.491 W. AsyncDIMM host MC area = 3.103 mm² (+3.7%), power = 0.613 W (+24.8%). Extra: offload logic, return unit, offload table (Return Unit structure ~10 KB).
  2. NMA MC (per rank): conventional = 0.662 mm², 0.045 W. AsyncDIMM = 0.713 mm² (+7.7%), 0.079 W (+75.6%). Extra: SR unit, mixed FIFOs (reusing existing FIFO, adding SR logic), return unit.
  3. Total for quad-core: 1 host MC + 4 NMA MCs. Area = 3.103 + 4×0.713 = 5.955 mm². Compare to conventional: 2.991 + 4×0.662 = 5.639 mm². Overhead = +5.6% (0.316 mm² / 5.639 mm²). As fraction of typical DRAM chip (100 mm²): <1%.
  4. Power: 0.613 + 4×0.079 = 0.929 W (conventional: 0.491 + 4×0.045 = 0.671 W, overhead +38.5%). But typical DRAM chip power = 10 W, so AsyncDIMM overhead is ~2.6% of chip power.
  5. Critical path: none of the new logic (offload/return/SR) is on the timing-critical path (host MC command issue, DRAM row access). All off-path, so no frequency penalty.
🍯
For a dual-core system, overhead is even lower (<0.4% area). The paper's synthesis is on 28nm CMOS; newer nodes would shrink it further. The power overhead is acceptable given the performance gain (1.19×–1.92× speedup).
Test yourself

Practice problems

Try each before opening the answer. Every one ends with the misconception it checks.

Problem 1 · easy
Why can't host MC and NMA MC both issue ACT commands to the same bank at the same time, even if they target different rows?
Show answer
DRAM banks have a single row buffer — only one row can be open (active) at a time. Issuing ACT to row A while row B is already open would require closing B first (PRE), then opening A. Two simultaneous ACTs = protocol violation, causing data corruption. The physical DRAM array doesn't support it.Misconception: that DRAM banks can open multiple rows in parallel (like a cache with multiple ways). DRAM is single-row-open per bank, inherently serial within a bank.
Problem 2 · medium
Host MC offloads ACTO (bank 0, row 3), RDO (bank 0, col 0), RDO (bank 0, col 8), PREO (bank 0). NMA MC is currently draining NMA REQ FIFO for bank 0 (row 5 open). When does the NMA MC switch to the host CMD FIFO, and what recovery command(s) does it issue?
Show answer
The NMA MC switches when the REQ FIFO empties or issues a PRE (row 5 closed). At that point, the bank is closed (last NMA BSC = PRE). Host's last BSC = PREO (also expects closed). State matches → no recovery needed. NMA MC directly issues ACTO (row 3), RDO×2, PREO.Misconception: that recovery is always needed. It's only needed when the DRAM state doesn't match the host FSM's expectation. If both sides ended with PRE (closed), no recovery.
Problem 3 · medium
Host offloaded 8 RDO commands to NMA-0. NMA MC completes them all before host issues RT. Host detects one interrupt (cycle 0 = NMA-0). What batch size should host use in the RT command, and why?
Show answer
Batch size = 1 (the number of interrupts seen). Even though 8 reads are done, the NMA MC only issued 1 interrupt (for the oldest offloaded read). AsyncDIMM's interrupt protocol enforces offload order — NMA MC won't issue the next interrupt until the previous one is acknowledged (RT issued). So host issues RT(batch=1), gets first read's data, then NMA MC issues 7 more interrupts (host collects them, issues RT(batch=7) or 7×RT(batch=1)).Misconception: that batch size = number of completed reads. Batch size = number of interrupts host has seen since last RT, enforcing order and preventing data loss.
Problem 4 · hard
Why does AsyncDIMM's timing optimization (omitting tRCD for ACTO) not cause a DRAM timing violation, even though DDR4 spec requires tRCD between ACT and RD?
Show answer
ACTO is an offload command sent to the NMA MC, not a real ACT sent to DRAM. The NMA MC queues it in the CMD FIFO. When the NMA MC later schedules a real ACT command to DRAM (based on the ACTO), it obeys full tRCD timing before issuing the real RD. The optimization applies only to the offload phase (host MC → NMA MC on the C/A bus), not to the actual DRAM access phase (NMA MC → DRAM). DRAM never sees the shortened timing.Misconception: that offload commands are directly issued to DRAM. They're not — they're queued in the NMA MC, which later issues real DDR commands with full timing constraints.
Problem 5 · medium
Explicit sync (H2N) updates NMA MC FSMs in real time during host mode. Why isn't implicit sync (N2H) also real-time? Why wait until mode switch?
Show answer
H2N is unidirectional (host → NMA) and passive (NMA just updates FSMs, doesn't issue commands). N2H would require the NMA MC to notify the host of every command it issues, which adds protocol complexity and latency. Instead, AsyncDIMM records the last host BSC command in the SR unit, and before switching back to host mode, the NMA MC checks if DRAM state matches that expectation. If not, it issues recovery commands. This is cheaper (1–2 commands per bank at most, often none) than continuous real-time sync.Misconception: that explicit and implicit sync are symmetric. They're not — H2N is passive tracking, N2H is active recovery at mode-switch boundaries.
Problem 6 · hard
Chopim (opportunistic access) tracks NMA accesses in real time at the host MC. Why does this hurt row locality, and how does AsyncDIMM avoid it?
Show answer
Chopim's host MC issues NMA accesses with a fixed probability (e.g., 1/4), interleaving them with host accesses at a fine-grained per-command level. This breaks row locality for both sides: host accesses to row A are interrupted by NMA accesses to row B in the same bank, forcing ACT+PRE overhead repeatedly. The paper shows 28–64% row-buffer hit-rate degradation. AsyncDIMM avoids this by segregating host and NMA commands in separate FIFOs and switching only when row locality is exhausted (FIFO empty or PRE). The switch method preserves row hits within each stream.Misconception: that fine-grained interleaving maximizes concurrency. It actually degrades locality. Coarser-grained switching (AsyncDIMM's per-FIFO-drain) is better for bandwidth.
Problem 7 · easy
UPMEM (shielding access) segregates ranks: host never touches NMA ranks during NMA execution. In the paper's 1:3 configuration (1 non-NMP rank, 3 NMP ranks), how does this compare to AsyncDIMM's approach?
Show answer
With 1:3 segregation, the host can only access 1 rank during NMA execution, losing parallelism. The paper reports AsyncDIMM achieves 1.34× higher host bandwidth on average compared to UPMEM. AsyncDIMM lets the host offload to all ranks concurrently, exploiting full channel bandwidth without rank segregation.Misconception: that segregating ranks is "free" for host. It limits host's available bandwidth and parallelism. AsyncDIMM's concurrent access avoids this tradeoff.
Problem 8 · medium
AsyncDIMM's SR method switches FIFOs only when a PRE is issued or FIFO empties. In what corner case could this cause starvation, and how is it prevented?
Show answer
Corner case: if one FIFO (e.g., host CMD FIFO) has continuous row-buffer hits (same row, many RDs) and never issues PRE, it would monopolize the bank, starving the NMA REQ FIFO. Prevention: the MC's open-page policy has a maximum-hit threshold (e.g., 16 consecutive hits to the same row), after which it forces a PRE. When that PRE is issued, the SR method switches FIFOs, giving the NMA side a turn. This is standard in MC designs and AsyncDIMM reuses it.Misconception: that the SR method has no fairness mechanism. It inherits the MC's existing max-hit threshold, which bounds starvation.
Quick answers

Frequently asked questions

Ten questions newcomers ask most. Tap to expand.

Q1What does "asynchronous execution" actually mean here?▾
Host and NMA execute concurrently (asynchronously) instead of in coarse-grained epochs (synchronously). The host offloads memory commands to the NMA MC, which schedules them alongside its own local requests. Both sides issue commands without waiting for the other to finish. The OSR mechanism ensures correctness (no bus collisions, no DRAM errors) despite independent operation.
Q2Why not just use packet-based access like HMC?▾
Packet-based (HMC-Link) requires new interface pins (LxRX/LxTX instead of DDR C/A/DQ), a NoC or crossbar between DIMM slots, and complex packet controllers. It's costly (area, power, signal integrity), limits plug-and-play (conventional DIMMs not compatible), and adds latency (packet header/tail overhead). AsyncDIMM reuses existing DDR buses — no interface changes, lower cost, lower latency. The paper shows AsyncDIMM outperforms HMC-Link by 1.32× (bandwidth) and 38% (latency).
Q3How does TDM interrupt work, concretely?▾
The alert_n signal is shared by all ranks (like C/A bus). Each rank's interrupt is valid only in a specific cycle window: e.g., rank 0: cycle 0–7, rank 1: cycle 8–15, etc. Host MC and NMA MCs synchronize their clocks during calibration. When host detects alert_n at cycle 5, it knows source = rank 0. No polling needed, latency = Nrank/2 cycles on average (vs. 50–80 cycles for polling all ranks). The paper uses 1-bit signal, so negligible overhead.
Q4Why does offload preserve row locality if host MC doesn't know the actual DRAM state?▾
Host MC's FSMs may be stale, but the host scheduler still applies row-locality heuristics (e.g., First-Ready FCFS, open-page policy) when generating commands. Those heuristics produce a stream with row hits clustered together. When offloaded commands arrive at the NMA MC's CMD FIFO, they're already pre-scheduled for locality. The FIFO doesn't reorder them, so the row hits are preserved. The SR method drains the FIFO until a PRE (row closed) or empty, maximizing those hits.
Q5Does AsyncDIMM support dynamic NMA workloads like BFS or SpMV?▾
Yes — unlike Chopim, which requires predetermined latency/pattern. The NMA MC doesn't care about the NMA CU's execution details; it just schedules whatever commands the CU generates. If BFS's loop length varies or SpMV's access pattern is irregular, row-buffer hits may drop, but the SR method still works (switches when PRE or FIFO empty). The paper tests both basic kernels (BK, regular) and complex kernels (CK, irregular) — AsyncDIMM wins on both.
Q6What is the Return Unit structure and how much storage does it require?▾
The Return Unit (RU) in the host MC tracks offloaded reads awaiting return. The paper mentions a structure of approximately 10 KB to support 8 DIMMs or 16 ranks with up to 64 offloaded memory accesses. This is implemented as embedded SRAM in the host MC, similar to read data buffers in commercial MCs.
Q7How does the host MC track bank states if it doesn't see NMA-issued commands in concurrent mode?▾
The host FSMs track bank state (open/closed) and the row address from the last command they issued. In concurrent mode, the host's FSMs update based on its own offloaded commands (ACTO/PREO), not the real DRAM state. The SR unit records the last host BSC command's row information along with the bank state. When switching back to host mode, implicit sync (N2H) uses this recorded information to issue the correct recovery commands (PRE to close, or ACT to the expected row) to align actual DRAM state with what the host FSMs expect.
Q8Can AsyncDIMM support more than one NMA per rank?▾
The paper evaluates one NMA MC per rank and does not design arbitration among multiple NMAs per rank. Extending AsyncDIMM to multiple NMAs per rank would require additional design work — for example, multiple REQ FIFOs with round-robin or priority-based arbitration at the NMA MC, and potentially additional return-path logic. The host side offload mechanism would likely remain unchanged, but the rank-side scheduling would need enhancement.
Q9What did the authors leave for future work?▾
The paper mentions reserved command bits for possible priority/fixed batch fields and evaluates high-performance HMC-Link-p as a point of comparison. Potential extensions could include priority-based scheduling for latency-critical accesses, supporting multiple threads or SMT cores, and exploring sophisticated NMA MC schedulers beyond FIFO. The paper does not explicitly flag specific future-work items beyond these implicit directions.
Q10How does write offload (WRO) differ from read offload in timing optimization?▾
Write offload cannot simply omit timing constraints like read offload because write data occupies the data bus. The paper uses read insertion — interleaving RDO commands with WRO commands — to avoid write-read (tWTR) and read-write (tRTW) timing penalties during the offload phase. This is distinct from the ACTO/RDO optimization (which omits tRCD/tRL entirely). For WRO, the optimization is scheduling-based rather than pure placeholder status.

Glossary & further reading

Every term in one place, then a path to go deeper.

Core terms
AsyncDIMMThis paper's mechanism: asynchronous execution in DIMM-NMP via offload-schedule-return (OSR) on existing DDR buses.
DIMM-NMPNear-memory processing where accelerators (NMAs) are integrated on the DIMM buffer chip, with local MCs.
OSR mechanismOffload host commands to NMA MC, schedule them with NMA requests (switch-recovery), return read data on demand (RT).
H2N explicit syncHost-to-NMA synchronization: every host command updates NMA MC FSMs in real time, keeping them aligned.
N2H implicit syncNMA-to-host synchronization: before switching back to host mode, NMA MC restores bank states to match host FSM expectations.
Switch-recovery (SR) methodNMA MC arbitration: drain current FIFO (host CMD or NMA REQ) until row locality exhausted, then switch; issue recovery commands to align DRAM state.
Offload commandsACTO, RDO, WRO, PREO — host versions of ACT, RD, WR, PRE sent to NMA MC instead of DRAM, with optimized timing (omit tRCD for ACTO, use read insertion for WRO).
Return (RT) commandHost asks NMA MC to send back completed offloaded read data, with batch size encoded in command.
Supporting terms
Synchronous executionBaseline: host and NMA modes alternate in coarse-grained epochs, never concurrent. Wastes bandwidth (37–76% rank idle time).
Asynchronous executionHost and NMA run concurrently, issuing commands at the same time (to different ranks or banks), exploiting idle bandwidth.
Memory controller (MC)Converts read/write requests to DDR commands (ACT, RD, WR, PRE), tracks bank states, enforces timing constraints.
Bank stateOpen (row active in row buffer) or closed (no row active). FSMs track this; SR unit aligns it across host/NMA MCs.
Row localityConsecutive accesses to the same open row (row-buffer hit), avoiding ACT+PRE overhead. AsyncDIMM preserves it via FIFOs and SR switching.
TDM interruptTime-division multiplexing on alert_n signal: each rank's interrupt valid in a specific cycle window, so host infers source rank from cycle timing (no polling).
Adaptive batchHost collects multiple completed offloaded reads (multiple interrupts) and retrieves them all with one RT command, reducing latency.
CMD FIFO / REQ FIFOPer-bank queues in NMA MC: CMD holds host offloaded commands, REQ holds NMA local requests. SR unit arbitrates between them.
BSC commandBank-state-change command (ACT or PRE). SR unit records the last BSC from host and NMA sides to guide recovery.
Timing constraintsDDR rules (tRCD, tBL, tRAS, etc.). Offloaded commands omit some (ACTO no tRCD) because they don't touch DRAM yet.
