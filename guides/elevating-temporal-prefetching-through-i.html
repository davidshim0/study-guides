Cache where it matters, not where it's asked

A few high-impact memory instructions cause most cache misses. Kairos finds them fast, prefetches from them selectively, and discards useless metadata—outperforming state-of-the-art with 251 bytes of fixed overhead.

Before you begin

Eight ideas the paper builds on

These primers cover the cache/prefetch concepts Kairos assumes. Tap any to expand.

critical instruction (high impact) low-impact instruction Kairos machinery prior approaches (Triangel, Triage)
1Temporal vs spatial prefetching▾

Spatial prefetchers learn patterns like strides or next-line accesses (address A, then A+64, then A+128…). They work well for regular patterns but fail on irregular pointer-chasing. Temporal prefetchers instead learn correlation: "address X was followed by Y, then Z"—no stride required.

Temporal prefetchers shine on graph workloads, linked lists, and multi-dimensional arrays where the "next" address depends on data values, not arithmetic.

Now you're ready: Kairos is a temporal prefetcher—it learns which addresses follow each other for a given instruction, even if they're scattered across memory.
2IP-based (per-instruction) learning▾

Different instructions in a program have different memory behaviors. One might stride through an array; another might chase pointers. If you lump them together (global history), their patterns interfere. Tracking patterns per IP (instruction pointer) keeps them clean.

Modern temporal prefetchers (Triage, Triangel, Kairos) all use per-IP metadata tables. Each instruction's miss history is stored separately, indexed by the instruction's address.

Now you're ready: Kairos detects which IPs matter most and only tracks metadata for those, avoiding pollution from rarely-reused instructions.
3Metadata storage — on-chip vs off-chip▾

Learning temporal correlations requires storing "last address → current address" pairs. Early prefetchers (STMS, Domino) kept this in DRAM (off-chip), introducing bandwidth overhead and latency. Recent work (Triage, Triangel, Kairos) moved everything on-chip for speed—but on-chip space is scarce, so what you store matters.

Kairos's breakthrough: store only metadata for critical instructions with proven prefetch utility, dynamically shrinking storage by ~99% vs off-chip approaches while keeping it all on-chip.

Now you're ready: The central challenge is deciding which metadata to keep on-chip—Kairos solves it with instruction correlation.
4Prefetch coverage and accuracy▾

Coverage = fraction of cache misses that a prefetch successfully hides (the prefetch arrived before the miss). Accuracy = fraction of issued prefetches that were actually used before eviction. High coverage means you're catching the misses; high accuracy means you're not polluting the cache.

Kairos prioritizes coverage over accuracy—it's willing to issue more prefetches (lowering accuracy ~10% vs Triangel) to raise coverage by 17.6%, because hiding more misses wins overall.

Now you're ready: Kairos tracks per-instruction prefetch coverage to determine which metadata to keep—the key innovation over Triangel.
5Critical instructions — the 10/90 rule▾

Figure 2 in the paper shows that in 623.xalancbmk_s, the top 10 IPs generate the vast majority of LLC misses; the remaining 3400 instructions collectively cause far fewer. This is the Pareto principle applied to memory: a tiny fraction of instructions dominate cache performance.

If you can identify those ~10 IPs quickly, you can focus all your metadata storage on them and ignore the noise. That's exactly what Kairos does.

Now you're ready: Kairos's Detecting Unit finds critical IPs in one observation window by thresholding each IP's contribution to total misses.
6Metadata pollution and replacement▾

If you store metadata for every miss address without filtering, rarely-reused patterns crowd out frequently-reused ones. This is metadata pollution. Like cache pollution, it degrades performance by evicting useful entries to make room for useless ones.

Triage used Hawkeye (a cache replacement policy) to manage metadata; Triangel pre-sampled to filter low-repetition IPs. Kairos instead uses coverage-based classification: if an IP's prefetches don't cover misses, stop generating metadata for it.

Now you're ready: Kairos combats pollution by marking IPs as positive/neutral/negative based on observed prefetch utility, then adjusting storage/prefetch degree accordingly.
7Dynamic partitioning of cache for metadata▾

Kairos stores its metadata in the LLC itself (treated as data). But how many ways should it steal from normal data cache? Too many → cache performance drops; too few → not enough metadata hits. The trade-off shifts as workload phases change.

Kairos uses a PID controller (feedback loop) to adjust the partition every 218 accesses, balancing miss rate (too little cache) against prefetch utility (too little metadata). This is simpler and cheaper than Triangel's multi-level sampling.

Now you're ready: The Partition Unit is Kairos's third component—it dynamically resizes metadata storage based on observed performance, avoiding over-allocation.
8Prefetch degree and timeliness▾

Degree = how many prefetches to issue per trigger (1, 2, 4…). Higher degree increases coverage but risks lower accuracy and bandwidth waste. Timeliness = issuing early enough to hide latency but not so early the prefetch is evicted before use.

Kairos adapts degree per instruction: positive IPs (high coverage) get degree 4; neutral IPs get degree 1; negative IPs get degree 0 (no prefetch). This balances aggressiveness with accuracy on a per-IP basis.

Now you're ready: Kairos's prefetch method ties degree to classification—another way it leverages instruction correlation to improve efficiency.

The whole story in one place

Kairos in a nutshell

Why existing temporal prefetchers leave performance on the table, and how instruction correlation fixes it.

Temporal prefetchers learn correlations between miss addresses to predict irregular access patterns. But storing all temporal metadata indiscriminately wastes precious on-chip space: rarely-reused patterns crowd out frequently-reused ones, and metadata for low-impact instructions consumes storage better spent on high-impact ones.

Kairos solves this with a three-phase approach grounded in a simple observation: a tiny fraction of instructions generate the vast majority of cache misses (Figure 2 in the paper: the top 10 IPs generate the vast majority of misses in xalancbmk). If you can identify those instructions fast and validate that their prefetches actually help, you can focus all metadata storage on proven winners.

Phase 1: Key IP Detection. The Detecting Unit tracks per-IP miss counts in a small table (32 entries, 680 bits). When an IP's contribution exceeds 12.5% of total misses in a window, it's marked as a key IP and forwarded to the Training Unit. A PLRU policy prevents early-detected IPs from blocking later ones. Reset every 512 misses (the counter saturates).

Phase 2: Prefetch Evaluation. The Training Unit (16 entries, 975 bits) stores each key IP's last miss address and two counters: miss_count (total misses) and useful_pref_count (prefetch hits). When miss_count saturates, Kairos computes prefetch coverage = useful ÷ miss. Coverage ≥87.5% → positive (aggressive: degree 4, metadata kept via LFU replacement); coverage <12.5% → negative (stop prefetching, discard metadata); else neutral (degree 1, standard retention).

Phase 3: Dynamic Partition. Metadata lives in the LLC (like Triage/Triangel). But how many ways to allocate? The Partition Unit runs a PID controller every 218 accesses, adjusting allocation based on miss rate change and prefetch utility (useful_pref ÷ miss). If utility is high and miss rate is dropping, grow metadata partition (max 1 way at a time); if miss rate rises despite prefetching, shrink it. This replaces Triangel's complex set-dueller with a 44-byte feedback loop.

The payoff: Kairos achieves 25.2% geomean speedup vs IP-stride baseline and 10.1% over Triangel (SOTA), with just 251 bytes of fixed overhead (vs Triangel's 17.6 KiB). Coverage jumps 17.6% (48.8% vs 31.2%) while accuracy drops only 10.8% (68.8% vs 79.6%)—a favorable trade. On CloudSuite, Kairos gains 7.3% (vs Triangel's 4.2%). Hardware cost: <1% area, nothing on critical path.

🍯 Key takeaways

  • Existing temporal prefetchers store all metadata equally—Kairos stores only critical IPs (top ~10) with proven prefetch coverage.
  • Detection is fast (one window, 12.5% threshold) vs Triangel's multi-phase sampling—fewer missed prefetch opportunities.
  • Classification by coverage (useful_pref ÷ miss) directly optimizes what matters: hiding misses, not just pattern repetition.
  • PID controller dynamically partitions LLC ways for metadata vs data, replacing Triangel's expensive set-dueller.
  • Result: 10.1% speedup over SOTA with 251 B fixed overhead vs 17.6 KiB.
  • Coverage +17.6%, accuracy -10.8%—the coverage gain dominates, proving aggressive prefetch from proven IPs wins.
Seven everyday pictures

How to think about Kairos

Each analogy stands alone. Read in any order; each ends with an honest "where it breaks."

The core idea · Pareto principle

Vital few, trivial many

In any system, ~20% of causes produce ~80% of effects. For Kairos, it's even more skewed: ~10 instructions (out of thousands) cause >90% of cache misses. Instead of tracking every instruction equally, Kairos finds that vital few fast (Detecting Unit), invests all metadata storage in them (Training Unit), and ignores the trivial many. Like focusing your debugging on the hot loop, not the entire codebase.

Where it breaks: if misses were uniformly distributed across all IPs, this approach would fail. The paper shows real workloads are highly skewed (Figure 2), but pathological cases exist.
Detection · Fast filtering

The loudest voices

In a meeting, you quickly notice who's talking most—no need to count every word. Kairos's Detecting Unit similarly finds critical IPs by watching miss counts in real-time. When an IP crosses 12.5% of the total-miss counter, it's "loud enough" to matter. A PLRU policy ensures late bloomers aren't blocked by early talkers. Reset every 512 misses to adapt to new phases. Simple, fast, effective.

Where it breaks: if a phase shift happens mid-window, the Detecting Unit won't see it until the next reset—a lag of up to 512 misses. Triangel's continuous sampling avoids this, at higher cost.
Classification · Coverage tiers

Batting average for prefetchers

A baseball player with a .900 average (90% hits) bats cleanup; one with .100 sits the bench. Kairos does this for IPs: positive IPs (coverage ≥87.5%) get aggressive prefetch (degree 4, LFU retention); negative IPs (<12.5%) get benched (degree 0, metadata discarded); neutral IPs (middle) get cautious treatment (degree 1). Coverage = useful_pref ÷ miss—the "batting average" for hiding misses.

Where it breaks: coverage is measured over a window (8-bit counter saturation). If an IP's behavior oscillates faster than the window, classification lags. The paper's counters reset every ~256 misses, a trade-off.
Partitioning · PID feedback

Thermostat for metadata

A thermostat adjusts heat based on temperature deviation, not on-off switches. Kairos's PID controller similarly adjusts metadata cache allocation: if prefetch utility is high and miss rate is dropping, grow the partition (give metadata more LLC ways); if miss rate rises despite prefetching, shrink it (reclaim ways for data). Proportional (current utility), Integral (miss rate delta), Derivative (delta-delta) terms smooth the response. Re-evaluates every 218 accesses.

Where it breaks: PID assumes the relationship between metadata size and performance is smooth. Pathological workloads with sharp phase boundaries might oscillate. The paper's conservative parameters (α=1, β=-0.3, γ=0.1) prioritize stability over agility.
Overhead · Minimal footprint

Pocket knife vs toolbox

Triangel carries a full toolbox (17.6 KiB: sampler, reuse buffer, set dueller, training table). Kairos brings a pocket knife (251 B: 32-entry Detecting Unit, 16-entry Training Unit, PID controller). Both solve the problem, but Kairos fits in your pocket. The secret: focus on proven critical IPs instead of sampling everything. Fixed structural overhead is 251 B vs Triangel's 17.6 KiB, 10.1% better performance.

Where it breaks: if you need to track 100s of critical IPs (extremely diverse workload), Kairos's 16-entry Training Unit might thrash. The paper's workloads have ≤16 critical IPs per phase, validating the design.
Adaptation · Phase tracking

Speedometer recalibration

Your car's speedometer doesn't average speed since ignition—it updates every second. Kairos resets its Detecting Unit every 512 misses and its Training Unit counters at saturation (8 bits → ~256 events). This lets it track phase changes: if a nested loop exits and a new one starts, the critical IPs shift. Old IPs fall out, new ones appear. The PID controller adjusts metadata allocation continuously, so the system adapts without manual tuning.

Where it breaks: extremely fast phase changes (every few hundred misses) might arrive between resets, causing Kairos to miss opportunities. The paper shows typical phases last longer, so the reset intervals work well in practice.
The full explanation

How it actually works

Step by step, with the paper's mechanisms spelled out in plain English.

1 · Detecting Unit — finding critical IPs fast

The Detecting Unit is a 32-entry table (10-bit IP tag, 1-bit key flag, 9-bit miss counter per entry, plus a 31-bit PLRU and 9-bit total miss counter). On every L2C miss:

  1. Hash the IP to an entry. If the key flag is already set (this IP was previously deemed critical), forward the IP to the Training Unit immediately—no further tracking needed.
  2. Otherwise, increment both the entry's miss counter and the total miss counter. Check: does this entry's count now exceed 12.5% of the total? If yes, set the key flag and forward to Training Unit.
  3. Update PLRU on every access, so frequently-missing IPs aren't evicted prematurely by one-hit wonders.
  4. When the total miss counter saturates (512 misses), reset all counters and begin a new detection window. This prevents early-detected IPs from permanently hogging space.

Why 12.5%? A deliberately modest threshold (1/8 of misses). It captures the top ~10 IPs without being so strict that only the absolute top IP qualifies. Sensitivity analysis (not detailed in the paper but implied) shows this balances detection speed with coverage.

2 · Training Unit — learning and classifying patterns

The Training Unit is a 16-entry table (10-bit IP tag, 32-bit last address, 8-bit useful_pref counter, 8-bit miss counter, 2-bit key tag). When a memory access hits the L2C (either a miss or a prefetch hit):

  1. Search for the IP. If found and the access is a miss or prefetch hit, increment the miss counter. If it's a prefetch hit specifically (cache hit with prefetch tag), also increment useful_pref.
  2. Generate correlation metadata: use the last address (stored in this entry) to create a prediction entry (XOR high/low bits for set index, low bits for tag). Insert into the metadata cache (LLC partition) with a replacement bit set according to the IP's current key tag.
  3. Update the last address field to the current miss address, preparing for the next correlation.
  4. When the miss counter saturates (256), compute coverage = useful_pref ÷ miss. If coverage ≥87.5%, set key tag = +1 (positive); if <12.5%, set key tag = -1 (negative); else key tag = 0 (neutral). Reset both counters and start a new window.

Replacement policy: metadata generated from positive IPs gets rpl=1 (LFU retention); negative IPs stop generating metadata entirely; neutral IPs get rpl=0 (standard LRU-like eviction). This keeps proven-useful metadata in the cache while discarding low-utility entries.

3 · Partition Unit — dynamic LLC allocation

The Partition Unit runs a PID controller every 218 accesses. It tracks three 18-bit counters: access_count, miss_count, useful_pref_count. At each interval:

  1. Compute current miss rate = miss / access, prefetch utility = useful_pref / miss, and delta miss rate vs previous window.
  2. Compute PID signal: δ = 1.0·utility − 0.3·Δmiss + 0.1·ΔΔmiss. The proportional term (utility) pushes for more metadata if prefetches help; the integral term (Δmiss) penalizes if misses are rising; the derivative term (ΔΔmiss) smooths oscillations.
  3. Decide: if δ > 0.25 or (useful_pref/access) > current partition ratio, grow metadata by 1 way. If δ < -0.5 and current miss rate > 1.2× previous, shrink by 1 way. Else maintain. Cap at ±1 way per interval to limit churn.
  4. If performance improvement is <5% for 2 consecutive intervals, hold steady to avoid oscillation during stable phases.

Parameter choice: α=1, β=-0.3, γ=0.1 are derived from control theory (proportional response, negative feedback to limit growth, damping). The paper argues these are system-agnostic because they're balancing a fundamental trade-off (metadata overhead vs prefetch benefit), not tuned to a specific cache size.

4 · Prefetch method — adaptive degree

On an L2C miss, if the IP is in the Training Unit and has metadata:

  1. Look up the current miss address in the metadata cache (the LLC partition). If found, the metadata entry contains the next predicted address.
  2. Issue prefetches based on the IP's key tag: positive → degree 4 (follow the correlation chain up to 4 hops); neutral → degree 1; negative → degree 0 (no prefetch).
  3. If a prefetch hits metadata that was itself prefetched and used, set that metadata's rpl bit to 1 (reinforcement—boost its retention).

Why variable degree? Positive IPs have proven high coverage, so aggressive prefetch (degree 4) is safe. Negative IPs waste bandwidth, so don't prefetch. Neutral IPs are still under evaluation—degree 1 is conservative exploration.

🍯
Why this beats Triangel: Triangel pre-samples IPs to verify repetition ≥threshold, requiring a multi-level structure (sampler, SSampler, reuse buffer, set dueller) totaling 17.6 KiB. Kairos instead detects critical IPs via miss contribution (one simple table) and validates utility directly via coverage tracking (another simple table). No pre-sampling = no lost prefetch opportunities during observation. Coverage-based classification = directly optimizing the goal (hiding misses), not a proxy (pattern repetition).
Follow the numbers, step by step

Worked examples

Four traces using the paper's actual data—detection thresholds, coverage ratios, PID decisions, and final speedup.

① Detecting a critical IP in xalancbmk

From Figure 2: the top 10 IPs generate the vast majority of misses in 623.xalancbmk_s. Walk through detection.

  1. Initial state: Detecting Unit has a 9-bit total miss counter (max 512). Assume the observation window has recorded 500 total misses so far.
  2. One critical IP contributes significantly: It has accumulated enough misses to exceed the 12.5% threshold early in the observation window.
  3. Check threshold: When an IP's miss count exceeds 12.5% of the total miss counter (62.5 out of 500), the key flag is set.
  4. From this point on, every miss from this IP goes directly to the Training Unit, bypassing further detection tracking. The Detecting Unit focuses on finding the next critical IP among the remaining candidates.
  5. Result: Critical IPs are classified as key IPs within one observation window, not after extensive sampling. Training begins immediately.
💡
Why so fast? The 12.5% threshold is deliberately low. An IP causing even 1/8 of misses is significant enough to track. Figure 2 shows the top IPs in xalancbmk dominate miss counts, so they trip the threshold quickly.

② Classifying an IP as positive in 450.soplex

An IP in the Training Unit has observed 200 misses and 180 useful prefetches. Compute coverage and classify.

  1. Counters read: miss_count = 200, useful_pref_count = 180. (The 8-bit counters saturate at 256; this IP is nearing saturation.)
  2. Compute coverage: 180 ÷ 200 = 0.90 = 90%.
  3. Check thresholds: 90% ≥ 87.5%, so this IP is classified as positive. Set key tag = +1.
  4. Consequences: future metadata generated by this IP will have rpl=1 (LFU retention), and future prefetches will use degree 4 (aggressive).
  5. Reset counters: both miss_count and useful_pref_count → 0, start a new window. If the next window also shows ≥87.5%, the IP remains positive; if it drops to 50%, it reverts to neutral.
📝
The paper reports Kairos achieves 1.53× speedup on 450.soplex (Figure 8), one of the highest gains. This is likely because soplex has a small set of critical IPs with very high coverage (>90%), allowing Kairos to focus all metadata storage on proven winners and issue aggressive prefetches.

③ PID controller deciding to grow the metadata partition

At interval k, Kairos observes: access_count = 200k, miss_count = 6k, useful_pref_count = 4.8k. Previous interval had miss_count = 7k (out of 200k accesses). Current metadata partition is 2 ways. Decide.

  1. Current metrics: miss rate = 6k ÷ 200k = 3.0%. Utility = 4.8k ÷ 6k = 80%. Previous miss rate = 7k ÷ 200k = 3.5%.
  2. Delta terms: Δmiss = (3.0% − 3.5%) = -0.5% (misses dropped). ΔΔmiss = change in Δmiss vs previous interval (assume previous Δmiss was -0.3%, so ΔΔmiss = -0.5% − (-0.3%) = -0.2%).
  3. Compute δ: δ = 1.0 × 0.80 + (-0.3) × (-0.005) + 0.1 × (-0.002) = 0.80 + 0.0015 − 0.0002 ≈ 0.80.
  4. Check thresholds: δ = 0.80 > 0.25 (θ_plus), so increase metadata partition by 1 way (2 → 3 ways).
  5. Reasoning: high utility (80%) and dropping miss rate signal that prefetching is working well and more metadata space could improve coverage further. The PID controller allocates more LLC ways to metadata.
🍯
The PID's role: unlike Triangel's set-dueller (which compares two fixed policies across sampled sets), Kairos's PID continuously adjusts based on actual observed performance. If prefetches help, grow; if they hurt (miss rate rises), shrink. This is simpler (44 bytes vs Triangel's set dueller overhead) and more direct.

④ End-to-end: xalancbmk speedup calculation

Figure 8 shows Kairos achieves ~1.85× speedup on 623.xalancbmk_s. Trace the mechanism: detection → classification → prefetch → speedup.

  1. Baseline (no prefetch): xalancbmk has high LLC miss rate. Cache misses incur significant latency penalty.
  2. Kairos detects top critical IPs (Figure 2: top IPs generate the vast majority of misses). These IPs are marked as key, trained, and likely classified as positive (Figure 9 shows Kairos coverage ~70% on xalancbmk).
  3. Prefetch coverage: 70% of misses are successfully hidden by prefetches, significantly reducing memory stall time.
  4. Accuracy consideration: Figure 10 shows Kairos accuracy ~65% on xalancbmk. The coverage gain dominates any minor cache pollution from lower accuracy.
  5. Speedup: The substantial reduction in memory stall time from hidden misses translates to the observed 1.85× speedup. The paper reports 1.85×, demonstrating the benefits of higher-degree prefetch chains and metadata cache hits reducing follow-on misses.
💡
The paper's geomean speedup (25.2% over baseline, 10.1% over Triangel) hides variation: xalancbmk is a best case (irregular, high temporal locality); omnetpp is harder (lower coverage). Kairos's strength is adapting per-workload via classification.

Problem 1 · easy

Show answer
50 ÷ 500 = 10%, which is <12.5%, so no, it is not yet marked as a key IP. Problem 5 · medium
Kairos achieves 48.8% coverage vs Triangel's 31.2% (+17.6%), but accuracy drops 79.6% → 68.8% (-10.8%). Why does this trade-off favor Kairos? Explain using relative costs.
Show answer
The trade-off favors Kairos because the benefit of hiding misses (via coverage) far outweighs the cost of some wasted prefetches (lower accuracy). Each hidden miss saves the full memory access latency (hundreds of cycles), while each wasted prefetch costs only the bandwidth and minor cache pollution of bringing in an unused line. Since Kairos hides significantly more misses (+17.6% coverage improvement) at the cost of moderately more wasted prefetches (-10.8% accuracy), the net performance gain is substantial. This is evidenced by Kairos's 10.1% speedup advantage over Triangel despite the lower accuracy.Misconception: that accuracy must stay high for good performance. The paper demonstrates that when miss penalties are much larger than prefetch costs, prioritizing coverage over accuracy yields better overall speedup.
Problem 6 · easy
The Detecting Unit has 32 entries and a PLRU policy. Why PLRU instead of true LRU, and what's the cost of this approximation?
Show answer
True LRU for 32 entries requires ~155 bits of state (log2(32!) ≈ 118, but practical implementations use ~5 bits per entry). PLRU (pseudo-LRU) uses a tree of bits (31 bits for 32 entries) to approximate LRU with lower overhead. The cost: PLRU can evict an entry that was accessed more recently than another, but in practice the error is small—PLRU's victim is among the ~least-recently-used, which is good enough. For Kairos, the 31-bit PLRU (vs 160-bit true LRU) saves ~16 bytes of overhead, keeping the Detecting Unit at 85 B total.Misconception: that PLRU is a significant performance degradation vs true LRU. Studies show PLRU matches LRU to within ~1–2% miss rate on typical workloads; the savings in implementation complexity are well worth it.
Problem 7 · hard
Kairos's PID parameters are α=1, β=-0.3, γ=0.1. The paper claims they're "system-agnostic" because they balance a fundamental trade-off, not tuned to specific hardware. Explain this claim using control theory principles, and identify a scenario where the parameters might need adjustment.
Show answer
In control theory, PID gains are chosen based on the process dynamics (how the system responds to input), not the absolute scale of the system. Here, the process is: "more metadata → higher prefetch utility → lower miss rate (but also less data cache)." The trade-off is inherent to any cache/metadata split, regardless of cache size. α=1 (proportional) makes the response linear in utility (normalized 0–1); β=-0.3 (integral) adds negative feedback to prevent runaway growth; γ=0.1 (derivative) damps oscillations. These gains work as long as utility and miss rate are normalized metrics, which they are. Scenario requiring adjustment: if the LLC replacement policy changed from LRU to something with different eviction dynamics (e.g., RRIP), the relationship between metadata size and miss rate might shift—then β/γ might need retuning to match the new response curve. But for standard LRU-based LLCs (most commercial systems), the parameters hold.Misconception: that "system-agnostic" means "one-size-fits-all." It means the parameters don't depend on cache size or bandwidth, but they do assume a certain replacement policy and miss cost model. Exotic systems (e.g., HBM with 10× bandwidth, or NUCA caches with non-uniform latency) might need different gains.
Problem 8 · medium
Figure 15 shows Kairos achieves 29.9% speedup on multi-core SPEC mixes vs 18.7% for Triangel. Why does the gap widen in multi-core (vs 10.1% in single-core)? Hint: think about LLC contention and metadata sharing.
Show answer
In multi-core, the LLC is shared among cores, so metadata from all cores competes for space. Triangel stores metadata for more IPs per core (no aggressive filtering by coverage), so with 4 cores, the metadata footprint is ~4× larger—leading to more evictions of useful metadata and interference between cores. Kairos stores only critical, high-coverage IPs per core, so its per-core metadata footprint is smaller. With 4 cores, the total metadata is still manageable, and the PID controller can allocate enough LLC ways to cover all cores' critical IPs without thrashing. Result: Kairos maintains high coverage (and thus speedup) in multi-core, while Triangel's larger footprint degrades performance under contention.Misconception: that multi-core simply scales single-core results linearly. Shared-resource contention (LLC, bandwidth) can amplify differences between prefetchers—Kairos's efficiency advantage (251 B overhead vs 17.6 KiB) matters more when resources are scarce.
Quick answers

Frequently asked questions

Ten questions newcomers ask most. Tap any to expand.

Q1What's the difference between "critical IP" and "key IP"?▾
The paper uses them interchangeably. Both refer to an instruction (identified by its IP = instruction pointer) that the Detecting Unit has flagged as contributing >12.5% of misses in the current window. Once flagged, the IP is forwarded to the Training Unit and its miss/coverage behavior is tracked.
Q2Why doesn't Kairos use Belady's optimal replacement (like Hawkeye) for metadata management?▾
Belady's algorithm requires knowing the future—specifically, which cache line will be re-referenced farthest in the future. For metadata, "future" means "will this IP generate more prefetchable misses?" Kairos doesn't predict that; instead, it uses observed past coverage (useful_pref ÷ miss) as a proxy. This is simpler than running a PC-based predictor (like Hawkeye's OPTgen) and directly targets the goal: keep metadata that covers misses.
Q3How does Kairos handle multi-threaded workloads?▾
The paper evaluates multi-core with multi-programmed workloads (each core runs a different application) and demonstrates strong performance gains (Figure 15: 29.9% speedup in 4-core mixes). Kairos tracks IPs per-core independently. The shared LLC metadata partition is dynamically sized by the PID controller to accommodate all cores' critical IPs.
Q4Why 12.5% for the detection threshold and 87.5%/12.5% for coverage classification?▾
12.5% = 1/8, a simple fraction that's low enough to catch the top ~10 IPs without being so strict that only the #1 IP qualifies. For coverage, 87.5% = 7/8 (high bar for "positive"), 12.5% = 1/8 (low bar for "negative"). These are deliberately conservative: an IP must consistently achieve ≥87.5% coverage to earn aggressive prefetch (degree 4), and must fail badly (<12.5%) to be benched. The paper doesn't report sensitivity to these exact values, but the geometric simplicity (powers of 2) suggests they're chosen for implementation ease (bit shifts) and robustness.
Q5What happens to metadata when an IP's classification changes from positive to negative?▾
When an IP is reclassified as negative (coverage drops below 12.5% in a new window), the Training Unit sets its key tag = -1, and future metadata generation for that IP stops. Existing metadata in the LLC partition (generated when the IP was positive/neutral) remains until it's evicted by the normal replacement policy (LRU-like, since rpl=0 for old entries). The IP's entry in the Training Unit stays, continuing to track coverage in case it improves and the IP returns to neutral/positive in a later window.
Q6How does Kairos compare to Berti (a leading spatial prefetcher)?▾
On irregular workloads (SPEC CPU Irregular, CloudSuite), Kairos significantly outperforms Berti (25.2% vs ~8% geomean speedup, estimated from Figure 8). On regular workloads (SPEC CPU Regular, GAP), Berti wins (Figure 16). The paper proposes a composite prefetcher (Kairos+Berti) that combines both: Kairos implemented at L2C for temporal patterns, Berti at L1D for spatial patterns. The two are orthogonal and don't interfere. Result: 0.3% gain on regular, 3.2% on GAP, no loss on irregular.
Q7Why does Kairos reset the Detecting Unit every 512 misses instead of, say, every 1M instructions?▾
512 misses ≈ one counter saturation (9 bits). Using misses (not instructions) as the reset trigger adapts to miss rate: high-MPKI workloads reset more frequently (more opportunities to detect phase changes); low-MPKI workloads reset less often (stable phases). An instruction-based reset would be phase-agnostic—a high-MPKI phase could persist across many resets, while a low-MPKI phase would reset too often. The paper's choice ties the observation window to the event of interest (misses), a common technique in adaptive hardware.
Q8Could Kairos's PID controller oscillate if the workload phases change rapidly?▾
Yes, but the paper's conservative parameters (β=-0.3, γ=0.1) are designed to damp oscillations. The derivative term (γ) penalizes rapid changes in Δmiss, stabilizing the response. Additionally, the decision logic includes a "hold steady" rule: if performance improvement is <5% for 2 consecutive intervals, maintain the partition. This prevents thrashing during marginal phases. Figure 17 shows that in practice, Kairos maintains the partition ~60–80% of the time across workloads—only adjusting when there's a clear signal (large δ or significant miss rate change).
Q9What's the bandwidth overhead of storing metadata in the LLC?▾
Figure 13 shows Kairos's LLC traffic is higher than Triangel's but lower than Triage's. Each metadata insertion (on a miss by a key IP) and lookup (on a miss) involves LLC accesses. The paper shows that despite this overhead, Kairos's overall performance improvement (25.2% speedup) justifies the bandwidth cost. Figure 14's energy analysis shows Kairos's dynamic energy overhead (~30.7%) is comparable to other prefetchers, and the speedup gain outweighs the energy cost.
Q10Why doesn't Kairos use off-chip storage like early prefetchers (STMS, Domino)?▾
Off-chip storage (DRAM) introduces two problems: (1) latency—fetching metadata from DRAM takes significant time, delaying prefetch issue and reducing timeliness; (2) bandwidth—every metadata access consumes precious DRAM bandwidth, interfering with demand traffic. Triage (2019) showed that on-chip storage, despite being smaller, outperforms off-chip by keeping metadata access latency low (cache hit latency). Kairos continues this trend, using the LLC (already on-chip) as metadata storage and dynamically sizing it via the PID controller. The result: metadata hits are fast, and the storage overhead is justified by the speedup (25.2%).

Connection to your research (LtRAM / CXL / tiering)

Coverage over accuracy for asymmetric costs

Kairos prioritizes coverage (+17.6%) over accuracy (-10.8%) because hiding a miss to slow memory is worth far more than a wasted prefetch to fast memory. In tiering, hiding a CXL/NVM miss similarly dominates the cost of a speculative DRAM prefetch. Lesson: aggressive prefetch to the slower tier is correct when latency gaps are large—accept some pollution for higher coverage.

Instruction-level correlation for irregular patterns

Kairos exploits the fact that a few instructions cause most misses. In OS-managed tiering, a few system calls or hot functions likely dominate tier migrations. Tracking per-PC or per-call-site migration history could identify "critical promotion candidates" faster than global LRU—mirroring Kairos's IP-based detection.

Dynamic partitioning via feedback

Kairos's PID controller adjusts metadata/data partition based on observed miss rate and prefetch utility. In tiering, you could use a similar controller to adjust DRAM/NVM capacity allocation: if promotion rate is high and NVM miss rate is dropping, grow DRAM partition; if DRAM is underutilized, shrink it. The PID parameters would need retuning for migration latency (μs scale) vs cache latency (ns scale), but the principle holds.

Multi-core contention scaling

Kairos's advantage grows in multi-core (10.1% → 11.2%) because its low overhead (251 B per core) avoids shared-resource thrashing. In a multi-core CXL system, per-core tiering metadata (promotion candidates, access counters) similarly competes for bandwidth and memory. A lightweight, coverage-based approach (promote only high-impact pages) could outscale a heavier, global approach (track all pages) under contention.

Temporal Memory Streaming hard

Wenisch et al. · ISCA 2005, TC 2010

The foundational work on correlation-based (temporal) prefetching. Introduced the off-chip metadata approach (STMS) that Kairos improves upon. Essential background for understanding why on-chip is necessary.

Practical Temporal Prefetching (Triage) medium

Wu et al. · MICRO 2019

First fully on-chip temporal prefetcher. Used Hawkeye for metadata management. Kairos builds on Triage's insight (on-chip is viable) but replaces Hawkeye with coverage-based classification for better targeting.

Triangel hard

Ainsworth & Mukhanov · ISCA 2024

The SOTA that Kairos outperforms. Pre-sampling approach to filter IPs by repetition rate. Read this to understand the comparison—Kairos's detection is simpler and faster, avoiding the sampling window's prefetch opportunity loss.

Berti medium

Navarro-Torres et al. · MICRO 2022

SOTA spatial prefetcher using IP-based offset learning. Orthogonal to Kairos (spatial vs temporal). The paper shows Kairos+Berti as a composite prefetcher captures both pattern types—relevant if your tiering system needs to handle both regular (stride) and irregular (pointer-chasing) access patterns.

Hawkeye Cache Replacement hard

Jain & Lin · ISCA 2016

Belady-optimal replacement via PC-based learning. Used by Triage for metadata management. Kairos's simpler coverage-based approach achieves similar goals (keep useful metadata) without Hawkeye's OPTgen overhead—worth understanding if you're designing a tiering replacement policy.
