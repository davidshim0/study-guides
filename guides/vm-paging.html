Virtual memory & the page-table walk

Every load is a promise the hardware keeps lazily. Master the walk from a virtual address to a physical frame — and the trap that fires when it can't — because that trap is exactly where your LtRAM tiering lives.

🧩 os.mem.virtmem + os.mem.paging 🐧 Linux 6.8 · x86-64 🎯 target: design
🗺️

The MMU walks your page tables in hardware to translate every address. When a translation is missing or violates permissions, it raises #PF and the kernel takes over — populating the mapping, copying a page, or migrating it. Your tier hooks that handoff.

The cast of characters

Eight structures this lesson leans on

You think in caches and TLBs already, so these are quick framings — what each thing is, and the hardware-vs-kernel side it lives on. Skim, or open whichever you want to nail down.

1Virtual vs physical address & the MMU HW▾

Every instruction emits a virtual address (VA). The MMU, in hardware, translates it to a physical address (PA) using page tables the kernel built. The CPU core never touches physical memory directly; isolation and relocation are free side-effects of that indirection.

Ready: the whole lesson is "how the MMU turns a VA into a PA, and what happens when it can't."
2Page & frame (the unit of mapping) HW▾

Translation happens at page granularity: 4 KiB (the base), or 2 MiB / 1 GiB huge pages. A virtual page maps to a physical frame. The low 12 bits (for 4 KiB) are the byte offset and pass through translation untouched — only the page number is translated.

Ready: 4 KiB ⇒ 12-bit offset; the rest of the VA indexes the page tables.
3The TLB (translation cache) HW▾

Walking page tables on every access would be ruinous, so the TLB caches recent VA→PA translations. A hit skips the walk entirely. A miss triggers the hardware page-table walk. TLB reach = entries × page size; huge pages exist largely to stretch it.

Ready: the page-table walk is the slow path behind a TLB miss — and editing a PTE means you must invalidate stale TLB entries (shootdown), which Lesson on os.mem.tlb covers.
4The PTE and its key bits SWHW▾

The leaf of the walk is a page-table entry (PTE): a physical frame number plus flag bits. The ones you care about: _PAGE_PRESENT (is it mapped?), _PAGE_RW (writable?), _PAGE_ACCESSED/young and _PAGE_DIRTY (set by hardware on use/write), NX (no-execute).

Ready: this is a preview — the next lesson (os.lmm.pte) is all about these bits, because the dirty bit is your placement signal and write-protect is your CoW trap.
5mm_struct & the VMA list SW▾

Each process has an mm_struct describing its address space, holding a set of vm_area_struct (VMAs). A VMA is one contiguous region (a mapping) with permissions in vm_flags (VM_READ/WRITE/EXEC). The VMAs are the kernel's intended map; the page table is the realized map.

Ready: on a fault the kernel finds the VMA (find_vma) to decide what the page should be — and your project reads vm_flags to route RO regions to the tier.
6The page fault (#PF) is a fault FAULT▾

A fault is a synchronous, restartable exception: the CPU pushes state, jumps to the handler, and on return re-executes the faulting instruction. The page fault (#PF, vector 14) hands the kernel the faulting address (in CR2) and an error code (present? write? user?). Not a crash — the normal way the kernel populates memory lazily.

Ready: "restartable" is why CoW and migration work: the kernel fixes the mapping, the instruction simply retries.
7x86-64 4-level paging HW▾

x86-64 uses a 48-bit canonical VA and a 4-level radix tree of tables: PML4 → PDPT → PD → PT (Linux names them PGD → PUD → PMD → PTE; P4D is folded at 4 levels). Each level is indexed by 9 bits (512 entries), and CR3 holds the physical base of the top table. (5-level "LA57" adds one more.)

Ready: 9+9+9+9+12 = 48 bits. The decoder on the "How It Works" tab splits any VA into exactly these fields.
8Your LtRAM tier as a NUMA node PROJECT▾

In your prototype the slow tier is ZONE_LTRAM on a separate NUMA node, removed from the general node mask so normal allocations avoid it. A page in the tier is just a frame on a different node — translation is identical. What differs is how pages get there (fault-time routing, migration) and what a write does (faults, then repatriates).

Ready: Lesson 1 gives you the translation + fault machinery; everything your kernel adds bolts onto it.
The whole lesson in one place

Address translation, and the trap behind it

What the hardware does on its own, where the kernel must step in, and the one diagram that ties it together.

Virtual memory buys three things at once from a single mechanism — indirection: each process gets a private, contiguous-looking address space (isolation), pages can live anywhere physical or nowhere yet (relocation & lazy allocation), and per-page permission bits give protection. The MMU realizes all three by walking page tables on every access.

The hardware does the common case alone: TLB hit, or TLB miss → walk the 4-level tree → a present PTE → done, fill the TLB. But the hardware cannot invent a mapping. When the PTE is not present, or the access violates its permission bits, the MMU does the only thing it can — raise #PF and hand control to the kernel. The kernel consults the VMA (the intended map) to decide: allocate a zero page, fault in a file page, copy a shared page, or — in your system — migrate a tier page back to DRAM. Then it returns, and the faulting instruction re-runs as if nothing happened.

That handoff — hardware walk → #PF → kernel fault handler → retry — is the spine of everything you built. CoW repatriation is a write that faults into do_wp_page. Fault-time placement is the kernel choosing a tier frame inside the handler. Lesson 1 is that spine.

🍯 Key takeaways

  • VM = one mechanism (paged indirection) delivering isolation + relocation + protection.
  • The MMU walks PML4→PDPT→PD→PT (PGD→PUD→PMD→PTE), 9 bits/level, 12-bit offset, CR3 at the root.
  • The TLB caches translations; the walk is the miss path; editing a PTE needs a TLB invalidation.
  • Hardware does present/permitted accesses; anything missing or disallowed traps to #PF.
  • The kernel uses the VMA as the source of truth for what a page should be, then fixes the PTE.
  • #PF is restartable — the instruction re-runs — which is what makes CoW & migration transparent.
  • Your tier hook is in the handler: place RO pages there at fault time; repatriate on a write fault.
Six framings

Mental models (and where they leak)

You don't need these to understand the mechanism — but they pin down the why. Each notes where the analogy stops being literally true.

🌳
Page table · radix trie

A four-deep address trie

The page table is a radix tree keyed by the bits of the VA: 9 bits pick a slot at each level, the leaf holds the frame. Sparse address spaces cost almost nothing because empty subtrees are simply absent (a null PGD entry prunes 512³ pages).

Leaks: unlike a software trie, the walk is done by hardware with fixed fan-out (512), and interior nodes can themselves be leaves (a PMD with _PAGE_PSE set is a 2 MiB huge page — the walk stops early).
📇
TLB · translation cache

A sticky-note of recent lookups

The TLB is just a cache, with the same hit/miss economics as any cache you know. Hit: instant. Miss: pay the multi-level walk. Locality is everything — hence huge pages to grow reach per entry.

Leaks: unlike a data cache, stale TLB entries aren't coherent with the page tables in memory. Edit a PTE and you must explicitly invalidate (INVLPG / a shootdown IPI) — there's no hardware snoop.
📜
VMA vs page table

Blueprint vs as-built

The VMA list is the blueprint — what the address space is supposed to be (this range is RO file-backed, that range is anon RW). The page table is the as-built — what's actually wired up right now. The fault handler reconciles as-built to blueprint, one page at a time.

Leaks: they can legitimately disagree (a not-yet-faulted VMA has no PTEs) — that's not a bug, it's lazy population. The blueprint is authoritative; the PTE is a cache of decisions.
🔔
#PF · restartable trap

A pause button, not a crash

A fault freezes the instruction mid-flight, runs kernel code, then resumes the same instruction. The CPU rewinds architectural state so the retry is invisible. This is what lets the kernel "lie" — present memory that wasn't there until you touched it.

Leaks: not every #PF is benign — a real protection violation with no VMA fix becomes SIGSEGV. The handler decides "fixable vs fatal" by consulting the VMA.
🏷️
Dirty/accessed bits

Hardware leaving fingerprints

You don't poll usage; the CPU stamps the PTE for you — sets accessed on any touch, dirty on a write. Software reads and clears them to age pages. Your scanner reads exactly these fingerprints.

Leaks: the stamps are one-bit and lossy (set-once until cleared), and clearing them costs a TLB flush to make the next access re-stamp. Detail lives in the next lesson (os.lmm.pte).
🏭
Your tier · a far warehouse

A second shelf, not a basement

A tier frame is reached by the identical translation path — it's just physically on a slower NUMA node. Reads are transparent. The interesting events are placement (kernel chooses a tier frame in the handler) and the write trap (write-protected ⇒ #PF ⇒ repatriate).

Leaks: "transparent reads" hides the latency: same path, slower frame. And unlike swap (which removes the PTE), your tier page stays present — only writes are gated, via the RW bit.
The mechanism, precisely

From a load to a frame (and a fault)

First principles, the exact x86-64 walk, the fault path in mm/memory.c, and where your kernel code hooks in. Grounded in Linux 6.8.

1 · Why the kernel, not just the MMU?

The MMU is a fast, dumb tree-walker: given CR3 and a VA, it fetches four entries and produces a frame, or faults. It has no policy. It can't decide that a not-present page should be a fresh zero page vs a file page vs a copy of a shared page; it can't allocate a frame; it can't know your tiering rules. Policy and population are the kernel's job. The hardware's contract is narrow and mechanical; everything interesting is the kernel filling in what the hardware found missing.

2 · The x86-64 4-level walk, exactly

A 48-bit canonical VA splits into five fields. Each 9-bit index selects one of 512 entries (each entry is 8 bytes, so a table is exactly one 4 KiB page):

  47        39 38        30 29        21 20        12 11         0
 +-----------+-----------+-----------+-----------+------------+
 |  PGD (9)  |  PUD (9)  |  PMD (9)  |  PTE (9)  | offset(12) |
 +-----------+-----------+-----------+-----------+------------+
   (PML4)       (PDPT)        (PD)        (PT)      byte in page

 CR3 ─▶ PML4[PGD] ─▶ PDPT[PUD] ─▶ PD[PMD] ─▶ PT[PTE] ─▶ frame
                                    │
                                    └─ if _PAGE_PSE: PMD is a 2 MiB leaf (stop here, 21-bit offset)

Worst case, a TLB miss costs four dependent memory loads before the access itself. Hardware paging-structure caches (page-walk caches) cache the upper levels, so most walks touch far fewer than four. Use the decoder below on a real address.

3 · The TLB and the cost of a miss

Concrete order-of-magnitude: an L1 dTLB of ~64 entries × 4 KiB ≈ 256 KiB of reach; blow past that with a striding scan and you eat a walk per ~page. A 2 MiB hugepage entry covers 512× the range per slot, which is the whole point of THP for big working sets. A TLB hit is ~1 cycle; a full walk is tens of cycles (cache-resident tables) to hundreds (cold). Your scanner walking many anon PTEs is, in effect, a software page-table walk — and clearing an accessed/dirty bit forces a TLB invalidation so the CPU re-stamps on the next touch.

4 · The fault path in the kernel

When the PTE is absent or the access violates it, the CPU raises #PF (vector 14), stashing the faulting VA in CR2 and an error code (present / write / user / instruction-fetch). The kernel entry and the generic handler chain (Linux 6.8, arch/x86/mm/fault.c then mm/memory.c):

// #PF (vector 14) → arch entry
exc_page_fault() // arch/x86/mm/fault.c; reads CR2 + error code
  └─ do_user_addr_fault()
       ├─ find_vma(mm, addr)        // the "blueprint": is this address mapped at all?
       │     └─ no VMA / perm mismatch ⇒ SIGSEGV
       └─ handle_mm_fault(vma, addr, flags)   // mm/memory.c — the policy engine
            └─ __handle_mm_fault() // descends PGD→PUD→PMD, allocating tables as needed
                 └─ handle_pte_fault()
                      ├─ !present, anon  ⇒ do_anonymous_page()  // fresh zero page
                      ├─ !present, file  ⇒ do_fault() / do_swap_page()
                      └─ present, write, !writable ⇒ do_wp_page()  // ← CoW / your repatriation
// handler returns → CPU re-executes the faulting instruction

Two things to burn in: the kernel allocates missing intermediate tables during the descent (a fresh mapping may need a new PMD page), and the decision of what to do is driven by the VMA plus the error code, not by the PTE alone.

5 · Where your LtRAM kernel hooks in PROJECT

🔧
Placement rides the not-present branch: in/after handle_pte_fault, for a read-only VMA (vm_flags without VM_WRITE, e.g. code/libs/RO file maps), your code steers the new frame to ZONE_LTRAM on the tier node instead of a normal DRAM node.

Repatriation rides the present-but-write-protected branch: every tier page is write-protected, so a store lands in do_wp_page. Your handler recognizes a tier page and migrates it back to DRAM before the write applies — reusing the same CoW machinery fork()/KSM use. The instruction then retries and writes the DRAM copy.

Both are just policy inserted at the handoff this lesson traced. Lessons os.lmm.faultpath, os.lmm.cow, and os.lmm.migration open each up.

Follow the path, step by step

Worked traces

Four end-to-end traces — a hit, a cold miss, a fresh fault, and the one that is your repatriation.

① A load that hits the TLB HW only

  1. Instruction issues a load of VA V. The page number (V >> 12) is presented to the TLB.
  2. TLB hit: the cached entry yields frame F and the permission bits; the access is allowed.
  3. Physical address = F·4096 + (V & 0xFFF). The walk never happens. ~1 cycle of translation overhead.
💡
99%+ of accesses look like this. Everything below is the miss machinery the fast path hides.

② A TLB miss → full walk → present PTE HW

  1. TLB miss. Hardware reads CR3 to find the PML4 base.
  2. Descend: PML4[V.PGD] → PDPT[V.PUD] → PD[V.PMD] → PT[V.PTE] — up to four dependent loads (fewer if page-walk caches hit).
  3. Leaf PTE is present and permits the access. Hardware sets accessed (and dirty if it's a write), fills the TLB, completes the load.
📐
If the PMD had _PAGE_PSE, the walk stops one level early at a 2 MiB frame with a 21-bit offset — one fewer load, 512× the TLB reach.

③ A load to an unmapped anon page → #PF FAULT

  1. PTE not present. The MMU raises #PF; CR2 = V, error code says not-present, read, user.
  2. exc_page_fault → do_user_addr_fault: find_vma(mm, V) returns an anonymous RW VMA covering V — fixable.
  3. handle_mm_fault → __handle_mm_fault descends, allocating any missing PMD/PT pages, reaches handle_pte_fault.
  4. do_anonymous_page allocates a zeroed frame, builds the PTE (present, RW, user), installs it.
  5. Handler returns; the CPU re-executes the same load. This time it walks to a present PTE and succeeds.
🍯
This is lazy allocation in the flesh: the page didn't exist until you read it. The instruction never "knows" it faulted.

④ A store to a write-protected tier page → repatriation YOUR PATH

  1. The page is present but write-protected (it lives in ZONE_LTRAM; your code cleared _PAGE_RW). A store executes.
  2. #PF fires: error code = present, write, user — the tell-tale of a write to a non-writable present PTE.
  3. handle_pte_fault routes to do_wp_page (the CoW handler). Normally CoW copies a shared page; your hook instead recognizes a tier page.
  4. Migrate to DRAM: isolate the page, copy contents to a fresh DRAM frame on a normal node, re-point the PTE (now present + RW on DRAM), invalidate stale TLB entries.
  5. Return; the store re-runs and lands on the DRAM copy. The application saw a slightly slower store and nothing else.
🔧
This is your whole "read-only to applications, repatriate on write" invariant, expressed in the fault machinery of Lesson 1. The only custom part is step 3–4; the trap, the VMA lookup, the retry are all stock kernel.
Probe your understanding

Practice (diagnose / design)

Try before revealing. These probe the level the course targets for this topic: design.

Calculate
For VA 0x00007f3c12345678 (4 KiB pages), give the PGD/PUD/PMD/PTE indices and the offset. (Check yourself with the decoder.)
Show answer
Split the low 48 bits into 9/9/9/9/12. Offset = 0x678 = 1656. Indices (each (VA >> shift) & 0x1FF): PTE@12, PMD@21, PUD@30, PGD@39. Working it through: PGD=254, PUD=240, PMD=145, PTE=325, offset=1656. The point isn't the arithmetic — it's that each 9-bit slice indexes exactly one table, and the bottom 12 bits never get translated.Misconception: that the offset is "part of the frame number." The offset bypasses translation entirely; only the page number (bits 47:12) is translated.
Diagnose
A process has a valid RW anonymous VMA over some range, but reading an address in it faults. Is this a bug? Walk the path.
Show answer
Not a bug — it's expected lazy population. The VMA (blueprint) exists, but no PTE has been installed yet (as-built is empty there). The read faults not-present; find_vma finds the RW anon VMA, handle_mm_fault → do_anonymous_page installs a zero page, and the read retries and succeeds. A VMA's existence ≠ a PTE's existence.Misconception: that "mapped" means "has a page table entry." Mapped means "covered by a VMA"; PTEs are filled on demand.
Diagnose
Exactly what does the hardware do on a store to a present, read-only PTE, and where does software take over? Which error-code bits identify this case?
Show answer
The MMU completes the walk, finds the PTE present but _PAGE_RW clear for a write, and raises #PF — it does not modify memory. Error code bits: present=1, write=1 (user=1 if from user mode). The CPU writes CR2 and traps; software takes over at exc_page_fault → … → handle_pte_fault, which sees present+write+!writable and dispatches do_wp_page. In your system that's the repatriation trigger.Misconception: that the hardware "tries and fails" the store. It detects the permission violation before any write and faults atomically — nothing is half-written.
Design
You want fresh read-only mappings (code, RO file maps) to land in the tier at fault time. Where in the fault path do you make that decision, and what input drives it?
Show answer
Inside the not-present branch of handle_pte_fault (e.g., in/around do_fault for file pages or where the frame is allocated), gated on the VMA's vm_flags: lacking VM_WRITE (and matching your policy for code/RO maps) ⇒ request the frame from ZONE_LTRAM on the tier node rather than a normal node. The decision needs the VMA (intent) — the PTE alone can't tell you the region is meant to be read-only. You also must keep ZONE_LTRAM out of the general fallback so ordinary allocations never drift in.Misconception: deciding placement from the PTE/page. Placement intent comes from the VMA; the page is just the frame you fill.
Design
In do_wp_page you re-point a tier page's PTE to a new DRAM frame on CPU 0. Another thread on CPU 1 had the old translation cached. What must you do, and what breaks if you skip it?
Show answer
You must invalidate the stale TLB entry on every CPU that could hold it — a TLB shootdown (flush_tlb_*, an IPI to CPU 1) — and order it correctly with the PTE update (update PTE, then flush, with the right barriers/ptep helpers). Skip it and CPU 1 keeps translating the VA to the old tier frame: it reads stale data, or writes to the page you're abandoning — a lost-update / use-after-migrate bug. The TLB is not coherent with page-table memory; only explicit invalidation fixes it. (Depth: os.mem.tlb and os.lmm.ptep.)Misconception: that updating the PTE in memory is enough. Cached translations survive until explicitly flushed.
Quick answers

FAQ

Precise answers to the questions this lesson tends to raise.

Q1Why four levels and not one big flat table?▾
A flat table for a 48-bit space at 4 KiB pages would need 2³⁶ entries (~512 GiB) per process — absurd, and almost all empty. The multi-level radix tree makes the table proportional to what's actually mapped: an unused PGD slot prunes a 512 GiB subtree at the cost of one null entry. You pay walk depth to save enormous space.
Q2Who sets the accessed and dirty bits — hardware or the kernel?▾
Hardware sets them: accessed on any access, dirty on a write, as a side effect of the walk. Software reads and clears them (pte_mkold, pte_mkclean) to age pages, and clearing requires a TLB flush so the next access re-stamps. Your placement scanner is built entirely on reading/clearing these. Full treatment in os.lmm.pte.
Q3What's in CR3 and CR2?▾
CR3 holds the physical base of the top-level table (PML4) for the current address space — switching it (on context switch) switches the whole mapping. CR2 holds the faulting virtual address the hardware captured for the most recent #PF, which the handler reads first.
Q4How is a tier page different from a swapped-out page?▾
A swapped page is not present — its PTE is a swap entry, and any access (read or write) faults to bring it back. Your tier page stays present on a slower node, so reads are direct (no fault); only writes are gated (you clear _PAGE_RW), faulting into do_wp_page to repatriate. Same fault machinery, different "missing" condition.
Q5Does every TLB miss cost four memory accesses?▾
Worst case yes, but rarely in practice. Paging-structure caches (page-walk caches) hold recently-used upper-level entries, so most walks resolve in one or two loads, and the tables themselves are often in the data caches. Huge pages also cut a level. The "four loads" is the ceiling, not the norm.
Q6What makes an address "canonical," and why do kernel addresses look like 0xffff…?▾
With 48-bit VAs, bits 63:48 must be a sign-extension of bit 47 — otherwise the CPU faults (non-canonical). So the space splits into a lower half (0x0000…, user) and an upper half (0xffff…, kernel). The decoder still uses only bits 47:0 for the indices; the high bits are just the canonical sign extension.
Q7Where do the intermediate page-table pages come from?▾
The kernel allocates them on demand during the fault descent (__handle_mm_fault / the p?d_alloc helpers): the first fault into a fresh region may allocate a new PMD and PT page before installing the leaf PTE. They're ordinary kernel pages, freed when the mapping is torn down.
Q8Is this x86-specific? My design should be portable.▾
The concepts (multi-level translation, TLB, present/RW/dirty, restartable faults) are universal; the specifics (4 levels, 9-bit indices, CR3, the exact bit positions) are x86-64. Linux abstracts them behind generic names (PGD/PUD/PMD/PTE, pte_dirty(), etc.) so your mm code is mostly arch-neutral — arch headers (arch/x86/include/asm/pgtable*.h) supply the real bits.
