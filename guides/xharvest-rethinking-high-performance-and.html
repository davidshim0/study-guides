Cutting SSD costs 31% by harvesting the idle host, not hoarding internal chips

SSDs bundle expensive ARM cores and DRAM to handle I/O bursts — yet sit idle 96% of the time. This is XHarvest, which moves firmware to a secure host enclave and uses CXL to grab host memory only when needed.

Before you begin

Ten ideas XHarvest builds on

Each primer clarifies a concept the paper assumes. Tap any to open; read in order first time, or jump to what's unfamiliar.

CXL (interconnect) TEE (security) dynamic harvesting static provisioning (baseline)
1Why SSDs have internal ARM + DRAM▾

A high-performance SSD bundles an ARM processor (e.g., 4.43× the power of PCIe 3.0 controllers for PCIe 5.0's 14 GB/s) and DRAM (e.g., 10 GB for a 7.5 TB drive) to run firmware — the flash translation layer (FTL) that maps logical to physical addresses, garbage collection, wear-leveling. These resources cost roughly 30% (compute) + 10% (memory) of a 1 TB SSD.

But Alibaba traces show I/O bandwidth below 25% for 96.64% of runtime, and above 80% for only 0.28%. The vast, expensive internal resources sit idle most of the time, plunging SSD design into a cost-utilization dilemma.

Now you're ready: XHarvest reserves only 25% of the internal compute power and 10% of the memory, then dynamically borrows host resources for bursts.
2Open-Channel SSD (OCSSD) — the prior attempt▾

OCSSD removed internal compute/memory entirely and moved firmware (pblk) to the Linux kernel, using host CPU and DRAM. It aimed for cost-efficiency but statically occupies host memory for the full mapping table and competes with applications for CPU, causing severe contention. Applications in Alibaba use >95% memory at low I/O, leaving little room for OCSSD's demands. The paper notes that this approach faces adoption obstacles due to firmware leakage risks and vendor resistance to exposing proprietary algorithms in the OS kernel. Linux deprecated it (kernel 5.15).

Now you're ready: XHarvest avoids static host resource occupation by harvesting dynamically, releasing memory when I/O is low.
3The flash translation layer (FTL) and metadata▾

Flash requires erase-before-write, so SSDs use out-of-place updates. The FTL maintains a mapping table (LPN → PPN at 4–16 KB granularity) in DRAM to hide this. On write, it allocates a new PPN and invalidates the old. Metadata for the entire device is stored on special flash pages (translation pages); the firmware buffers hot entries in DRAM for speed. Conventional SSDs fit the entire table in internal DRAM to avoid slow flash reads (e.g., 25 µs).

Now you're ready: XHarvest builds a unified FTL cache spanning host memory and modest internal DRAM, accessed via CXL for efficiency.
4Compute express link (CXL) — cache-coherent memory semantics▾

CXL is a new interconnect (atop PCIe PHY) that unifies the host CPU and peripheral devices (e.g., SSDs) in a cache-coherent domain. CXL.mem lets the host access device memory via load/store at cacheline granularity (instead of coarse DMA). CXL.cache lets devices cache host memory. Both use cache-coherent protocols, so no manual flushing or OS stack overhead. Final CXL latency target: 75 ns cross-NUMA (the paper's emulation setting).

Now you're ready: XHarvest exploits CXL.mem to let the host CPU (in enclave) directly read FTL entries from SSD internal DRAM, bypassing OS (10 µs) for a 200 ns memory shot.
5Trusted execution environment (TEE) — Intel SGX▾

TEE (e.g., Intel SGX, AMD SEV, ARM TrustZone) provides a secure container called an enclave where sensitive code/data run isolated from the OS or other processes. SGX encrypts the enclave's memory region (Enclave Page Cache, EPC) and decrypts on CPU cache loads, preventing unauthorized reads. Attestation proves the enclave runs on a genuine SGX CPU with the expected initial state. Ecall/ocall transitions between enclave and untrusted code incur ~20K cycles each.

Now you're ready: XHarvest runs the SSD firmware (proprietary FTL algorithms) in an enclave to protect secrets while harvesting the host CPU, avoiding firmware leakage.
6CXL 3.1 TEE Security Protocol (TSP)▾

CXL 3.1 introduces TSP (component measurement and authentication, CMA), which lets a TEE and a CXL device mutually authenticate and then encrypt their CXL traffic (e.g., AES-GCM). This prevents a compromised OS from reading the CXL data flowing between the enclave and the SSD. The AES-GCM overhead is ~5% latency, negligible against flash ops (25 µs read, 50 µs write).

Now you're ready: XHarvest uses TSP to secure the enclave–SSD collaboration: only the authenticated enclave accesses internal DRAM; traffic is encrypted. A compromised OS sees ciphertext.
7I/O burst interleaving in production▾

Alibaba data (4,000 servers, 8 days) show I/O bursts don't happen on all SSDs simultaneously. CDF of active VMs (I/O >50%) shows more than 5 VMs active concurrently only 2.20% of runtime. Storage allocation: multiple VMs share an SSD. So while individual SSDs may burst, few do so at the same time, allowing dynamic sharing of host memory instead of static reservation for all drives.

Now you're ready: XHarvest allocates host memory on-demand (harvesting) when a drive detects a burst, then releases it as load drops, avoiding simultaneous high memory use.
8CPU utilization at varying I/O loads▾

The paper's Figure 3d breakdown: when I/O BW is 25–40%, CPU is Low (0–40%) for 26.06%, Mid (40–60%) for 46.61%, High (60–100%) for 27.33%. As I/O grows to 85–100%, Low jumps to 45.69% (due to CPU waiting on I/O). This indicates spare host CPU during high I/O loads, safe to harvest for firmware without starving applications.

Now you're ready: XHarvest uses a single host core (in enclave) to serve bursts; the enclave pins to one core, leaving others for apps, avoiding heavy CPU contention.
9DRAMless SSD — a partial step▾

DRAMless SSDs omit internal DRAM and buffer the FTL cache in host memory (e.g., 128 MB for 1 TB). But they retain full internal compute, must transfer pages via DMA (4 KB granularity, misaligned with 8 B entries), and encrypt metadata in host memory to avoid leakage. The paper's evaluation shows DRAMless degrades by 4.02% vs. ConvSSD in 50th latency and by 83.28% when host memory is scarce.

Now you're ready: XHarvest improves on DRAMless by also reducing internal compute (to 25%) and using CXL's fine-grained, cache-coherent access (no DMA, no unnecessary entry transfer).
10The naive OCSSD-like solution's three challenges▾

Retain moderate internal DRAM/CPU to alleviate host contention: (1) Trade-off: more internal DRAM → less contention but higher cost; less → cheaper but more contention. (2) Hampered collaboration: PCIe separates cache-coherency domains; host can't directly access internal DRAM without OS stack (10 µs penalty). (3) Firmware leakage: unprotected host-based firmware exposes proprietary algorithms, preventing vendor adoption. XHarvest tackles all three with CXL (unified memory), dynamic harvesting (on-demand), and TEE (secure execution).

Now you're ready: XHarvest combines modest internal resources with CXL-enabled dynamic harvesting and SGX-based secure firmware execution to solve the dilemma.
The whole story in one place

XHarvest in a nutshell

The problem, the dynamic harvesting solution, and how it all fits together.

SSDs for peak I/O (e.g., PCIe 5.0, 14 GB/s) embed expensive ARM processors (4.43× more compute vs. PCIe 3.0) and large DRAM (10 GB for 7.5 TB drive) to run firmware (FTL, GC, wear-leveling) at top speed. Yet production clusters (Alibaba, Google, Facebook) show I/O bandwidth utilization below 25% for 96.64% of runtime, and above 80% for only 0.28%. These costly internal resources sit idle most of the time — a cost-utilization dilemma.

Open-Channel SSD (OCSSD) tried removing internal resources entirely, running firmware (pblk) in the Linux kernel on host CPU/DRAM. But it statically occupies host memory (for the entire FTL mapping table) and CPU, competing with memory-intensive applications (Alibaba: >95% memory use at low I/O). This contention, plus adoption obstacles from firmware leakage risks and vendor resistance to putting proprietary firmware in the OS kernel, led to Linux deprecating it in kernel 5.15.

XHarvest shifts from static to dynamic resource allocation. It reserves only 25% internal compute and 10% memory (enough for regular I/O) and removes the rest. When I/O bursts (>60% bandwidth over 5 ms window), a load detector signals a daemon thread to launch a secure enclave that harvests a single host CPU core and allocates EPC (encrypted host memory) for a larger FTL cache. The enclave executes the firmware (protecting proprietary algorithms), polls requests via a CXL-driven communication mechanism, and accesses the unified metadata cache spanning host EPC and SSD internal DRAM via cache-coherent CXL.mem (no OS stack, fine-grained). As load drops, the enclave flushes the cache and releases resources.

The payoff: 31.50% cost savings for a 1 TB SSD (vs. ConvSSD), 5.02% higher throughput on real traces, and 2.27× execution time reduction over OCSSD in memory-intensive tasks (Terasort with 64 GB memory). XHarvest's dynamic harvesting exploits burst interleaving (only a few SSDs peak simultaneously) and idle host CPU (45.69% Low utilization at 85–100% I/O), while TEE + CXL TSP secures the collaboration. Overhead: negligible on-path latency (secure CXL traffic: ~5%).

🌾 Key takeaways

  • Conventional SSDs provision costly internal compute (30% of cost) and DRAM (10%) for peak I/O but sit idle 96.64% of the time.
  • OCSSD removed internal resources but statically occupied host CPU/memory, causing severe contention (>95% memory use by apps).
  • XHarvest reserves modest internal resources (25% compute, 10% memory) and dynamically harvests host CPU and DRAM only during bursts.
  • Burst interleaving in production: few SSDs peak simultaneously (>5 active VMs only 2.20% of runtime), enabling on-demand sharing.
  • TEE (SGX enclave) protects proprietary firmware on the host; CXL 3.1 TSP secures enclave–SSD traffic (AES-GCM, ~5% overhead).
  • CXL.mem unifies host and SSD memory spaces with cache-coherent, fine-grained access (200 ns vs. 10 µs OS stack).
  • Result: 31.50% cost cut, 5.02% throughput gain (vs. ConvSSD), 2.27× faster than OCSSD in memory-intensive workloads.
Seven everyday pictures

How to think about XHarvest

Each analogy stands alone — read in any order. Every card ends with where the picture breaks down.

The core idea · dynamic vs. static

A personal generator vs. the grid

You could buy a huge backup generator to handle peak power demand (static provisioning), but it sits idle 96% of the time. Instead, install a small generator for normal loads and tap the city grid during rare blackouts (dynamic harvesting). XHarvest keeps modest internal compute (25%) and DRAM (10%) for regular I/O, then borrows host CPU and memory only during bursts (which last <2.20% of runtime for >5 concurrent VMs).

Where it breaks: the grid is always-on; host resources must be free (low CPU/memory use). XHarvest detects this (45.69% Low CPU at high I/O), but if host is fully loaded, harvesting fails. The paper shows this is rare in typical datacenters.
CXL's role · unified memory

A shared library catalog

Old system: two separate buildings (host, SSD), each with its own card catalog (DRAM). To check a book in the other building, you call (OS stack) and wait 10 µs. CXL.mem merges both catalogs into one searchable database — the host CPU walks into the SSD building and checks the shelf directly (200 ns). XHarvest's enclave reads FTL entries from internal DRAM via CXL.mem, no OS involvement, fine-grained (8 B entry vs. DMA's 4 KB page).

Where it breaks: a library catalog is static; the FTL cache is dynamic (LRU eviction). Also, CXL.mem adds 75 ns latency (cross-NUMA), but this is negligible vs. flash ops (25 µs read, 50 µs write).
Security · TEE (SGX enclave)

A lockbox inside the vault

The host is a bank vault (untrusted OS, malicious apps). You bring a lockbox (enclave) with your valuables (firmware, FTL data). The vault's security cameras (OS) can't see inside the lockbox — it's encrypted (EPC). Only you (the enclave) have the key (CPU decrypts on LLC load). A bank inspector (attestation service) verifies the lockbox is genuine SGX, not a fake. XHarvest stores proprietary FTL algorithms in the enclave, invisible to the OS or other processes.

Where it breaks: a lockbox is opened once; the enclave continuously decrypts EPC on CPU cache loads (~5.09% overhead in 64K random write when using EPC vs. regular memory). Also, ecall/ocall (~20K cycles) is expensive; XHarvest replaces them with CXL-driven communication.
Secure traffic · CXL TSP

Encrypted postcards

The enclave and SSD exchange messages (requests, FTL entries) via CXL. But the OS (compromised postmaster) can snoop. CXL TSP encrypts each postcard (CXL flit) with AES-GCM: mutual authentication first (enclave ↔ SSD verify certs), then symmetric key exchange. The OS sees only ciphertext. MAC codes prevent tampering. Overhead: ~5% latency (1 CPU cycle per byte encryption), negligible vs. flash ops. XHarvest uses TSP to protect all enclave–SSD collaboration.

Where it breaks: postcards are one-way; CXL traffic is bidirectional (requests, responses, FTL entries). Also, TSP secures the wire but not the endpoints — the enclave and SSD must already be trusted (attestation ensures this).
Memory efficiency · fine-grained CXL vs. DMA

Delivering a letter vs. a truck

DMA (baseline, DRAMless SSD) transfers data in 4 KB chunks (truck) even if you need one 8 B mapping entry (letter). This wastes bandwidth and pollutes the SRAM cache with unnecessary entries. CXL.mem delivers at cacheline granularity (64 B), aligned with entry size (8 B) — just the letter. XHarvest cuts metadata traffic by 33.97% over DRAMless SSD (Figure 20a) and reduces FTL latency by 55.95% (25% hit ratio) to 79.27% (100% hit ratio).

Where it breaks: a letter is discrete; CXL transfers cachelines (64 B), which may still carry multiple entries. But 64 B ≪ 4 KB, so the waste is minimal. Also, CXL adds 75 ns vs. DMA's overhead (10 µs OS stack), a net win.
Load detection · dynamic launch

A sprinkler system with a rain sensor

You water your garden (regular I/O) with a small pump (25% internal compute). When rain comes (I/O burst >60% over 5 ms window), a sensor (load detector) turns on the hose (harvests host CPU + EPC). When rain stops, the sensor turns it off (enclave flushes cache, releases resources). XHarvest's detector monitors I/O in 5 ms windows; if load exceeds 60%, it signals a daemon thread to launch the enclave. Enclave startup is decoupled from the critical path (pre-created at boot, re-entered via ecall) to avoid latency spikes.

Where it breaks: a rain sensor is instant; enclave launch (attestation, firmware decryption) takes time. XHarvest pre-initializes the enclave at boot and only allocates EPC on burst, but the first 5 ms after burst sees a transient latency increase (no impact on tail latency over 60 s, Figure 23).
Coordination · resource partitioning

Two chefs in a kitchen

A slow cook (SSD ARM, 1× power) and a fast chef (host CPU core, 6× power) share a kitchen (flash channels, memory). If both grab ingredients randomly, they collide. XHarvest assigns resources proportional to compute power (approximately 1:6 ratio, §5.4). I/O loads are divided by logical address ranges (2 MB units, round-robin) to match these resource assignments. This balances resource usage and helps avoid contention while exploiting parallelism.

Where it breaks: chefs don't share data; the enclave and SSD must coordinate metadata updates (via CXL-driven communication). Also, read requests are directed by the physical page number (PPN) rather than strictly following partitioning, which can introduce potential contention over memory and flash channels.
The full explanation

How it actually works

Step by step, with the paper's key mechanisms explained at a glance.

1 · The problem — static provisioning wastes money

High-performance SSDs (e.g., PCIe 5.0, 14 GB/s) embed powerful ARM processors (4.43× more compute than PCIe 3.0) and large DRAM (10 GB for 7.5 TB) to run firmware (FTL, GC, wear-leveling) at top speed. These resources cost ~30% (compute) + 10% (memory) of a 1 TB SSD. But Alibaba traces (4,000 servers, 8 days) show I/O bandwidth utilization below 25% for 96.64% of runtime, and above 80% for only 0.28%. The expensive internal resources sit idle most of the time — a cost-utilization dilemma.

💡
Why not OCSSD? Open-Channel SSD removed internal resources entirely and ran firmware (pblk) in the Linux kernel, using host CPU/DRAM. But it statically occupied host memory (full FTL mapping table) and CPU, competing with memory-intensive applications (Alibaba: >95% memory use at low I/O). This contention, plus adoption obstacles from firmware leakage risks and vendor resistance to putting proprietary firmware in the OS kernel, led to Linux deprecating it (kernel 5.15).

2 · The shift — dynamic harvesting

XHarvest shifts from static to dynamic resource allocation. It reserves only 25% internal compute (modest ARM) and 10% memory (enough for a small FTL cache) to serve regular I/O (the 96.64% low-load time). For the rare I/O bursts (>60% bandwidth over 5 ms window), it harvests a single host CPU core and allocates EPC (encrypted host memory, e.g., 128 MB) for a larger FTL cache. When load drops, it releases these resources. This exploits two observations:

ObservationEvidence (Alibaba)XHarvest's Exploit
Burst interleavingCDF: >5 active VMs (I/O >50%) occur only 2.20% of runtimeDynamic memory allocation on-demand (not all SSDs burst simultaneously)
Idle CPU at high I/OCPU Low (0–40%) = 45.69% at 85–100% I/O BWHarvest 1 host core for enclave during bursts without starving apps
🌾
The key insight: I/O bursts are occasional and sparse (few SSDs peak at once). XHarvest allocates host resources on-demand and releases them, avoiding simultaneous high memory use and reducing contention by 2.27× execution time over OCSSD (Terasort, 64 GB memory).

3 · Secure execution — running firmware in an enclave

To protect proprietary FTL algorithms, XHarvest runs the firmware in an SGX enclave. The SSD stores the encrypted firmware binary. When the load detector (§5.4) signals a burst, a daemon thread on the host creates the enclave, which:

  1. Attestation: The SSD requests attestation (e.g., Intel Attestation Service) to verify the enclave runs on a genuine SGX CPU with the expected initial state. Communication is encrypted to prevent daemon thread attacks.
  2. Firmware loading: After successful attestation, the enclave acquires the encrypted firmware binary and decryption key from the SSD, decrypts it, and loads it into the EPC (encrypted host memory).
  3. I/O processing: The enclave pins to a single host CPU core and continuously polls I/O request queues within the SSD (via CXL-driven communication, §5.2). It translates LPNs to PPNs using the unified FTL cache (§5.3) and dispatches PPNs to the flash backbone. The enclave handles ~6× more I/O than the SSD firmware due to the powerful host CPU.
⚠
Challenge: Enclave cannot directly access peripheral devices (managed by OS). Traditional ecall/ocall (~20K cycles each) to interact with the SSD would degrade throughput by 96.13% (4K read, Figure 6). XHarvest solves this with CXL-driven communication (next section).

4 · CXL-driven communication — bypassing OS

CXL.mem allows the host CPU to map SSD internal DRAM as cacheable memory (HDM), accessible via load/store instructions at cacheline granularity. But this exposes internal DRAM to untrusted host software. XHarvest secures it with CXL 3.1 TSP (TEE Security Protocol):

# Step 1: Mutual authentication (TSP CMA)
enclave ↔ SSD: certificate exchange, verify identity
# Step 2: Symmetric key exchange
enclave ↔ SSD: exchange AES-GCM key for encrypting CXL flits
# Step 3: Encrypted traffic
for each CXL flit (enclave ↔ SSD):
    encrypt with AES-GCM (1 CPU cycle/byte, ~5% overhead)
    generate MAC code for integrity check

With secure CXL traffic, XHarvest builds a message-passing communication mechanism. A portion of SSD internal DRAM is a ring buffer (64 B messages). The enclave and SSD maintain tail/head pointers. To send a message, the enclave writes to the tail position and sets a ready bit. The SSD polls the ready bit at the head position (via CXL.mem), retrieves the message, and vice versa for reverse communication. This replaces ecall/ocall, delivering 3700K OPS (single SSD) and scaling to 3.85× with four SSDs (Figure 24).

💡
Why TSP matters: Without encryption, a compromised OS could read enclave–SSD traffic (requests, FTL entries) via CXL.mem snooping, leaking proprietary algorithms. TSP encrypts the wire; attestation ensures the endpoints (enclave, SSD) are trusted.

5 · Unified FTL cache — CXL.mem spans host and SSD

XHarvest builds a unified metadata cache spanning host EPC (e.g., 128 MB) and SSD internal DRAM (e.g., 10% of full table). Both the enclave and SSD firmware can access this cache, but they use different methods:

WhoWhereHowSecurity
EnclaveEPC (host)Load/store → CPU decrypts on LLC loadSGX encrypts EPC (opaque to OS)
EnclaveSSD DRAMCXL.mem → secure CXL traffic (TSP)AES-GCM encryption (~5% overhead)
SSD firmwareInternal DRAMDirect DRAM access (no encryption needed)Internal to SSD (trusted)
SSD firmwareHost memoryCXL.cache → self-encrypt entriesFirmware encrypts individual entries + CRC/ECC

The cache stores translation pages (each page = multiple mapping entries for adjacent LPNs). The enclave maintains a hash table index to track buffered pages. On lookup, it searches the hash table; if found, it accesses the entry (in EPC or SSD DRAM) via load/store (CXL.mem for SSD DRAM). If not found, it evicts a victim page (LRU policy) and reads the target page from flash. This unified cache improves hit ratio and reduces flash reads by 55.95% (25% hit) to 79.27% (100% hit) over DRAMless SSD (Figure 20).

🌾
CXL's advantage: Fine-grained (8 B entry vs. DMA's 4 KB page), cache-coherent (no manual flush), and direct access (200 ns vs. 10 µs OS stack). XHarvest cuts metadata traffic by 33.97% and FTL latency by 55.95% over DRAMless SSD.

6 · Host-SSD coordination — resource partitioning

The enclave (host CPU) and SSD firmware can process I/O concurrently, but they share flash channels and memory. To avoid contention, XHarvest partitions resources proportional to compute power, with resources assigned approximately in a 1:6 proportion (§5.4):

# Flash channel allocation (example: 8 total channels, Table 1)
Assign channels to firmware and enclave approximately 1:6 by compute power
Write requests: respect partitioning (avoid conflicts)
Read requests: directed by PPN (can introduce contention)

# Memory allocation
SSD internal DRAM: partition ~1:6 (firmware : enclave)
host memory (EPC): allocated only by enclave (harvested on burst)

# Load distribution (by logical address space)
divide LPN space into 2 MB units (1 translation page)
round-robin assign units ~1:6 (firmware : enclave)
each side handles I/O requests from its assigned address ranges

This balances resource usage and helps exploit parallelism. Figure 22 shows computation resource usage for the SSD controller and harvested CPU core during operation.

💡
Why 2 MB units? Aligns with translation page size (covers 2 MB of LPN space at 8 B entry/4 KB page). Fine-grained division (2 MB ≪ TB capacity) distributes I/O load proportionally to resources, avoiding hotspots.

7 · Dynamic enclave launch — on-demand harvesting

A load detector in the SSD monitors I/O bandwidth in 5 ms windows. If load exceeds 60%, it activates a flag variable in host memory. A daemon thread polls this flag every 1 ms (0.1% CPU). On activation, the thread re-enters the pre-created enclave (via ecall) and allocates EPC for the FTL cache. When load drops, the enclave flushes the cache (write dirty pages to flash), releases EPC, and exits (via ocall). Enclave startup is decoupled from the critical path (pre-initialized at boot) to avoid latency spikes. Figure 23 shows transient real-time latency increase limited to 5 ms, with negligible impact on tail latency over 60 s.

🌱
Why pre-create the enclave? Attestation + firmware decryption takes time. Pre-creation at boot hides this latency; re-entering via ecall (~20K cycles) is fast compared to the 5 ms detection window.

Four traces using the paper's real data — computing cost savings, latency reduction, and memory contention relief.

① Computing the cost savings (1 TB SSD)

The paper says XHarvest saves 31.50% cost for a 1 TB SSD. Break down the calculation using market prices from the paper's sources.

  1. ConvSSD cost (baseline): NAND flash (128 GB × 8 = 1 TB) = 8 × $5.248 = $41.98. Controller (full ARM + DMA) = $23. Memory (1 GB DRAM for FTL) = $7.5. Other (packaging, etc.) = $4. Total = $76.48.
  2. XHarvest cost: NAND flash (same) = $41.98. Controller (25% compute) ≈ $23 × 0.25 = $5.75. Memory (10% DRAM for base FTL) = $7.5 × 0.10 = $0.75. Other = $4. Total = $52.48.
  3. Savings: ($76.48 − $52.48) / $76.48 = $24.00 / $76.48 = 31.38% (paper rounds to 31.50%).
  4. Host memory cost? XHarvest uses 128 MB host EPC during bursts (0.28% of runtime). Not counted in SSD cost (shared across drives). But if amortized: 128 MB × $7.5/GB = $0.96, negligible vs. savings.
🌾
Insight: Compute (30% → 7.5%) and memory (10% → 1%) reductions dominate the savings. NAND flash (54.9% of cost) is fixed. For larger drives (4 TB, 16 TB), flash % rises, so savings drop to 19.83% and 15.49% (Figure 17).

② Calculating the CXL latency benefit

The paper says CXL.mem reduces enclave–SSD collaboration latency from 10 µs (OS stack) to 200 ns (memory shot). Verify this with the paper's Figure 20 data (FTL latency breakdown).

  1. Baseline (DRAMless SSD): 25% FTL cache hit ratio. FTL latency (en/decrypt + DMA transfer) = 0.0 normalized units (baseline). Data latency (flash ops) = 0.7. Firmware (FTL processing) = 0.3. Total = 1.0.
  2. XH-Host (enclave with CXL.mem): FTL latency = −0.60 vs. baseline (55.95% reduction). Data = 0.7 (same). Firmware = 0.3 − 0.07 (powerful CPU) = 0.23. Total = 0.33.
  3. Latency reduction: FTL part: 0.6 / 1.0 = 60% of total latency saved. If FTL was 10 µs (OS stack + DMA), now 10 µs × (1 − 0.60) = 4 µs. CXL.mem access (cacheline) ≈ 200 ns (75 ns CXL-induced + 125 ns DRAM) per entry. At 25% hit, multiple entries fetched → 4 µs total latency (including communication overhead).
  4. At 100% hit ratio: All entries in cache (no flash reads). FTL latency reduction = 79.27% (Figure 20c). Total latency = 0.21 (vs. 1.0), dominated by pure CXL.mem access (200 ns per entry).
💡
The 55.95%–79.27% FTL latency reduction comes from bypassing the OS stack (10 µs → 200 ns) and fine-grained access (8 B entry vs. 4 KB DMA). The traffic reduction shown in Figure 20a reflects the measured communication overhead for XHarvest's approach.

③ Memory contention relief in Terasort

The paper says XHarvest reduces execution time by 2.27× over OCSSD-H (high contention) in Terasort with 64 GB memory. Show the calculation using Figure 16b data.

  1. OCSSD-H setup: Terasort gets 64 GB memory (application). OCSSD-H statically reserves 80% of XHarvest's host memory (due to excessive static occupation), so Terasort effectively has 64 GB − (80% × 128 MB) ≈ 63.9 GB. High contention → frequent page swapping.
  2. XHarvest setup: Terasort gets 64 GB. XHarvest harvests 128 MB only during bursts (~0.28% of time). No static occupation → Terasort has full 64 GB most of the time. Low contention.
  3. Execution time (Figure 16b, normalized to XHarvest = 1.0): OCSSD-H = 2.27 (i.e., 2.27× slower). DLSSD = 1.14 (less contention than OCSSD-H but still static 128 MB). XHarvest = 1.0 (baseline, dynamic harvesting).
  4. Time reduction: OCSSD-H: 2.27 / 1.0 = 2.27× slower. If XHarvest takes 100 s, OCSSD-H takes 227 s. Savings: 127 s (56% of OCSSD-H's time).
🌾
Key insight: Terasort's working set (70 GB) exceeds memory, so contention forces swapping. XHarvest's dynamic harvesting releases memory when I/O is low, reducing swap frequency. OCSSD's static occupation amplifies the penalty.

④ Throughput gain over ConvSSD (macrobenchmark)

The paper says XHarvest improves bandwidth by 5.02% over ConvSSD in macrobenchmarks (Figure 13). Where does this gain come from, given ConvSSD has a full FTL table in DRAM?

  1. ConvSSD: Full ARM processor (100% compute), full DRAM (10 GB for 7.5 TB SSD), entire FTL mapping table cached. No flash reads for translation pages. Serves I/O at ARM speed.
  2. XHarvest: Modest ARM (25%), modest DRAM (10%). Enclave (host CPU) handles 6× more I/O than ARM. Unified FTL cache (EPC + internal DRAM) has higher hit ratio due to LRU policy and larger effective size (128 MB EPC + 1 GB internal vs. 10 GB internal).
  3. Where XHarvest wins: Sequential workloads: enclave's superior compute power (6× ARM) exhausts NAND flash parallelism faster. ConvSSD's ARM bottlenecks. XHarvest gains 11.52% in sequential (Figure 12). Random workloads: spatial locality in real traces (e.g., lun0, prn1) reduces translation page misses. XHarvest's hit ratio ≈ ConvSSD's (both ~100% for hot data). Enclave still wins on compute. Average gain: 5.02%.
  4. Where ConvSSD wins: Purely random, no locality (microbenchmark). ConvSSD's full DRAM avoids all translation page reads. XHarvest lags by 63.04% (Figure 12, 64K-RandRd). But real workloads (Figure 13) show locality, narrowing the gap to +5.02% for XHarvest.
💡
XHarvest's gain comes from superior host CPU (6× ARM) exploiting flash parallelism, not from memory capacity (ConvSSD has 10× more DRAM). Real workloads' spatial locality mitigates the translation page read penalty.

Problem 1 · easy
XHarvest reserves 25% internal compute and 10% memory. Why not 0% (like OCSSD) to maximize cost savings?
Show answer
Reserving 0% (OCSSD) forces static host CPU/memory occupation for all I/O, even low loads (96.64% of runtime). This causes severe contention with applications (>95% memory use). XHarvest's 25% compute + 10% memory serve regular I/O internally, isolating SSD tasks from apps. Only bursts (0.28% of time) harvest host resources dynamically. This balance reduces contention (2.27× vs. OCSSD) while achieving 31.50% cost savings vs. ConvSSD (100% internal).Misconception: that 0% internal = maximum efficiency. Static host occupation always competes with apps, even at low I/O. Modest internal resources provide isolation.
Problem 2 · medium
The paper says CXL.mem reduces metadata traffic by 33.97% over DRAMless SSD (Figure 20a). Explain how this measured reduction comes from XHarvest's approach.
Show answer
DRAMless SSD uses DMA to transfer 4 KB translation pages even when only one 8 B entry is needed. XHarvest uses CXL.mem with cacheline granularity (64 B), which is far more efficient. The 33.97% reduction shown in Figure 20a is the measured traffic comparison at 25% hit ratio, accounting for real access patterns including communication overhead (request messages, PPN returns), encryption (AES-GCM MAC), and eviction behavior. The measurement reflects the actual traffic generated by each system's access pattern, not a simple arithmetic calculation.Misconception: that the 33.97% can be derived from basic arithmetic like (4KB - 64B)/4KB. The figure shows measured traffic including all system overheads (communication, encryption, evictions) under realistic access patterns.
Problem 3 · hard
XHarvest's enclave harvests 1 host CPU core during bursts. If a server has 24 SSDs, could all 24 burst simultaneously and starve the host's 128 cores? Calculate the worst-case CPU demand.
Show answer
Worst case: all 24 SSDs burst simultaneously (I/O >60%). Each enclave needs 1 core → 24 cores total. Host has 128 cores → 24 / 128 = 18.75% CPU. Applications get 104 cores (81.25%). Is this realistic? No. Alibaba traces (Figure 3c) show >5 active VMs (I/O >50%) only 2.20% of runtime. Storage allocation: multiple VMs share an SSD. If 24 SSDs serve 96 VMs (4 VMs/SSD), then >20 SSDs bursting (80+ VMs active) is astronomically rare (CDF → 0%). Expected: ~2–5 SSDs burst simultaneously → 2–5 cores (1.56%–3.91% CPU). XHarvest's dynamic harvesting exploits burst interleaving, not worst-case provisioning. In practice, 24 SSDs require ≤8 cores (6.25% CPU) under typical moderate loads (paper's note, §6.2).Misconception: that dynamic harvesting must provision for worst-case (all drives burst). Production data show bursts are sparse and interleaved (few drives peak simultaneously), so on-demand allocation safely shares host CPU.
Problem 4 · medium
The paper says secure CXL traffic (AES-GCM encryption) adds only ~5% latency overhead. Why is this negligible compared to flash operations?
Show answer
AES-GCM encryption: ~1 CPU cycle per byte (modern CPUs with AES-NI instructions). For a 64 B cacheline (CXL.mem transfer), encryption = 64 cycles. At 2.8 GHz CPU (Intel Xeon 8562Y+), 1 cycle = 0.357 ns. Encryption latency = 64 × 0.357 ns = 22.85 ns. CXL.mem access latency (cross-NUMA): 75 ns (CXL-induced) + 125 ns (DRAM) = 200 ns. Encryption overhead: 22.85 / 200 = 11.43%. But compared to flash ops: Read latency = 25 µs = 25,000 ns. Write latency = 50 µs = 50,000 ns. Encryption: 22.85 / 25,000 = 0.09% (read) or 0.046% (write). Negligible. Even if CXL.mem is on the critical path (FTL cache hit), 11.43% of 200 ns = 22.85 ns is dwarfed by firmware processing (~100s of ns) or flash ops. The ~5% paper quote accounts for end-to-end latency (including firmware, flash), not just CXL.mem.Misconception: that encryption overhead is measured only against CXL.mem latency (200 ns), where it's 11.43%. In reality, flash ops (25–50 µs) dominate SSD latency, so encryption is <0.1% of total.
Problem 5 · easy
XHarvest's load detector monitors I/O bandwidth in 5 ms windows. If a burst lasts 3 ms, will the detector miss it?
Show answer
The detector checks every 5 ms window. If a burst lasts 3 ms, it falls entirely within one window. The detector computes cumulative I/O over 5 ms; if the burst pushes cumulative BW >60%, the flag activates. So a 3 ms burst can be detected if it's intense enough to raise the 5 ms average above 60%. Example: Burst at 100% BW for 3 ms + idle 2 ms = (100% × 3 + 0% × 2) / 5 = 60% average → detected. But if burst is 50% for 3 ms, average = 30% → not detected. The 5 ms window smooths noise (prevents false positives from brief spikes) while catching sustained bursts. Paper (§5.4) notes transient latency increase limited to 5 ms, implying detection is fast enough for bursts lasting >5 ms (typical in production).Misconception: that 5 ms window can't detect sub-5 ms bursts. It can if the burst is intense enough to raise the window average above 60%. The window is a time-averaged threshold, not a minimum burst duration.
Problem 6 · hard
XHarvest partitions resources approximately 1:6 (SSD firmware : enclave) based on compute power. But read requests are directed by PPN rather than following partitioning. Could this cause contention? Explain.
Show answer
Write requests respect partitioning: firmware and enclave write to their assigned portions of the resources. Read requests are directed by the physical page number (PPN), meaning both firmware and enclave may access the same flash channels and memory to retrieve data. The paper explicitly states that read requests do not follow flash-channel partitioning and introduces the potential for contention over memory and flash channels. However, XHarvest's I/O load distribution (approximately 1:6 by LPN ranges, §5.4) means firmware generates fewer reads (1/7 of total) and enclave generates more (6/7), reducing the likelihood of simultaneous access to the same resources. Figure 22 shows computation resource usage for both components during operation.Misconception: that read requests cause no contention because they don't modify state. While reads are non-mutating, they still consume channel bandwidth and memory bandwidth, introducing potential contention when both firmware and enclave access the same channels simultaneously.
Problem 7 · medium
The paper says XHarvest's enclave incurs 5.09% throughput degradation vs. non-EPC memory (Figure 21, 64K random write). Why is this overhead so small?
Show answer
EPC (Enclave Page Cache) is encrypted by SGX. On CPU cache load, the processor decrypts the cacheline; on eviction, it encrypts. Encryption overhead depends on access frequency. FTL tasks are simple for XHarvest: lookup mapping entry (1 entry per I/O request), update PPN on write, evict victim page on cache miss. Each I/O request touches 1 FTL entry (8 B). For 64K write, the enclave processes ~16 K requests/sec (Figure 21). Encryption: 16 K × 8 B = 128 KB/sec encrypted data. At 1 CPU cycle/byte (AES-NI), 128 KB × 1 cycle/byte = 131,072 cycles/sec = 0.047 ms/sec at 2.8 GHz. Negligible CPU load. The 5.09% degradation comes from cache pollution: EPC pages compete with other LLC entries (firmware code, flash metadata). More LLC misses → slightly slower firmware execution. For 64K read (simpler FTL processing), EPC overhead is <0.1% (Figure 21).Misconception: that EPC encryption always incurs heavy overhead. Overhead scales with encrypted data volume. XHarvest's simple FTL access pattern (1 entry/request) generates low encryption load, so overhead is ~5% in the worst case (write-heavy, small requests).
Problem 8 · easy
XHarvest harvests 128 MB host EPC during bursts. If multiple SSDs burst simultaneously, could they exhaust host memory (e.g., 480 GB DRAM)?
Show answer
Host DRAM: 480 GB. One SSD harvests 128 MB. Max SSDs before exhaustion: 480 GB / 128 MB = 3,750 SSDs. A typical storage server has 24 SSDs (§6.2). Even if all 24 burst simultaneously (astronomically rare, §Practice 3), total harvest = 24 × 128 MB = 3 GB (0.625% of 480 GB). Applications get 477 GB (99.375%). In reality, burst interleaving (2.20% of time >5 VMs active) means ~2–5 SSDs burst → 256 MB – 640 MB (0.05%–0.13% of DRAM). XHarvest's dynamic harvesting ensures low simultaneous memory use, avoiding exhaustion.Misconception: that harvesting host memory is risky for memory exhaustion. XHarvest harvests on-demand (few SSDs burst simultaneously) and releases when idle. Even 24 SSDs (worst case) use <1% of host DRAM.
Quick answers

Frequently asked questions

Ten questions newcomers ask most. Tap any to expand.

Q1What is the main problem XHarvest solves?▾
Conventional SSDs provision expensive internal compute (30% cost) and DRAM (10%) to handle peak I/O bursts, but these resources sit idle 96.64% of the time (Alibaba: I/O BW <25%). XHarvest shifts to dynamic allocation: reserve modest internal resources (25% compute, 10% memory) for regular I/O, then harvest host CPU and DRAM only during bursts (0.28% of time I/O >80%). This saves 31.50% cost while maintaining performance.
Q2Why not just use Open-Channel SSD (OCSSD)?▾
OCSSD removed internal resources but statically occupied host CPU and memory for all I/O, causing severe contention with applications (Alibaba: >95% memory use at low I/O). XHarvest's dynamic harvesting releases host resources when I/O is low (96.64% of time), reducing contention by 2.27× execution time (Terasort). Also, OCSSD faced adoption obstacles from firmware leakage risks and vendor resistance to putting proprietary firmware in the OS kernel. XHarvest uses SGX enclave to protect secrets.
Q3How does CXL help XHarvest?▾
CXL.mem unifies host memory and SSD internal DRAM in a cache-coherent domain, enabling fine-grained (8 B entry vs. DMA's 4 KB page) and direct access (200 ns vs. 10 µs OS stack). This eliminates unnecessary metadata transfer (33.97% traffic reduction) and bypasses ecall/ocall overhead (96.13% throughput penalty without CXL). XHarvest builds a unified FTL cache spanning host EPC + internal DRAM, accessed via load/store instructions (CXL.mem).
Q4What is the enclave's role?▾
The enclave (Intel SGX) runs the SSD firmware (proprietary FTL algorithms) on the host CPU in an isolated, encrypted container (EPC). This protects firmware secrets from the OS or malicious apps while exploiting the powerful host CPU (6× more compute than SSD ARM). Attestation verifies the enclave is genuine; CXL TSP encrypts enclave–SSD traffic (AES-GCM, ~5% overhead). The enclave harvests 1 host CPU core during bursts and releases it when idle.
Q5How does XHarvest avoid memory contention?▾
XHarvest harvests host memory (128 MB EPC) dynamically only during bursts (few SSDs peak simultaneously, §3.3). At low I/O (96.64% of time), it uses only internal DRAM (10% of full table), leaving host memory free for applications. Burst interleaving (>5 active VMs only 2.20% of time) ensures low simultaneous memory use (e.g., 24 SSDs → <3 GB, <1% of 480 GB DRAM). This reduces Terasort execution time by 2.27× vs. OCSSD's static occupation.
Q6What is CXL TSP and why is it needed?▾
CXL 3.1 TSP (TEE Security Protocol) encrypts CXL traffic between the enclave and SSD using AES-GCM. Without it, a compromised OS could snoop CXL.mem traffic (requests, FTL entries) and leak proprietary algorithms. TSP performs mutual authentication (enclave ↔ SSD verify certs) and exchanges symmetric keys. Encryption adds ~5% latency (1 CPU cycle/byte), negligible vs. flash ops (25 µs read, 50 µs write). This secures the host-SSD collaboration while maintaining efficiency.
Q7How does the load detector work?▾
A circuit in the SSD monitors I/O bandwidth in 5 ms windows. If cumulative BW exceeds 60%, it sets a flag in host memory (via CXL.mem). A daemon thread polls the flag every 1 ms (0.1% CPU). On flag activation, the thread re-enters the pre-created enclave (via ecall) and allocates EPC for FTL caching. When load drops, the enclave flushes dirty pages, releases EPC, and exits. Pre-creation at boot hides attestation latency; 5 ms detection window limits transient latency spikes (no impact on tail latency over 60 s, Figure 23).
Q8Why does XHarvest outperform ConvSSD by 5.02%?▾
ConvSSD has full internal ARM + DRAM but is bottlenecked by ARM compute power. XHarvest's enclave (host CPU) is 6× faster, exhausting flash parallelism more efficiently. In sequential workloads, XHarvest gains 11.52%; in random workloads with spatial locality (real traces), it gains 5.02%. ConvSSD wins only in purely random microbenchmarks (no locality) due to its full DRAM (no translation page reads), but such patterns are rare in production.
Q9What are the hardware overheads?▾
On-path latency: secure CXL traffic (TSP AES-GCM) adds ~5%; EPC encryption adds ~5.09% in worst case (64K write). Total <10%, negligible vs. 31.50% cost savings. Off-path: load detector (5 ms window), daemon thread (0.1% CPU polling), enclave startup (pre-created at boot, re-entered via ecall ~20K cycles). No hardware redesign needed (adheres to CXL/NVMe specs).
Q10Does XHarvest always beat OCSSD?▾
In ideal conditions (no contention, sufficient host CPU/memory), OCSSD is 39.29% faster (Figure 12, microbenchmark) due to full host resource exploitation. But in realistic workloads with memory-intensive applications (>95% memory use), OCSSD causes severe contention, slowing execution by 2.27× (Terasort). XHarvest's dynamic harvesting avoids this penalty, outperforming OCSSD-H by 2.27× in memory-intensive tasks and matching OCSSD in low-contention scenarios (only 0.70% behind in macrobenchmarks, Figure 13).

Glossary & further reading

Core terms
XHarvestThe paper's SSD architecture: reserve modest internal resources (25% compute, 10% memory), dynamically harvest host CPU/DRAM during bursts via CXL and SGX.
Dynamic harvestingOn-demand allocation of host CPU/memory only during I/O bursts (>60% BW over 5 ms window), releasing them when idle. Contrasts with static provisioning (always occupy).
CXL (Compute Express Link)Interconnect standard (atop PCIe PHY) unifying host and devices in a cache-coherent domain. CXL.mem: host accesses device memory; CXL.cache: device caches host memory.
CXL TSP (TEE Security Protocol)CXL 3.1 feature encrypting CXL traffic (AES-GCM) between a TEE (enclave) and a device (SSD) after mutual authentication. Prevents OS snooping.
SGX (Software Guard Extensions)Intel TEE isolating code/data in an encrypted enclave (EPC). CPU decrypts on LLC load; attestation verifies genuineness. Protects firmware secrets on untrusted host.
EnclaveSecure container (SGX) where XHarvest runs firmware on the host CPU. Pins to 1 core, polls SSD requests via CXL, accesses unified FTL cache, releases on idle.
Unified FTL cacheMetadata cache spanning host EPC (128 MB) + SSD internal DRAM (10% of full table). Accessed via CXL.mem (enclave) or direct DRAM (SSD firmware). LRU eviction.
Load detectorCircuit in SSD monitoring I/O BW in 5 ms windows. If >60%, signals daemon thread (via CXL.mem flag) to launch enclave. Decoupled from critical path (pre-created at boot).
Supporting terms
FTL (Flash Translation Layer)Firmware mapping logical (LPN) to physical (PPN) addresses. Out-of-place updates due to erase-before-write. Stored in DRAM (ConvSSD) or unified cache (XHarvest).
Translation pageSpecial flash page storing multiple FTL mapping entries (adjacent LPNs). XHarvest caches these in EPC + internal DRAM (LRU policy). Covers ~2 MB LPN space.
OCSSD (Open-Channel SSD)Removed internal compute/DRAM, ran firmware (pblk) in Linux kernel. Statically occupied host resources → severe contention. Faced adoption obstacles from firmware leakage risks. Deprecated in kernel 5.15.
DRAMless SSDOmits internal DRAM, buffers FTL cache in host memory (128 MB). Retains full ARM, uses DMA (4 KB granularity, misaligned with 8 B entries), encrypts host metadata.
ConvSSD (Conventional SSD)Full internal ARM (100% compute) + DRAM (e.g., 10 GB for 7.5 TB). Entire FTL table cached in DRAM. Baseline for cost/performance comparison.
Burst interleavingProduction pattern: few SSDs peak simultaneously (Alibaba: >5 VMs active only 2.20% of time). Enables dynamic memory sharing instead of static reservation.
EPC (Enclave Page Cache)Encrypted host memory region for SGX enclaves. CPU decrypts on LLC load; OS sees ciphertext. XHarvest allocates 128 MB EPC on burst, releases on idle.
AttestationProof-based verification (e.g., Intel Attestation Service) that the enclave runs on genuine SGX hardware with expected initial state. Ensures trustworthiness.
Ecall / ocallTransitions between enclave (trusted) and untrusted code. ~20K cycles each. XHarvest replaces with CXL-driven communication to avoid this overhead.
CXL-driven communicationMessage-passing mechanism via ring buffer in SSD internal DRAM (64 B messages). Enclave writes message, sets ready bit; SSD polls via CXL.mem. 3700K OPS (single SSD).
Host-SSD coordinationPartitions flash channels, memory, and I/O load approximately 1:6 (firmware : enclave) proportional to compute power (§5.4). Helps avoid contention while exploiting parallelism.
