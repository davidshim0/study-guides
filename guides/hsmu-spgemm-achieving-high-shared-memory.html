Giving shared memory to the right accumulator, not just any accumulator

Hash-based SpGEMM on GPUs wastes shared memory to avoid collisions. This is the plain-English story of HSMU-SpGEMM, a binary-search accumulator that uses exact-size arrays for its logical structure — and runs up to 131× faster.

📄 "HSMU-SpGEMM" · Wu et al. (HPCA'25) 🎓 GPU basics assumed — shared memory, warps, SMs ⏱ ~25 min read
🔍

Give each slot in shared memory to a real NZ element, not to empty hash space.
Keep a sorted column list; binary-search replaces hash-lookup — zero collisions, exact-size accumulator arrays.

Before you begin

Quick glossary and ten building blocks

Key acronyms defined up front, then ten concepts HSMU-SpGEMM builds on. Expand any card; read in order the first time.

🔑 Quick glossary

SpGEMM Sparse General Matrix-Matrix Multiplication (C = A × B, all sparse)
NZ Non-Zero element in a sparse matrix
NNZ Number of Non-Zeros in a matrix or row
SM Streaming Multiprocessor — GPU's parallel execution unit
Warp Group of 32 GPU threads executing together
Shared memory Fast on-chip cache/scratchpad per SM (~128 KB)
Hash table Key-value structure for fast lookups (prone to collisions)
CSR Compressed Sparse Row format (ptr, col, val arrays)
1What SpGEMM is and why it matters▾

SpGEMM (Sparse General Matrix-Matrix Multiplication) computes C = A × B where all three matrices are sparse — most entries are zero. Storing only NZ elements saves space but makes multiplication complex.

It's a core primitive in scientific simulations (algebraic multigrid solvers, iterative methods), graph processing (breadth-first search, Markov clustering, triangle counting), and machine learning (sparse neural networks). Performance matters because real matrices can be gigantic (millions of rows) yet very sparse (<0.01% nonzero).

Now you're ready: HSMU-SpGEMM is about making this multiplication fast on GPUs by using shared memory efficiently during the accumulation step.
2Gustavson's algorithm — the row-row method▾

Gustavson's algorithm (1978) computes SpGEMM row-by-row: for each row i of A, and each NZ element aij in that row, multiply it with every element in row j of B to generate intermediate products. These products all belong to row i of the output matrix C.

The core challenge: many intermediate products share the same column index (because different paths through the matrices can multiply to the same output cell). An accumulator collects and sums these products efficiently. The better the accumulator, the faster SpGEMM runs.

Now you're ready: The entire paper is about designing a better accumulator — one that stores intermediate products in GPU shared memory without wasting space.
3Symbolic vs numeric stages▾

Most SpGEMM implementations split work into two phases. Symbolic stage: figure out where the NZ elements of C will be (how many per row, which column indices) without computing actual values. Allocate storage. Numeric stage: compute the actual NZ values and store them.

Why separate? Because you need to know the output size before allocating memory. The symbolic stage often runs a cheaper "trial" version of the multiply to count NZs; the numeric stage then does the real work.

Now you're ready: HSMU-SpGEMM's symbolic stage does more than count — it builds a sorted column index array for each row (via mask generation and extraction), which the numeric stage then uses for collision-free accumulation.
4GPU shared memory — fast but tiny▾

Each Streaming Multiprocessor (SM) on a GPU has a small on-chip memory (~128 KB on NVIDIA RTX 3090 Ti) accessible to all threads in a block. Shared memory is ~100× faster than global DRAM but limited in size.

Accumulators want to live in shared memory for speed. But if the accumulator structure is too big (e.g., a hash table 2× the needed size to reduce collisions), it spills to global memory — killing performance. The paper's metric: shared memory utilization = actual data / allocated space. Higher is better.

Now you're ready: Traditional hash tables in prior work allocate extra space to reduce collisions — OpSparse uses roughly 2× the needed size (averaging 35% utilization), while spECK's fill is capped at 66% though actual utilization can be inefficient for some matrices.
5Hash-based accumulators and their trade-off▾

A hash accumulator uses a hash function to map each intermediate product's column index to a position in a hash table. If two products hash to the same slot (a collision), you must search for an empty slot via linear probing — slow.

The classic fix: make the hash table bigger than the number of NZs (e.g., roughly 2× for OpSparse, capped at 66% fill for spECK). Fewer collisions but wasted shared memory — some slots stay empty. This inherent trade-off (collision rate vs utilization) is what existing libraries accept.

Now you're ready: HSMU-SpGEMM breaks the trade-off: exact-size logical arrays (every slot holds real data) and zero collisions (by using binary search on a sorted array instead of hashing).
6CSR format — how sparse matrices are stored▾

Compressed Sparse Row (CSR) stores a sparse matrix in three arrays: ptr (row start indices), col (column indices of NZs), and val (values). For a matrix with m rows and nnz nonzeros, ptr is length m+1, col and val are length nnz.

Example: a 4×4 matrix with NZs at (0,1), (0,3), (2,2) becomes ptr=[0,2,2,3,3], col=[1,3,2], val=[a,b,c]. Row 0 has elements from index 0 to 2 (exclusive) in col/val.

Now you're ready: HSMU-SpGEMM works with CSR input and output, but internally uses a new "compressed mask" structure during symbolic stage for large matrices.
7Intermediate products — the flood to accumulate▾

When computing row i of C, you generate many intermediate products: for each NZ aij in row i of A, multiply it with every NZ in row j of B. If B's row j has 1000 NZs, you get 1000 products from just one aij.

The compression ratio (paper Table II) is the ratio of total intermediate products to final NZs in C. Ratios of 10–100 are common — e.g., matrix gupta3 has compression 113 (30.7 billion intermediate products collapse to 270.9 million NZs). The accumulator must handle this flood efficiently.

Now you're ready: HSMU-SpGEMM's accumulator maps each intermediate product to its final position using binary search on a pre-sorted column array — no matter how many products arrive.
8Warp divergence and why uniformity helps▾

A GPU warp executes 32 threads in lockstep. If threads take different branches (e.g., some hit, some search for a slot), the warp serializes both paths — effectively cutting parallelism. This is warp divergence.

Hash collisions cause divergence: some threads find their slot immediately, others loop to probe. Binary search also branches, but the paper's Fig. 13 shows similar "average active threads per warp" (~20) for HSMU-SpGEMM, spECK, and OpSparse — divergence is comparable, not worse.

Now you're ready: Binary search doesn't hurt warp efficiency more than hash probing does — and it gains zero collisions and higher shared memory utilization.
9Batch processing and binning for load balance▾

Rows of C vary wildly in NNZ (some have 10, others 10,000). Processing them one thread-block per row would leave some blocks idle. Instead, libraries group rows into batches of similar size and assign batches to SMs for better load balance.

HSMU-SpGEMM uses Nsparse's binning strategy: sort rows into bins by NNZ, then process each bin with appropriate resources (e.g., many small rows per block, or one huge row per block). This keeps SMs busy and shared memory well-utilized.

Now you're ready: The batch model is why shared memory utilization matters — each batch allocates a chunk of shared memory, and wasting it means fewer concurrent batches (lower throughput).
10The mask matrix trick — bits, not values▾

HSMU-SpGEMM's symbolic stage uses a clever auxiliary structure: a mask matrix where each bit represents "is there a NZ at this position?" For small matrices, each row is a bitstring (64 bits per unit). For large matrices, it's compressed into a sparse format (only store NZ units).

Why? Bitwise OR is fast and the mask reveals NZ positions without computing values. After OR-ing mask rows from B according to A's NZs, you count bits (via __popcll()) to get NNZ and extract column indices (via __ffsll()) — already sorted.

Now you're ready: This mask is the secret to zero-collision accumulation — the sorted column array comes from the mask, not from hashing or merging.
The whole story in one place

HSMU-SpGEMM in a nutshell

The problem, the measurement fix, and how it all connects.

Hash-based SpGEMM libraries (Nsparse, spECK, OpSparse, cuSPARSE) store intermediate products in a hash table during Gustavson's row-row algorithm. But hash collisions force a choice: either accept a high collision rate (Nsparse: 400% on matrix cant) or allocate larger tables to reduce collisions — OpSparse uses roughly 2× the needed size (averaging 35% utilization), while spECK's table fill is capped at 66%.

HSMU-SpGEMM's fix: replace the hash table with a sorted column array (length = exact NNZ of the row) and a value array (also exact NNZ). To position an intermediate product, binary-search the sorted columns for its column index — guaranteed to find it (zero collisions) in O(log N) time. This uses exact-size arrays with real data in the binary-search accumulator's logical structure.

The sorted column array comes from the symbolic stage: for small matrices, HSMU-SpGEMM builds a bitmask of B, ORs rows per A's NZs to get C's mask, then extracts sorted column indices via __ffsll(). For large matrices, it uses a compressed mask (CSR-like for masks) and calls OpSparse's symbolic kernel modified to do OR instead of multiply-add. This preprocessing has non-trivial cost (Fig. 17 reports "Generate auxiliary mask" averages 26.40% of total time and "Generate sorted Ccol" averages 15.83%).

The numeric stage loads the sorted column array into shared memory, initializes a value array, and processes intermediate products: each warp handles one NZ of A, binary-searches the column array to find positions, and atomically adds to the value array. Dense rows (>4096 NZ) use spECK's dense accumulator instead.

The payoff: on 338 matrices from SuiteSparse, HSMU-SpGEMM achieves geometric mean speedup of 3.19× over Nsparse, 1.28× over spECK, 2.34× over OpSparse, and 5.06× over cuSPARSE (NVIDIA RTX 3090 Ti). Max speedup: 131.64× (vs Nsparse). Measured shared memory utilization: HSMU-SpGEMM 71.69% on matrix cant, about 68% average (Fig. 11); OpSparse 35.35%, Nsparse 70.67%. Peak memory often lower than OpSparse for matrices with many long rows.

🍯 Key takeaways

  • Hash tables trade collision rate for utilization; HSMU-SpGEMM gets both: 0% collisions, exact-size logical arrays in the binary-search accumulator.
  • Binary search on a sorted column array replaces hash lookup — stable O(log N) cost, no chain collisions.
  • Sorted columns come from the symbolic stage via bitmask OR and __ffsll() extraction, but this preprocessing has cost (mask generation ~26% of time, sorted col extraction ~16%).
  • Numeric stage: one warp per NZ of A, binary-search for positions, atomic-add to value array in shared memory.
  • Dense rows (>4096 NZ) switch to spECK's dense accumulator — hybrid approach for wide support.
  • Result: up to 131× (avg 3.2×) faster than Nsparse, 68% avg shared memory utilization, lower peak memory for long rows.

How to think about HSMU-SpGEMM

Each picture stands alone. Every card ends with where the analogy breaks.

The core idea · Utilization vs collisions

Parking lot assignment

A hash-based lot assigns cars by license-plate hash — but reserves extra empty spots to reduce collisions (if two hash to slot 3, one probes to slot 4). A sorted-list lot assigns by alphabetical order — every spot holds a real car (100% full), and you find yours by binary search in the directory. Same cars, zero wasted spaces.

Where it breaks: real parking doesn't pre-sort arrivals; HSMU-SpGEMM's "pre-sort" happens in the symbolic stage (building the mask and extracting sorted columns), which has its own cost (~26% + ~16% of total time per Fig. 17).
The lookup method · Binary search

Phone book vs hash table

Finding "Smith" in a phone book: open to the middle, too far? go left half; repeat — log N steps. A hash table would compute H("Smith") and jump straight there if no collision. Binary search is slower per lookup (log N vs O(1)) but needs zero extra space and never collides. For GPU shared memory, space is the bottleneck.

Where it breaks: if you're adding/removing names constantly, hashing wins; but HSMU-SpGEMM's sorted array is fixed after symbolic stage — no insertions, so the hash advantage disappears.
The symbolic stage · Bitmask OR

Overlapping transparencies

Place two transparencies (each with some black dots) on top of each other — the combined image shows a dot wherever either original had one. HSMU-SpGEMM's mask B has a 1-bit for each NZ column in B's rows; OR-ing rows from mask B (according to A's NZs) produces mask C's row — already marking exactly which columns have NZs in C.

Where it breaks: transparencies are physical; HSMU-SpGEMM's mask is a compressed bitstring (64-bit units for small matrices, CSR-like for large). Also, OR is fast, but __ffsll() extraction adds overhead.
Warp divergence · Branch uniformity

Concert crowd turning heads

A warp is like 32 people in lockstep: if half turn left and half turn right, you wait for both groups to finish before moving on. Binary search branches log N times, but so does hash probing with collisions. The paper's Fig. 13 shows both average ~20 active threads/warp — divergence is comparable, not worse.

Where it breaks: real people don't execute in SIMD; GPU threads do. And if hash collisions are rare, hash might diverge less — but then you waste shared memory to make collisions rare.
Shared memory · Precious resource
Hotel rooms in Manhattan

Shared memory is like hotel rooms in Manhattan — expensive and scarce (~128 KB per SM). If your accumulator reserves 2× the needed rooms "just in case," you halve the number of batches that can run concurrently. HSMU-SpGEMM's exact-size logical arrays mean more batches fit, higher throughput.

Where it breaks: hotel rooms are rented continuously; GPU batches launch in waves. But the analogy holds: wasting shared memory reduces occupancy, which reduces parallelism.
Hybrid approach · Dense rows
Toolbox with specialized tools

You wouldn't use a wrench to hammer nails. HSMU-SpGEMM uses binary search for sparse/medium rows (<4096 NZ) but switches to spECK's dense accumulator for very dense rows — a direct array indexed by column. Hybrid design handles the full range of sparsity patterns efficiently.

Where it breaks: real tools don't auto-switch; HSMU-SpGEMM sets a threshold (4096 NZ) based on empirical tuning (Fig. 6). Different matrices might prefer different thresholds.
The full explanation

How it actually works

Section by section, with the paper's key diagrams rebuilt.

1 · The problem — hash trade-off is inherent

Traditional hash-based accumulators (Nsparse, spECK, OpSparse, cuSPARSE) map intermediate products to a hash table via hash = (col_index * scale) MOD table_length. Collisions (different columns hash to the same slot) are resolved by linear probing — searching forward for an empty slot.

The trade-off: small tables → many collisions → slow probing. Large tables (roughly 2× needed for OpSparse, capped at 66% fill for spECK) → fewer collisions but wasted shared memory. On matrix cant, Nsparse (1× table) has 400% collision ratio (must probe 4× on average); OpSparse (2× table) has 2.75% collisions but only 35% shared memory utilization.

🌱
Concretely (Table I): Nsparse: 70.67% utilization, 1.08 billion collisions, 11.30 ms. OpSparse: 35.35% utilization, 7.4 million collisions, 9.02 ms. HSMU-SpGEMM: 71.69% utilization, zero collisions, 7.80 ms.

2 · The solution — sorted array + binary search

HSMU-SpGEMM replaces the hash table with two exact-size arrays: col[] (sorted column indices of NZs, length = NNZ of row) and val[] (values, also length NNZ). To position an intermediate product with column c:

function findInSorted(col[], start, c):
    # binary search col[] for c, starting at start
    lo = start; hi = len(col) - 1
    while lo <= hi:
        mid = (lo + hi) / 2
        if col[mid] == c: return mid
        if col[mid] < c: lo = mid + 1
        else: hi = mid - 1
    return -1  # not found (shouldn't happen if mask correct)

This guarantees O(log N) lookup, zero collisions (every column in col[] is a real NZ), and exact-size arrays in the binary-search accumulator's logical structure. The catch: you need col[] sorted before the numeric stage — which is what the symbolic stage computes.

3 · Symbolic stage (small matrices) — bitmask method

For matrices where each row fits in <2048 columns (so 2048 bits = 32×64-bit units fit in shared memory), HSMU-SpGEMM uses a direct bitmask. Three steps:

  1. Build mask B: For each row of B, create a bitstring where bit j = 1 if column j has a NZ, 0 otherwise. Store as 64-bit integers. Example: row [0,0,a,0,b] → bits 00101 (positions 2,4 set).
  2. Build mask C: For each row i of A, for each NZ aij in that row, OR mask B's row j into mask C's row i. This marks all columns where C will have NZs. Embarrassingly parallel across rows of A.
  3. Extract sorted columns: For mask C's row i, count 1-bits with __popcll() (gives NNZ), then extract positions of 1-bits with __ffsll() (find first set bit, clear it, repeat). Positions come out sorted left-to-right → sorted column array.
💡
Why sorted? Bits are stored left-to-right in the integer, and __ffsll() scans from low to high bit — so extracted positions are inherently sorted. No explicit sort needed.

4 · Symbolic stage (large matrices) — compressed mask

For matrices >2048 columns, the bitmask would be too large for shared memory. HSMU-SpGEMM switches to a compressed mask format similar to CSR: store only the 64-bit units that contain 1-bits, indexed by a tilePtr and tileCol array.

The process: build compressed mask B (scan B's CSR, mark which 64-bit tiles have NZs), then call OpSparse's symbolic kernel modified to do bitwise OR instead of multiply-add. This produces compressed mask C. Extract NNZ and sorted columns the same way as small matrices (via __popcll() and __ffsll() on the stored tiles).

🌱
The compressed mask adds overhead (Fig. 17 shows "Generate auxiliary mask" averages 26.40% of total time and "Generate sorted Ccol" averages 15.83%). But it scales to huge matrices where the simple bitmask would overflow shared memory.

5 · Numeric stage — binary search + atomic-add

With sorted col[] in hand from symbolic stage, the numeric stage (Algorithm 2 in paper) processes each row of C in a thread block:

  1. Load sorted col[] into shared memory; initialize shared val[] to zero.
  2. Assign warps to NZs of A's row: warp w handles A[row_i][w] (stride by wnum = warps/block).
  3. Each thread in the warp processes one NZ of B's corresponding row: binary-search shared col[] for the product's column index, atomic-add the value to shared val[pos].
  4. After all warps finish, shared val[] contains the final NZ values for row i of C — copy to global memory.

Key optimization: the pos variable in binary search is updated after each find, so the next search starts from the last position found (columns tend to cluster) — reduces search space.

🍯
The atomic-add is necessary because multiple threads/warps can accumulate to the same column index. Shared memory atomics are fast; the binary search (O(log N)) is the main cost, but it's stable and collision-free.

6 · Hybrid: dense accumulator for long rows

Rows with >4096 NZ elements use spECK's dense accumulator instead: a direct-indexed array (length = number of columns in C) where val[c] is the value for column c. No search needed, but requires more memory and a final scan to extract NZs.

The threshold (4096) was tuned empirically (Fig. 6): too low and many rows use dense unnecessarily (slow); too high and long rows suffer from excessive binary searches. At 4096, HSMU-SpGEMM achieves max geometric mean GFLOPS (32.94 on 3090 Ti).

② Bitmask symbolic stage on toy example

Compute row 0 of C for 4×4 matrices A and B (paper Fig. 3). A's row 0 = [0, a, 0, b]. B is 4×4 with NZs shown in the figure. Build mask C.

  1. Mask B: Row 1 of B has NZ at cols 0,2 → bits 1010. Row 3 has NZ at col 1 → bits 0100. (4-bit example; real uses 64-bit units.)
  2. Compute mask C row 0: A's row 0 has NZ at cols 1,3 (indices of a,b). OR mask_B[1] with mask_B[3]: (1010) OR (0100) = 1110.
  3. Extract NNZ: __popcll(1110) = 3 → row 0 of C has 3 NZs.
  4. Extract sorted columns: __ffsll(1110) = 0 (bit 0 set) → col[0]=0. Clear bit 0 → 1100. __ffsll(1100) = 1 → col[1]=1. Clear → 1000. __ffsll(1000) = 2 → col[2]=2. Sorted col[] = [0,1,2].
💡
The numeric stage would then load col[] = [0,1,2] into shared memory, initialize val[] = [0,0,0], and binary-search to accumulate products. Zero collisions, exact fit.

Practice problems

Try before opening the answer. Each checks a common misconception.

Problem 1 · easy
A hash table with length 2000 holds 1000 NZ elements. What is its shared memory utilization, and why does HSMU-SpGEMM avoid this waste?
Show answer
Utilization = 1000 / 2000 = 50%. Half the allocated slots are empty (reserved to reduce collisions). HSMU-SpGEMM allocates exactly 1000 slots (sorted col[] and val[]) for its binary-search accumulator's logical structure, using exact-size arrays — no wasted space, no collisions (binary search finds the right slot).Misconception: that larger tables are always better. They reduce collisions but waste shared memory, reducing GPU occupancy (fewer concurrent batches).
Problem 2 · medium
Why does HSMU-SpGEMM's binary search not suffer from worse warp divergence than hash-based methods, despite branching log N times?
Show answer
Hash collisions also cause divergence: some threads find their slot immediately, others loop to probe. Binary search branches uniformly (all threads do the same log N comparisons on different data). Fig. 13 shows similar "avg active threads/warp" (~20) for HSMU-SpGEMM, spECK, and OpSparse — divergence is comparable, not worse. The key is that HSMU-SpGEMM's branches are predictable (always log N steps), while hash collisions are data-dependent.Misconception: that binary search always hurts divergence more than hashing. It depends on collision rate and access patterns.
Problem 3 · medium
A row of C has NNZ=5000. OpSparse allocates a hash table of length 10,000 (2×). HSMU-SpGEMM allocates what, and how does the dense accumulator threshold (4096) affect this row?
Show answer
HSMU-SpGEMM checks: NNZ=5000 > 4096 → use dense accumulator instead of binary search. Allocates a direct-indexed array (length = num columns in C, e.g., 10,000 if C is 10k×10k), initializes to zero, then indexes directly by column (no search). After computation, scans the array to extract the 5000 NZs. This is faster than 5000 binary searches (each O(log 5000) ≈ 12 steps) but uses more memory. OpSparse's hash table for this row would be in global memory (too big for shared) → very slow.Misconception: that HSMU-SpGEMM only uses binary search. It's a hybrid: binary search for sparse/medium rows, dense for very dense rows (threshold tuned to 4096 for best avg performance).
Problem 4 · hard
Matrix vsp_model1_crew1_cr42_south31 (paper Fig. 8a) has extremely compact NZ distribution and gives HSMU-SpGEMM a 131× speedup over Nsparse. Explain why hash-based methods fail here and HSMU-SpGEMM succeeds.
Show answer
Compact distribution → many NZs with similar column indices → they hash to nearby slots. Linear probing chains collisions → if slot k is full, try k+1, k+2, ... This creates clustering: a collision at k affects k+1, k+2, etc., causing chain reactions. Nsparse's 1× table hits extreme clustering (400% collision ratio on cant, worse on this matrix). HSMU-SpGEMM is unaffected: compact NZs just means the sorted col[] has sequential indices (e.g., [100,101,102,...]). Binary search finds them in log N steps regardless of compactness. Plus, the symbolic stage's OR operations on compact masks are faster (fewer distinct 64-bit units).Misconception: that hash tables handle all patterns equally. Clustering destroys hash performance but doesn't affect binary search.
Problem 5 · medium
The paper's Fig. 19 shows format conversion overhead (CSR → compressed mask) is mostly under one-tenth the cost of a single SpGEMM. Why does this matter for iterative solvers like algebraic multigrid?
Show answer
In AMG or Krylov methods, the paper notes you can amortize format conversion in iterative scenarios — you could run SpGEMM many times on the same matrix structure. If conversion is mostly under 1/10 per call and HSMU-SpGEMM is 3× faster per SpGEMM (geometric mean speedup), the conversion overhead becomes small relative to the overall gain in iterative workloads.Misconception: that symbolic stage overhead kills performance. The paper says it's mostly under one-tenth of one SpGEMM and can be amortized in iterative contexts.
Problem 6 · hard
Matrix in-2004 (paper Fig. 8b) has highly scattered NZ distribution and gives HSMU-SpGEMM its minimum speedup. Explain why scattered patterns favor hash-based methods.
Show answer
Scattered NZs → column indices spread across the matrix (e.g., [5, 203, 1845, 3002, ...]). Hash function distributes them uniformly → low collision rate even with a 1× table. OpSparse/spECK see few collisions, fast probing. For HSMU-SpGEMM: 1) Symbolic stage overhead increases: more distinct 64-bit units in the compressed mask (each unit covers a 64-column span; scattered NZs hit many spans) → more OR operations. 2) Dense accumulator (if row > 4096 NZ) must scan a large direct array to extract scattered NZs (many zero gaps). Binary search itself is fine (O(log N) is stable) but the symbolic overhead and extraction cost add up. Hash methods win on these matrices.Misconception: that HSMU-SpGEMM is always fastest. It wins on avg (3.2× geom mean) but loses on some scattered matrices where hash tables work well and symbolic overhead is high.
Problem 7 · easy
Why does HSMU-SpGEMM use __ffsll() (find first set bit) instead of a general sort to extract column indices from the bitmask?
Show answer
__ffsll() scans the 64-bit integer from low bit to high bit, finding the position of the first 1. Clear that bit, repeat → positions come out in ascending order automatically (bit 0, then bit 5, then bit 12, ...). No explicit sort needed; the intrinsic function does it in hardware. A general sort would be O(N log N); __ffsll() loop is O(NNZ) with low constant factor (hardware instruction).Misconception: that sorted columns require a sort pass. The bitmask representation and __ffsll() give sorted columns as a byproduct.
Problem 8 · medium
The paper's Table V shows HSMU-SpGEMM uses 3.61 GB peak memory for matrix gupta3, while OpSparse uses 9.81 GB. Both compute the same output. Where does OpSparse's extra memory go?
Show answer
Matrix gupta3 has 16,762 rows in C with >4096 NZ each (long rows). OpSparse can't fit these in shared memory, so it allocates global memory hash tables (roughly 2× row NNZ each) for all 16,762 rows simultaneously. At roughly 2× overhead and large NNZ, this balloons to 9.81 GB. HSMU-SpGEMM uses the dense accumulator for long rows, which processes them in batches (not all at once) and only stores the final NZ values (not 2× hash table overhead) → 3.61 GB. The paper notes OpSparse fails to complete case39 and TSOPF_FS_b300_c2 due to out-of-memory, while HSMU-SpGEMM completes them.Misconception: that OpSparse's roughly 2× table is always small. For matrices with many long rows, the global memory overhead becomes prohibitive.
Quick answers

Frequently asked questions

Ten questions newcomers ask most. Tap any to expand.

Q1What does "shared memory utilization" actually mean?▾
Ratio of space used by real data to space allocated in shared memory. A hash table of length 2000 holding 1000 NZs has 50% utilization — half the slots are empty (reserved to avoid collisions). HSMU-SpGEMM allocates exactly 1000 slots (sorted col[] + val[]) for its binary-search component → exact-size logical arrays. Measured avg utilization is about 68% (including dense rows overhead). Higher utilization → more batches fit in shared memory → higher GPU occupancy → faster.
Q2Why is binary search stable but hash probing isn't?▾
Binary search always takes O(log N) steps — predictable, no data-dependent variance. Hash probing depends on collisions: if few, it's O(1); if many (clustering), it can degenerate to O(N). Compact NZ patterns (paper Fig. 8a) cause hash clustering → chain collisions → OpSparse slows to 107× worse than HSMU-SpGEMM. Binary search is unaffected by NZ pattern.
Q3How does the symbolic stage "know" the sorted columns before computing values?▾
It doesn't compute values — only positions. Symbolic stage builds a bitmask (or compressed mask) marking "there's a NZ here" without computing the actual NZ. Bitwise OR is fast; extract positions with __ffsll() → sorted column indices. However, this preprocessing has cost: Fig. 17 reports "Generate auxiliary mask" averages 26.40% of total time and "Generate sorted Ccol" averages 15.83%. The numeric stage then computes the values for those positions via binary search + atomic-add.
Q4Why does HSMU-SpGEMM switch to a dense accumulator at 4096 NZ?▾
Tuned empirically (paper Fig. 6). Below 4096, binary search (log N ≈ 12 steps) is faster than scanning a large direct array. Above 4096, the direct array wins: index by column (O(1)), then scan to extract NZs once. The threshold balances search cost vs scan cost. Different GPUs or matrix types might prefer different thresholds, but 4096 maximizes avg GFLOPS on their test set.
Q5Does HSMU-SpGEMM always beat hash-based methods?▾
No — it wins on average (3.2× geom mean speedup on 338 matrices, paper Table III) but loses on some scattered matrices (e.g., in-2004, paper Fig. 8b). Scattered NZs → hash tables have low collisions (fast), while HSMU-SpGEMM's symbolic stage has higher overhead (more distinct 64-bit mask units to OR). Trade-off: hash wins on ~10% of matrices, HSMU-SpGEMM wins on ~62% (paper Table IV).
Q6What's the overhead of format conversion (CSR → compressed mask)?▾
Paper Fig. 19 shows it's mostly under one-tenth the cost of a single SpGEMM on most matrices. The paper says this can be amortized in iterative solvers (AMG, Krylov) where you run many SpGEMMs on the same matrix structure. Even at about 1/10 per call, HSMU-SpGEMM's 3× numeric speedup makes the conversion overhead manageable in iterative workloads.
Q7Why doesn't cuSPARSE use a similar approach?▾
cuSPARSE v11.4 uses a hash-based accumulator (closed source, so details unclear). It's optimized for general-purpose use and backward compatibility. The paper shows cuSPARSE is 5–7× slower than HSMU-SpGEMM on avg (geom mean). Research prototypes like HSMU-SpGEMM push the frontier; commercial libraries follow later (with more testing/validation).
Q8Does HSMU-SpGEMM work for non-square matrices or A × B^T?▾
The evaluation uses 338 sparse square matrices from SuiteSparse and additionally A × A^T on eight non-symmetric matrices (Fig. 16). The method applies to sparse × sparse in general; the mask matrix adapts to the dimensions. Performance depends on sparsity pattern.
Q9How does HSMU-SpGEMM handle extremely large matrices that don't fit in GPU memory?▾
The paper's evaluation is on matrices that fit on the tested GPUs. The largest tested matrix is delaunay_n24 (16.8M rows, 100.7M NNZ, 633.9M intermediate products) — fits in 24 GB VRAM of RTX 3090 Ti. Out-of-core execution (doesn't fit) is not covered in the paper.
Q10What did the authors leave for future work?▾
The conclusion says future work is applying HSMU-SpGEMM in real-world scenarios such as accelerating deep neural network execution. Implicit extensions could include: 1) Auto-tuning the dense threshold (4096) per matrix or GPU architecture. 2) Handling FP32/FP16 (paper tests FP64 only). 3) Multi-GPU scaling (current is single-GPU). 4) Integration with higher-level libraries.

Glossary & further reading

Every term in one place, then a path to go deeper.

Core terms
HSMU-SpGEMMHigh Shared Memory Utilization SpGEMM — this paper's method. Uses sorted arrays + binary search instead of hash tables.
SpGEMMSparse General Matrix-Matrix Multiplication. C = A × B where all are sparse (most entries zero).
Shared memory utilizationRatio of used slots to allocated slots in GPU shared memory. Higher = less waste, more concurrent batches.
Hash collisionWhen two column indices map to the same hash table slot. Resolved by linear probing (slow, causes warp divergence).
Binary searchO(log N) lookup in a sorted array. Stable cost, zero collisions — HSMU-SpGEMM's core trick.
Bitmask / mask matrixAuxiliary structure where each bit = "is there a NZ here?" Used in symbolic stage to compute NZ positions via OR.
Symbolic stageFirst phase: determine where NZs are (positions, count) without computing values. Allocates memory.
Numeric stageSecond phase: compute actual NZ values via multiply-accumulate, using positions from symbolic stage.
Supporting terms
CSR (Compressed Sparse Row)Sparse matrix format: ptr, col, val arrays. Row i's NZs are col[ptr[i]:ptr[i+1]], val[...].
NZ / NNZNon-Zero element / Number of Non-Zeros in a matrix or row.
Intermediate productTemporary result aij × bjk during row-row multiply. Many products share the same column index (need accumulation).
Compression ratioRatio of intermediate products to final NZs. High ratio (10–100×) means heavy accumulation load.
SM (Streaming Multiprocessor)GPU's parallel execution unit. Has ~128 KB shared memory (RTX 3090 Ti), many CUDA cores.
WarpGroup of 32 GPU threads executing in SIMD lockstep. Divergence (different branches) serializes execution.
Warp divergenceWhen threads in a warp take different code paths (e.g., hit vs probe in hash table). Reduces parallelism.
Batch processingGroup rows of similar size, process in batches on SMs. Improves load balance and shared memory reuse.
BinningGrouping rows by NNZ into bins (e.g., 0–100, 100–1000, ...). Each bin uses appropriate resources (warp/block config).
Dense accumulatorDirect-indexed array (length = num columns). Fast for very dense rows but needs more memory.
Compressed maskCSR-like format for the bitmask (large matrices). Store only non-zero 64-bit units, indexed by tilePtr/tileCol.
__popcll() / __ffsll()CUDA intrinsics: count bits / find first set bit in 64-bit int. Used to extract NNZ and sorted columns from mask.

Related work

Gustavson's Algorithm (1978) foundational

ACM Trans. Math. Software · row-row SpGEMM

The classic row-by-row method HSMU-SpGEMM builds on. Introduced the accumulator concept — this paper improves it with sorted arrays.

Nsparse (ICPP'17) baseline

Nagasaka et al. · hash 1×, binning strategy

High collision rate (400% on cant) but good binning. HSMU-SpGEMM reuses the binning, replaces the hash accumulator.

spECK (PPoPP'20) baseline

Parger et al. · hash capped at 66% fill, hybrid with dense

First to combine hash + dense accumulators. HSMU-SpGEMM adopts the dense part, replaces hash with binary search for better utilization.

OpSparse (IEEE Access'22) baseline

Du et al. · hash roughly 2×, global mem for long rows

Hash-based utilization averages about 35%. HSMU-SpGEMM's exact-size arrays win on avg, lower peak memory for long rows.

cuSPARSE v11.4 (NVIDIA) commercial baseline

Closed source · cusparseSpGEMM

NVIDIA's official library. HSMU-SpGEMM is 5–7× faster (geom mean) — shows research prototypes can exceed commercial state-of-the-art.

bhSPARSE (IPDPS'14) non-hash baseline

Liu & Vinter · merge-based + ESC

Different approach (merge rows, sort, compress). HSMU-SpGEMM's numeric stage is faster (paper Fig. 14) but symbolic overhead is higher — trade-off.
