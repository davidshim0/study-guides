When thirty-two walkers can't keep up with thousands of threads

GPUs running irregular workloads spend most of their page-walk time waiting in line for a free hardware walker. SoftWalker fixes it by launching the walks as software threads, leveraging the GPU's massive parallelism to reduce queueing delay.

📄 "SoftWalker" · Jang et al., MICRO 2025 🎓 GPU virtual memory + TLB basics assumed ⏱ ~25 min read
🚶

When TLB misses spike, don't build more fixed hardware walkers — launch dynamic software threads on idle cores to walk in parallel, repurposing stall cycles into productive translation work.

Before you begin

Seven GPU translation concepts SoftWalker builds on

Each card clears up one foundation. Read in order the first time, or jump to whatever you need.

1GPU virtual memory and the TLB hierarchy▾

Modern GPUs use virtual memory to abstract physical addresses and enable features like unified memory and demand paging. A TLB (Translation Lookaside Buffer) caches recent virtual-to-physical mappings. GPUs have a two-level hierarchy: each SM has a private L1 TLB, and all SMs share a large L2 TLB (1024 entries in the paper's config, covering ~64 MB with 64 KB pages).

On an L2 TLB miss, the GPU must perform a page table walk — traversing multiple levels of in-memory page tables to find the mapping — which is slow (~400 cycles in this setup).

Now you're ready: SoftWalker targets the bottleneck of handling those L2 TLB misses, which are frequent for irregular workloads.
2Page table walks and hardware Page Table Walkers (PTWs)▾

A page table walk is the process of translating a virtual page number (VPN) into a physical page number (PPN) by reading a multi-level radix page table. For a four-level table, the walker reads the page directory at each level, using bits of the VPN as an offset.

Modern GPUs have a fixed pool of hardware PTWs (32 in the baseline) that handle these walks in dedicated logic. When TLB misses exceed the number of walkers, requests queue in a Page Walk Buffer (PWB) until a walker is free.

Now you're ready: The contention when 32 walkers face thousands of concurrent misses is the core problem SoftWalker solves.
3Irregular GPU workloads and why TLB pressure spikes▾

Regular workloads (dense matrix ops, convolutions) access memory in a coalesced, sequential pattern — threads in a warp touch the same page, so TLB hit rates are high. Irregular workloads (graph processing, sparse linear algebra, hash lookups) scatter accesses across a wide address range with poor spatial locality.

The paper shows that for irregular apps like bfs and nw, each warp instruction can trigger up to 32 distinct TLB lookups, easily exceeding L2 TLB capacity (1024 entries) and causing frequent misses.

Now you're ready: These workloads generate hundreds of concurrent page walk requests, saturating the 32 hardware walkers and creating a queue.
4Queueing delay — the dominant component of walk latency▾

The paper defines page walk latency as the sum of two parts: 1) queueing delay (time waiting in the PWB for a free walker) and 2) page table access latency (the actual memory reads to traverse the page table). For irregular workloads on the baseline, the paper shows that queueing delay is the dominant fraction of total walk latency — the walkers themselves are fast, but there aren't enough of them.

Doubling PTWs to 64 cuts queueing delay in half; going to 1024 PTWs nearly eliminates it (but at huge area/power cost).

Now you're ready: SoftWalker's goal is to reduce queueing delay by making the number of concurrent walkers scale with demand.
5MSHRs — tracking outstanding TLB misses▾

A Miss Status Holding Register (MSHR) is a small buffer that tracks in-flight requests. Each L2 TLB MSHR entry holds metadata for a pending miss: which SM/warp/thread caused it, and what VPN is being translated. The baseline has 128 L2 TLB MSHRs, each supporting up to 46 merges (coalescing requests for the same page).

When MSHRs fill, new TLB misses fail — they stall the pipeline until an MSHR frees. This is a bottleneck independent of the PTW count.

Now you're ready: SoftWalker introduces In-TLB MSHRs to expand MSHR capacity by repurposing idle TLB entries.
6GPU thread-level parallelism and warp scheduling▾

A GPU SM can interleave thousands of threads with minimal context-switch overhead (as low as one cycle). A warp is a group of 32 threads that execute in lockstep. The warp scheduler picks a ready warp each cycle; if none are ready (e.g., all waiting on memory), the pipeline stalls.

For irregular workloads, the paper shows ~90% of warp scheduler cycles are stalled on memory/scoreboard — the SM is mostly idle, waiting for data or for address translation to complete.

Now you're ready: SoftWalker leverages these idle cycles by launching dedicated page-walk threads (PW Warps) that run on the same SM pipeline.
7Why scaling hardware PTWs is expensive▾

A naive solution is to add more hardware PTWs. But the PWB is a Content-Addressable Memory (CAM) structure (to support associative lookup and merging), which has super-linear area growth. Prior work [50] shows 192 PTWs with 18 PWB ports consume 3.9% of total GPU area.

The paper's analysis (Figure 15) shows that for the same area budget, SoftWalker delivers ~2.6× speedup, while 128 hardware PTWs reach only ~2.1× — software is more area-efficient.

Now you're ready: SoftWalker's software approach avoids the CAM cost by using lightweight software-managed buffers in shared memory.
The whole story in one place

SoftWalker in a nutshell

The bottleneck, the fix, and how thousands of concurrent walks reduce the queue.

GPUs with irregular workloads generate hundreds of concurrent TLB misses. With only 32 hardware Page Table Walkers (PTWs), most misses queue in the Page Walk Buffer (PWB), and queueing delay dominates page walk latency. Scaling to 1024 PTWs would fix it but costs ~64× more area (mostly in the CAM-based PWB).

SoftWalker's insight: the GPU itself has thousands of threads and 90% idle cycles during memory stalls. Instead of fixed hardware walkers, launch lightweight software threads (PW Warps) on the SMs to perform page table walks. Each SM gets a dedicated PW Warp slot (isolated from user warps) and a small software-managed buffer (SoftPWB) in shared memory. When an L2 TLB miss occurs, a Request Distributor forwards it to an idle SM; the SoftWalker Controller on that SM fills the request into SoftPWB and launches the PW Warp, which executes a simple loop: load VPN, compute page table offsets, issue memory reads (new LDPT instruction), and fill the L2 TLB (FL2T instruction).

Because the GPU has 46 SMs and can launch ~32 PW Warp threads per SM, SoftWalker supports up to 1472 concurrent walkers — far exceeding the 32-PTW baseline. This reduces queueing delay. The trade-off is a modest increase in per-walk latency (~2× vs. ideal hardware, due to instruction execution and SM↔L2 TLB communication), but that's smaller compared to the queueing delay it avoids.

To prevent MSHR saturation (the second bottleneck), SoftWalker introduces In-TLB MSHRs: when the 128 dedicated MSHR entries fill, the L2 TLB repurposes idle TLB entries themselves to hold pending miss metadata (via a pending bit). This expands MSHR capacity to ~1024 on demand, allowing more walks to proceed in parallel.

The payoff: on 20 workloads, SoftWalker reduces page walk latency by 72.8% on average and achieves 2.24× speedup (3.94× for irregular apps), with under 1% area overhead (mainly SoftWalker Controller and In-TLB MSHR logic) and nothing on the critical path.

🍯 Key takeaways

  • For irregular workloads, queueing delay dominates page walk latency, not the walk itself.
  • SoftWalker shifts page walks from 32 fixed hardware PTWs to thousands of dynamic software threads (PW Warps).
  • Each SM gets a dedicated PW Warp slot (isolated, high-priority) and a software-managed SoftPWB buffer.
  • New ISA: LDPT (load page table), FL2T (fill L2 TLB), FPWC (fill page walk cache), FFB (fault buffer).
  • In-TLB MSHRs repurpose idle TLB entries as MSHR slots, expanding capacity from 128 to ~1024 on demand.
  • Result: 72.8% lower walk latency, 2.24× speedup (3.94× irregular), under 1% area, no critical-path impact.
  • A hybrid mode retains hardware PTWs for regular workloads; software walkers activate only under high pressure.
Seven everyday pictures

How to think about SoftWalker

Each picture stands alone. Every card ends with where the analogy stops being true.

The core problem · Queueing delay

Two checkout lanes, fifty shoppers

A store with only two cashiers (hardware PTWs) serves fifty customers (TLB misses). The actual checkout is fast (page table access ~400 cycles), but each shopper waits in line (queueing delay). Opening 1000 lanes would fix it, but the store can't afford the space. SoftWalker is like training all the waiting shoppers to check themselves out using mobile scanners — the line shrinks, and everyone's done faster.

Where it breaks: real shoppers can't suddenly become cashiers; PW Warps are pre-trained threads that exist from kernel launch, not ad-hoc conversions.
Leveraging idle resources · Thread-level parallelism

A factory floor with idle machines

A GPU SM is like a factory floor with dozens of machines (execution units). For irregular workloads, 90% of the time those machines are idle, waiting for deliveries (memory latency). Instead of letting them sit, SoftWalker assigns one machine per floor (PW Warp) to run a simple translation task during downtime. The floor stays productive, and the specialized task doesn't interfere with the main assembly line (user warps).

Where it breaks: a real factory ramp-up has overhead; GPU context switches are <1 cycle, so PW Warps activate instantly.
Software-managed buffer · SoftPWB

A whiteboard for task handoffs

Instead of a fixed hardware buffer (CAM-based PWB), SoftWalker uses a small region of shared memory (SoftPWB, 96 bits per entry) as a whiteboard. When the L2 TLB has a miss, it writes the task (VPN, page table base) on the board; the PW Warp reads it, completes the walk, and erases it. The board is cheap (just memory), scales trivially (expand by allocating more shared memory), and avoids the CAM's super-linear area cost.

Where it breaks: a real whiteboard has no access control; SoftPWB is in the PW Warp's private shared memory space, isolated by GPU hardware.
Expanding MSHR capacity · In-TLB MSHR

Parking spots that double as waiting areas

A parking lot (L2 TLB) has 1024 spots. Normally 128 are reserved as "valet holding" (dedicated MSHRs) for cars waiting to be parked. But for irregular workloads, many regular spots sit empty (2.4% L2 TLB hit rate). In-TLB MSHR lets the valet use empty regular spots as temporary holding when the 128 dedicated spots fill. A single bit per spot (pending bit) marks it "waiting, not parked," expanding capacity to ~1024 on demand without building new structures.

Where it breaks: real parking spots can't hold metadata; In-TLB MSHR entries store VPN + requesting SM/warp, not actual translation data.
Isolation · PW Warp vs. user warps

A VIP lane at airport security

Regular travelers (user warps) queue in the main lanes; flight crew (PW Warps) have a dedicated VIP lane with its own scanners (dedicated warp slot, instruction buffer, scoreboard). The crew doesn't consume a passenger lane, and passengers can't access the crew lane. Similarly, PW Warps are structurally isolated — they don't reduce SM occupancy for user kernels, and user warps can't access the privileged page table data in PW Warp registers or SoftPWB.

Where it breaks: real VIP lanes might share scanners; PW Warps have dedicated architectural resources (1470 bits per SM).
Software walk routine · Page table traversal

Following a GPS route step-by-step

A radix page table is like nested address books: the VPN gives you a page number in book 1, which points to a page in book 2, and so on for four levels. The PW Warp's code (Figure 14) is the GPS route: loop over levels, compute offset from VPN bits, issue LDPT (load the next book entry), check for invalid (FFB for page fault), update the shortcut cache (FPWC), and finally install the destination (FL2T). It's a simple loop, compiled to ~16 registers and ~30 SASS instructions.

Where it breaks: a GPS recalculates on detours; PW Warps follow a fixed path (PWC lookups happen at the Request Distributor, not in the warp).
Hybrid mode · Best of both worlds

Auto + manual transmission

Some drivers prefer manual control for precision (regular workloads with low TLB pressure); others need auto for heavy traffic (irregular workloads). SoftWalker's hybrid mode keeps the 32 hardware PTWs (auto) and activates software walkers (manual) only when the hardware queue fills. The Request Distributor routes to hardware first; if all are busy, it switches to software. Regular apps see minimal slowdown, irregular apps get massive parallelism.

Where it breaks: real transmissions don't switch mid-drive; the hybrid uses both simultaneously, with a counter to track which is handling each miss.
The full explanation

How it actually works

Mechanism by mechanism, with the paper's key numbers and architectural details.

1 · The baseline architecture and its bottleneck

The baseline GPU (modeled after NVIDIA RTX 3070) has 46 SMs, each with a private L1 TLB (32 entries, 64 KB pages). All SMs share an L2 TLB (1024 entries, 16-way) with 128 MSHRs (each supporting 46 merges). On an L2 TLB miss, the request is buffered in a Page Walk Buffer (PWB) and assigned to one of 32 hardware PTWs, which traverse a four-level radix page table. Each level uses 9 bits of the VPN as an offset (bits 48-40, 39-31, 30-22, 21-13), with the final 16 bits (12-0 for 64 KB pages, plus the page-table index bits) completing the 49-bit virtual address. Each walk takes ~400 cycles of memory access (including L2 cache hits for recently-accessed page table pages).

For irregular workloads, the paper shows the L2 TLB MPKI ranges from 4.8 (st2d) to 2517 (spmv), generating hundreds of concurrent misses. With only 32 PTWs, the PWB saturates, and queueing delay dominates: the paper shows that queueing delay is the dominant fraction of total page walk latency for irregular workloads.

🌱
Concretely: for bfs, L2 TLB MPKI is 22.2. With ~48 warps/SM × 46 SMs, that's many L2 TLB misses. At any moment, many are pending. The 32 PTWs can handle 32 walks in parallel, so the rest queue. SoftWalker's 1472 walkers process more misses in parallel, cutting queueing delay substantially.

2 · PW Warp — a dedicated thread for page walks

Each SM allocates one Page Walk Warp (PW Warp): a special warp slot isolated from user warps. It has its own instruction buffer, scoreboard entry, and SIMT stack entries (total 1470 bits per SM), plus 16 registers from the underutilized register file. The PW Warp is launched at kernel start by the host driver, which preloads the page walk code into device memory. The warp enters a wait loop, sleeping until the SoftWalker Controller signals a new request.

The PW Warp runs at the highest warp priority — address translation is on the critical path, so delaying it stalls the entire pipeline. Because it's structurally separate, it doesn't consume a user warp slot or reduce SM occupancy. Security is maintained via GPU hardware: the PW Warp's registers and SoftPWB (in shared memory) are scoped to its thread block, invisible to user warps.

💡
Why not reuse user warp slots? That would reduce parallelism (fewer user warps per SM). Allocating a dedicated slot adds minimal overhead (1470 bits + 16 regs) and ensures page walks don't compete with application threads.

3 · ISA extensions — four new instructions

To enable PW Warps to walk page tables, the paper extends GPU ISA with four instructions:

InstructionOperationWhy it's needed
LDPT dst, base, offLoad PTE from page table at base + offset, bypassing TLBNormal loads go through TLB; walkers must access physical addresses directly
FL2T vpn, pteFill L2 TLB entry with VPN→PTE mappingWarp must install the final translation and resolve the MSHR
FPWC vpn, lv, pteFill Page Walk Cache with intermediate PTE at level lvCache intermediate levels to skip walks for nearby VPNs
FFB vpn, lvStore invalid PTE in Fault Buffer for page fault handlerSignal OS when a page is not present

These instructions execute through the standard SM pipeline. LDPT issues a memory read like a normal load but uses a physical address; FL2T sends a message to the L2 TLB controller to resolve the MSHR and wake waiting threads.

4 · Request management — SoftPWB and SoftWalker Controller

Each SM has a SoftPWB: a 32-entry buffer in shared memory (96 bits per entry: 33-bit VPN, 31-bit page table base PFN, 2-bit current level). When the Request Distributor forwards a miss to this SM, the SoftWalker Controller (a small FSM) fills the next empty SoftPWB slot and updates a 2-bit SoftPWB Status Bitmap (64 bits total): 00 = invalid, 01 = valid (ready to walk), 10 = processing (walk in progress). When a valid entry exists and the PW Warp is idle, the controller signals the warp scheduler to issue the PW Warp, which loads the request from SoftPWB and begins the walk.

The Request Distributor sits at the L2 TLB and maintains a per-SM counter. When an L2 TLB miss occurs, it selects the SM with the lowest counter (round-robin), increments that counter, and forwards the request. When the SM's PW Warp completes (sends FL2T), the counter decrements. This prevents overflowing any single SM's SoftPWB.

🍯
Why shared memory for SoftPWB? It's software-accessible (PW Warp can read via standard load) and cheap. A CAM-based PWB for 32×46=1472 entries would be huge; repurposing shared memory costs nothing (shared memory is already there and underutilized).

5 · In-TLB MSHR — expanding MSHR capacity on demand

The 128 dedicated L2 TLB MSHRs become a bottleneck when thousands of PW Warps are active. The paper adds a pending bit to each L2 TLB entry (1 bit per entry, 1024 bits total). When MSHRs fill and a new miss arrives, the TLB controller selects a victim entry (LRU) and sets its pending bit, converting it into an MSHR slot. The entry now holds miss metadata (SM ID, warp ID) instead of a valid translation. When the walk completes (FL2T), the controller clears the pending bit of all matching ways (tag match) in that set, fills the PTE into one way, and resolves all coalesced misses.

This design allows up to 1024 concurrent outstanding misses (the full TLB capacity) without expanding the dedicated MSHR structure. For irregular workloads with low L2 TLB hit rates (2.4% on average), the TLB entries are mostly idle anyway, so repurposing them costs nothing.

⚠
Trade-off: If L2 TLB hit rate were high, In-TLB MSHR would evict useful translations. But irregular workloads thrash the TLB, so the entries are underutilized. For regular workloads, the hybrid mode keeps hardware PTWs and doesn't fill MSHRs, so In-TLB MSHR never activates.

6 · The walk routine — code sequence in Figure 14

The PW Warp executes the following loop (simplified; full SASS code in the paper):

// Load request from SoftPWB (indexed by threadIdx)
int idx = threadIdx.x;
uint128_t req = softPWB[idx];       // VPN | pt_base | level
uint64_t vpn = (req >> 64) & 0x1FFFFFFFFFF;
uint64_t pt_base = req & 0xFFFFFFFFFFF;
int level = (req >> 60) & 0x3;

// Walk each level (starting from PWC hit level)
while (level < 4) {
    int offset = (vpn >> ((3-level)*9)) & 0x1FF;  // 9-bit offset
    uint64_t pte = __load_page_table(pt_base, offset);  // LDPT
    if (__page_fault(pte)) {
        __fill_fault_buffer(vpn, level);              // FFB
        return;
    }
    __fill_page_walk_cache(vpn, level, pte);          // FPWC
    pt_base = pte;
    level++;
}
__fill_l2_tlb(vpn, pt_base);                          // FL2T

Each thread in the PW Warp handles one miss. The warp is launched with a mask indicating which threads are active (corresponding to valid SoftPWB entries). Threads that finish early (e.g., page fault) mark their SoftPWB slot invalid and exit. The warp completes when all active threads finish.

Follow the numbers, step by step

Worked examples

Four traces with the paper's real numbers, walking through the mechanisms.

① Baseline queueing delay calculation

For irregular workloads, queueing delay dominates page walk latency on the baseline. Estimate how the baseline's 32 PTWs create this bottleneck.

  1. Concurrency: with 46 SMs × 48 warps/SM × 32 threads/warp = ~70,000 threads, many misses are in-flight simultaneously. For irregular workloads, the paper shows many concurrent misses.
  2. PTW throughput: 32 walkers handle 32 misses in parallel. Each walk takes ~400 cycles. So the system processes 32 misses every 400 cycles.
  3. Queue forms: with many misses arriving and only 32 processed simultaneously, the queue saturates. The paper's Figure 18 shows queueing delay is the dominant component of total walk latency for irregular workloads.
  4. Total walk latency: queueing delay plus walk time. The paper shows 72.8% average latency reduction with SoftWalker, indicating substantial baseline queueing delay.
🍯
Insight: the 32 PTWs are fast but overwhelmed. SoftWalker's 1472 walkers process more concurrent misses in parallel, cutting queueing delay substantially.

② SoftWalker latency breakdown

For irregular workloads with SoftWalker, estimate the total page walk latency, accounting for communication and instruction overhead.

  1. Page table access latency: still ~400 cycles (4 levels × ~100 cycles/level, with L2 cache hits on page table pages).
  2. SM↔L2 TLB communication: Request Distributor forwards the miss to the SM (one L2 TLB access latency). PW Warp sends FL2T back to L2 TLB. Communication overhead exists.
  3. Instruction execution: the PW Warp loop has instructions per level (offset computation, LDPT, FPWC, conditional checks). Execution adds overhead.
  4. Total per-walk latency: page table access + communication + execution overhead.
  5. Queueing delay: with 1472 walkers and many concurrent misses, more misses are serviced immediately. Queueing delay reduces substantially.
  6. Latency reduction: the paper's Figure 18 shows a 72.8% average latency reduction.
💡
The communication and execution overhead is smaller compared to the queueing delay it eliminates. That's the key trade-off: slightly longer per-walk time, much shorter total time.

③ In-TLB MSHR capacity calculation

The baseline has 128 dedicated L2 TLB MSHRs, each supporting 46 merges. In-TLB MSHR can use up to 1024 TLB entries. For spmv (L2 TLB MPKI = 2517), estimate the capacity expansion.

  1. Baseline MSHR capacity: 128 MSHR entries × 46 merges = 5,888 concurrent misses before MSHR failure.
  2. spmv miss rate: High L2 TLB misses. The baseline MSHRs can saturate.
  3. With In-TLB MSHR: up to 1024 entries. Assuming the same 46 merges/entry, capacity = 1024 × 46 = 47,104 concurrent misses.
  4. Effective expansion: 47,104 ÷ 5,888 = 8× capacity. The paper's Figure 17 shows In-TLB MSHR eliminates 95.3% of MSHR failures on average.
  5. Cost: 1 bit per TLB entry = 1024 bits = 128 bytes. Plus control logic (0.0061 mm² in 28nm). Negligible compared to the TLB itself (~1 kB).
🍯
Insight: for irregular workloads with low L2 TLB hit rates (2.4%), the TLB entries are idle. Repurposing them as MSHRs is nearly free and unlocks huge MSHR capacity.

④ Area overhead calculation

The paper claims under 1% area overhead for a 46-SM GPU. Verify this using the paper's numbers.

  1. SoftWalker Controller per SM: 64 bits (SoftPWB Status Bitmap) + 64 bits (instruction buffer) + 126 bits (scoreboard) + 8×160 bits (SIMT stack) = 1,470 bits per SM. Total: 46 SM × 1,470 = 67,620 bits = 8.45 kB.
  2. In-TLB MSHR: 1 bit per TLB entry × 1024 entries = 1,024 bits = 128 bytes. Plus control logic: 0.0061 mm² (28nm). For a 7nm chip, scale by (28/7)² ≈ 16×, so ~0.0004 mm². Negligible.
  3. PW Warp registers: 16 registers × 32 bits × 46 SMs = 23,552 bits = 2.94 kB. This is carved from the existing register file (typically several MB), so it's repurposing, not new area.
  4. Total new area: 8.45 kB (controller) + 0.13 kB (In-TLB MSHR bits) ≈ 8.6 kB of storage, plus 0.0004 mm² of logic. The paper's baseline GPU (RTX 3070, GA102) has a total area of 628.4 mm² (7nm). The controller logic is likely <0.01 mm², so total overhead is <0.1 mm² / 628.4 mm² = 0.016% area.
  5. Including SoftPWB: 32 entries × 96 bits × 46 SMs = 140,544 bits = 17.6 kB. This is in shared memory (128 kB per SM, total ~6 MB). It's repurposing, not new area. If we count it as "overhead" (unusable by user kernels), 17.6 kB out of ~6 MB shared memory is a small fraction.
💡
The "under 1%" claim is conservative. The actual new silicon (controller + In-TLB MSHR logic) is <0.02% of chip area — essentially free.

Practice problems

Try each before opening the answer. Every one includes the misconception it checks.

Problem 1 · easy
Why does SoftWalker use dedicated PW Warp slots instead of reusing existing user warp contexts (like CABA's assist warps)?
Show answer
Reusing user warp slots would reduce SM occupancy (fewer warps available for the application kernel). For latency-sensitive page walks, this would degrade performance. Dedicated PW Warp slots (1470 bits per SM) add minimal overhead and ensure the PW Warp doesn't compete with user warps for scheduling. Additionally, PW Warps need privileged access to page table data (physical addresses, TLB control), which can't be safely shared with user warps.Misconception: that assist warps and PW Warps are the same. Assist warps (CABA) run compression/decompression code that's not privileged and can share user contexts. PW Warps access physical memory and must be isolated.
Problem 2 · medium
The paper shows that queueing delay is the dominant fraction of total page walk latency for irregular workloads. Show that this ratio would drop if the page table access latency were 10× higher (4000 cycles instead of 400).
Show answer
If queueing delay is D and walk latency is W=400, and D dominates (D >> W), total is approximately D+W. If walk latency rises to W'=4000, total becomes D+W'. The queueing delay D remains similar (same 32 PTWs, same arrival rate), but now W' is much larger. So the fraction D/(D+W') drops. For example, if D=2100 and W=400, D/(D+W)=84%. With W'=4000, D/(D+W')=2100/6100=34%. The ratio drops because the walk itself becomes a larger component. However, SoftWalker would still help by parallelizing the longer walks, reducing total time.Misconception: that queueing delay is always the bottleneck. The paper's setup has fast walks (~400 cycles) and slower queues. If walks were slower, the queue would matter less — but SoftWalker's parallelism would still eliminate contention.
Problem 3 · medium
Why does In-TLB MSHR work well for irregular workloads but not for regular ones?
Show answer
Irregular workloads have low L2 TLB hit rates (~2.4% on average), so most TLB entries are invalid or rarely used. Repurposing them as MSHR slots (via the pending bit) costs nothing. Regular workloads have high L2 TLB hit rates (>90%), so the TLB entries are actively holding valid translations. Converting them to MSHR slots would evict useful mappings and degrade performance. The hybrid mode keeps hardware PTWs for regular workloads, so In-TLB MSHR never activates (MSHRs don't fill).Misconception: that In-TLB MSHR is always beneficial. It's a trade-off: if the TLB is underutilized (irregular), repurposing is free; if the TLB is active (regular), it hurts.
Problem 4 · hard
The paper shows SoftWalker achieves 2.24× speedup on average, but the ideal PTW config (1024 PTWs + 1024 MSHRs) achieves 2.58×. Why doesn't SoftWalker match the ideal, even though it has more walkers (1472)?
Show answer
SoftWalker's per-walk latency is higher than ideal hardware (due to communication and instruction execution). For some workloads, this overhead partially offsets the queue savings. Additionally, the Request Distributor's round-robin policy may not perfectly balance load across SMs (some SMs may have more pending walks than others, creating mini-queues). The ideal hardware config has zero queueing delay and the lowest per-walk latency. SoftWalker trades slightly higher per-walk latency for massive parallelism, which is a net win but not perfectly optimal. The gap (2.58× vs 2.24×) is small enough that SoftWalker is area-efficient (achieves 87% of ideal performance at <1% of ideal area).Misconception: that more walkers = always better. The per-walk latency matters too. SoftWalker's software approach has unavoidable overhead (instruction execution), so it can't perfectly match ideal hardware, but it's close enough to be practical.
Problem 5 · medium
Why does the paper use shared memory for SoftPWB instead of a dedicated SRAM buffer?
Show answer
Shared memory is already on-chip (128 kB per SM, total ~6 MB) and underutilized by many irregular workloads (which have low data reuse and don't need much shared memory). Repurposing a small slice (32 entries × 96 bits = 384 bytes per SM) is nearly free. A dedicated SRAM buffer would require new area (46 SMs × 384 bytes = 17.6 kB), plus the control logic to access it. Shared memory is software-accessible via standard load/store, so the PW Warp can read SoftPWB entries using existing instructions. The trade-off is that user kernels lose ~0.3% of shared memory, but the paper argues this is negligible for irregular apps.Misconception: that repurposing shared memory reduces user kernel performance. For irregular workloads, shared memory is already underutilized (most parallelism comes from threads, not shared data). The 384-byte SoftPWB is a tiny fraction of the 128 kB per SM.
Problem 6 · hard
The hybrid mode keeps 32 hardware PTWs and activates software walkers only when hardware is saturated. Construct a scenario where the hybrid performs worse than pure SoftWalker.
Show answer
Scenario: a workload with bursty TLB misses — long periods of low pressure (few misses) followed by sudden spikes (hundreds of concurrent misses). During the low-pressure phase, the hybrid uses hardware PTWs (fast, low latency). When the spike hits, hardware PTWs saturate, and the Request Distributor switches to software. But the software walkers take time to launch (PW Warp must be scheduled, SoftPWB must be filled), so there's a brief period where misses queue even though software walkers are available but not yet active. In contrast, pure SoftWalker always uses software, so the walkers are pre-launched and ready. The hybrid's latency would be slightly higher during the transition. However, the paper's evaluation shows the hybrid matches pure SoftWalker for irregular apps (because the spikes are sustained, not bursty), and outperforms it for regular apps (avoiding software overhead). So this scenario is theoretical.Misconception: that the hybrid is strictly better. It's a design trade-off: the hybrid optimizes for the common case (regular apps use hardware, irregular apps saturate and switch to software), but has a potential weakness on bursty workloads with rapid pressure changes.
Problem 7 · easy
Why does SoftWalker require new ISA instructions (LDPT, FL2T, etc.) instead of using existing GPU load/store?
Show answer
Normal GPU load/store goes through the TLB — it translates virtual addresses to physical. But the PW Warp is *doing* the translation, so it must access the page table using physical addresses, bypassing the TLB. LDPT is a privileged load that uses a physical address directly. FL2T is a special store that doesn't write to memory but instead sends a message to the L2 TLB controller to install a translation. These operations can't be done with standard instructions.Misconception: that page walks are just memory reads. They are, but they must bypass the TLB (to avoid infinite recursion) and update TLB state (which normal stores can't do). The ISA extensions give PW Warps the necessary low-level control.
Problem 8 · medium
The paper shows that some regular apps (2dc, histo, red, scan) slow down by up to 10.9% with pure SoftWalker. Why, and how does the hybrid fix it?
Show answer
Regular apps have low TLB pressure (L2 TLB MPKI < 1), so queueing delay is negligible. The baseline's 32 hardware PTWs are sufficient. SoftWalker introduces communication overhead (SM↔L2 TLB) and instruction execution overhead, which is pure waste when there's no queue. The overhead on a small number of walks can translate to overall slowdown. The hybrid mode routes misses to hardware PTWs first; only when all 32 are busy (rare for regular apps) does it use software. So regular apps see hardware-only performance (minimal slowdown), and irregular apps see software-only performance (large speedup). The hybrid's Request Distributor tracks hardware PTW availability via a counter and switches accordingly.Misconception: that SoftWalker is always faster. It's faster when queueing delay dominates, but slower when per-walk latency matters. The hybrid adapts to the workload's pressure level.
Quick answers

Frequently asked questions

Twelve questions newcomers ask most. Tap any to expand.

Q1What exactly is a "page walk" in a GPU?▾
It's the process of translating a virtual page number (VPN) to a physical page number (PPN) by reading a multi-level radix page table in memory. GPUs typically use a four-level table (like x86-64): each level uses 9 bits of the VPN as an index into a 512-entry table, which points to the next level's base address. The final level gives the physical page frame number. A hardware Page Table Walker (PTW) or software PW Warp performs this traversal, issuing 4 memory reads (one per level) to get the final translation.
Q2Why do irregular workloads have such high TLB miss rates?▾
Irregular apps (graph processing, sparse linear algebra, hash lookups) access memory in a scattered, non-sequential pattern with poor spatial locality. Each warp instruction can touch up to 32 distinct pages (one per thread), easily exceeding the L2 TLB's 1024-entry capacity (which covers only ~64 MB with 64 KB pages). For example, breadth-first search (bfs) randomly accesses graph nodes spread across gigabytes of memory, causing L2 TLB MPKI of 22.2. Regular apps (dense matrix ops) access contiguous pages, so coalescing keeps TLB hit rates high.
Q3How does SoftWalker avoid interfering with user kernel performance?▾
PW Warps are structurally isolated: they have dedicated architectural slots (instruction buffer, scoreboard, SIMT stack) that don't consume user warp contexts. The 16 registers per PW Warp are carved from the underutilized register file (most GPU apps don't max out registers). SoftPWB is in shared memory, which irregular apps underutilize. PW Warps run at highest priority because address translation is on the critical path — delaying it stalls the whole pipeline. The total overhead is <1% of chip area and ~0.3% of shared memory, negligible for irregular workloads.
Q4What happens if a PW Warp encounters a page fault (invalid PTE)?▾
The PW Warp executes the FFB (Fill Fault Buffer) instruction, which stores the VPN and page table level in a small fault buffer (similar to existing GPU page fault handling). The OS's page fault handler reads the fault buffer, allocates a physical page, updates the page table, and signals the GPU to retry the translation. From the OS's perspective, this is identical to a hardware PTW-generated fault — the mechanism is unchanged. The paper notes compatibility with unified virtual memory (UVM) systems.
Q5Why does scaling hardware PTWs have super-linear area cost?▾
The Page Walk Buffer (PWB) is a Content-Addressable Memory (CAM) that supports associative lookup (to merge requests for the same VPN) and multi-port access (so all PTWs can read simultaneously). CAM area grows as entries × ports², and power grows similarly. Doubling PTWs from 32 to 64 requires doubling PWB entries and likely doubling ports (from 2 to 4), so area increases by 2 × 2² = 8×. Prior work shows 192 PTWs with 18 ports consume 3.9% of chip area — a significant cost. SoftWalker's SoftPWB is just SRAM (shared memory), with linear area scaling.
Q6How does the Request Distributor decide which SM gets a miss?▾
It maintains a per-SM counter (initially 0). When a miss occurs, it picks the SM with the lowest counter (round-robin tie-break), increments that counter, and forwards the request. When the SM's PW Warp completes (sends FL2T), the counter decrements. This prevents overflowing any single SM's SoftPWB (which has 32 entries). The paper evaluates alternative policies (random, stall-aware) and finds minimal performance difference, so it uses simple round-robin for low overhead.
Q7Why does In-TLB MSHR use a pending bit instead of a separate flag?▾
Each L2 TLB entry has a valid bit (1 = holds a translation, 0 = empty). Adding a pending bit creates three states: valid=1, pending=0 → normal translation; valid=0, pending=1 → acting as MSHR; valid=0, pending=0 → invalid. This encoding reuses the existing valid bit, so the overhead is just 1 bit per entry (1024 bits total). A separate flag would double the overhead. The controller checks the pending bit to distinguish TLB entries from MSHR entries.
Q8Can SoftWalker handle large pages (2 MB) or huge pages (1 GB)?▾
Yes. Large pages reduce TLB pressure by covering more memory per entry (32× more than 64 KB pages for 2 MB pages), so the baseline's 32 PTWs become sufficient for more workloads. The paper evaluates large pages (Figure 25) and shows that even with 2 MB pages, some irregular apps (xsb, spmv, gups) still benefit from SoftWalker (5.1×, 4.5×, 7.0× speedup). The PW Warp code adjusts by using fewer page table levels (e.g., 2-level for 2 MB pages instead of 4-level). SoftWalker is complementary to large pages.
Q9What's the energy cost of SoftWalker vs. hardware PTWs?▾
The paper doesn't measure energy. Hardware PTWs are fixed-function logic (low power per walk, but many idle during low pressure). SoftWalker uses SM pipelines (higher power per walk due to instruction execution), but only activates PW Warps when needed. For irregular workloads with sustained high pressure, SoftWalker's walkers are constantly active (similar to hardware), but the elimination of queueing delay reduces overall execution time. For regular workloads, the hybrid keeps hardware PTWs, so energy is unchanged.
Q10How does SoftWalker handle security (e.g., can user warps read page tables)?▾
No. PW Warps are isolated: their registers and SoftPWB (in shared memory) are scoped to the PW Warp's thread block, invisible to user warps. GPU hardware enforces strict memory protection — user warps can't access another block's shared memory or registers. The ISA extensions (LDPT, FL2T, etc.) execute through the standard SM pipeline. The paper notes that the PW Warp follows cleanup guidelines to prevent register leakage, and mentions compatibility with register file uninitialization fixes (Pustelnik et al., 2024), ensuring stale PTE data doesn't leak across kernel invocations.
Q11What are the failure modes? When does SoftWalker perform worse?▾
1) Regular workloads with low TLB pressure: SoftWalker's communication overhead is pure waste. The hybrid fixes this by using hardware PTWs. 2) Workloads with very high L2 TLB hit rates: In-TLB MSHR would evict useful translations. The hybrid keeps In-TLB MSHR inactive. 3) Workloads with rapidly changing phase behavior: if TLB pressure spikes and drops within a few thousand cycles, the Request Distributor may lag behind, routing to software when hardware would suffice (or vice versa). The paper's evaluation shows this is rare. 4) Memory bandwidth saturation: if the memory system is already bottlenecked, adding more page table reads (from thousands of PW Warps) could degrade performance. The paper shows this doesn't happen because irregular apps use only ~7% of bandwidth in the baseline (the memory system is underutilized due to pipeline stalls).
Q12Could SoftWalker be applied to CPUs?▾
Unlikely to be beneficial. CPUs have low thread counts (tens, not thousands) and low context-switch tolerance (hundreds of cycles for privilege transitions). Software-managed TLBs on early MIPS CPUs were abandoned because the OS overhead (trap, walk, return) was higher than hardware PTWs. GPUs succeed with SoftWalker because 1) massive parallelism (46 SMs × 32 threads = 1472 walkers), 2) low context-switch cost (<1 cycle), and 3) abundant idle cycles (90% for irregular apps). CPUs have none of these properties. However, the In-TLB MSHR idea (repurposing idle TLB entries as MSHR slots) could apply to CPUs with large TLBs.

Glossary & further reading

Every term in one place, then paths to go deeper.

Core terms
SoftWalkerThe paper's framework: perform GPU page table walks in software using lightweight threads (PW Warps) instead of fixed hardware PTWs.
Page Walk Warp (PW Warp)A dedicated, isolated warp on each SM that executes page table walk code. Has its own architectural slot (inst buffer, scoreboard, SIMT stack) and 16 registers.
Queueing delayTime a TLB miss spends waiting in the Page Walk Buffer (PWB) for a free hardware PTW. For irregular workloads, the dominant component of total page walk latency.
SoftPWB (Software Page Walk Buffer)A 32-entry buffer in shared memory (96 bits per entry) that holds pending page walk requests for a PW Warp. Replaces the CAM-based hardware PWB.
SoftWalker ControllerA small FSM per SM that manages the SoftPWB Status Bitmap, fills requests into SoftPWB, and launches PW Warps. Total 1470 bits per SM.
In-TLB MSHRA mechanism that repurposes idle L2 TLB entries (via a pending bit) to hold metadata for outstanding misses when dedicated MSHRs fill. Expands capacity from 128 to ~1024.
Request DistributorA module at the L2 TLB that forwards misses to SMs using a round-robin counter. Ensures no SM's SoftPWB overflows.
Hybrid modeA variant that keeps 32 hardware PTWs and activates software walkers only when hardware saturates. Optimizes for both regular and irregular workloads.
Supporting terms
Page Table Walker (PTW)Fixed-function hardware that traverses a radix page table to translate VPN→PPN. The baseline has 32 PTWs.
Page Walk Buffer (PWB)A CAM-based buffer that holds pending page walk requests when PTWs are busy. Supports associative lookup to merge requests for the same VPN.
MSHR (Miss Status Holding Register)A buffer entry that tracks metadata for an in-flight TLB miss (SM ID, warp ID, VPN). The baseline L2 TLB has 128 MSHRs, each supporting 46 merges.
Radix page tableA multi-level tree where each level uses a portion of the VPN as an index. GPUs use 4-level with a 33-bit VPN (9+9+9+9 bits for level offsets, plus 16-bit page offset for 64 KB pages).
Page Walk Cache (PWC)A cache for intermediate page table entries (PDEs). Lets walkers skip levels if the PDE is cached. Consulted before dispatching a walk.
Irregular workloadApps with scattered memory access patterns (graph processing, sparse linear algebra, random lookups) that cause high TLB miss rates.
LDPT / FL2T / FPWC / FFBNew ISA instructions for PW Warps: load page table (bypassing TLB), fill L2 TLB, fill PWC, fill fault buffer.
Pending bitA 1-bit flag per L2 TLB entry that marks it as acting as an MSHR (holding miss metadata) instead of a valid translation.
SoftPWB Status BitmapA 2-bit per thread bitmap (64 bits total) that tracks the state of each SoftPWB entry: invalid (00), valid (01), or processing (10).
Warp schedulerThe GPU's instruction issue logic that picks a ready warp each cycle. For irregular apps, 90% of cycles are stalled (no ready warps).

Related papers (GPU address translation)

Architectural Support for Address Translation on GPUs medium

Pichai, Hsu, Bhattacharjee · ASPLOS 2014

The foundational paper on GPU TLB design. Introduces the two-level TLB hierarchy and hardware PTWs, which SoftWalker extends.

Neighborhood-Aware Address Translation (NHA) medium

Shin, Cox, et al. · ISCA 2018

Coalesces page walk requests targeting the same cacheline (up to 16× reduction). SoftWalker is complementary — it increases throughput beyond what coalescing achieves.

Mosaic: TLB for Multiple Page Sizes medium

Ausavarungnirun, Landgraf, et al. · MICRO 2017

Transparent support for large pages (2 MB) in GPUs. Reduces TLB pressure but can't eliminate it for apps with footprints >2 GB. SoftWalker handles the residual pressure.

Fixed-Size Hashed Page Tables (FS-HPT) hard

Jang, Park, et al. · PACT 2024

Replaces radix page tables with hash-based lookup (1 memory access instead of 4). Reduces per-walk latency but doesn't increase PTW throughput — still bottlenecked by 32 walkers.

Avatar: Speculative TLB for GPUs hard

Park, Kwon, et al. · MICRO 2024

Speculates physical addresses and validates via embedded PTEs in cachelines. Eliminates some page walks but requires validation (still needs PTWs for uncompressed data).

IDYLL: Light-Weight PTE Invalidations in Multi-GPUs hard

Li, Guo, et al. · MICRO 2023

Optimizes TLB shootdowns in multi-GPU systems. Orthogonal to SoftWalker (focuses on coherence, not walk throughput).

All numbers (latencies, speedups, area) are from the paper's simulation (Accel-Sim, RTX 3070 config) or cited prior work.
