Storing two bits where one lived, without losing what matters

Transformers demand massive memory. MLC RRAM packs twice the data, but noise kills accuracy. This is how HyFlexPIM reshapes the model so only 5–10% of weights need expensive precision — and the rest run cheap.

Quick glossary of key acronyms
PIMProcessing-In-Memory. Compute happens inside the memory array, not in a separate ALU — no data bus trip.
RRAMResistive RAM. Non-volatile memory that stores data as resistance levels; can do analog multiply-accumulate.
SLCSingle-Level Cell. One bit per cell (high/low resistance), high noise margin, high area & energy cost.
MLCMulti-Level Cell. 2+ bits per cell via multiple resistance levels. Denser, faster, cheaper — but noisy.
SVDSingular Value Decomposition. W = U·Σ·V^T, exposing a diagonal Σ of importance ranks.
GEMVGeneral matrix-vector multiply. The core operation PIM accelerates: y = W·x.
Gradient redistributionFine-tuning after truncation so a small subset of weights/ranks have dominantly large gradients.
Hard thresholdTruncate SVD to rank k = (m×n)/(m+n), preserving MAC count vs. the original matrix.
1What PIM is and why it matters for Transformers▾

Processing-In-Memory does the multiply-add inside the memory array instead of fetching data to a CPU/GPU. For Transformers, 70% of ops are static-weight GEMVs (y = W·x), reused over many inputs. Analog RRAM PIM does each MAC in ~1 cycle at low voltage, avoiding the energy/latency of data movement.

The catch: analog is noisy. So we need to decide which data can tolerate noise (→ MLC) and which needs precision (→ SLC).

Now you're ready: HyFlexPIM's whole trick is maximizing the portion that can be MLC by reshaping the model in software first.
2Single vs. multi-level cells — the efficiency vs. accuracy trade-off▾

SLC: one bit per cell (resistance either high or low). Big gap → high noise margin → accurate, but 2× area & energy vs. 2-bit MLC. MLC: 2 bits per cell (four resistance levels: 00, 01, 10, 11). Half the area, double the throughput (two bits at once), half the analog energy — but smaller gaps → BER ~4% after one day, vs. SLC's <0.5%.

Brute-force MLC for all weights kills accuracy (40% drop for BERT on MRPC). Brute-force SLC everywhere wastes the efficiency MLC offers. The question is: which weights can be MLC?

Now you're ready: HyFlexPIM's hybrid approach: protect the critical 5–10% in SLC, run the rest in MLC for maximum efficiency with <1% accuracy loss.
3Why Transformers are memory-bound, not compute-bound▾

A BERT-Base layer is 768×768 weights (4.7 MB for INT8) plus 768×3072 FFN weights (18 MB). Moving that from DRAM costs ~400 pJ/B; computing with it (an 8-bit MAC) costs ~1 pJ. The data movement dominates. GPUs spend most cycles idle, waiting for memory.

PIM sidesteps this: the weights live in the array, the input vector is broadcast once, and the output appears 64 cycles later (for a 64-row array) with near-zero movement cost.

Now you're ready: That's why PIM wins for Transformers — and why noisy MLC is acceptable if we can isolate the 5% that really needs SLC precision.
4SVD decomposition: W = U·Σ·V^T and what it exposes▾

Any matrix W (m×n) can be written as U (m×m) · Σ (m×n diagonal) · V^T (n×n), where Σ has descending singular values σ₁ ≥ σ₂ ≥ ... The first few σ capture most information; small σ are noise.

After truncation to rank k, the matrices are U (m×k), Σ (k×k), V (k×n). We merge Σ·V^T into a single k×n matrix to store. Now there are only k "importance channels," and we can identify which ones matter most.

Now you're ready: SVD + truncation is the starting point. Fine-tuning (next primer) will then concentrate the importance into even fewer ranks.
5Hard threshold: keeping MAC count the same▾

Original W (m×n) costs L·m·n MACs per input (sequence length L). Naïve SVD → U (m×k) and V (k×n) costs L·m·k + L·k·n MACs. If k is arbitrary, this can be more or less than the original.

To preserve compute, set k = (m·n)/(m+n). Then m·k + k·n = m·n, so MAC count unchanged. E.g., for a 768×768 layer, k = (768×768)/(768+768) = 384. This also keeps parameter count (m·k + k·n) = m·n.

Now you're ready: The hard threshold ensures no compute overhead from SVD. The next step is fine-tuning to recover accuracy lost from truncation.
6Gradient redistribution: fine-tuning to concentrate importance▾

Truncating Σ to rank k loses information → accuracy drop (e.g., BERT-Large on RTE: −20%). Fine-tune for 1–3 epochs to recover. During fine-tuning, gradients ∂L/∂σᵢ reveal which ranks are critical. Before SVD, gradients are spread evenly across all weights. After SVD+truncation+fine-tuning, the first few ranks (high σ) get massive gradients, while the rest have tiny gradients.

This happens because fine-tuning tries to pack the lost information back into the untruncated ranks, and the principal (largest σ) ranks absorb it. Now we have a clear demarcation: top 5–10% of ranks → SLC; rest → MLC.

Now you're ready: This is the key software transformation. Only a handful of ranks are truly critical; the rest can tolerate MLC noise.
7Analog vs. digital PIM: when to use which▾

Analog PIM: extremely parallel (64 rows × 128 cols multiply-add simultaneously), low-voltage (tiny energy), but noisy and MLC writes are slow (iterative verify). Use for: static weights (written once, reused forever) where you can tolerate some noise after SVD. Digital PIM: bit-serial NOR operations, slower than analog but noiseless, no ADC cost, and SLC writes are fast. Use for: dynamic data (Q·K^T, attention scores) that changes every token, or high-precision ops (12-bit V).

HyFlexPIM uses digital PIM for the orange box (Q·K, softmax, ×V) and analog PIM for all linear layers (WQ, WK, WV, Proj, FFN1, FFN2).

Now you're ready: The mixed-signal split: digital for precision & dynamic, analog for throughput & static. Within analog, hybrid SLC/MLC for the final efficiency knob.
8ADC resolution & reconfigurability▾

Analog PIM accumulates current on bitlines; an ADC converts it to digital. Required precision: ⌈log₂(R)⌉ + w − 1, where R = rows, w = bits/cell. For 64 rows, SLC (w=1) needs 6-bit ADC; MLC (w=2) needs 7-bit. HyFlexPIM uses a reconfigurable 7-bit SAR ADC that can operate as 6-bit by bypassing the MSB capacitor (C7), costing <1% extra vs. a fixed 6-bit design.

Energy: going 6→7 bits doubles ADC cost, but MLC generates half the outputs (two bits at once), so total ADC energy stays flat vs. SLC.

Now you're ready: The ADC is not a bottleneck. The real win is MLC's 2× throughput and area density with negligible accuracy loss after gradient redistribution.
The whole story in one place

HyFlexPIM in a nutshell

The problem, the software trick that fixes it, the hardware that executes it, and the payoff.

Transformers are 70% static-weight GEMVs — perfect for PIM. MLC RRAM offers 2× density, 2× throughput, half the analog energy vs. SLC, but brute-force MLC kills accuracy (−40% for BERT on MRPC). Brute-force SLC everywhere wastes MLC's efficiency. The question: which weights can be MLC without losing accuracy?

HyFlexPIM's answer: gradient redistribution via SVD. Decompose W = U·Σ·V^T, truncate to rank k = (m·n)/(m+n) (preserving MAC count), fine-tune 1–3 epochs. This concentrates importance into a small subset: after fine-tuning, only 5–10% of ranks (for encoders) or 5–20% (for decoders) have dominantly large gradients. Store those in SLC; run the rest (90–95%) in MLC. Hardware: each analog PIM module is reconfigurable SLC/MLC (sharing the same WL driver, bitline circuitry, S&A, just a 6/7-bit flex ADC). Digital PIM handles attention (Q·K^T, softmax, ×V) with reliable bit-serial ops.

The payoff: BERT-Base/Large, GPT-2, Llama3, ViT-Base — all within 1% accuracy (encoders/vision) or <10% loss increase (decoders) vs. full-SLC baseline, while achieving 1.86× throughput and 1.45× energy efficiency over state-of-the-art (ASADI†), 5.4× energy & 10.6× speedup over SPRINT, for <1% area overhead (reconfigurable ADC + owner bits).

🔑 Key takeaways

  • Brute-force MLC kills accuracy (−40% for BERT); brute-force SLC wastes efficiency. Need a software-hardware co-design.
  • Gradient redistribution: SVD + hard-threshold truncation + fine-tuning → only 5–10% of ranks have huge gradients (error-sensitive). Protect those in SLC; rest → MLC.
  • Mixed-signal PIM: digital for attention (dynamic, high-precision), analog for linear layers (static, 70% of ops).
  • Hybrid SLC/MLC analog: single module reconfigures (same WL driver, just 6/7-bit ADC flex). <1% area overhead.
  • MLC writes are slow (iterative verify), but static weights are written once offline; inference is pure reads → no endurance concern.
  • Result: 1.86× throughput, 1.45× energy vs. ASADI†; 5.4× energy, 10.6× speed vs. SPRINT; <1% accuracy drop (BERT), <10% loss increase (GPT-2).
Seven everyday pictures

How to think about HyFlexPIM

Each picture stands alone — read in any order. Every card ends with where the analogy breaks.

The core idea · Gradient-based protection

Guarding the crown jewels

You have a warehouse of goods — 5% are fragile antiques (high gradient ranks), 95% are sturdy crates (low gradient). Brute-force premium insurance (SLC) for all is wasteful; brute-force cheap insurance (MLC) for all breaks the antiques. HyFlexPIM uses SVD+fine-tuning to identify the antiques, insures only those expensively, and ships the rest cheap. Result: warehouse runs 2× faster (MLC throughput), costs half as much (MLC energy), nothing breaks (<1% loss).

Where it breaks: real antiques don't change identity; after fine-tuning, the "fragile" ranks are stable only for this model/task. Retrain on new data → re-identify the top 5%.
SLC vs. MLC · The efficiency vs. noise trade-off

Wide lanes vs. narrow lanes on a highway

SLC is a single wide lane (high/low resistance) — large gap, easy to read, but only one car (bit) at a time. MLC is two narrow lanes (four resistance levels: 00, 01, 10, 11) — twice the throughput (two cars side-by-side), but smaller gaps so noise can cause a car to drift into the wrong lane (bit error). On a highway with moderate traffic (low gradient ranks), the occasional drift is tolerable. On a critical route (high gradient), you need the wide lane (SLC) to guarantee no drift.

Where it breaks: real lanes don't reconfigure; HyFlexPIM's single RRAM module switches SLC↔MLC by reprogramming the same cells. The analogy also misses that MLC writes are slower (iterative verify), but since weights are static, that's a one-time offline cost.
SVD decomposition · Exposing importance

Sorting cargo by weight

SVD decomposes W into U·Σ·V^T, where Σ is a diagonal list of "weights" (singular values σ) in descending order: σ₁ ≥ σ₂ ≥ ... Before SVD, the cargo (weights) is mixed — no clear order. After SVD, the heaviest crates (largest σ) are at the top. Truncate the light crates (small σ), fine-tune to recover accuracy, and now you have a sorted warehouse: protect the top 5% (heavy crates) in SLC, ship the rest in MLC. The sorting is automatic — SVD exposes it, fine-tuning concentrates it.

Where it breaks: real cargo has fixed weight; in SVD, the σ values change during fine-tuning (they grow for the top ranks as they absorb lost info). Also, you merge Σ·V^T offline, so the hardware never sees three separate matrices — just U and a compressed matrix.
Hard threshold · Preserving MAC count

Keeping the same number of trips

Original W (m×n) requires L·m·n deliveries (MACs) per input (sequence length L). Naïve SVD → two smaller matrices U, V could require more or fewer trips. The hard threshold k = (m·n)/(m+n) ensures the total trips (m·k + k·n) exactly equals m·n — same workload, just reorganized. For a 768×768 layer, k=384: you now have two smaller routes (768×384 and 384×768) that sum to the same total mileage as the original 768×768 route.

Where it breaks: real delivery routes have geography; MAC count is purely arithmetic. Also, the analogy misses that the two-matrix structure (U, Σ·V^T) is what enables gradient-based splitting — before SVD, all weights were equally "anonymous."
Separate monitor · Clean measurement

Two thermometers for two rooms

To learn how warm one room runs, don't measure it while a second room's heater blasts through an open door — your reading is contaminated. HyFlexPIM's gradient calculation happens in a separate "room" (during fine-tuning, using clean FP32 weights), then the final mapping to SLC/MLC hardware uses that clean gradient info. An earlier in-memory scheme (ASADI's rival) measured importance inside the noisy shared cache → only +4% gain. HyFlexPIM's clean measurement → +11% gain, at the cost of storing gradients offline (30 GB one-time, not in inference hardware).

Where it breaks: thermometers are passive; HyFlexPIM's "measurement" is active fine-tuning (1–3 epochs). Also, once the model is deployed, gradients aren't recalculated — the SLC/MLC split is frozen until you retrain.
Reconfigurable ADC · One design, two modes

A camera with a switchable lens

SLC needs a 6-bit ADC (64 rows, 1 bit/cell → 2⁶=64 levels); MLC needs 7-bit (64 rows, 2 bits/cell → 2⁷=128 levels). Instead of two cameras (two ADCs), HyFlexPIM uses a single 7-bit SAR ADC that can run as 6-bit by bypassing the largest capacitor (C7, the MSB). Overhead: one mux, <1% area. Energy: MLC generates half the outputs (two bits at once), so total ADC energy is flat vs. SLC despite the extra bit.

Where it breaks: a real lens swap is manual; HyFlexPIM's ADC mode is set per subarray (controlled by which weights are SLC vs. MLC) and doesn't change during inference. Also, the analogy misses that the ADC is shared (128 bitlines → one ADC via S&H + mux), so it's already amortized.
Mixed-signal split · When to use which

Conveyor belts vs. craftsmen

Analog PIM is a high-speed conveyor belt (64×128 cells multiply-add simultaneously, low voltage, tiny energy) — great for bulk goods (static weights, written once, reused forever). Digital PIM is a skilled craftsman (bit-serial NOR, slower but noiseless, no ADC) — needed for custom work (dynamic Q·K^T every token, high-precision 12-bit V). HyFlexPIM uses the conveyor for linear layers (WQ/WK/WV, FFN: 70% of ops) and craftsmen for attention (orange box: Q·K, softmax, ×V). Don't use craftsmen for bulk goods (too slow); don't use conveyors for custom work (noisy & MLC writes are slow).

Where it breaks: conveyors are one-directional; RRAM PIM is bidirectional (read/write), but writes are rare (static weights). Also, the analogy misses that digital PIM still stores data in RRAM (1024×1024 arrays), just uses bit-serial compute instead of analog current summation.
The full explanation

How it actually works

Section by section, with the paper's key diagrams rebuilt step-by-step.

1 · The problem — brute-force MLC kills accuracy

Transformers are 70% static-weight GEMVs (y = W·x for WQ, WK, WV, Proj, FFN1, FFN2). Each GEMV reuses the same W over many inputs, making PIM ideal: write W once, read forever. 2-bit MLC offers 2× density (half the area), 2× throughput (two bits per cell), half the analog energy vs. 1-bit SLC. The catch: MLC's four resistance levels (00, 01, 10, 11) have smaller gaps → higher bit error rate (BER ~4% after one day vs. SLC's <0.5%).

Brute-force MLC for all weights: BERT-Base on MRPC drops from 84.8% to 44.8% accuracy (−40%). Brute-force SLC everywhere: wastes MLC's 2× throughput and area. The question: which weights can be MLC?

🔍
Concrete numbers: BERT-Large has 24 layers × (768×768 WQ/WK/WV + 768×3072 FFN1 + 3072×768 FFN2) = ~335M parameters. At INT8, that's 335 MB. MLC stores it in half the area (167 MB worth of cells), doubling throughput, but naive mapping → catastrophic accuracy loss.

2 · SVD decomposition and hard-threshold truncation

Any matrix W (m×n) can be written as W = U·Σ·V^T, where U (m×m) and V (n×n) are orthogonal, and Σ (m×n diagonal) has descending singular values σ₁ ≥ σ₂ ≥ ... ≥ 0. Truncate to rank k: U_k (m×k), Σ_k (k×k), V_k (k×n). Merge Σ_k·V_k^T into a single k×n matrix to store.

To preserve MAC count (L·m·n per input), set k = (m·n)/(m+n). For a 768×768 layer, k = (768×768)/(768+768) = 384. Now the two matrices U (768×384) and Σ·V^T (384×768) require 768×384 + 384×768 = 589,824 MACs, exactly equal to the original 768×768 = 589,824.

# original W
W_original: (m, n)  →  L * m * n MACs

# post-SVD
U: (m, k),  ΣV^T: (k, n)  →  L * (m*k + k*n) MACs

# hard threshold k = (m*n)/(m+n) ensures:
m*k + k*n = m*n  → same MAC count

Truncation loses information → accuracy drop (BERT-Large on RTE: −20%). Next: fine-tune to recover.

3 · Gradient redistribution via fine-tuning

Fine-tune for 1–3 epochs (BERT-Base/Large: 1 epoch on simple tasks, 2–3 on complex). During fine-tuning, gradients ∂L/∂σᵢ reveal which ranks are critical. Before SVD (Figure 11a): gradients are evenly spread across all weights (order-wise in a row). After SVD+truncation, no fine-tuning (Figure 11b): gradients differ by rank, but not dramatically. After fine-tuning (Figure 11c): the top ranks (largest σ) absorb the lost information and their gradients explode (10–100× larger than the rest).

Why? Fine-tuning tries to recover the truncated information by packing it into the untruncated ranks. The principal (largest σ) ranks have the most "capacity" to absorb it, so they get the bulk of the gradient updates. Now only 5–10% of ranks (encoders) or 5–20% (decoders) have dominantly large gradients. Those are the error-sensitive ranks → map to SLC. Rest → MLC.

💡
Key insight: Gradient redistribution isn't just a side effect — it's the goal. By forcing the model to represent itself via truncated SVD, we create a clear importance hierarchy that didn't exist in the original flat weight matrix.

4 · Hybrid SLC/MLC analog PIM module

Each analog PIM module is 512 RRAM arrays, each 64 rows × 128 columns. SLC mode: each cell stores 1 bit (high/low resistance), requires 4 columns per 4-bit weight (w₃, w₂, w₁, w₀ mapped vertically). MLC mode: each cell stores 2 bits (four resistance levels), requires 2 columns per 4-bit weight (w_MSB = w₃w₂, w_LSB = w₁w₀). Same WL driver (applies input bits sequentially), same bitline circuitry, same S&A module (just different shift factors: SLC uses 1×, 2×, 4×, 8× for w₀...w₃; MLC uses 1×, 2×, 4×, 16× for w_LSB, w_MSB). Difference: ADC precision (6-bit for SLC, 7-bit for MLC).

The reconfigurable ADC is a 7-bit SAR with a bypass mux for C7 (MSB capacitor). For SLC, bypass C7 → 6-bit operation. For MLC, enable all capacitors → 7-bit. Overhead: one mux + control logic, <1% area. Energy: MLC generates half the outputs (two bits at once), so total ADC energy is flat vs. SLC.

🔑
Critical detail: MLC writes are slow (iterative verify-read-write to hit the exact target resistance), but static weights are written once offline before inference. Inference is pure reads → fast, no endurance concern (RRAM endurance: 10⁸ cycles, far exceeding server lifespan).

5 · Mixed-signal PIM: digital for attention, analog for linear layers

Transformers have two categories of ops: 1) Static-weight GEMVs (WQ, WK, WV, Proj, FFN1, FFN2: 70% of FLOPs), written once, reused forever → analog PIM. 2) Dynamic ops (Q·K^T, softmax, ×V), where both operands change every token → digital PIM.

Why digital for attention? Q·K^T is matrix-matrix (not GEMV), softmax needs transcendental functions, and ×V requires 12-bit precision (high bit-width to avoid cumulative error across sequence). Analog PIM excels at GEMV but is noisy and expensive for high-precision ops. Digital PIM uses bit-serial NOR (Figure 3(d)): Q (INT8) × K (INT8) via 64 NOR ops, 3 columns each (two operand bits, one output), 5 cycles per row (4 write, 1 read). Noiseless, flexible precision, no ADC cost.

Digital PIM also stores intermediate results (Q, K, V tokens after the linear layers, attention scores after Q·K^T) in its own 256 RRAM arrays (1024×1024 each, 128 KB per array). This avoids costly off-chip DRAM fetches for the KV cache.

6 · End-to-end dataflow

Input token embedding (768-dim for BERT-Base) enters via the global bus (PCIe-6.0, 128 GB/s). Each PU processes one layer. Step 1: Analog PIM computes WQ·x, WK·x, WV·x in parallel (three of the 24 analog modules per PU). Post-SVD, each linear layer is two GEMVs: U·(Σ·V^T·x). Critical ranks (top 5–10% by gradient) are in SLC subarrays, rest in MLC. Step 2: Q, K, V tokens are written to digital PIM arrays. Step 3: Digital PIM computes Q·K^T (64 NOR ops per output element), applies softmax in the SFU (Taylor series for exp, pipelined FP ops), then ×V (bit-serial again). Step 4: Attention output goes to analog PIM for Proj·x. Step 5: FFN1·x, FFN2·x in analog PIM. Step 6: Output (768-dim) sent to next PU via inner-unit shared bus (1000 GB/s OCI).

For multi-PU (tensor parallelism) or multi-chip (pipeline parallelism), partial sums (<3 KB per PU) are aggregated via global bus (24 cycles overhead). For Llama3 (dual-chip), only a single hidden layer (0.75–2 KB) is sent between chips (6–16 cycles over PCIe-6.0).

Follow the numbers, step by step

Worked examples

Four traces, each carried all the way through — SVD decomposition, gradient ranking, SLC/MLC mapping, and the final accuracy check.

① SVD decomposition and hard-threshold truncation for a 768×768 layer

BERT-Base WQ layer: 768×768 weights (589,824 parameters). Apply SVD, truncate to preserve MAC count. What's k, and how many parameters remain?

  1. Original MAC count: For sequence length L=128, each GEMV costs L·m·n = 128×768×768 = 75,497,472 MACs per input.
  2. Hard threshold k: k = (m·n)/(m+n) = (768×768)/(768+768) = 589,824/1,536 = 384.
  3. Post-SVD matrices: U (768×384), Σ (384×384 diagonal), V^T (384×768). Merge Σ·V^T → 384×768.
  4. New MAC count: L·(m·k + k·n) = 128×(768×384 + 384×768) = 128×589,824 = 75,497,472. Same as original.
  5. Parameter count: U has 768×384 = 294,912, Σ·V^T has 384×768 = 294,912. Total = 589,824, same as original.
🔑
Insight: The hard threshold ensures zero compute overhead. The reorganization (from one 768×768 matrix to two 768×384 and 384×768) exposes rank importance via the diagonal Σ — that's what SVD buys you.

② Gradient redistribution: identifying the top 5% ranks

After SVD+truncation, fine-tune BERT-Base WQ for 1 epoch. Gradients ∂L/∂σᵢ are saved. How many ranks are protected in SLC at 5%, and what's the threshold?

  1. Truncated rank k = 384. Top 5% means 5% × 384 = 19.2 → round to 19 ranks are SLC.
  2. Sort gradients: |∂L/∂σ₁| = 0.0035, |∂L/∂σ₂| = 0.0032, ..., |∂L/∂σ₁₉| = 0.0012, |∂L/∂σ₂₀| = 0.00008 (100× drop after rank 19). Set threshold at 0.0012.
  3. SLC mapping: Ranks 1–19 (columns 1–19 of Σ·V^T and rows 1–19 of U) → SLC subarrays. Each rank is 768+768 = 1,536 parameters.
  4. MLC mapping: Ranks 20–384 (365 ranks, 365×1,536 = 560,640 parameters, 95% of total) → MLC subarrays.
  5. Hardware implication: 19 SLC subarrays (each 64×128, 1-bit/cell) + 365 MLC subarrays (64×128, 2-bit/cell). MLC subarrays are half the area of SLC, so total area ≈ 19 + 365/2 = 201.5 equiv. SLC arrays vs. 384 full SLC. 47.6% area savings.
💡
The gradient threshold is task-specific. For BERT-Large on RTE (harder task), top 10% (38 ranks) are needed; for MRPC (easier), 5% suffices. The paper sweeps 5%, 10%, 30%, 40%, 50%, 100% SLC and plots accuracy vs. protection rate (Figure 12).

③ Noise injection and accuracy recovery

BERT-Base on MRPC, 5% SLC mapping (19/384 ranks). Inject Gaussian noise η ~ N(0, σ²) into MLC weights to simulate BER=4.04%. What's σ, and what's the final accuracy?

  1. BER calibration: Paper uses real RRAM data from [15]: BER=4.04% after one day. Reverse-calculate σ: for a 2-bit MLC cell, BER=4.04% means ~4 out of 100 cells drift to wrong level. σ ≈ 0.025 (standard deviation relative to the gap between levels).
  2. Noise formula: W̃ = W ⊙ (1 + η), where η ~ N(0, 0.025²). Apply only to MLC ranks (20–384).
  3. SLC ranks (1–19): No noise (or minimal σ=0.0025, 10× lower). These are the critical ranks with large gradients.
  4. Inference accuracy: BERT-Base on MRPC, 5% SLC → 84.31% accuracy (baseline without PIM noise: 84.80%, drop of 0.49%). Compare: 100% MLC → 44.8% (−40%), 100% SLC → 84.80% (no drop but double the area).
  5. Why it works: The top 5% ranks (SLC) absorb the gradient-critical weights. The rest (95% in MLC) have tiny gradients, so noise has minimal impact on loss.
🔑
Key result: By protecting only 5%, you recover 99.4% of the baseline accuracy while running 95% of the weights in MLC (2× throughput, 2× density, half energy). That's the power of gradient redistribution.

④ Energy and throughput calculation for BERT-Base WQ layer

Compare full-SLC vs. 5%-SLC hybrid for one GEMV (768×768 weights, INT8 input, sequence length L=128). What's the throughput and energy?

  1. Full-SLC: 384 arrays (each 64×128, 1-bit/cell). Each GEMV takes 64 cycles (bit-serial for 8-bit input) × 4 columns (4-bit weight split) = 256 cycles per output. For 768 outputs, 256 cycles (pipelined). Throughput: 128×768×768 MACs / 256 cycles = 295,488 MACs/cycle.
  2. 5%-SLC hybrid: 19 SLC arrays (256 cycles) + 365 MLC arrays (128 cycles, 2× faster due to 2 bits/cell). Bottleneck: SLC (256 cycles). But only 19/384 = 4.9% of outputs wait full 256; rest done at 128. Effective cycles ≈ 0.049×256 + 0.951×128 = 134 cycles. Throughput: 75,497,472 MACs / 134 cycles = 563,414 MACs/cycle (1.91× faster).
  3. Energy (analog computation only): SLC: 60.78 mW per array (from Table 2) × 384 = 23,339 mW. Hybrid: 19×60.78 (SLC) + 365×30.39 (MLC, half the energy) = 1,155 + 11,092 = 12,247 mW (52% of full-SLC).
  4. ADC energy: SLC: 512 mW per 6-bit ADC × 384 = 196,608 mW. Hybrid: 19×512 (6-bit) + 365×1024 (7-bit, but half the conversions) = 9,728 + 186,880 = 196,608 mW (same, because MLC does half the conversions at 2× cost per conversion).
  5. Total energy: Full-SLC: 23,339 + 196,608 = 219,947 mW. Hybrid: 12,247 + 196,608 = 208,855 mW (95% of full-SLC). But throughput is 1.91× higher, so energy-delay product is 95% / 1.91 = 49.7% of full-SLC (2.01× better).
💡
The throughput win is sublinear (1.91× instead of 2×) because the 5% SLC bottleneck still takes 256 cycles. For higher SLC rates (e.g., 30%), throughput drops to ~1.4×. The energy win is also sublinear (1.05× instead of 2×) because ADC energy is flat. But combined, energy-delay is 2× better, and that's with only 5% SLC.

Practice problems

Try each before opening the answer. Every one ends with the common misconception it's checking for.

Problem 1 · easy
Why does brute-force MLC (100% MLC, no SLC) fail so catastrophically for BERT? (Hint: it's not just "noise is bad.")
Show answer
Transformers have cascaded layers (12–24) with feedback loops (residual connections). Each layer's error compounds: a 1% error in layer 1 becomes 1.01¹² ≈ 12.7% by layer 12. MLC's 4% BER per layer → 1.04¹² ≈ 60% error by the end. Also, attention (softmax) is exponentially sensitive to small input changes: a ±0.01 shift in Q·K^T can flip the argmax, destroying the attention pattern. SLC's <0.5% BER stays within tolerable bounds; MLC's 4% doesn't.Misconception: that noise is uniformly bad. It's bad for critical paths (high-gradient ranks, early layers, attention keys). HyFlexPIM isolates those in SLC.
Problem 2 · medium
A 1024×1024 layer has hard-threshold k = 512. After fine-tuning, the top 10% (51 ranks) have 90% of the total gradient magnitude. If you protect only those 51 ranks in SLC and run the rest in MLC, what's the MLC ratio, and what's the expected area savings vs. full-SLC?
Show answer
MLC ratio = (512 − 51) / 512 = 461/512 = 90.0%. Full-SLC area: 512 equiv. arrays. Hybrid: 51 SLC + 461 MLC (each MLC = 0.5× area) = 51 + 461×0.5 = 51 + 230.5 = 281.5 equiv. arrays. Savings: (512 − 281.5)/512 = 45.0%. But throughput is ~1.8× (bottleneck on 51 SLC), so effective area-throughput = 281.5/1.8 = 156.4 equiv. arrays (69.5% savings).Misconception: that 90% MLC → 90% area savings. MLC is only 50% denser, and the SLC bottleneck limits throughput, so you must amortize over throughput to get the true efficiency win.
Problem 3 · hard
Why does gradient redistribution work — i.e., why do the top ranks absorb the lost information during fine-tuning, instead of spreading it evenly across all untruncated ranks?
Show answer
SVD orders ranks by variance explained: rank 1 (largest σ₁) captures the direction of maximum variance in the data, rank 2 (σ₂) the second-max orthogonal direction, etc. When you truncate ranks k+1...n, you lose variance in those directions. Fine-tuning tries to minimize loss L by adjusting the remaining ranks. The top ranks (large σ) already span the high-variance subspace, so they can "stretch" slightly to approximate the lost variance with minimal distortion. Lower ranks (small σ) span low-variance directions and can't meaningfully absorb high-variance info without breaking orthogonality. Result: gradients ∂L/∂σᵢ are largest for i=1...k_critical, exponentially smaller for i > k_critical.Misconception: that gradients are a hyperparameter you set. They're the outcome of fine-tuning on a truncated basis. The concentration happens naturally because the SVD basis is ordered by importance.
Problem 4 · medium
HyFlexPIM uses a reconfigurable 7-bit ADC that can run as 6-bit by bypassing C7. Why doesn't this double the energy cost (since going 6→7 bits normally doubles ADC energy)?
Show answer
SLC (6-bit): 64 rows, 1 bit/cell → 128 bitline outputs per GEMV (128 columns). Each needs one ADC conversion → 128 conversions. MLC (7-bit): 64 rows, 2 bits/cell → 64 outputs (two bits per column) → 64 conversions. Energy per conversion: 6-bit ≈ E, 7-bit ≈ 2E. Total: SLC = 128×E = 128E, MLC = 64×2E = 128E (same). The extra bit precision is exactly offset by half the number of outputs.Misconception: that MLC's 7-bit ADC is "more expensive." It's more expensive per conversion, but you do half as many conversions, so total energy is flat.
Problem 5 · easy
Why is the hard threshold k = (m·n)/(m+n) instead of, say, k = min(m, n) or k = √(m·n)?
Show answer
Goal: preserve MAC count. Original W (m×n) costs m·n MACs. Post-SVD, two GEMVs: U (m×k) costs m·k, Σ·V^T (k×n) costs k·n. Total: m·k + k·n. Set this equal to m·n: m·k + k·n = m·n → k(m + n) = m·n → k = (m·n)/(m+n). If k = min(m,n), you'd have m·k + k·n ≠ m·n (unless m=n). If k = √(m·n), same issue. The harmonic-mean-like formula (m·n)/(m+n) is the only choice that balances the two GEMVs to sum to the original MAC count.Misconception: that k is arbitrary. It's the unique value that preserves compute. Any other k either adds overhead (k too large) or loses capacity (k too small, forcing more aggressive truncation).
Problem 6 · hard
HyFlexPIM's 5% SLC mapping works for BERT-Base on MRPC but fails (accuracy drops >5%) for GPT-2 on WikiText-2. Why does the decoder need more SLC (20%) than the encoder (5%)?
Show answer
Encoders (BERT) are bidirectional: attention sees the full sequence at once, so errors in one token are "averaged out" by the context. Decoders (GPT-2) are autoregressive: each token depends on all previous tokens, so an error at position t propagates to t+1, t+2, ..., T (sequence length T). This is error accumulation (like compound interest). For T=1024 (WikiText-2), a 1% per-token error becomes 1.01¹⁰²⁴ ≈ ∞. MLC's 4% BER is tolerable for 5% SLC in encoders (errors don't compound across the sequence) but not in decoders (they do). Solution: protect 20% in SLC (the critical ranks for early layers + high-gradient FFN weights), giving the autoregressive chain enough precision to avoid runaway error.Misconception: that encoders and decoders are "the same architecture." They differ in error propagation structure: bidirectional (parallel, error-dampening) vs. autoregressive (serial, error-compounding). That's why decoders need more SLC.
Problem 7 · medium
Why does HyFlexPIM use digital PIM for Q·K^T instead of analog PIM, even though analog PIM is faster?
Show answer
Three reasons: 1) Q·K^T is matrix-matrix (L×D_h · D_h×L → L×L), not GEMV. Analog PIM excels at GEMV (one vector broadcast to all rows); for matrix-matrix, you'd need L sequential GEMVs, no parallelism win. 2) Q, K change every token (dynamic data). MLC writes are slow (iterative verify), so you'd waste cycles writing. SLC writes are fast but noiseless digital PIM is even faster (bit-serial, no ADC). 3) Attention is sensitive: softmax(Q·K^T/√d) exponentially amplifies small errors. A ±0.01 shift in Q·K^T can flip the argmax, destroying the attention pattern. Digital PIM (INT8 × INT8 via NOR) is noiseless, avoiding this issue.Misconception: that analog PIM is always faster. It's faster for GEMV with static weights. For dynamic matrix-matrix with noise-sensitive ops, digital PIM's noiseless bit-serial approach wins.
Problem 8 · easy
If gradient redistribution is so effective, why not apply it without SVD — just fine-tune the original model and map the top 5% weights (by gradient magnitude) to SLC?
Show answer
Without SVD, the weights are a flat m×n matrix with no inherent ordering. Gradients ∂L/∂W_ij are computed, but they're spread across all m·n elements with no structure. Even after fine-tuning, the distribution stays roughly uniform (see Figure 11a) — no clear "top 5%" emerges. SVD creates the ordering via Σ (diagonal of singular values), and truncation forces the model to pack importance into a few ranks. Without SVD, you have no basis to demarcate critical vs. non-critical weights, and brute-force gradient thresholding would protect random scattered weights (no spatial locality in the RRAM array → no area/throughput win).Misconception: that gradients alone are enough. Gradients reveal importance, but SVD provides the structure (rank ordering) that makes gradient-based splitting practical in hardware.
Quick answers

Frequently asked questions

Twelve questions newcomers ask most. Tap any to expand.

Q1What's the difference between SLC and MLC in one sentence?▾
SLC stores 1 bit per cell (high/low resistance, high noise margin, high cost); MLC stores 2+ bits per cell (multiple resistance levels, 2× denser, 2× throughput, half energy, but noisy).
Q2Why does brute-force MLC fail for Transformers but work for CNNs/SNNs?▾
CNNs/SNNs have inherent error resiliency (local spatial features, spiking = binary). Transformers are cascaded (12–24 layers) with attention (exponentially sensitive to small changes in Q·K^T) — errors compound catastrophically. MLC's 4% BER is tolerable for 1–2 layer CNNs, not for 24-layer BERT.
Q3What is SVD actually doing here — why not just prune weights directly?▾
SVD decomposes W = U·Σ·V^T, exposing a diagonal Σ of ordered importance (largest σ₁ = most important). Truncate small σ, fine-tune to recover accuracy, and gradients naturally concentrate into the top ranks. Direct pruning (zero out small weights) doesn't create this ordering — it just makes the matrix sparse, with no clear demarcation for SLC/MLC mapping.
Q4How is the hard threshold k = (m·n)/(m+n) derived?▾
Original W (m×n) costs m·n MACs. Post-SVD, U (m×k) and Σ·V^T (k×n) cost m·k + k·n. Set equal: m·k + k·n = m·n → k(m+n) = m·n → k = (m·n)/(m+n). This is the unique value that preserves MAC count. It's also the harmonic mean of m and n (up to a factor of 2).
Q5Why does fine-tuning concentrate gradients into the top ranks?▾
Truncation loses variance in directions k+1...n. Fine-tuning tries to recover it by adjusting the untruncated ranks. The top ranks (large σ) span the high-variance subspace and can "stretch" to approximate the lost info. Lower ranks (small σ) can't, so they get tiny gradient updates. Result: ∂L/∂σᵢ is large for i=1...k_critical, exponentially smaller for i > k_critical.
Q6What's the reconfigurable ADC, and why does it cost <1% overhead?▾
A 7-bit SAR ADC with a bypass mux for C7 (MSB capacitor). For SLC, bypass C7 → 6-bit. For MLC, enable all caps → 7-bit. The mux + control logic is one transistor, <1% of the ADC area (which is already <1% of the total chip area per Table 2). Energy: MLC does half the conversions, so total ADC energy is flat vs. SLC.
Q7Why use digital PIM for attention instead of analog PIM?▾
Three reasons: 1) Q·K^T is matrix-matrix, not GEMV (no parallelism win). 2) Q, K are dynamic (written every token), and MLC writes are slow. 3) Attention is noise-sensitive (softmax amplifies errors). Digital PIM is noiseless, fast for bit-serial ops, and doesn't require ADCs.
Q8What's the BER (bit error rate) for SLC vs. MLC, and why does it matter?▾
SLC: <0.5% BER after one day (measured from [15]). MLC: ~4% BER. For a 12-layer BERT, 1.005¹² ≈ 1.06 (6% cumulative error, tolerable). 1.04¹² ≈ 1.60 (60% cumulative error, catastrophic). That's why MLC needs gradient-based protection: isolate the 5% critical weights in SLC, tolerate 4% BER in the rest.
Q9Does HyFlexPIM always beat ASADI/SPRINT?▾
No — on very short sequences (N<128) where FFN dominates, ASADI's token pruning can win. On very long sequences (N>8192) where attention dominates, SPRINT's analog Q·K pre-filtering helps. HyFlexPIM's sweet spot is moderate N (512–2048), where linear layers are 70% of ops and MLC's 2× throughput matters most. See Figure 17 for scaling analysis.
Q10Why doesn't MLC's slow write (iterative verify) hurt inference?▾
Static weights are written once offline before deployment. Inference is pure reads (fast, <100 ns per GEMV). MLC writes take ~10 µs (100× slower than SLC), but that's a one-time cost during model loading, not in the critical path. For dynamic data (Q, K, V), we use digital PIM (fast SLC writes) or avoid MLC entirely.
Q11What's the area breakdown of HyFlexPIM?▾
Per Table 2: analog PIM module is 0.47 mm² (64.2% ADC, 22% S&A, 9.6% RRAM array, 3.7% WL driver, rest IR/OR/S&H). Digital PIM module is 8.01 mm² (59.8% SFU, 35.8% RRAM, 2.6% S&A, 1.8% WL driver). Total chip: 24 analog + 8 digital per PU × 24 PUs = ~200 mm² at 65 nm. Reconfigurable ADC adds <1% (0.003 mm² per module).
Q12Can you apply gradient redistribution to other models (ResNet, Llama3, ViT)?▾
Yes, but the optimal SLC rate varies. ResNet (conv layers): 5% SLC works (convs are error-tolerant). Llama3 (decoder, autoregressive): needs 20% SLC (error compounds across sequence). ViT (encoder, vision): 5% SLC works (similar to BERT). The principle is universal: SVD + truncate + fine-tune → gradient redistribution → protect top-k% in SLC. The k depends on error propagation structure (bidirectional vs. autoregressive).

Glossary & further reading

Every term in one place, then a path to go deeper.

Core terms
HyFlexPIMHybrid (SLC/MLC), Flexible (reconfigurable), Processing-In-Memory accelerator for Transformers via gradient redistribution.
Gradient redistributionFine-tuning post-SVD to concentrate importance into a small subset of ranks, enabling clear SLC/MLC demarcation.
SLC (Single-Level Cell)1 bit per cell (high/low resistance). High noise margin, high area/energy cost. Used for critical 5–10% of ranks.
MLC (Multi-Level Cell)2+ bits per cell via multiple resistance levels. 2× density, 2× throughput, half energy, but BER ~4% vs. SLC's <0.5%.
SVD (Singular Value Decomposition)W = U·Σ·V^T. Exposes rank importance via diagonal Σ (σ₁ ≥ σ₂ ≥ ...). Truncate small σ, fine-tune to recover accuracy.
Hard thresholdTruncate SVD to rank k = (m·n)/(m+n), preserving MAC count vs. original matrix.
Reconfigurable ADC7-bit SAR ADC that can run as 6-bit by bypassing C7. <1% overhead. Energy-flat vs. SLC due to half the conversions in MLC.
Mixed-signal PIMDigital PIM for attention (Q·K^T, softmax, ×V), analog PIM for linear layers (WQ/WK/WV, FFN). 70% of ops in analog.
Supporting terms
GEMVGeneral matrix-vector multiply: y = W·x. The core operation analog PIM accelerates.
BER (Bit Error Rate)Probability a cell's resistance drifts to the wrong level. SLC: <0.5%, MLC: ~4% (measured from real RRAM [15]).
Attention bottleneckQ·K^T is matrix-matrix (not GEMV), dynamic (changes every token), and noise-sensitive (softmax). Why digital PIM.
Static weightsWQ, WK, WV, Proj, FFN1, FFN2. Written once, reused forever. 70% of Transformer FLOPs. Perfect for analog PIM.
Shift & Add (S&A)Digital module that combines bit-serial results (e.g., w₀×1 + w₁×2 + w₂×4 + w₃×8) to form the final MAC output.
SAR ADCSuccessive-Approximation-Register ADC. Binary search (C7→C1) for 7-bit conversion in ~7 cycles. Energy: ~512 mW per 6-bit, ~1024 mW per 7-bit.
Tensor parallelismMultiple PUs collaborate to process one layer (for large hidden dims or long sequences). Partial sums aggregated via OCI.
Pipeline parallelismMultiple chips cascade for deep models (>24 layers). Hidden layer output (0.75–2 KB) sent via PCIe-6.0 (6–16 cycles).
SFU (Special Function Unit)Handles non-linear ops (softmax, LayerNorm, GELU) via pipelined FP ops (Taylor series for exp). 256 inputs/cycle, 138.89 mW.
