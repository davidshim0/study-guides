Splitting memory's job: accelerate retrieval and generation together

Retrieval-augmented language models ask memory for two things at once: big matrices for attention, and vector lookups for search. This is the plain-English story of MNM, a hardware architecture that puts each where it fits best — PIM in the DRAM and PNM near the DRAM — delivering 29.2× speedup.

📄 "Accelerating Retrieval Augmented Language Model via PIM and PNM Integration" · Jang et al. 🎓 Assumes tiering/CXL background ⏱ ~25 min read
🧩

Put bandwidth-starved GEMV for attention in the memory banks (PIM) and lookup-heavy retrieval on the logic die (PNM) — and schedule them to overlap.
Same memory stack, two specialized jobs, maximum throughput.

Eight ideas the paper assumes you know

Each card clears up one concept MNM builds on. Read in order the first time, or jump to anything unfamiliar.

PIM (in-memory acceleration) PNM (near-memory logic) MNM (both together) GPU baseline
1Retrieval-Augmented Language Model (RALM) and RETRO▾

A RALM couples a token generator (a transformer LM) with a retriever that fetches relevant external data on-demand. The RETRO model queries a vector database every few tokens, encoding the retrieved data and feeding it into the decoder's chunked cross-attention layer. This decouples knowledge from model parameters, reduces hallucination, and keeps answers current.

Now you're ready: MNM targets both the generator's memory-bound attention kernels and the retriever's compute-inefficient lookups, accelerating them simultaneously.
2GEMV vs GEMM in attention — why batching is hard▾

In token generation, self-attention and cross-attention compute Q×KTV with unique K, V matrices per request. Since K and V differ across inputs, you can't batch these into fat matrix multiplies (GEMM). Instead, each head reads its own K and V slices and performs GEMV (General Matrix-Vector multiplication), which has low arithmetic intensity and is severely memory-bound.

Now you're ready: PIM units tackle GEMV by leveraging the high internal bandwidth inside each DRAM bank, where parallelism is abundant but logic is constrained.
3Vector search with IVF-PQ — clustering, quantization, and lookups▾

IVF (Inverted File index) partitions the dataset into clusters and searches only the top-nprobe clusters closest to the query, narrowing the scan. PQ (Product Quantization) compresses each vector into compact codes by splitting it into sub-vectors and mapping each to a codebook index. Searching then involves LUT lookups to reconstruct distances, followed by intensive top-k selection over millions of candidates.

Now you're ready: IVF-PQ is memory-intensive (frequent reads) and sorting-heavy (top-k selection), which GPUs parallelize poorly. PNM logic on the HBM logic die attacks both.
4HBM's two-tier structure: core die and logic die▾

A modern HBM3 stack comprises multiple core (memory) dies stacked on one logic (buffer) die. The core dies hold the actual DRAM cells; the logic die handles I/O, TSVs (through-silicon vias), and interfaces. The paper exploits this: PIM logic (simple MAC units) sits in the banks on each core die, while PNM accelerators (more complex sorters and scanners) sit on the logic die.

Now you're ready: Placing logic in both tiers lets you use core-die bandwidth for GEMV and logic-die aggregate bandwidth plus extra transistors for retrieval.
5Processing In Memory (PIM) basics — banks, rows, and MAC units▾

PIM integrates compute directly into DRAM banks. In MNM, each bank has a MAC unit (16-bit FP) that reads a row into a buffer and a query vector from a global bus, then dot-products them in parallel across all banks. A bank's internal bandwidth is orders of magnitude higher than off-chip, so GEMV (Q·K per head) can be fast if you organize data in rows and broadcast Q slices globally.

Now you're ready: Self-attention's Q·KT and V-weighted sum both become PIM-friendly: load columns of K or V as rows, broadcast Q or softmax outputs, and compute in the banks.
6Processing Near Memory (PNM) basics — specialized logic on the buffer die▾

PNM sits on the logic die, between the core dies and the host. It's fabricated with standard logic processes (e.g., 7nm), so it can implement more complex circuits (e.g., sorters, codebook lookups, distance adders) than PIM. PNM reads data from memory, processes it with dedicated accelerators, and writes results back or hands them off to the host — offloading tasks that don't map well to GPU.

Now you're ready: The retriever's PQ code scanning and top-k sorting are a perfect PNM use case: they need high memory throughput and specialized logic, but minimal GPU cores.
7Dual row buffer — concurrency without conflict▾

A typical bank has one row buffer that holds the activated row. When PIM is computing (e.g., MAC operations on that row), a memory request for another row would cause a conflict. The paper adopts a dual row buffer scheme: one buffer for PIM operations and one for normal I/O or PNM reads. This allows PIM (for attention) and PNM (for retrieval) to run simultaneously on different rows.

Now you're ready: MNM schedules token generation (PIM) and retrieval (PNM) to overlap, hiding latencies by exploiting the dual buffer's concurrency.
8Selective batching and early generation — overlapping requests▾

In standard batched serving, all requests wait for the slowest retrieval to finish before continuing generation, wasting GPU cycles. Selective batching lets requests resume generation individually as their retrievals complete. Early generation goes further: if one request's retrieval is running, others can generate a few extra tokens in advance, keeping the GPU busy. Because MNM's retrieval is faster, early-generated tokens don't degrade accuracy much.

Now you're ready: MNM's scheduling strategy uses both techniques to maximize throughput — the software counterpart to the hardware acceleration.
The whole story in one place

MNM in a nutshell

The problem, the split-location fix, and how scheduling completes the picture.

A Retrieval-Augmented Language Model (RALM) interleaves token generation (language model) with vector search (retriever) to ground its outputs in external knowledge. Profiling RETRO on an H100 GPU reveals that both phases are memory-bound, yet have fundamentally different bottleneck characteristics. The language model's attention layers (self-attention and chunked cross-attention) rely on unique per-request K and V matrices, forcing GEMV operations with low data reuse and high bandwidth demand. Meanwhile, the retriever's IVF-PQ search involves frequent LUT lookups and intensive sorting (top-k selection), operations that are hard to parallelize on GPU and exhibit low compute utilization.

The paper proposes MNM, a heterogeneous architecture integrating Processing In Memory (PIM) on the HBM core dies and Processing Near Memory (PNM) on the logic die. PIM units (16-bit FP MAC per bank, ~650 MHz) accelerate GEMV by exploiting each bank's high internal bandwidth: Q vectors are broadcast over a global bus, while KT and V columns are read from the PIM row buffers and dot-producted in parallel across all banks. PNM modules on the logic die handle retrieval tasks: PQ code scanners perform lookups into precomputed tables and reconstruct distances, while a top-k sorter (using odd-even merge sort) selects the closest candidates. A dual row buffer design allows PIM and PNM to access memory concurrently, enabling overlap of generation and retrieval.

To exploit this hardware, the paper introduces a scheduling strategy combining selective batching and early generation. In selective batching, requests resume token generation individually as soon as their retrieval completes, avoiding batch-wide synchronization stalls. Early generation allows requests waiting for another's retrieval to produce a few tokens ahead, keeping the GPU busy. Because MNM shortens retrieval latency (up to ~8× vs. GPU for Config 5, nprobe=256), the number of early tokens remains bounded and accuracy loss is minimal (perplexity trends from Figure 8 suggest small increases with batch size 256 vs. baseline).

The payoff: Across RETRO-0.5B to RETRO-7.5B and various dataset sizes, MNM with early generation delivers up to 29.2× E2E speedup (Config 5, 140M dataset) over GPU, 71.5% energy savings (Config 5, 700M dataset), and scales to 310B parameters with projected 2.95× speedup over 8× GB200 GPUs — all with 15.0% HBM core die overhead for PIM and 2.0% logic die overhead for PNM.

🍯 Key takeaways

  • RALM has two memory-bound bottlenecks: GEMV-based attention (unique K,V per request) and IVF-PQ retrieval (LUT lookups + sorting).
  • PIM (in-bank MAC units) accelerates attention by exploiting high internal bandwidth; PNM (logic-die accelerators) handles retrieval.
  • Dual row buffers allow PIM and PNM to run concurrently, overlapping token generation and vector search.
  • Selective batching + early generation keeps the system busy: requests resume individually, and waiting requests generate extra tokens.
  • MNM achieves up to 29.2× speedup, 71.5% energy reduction, and scales to 310B parameters with 15% core die + 2% logic die overhead.
  • Early generation is viable because MNM shrinks retrieval latency enough that extra tokens don't push perplexity beyond acceptable bounds.
Seven everyday pictures

How to think about MNM

Each picture stands alone. Every card ends with an honest note about where the analogy breaks.

Core idea · Split by location

Two specialists in one building

Imagine a library with two staff: one shelves books in the stacks (PIM, in the core), one handles patron queries at the front desk (PNM, on the logic die). Each stays where the data lives. PIM works in the banks with tons of books; PNM sits near the catalog with lookup tools. MNM is that building design: put compute where bandwidth or logic complexity fits best.

Where it breaks: Libraries have physical constraints; MNM's constraint is fabrication process (DRAM for PIM, standard logic for PNM), not just location.
PIM · All-bank parallelism

Assembly line with parallel stations

Each DRAM bank is a worker on an assembly line. PIM broadcasts a query vector down the line (global bus), and every worker dot-products it with their own row of data. All workers compute simultaneously — 128 banks × 16 channels = massive parallelism. This is why GEMV (Q·K) runs so fast: the work splits across all banks, each using its own high-speed row buffer.

Where it breaks: Assembly lines produce identical outputs; PIM workers produce partial dot-products that sum into the final attention scores.
PNM · PQ code scanning

Recipe lookup in a cookbook

Each PQ codeword is like a recipe ID: "sub-vector 0 = entry 42, sub-vector 1 = entry 7…" The PNM logic flips to those pages in a precomputed cookbook (LUT) and reads the distances, then sums them into a total. Because the cookbook lives on the logic die (fast SRAM), lookups are cheap. PNM does 16 recipes in parallel (one per scanner), churning through millions of candidates.

Where it breaks: Cookbooks are static; the precomputed table is cluster-dependent and updated per retrieval, but the lookup flow is the same.
PNM · Top-k sorting

Sifting the winners

After computing distances for a batch of PQ codes, PNM's top-k sorter uses odd-even merge sort to keep only the k closest. Think of it as a sieve that holds 16 candidates at a time, compares pairs, and swaps them into order. Each batch is merged with the running top-16 register, and the worst are discarded. By the end, only the nearest neighbors remain.

Where it breaks: A sieve is passive; PNM's sorter is active hardware (compare-and-swap units) that processes 16 distances per cycle at 650 MHz.
Dual row buffer · Concurrency

Two lanes on a highway

Each bank has two row buffers: one for PIM operations (holding a K or V row for attention), one for regular I/O (reading PQ codes for retrieval). PIM and PNM can thus run simultaneously without conflicts — like two lanes of traffic moving in parallel. This is what lets MNM overlap token generation (PIM) and retrieval (PNM), hiding latencies.

Where it breaks: Highway lanes carry independent cars; the two buffers serve related but distinct memory requests (MAC data vs. read data). Also, dual buffers add ~3.8 mm² area overhead per core die.
Selective batching · Resume individually

Checkout lines, not a single queue

In standard batching, all requests wait for the slowest retrieval to finish before the batch resumes generation — like everyone queueing behind the slowest shopper. Selective batching lets each request resume generation the moment its own retrieval completes, like separate checkout lanes. No one waits for someone else's slow scan.

Where it breaks: Checkout lanes are independent; MNM's batch still shares GPU resources (SMs, memory controllers), but the scheduler dynamically regroups requests rather than forcing synchronization.
Early generation · Fill idle cycles

Prep work while you wait

If request B's retrieval is running, requests A and C can generate a few extra tokens ahead of schedule — like chopping vegetables for the next dish while the oven is busy. Because MNM's retrieval is fast, those extra tokens don't push the actual retrieval interval too far, so accuracy stays high. The GPU stays busy instead of idling.

Where it breaks: Chopping ahead doesn't depend on the oven's speed; early generation's viability hinges on MNM shortening retrieval enough that perplexity increase remains small (Figure 8 trends). On a slow retriever, too many early tokens would degrade quality.
The full explanation

How it actually works

Section by section, with the paper's key details rebuilt as plain-English steps.

1 · The problem — two memory-bound bottlenecks

Profiling RETRO (0.5B–7.5B parameters) on an H100 NVL GPU with IVF-PQ retrieval (configs 1–5, nlist from 32768 down to 2048, nprobe from 16 up to 256) reveals that as retrieval quality increases (smaller nlist, larger nprobe), the retriever portion grows from 4.4% (Config 1) to 88.5% (Config 5) of E2E latency. Within the language model, Multi-Head Attention (MHA) dominates: self-attention and chunked cross-attention together consume 54.8%–70.4% of LM time. These layers generate unique K and V matrices per request, forcing GEMV operations (Q·KT and softmax·V) that have low arithmetic intensity (~10-1 Op/B for MHA) and sit in the memory-bound region of the H100 roofline.

On the retriever side, PQ code scanning and top-k selection together take 44.2%–69.5% of retrieval time. PQ code scan performs LUT lookups per sub-vector to reconstruct distances (d = ||x−yC||² + precomputed term − 2x·yR), and top-k selection sorts millions of candidates to find the k nearest. Both exhibit even lower arithmetic intensity (~10-2 Op/B) and underutilize GPU compute (low TFLOPS). Batching retrieval is hard because each request accesses unique clusters, and sequential retrieval creates synchronization stalls.

🌱
Concretely: In Config 5 with RETRO-0.5B, retrieval takes 88.5% of time. MHA (Q·K and softmax·V) is 70.4% of the LM. Top-k selection alone is 29.6% of retrieval. All are memory-bound, but MHA needs bandwidth (PIM solution), while retrieval needs specialized logic (PNM solution).

2 · Design decision — where to put the logic

The paper chooses to place PIM logic (simple 16-bit FP MAC units) in each DRAM bank on the core die, because attention's GEMV (Q·K, softmax·V) benefits from high internal bandwidth but requires only basic MAC operations. Complex operations like softmax (exponent, division) are offloaded to a Softmax Calculator on the PNM logic die, which is fabricated with a standard 7nm process and can support more transistors.

For retrieval, placing PQ code scanners in the banks (PIM) would require storing a full copy of the PQ codebook and precomputed tables in each bank group, incurring ~68.8 mm² area overhead per core die and 4.1 W extra power (totaling 32.8 W for 8 core dies). Instead, MNM puts PQ Code Scanners and a Top-k Selector on the PNM logic die (7nm), where a single shared codebook can serve all channels via the high aggregate bandwidth of HBM (up to 4 TB/s in H100 NVL).

To avoid row buffer contention between PIM and PNM, MNM adopts a dual row buffer architecture: one buffer for PIM MAC operations, one for normal I/O and PNM reads. This costs ~3.8 mm² per core die but enables concurrent PIM (attention) and PNM (retrieval) operations.

🍯
Area overhead summary: PIM (MAC + dual row buffer) = 18.2 mm² per core die (15.0% of 121 mm²). PNM (scanners + sorter + softmax) = 2.4 mm² on the logic die (2.0% of 120 mm² available). Total per-stack overhead distributed across dies.

3 · MNM PIM — accelerating attention GEMV

Each HBM bank integrates a 16-bit FP MAC unit that reads 16 FP16 elements from the PIM row buffer and 16 from a global vector buffer, then performs a dot-product. For self-attention, the K matrix (rows of keys) is stored in memory such that each K column (across tokens) becomes a row in a bank. To compute Q·KT, the GPU writes Q to the global vector buffer via PIM_WR_GB, then issues PIM_ACT_AB to activate the K row in each bank, followed by PIM_MAC_AB to compute the dot-product across all banks in parallel. The partial sums accumulate in a result register, which is then transferred to the softmax calculator on the PNM die via PIM_MV_SB. After softmax, the output is written back to the global buffer, and another PIM_MAC_AB pass computes the context vector (softmax·V).

Chunked cross-attention works similarly, except the K and V matrices are computed from encoded retrieval data and cached for the retrieval interval. Because each request has unique K and V, batching is limited — but PIM's all-bank parallelism (128 banks × 16 channels) compensates by exploiting internal bandwidth (e.g., 8× higher than off-chip for HBM3).

💡
Why this works: GEMV is memory-bound. PIM moves the compute to where the data lives (row buffer), so each bank reads locally at high speed instead of shipping data off-chip. The dual row buffer lets PIM continue MAC operations while PNM reads PQ codes from other rows.

4 · MNM PNM — accelerating retrieval

Cluster selection (coarse search) runs on the GPU: compute ||x−yC||² for each cluster centroid and select the top nprobe clusters. Before starting PQ code scan, the host writes the query x and ||x−yC||² for each selected cluster into a Query Register on the PNM die (via PNM_WR_MMIO), and populates the Precomputed Table (||yR||² + 2(yC·yR)) for that cluster.

PQ code scan: The GPU issues PNM_RET_INIT, which loads PQ codewords from all 16 HBM channels into a PQ code buffer on the logic die (1024 B every 4 cycles). Each PQ codeword consists of M indices (M=64 sub-vectors), each 1 B. The PNM has 16 PQ Code Scanners, each with 24 MAC units for 384-D FP16 dot-products. Each scanner looks up the precomputed LUT for each sub-vector index, reconstructs yR by concatenating sub-vector centroids, computes x·yR, and sums it with the precomputed term and ||x−yC||² from the Query Register to get the final distance d.

Top-k selection: The 16 scanners produce 16 distances per cycle. These are fed into a Full Sorter (odd-even merge sort on 16 elements), then a Partial Sorter merges the sorted batch with the current top-16 from a Top-k Register, keeping only the 16 smallest. After scanning all PQ codes in a cluster, the GPU reads the top-16 distances and IDs, then moves to the next cluster. Final top-k selection (across all nprobe clusters) happens on the GPU.

🍯
Why precomputed LUT: Without LUT, each distance calculation would require 2M FP ops (per sub-vector dot-product and norm). With LUT, it's M lookups + 1 sum. The paper shows EDAP (Energy-Delay-Area Product) improves 13%–36% with LUT, and thermal rise is only 0.38–0.47°C.

5 · Scheduling — selective batching + early generation

In standard RALM, all requests wait for the slowest retrieval to finish before resuming generation, causing idle GPU cycles. MNM uses selective batching: each request resumes generation individually as its retrieval completes, forming dynamic batches. Early generation goes further: if request B's retrieval is running, requests A and C can generate extra tokens ahead of schedule, filling GPU cycles. The number of early tokens is bounded by the system's measured per-token generation time Tgen and retrieval time Tret,max: Nret = ⌊Tgen/Tret,max⌋ retrievals can fit in one generation window.

The Request Manager maintains a Retrieval Group Queue (groups of Nret requests each) and a Request Pool tracking early tokens per request. After each retrieval round, the manager calls updateQueue() to reorder the queue: requests with more early tokens (closer to needing retrieval) move to the front. The host issues MNM::Ret API for the next group's retrieval, and MNM::Gen API for batched generation.

⚠
Accuracy trade-off: Early generation extends the effective retrieval interval. However, because MNM shortens retrieval latency (e.g., ~8× vs. GPU for Config 5), the number of early tokens remains small. Figure 8 trends suggest perplexity increases remain modest at batch size 256, which is acceptable.
Follow the numbers, step by step

Worked examples

Four traces, each carried through — PIM MAC, PNM scan, top-k selection, and scheduling.

① PIM attention GEMV (Q·KT for one head)

384-D embeddings, 16-way attention, seq_len = 2048. How does PIM compute Q·KT for one head?

  1. Split Q by head: 384 / 16 = 24 dims per head. Q vector for head h is 24 FP16 elements.
  2. K matrix stored in banks: K is 2048×24 (seq_len × head_dim). Each row of K (24 elements, one token's key) is stored across banks. With sufficient banks (128 per channel × 16 channels), K can be distributed row-wise.
  3. Write Q to global buffer: GPU issues PIM_WR_GB, broadcasting Qh (24 elements) to all banks.
  4. Activate K rows: PIM_ACT_AB loads K rows into each bank's PIM row buffer. Each bank now has 16 FP16 elements (partial row).
  5. Compute dot-product: PIM_MAC_AB triggers each bank's MAC unit to multiply Qh × K row and accumulate. This runs in parallel across 128 banks (8 groups × 4 banks × 4 pseudochannels) per HBM channel.
  6. Aggregate results: Each bank's partial sum (Qh · Ki) is collected, summed across banks, giving one attention score per token. Repeat for all 2048 tokens.
  7. Speedup: PIM is memory-bound, not compute-bound, so the gain comes from avoiding 2048× off-chip reads of K rows. Internal bandwidth is ~8× higher than off-chip, so Q·K latency drops substantially.
🍯
Why it works: PIM moves compute to the data (row buffer), exploiting the massive parallel bandwidth of all banks. Each bank reads its K row locally at high speed, avoiding the off-chip bottleneck.

② PNM PQ code scan (computing distance for one vector)

384-D database vectors, M=64 sub-vectors (6-D each), ksub=256 codebook entries. Query x, selected cluster yC, PQ codeword = [42, 7, 191, …] (64 indices). How does PNM compute d?

  1. Precompute LUT: Before scan, host writes ||x−yC||² = 5.2 (computed on GPU) to Query Register. For each sub-vector position m, compute the precomputed term: LUT[m, idx] = ||yR,sub_m||² + 2(yC,sub_m · yR,sub_m) for all 256 codebook entries. Example: LUT[0, 42] = 3.1.
  2. Load PQ code: The PNM logic reads the codeword [42, 7, 191, …] from HBM into the PQ code buffer (1024 B every 4 cycles, so 64 indices fit rapidly).
  3. Lookup + reconstruct yR: For sub-vector 0, index 42 → look up codebook entry 42 (a 6-D FP16 vector). For sub-vector 1, index 7 → entry 7. Concatenate all 64 sub-vector centroids → reconstructed yR (384-D).
  4. Compute x·yR: The PNM PQ scanner has 24 MAC units (for 384-D FP16 dot-product). Compute x·yR = 12.7.
  5. Assemble distance: d = ||x−yC||² + LUT_sum − 2(x·yR). LUT_sum = sum over 64 sub-vectors of LUT[m, codeword[m]]. Example: 5.2 + 98.4 − 2(12.7) = 78.2.
  6. Speedup: Without LUT, each sub-vector distance would require 2M FP ops (dot-product + norm). With LUT, it's M lookups (64) + 1 FP addition. LUT is in SRAM (7nm logic die), so lookups are fast. The paper reports EDAP improvements with LUT.
💡
Why precomputed LUT wins: The paper measured EDAP (Energy-Delay-Area Product) for 192–1536-D embeddings. With LUT, EDAP improves 13%–36% because LUT storage cost (ksub×M floats) is smaller than the saved MAC operations. Thermal rise is only 0.38–0.47°C.

③ Top-k selection (sorting 16 distances, merging with top-16)

16 PQ scanners produce 16 new distances: [78.2, 65.1, 92.3, …, 54.7]. Current top-16 register holds [50.0, 51.2, …, 80.1]. How does the sorter update top-16?

  1. Full sort: The 16 new distances are fed into the Full Sorter (odd-even merge sort). It compares pairs (0 vs 1, 2 vs 3, …), swaps if out of order, then compares odd-even pairs, repeats. After log₂(16) = 4 stages, the 16 distances are sorted: [54.7, 65.1, 78.2, …, 92.3].
  2. Partial merge: The Partial Sorter merges the sorted 16 with the top-16 register (also sorted). It has 32 inputs, performs a merge (compare first elements, take smaller, advance pointer), producing 32 sorted distances.
  3. Keep top-16: Discard the worst 16, update the top-16 register with the 16 smallest. Example: if old top-16 was [50.0, …, 80.1] and new sorted is [54.7, …], merged result is [50.0, 51.2, 54.7, …]. The worst 16 (> 80) are dropped.
  4. Repeat: For each batch of 16 distances from PQ scanners, repeat steps 1–3. After scanning all nprobe clusters × MaxClusterSize vectors, the top-16 register holds the 16 nearest neighbors.
  5. Final top-k on GPU: The GPU reads the top-16 from each cluster (nprobe = 256 → 256 × 16 = 4096 candidates) and does a final sort to select top-k (e.g., k=16 for Config 5, Table 1). This is a small GPU kernel (4096-element sort), much cheaper than sorting all 140M vectors.
🍯
Why sorter is fast: Odd-even merge sort parallelizes compare-and-swap operations. Each stage runs rapidly at 650 MHz. GPU-based sorting would require shipping data to SMs and back, adding latency.

④ Scheduling (selective batching + early generation for batch size 32)

Batch size = 32 requests, Config 5 (nprobe=256), RETRO-1.5B. Tgen is the per-token generation time after the first token, Tret,max is the longest retrieval in the batch. How does scheduling work?

  1. Form retrieval groups: Nret = ⌊Tgen / Tret,max⌋. If Tgen (per-token time) × retrieval_interval fits multiple retrievals, groups can be formed accordingly. Assume Nret = 2 for this example, so 32 requests → 16 groups of 2 requests each (Gr 0 to Gr 15).
  2. Initial retrieval queue: [Gr 0, Gr 1, Gr 2, …, Gr 15]. On-Ret Gr = Gr 0. Host calls MNM::Ret for Gr 0 (2 requests), GPU issues PNM commands. Tret for Gr 0 varies by cluster.
  3. While Gr 0 retrieves: Remaining 30 requests (Gr 1–15) can generate tokens. If Tgen allows, they can generate extra tokens (early generation) filling GPU cycles while waiting.
  4. Gr 0 finishes retrieval: Host reads top-k IDs, looks up raw data, calls MNM::Gen for Gr 0 + batched ongoing generation. GPU generates next tokens for all active requests. Request Manager updates On-Ret Gr ← Gr 1.
  5. Reorder queue: After all groups complete one round, updateQueue() checks the Request Pool: which groups have the most early tokens? Groups with more early tokens (closer to needing retrieval) move to the front. Reorder queue accordingly.
  6. Next round: On-Ret Gr moves to the next group. Host calls MNM::Ret. Cycle repeats. MNM's fast retrieval bounds early token count, keeping perplexity increases modest.
⚠
Accuracy check: Early generation extends retrieval interval. With MNM's fast retrieval, Figure 8 trends suggest perplexity increases remain acceptable at batch size 256. On GPU, retrieval is slower, so early tokens would degrade perplexity more.

Problem 1 · easy
Why can't you batch MHA (Q·KT) into a fat GEMM across requests in RALM?
Show answer
Because each request has unique K and V matrices (specific to that input sequence). You can't stack them into one big matrix and GEMM, because the K rows differ per request. Instead, you're forced to do GEMV (one Q vector × one K matrix) per request, which is memory-bound and has low data reuse. This is why PIM helps: it exploits internal bandwidth to accelerate GEMV, rather than trying to make GEMM faster.Misconception: that batching always works. It does for FFN (same weight matrices WFFN) but not for MHA (unique K, V per request).
Problem 2 · medium
The paper reports PIM has 16-bit FP MAC units at 650 MHz. Each bank MAC reads 16 FP16 elements. With six MNM-enabled HBMs connected to an H100 NVL, describe the PIM architecture per Table 3.
Show answer
Table 3 specifies the HBM organization: each HBM has 16 channels, each channel has 8 bank groups, each bank group has 4 banks. So one HBM has 16 × 8 × 4 = 512 banks. Each bank has one MAC unit reading 16 FP16 elements per operation. With six HBMs, total banks = 6 × 512 = 3072 banks. The MAC units run at 650 MHz, enabling massive parallelism for GEMV operations across all banks.Misconception: that PIM needs more compute. It needs more bandwidth, which all-bank parallelism provides.
Problem 3 · medium
Why does MNM use a precomputed LUT for IVF-PQ retrieval instead of computing distances on-the-fly? Give one number from the paper to support.
Show answer
Without LUT, each sub-vector distance requires 2M FP ops (dot-product for x·yR,sub + norms). With LUT, it's M lookups (each fast from SRAM) + 1 FP addition. The paper measured EDAP (Energy-Delay-Area Product) and found improvements of 13%–36% across 192–1536-D embeddings (Fig. 5). Thermal rise is only 0.38–0.47°C, well within HBM3's 105°C limit. So LUT storage cost (ksub×M floats) is smaller than the saved operations.Misconception: that LUT is always free. It costs area (PQ codebook + precomputed table on logic die), but the energy-delay product improves, so it's worth it.
Problem 4 · hard
The dual row buffer costs ~3.8 mm² per core die. With 8 core dies per HBM stack, that's ~30 mm² total. Why is this overhead acceptable?
Show answer
Because it enables concurrent PIM (attention) and PNM (retrieval) operations. Without dual buffers, PIM MAC operations would conflict with PNM reads, forcing serialization. The paper shows that overlapping generation and retrieval via selective batching + early generation improves E2E speedup by up to 29.2× (Config 5, 140M dataset). The 30 mm² overhead (~3.1% per core die of 121 mm²) is far less than the performance gain. Also, MNM reduces E2E latency enough that early generation doesn't degrade perplexity significantly (Figure 8 trends), so the system stays accurate.Misconception: that dual buffers are just for performance. They're essential for the scheduling strategy; without them, MNM would be serial, like GPU.
Problem 5 · medium
Early generation extends the effective retrieval interval. For RETRO-1.5B, batch size 256, Config 5 (nprobe=256), explain why perplexity increase is acceptable, referencing MNM's speedup.
Show answer
MNM's retrieval is ~8× faster than GPU for Config 5 (PNM-based PQ scan + top-k vs. GPU kernels). So even though early generation allows extra tokens per request, the actual retrieval interval stays closer to the configured value than on GPU (where slow retrieval would push intervals much higher). Figure 8 trends show that at batch size 256, MNM-based early generation keeps perplexity increases modest compared to GPU-based early generation. The bounded early token count maintains higher accuracy than GPU despite using early generation.Misconception: that early generation always degrades accuracy. It does if retrieval is slow (GPU), but MNM's fast retrieval bounds the number of early tokens, keeping perplexity low.
Problem 6 · hard
The paper projects MNM to 310B-parameter RETRO models and claims 2.95× speedup over 8× GB200 GPUs. Explain the main bottleneck in the 8× GPU baseline that MNM avoids.
Show answer
In the 8× GB200 baseline, MHA operations (Q·KT, softmax·V) for a 310B model require distributing K and V matrices across GPUs and aggregating results via NVLink. The paper notes (Fig. 13b) that inter-GPU communication for MHA constitutes up to 31.0% of latency in the 8× GPU system. MNM offloads MHA to PIM, which runs locally in each HBM stack without inter-GPU transfers. For retrieval, the 8× GPU system batches requests across GPUs, but each GPU still runs slow IVF-PQ kernels; MNM uses PNM logic to accelerate PQ scan and top-k, reducing retrieval latency. The 2.95× speedup comes from both avoiding MHA communication overhead and accelerating retrieval.Misconception: that more GPUs always help. For memory-bound MHA, adding GPUs adds communication overhead (NVLink latency), which MNM avoids by keeping compute in-memory.
Problem 7 · easy
Why does MNM place PQ code scanners on the PNM logic die instead of in each PIM bank?
Show answer
Placing PQ scanners in each bank (PIM) would require storing a full copy of the PQ codebook and precomputed tables in every bank group, incurring ~68.8 mm² area overhead per core die and 4.1 W extra power (totaling 32.8 W for 8 core dies). Instead, MNM puts scanners on the PNM logic die (7nm), where a single shared codebook serves all channels via HBM's high aggregate bandwidth (4 TB/s in H100 NVL). This reduces area overhead to ~1.0 mm² for 16 scanners on the logic die (2% of 120 mm² available), and power to ~0.99 W.Misconception: that PIM should handle everything. PIM is DRAM process, limited to simple logic (MAC units). PNM (standard logic process) can implement complex scanners and sorters.
Problem 8 · medium
The paper uses selective batching. Compare its behavior to standard batching when one retrieval is 3× slower than the others.
Show answer
In standard batching, all 32 requests wait for the slowest retrieval (3× slower) to finish before resuming generation, wasting GPU cycles. If the slow retrieval takes 60 ms and others take 20 ms, the batch stalls for 60 ms. In selective batching, each request resumes generation individually as its retrieval completes. The 31 fast requests resume after 20 ms and generate tokens; only the 1 slow request waits 60 ms. Meanwhile, early generation allows the 31 to produce extra tokens while the slow one retrieves. Total E2E time for selective batching ≈ 60 ms (slow retrieval) + generation time for 32 requests (batched), vs. standard ≈ 60 ms + generation time (serial). Speedup depends on batch size, but selective batching reduces idle time from 31×40 ms (wasted) to near zero.Misconception: that batching is all-or-nothing. Selective batching allows partial batch progress, hiding retrieval latency variations.
Quick answers

Frequently asked questions

Twelve questions newcomers ask most. Tap any to expand.

Q1What is the fundamental difference between PIM and PNM?▾
PIM (Processing In Memory) places compute in the DRAM banks on the core die, fabricated with DRAM process. It's limited to simple logic (e.g., MAC units) but exploits high internal bandwidth (8× off-chip). PNM (Processing Near Memory) sits on the HBM logic die, fabricated with standard logic process (e.g., 7nm), so it can implement more complex circuits (sorters, scanners) and uses the high aggregate bandwidth of HBM (4 TB/s). MNM uses both: PIM for bandwidth-hungry GEMV (attention), PNM for logic-heavy retrieval (IVF-PQ).
Q2Why can't GPU handle RALM efficiently?▾
GPU's off-chip bandwidth (e.g., 3.35 TB/s for H100 NVL) is insufficient for RALM's memory-bound operations. MHA (Q·KT) generates unique K, V per request, forcing GEMV with low data reuse. IVF-PQ retrieval involves frequent LUT lookups (scattered reads) and sorting (low GPU utilization). Both underutilize GPU's compute parallelism (low TFLOPS) and saturate memory bandwidth. MNM offloads these to PIM/PNM, which exploit higher bandwidth (internal for PIM, aggregate for PNM) and specialized logic (sorters).
Q3How does MNM avoid row buffer conflicts between PIM and PNM?▾
MNM adopts a dual row buffer architecture: one buffer for PIM MAC operations (holding K or V rows for attention), one for normal I/O and PNM reads (reading PQ codewords for retrieval). This allows PIM (attention) and PNM (retrieval) to run concurrently on different rows, enabling overlap of generation and retrieval. The dual buffer costs ~3.8 mm² per core die (~3.1% area overhead) but is essential for scheduling efficiency.
Q4What is the precomputed LUT in IVF-PQ, and why use it?▾
For each cluster, the precomputed LUT stores ||yR||² + 2(yC·yR) for all ksub=256 codebook entries × M sub-vectors. This term doesn't depend on the query x, so it's computed once before scanning. During scan, each PQ codeword index lookups the LUT to avoid recomputing norms and dot-products. The paper shows EDAP improves 13%–36% with LUT (Fig. 5), as lookup cost (fast from SRAM) is cheaper than 2M FP ops per sub-vector.
Q5How does selective batching differ from continuous batching in LLM serving?▾
Continuous batching (Orca-style) lets new requests join the batch as soon as a slot frees up (e.g., when a request finishes). Selective batching in MNM is similar but focuses on retrieval completion: each request resumes generation individually as its retrieval finishes, rather than waiting for the whole batch. The paper extends the idea with early generation: requests waiting for another's retrieval generate extra tokens, filling GPU cycles. Both techniques aim to maximize throughput, but MNM's applies specifically to RALM's interleaved gen-ret phases.
Q6What limits the number of early-generated tokens per request?▾
The number is bounded by Nret = ⌊Tgen / Tret,max⌋, where Tgen is per-token generation time after the first token and Tret,max is the longest retrieval in the batch. Requests waiting can generate extra tokens during others' retrievals. Because MNM shortens Tret (e.g., ~8× vs. GPU), early tokens remain bounded, keeping perplexity increases modest (Figure 8 trends).
Q7Does MNM work for other RALMs besides RETRO?▾
The paper profiles RETRO specifically, demonstrating how MNM accelerates its two core bottlenecks: GEMV-based attention (self-attention, cross-attention) and IVF-PQ retrieval. The architecture targets these common patterns found in transformer-based RALMs. While the paper discusses broader applicability qualitatively, the empirical results focus on RETRO configurations. The scheduling strategy (selective batching + early generation) applies to retrieval-interleaved generation flows common in RALMs.
Q8How does MNM scale to larger models (e.g., 310B parameters)?▾
The paper projects MNM to 89B, 175B, and 310B-parameter RETRO models, comparing against 4× and 8× GB200 systems (Fig. 13b). For 310B, MNM achieves 2.95× speedup over 8× GB200 because: (1) PIM offloads MHA, avoiding inter-GPU communication (which is 31% of 8× GPU latency); (2) PNM accelerates retrieval, which scales linearly with database size (32B vectors). MNM distributes database clusters across HBM stacks, so retrieval parallelism scales with memory bandwidth, not GPU count.
Q9What is the area and power overhead of MNM?▾
PIM (MAC + dual row buffer) = 18.2 mm² per core die (15.0% of 121 mm²). With 8 core dies per stack, that's 145.6 mm² per HBM, but it's distributed (not on one die). PNM (scanners + sorter + softmax) = 2.4 mm² on the logic die (2.0% of 120 mm²). Power: PIM = 232.7 mW per core die (MAC 134.5 mW + dual buffer 98.2 mW). PNM = 1.14 W (scanners 990 mW + softmax 154 mW). Total ~5.35 W under max utilization (4.6% of 116 W HBM power budget).
Q10Does MNM improve TTFT (Time To First Token)?▾
Yes. TTFT = prefill latency + retrieval latency. MNM accelerates retrieval via PNM (PQ scan + top-k), reducing retrieval time by ~8× (Config 5). Prefill is GEMM-dominant (not GEMV), so PIM doesn't help prefill directly, but the paper shows TTFT speedup of 1.5–11× depending on input length (Fig. 11). At shorter inputs (32–128 tokens), retrieval is a larger fraction of TTFT, so PNM's gain is significant. At longer inputs (1024–2048 tokens), prefill dominates, and 2× GPU baseline catches up.
Q11Can MNM handle exact kNN search (not approximate)?▾
Yes, with modifications. Exact kNN doesn't use IVF-PQ (no clustering, no quantization), so you'd scan all vectors and compute exact distances (L2 or cosine). PNM's distance computation logic (MAC units for dot-products) can handle this, but the precomputed LUT wouldn't apply (no PQ codes). The top-k sorter still works. Exact kNN is more compute-intensive (no approximation), but PNM's parallel scanners and high aggregate bandwidth make it feasible for smaller databases (e.g., <10M vectors). For 140M vectors, approximate IVF-PQ is faster.
Q12What are the paper's future directions?▾
The paper's discussion mentions exploring diverse RALM paradigms beyond RETRO, exact kNN search alternatives to IVF-PQ, and cache systems such as RAG Cache and Cache-Craft that could complement MNM's acceleration. These represent potential directions for extending the architecture to broader retrieval-augmented applications.

Glossary & further reading

Every term in one place, then a path to go deeper.

Core terms
MNM (Mixed Near/In-Memory)The paper's heterogeneous architecture integrating PIM (in-bank MAC units) and PNM (logic-die accelerators) for RALM acceleration.
RALM (Retrieval-Augmented Language Model)A language model that queries an external database (vector search) during generation to ground outputs in verifiable, up-to-date information.
RETROGoogle DeepMind's RALM (2022), using IVF-PQ retrieval every chunk_size tokens. The paper profiles RETRO-0.5B to 7.5B parameters.
PIM (Processing In Memory)Compute integrated into DRAM banks on the core die. MNM uses 16-bit FP MAC units (650 MHz) to accelerate GEMV for attention.
PNM (Processing Near Memory)Specialized logic on the HBM logic die (7nm process). MNM uses PQ code scanners + top-k sorter for retrieval.
GEMV (General Matrix-Vector Multiplication)Q·KT for attention, where each request has unique K. Memory-bound (low arithmetic intensity ~10-1 Op/B), accelerated by PIM.
IVF-PQ (Inverted File index with Product Quantization)FAISS's approximate nearest-neighbor search. IVF partitions dataset into clusters; PQ compresses vectors into codes. Bottlenecks: LUT lookups + sorting.
Dual row bufferTwo row buffers per bank: one for PIM MAC operations, one for normal I/O / PNM reads. Enables concurrent PIM (attention) and PNM (retrieval).
Selective batchingEach request resumes generation individually as its retrieval completes, avoiding batch-wide synchronization stalls.
Early generationRequests waiting for another's retrieval generate extra tokens ahead of schedule, filling GPU cycles. Viable because MNM shortens retrieval latency.
Supporting terms
HBM3 (High Bandwidth Memory 3)Stacked DRAM with 8 core dies + 1 logic die. H100 NVL has 6 HBM3 stacks, 3.35 TB/s aggregate bandwidth, 94 GB capacity.
Core dieMemory die in HBM stack, fabricated with DRAM process. MNM places PIM logic (MAC units) here.
Logic die (buffer die)Bottom die in HBM stack, handles I/O and TSVs. Fabricated with standard logic process (7nm). MNM places PNM accelerators here.
Self-Attention (SA)Transformer layer computing Q·KT·V using input embeddings. In RETRO, each decoder block has SA + CCA + FFN.
Chunked Cross-Attention (CCA)RETRO-specific layer using Q from SA output and K, V from encoded retrieval data. Cached for retrieval_interval tokens.
PQ codebookCollection of sub-vector centroids (ksub=256 entries × M sub-vectors). Each PQ codeword is M indices pointing into the codebook.
Precomputed LUT||yR||² + 2(yC·yR) for all codebook entries × sub-vectors. Computed once per cluster before scanning, stored in SRAM on PNM die.
Top-k selectionSorting to find k nearest neighbors. MNM uses odd-even merge sort (parallel compare-and-swap) on PNM logic die.
Retrieval Group QueueQueue of Nret-sized groups of requests. Request Manager schedules retrieval sequentially through the queue, while generation overlaps.
NretNumber of retrievals that fit in one generation window: Nret = ⌊Tgen / Tret,max⌋. Used to partition batch into retrieval groups.
MMIO (Memory-Mapped I/O)Registers and LUTs on PNM die exposed as memory addresses. GPU writes query, precomputed table via PNM_WR_MMIO command.
PerplexityMetric for RALM quality: exp(−(1/N) Σ log P(token)). Lower is better. Early generation extends retrieval interval; Figure 8 shows perplexity trends.
