ISCA 2025 · A Field Guide

Mixing cheap DIMM with fast HBM to accelerate RAG systems

Retrieval-augmented generation demands both huge memory capacity (for knowledge bases) and high bandwidth (for random access and model inference). This is the plain-English story of HeterRAG, which combines DIMM-based and HBM-based PIM to deliver both — cheaply.

📄 "HeterRAG: Heterogeneous PIM Acceleration for RAG" · Liu et al. 🎓 Some familiarity with RAG and memory hierarchies assumed ⏱ ~25 min read
🧩

Use DIMM-based PIM for the retrieval stage (large capacity, random access) and HBM-based PIM for the generation stage (high bandwidth, sequential access) — scaling each independently instead of forcing both through a single expensive HBM stack.

Before you begin

Eight concepts the paper builds on

Each card covers one idea HeterRAG assumes. Read them in order the first time, or jump to what's unfamiliar.

DIMM-based PIM (retrieval, capacity) HBM-based PIM (generation, bandwidth) Hybrid design (both devices)
1What RAG is and why it matters▾

Retrieval-augmented generation (RAG) combines a retrieval system (which finds relevant documents from a knowledge base) with a language model (which generates answers). Unlike a pure LLM trained on static data, RAG can pull in up-to-date, domain-specific information at query time, making it more accurate for knowledge-intensive tasks.

RAG has two stages: (1) retrieval — find the top-k most relevant documents for a user query using approximate nearest-neighbor search; (2) generation — feed those documents plus the query into an LLM to produce a response.

Now you're ready: HeterRAG accelerates both stages — retrieval on DIMM-based PIM, generation on HBM-based PIM — instead of bottlenecking on a single memory type.
2ANNS and graph-based search▾

Approximate Nearest Neighbor Search (ANNS) finds vectors close to a query without checking every candidate. Graph-based ANNS (e.g. HNSW) organizes vectors as vertices in a graph, then traverses edges from a starting vertex, iteratively fetching neighbors, computing distances, and updating a priority queue until the top-k closest neighbors are found.

Each iteration involves three steps: (1) neighbor fetching — load a vertex's neighbors from memory, (2) distance computation — calculate the distance between the query and each neighbor, (3) queue updating — sort and keep the k-best candidates.

Now you're ready: The paper offloads distance computation — the most memory-intensive operation — to DIMM-based PIM, leaving neighbor fetching and queue updating on the host.
3Why LLM inference is memory-bound▾

Transformer-based LLMs (GPT, LLaMA) consist of decoder blocks, each containing multi-head attention and feedforward layers. Inference has two phases: prefilling (process all input tokens in parallel — dominated by matrix-matrix multiplication) and decoding (generate one token at a time autoregressively — dominated by matrix-vector multiplication).

Decoding is memory-bound because GEMV operations have low arithmetic intensity: weights are loaded once and used for a single output, so performance depends entirely on memory bandwidth. Limited on-chip cache can't hold all weights, forcing frequent off-chip accesses.

Now you're ready: HBM-based PIM alleviates this bottleneck by processing GEMV operations in-memory, near the data, instead of shuttling weights back and forth.
4Processing-in-Memory (PIM) basics▾

PIM places computation near or inside memory, reducing data movement and energy consumption. Digital PIM (the focus here) embeds processing units — small compute cores or arithmetic units — at different memory hierarchy levels (e.g. bank, rank, channel).

Two main flavors: DIMM-based PIM uses 2D DDR layouts, is cheaper and higher capacity but lower bandwidth; HBM-based PIM uses 3D-stacked memory, offers higher bandwidth and lower energy per bit but is expensive and limited in capacity.

Now you're ready: HeterRAG combines both: DIMM for storing large knowledge bases (cheap, scalable capacity) and HBM for bandwidth-hungry generation (fast, energy-efficient).
5HBM vs DDR — bandwidth, capacity, cost▾

An 8-Hi HBM2 stack delivers ~256GB/s bandwidth (13× faster than DDR4-2400's 19.2GB/s) and consumes roughly half the energy per bit transferred. But it provides only 16–24GB per stack, costs ~$110/GB, and scales expensively.

A DDR4 DIMM offers up to 64GB per module at ~$20/GB (one-fifth of HBM's cost). DDR5 further improves bandwidth while staying affordable. For a 100GB knowledge base, HBM would cost $11,000; DIMMs cost about $2,000.

Now you're ready: RAG's dual needs — large knowledge base storage and high-bandwidth inference — justify a heterogeneous design: use cheap DIMMs where capacity matters, HBM where bandwidth matters.
6Why a naive "DIMM + HBM" combo fails▾

You might think: store the knowledge base on DIMMs, copy data to HBM-based PIM for processing. But that transfer is constrained by the interconnect (e.g. PCIe 5.0 ×8 = ~32GB/s), which is far slower than HBM's internal bandwidth (~256GB/s).

For both retrieval and generation, moving data from DIMM to HBM-PIM becomes a new bottleneck, negating PIM's benefits. The paper's experiments show a naive HBM-only PIM baseline (with DIMM as external storage) performs worse than a CPU-GPU baseline.

Now you're ready: HeterRAG fixes this by putting processing where the data lives — retrieval on DIMM-PIM, generation on HBM-PIM — so data never crosses the slow interconnect during compute.
7CXL and device interconnects▾

Compute Express Link (CXL) is a high-speed interconnect for CPUs, accelerators, and memory. It supports three sub-protocols: cxl.io (PCIe-like device control), cxl.cache (coherent caching), and cxl.mem (memory pooling). Devices connect to a CXL switch, which scales to thousands of nodes.

In HeterRAG, both AccelDIMM and AccelHBM attach as CXL Type 2 devices, enabling flexible scaling: add more AccelDIMM devices to expand knowledge base capacity, add more AccelHBM devices to handle larger models or higher throughput.

Now you're ready: CXL lets HeterRAG scale heterogeneous PIM devices independently, unlike a monolithic GPU-centric design where everything must share the same memory type.
8Locality in RAG workloads▾

Real-world RAG exhibits strong locality: (1) a small fraction of documents (e.g. top 3%) account for most queries (power-law distribution), (2) in iterative RAG, successive queries reuse the same documents. Despite this, current systems recompute everything from scratch.

Document reuse means certain vectors are fetched repeatedly (in retrieval) and certain KV tensors are regenerated over and over (in generation). Caching frequently used data can drastically reduce memory access and computation.

Now you're ready: HeterRAG exploits locality with two optimizations: cache popular vertex vectors in AccelDIMM's RPMs, and cache KV tensors for popular documents in AccelHBM using a prefix tree + selective recomputation scheme.
The whole story in one place

HeterRAG in a nutshell

The problem, the heterogeneous solution, and how it all connects.

RAG systems face a dual bottleneck: retrieval suffers from random, irregular memory access (graph traversal has low bandwidth utilization), while generation is limited by GEMV's frequent weight loading (low data reuse). Both are memory-bound. Worse, RAG's knowledge bases can be hundreds of gigabytes to terabytes — far beyond what a single HBM stack can hold.

Existing HBM-based PIM works well for LLMs but fails for RAG: storing a 100GB knowledge base on HBM costs $11,000; using DIMM as external storage and copying data to HBM-PIM for processing creates a PCIe bottleneck (~32GB/s vs HBM's 256GB/s).

HeterRAG's solution: combine two types of PIM devices. AccelDIMM uses DIMM-based PIM for retrieval — it offers cheap, scalable capacity (64GB per module) and offloads distance computation to rank-level processing modules. AccelHBM uses HBM-based PIM for generation — it provides high bandwidth for GEMV and offloads these operations to bank-level processing modules. Each device type scales independently via CXL interconnect.

Three software-hardware co-optimizations further boost performance: (1) locality-aware retrieval caches frequently accessed vertex vectors and reuses search results from prior iterations as starting vertices; (2) locality-aware generation caches KV tensors for popular documents using a prefix tree + selective recomputation (storing dense KV for each document and sparse KV when it appears in a sequence); (3) fine-grained parallel pipeline starts generation as soon as partial retrieval results are available, overlapping the two stages.

The payoff: on 20 workloads, HeterRAG achieves 6.6–26.5× (13.5× avg) higher throughput, 8.7–27.6× (13.7× avg) lower latency, and up to 2.8× better energy efficiency vs CPU-GPU baselines. Area overhead is ~11.3% for AccelHBM's DRAM die and acceptable (~7.8mm² per DIMM, small vs typical >100mm² buffer chips) for AccelDIMM.

🍯 Key takeaways

  • RAG is memory-bound in both retrieval (random access → low BW utilization) and generation (GEMV → frequent weight loading).
  • RAG demands huge capacity for knowledge bases (100GB–TB), making HBM-only designs impractical ($11k for 100GB).
  • HeterRAG splits the workload: AccelDIMM (DIMM-based PIM) for retrieval, AccelHBM (HBM-based PIM) for generation.
  • Each device type scales independently via CXL; data never crosses the slow PCIe link during compute.
  • Locality-aware retrieval: cache popular vertex vectors; reuse prior search results as starting vertices.
  • Locality-aware generation: prefix tree + selective KV recomputation (dense KV per document, sparse KV per sequence).
  • Fine-grained pipeline: start generation on partial retrieval results, overlapping stages for better utilization.
  • Result: 13.5× avg throughput, 13.7× avg latency reduction, 2.8× energy efficiency vs CPU-GPU.
Six everyday pictures

How to think about HeterRAG

Each picture stands alone — read them in any order. Every card ends with an honest note about where the picture stops being true.

The core idea · Heterogeneous memory

A two-refrigerator kitchen

You have a large walk-in fridge (DIMM) that holds a week's groceries cheaply, and a small speed-fridge (HBM) next to the stove for instant access to tonight's ingredients. HeterRAG does this for data: store the massive knowledge base in cheap DIMM-PIM, keep hot inference weights in fast HBM-PIM. Each serves its purpose; neither bottlenecks the other.

Where it breaks: a kitchen has one cook; HeterRAG has independent devices processing in parallel, so the analogy undersells the concurrency and the CXL interconnect that orchestrates them.
Retrieval stage · Graph traversal

Hunting for treasure in a maze

Graph-based ANNS is like searching a maze where each room (vertex) has doors to a few neighbors. You start at an entrance, check each neighbor's distance to your target, pick the closest unchecked room, and repeat. The path is random and unpredictable — you can't prefetch the next room until you've checked the current one. DIMM-PIM handles this by offloading distance computation to rank-level modules right where the vertex data lives.

Where it misleads: a maze has walls; a graph's "neighbors" are just pointers in memory. The randomness is in the access pattern, not physical obstacles. And you're searching for similarity, not a fixed location.
Generation stage · GEMV bottleneck

Reading a huge encyclopedia one sentence at a time

Decoding is like writing an essay where you must consult an encyclopedia (model weights) for every single word. You read one page (a weight matrix), write one word (generate one token), then load the next page. The pages are too big to keep open all at once, so you're constantly fetching them. HBM-PIM fixes this by moving the computation to the pages themselves — read-and-compute in place.

Where it breaks: an encyclopedia is passive; HBM-PIM actively computes. And the "pages" (weights) are reused across tokens in a batch, so the analogy undersells the benefit of KV caching for multi-turn or multi-document scenarios.
Scaling · Independent growth

Adding more filing cabinets vs more CPUs

If your knowledge base grows from 100GB to 1TB, add more AccelDIMM devices (more filing cabinets) — no need to upgrade your generation hardware. If you need to serve a larger model or higher query throughput, add more AccelHBM devices (more CPUs) — no need to expand your knowledge storage. CXL lets you scale each resource independently, like a modular data center.

Where it breaks: filing cabinets are passive; AccelDIMM devices actively process queries in parallel. And CXL switch latency is non-zero, so scaling isn't perfectly linear (though the paper shows near-superlinear speedup for retrieval due to reduced per-device workload).
Locality-aware optimization · Caching hot data

Bookmarking your favorite pages

In a library, you don't re-shelve a book you'll reread tomorrow. HeterRAG caches frequently accessed vertex vectors (in retrieval) and KV tensors (in generation) so they're immediately available on the next query. For iterative RAG, it even reuses the previous search's results as starting points — like resuming from a bookmark instead of starting from page 1.

Where it misleads: a bookmark is static; HeterRAG's caches use LRU eviction and are dynamically updated. And the prefix tree for KV caching is more sophisticated than a simple "most-used" list — it shares common prefixes and selectively recomputes sparse KV.
Pipeline · Overlapping stages

Cooking dinner while the oven preheats

Strict sequential RAG waits for all retrieval to finish before starting generation — like waiting for the oven to fully preheat before you even start chopping vegetables. HeterRAG's fine-grained pipeline starts generation as soon as partial retrieval results arrive, overlapping the two stages. This keeps both AccelDIMM and AccelHBM busy, raising hardware utilization from ~32% to ~42%.

Where it breaks: cooking is serial (one person, one stove); HeterRAG has multiple independent devices. And the oven doesn't "send partial results" — but HeterRAG's host aggregates retrieval results at fixed intervals, so the analogy captures the spirit of not waiting idly.
The full explanation

How it actually works

Section by section, with the paper's key mechanisms and design choices spelled out.

1 · The problem — RAG's dual bottleneck

RAG systems run on two stages. Retrieval: graph-based ANNS traverses a graph index with random memory access, causing low bandwidth utilization despite needing to check many vertices. Generation: autoregressive decoding performs GEMV operations with limited weight/KV reuse, forcing frequent off-chip memory access. Both stages are memory-bound.

Worse, RAG's knowledge bases are massive. A Wikipedia corpus (tens of millions of documents) plus vectors and graph indices can be 100GB–1TB. Storing this on HBM costs ~$110/GB; a 100GB base would cost $11,000. Using DIMM as external storage and copying to HBM-PIM for processing creates a PCIe bottleneck (~32GB/s vs HBM's 256GB/s).

🌱
Paper measurement: on GPT2-XL with a 20M-entry knowledge base, retrieval takes 30–40% of execution time; with LLaMA2-7B and iterative RAG (16 iterations), retrieval dominates. Both need optimization. (Paper, Fig. 2)

2 · HeterRAG architecture — two device types

HeterRAG splits the workload across two PIM devices connected via CXL: AccelDIMM (DIMM-based PIM) for retrieval and AccelHBM (HBM-based PIM) for generation. The host orchestrates: it broadcasts queries to all AccelDIMM devices, aggregates top-k results, maps vector IDs to documents, and sends documents to AccelHBM for generation.

Both device types scale independently. Add more AccelDIMMs to expand knowledge base capacity (data is partitioned at the vertex level across devices). Add more AccelHBMs to handle larger models (using tensor or pipeline parallelism). The CXL switch supports up to 4K nodes, ensuring scalability.

3 · AccelDIMM design — rank-level distance computation

AccelDIMM offloads distance computation (80%+ of retrieval time) to DIMM-based PIM, leaving neighbor fetching and queue updating on the host. Each AccelDIMM has multiple memory channels, each with PIM-enabled DIMMs. Data is partitioned across ranks for balanced workload.

Hardware components: (1) Top-level Processing Module (TPM) — decodes host instructions, manages neighbor fetching and queue updating; (2) DIMM-level Processing Module (DPM) — dispatches PIM instructions to ranks, buffers results; (3) Rank-level Processing Module (RPM) — performs distance computation (inner product), includes a vertex cache to reduce DRAM access for hot vectors.

💡
Why rank-level not bank-level? Graph traversal's irregular access leaves many banks idle if computation is per-bank. Rank-level deployment (one RPM per rank) balances area overhead with utilization. (Paper, §4.2)

4 · AccelHBM design — bank-level GEMV

AccelHBM offloads GEMV operations (the bottleneck in decoding) to HBM-based PIM, handling GEMM and other ops (LayerNorm, Softmax) in the top-level processing module. Each AccelHBM has multiple HBM channels, each with PIM-enabled HBM.

Hardware components: (1) Top-level Processing Module (TPM) — includes a systolic array for GEMM, VLIW processors for other ops, scratchpad memory, and custom units (tree search, KV substitution, token filtering) for locality-aware generation; (2) Bank-level Processing Module (BPM) — performs GEMV (two vector inner product units per BPM), reads from row buffer and global buffer.

🍯
Area overhead: one BPM per two banks adds ~6.016mm² on DRAM die (~11.31% of 53.15mm² HBM2). For AccelDIMM, each RPM is ~3.4mm², each DPM ~0.8mm², totaling ~7.817mm² per DIMM — acceptable given typical buffer chip area >100mm². (Paper, §5.6)

5 · Locality-aware retrieval — caching and reuse

Real-world RAG exhibits strong locality: 60% of queries hit the top 3% of documents. HeterRAG exploits this with two techniques: (1) Vertex cache in each RPM (LRU policy) — frequently accessed vertex vectors stay on-chip, avoiding DRAM reads; (2) Reuse prior search results — in iterative RAG, the result buffer from iteration N becomes the initial buffer for iteration N+1, so the search starts near the answer instead of from scratch.

The cache is split into local (per-query) and global (merged after query completes) parts to avoid polluting global cache with one-off accesses.

6 · Locality-aware generation — prefix tree + selective KV

Naive KV caching stores a separate copy for each sequence, wasting memory when documents appear multiple times. HeterRAG uses a prefix tree to share KV across sequences with common prefixes. For each document, it stores: (1) dense KV — computed once based on the document alone, independent of sequence; (2) sparse KV — a small subset (10–20% of tokens) recomputed when the document appears in a new sequence, capturing position-dependent attention.

When a new sequence {D2, D4, D6} arrives, the system searches the tree for matches. For matched documents, it loads dense + sparse KV and substitutes selectively. For unmatched documents, it checks if dense KV exists (recompute only sparse) or computes from scratch. Eviction uses LRU on sparse KV entries.

🍯
Why it works: sparse KV retains the 10–20% of tokens with the highest attention weights. Dense KV is still used; sparse KV selectively replaces certain token representations to capture position-dependent context shifts. This approach causes just 0.2% attention deviation — negligible impact on generation quality. (Inspired by prior work, cited in paper §4.4)

7 · Fine-grained parallel pipeline — overlap stages

Traditional RAG waits for all retrieval to finish before starting generation. HeterRAG starts generation as soon as partial results are available. The host aggregates retrieval results at fixed intervals: completed searches send all top-k results; in-progress searches send high-confidence results (those near the head of the result buffer, unlikely to change).

AccelHBM begins prefilling immediately on partial results. For completed searches, it proceeds to decoding. This overlap raises hardware utilization from 32–38% (sequential) to 39–45% (pipelined). (Paper, Table 3)

🌱
Software stack: HeterRAG extends ML frameworks with two APIs — ANNS-ACC and LLM-ACC — that abstract device management. Programs are compiled into host executables and device-specific binaries, dispatched via drivers. (Paper, Fig. 11)
Follow the numbers, step by step

Worked examples

Four traces using the paper's real data.

① HBM vs DIMM cost for a 100GB knowledge base

Calculate the hardware cost difference between HBM-only and DIMM-based storage for a 100GB knowledge base.

  1. HBM option: each 8-Hi HBM2E stack provides 16GB, costs ~$110/GB. For 100GB, need 100÷16 ≈ 6.25 stacks → 7 stacks. Cost = 7 × 16GB × $110/GB = $12,320.
  2. DIMM option: each DDR4 DIMM provides 64GB, costs ~$20/GB. For 100GB, need 100÷64 ≈ 1.6 modules → 2 modules. Cost = 2 × 64GB × $20/GB = $2,560.
  3. Cost ratio: $12,320 ÷ $2,560 ≈ 4.8×. HBM is nearly 5× more expensive for the same capacity.
  4. Capacity ceiling: HBM tops out at ~24GB per stack; scaling to 1TB requires dozens of stacks (impractical). DIMM scales to 64GB modules (128GB DDR5 modules are emerging but not cited in the paper).
🍯
Insight: the 5× cost difference motivates HeterRAG's heterogeneous design. Store large knowledge bases on DIMM-PIM, keep only hot inference data in HBM-PIM. (Paper, §1, §3.2)

② Bandwidth bottleneck in naive DIMM+HBM design

Why does the "naive" design (DIMM storage + HBM-PIM for processing) underperform? The paper supports the qualitative PCIe bottleneck (32GB/s vs 256GB/s internal HBM bandwidth) but does not provide specific per-query transfer measurements.

  1. Setup: naive design stores the knowledge base on DIMM but attempts to process retrieval on HBM-PIM by transferring vertex data over the interconnect.
  2. Interconnect bandwidth: PCIe 5.0 ×8 provides ~32GB/s. HBM's internal bandwidth is ~256GB/s — 8× faster.
  3. The bottleneck: transferring vertex data from DIMM to HBM-PIM over PCIe is 8× slower than HBM's internal processing speed, negating PIM's benefit.
  4. Paper evidence: experiments show NaiveHBM (DIMM storage + HBM-PIM) performs worse than CPU-GPU baseline because the transfer time dominates.
  5. HeterRAG fix: AccelDIMM processes retrieval on the DIMM side (no transfer), only sending tiny top-k results (8KB for 1000 entries) back to the host.
💡
The paper demonstrates the naive design underperforms due to the PCIe bottleneck; HeterRAG eliminates this by processing data where it lives. (Paper, §5.2, Fig. 12)

③ Throughput improvement from locality-aware retrieval

The paper reports an average 1.25× throughput gain from locality-aware retrieval optimizations across workloads (Fig. 15).

  1. Baseline: HeterRAG-base (no optimizations) serves as the reference for measuring locality-aware retrieval gains.
  2. Vertex cache hit rate: for power-law workloads, the 128KB cache (in each RPM) achieves measurable hit rates on frequently accessed vertices (paper cites 60% of queries hitting top 3% of documents).
  3. Reuse prior results: in iterative RAG (16 iterations), reusing the previous iteration's results as starting vertices reduces redundant graph traversal.
  4. Measured speedup: the paper reports locality-aware retrieval achieves a 1.25× average throughput gain across all workloads (Fig. 15).
🍯
Insight: real-world locality (60% of queries hit top 3% of documents) makes caching and result reuse highly effective. The paper measures 1.25× avg throughput gain from locality-aware retrieval. (Paper, §5.3, Fig. 15)

④ Energy efficiency comparison — HeterRAG vs CPU-GPU

The paper reports normalized energy reductions: 35.50–63.76% across workloads, with a 49.8% average reduction (Fig. 14).

  1. Measurement approach: the paper normalizes energy consumption to the CPU-GPU baseline (Intel Xeon Gold 5117 + NVIDIA V100).
  2. Energy reduction range: HeterRAG achieves 35.50–63.76% energy reduction depending on the workload mix (retrieval vs generation dominated).
  3. Average reduction: across all workloads (LLaMA2-7B, iterative and one-pass RAG), HeterRAG averages 49.8% energy reduction.
  4. Breakdown: retrieval saves energy via near-memory computation (no PCIe transfer); generation saves energy via HBM's low energy-per-bit (0.5pJ/bit vs DDR's 1pJ/bit).
  5. Best case: for one-pass RAG (generation-dominated), HeterRAG achieves up to 63.76% energy reduction. (Paper, Fig. 14)
🍯
Insight: PIM reduces data movement (the largest energy consumer in von Neumann architectures). HeterRAG averages 49.8% energy reduction across all workloads. (Paper, §5.2, Fig. 14)

Practice problems

Try each before opening the answer. Every one ends with the common misconception it checks.

Problem 1 · easy
HeterRAG uses DIMM-based PIM for retrieval and HBM-based PIM for generation. Why not use HBM-based PIM for both stages?
Show answer
Capacity and cost. A 100GB knowledge base on HBM costs ~$11,000 (at ~$110/GB); on DIMM it's ~$2,000 (at ~$20/GB) — a 5× difference. Retrieval's random access doesn't need HBM's full bandwidth, and DIMM provides adequate BW for graph traversal. Generation (GEMV) is more bandwidth-sensitive, justifying HBM's higher cost for that stage.Misconception: that "more bandwidth is always better." Retrieval is bottlenecked by random access patterns (low BW utilization), not raw BW. DIMM-PIM is sufficient and far cheaper.
Problem 2 · medium
The naive design stores knowledge base on DIMM but processes retrieval on HBM-PIM. Why does this underperform CPU-GPU? Identify the bottleneck.
Show answer
The PCIe interconnect. Transferring data from DIMM to HBM-PIM goes over PCIe 5.0 ×8 (~32GB/s), which is 8× slower than HBM's internal 256GB/s. The paper's qualitative analysis shows the transfer time dominates processing, negating PIM's benefit. HeterRAG fixes this by processing retrieval on the DIMM side (AccelDIMM), eliminating the transfer.Misconception: that "PIM always wins." PIM only helps if data stays near the compute. Moving data across a slow interconnect negates the benefit.
Problem 3 · medium
AccelDIMM deploys distance computation at the rank level, not bank level. Explain the trade-off and why rank-level was chosen.
Show answer
Graph traversal's irregular access patterns leave many banks idle if computation is per-bank. Deploying at bank level wastes area (too many compute units sit unused). Rank-level deployment (one RPM per rank) balances area overhead with utilization — multiple ranks process in parallel (per-channel), but each rank is fully utilized. This keeps area overhead at ~7.817mm² per DIMM while enabling effective parallelism.Misconception: that "finer-grained parallelism is always better." For irregular workloads, over-provisioning compute causes idle resources and wasted area.
Problem 4 · hard
Locality-aware generation uses a prefix tree to share KV across sequences. But sequences {D2,D3,D5} and {D2,D4,D5} share only D2. Does the prefix tree still help? Walk through how it works.
Show answer
Yes. The tree stores D2's dense KV (computed once) and sparse KV for each branch (D3 vs D4). When {D2,D4,D5} arrives, it finds D2 (load dense+sparse), finds D4 (load dense+sparse), but D5 is new. For D5, the system checks if dense KV exists elsewhere in the tree (from {D2,D3,D5}). If yes, it loads dense and recomputes only sparse (10–20% of tokens); if no, it computes from scratch. Either way, D2's KV is fully reused, and D5's dense might be reused, saving most computation.Misconception: that prefix trees only help with long common prefixes. The split into dense (document-only) vs sparse (position-dependent) KV lets the tree share even when sequences diverge early.
Problem 5 · medium
Fine-grained parallel pipeline starts generation on partial retrieval results. How does the host decide which partial results are "safe" to send?
Show answer
The host aggregates retrieval results at fixed intervals. For in-progress searches, it checks the result buffer: if a newly discovered vertex appears near the head (high confidence — unlikely to be displaced by future iterations), it's sent immediately. If a new vertex appears near the tail (low confidence — may be bumped by better candidates), the host waits. Completed searches send all top-k results. This heuristic balances early generation start (to overlap stages) with accuracy (avoiding sending stale results).Misconception: that partial results are "guesses." They're real top-k candidates at the current iteration, and for most queries the early candidates don't change much in later iterations (empirically validated in the paper's workloads).
Problem 6 · hard
HeterRAG achieves near-superlinear speedup when scaling AccelDIMMs (e.g. 4→8→16→32 devices). Explain why, even though there's no communication between AccelDIMMs.
Show answer
Data parallelism + reduced per-device workload. The knowledge base is partitioned across all AccelDIMMs. Each device processes the same query independently on its local data. Doubling the device count halves each device's data size, which non-linearly reduces search time because graph-based ANNS's iteration count grows sublinearly with dataset size (the graph is more connected in smaller subsets, so convergence is faster). Additionally, more devices reduce cache pressure per device, raising hit rates. Combined effect: slightly better than linear speedup.Misconception: that perfect data parallelism always gives linear speedup. ANNS's sublinear scaling with dataset size can yield super-linear speedup when partitioning across devices.
Problem 7 · medium
Selective KV recomputation keeps only 10–20% of tokens (sparse KV) and claims 0.2% attention deviation. Why doesn't this hurt generation quality?
Show answer
Attention sparsity. In transformer models, most tokens contribute negligibly to attention scores — only a small subset (high-attention tokens) actually matter. Sparse KV retains the 10–20% of tokens with the highest attention weights. Dense KV is still used; sparse KV selectively replaces certain token representations when the document appears in a new sequence. This preserves the important information while causing only 0.2% deviation (measured empirically). This is the key insight from attention sparsity research that HeterRAG leverages.Misconception: that sparse KV "discards 80–90% of tokens" entirely. Dense KV is still used; sparse KV selectively replaces high-attention tokens to capture position-dependent context shifts.
Problem 8 · easy
A user wants to scale HeterRAG to support a 2TB knowledge base and LLaMA2-70B. Which device type(s) need to be added, and how many?
Show answer
For the 2TB knowledge base, add more AccelDIMM devices. Each holds ~64GB (typical DIMM capacity), so need 2TB÷64GB = 32 AccelDIMMs. For LLaMA2-70B (~140GB in FP16), add AccelHBM devices. If each AccelHBM has 80GB HBM, need 140GB÷80GB = 2 AccelHBMs (using tensor parallelism). Or use 3 AccelHBMs with pipeline parallelism if latency is critical. The two scaling dimensions are independent — knowledge base size doesn't affect model size requirements.Misconception: that you must scale both device types together. HeterRAG's heterogeneity lets you scale retrieval (AccelDIMM) and generation (AccelHBM) independently based on workload.
Quick answers

Frequently asked questions

Twelve questions newcomers ask most.

Q1What is PIM and why does it help RAG?▾
PIM (Processing-in-Memory) places computation near or inside memory, reducing data movement. RAG is memory-bound in both retrieval (random access) and generation (GEMV). Moving compute to the data — instead of shuttling data to a distant processor — cuts latency, boosts bandwidth utilization, and saves energy. HeterRAG uses two PIM flavors: DIMM-based for retrieval (capacity + cost), HBM-based for generation (bandwidth + energy efficiency).
Q2Why not just use more HBM stacks for the knowledge base?▾
Cost and capacity limits. Each HBM2E stack provides ~16GB at ~$110/GB. A 100GB knowledge base costs $11,000; a 1TB base would cost $110,000 — impractical. HBM also caps at ~24GB per stack, so scaling to TB-scale requires dozens of stacks. DIMM provides 64GB per module at ~$20/GB, 5× cheaper. HeterRAG uses DIMM for storage, HBM only for hot inference data.
Q3How does HeterRAG avoid the PCIe bottleneck?▾
By processing data where it lives. AccelDIMM performs retrieval on the DIMM side, never moving vertex data to HBM. Only tiny top-k results (8KB for 1000 entries) cross the interconnect to the host. AccelHBM processes generation on the HBM side, where model weights reside. No bulk data transfers happen during compute. The CXL interconnect (~32GB/s) only handles orchestration and result aggregation.
Q4What is graph-based ANNS and why is it hard to accelerate?▾
Graph-based ANNS (e.g. HNSW) organizes vectors as vertices in a graph, then traverses edges to find nearest neighbors. It's hard to accelerate because: (1) the graph is sparse, so access patterns are unpredictable (can't prefetch); (2) each iteration depends on the previous one (limited parallelism); (3) distance computation dominates (80%+ of time) but involves scattered memory reads. HeterRAG offloads distance computation to DIMM-based PIM and caches hot vertices.
Q5Why deploy distance computation at rank level, not bank level?▾
Graph traversal's irregular access leaves many banks idle if computation is per-bank. Rank-level deployment (one RPM per rank) balances area overhead with utilization — multiple ranks process in parallel (per-channel), but each rank is fully utilized. Bank-level would waste area (too many idle compute units). Rank-level keeps area overhead at ~7.817mm² per DIMM while enabling effective parallelism.
Q6How does locality-aware generation reduce KV recomputation?▾
Prefix tree + selective KV. Each document stores dense KV (computed once, independent of sequence) and sparse KV (10–20% of tokens, recomputed per sequence to capture position-dependent attention). When a document appears in a new sequence, load dense KV, recompute only sparse. If the document matches a tree path, load existing sparse; if not, recompute it. This cuts KV recomputation significantly while causing only 0.2% attention deviation (due to attention sparsity).
Q7What is "fine-grained parallel pipeline" and how much does it help?▾
Traditional RAG waits for all retrieval to finish before starting generation. Fine-grained pipeline starts generation as soon as partial retrieval results are available (host aggregates results at fixed intervals). This overlaps the two stages, raising hardware utilization from 32–38% to 39–45%. The paper measures a 1.20× avg throughput gain from pipelining alone (Fig. 15). Combined with locality optimizations, total speedup is 1.25 × 1.22 × 1.20 ≈ 1.8×.
Q8How does HeterRAG scale to larger models like LLaMA2-70B?▾
Add more AccelHBM devices. LLaMA2-70B needs ~140GB (FP16). If each AccelHBM has 80GB HBM, use 2 devices with tensor parallelism (each computes partial results, then aggregates). Or use 3 devices with pipeline parallelism (each handles a different model stage). The paper evaluates both; tensor parallelism is default. Scaling retrieval (knowledge base size) is independent — add AccelDIMMs as needed.
Q9What if my model is too large for HBM capacity?▾
Three options: (1) add more AccelHBM devices (scales to arbitrary model size via tensor/pipeline parallelism); (2) use standard DIMMs as external memory (cheaper but introduces PCIe bottleneck); (3) combine both — store cold weights on DIMM, hot weights on HBM. The paper suggests (1) is preferred for performance; (2) is acceptable if cost is critical. For RAG, retrieval dominates memory needs, so AccelHBM count is typically modest.
Q10Does HeterRAG always beat CPU-GPU?▾
In the paper's evaluated configurations, HeterRAG consistently outperforms CPU-GPU, achieving 6.62–26.53× throughput improvement (13.5× avg) and 8.7–27.6× latency reduction (13.7× avg). The largest gains appear in iterative RAG with large knowledge bases (where retrieval dominates). One-pass RAG with small knowledge bases sees smaller but still substantial gains (6–8×). The paper does not report scenarios where HeterRAG underperforms CPU-GPU in its experimental setup.
Q11How does this compare to other RAG accelerators like Chameleon or PipeRAG?▾
HeterRAG outperforms both. Chameleon (FPGA+GPU) uses PQ-based ANNS (less accurate than graph-based) and doesn't address the memory bottleneck — HeterRAG achieves 6.17× higher throughput. PipeRAG (CPU+GPU) aggressively reuses retrieval results but still suffers from memory BW limits — HeterRAG achieves 5.48× lower latency. HeterRAG's PIM design fundamentally eliminates the memory bottleneck, while others optimize around it. (Paper, §5.5, Fig. 17)
Q12What did the authors leave for future work?▾
The paper's conclusion restates the HeterRAG architecture and its evaluation results. The authors do not explicitly enumerate future work directions in the paper text. Potential extensions could include applying HeterRAG to other workloads, exploring different PIM substrates, or optimizing for additional RAG variants, but these are not discussed as explicit future work items in the source material.

Core terms
RAG (Retrieval-augmented Generation)A method combining retrieval (find relevant documents) with generation (LLM produces answer), enabling real-time, domain-specific responses.
ANNS (Approximate Nearest Neighbor Search)Find vectors close to a query without checking every candidate. Graph-based ANNS uses a pre-built graph index to traverse efficiently.
HeterRAGThe paper's system: combines DIMM-based PIM (retrieval) and HBM-based PIM (generation) for high performance, capacity, and cost-efficiency.
AccelDIMMDIMM-based PIM device for retrieval. Offloads distance computation to rank-level processing modules; stores knowledge base.
AccelHBMHBM-based PIM device for generation. Offloads GEMV to bank-level processing modules; handles LLM inference.
PIM (Processing-in-Memory)Compute near or inside memory to reduce data movement. Digital PIM embeds processing units at memory hierarchy levels (rank, bank, channel).
CXL (Compute Express Link)High-speed interconnect for CPUs, accelerators, memory. Supports cxl.io (device control), cxl.cache (coherency), cxl.mem (memory pooling).
Locality-aware retrievalCache frequently accessed vertex vectors; reuse prior search results as starting vertices in iterative RAG.
Locality-aware generationPrefix tree + selective KV recomputation. Store dense KV (per document) and sparse KV (per sequence, 10–20% of tokens).
Fine-grained parallel pipelineStart generation on partial retrieval results, overlapping stages to raise hardware utilization from ~32% to ~42%.
Supporting terms
GEMV / GEMMGeneral Matrix-Vector / Matrix-Matrix Multiplication. GEMV dominates decoding (memory-bound); GEMM dominates prefilling (compute-bound).
KV cacheCached Key-Value tensors from prior tokens, enabling autoregressive decoding without recomputing attention for every token.
HNSW (Hierarchical Navigable Small World)A graph-based ANNS algorithm. Organizes vectors in a hierarchical graph, searches by traversing edges from a starting vertex.
RPM (Rank-level Processing Module)In AccelDIMM: performs distance computation, includes vertex cache. One RPM per rank.
BPM (Bank-level Processing Module)In AccelHBM: performs GEMV (vector inner product), reads from row buffer and global buffer. One BPM per two banks.
Dense KVKey-Value tensors computed once per document, independent of sequence. Shared across all sequences containing that document.
Sparse KVSubset of KV (10–20% of tokens with highest attention weights) recomputed per sequence to capture position-dependent attention. Stored per tree path.
Attention sparsityMost tokens contribute negligibly to attention scores; only a small subset matters. Enables selective KV recomputation with 0.2% deviation.
PCIe bottleneckTransferring data from DIMM to HBM-PIM over PCIe 5.0 ×8 (~32GB/s) is 8× slower than HBM's internal 256GB/s, negating PIM's benefit.
Power-law distributionIn RAG, a small fraction of documents (e.g. top 3%) account for most queries (e.g. 60%), enabling effective caching.

PIM eliminates data movement energy

HeterRAG reduces energy consumption by 35.5–63.76% (49.8% avg). HBM's low energy-per-bit (0.5pJ vs DDR's 1pJ) and near-memory compute cut total energy. (Paper, §5.2, Fig. 14)

4→32 AccelDIMM devices · 1B-entry knowledge base

Increasing AccelDIMMs from 4 to 32 yields better-than-linear speedup due to reduced per-device workload and ANNS's sublinear scaling with dataset size. (Paper, §5.4, Fig. 16)

AccelDIMM: 7.817mm² · AccelHBM: 6.016mm²

AccelDIMM adds ~7.817mm² per DIMM (acceptable given buffer chip >100mm²). AccelHBM adds ~6.016mm² per DRAM die (~11.31% of 53.15mm² HBM2). (Paper, §5.6)

Related work

TransPIM: HBM-based PIM for transformers

Zhou et al. · HPCA 2022

Early work on HBM-based PIM for LLM inference. HeterRAG extends this by adding DIMM-based PIM for retrieval and introducing heterogeneous scaling.

AttAcc: Attention acceleration with PIM

Park et al. · ASPLOS 2024

Optimizes attention mechanism using HBM-based PIM. HeterRAG adopts similar bank-level GEMV offloading for AccelHBM but adds retrieval acceleration and locality optimizations.

RecNMP: Near-memory processing for recommendation

Ke et al. · ISCA 2020

DIMM-based PIM for embedding lookups. HeterRAG borrows instruction compression techniques for AccelDIMM but targets graph-based ANNS instead of embedding tables.

Chameleon: Heterogeneous FPGA+GPU for RAG

Jiang et al. · arXiv 2023

Uses FPGA for retrieval (PQ-based ANNS), GPU for generation. HeterRAG outperforms it 6.17× avg by using PIM (higher BW) and graph-based ANNS (higher accuracy).

PipeRAG: CPU+GPU with aggressive result reuse

Jiang et al. · arXiv 2024

Reuses retrieval results across iterations. HeterRAG outperforms it 5.48× avg by adding PIM acceleration and locality-aware optimizations.
