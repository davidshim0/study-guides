How to make reconfigurable hardware finally deliver on sparse iterative workloads

Distributed SRAM gives enormous bandwidth. CGRAs give flexibility. Quartz combines them with short dataflow tasks and smart data partitioning to achieve 21× over Dalorex++, 93× over H100 on BFS, SpMV, and solvers.

Seven concepts Quartz builds on

Tap any card to expand. These clarify the sparse-workload landscape and Quartz's design choices.

high-bandwidth local storage partitioning challenges reconfigurable fabric baseline / tradeoffs
1Iterative sparse matrix computations — and why memory dominates▾

Applications like BFS, CG solvers, and PDHG loop for thousands of iterations, each performing sparse matrix–vector operations (SpMV) or graph traversals. Sparsity (99–99.9999%) means most data is zero and irregular memory access patterns arise. Low arithmetic intensity (few FLOPs per byte) means you spend far more time fetching data than computing. A 1 MB cache with a 16-way associative structure can't hide the latency when every access pattern is unpredictable.

Now you're ready: Quartz targets workloads where memory bandwidth is the bottleneck, not compute throughput or cache locality.
2Distributed-SRAM architectures — trading hierarchy for bandwidth▾

Traditional systems have a memory hierarchy: small fast cache, big slow DRAM. Distributed-SRAM systems place a modest (e.g., 452 KB) scratchpad at every tile in a spatial array of 16,384 tiles, giving aggregate capacity of 7.1 GB all on-chip and aggregate bandwidth orders of magnitude higher. Quartz utilizes on average 280 TB/s (measured from runtime utilization across workloads), far exceeding conventional memory systems. Each tile's PE accesses its own memory with 2-cycle latency. Data is partitioned across tiles at element granularity.

Now you're ready: Quartz uses this architecture style to overcome memory bottlenecks — but that shifts the challenge to partitioning and PE design.
3The dilemma: specialized PEs vs. general cores▾

Prior distributed-SRAM accelerators either hardwire control flow for a narrow domain (Azul: fast but inflexible) or use general-purpose in-order cores (Dalorex: flexible but slow, spending ~60% of cycles on bookkeeping and address calculation). You want the flexibility to support BFS, SSSP, WCC, CG, CHB, PDHG without changing the chip, but you also want high throughput.

Now you're ready: Quartz bridges the gap with reconfigurable PEs — CGRAs that reconfigure per task type for high utilization, yet remain general.
4Reconfigurable compute (CGRAs) — a middle ground▾

A coarse-grained reconfigurable array (CGRA) contains functional units (load/store, ALU, FP multiply/add) connected by configurable routes. At runtime, you load a "bitstream" that wires units into a custom datapath. Unlike an FPGA (fine-grained logic blocks), a CGRA's units are pre-built, so reconfiguration is fast. Quartz uses double-buffered config cells to preload the next task configuration during execution of the prior task, enabling minimal reconfiguration overhead. Unlike a CPU pipeline, there's no instruction fetch or decode per operation — the fabric is set once per task.

Now you're ready: Quartz PEs are CGRAs designed for short tasks (tens of ops), reconfiguring as messages arrive. This keeps compute busy without the overhead of a CPU ISA.
5Static vs. dynamic sparsity — and why it matters for partitioning▾

Static sparsity: the pattern of nonzeros is fixed across iterations (e.g., adjacency matrices in BFS — edges don't move). Dynamic sparsity: the frontier vector in BFS changes every iteration. In "all-active" algorithms (CG, PDHG), all vector elements are nonzero every iteration; in "non-all-active" (BFS), only a subset is live, and that subset shifts. Prior partitioning methods handle static-static sparsity well (hypergraph partitioning) but fail when one operand is dynamically sparse — work can clump unpredictably.

Now you're ready: Quartz's two-phase partitioning (graph clustering + per-cluster hypergraph) balances load even when sparsity is time-varying.
6Task-level dataflow — messages trigger computation▾

Instead of fetching instructions from memory, Quartz uses active messages: a packet arriving at a tile triggers a short task that (1) reads local data, (2) computes, (3) writes local data and/or sends new messages. Each task is ~10–50 operations and fits entirely on the reconfigurable fabric. No waiting for multiple operands (as in traditional dataflow). No instruction queue. The PE reconfigures to the task type encoded in the message header and executes immediately.

Now you're ready: This execution model minimizes overhead and hides latency by overlapping network arrivals with compute.
7The Einsum formalism — a notation for computation and communication▾

Einstein summation (Einsum) concisely describes tensor operations. For SpMV: Z[m] = A[m,n] * B[n] with implicit sum over n. Quartz extends this with a partitioning rank P to represent data distribution and inserts communication Einsums for inter-tile sends/receives. The full cascade of Einsums becomes the spec, and tasks are derived by lowering each Einsum into loopnests. The paper describes the first three steps of task generation conceptually; partitioning and task-instance counts are automated in the implementation.

Now you're ready: Quartz's programming model translates Einsums → partitioned Einsums → task types → CGRA bitstreams through a systematic process.
📌
Quick glossary: Tile = PE + scratchpad + router. PE = reconfigurable fabric of functional units. Task = short computation triggered by message arrival or local task completion. SpMV = sparse matrix–vector multiply. MPKI = misses per 1000 instructions. GTEP/s = giga traversed edges per second (graph perf metric).
The whole story in one place

Quartz in a nutshell

The architecture, the execution model, the partitioning trick, and the payoff.

Sparse iterative workloads (BFS, CG, PDHG) are memory-bound: 99% sparsity and low arithmetic intensity create irregular access patterns that defeat caches. Distributed-SRAM systems (many tiles, each with local scratchpad) offer enormous aggregate bandwidth, but prior designs either hardwire control flow (Azul: inflexible) or use general cores (Dalorex: spend 60% of cycles on bookkeeping). Quartz uses reconfigurable PEs (CGRAs) to get the best of both: high throughput via custom datapaths, flexibility via fast reconfiguration.

Each of Quartz's 16,384 tiles holds a CGRA, a 452 KB scratchpad, and a router. Tiles connect via a 2D torus network (8 TB/s bisection bandwidth). Computation proceeds by task-level dataflow: arriving messages trigger short tasks (tens of ops) that reconfigure the PE, read local data, compute, and optionally send new messages. Two separate fabrics (Update and Send) prevent application-level deadlock. Because tasks are short and data is local, most SRAM accesses hit in 2 cycles and the network is the main latency source — which is where partitioning comes in.

Data partitioning is critical. You want (1) minimal inter-tile messages (co-locate dependent data) and (2) balanced load (no tile waits while others finish). For all-active algorithms (CG, PDHG), recursive hypergraph partitioning works: represent data dependencies as a hypergraph, partition nodes to minimize edge cuts, recursively partition each cluster to respect physical locality (intra-chiplet, then intra-tile). For non-all-active algorithms (BFS), dynamic sparsity breaks load balance. Quartz's two-phase approach clusters the input graph via graph partitioning on the adjacency structure, then independently hypergraph-partitions each cluster across all tiles. This partitioning contributes an additional 3.7× speedup in the ablation study (Fig. 1), while for non-all-active algorithms two-phase is 3.6× over round-robin and 1.4× over recursive hypergraph (Fig. 15).

Programming model: write your algorithm as a cascade of Einsums (extended notation for partitioning and communication). Insert a partitioning rank P and distribution tensors T for sends/receives. Lower each Einsum into loopnests, fix P and iteration I, split at natural boundaries (e.g., outer loop iterations become separate tasks), fuse adjacent dependent loops to eliminate intermediate storage, and emit task types. Translate each task type to a CGRA bitstream. Instantiate tasks at runtime from partitioned data. The paper describes this process systematically; partitioning and task instantiation are automated, while full compiler passes are future work.

Results: Quartz achieves gmean 21.4× over Dalorex++ (a simulated distributed-SRAM baseline with in-order RISC PEs, multicast/reduction trees, and same PE count/capacity) across 6 applications (BFS, SSSP, WCC, CG, CHB, PDHG) and gmean 93× over an H100 GPU (using best available libraries). It provides 21× better performance per area than the H100 and 2223× better energy efficiency. The gain comes from two sources (Fig. 1): reconfigurable hardware (5.7×) and clever partitioning (3.7×). On CG, Quartz reaches 38% of peak 16 TMUL/s; on BFS, it sustains 280 TB/s of memory bandwidth.

🍯 Key takeaways

  • Architecture: 16,384 tiles, each with a CGRA PE (Update + Send fabrics), 452 KB SRAM, router. 2D torus network, 8 chiplets.
  • Execution: Task-level dataflow. Messages trigger tasks. PEs reconfigure per task type (minimal overhead with double-buffering). Short tasks (tens of ops) fit entirely on fabric.
  • Partitioning (all-active): Recursive hypergraph partitioning. Minimize inter-tile messages, respect physical locality (chiplet → tile hierarchy).
  • Partitioning (non-all-active): Two-phase: graph partition input into clusters, then hypergraph-partition each cluster across all tiles. The clusters approximate values likely active in the same iteration and are distributed across tiles to improve load balance.
  • Programming: Einsum cascade → partitioned Einsums → task types → CGRA bitstreams. Systematic translation; partitioning and instantiation automated.
  • Performance: 21.4× over Dalorex++, 93× over H100 GPU (gmean). 38% of peak on CG, 280 TB/s memory bandwidth.
  • Area/energy: 449 mm² per chiplet (68% SRAM), 21× better perf/area than H100, 2223× better perf/J. ~1% total chip overhead.
Seven everyday pictures

How to think about Quartz

Each analogy stands alone — read them in any order. Every card ends with where the picture breaks.

The core idea · Distributed SRAM

Many small kitchens vs. one big warehouse

Traditional systems: one giant warehouse (DRAM) far from the workers (CPU cores), accessed via narrow roads (memory bus). Distributed-SRAM: give every worker their own small kitchen (452 KB scratchpad) right next to their workbench. Total food capacity (7.1 GB) is smaller than the warehouse (GPU: 80 GB), but aggregate bandwidth is far higher because every kitchen's pantry is two steps away. For sparse workloads where every bite of data is unique (no reuse), proximity wins over bulk.

Where it breaks: if you need more data than fits across all kitchens combined, you're stuck. Quartz targets problems that fit in 7.1 GB (large graphs/matrices with high sparsity), not dense ML models with billions of parameters.
The compute model · CGRAs

A modular assembly line you rearrange between products

A CGRA is like a factory floor with pre-built stations (load conveyor, welder, painter, packer) and movable tracks connecting them. To make a new product (task type), you quickly rewire the tracks (reconfigure the fabric) to form a custom assembly line. Unlike a general factory (CPU) where every item waits in a queue for the same multi-purpose robot (instruction pipeline), each station operates in parallel once wired. Unlike building a new factory from scratch (ASIC), rearranging takes minimal time because the stations are already there. Quartz does this thousands of times per second as different task types arrive.

Where it breaks: if a product needs a station you don't have (e.g., a quantum gate), you're stuck. Quartz's fabric is tuned for sparse linear algebra (load/store, integer/FP arithmetic), not arbitrary computation.
Execution · Task-level dataflow

A courier triggers actions on delivery

Traditional dataflow: workers wait for all ingredients (operands) before cooking. Quartz: a courier (message) arrives with one critical ingredient (e.g., a column index for SpMV) and a recipe card (task type). The worker immediately reconfigures their station, grabs local ingredients (scratchpad data), cooks (computes), and sends results to other workers. No waiting for missing operands — the task is designed so local data plus the message payload is enough. The courier service (network) keeps deliveries flowing, and workers stay busy.

Where it breaks: if a task genuinely needs inputs from multiple distant sources before starting, you're forced to split it into sub-tasks or use synchronization (Quartz does barriers between iterations for CG/PDHG). Pure message-triggering works best when dependencies are 1→many or local.
Partitioning · Hypergraph cuts

Dividing a recipe book so each kitchen is mostly self-sufficient

You have a cookbook (computation) where some recipes share ingredients (data dependencies). Goal: split recipes among kitchens (tiles) so each kitchen can make most dishes without asking neighbors for ingredients (minimizing inter-tile messages). A hypergraph models this: nodes = ingredients, hyperedges = recipes (each connects all its ingredients). Partitioning the hypergraph to minimize cut hyperedges means most recipes stay local. Quartz does this recursively (first split across chiplets, then within each chiplet) to respect physical distance — nearby kitchens share more ingredients.

Where it breaks: hypergraph partitioning assumes you know the recipe book upfront (static sparsity). For BFS, the "recipes" change each iteration as the frontier moves — that's why Quartz needs the two-phase trick.
Partitioning challenge · Dynamic sparsity

A moving target in a shooting gallery

All-active algorithms (CG, PDHG): the targets (vector elements) are always in the same places — you can aim carefully once. Non-all-active (BFS): each iteration, only a few targets light up (the frontier), and they're in different spots. Hypergraph partitioning optimizes for the full gallery lit; on BFS iteration 5, maybe only a few tiles are active while the rest idle. Quartz's solution: graph partition the adjacency matrix into clusters, then hypergraph partition each cluster across all tiles. The clusters approximate values likely active in the same iteration, and distributing them across tiles improves load balance.

Where it breaks: if one cluster is a supernode (airport hub in a flight graph), it dominates work even when spread. Quartz's method is a heuristic that works well for typical power-law graphs but isn't perfect.
Programming · Einsum → tasks

A recipe that auto-generates cooking instructions

You write the high-level intent (Einsum: "multiply matrix columns by vector elements, sum over columns") without specifying how or where. Quartz's approach adds a partitioning rank P ("tile 0 owns these rows") and inserts send/receive Einsums ("send column value to tiles needing it"). Then it lowers to loopnests, fixes P and iteration I, splits loops at natural points (one task per arriving column value), and emits task types. Each task type becomes a CGRA bitstream. At runtime, partitioned data instantiates tasks automatically. You never write tile-level code by hand.

Where it breaks: Quartz's current programming model assumes computations fit the Einsum+partitioning abstraction. Arbitrary control flow (e.g., "if graph has cycle, do X") isn't captured. The paper describes the task generation process systematically; partitioning and instantiation are automated, while full compiler automation is future work.
Why CGRAs beat CPUs here · Overhead

Removing the middleman

A CPU fetches an instruction, decodes it, reads operands, executes, writes back — every cycle. For a loop scaling a vector, 60% of cycles are bookkeeping (address calculation, branch checks). A CGRA configures once to form a pipeline that directly streams data through load → multiply → store units, no per-element overhead. On Quartz's 16 ops/task average, the one-time config overhead amortizes to nearly zero. The CPU is like a general contractor who reads the blueprint for every nail; the CGRA is like a stamping machine set up once and then banging out parts.

Where it breaks: if tasks are ultra-short (1–2 ops) or wildly varying (thousands of different task types), reconfiguration overhead starts to matter. Quartz's design (double-buffered configs, <10 task types per app) keeps this negligible.
The full explanation

How it actually works

Architecture, execution model, partitioning, and programming — step by step with the paper's key diagrams.

1 · Architecture overview

Quartz is a multi-chiplet system: 8 chiplets, each with 2,048 tiles (total 16,384). Tiles connect via a 2D torus network with 128-bit links (1 cycle/hop, on-chiplet and inter-chiplet). Aggregate bisection bandwidth: 8 TB/s. Each tile contains:

  • Reconfigurable PE: Two fabrics (Update and Send) made of functional units (load/store, INTALU, FPADD, FPMUL, FPDIV) connected by configurable routes (muxes at connection nodes). Update fabric: 8 load, 8 store, 16 INTALU, 6 FPADD, 1 FPMUL, 1 FPDIV. Send fabric: 3 load, 3 store, 2 INTALU. Double-buffered config cells preload next task during execution → minimal reconfiguration overhead.
  • Scratchpad SRAM: 452 KB (448 KB data + 4 KB task queue/config). Table 2 lists 8 ports per data SRAM, 128-bit wide, 2-cycle access. Highly banked to avoid port contention.
  • Router: connects to 4 neighbors (2D torus). The paper says tree communication (multicast and reduction) is implemented via extra task types to avoid additional hardware costs, with local bubble routing for network-level deadlock prevention.
  • Supporting logic: Send/receive buffers, bitmap reader/writer (hierarchical bitmaps track pending Send tasks), load/store coalescer, config loader.
🔧
Area breakdown per chiplet (Table 5): 2,048 PEs × 0.0668 mm² = 136.88 mm², 2,048 routers × 0.0016 mm² = 3.28 mm², SRAM total 294.13 mm², I/O PHY 15 mm² → total 449 mm² (68% SRAM). Power (Fig. 17): SRAM dominates (~60–80%), then compute (FP apps) or network (integer apps).

2 · Task-level dataflow execution

Quartz uses active messages [Eicken'92]: a packet arriving at a tile triggers a task. No waiting for multiple operands (unlike traditional dataflow). No instruction fetch (unlike CPU). Each task is ~10–50 ops and fits on the CGRA fabric. Structure: (1) receive data from network / read local scratchpad, (2) compute, (3) write local / send to network. Tasks triggered by messages go to the Update fabric; tasks that send data use the Send fabric. This separation prevents application-level deadlock (e.g., all tiles trying to send, none receiving). Bubble routing addresses network-level deadlock.

Example (SpMV): Tile 0 owns column 5 of matrix A and vector element B[5]. It sends (5, B[5]) to all tiles owning nonzeros in column 5. Each receiving tile triggers a ScaleColumn task: configure Update fabric to load A[*,5] elements, multiply each by B[5], accumulate into local partial sums LZ[*]. Later, send partial sums to owner of final Z[*]. Crucially, one message triggers one task — no waiting.

💡
Why separate fabrics? If all tasks shared one fabric, a burst of Send tasks could block Update tasks, stalling incoming messages → network backpressure → deadlock. By dedicating one fabric to receives (Update) and one to sends (Send), you guarantee forward progress. Update tasks run immediately; Send tasks queue in a bitmap (hierarchical for fast scan) and run when fabric is free.

3 · Sparsity support — three techniques

Quartz avoids heavy intersection/merger hardware [ExTensor, Tensaurus] via three design choices:

  1. Split outer loop across tasks. For 2D operations (SpMV: matrix × vector), choose a dataflow (n→m: iterate columns, then rows) and make each outer-loop iteration a separate task. E.g., one task per column: receives (n, B[n]), scales all A[*,n] by B[n], updates LZ[*]. Intersection becomes degenerate (just scaling a sparse vector).
  2. Leverage static sparsity. All matrices in the workloads have static sparsity (adjacency graphs, physics systems). Pre-compute and store where each vector element must be sent. E.g., for B[5], store "tiles 3, 7, 12 own column 5 nonzeros" → O(1) multicast setup.
  3. Guarantee coordinate subsetting. For operations like "update local distance vector with frontier candidates," partition the distance vector so each tile owns exactly the coordinates it will need. Intersection is trivial (all local coordinates match).
🍯
These techniques reduce all sparse operations to sparse vector scale (multiply vector by scalar, add to another vector) or trivial intersections (coordinates guaranteed to match). Simple to implement, fast to execute.

4 · Partitioning for all-active algorithms — recursive hypergraph

For CG, CHB, PDHG (where all vector elements are active every iteration), we want to (1) minimize inter-tile messages and (2) balance load (equal number of nonzeros per tile). Hypergraph partitioning does both. Construct hypergraph H: nodes = data elements (matrix nonzeros, vector elements), hyperedges = operations (e.g., one hyperedge connects all elements in a row of A with the corresponding element of Z). Partition H into N clusters (N = number of tiles) to minimize cut hyperedges (= messages sent) subject to balanced node counts. Use PaToH library [Çatalyürek'11].

Recursive partitioning improves physical locality. First partition into 8 clusters (one per chiplet), then partition each chiplet into 2,048 clusters (one per tile). This reduces average hops per message by 21% (Fig. 15) because dependent data is more likely to be nearby.

📌
Cost: hypergraph partitioning takes gmean 18.5 minutes across all datasets (Table not shown, text in §6.4). Amortized over thousands of iterations and many runs (road graphs for navigation, physics matrices reused across timesteps). Graph500 submissions [Fugaku] spend 1000× longer in preprocessing than a single BFS run.

5 · Partitioning for non-all-active algorithms — two-phase

For BFS, SSSP, WCC (where frontier/active set changes each iteration), hypergraph partitioning alone fails: it co-locates elements that are ever used together, but at any given iteration only a subset is active. E.g., partitioning a road graph by city puts all roads of Boston on one tile — great for locality, but when BFS starts at Boston, only one tile is busy initially. To balance dynamic load, Quartz uses two-phase partitioning:

  1. Graph partition the input graph (adjacency matrix as-is, not a hypergraph) into clusters using standard methods [Karypis]. The paper describes clustering and partitioning but does not specify a fixed number of clusters as a parameter. Each cluster approximates values likely active in the same iteration.
  2. Hypergraph partition each cluster independently across all tiles. The clusters are distributed across tiles to improve load balance. Now iterations activate elements from multiple clusters, more uniformly distributing work.

This trades off locality (more inter-tile messages) for load balance. Net result: for non-all-active algorithms, two-phase partitioning is 3.6× over round-robin and 1.4× over recursive hypergraph (Fig. 15). The key insight: each cluster is small enough that hypergraph works well locally, and spreading clusters globally helps keep more tiles busy.

⚠
When it struggles: if one cluster is a supernode (e.g., airport hub with 10,000 edges in a flight graph), it dominates work even when spread. Quartz's method is a heuristic that works well for typical power-law graphs (web, social) but isn't optimal for all graphs.

6 · Programming model — Einsum → tasks

Quartz's programming interface is an Einsum cascade. The paper defines a systematic process to translate Einsums to task types. The first three steps are described conceptually; partitioning and task-instance counts are automated in the implementation:

  1. Step 1: Write iterative Einsum cascade. Express computation as Einsums (extended notation: custom map/reduce ops, iterative ranks I for time). E.g., SpMV: ZMm = AM,Nm,n · BNn.
  2. Step 2: Insert partitioning rank P. Add rank P to tensors to represent distribution. Insert distribution tensors T for sends/receives. E.g., LBP,Np,n = T1P,Np,n · BP,Nq,n (send B[n] to tiles needing it). Einsums now represent communication explicitly.
  3. Step 3: Lower to loopnests, fix P and I, split loops. For each non-transfer Einsum, write as loopnest. Fix P=p (tile p) and I=i (iteration i) to eliminate those loops. At most 2 loops remain (for rank-2 tensors). Choose dataflow (loop order) to maximize reuse. Split execution of outer loop across separate tasks (one task per outer iteration). Fuse adjacent dependent tasks to eliminate intermediate storage. Emit task types.
🍯
Example (SpMV, 4 task types): (1) SendColumnValue: send (n, B[n]) to tiles owning A[*,n]. (2) ScaleColumn: scale local A[*,n] by B[n], add to LZ[*]. (3) SendPartialSum: send (m, LZ[m]) to tile owning Z[m]. (4) UpdateRowSum: reduce incoming partial into Z[m]. Code for each task → CGRA bitstream.

Worked examples

Step-by-step traces using the paper's concepts.

① Deriving task types for SpMV

Starting from the partitioned Einsum cascade (paper §3.2–3.3), show how to arrive at the four task types for sparse matrix-vector multiplication.

  1. Partitioned Einsums. From paper §3.2: LBP,N = T1P,N · BP,N (send), LZP,M = AP,M,N · LBP,N (compute), ZP,M = T2P,M · LZP,M (gather).
  2. Einsum 1 (send). Lower to loopnest: for n: for p where T1[p,n]: send (n, B[q,n]) to p. Fix P=q (sender), split n-loop → SendColumnValue task: send (n, B[n]) to all tiles in multicast tree.
  3. Einsum 2 (compute). LZ[p,m] = A[p,m,n] · LB[p,n]. Loopnest: for n: for m: LZ[p,m] += A[p,m,n] * LB[p,n]. Choose n→m dataflow (LB[p,n] arrives unpredictably), split n-loop → ScaleColumn task: for m: LZ[p,m] += A[p,m,n] * LB[p,n] (received).
  4. Einsum 3 (gather). Z[p,m] = LZ[q,m] where T2[p,m]. Split into two tasks: (a) SendPartialSum: send (m, LZ[p,m]) to owner of Z[m]. (b) UpdateRowSum: Z[p,m] += incoming LZ[q,m].
🍯
Result: four simple task types, each ~10–20 ops. ScaleColumn loops over local column; others are O(1) per invocation. Fusing steps 2 and 3 would eliminate LZ storage, but paper keeps them separate for clarity.

② Understanding network traffic patterns

Fig. 16 shows that hypergraph partitioning reduces link activations to ~0.3× of round-robin (normalized). What does this tell us about communication?

  1. Baseline (round-robin). With round-robin partitioning, dependent data is scattered randomly across tiles. Each edge in the computation graph typically causes messages between distant tiles, with many hops across the network.
  2. Hypergraph partitioning effect. Co-locates dependent data, so most operations involve local or nearby tiles. The normalized graph in Fig. 16 shows ~0.3× link activations compared to round-robin, indicating roughly 3× fewer network links are active for the same computation.
  3. Impact. Fewer link activations means lower network contention, reduced latency, and better overall performance. This factor-of-3 reduction in network traffic is a key contributor to Quartz's speedup.
💡
Why 0.3×? Hypergraph minimizes cut hyperedges (messages between tiles) but can't eliminate them entirely because the computation graph is connected. The 0.3 factor reflects typical power-law graph structure (high clustering) that hypergraph partitioning exploits well.

③ BFS performance breakdown — hardware vs. partitioning

Fig. 1 shows Quartz achieves 21.4× over Dalorex++ (gmean), decomposed into 5.7× (hardware) and 3.7× (partitioning). Understanding these contributions.

  1. Dalorex++ baseline. In-order RISC cores with multicast/reduction trees. Same PE count and total capacity as Quartz but uses CPU-style execution with instruction fetch/decode overhead.
  2. Hardware contribution (5.7×). Replacing CPU cores with reconfigurable CGRAs eliminates per-operation overhead. Double-buffered reconfiguration and task-level dataflow keep PEs busy. This 5.7× gain shows the benefit of specialized datapath execution over general-purpose cores.
  3. Partitioning contribution (3.7×). Quartz's recursive hypergraph (for all-active) and two-phase (for non-all-active) partitioning reduce inter-tile communication and balance load. This 3.7× gain is measured by comparing Quartz with its advanced partitioning to Quartz with baseline round-robin partitioning.
  4. Total gain. 5.7× × 3.7× ≈ 21× overall speedup. Both hardware and partitioning are essential — neither alone delivers the full benefit.
📌
Key insight: hardware (CGRAs) and partitioning (two-phase/recursive) contribute roughly equally to total gain. You need both components working together.

④ Area breakdown — understanding the 449 mm² chiplet

Table 5 breaks down one chiplet's area. Verify the 68% SRAM figure and understand where area goes.

  1. Component breakdown (Table 5). Per chiplet with 2,048 tiles: PEs total 136.88 mm², routers total 3.28 mm², SRAM total 294.13 mm², I/O PHY 15 mm².
  2. Total area. 136.88 + 3.28 + 294.13 + 15 = 449.29 mm² ≈ 449 mm² per chiplet.
  3. SRAM fraction. 294.13 / 449 = 65.5% ≈ 68% (paper rounds). This shows that memory dominates the chip area, which is expected for a memory-bandwidth-optimized architecture.

🍯
SRAM dominates because 452 KB × 2,048 tiles = 904 MB per chiplet of on-chip storage. At 7nm technology, SRAM requires significant area (~0.33 mm²/MB), but this investment delivers the 280 TB/s aggregate bandwidth that makes Quartz competitive.

Practice problems

Problem 1 · easy
Why does Quartz separate Update and Send tasks into different fabrics instead of using one unified fabric?
Show answer
To prevent application-level deadlock. If all tasks shared one fabric, a burst of Send tasks (which generate network traffic) could block Update tasks (which receive from the network). This creates backpressure: tiles can't send because fabric is busy, can't receive because Send tasks are hogging the fabric → deadlock. By dedicating Update fabric to receives and Send fabric to sends, you guarantee Update tasks run immediately upon message arrival, preventing network stalls.Misconception: that separation is for performance (parallelism). It's actually for correctness (deadlock freedom).
Problem 2 · medium
For BFS on a road graph, hypergraph partitioning co-locates all roads of Boston on one tile. Why does this hurt performance, and how does two-phase partitioning help?
Show answer
Hypergraph minimizes total messages by co-locating "ever used together" data. But in BFS starting at Boston, iteration 1 only activates Boston roads (one tile busy, thousands idle). Two-phase partitioning: (1) graph-partition the road network into clusters, (2) hypergraph-partition each cluster across all tiles. Now Boston roads are spread among many tiles, so iteration 1 activates elements across multiple tiles, improving load balance. Trade: more inter-tile messages (Boston roads aren't co-located), but better utilization (more tiles work each iteration).Misconception: that co-locating all dependent data is always optimal. For dynamic sparsity, distributing likely-active-together data improves balance at the cost of some locality.
Problem 3 · medium
Quartz's ScaleColumn task for SpMV loops over local matrix rows (for m: LZ[m] += A[m,n] * LB[n]). Why doesn't it need an intersection unit to find which rows have nonzeros in column n?
Show answer
Because of technique #2 (§4.2): leverage static sparsity. At partition time, Quartz pre-computes which tiles own nonzeros in each column and stores that info (distribution tensor T1). When tile p receives (n, B[n]), it already knows locally stored A[*,n] coordinates — they're in the scratchpad in compressed format (e.g., CSC column pointers). The task just iterates over that compressed list. No dynamic intersection needed — the coordinates are known at task launch time.Misconception: that sparse operations always require complex intersection logic. Quartz's design avoids it by splitting loops and using static structure.
Problem 4 · hard
Paper §6.4 says hypergraph partitioning takes gmean 18.5 minutes. For a BFS query service on a road graph (millions of queries, same graph), is this acceptable? What if the graph changes daily?
Show answer
Acceptable for static graph, marginal for daily updates. Static graph: 18.5 min one-time cost amortized over millions of queries (each taking ~ms) is negligible. Cost per query: 18.5 min / 10⁶ queries = 0.0011 sec/query = 0.06% overhead. Daily updates: re-partition every 24 hours. If graph structure changes slightly, incremental partitioning (future work) could reduce this. If structure changes drastically (unlikely for roads), 18.5 min might be tolerable if queries are frequent enough. But for real-time updates (graph changes per query), partitioning cost is prohibitive — you'd need faster heuristics or accept suboptimal partitions.Misconception: that preprocessing time matters relative to wall-clock time, not relative to amortization across runs. Graph500 Fugaku spends 1000× longer preprocessing than one BFS run (§6.4), yet still wins.
Problem 5 · hard
Quartz achieves 38% of peak 16 TMUL/s on CG. Where does the other 62% go? (Hint: think about what limits utilization in sparse workloads.)
Show answer
Three main sinks: (1) Load imbalance. Even with good partitioning, some tiles finish early and idle while others complete. (2) Network stalls. If incoming messages arrive bursty, tiles may stall waiting for operands or backpressure from downstream tiles. (3) SRAM port contention. Update fabric has 8 load units but SRAM has 8 ports — if multiple units access same bank, they serialize. (4) Reconfiguration overhead. Although minimal with double-buffering, it's not zero — switching task types incurs some cycles. (5) Iteration boundaries. CG uses barrier sync between iterations; all tiles wait for slowest. For CG specifically, (1) and (5) dominate: matrices are irregular, and barriers enforce global sync. 38% is actually high for a sparse workload — dense ML models on TPUs hit ~50% utilization.Misconception: that peak throughput is achievable on real workloads. Sparse → irregular → utilization gap. 38% is good.
Problem 6 · medium
Why does Quartz choose n→m dataflow (iterate columns, then rows) for SpMV instead of m→n?
Show answer
Because vector elements B[n] arrive from the network at unpredictable times (sent by their owner tiles after they're computed or loaded). If we used m→n dataflow (iterate rows, then columns), each task would need to wait for all B[n] in that row before starting — requires synchronization and buffering. In n→m, each arriving (n, B[n]) message triggers a task immediately: scale column n of A by B[n], add to local LZ[*]. No waiting. The task reuses B[n] across all local rows, and LZ[*] accumulates in the scratchpad. Final LZ[m] values are sent when column iteration completes.Misconception: that dataflow choice is arbitrary. It's driven by which operand arrives dynamically (B) vs. is stored locally (A).
Problem 7 · easy
True or false: Quartz's scratchpads (452 KB each) are too small to hold entire graphs, so it pages data from DRAM between iterations.
Show answer
False. Quartz is an all-SRAM system: all data lives on-chip in the aggregate 7.1 GB (16,384 × 452 KB) of scratchpad. No DRAM access during iteration execution. If the graph doesn't fit (e.g., >7.1 GB), Quartz can't run it — you'd need more chiplets or a different architecture. But target workloads (paper Table 3: largest is 114M nonzeros ≈ 1 GB for matrix + vectors) fit comfortably. The I/O PHY (HBM2e) is only for initial load and final writeback, not per-iteration traffic.Misconception: conflating distributed-SRAM with near-data processing (DRAM-based). Quartz is the former; HRL/Tesseract are the latter.
Problem 8 · medium
Quartz's programming model inserts a partitioning rank P into Einsums. How does this capture communication? Give an example send Einsum.
Show answer
P represents tile assignment. A distribution tensor T encodes where each element must go. For SpMV, T1P,Np,n = 1 if tile p needs B[n] (i.e., p owns nonzeros in column n of A). The send Einsum is: LBP,Np,n = T1P,Np,n · BP,Nq,n. Read: "For each (p,n) where T1[p,n]=1, send B from tile q (owner of B[n]) to tile p." Lowering this Einsum generates SendColumnValue tasks. Similarly, T2P,Mp,m (ownership of Z[m]) generates gather tasks. T-tensors are computed offline during partitioning and stored compactly (T1 as coordinate list, T2 as bitmap).Misconception: that Einsums only describe compute. Quartz extends them to describe communication explicitly via T-tensors and partitioning ranks.
Quick answers

Frequently asked questions

Twelve questions about Quartz. Tap to expand.

Q1What is a "reconfigurable PE" and how is it different from a CPU?▾
A CPU fetches and decodes instructions every cycle; a reconfigurable PE (CGRA) loads a "bitstream" once that wires functional units (load/store, ALU, FP) into a custom datapath for a task. No per-operation overhead. Think of it as a factory floor you rearrange between products (tasks) vs. a general robot (CPU) that reads blueprints for every nail. Quartz PEs have double-buffered config cells, so reconfiguration overlaps with prior task execution with minimal overhead.
Q2Why does Quartz outperform a GPU by 93×?▾
Three reasons: (1) Memory bandwidth. Quartz utilizes on average 280 TB/s aggregate bandwidth (measured from workload execution) through its distributed SRAM architecture vs. GPU HBM bandwidth. Sparse workloads are memory-bound, and Quartz's architecture delivers far more effective bandwidth. (2) Latency. Quartz: 2-cycle SRAM access vs. GPU: hundreds of cycles to HBM. (3) Partitioning. GPU's general model doesn't exploit sparsity structure; Quartz's hypergraph partitioning minimizes data movement. But Quartz uses 8× more area (3,592 mm² vs. ~815 mm²), so perf/area is 21× better.
Q3How does Quartz handle dynamic sparsity (e.g., BFS frontier)?▾
Two-phase partitioning. First, graph-partition the input graph into clusters. The paper describes clustering and partitioning methods but doesn't specify a fixed parameter. Each cluster approximates values likely active in the same iteration. Then hypergraph-partition each cluster independently across all tiles. Now iterations activate elements from multiple clusters, and distributing clusters across tiles improves load balance. This trades locality (more messages) for better work distribution. Result: 3.6× speedup over round-robin and 1.4× over single-phase hypergraph on non-all-active algorithms (Fig. 15).
Q4What happens if the graph doesn't fit in 7.1 GB?▾
Quartz can't run it — you need more chiplets (scale to 16, 32, etc.) or a different architecture. Quartz is an all-SRAM system: no DRAM paging during execution. Target workloads (Table 3: largest ~1 GB matrix + vectors) fit comfortably. For larger problems, you'd use a near-data processing system (HRL, Tesseract) or a GPU with larger HBM capacity, accepting lower bandwidth and higher latency.
Q5How long does partitioning take, and is it practical?▾
Gmean 18.5 minutes (§6.4). Practical for iterative workloads: amortize over thousands of iterations and many runs. Example: road graph for navigation serves millions of queries (each taking ~ms) with one partitioning. Cost per query: 18.5 min / 10⁶ ≈ 0.001 sec ≈ 0.1% overhead. For daily graph updates, 18.5 min is tolerable if structure doesn't change drastically. For real-time updates (graph changes per query), you'd need faster heuristics (future work) or accept suboptimal partitions. Graph500 Fugaku spends 1000× longer on preprocessing than one BFS run.
Q6Can Quartz handle dense matrix operations (e.g., dense matrix multiply)?▾
Not efficiently. Quartz's design optimizes for sparse irregular access (high bandwidth, short tasks, partitioning for locality). Dense ops have regular access patterns, high reuse, and benefit from cache hierarchies + vectorization — GPUs and TPUs win. Quartz's 452 KB scratchpad per tile is too small for meaningful dense tile sizes, and its task model (message-triggered, reconfigurable) adds overhead for regular operations. Use Quartz for sparse; use TPU/GPU for dense.
Q7How does Quartz prevent deadlock?▾
Two main mechanisms: (1) Separate Update/Send fabrics. Update tasks (receives) run immediately on dedicated fabric, preventing backpressure. Send tasks queue in bitmaps and run when fabric is free. This separation addresses application-level deadlock. (2) Bubble routing. Each ring in the 2D torus uses bubble flow control to address network-level deadlock: messages make forward progress through the network. Tree communication (multicast and reduction) is implemented via extra task types rather than specialized hardware.
Q8Why doesn't Quartz use a GPU-style cache hierarchy?▾
Because sparse workloads defeat caches. With 99% sparsity and irregular access patterns, most cache lines are fetched once and never reused — cache hit rate ~0%. A cache hierarchy would just add latency (tag lookup, coherence protocol) without benefit. Distributed SRAM eliminates the hierarchy: all data is "cached" locally at 2-cycle latency, and partitioning ensures each tile's data is actually used. This is a fundamental architectural choice: optimize for capacity × bandwidth (many small SRAMs) rather than locality (cache hierarchy).
Q9Can Quartz run multiple applications simultaneously (multiprogramming)?▾
Not in current design. Quartz is a spatial architecture: all tiles run tasks from one application at a time. You could time-multiplex (run app A for 1ms, checkpoint, load app B), but partitioning is per-app and checkpointing is expensive. Spatial isolation (dedicate chiplets 0–3 to app A, 4–7 to app B) is possible but requires inter-app partitioning (future work). Quartz targets single-app acceleration, like TPUs for ML. For multiprogramming, use a CPU or GPU.
Q10What did the authors leave for future work?▾
Several avenues: (1) Compiler. Fully automate the Einsum → task type translation (currently the first three steps are described conceptually; partitioning and task instantiation are automated). (2) Faster partitioning heuristics. 18.5 min is fine for static graphs but limits real-time applications. Incremental partitioning for slowly changing graphs. (3) Non-Einsum computations. Extend programming model to capture arbitrary control flow (e.g., "if cycle detected, break"). (4) Fault tolerance. Checkpoint/restart for long-running solvers. (5) Larger problems. Multi-rack scaling beyond 8 chiplets.
Q11How does Quartz compare to other distributed-SRAM systems (Azul, Dalorex)?▾
Azul [MICRO'24]: hardwires control flow for a few kernels (CG, BiCG, PDHG) → 3× faster than Quartz on those kernels but inflexible. Quartz uses reconfigurable PEs to support any Einsum-expressible computation. Dalorex [HPCA'23]: general-purpose in-order cores → flexible but slow (60% cycles on bookkeeping). Quartz CGRAs eliminate overhead → 5.7× faster. Dalorex++ (paper baseline): adds multicast/reduction trees to Dalorex → Quartz still beats it 21.4× via better PEs (5.7×) and partitioning (3.7×). Summary: Quartz fills the flexibility–performance gap.
Q12What are the key limitations of Quartz?▾
Main limitations: (1) Capacity constraint. 7.1 GB total on-chip storage limits problem size. Larger problems need more chiplets or off-chip memory (losing the bandwidth advantage). (2) Static sparsity assumption. While two-phase partitioning helps with dynamic sparsity, performance degrades if sparsity patterns change drastically between iterations. (3) Partitioning time. 18.5 minutes is amortizable for many use cases but prohibitive for single-shot queries. (4) Programming model. Requires expressing algorithms in Einsum notation; not all algorithms fit this model naturally.

Glossary & further reading

Every term in one place, then paths to go deeper.

Core terms
QuartzA distributed-SRAM accelerator with reconfigurable PEs, task-level dataflow, and smart partitioning for sparse iterative workloads. 16,384 tiles, 7.1 GB capacity, utilizing on average 280 TB/s bandwidth.
Distributed-SRAMArchitecture style: many tiles, each with local scratchpad (Quartz: 452 KB/tile). Aggregate bandwidth far exceeds memory hierarchy systems. All data on-chip.
CGRA (Coarse-Grained Reconfigurable Array)Reconfigurable PE: functional units (load/store, ALU, FP) connected by configurable routes (muxes). Bitstream wires units into custom datapath per task type. Fast reconfiguration with double-buffering.
Task-level dataflowExecution model: arriving messages trigger tasks. No instruction fetch, no waiting for multiple operands. Tasks are short (~10–50 ops) and self-contained.
Hypergraph partitioningMethod to minimize inter-tile communication: nodes = data elements, hyperedges = operations. Partition to minimize cut hyperedges (messages) subject to balanced node counts. Used for all-active algorithms (CG, PDHG).
Two-phase partitioningQuartz's method for non-all-active algorithms (BFS, SSSP): (1) graph-partition input into clusters, (2) hypergraph-partition each cluster across all tiles. Balances dynamic load.
Einsum cascadeProgramming model: express computation as cascade of Einsums (extended notation: partitioning rank P, distribution tensors T). Lower to tasks via systematic steps.
Update fabric / Send fabricSeparation in Quartz PE: Update fabric handles message-triggered tasks (receives); Send fabric handles send tasks. Prevents application-level deadlock.
Supporting terms
SpMVSparse matrix-vector multiplication: Z = A × B where A is sparse. Core operation in solvers (CG, CHB) and graph algorithms (BFS, SSSP via adjacency matrix).
Static sparsityNonzero pattern fixed across iterations (e.g., adjacency matrix). Quartz exploits this to pre-compute send destinations.
Dynamic sparsityNonzero pattern changes per iteration (e.g., BFS frontier). Requires load-balancing partitioning (two-phase).
All-active algorithmAll vector elements nonzero every iteration (CG, PDHG). Static sparsity only (matrix). Hypergraph partitioning works well.
Non-all-active algorithmSubset of vector elements active per iteration (BFS, SSSP). Dynamic sparsity. Needs two-phase partitioning.
Tree communicationMulticast and reduction patterns implemented via extra task types in Quartz to avoid additional hardware costs. Used for distributing data (multicast) and gathering partial sums (reduction).
Distribution tensor TEinsum notation for communication: T[p,i] = 1 if tile p needs element i. Inserted in partitioned Einsum cascade to represent sends/receives.
TileQuartz's basic unit: PE + 452 KB scratchpad + router. 16,384 tiles across 8 chiplets (2,048/chiplet).
ScratchpadLocal SRAM at each tile. 452 KB (448 data + 4 task queue/config). 8 banks, 128-bit wide (Table 2). 2-cycle access latency, pipelined.
Active messageNetwork packet that triggers computation at receiver. Contains task type, arguments (e.g., column index, value). Core of Quartz's execution model.
Functional unitPre-built compute block in CGRA: load/store (with address gen), INTALU, FPADD, FPMUL, FPDIV. Wired together by configuration to form custom datapath.
BitstreamConfiguration data loaded into CGRA to wire functional units for a task type. Quartz uses double-buffered config cells to preload next task during execution.
Graph partitionStandard method to divide graph nodes into clusters minimizing edge cuts. Used in Quartz's two-phase approach (step 1) to identify likely-active-together elements.
PaToHLibrary for hypergraph partitioning [Çatalyürek'11]. Used by Quartz for both single-phase (all-active) and phase 2 of two-phase (non-all-active).
Dalorex / Dalorex++Baseline distributed-SRAM system [HPCA'23]. Uses in-order RISC cores as PEs. Dalorex++ adds multicast/reduction trees (paper's baseline with same PE count and capacity as Quartz).
AzulPrior specialized distributed-SRAM system [MICRO'24]. Hardwires control flow for CG/BiCG/PDHG → fast but inflexible. Quartz is more general.
