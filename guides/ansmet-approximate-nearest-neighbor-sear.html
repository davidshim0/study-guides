Stop fetching when you already know the answer is too far

Finding the nearest neighbors in large-scale vector databases (some with billions of vectors) is slow and memory-hungry. ANSMET offloads comparisons to near-memory logic and terminates early — saving memory accesses by estimating distance lower bounds from partial data — achieving strong speedups without losing accuracy.

📄 "ANSMET" · Li, Jin, Tian, Zhang, Gao (Tsinghua) 🎓 Assumes DRAM/NDP memory basics ⏱ ~25 min read
🔍

If the partially fetched bits already prove a vector is beyond the threshold, stop accessing the rest — you've saved memory bandwidth without sacrificing accuracy.

Before you begin

Six concepts ANSMET builds on

These explainers cover the memory-tiering and vector-search background. Read them in order or skip to the unfamiliar ones.

1Vector databases and approximate nearest neighbor search (ANNS)▾

High-dimensional vectors (hundreds of dimensions) represent embeddings in ML — text, images, audio. A vector database stores billions of such vectors. The key operation is kNN search: find the 𝑘 nearest vectors to a query. Brute force is too slow (O(𝑁𝐷) for 𝑁 vectors of 𝐷 dims), so ANNS trades perfect accuracy for speed by using indexes like graph-based HNSW (hierarchical navigable small worlds) or cluster-based IVF (inverted file index) to skip most vectors and only check promising candidates.

Now you're ready: ANSMET accelerates the distance-comparison phase of ANNS — the part where you fetch and compare the candidate vectors identified by the index.
2Why ANNS is memory-bound▾

Even with good indexes, a query touches hundreds of high-dimensional vectors (e.g., 617 vectors × 128 dims × 2 B = 158 kB per query in HNSW-SIFT). Each comparison fetches the full vector from DRAM and does a simple distance calculation (subtract, square, sum for Euclidean) with low arithmetic intensity (~0.1 op/byte). So a significant portion of execution time is memory-bound distance comparison, not index traversal or sorting.

Now you're ready: ANSMET's hardware answer is near-data processing (bring compute to memory); its software answer is early termination (reduce data fetched per comparison).
3Near-data processing (NDP) for DRAM▾

Instead of moving data to the CPU, place compute logic near the data in the memory system. For DRAM DIMMs, this means adding a buffer chip per rank with small processing units (e.g., adders, multipliers) that see the full internal bandwidth (8× higher than the CPU-side channel). DIMM-based NDP keeps commodity DRAM capacity (terabytes) but gives local bandwidth for memory-bound tasks. It's been used for recommendation systems (aggregate embeddings) and graph processing.

Now you're ready: ANSMET is the first DIMM-NDP design specialized for ANNS, offloading distance comparison to rank-level logic while the CPU does index traversal.
4The opportunity: many vectors are rejected▾

In ANNS, you maintain a threshold — the 𝑘th smallest distance so far. A new vector is only "accepted" if its distance is below this threshold. The paper shows that a large fraction of comparisons (often 50–90%) are rejected (the vector is too far), meaning both the memory accesses and the computations were wasted. This is unique to ANNS: unlike recommendation systems that must aggregate all required vectors, ANNS can stop early once a vector proves to be out of range.

Now you're ready: early termination leverages this rejection pattern to stop fetching a vector partway through if the partial data already proves it's too far.
5Estimating distance from partial data▾

Suppose you've fetched only 2 of 4 dimensions: (1, 2, ?, ?) vs query (4, −2, 6, −1). You can compute a distance lower bound by setting the missing values to minimize distance: for Euclidean, if fetched bits match the query, set missing bits the same; if fetched > query, set missing to all 0s; if fetched < query, set to all 1s. Example: for (00␣␣, 01␣␣) vs (0110, 0101), the lower bound sets missing bits to (0011, 0101), giving distance ≥ 3. If this lower bound already exceeds the threshold, reject without fetching more.

Now you're ready: ANSMET combines partial-dimension and partial-bit early termination (hybrid ET) to maximize savings with zero accuracy loss.
6Data layout for efficient partial fetches▾

Memory fetches are 64 B granularity. To enable early termination, you need to fetch the most informative bits first: high bits (sign, exponent for FP32) have more impact than low bits. ANSMET transforms the layout offline so each 64 B chunk contains the next high 𝑛 bits of multiple dimensions (e.g., highest 8 bits of 64 dims), not the full vector sequentially. After fetching one chunk, you estimate the lower bound; if it's too high, stop. The layout also eliminates common prefixes (like quantization) to save space and fetches.

Now you're ready: the data layout is determined by sampling 100 vectors offline to decide bit chunk sizes and common prefix length, balancing efficiency and overhead.

The whole story in one place

ANSMET in a nutshell

The problem, the hardware + software fix, and the measured payoff.

ANNS distance comparison is memory-bound (a significant fraction of time, hundreds of kB per query) with low arithmetic intensity (~0.1 op/byte). Worse, 50–90% of vectors are rejected (beyond the threshold), making their accesses and computations wasted. ANSMET addresses both inefficiencies with a hardware-software co-design.

At the hardware level, ANSMET uses DIMM-based NDP — placing distance-computing logic in the buffer chip of each memory rank (32 NDP units total across 4 channels × 2 DIMMs × 4 ranks = 32 ranks). This exploits 8× higher internal bandwidth and parallelism to offload the memory-intensive comparison tasks from the CPU, which continues to do index traversal and result sorting.

At the software level, ANSMET introduces hybrid partial-dimension/bit early termination. As a vector is fetched in 64 B chunks, we compute a conservative distance lower bound using the partially fetched data (e.g., highest 8 bits of 64 dims). If this lower bound already exceeds the current threshold, we stop issuing future accesses for that vector. The key is fetching high-impact bits first and setting missing bits to minimize distance — so the lower bound is always correct and accuracy is never lost.

To maximize savings, ANSMET transforms the data layout offline: packing the next high 𝑛 bits of multiple dimensions in each 64 B chunk (rather than storing full vectors sequentially). It uses dual-granularity fetch (coarse for common prefix, fine for high-entropy range) and common prefix elimination (storing shared high bits once). Both are decided by sampling 100 vectors to measure prefix entropy and early termination frequency. Vectors are partitioned across ranks with a hybrid scheme (1 kB sub-vectors per rank) plus replication of hot vectors (entry points, top-layer graph vertices) to balance load.

The payoff: on 7 datasets (BigANN, SPACEV, DEEP, and Txt2Img are billion-scale; SIFT, GloVe, and GIST are in the millions) with HNSW, ANSMET achieves an average 5.26× speedup from NDP (up to 6.40× for DEEP) and another 1.52× from early termination on top of that (up to 2.24× for GIST). These are separate, multiplicative effects measured in the evaluation. Energy drops by 78% with NDP alone, then 31% more with ET. Hardware cost is under 0.06 mm² per NDP unit (2 kB QSHRs + 16-wide adders/multipliers at 22 nm). Offline preprocessing adds only 1.6% to graph construction time.

🍯 Key takeaways

  • ANNS distance comparison is memory-bound (significant execution time) with 50–90% of vectors rejected — both inefficiencies ANSMET targets.
  • DIMM-based NDP offloads comparison to rank-level logic (32 units) with 8× bandwidth; CPU does index traversal.
  • Hybrid partial-dimension/bit early termination stops fetching partway if the lower bound proves a vector is too far — zero accuracy loss.
  • Data layout transforms: pack high 𝑛 bits of multiple dims per 64 B, use dual-granularity fetch (coarse → fine), eliminate common prefixes.
  • Layout decided by sampling 100 vectors offline to measure prefix entropy (low → common prefix) and ET frequency (high → fine-grained).
  • Hybrid partitioning (1 kB sub-vectors per rank) + hot vector replication (entry points, top graphs) balances load.
  • Results: 5.26× speedup from NDP, 1.52× from ET (separate multiplicative effects), 78% + 31% energy reduction, <0.06 mm² per NDP unit.
Seven everyday pictures

How to think about ANSMET

Each picture stands alone. Every card ends with an honest note about where the analogy breaks.

The core idea · Early termination

Measuring a too-long stick

You need sticks under 10 cm. You start measuring one — at 6 cm it's still possible, but at 11 cm you know it's too long without measuring to the end. Early termination does this for vectors: if the partial distance (from the first few dimensions or bits) already exceeds the threshold, stop fetching the rest. You saved bandwidth and time, and the answer is still correct because you used a lower bound — the true distance can only be higher.

Where it breaks: a stick's length doesn't change; a vector's "true" distance is fixed once you fetch it all, but ANSMET bets you don't need to fetch it all to reject it.
Partial bits · High to low

Weighing heavy items first

You have a 10 kg limit. Start with the heaviest items (sign bit, exponent for floating-point). If a 5 kg box already puts you at 8 kg and another 3 kg box is coming, you know you'll exceed 10 kg without weighing the rest. ANSMET fetches high bits first (most significant) — they have the biggest impact on distance. If the partial distance from high bits already exceeds the threshold, stop.

Where it breaks: weights are independent; bits are positional. The analogy captures "big contributions first" but not the bit-level logic (setting missing bits to minimize distance).
Layout choice · Dimensions vs bits

Packing a suitcase

You have 64 bytes per trip (the 64 B memory access). Pack more dimensions but fewer bits per dimension (e.g., 2 bits × 256 dims), or more bits but fewer dimensions (e.g., 16 bits × 32 dims)? The optimal choice depends on the data: if all dimensions have similar common prefixes (low entropy), pack more dims with coarse bits to skip the prefix fast. If entropy is high early, pack fewer dims with fine bits to detect rejections sooner. ANSMET samples 100 vectors to decide.

Where it breaks: suitcase packing is a one-time choice; ANSMET uses dual-granularity (coarse then fine) and adapts per dataset, not a single packing.
Common prefix elimination · Save space

Dropping country codes

If 99% of your phone numbers share the same country code (+1), don't store it for each — store it once. ANSMET does this for vector data: if 99.9% of elements share the highest 8 bits (e.g., 11002 prefix), store that prefix once in the NDP unit and omit it from every vector, saving 8 bits × dimension. Outliers (0.1%) are flagged with an extra bit and stored in full. This is like quantization but optimized for early termination.

Where it breaks: phone codes are discrete; bit prefixes are continuous. The analogy captures "common data stored once" but not the bit-level prefix matching and outlier handling.
Hybrid partitioning · Load balance

Splitting a team project

You have 8 workers (memory ranks) and a big task (vector database). Vertical split: each worker does a piece of every vector (e.g., dimensions 0–31, 32–63, ...). Fast but needs aggregation. Horizontal split: each worker handles a subset of vectors. Avoids aggregation but risks imbalance if some vectors are hot. ANSMET uses hybrid: split each vector into 1 kB sub-vectors (vertical) and assign sub-vectors to rank groups (horizontal), then replicate hot vectors (entry points in HNSW) to all groups to balance load.

Where it breaks: teams coordinate verbally; memory ranks need data movement (cross-rank communication is slow). The analogy captures "split work, balance load" but not the bandwidth/latency tradeoffs.
Heterogeneous processing · CPU + NDP

Chef and prep cook

The chef (CPU) decides the menu and plates the dishes (index traversal, sorting results) — complex, irregular tasks. The prep cook (NDP) does the repetitive chopping and measuring (distance comparison) — simple, parallel tasks. They work simultaneously but on different parts of the job. ANSMET offloads the memory-bound comparison to NDP (32 units with 8× bandwidth) while the CPU handles the logic-heavy index traversal and result sorting.

Where it breaks: chefs and cooks share a kitchen; CPU and NDP share memory but not seamlessly (NDP has local DRAM; CPU polls results). The analogy captures "different tasks, different processors" but not the coordination overhead.
Adaptive polling · Result retrieval

Checking the oven

You put cookies in the oven (offload task to NDP). If you poll too early, you waste time opening the door (bandwidth). If too late, they burn (latency). ANSMET uses adaptive polling: estimate the expected latency from the early termination frequency distribution (measured during offline sampling), and poll at that time ± a small buffer. This saves 62% of result-collection overhead vs fixed polling (100 ns interval).

Where it breaks: ovens are predictable; NDP latency varies due to early termination randomness. The analogy captures "poll at the right time" but not the statistical estimation from the sampled distribution.
The full explanation

How it actually works

Section by section, with the paper's key mechanisms rebuilt as step-by-step explainers.

1 · The baseline: NDP without early termination

Start with a DIMM-based NDP architecture for ANNS. The system has 32 ranks total (4 channels × 2 DIMMs × 4 ranks per DIMM), and each rank has a buffer chip augmented with a NDP unit: 32 QSHRs (query status handling registers, 2148 B each = 1 kB query + 1 kB vector + 8 × (4 B addr + 4 B dist)) and a distance computing unit (16 × 32-bit adders/multipliers). The CPU does index traversal (HNSW graph search) and offloads distance comparison to NDP via encoded DDR commands:

configure(dim, metric, ET_params)         # once at start
set_query(qshr_id, query_vector)          # 16 × 64B WRITE
set_search(qshr_id, 8 × (addr, thr))     # 1 × 64B WRITE
# NDP fetches vector, computes distance, writes result
result = poll(qshr_id)                     # READ when done

Vector data are partitioned hybrid: each 128-dim FP16 vector (256 bytes) can be split into sub-vectors of 256 dims × FP16 = 512 bytes per sub-vector (2 sub-vectors per vector if needed), with sub-vectors assigned to different ranks within a rank group (vertical), while different vectors go to different rank groups (horizontal). Hot vectors (HNSW entry points, top-layer vertices) are replicated to all rank groups. This gives 5.26× speedup over CPU (Figure 6) by exploiting 8× bandwidth and rank-level parallelism.

2 · Hybrid early termination: partial dimensions + partial bits

For each vector, we fetch it in 64 B chunks. After each chunk, compute a distance lower bound 𝑑LB using the partially fetched data. If 𝑑LB already exceeds the current threshold (max distance in result set), reject and stop fetching.

Setting missing data conservatively: for Euclidean distance, compare fetched bits to query. If match, set missing bits same as query; if fetched > query, set missing to 0s; if fetched < query, set to 1s. For inner product (negative sum), set to 1s if unsigned or same sign, else 0s for a conservative lower bound. Example for query (0110₂, 0101₂):

partial (00␣␣, 01␣␣) → set to (0011, 0101)  # match then same
partial (11␣␣, 01␣␣) → set to (1100, 0101)  # 11 > 01 → 00
𝑑LB = √[(0011−0110)² + (0101−0101)²] = 3

This lower bound is always ≤ true distance, so rejecting on 𝑑LB > threshold is safe (zero accuracy loss). The paper shows 50–90% vectors are rejected, and early termination saves 25–53% of memory accesses (Figure 10).

3 · Data layout: dual-granularity fetch + common prefix elimination

To maximize early termination effectiveness, transform the layout offline so each 64 B chunk contains the most informative bits for many dimensions. The paper uses a dual-granularity scheme: after the common prefix, fetch coarse chunks (e.g., 8-bit steps for 𝑇C times) to quickly skip low-entropy range, then fine chunks (e.g., 2-bit steps) in high-entropy range to catch early terminations. The bit step sizes 𝑛C, 𝑛F, 𝑇C are chosen by sampling 100 vectors and minimizing total fetches using their early termination positions.

Common prefix elimination: if 99.9% of elements share the highest 𝑘 bits (e.g., 1100₂ for FP32), store that prefix once in the NDP unit and omit it from each vector, saving 𝑘 bits per dimension. Outliers (0.1%) are flagged with 1 OlVec bit per vector and 1 OlElm bit per element, and store their partial-match length + remaining bits in place. To ensure no accuracy loss, store non-compressed outlier vectors in a separate backup area and re-check if the compressed version is accepted (a few extra accesses, but rare).

🍯
Offline preprocessing: sample 100 vectors to measure prefix entropy and early termination frequency distribution. Use these to decide: (1) common prefix length (99.9% elements match), (2) dual-granularity bit steps (𝑛C, 𝑛F, 𝑇C), (3) distance threshold for sampling (90th percentile of pairwise distances, representing the 10% largest distances). Adds only 1.6% to graph construction time.

4 · Hybrid partitioning: 1 kB sub-vectors + hot replication

Purely vertical partitioning (split dimensions) maximizes parallelism but requires CPU aggregation of partial distances and wastes bandwidth if sub-vectors are small. Purely horizontal (split vectors) avoids aggregation but suffers load imbalance. ANSMET uses hybrid: split each vector into 1 kB sub-vectors (e.g., 2 sub-vectors for 128-dim FP16) and assign them to different ranks within a rank group (vertical), then assign vectors to different rank groups (horizontal). 1 kB is chosen because early termination is more effective with longer sub-vectors (can save more future fetches per termination).

To balance load, replicate hot vectors to all rank groups. Unlike recommendation systems where hot vectors are arbitrary, ANNS has clear hints: entry points and top-layer vertices in HNSW, centroids in IVF. For GIST, replicating the top 4 HNSW layers (5.27 MB, 0.14% of data) reduces load imbalance ratio from 1.49× to 1.05×. For zipf-skewed queries (𝑎=2.0), from 2.19× to 1.09×.

5 · Adaptive polling: estimating NDP latency

The CPU polls NDP units to retrieve results. Fixed polling (e.g., every 100 ns) wastes bandwidth if done too early or adds latency if too late. ANSMET uses adaptive polling: from the offline sampling, we have the early termination frequency distribution (Figure 3). This also gives the latency distribution of each comparison task. For multiple tasks, sum their distributions (considering issue time differences) to estimate the expected completion time, and poll at that time. This reduces result-collection overhead by 62% vs fixed polling (Figure 9).

Follow the numbers, step by step

Worked examples

Four traces with the paper's real numbers, carried through early termination and data layout decisions.

① Computing the distance lower bound from partial bits

Query: (0110₂, 0101₂). Vector partially fetched: (01␣␣₂, 01␣␣₂). What's the Euclidean distance lower bound, and when should we reject?

  1. Match the fetched bits: 01 (query) vs 01 (fetched) — they match in both dimensions.
  2. Set missing bits conservatively: when they match, set missing to same as query: (0110, 0101).
  3. Compute lower bound: 𝑑LB = √[(0110−0110)² + (0101−0101)²] = 0.
  4. Compare to threshold: if threshold is 2.0, 0 < 2.0 → continue fetching. If threshold is 0.5 after next fetch, re-check.
  5. General rule: reject when 𝑑LB > threshold. Here we didn't reject yet because 𝑑LB = 0 is always ≤ threshold.
🍯
Insight: the lower bound is always ≤ true distance because we set missing bits to minimize distance. So rejecting on 𝑑LB > threshold is safe (no false rejections, zero accuracy loss).

② Deciding dual-granularity bit steps from sampling

Dataset DEEP (96-dim FP32). Sample 100 vectors. Measure prefix entropy (Figure 3): low for bits 0–8, high for 9–28. How do we choose 𝑛C (coarse), 𝑛F (fine), 𝑇C (coarse rounds)?

  1. Identify low-entropy range: bits 0–8 (prefix entropy < 0.3). Use coarse steps to skip this fast.
  2. Identify high-termination range: bits 9–28 (ET frequency > 1%). Use fine steps to catch early terminations.
  3. Try different 𝑛C, 𝑛F, 𝑇C combinations: for each sampled vector, find its early termination position 𝑝ET (first bit where 𝑑LB > threshold). Calculate total fetches using Eq. 1 in paper: 64 × (⌈𝐷/𝑚C⌉ × 𝑇C + ⌈𝐷/𝑚F⌉ × ⌈(𝑝ET − 𝑛C𝑇C)/𝑛F⌉) where 𝑚C,F = ⌊512/𝑛C,F⌋ (64 B = 512 bits).
  4. Pick the minimum: sum total fetches over all 100 vectors. The 𝑛C, 𝑛F, 𝑇C that minimize this sum are chosen. For DEEP, this gives 𝑛C=8, 𝑛F=2, 𝑇C=1 (one coarse fetch of 8 bits, then fine fetches of 2 bits).
  5. Threshold for sampling: use 90th percentile of pairwise distances in the 100 vectors (representing the 10% largest distances, empirically optimal, Table 5 KL divergence).
💡
Why dual-granularity helps: uniform bit steps (e.g., always 4-bit) would fetch too slowly in low-entropy (wasting time) or too coarsely in high-termination (missing early rejections). Dual-granularity adapts to the data distribution.

③ Common prefix elimination with outliers

Dataset SPACEV (100-dim INT8). Sample 100 vectors. 99.9% of elements have prefix 1100₂ (4 bits). How much space is saved, and how are outliers handled?

  1. Space saved per normal vector: 4 bits (prefix) × 100 dims − 1 bit (OlVec flag) = 400 − 1 = 399 bits = 49.875 bytes per vector.
  2. Outlier overhead: 0.1% of 100 dims × 1 billion vectors = 100M outlier elements (not 100k). Each needs 1 OlElm bit + log₂(4) = 2 bits (partial match length) + remaining bits. Say 8 bits total per outlier element = 100M × 8 bits ≈ 100 MB extra.
  3. Backup vectors: if we want zero accuracy loss, store non-compressed outlier vectors in a separate area. For 0.1%, that's 1M vectors × 100 B = 100 MB extra (1% overhead). On accept, re-check the backup.
  4. Net result: 37.5% space saved (Table 5) plus 1.1% extra backup space with no accuracy loss under the paper's no-accuracy-loss setting. This differs from a simple calculation because the paper accounts for backup overhead and fine-grained bit handling.
🍯
Insight: common prefix elimination is like quantization but optimized for early termination (pack high bits separately) and flexible on accuracy (backup outliers for zero loss or not for more savings).

④ Hybrid partitioning: 1 kB sub-vectors + hot replication

Dataset GIST (960-dim FP32 = 3840 bytes per vector). 8 ranks. How is the data partitioned, and how much does hot replication help load balance?

  1. Sub-vector size: 1 kB chosen (Section 7.3). Each 3840 B vector is split into 4 sub-vectors of 960 B (240 dims × 4 B) each.
  2. Hybrid partitioning: assign 4 sub-vectors to 4 ranks within one rank group (vertical). Other vectors go to other rank groups (horizontal). With 8 ranks, we have 2 rank groups of 4 ranks each.
  3. Hot replication: HNSW top 4 layers (5.27 MB, 0.14% of data) are replicated to both rank groups. Without replication, load imbalance ratio (max load / avg load) is 1.49×. With replication, it's 1.05× (Table after Section 5.3).
  4. Why 1 kB works: early termination is more effective with longer sub-vectors (can save more future fetches per termination). 1 kB is the sweet spot (Figure 12): smaller (256 B, 512 B) is less effective, larger (2 kB) gives diminishing returns and complicates partitioning.
💡
Why hot replication helps ANNS: unlike recommendation systems where hot vectors are arbitrary, ANNS has clear hints (entry points, top-layer vertices). Replicating just 0.14% of data reduces imbalance from 1.49× to 1.05×, enabling near-linear scaling to 32 NDP units.

Problem 1 · easy
Query (0101₂, 1001₂) vs vector partially fetched (01␣␣₂, 10␣␣₂). What's the Euclidean distance lower bound 𝑑LB?
Show answer
Compare fetched bits: 01 vs 01 (match), 10 vs 10 (match). Set missing bits same as query: (0101, 1001). 𝑑LB = √[(0101−0101)² + (1001−1001)²] = 0. (Matches always give 𝑑LB = 0 until they diverge.)Misconception: that partial bits always give non-zero lower bound. If they match so far, the lower bound is 0 until a mismatch appears.
Problem 2 · medium
ANSMET uses dual-granularity fetch (coarse 8-bit, fine 2-bit). For a 96-dim vector with 32-bit elements, how many dimensions fit in one 64 B coarse fetch, and one fine fetch?
Show answer
Coarse: 𝑚C = ⌊64 × 8 / 8⌋ = ⌊512 / 8⌋ = 64 dims. Fine: 𝑚F = ⌊512 / 2⌋ = 256 dims (all 96 fit, with padding). So coarse fetch gets high 8 bits of 64 dims; fine fetch gets next 2 bits of all 96 dims.Misconception: that coarse/fine refers to number of dimensions. It's bits per dimension (8 vs 2), not dims per fetch.
Problem 3 · medium
Common prefix elimination saves 4 bits per dimension for 100-dim vectors. Dataset has 1B vectors. How much space is saved if 0.1% are outliers (stored in full)?
Show answer
Normal vectors: 1B × 99.9% = 999M vectors × (4 × 100 − 1) bits = 999M × 399 bits ≈ 49.875 GB saved. Outliers: 0.1% × 100 dims × 1B vectors = 100M outlier elements. Each needs extra bits (OlElm flag + partial match + remaining), approximately 100M elements × 1 byte = 100 MB extra. Net: significant space reduction. The paper reports 37.5% saved space plus 1.1% backup overhead at 0.1% outliers for zero accuracy loss (Table 5).Misconception: confusing 0.1% outlier vectors with 0.1% outlier elements. The paper measures outlier elements (individual dimension values), not entire vectors.
Problem 4 · hard
Why does ANSMET use 1 kB sub-vectors for hybrid partitioning, not 64 B (one access granularity)? Explain in terms of early termination effectiveness.
Show answer
With 64 B sub-vectors (e.g., 16 dims × 4 B), each rank only has 16 dims. Early termination on this rank can only save future fetches of those 16 dims. Even if 𝑑LB > threshold locally, other ranks must still fetch their 16 dims because there's no fast cross-rank communication. So ET saves at most 1/N of accesses (N = number of ranks). With 1 kB sub-vectors (256 dims), each rank has enough dims that its local 𝑑LB is often sufficient to reject the entire vector, saving all future fetches on this rank. The paper finds 1 kB is the sweet spot: smaller is less effective, larger gives diminishing returns (Figure 12).Misconception: that smaller sub-vectors are always better (less fetch per access). They reduce ET effectiveness because the partial distance is less informative.
Problem 5 · medium
ANSMET uses adaptive polling instead of fixed-interval polling (e.g., every 100 ns). How does it estimate the polling time, and why does this reduce overhead?
Show answer
During offline sampling, ANSMET measures the early termination frequency distribution (Figure 3). This also gives the latency distribution of each comparison task (more early terminations = shorter latency). For a batch of tasks, sum their latency distributions (considering issue time differences) to estimate expected completion time. Poll at that time ± a small buffer. This avoids polling too early (wastes bandwidth) or too late (adds latency). The paper shows this reduces result-collection overhead by 62% vs fixed 100 ns polling (Figure 9).Misconception: that polling time is fixed by the hardware. It depends on early termination frequency, which varies by dataset and threshold.
Problem 6 · hard
Why does ANSMET replicate only the top 4 HNSW layers (5.27 MB, 0.14% of data) to all rank groups, not replicate all vectors? What's special about top layers?
Show answer
In HNSW, every query starts at the top layer entry point and traverses down. Top-layer vertices are accessed by all queries, making them hot. Replicating them to all rank groups balances load: no single rank group is overloaded with entry-point accesses. Lower layers have more vertices but are accessed less frequently per vertex (only by queries whose traversal path reaches them). Replicating all would waste space (100 GB → 200 GB for 2 rank groups) for little load-balancing benefit. The paper shows 0.14% replication reduces imbalance from 1.49× to 1.05×, which is near-perfect.Misconception: that all hot vectors are equally hot. In ANNS, the index structure (HNSW top layers, IVF centroids) provides clear hints about which vectors are most hot.
Problem 7 · easy
ANSMET achieves 5.26× speedup from NDP and 1.52× from early termination. Are these multiplicative (5.26 × 1.52 = 8×) or additive (5.26 + 1.52 = 6.78×)?
Show answer
Multiplicative. The 5.26× is vs CPU-Base (no NDP, no ET). The 1.52× is NDP-ETOpt vs NDP-Base (NDP but no ET). These are separate effects measured in the evaluation: 5.26× is average/geomean NDP speedup, and 1.52× is additional ET speedup. The paper reports them as separate results, not a directly measured combined 8×.Misconception: that the paper directly measured or stated an exact combined 8× headline. The paper reports separate NDP and ET effects; multiplication is a reader's arithmetic combination.
Problem 8 · medium
Figure 10 shows fetch utilization improves from 6.0% (NDP-Base) to 11.1% (NDP-ETOpt). Why is it still so low (~11%)? Where are the remaining 89% of fetches going?
Show answer
The 89% are still ineffectual (rejected vectors), but early termination can't catch them all. Two reasons: (1) At the start of a query, the threshold is loose (based on initial entry points, not final 𝑘-NN), so many vectors are within the threshold until it converges. (2) Some vectors are close to the threshold — their 𝑑LB doesn't exceed it until late in the fetch sequence (e.g., last 1–2 fetches), so ET saves little. The paper notes this gap is due to "over-estimated distance bounds at the beginning of each query" (Section 7.2). Further improvement would require tighter initial thresholds (e.g., learned from similar queries).Misconception: that early termination can eliminate all ineffectual fetches. It can only reject vectors whose 𝑑LB exceeds the threshold early; those close to the threshold or when threshold is loose are not caught.
Quick answers

Frequently asked questions

Ten questions about ANSMET's design and tradeoffs. Tap any to expand.

Q1What does "zero accuracy loss" mean? Isn't ANNS already approximate?▾
ANNS is approximate in that it uses an index (HNSW, IVF) to avoid checking all vectors, trading accuracy for speed. But given the set of vectors the index selected to check, early termination must not introduce additional approximation errors. "Zero accuracy loss" means ANSMET's lower-bound rejection is conservative: if 𝑑LB > threshold, the true distance is definitely > threshold, so rejection is safe. This is unlike prior ET methods that predict rejection heuristically and may miss true positives (Section 3, Section 4.1).
Q2Why DIMM-based NDP instead of HBM or 3D-stacked memory?▾
HBM and 3D-stacked (e.g., HMC) offer extremely high bandwidth (up to 1 TB/s) but limited capacity (tens of GB). Vector databases are often terabyte-scale (1B vectors × 1 kB = 1 TB). DIMM-based NDP uses commodity DRAM modules with capacity up to 1 TB per DIMM (Samsung DDR5, cited in paper), at the cost of lower per-DIMM bandwidth. But with 8 ranks × 8 channels (64 ranks total), ANSMET's aggregate bandwidth is still high. The paper argues DIMM-NDP is the better balance for ANNS (Section 2.2).
Q3What's the hardware cost of ANSMET? Is 0.06 mm² per NDP unit realistic?▾
Each NDP unit has 32 QSHRs (2148 B each = 67 kB SRAM) + 16-wide 32-bit adders/multipliers. The paper cites CACTI at 22 nm for 0.06 mm² per unit, which is conservative (older node). 32 units total = 1.92 mm², compared to a 100 mm² DIMM buffer chip (Section 6). Energy is 300 mW per unit × 32 = 9.6 W, acceptable for a memory system. The paper also notes this is smaller than prior DIMM-NDP designs (TensorDIMM, RecNMP) which have 20+ kB of buffers (Section 6).
Q4How much does offline preprocessing cost? Is 1.6% overhead acceptable?▾
Preprocessing includes: (1) sampling 100 vectors to decide layout parameters (~seconds), (2) transforming data layout for all vectors (parallelizable, ~10s for SIFT 1M, ~45s for BigANN 1B), (3) graph construction (HNSW) which dominates (240s for SIFT 1M, >5000s for BigANN 1B). The layout transformation adds only 1.6% to graph construction time (Table 4). Since preprocessing is done once offline and amortized over many online queries, this is acceptable. (Note: dynamic vector insertion would need incremental layout updates, not addressed in paper.)
Q5Why sample 100 vectors, not more? Is 1e−7 sampling rate (for 1B dataset) enough?▾
The paper empirically finds 100 vectors balance sampling cost and accuracy. Figure 11(a) shows KL divergence between sampled ET distribution and true distribution: 100 vectors give KL = 0.01 (very close to true), while 10 vectors give 0.45 (worse). More than 100 (200) gives no further improvement. Sampling rate of 1e−7 is tiny but sufficient because the goal is to measure distribution (prefix entropy, ET frequency), not individual vector properties. The dataset is assumed to be i.i.d. (common in ML), so 100 samples capture the population statistics well.
Q6What happens if a vector is an outlier (doesn't match the common prefix)?▾
The paper allows up to 0.1% of elements to be outliers (Table 5). Each outlier is flagged with 1 OlElm bit, stores its partial-match length (log₂ of prefix length, e.g., 2 bits for 4-bit prefix), and remaining bits in place. Lowest bits are dropped to fit. On distance computation, these dropped bits are recovered conservatively (set to minimize distance, as in Section 4.1). If you want zero accuracy loss, store non-compressed outlier vectors in a separate backup area and re-check on accept (a few extra accesses, but rare since outliers are 0.1%). The paper defaults to this backup approach (Section 4.3).
Q7Why does ANSMET use dual-granularity fetch (coarse → fine) instead of uniform bit steps?▾
Figure 3 shows that prefix entropy is low early (common prefix zone) and high later (high-termination zone). Uniform steps (e.g., always 4-bit) would waste time in low-entropy (should skip with coarse 8-bit) or miss early rejections in high-entropy (should use fine 2-bit to detect divergence sooner). Dual-granularity adapts: coarse (e.g., 8-bit × 1 round) to skip low-entropy fast, then fine (e.g., 2-bit × many rounds) to catch early terminations. The paper shows this is more effective than simple heuristic (NDP-ET vs NDP-ET+Dual: 8.2% extra speedup, Section 7.1).
Q8How does ANSMET handle multiple distance metrics (Euclidean, inner product, cosine)?▾
The distance computing unit in Figure 5(d) has 16 adders and multipliers, each 32-bit, designed to support multiple metrics (like prior work, CXL-ANNS). For Euclidean, use subtraction (VSUB) → square (VMUL) → sum (VADD) → sqrt (VSQRT). For inner product, skip subtraction and just do element-wise multiply → sum. For cosine, normalize vectors during preprocessing (convert to unit vectors), then inner product = cosine. The missing-bit recovery logic also changes per metric: for Euclidean, set bits to minimize |diff|; for inner product, set to 1 if unsigned or same sign, else 0 for conservative lower bound. This is configured once at query start via the configure instruction (Section 5.2).
Q9Why does NDP-BitET (bit-serial ET from BitNN) perform worse than ANSMET on some datasets?▾
NDP-BitET uses fixed 1-bit steps (bit-serial), which is only effective for datasets with sufficiently large dimensions. For SIFT (128-dim), each 64 B fetch contains only 128 bits (1 bit × 128 dims), wasting 75% of the fetch (64 B = 512 bits). For GIST (960-dim), 1-bit × 960 dims = 120 bytes > 64 B, so it needs multiple fetches per bit step, but still better than baseline. ANSMET's hybrid approach (partial dims + partial bits with adaptive granularity) works across all dimension sizes by packing more bits of fewer dims (e.g., 8 bits × 64 dims) when dims are small, or more dims of fewer bits (e.g., 2 bits × 256 dims) when dims are large (Figure 6).
Q10What are the limitations of ANSMET? When would it not help?▾
Three main limitations: (1) Low rejection rate: if most vectors are within the threshold (e.g., very large 𝑘 or loose efSearch), early termination saves little. (2) Non-i.i.d. data: if vectors have wildly different distributions per cluster, the sampled common prefix may not apply globally (high outlier rate). (3) Dynamic insertion: the paper focuses on static databases; inserting new vectors would need incremental layout updates, which is expensive. The authors note these as future work (Section 8). Also, ANSMET targets billion-scale datasets where memory bandwidth is the bottleneck; for small datasets that fit in LLC, CPU is already fast enough.

Glossary & further reading

Core terms
ANSMETApproximate Nearest Neighbor Search with Near-Memory Processing and Hybrid Early Termination — this paper's system.
ANNS (Approximate kNN Search)Find 𝑘 nearest neighbors to a query vector, trading accuracy for speed using indexes (HNSW, IVF).
HNSWHierarchical Navigable Small Worlds — a graph-based index with multiple layers for fast traversal.
NDP (Near-Data Processing)Compute logic placed near memory (here in DIMM buffer chip) to exploit high bandwidth.
Early termination (ET)Stop fetching/computing partway if partial data proves the result is out of range.
Hybrid partial-dimension/bit ETANSMET's novel ET: fetch subset of dims or bits, estimate distance lower bound, reject if exceeds threshold.
Distance lower bound (𝑑LB)Conservative estimate of full distance from partial data; always ≤ true distance, so rejecting on 𝑑LB > threshold is safe.
Dual-granularity fetchUse coarse bit steps (e.g., 8-bit) in low-entropy range, fine steps (e.g., 2-bit) in high-termination range.
Common prefix eliminationOmit shared high bits (e.g., 1100₂) from all vectors, store once in NDP unit to save space and fetches.
Hybrid partitioningSplit each vector into 1 kB sub-vectors (vertical) and assign to rank groups (horizontal) to balance load.
QSHR (Query Status Handling Register)Per-NDP-unit buffer (2148 B) storing query vector, search tasks, and status for distance comparison.
Adaptive pollingEstimate NDP completion time from ET frequency distribution, poll at that time to reduce result-collection overhead.
Supporting terms
IVF (Inverted File Index)Cluster-based index grouping vectors by centroids; search only within top-𝑘 clusters.
Euclidean distance𝑑 = √Σ(𝑎ᵢ−𝑏ᵢ)² — the L2 norm between two vectors.
Inner-product distance𝑑 = −Σ𝑎ᵢ𝑏ᵢ — negated dot product (maximizing it = minimizing distance).
Cosine distanceNormalized inner product by vector magnitudes; after preprocessing normalization, same as inner product.
Prefix entropy−Σ𝑃ᵢlog𝑃ᵢ where 𝑃ᵢ is frequency of 𝑖th prefix value; measures diversity of prefixes.
ET frequencyHow often a given prefix length triggers early termination (𝑑LB > threshold).
Outlier vectorVector whose elements don't match the common prefix; flagged and stored differently.
RankOne independently addressable set of DRAM chips on a DIMM; ANSMET has 32 ranks total.
Vertical partitioningSplit dimensions of each vector across ranks; maximizes parallelism but needs aggregation.
Horizontal partitioningAssign different vectors to different ranks; avoids aggregation but risks load imbalance.
Hot vector replicationDuplicate frequently accessed vectors (HNSW entry points, top layers) to all rank groups to balance load.
Sampling-based preprocessingUse 100 vectors to measure prefix entropy, ET frequency, and decide layout parameters offline.
