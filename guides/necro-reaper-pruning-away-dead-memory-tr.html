ASPLOS 2025 · A Field Guide

Stop writing ghost data to memory

Up to 50% of memory bandwidth in warehouse-scale computing is wasted writing dead objects and reading uninitialized bytes (geomean 26%). This is the story of Necro-reaper, a profile-guided software/hardware co-design that prunes it away.

📄 "Necro-reaper" · Apostolakis, Kennelly, Li & Ranganathan (Google) 🎓 Some cache & memory tiering background assumed ⏱ ~25 min read
💀

A cache line for deallocated memory needn't be written back. A fetch-on-miss for freshly-allocated space is reading garbage. Hardware can't see this alone; software must tell it when memory dies and is born.

Before you begin

Seven concepts the paper assumes

Each card covers one building block for Necro-reaper. Expand any to read — order matters for newcomers.

dead data / writebacks uninitialized / fetch-on-miss live traffic Necro-reaper's machinery
1Why memory bandwidth saturates WSC machines▾

Per-core memory bandwidth has plateaued over the last decade (~3–5 GB/s) while working sets keep growing. For many latency-sensitive WSC workloads, some state-of-the-art server platforms can saturate memory bandwidth at even 50% CPU utilization — bandwidth, not compute, is the bottleneck. For your tiering/CXL research context: bandwidth pressure makes memory-tier placement critical.

Now you're ready: Necro-reaper targets the wasted portion (up to 50%, geomean 26%) of that scarce bandwidth in WSC — bandwidth spent moving dead or uninitialized data.
2LLC writebacks and fetch-on-write misses▾

A write-back last-level cache (LLC) writes dirty evicted lines to memory. A write-allocate policy fetches a line from memory on a write miss (even if the write will overwrite the whole line later). Both are standard cache policies that create memory traffic.

The paper's observation: when that line corresponds to deallocated memory (a writeback) or freshly-allocated but uninitialized space (a fetch-on-miss), the traffic is semantically useless.

Now you're ready: Necro-reaper's goal is to suppress these two categories of useless memory traffic — dead writebacks and dead fetches.
3Allocation patterns in WSC vs SPEC CPU▾

SPEC CPU 2017 allocates ~0.04 times per 1000 instructions (APKI); allocations happen mostly at startup. For WSC workloads, Necro-reaper measured 0.62 APKI during steady state — an order of magnitude higher — and many are short-lived objects ≥64B. High allocation rate + short lifetimes + cache-line-sized+ objects = high dead-traffic exposure.

Now you're ready: Necro-reaper shows that 14–50% (geomean 32%) of allocations in WSC are both short-lived and bigger than a cache line — the sweet spot for dead traffic.
4The hardware is oblivious to (de)allocations▾

A cache controller sees bytes, addresses, and dirty bits. It cannot tell when a line corresponds to freed memory or uninitialized bytes — those are software-level semantic concepts. Conversely, the software (allocator, compiler) doesn't see cache evictions or misses. Neither alone can fix dead traffic.

Now you're ready: Necro-reaper introduces new ISA instructions so the software can tell the hardware "this line is dead, don't write it back" and "install a clean line on allocation, don't fetch."
5Profile-guided optimization (PGO) in WSC▾

WSC increasingly uses profile-guided compilation: collect runtime profiles (branch frequencies, heap access patterns, PMU events) on representative loads, then recompile to optimize hot paths. AutoFDO, BOLT code layout, and heap profiling are production techniques at Google and elsewhere.

Now you're ready: Necro-reaper extends this profile-guided approach to heap (de)allocation and access patterns, guiding when to install and invalidate lines — a pragmatic fit for WSC's existing vertical integration.
6C++ vs managed languages — no zero-init requirement▾

Java requires zero-initialization of new allocations (often 25% of total memory traffic). C++ does not — reading uninitialized memory is undefined behavior, so the language permits skipping zero-init. Thus, any optimization for explicit languages must be selective: install only the portion that will be used soon, or pay a cache pollution penalty.

Now you're ready: Necro-reaper's profile-guided installation heuristic installs the first N% of an allocation (determined by profile or static heuristic), balancing dead-fetch savings against cache pollution.
7Existing cache-control ISA instructions (limited)▾

AArch64 has DC ZVA (zero-install a line without fetch, but marks it dirty → unnecessary writeback). x86 has clflush (flushes + writes back). AArch64 has DC IVAC (invalidate without writeback, but only in privileged execution). No userspace instruction can both install clean and invalidate without writeback.

Now you're ready: Necro-reaper proposes DC IZVA (install clean zero line) and enables DC IVAC for userspace (with security hardening, discussed later) to close the gap.

The whole story in one place

Necro-reaper in a nutshell

The problem, the fix, the profile-guided approach, and the results — all in about three minutes.

Some state-of-the-art server platforms can saturate memory bandwidth at even 50% CPU utilization in WSC. Studying that traffic, the authors discovered up to 50% (geomean 26%) is dead: writebacks of deallocated memory and fetches of uninitialized bytes. For C++ WSC workloads with high allocation rates (0.62 APKI, vs 0.04 for SPEC CPU) and many short-lived objects ≥64B, this dead traffic is both large and fixable.

The hardware can't fix it alone (it doesn't know when memory is allocated or freed). The software can't fix it alone (it doesn't see cache evictions or misses). Prior work [36,37,45,56] proposed ISA instructions but required hardware tracking tables or were tied to garbage collection. Necro-reaper takes a pragmatic WSC-tailored approach: profile-guided software tells the hardware when to act, avoiding hardware tracking.

Two new ISA instructions (variants of AArch64's existing instructions): DC IZVA installs a clean zero line without fetching from memory (unlike DC ZVA, which marks lines dirty); and DC IVAC (userspace-enabled with hardening) invalidates a line without writeback. The memory allocator inserts these instructions: on allocation, install the profiled-or-heuristic portion of the object; on deallocation, invalidate lines if the profile shows they won't be quickly reused. The compiler instruments the workload to profile heap access patterns (max offset accessed per allocation site) and reuse distance per size-class, then replaces allocation calls with extended variants that pass the install-portion and skip-invalidate hints to the allocator.

The payoff (10 diverse WSC workloads, AArch64 traces, 32MB LLC): 26% geomean reduction in memory traffic (up to 50% for some workloads), translating to 6% geomean IPC speedup (up to 13%). Profile-guided installation-only (without invalidation) yields 3.9% IPC speedup; static heuristics (75% of object size, ≤128KB, no profile) yield 2.1%. For context, the oracle strategy (install on initialization misses with perfect detection, profile-guided invalidation) yields 9% IPC speedup, so Necro-reaper captures 63% of the headroom with pragmatic, deployable techniques.

🍯 Key takeaways

  • Up to 50% (geomean 26%) of memory traffic in WSC is dead — wasted bandwidth writing deallocated objects and reading uninitialized bytes.
  • High allocation rate (0.62 APKI vs 0.04 for SPEC CPU) + short-lived objects ≥64B = high dead-traffic exposure in WSC.
  • New ISA instructions: DC IZVA (install clean zero line) and DC IVAC (userspace-enabled invalidate without writeback).
  • Profile-guided heuristics for installation (install N% of object based on profiled access pattern) and invalidation (skip if reuse distance <1MB).
  • Static fallback: install first 75% of object, ≤128KB total, if no profile available.
  • Geomean IPC speedup: 6% (up to 13%); memory traffic reduction: 26% (up to 50%).
  • Profile-guided installation alone (no invalidation): 3.9% IPC speedup — shows installation is the bigger win.
Five everyday pictures

How to think about Necro-reaper

Each picture stands alone. Every card ends with where the analogy breaks.

The core problem · Dead traffic

Hauling away a demolished house

A house is demolished (memory deallocated), and its remains sit on the lot (dirty line in LLC). The garbage truck (writeback on eviction) hauls the rubble to the landfill (memory) — but the lot is already cleared; the hauling trip is wasted. Similarly, reading a blueprint for a not-yet-built house (fetch-on-miss for uninitialized allocation) is reading meaningless data. Necro-reaper tells the truck "skip the demolished lot" (invalidate) and says "the blueprint is blank, don't fetch" (install clean).

Where it breaks: real garbage trucks can see the rubble; a cache controller can't tell a line is dead without software hints. Also, the analogy doesn't capture when to invalidate (profile-guided reuse-distance check).
Installation · Selective prefetch

Stocking a new apartment

You move into a new apartment (allocation). A static heuristic says "stock 75% of the kitchen items immediately" — good for most people, but you might not use some items for months (cache pollution). A profile-guided heuristic says "you only ever use 6% of the kitchen in the first week, stock that" — no waste, but requires knowing your habits (profiling). Necro-reaper does the latter: install only the portion of the allocation the profile says you'll access soon.

Where it breaks: real apartments don't magically unstock items if unused; caches do evict under pressure. Also, the analogy suggests "stocking" is free; installing lines costs instruction overhead (though the paper shows it's worth it).
Invalidation · Reuse distance

Hotel checkout and the next guest

A guest checks out (deallocation). If the hotel knows the room will be rebooked within an hour (reuse distance <1MB), skip cleaning/invalidation — the next guest (allocation) will immediately "write over" the room anyway. If the room will sit empty for days, clean it now (invalidate) to free up cleaning staff (cache space). Necro-reaper profiles reuse distance per size-class: if median reuse is small, skip invalidation; else, invalidate on free.

Where it breaks: real hotel rooms have a physical location; address space is virtual and can be mapped anywhere. Also, the analogy suggests cleaning is always visible; cache invalidation is a performance hint the hardware can ignore if benign.
Co-design · Software/hardware split

A thermostat and the HVAC

The thermostat (software: compiler + allocator) reads the room temperature (profiles heap access patterns) and tells the HVAC (hardware: cache + ISA instructions) when to turn on/off. The HVAC can't sense room temperature alone; the thermostat can't move air. Necro-reaper's co-design: software profiles and inserts hints (DC IZVA, DC IVAC), hardware executes them (install/invalidate lines). Neither alone suffices.

Where it breaks: a thermostat continuously senses; Necro-reaper profiles once (or periodically with AutoFDO-style re-profiling). Also, the analogy suggests the HVAC trusts the thermostat blindly; the hardware can ignore hints if they'd cause correctness issues (though the paper ensures they won't).
Scaling · LLC size sensitivity

Bigger warehouse, more ghost inventory

A small warehouse (8MB LLC) has little room; items (cache lines) are quickly evicted, so only very-short-lived allocations survive to become dead traffic. A giant warehouse (128MB LLC) keeps items longer; more objects survive to deallocation → more dead writebacks. The paper shows headroom (oracle) savings increase 78% from 8MB to 128MB LLC — bigger cache = more dead traffic opportunity.

Where it breaks: real warehouses don't automatically write inventory to a ledger on eviction; LLCs do (writeback). Also, the analogy suggests "ghost inventory" is proportional to size; the paper shows it depends on allocation lifetimes vs cache residency, not just size.
The full explanation

How it actually works

Section by section, with the paper's key steps and numbers.

1 · The problem — 26–50% of traffic is dead

The paper traces 10 diverse WSC workloads (search, databases, ads, data-processing) during steady-state loadtests on AArch64. For a 32MB inclusive LLC, 26% geomean (up to 50% for some workloads) of total memory traffic is dead: heap writebacks of deallocated objects + fetches-on-miss for uninitialized allocations. WSC workloads allocate 0.62 APKI (vs 0.04 for SPEC CPU), and 14–50% of allocations (geomean 32%) are both short-lived (lifetime ≤ avg LLC residency) and ≥64B — the sweet spot for dead traffic.

🌱
Why WSC is exposed: high allocation rate + short lifetimes + large objects mean lines enter the LLC (on init miss), sit there until the object dies, then get written back as dead data. Lower allocation rates (SPEC CPU) or smaller objects (<64B, not targeted) don't exhibit this pattern as strongly.

2 · New ISA instructions — DC IZVA and userspace DC IVAC

Necro-reaper proposes two AArch64 ISA instruction variants:

InstructionActionDifference from existing
DC IZVAInstall a clean zero line in L2/LLCExisting DC ZVA marks the line dirty, causing writeback on eviction if not overwritten
DC IVACInvalidate (evict) a line without writebackExisting DC IVAC is privileged-only (EL>1); Necro-reaper enables it for userspace (EL0) with hardening

Both are treated as stores for memory ordering and speculation (no out-of-order or mispeculated overwrites). For cache coherence (MESI): DC IZVA transitions to E (exclusive clean) if no dirty copy exists, else M (modified, to preserve security — avoid rolling back a dirty line). DC IVAC transitions to I (invalid) and issues a BusInv to invalidate other copies without writing back any dirty copies.

⚠
Security hardening: userspace DC IVAC could leak cross-process data (OS zeros a page, attacker invalidates before reading → sees victim's old data). The paper proposes OS-based hardening (non-temporal zero by OS) or a hardware "invalidatable" bit (set on DC IZVA, reset on privileged writes) so DC IVAC is a no-op on non-installed lines. See §4.3.3 and FAQ Q7 for details.

3 · Profile-guided installation — install N% of the object

Install lines on allocation. But which lines? The paper explores three strategies:

  • Oracle (headroom): install on initialization miss (requires hardware tracking of uninitialized memory, as in prior work [36,45]). Assumes 100% detection of init writes. Geomean IPC speedup: 9.0%.
  • Static heuristic: install the first 75% of the object, ≤128KB total. Determined by studying initialization patterns (Fig. 4 in paper): allocations are mostly but not fully initialized, and big/small objects initialize at similar paces. Geomean IPC speedup: 2.1%.
  • Profile-guided (Necro-reaper): instrument the workload to track max offset accessed per allocation site (callstack+source location), average across dynamic instances, install that portion (rounded up to cache-line granularity). Profile coverage: 85% geomean (51% for bigquery, 98% for bigtable). Falls back to static heuristic if no profile. Geomean IPC speedup: 5.9% (full optimization with invalidation), 3.9% (installation-only).

Why profile-guided wins: it installs just enough to cover the accessed portion without overfilling the cache (cache pollution). For example, the search workload: profile shows allocation at main:1:8 accesses only 6% of its 128B object → install 1 line (64B); allocation at main:6:8 (no profile, input-dependent path) → static heuristic installs 3 lines (75% of 256B).

4 · Profile-guided invalidation — skip if reused quickly

Invalidate lines on deallocation. But when? Always invalidating generates coherence traffic and can hurt performance if the address space is quickly reused by the allocator (next allocation doesn't get installed → miss). Necro-reaper profiles reuse distance per power-of-two size-class: total allocated bytes between deallocation of a virtual address and its reuse by the allocator. The paper reports median reuse distances (e.g., 7.2KB for search, 4.0MB for flume in size class (64,128]) and uses them to guide skip-invalidation decisions.

Why per-size-class? Reuse patterns depend on the allocator (TCMalloc's size classes) and the workload's allocation patterns, not the deallocation callstack. For example, search has median reuse of 7.2KB for size (64,128], 28.7MB for (2048,4096]; flume has 4.0MB and 13.4KB respectively — no one static heuristic works.

🍯
Invalidation headroom vs overhead: profile-guided invalidation adds 2.0% IPC speedup on top of installation-only's 3.9% (total 5.9%). Without the profile-guided skip, the bigtable workload sees a slowdown due to invalidating quickly-reused lines. Profile coverage for invalidation: 100% (coarse-grained per-size-class aggregation).

5 · Compiler & allocator integration

The compiler (LLVM-based, profile-guided flow similar to AutoFDO [21]):

  1. Instruments the target workload to profile heap allocations and accesses: on each allocation, log the callstack+source location; on each access, update the max offset accessed for that allocation; on deallocation, record the reuse distance for the size class.
  2. Profiles the instrumented workload with representative inputs (ideally production-like to minimize mismatch).
  3. During optimization, replaces allocation calls (aligned_alloc, malloc) with extended variants (aligned_alloc_install(size, install_portion)) that take the profiled initialization portion (or static heuristic if no profile). Replaces deallocation calls (free) with free_invalidate(p) that checks the size-class profile.

The memory allocator (TCMalloc, extended):

  • aligned_alloc_install(size, install_portion): calls the underlying allocator, then loops from offset 0 to size * install_portion (rounded up to cache-line boundary), issuing DC IZVA for each cache line.
  • free_invalidate(p): looks up the size (from TCMalloc's size-class metadata), calls free(p, size), then checks if the size class is marked "skip invalidation" in the profile; if not, loops over all cache lines fully contained within the deallocated object, issuing DC IVAC for each.
💡
Why the allocator, not the compiler? The compiler doesn't see the runtime size (often not compile-time constant). The allocator knows the size (size-class lookup or sized deletion [25]) and is already a custom component in WSC (TCMalloc). Thus, the allocator is the natural place to insert the ISA instructions, guided by compiler-provided hints.

6 · Performance results — 6% IPC, 26% traffic reduction

Evaluation: 10 WSC workloads, AArch64 traces (50s steady-state, DynamoRIO memtrace), gem5 single-core simulation (Arm Neoverse V2 model, 8-issue OOO, 2MB L2, 1MB exclusive SLC slice, 4GB/s per-core bandwidth, 32MB LLC for traffic analysis). Key results:

MetricNecro-reaperStaticOracle
Memory traffic reduction26% geomean (up to 50%)19% geomean41% geomean (headroom)
IPC speedup5.9% geomean (up to 13.4%)2.1% geomean9.0% geomean

Peak mem BW reduction12.4% geomean (p99)——
L2 MPKI reduction12.8% geomean (full Necro-reaper, mostly store misses)——

Necro-reaper captures 63% of oracle headroom (5.9% / 9.0% IPC) with pragmatic, deployable techniques. Profile-guided invalidation adds 2.0% IPC on top of installation-only's 3.9%. The static heuristic (no profile) yields 2.1% IPC, showing that even without profiling, the approach is beneficial.

🍯
Why speedup < traffic reduction: 26% traffic reduction → 6% IPC speedup (not 26%) because (i) not all traffic is on the critical path; (ii) bandwidth is constrained but not always saturated; (iii) some workloads are compute-bound. For bandwidth-sensitive workloads (search, flume), speedup is higher (8–13%).
Follow the numbers, step by step

Worked examples

Four traces from the paper's evaluation and methodology, carried all the way through.

① Computing the static heuristic from initialization patterns

The search workload (32MB LLC): what static heuristic maximizes init-miss coverage while minimizing cache pollution?

  1. Observation 1 (Fig. 4a): 35% of LLC init misses come from objects ≥16KB. Thus, we can't just "install all small objects"; we need to install part of big ones too.
  2. Observation 2 (Fig. 4b): median and last-line initialization times are similar across sizes (small and big objects initialize at the same pace). So a fixed percentage of size (not fixed bytes) makes sense.
  3. Observation 3 (Fig. 4c): ~75% of objects are initialized across all sizes. Thus, "install first 75%" covers most accesses.
  4. Upper bound: ≤128KB total installed bytes (based on distribution of init misses and LLC size). For a 200KB object, install 75% = 150KB, but cap at 128KB.
  5. Result: the static heuristic is "install first 75% of object, rounded up to cache-line boundary, ≤128KB total."
💡
Why 128KB cap? The cap is 128KB = 0.39% of the 32MB LLC (not 4%, which would be 1.28MB). This avoids installing too much per allocation, limiting cache pollution risk for most allocations (75% < 128KB).

② Profile-guided installation for a single allocation site

Allocation at main:1:8 allocates 128B (2 cache lines), but profile shows only 8 bytes (6%) accessed. How does Necro-reaper decide?

  1. Profile: max offset accessed = 8B (during profiling, the first 8 bytes of the 128B object were touched).
  2. Compute install portion: 8B / 128B = 6.25%, round up to next cache line = 1 line (64B).
  3. Compiler: replaces p1 = aligned_alloc(128) with p1 = aligned_alloc_install(128, 0.0625).
  4. Allocator: aligned_alloc_install(128, 0.0625) calls the underlying allocator, then loops from offset 0 to 128*0.0625 = 8B (round up to 64B = 1 line), issuing DC IZVA for 1 line.
  5. Hardware: installs 1 clean zero line at p1+0, avoiding a fetch-on-miss for the first access (the *p1 = val write hits in cache).
🍯
Compare to static: static heuristic would install 75% of 128B = 96B = 2 lines, wasting 1 line (cache pollution) since only the first 8B are ever accessed. Profile-guided installs 1 line, saving traffic without pollution.

③ Profile-guided invalidation for a size class

Size class (64B, 128B] has median reuse distances of 7.2KB for search, 4.0MB for flume. Should Necro-reaper invalidate on free?

  1. Profile: search's median reuse distance for (64B,128B] is ~7.2KB (small reuse distance) → skip invalidation. flume's median is ~4.0MB (large) → do invalidate.
  2. Compiler: for search, marks size class (64B,128B] as "skip invalidation"; for flume, marks as "do invalidate".
  3. Allocator (search): free_invalidate(p) looks up size = 128B, checks profile → skip. No DC IVAC issued. The freed line stays in cache until evicted or reused by next allocation (which happens quickly, within 7.2KB allocated).
  4. Allocator (flume): free_invalidate(p) looks up size = 128B, checks profile → do invalidate. Issues DC IVAC for the 1 fully-contained line (p+0 to p+64 if p is aligned). Line is evicted from cache without writeback (if dead), freeing space.
⚠
Why not always invalidate? If the address space is quickly reused (next allocation within L2 residency), invalidating the old line and not installing the new one (due to profile or heuristic) causes a miss. For search's (64B,128B] size class, this happens often → skipping invalidation avoids that miss.

④ Tallying memory traffic savings for one workload (approximate)

For the bigtable workload (32MB LLC, Necro-reaper profile-guided), what's the breakdown? (Note: these numbers are estimated from Figure 6 in the paper and should be treated as approximate.)

  1. Baseline traffic: 100% (normalized). Heap traffic: estimated ~35% of total (rest is instruction/file-backed/unrecognized).
  2. Heap reads (fetches on init miss): Necro-reaper installs lines on allocation (profile-guided: 98% coverage for bigtable). For lines installed, no fetch → saves those misses. Heap read traffic reduction: estimated ~40% of baseline heap reads.
  3. Heap writes (dead writebacks): Necro-reaper invalidates dead lines (profile-guided: skip for quickly-reused size classes). For lines invalidated, no writeback → saves those writes. Heap write traffic reduction: estimated ~30% of baseline heap writes.
  4. Total traffic reduction: 35% for bigtable (from paper Fig. 6), covering both reads and writes. Static heuristic: 19% (lower due to cache pollution from over-installation).
  5. IPC speedup: 13% for bigtable (one of the highest). Why? bigtable is bandwidth-sensitive, and 35% traffic reduction → reduced memory latency for critical loads.
🍯
Static vs profile-guided for bigtable: static heuristic hurts performance (slight slowdown) because it invalidates all freed objects, including quickly-reused ones. Necro-reaper's profile-guided invalidation skips those → 13% speedup. This demonstrates the importance of the reuse-distance profile.

Practice problems

Try each before opening the answer. Every problem ends with the misconception it checks.

Problem 1 · easy
A WSC workload allocates 1.2 times per 1000 instructions (1.2 APKI). Is this high or low compared to the paper's measurements, and what does it predict for dead traffic exposure?
Show answer
High. The paper measured 0.62 APKI geomean for WSC workloads (vs 0.04 for SPEC CPU). 1.2 APKI is nearly 2× the WSC average → even higher allocation rate → likely more dead traffic exposure (if allocations are short-lived and ≥64B). The paper shows dead traffic correlates with allocation rate + short lifetimes + large objects (Fig. 8).Misconception: that allocation rate alone determines dead traffic. It's allocation rate times the fraction of short-lived large objects. A high APKI of tiny objects (<64B) wouldn't exhibit much dead traffic (Necro-reaper doesn't install/invalidate partial lines).
Problem 2 · medium
The static heuristic is "install first 75%, ≤128KB." For a 200KB allocation, how many cache lines (64B each) are installed, and why not more?
Show answer
75% of 200KB = 150KB, but the cap is 128KB → install 128KB = 2048 cache lines (128KB / 64B). Why not more? Installing >128KB per allocation risks cache pollution. The 128KB cap is 0.39% of the 32MB LLC; installing many such allocations could still affect the cache, but the cap limits the per-allocation footprint. The paper set 128KB based on the distribution of init misses (Fig. 4a shows most allocations benefit from partial installation at this threshold).Misconception: that the cap is arbitrary or that it's 4% of the LLC. It's 0.39% of 32MB (128KB), not 4% (which would be 1.28MB). The cap balances coverage and pollution.
Problem 3 · medium
Why does the oracle strategy (install on init miss) outperform Necro-reaper's profile-guided installation-on-allocation, and why is oracle impractical?
Show answer
Oracle installs exactly when needed (on the first write to each line), avoiding cache pollution from early installation. Profile-guided installs on allocation, so lines may sit in cache unused for a while (allocation-to-access time) and may be evicted before access (cache pollution). Oracle IPC speedup: 9.0%; Necro-reaper: 5.9%. Oracle is impractical because it requires hardware tracking of uninitialized memory (lookup tables, per prior work [36,45]) and 100% detection of initialization writes (impossible statically in C++). Necro-reaper avoids hardware tracking by using profiles and static heuristics.Misconception: that oracle is "just better profiling." Oracle needs runtime hardware tracking to detect init writes on-the-fly, not just offline profiling. Necro-reaper trades some precision for deployability (no hardware tracking).
Problem 4 · hard
A size class has median reuse distance 500KB. Should Necro-reaper invalidate on free based on the paper's approach, and what's the tradeoff?
Show answer
The paper reports median reuse distances and uses them to guide skip-invalidation decisions, but does not specify a single numerical threshold like "1MB = skip, else invalidate." Based on the examples (7.2KB for search → skip; 4.0MB for flume → invalidate), smaller median values lead to skipping. For 500KB (intermediate), the decision depends on the profile and workload characteristics. Tradeoff: invalidating frees cache space and avoids dead writebacks for long-reuse allocations, but causes misses for quickly-reused allocations if the new allocation isn't installed. The median-based heuristic balances these concerns.Misconception: that reuse distance decisions are binary with a fixed threshold. The paper uses median reuse distances per size-class to inform skip-invalidation, not a single global threshold. The threshold concept is a simplification; actual decisions are profile-driven and workload-specific.
Problem 5 · medium
The paper shows 26% memory traffic reduction → 6% IPC speedup (not 26% speedup). Explain why speedup < traffic reduction.
Show answer
Three reasons: (1) Not all traffic is on the critical path. Some loads are prefetched or off critical path; reducing their latency doesn't help IPC. (2) Bandwidth isn't always saturated. The evaluation uses 4GB/s per-core; some workloads use less (e.g., Fig. 10 shows median usage often <4GB/s). Traffic reduction only helps IPC when bandwidth is a bottleneck. (3) Some workloads are compute-bound. Reducing memory traffic doesn't help if the bottleneck is ALU/FPU. For bandwidth-sensitive workloads (search: peak 50GB/s in unconstrained setting), the traffic reduction → speedup conversion is higher (8–13% IPC speedup for search, flume).Misconception: that traffic reduction directly translates to speedup. It only helps if (a) the reduced traffic is on critical loads, and (b) memory is the bottleneck. The paper's 6% geomean is averaged across diverse workloads; some gain more, some less.
Problem 6 · hard
Why does Necro-reaper's static heuristic cause a slowdown for bigtable, while the profile-guided approach yields +13% speedup?
Show answer
The static heuristic invalidates all freed objects (no reuse-distance profile). For bigtable, many deallocations are quickly reused by the allocator (short reuse distance for some size classes). Invalidating those lines and not reinstalling them (because profile/heuristic says install <100%) causes a miss on the next allocation → performance loss. The profile-guided invalidation checks reuse distance per size class and skips invalidation for quickly-reused classes → avoids those misses → +13% speedup. This demonstrates that profile-guided invalidation is critical for workloads with mixed reuse patterns.Misconception: that "more invalidation = better." Invalidation only helps if the lines won't be reused soon. Over-invalidation (as in static) can hurt by evicting lines that the next allocation would have used (if not reinstalled). The profile-guided approach balances both.
Problem 7 · easy
The paper proposes DC IZVA (install clean zero line). Why is installing a clean line better than installing a dirty line (as DC ZVA does)?
Show answer
A clean line is not written back on eviction unless it's modified (written to). If the installed line is never written (e.g., the object is deallocated before that line is accessed), a clean line avoids a useless writeback. DC ZVA marks the line dirty → writeback on eviction even if never written → unnecessary traffic. DC IZVA marks the line clean → no writeback unless written → avoids dead writeback for unused installed lines. This is especially important for profile-guided installation, where some installed lines may not be accessed (if profile is imperfect or phase-dependent).Misconception: that "zero-initialized = dirty." Zero-initialization can be clean (DC IZVA) or dirty (DC ZVA). The difference is whether the line is marked dirty on installation. Dirty → writeback; clean → no writeback unless written.
Problem 8 · hard
The paper mentions a security concern with userspace DC IVAC: an attacker could invalidate a page the OS just zeroed and read the victim's old data. How does the "invalidatable bit" hardware hardening fix this?
Show answer
The "invalidatable bit" is a per-line bit in the cache, set by DC IZVA (clean installation) and reset by any privileged write (e.g., OS zero-initialization). DC IVAC (userspace) is only allowed to invalidate lines with the invalidatable bit set (i.e., lines installed by the user via DC IZVA). For lines with the bit unset (e.g., OS-zeroed pages), DC IVAC is a no-op. This prevents the attack: OS zeros a page (resets invalidatable bit) → attacker tries DC IVAC → no-op → attacker reads zeros (not victim's old data). The cost: an extra bit per cache line (8 bits per 64B → 1.56% cache area overhead) + logic to check the bit. The paper shows this adds 8.5% overhead to Necro-reaper's IPC speedup (5.4% vs 5.9% without hardening) — tolerable but less preferable than OS-based hardening (non-temporal zero by OS).Misconception: that the invalidatable bit prevents all side-channels. It only prevents cross-process data leaks via DC IVAC. Other side-channels (flush-and-reload with clflush-equivalent instructions) are orthogonal. The paper notes that DC IVAC doesn't introduce new side-channels beyond what clflush already enables.
Quick answers

Frequently asked questions

Ten questions readers ask most. Tap any to expand.

Q1How does this relate to my tiering/CXL research?▾
Necro-reaper targets LLC-to-memory traffic for WSC heap behavior. For DRAM/NVM tiering: dead writebacks to DRAM are analogous to dead writebacks to NVM (wasting write endurance + bandwidth). For CXL: dead traffic over the CXL link is even more costly (higher latency, lower bandwidth than local DRAM). The profile-guided approach could extend to "don't promote dead data to DRAM tier" or "invalidate dead lines in CXL-attached memory." The paper evaluates LLC-memory traffic; extending to tiering would require additional evaluation of promotion policies, DRAM/NVM endurance, and CXL-link traffic.
Q2Why not just zero-initialize all allocations like Java?▾
For Java, zero-initialization is mandatory (language semantics) and accounts for ~25% of memory traffic [69]. For C++, zero-init is not required (reading uninitialized memory is UB), so it's wasted traffic if the allocation is initialized by the program anyway. Necro-reaper installs only the portion that will be used (profile-guided or heuristic), avoiding the full-zero overhead. Also, zero-init doesn't solve dead writebacks (you still need invalidation).
Q3Does Necro-reaper work for non-C++ languages?▾
Partially. The installation part (avoiding dead fetches) works for any explicitly memory-managed language (C, Rust, etc.). For managed languages (Java, Go), prior work [56,37] proposes GC-integrated schemes, but Necro-reaper's profile-guided approach could adapt (profile allocation-site access patterns even for GC languages). The invalidation part works for any language with explicit or GC-based deallocation. The challenge: GC languages have less predictable deallocation timing, so reuse-distance profiling is harder.
Q4How does this compare to prior work (ESKIMO [36], Sartor et al. [56])?▾
Prior work (Table 3 in paper): ESKIMO [36] and Lewis [45] require hardware tracking of uninitialized memory (lookup tables); Sartor [56] is GC-specific and coarse-grained (32KB regions); Ismail [37] requires hardware tracking. Necro-reaper is the first to combine (a) explicit memory management (C++), (b) fine-grained per-line installation/invalidation, (c) no hardware tracking (profile-guided software), (d) security-hardened userspace invalidation, and (e) evaluation on WSC workloads. The pragmatic fit for WSC's vertical integration is the key differentiator.
Q5What's the profile mismatch risk?▾
The paper profiles the to-be-optimized workload with the same configuration (similar to AutoFDO [21]) to minimize mismatch. The risk: if the profiling inputs don't cover all code paths, some allocation sites have no profile (e.g., bigquery: 51% coverage → lower speedup than other workloads). For those, Necro-reaper falls back to the static heuristic. The paper's 85% geomean profile coverage suggests most paths are covered. For production deployment, continuous profiling (like BOLT's PMU-based profiling [52,53]) could address mismatch.
Q6Why does installation-only (no invalidation) yield 3.9% IPC vs 5.9% full?▾
Installation (avoiding dead fetches) is responsible for ~65% of the speedup (3.9% / 5.9%). Invalidation (avoiding dead writebacks) adds ~35% (2.0% additional IPC). Why? (1) Fetch-on-miss is on the critical path (load latency); writeback is often off critical path (asynchronous eviction). (2) Installation reduces L2 MPKI (12.8% reduction for full Necro-reaper, mostly store misses), directly helping IPC; invalidation frees cache space (indirect benefit, only helps if cache is thrashed). (3) For some workloads (bigtable), invalidation is critical to avoid over-eviction; for others, it's a smaller win.
Q7What's the security threat model for userspace DC IVAC?▾
Threat: attacker process gets a page that the OS just zeroed (from a victim process). OS zeros the page in DRAM, but a stale dirty copy might be in the attacker's cache (if the victim's line was in a shared cache and not flushed). Attacker issues DC IVAC → evicts the line without writeback → reads from DRAM, expecting zeros, but gets victim's old data (the dirty line was never written back). Mitigation: (1) OS-based: OS uses non-temporal writes to zero pages (bypassing cache) or forces writeback (clwb); (2) hardware-based: invalidatable bit ensures DC IVAC is no-op on OS-zeroed pages. The paper proposes both; hardware-based adds 8.5% overhead (tolerable but less ideal).
Q8Does this help for file-backed data or stack allocations?▾
Not directly. The paper targets heap allocations only (malloc/free). File-backed data (mmap) could benefit from similar techniques (e.g., invalidate on munmap), but the paper doesn't evaluate it. Stack allocations are too short-lived and small (typically <64B or <1KB) to benefit much — they rarely survive to LLC eviction. The paper's focus on heap is justified by WSC's high heap allocation rate (0.62 APKI) and the fact that heap traffic is 35–50% of total memory traffic (Fig. 6).
Q9How does LLC size affect the results?▾
Fig. 7 in paper: headroom (oracle) savings increase 78% from 8MB to 128MB LLC (geomean). Why? Bigger LLC → longer residency → more objects survive to deallocation → more dead writebacks. For dead fetches, the effect is mixed: bigger LLC reduces compulsory misses for other data, increasing the relative importance of init misses. The paper evaluates oracle LLC-size sensitivity for 8/32/128MB LLCs but does not make broader claims about DRAM/CXL capacity or modern server LLCs beyond the oracle sensitivity results.
Q10What's the instruction overhead of inserting DC IZVA/DC IVAC?▾
The paper does not quantify instruction count overhead. While one can estimate overhead for small allocations (e.g., a few extra instructions per allocation), the paper explicitly does not provide this analysis and suggests range instructions to reduce code bloat for large allocations/invalidation loops. The IPC speedup (5.9%) demonstrates net benefit, but instruction overhead (especially for large allocations) is not directly measured. The bigger cost is cache coherence messages (BusInstall, BusInv), which the paper models in gem5's CHI protocol.

Glossary & further reading

Every term in one place, then papers to deepen your understanding.

Core terms
Dead dataData corresponding to deallocated memory (not the "not-to-be-read" definition from dead-block prediction work [42,46,67]).
Dead writebackAn LLC writeback of a cache line containing dead data — semantically useless memory traffic.
Dead fetch-on-missA fetch from memory on an LLC write miss when the line is fully uninitialized (deallocated memory being read).
Dead memory trafficTotal bytes transferred LLC↔memory for dead writebacks + dead fetches. Paper measures 26–50% of total traffic (geomean 26%).
Install cache lineDirectly place a line into a cache without a memory fetch. Necro-reaper uses DC IZVA (install clean zero line).
Invalidate cache lineEvict a cache line from the hierarchy without writing it back. Necro-reaper enables DC IVAC for userspace.
APKIAllocations Per Kilo Instructions. WSC: 0.62 APKI (geomean); SPEC CPU 2017: ~0.04 APKI.
Short-lived allocationLifetime ≤ average LLC residency for a given workload. Likely to survive in LLC from allocation to deallocation.
Reuse distanceTotal allocated bytes between deallocation of a virtual address and its reuse by the allocator for a new allocation.
Implementation terms
DC IZVANecro-reaper's proposed ISA instruction: install a clean zero line (vs DC ZVA, which installs dirty) in L2/LLC.
DC IVAC (userspace)Necro-reaper enables AArch64's DC IVAC for EL0 (userspace) to invalidate without writeback, with security hardening.
Invalidatable bitHardware hardening: per-line bit set by DC IZVA, reset by privileged writes; DC IVAC is no-op if bit unset.
TCMallocGoogle's size-class-based memory allocator, extended by Necro-reaper to insert DC IZVA/DC IVAC instructions.
Profile-guided installationInstall N% of an allocation (N = profiled max-offset / size), rounded up to cache-line boundary.
Profile-guided invalidationSkip invalidation for size classes with small median reuse distance; else invalidate on free.
Static heuristicFallback if no profile: install first 75% of object, ≤128KB total; always invalidate (no reuse-distance check).

Profile-Guided Optimization (AutoFDO) easy

Chen, Li, Moseley · CGO 2016

The profile-guided compilation framework Necro-reaper extends. AutoFDO collects branch/edge profiles, Necro-reaper adds heap access profiles. Understanding AutoFDO's callstack-based aggregation is key to Necro-reaper's approach.

Memory Hierarchy for Web Search medium

Ayers et al. · HPCA 2018

Characterizes WSC workload memory patterns. Shows large working sets, high miss rates → motivation for why WSC is bandwidth-saturated. Complements Necro-reaper's dead-traffic focus.

ESKIMO: Energy Savings Using Semantic Knowledge hard

Isen & John · MICRO 2009

Prior work on avoiding dead traffic for C/C++. Uses hardware tracking tables for uninitialized memory. Necro-reaper's comparison shows profile-guided software can replace hardware tracking for WSC.

Cooperative Cache Scrubbing hard

Sartor et al. · PACT 2014

Dead-traffic elimination for Java/GC languages. Uses coarse-grained (32KB) regions and GC integration. Necro-reaper's fine-grained, profile-guided approach is more applicable to WSC's C++ codebases.

Limoncello: Prefetchers for Scale medium

Jain et al. · ASPLOS 2024

Recent Google work on bandwidth-aware prefetching for WSC. Shows 3–5GB/s per-core bandwidth constraints. Necro-reaper's 26% traffic reduction complements prefetching by freeing bandwidth for useful prefetches.

Characterizing a Memory Allocator at Warehouse Scale medium

Zhou et al. · ASPLOS 2024

Profiles TCMalloc at Google scale. Provides context for Necro-reaper's allocator extensions and size-class reuse patterns. Relevant for understanding the reuse-distance profile.
