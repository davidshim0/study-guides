HPCA 2025 · A Field Guide

Cache what you actually reuse, not what you merely touch

Serverless functions hammer global storage. Local caching helps—but existing schemes either prohibit replicas (causing remote reads) or check versions on every read (slower than fetching small data). This is the story of Concord, which brings hardware-style invalidation coherence to distributed software caches.

📄 "Concord: Rethinking Distributed Coherence..." · Stojkovic et al. 🎓 Some OS/cache basics assumed — primers cover them ⏱ ~25 min read
💾

Allow multiple cached copies of small data items across nodes, keep them coherent with invalidations (not version checks), and minimize coherence traffic by routing function invocations to nodes that already cache the needed data.

Before you begin

Eight ideas the paper assumes you know

Each card clears up one concept Concord builds on. Read in order the first time, or jump to whatever looks unfamiliar.

Concord (invalidation-based coherence) versioning (Faa$T) distributed directory / cache agent single-home (OFC / no coherence)
1Serverless / FaaS: stateless by design▾

In Function-as-a-Service, users upload code. The provider wraps it in a container, runs it, then discards all local state when the function ends. Any durable data must live in global storage (e.g., Azure Blob Storage). For security, functions can't talk directly to each other—they pass data through storage. The result: applications spend 35–93% of their response time on storage reads/writes.

Now you're ready: Concord's job is to cache data in node memory so functions can skip the slow trip to global storage, while keeping caches coherent when the same data is written elsewhere.
2The baseline problem: OFC (single-home caching)▾

One early scheme (OFC) assigns each data item a home node via consistent hashing. That data item can only be cached in its home. If a function on a non-home node reads the item, the read goes remotely to the home. This eliminates any need for coherence (there's at most one cached copy), but causes frequent remote accesses—in single-home schemes, remote reads and writes can account for up to 82% of response time.

Now you're ready: Concord's first departure is allowing the same data item to be cached in multiple nodes at once, so local reads can hit locally. That introduces a coherence problem, which Concord solves with invalidations.
3The versioning alternative: Faa$T▾

Faa$T allows replicas and keeps them coherent via version numbers. Each data item has a home that stores the latest version number. A non-home node that caches the item first fetches the version from the home and compares it to its local version. If they match, the local copy is valid. If not, the node fetches the data. Writes propagate to the home, which increments the version. No invalidations are sent.

This works well for large data where fetching a small version number is cheaper than fetching the whole item. But in serverless, 80% of data items are ≤12KB. Version access and checking can cost up to 78% of application response time for these small items.

Now you're ready: Concord skips version checks. On a local read hit, it just reads—no remote message—because invalidations ensure the local copy is always valid.
4Hardware cache coherence protocols (MESI family)▾

Multicore CPUs keep their private caches coherent with protocols like MESI: Modified, Exclusive, Shared, Invalid. A directory (centralized or distributed) tracks which caches hold a given block and in what state. On a write, the home (directory holder) sends invalidations to all sharers, waits for acknowledgments, then allows the write to complete. Hardware protocols are fast because latencies are ~cycles and the number of sharers is modest.

Now you're ready: Concord adapts this model to software, using per-application distributed directories and invalidation messages over the network. It's feasible because serverless workloads have only a few sharers per data item (typically <11 nodes on a 16-node cluster).
5Per-application caches vs shared caches▾

Early designs (OFC, InfiniCache) used a single cache per node shared by all applications. But Azure production data shows 99.7% of data objects are not shared across applications. Strong data reuse happens within an application. Concord gives each application its own cache instances (one per node where the app runs), isolated from other apps.

This also means when all of an app's functions leave a node, the app's cache instance is discarded—no leftover state, and no charges for unused memory across apps.

Now you're ready: Concord's coherence domain is per-application. Writes from App A don't invalidate caches of App B, even if both are on the same node.
6Write-through caches for fault tolerance▾

A write-through cache propagates every update immediately to the backing store (here, global storage) before acknowledging the write. A write-back cache would buffer writes and flush later. Write-through is slower on writes but simpler for fault tolerance: if a cache crashes, no data is lost—storage is always up-to-date. Concord uses write-through to avoid complex recovery of dirty data.

Now you're ready: Concord's protocol is write-through. On a write, the home sends invalidations to sharers in parallel with the update to storage, so the invalidation latency is often hidden by storage latency (~30ms).
7Consistent hashing for dynamic domains▾

In a distributed system, nodes join and leave over time. Consistent hashing maps keys (data addresses) to nodes in a way that minimizes rehashing when membership changes. Each node and each key get a position on a hash ring; a key's home is the first node clockwise from the key's position. When a node leaves, its keys move to the next node on the ring—only that node's keys are affected, not the whole system.

Now you're ready: Concord uses consistent hashing to assign data item homes. When a cache instance is created or removed (because functions are added/removed), only the items homed in that instance change homes—and Concord's protocol handles the rehashing transparently.
8Serverless functions are idempotent & retry-tolerant▾

Serverless platforms (AWS Lambda, Azure Functions, OpenWhisk) automatically retry failed function invocations. To make this safe, functions must be idempotent: re-executing a function with the same inputs must produce the same result. Storage APIs for serverless are also idempotent (key-value, versioned objects). This means serverless is already designed to tolerate failures.

Now you're ready: Concord's coherence protocol ensures data consistency on node failure, verified with TLA+. If a cache instance crashes mid-write, Concord detects it (via ZooKeeper heartbeats), evicts any stale cached copies of data items homed there, and re-homes the items. The function invocation that was writing might fail and be retried—standard serverless behavior.

🔑 Quick glossary of Concord-specific terms

Cache Agent (CA)
Per-cache-instance controller that handles coherence messages and manages directory entries for homed data.
Data Directory
Distributed across cache agents; each entry tracks sharers and state (E/S) for one data item.
Coherence Domain
The set of all cache instances for one application; each app has its own domain, isolated from other apps.
Home Node
The node (cache instance) that holds the directory entry for a data item, determined by consistent hashing of the item's address.
Sharer
A cache instance that has a valid (E or S) copy of a data item.
E/S/I States
Exclusive (single cached copy, writable), Shared (multiple copies, read-only), Invalid (no cached copy).
The whole story in one place

Concord in a nutshell

The problem, how invalidation-based coherence fixes it, and how it all fits together.

Serverless functions are stateless: all durable data lives in global storage (Azure Blob, etc.), causing 35–93% of response time to be storage accesses. Prior caching schemes either prohibit replicas (OFC—remote reads/writes can account for up to 82% of response time) or check version numbers on every read (Faa$T—version access and checking can cost up to 78% of application response time for the small items (≤12KB) that dominate serverless).

Concord allows multiple cached copies of the same data item across nodes and keeps them coherent with a directory-based invalidation protocol inspired by hardware cache coherence (MESI without M). Each application has its own per-node cache instances, isolated from other apps. A Cache Agent (CA) per cache instance manages coherence: it holds directory entries for data items homed locally (tracking which other cache instances are sharers) and sends/receives invalidations on writes. Caches are write-through (updates propagate immediately to storage), so global storage is always up-to-date. On a local read hit, a function just reads—no remote message—because Concord's invalidations ensure the local copy is always valid. This is the key win: for the 83% of reads that hit locally (on average), Concord is a pure local access with zero coherence traffic.

Concord minimizes coherence messages in three ways: (1) Coherence-aware invocation scheduling routes functions to nodes that likely cache the needed data (based on hashing the function's input parameters), raising local hit rates. (2) Write-through + parallel invalidations: invalidations are sent in parallel with the storage update, so the ~30ms storage latency hides the ~2ms per-invalidation latency. (3) Modest sharer counts: in practice, a data item is typically cached in only 1.7–5.0 nodes (max 6–15 on a 16-node cluster under high load), so the number of invalidations per write averages 1.2 (max 4.9). For fault tolerance, a distributed coordination service (ZooKeeper) monitors node health; if a cache instance crashes, other instances evict data items homed there and recompute new homes via consistent hashing.

Concord also uses unused memory already allocated to functions (users request ~5× more memory than needed, so 100s of MBs per node sit idle) as cache space—so no extra charges. Finally, Concord's coherence protocol enables two new features: transactional storage accesses (detect conflicts via coherence messages, buffer speculative state in cache, commit atomically) and communication-aware function placement (learn producer-consumer pairs from coherence traffic, co-locate them to reduce network overhead).

The payoff: On a 16-node cluster running 7 serverless benchmarks, Concord speeds up execution by 2.4× (vs OFC) and 2.5× (vs Faa$T) on average, and improves throughput by 1.7×. Cache instances average 6.2MB in size (4.8% of allocated memory), and local read hits (83% of reads) are ~1.6ms vs ~3.1ms for remote read hits and ~32ms for storage misses.

🍯 Key takeaways

  • Serverless functions spend 35–93% of time on storage I/O; caching helps, but prior schemes are suboptimal for small, read-heavy data.
  • OFC (single-home) avoids coherence but remote reads/writes can account for up to 82% of response time. Faa$T (versioning) allows replicas but version access and checking can cost up to 78% of application response time for ≤12KB items.
  • Concord allows replicas and keeps them coherent with invalidations (like hardware MESI), not version checks. Local read hits (83% of reads) are pure local—no remote message.
  • Per-application caches + distributed directory: each app's cache instances form a coherence domain; directory is sharded across homes via consistent hashing.
  • Write-through + parallel invalidations: writes update storage and send invalidations in parallel, hiding invalidation latency (~2ms) behind storage latency (~30ms).
  • Coherence-aware scheduling routes invocations to nodes that cache their data (based on input hash), raising local hit rate to 83% (vs 75% without).
  • Fault tolerance: write-through + ZooKeeper heartbeats detect failures; crashed cache instances' data items are re-homed via consistent hashing; Concord ensures data consistency, verified with TLA+.
  • Bonus features: transactions (detect conflicts via coherence messages) and communication-aware function placement (learn producer-consumer pairs from coherence traffic).
  • Result: 2.4× faster than OFC, 2.5× faster than Faa$T, 1.7× throughput gain, using 6.2MB average cache instance size (4.8% of allocated memory).
Six everyday pictures

How to think about Concord

Each picture stands alone—read them in any order. Every card ends with an honest note about where the picture stops being true.

The core idea · Invalidation vs versioning

Borrowing a library book

You checked out a textbook last week and it's still on your desk. You want to read Chapter 3. With versioning (Faa$T), you'd first call the library to confirm the edition number hasn't changed since you borrowed it—even though you already have the book. With invalidation (Concord), the library calls you if someone submits a newer edition, telling you to discard your copy. Until that happens, you just read. For short books (≤12KB) where calling the library takes as long as re-borrowing the book, Concord's "read now, invalidate later" is much faster.

Where it breaks: if books were huge (GBs) and edition numbers tiny (bytes), versioning would win—but serverless data is mostly small, so invalidation wins.
Per-application isolation · Coherence domains

Separate filing cabinets

A shared office has three teams. Each team has its own filing cabinet (cache instance) in the office, holding only that team's files. A write by Team A invalidates only the other copies of Team A's files, not Team B's or Team C's. If Team A leaves the office, their cabinet is removed—no leftover files. In Concord, each application has its own per-node cache instance, and coherence operates per-application: writes from App1 don't trigger invalidations in App2's cache, even if both are on the same node.

Where it breaks: real office cabinets take physical space; Concord's per-app caches share the node's memory dynamically, using only what's idle—so no extra charges.
Consistent hashing · Dynamic homes

A clock with moving hands

Imagine a 12-hour clock face where each hour position is a node. A data item's "home" is the first hour hand clockwise from the item's position on the dial. If the 3 o'clock hand disappears (node leaves), items between 2 and 3 o'clock now belong to the 4 o'clock hand—only those items move. Concord uses consistent hashing to assign homes: when a cache instance joins or leaves a coherence domain, only the items homed in that instance (and their immediate neighbors on the hash ring) are affected—no global reshuffling.

Where it breaks: a clock face is static; Concord's "ring" is a software construct that nodes recompute on membership changes, using a two-phase protocol to avoid races during rehoming.
Write latency · Hiding coherence cost

Mailing a letter with carbon copies

You write a memo and need to send the original to HQ (global storage) and carbon copies to three colleagues (sharers). HQ is a 30-minute drive; your colleagues are 2 minutes away by messenger. You send all four messages at once. The memo is "delivered" when HQ acknowledges it—but since the colleagues' copies arrive in 2 minutes (well before the 30-minute HQ round-trip), the HQ latency hides the colleague latency. Concord does the same: writes update storage (30ms) and send invalidations (2ms each) in parallel, so the storage latency dominates—invalidations are nearly free.

Where it breaks: if there were 100 colleagues and each invalidation required a separate 30ms round-trip, the parallelism wouldn't help—but Concord's workloads have ≤15 sharers (avg 3–5), so invalidations finish before storage does.
Scheduling · Coherence-aware placement

Sending customers to the same cashier

A store has three cashiers. Most customers buy apples or oranges. You (the manager) notice that the apple customers always ask for a receipt, while orange customers don't—so you route all apple customers to Cashier 1 (who knows the receipt procedure) and orange customers to Cashier 2. Over time, Cashier 1 becomes an expert on apples. Concord does this with function invocations: it hashes the invocation's inputs and routes invocations with the same inputs to the same node, betting that node already caches the data those inputs need. Result: local hit rate rises from 75% (random scheduling) to 83% (coherence-aware).

Where it breaks: if invocation inputs don't predict data access patterns (e.g., a function that randomly samples from a large dataset), hashing inputs won't help—but in practice, serverless functions have strong input-to-data locality.
Fault tolerance · Write-through + directory recovery

A whiteboard with a notebook backup

A team uses a whiteboard (cache) for quick notes but immediately copies every note into a durable notebook (storage) before erasing the whiteboard. If someone accidentally wipes the whiteboard (node crash), the notebook still has everything—no data is lost. The only lost info is who had which notes on their whiteboards (directory state). Concord's write-through caches do the same: every write updates storage immediately, so if a cache instance crashes, the data is safe in storage. The coherence protocol just evicts any stale cached copies (by rehoming directory entries) and moves on.

Where it breaks: a real notebook might take hours to write in; Concord's storage is ~30ms, which is acceptable for serverless latency budgets—but still slower than a write-back cache (which Concord avoids for simplicity).
The full explanation

How it actually works

Section by section, with the paper's key mechanisms rebuilt as visuals you can read at a glance.

1 · The problem — storage I/O dominates serverless

Serverless functions are stateless: all durable data lives in global storage (Azure Blob, AWS S3, etc.). Security rules prohibit direct inter-function communication, so data must be passed through storage. Result: applications spend 35–93% of response time on storage reads/writes. Local caching can help—but prior schemes are suboptimal.

🌱
Concretely: The paper measures 7 benchmarks (TrainT, eShop, ImgProc, VidProc, HotelBook, MediaServ, SocNet) on a 16-node cluster with OpenWhisk. Without caching, avg response time is 307ms (mean) with 63% spent on storage. OFC (single-home caching) still has remote reads/writes accounting for significant overhead. Faa$T (versioning) has version access and checking consuming substantial response time. Concord cuts total response time to 25ms (mean)—a 2.4× speedup over OFC.

2 · Concord's architecture — per-app caches + cache agents

Concord gives each application its own cache instances, one per node where the app runs. A Cache Agent manages each cache instance. The CA has three roles:

  1. Intercept storage requests from local functions and satisfy them from cache or forward to remote CAs/storage.
  2. Manage the Data Directory for data items homed locally (track sharers, send invalidations).
  3. Re-purpose unused memory from co-located functions into cache space—so no extra charges.

An Application Controller (centralized) maintains a Node Directory listing which nodes have cache instances for each app. When a cache instance is created/removed, it informs the controller, which propagates the membership change to other instances of that app.

3 · Coherence protocol (ESI states) — reads and writes

Concord uses an ESI protocol (Exclusive, Shared, Invalid)—MESI without the Modified state, because caches are write-through. Each cached data item has a state: E (single cached copy, writable), S (multiple copies, read-only), or I (not cached). Six operations:

Local read hit

The function reads a data item that's cached locally (E or S state). The CA provides it directly—no remote message. This is the common case: 83% of reads on average.

Remote read hit

The function reads a data item not in the local cache. The CA forwards the read to the home CA ( 1 ). The home checks the directory entry: if it's in E state, the home fetches the data from the owner (which downgrades to S), marks both requester and former owner as sharers in S, and sends the data to the requester. If it's already in S state, the home fetches the data (from local cache if homed locally, else from storage), adds the requester as sharer, and sends it. Avg latency: 3.1ms. (Paper Sec. III-C2, Fig. 5a-2)

Read miss

The function reads a data item not in the local cache, and the home has no directory entry (data was never cached). The home CA fetches from storage ( 4 ), creates a directory entry with the requester as sharer in E state, and sends the data. Avg latency: 32ms (storage round-trip). (Paper Sec. III-C2, Fig. 5a-3)

Local write hit (E state)

The function writes a data item cached locally in E state. The CA updates the local cache and propagates the write directly to storage, bypassing the home—because E guarantees no other sharers. Faster path for repeated writes to the same item. (Paper Sec. III-C2)

Local write hit (S state)

The function writes a data item cached locally in S state (multiple sharers). The CA updates the local cache and forwards the write to the home CA ( 3 ). The home sends invalidations to all sharers except the writer ( 4a ) and propagates the write to storage ( 4b ) in parallel. When all sharers acknowledge ( 5a ) and storage acknowledges ( 5b ), the home updates the directory to mark only the writer as owner in E state ( 6 ) and sends an ack to the writer ( 7 ). Avg write latency: 30–32ms (dominated by storage, not invalidations). (Paper Sec. III-C2, Fig. 5b)

Remote write hit / write miss

The function writes a data item not in the local cache. The CA forwards the write to the home. If the home has a directory entry (remote write hit), it sends invalidations to existing sharers and updates storage in parallel, then marks the writer as owner in E. If the home has no entry (write miss), it just updates storage, creates an entry with the writer as owner in E, and sends an ack. (Paper Sec. III-C2)

4 · Minimizing coherence traffic — three techniques

① Coherence-aware invocation scheduling

When the load balancer receives a function invocation, it computes hash(invocation_inputs) and uses the result to pick one of the nodes that has an instance of that function. Rationale: invocations with the same inputs likely access the same data, so routing them to the same node increases local cache hits. This raises the local hit rate from 75% (random scheduling) to 83% (coherence-aware). (Paper Sec. III-G, Table III)

# on incoming invocation
node = hash(inputs) mod num_nodes_with_function
if node.overloaded():
    node = try_another_hash() or random_non_overloaded_node()
route_to(node)

② Write-through + parallel invalidations

Writes update storage (30ms round-trip) and send invalidations (2ms each) in parallel. Because invalidations finish before storage, the storage latency hides the invalidation latency. Avg 1.2 invalidations per write (max 4.9) on a 16-node cluster, so even with 5 invalidations, the protocol adds only ~2ms beyond the 30ms storage latency. With 30 nodes, measured write latency is 32.4ms—only 8% overhead over storage alone. (Paper Sec. III-C2, Fig. 9, 11)

③ Modest sharer counts

Despite 16 nodes, the avg number of sharers per data item is 1.7–3.0 (low/medium load) and 3.0–5.0 (high load), with max 6–15. This is due to (a) coherence-aware scheduling packing invocations that operate on the same data onto few nodes, and (b) serverless functions having strong data locality. Few sharers → few invalidations → low coherence traffic. (Paper Sec. II-B, Table I)

5 · Fault tolerance — write-through + ZooKeeper + recovery

Concord handles node crashes in three steps:

  1. Detection: A distributed coordination service (ZooKeeper) sends heartbeats to all cache agents of each application. If a node stops responding, ZooKeeper marks it failed and notifies other cache instances of that application only (not other apps).
  2. Eviction: Surviving cache instances check if they cache any data items homed in the failed node. They evict those items from their local caches (since the directory is lost). Because caches are write-through, storage is up-to-date—no data loss.
  3. Rehoming: Surviving cache instances form a new consistent hash ring (without the failed node). Data items previously homed in the failed node are now homed in the next node clockwise on the ring. When a future read misses for such an item, it goes to the new home, which fetches from storage and creates a new directory entry.

For writes in flight when a node crashes, Concord ensures correctness: no cache instance is allowed to read from storage for a data item homed in the failed node until recovery is complete (all stale cached copies evicted). This prevents reading a new value from storage while some cache still has the old value. Serverless functions are idempotent, so if a function's write was interrupted by a crash, the function is retried—standard behavior. (Paper Sec. III-F, Fig. 6)

6 · Dynamic coherence domains — cache instances join/leave

A cache instance is created when the first function of an app is loaded onto a node, and destroyed when all functions of that app are removed (after a grace period or due to space pressure). Concord uses a two-phase protocol to add/remove cache instances from a coherence domain:

  1. Phase 1 (synchronize): The joining/leaving cache instance informs all other instances of its membership change. Other instances mark the change as pending and pause accepting new coherence operations involving data items that might change homes.
  2. Phase 2 (rehome): All instances recompute the consistent hash ring. For a leaving instance, its directory entries are sent to the new homes (the next node clockwise on the ring). For a joining instance, some directory entries move from existing homes to the new instance. Once all entries are moved and acknowledged, instances resume normal operation.

This protocol ensures no data races during domain changes. Latency is ~120ms for a 16-node domain, but only the joining/leaving node is blocked—other nodes continue serving reads/writes for data items whose homes don't change. (Paper Sec. III-D)

Follow the numbers, step by step

Worked examples

Four traces, each carried all the way through—computing speedups, analyzing invalidations, transaction conflicts, and memory use.

① Computing the speedup: Concord vs Faa$T

TrainT application on a 16-node cluster at medium load. OFC avg response time = 22ms. Faa$T = 21ms. Concord = 13ms. What's the speedup?

  1. Speedup over OFC: 22ms / 13ms = 1.69× (Concord is 69% faster).
  2. Speedup over Faa$T: 21ms / 13ms = 1.62× (Concord is 62% faster).
  3. Why Concord wins: OFC forces remote reads to single homes (remote reads/writes can account for up to 82% of response time in single-home schemes). Faa$T checks versions on every read (version access and checking can cost up to 78% of application response time for ≤12KB items). Concord serves 83% of reads from local cache with no remote message—pure local access.
  4. Verification: Paper reports Concord avg response time across all 7 apps at medium load = 20ms (vs 48ms for OFC, 49ms for Faa$T), which is 2.4× and 2.5× speedup. TrainT's 1.69× is consistent with this range.
🍯
Insight: The speedup comes from eliminating remote messages on local reads. Concord's invalidation-based protocol ensures local reads are always valid, so no version check is needed.

② Analyzing invalidation overhead: write to B with 3 sharers

Data item B (size 8KB) is cached in 3 nodes (N0, N1, N2) in S state. N1 is B's home. N0 writes B. How long does the write take?

  1. N0 writes B: Updates local cache, forwards write to home N1 ( 3 ).
  2. N1 (home) actions in parallel: (a) Sends invalidations to sharers other than N0 (N1, N2) ( 4a ). (b) Updates storage ( 4b ).
  3. Invalidation latency: Each invalidation + ack is ~2ms (measured). 2 invalidations (N1, N2) sent in parallel → total 2ms.
  4. Storage latency: Azure Blob Storage round-trip ~30ms.
  5. Total write latency: max(2ms, 30ms) = 30ms (storage dominates).
  6. Contrast with Faa$T: Faa$T write also takes ~30ms (update home + storage), but Faa$T's subsequent reads by N1 or N2 require version checks (~2–3ms each), while Concord's reads are local (~1.6ms) with no version check.
💡
Why invalidations scale: With ≤15 sharers (typical max in serverless), invalidations are sent in parallel with storage. The paper shows that with 30 nodes, write latency is 32.4ms—only 8% overhead over storage alone. Invalidations sent in parallel are hidden by storage latency.

③ Transaction conflict detection via coherence

Process P1 starts a transaction that reads data items A and B, then writes C. Meanwhile, Process P2 (on a different node) writes B. Does Concord detect the conflict?

  1. P1 reads A, B: Local cache instance marks A and B as "Speculatively Read by P1" and records P1's ID.
  2. P2 writes B: The write propagates to B's home CA, which sends an invalidation to all sharers of B, including P1's cache instance.
  3. Conflict detected: P1's cache agent receives the invalidation for B (which is marked "Speculatively Read by P1"). The agent squashes P1's transaction immediately—before P1 tries to commit.
  4. Rollback: P1's speculative reads/writes are discarded from the local cache. P1's transaction is re-executed with exponential back-off.
  5. Contrast with Saga/Beldi: Saga detects conflicts by re-reading data from storage after the transaction completes (adding ~30ms per item). Beldi logs all accesses and checks logs on commit (adding storage I/O overhead). Concord detects conflicts during the transaction via coherence messages—no extra storage accesses. Result: 54% faster than Saga, 20% faster than Beldi (Paper Fig. 15).
🍯
Insight: Concord's coherence protocol is a distributed conflict detector that comes for free with invalidations. Transactions just piggyback on existing coherence infrastructure.

④ Memory use: cache instance size vs unused memory

A node runs 3 functions of App1, each allocated 128MB (total 384MB). Functions actually use 72MB, 60MB, 55MB (total 187MB). How much cache can App1's cache instance use?

  1. Total allocated memory: 384MB (charged to user).
  2. Total used memory: 187MB (actual function footprint).
  3. Unused memory: 384 − 187 = 197MB (idle, available for cache).
  4. Cache size (paper avg): Concord cache instances average 6.2MB in size (max 12.6MB). This is 3.1% of the 197MB unused memory, or 1.6% of the 384MB total allocated memory.
  5. Dynamic sizing: If function memory use rises (e.g., from 187MB to 370MB), unused memory drops to 14MB. Concord shrinks the cache to fit (e.g., evict down to 6MB). If memory use drops, cache grows again.
  6. Verification: Paper reports avg unused memory per app per node = 56.8MB, and avg cache instance size = 6.2MB, which is 10.9% of unused memory—consistent with this example's 3.1% (our functions left more unused memory than the paper's average).
💡
Why this matters: Users are already charged for the 384MB, whether functions use it or not. Concord re-purposes the idle 197MB into cache—so the cache is "free" from the user's perspective. No extra charges, no extra memory footprint.

Problem 1 · easy
Data item X is cached in nodes N0, N1, N2 in S state. N1 is X's home. N2 writes X. Which nodes receive invalidations, and what happens to X's directory entry?
Show answer
N1 (home) sends invalidations to sharers other than the writer: N0 and N1. After all sharers acknowledge and storage is updated, N1 updates X's directory entry to mark only N2 as owner in E state. The directory entry now shows "X: sharers=[N2], E".Misconception: that N2 (the writer) receives an invalidation. Concord invalidates sharers other than the writer; the writer's local cache is updated directly and becomes the sole E owner.

(Assume remote read hit = 3.1ms, remote miss = 32ms, and 7% remote hits, 18% misses in both.)
Show answer
Concord: (0.83 × 1.6) + (0.07 × 3.1) + (0.10 × 32) = 1.33 + 0.22 + 3.2 = 4.75ms. Faa$T: (0.75 × 3.8) + (0.07 × 3.1) + (0.18 × 32) = 2.85 + 0.22 + 5.76 = 8.83ms. Concord is 1.86× faster on reads.Misconception: that Faa$T's version check overhead is negligible. For ≤12KB data (80% of items), version access and checking can cost up to 78% of application response time—so the overhead is substantial.
Problem 3 · medium
A cache instance crashes mid-write to data item Y. Y was in E state in the crashed cache, and the write had updated storage but not sent invalidations (since E writes bypass the home). What happens to Y's directory entry and cached copies?
Show answer
Y was in E state, so there's only one cached copy (in the crashed cache). When ZooKeeper detects the crash, other cache instances check if they have Y cached locally—they don't (since Y was E). Y's directory entry was in the crashed cache's CA, so it's lost. Next read of Y will be a remote miss: the new home (determined by consistent hashing) fetches Y from storage (which has the latest value, since the write was write-through) and creates a new directory entry. No inconsistency occurs.Misconception: that E state writes are unsafe on crash. They're safe because E guarantees no other sharers exist, and writes update storage before acknowledging—so storage is always up-to-date.
Problem 4 · hard
A node has 3 cache instances (App1, App2, App3). App1 writes data item A. App2 reads data item A (different A—different key, but same address in storage). Does App2 receive an invalidation from App1's write?
Show answer
No. Concord's caches are per-application and isolated. App1's write to A triggers invalidations only in App1's coherence domain (cache instances of App1 on other nodes). App2's cache is a separate instance with its own directory and own copy of "A" (which is a different data item—different key in App2's namespace). No cross-app invalidations occur.Misconception: that caches are shared across apps on the same node. Early designs (OFC, InfiniCache) did this, but Concord isolates per-app caches because 99.7% of data objects are not shared across apps.
Problem 5 · medium
Concord's coherence-aware invocation scheduling hashes the invocation's inputs and routes it to the corresponding node. Why does this increase local cache hits? Give a scenario where it doesn't help.
Show answer
Hashing inputs assumes invocations with the same inputs access the same data—so routing them to the same node reuses that node's cached data. E.g., function `process_order(order_id=123)` likely accesses `orders/123`, so routing all `order_id=123` invocations to Node X keeps `orders/123` cached in X. This raises local hit rate from 75% (random) to 83% (coherence-aware). However, it doesn't help if inputs don't predict data access—e.g., a function that randomly samples from a large dataset (`random.choice(items)`) has no input-to-data correlation, so hashing inputs is no better than random.Misconception: that input-based hashing is a general-purpose cache optimization. It's specific to workloads with input-to-data locality, which serverless functions typically have (because inputs often encode keys or IDs).
Problem 6 · hard
Concord's write latency is dominated by storage latency (30ms), not invalidation latency (2ms per node). The paper measures 32.4ms write latency with 30 nodes. Explain why invalidations don't accumulate sequentially.
Show answer
Invalidations are sent in parallel with storage update. The home CA sends all invalidations simultaneously (not sequentially), so they all complete within ~2ms total (not 2ms × N). The 30ms storage latency dominates. With 30 nodes, the paper measures 32.4ms—only 2.4ms (8%) overhead over storage alone. Invalidations are hidden by storage latency because they're parallelized and faster than storage. If sharers were 100+ and invalidations were sent sequentially (or storage were instant), invalidations could dominate—but Concord's design (parallel sends, modest sharer counts) avoids this.Misconception: that invalidation latency accumulates linearly with sharer count. Parallel sends and hiding behind storage latency make invalidations nearly free in Concord's workloads.
Problem 7 · medium
Faa$T's version check on every read adds overhead. Concord's local read hit is 1.6ms. The paper does not evaluate very large items (1MB+). Based on the paper's mechanisms, would version checking be faster than invalidation-based coherence for 1MB items? Why or why not?
Show answer
The paper evaluates items up to 256KB (Fig. 3) and notes that 80% of serverless data is ≤12KB. For 1MB items: Version check would still be ~3.8ms (version number is small). Fetching 1MB data over the network would be slower (~tens of ms). If the version matches (no update since last read), Faa$T avoids the 1MB fetch—saving time. However, if updates are frequent (20% writes in the paper's workload), invalidation-based coherence would still win because Concord's local reads (1.6ms) are faster than Faa$T's version checks (3.8ms), regardless of item size. The paper does not evaluate 1MB items, so this conclusion is extrapolated beyond the paper's scope.Misconception: that version checking is always overhead. For very large items with rare updates, version checking can be cheaper than invalidation-based protocols—but that's not the serverless workload the paper studies (small, frequently-updated items).
Problem 8 · easy
Concord uses write-through caches. Why not write-back (buffer writes locally, flush later) for even lower latency?
Show answer
Write-back would reduce write acknowledgment latency to ~2ms (local cache update only) vs Concord's 30ms (storage + invalidations). But write-back requires complex fault tolerance: if a cache crashes before flushing, dirty data is lost. You'd need (a) logging dirty data to durable storage (adding I/O), (b) replicating dirty data across nodes (adding network traffic), or (c) accepting data loss (unacceptable for serverless). Concord chooses write-through for simplicity: storage is always up-to-date, so crashes are handled by just evicting stale cached copies (no data recovery). The 30ms write latency is acceptable for serverless (which already has 30–300ms response times). The paper does not evaluate write-back latency, so the "~2ms" claim is unsupported.Misconception: that write-back is always better. It's faster but adds fault tolerance complexity—Concord trades write latency for simplicity and safety.
Quick answers

Frequently asked questions

Ten questions newcomers ask most. Tap any to expand.

Q1Why is invalidation-based coherence feasible for serverless but not for large distributed systems?▾
Two reasons: (1) Serverless apps have small coherence domains (≤16 nodes per app, typically 3–5 sharers per data item), so the number of invalidations is modest (avg 1.2, max 4.9). Large systems (100s–1000s of nodes) can't afford O(N) invalidations per write. (2) Serverless functions are idempotent and retry on failure, so Concord's fault tolerance (write-through + evict stale copies) is acceptable. Large systems (e.g., distributed databases) need stronger consistency guarantees, which invalidation-based protocols struggle to provide at scale.
Q2How does Concord handle external writes to global storage (e.g., from a non-serverless app)?▾
Users specify which storage locations (e.g., Azure Blob folders) are used by their serverless apps. Concord registers a listener FaaS function that is invoked on every storage update in those locations (Azure supports this via triggers). When the listener detects an external write, it forwards the update to the application controller, which routes it to the home CA. The home CA sends invalidations to all sharers, ensuring cached copies are invalidated. Functions never read stale data from cache.
Q3What happens if the home node of a data item crashes during a write?▾
Concord ensures correctness via write-through + recovery protocol. If the home crashes after updating storage but before sending invalidations, some caches may still have stale S copies. The recovery protocol (Sec. III-F) prevents reads from those stale copies: when ZooKeeper detects the crash, other cache instances evict all data items homed in the crashed node. The next read will go to the new home (determined by consistent hashing), which fetches the latest value from storage (which is up-to-date due to write-through). No inconsistency occurs. The function that was writing may be retried (standard serverless behavior).
Q4Why does Concord use consistent hashing instead of a centralized directory?▾
A centralized directory would be a single point of failure and a bottleneck (all reads/writes must consult it). Consistent hashing distributes the directory across all cache instances: each data item has a home (determined by hash of address), and only the home manages that item's directory entry. This scales horizontally (more nodes → more directory capacity) and tolerates failures (crashed home's entries move to the next node on the hash ring). The cost is rehoming some items when nodes join/leave, but Concord's two-phase protocol handles this transparently.
Q5How does Concord's transaction support compare to Saga or Beldi?▾
Saga patterns require users to write compensating functions to roll back failed transactions—manual, error-prone. Beldi logs all storage accesses to enable rollback, adding storage I/O overhead. Concord detects conflicts automatically via coherence messages: if another process writes a data item you've read speculatively, you receive an invalidation and your transaction is squashed—no extra storage accesses. Concord buffers speculative state in local cache, commits atomically (with a global lock), and uses exponential back-off for fairness. Result: 54% faster than Saga, 20% faster than Beldi (Paper Fig. 15). Limitation: users must ensure no side effects beyond storage accesses (e.g., no HTTP calls during transactions).
Q6What is communication-aware function placement, and how does it differ from coherence-aware invocation scheduling?▾
Coherence-aware scheduling (Sec. III-G) routes invocations to nodes based on input hash—trying to co-locate invocations that operate on the same data. Communication-aware placement (Sec. IV-B) co-locates functions that frequently communicate (producer-consumer pairs) on the same node, even if they have different inputs. Concord learns these pairs by monitoring coherence traffic: if function F1 often writes data that function F2 reads, Concord places F1 and F2 on the same node, reducing network overhead (and enabling shared-memory communication). Result: 25% avg latency reduction for workflows with producer-consumer patterns (Paper Fig. 16).
Q7How much memory does Concord's directory consume per cache instance?▾
The paper reports avg cache instance size = 6.2MB (max 12.6MB), which includes both cached data and directory entries. The paper does not provide a per-directory-entry byte size or calculate directory overhead as a percentage of cache size. Estimates of directory entry size or overhead percentages are not supported by the paper.
Q8Does Concord support multi-threaded functions that share data via cache?▾
Concord is designed for multiprogrammed workloads (independent functions of the same app), not multi-threaded functions. If a function spawns threads that share data, Concord's coherence protocol operates at the granularity of cache instances (per-node), not per-thread. Multiple threads on the same node access the same cache instance, so no coherence messages are needed for intra-node sharing. However, Concord doesn't provide thread-level synchronization (locks, barriers)—users must use standard threading primitives. The paper does not discuss multi-threaded workloads.
Q9Why does Concord use ESI (no Modified state) instead of full MESI?▾
The Modified state (M) in MESI allows a cache to buffer writes without immediately updating storage—dirty data. Concord removes M for fault tolerance: if a cache with dirty M data crashes, the data is lost unless you add complex recovery (logging, replication). By using write-through (E state writes update storage immediately), Concord ensures storage is always up-to-date—no dirty data, no recovery complexity. The cost is higher write latency (30ms vs potentially lower for write-back), but serverless latencies are 30–300ms, so the overhead is acceptable.
Q10How does Concord compare to hardware CXL disaggregated memory schemes (e.g., Apta)?▾
Apta (a CXL cache coherence protocol) uses lazy invalidations: writes complete before invalidations are sent, allowing stale reads for a short window. Apta then prevents scheduling functions on nodes with stale data (coherence-aware scheduling). Concord uses eager invalidations: writes wait for invalidations before completing, so no stale reads occur. Result: Concord allows all nodes to schedule functions without interruption, while Apta's scheduler has only 8.9/15 nodes available at a time. Concord is 47% faster than Apta (Paper Fig. 17, Sec. VII). Trade-off: Apta's lazy invalidations are faster on writes, but Concord's eager invalidations are simpler and avoid scheduler bottlenecks.

Glossary & further reading

Every term in one place, then a path to go deeper.

Core terms
ConcordA distributed software caching system for FaaS that allows replicas and keeps them coherent with invalidations (inspired by hardware MESI).
Cache Agent (CA)Per-cache-instance controller that handles coherence messages, manages directory entries for homed data, and intercepts storage requests from local functions.
Data DirectoryDistributed across cache agents; each entry tracks sharers (list of nodes with cached copy) and state (E/S) for one data item.
Coherence DomainThe set of all cache instances for one application; each app has its own domain, isolated from other apps.
Home NodeThe node (cache instance) that holds the directory entry for a data item, determined by consistent hashing of the item's address.
SharerA cache instance that has a valid (E or S) copy of a data item.
E/S/I StatesExclusive (single cached copy, writable), Shared (multiple copies, read-only), Invalid (not cached). Concord uses ESI (MESI without M) for write-through caches.
InvalidationA coherence message sent by the home CA to all sharers of a data item when the item is written, instructing them to discard their cached copy.
Supporting terms
OFCSingle-home caching: each data item cached in only one node (its home). No coherence, but remote reads/writes can account for up to 82% of response time.
Faa$TVersioning-based coherence: each data item has a version number; non-home nodes check version on every read. Version access and checking can cost up to 78% of application response time for small items (≤12KB).
Write-throughWrites update storage immediately (before acknowledging), so storage is always up-to-date. Slower than write-back but simpler for fault tolerance.
Consistent hashingDistributes keys (data addresses) to nodes in a way that minimizes rehashing when nodes join/leave. Concord uses it to assign data item homes.
Coherence-aware invocation schedulingRoute invocations to nodes based on hash of inputs, betting that invocations with same inputs access same data. Raises local hit rate 75% → 83%.
Communication-aware function placementCo-locate functions that frequently communicate (learned from coherence traffic) on the same node. Reduces network overhead by 25%.
ZooKeeperDistributed coordination service that monitors node health via heartbeats and notifies cache instances of failures.
Two-phase protocolConcord's mechanism for adding/removing cache instances: Phase 1 synchronizes membership change, Phase 2 rehomes directory entries.

Romero et al. · SoCC '21

The versioning-based coherence scheme that Concord outperforms. Allows replicas, keeps coherent via version numbers. Single-home caching: each data item cached in only one node (its home). No coherence overhead, but remote reads/writes can account for up to 82% of response time. Concord vs OFC shows the benefit of allowing replicas.

A transaction system for serverless that logs all storage accesses to enable rollback. Concord's transaction support (Sec. IV-A) is 20% faster by detecting conflicts via coherence messages instead of logging.

The original paper on consistent hashing, used by Concord to assign data item homes and handle dynamic coherence domains (cache instances joining/leaving).
