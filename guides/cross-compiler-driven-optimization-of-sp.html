Mixing dense and sparse for real speedup, not paper gains

Most sparse kernels underperform cuBLAS until sparsity hits 76–98%. CROSS, a compiler framework, exploits non-uniform sparsity distribution to dynamically blend dense and sparse compute, achieving up to 3.75× speedup over cuBLAS and 2.51× over Sputnik at 70% sparsity where rivals fail.

📄 "CROSS" · Liu, Huang, Yang, Wang, Li, Jiang (HPCA '25) 🎓 Assumes sparse matrix basics; familiarity with GPU kernels helpful ⏱ ~25 min read
🧩

Give dense blocks to dense compute (cuBLAS), sparse blocks to sparse compute (Sputnik), and swing blocks to whichever balances the load — not the same format everywhere.

1Sparsity in DNNs — and the accuracy vs. compression trade-off▾

Modern transformers (BERT, GPT-2, Llama-2) have billions of parameters. Pruning — setting a fraction of weights to zero — compresses the model while (if done carefully) preserving accuracy. For example, Llama-2-7B at 68.98% sparsity loses 10.69 perplexity points (from 61.87 to 72.56) but shrinks memory footprint ~3×.

The paper studies fine-grained unstructured sparsity: any element in a weight matrix can be zero, with no constraints on pattern. This is more flexible than block sparsity (which requires square blocks and can degrade accuracy above 60% sparsity) but harder to accelerate.

Now you're ready: CROSS targets unstructured sparsity at 70–90%, where most existing sparse kernels underperform cuBLAS (dense).
2Why sparse kernels lose to dense — the 76–98% threshold problem▾

Sparse computation saves work (fewer multiplies) but adds overhead: index decoding, irregular memory access, load imbalance. The paper's experiments (Fig. 1) show Sputnik (state-of-the-art) beats cuBLAS only above 76% sparsity; cuSPARSE needs 98%. Below those thresholds, the overhead exceeds the savings.

This means sparse models with 70% sparsity (common in practice) often run slower with generic sparse kernels than with dense kernels that ignore the zeros.

Now you're ready: CROSS's insight is that not all parts of a 70%-sparse matrix should be handled the same way — some regions are too dense for sparse compute, others are sparse enough.
3Non-uniform sparsity distribution — the locality CROSS exploits▾

The paper profiled Llama-2-7B (70% overall sparsity) and found row sparsity (the fraction of zeros in each row) ranging from ~29% to ~99%. Figure 3 shows the distribution is highly non-uniform: 11.4% of rows in layer 31's Wq have 38–40% nonzeros (too dense for sparse), while 35.5% are above 85% sparsity (perfect for sparse).

Observation 1: storing a low-sparsity block in CSR can use more memory than dense storage (indices + values exceed raw values). Observation 2: generic sparse kernels use CSR everywhere, missing optimization opportunities.

Now you're ready: CROSS divides the matrix into tiles, measures local sparsity per tile, and chooses the encoding (dense vs. CSR) dynamically based on BeTT's sparsity thresholds (67.5% dense, 83.5% sparse for 256×256 blocks).
4Load imbalance from heterogeneous compute▾

If block A (85% sparse) is assigned to a Sputnik kernel and block B (40% sparse) to cuBLAS, but both run on the same SM, block B finishes first and idles while block A drags. Or worse: if both are forced into Sputnik, block B is inefficient; if both into cuBLAS, block A wastes cycles on zeros.

The paper's experiments (Fig. 9) show that without load balancing, the compute units handling dense blocks finish early, while units handling sparse blocks lag — overall throughput is limited by the slowest unit.

Now you're ready: CROSS introduces intra-batch (assign swing blocks to balance within one batch) and inter-batch (interleave batches to spread load) scheduling.
5The Block Execution Time Table (BeTT) — offline profiling as a compiler guide▾

CROSS constructs BeTT by offline-profiling SpMM (Sputnik) and GEMM (cuBLAS) for various block shapes (8×8, 16×16, …, 1024×1024) and sparsity rates (55%–99%, sampled at 1% intervals). For each configuration, the paper runs 10 trials, measures execution time, and fits a 5th-order polynomial to the sparsity → time curve.

From the fitted curves, CROSS computes two thresholds: dense threshold (sparsity below which GEMM < SpMM) and sparse threshold (above which SpMM reliably < GEMM). The region in between is the swing band. BeTT stores: (row_length, col_length, dense_thresh, sparse_thresh, polyfit_coeffs). Construction takes ~2 hours for a 16×16 minimum block on A100; storage ~80 kB.

Now you're ready: BeTT is the decision oracle — it tells CROSS which blocks should be dense and which sparse, based on measured performance, not heuristics.
6Coordinate space transformation — from 2D matrix to 1D kernel input▾

In the graph-level IR (e.g., TVM's Relay), matrix data is indexed by (row, col) coordinates. In low-level IR (CUDA kernel input), data is a 1D buffer. After CROSS splits the matrix into blocks and decides which are dense/sparse, it must map each block's 2D coordinates to the 1D position in the encoded buffer.

For dense blocks, this is straightforward (row-major or column-major offset). For sparse blocks (CSR), the transformation also encodes row pointers and column indices. CROSS performs this as the final Block-level IR → Low-level IR lowering step.

Now you're ready: coordinate transformation is the mechanical step that converts CROSS's high-level decisions (which blocks are dense/sparse) into actual kernel inputs.
The whole story in one place

CROSS in a nutshell

The problem, the compile-time measurement, the block-level decision, the load balancing, and how it all connects.

Sparse DNN inference (e.g., 70%-sparse Llama-2-7B) fails to see speedup with existing sparse kernels (Sputnik, cuSPARSE, SparseTIR) because they apply sparse encoding uniformly, even though large regions of the matrix are too dense to benefit (below 76% sparsity for Sputnik). The result: slower than cuBLAS (dense), despite having 70% zeros.

CROSS, an end-to-end compiler framework integrated into TVM, fixes this with format-variable tensor representation. At compile time, it (1) divides the weight matrix into fixed-size blocks; (2) measures each block's sparsity; (3) queries BeTT (offline-profiled lookup table) to classify each block as dense, sparse, or swing (in between); (4) fuses adjacent blocks of the same type to improve locality; (5) shrinks block boundaries if edges are fully sparse; (6) encodes dense blocks as raw arrays (for cuBLAS) and sparse blocks in CSR (for Sputnik).

Then, at the low-level IR, CROSS applies intra-batch load balancing: assign swing blocks to whichever compute paradigm (dense or sparse) currently has lower total load, ensuring the two don't finish at vastly different times. For multi-batch inference, inter-batch load balancing interleaves batches — sort one batch by ascending load, the next by descending, then concatenate — so high-load blocks from one batch pair with low-load blocks from another.

The payoff: on 5 models (BERT-base/large, GPT-2, Llama-2-7B, BART-large) at 70–90% sparsity, CROSS achieves 1.63–2.51× speedup over Sputnik (the best rival) and 1.64–3.75× over cuBLAS, with only 6.07% extra memory on average (from encoding some swing blocks twice). The cost: ~2 hours offline BeTT construction (one-time per GPU) + compile-time profiling.

🍯 Key takeaways

  • Not all sparse is sparse: at 70% overall sparsity, many tiles are 40–50% dense — worse for CSR than dense storage.
  • BeTT (Block Execution Time Table): offline-profiled lookup (block_shape, sparsity) → expected runtime + dense/sparse/swing classification.
  • Format-variable representation: dense blocks → raw arrays (cuBLAS); sparse blocks → CSR (Sputnik); swing → assigned for balance.
  • Homogeneous block fusion: merge adjacent dense (or sparse) blocks to improve data locality; re-query BeTT after fusion to confirm type.
  • Block shape shrinking: if block boundaries are fully sparse, compress them to reduce computation; update BeTT classification.
  • Intra-batch balancing: assign swing blocks to the compute set (dense or sparse) with lower cumulative latency.
  • Inter-batch balancing: sort batch₀ ascending, batch₁ descending, concatenate → high-load + low-load pairs across batches.
  • Result: up to 2.51× over Sputnik, 3.75× over cuBLAS at 70% sparsity; +6.07% memory overhead; all without custom hardware.
Six everyday pictures

How to think about CROSS

Each picture stands alone. Every card ends with an honest note about where the picture stops being true.

The core idea · Mixed storage and compute

Three different roads for three different terrains

You're paving a route through mountains, plains, and swamps. One type of pavement (asphalt = dense) is fastest on plains but terrible in swamps; another (gravel = sparse) works in swamps but wastes material on plains. CROSS uses asphalt for plains (dense blocks), gravel for swamps (sparse blocks), and picks whichever is cheaper for the borderline terrain (swing blocks), balancing the crew's workload. One-size-fits-all (CSR everywhere) overbuilds the plains and underperforms in swamps.

Where it breaks: real road-building switches materials at fixed points; CROSS can reassign swing blocks dynamically (intra-batch) or across batches (inter-batch), which is more like re-routing traffic than changing pavement mid-construction.
BeTT construction · Offline measurement

Tasting every jar in the pantry once

You have a pantry of jars (block shapes × sparsity levels). Before cooking (compilation), you taste each jar once (offline profiling) and label it: "sweet" (dense wins), "salty" (sparse wins), or "umami" (swing). At cooking time, you just read the labels — no re-tasting. CROSS spends ~2 hours building BeTT (one-time per GPU), then queries it in milliseconds during compilation.

Where it breaks: tasting is instant; profiling a 1024×1024 block at 99 sparsity levels takes seconds per configuration. Also, the labels assume the same GPU model — move to a different accelerator (e.g., H100) and you must rebuild BeTT.
Block fusion · Merging same-type neighbors

Combining adjacent fields under the same crop

Two adjacent fields both grow corn (both dense blocks). Harvest them separately and you make two trips; merge them into one larger cornfield and you make one trip with better tractor efficiency. CROSS fuses adjacent same-type blocks (e.g., two 256×256 dense → one 256×512 dense) if BeTT confirms the fused block remains the same type. This improves data locality and reduces kernel-launch overhead.

Where it breaks: real fields have fixed soil; CROSS's "type" can change after fusion if the combined sparsity crosses a threshold (e.g., two 68%-sparse swing blocks might fuse into a 68%-sparse block that BeTT reclassifies as dense). If that happens, CROSS aborts the fusion.
Block shape shrinking · Trimming the fat

Cropping the blank margins off a photo

A photo has white borders (fully sparse block edges). You crop them off before printing to save paper (computation). CROSS checks if a block's top/bottom/left/right edges are entirely zero; if so, it shrinks the block boundary and updates its shape in BeTT. The cropped block may then be reclassified (e.g., from swing to sparse if the remaining core is denser).

Where it breaks: cropping a photo is free; shrinking a block requires re-encoding (updating row pointers in CSR if sparse, or recalculating the dense array bounds). The paper reports this overhead is small (~10% of total preprocessing), but it's not zero.
Intra-batch balancing · Assigning swing blocks to balance load

Distributing dishes to balance two kitchen crews

Crew A (dense compute) has 2 seconds of work; crew B (sparse compute) has 5 seconds. You have a swing dish (swing block) that takes 3 seconds either way. Assign it to crew A → both finish in 5 seconds (balanced). Assign it to crew B → A finishes in 2s, B in 8s (imbalanced). CROSS sums expected latency for each crew, assigns swing blocks to whichever is shorter, minimizing the max latency (the bottleneck).

Where it breaks: real kitchen crews can't start a new dish mid-batch; CROSS's assignment is compile-time (fixed per batch). Also, the analogy assumes crews finish their entire load before the next batch; in reality, GPU SMs pipeline instructions, so the boundary between batches blurs — which is why inter-batch balancing helps.
Inter-batch balancing · Interleaving batches to spread load

Shuffling two decks of cards to even out high and low values

Deck A (batch₀) has blocks sorted by ascending load: 2, 5, 8, 10. Deck B (batch₁) descending: 9, 7, 4, 1. Interleave them: (2, 9), (5, 7), (8, 4), (10, 1) — each pair sums to ~11, balancing the load across compute units over time. CROSS sorts one batch ascending, the next descending, then concatenates, so high-load blocks from batch₀ pair with low-load blocks from batch₁.

Where it breaks: real cards are discrete; GPU batches aren't rigidly paired — SMs fetch work from a queue. The interleaving is a heuristic that statistically spreads load, not a guarantee that every SM processes exactly one high + one low block. Also, odd-sized batches leave the last batch un-interleaved.
The full explanation

How it actually works

Section by section, with the paper's key diagrams rebuilt as visuals you can read at a glance.

1 · Offline profiling — constructing BeTT

Before compilation, CROSS builds the Block Execution Time Table (BeTT) by profiling SpMM (Sputnik) and GEMM (cuBLAS) for a grid of (block_shape, sparsity) pairs. For example, shapes = {8×8, 16×16, 32×32, …, 1024×1024} and sparsity = 55%, 56%, …, 99% (sampled at 1% intervals). For each configuration, the paper runs 10 trials, measures execution time, and fits a 5th-order polynomial to the sparsity → time curve.

From the fitted curves, CROSS computes two thresholds: dense threshold (sparsity below which GEMM < SpMM) and sparse threshold (above which SpMM reliably < GEMM). The region in between is the swing band. BeTT stores: (row_length, col_length, dense_thresh, sparse_thresh, polyfit_coeffs). Construction takes ~2 hours for a 16×16 minimum block on A100; storage ~80 kB.

🌱
Concretely: for a 256×256 block at 72% sparsity, BeTT lookup returns: type = "swing" (between 67.5% dense_thresh and 83.5% sparse_thresh), expected_time_if_sparse = polyfit(0.72), expected_time_if_dense = GEMM_mean (constant). CROSS uses this to decide whether to encode as CSR or dense.

2 · Sparsity profiling & block classification

At compile time, CROSS divides the weight matrix (e.g., Llama-2 layer 31 Wq, shape 4096×4096) into fixed-size blocks (default 256×256). For each block, it calculates the local sparsity rate (1 - nonzeros / total_elements), queries BeTT with (block_shape, sparsity), and gets back: block_type (dense / swing / sparse) + expected_time.

The paper's profiling (Fig. 3) shows Llama-2 layer 31 Wq has row sparsity ranging from ~29% to ~99%. With dense_thresh = 67.5% and sparse_thresh = 83.5%, blocks below 67.5% sparsity are classified as dense, those above 83.5% as sparse, and those in between as swing. This non-uniform distribution is what CROSS exploits.

3 · Fusion of homogeneous blocks

Adjacent blocks of the same type (e.g., two 256×256 dense blocks side-by-side) are fused into one larger block (256×512 dense) if: (1) BeTT confirms the fused block retains the same type, and (2) the fused block doesn't exceed a maximum size (to avoid kernel register pressure). Fusion improves data locality (fewer kernel launches, larger GEMM tiles for cuBLAS) and reduces metadata overhead (fewer block headers).

For example, if block A (256×256, 40% sparse, type = dense) and block B (256×256, 45% sparse, type = dense) are adjacent in the row dimension, CROSS tentatively fuses them into AB (256×512, 42.5% sparse). It then queries BeTT: if 42.5% is still below dense_thresh (67.5%), AB is classified as dense → fusion accepted. If AB crosses into swing, fusion is rejected and the blocks remain separate.

💡
Why re-query BeTT after fusion? The combined sparsity is the average of the two blocks, but BeTT's thresholds may differ for the larger shape (256×512 vs. 256×256). The paper's experiments (Section V-E) show fusion improves performance by ~10% on average (measured as reduced preprocessing overhead + better kernel efficiency).

4 · Block shape optimization (shrinking)

After fusion, CROSS checks if a block's boundaries (top/bottom/left/right edges) are entirely zero. If so, it shrinks the block by trimming those rows/columns and updates the block's metadata: new shape, new sparsity rate (now computed over the smaller block), new BeTT classification.

For example, a 256×256 block with 90% sparsity (sparse block) has its top 32 rows and left 64 columns fully zero. CROSS shrinks it to (224×192, 78% sparsity). Re-querying BeTT: 78% falls in the swing band (67.5–83.5%) → reclassified as swing. The shrunk block is now smaller (less computation) but may be reassigned to dense or sparse depending on load balancing.

5 · Coordinate space transformation & sparse encoding

Each block is now tagged with: type, position (LocX = [row_start, col_start]), shape (shapeX = [rows, cols]), and value pointer. CROSS encodes:

  • Dense blocks: stored as raw row-major arrays (or column-major, depending on cuBLAS's preference). No indices.
  • Sparse blocks: encoded in CSR format: row_ptr (offsets into col_indices), col_indices, values. The encoding uses the block's LocX to compute global column indices (local_col + col_start).
  • Swing blocks: encoded in both dense and CSR formats (dual encoding). The intra-batch balancing step will pick one at Low-level IR.

The dual encoding of swing blocks adds ~6.07% memory overhead on average (measured across 5 models at 70–90% sparsity), but the paper argues this is acceptable given the performance gain (1.63–2.51× over Sputnik).

6 · Intra-batch load balancing

After encoding, CROSS has three sets of blocks per matrix slice (a horizontal band of blocks assigned to one compute unit): dense_set, swing_set, sparse_set. It computes the cumulative latency of dense_set (sum of BeTT-predicted times for all dense blocks) and sparse_set. Then, for each swing block, it assigns it to whichever set currently has lower cumulative latency. The pseudo-code (Algorithm 1 in the paper):

for each swing_block in swing_set:
    if dense_lat >= sparse_lat:
        sparse_set.insert(swing_block)
        sparse_lat += swing_block.overhead
    else:
        dense_set.insert(swing_block)
        dense_lat += swing_block.overhead

This greedy assignment minimizes max(dense_lat, sparse_lat), the bottleneck latency. The assigned swing blocks are then encoded in the chosen format (if assigned to dense_set, use the dense encoding; if to sparse_set, use CSR).

7 · Inter-batch load balancing

For multi-batch inference (batch_size > 1), CROSS interleaves batches to spread load. The algorithm (Algorithm 2):

  1. Sort batch₀'s slices by latency (dense_lat + sparse_lat) in ascending order.
  2. Sort batch₁'s slices in descending order.
  3. Concatenate corresponding slices from batch₀ and batch₁ to form a new "merged batch".
  4. Repeat for batch₂ (ascending), batch₃ (descending), etc.

The result: slice₀ from batch₀ (lowest load) is paired with slice₀ from batch₁ (highest load), slice₁ from batch₀ (second-lowest) with slice₁ from batch₁ (second-highest), etc. This statistically balances the workload over time, as the GPU scheduler processes merged batches in parallel.

🍯
The paper's experiments (Fig. 13) show inter-batch balancing yields 1.14–1.58× additional speedup over intra-batch alone when batch_size ≥ 8, and up to 4.10× over cuBLAS at batch_size = 128 (90% sparsity).
Follow the numbers, step by step

Worked examples

Four traces using the paper's real numbers — BeTT lookup, block fusion, load balancing, and the memory overhead calculation.

① BeTT lookup for a 256×256 block at 72% sparsity

You have a 256×256 block with 72% sparsity (28% nonzero). Query BeTT to classify it and predict execution time.

  1. Lookup thresholds. BeTT entry for 256×256 (from Fig. 6, M=K=N=256): dense_thresh = 67.5%, sparse_thresh = 83.5%.
  2. Classify. 72% is between 67.5% and 83.5% → type = "swing".
  3. Predict time if sparse. Use polyfit coeffs (5th-order): exec_time ≈ a₀ + a₁(0.72) + a₂(0.72)² + … The paper's Fig. 6 shows the mean curve drops from ~0.035 us at 55% to ~0.012 us at 95%. At 72%, interpolate: ~0.022 us.
  4. Predict time if dense. GEMM_mean (constant) ≈ 0.020 us (from Fig. 6, cuBLAS line).
  5. Decision. Since 0.022 us (sparse) > 0.020 us (dense) marginally, and the block is in the swing band, CROSS will assign it based on load balancing — if dense_set has lower cumulative latency, encode as dense; otherwise CSR.
🍯
Insight: BeTT doesn't just classify; it predicts execution time for both compute paradigms, enabling greedy load-balancing decisions without re-profiling at runtime.

② Fusing two adjacent dense blocks

Block A (256×256, 40% sparse, type=dense) and Block B (256×256, 45% sparse, type=dense) are adjacent in the row dimension. Should they be fused?

  1. Tentative fusion. Fused block AB shape = (256, 256+256) = 256×512. Combined sparsity = (40% × 256 + 45% × 256) / 512 = 42.5%.
  2. Query BeTT. Look up (256×512, 42.5%). The dense_thresh for 256×512 is unknown from the paper (not provided), but assume it's similar to 256×256 (~67.5%). 42.5% < 67.5% → type = dense.
  3. Decision. AB is still classified as dense → fusion accepted. The fused block is now (256×512, 42.5% sparse, type=dense), encoded as one contiguous dense array.
  4. Benefit. Fusion reduces kernel-launch overhead (one cuBLAS call instead of two) and improves data locality (larger GEMM tile). The paper's experiments show fusion yields ~10% performance gain on average.
💡
If the combined sparsity had crossed into the swing band (e.g., 70%), BeTT would reclassify AB as swing, and CROSS would abort the fusion to keep the blocks' type consistent with their original classification.

③ Intra-batch load balancing — assigning a swing block

A slice has dense_lat = 2.0 us, sparse_lat = 5.0 us. A swing block with overhead = 3.0 us (if assigned to either set) arrives. Where should it go?

  1. Compare cumulative latency. dense_lat (2.0) < sparse_lat (5.0).
  2. Assign to the shorter set. Insert swing block into dense_set. New dense_lat = 2.0 + 3.0 = 5.0 us.
  3. Final latencies. dense_lat = 5.0 us, sparse_lat = 5.0 us → balanced. Max(dense_lat, sparse_lat) = 5.0 us (the bottleneck).
  4. Counterfactual. If assigned to sparse_set instead: dense_lat = 2.0, sparse_lat = 5.0 + 3.0 = 8.0 → max = 8.0 us (worse).
🍯
The greedy assignment minimizes the max latency, which is the overall slice execution time (since dense and sparse compute happen in parallel, the slower one dominates).

④ Memory overhead from dual-encoding swing blocks

A 256×256 swing block (70% sparse, 30% nonzero) is dual-encoded. Calculate the overhead.

  1. Dense encoding. 256 × 256 × 4 bytes (FP32) = 262,144 bytes.
  2. CSR encoding (sparse). Nonzeros = 0.3 × 65,536 = 19,661. Values: 19,661 × 4 = 78,644 bytes. Col_indices: 19,661 × 4 = 78,644 bytes. Row_ptr: 257 × 4 = 1,028 bytes. Total CSR = 158,316 bytes.
  3. Dual encoding. 262,144 (dense) + 158,316 (CSR) = 420,460 bytes.
  4. Baseline (CSR only). 158,316 bytes. Overhead = (420,460 − 158,316) / 158,316 = 165.5% extra.
  5. Whole-model average. The paper reports +6.07% overhead on average because: (a) not all blocks are swing (many are dense-only or sparse-only), and (b) the dual encoding applies only at compile time — after intra-batch balancing, one encoding is discarded per swing block.
🌱
The 6.07% average is computed across all weight matrices in BERT-base, BERT-large, GPT-2, Llama-2, BART-large at 70–90% sparsity. The worst-case (single swing block) is much higher, but most blocks are not swing.

Practice problems

Try each before opening the answer. Every one ends with the common misconception it's checking for.

Problem 1 · easy
A 128×128 block has 85% sparsity. BeTT says dense_thresh = 70%, sparse_thresh = 88%. What type is it, and which compute paradigm should CROSS use if load balancing doesn't apply?
Show answer
Type = "swing" (85% is between 70% and 88%). If load balancing doesn't apply (e.g., only one block in the slice), CROSS defaults to whichever has lower expected execution time. Query BeTT: polyfit(0.85) for sparse, GEMM_mean for dense. Since 85% is closer to sparse_thresh than dense_thresh, sparse is likely faster → use Sputnik (CSR). However, the paper's experiments (Fig. 21) show the optimal benefit is around 68–72% (not 85%), so at 85%, sparse should reliably win.Misconception: that "swing" means "always use sparse." Swing blocks can go either way; CROSS picks based on BeTT-predicted time + load balance, not a fixed rule.
Problem 2 · medium
Two 256×256 blocks (A: 65% sparse, B: 75% sparse) are fused into AB (256×512, 70% sparse). BeTT for 256×512 says dense_thresh = 68%, sparse_thresh = 84%. Should the fusion be kept?
Show answer
Before fusion: A (65% sparse) is below 68% dense_thresh → type = dense. B (75% sparse) is between 68% and 84% → type = swing. After fusion: AB (70% sparse) is between 68% and 84% → type = swing. A changed type (dense → swing), so CROSS aborts the fusion and keeps A and B separate. The fusion is rejected.Misconception: that fusion is always beneficial if the combined sparsity is in a "reasonable" range. CROSS requires the fused block to retain the same type as the original blocks; if not, the fusion breaks the classification invariant and is aborted.
Problem 3 · medium
A 256×256 block (80% sparse, swing) has all zeros in the top 32 rows and left 64 columns. After shrinking, it becomes 224×192. The new sparsity is 68%. BeTT for 224×192 says dense_thresh = 65%, sparse_thresh = 82%. What happens?
Show answer
Before shrinking: 256×256, 80% sparse, type = swing (assuming thresholds ~67.5–83.5%). After shrinking: 224×192, 68% sparse. Query BeTT: 68% is between 65% and 82% → still swing. The shrunk block's type didn't change, so the shrinking is kept. The block is now smaller (less computation: 224×192 vs. 256×256), which improves efficiency.Misconception: that shrinking always changes the block type. Shrinking can reclassify a block if the new sparsity crosses a threshold, but if it stays in the same band (swing → swing), the type is unchanged and the shrinking is beneficial.
Problem 4 · hard
Intra-batch balancing assigns swing blocks to minimize max(dense_lat, sparse_lat). Prove or give a counterexample: the greedy algorithm (always assign to the currently shorter set) is optimal for minimizing the max latency.
Show answer
The greedy algorithm is optimal for this problem. Proof sketch: at each step, assigning the swing block to the currently shorter set increases both sets' latency, but it minimizes the max increase. If you assign to the longer set, the max increases by more. By induction, greedy minimizes the final max. Formal proof: this is a variant of the "load balancing on two machines" problem (Graham, 1966), where greedy (Longest Processing Time first, or here, assign to Min-Load machine) is a 4/3-approximation for makespan, but for identical processing times (all swing blocks have the same overhead), greedy is exact optimal.Misconception: that greedy is suboptimal because it's "myopic." For this specific problem (two sets, minimize max), greedy is provably optimal. However, if there were more than two compute paradigms (e.g., dense, sparse, quantized), the problem becomes bin-packing and greedy may not be optimal.
Problem 5 · medium
Inter-batch balancing sorts batch₀ ascending and batch₁ descending, then concatenates. Why does this help, and what's the worst-case scenario where it doesn't?
Show answer
Interleaving high-load (batch₀'s tail) with low-load (batch₁'s head) spreads the workload over time, so GPU SMs don't all process high-load blocks simultaneously (which would cause a bottleneck). The worst case: if all slices in batch₀ have identical load and all in batch₁ have identical load, sorting and interleaving has no effect (the load is already uniform). Also, if the GPU scheduler processes batches out-of-order (e.g., fetches slices randomly from a queue), the interleaving heuristic loses its benefit.Misconception: that inter-batch balancing is a guarantee of balanced load. It's a statistical heuristic that works when (a) loads vary significantly within a batch, and (b) the GPU scheduler processes slices in a predictable order. The paper's experiments (Fig. 13) show it helps on average, but it's not a universal fix.
Problem 6 · hard
The paper reports CROSS beats Sputnik by 1.63–2.51× at 70% sparsity but only 1.39–1.63× at 90%. Why does the speedup ratio decrease as sparsity increases, when sparse compute should become more effective?
Show answer
At 90% sparsity, most blocks are classified as sparse (above sparse_thresh ≈ 83.5%), so CROSS ends up using Sputnik for almost all blocks — the same as plain Sputnik. The speedup comes primarily from: (a) block fusion (slightly larger tiles), (b) inter-batch balancing (modest gain), and (c) avoiding the overhead of encoding the few remaining dense blocks in CSR. At 70% sparsity, many blocks are in the swing band or dense, so CROSS's mixed compute (cuBLAS + Sputnik) has more opportunity to beat uniform Sputnik (which forces all blocks into CSR, even the dense ones). The diminishing returns at high sparsity are expected — CROSS's advantage is in the non-uniform regime.Misconception: that CROSS's speedup should increase with sparsity because "sparse compute is faster at higher sparsity." True, but CROSS's edge is in avoiding sparse compute for dense blocks. At 90%, there are fewer dense blocks to avoid, so the gap narrows.
Problem 7 · easy
The paper says BeTT construction takes ~2 hours on A100 for a 16×16 minimum block with 1% sparsity sampling. If you increase the minimum block to 32×32, does construction time go up or down, and why?
Show answer
Construction time goes down. A larger minimum block means fewer shapes to profile (e.g., 16×16, 32×32, 64×64, … has more entries than 32×32, 64×64, …). Fewer shapes × 99 sparsity levels = fewer configurations to profile. The paper's Fig. 20a shows that 32×32 minimum has ~60% less construction time than 8×8, confirming this.Misconception: that larger blocks take longer to profile because they have more elements. While each individual profile run may take slightly longer (larger GEMM), the reduction in the number of configurations dominates, so total time decreases.
Problem 8 · medium
The paper says cuSPARSE needs 98% sparsity to beat cuBLAS, while Sputnik needs 76%. Why the huge gap, and what does Sputnik do differently?
Show answer
cuSPARSE uses a generic CSR kernel with no tiling optimization, so it suffers from: (a) irregular memory access (jumping through col_indices), (b) low warp utilization (threads in a warp access different rows), and (c) no data locality. Sputnik introduces hierarchical 1D tiling: it tiles the sparse matrix into row blocks, loads each block's nonzeros into shared memory, and processes them with better locality. This reduces the sparsity threshold from 98% to 76%. However, Sputnik still has overhead (index decoding, load imbalance), so it loses to cuBLAS below 76%.Misconception: that all sparse kernels are equally efficient. Sputnik is state-of-the-art for unstructured sparsity, but even it has a high threshold (76%). CROSS's insight is to bypass the threshold entirely for dense blocks.
Quick answers

Frequently asked questions

Ten questions newcomers ask most. Tap any to expand.

Q1How does CROSS differ from SparseTIR, which also uses multiple formats?▾
SparseTIR (ASPLOS '23) introduces a composable format abstraction (e.g., CSR + BSR) but requires manual selection of the optimal format combination, which is impractical at scale. CROSS automates the decision using BeTT: it profiles offline, classifies blocks as dense/swing/sparse at compile time, and dynamically assigns swing blocks for load balancing. SparseTIR also doesn't handle load balancing or inter-batch scheduling. The paper's comparison (Fig. 17) shows CROSS is 1.78–2.37× faster than SparseTIR-CSR+BSR.
Q2Why not just use block-sparse formats (BSR) like NVIDIA's Block-SpMM?▾
Block-sparse (BSR) requires uniform block sizes (e.g., all 32×32) and a minimum sparsity (~60%) to beat cuBLAS, but it degrades accuracy above 60% (prior work shows > 2% accuracy loss). CROSS uses fine-grained unstructured sparsity (any element can be zero), which preserves accuracy at 70–90% sparsity. BSR is efficient for structured patterns (e.g., attention heads), but CROSS targets the more general unstructured case where BSR fails.
Q3What happens if the sparsity distribution changes at runtime (e.g., dynamic pruning)?▾
CROSS is a compile-time framework — it assumes the sparsity pattern is fixed (e.g., a pruned model loaded from disk). If sparsity changes dynamically, CROSS would need to recompile (re-profile, re-classify, re-encode). For dynamic scenarios, a runtime approach (e.g., adaptive block selection) would be needed, but the paper doesn't address this. The paper's scope is static sparsity (the common case in inference).
Q4How does CROSS handle very small matrices (e.g., 64×64)?▾
For small matrices, the block size (default 256×256) exceeds the matrix size, so CROSS tiles into a single block and treats it as one unit (dense, swing, or sparse based on its overall sparsity). This works, but there's no opportunity for mixed compute (only one block), so the benefit is limited. The paper's experiments focus on large matrices (4096×4096 in Llama-2), where tiling into multiple blocks is effective.
Q5Why does CROSS use CSR instead of other sparse formats (COO, ELL)?▾
CSR (Compressed Sparse Row) is the format Sputnik optimizes for, and Sputnik is the state-of-the-art sparse kernel CROSS builds on. The paper could have used COO (coordinate format, simpler but more storage) or ELL (fixed-width, wastes space if rows have varying nonzeros), but CSR is a good balance: compact (row pointers + col indices) and cache-friendly (sequential access within a row). CROSS's innovation is not the format itself but the decision of when to use it (sparse blocks) vs. when to avoid it (dense blocks).
Q6What's the overhead of AtomicAdd when mixing dense and sparse results in one slice?▾
If a slice has both dense and sparse blocks, their results must be accumulated into the same output coordinates, requiring AtomicAdd (GPU atomic operations to avoid write conflicts). The paper doesn't quantify this overhead separately, but it's included in the overall execution time. Since AtomicAdd is expensive (serializes writes), CROSS tries to minimize it by: (a) fusing blocks of the same type (fewer slices with mixed types), and (b) ensuring most slices are homogeneous (all dense or all sparse) after intra-batch balancing. The experiments (Fig. 11) show the final speedup already accounts for AtomicAdd.
Q7Does CROSS work on GPUs other than A100 (e.g., V100, H100)?▾
The paper evaluates CROSS on NVIDIA A100. BeTT must be rebuilt for each GPU model (different compute capability, memory hierarchy, Tensor Core specs → different dense/sparse thresholds). For V100 or H100, you'd re-run the offline profiling (~2 hours) to construct a new BeTT. The rest of CROSS (graph-level IR, block-level fusion, load balancing) is GPU-agnostic and works as-is.
Q8Why does CROSS underperform at 50% sparsity (Fig. 11, bottom row)?▾
At 50% sparsity, most blocks are dense (sparsity < dense_thresh ≈ 67.5%), so CROSS falls back to cuBLAS for almost all blocks — effectively the same as plain cuBLAS. There's no speedup because there's no sparse compute to exploit. The paper's conclusion: sparsity below 50–60% is too low for any sparse optimization, including CROSS. The sweet spot is 70–90% sparsity, where CROSS's mixed compute shines.
Q9How does CROSS compare to dedicated sparse accelerators (e.g., Sanger, SpAtten)?▾
Sanger (MICRO '21) and SpAtten (HPCA '21) are hardware accelerators with custom datapaths for specific sparsity patterns (e.g., token-level pruning, reconfigurable systolic arrays). They offer higher speedup (3–10×) but are inflexible (fixed to one model structure) and require custom silicon. CROSS is a software-only solution (TVM compiler framework) that works on commodity GPUs with no hardware changes. The trade-off: CROSS is slower than a dedicated accelerator but more adaptable to evolving model architectures.
Q10What are the limitations and future work directions for CROSS?▾
The paper demonstrates CROSS on NVIDIA A100 for static unstructured sparsity at inference time. Extensions would include: (1) adapting BeTT thresholds per model (currently fixed from offline profiling), (2) handling convolution layers (currently converted to matrix multiplies with img2col), (3) multi-GPU parallelism (CROSS is single-GPU), and (4) combining with quantization (e.g., INT8 sparse). The core framework targets the common case (static sparsity, single GPU) but is not exhaustive.

Related papers

Sputnik: Fast Sparse Matrix Multiplication easy

Gale, Zaharia, Young, Elsen · SC '20

State-of-the-art sparse kernel with hierarchical 1D tiling. CROSS builds on this for sparse blocks. The 76% threshold is from their experiments.

SparseTIR: Composable Abstractions for Sparse Compilation medium

Ye, Lai, Shao, Chen, Ceze · ASPLOS '23

Introduces composable sparse formats (CSR+BSR, ELL+COO) but requires manual tuning. CROSS automates the choice using BeTT.

ASpT: Adaptive Sparse Tiling for Sparse Matrix Multiplication medium

Hong, Sukumaran-Rajam, Nisa, Singh, Sadayappan · ICS '19

Reorders matrices and metadata for better locality. CROSS doesn't reorder (keeps original layout) but achieves similar gains via mixed compute.

cuSPARSE: NVIDIA Sparse Matrix Library easy

NVIDIA · 2021 (docs)

Baseline generic sparse library. The paper shows it needs 98% sparsity to beat cuBLAS — the motivation for CROSS.

Sanger: A Co-Design Framework for Sparse Attention hard

Lu et al. · MICRO '21

Hardware accelerator with reconfigurable systolic array for sparse attention. CROSS is software-only (TVM) but targets the same domain (transformers).

SpAtten: Token-Level Pruning in Transformers hard

Wang et al. · HPCA '21

Hardware unit for fetching/reordering sparse tokens. CROSS handles weight sparsity (not activation), a complementary optimization.

TVM: An Automated End-to-End Optimizing Compiler medium

Chen et al. · OSDI '18

The compilation framework CROSS is built on. Provides graph-level IR (Relay), low-level IR (TIR), and Ansor auto-scheduler.
