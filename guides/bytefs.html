ASPLOS 2025 · A Field Guide

One drive, two doors — and a file system that knows which to use

A memory-semantic SSD lets you reach NAND flash two ways at once: byte-by-byte over CXL/PCIe load-stores, or 4 KB blocks over NVMe. ByteFS is the first file system that uses both on purpose — picking the cheaper door per data structure, and reshaping the SSD's own firmware so the byte door doesn't quietly explode into flash writes.

📄 "ByteFS: System Support for (CXL-based) Memory-Semantic SSDs" ✍ Li · Zhou · Ren · Huang · UIUC 🎓 Some storage/kernel basics assumed — primers cover them ⏱ ~35 min read
🚪

For each write, take the BYTE door when only a handful of bytes really changed; take the BLOCK door when a whole page is dirty — then absorb the byte writes in an in-SSD log so the page-granular flash underneath never sees the amplification.

Before you begin

Ten ideas the paper assumes you know

Each card clears up one concept ByteFS builds on. They're collapsed — tap any open. Read in order the first time, or jump to whatever looks unfamiliar. (If you work on OS-managed DRAM/NVM tiering and CXL, several will feel like home — the framing notes connect them to your world.)

byte path / byte interface block path / block interface SSD-DRAM firmware machinery NAND flash (durable backing store)
1What a memory-semantic SSD (M-SSD) is▾

A normal SSD is a block device: you talk to it in 4 KB blocks over NVMe. A memory-semantic SSD additionally exposes its internal DRAM as a memory region the CPU can reach with ordinary load/store instructions over PCIe (MMIO) or CXL. So one device offers two interfaces at once: byte-granular memory access and the usual block access.

The paper's prototype is a real OpenSSD FPGA board (1 TB flash, 16 channels, 1 GB DRAM, on-board ARM core running the firmware). Samsung's CMM-H is the commercial example it cites.

Now you're ready: ByteFS is a file system designed specifically for this dual-interface device — and the whole paper is about using both interfaces well.
2CXL.mem and load/store over an interconnect▾

Historically, anything across PCIe was reached by DMA in blocks. CXL (Compute Express Link) adds a cache-coherent memory protocol, CXL.mem, that lets the CPU issue cacheable load/store accesses to a device as if it were memory. The M-SSD's byte interface can ride either plain PCIe MMIO or CXL.mem.

The paper measures the byte path at 4.8 µs read / 0.6 µs write per cacheline on its PCIe-3.0×4 prototype, and notes CXL drops cacheline latency further (it models 175 ns in a sensitivity study).

Now you're ready: the "byte interface" is exactly the load/store-over-CXL/PCIe path. For a tiering/LtRAM reader, it's the same load-store device door you already think about — only the backing store is flash, not DRAM/NVM.
3Byte-addressable persistence (and why it's hard here)▾

"Persistent" means a write survives power loss. On NVM you flush a cacheline (clflush/clwb) and it's durable. On an M-SSD it's subtler: the byte write lands in the device's volatile DRAM buffer, and the device uses a battery to flush DRAM to flash on power loss. ByteFS also issues a zero-byte write-verify read after the flush to force any posted PCIe write to actually complete (a non-posted read serializes behind prior writes in the root complex).

Now you're ready: "byte-granular persistence" on an M-SSD = cacheline store → clwb → write-verify read, backed by battery-backed DRAM. Not free, but far cheaper than a 4 KB block write.
4The block-vs-byte interface gap▾

The device can be addressed by the byte, but the NAND flash inside it can only be read/written a whole page at a time (4 KB here) — a hard physical limit. So even a 64 B byte write would, naïvely, force a 4 KB read-modify-write of a flash page. The byte door at the top doesn't match the block-only store at the bottom.

This internal granularity gap is distinct from the external one (host talks bytes, legacy FS talks blocks). ByteFS has to close both.

Now you're ready: the single hardest problem in the paper is bridging byte-granular access on top of page-granular flash — solved with a log-structured write buffer inside the SSD.
5I/O amplification (read & write)▾

I/O amplification = bytes actually moved to/from the device ÷ bytes the application asked for. Persisting a 128 B inode field as a 4 KB block write is 32× write amplification. The paper measures real file systems (Ext4, F2FS) at 1.1–6.2× write and 1.1–1.7× read amplification across workloads (Table 2).

The biggest culprits are tiny metadata updates — inodes, bitmaps, dentries, data pointers, journals — that each ride a full 4 KB block.

Now you're ready: ByteFS's headline win is cutting this amplification by using byte writes for the small stuff. It reports up to 5.1× less write traffic.
6The kernel I/O paths: buffered, direct, mmap▾

Three ways an app reaches storage. Buffered I/O goes through the page cache: writes dirty a cached page, written back later. Direct I/O (O_DIRECT) bypasses the page cache and goes straight to the device. mmap maps file pages into the address space so loads/stores hit them directly.

ByteFS handles all three, and chooses byte vs block differently in each — e.g. in Direct I/O it just checks the request size (≤ 512 B → byte); in buffered I/O it can't see the true write size, so it measures how much of a page actually changed.

Now you're ready: "interface selection" is path-specific. The clever part is buffered I/O, where ByteFS uses copy-on-write to discover the real modified ratio.
7The page cache & copy-on-write (CoW)▾

The page cache holds file pages in host DRAM; dirty ones are flushed on fsync/msync, by the flush daemon, or under memory pressure. Problem: by writeback time the kernel only knows the page is "dirty," not which bytes changed.

ByteFS solves this with copy-on-write: on first modification it snapshots the original page (tracked in an XArray keyed by file offset). At writeback it XORs original vs current (AVX2, ~936 cycles/op) to find exactly the dirty 64 B cachelines.

Now you're ready: CoW + XOR is how ByteFS recovers byte-level change info that the page cache normally throws away — costing ~16% extra page-cache space.
8Log-structured writes & data coalescing▾

A log-structured store appends every write to the end of a log instead of updating in place. Many small appends to nearby addresses can then be coalesced — merged — into one big sequential write to the slow medium, and a background cleaner reclaims stale log space.

ByteFS applies this inside the SSD firmware: byte writes append to a 256 MB circular log in SSD DRAM at 64 B granularity; a background cleaner (kicks in at 85% utilization) coalesces and flushes whole pages to flash.

Now you're ready: the in-SSD log is what turns a storm of 64 B byte writes into a few coalesced 4 KB flash writes — bridging the internal granularity gap.
9SSD firmware: the FTL and its DRAM▾

Every SSD runs firmware with a Flash Translation Layer (FTL) that maps logical addresses to physical flash pages, does wear-leveling and garbage collection, and uses the device's internal DRAM for the mapping table and write buffers. On the prototype, the address-mapping table alone uses 512 MB of the 1 GB SSD DRAM; the write buffer 16 MB.

ByteFS doesn't replace the FTL — it extends the firmware (≈1.5 K lines of C, no hardware change) to add the log, its index, and transaction support alongside the existing FTL.

Now you're ready: ByteFS is a hardware/software co-design — host file system and SSD firmware change together. The firmware side is where the byte/flash gap gets closed.
10Crash consistency, journaling & redo logs▾

A file system must survive a crash mid-update without corrupting itself. The classic tool is a journal: write the change to a log first, then apply it in place — so on reboot you replay or discard. Ext4's ordered-mode journaling alone is 30.7% of its traffic on average (double writes of critical metadata).

ByteFS reuses its in-SSD write log as a redo log: metadata updates are already in the log, so it needn't double-write them. A 4 B transaction ID (TxID) tags each update; a 2 MB TxLog in SSD DRAM records commit order; RECOVER() replays committed entries on reboot.

Now you're ready: ByteFS gets crash consistency almost for free by making the byte-write log double as the journal — no separate double write.
The whole story in one place

ByteFS in a nutshell

The mismatch, the two-sided fix, and how the pieces connect.

A memory-semantic SSD offers two doors to the same NAND flash: a byte interface (cacheline load/store over PCIe MMIO or CXL.mem) and a block interface (4 KB NVMe). Existing file systems use exactly one. Block-based ones (Ext4, F2FS) waste the byte door and pay 1.1–6.2× write amplification persisting tiny metadata as full pages. NVM file systems (NOVA, PMFS) assume the byte door is near-DRAM-fast and lean on it exclusively — but on an M-SSD the byte path is microseconds, not nanoseconds, and flash is page-granular, so that assumption breaks.

ByteFS's insight is that the right granularity is per-data-structure and per-access, and it must be enforced end to end. On the host side, it characterizes every core file-system structure (Table 3) and routes each through the cheaper door: small metadata updates (inode fields, bitmaps, dentries, extents) go byte; bulk reads and large/dirty pages go block. For buffered writes, where the kernel only knows "page is dirty," it uses copy-on-write + an XOR to recover the true modified ratio and pick the door.

But a byte write still has to land on page-granular flash. So ByteFS also rewrites the SSD firmware: it turns the in-device DRAM into a log-structured write buffer (256 MB circular log, 64 B entries), indexed by a three-layer skip list, with a background cleaner that coalesces many small byte writes into a few sequential flash page writes. The same log doubles as a redo log for crash consistency, avoiding journaling's double writes. And to spend the precious SSD DRAM well, ByteFS uses coordinated caching: flash pages are cached only in the host page cache, never duplicated in SSD DRAM, leaving that DRAM entirely for the write log.

The payoff: versus Ext4, F2FS, NOVA, and PMFS, ByteFS is faster by up to 2.7× (e.g. 6.0× on file create vs Ext4, 4.1× on OLTP), and it cuts write traffic to the SSD by up to 5.1× — all while preserving crash consistency and recovering in ~4.2 s.

🍯 Key takeaways

  • An M-SSD exposes byte (CXL/PCIe load-store) and block (NVMe) interfaces to the same flash; ByteFS is the first file system to use both, adaptively.
  • Per-structure routing: small metadata → byte writes; bulk reads & large dirty pages → block (Table 3 characterization).
  • Buffered writes use copy-on-write + XOR to find the modified ratio; if < 1/8 of a page (512 B) is dirty, take the byte door.
  • The SSD firmware manages its DRAM as a 256 MB log-structured buffer, indexed by a three-layer skip list, that coalesces byte writes into page-granular flash writes.
  • The write log doubles as a redo log → crash consistency without journaling's double writes (Ext4 journaling alone is 30.7% of traffic).
  • Coordinated caching: flash pages cached only in the host page cache, never duplicated in SSD DRAM — that DRAM is reserved for the write log.
  • Result: up to 2.7× faster and 5.1× less write traffic vs SOTA, with crash consistency preserved and ~4.2 s recovery.
🔗
If you work on OS-managed tiering / LtRAM: ByteFS is the storage-stack mirror of your problem. You decide DRAM-vs-NVM placement; ByteFS decides byte-vs-block granularity for the same bytes. Its CoW+XOR modified-ratio detector is essentially a hot/cold classifier at cacheline grain inside the page cache — a mechanism you could lift for page promotion/demotion. And its core lesson echoes tiering: a device's load-store interface is not automatically the cheap one; the cost model (here, page-granular flash underneath) has to drive the policy.
Seven everyday pictures

How to think about ByteFS

Each picture stands alone — read them in any order. Every card ends with an honest note about where the picture stops being true.

The core idea · Dual interface

A narrow door and a loading dock

A warehouse has two entrances to the same storeroom: a narrow side door for one person carrying a small parcel, and a big loading dock for a forklift and a pallet. Sending one envelope through the loading dock means powering up a forklift for nothing; carrying a whole pallet through the side door means a hundred trips. ByteFS sizes the entrance to the shipment: a 64 B inode field takes the byte door; a freshly-written 16 KB file takes the block dock.

Where it misleads: a real warehouse's two doors are independent. Here both doors lead to the same page-granular shelves (flash), so even the byte door's parcels eventually get boxed into pallets — that's the in-SSD log's job, not visible in this picture.
The in-SSD log · Coalescing

Bundling errands into one trip

Five small errands, all near each other across town. Driving out and back for each is five tanks of gas. Instead you jot them on a running list, and once the list is full you make a single loop that hits them all. ByteFS's firmware does this: byte writes are appended to a 256 MB log; a background cleaner waits until the log is ~85% full, then coalesces nearby updates into one sequential 4 KB flash write.

Where it breaks: errands you batch can be done in any order; the log must respect commit order and only flush committed transactions — so the cleaner isn't free to reorder arbitrarily. Also, batching adds latency the picture hides; ByteFS relies on the flush being fully in the background.
Buffered writes · CoW + XOR

Spot-the-difference on two photos

You photocopy a page before editing it, then lay the copy over the original on a lightbox: only the changed lines stand out. ByteFS keeps a copy-on-write snapshot of each dirty page and XORs it against the current version at writeback. Wherever the XOR is non-zero, a 64 B cacheline changed. Few changes (< 512 B of the 4 KB page) → byte door; lots → block door.

Where it breaks: spotting differences on paper is free; here the snapshot costs real DRAM (~16% of the page cache) and the XOR costs CPU (~936 cycles/page). It's worth it because it converts a "page is dirty" bit into precise byte-level knowledge — but it isn't a free lunch.
Finding data in the log · Skip list

A book's index, in three depths

To find a fact you don't read the whole book — the index sends you to a shelf, then you flip to a page, then scan its lines. ByteFS locates a byte in its 256 MB write-log the same way, in three hops: layer 1 divides the address to pick 1 of ~16 partitions (no search — just walking to the right shelf); layer 2 is a skip list keyed by page, whose stacked "express-lane" pointers let you skip over the pages you don't want and land on yours in ~log(n) hops; layer 3 is a short sorted list of the writes inside that page. ~89 ns total — trivial next to microsecond flash.

The word "skip" trips people up: it's the name of the data structure — a skip list jumps over sorted entries during a search; nothing is dropped from storage. And unlike a book's index this one is rebuilt constantly as writes append and the cleaner reclaims space (~21 MB of SSD DRAM); the three depths also bound each skip list's size so no single structure grows unwieldy.
Coordinated caching · No duplication

Don't keep two copies of one file

Two people sharing a project don't each keep a private copy of every document — they agree on one canonical location. ByteFS refuses to cache flash pages in both the host page cache and the SSD DRAM. Read pages live only in the host cache; the scarce SSD DRAM is reserved entirely for the write log that absorbs persistent byte writes — the resource it actually needs.

Where it breaks: for a co-runner with great write locality, a page-granular SSD-side cache can absorb more writes (NOVA/PMFS do this). ByteFS deliberately gives that up to keep SSD DRAM for the log; the paper shows it sometimes pays slightly more flash traffic as a result, but wins overall because flushing is in the background.
Crash consistency · Redo log reuse

The to-do list that's also the receipt

You jot every task before doing it, then check it off. If you're interrupted, the list itself tells you what was finished and what to redo — no separate ledger needed. ByteFS's byte-write log already holds every metadata update, so it doubles as the journal: each entry carries a 4 B TxID, a 2 MB TxLog records commit order, and recovery replays only committed entries. No second "double write" of metadata.

Where it breaks: a paper list never gets corrupted mid-stroke; here durability hinges on battery-backed DRAM actually flushing the log on power loss, plus the write-verify read forcing posted writes to complete. The "free" journal assumes that hardware contract holds.
Why not just reuse NOVA · The latency gap

A sports car on a dirt road

NVM file systems (NOVA, PMFS) are tuned for a medium where every byte access is ~100 ns — a smooth highway, so they floor the byte interface and bypass caches. Bolt that onto an M-SSD and you're driving a sports car down a dirt road: the byte path is microseconds, and a cache miss means a slow flash page fetch. ByteFS instead designs for the rough surface — block reads to exploit locality, byte writes only where they pay, coalescing to hide flash latency.

Where it breaks: it's not that NOVA is badly built — it's optimal on its intended road. The analogy can imply NVM FS are "slow," but on real NVM they win. The point is fit-to-device, not absolute speed. With CXL lowering byte latency (175 ns), the road gets smoother and the gap narrows.
The full explanation

How it actually works

Section by section through ByteFS's design, with the paper's key diagrams rebuilt as visuals you can read at a glance, and pseudocode where it helps.

1 · The motivation — where amplification actually comes from (§3)

Before designing anything, the authors profile Ext4 and F2FS per data structure (Table 3 + Fig. 1). The verdict: most metadata updates are tiny but ride a full block, while most reads benefit from block locality. Concretely, from the paper:

The design rule that falls out: use the byte interface for small metadata writes; use the block interface for reads (to exploit host-cache locality) and for large/dense writes. That single table drives every routing decision in §4.5–4.6.

2 · Enabling byte access & persistence (§4.2)

The host maps the whole SSD into its address space via the PCIe BAR, so any SSD address is reachable by load/store (MMIO), or with CXL.mem as cacheable load/store. ByteFS notes it needs no cache coherency between SSD DRAM and host CPU cache — the host always holds the latest copy and the firmware never modifies host-written data behind its back.

Persisting a byte write is two steps, because the CPU's Write-Combining buffers and PCIe posting can leave a write in flight:

# persist a byte (cacheline) write to the M-SSD
store(mapped_addr, data)        # MMIO / CXL.mem store
clflush / clwb(mapped_addr)       # 1) push the dirty line out of CPU cache
x = read(mapped_addr, 0 bytes)     # 2) write-verify read: a non-posted read
                                #    serializes behind the posted write in
                                #    the PCIe root complex → write completed
💡
Tiering connection: this is the M-SSD analogue of the persist barrier you know from NVM (clwb+sfence), but with an extra device round-trip (the write-verify read) because the durability point is across PCIe, not on the memory bus. It's a big reason the byte path costs 0.6 µs, not 90 ns.

3 · The in-SSD log-structured memory (§4.3)

The crux. Flash is page-granular; byte writes are not. ByteFS reorganizes the SSD's DRAM into a log-structured write buffer at cacheline granularity. Every byte write is appended as a 64 B-aligned entry to a 256 MB circular log (head/tail pointers) — so it never touches flash on the critical path. That log holds the data; a separate three-layer index (~21 MB) holds only pointers that locate the current bytes for an address without scanning all 256 MB. A background cleaner coalesces the log into flash pages.

The log holds data; the index holds only the map. A log entry is the real written bytes (≤64 B) plus its address tag — a read ultimately lands here. A chunk (index) entry is just 9 B of pointer — block-offset (1 B) · log-offset (4 B) · length (4 B) — saying "the 64 B block at offset N lives there in the log, this long." Index = table of contents; log = the book.

Worked read — the app reads 8 bytes at page 1, byte 4:

  1. Layer 1: divide the address → partition P0. Pure arithmetic, no search.
  2. Layer 2: skip-list search keyed by page → skip past pages 0, 12, … and land on page 1's node. This is the only "skipping" — hopping over sorted entries, not dropping data.
  3. Layer 3: page 1's chunk list is sorted by in-page offset; binary-walk to the chunk covering byte 4 → {blkoff, logoff = 0x3A00, len = 8}.
  4. Follow logoff into the 256 MB log and read the 8 bytes. ≈ 89 ns total.

Overwrite that block later? A fresh entry is appended at the log's tail and the chunk's log-offset is repointed to it — the index always names the current location; the background cleaner later garbage-collects the stale bytes.

The cleaner (Algorithm 1 in the paper) runs in the background once log utilization passes 85%. It walks the layer-2 skip lists, and for each modified page: allocates a write-buffer slot, loads the old flash page only if the update is partial, merges the latest committed chunk versions, and writes coalesced 4 KB pages out with channel-level parallelism. Double buffering serves new writes in a fresh log region while the old one drains, so cleaning never blocks foreground I/O.

# Algorithm 1 (paraphrased): flush/clean the log
for page in traverse(layer2_skiplists):
    buf = alloc_write_buffer(page)
    if partial_update(page):
        read_flash_page(page, buf)          # read-modify-write only if needed
    for chunk in page.chunk_list:             # layer-3, ordered
        if not committed(chunk):
            migrate_to_new_log(chunk)         # keep uncommitted data
        entry = latest_committed_version(chunk)
        memcpy(buf + chunk.off, entry.data, entry.size)
    if buf_full: flush_to_flash(buf)          # coalesced page write
clean_log_region()                            # reclaim space + index

4 · Read & write through the dual interface (§4.4)

Putting the two doors together at the device:

Byte interfaceBlock interface
ReadMMIO load → firmware looks up log; hit returns data; miss fetches flash page, returns only the requested cachelineNVMe read → load page to transfer buffer; merge any dirty log entries; return full 4 KB page
Write64 B store + clwb; firmware appends to log, inserts chunk entry; concurrent writers update the chunk to newest version4 KB block via PCIe → FTL write buffer → flash; firmware invalidates any overlapping log entries (page is now authoritative)
⚠
Subtlety the paper is careful about: a block write makes the whole page authoritative, so existing log entries for that page are invalidated immediately — safe because ByteFS guarantees written-back pages from the host page cache are up to date. Mixing byte and block writes to the same page without this rule would risk a stale log entry resurrecting old bytes.

5 · Host-side metadata & the modified-ratio detector (§4.5–4.6)

On the host, ByteFS reshapes each metadata structure so the common update is a clean 64 B byte write, and caches them in host DRAM (radix trees), loading on a miss via block:

  • Inode (128 B) is split into a lower 64 B (hot: size, mtime, access rights) and upper 64 B (cold). A typical update is one atomic 64 B byte write.
  • Bitmaps are divided into 64 B groups as the update unit; allocation flips just a group via byte writes.
  • Dentries (inode# 4 B + type 2 B + namelen 2 B + name ≤256 B) update in 64 B–320 B byte writes.
  • Data pointers use Ext4-style extents (leaf node 16 B = offset 8 B + LBA 4 B + length 4 B), updated in place via byte writes.

For buffered I/O, the kernel only knows a page is dirty, not which bytes. ByteFS recovers that with copy-on-write: snapshot the page on first write, then at writeback XOR snapshot vs current to compute the modified ratio r = Modified / total (counts of dirty 64 B cachelines). The rule:

# buffered-write interface selection (per dirty page)
dup   = cow_snapshot(page)               # tracked in an XArray by file offset
diff  = avx2_xor(page, dup)            # ~936 cycles, up to 14 GB/s
r     = popcount_dirty_cachelines(diff) / 64
if r < 1/8:                              # < 512 B of the 4 KB page changed
    persist via BYTE interface          # write only the dirty 64B chunks
else:
    persist via BLOCK interface         # whole-page writeback is cheaper
🌱
For Direct I/O the size is known up front, so the rule is just: request ≤ 512 B → byte, else block. mmap reuses the buffered-I/O machinery (CoW on mapped pages). The CoW snapshots cost ~16% extra page-cache space on average (max page cache 8 GB in their runs).

6 · Crash consistency for free, and recovery (§4.7)

Because every metadata update is already in the in-SSD write log, ByteFS uses that log as a redo log — no separate double write. A transaction gets a 4 B TxID from a global counter; the host's TxTable tracks in-flight transactions; updates are persisted into the log tagged with their TxID. On commit, a custom NVMe COMMIT(TxID) appends a 4 B entry to the 2 MB TxLog in SSD DRAM, fixing commit order. The firmware later propagates committed updates to flash in TxLog order via log cleaning.

# transaction commit (firmware-assisted)
tx = new_transaction()                 # 4B TxID from global counter; tracked in host TxTable
for upd in tx.updates:
    append_to_log(encode(upd, tx.id))  # 1  persist (byte interface)
nvme_cmd(COMMIT, tx.id)               # 2  firmware appends TxID to 2MB TxLog
# 3  firmware flushes committed updates to flash in TxLog order (background)

Recovery (RECOVER()): scan the log; for each entry check its trailing 4 B TxID against the TxLog — discard if absent (uncommitted), else flush remaining committed entries to flash in TxID order; then replay data journals if enabled. Measured recovery is 4.2 s on average: 0.9 s to load SSD DRAM content + 2.7 s to scan the log + TxLog and flush committed entries.

💡
For data (not just metadata), small writes ride the same byte-log transaction mechanism; large writes combine JBD2 (Ext4's journaling layer) with a ByteFS commit entry carrying the TxID — so ByteFS offers the equivalent of Ext4 ordered-mode plus optional data journaling.

7 · Coordinated caching (§4.3, end)

SSD DRAM is scarce and shared with the FTL (512 MB mapping table + 16 MB write buffer on the prototype). ByteFS refuses to cache flash pages on both sides: it keeps no page-granular cache inside the SSD, caching loaded pages only in the host page cache. That frees the SSD DRAM to be the 256 MB write log — the resource that actually moves the needle for persistent byte writes. The trade is honest: with poor host-cache locality this can cost extra flash reads, and ByteFS sometimes pays more flash write traffic than NOVA/PMFS — but those flushes are in the background and are outweighed by the write-log's benefits.

Follow the numbers, step by step

Worked examples

Five traces, each carried all the way through with the paper's actual numbers — an inode update, the modified-ratio decision, a full open/write/fsync trace, the firmware storage budget, and a recovery tally. Arithmetic is recomputed here.

① The inode-update write amplification, computed

An application updates a file's size and mtime — a few fields of its 128 B inode. Compare Ext4's block path with ByteFS's byte path.

  1. Ext4 (block): the inode lives in a 4 KB on-disk inode block. Persisting any field forces a whole 4 KB write. Amplification = 4096 / 128 = 32× over the 128 B inode (and far more over the handful of bytes that truly changed).
  2. ByteFS splits the inode into a hot lower 64 B (size, mtime, access rights) and a cold upper 64 B. The changed fields are all in the lower half.
  3. ByteFS (byte): persist just the 64 B lower region with one atomic byte write. Versus Ext4's 4 KB: 4096 / 64 = 64× less data moved for that update.
  4. Why this dominates: inodes are 35% (Ext4) and 24.4% (F2FS) of device write traffic — bytes actually hitting flash, not app-requested bytes. It's this high because an inode is touched on nearly every file/dir op and each 128 B update is inflated to a full 4 KB block. (Block writes are all 4 KB, so "% of writes" and "% of bytes written" coincide here.) The inode's logical size is tiny, but its device footprint isn't — which is exactly why shrinking each write 4 KB → 64 B is the single biggest lever.
🍯
Insight: the win isn't a clever algorithm, it's matching the write size to the change size — and the inode split makes "the change" reliably fit in one 64 B cacheline so the byte door is always available.

② The modified-ratio decision (buffered write)

A cached 4 KB page is written back. ByteFS must choose byte vs block. Recompute the threshold and decide for two cases.

  1. Page geometry: a 4 KB page = 4096 / 64 = 64 cachelines of 64 B each.
  2. The rule (§4.6): use the byte interface when r = Modified/total < 1/8. That threshold in bytes is 4096 / 8 = 512 B, i.e. fewer than 512/64 = 8 dirty cachelines.
  3. Case A — a 200 B record updated: touches ⌈200/64⌉ = 4 cachelines → r = 4/64 = 0.0625 < 0.125. Byte door: write ~256 B instead of 4 KB (≈16× less).
  4. Case B — a 1 KB region rewritten: touches 16 cachelines → r = 16/64 = 0.25 ≥ 0.125. Block door: a single 4 KB writeback beats 16 separate byte persists.
  5. The cost of knowing: finding the dirty cachelines needs the CoW snapshot + an AVX2 XOR (~936 cycles/page, up to 14 GB/s) and ~16% extra page-cache DRAM.
💡
Slide the interactive on the "How It Works" tab to 8 cachelines and watch the verdict flip from byte to block — that's exactly the r = 1/8 knee computed here.

③ A full open → write → fsync trace (Fig. 5)

The paper's running example: open and read a 16 KB file, overwrite the first 1 KB, then fsync. Follow the doors.

  1. open (R1–R3): path lookup walks the tree; uncached components are fetched via the block interface (whole block, to exploit dentry/inode locality in the host cache).
  2. read 16 KB (R4): file data read in blocks and cached in the host page cache — block reads everywhere, by design.
  3. write first 1 KB (W1–W2): ByteFS finds the cached page via the inode's address_space and does copy-on-write on it, recording the duplicate.
  4. persist inode (W3): a transaction (TxID) is opened; the inode update goes to the firmware write log via the byte interface — atomic 64 B.
  5. fsync (S1–S5): XOR original vs copied page (S2) → here 1 KB dirty = 16 cachelines = r = 0.25 ≥ 1/8, so the block interface is chosen for the data writeback (S3); since metadata is already persisted, TxCommit(TxID) finalizes after the block I/O (S4); committed data flushes to flash in the background (S5).
🍯
Insight: a single syscall sequence uses both doors — block for the bulk 16 KB read and the 1 KB data writeback, byte for the small inode metadata — and the commit is ordered through the firmware log so a crash mid-fsync is recoverable.

④ The firmware storage budget on the prototype

Where does the 1 GB of SSD DRAM go, and how big is the log index? Add it up from §4.3 and §4.9.

  1. FTL essentials: address-mapping table = 512 MB; FTL write buffer = 16 MB. (For a 1 TB SSD at 4 KB pages that's 1 TB/4 KB = 268,435,456 pages → 512 MB / that ≈ 2 B of mapping per page, a sane page-table-entry size.)
  2. ByteFS write log: 256 MB circular region of 64 B-aligned entries.
  3. Log index: ~21 MB on average for a 256 MB log → index overhead ≈ 21/256 = 8.2% of the log region. Each chunk entry is just 1 + 4 + 4 = 9 B.
  4. TxLog: 2 MB, holding 4 B commit entries → room for ~500 K in-flight committed TxIDs.
  5. Tally: 512 + 16 + 256 + 21 + 2 ≈ 807 MB of the 1 GB, leaving headroom — and notably no page-granular SSD-side data cache (coordinated caching reclaims that space for the log).
💡
The punchline of the budget: the design spends DRAM on indexing and buffering writes, not on a second copy of read data. That's the coordinated-caching decision, made concrete in bytes.

⑤ Recovery time, decomposed

After a power cut mid-YCSB, how long is reboot-to-consistent, and what dominates? (§5.5)

  1. Load SSD DRAM content from battery-backed DRAM: 0.9 s.
  2. Scan the global log + TxLog, discard uncommitted entries (TxID absent from TxLog), flush committed entries to flash in TxID order: 2.7 s.
  3. Sum of the two named phases: 0.9 + 2.7 = 3.6 s.
  4. Reported average total: 4.2 s — so ≈ 0.6 s is the remaining work (data-journal replay if enabled, log-region + TxLog cleanup, bring-up). The log scan dominates, as expected for a redo-log recovery.
🌱
Honesty note: the paper states the 0.9 s and 2.7 s sub-steps and the 4.2 s average; the ~0.6 s residual is our subtraction, attributable to the §4.7 steps the paper lists (data-journal recovery, cleanup) but doesn't separately time.
Test yourself

Practice problems

Try each before opening the answer. Every one ends with the common misconception it's checking for.

Problem 1 · easy
An M-SSD exposes a byte (load/store) interface. Why can't ByteFS just treat it like NVM and persist every write byte-by-byte, the way NOVA does?
Show answer
Two reasons. (1) The byte path is microseconds, not nanoseconds — 0.6 µs write / 4.8 µs read per cacheline on the prototype, vs ~90–300 ns for NVM — so spamming it for large writes is slow. (2) The flash underneath is page-granular (4 KB), so a byte write that misses the in-SSD buffer triggers a flash page access anyway. ByteFS therefore uses byte writes only where they pay (small metadata) and block for bulk, and coalesces byte writes in an in-SSD log so flash sees page-granular traffic.Misconception: that a load/store interface implies DRAM-like cost. The interface is byte-addressable; the medium and the interconnect are not byte-cheap.
Problem 2 · medium
A 4 KB page is written back in buffered mode with 300 B of changes spread across 5 separate 64 B cachelines. Does ByteFS pick byte or block? Show the threshold computation.
Show answer
A page is 4096/64 = 64 cachelines. The threshold is r < 1/8, i.e. fewer than 8 dirty cachelines (< 512 B). Here 5 cachelines are dirty → r = 5/64 ≈ 0.078 < 0.125 → byte interface. ByteFS persists ~5×64 = 320 B of byte writes instead of a 4 KB block — about 12.8× less data. Note it's the number of dirty cachelines, found by the CoW+XOR, not the 300 raw bytes, that matters.Misconception: that 300 B "obviously fits" so byte is trivial — but if those 300 B were smeared across 9+ cachelines, r would exceed 1/8 and block would win. Granularity, not raw byte count, decides.
Problem 3 · medium
ByteFS reuses its in-SSD write log as a redo log for crash consistency. What concrete cost does this avoid, and roughly how big was that cost in Ext4?
Show answer
It avoids the double write of journaling: a journaled FS writes critical metadata once to the journal and again in place. Since ByteFS's metadata updates are already durably in the byte-write log, the log is the journal — only a 4 B TxID and a TxLog commit entry are added. In Ext4 ordered mode, journaling is 30.7% of total traffic on average, so eliminating the double write is a large saving on top of the per-update amplification win.Misconception: that crash consistency requires a dedicated, separate journal. Here a structure built for another purpose (absorbing byte writes) provides the ordering and durability the journal would.
Problem 4 · hard
Suppose ByteFS allowed a byte write and a 4 KB block write to the same page to coexist in the log without invalidation. Construct a sequence where this resurrects stale data, and state the rule ByteFS uses to prevent it.
Show answer
Sequence: (1) byte-write cacheline X of page P → log entry e₁ = {P, off=X, new}. (2) later, the host page cache writes the whole page P back via the block interface with an even newer X. If e₁ is left in the log, a subsequent read or a log flush could merge e₁ over the block-written page and restore the old X — stale data resurrected. ByteFS's rule (§4.4): on a block write of page P, the firmware scans the skip list and invalidates all log entries for P immediately, because a written-back page from the host cache is guaranteed up to date. So the page becomes the single source of truth and no log entry can override it.Misconception: that byte and block writes are independent fast paths. They alias the same page, so block writes must authoritatively supersede overlapping byte-log entries.
Problem 5 · medium
ByteFS deliberately keeps no page-granular cache inside the SSD, caching read pages only in the host page cache. Name one workload pattern where this hurts ByteFS, and explain why it accepts the cost.
Show answer
A write-heavy workload with strong page-level write locality (the paper points to create/Fileserver for reads and mkdir/rmdir for writes). A page-granular SSD-side cache — which NOVA/PMFS keep — could absorb repeated writes to the same page before it ever reaches flash, so ByteFS may incur higher flash traffic via its read-modify-write during log cleaning. ByteFS accepts this because (a) the freed SSD DRAM becomes the 256 MB write log, which is the bigger win, and (b) the extra flash work happens in the background via double-buffered cleaning, off the critical path.Misconception: that "no duplication" is always a pure win. It's a deliberate trade — give up SSD-side absorption to fund the write log; net positive on average, not universally.
Problem 6 · hard
The byte-write persist sequence is store → clwb → zero-byte "write-verify" read. Why is the write-verify read necessary, and what hardware property makes a plain read sufficient to guarantee the prior write completed?
Show answer
PCIe posted writes return immediately and may still be in flight (buffered in CPU Write-Combining buffers or pending in the root complex) after the store/clwb retire — so durability isn't yet guaranteed. The fix exploits ordering in the PCIe root complex: reads are non-posted and are serialized behind earlier posted writes to the same path. Issuing a (zero-byte) read therefore cannot complete until the preceding posted write has drained to the device. The returning read is proof the write landed. This is the M-SSD analogue of an NVM persist barrier, but with a device round-trip — a major reason byte-writes cost ~0.6 µs.Misconception: that clwb alone makes the write durable on an M-SSD. clwb evicts from CPU cache, but the durability point is across PCIe; you need the read to flush the posted write to the device.
Problem 7 · easy
In a sensitivity run, the authors emulate a CXL-based M-SSD with 175 ns cacheline access instead of the prototype's 4.8 µs. Qualitatively, what should happen to ByteFS's advantage, and why doesn't it vanish?
Show answer
Lower byte-path latency (CXL.mem) makes the byte interface cheaper, so both ByteFS and NOVA improve on some workloads. But ByteFS's advantage doesn't vanish because its wins also come from sources CXL doesn't touch: reduced flash traffic via coalescing, fewer 4 KB writes for tiny metadata, and hiding flash write latency by background log flushing. NOVA still doesn't optimize for the high flash access latency, so ByteFS stays ahead.Misconception: that ByteFS's benefit is purely "byte writes are faster." Much of it is reducing the amount of flash I/O and ordering it well — orthogonal to interconnect speed.
Problem 8 · medium
The three-layer skip list resolves a lookup in a full 256 MB log in ~89 ns, while a flash page access is microseconds. Why did the authors still bother making the index three layers and per-partition instead of one big skip list or a hash table?
Show answer
Two reasons given in §4.3. (1) Splitting into 16 MB partitions keeps each skip list small, bounding lookup latency and avoiding one giant structure — layer 1 is a cheap divide to pick the partition. (2) A skip list (vs a hash table) supports arbitrary chunk sizes and range lookups: file systems issue variable-size accesses, and an ordered chunk list (layer 3, by in-page block offset) lets a range request be served without breaking it into fixed-size keys. 89 ns is already negligible vs flash, but the structure also keeps memory bounded (~21 MB) and supports the access patterns hash tables handle poorly.Misconception: that since 89 ns ≪ flash latency, index design doesn't matter. It matters for memory footprint, range queries, and keeping worst-case lookup bounded as the log fills.
Quick answers

Frequently asked questions

Twelve questions a systems reader asks first. Tap any to expand.

Q1What exactly is a "memory-semantic SSD," and how is it different from NVM?▾
An M-SSD is a flash SSD that also exposes its internal DRAM as a memory region the CPU reaches by load/store over PCIe MMIO or CXL.mem — so it has both a byte and a block interface to the same NAND. Unlike NVM (e.g. Optane), which sits on the processor memory bus with near-DRAM latency and a PCM-like medium, an M-SSD is PCIe-attached, microsecond-latency on the byte path, built on commodity NAND, and ~40× cheaper per GB (~$0.22/GB vs NVM's ~$3.6/GB in the paper's Table 1).
Q2Why not just run Ext4 or NOVA on the M-SSD?▾
Ext4/F2FS use only the block interface, so they pay 1.1–6.2× write amplification persisting tiny metadata as 4 KB blocks. NOVA/PMFS are built for NVM and assume the byte path is DRAM-fast, leaning on it exclusively and bypassing caches — but on an M-SSD the byte path is microseconds and flash is page-granular, so that assumption misfires. Neither uses both interfaces adaptively or addresses the internal byte/page granularity gap. ByteFS is designed for exactly this device.
Q3How does ByteFS decide byte vs block for a given write?▾
It depends on the I/O path. Direct I/O: by request size (≤ 512 B → byte). Buffered I/O: it can't see the true write size, so it copy-on-writes the page, XORs original vs current at writeback to find the dirty 64 B cachelines, and uses byte if the modified ratio r < 1/8 (under 512 B of the 4 KB page), else block. mmap reuses the buffered path. Metadata structures are reshaped so their common update is a clean 64 B byte write.
Q4How does a byte write avoid blowing up into a 4 KB flash write?▾
The SSD firmware turns its DRAM into a 256 MB log-structured write buffer at 64 B granularity. Byte writes are appended to the log (never touching flash on the critical path) and indexed by a three-layer skip list. A background cleaner, triggered at 85% utilization, coalesces many nearby byte writes into a few sequential 4 KB flash page writes — so the page-granular flash sees coalesced traffic, not a storm of tiny RMWs.
Q5Where does CXL come in?▾
CXL.mem lets the host issue cacheable load/store accesses to the device, a faster realization of the byte interface than plain PCIe MMIO. ByteFS's byte path works over either; the paper's prototype uses PCIe 3.0 ×4 (byte writes 0.6 µs), and a sensitivity study models a CXL-based M-SSD with 175 ns cacheline latency, where byte access (and thus several workloads) gets faster. CXL is the trajectory that makes the byte interface more attractive over time.
Q6How does ByteFS stay crash-consistent without journaling's double write?▾
Its in-SSD byte-write log doubles as a redo log. Each update is tagged with a 4 B TxID and persisted in the log; a custom NVMe COMMIT(TxID) appends to a 2 MB TxLog that fixes commit order; the firmware later flushes committed updates to flash in that order. On reboot, RECOVER() scans the log, discards entries whose TxID isn't in the TxLog (uncommitted), and flushes the rest. So the structure that absorbs byte writes also provides atomicity and ordering — no separate double write (Ext4 journaling alone is 30.7% of traffic).
Q7What is the "coordinated caching" decision and why does it matter?▾
SSD DRAM is scarce (the FTL mapping table alone is 512 MB of 1 GB on the prototype). ByteFS keeps no page-granular data cache inside the SSD — read pages are cached only in the host page cache — so the SSD DRAM can be the 256 MB write log. It's a deliberate trade: you give up SSD-side write absorption (which NOVA/PMFS keep) to fund the log. Net positive because the log + coalescing + background flushing outweigh the occasional extra flash traffic.
Q8What does CoW + XOR cost, and is it worth it?▾
The copy-on-write snapshots occupy ~16% of the page cache on average (max page cache 8 GB in their runs), and each writeback XOR is ~936 CPU cycles (AVX2, up to 14 GB/s). It's worth it because the page cache otherwise discards which bytes changed; recovering that lets ByteFS take the byte door for lightly-modified pages, which is where the amplification savings live. The paper judges the DRAM cost "less of a concern" against the speedups.
Q9How big are the actual speedups, and where do they come from?▾
Up to 2.7× over state-of-the-art file systems. Micro-benchmarks: 2.5× over Ext4, 1.48× over F2FS on average; file create 6.0× (Ext4) / 2.4× (F2FS). Macro: OLTP 4.1× over Ext4, Varmail 1.9× over F2FS, Fileserver 2.2× over Ext4. Write traffic to the SSD drops up to 5.1×; metadata traffic up to 25.3× (Ext4) / 17.2× (F2FS); flash traffic 2.1–3.2× across baselines. Gains come from byte writes for small metadata, dynamic interface selection for data, and the firmware log's coalescing + cheap consistency.
Q10Does ByteFS ever lose, or cost more, than the baselines?▾
Yes, honestly reported. On read-only/range workloads (YCSB-C, YCSB-E) it's roughly tied — there's little write amplification to remove. On create and Fileserver it can incur higher flash read traffic than Ext4/F2FS, and on mkdir/rmdir higher flash write traffic than NOVA/PMFS, due to the read-modify-write pattern during log cleaning and its no-SSD-side-cache choice. It also spends ~16% extra page-cache DRAM for CoW. These are outweighed on average because cleaning is in the background.
Q11Is this real hardware, and how much code is it?▾
Both. A real prototype on an OpenSSD FPGA board (1 TB flash, 16 channels, 1 GB DRAM, on-board ARM running the firmware), plus an FEMU-based emulator for sensitivity studies. ByteFS is built on Ext4 in Linux 5.1: ~3.9 K LoC host file system (1.3 K to reorganize on-disk metadata, 0.6 K for CoW), plus ~1.5 K lines of C in the SSD firmware (0.8 K for the log + skip list, 0.4 K for cleaning + transactions, 0.1 K for the custom NVMe byte command). No SSD hardware changes.
Q12Why should a DRAM/NVM tiering (LtRAM) researcher care?▾
ByteFS is the storage-stack counterpart to memory tiering. Its CoW+XOR modified-ratio detector is a per-page, cacheline-grain hot/cold classifier living inside the page cache — directly reusable for promotion/demotion decisions. Its core thesis — a load/store-capable device's "cheap" interface isn't automatically cheap; the cost model of the backing medium (page-granular flash) must drive policy — is the same lesson tiering keeps learning about NVM and CXL memory. And as an M-SSD is itself a CXL memory device with byte access, it's a natural tier below DRAM/NVM in an LtRAM hierarchy.
Q13Where does the 1/8 (512 B) byte-vs-block threshold come from — derived, or a heuristic?▾
It's a tunable cost threshold, not a fundamental constant. It marks the crossover where the byte door stops paying off: below ~1/8 of a page dirty, logging just the changed 64 B cachelines (little amplification) beats rewriting the whole 4 KB page; above it, the byte path's per-cacheline latency plus the eventual read-modify-write to coalesce a sparsely-dirtied page back into flash make one block write cheaper. The specific 1/8 falls out of the prototype's measured costs (0.6 µs per cacheline byte-write vs a 4 KB block op) and empirical validation — a device with a different byte/block latency ratio, or a very different workload, would shift it. So: informed by a cost model, but chosen and workload-dependent, not a law.
Q14When metadata is "35% of write traffic," is that a count of writes or actual bytes?▾
Bytes written to the device (flash) — not app-requested bytes, and not a raw op-count (though here they coincide). Because a block FS writes everything in uniform 4 KB blocks, "35% of write ops are inode writes" and "35% of bytes written are inode blocks" are the same number. The point is that it's measured at the device: each 128 B inode update is inflated to a 4 KB block, so inodes consume 35% of flash write bandwidth/endurance despite being logically tiny. Count them as app-logical bytes instead and they'd look negligible — the amplification is the entire reason shrinking each write 4 KB → 64 B is the biggest lever.

Glossary, results & source notes

Every term in one place, the headline numbers, an honest provenance block, and where to read next.

Core terms
ByteFSThe paper's file system for memory-semantic SSDs: a host/firmware co-design that adaptively uses byte and block interfaces and reshapes SSD firmware to bridge the byte/flash granularity gap.
Memory-semantic SSD (M-SSD)A flash SSD that exposes its internal DRAM as a CPU-addressable memory region (byte interface) in addition to the normal NVMe block interface.
Byte interfaceCacheline-granular load/store access to the SSD via PCIe MMIO or CXL.mem; 0.6 µs write / 4.8 µs read per cacheline on the prototype.
Block interfaceConventional 4 KB NVMe access; used for reads (locality) and large/dense writes.
InodeThe file system's per-file record (~128 B): size, timestamps, permissions, owner, and pointers to the file's data blocks — everything about a file except its name and contents. Lives in a 4 KB on-disk inode block, so touching one field rewrites the whole block. Nearly every file/dir op updates one.
Dentry (directory entry)The name→inode link that builds the directory tree: maps a filename to an inode number (+ type, name length). Resolving a path like /a/b/c walks dentries; the kernel caches them (the dcache) so lookups don't hit storage.
Buffered I/OThe default read/write path: data goes through the page cache in host DRAM (a write dirties a cached page, flushed to the device later) instead of straight to the device (that's Direct I/O / O_DIRECT). ByteFS's hard case — at writeback the kernel knows a page is dirty but not which bytes changed.
Page cacheA slice of host DRAM the kernel uses to hold recently-used file pages, so repeat reads/writes hit RAM, not storage. Dirty pages are written back on fsync, by the flush daemon, or under memory pressure.
I/O amplificationBytes moved to/from the device ÷ bytes the app requested. Ext4/F2FS: 1.1–6.2× write, 1.1–1.7× read.
Log-structured write bufferByteFS's reorganization of SSD DRAM: a 256 MB circular log of 64 B entries that absorbs byte writes, coalesced into flash page writes by a background cleaner.
Three-layer skip listThe log index: partition table (16 MB partitions) → per-page skip list (by LPA) → ordered chunk list (by in-page offset). ~89 ns lookup, ~21 MB DRAM.
Modified ratio (r)Fraction of a 4 KB page's 64 B cachelines that are dirty, found by CoW + XOR. r < 1/8 (< 512 B) → byte interface; else block.
Supporting terms
CXL.memCompute Express Link's memory protocol: cacheable load/store to a device over the interconnect. A faster realization of the byte interface (175 ns modeled).
Copy-on-write (CoW)Snapshotting a page on first modification (tracked in an XArray by offset) so the writeback XOR can recover which bytes changed. ~16% page-cache overhead.
Coalescing / log cleaningThe background process (≥85% utilization) that merges nearby byte-log entries into sequential 4 KB flash writes and reclaims log space; double-buffered so it never blocks foreground I/O.
Coordinated cachingCaching flash pages only in the host page cache, never duplicated in SSD DRAM, freeing that DRAM for the write log.
Redo log / TxID / TxLogThe write log doubles as a redo log; a 4 B TxID tags each transaction; a 2 MB TxLog records commit order for consistency and recovery.
Write-verify readA zero-byte non-posted read after a byte write, exploiting PCIe root-complex ordering to force the posted write to complete (durability point).
FTLFlash Translation Layer: SSD firmware that maps logical to physical pages. ByteFS extends, not replaces, it (mapping table 512 MB, write buffer 16 MB on the prototype).
JBD2Ext4's journaling block device layer; ByteFS combines it with its own commit entries for large-write data journaling.
Inode splitDividing the 128 B inode into a hot lower 64 B (size, mtime, rights) and cold upper 64 B, so the common update is one atomic 64 B byte write.
NOVA / PMFSState-of-the-art NVM file systems used as baselines; byte-only, tuned for near-DRAM NVM, suboptimal on the microsecond/page-granular M-SSD.
