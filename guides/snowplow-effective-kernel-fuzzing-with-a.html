ASPLOS 2025 · A Field Guide

Teaching a fuzzer to predict the argument that unlocks new code

Kernel fuzzers mutate tests by changing system-call arguments — but they guess mostly at random. Snowplow trains a graph neural network on millions of real mutations to predict which argument, when changed, actually reaches the kernel code you're hunting.

📄 Snowplow: Effective Kernel Fuzzing with a Learned White-box Test Mutator · Gong, Wang, Altınbüken, Fonseca, Maniatis 🎓 For OS-focused researchers familiar with fuzzing & Linux kernel basics ⏱ ~20 min read
🧠

Train a model to predict which argument to mutate, given the test, its kernel coverage, and the target block you want to reach — not which value to assign (leave that to Syzkaller's instantiation), just where to mutate. Repeat for a dataset of 44.3M successful mutations; deploy at inference time as a localization oracle.

Background primers

Six concepts Snowplow builds on. Tap to expand each.

1Kernel fuzzing: tests are user-space programs▾

A kernel test is a small C program (or Syzkaller's Syzlang snippet) that invokes system calls. A fuzzer generates thousands of these, executes them, and measures kernel code coverage. The fuzzing corpus is the set of tests that have already discovered unique coverage.

The fuzzer picks a base test, mutates it (changes arguments, inserts/removes calls, etc.), executes the mutant, and if new coverage appears, adds it to the corpus. The mutation engine is the heart of the fuzzer.

Now you're ready: Snowplow learns to guide one mutation type — choosing which argument to mutate when Syzkaller decides to perform an argument mutation.
2Mutation types: type selection, localization, instantiation▾

For each mutation of a base test, the fuzzer decides: (1) type selection — insert a syscall, mutate an argument, remove a call, etc.; (2) localization — which syscall/argument; (3) instantiation — the concrete new value (e.g., flip a flag from O_CREAT to O_CREAT|O_RDWR).

Syzkaller uses heuristics + randomness for all three. For instance, argument localization picks a random argument from the syscall with the most arguments (a simple heuristic).

Now you're ready: Snowplow replaces the localization step (for argument mutations) with a learned predictor, PMM. Syzkaller still does type selection and instantiation.
3The search space is vast: nested arguments▾

A Linux syscall may take 3–6 direct arguments, but each can be a pointer to a struct with its own fields, which can be nested structs, buffers, etc. Syzkaller's Syzlang encodes all these sub-level parameters as "arguments" — so a single sendmsg() test can have ~60 mutable arguments.

On average, a kernel test contains >60 arguments available for mutation. Picking the right one is non-trivial — Syzkaller discovers only ~44 coverage-increasing mutants per 1000 argument mutations (§5.1 in the paper).

Now you're ready: PMM learns from a dataset of 44.3 million successful mutations to predict which of those ~60 arguments, when changed, leads to new coverage. The payoff: 5× faster coverage discovery.
4Successful mutations & the dataset▾

A successful mutation = one that covers at least one new kernel code block vs. the base test. To build a training dataset, Snowplow performs 1000 random argument mutations per base test (from a 1M-test corpus), executes each, collects kernel coverage, and records the argument and new coverage when new blocks appear.

Result: 44.3M successful mutations discovered on Linux 6.8, averaging ~45 per base test. Each is a training example: ⟨base test, base coverage, mutated argument(s), newly covered blocks⟩.

Now you're ready: Snowplow "inverts" these examples: given test + old coverage + desired target blocks, predict which argument to mutate. This is the localization oracle.
5Graph representation: bridging test & kernel coverage▾

To train an ML model, Snowplow needs a uniform input format for both the user-space test (a program tree) and kernel coverage (a CFG). It represents both as a single graph: nodes for syscalls, arguments, and kernel basic blocks; edges for control/data flow within the test, control flow in the kernel, and kernel-user transitions (syscall entry/exit).

This lets a Graph Neural Network propagate information across the user-kernel boundary and learn which arguments influence which kernel branches.

Now you're ready: PMM is a GNN trained on these graphs. It predicts a MUTATE/NOT-MUTATE label for each argument node, given the base test, its coverage, and target blocks.
6Directed fuzzing: reaching a specific target▾

Undirected fuzzing aims for maximum overall coverage. Directed fuzzing aims to reach a specific code location (e.g., a patched line to validate a fix, or a bug-related block). Directed fuzzers (like SyzDirect) use sophisticated heuristics to prioritize mutations that move toward the target.

But they still rely on semi-random argument selection once they decide to mutate arguments. Snowplow's learned localizer can accelerate directed fuzzing too, by predicting which argument brings the test closer to the target.

Now you're ready: Snowplow-D (Snowplow integrated into SyzDirect) reaches 21/24 target locations (vs. 19 for SyzDirect alone) and is 8.5× faster on the 19 shared targets (§5.4).

The whole story

Snowplow in a nutshell

The problem, the ML solution, how training and inference work, and what it achieves.

Kernel fuzzers like Syzkaller mutate tests by changing system-call arguments, inserting/removing calls, etc. Argument mutations are critical — they change the data and control flow inside the kernel — but the search space is enormous. A typical test has >60 mutable arguments (nested struct fields, buffer pointers, etc.), and Syzkaller uses simple heuristics (pick the call with most arguments) plus randomness to select one.

Result: on average, only 44 out of 1000 argument mutations discover new coverage (§5.1). The rest are wasted cycles.

Snowplow's core insight: we have the data to do better. By running Syzkaller's random mutations at scale (1000 per base test × 1M tests), we can collect millions of successful mutations and train a model to predict them. Snowplow trains PMM (Program Mutation Model), a Graph Neural Network that inputs the base test, its kernel coverage, and desired target blocks, and predicts which argument(s) to mutate.

Training. From the 44.3M successful mutations, Snowplow constructs a graph representation for each: syscall and argument nodes from the test, kernel basic-block nodes from coverage (both covered and one-branch-away alternatives), and edges for control/data flow + kernel-user transitions. The model learns to assign MUTATE/NOT-MUTATE labels to argument nodes. To handle uncertainty, Snowplow's targets include a noisy mix: all uncovered blocks one branch away from base coverage, plus a sample (1/25%/50%/75%/100%) of actual newly covered blocks (only those one branch away), ensuring overlap with true new coverage.

Inference. At fuzzing time, when Syzkaller decides to mutate arguments, Snowplow sends a mutation query (test + coverage + targets) to PMM, which predicts promising arguments. Syzkaller then mutates and executes them using its own instantiation logic. Inference runs asynchronously on a GPU server; the fuzzing throughput is ~383 tests/sec (vs. 390 for baseline Syzkaller — negligible overhead).

Results. On Linux 6.8 (trained), Snowplow achieves 7.0% higher coverage in 24 hours and is 5.2× faster to reach Syzkaller's 24-hour coverage. On newer kernels 6.9 and 6.10 (untrained), it's 4.8–5.0× faster with 7.7–8.6% more coverage. In a 7-day campaign, Snowplow found 86 unique crashes (57 reproducible), all new to Syzbot; Syzkaller found zero. For directed fuzzing, Snowplow-D reaches 21 of 24 targets (vs. 19 for SyzDirect) and is 8.5× faster on the 19 shared targets.

🍯 Key takeaways

  • Syzkaller's argument localization is semi-random; Snowplow learns it from a dataset of 44.3M successful mutations.
  • PMM is a GNN trained on a graph representation that unifies test programs (syscalls/arguments) and kernel coverage (CFG blocks + transitions).
  • Training dataset: 1M base tests × 1000 mutations each → 44.3M successful mutations (avg. 45 per test).
  • Graph inputs: syscall/argument nodes, kernel block nodes (covered + one-branch alternatives), control/data/transition edges.
  • PMM predicts MUTATE/NOT-MUTATE for argument nodes; Syzkaller instantiates the mutation.
  • 5.2× speedup on trained kernel, 4.8–5.0× on newer kernels; 7–8.6% higher 24-hour coverage.
  • 86 new crashes (57 reproducible) in 7 days; 4 confirmed bugs, 2 fixed (including a 20-year-old out-of-bounds write in the ATA driver).
  • Directed fuzzing: 8.5× faster to reach 19 targets, 2 additional targets vs. SyzDirect.
Seven everyday pictures

How to think about Snowplow

Each analogy stands alone. Every card ends with where the picture breaks.

The core problem · Random vs. learned localization

Searching a dark warehouse

Syzkaller flips a coin at each fork: "Mutate argument 3 or 17?" It discovers ~44 successes per 1000 tries — like searching a dark warehouse by throwing darts. Snowplow turns on the lights: it's seen 44 million successful paths and learned the map. Given a test and target, it predicts which argument unlocks the door.

Where it breaks: a warehouse has static geometry; kernel behavior shifts (phases). Snowplow trains on one kernel (6.8) but generalizes to 6.9 and 6.10 without retraining — suggesting the learned map is robust.
The dataset · 44M successful mutations

A lab notebook of experiments

Imagine a lab that tried 1 billion chemistry experiments, recording only the 44 million that produced a reaction. Snowplow's training dataset is exactly that: 1M base tests × 1000 mutations each, filtered to successful ones (new coverage). The notebook says "when test X had argument Y changed, kernel block Z lit up."

Where it breaks: a chemistry notebook is noise-free; kernel coverage can be non-deterministic (threads, interrupts). Snowplow mitigates this with VM snapshots and sequential execution, but some noise remains.
The graph representation · Unified view

A roadmap with borders

A test is a user-space program; coverage is kernel CFG. Snowplow glues them into one graph — syscall nodes, argument nodes, kernel block nodes — connected by "border crossings" (syscall entry/exit edges). The GNN traverses this map, learning that changing argument A propagates through syscall B to flip branch C in the kernel.

Where it breaks: a roadmap is 2D and static; control flow has cycles (loops) and dynamic state (memory, file descriptors). The graph is a projection that omits runtime values — Snowplow uses argument types, not literal constants.
Model task · Binary classification per argument

Labeling sticky notes

For each argument in the test, PMM predicts a label: MUTATE (this one leads to the target) or NOT-MUTATE (skip it). It's like sorting 60 sticky notes into two piles. Training examples show the ground truth (which arguments actually unlocked new coverage), and the GNN learns the patterns.

Where it breaks: sticky notes are independent; arguments interact (mutating arg 3 may only matter if arg 7 is set first). PMM learns some interactions via graph structure, but doesn't explicitly model argument dependencies.
Training targets · Noisy set

A fuzzy destination

At training time, Snowplow doesn't give PMM the exact newly covered blocks — it adds noise: all blocks one branch away from old coverage + a sample of the actual new blocks (only those one branch away). Why? At inference time, we don't know exactly what's reachable. The noise teaches robustness: "predict arguments that reach this general area, not one precise block."

Where it breaks: a fuzzy destination is still local (one branch away). The paper frames one-hop alternatives as enabling incremental coverage; directed-fuzzing results show PMM helps reach deeper handler code, suggesting value beyond immediate neighbors.
Hybrid fuzzer · Division of labor

A relay race

Snowplow doesn't replace Syzkaller — it partners with it. PMM picks the argument (localization); Syzkaller mutates the argument's value (instantiation) and executes the test. Meanwhile, Syzkaller also does other mutation types (insert/remove calls) unchanged. It's a relay: PMM runs the first leg, Syzkaller the rest.

Where it breaks: a relay has clean handoffs; PMM runs asynchronously (inference takes ~0.7 sec). Syzkaller doesn't wait — it tries other mutations while PMM thinks. The predicted arguments catch up later.
Generalization · Train once, fuzz many

A universal key

Snowplow trains PMM on Linux 6.8 (March 2024) but deploys it on 6.9 (May) and 6.10 (July) without retraining. It still achieves 4.8–5× speedup — like a key cut for one door that works on similar locks. Why? The learned patterns (e.g., "mutating the mode flag of open() often reaches filesystem branches") transfer across kernel versions.

Where it breaks: a universal key won't work on a radically different lock. If the kernel API changes (new syscalls, deprecated arguments), PMM may degrade. The paper doesn't test year-old kernels — generalization limits are unclear.
The full explanation

How it actually works

Step by step, with the paper's key numbers.

1 · Collecting the dataset

Snowplow starts with a seed corpus of 1M unique kernel tests, sampled from Syzbot's public artifacts. For each base test, it executes the test to collect baseline coverage (0.98M tests complete; some crash/hang and are excluded). Then it performs 1000 random argument mutations per base test, using Syzkaller's existing argument mutator to change values.

For each mutation, Snowplow executes the mutant, compares its coverage to the base, and if it covers new kernel blocks (i.e., blocks not in the base coverage), records a sample: ⟨base test, base coverage, mutated argument(s), newly covered blocks⟩. In total, 44.3M successful mutations are discovered — averaging ~45 per base test. The hit rate: 44 out of 1000 mutations succeed (4.4%).

🔧
Determinism hacks. To reduce noise, Snowplow: (1) loads a VM snapshot before each test (common initial state); (2) executes syscalls sequentially (no thread interleaving); (3) replaces network RPC with virtio (fewer interrupts). Result: cleaner coverage traces.

2 · Graph representation

For each sample, Snowplow builds a mutation query graph. The graph has three types of nodes: (1) syscall nodes (e.g., "open", "read"), (2) argument nodes (e.g., "flag", "ptr", nested struct fields), (3) kernel block nodes (covered blocks + alternative blocks one branch away). Edges encode: (1) argument ordering (within a syscall), (2) call ordering (syscall sequence), (3) argument in/out (data flow), (4) kernel control flow (CFG edges), (5) kernel-user transitions (syscall entry/exit).

On average, a graph contains 2372 vertices (5 syscall, 62 argument, 1631 covered kernel blocks, 674 alternatives) and 2989 edges. The large kernel subgraph is expected — a single test triggers hundreds of blocks. Alternatives are detected via static CFG analysis (Angr): for each covered block, find outgoing edges to uncovered blocks.

💡
Why graph, not text? A test is a program (parse to AST); coverage is a CFG. Separate representations make learning hard. The graph unifies them, letting the GNN propagate information across user-kernel boundaries via the transition edges.

3 · Training PMM

PMM is a Graph Neural Network with three learned components: (1) θ_GNN (the GNN backbone, e.g., GCN), (2) θ_TRANSFORMER (a BERT-style encoder for kernel basic-block assembly, pre-trained on all X86 assembly in the kernel), (3) θ_Emb (embedding matrices for syscall names, argument types, edge types). The model predicts a binary label (MUTATE / NOT-MUTATE) for each argument node.

Training examples are inverted from the forward dataset: given ⟨test, coverage, newly covered blocks⟩, Snowplow constructs a target set ˆc by sampling from: (a) all uncovered blocks one branch away from the base coverage, (b) the actual newly covered blocks, but only those one branch away. Then it samples 1, 25%, 50%, 75%, or 100% of this noisy set, ensuring overlap with the actual new blocks. This adds robustness — the model learns to predict arguments that reach "this area" rather than exact blocks.

Hyperparameter search evaluates 112 configurations (each trained up to 72 hours on 8×A100) using mean F1 score on a validation set (10K base tests). Loss function: binary cross-entropy on argument labels. Best model achieves 84.2% F1 on the evaluation set (10K unseen tests), vs. 30.3% for a random-8-argument baseline.

🍯
Training cost. 3 days on 8×A100 GPUs. Amortizes over weeks of continuous fuzzing — kernel development runs Syzbot 24/7, so training once per kernel version is a drop in the ocean.

4 · Inference at fuzz time

Snowplow integrates into Syzkaller's mutation loop. When Syzkaller picks a base test to mutate, Snowplow asynchronously sends a mutation query (test + coverage + targets) to a GPU server running PMM via gRPC. The model predicts promising arguments and returns them to Syzkaller, which instantiates and executes the mutations.

Key design choices: (1) Asynchronous inference — Syzkaller doesn't block; it tries other mutation types (insert/remove calls) while waiting. (2) Dynamic mutation budget — if PMM predicts many interesting arguments, Snowplow mutates more; if few, fewer. (3) Fallback to random — with low probability, Syzkaller still does random argument mutations (safety net if PMM misses something).

At saturation, PMM serves 57 queries/sec with 0.69-sec average latency (on 8×L4 GPUs). Fuzzing throughput: 383 tests/sec (Snowplow) vs. 390 (Syzkaller) — negligible overhead.

5 · Evaluation: undirected fuzzing

Snowplow and Syzkaller fuzz three kernels (6.8, 6.9, 6.10) for 24 hours each, repeated 5 times. Metric: edge coverage (unique directed pairs of basic blocks). Setup: 42 VMs per fuzzer (2 vCPUs each), same initial seed per run.

Linux 6.8 (trained): Snowplow achieves 7.0% higher coverage at 24h and reaches Syzkaller's 24h coverage in 4.6 hours (5.2× speedup). Syzkaller takes the full 24 hours to reach what Snowplow achieves in under 5 hours.

Linux 6.9 and 6.10 (untrained): Snowplow generalizes without retraining. Coverage improvement: 8.6% (6.9), 7.7% (6.10). Speedup: >4.8× on both. Coverage bands don't overlap, showing consistent superiority.

🍯
Snowplow is faster and more consistent. Syzkaller's coverage varies widely across runs (wide bands); Snowplow's is tight (narrow bands). This is expected — Snowplow's mutations are guided by a learned model, not semi-random heuristics.

6 · Evaluation: finding bugs

In a 7-day fuzzing campaign (repeated 2×), Snowplow discovers 86 unique new crashes (67 in run 1, 46 in run 2, with some overlap). Syzkaller finds zero new crashes — expected, since Syzbot has been fuzzing these kernels continuously. Snowplow's crashes are "new" = not in Syzbot's crash database (all bugs since 2018). Of the 86, 57 are reproducible by Syzkaller's syz-repro tool (66% reproducibility vs. 32% for Syzbot overall). Non-reproducible crashes often involve concurrency.

Crash categories (Table 3): null-pointer dereference (10), paging fault (23), general protection fault (39), out-of-bounds access (1), assertion violations (4), warnings (8), other (2). Most are serious — crashes that corrupt kernel state or exploit security issues.

Case study: ATA driver out-of-bounds write. Snowplow found a 20-year-old bug (fixed in 2025) where the ATA driver writes arbitrary data to kernel memory. Trigger: ioctl(fd, SCSI_IOCTL_SEND_COMMAND, ...) with carefully crafted arguments: (1) SCSI device FD, (2) cmd = SCSI_IOCTL_SEND_COMMAND, (3) SCSI command = ATA_16 PASS-THROUGH, (4) ATA command = ATA_NOP, (5) protocol = ATA_PROT_PIO, (6) incorrect data length. Syzkaller rarely finds this because the argument constraints are strict; PMM systematically identifies the critical arguments and keeps mutating them.

📊
Of 57 reproducible crashes, 45 involve ioctl() with SCSI_IOCTL_SEND_COMMAND — likely stemming from the same ATA bug. The remaining 12 (including 6 already diagnosed) are distinct issues. Snowplow reported 7 crashes to kernel developers; 4 confirmed, 2 fixed.

7 · Evaluation: directed fuzzing

Snowplow-D integrates PMM into SyzDirect (state-of-the-art directed fuzzer). Evaluation: reach 24 bug-related kernel code locations (from SyzDirect's dataset). Important context: the SyzDirect dataset originally had 100 target locations, but only 24 were runnable because patches failed on 76. Metric: time to first coverage of the target. Both fuzzers run 24h (repeated 5×).

Results: SyzDirect reaches 19 targets; Snowplow-D reaches 21 (2 additional). On the 19 shared targets (covering both easy and hard targets), Snowplow-D is 8.5× faster on average. Easy targets (entry points of syscalls) show no speedup — PMM doesn't help when the target is immediate. Hard targets (deep in syscall handlers) show large speedups: e.g., net/rxrpc/sendmsg.c:747 = INF (SyzDirect never reaches it), drivers/scsi/scsi_ioctl.c:357 = 242× faster.

💡
Why the split? Easy targets don't need argument mutations — just invoke the right syscall. Hard targets need precise arguments to navigate deep branches. PMM shines on the latter.

Worked examples

Four traces using the paper's exact measurements.

① Building a training example from a successful mutation

Base test: open("./file0", O_CREAT, 0777); read(r0, buffer, 42). Mutate the flag argument to O_CREAT|O_RDWR, execute, and the mutant covers a new block in fs/ext4/file.c. How does this become a training example?

  1. Base coverage 𝑐ᵢ: the kernel blocks triggered by the base test (green nodes in the graph).
  2. Mutant coverage 𝑐ᵢⱼ: execute the mutant; collect new blocks 𝑐ᵢⱼ \ 𝑐ᵢ (the ext4 block).
  3. Target set ˆ𝑐ᵢⱼ: construct a noisy target by: (a) adding all uncovered blocks one branch away from 𝑐ᵢ, (b) sampling from the actual new blocks (only those one branch away). Sample 50% → ˆ𝑐ᵢⱼ includes the ext4 block + some distractors.
  4. Localization 𝑎ᵢⱼ: the flag argument of open() (the one mutated).
  5. Training example: ⟨sᵢ (base test), 𝑐ᵢ (base coverage), 𝑎ᵢⱼ (flag arg), ˆ𝑐ᵢⱼ (noisy targets)⟩.
🍯
Key insight: the target set includes distractors (nearby uncovered blocks) to teach robustness. At inference time, we don't know exactly which blocks are reachable — the model learns to predict arguments that explore the general area.

② Why random localization fails

A test has 60 arguments. Syzkaller picks 8 at random. PMM predicts 8 based on the graph. Compare their F1 scores (precision/recall on the true set of promising arguments).

  1. Random baseline (Rand.8): selects 8 arguments uniformly. F1 = 30.3%, Jaccard = 19.9%. Precision = 36.6%, Recall = 37.0%. The low scores show that random selection rarely overlaps with the true promising set.
  2. PMM: predicts 8 arguments using the graph. F1 = 84.2%, Jaccard = 76.1%. Precision = 91.2%, Recall = 81.2%. Nearly 3× better than random.
  3. Why the gap? Random doesn't see the kernel coverage or targets — it's blind. PMM propagates information from targets → kernel blocks → syscall entry edges → argument nodes, learning which arguments influence which branches.
💡
The 60-argument search space is the crux. Syzkaller's heuristic (pick the call with most args) is slightly better than uniform random, but still semi-random. PMM shrinks the effective search space by 3× (F1 improvement).

③ Coverage speedup calculation

On Linux 6.8, Syzkaller reaches 125K edge coverage after 24 hours. Snowplow reaches the same 125K in 4.6 hours. Compute the speedup.

  1. Baseline time: 24 hours = 1440 minutes.
  2. Snowplow time: 4.6 hours = 276 minutes.
  3. Speedup: 1440 / 276 = 5.2×.
  4. Coverage improvement: at 24h, Snowplow reaches ~134K edges (from Figure 6a) vs. Syzkaller's 125K. Improvement = (134K - 125K) / 125K = 7.0%.
  5. Consistency: Snowplow's 5 runs have a tight band (min 132K, max 136K); Syzkaller's is wide (min 120K, max 128K). No overlap after 5 hours.
🍯
The speedup metric answers: "How much faster does Snowplow reach the same coverage?" The improvement metric answers: "How much more coverage does Snowplow get in the same time?" Both matter — one for resource efficiency, one for thoroughness.

④ Directed fuzzing: time to target

SyzDirect tries to reach drivers/scsi/scsi_ioctl.c:357 and takes 24,487 seconds (average of 1/5 runs; 4/5 fail). Snowplow-D takes 101 seconds (average of 1/5 runs). Compute the speedup and explain the discrepancy.

  1. SyzDirect: 24,487 sec = ~6.8 hours. Success rate: 1/5 (20%).
  2. Snowplow-D: 101 sec = ~1.7 minutes. Success rate: 1/5 (20%).
  3. Speedup: 24,487 / 101 = 242.4×.
  4. Why so large? The target (drivers/scsi/scsi_ioctl.c:357) is deep in the ioctl handler, requiring precise arguments (SCSI_IOCTL_SEND_COMMAND + correct SCSI/ATA commands). SyzDirect's heuristics bias toward ioctl() but don't localize the critical arguments. PMM does.
  5. Why same success rate? The target is still hard — only 1/5 runs reach it even with PMM. But when reached, it's 242× faster. This is the nature of directed fuzzing: success isn't guaranteed, but PMM accelerates the search.
🔍
The paper reports "INF" speedup for targets SyzDirect never reaches (0/5) but Snowplow-D does (1/5). Technically, divide-by-zero is undefined, but the intent is clear: PMM enables reaching targets SyzDirect can't.

Practice problems

Try before revealing the answer. Each ends with the misconception it checks.

Problem 1 · easy
Snowplow trains PMM on Linux 6.8 and deploys it on 6.9 and 6.10 without retraining. Does this violate the ML principle "test on unseen data"? Explain.
Show answer
No. The training set is drawn from 1M tests on 6.8; the evaluation tests PMM's predictions on new tests on 6.9/6.10. The "data" is the test-coverage-target tuples, not the kernel version. PMM learns patterns (e.g., "mutating open's mode flag often reaches fs branches") that generalize across kernel versions, as long as the API doesn't radically change. The 6.9/6.10 results (4.8–5× speedup) validate this generalization.Misconception: that kernel version is the "test set." The test set is unseen base tests + unseen coverage patterns, even on the same kernel. Generalizing to new kernels is a bonus.
Problem 2 · medium
PMM achieves 84.2% F1 on argument localization. Why doesn't Snowplow achieve 5× speedup on all targets in directed fuzzing? (Hint: consider the easy vs. hard target split.)
Show answer
Easy targets (syscall entry points) don't need argument mutations — just invoke the right call. PMM's localization is irrelevant there, so no speedup. Hard targets (deep in handlers) require precise arguments; PMM shines, yielding 8.5–242× speedup. The overall average (8.5×) is a mix. Also, 84% F1 means PMM sometimes mispredicts (false positives/negatives), so even on hard targets, not every mutation succeeds.Misconception: that F1 score directly translates to speedup. Speedup depends on both localization accuracy and whether localization is the bottleneck (it's not for easy targets).
Problem 3 · hard
Snowplow adds "noisy" targets (uncovered blocks one branch away, plus distractors) to training examples instead of just the exact newly covered blocks. Construct a scenario where exact targets would fail at inference time but noisy targets succeed.
Show answer
Scenario: a test covers block A, and mutating arg X unlocks blocks B and C (both one branch away). Training example with exact targets ˆc = {B, C} teaches "arg X → B and C." At inference time, the fuzzer asks "reach block D" (also one branch from A, but uncovered by this mutation). If trained on exact targets, PMM never saw D and may ignore arg X. With noisy targets ˆc = {B, C, D, ...}, PMM learns "arg X explores this region," so it still predicts X for target D. The noise adds robustness to uncertainty about exactly what's reachable.Misconception: that exact supervision is always better. In fuzzing, the target is unknown (we don't know what's reachable until we try), so noisy supervision teaches "explore this area" rather than "reach this exact block."
Problem 4 · medium
Snowplow's inference takes 0.69 sec per query (57 queries/sec at saturation). Syzkaller generates ~390 tests/sec. Why doesn't inference latency bottleneck the fuzzer?
Show answer
Asynchronous design. When Syzkaller picks a base test, Snowplow sends a mutation query to PMM but doesn't block — Syzkaller continues with other mutation types (insert/remove calls). Once PMM returns, Syzkaller catches up with the predicted arguments. Also, not every test triggers a PMM query — only when Syzkaller's type selector chooses argument mutation. The fuzzing throughput (383 tests/sec) is nearly the same as baseline (390), showing the overhead is absorbed.Misconception: that inference latency directly slows fuzzing. The fuzzer is multi-threaded and doesn't wait for PMM — it overlaps inference with other work.
Problem 5 · easy
Snowplow's graph has "kernel-user transition" edges connecting syscall nodes to kernel entry/exit blocks. Why are these edges critical for PMM to work?
Show answer
Without them, the test graph (syscalls/arguments) and kernel graph (CFG blocks) are disconnected. The GNN can't propagate information from target blocks → kernel entry → syscall → arguments. The transition edges explicitly model the context switch, enabling the GNN to learn "this syscall enters here, so arguments of this call influence these branches." Remove them, and PMM is blind to the kernel-user boundary — it degenerates to a model that ignores coverage.Misconception: that the GNN can infer the boundary from node types. Graph neural networks propagate information along edges, not by "guessing" connections. Explicit edges are required.
Problem 6 · hard
Snowplow discovers 86 new crashes, but 45 of them likely stem from the same ATA driver bug. Are those 45 redundant? Should they count as 1 bug?
Show answer
Depends on the goal. For bug counting (e.g., "how many distinct root causes?"), yes, count as 1. But for fuzzing evaluation, they're non-redundant: each crash has a unique trigger (different ioctl arguments, different memory corruption patterns). Discovering 45 paths to the same bug shows Snowplow's thoroughness in exploring the ATA code region. Also, diagnosing them (to confirm they're the same bug) required manual effort — the reproducers don't obviously cluster. The paper is transparent: it reports 86 total, notes 45 are likely related, and separately reports 7 diagnosed + 4 confirmed + 2 fixed.Misconception: that "crash" = "bug." A crash is a symptom; a bug is the root cause. Multiple crashes can share a root cause, but each crash is still a distinct test failure worth investigating.
Problem 7 · medium
PMM uses a BERT-style Transformer to embed kernel basic-block assembly. Why not just use the block address (e.g., hash the PC) as a unique ID?
Show answer
Block addresses are arbitrary and uninformative. Two blocks with similar assembly (e.g., both call kmalloc()) should have similar embeddings — this helps the GNN generalize. A hash treats them as unrelated. Also, kernel addresses change across builds/versions; embeddings learned on 6.8 can transfer to 6.9 because the assembly patterns (e.g., "mov rax, [rdi]") are stable. The Transformer pre-training on all kernel assembly learns these patterns, so the GNN gets rich semantic features, not just IDs.Misconception: that unique IDs are sufficient for node embeddings. Embeddings should encode similarity — blocks that behave similarly should be close in embedding space.
Problem 8 · easy
Snowplow only learns argument localization, not instantiation (the concrete value). Why doesn't it learn both?
Show answer
Simplicity and division of labor. Localization (which arg?) is a classification problem with ~60 options per test — tractable for a GNN. Instantiation (which value?) is a combinatorial explosion: a flag can be any bitwise OR of enum values, a pointer any offset in a buffer, an integer any 32/64-bit value. Learning this would require a generative model (e.g., VAE or diffusion model) and is much harder. Instead, Snowplow leaves instantiation to Syzkaller, which has hand-crafted heuristics (e.g., flip one bit, replace with a magic constant, shift a pointer by N bytes). The hybrid design is pragmatic: learn what's learnable, delegate the rest.Misconception: that ML should replace all heuristics. Snowplow is a hybrid fuzzer — it uses ML where it helps (localization) and heuristics where they're good enough (instantiation).
Quick answers

Frequently asked questions

Twelve questions newcomers ask most. Tap to expand.

Q1What is a "mutation query" exactly?▾
It's the input to PMM at inference time: a graph containing (1) the base test (syscalls + arguments), (2) its kernel coverage (executed blocks), (3) target blocks (uncovered blocks we want to reach). PMM predicts which argument nodes to mark MUTATE. The fuzzer then mutates those arguments and executes the test.
Q2Why does Snowplow only handle argument mutations, not other types (e.g., syscall insertion)?▾
Focused scope for the paper. The authors demonstrate the methodology on argument localization because (1) it's a high-impact mutation type (contributes significantly to coverage), (2) the localization problem is well-defined (which of ~60 args?), (3) the dataset is natural (44M successful arg mutations). The approach generalizes — syscall insertion localization needs no representation/training changes, while insertion instantiation would predict one of the thousand or so system-call mutants. Future work.
Q3How does Snowplow handle arguments with literal values (e.g., filename strings, buffer contents)?▾
It doesn't embed them. Argument nodes contain the argument type (e.g., "string", "integer", "buffer pointer") but not literal values. This is intentional: literal values are infinite and non-generalizable. PMM learns "mutating a string argument here often helps," not "mutating '\/tmp\/foo' to '\/tmp\/bar' helps." Instantiation (choosing the literal) is left to Syzkaller's heuristics.
Q4What is the "one branch away" restriction for alternative blocks, and why impose it?▾
At training time, Snowplow adds uncovered blocks that are reachable in a single control-flow jump from the base coverage (e.g., the "else" branch of an "if" that the base test took the "then" branch of). Why? (1) Locality: mutating arguments typically affects nearby branches, not distant ones. (2) Noise control: including all uncovered blocks would add too many distractors. (3) Incrementalism: fuzzing progresses stepwise — cover block A, then blocks one hop from A, etc.
Q5How does Snowplow ensure deterministic coverage during dataset collection?▾
Three techniques: (1) VM snapshots — load a common initial state before each test; (2) sequential execution — disable Syzkaller's thread pool, execute syscalls one at a time (avoids interleaving non-determinism); (3) virtio for host-guest communication — replace network RPC (which triggers guest kernel network code non-deterministically). These reduce but don't eliminate non-determinism (e.g., hardware interrupts remain). The paper accepts residual noise, relying on the large dataset (44M examples) to average it out.
Q6What is the training cost in dollar terms?▾
External estimate (GCP pricing, 2024, time-sensitive): Training: 3 days on 8×A100 (40GB) GPUs. ~$3.67/hr per A100 → $3.67 × 8 × 72 hours = ~$2,100. Inference: 24h fuzzing on 8×L4 GPUs → ~$0.86/hr × 8 × 24 = ~$165/day. Amortization: if fuzzing continuously for 30 days, inference cost = $4,950; training is 42% of that. But the trained model can fuzz multiple kernel versions (6.8, 6.9, 6.10), so the cost amortizes further. (Note: these are rough external calculations using public cloud pricing; the paper does not provide cost estimates.)
Q7Why does PMM outperform random-8 by 3× in F1 but Snowplow only achieves 5× speedup in coverage?▾
Speedup depends on more than localization accuracy. (1) Instantiation: even if PMM picks the right argument, Syzkaller must mutate it to a useful value — if instantiation fails, no new coverage. (2) Execution time: running the test and collecting coverage takes time, diluting the speedup. (3) Other mutation types: Syzkaller also inserts/removes calls, which PMM doesn't touch. (4) Easy targets: some coverage is trivial to reach (e.g., syscall entry points) — PMM doesn't help there. The 5× speedup is a system-level metric aggregating all these factors.
Q8Can PMM be retrained on the fly as new successful mutations are discovered during fuzzing?▾
The paper doesn't do this (training is upfront), but online learning is feasible in principle. Challenges: (1) computational cost — retraining takes hours; (2) data imbalance — early fuzzing finds easy mutations, later fuzzing finds hard ones (distribution shift); (3) catastrophic forgetting — online updates may degrade performance on earlier patterns. A more practical approach: periodically (e.g., weekly) retrain on accumulated data from multiple fuzzing campaigns.
Q9What happens if Snowplow encounters a syscall variant not in the training data?▾
The syscall name is embedded via a learned matrix (θ_Emb). If the syscall is unseen, its embedding is random (untrained). However, the GNN also sees the syscall's kernel entry block (via the transition edge), so it can infer something from the kernel-side assembly. Also, many syscalls share structure (e.g., all filesystem syscalls enter similar VFS code), so the model may generalize. Performance likely degrades on truly novel syscalls, but the paper doesn't test this — an open question.
Q10Why does Snowplow's coverage band narrow while Syzkaller's widens (Figure 6)?▾
Randomness vs. learning. Syzkaller's semi-random mutations are sensitive to the initial seed — lucky seeds explore productive regions, unlucky ones get stuck. Snowplow's mutations are guided by PMM, which is deterministic given the same input graph (the fuzzer's state). So different seeds still converge to similar coverage. The narrow band shows consistency, not just higher average — a desirable property for CI/CD fuzzing where reliability matters.
Q11Can Snowplow be used for userspace fuzzing (e.g., AFL on binaries)?▾
Yes, conceptually. The approach is general: (1) collect a dataset of successful mutations (which input bytes → new coverage), (2) build a graph representation (program CFG + input structure), (3) train a model to predict which input bytes to mutate. The kernel-specific parts are (a) the syscall/argument representation (userspace would use input bytes or protocol fields) and (b) the kernel-user transition edges (userspace doesn't have this boundary). The GNN architecture is agnostic. Adapting Snowplow to AFL is plausible future work.
Q12What's the biggest limitation of Snowplow?▾
Training cost and one-time nature. Snowplow trains on 1M tests × 1000 mutations = 1 billion test executions (4 days on 120×60-vCPU VMs). This upfront cost is amortized only if you fuzz continuously for weeks. For one-off fuzzing campaigns (e.g., fuzzing a new driver for a day), the ROI is poor. Also, the model doesn't adapt during fuzzing — it's frozen at training time. If the kernel changes dramatically (e.g., a major subsystem rewrite), retraining is required. The paper doesn't quantify how often this is needed.

Context: related fuzzers & techniques

Syzkaller: the baseline easy

Google · open-source kernel fuzzer

State-of-the-art coverage-guided fuzzer for Linux/BSD/etc. Uses heuristics + randomness for mutations. Syzbot runs Syzkaller continuously, finding >6700 bugs. Snowplow builds on it, replacing argument localization with PMM.

HFL: Hybrid Fuzzing on Linux hard

Kim et al., NDSS 2020

Uses symbolic execution (S2E) to guide argument mutations, but only as a fallback when a branch is repeatedly missed. Expensive — symbolic engine runs infrequently. Snowplow uses ML as the first-choice mutator, not a fallback.

SyzDirect: directed greybox fuzzing medium

Tan et al., CCS 2023

State-of-the-art directed kernel fuzzer. Uses static analysis to infer resource dependencies (e.g., "call open() before read()") and prioritizes mutations that move toward the target. Snowplow-D integrates PMM into SyzDirect, achieving 8.5× speedup on hard targets.

Moonshine & Healer: implicit dependencies medium

Pailoor et al., USENIX Security 2018; Sun et al., SOSP 2021

Infer implicit control dependencies between syscalls (e.g., "ioctl() on an FD opened by socket()") and use them to guide syscall insertion. Orthogonal to Snowplow — they optimize "which call to insert," Snowplow optimizes "which argument to mutate."

Snowcat: kernel concurrency testing hard

Gong et al., SOSP 2023 (same first author as Snowplow)

Uses ML to predict kernel coverage of multi-threaded tests and select promising interleavings. Complementary to Snowplow: Snowcat optimizes test selection in concurrency testing; Snowplow optimizes mutation in coverage-directed fuzzing.

ML for systems: broader context

SmartChoices: ML in programming easy

Carbune et al., ICML RL4RealLife 2019

A framework for replacing heuristics in programs with ML models. Snowplow is an instance of this philosophy: replace Syzkaller's argument-selection heuristic with a learned model.

NEUZZ: neural program smoothing hard

She et al., IEEE S&P 2019

Uses a neural network to predict gradients in the input space (for gradient-based mutation in fuzzing). Different approach than Snowplow (gradient-based vs. classification-based), but shares the goal of ML-guided fuzzing.
