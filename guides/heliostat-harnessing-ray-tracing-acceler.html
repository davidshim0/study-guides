Borrowing idle ray tracers to translate addresses faster

GPUs struggle with virtual memory: the GMMU can't keep up. Instead of overprovisioning PTWs, Heliostat repurposes underutilized ray tracing hardware to accelerate page table walks — achieving 1.93× speedup with 1.53% area, 5.8% power.

Before you begin

Ten ideas the paper assumes you know

Each collapsed primer covers one concept. Read in order the first time, or jump to whatever looks unfamiliar.

ray tracing concepts page table walks RTA hardware GMMU baseline
1GPU virtual memory and the GMMU▾

Like CPUs, GPUs use virtual memory: each program sees its own address space, and a GPU Memory Management Unit (GMMU) translates virtual addresses to physical. The GMMU contains page table walkers (PTWs) — typically 16 of them — that traverse multi-level page tables on TLB misses. Each PTW can handle one translation at a time.

The problem: with hundreds of compute units running thousands of threads, the GMMU's 16 PTWs are a severe bottleneck. Some workloads spend more time queuing for a PTW than actually doing the walk.

Now you're ready: Heliostat's goal is to expand PTW bandwidth without overprovisioning the GMMU — by borrowing idle hardware that can also traverse trees.
2TLB hierarchy and miss penalties▾

A GPU has private L1 TLBs per compute unit and a shared L2 TLB. On an L2 TLB miss, the translation request goes to the GMMU. Miss rates vary wildly: some apps have 1%, others over 50%. The miss penalty has four parts: porting (sending the request), queuing (waiting for a free PTW), page table traversal (memory latencies), and page walk cache lookups.

The paper's key finding: in most applications, queuing delay dominates. The GMMU is simply starved for PTW bandwidth.

Now you're ready: Heliostat adds PTW capacity by making RTAs handle queued translations instead of waiting.
3Multi-level page tables and tree traversal▾

Page tables are organized as a radix tree with four levels (PML4 → PDP → PD → PT). Each level's index comes from a slice of the virtual address. A PTW starts from the page table base register (PTBR), fetches the first-level entry, extracts the next-level base address, continues down four levels, and retrieves the physical frame number (PFN).

This is a classic depth-first tree traversal — the same operation that ray tracing hardware is built to accelerate.

Now you're ready: Heliostat exploits operational similarity: page table walks and BVH traversal both involve depth-first tree searches, though the data structures and termination conditions differ.
4Ray tracing and BVH trees▾

Ray tracing simulates lighting by tracing rays through a 3D scene and testing intersections with geometry. To avoid testing every triangle, scenes are organized into a Bounding Volume Hierarchy (BVH): a tree where each internal node is an axis-aligned bounding box, and leaf nodes contain actual primitives. A ray traverses the BVH via depth-first search, skipping subtrees the ray doesn't intersect.

Modern GPUs have dedicated Ray Tracing Accelerators (RTAs) — fixed-function units that traverse BVH trees in hardware, far faster than general-purpose cores.

Now you're ready: RTAs are specialized tree-traversal engines. Heliostat repurposes them for page table trees.
5RTA architecture: warps, buffers, and operation units▾

An RTA has: a Warp Buffer (holds 4 concurrent warps), a Ray Buffer (stores ray properties per warp), a Mem Access FIFO (dedicated memory unit for fetching BVH nodes), a Node Decoder, and Operation Units (ray-box intersection, ray-triangle intersection, etc.). The RTA scheduler picks a warp, fetches the next BVH node, decodes it, and dispatches to the appropriate operation unit.

Crucially, the RTA can issue memory transactions independently of the compute units' regular load/store path.

Now you're ready: Heliostat adds one more operation unit — the PTE Decoding Unit (PDU) — to handle page table entries instead of BVH nodes.
6RTA underutilization▾

RTAs are only used for ray tracing kernels. Most GPU workloads — deep learning, scientific computing, graph analytics — never invoke ray tracing APIs. For those applications, the RTAs sit completely idle throughout execution, consuming only static leakage power.

The paper targets exactly this scenario: when a general-purpose (non-RT) application runs, the RTAs are guaranteed free.

Now you're ready: Heliostat democratizes RTAs for virtual memory operations, turning idle hardware into PTW bandwidth.
7Page Walk Cache▾

Because upper-level page table entries (PML4, PDP) are shared by many translations, GMMUs use a small Page Walk Cache (128 entries shared by 16 PTWs in the baseline) to avoid re-fetching them. This cache is critical for reducing memory traffic during walks.

Heliostat needs an equivalent for RTA-based translations — but sharing the compute units' L1 data cache causes pollution, and RTAs have no built-in shared cache.

Now you're ready: Heliostat reserves space in the underutilized L1 Scalar cache as an RT-PTW cache, providing both isolation and cross-RTA sharing.
8L1 Scalar Cache (L1S)▾

Each GPU Shader Array has an L1 Scalar (L1S) cache shared by all compute units in that array. It stores read-only data like kernel arguments and constant-qualified variables. Unlike the L1 Vector (data) cache, which is per-CU and heavily used, the L1S is typically underutilized (1–19% in tested workloads).

The key insight: L1S is shared across CUs in the array and has spare capacity — perfect for caching page tables used by multiple RTAs.

Now you're ready: Heliostat reserves one way per set in L1S for RTA page table walks, avoiding L1V pollution and enabling translation locality.
9Page size, huge pages, and the PS bit▾

Most OSes default to 4KB pages (4-level traversal). But GPUs also support huge pages (64KB, 2MB), which reduce TLB misses by covering more address space per entry. With huge pages, the page table tree is shallower — the walk stops earlier. To detect this, PTEs have a PS (page size) bit: if set, the current level is the last.

Heliostat's PDU checks the PS bit at each level (or, for 4KB-only systems, simply counts to depth 4).

Now you're ready: Heliostat supports any page size by checking PS — no hard-coded assumptions about tree depth.
10Lookahead translation (Heliostat+)▾

Beyond on-demand translations, ray tracing has a secondary ray feature: when a primary ray hits a reflective surface, a secondary ray is spawned to trace reflections/refractions. The secondary ray inherits traversal state from the primary ray, sharing the upper BVH levels until their paths diverge.

Heliostat+ adds lightweight hardware support to enable lookahead translation by leveraging this secondary-ray mechanism: predict the next address (via stride), spawn a secondary translation thread that shares upper page table levels with the on-demand translation, and cache the result in L1S for future use.

Now you're ready: Heliostat+ adds proactive translation prefetching, saving memory bandwidth and latency.

The whole story in one place

Heliostat in a nutshell

The problem, the cheap fix, and how it all connects.

GPUs have a virtual memory bottleneck: the GMMU with 16 PTWs can't keep up with hundreds of compute units. Overprovisioning PTWs costs 7.8× area and 6.8× power. But most of the queuing delay occurs when Ray Tracing Accelerators (RTAs) sit idle — because RTAs are only used for ray tracing kernels.

The paper's fix, Heliostat, repurposes RTAs to handle page table walks. The insight: both ray tracing and page table walk involve depth-first tree traversals, though BVH trees and page table radix trees differ in structure and encoding. To support PTW operations, Heliostat adds a lightweight PTE Decoding Unit (PDU) to the RTA's operation units and designs an RT-PTW Forwarding Unit (RFU) to route translations to idle RTAs when GMMU PTWs are busy. To avoid cache pollution and leverage translation locality, Heliostat reserves space in the underutilized L1 Scalar cache as an exclusive RT-PTW cache.

Heliostat+ adds lightweight hardware support to enable lookahead translation: predict the next address via stride, spawn a secondary translation thread that shares upper page table levels with the on-demand translation, and cache the result. Future on-demand translations can hit in the lookahead buffer and skip the walk entirely.

The payoff: Heliostat achieves 1.93× speedup over baseline and 1.92× / 1.66× over two state-of-the-art TLB optimizations (Valkyrie, BarreChord). Heliostat+ further improves to 2.39× over baseline. Compared to a 128-PTW GMMU (which achieves similar speedup), Heliostat uses only 1.53% area and 5.8% power.

🔮 Key takeaways

  • The bottleneck is PTW bandwidth (queuing delay), not individual walk latency — adding more PTWs helps, but costs too much.
  • RTAs are underutilized tree-traversal engines — they sit idle for general-purpose workloads but can handle page table walks.
  • Operational similarity: both BVH traversal and page table walks use depth-first tree searches, though structures differ.
  • Heliostat adds a PTE Decoding Unit (PDU) to the RTA pipeline and an RFU to route translations based on PTW availability.
  • L1 Scalar cache is underutilized (1–19%) — reserving one way per set provides an isolated, shared RT-PTW cache.
  • Heliostat+ adds lightweight hardware to enable lookahead translation — sharing upper page table levels and caching results.
  • Result: up to 2.39× speedup (Heliostat+), 1.53% area, 5.8% power vs. 128-PTW GMMU.
Seven everyday pictures

How to think about Heliostat

Each picture stands alone — read them in any order. Every card ends with an honest note about where the picture stops being true.

The core idea · Underutilized hardware

Borrowing a neighbor's garage

You have one tiny garage (16 PTWs) and 128 cars (CUs). Your neighbor has 128 garages (RTAs) they rarely use — only when they're doing carpentry (ray tracing). For general errands (general-purpose workloads), their garages sit empty. Heliostat borrows the neighbor's garages when yours is full. Same function (storing cars), just repurposed hardware.

Where it misleads: a garage is a passive space; an RTA is active logic. Heliostat modifies the RTA's pipeline to add PTW capability — it's more like converting a woodshop into a garage than just borrowing it.
Operational similarity · Tree traversal

Following a street address

A page table walk is like finding a house: start with the country (PML4), narrow to state (PDP), city (PD), street (PT), and finally the house number (offset). A BVH tree does the same for ray tracing: narrow from world bounding box to region to sub-region to triangle. Both are hierarchical searches that eliminate branches early. The RTA hardware that speeds up one naturally speeds up the other.

Where it breaks: street addresses have fixed depth (you always go 4 levels); BVH trees can be arbitrarily deep and unbalanced. Page tables are closer to street addresses (fixed depth), so they fit RTAs better than arbitrary BVH.
Routing · The RFU

A restaurant host

You walk into a restaurant with a main dining room (GMMU) and a private party room (RTAs). The host (RFU) checks the dining room first — if there's space, you sit there. If it's full, the host seats you in the party room, which is empty most nights. No reservation needed, no wasted tables. Heliostat's RFU is that host: check GMMU first, route to RTA if busy.

Where it breaks: a host seats you once; the RFU makes this decision per translation, thousands per second. But the analogy captures the core logic: prefer the main resource, overflow to the spare.
Cache reservation · L1S space

Marking a shelf

A shared office bookshelf has 16 shelves. Most people only use 2–3 shelves (1–19% utilization). You mark one shelf "RTA page tables only" — regular office supplies (L1V data) can't go there, but multiple departments (RTAs in the array) can use it. No pollution, dedicated space, minimal cost. That's the L1S reservation: one way per set, exclusively for page table entries.

Where it breaks: a bookshelf doesn't evict old books automatically; a cache does (LRU). But the analogy captures the key: reserve underutilized space to avoid interference.
Heliostat+ · Secondary ray for lookahead

A chess player thinking ahead

You're playing chess. Your opponent makes a move (on-demand translation). While computing your response, you also think "if I move here, they'll probably do X" — you lookahead one step (secondary ray). You walk the tree for your move and the predicted next move simultaneously, sharing the early search. If your prediction is right, you've already computed half the work for the next turn. Heliostat+ does this for page translations: fork a lookahead thread that shares upper page table levels with the on-demand translation, cache the result, and skip the walk when the predicted address arrives.

Where it breaks: chess has perfect information; memory accesses are noisier. Heliostat+ uses simple stride-based prediction (1, 64, 128, 256 pages), which works well for regular access patterns but can miss irregular workloads.
Sharing upper levels · Memory bandwidth

Taking the same elevator

Two people need to reach floors 10 and 12. If they take separate elevators from the lobby, both ride floors 1–9. Wasteful. Instead, they share an elevator to floor 9, then one continues to 10, the other to 12. Heliostat+ does this for page table walks: the on-demand and lookahead translations share the upper three levels (PML4, PDP, PD), fork only when their paths diverge at the PT level. This saves memory bandwidth — you only fetch the shared upper-level PTEs once.

Where it breaks: elevators move continuously; page table walks fetch discrete levels. But the analogy captures the key: share the common path, split when necessary.
Hardware cost · Lightweight modification

Adding a mailbox vs. building a post office

To increase mail capacity, you could build a new post office (128-PTW GMMU: 7.8× area, 6.8× power). Or you could install 128 mailboxes on existing houses (RTAs), adding a small slot to each (PDU). Same capacity, 1.53% the area. Heliostat is the mailbox approach: leverage existing infrastructure (RTAs are already there), add a tiny new function (PDU), and avoid overprovisioning.

Where it breaks: mailboxes can't sort or deliver mail; they just hold it. RTAs can fully process page table walks (fetch, decode, translate) — they're more capable than passive storage. But the analogy captures the cost asymmetry: repurposing is far cheaper than building new.
The full explanation

How it actually works

Section by section, with the paper's key mechanisms rebuilt as concise visuals.

1 · The problem — PTW bandwidth bottleneck

The paper starts by measuring where L2 TLB miss penalty goes. Result: queuing delay dominates in most applications. That means translation requests sit waiting for an available PTW, not because walks are slow, but because all 16 PTWs are busy. Adding 128 PTWs would fix it (comparable speedup to Heliostat+) but costs 7.8× area and 6.8× power. Most applications with low translation intensity don't benefit, so the hardware is wasted.

📊
Measured: applications with high L2 TLB MPKI (e.g., gesummv: 4784.5, gups: 557.4) show over 3× speedup with enough PTWs. But low-intensity apps (doitgen, syrk) barely improve — so overprovisioning wastes silicon.

2 · The insight — RTAs are tree traversal engines

Both ray tracing and page table walk involve depth-first tree traversals, though the structures differ. A BVH tree has internal nodes (axis-aligned bounding boxes) and leaf nodes (primitives). A page table has internal nodes (PTEs pointing to next-level tables) and leaf nodes (PTEs with PFN). The operations differ (ray-box intersection vs. PTE decoding), but both follow similar traversal patterns: fetch a node, decode it, fetch the next level, repeat until leaf.

RTAs have: (1) a Mem Access FIFO for independent memory transactions, (2) a Ray Buffer for storing traversal state, (3) a Node Decoder for determining next steps, (4) Operation Units for executing node-specific logic. All four pieces can be repurposed for page table walks with small modifications.

💡
Key observation: RTAs already exist (one per CU = 128 total), they're idle for general-purpose workloads, and they're fundamentally tree-walkers. Why not use them for page table walks?

3 · RTA modification — adding the PDU

Heliostat adds one new operation unit to the RTA: the PTE Decoding Unit (PDU). The PDU handles three functions:

  1. Check if last-level PTE: For 4KB pages, count to depth 4. For huge pages, check the PS bit. Two comparators.
  2. Calculate next-level address: Extract the level's index from the VPN, shift (multiply by 8), add to the base address from the current PTE. One adder, one shifter.
  3. Extract PFN and check access control: On last-level PTE, extract PFN, check valid bit and rwx. Two comparators.

The PDU uses the existing Ray Buffer to store RayPTWProperty (VPN, PID, rwx, level, physical address) instead of ray origin/direction. The buffer already exists, so no extra storage. The PDU outputs either (a) next-level page table address (to Mem Access FIFO), or (b) translated physical address (to RFU), or (c) page fault (replayed by GMMU).

🔮
Hardware cost: one adder, one shifter, four comparators per PDU. Four PDUs per RTA. 128 RTAs × 4 PDUs = 0.024 mm² total — 1.53% area and 5.8% power compared to a 128-PTW GMMU.

4 · Routing — the RFU

The RT-PTW Forwarding Unit (RFU) sits between the L2 TLB and the rest of the system. On an L2 TLB miss, the RFU checks: are there available PTWs in the GMMU? If yes, send the request to GMMU. If no, route the request to the requester CU's RTA via the inter-CU network. The RFU is a simple arbiter: monitor GMMU occupancy, multiplex the request stream.

🔁
Policy: prefer GMMU (it's simpler, optimized for PTW), overflow to RTA only when GMMU is busy. Measured: 65.3% of translations go to RTAs in Heliostat, 68.5% in Heliostat+.

5 · Cache reservation — L1S for RT-PTW cache

Page table walks benefit from caching upper-level PTEs (PML4, PDP entries are shared by many translations). Baseline GMMU has a 128-entry Page Walk Cache. RTAs need something similar, but sharing the L1V (data) cache causes pollution and RTAs have no built-in shared cache.

Solution: reserve space in the L1 Scalar (L1S) cache. L1S is shared by all CUs in a Shader Array, stores read-only constants, and is underutilized (1–19%). Heliostat reserves one way per set (way-level partitioning) for RTA page tables. Regular L1S operations skip the reserved way (an access mask register with one bit per way). RTAs use the reserved way exclusively.

Measured L1S hit rates: baseline (L1V shared) 24.6%, Heliostat (L1S reserved) 48.6%. The isolation doubles hit rate while avoiding pollution.

🔮
Key benefit: L1S is shared across CUs in the array, so page tables cached by one RTA's translation are visible to other RTAs in the array. This enables cross-RTA locality without extra hardware.

6 · Heliostat+ — lookahead translation

Beyond on-demand translations, Heliostat+ adds lightweight hardware support to enable lookahead translation. In ray tracing, when a ray hits a reflective surface, a secondary ray is spawned from that point in a new direction. The secondary ray inherits the traversal stack from the primary ray, sharing the upper BVH levels until their paths diverge.

Heliostat+ does the same for page tables: (1) predict the next address via stride (1, 64, 128, or 256 pages), (2) spawn a lookahead translation thread (secondary ray) that shares the upper page table levels (PML4, PDP, PD) with the on-demand translation, (3) fork only when their PT-level indices differ, (4) cache the lookahead PTE in a dedicated L1S lookahead buffer.

The lookahead buffer stores eight PTEs per cache line, all with consecutive VPNs at a fixed stride. The tag is the base VPN plus a 2-bit stride indicator. On a future on-demand translation, if the VPN hits in the lookahead buffer, the translation is served from cache (skip the walk). If it misses, the RTA runs the translation as usual.

💡
Why this works: GPU memory access patterns often follow strides (e.g., one page per CU = stride 128, one page per two CUs = stride 64). Heliostat+ detects these strides in a 200-entry circular queue, predicts the next access, and prefetches it. Measured: 50% of lookahead entries hit before eviction, reducing 31% of on-demand translations.

7 · Lookahead prediction — stride detection and RPF

Heliostat+ detects strides by monitoring the last 200 on-demand translations in a circular queue. It compares VPNs within each "similar region" and calculates candidate strides. The current design supports four common strides: 1, 64, 128, and 256 pages (chosen based on GPU memory mapping policies: one page per CU, per two CUs, etc.). The stride is encoded in the 2 unused bits of the translation request packet.

To share lookahead results globally, the RFU includes an RT-PTW lookahead filter (RPF) — a cuckoo filter per Shader Array that tracks which RTAs have cached which VPNs. On a translation request, the RFU checks all RPFs; if any return a hit, the request is routed to that Shader Array's RTA (not necessarily the requester's). This enables cross-array lookahead reuse.

🔍
RPF overhead: 32 cuckoo filters × (16 rows × 4 ways × 8-bit fingerprint) = 2 KB total. False positive rate: 1.53% theoretical, 0.38% measured. On false positive, the RTA simply doesn't find the lookahead entry and runs the translation normally.
Follow the numbers, step by step

Worked examples

Four traces, each carried all the way through — reading miss rates, calculating overhead, understanding lookahead, and tallying hardware cost.

① Quantifying the PTW bottleneck

Application gesummv has L2 TLB MPKI = 4784.5. With baseline 16 PTWs, what fraction of miss penalty is queuing delay, and how much speedup does adding 128 PTWs provide?

  1. Measure the baseline miss penalty breakdown. The paper's Fig. 5 shows: queuing delay ≈ 80% of total miss penalty for high-intensity apps. For gesummv, almost every L2 TLB access misses (near 100% miss rate).
  2. Interpret the bottleneck: 80% queuing means 80% of the time, translation requests sit waiting for a free PTW. The actual page table walk (memory latencies) is only 20% of the total.
  3. Add 128 PTWs (or equivalent RTA capacity). From Fig. 1, gesummv achieves ~5.5× speedup with 128 PTWs over 16 PTWs. The queuing delay drops to near zero.
  4. Heliostat's approach: instead of adding 128 PTWs to the GMMU (7.8× area, 6.8× power), repurpose the 128 existing RTAs. Measured speedup: 2.39× (Heliostat+) at 1.53% area, 5.8% power.
🔮
Insight: the queuing bottleneck is real, and adding PTW capacity fixes it. Heliostat achieves comparable speedup by borrowing idle RTAs instead of overprovisioning the GMMU.

② Walking through an RTA-based translation

A translation request for VPN 0x12345678 arrives at the RFU. GMMU has 16 busy PTWs. Walk through Heliostat's steps to translate the address.

  1. RFU checks GMMU monitor: all 16 PTWs busy. Route the request to the requester CU's RTA.
  2. RTA warp scheduler: assign a warp, create a RayPTWProperty entry in the Ray Buffer with V_addr = 0x12345678, level = 0.
  3. Fetch PML4 entry: use PTBR as base, extract PML4 index from VPN bits [39:36], fetch PTE from memory via Mem Access FIFO.
  4. PDU checks PS bit: not set, level ≠ 4. Calculate next-level base address from PTE, increment level to 1, fetch PDP entry.
  5. Repeat for PDP, PD, PT: PDU traverses four levels, checking L1S RT-PTW cache at each step. Measured: 48.6% hit rate on upper-level PTEs.
  6. Last-level PT: PDU detects level == 4, extracts PFN from PTE, checks valid bit and rwx. Translation complete.
  7. Response: RTA sends translated physical address to RFU → L2 TLB → L1 TLB.
💡
Key observation: the RTA's existing Mem Access FIFO and Ray Buffer handle all the fetching and state. The PDU just decodes PTEs instead of BVH nodes — the rest of the pipeline is unchanged.

③ Lookahead translation saves bandwidth

An on-demand translation for VPN 1000 arrives with stride 128. Heliostat+ spawns a lookahead for VPN 1128. How many memory fetches are saved by sharing upper levels?

  1. On-demand translation fetches PML4: PTBR + PML4 index from VPN 1000. One memory fetch.
  2. Lookahead inherits PML4: no separate fetch needed. Savings: 1 fetch.
  3. On-demand fetches PDP: base from PML4 entry + PDP index. One memory fetch.
  4. Lookahead inherits PDP: VPN 1000 and 1128 share the same PDP index (both in the same 512GB region). Savings: 1 fetch.
  5. On-demand fetches PD: base from PDP entry + PD index. One memory fetch.
  6. Lookahead forks here: for typical 4KB pages with VPNs differing by 128, paths diverge at the PD level. Lookahead spawns a separate thread, fetches its own PD and PT. Total: 2 fetches for lookahead.
  7. Total memory fetches: on-demand (4) + lookahead (2) = 6. Baseline without sharing: 4 + 4 = 8. Savings: 2 fetches (25%).
🔮
Measured impact: paper reports 7% extra memory traffic for lookahead translations, but 31% reduction in on-demand translations (lookahead buffer hits). Net: bandwidth-neutral, latency reduced.

④ Tallying the hardware cost

Compare the area and power of three designs: baseline 16-PTW GMMU, 128-PTW GMMU, and Heliostat with 128 RTAs. Use the paper's RTL synthesis results.

  1. Baseline GMMU: 16 PTWs, 16-entry single-ported CAM for Page Walk Cache. Synthesized area: 0.39 mm², power: 77.2 mW.
  2. 128-PTW GMMU: scale linearly (128/16 = 8×). Area: 0.39 × 7.8 = 3.04 mm², power: 77.2 × 6.8 = 525 mW. (Measured overhead from Fig. 1.)
  3. Heliostat: add PDU to 128 RTAs. Each PDU: 4 comparators, 1 adder, 1 shifter. Four PDUs per RTA. Total: 128 × 4 × (6 gates) = 0.024 mm², 4.484 mW.
  4. Heliostat vs. 128-PTW GMMU: The paper reports Heliostat achieves 1.53% area and 5.8% power compared to the 128-PTW GMMU (accounting for RFU and L1S modifications in addition to PDU hardware).
  5. Speedup comparison: 128-PTW GMMU achieves similar speedup to Heliostat+. Both: approximately 2.39×.
💡
Bottom line: repurposing existing RTAs is two orders of magnitude cheaper than overprovisioning the GMMU, while achieving the same performance.
Test yourself

Practice problems

Try each before opening the answer. Every one ends with the common misconception it's checking for.

Problem 1 · easy
Why does overprovisioning the GMMU with 128 PTWs waste area for low-intensity applications (e.g., doitgen, L2 TLB MPKI = 0.015)?
Show answer
Low-intensity apps rarely miss the L2 TLB (0.015 misses per 1000 instructions). They barely use the baseline 16 PTWs, so adding 112 more PTWs provides zero benefit — those PTWs sit idle, consuming static leakage power. The area cost (7.8×) is paid for hardware that's never used. Heliostat's approach (repurpose idle RTAs) avoids this waste because RTAs already exist and are idle anyway.Misconception: that more PTWs always help. They only help when the GMMU is a bottleneck. For low-intensity apps, the TLB is sufficient.
Problem 2 · medium
Application nw has L2 TLB MPKI = 21.1 and achieves 3× speedup with Heliostat. The miss penalty breakdown shows 70% queuing, 20% page table traversal, 10% other. If you reduced traversal latency by 50% (e.g., faster memory), how much would IPC improve?
Show answer
Traversal is only 20% of the miss penalty. Halving it saves 10% of total penalty. Queuing (70%) is the dominant component, which Heliostat addresses by adding PTW bandwidth. Reducing traversal latency would improve IPC by at most 10% × (fraction of time spent on TLB misses) — negligible compared to Heliostat's 3× speedup from eliminating queuing. Misconception: that faster memory always helps. The bottleneck is PTW bandwidth (queuing), not latency (memory speed).
Problem 3 · medium
Why does Heliostat reserve space in L1S instead of L1V for the RT-PTW cache, even though L1V is larger and has more bandwidth?
Show answer
Two reasons: (1) Pollution: L1V is the main data cache for general-purpose operations. Storing page tables there evicts useful application data, hurting performance. Measured: baseline (L1V shared) gets only 24.6% hit rate because page table entries are constantly evicted by regular loads/stores. (2) Sharing: L1S is shared across all CUs in a Shader Array, so page tables cached by one RTA's translation are visible to other RTAs. L1V is per-CU, so no sharing. Heliostat's L1S reservation achieves 48.6% hit rate with isolation and cross-RTA sharing.Misconception: that bigger is better. L1V is larger but wrong for this use case — isolation and sharing matter more than raw capacity.
Problem 4 · hard
Heliostat's PDU checks the PS bit to detect huge pages. For a system that uses only 4KB pages (no huge pages), explain how the PDU simplifies and whether the RTA's existing BVH traversal logic could handle this case natively without a PDU.
Show answer
For 4KB-only, the PDU skips the PS bit check and simply counts to level == 4 (statically determined depth). This saves one comparator. However, the RTA's BVH logic cannot handle this natively because: (1) BVH node format (AABB min/max, child indices) ≠ PTE format (PFN, flags). The Node Decoder expects BVH-specific fields. (2) BVH operations (ray-box intersection) ≠ PTE operations (extract PFN, check rwx). The Operation Units are hardwired for ray tracing. The PDU is necessary to decode PTEs and execute page table logic, even for fixed-depth trees.Misconception: that structural similarity (both are trees) implies functional similarity (same operations). The data formats and operations differ; the PDU bridges this gap.
Problem 5 · medium
Heliostat+ uses stride-based lookahead prediction with four strides: 1, 64, 128, 256. Why these specific values, and what access pattern would cause high false positive rate in the lookahead buffer?
Show answer
The four strides match common GPU memory mapping policies: stride 1 (linear access), stride 64 (one page per two CUs in a 128-CU GPU), stride 128 (one page per CU), stride 256 (one page per CU with 2× interleaving). These patterns arise from how kernels distribute work across CUs. An access pattern with irregular, non-strided jumps (e.g., pointer chasing, hash table lookups) would cause high false positive rate — the lookahead buffer would cache VPNs that never get accessed, wasting space and L1S bandwidth. Measured: most tested apps have regular strides (50% hit rate), but irregular workloads would benefit less.Misconception: that stride prediction works for all access patterns. It's effective only for regular, predictable patterns — which fortunately are common in GPU computing.
Problem 6 · hard
The RFU routes translations to RTAs based on PTW availability. Construct a scenario where this policy causes pathological behavior (e.g., load imbalance, excessive NoC traffic) and propose a fix.
Show answer
Scenario: suppose 16 CUs in one corner of the GPU simultaneously issue L2 TLB misses. All 16 PTWs are busy, so the RFU routes all 16 requests to those CUs' RTAs. But those 16 RTAs are also in the same corner, creating a hotspot. Other RTAs across the GPU sit idle. Pathology: (1) load imbalance — 16 RTAs overloaded, 112 idle; (2) NoC congestion — all responses funnel back through the same network path. Fix: add a simple load-balancing policy to the RFU — track RTA occupancy (e.g., how many translations each RTA is handling), and route new requests to the least-loaded RTA in the system (or within a local region), not necessarily the requester's. Trade: slightly more NoC traffic for better load distribution. Paper doesn't implement this but acknowledges the RFU could be enhanced.Misconception: that "requester's RTA" is optimal. Locality helps for L1S cache, but load balance matters for throughput.
Problem 7 · easy
Heliostat repurposes RTAs for general-purpose workloads. What happens when a ray tracing kernel runs? Does Heliostat break ray tracing performance?
Show answer
When a ray tracing kernel is invoked, Heliostat is disabled — the RTAs revert to their original function. The RFU stops routing translations to RTAs and only uses the GMMU. No performance impact on ray tracing. The paper notes this can be controlled via driver option or by detecting RT API calls. Technically, Heliostat and RT could run concurrently if the RTA's warp buffers aren't fully utilized (Heliostat uses dedicated PDU operation units, separate from ray-box/ray-tri units), but the paper conservatively disables Heliostat during RT to avoid any potential interference.Misconception: that repurposing RTAs permanently disables ray tracing. The hardware is unchanged; the PDU is just one more operation unit that's activated only for PTW.
Problem 8 · medium
The paper reports 41.5% reduction in page translation latency with Heliostat, but only 1.93× IPC speedup. Why doesn't a 58.5% latency reduction translate to 1.7× speedup?
Show answer
Two reasons: (1) Not all instructions are memory-bound. Translation latency only affects instructions that miss the TLB. For apps with low L2 TLB MPKI, most instructions are unaffected. The 41.5% latency reduction applies only to the miss penalty, which is a fraction of total execution time. (2) Speedup is geometric mean across 23 apps. Some apps (low intensity) barely benefit, dragging down the average. High-intensity apps (e.g., gesummv) show 5.5× speedup, but low-intensity apps (e.g., doitgen) show 1.0× — the geometric mean is 1.93×. The latency reduction is averaged across all apps, including those that don't bottleneck on translation.Misconception: that latency reduction = speedup. Speedup depends on how much time is spent on the bottleneck, not just the bottleneck's latency.
Quick answers

Frequently asked questions

Twelve questions newcomers ask most. Tap any to expand.

Q1Why can't we just make the TLB bigger instead of adding PTW bandwidth?▾
The paper measured translation reuse distances (Fig. 6). High-intensity apps (e.g., gups, gesummv) have reuse distances from 2⁹ to 2¹⁶ entries — far beyond the baseline 512-entry L2 TLB. To cover those apps, you'd need a TLB with tens of thousands of entries, which costs huge area and power. More critically, low-intensity apps have short reuse distances (<512) and wouldn't use the extra TLB capacity — so you'd waste silicon. Heliostat's approach (add PTW bandwidth) helps all apps that bottleneck on translation, regardless of reuse distance.
Q2What's the difference between Heliostat and Heliostat+?▾
Heliostat handles on-demand translations only — it routes TLB miss requests to idle RTAs. Heliostat+ adds lookahead translation: predict the next address (via stride), spawn a secondary translation thread that shares upper page table levels with the on-demand translation, and cache the result in the L1S lookahead buffer. Future on-demand translations can hit in the buffer and skip the walk. Measured: Heliostat 1.93×, Heliostat+ 2.39× speedup.
Q3How does Heliostat know when to use an RTA vs. the GMMU?▾
The RFU (RT-PTW Forwarding Unit) monitors the GMMU's PTW occupancy. If there's an available PTW, the request goes to the GMMU (simpler, optimized for PTW). If all 16 PTWs are busy, the RFU routes the request to an idle RTA. Measured: 65.3% of translations go to RTAs in Heliostat, 68.5% in Heliostat+. This means the GMMU is heavily oversubscribed — RTAs handle the overflow.
Q4Why reserve L1S space instead of adding a separate PT cache like the baseline GMMU?▾
Two reasons: (1) Cost: a separate 128-entry Page Walk Cache per Shader Array would add area and power. The L1S is already there and underutilized (1–19% measured). (2) Sharing: L1S is shared across all CUs in a Shader Array, so page tables cached by one RTA's translation are visible to other RTAs. A per-RTA cache would miss this cross-RTA locality. Reserving L1S space is essentially free (no new hardware) and provides better sharing.
Q5What happens if the L1S cache is heavily used by the application (e.g., lots of constant data)?▾
The paper measured L1S utilization across 23 apps: 1–19%. Even in the worst case (19%), reserving one way per set (6.25% of capacity, assuming 16-way L1S) leaves 12.75% unused. However, if an app genuinely saturates L1S, reserving a way would cause some constant data to be evicted to L2, adding latency. The paper doesn't test this edge case but notes it's rare. Heliostat could dynamically enable/disable the L1S reservation based on monitored L1S miss rate — a future enhancement.
Q6Does Heliostat support multiple page sizes (4KB, 64KB, 2MB)?▾
Yes. The PDU checks the PS (page size) bit in each PTE at every level. If the PS bit is set, the current PTE is a leaf (huge page); extract the PFN and finish. If not, continue to the next level. For systems that use only 4KB pages, the PDU simplifies by checking level == 4 instead of the PS bit. The paper evaluates Heliostat+ with 64KB and 2MB pages (Fig. 26): at base application scale, Heliostat+ achieves 7% and 0.3% speedup for 64KB and 2MB respectively; when scaling applications 16×, speedup increases to 31% and 4.2%.
Q7How does Heliostat+ predict the stride? What if the app has multiple strides?▾
The RFU maintains a 200-entry circular queue of recent on-demand translations. It compares VPNs within each "similar region" and calculates candidate strides. The current design supports four strides: 1, 64, 128, 256 (chosen empirically based on GPU memory mapping). If an app has multiple strides (e.g., alternating 1 and 128), Heliostat+ picks the most frequent stride in the recent window. The paper doesn't implement per-pattern stride tracking (future work) but notes that most GPU apps have dominant strides due to regular memory access patterns.
Q8What's the RPF (RT-PTW lookahead filter) and why is it needed?▾
The lookahead buffer is stored in L1S, which is shared per Shader Array. An on-demand translation in one Shader Array might benefit from a lookahead entry cached in a different Shader Array. The RPF (a cuckoo filter per Shader Array) tracks which VPNs are cached where. On a translation request, the RFU checks all RPFs; if any return a hit, the request is routed to that Shader Array's RTA (not the requester's). This enables cross-array lookahead reuse. False positive rate: 0.38% measured (1.53% theoretical) — low enough to be negligible.
Q9Does Heliostat work for multi-tenancy (multiple apps running concurrently)?▾
Yes. The paper tested multi-programming by running app combinations across different translation intensities (low/mid/high). Heliostat+ achieves 1.47× speedup over baseline. Each process has its own PTBR and PID, and the PDU uses the PID to differentiate processes. The lookahead buffer tags include the PID to avoid cross-process false hits. The main challenge is fairness: high-intensity apps might monopolize RTA bandwidth, starving low-intensity apps. Heliostat doesn't implement per-process QoS (future work) but notes the RFU could track per-process latency and throttle if needed.
Q10What about TLB shootdowns (page invalidations)? Does the lookahead buffer cause stale PTE issues?▾
On page faults, migrations, or permission changes, the OS issues a TLB shootdown (flush all TLBs). Heliostat+ treats the lookahead buffer as part of the TLB hierarchy: on a shootdown, flush both TLBs and the lookahead buffer. The paper notes this is a system-level command (no per-entry invalidation needed). Since GMMU and RTAs traverse the same page table copy in memory, coherence between them is inherently maintained — they see the same updated PTEs after a shootdown.
Q11Can Heliostat be applied to CPUs or is it GPU-specific?▾
Heliostat's core idea (repurpose underutilized tree-traversal hardware for PTW) is general, but the implementation is GPU-specific for two reasons: (1) GPUs have hundreds of CUs and one GMMU — the PTW bandwidth bottleneck is unique to massively parallel architectures. CPUs have fewer cores and more PTWs per core, so queuing delay is less severe. (2) GPUs have RTAs (one per CU) that sit idle for general-purpose workloads. CPUs don't have equivalent idle accelerators. A CPU analogy might be repurposing AVX512 units for PTW, but AVX512 isn't idle as often and doesn't match the tree-traversal structure as cleanly as RTAs.
Q12What are the main limitations or future work?▾
The paper flags: (1) Load balancing: the RFU routes to the requester's RTA, which can cause hotspots. Future: route to least-loaded RTA. (2) RT concurrency: Heliostat conservatively disables during RT kernels. Future: allow concurrent RT and PTW if warp buffers have spare capacity. (3) Stride prediction: Heliostat+ uses simple strides (1, 64, 128, 256). Future: per-pattern stride history or ML-based prediction. (4) QoS: no fairness guarantees for multi-tenancy. Future: per-process bandwidth limits. (5) L1S saturation: edge case where L1S is fully utilized by constants. Future: dynamic reservation based on monitored miss rate.

Glossary & further reading

Every term in one place, then a path to go deeper.

Core terms
HeliostatThe paper's mechanism: repurpose underutilized ray tracing accelerators (RTAs) to handle page table walks, expanding PTW bandwidth without overprovisioning the GMMU.
GMMUGPU Memory Management Unit — the baseline virtual memory hardware with 16 page table walkers (PTWs) that traverse multi-level page tables on TLB misses.
PTWPage Table Walker — a circuit that traverses the radix-tree-based page table to translate one virtual address to physical at a time.
RTARay Tracing Accelerator — specialized hardware per compute unit that traverses BVH trees for ray tracing. Heliostat repurposes RTAs for page table walks.
PDUPTE Decoding Unit — Heliostat's new operation unit added to the RTA pipeline. Handles page table entry decoding, address calculation, and access control checks.
RFURT-PTW Forwarding Unit — arbitrator that routes translation requests to GMMU or idle RTAs based on PTW availability. Monitors GMMU occupancy and distributes load.
L1SL1 Scalar cache — shared read-only cache per Shader Array, typically used for constant data. Heliostat reserves one way per set for RT-PTW cache (page tables).
Heliostat+Enhanced version of Heliostat that adds lookahead translation with lightweight hardware support: predict next address, share upper page table levels, cache result.
Lookahead translationProactive translation for predicted future requests. Heliostat+ spawns a secondary translation thread (like a secondary ray) that shares upper page table levels with on-demand translation.
Supporting terms
BVHBounding Volume Hierarchy — tree structure for ray tracing where internal nodes are axis-aligned bounding boxes, leaf nodes are primitives. Operationally similar to page tables.
Ray BufferInternal buffer in RTA that stores ray properties (origin, direction) or, in Heliostat, RayPTWProperty (VPN, PID, rwx, level, P_addr).
RayPTWPropertyData structure stored in Ray Buffer for PTW operations: VPN (64b), PID (32b), rwx (8b), level (8b), P_addr (64b). Total 112 bits vs. 256-bit original RayProperty.
PS bitPage Size bit in a PTE. If set, the current PTE is a last-level entry for a huge page (64KB, 2MB). PDU checks this to support multiple page sizes.
RT-PTW cacheReserved space in L1S cache for caching page table entries accessed by RTA-based translations. Avoids L1V pollution and enables cross-RTA sharing.
Lookahead bufferDedicated portion of L1S cache for storing lookahead translation results. Format: 8 PTEs per cache line, all with consecutive VPNs at fixed stride.
Secondary rayRay tracing feature: when a primary ray hits a reflective surface, a secondary ray is spawned from that point in a new direction. Heliostat+ leverages this for lookahead translation.
StrideAddress offset between consecutive memory accesses. Heliostat+ detects strides (1, 64, 128, 256 pages) to predict next translation request.
RPFRT-PTW lookahead filter — cuckoo filter per Shader Array that tracks which VPNs are cached in that array's lookahead buffer. Enables cross-array lookahead reuse.
Node DecoderRTA component that decodes BVH node type (internal/leaf). Modified in Heliostat to check PS bit for huge pages and activate PDU for PTE operations.
Mem Access FIFODedicated memory unit in RTA that issues memory transactions to fetch BVH nodes. In Heliostat, used to fetch page table entries.

Concepts to shore up first

GPU architecture and memory hierarchy easy

explainer · 30–45 min

How GPUs organize compute units, caches, and memory. Search "AMD GCN architecture" or "NVIDIA GPU memory hierarchy" (AMD/NVIDIA whitepapers). Heliostat targets AMD-style GPUs (Shader Arrays, L1S cache) but concepts apply broadly.

Virtual memory and page tables easy

explainer · 20–30 min

How virtual addresses translate to physical via multi-level page tables. Search "x86 page table walk" (Intel SDM, Hennessy & Patterson Ch. 5). Heliostat assumes you know the 4-level radix tree (PML4, PDP, PD, PT).

Ray tracing and BVH trees medium

explainer · 45–60 min

How ray tracing works and why BVH trees accelerate it. Search "BVH traversal" (NVIDIA RTX whitepaper, Pharr et al. "Physically Based Rendering"). Heliostat's insight is that BVH traversal and page table walks are operationally similar.

Related papers

Mosaic: Multi-Page GPU Allocation medium

Ausavarungnirun et al. · MICRO 2017

Addresses GPU virtual memory bottlenecks via large pages and transparent migration. Heliostat complements by adding PTW bandwidth instead of reducing TLB misses.

Valkyrie: Inter-TLB Sharing medium

Baruah et al. · PACT 2020

Enables L1 TLB sharing within Shader Arrays via lightweight NoC. Heliostat outperforms (1.92×) by adding PTW bandwidth, not just sharing cached translations.

BarreChord: Multi-GPU Page Coalescing hard

Feng et al. · ISCA 2024

Allocates temporally similar pages to identical physical locations across GPU chiplets, enabling translation without PTW. Heliostat outperforms (1.66×) by handling arbitrary page mappings.

RTNN: Ray Tracing for kNN hard

Zhu · PPoPP 2022

Reformulates k-nearest neighbor search as a ray tracing problem to leverage RTAs. Heliostat extends this democratization idea to system operations (PTW) instead of application algorithms.

TTA: Tree Traversal Accelerator hard

Ha et al. · MICRO 2024

Modifies RTAs to traverse general-purpose tree structures beyond BVH. Heliostat targets page tables specifically, leveraging radix-tree structure for simpler PDU design.
