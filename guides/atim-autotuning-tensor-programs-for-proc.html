Making DRAM-PIM programmable — without giving up performance

DRAM chips can now compute on their own data. The software problem: every kernel needs hundreds of lines of hand-tuned C. This is the plain-English story of ATiM, a tensor compiler that writes optimized DRAM-PIM code for you.

1What DRAM-PIM is and why it exists▾

Modern workloads (LLMs, databases) are memory-intensive — they spend more time waiting for data from DRAM than computing on it. The "memory wall": limited memory channels bottleneck everything, no matter how many CPU cores you have.

DRAM-PIM puts a small processor (DPU) right next to each DRAM bank, so data can be computed in-place at high internal memory bandwidth without crossing channels. DRAM-PIM products provide up to 16× higher in-memory compute bandwidth. The evaluated UPMEM server has 2048 DPUs — each bank becomes an independent compute node.

Now you're ready: ATiM targets UPMEM, which treats DRAM-PIM like a massively parallel distributed system — thousands of small in-order cores, each with 64 KB working memory.
2Why DRAM-PIM is hard to program today▾

Unlike GPUs, which have a unified memory abstraction and hardware schedulers to manage thousands of threads, UPMEM requires you to manually (1) distribute tensor data across thousands of DPUs, (2) write explicit C kernel code for each DPU, (3) manage 64 KB WRAM by hand (when to cache, how much), and (4) aggregate results back on the CPU host.

Current practice: hand-write ≈200 lines of C per operation, tuning every parameter by trial-and-error. Libraries like PrIM provide pre-tuned kernels but cover only a narrow set of shapes and operations.

Now you're ready: ATiM automates all four steps — generating host and kernel code from high-level tensor operations and searching for optimal tiling, caching, and parallelism parameters.
3Tensor compilers and schedule primitives▾

Tensor compilers (like TVM) separate what you compute (e.g., C[i] = A[i,j] · B[j]) from how. You write the computation once in a high-level IR; schedule primitives — Python-level directives like split(loop, factors=[16,4]) or cache_read(block, "local") — reshape the loops without touching the algorithm.

This separation enables autotuning: generate hundreds of candidate schedules by varying parameters (tile sizes, unroll factors, caching locations), compile and measure each, keep the best. It's how TVM gets GPU kernel performance that rivals hand-tuned cuBLAS.

Now you're ready: ATiM extends TVM's schedule primitives to simultaneously control host-to-DPU data distribution and per-DPU kernel structure — a joint search space prior work couldn't handle.
4The joint optimization problem for DRAM-PIM▾

For a GPU, autotuning focuses on the kernel — thread-block sizes, caching strategy. The host just launches it. But for UPMEM, host-side decisions determine kernel performance: if you tile a 8192×8192 matrix into 32×32 chunks and send them to 2048 DPUs, each DPU runs one tile sequentially — no parallelism. Tile into 16×16 chunks and you get 4× more tiles to distribute, but higher data transfer overhead.

The problem is interdependent: host tiling controls inter-DPU parallelism; kernel tiling (inside each DPU) controls intra-DPU parallelism and WRAM usage. Optimizing one without the other leaves performance on the table.

Now you're ready: ATiM's autotuner samples from a joint space of host and kernel parameters — e.g., "tile the reduction dimension into 4 chunks across DPUs, then inside each DPU cache 64 elements at a time."
5Branch penalties on simple in-order cores▾

A DPU is a 32-bit RISC core with no branch predictor, no reorder buffer — every mispredict stalls the pipeline. This is fine when executing tight loops with predictable branches, but boundary checks — conditionals to ensure tile accesses don't overflow tensor bounds — can trigger frequent stalls if not optimized away.

Low-level compilers (LLVM) can hoist loop-invariant conditions but can't safely eliminate boundary checks without high-level semantic knowledge. On CPU/GPU, advanced hardware hides these penalties; on DPU, a redundant branch can cost 20% runtime.

Now you're ready: ATiM implements three tensor-level passes (DMA-aware boundary check elimination, loop-bound tightening, invariant branch hoisting) that aggressively remove or reduce boundary checks by exploiting known loop structure and UPMEM's DMA features.
6The role of DMA in UPMEM▾

Each DPU moves data between MRAM (64 MB per bank) and WRAM (64 KB working memory) via a DMA engine — much faster for bulk transfers than element-by-element loads. But DMA transfers require contiguous data without conditional statements, which is blocked by boundary checks around the loop.

ATiM's boundary check elimination pass can remove those conditionals (by padding MRAM tiles locally, ensuring out-of-bounds reads are safe), turning slow loops into fast DMA calls. It's an algorithm-level optimization a general compiler can't do without knowing the tile padding strategy.

Now you're ready: ATiM's code generator translates cache_read/write schedule primitives into DMA intrinsics after eliminating boundary checks — a transformation enabled by tensor-level semantic knowledge.
The whole story in one place

ATiM in a nutshell

The problem, the insight, and how it all fits together — from tensor operation to optimized DRAM-PIM binary.

Today's DRAM-PIM software stacks provide only low-level C programming with hand-tuned libraries, making every new operation a multi-day engineering task. Even library-based approaches require explicit host and kernel code for data distribution, kernel execution, and result aggregation — and lack systematic optimization support.

The key insight: tensor IRs and autotuning can bridge this gap. Tensor operations are naturally loop-based (iterate over array elements, apply a computation), making them a perfect fit for DRAM-PIM's distributed in-memory execution. And the interdependence of host tiling and kernel structure creates a large, complex optimization space that search-based methods (autotuning) can navigate better than manual tuning.

ATiM extends the TVM compiler to provide fully automated code generation for UPMEM. It repurposes TVM's schedule primitives to define a joint search space including host-to-DPU data distribution (inter-DPU parallelism) and per-DPU kernel optimizations (loop tiling, caching, intra-DPU parallelism). A refined evolutionary search explores this space, filtering infeasible candidates early using UPMEM hardware constraints, and measuring promising ones on hardware to converge to optimal schedules.

Once a schedule is selected, ATiM's TIR lowering passes generate (1) per-DPU address calculations for tiled data, (2) host data transfer code with bulk and parallel optimizations, (3) hierarchical reduction loops across DPUs and host, and (4) post-processing for aggregating results. Three PIM-aware optimization passes then eliminate or reduce boundary checks by exploiting UPMEM's DMA features and known loop structures.

The payoff: ATiM-compiled kernels outperform hand-tuned PrIM by up to 6.18× on benchmark operations and 8.21× on GPT-J layers, with an average speedup of 2.49× over PrIM and 1.85× over PrIM(E). When PrIM's parameters are grid-searched (PrIM+search), ATiM achieves an average speedup of 1.84×. Against CPU-autotuned TVM, ATiM achieves up to 23.3× speedup for large tensors (≥64 MB) by eliminating data movement bottlenecks.

🍯 Key takeaways

  • Hand-tuning DRAM-PIM is tedious; current high-level solutions lack full automation or systematic optimization.
  • ATiM repurposes TVM schedule primitives to define a joint search space for host and kernel code — simultaneously optimizing inter-DPU and intra-DPU parallelism.
  • A balanced evolutionary search with adaptive epsilon-greedy strategy handles the expanded search space, preventing premature convergence to suboptimal (rfactor-biased) candidates.
  • PIM-aware optimizations (DMA-aware boundary check elimination, loop-bound tightening, invariant branch hoisting) aggressively reduce branch overheads on in-order DPU cores by up to 23.7%.
  • Result: 6.18× over PrIM, 8.21× on GPT-J layers, 1.84× over PrIM+search, 23.3× over CPU for large tensors — with fully automated, autotuning-integrated code generation.
Five everyday pictures

How to think about ATiM

Each analogy stands alone — read them in any order. Every card ends with where the picture breaks.

Core idea · Joint optimization

Scheduling a team project

You assign tasks to team members (host-to-DPU tiling), and each member organizes their own work (kernel structure). If you assign huge chunks to one person while others sit idle, the project drags. If you split tasks too finely, coordination overhead explodes. ATiM searches both assignment and per-person organization together, measuring the real completion time to find the best combination — a joint optimization that beats tuning either level independently.

Where it breaks: real teams have unpredictable human factors; ATiM's search space is deterministic, with hardware measurements as ground truth.
Autotuning · Evolutionary search

Breeding better solutions

Start with a population of random schedules (different tile sizes, caching locations). Measure each on hardware; keep the top performers. Generate new candidates by "mutating" the winners — tweak a parameter, try a different loop ordering. Repeat for ≈1000 generations. The cost model (trained on past measurements) predicts which mutations are promising, focusing search effort there. It's evolution guided by a learned intuition, converging to solutions that outperform hand-tuning.

Where it breaks: biological evolution has no cost model; ATiM's search is accelerated by a trained predictor, making it far faster than blind mutation.
Schedule primitives · Separation of concerns

Blueprint vs. construction

An architect draws a blueprint (the computation: C[i] = A[i,j] · B[j]); a contractor decides how to build it (loop order, tiling, parallelism). TVM's schedule primitives are the contractor's instructions: "tile the loop into 16×4 chunks, cache the inner tile, parallelize the outer." The same blueprint can be built many ways — ATiM generates hundreds of contractor plans, measures the build time, and picks the fastest. The separation lets you automate the "how" without rewriting the "what."

Where it breaks: construction blueprints don't change the result; different schedules produce the same numerical answer but with vastly different performance.
Boundary checks · DMA-aware elimination

Removing safety guardrails

When loop tiles don't align perfectly with tensor boundaries, compilers insert conditionals ("if this index is valid, proceed") to prevent out-of-bounds accesses. On a CPU with branch prediction, these are cheap; on an in-order DPU core, each check stalls the pipeline. ATiM's solution: pad MRAM tiles locally so out-of-bounds reads land in harmless memory — making the check unnecessary. It's like installing a soft bumper so you don't need to keep stopping to verify you haven't hit a wall.

Where it breaks: the bumper (padding) takes extra space; ATiM trades a bit of memory for significant execution speedup (up to 23.7%).
Code generation · DMA intrinsics

Moving furniture in bulk

Moving one box at a time is slow; renting a truck and loading everything at once is fast — but you can't do it if you need to check each item individually. ATiM's boundary check elimination clears the way for DPU DMA engines to transfer data in bulk (64-element chunks instead of 1-by-1 loads). The code generator translates high-level cache_read primitives into these DMA intrinsics after verifying conditionals can be safely removed, turning slow scalar loops into fast vectorized transfers.

Where it breaks: you can't rent a truck for every single item; ATiM only uses DMA when tile access patterns are contiguous and boundary-safe.
The full explanation

How it actually works

Section by section, with key steps and design choices explained.

1 · The joint search space

ATiM extends TVM's schedule primitives to define tunable host and kernel operations. For host: split and reorder define tile shapes, bind maps tiles to DPUs, and rfactor enables hierarchical reduction (partial reduction per DPU, final reduction on host). For kernel: split and reorder perform multi-level tiling, bind assigns tasklets, and cache_read/write + compute_at specify caching locations and sizes.

Example: splitting a 8192-row GEMV into 16 DPU tiles of 512 rows each, then inside each DPU tiling into 16 tasklets × 32-element cache chunks. Both levels are sampled during autotuning — ATiM searches tile sizes, DPU counts, tasklet parallelism, and caching granularity together.

2 · Balanced evolutionary search

The expanded search space introduces a bias: with ≫ more DPUs than tasklets per DPU (2048 vs. 24), the autotuner can favor inter-DPU parallelism (rfactor schedules) over intra-DPU optimizations, prematurely dropping non-rfactor candidates. ATiM corrects this with two mechanisms:

  1. Balanced sampling: during the first 40% of autotuning trials, extract equal numbers of top-K candidates from rfactor and non-rfactor design spaces, ensuring both are explored.
  2. Adaptive epsilon-greedy: start with high epsilon (0.5) to allow diverse candidates into the best-candidate database, then linearly decay to 0.05 after 40% of trials to accelerate convergence.

Combined, these techniques prevent the search from collapsing to a local optimum, delivering 21.2% higher final performance than TVM's default search.

3 · TIR lowering and code generation

Once a schedule is selected, ATiM lowers it to loop-based TIR, then to separate host and kernel programs:

  • Address calculation: map global tensor indices to per-DPU MRAM offsets by leveraging the local scope introduced by cache_read/write — base address (outer-loop index × tile size) + per-DPU offset (inner-loop index).
  • Data transfer code: generate host-to-device and device-to-host transfer loops, then apply bulk transfer (unroll contiguous loops into single DMA call) and bank-parallel transfer (for UPMEM's parallel data transfer APIs) optimizations.
  • Reduction code: translate rfactor primitives to hierarchical reduction: parallel partial reduction across DPUs, final reduction on host (which can be further tiled and parallelized on multi-core CPUs).

4 · PIM-aware optimizations

Three tensor-level passes target boundary checks — conditionals inserted when tile sizes don't perfectly divide tensor dimensions:

  1. DMA-aware boundary check elimination: remove checks guarding load/store to WRAM by locally padding MRAM tiles, making out-of-bounds accesses safe. This enables replacing scalar loops with DMA intrinsics.
  2. Loop-bound tightening: merge boundary conditions with loop upper bounds by solving linear inequalities in compile time, shrinking the loop to skip iterations that would fail the check.
  3. Invariant branch hoisting: move loop-invariant boundary checks outside the loop, then integrate with partial dead code elimination (PDCE) to hoist dependent DMA instructions under the condition, further reducing dynamic branch count (by 40× in paper's example).

Combined, these passes provide up to 23.7% speedup on UPMEM by eliminating branch stalls — an optimization infeasible for low-level compilers without high-level semantic knowledge.

Follow the numbers, step by step

Worked examples

Three traces using the paper's real numbers — schedule generation, address calculation, and boundary check elimination.

① Generating a schedule for 8192×8192 GEMV

How does ATiM sample a candidate schedule for C[i] = A[i,j] · B[j]?

  1. Host tiling: split the 8192-row loop into (DPU, row_per_dpu) = (128, 64). Bind DPU loop to blockIdx.x → distributes 128 tiles to 128 DPUs.
  2. Reduction strategy: split the 8192-column loop into (col_dpu, col_in) = (16, 512). Apply rfactor on col_dpu → parallel reduction across 16 chunks, each DPU reduces 512 elements.
  3. Kernel tiling: split row_per_dpu into (tasklet, row_cache, row_in) = (16, 4, 1) and col_in into (col_cache, col_in_inner) = (64, 8). Bind tasklet to threadIdx.x → 16 tasklets per DPU.
  4. Caching: cache_read(A, "local") at col_cache, cache_read(B, "local") at col_cache, cache_write(C, "local") at row_cache. Each tasklet fetches 64×8 A tiles and 64×1 B tiles per iteration.
  5. Result: 128 DPUs × 16 tasklets, each processing 4 rows × 64 columns at a time, with intermediate outputs partially reduced per DPU, then aggregated on host.

② Address calculation for tiled data

How does a DPU kernel access A[i,j] when A is tiled across DPUs?

  1. Global indexing: A is 8192×8192, tiled into 128 DPU chunks of 64×8192. DPU index = blockIdx.x; row offset in DPU = threadIdx.x × 4 + row_cache.
  2. Local MRAM address: global row = blockIdx.x × 64 + threadIdx.x × 4 + row_cache. Global address = A_base + (global_row × 8192 + j) × sizeof(float).
  3. WRAM caching: cache_read creates A_local buffer of size 64×8 floats. Base address for WRAM: A_local_base + (row_cache × 8 + col_in_inner) × sizeof(float).
  4. Lowering: ATiM's address calculation pass inserts outer-loop index (blockIdx.x × 64) as base, inner-loop index (threadIdx.x × 4 + row_cache) as per-DPU offset, all resolved at compile time.

③ Eliminating a boundary check for GEMV

Apply DMA-aware boundary check elimination + loop-bound tightening for a 7×40 GEMV with 2×16 tiling.

  1. Baseline: kernel caches 16 elements per iteration; inner loop has boundary check if (j * 16 + k < 40) to avoid overflow. This check repeats 96 times (3 outer × 2 rows × 16 inner).
  2. DMA-aware elimination: remove the check for load/store to WRAM by locally padding MRAM tiles to 16-byte alignment — out-of-bounds reads land in safe padding. Replace scalar loop with DMA intrinsic → 96 1-by-1 stores become 12 bulk transfers.
  3. Loop-bound tightening: merge remaining boundary check (for computation) with loop upper bound. Solve: k < 16 AND j * 16 + k < 40 ⇒ k < min(16, 40 − j * 16). Shrinks loop bounds to k < 16 for j=0, k < 16 for j=1, k < 8 for j=2 → total 40 iterations (16+16+8) instead of 48 (3×16).
  4. Result: branch count drops from 288 to 6; DMA count drops from 96 to 6; compute iterations 96 → 40. Total speedup ≈ 20% (paper Fig. 8).
Test yourself

Practice problems

Try each before opening the answer. Every problem ends with the misconception it's checking.

Problem 1 · easy
Why can't you just autotune the DPU kernel code independently and use a fixed host tiling strategy?
Show answer
Because host tiling determines how many tiles each DPU gets, which directly affects kernel execution time and memory footprint. If the host distributes too few large tiles, each DPU runs sequentially with no intra-DPU parallelism; if it distributes too many small tiles, data transfer overhead dominates. The optimal kernel structure (caching size, tasklet count) depends on tile size — so you must search both together.Misconception: that kernel optimization is separable from data distribution. For DRAM-PIM, inter-DPU and intra-DPU optimizations are tightly coupled.
Problem 2 · medium
ATiM's balanced sampling ensures equal representation of rfactor and non-rfactor candidates for the first 40% of trials. What happens if you remove this balancing?
Show answer
The search biases toward rfactor candidates (which exploit inter-DPU parallelism) because there are orders of magnitude more DPUs than tasklets per DPU (2048 vs. 24). Non-rfactor candidates (which rely on intra-DPU optimizations) get dropped early, and the search converges to a suboptimal local optimum that under-utilizes tasklet parallelism. The paper shows this results in 21.2% lower final performance.Misconception: that evolutionary search naturally balances exploration. Without explicit balancing, the search can collapse to over-explore one region of the space.
Problem 3 · medium
A DPU kernel has a boundary check if (i < N) inside a loop. Why can't LLVM optimize this away with loop-invariant code motion (LICM)?
Show answer
LICM can hoist the computation of the boundary condition (i < N) outside the loop if i and N are loop-invariant. But it cannot remove the conditional branch itself — that would require proving the branch always evaluates the same way, which LLVM cannot do without knowing the relationship between loop bounds and tensor dimensions. ATiM's loop-bound tightening does have this knowledge (from TIR semantics) and can tighten the loop upper bound to eliminate the branch entirely.Misconception: that low-level compilers can eliminate all redundant branches. High-level semantic knowledge is often required for aggressive optimization.
Problem 4 · hard
For a 4096×4096 GEMV, ATiM chooses 256 DPUs with 16 tasklets and 2D tiling (32×64 tiles per DPU), while PrIM uses 256 DPUs with 24 tasklets and 1D tiling (16 rows per DPU, full column). Estimate the performance difference and explain which configuration wins.
Show answer
PrIM's 1D tiling assigns 16 rows × 4096 columns to each DPU — each tasklet processes one row sequentially, no reduction parallelism. Data transfer: 16 × 4096 floats (256 KB) per DPU. ATiM's 2D tiling assigns 32×64 tiles — partial reduction on 64 columns, with 16 tasklets parallelizing the 32 rows. Data transfer per DPU: 32 × 64 floats (8 KB A matrix, plus 64-element B vector). Result: ATiM transfers 32× less data per DPU, enables hierarchical reduction to save host aggregation cost, and fully utilizes tasklet parallelism. Paper shows ≈4× speedup for this scenario.Misconception: that more tasklets (24 vs. 16) always wins. ATiM's 2D tiling and reduction strategy outweigh PrIM's higher tasklet count.
Problem 5 · medium
ATiM's DMA-aware boundary check elimination removes checks by padding MRAM tiles. What's the memory overhead for a 8192×8192 GEMV tiled into 128×64 chunks?
Show answer
For a 8192×8192 tensor with 128×64 tiling, 8192 is exactly divisible by both 128 and 64, so no padding is needed in this case. Padding overhead only applies when tile sizes don't perfectly divide tensor dimensions. For example, a 8100×8100 tensor with 128×64 tiles would require padding the last row-tile to 128 elements and last column-tile to 64 elements. The overhead depends on the specific mismatch between tensor size and tile size, and is typically negligible (<1% for large tensors) compared to the 20+% speedup from eliminating branches and enabling DMA.Misconception: that padding always incurs significant overhead. For most real-world tensor sizes, the memory cost is minimal.
Quick answers

Frequently asked questions

Ten questions newcomers ask most. Tap any to expand.

Q1What makes DRAM-PIM different from GPU offloading?▾
GPUs have hardware schedulers to manage threads, a unified memory abstraction, and large register files. DRAM-PIM (UPMEM) has no runtime scheduler — you must explicitly distribute data to 2048 DPUs and manage each DPU's 64 KB WRAM manually. Data movement is also different: GPUs use PCIe; DRAM-PIM accesses data in-place at internal memory bandwidth. The programming model is closer to distributed computing (MPI) than GPU programming (CUDA).
Q2Why not just use a heuristic for host-to-DPU tiling instead of autotuning?▾
Heuristics (e.g., "tile to maximize DPU utilization") work for some cases but fail for others. For small tensors, using all 2048 DPUs can add more overhead than benefit; for operations with reduction dimensions, 1D tiling leaves parallelism unexploited. Autotuning discovers configurations that depend on tensor shape, operation type, and hardware constraints — rules that are hard to codify in a heuristic. Paper shows autotuning beats heuristic-based PrIM+search by 1.84× on average.
Q3How does ATiM know which boundary checks are safe to eliminate?▾
ATiM leverages TensorIR's structural guarantees: after schedule primitives are lowered to loop-based TIR, all consumers of a caching loop are under that loop's constraints. This means boundary checks guarding WRAM loads/stores can be removed if MRAM tiles are padded (out-of-bounds reads land in padding, not other data). Checks guarding computation must remain until proven redundant by loop-bound tightening. It's a multi-step analysis using both IR structure and hardware knowledge.
Q4Does ATiM work for other DRAM-PIM architectures besides UPMEM?▾
The core ideas — joint host/kernel search, schedule primitive abstraction, PIM-aware optimizations — are general. But ATiM's current implementation is UPMEM-specific: DPU binding semantics, MRAM/WRAM memory hierarchy, DMA intrinsics. Extending to HBM-PIM (Samsung) or GDDR6-AiM (SK Hynix) would require redefining bank mapping policies, adding vector intrinsics for MAC units, and adjusting the autotuner for different constraints (e.g., two-bank PUs vs. one-bank DPUs). Paper discusses this in Section 8.
Q5What's the autotuning overhead — how long does it take to compile?▾
For 1000 trials, ATiM takes 11.5–18.5% longer than CPU autotuning due to (1) measuring on hardware (UPMEM) vs. CPU, (2) larger search space (joint host/kernel). However, the final result is often 6× faster than hand-tuned code, so the overhead is justified. For production use, you autotune once per operation/shape, save the result, and reuse it. Paper Fig. 15 shows per-iteration breakdown.
Q6Can ATiM optimize graph-level (multi-kernel) workloads?▾
Currently, ATiM optimizes individual tensor operations (single kernel). For multi-layer models (e.g., GPT-J), you must autotune each operation separately. Future work (paper Section 8) includes extending ATiM's frontend to perform inter-kernel optimizations like layer fusion and reordering to reduce DPU launch overhead and intermediate data transfers. This would require integrating with TVM's Relay or Relax graph IR.
Q7How does hierarchical reduction (rfactor) reduce host overhead?▾
The paper shows that 2D tiling with rfactor reduces both host-to-device data transfer and kernel execution time by enabling parallel reduction across DPUs. This approach can slightly increase post-processing time on the host (due to hierarchical aggregation), but the overall benefit comes from reduced data movement and better parallelism. The exact speedup depends on the tensor shape and reduction dimensions — for MTV/GEMV, the paper reports up to 8.37× speedup with hierarchical reduction.
Q8Why does ATiM beat CPU-autotuned TVM by 23× for large tensors?▾
CPUs are bottlenecked by DRAM bandwidth — even with multi-core parallelism and cache optimizations, they must shuttle data through limited memory channels. DRAM-PIM computes in-place at internal memory bandwidth (up to 16× higher according to the paper), with bank-level parallelism across 2048 DPUs. For ≥64 MB tensors, ATiM's data movement savings outweigh CPU's compute efficiency. For <4 MB tensors, CPU wins due to lower parallel overhead.
Q9Can I use ATiM without knowing TensorIR or schedule primitives?▾
ATiM currently requires users to write tensor operations at the TensorIR level. The paper lists integration with high-level frameworks (Relay, ONNX) as future work — this would allow users to define operations in familiar APIs and have ATiM automatically lower and optimize them. For now, basic TensorIR knowledge is needed to specify operations, though ATiM's autotuner handles schedule generation automatically once the computation is defined.
Q10What about SMT (simultaneous multithreading) or multi-threaded workloads on DPUs?▾
UPMEM DPUs support up to 24 tasklets (hardware threads) per DPU, which ATiM exploits via tasklet binding for data parallelism. The paper does not discuss extending ATiM to handle more complex multi-threaded workloads with shared data and synchronization primitives — the current focus is on data-parallel tensor operations where tasklets process independent data tiles.

TensorIR design paper — separation of computation and schedule, and how loop transformations are applied during TIR lowering.

Benchmarking a New Paradigm: Experimental Analysis and Characterization of a Real Processing-in-Memory System

Gómez-Luna et al. · IEEE Access 2022 (PrIM)

Hand-tuned UPMEM benchmark suite — the baseline ATiM beats by up to 6×.

SimplePIM: A Software Framework for Productive and Efficient Processing-in-Memory

Chen et al. · PACT 2023

High-level UPMEM programming framework with map/reduce primitives — compared in paper as another baseline.

CINM: A Compilation Infrastructure for Heterogeneous Compute In-Memory and Compute Near-Memory Paradigms

Khan et al. · arXiv 2023

MLIR-based PIM compiler for multiple architectures — supports tensor operations but lacks autotuning integration.
