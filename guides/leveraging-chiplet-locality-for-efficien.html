Page size is not one-size-fits-all in MCM GPUs

Multi-chip GPUs let each chiplet hold its own memory. Default memory mapping can exile a thread's data to the wrong chiplet—but CLAP discovers each workload's natural page grouping and maps it accordingly, delivering both translation speed and data locality.

📄 CLAP (Park et al., MICRO '25) 🎓 GPU memory-system basics assumed ⏱ ~25 min read
🗂️

Map contiguous pages that are accessed by the same chiplet to contiguous frames in that chiplet—then treat the group as a large page.
Measure how pages cluster ("chiplet-locality"), decide the cluster size, then map it in one go.

Before you begin

Quick glossary

Six core acronyms and four key concepts the paper uses. Expand any that look unfamiliar.

fine-grained / high locality intermediate / adaptive CLAP mechanisms coarse / large pages
1MCM GPU — multi-chip module▾

A multi-chip module (MCM) GPU integrates multiple GPU chiplets in a single package. Each chiplet has local memory partitions. A thread can access local memory (same chiplet, ∼low latency) or remote memory (another chiplet, ∼higher latency + off-chip BW cost). The paper's baseline uses four chiplets.

Now you're ready: CLAP tackles the NUMA (Non-Uniform Memory Access) challenge: keeping data near the threads that use it.
2Page & page size▾

Virtual memory divides the address space into fixed-size blocks called pages. Common sizes: 4KB, 64KB, 2MB. A large page (e.g., 2MB) maps a big virtual region to one physical frame—reducing TLB pressure but forcing coarse-grained placement. A small page (e.g., 64KB) allows fine-grained placement but can strain address translation (more TLB misses).

Now you're ready: CLAP picks page size (really, the granularity of contiguous-page groups) per data structure to balance translation efficiency and locality.
3Chiplet-locality▾

Chiplet-locality: GPU workloads often access data in patterns where specific groups of virtually contiguous pages are primarily used by threads on the same chiplet. The size of these groups (e.g., 64KB, 256KB) tends to remain consistent within a single data structure. This stems from parallel programming models where adjacent threads work on adjacent data.

Now you're ready: CLAP measures chiplet-locality and uses it to decide page-group size—so pages that belong together get mapped together.
4First-touch (FT) policy▾

On a page fault, the first-touch policy places the faulting page in the chiplet where the first accessing thread resides. This simple heuristic often captures locality—the first thread is likely on the chiplet that will use the page most. The paper uses FT as its baseline data-placement strategy, combined with contiguous threadblock scheduling.

Now you're ready: FT naturally exposes chiplet-locality during CLAP's profiling phase, letting CLAP see which pages cluster together per chiplet.
5PMM — partial memory mapping▾

PMM is CLAP's lightweight profiling step. At kernel launch, the driver maps only a fraction (default: 20%) of each data structure using small pages (64KB). This creates a sample mapping that reveals how pages are accessed across chiplets, without paying the full cost of mapping everything up front.

Now you're ready: PMM gathers the usage pattern; MMA (memory mapping analysis) then picks the right page size; the remainder of the structure is mapped accordingly.
6MMA — memory mapping analysis▾

MMA is CLAP's decision engine. After PMM completes, MMA examines the sample mapping using a tree-based algorithm. It computes a locality score at each page-group size (64KB, 128KB, 256KB, …) and picks the largest size where virtually all pages land in the chiplet they're supposed to. Hardware counters (from the Remote Tracker) refine this choice if the structure is inherently shared.

Now you're ready: MMA outputs "use 256KB groups for this structure"—and the driver maps the rest in 256KB contiguous chunks.
7RT — Remote Tracker▾

A small hardware table (32 entries, ∼288 bytes) in each chiplet's GMMU. RT logs access count and remote-access count for each data structure during PMM. When MMA runs, it aggregates RT stats across chiplets to compute the remote-access ratio. If a structure is inherently shared (high remote ratio), MMA relaxes locality requirements and favors larger pages.

Now you're ready: RT's counters let CLAP distinguish "shared by design" (pick large pages) from "misplaced by accident" (use fine-grained groups).
8TLB coalescing▾

Because CLAP deliberately maps virtually contiguous pages to physically contiguous frames, the hardware can coalesce up to 16 contiguous PTEs (fetched in one cache line) into a single TLB entry. This entry covers the full group (up to 1MB if 16×64KB pages). Result: the group behaves like a large page for translation—low TLB pressure—without losing fine-grained placement control.

Now you're ready: Coalescing is the payoff—CLAP's mapped groups act as large pages for address translation, but they're built from small pages placed exactly where they're needed.
9OLP — opportunistic large paging▾

During PMM, if all 64KB pages in a 2MB VA block happen to land in the same chiplet, the driver promotes that block to a true 2MB page. If pages land in different chiplets, the driver releases the 2MB reservation and falls back to separate 64KB mappings. This recovers large-page benefits when locality is naturally strong, without waiting for MMA.

Now you're ready: OLP is an optimization that speeds up clear-cut cases during profiling; MMA handles the complex ones afterward.
10Remote access ratio vs. locality score▾

The remote access ratio (from RT) tells you what fraction of page-walk accesses to a structure hit remote chiplets during PMM. The locality score (from MMA's tree) tells you what fraction of pages in a candidate group are mapped to the intended chiplet. MMA combines both: score_avg(level) ≥ threshold − remote_ratio. High remote ratio lowers the bar for accepting a larger page size.

Now you're ready: If a structure is shared (high remote ratio), MMA selects a bigger group size; if it's private (low remote ratio), MMA picks the tightest fit that preserves locality.
🧩
Connecting to your work: If you've dealt with Linux's autonuma or THP (transparent huge pages) on CPUs, CLAP is the GPU analog—but GPU parallelism makes memory access patterns more predictable, so CLAP can profile once (during PMM) rather than react continuously. If you've used NVIDIA UVM, CLAP lives in that same demand-paging layer, deciding page size rather than just placement. If you've thought about CXL-attached memory, CLAP's chiplet-locality idea applies wherever you have tiered or NUMA memory with page-level control.
The whole story in one place

CLAP in a nutshell

The problem, the measurement that solves it, and the payoff.

In a 4-chiplet MCM GPU, threads on chiplet-0 can access memory mapped to chiplet-1, chiplet-2, or chiplet-3—but those remote accesses incur higher latency and consume off-chip bandwidth. To minimize remote traffic, data should land near the threads using it. Conventional wisdom: use small pages (e.g., 64KB) so each page can be placed precisely. But small pages strain address translation—more TLB misses, more page walks. Large pages (e.g., 2MB) ease translation but force a contiguous 2MB virtual region to map to a single chiplet, even if different sub-regions are used by different chiplets. This coarse-grained placement can exile data to the wrong chiplet.

Prior work either assumes a fixed page size for the entire system (suboptimal for mixed workloads) or tries runtime page migration (expensive TLB shootdowns, cache flushes). CLAP takes a different approach: it predicts the right page size (really, the right page-group granularity) per data structure, then maps pages in that granularity from the start—no migration needed.

The key insight is chiplet-locality: GPU workloads naturally access data in chunks where adjacent pages are used by the same chiplet, and the chunk size tends to be consistent within a data structure. CLAP profiles this during PMM (partial memory mapping), where 20% of each structure is mapped with small pages in a sample run. A tree-based algorithm in MMA analyzes how those sample pages cluster per chiplet and selects the largest group size where nearly 100% of pages land in their intended chiplet. (Hardware counters—Remote Tracker—refine the choice if a structure is inherently shared.) The driver then maps the remaining 80% of the structure in groups of the selected size, reserving contiguous physical frames in the target chiplet.

Because the groups are deliberately contiguous in both virtual and physical space, the hardware can coalesce up to 16 PTEs into a single TLB entry. Result: these groups behave like large pages for address translation (low TLB pressure) while preserving fine-grained per-chiplet placement. If a structure's optimal size is exactly 2MB, CLAP promotes it to a true 2MB page; otherwise, it uses coalesced 64KB entries.

The payoff: On 15 workloads spanning irregular graph algorithms and ML kernels, CLAP delivers an average 17.5% speedup over static 64KB paging and an average 19.2% over static 2MB paging, while incurring under 1% area overhead (RT: 0.0124 mm²; coalescing logic: 0.0024 mm², totaling ∼0.0148 mm²; extended L2 TLB area increase: 8.98%). The design is orthogonal to remote caching schemes (NUBA, SAC), boosting their gains by 18–19 percentage points when combined.

🍯 Key takeaways

  • Page size is critical: 2MB pages reduce TLB pressure but can force data onto the wrong chiplet (up to 100% remote access in worst cases). 64KB pages preserve locality but suffer high translation overhead. The right answer depends on workload.
  • Chiplet-locality is real: 93.5% of GPU data structures (by address range) exhibit consistent page-grouping patterns—adjacent pages accessed by the same chiplet, with group size stable per structure.
  • CLAP profiles once, maps forever: PMM (20% sample) + MMA (tree-based scoring) determines the optimal page-group size (64KB to 2MB, including intermediate sizes like 256KB). No runtime migration.
  • Coalescing delivers large-page benefits: Deliberately contiguous mappings let the TLB merge 16 PTEs → 1 entry (up to 1MB coverage), acting as a large page without the placement penalty.
  • Hardware cost is tiny: RT adds 0.0124 mm²; coalescing logic adds 0.0024 mm²; extended L2 TLB increases area by 8.98% (but yields 4× the benefit of an equivalent-area TLB size bump).
  • Orthogonal to caching: CLAP reduces remote traffic before caching kicks in, amplifying the effectiveness of prior schemes (NUBA +18.1pp, SAC +19.8pp).
Six everyday pictures

How to think about CLAP

Each picture stands alone. Every card ends with where the analogy stops being true.

The core idea · Chiplet-locality

Warehouse aisles

A warehouse has four zones (chiplets). A shipment (data structure) contains 100 boxes (pages). Your workers (threads) in zone A will pick boxes 1–50; zone D picks 51–100. A naive system stacks all 100 boxes in zone A (one big pallet = 2MB page). Now zone-D workers walk across the warehouse for every box. CLAP says: split the shipment into two pallets (128KB groups), place each in its zone. Workers stay local; forklifts (interconnect) idle.

Where it breaks: Real workloads aren't perfectly split 50/50. CLAP handles uneven access (e.g., 80% by one chiplet, 20% by another) by weighting the locality score with remote-access stats from RT.
Profiling · Tree-based scoring

Stacking boxes, then measuring fit

You have 32 small boxes (64KB pages) to ship. Stack them in fours (4×64KB = 256KB candidate group). MMA's tree algorithm checks: do all four boxes in each stack go to the same city? If yes (score = 1.0), that stack is a valid group. If three go to NY and one to LA (score = 0.75), the stack is broken. CLAP picks the largest stack size where (almost) every stack is valid—that's your page-group size.

Where it breaks: Real trees are built bottom-up (64KB leaf nodes → 128KB internal nodes → 256KB → …). MMA computes score_avg(level) and picks the highest level meeting the threshold. The analogy simplifies the averaging.
Lightweight profiling · PMM

Tasting the stew

You don't eat the whole pot to check seasoning—you taste a spoonful. PMM maps 20% of each data structure using small pages and the first-touch policy. That sample reveals which pages land in which chiplet. A well-stirred pot (low variance) means a small sample suffices; CLAP uses 32 sampled sets (of 1024) in the underlying TLB monitor. The remaining 80% is then mapped in the chosen page-group size, without further sampling.

Where it breaks: Stew seasoning is uniform; GPU data structures can have phase changes (though GPU schedulers don't migrate threads, so patterns are stable within a kernel). If a structure changes usage later, CLAP's static choice may lag—but the paper shows this is rare (<1% workloads).
Translation efficiency · Coalescing

Bundle discounts

A store charges $1 per item, but if you buy four in a bundle, you get a single receipt covering all four (one transaction). TLB coalescing does this for page translation: 16 contiguous PTEs (fetched in one 128B cache line) → 1 TLB entry. If CLAP maps a 256KB group as 4×64KB pages, the TLB can bundle them into one entry (instead of four). You get large-page-like TLB coverage without losing per-64KB placement control.

Where it breaks: The store analogy suggests pure cost savings. In reality, coalescing also requires contiguity—both virtual and physical. CLAP ensures this via reservation. If pages aren't contiguous (e.g., interleaved placement), coalescing can't happen.
Hardware counters · Remote Tracker

Separate thermometers

To measure room temperature accurately, you don't open the door to the adjacent room—that contaminates your reading. RT (Remote Tracker) logs access counts per data structure in each chiplet's GMMU, isolated from other chiplets. If structure X is accessed 100 times by chiplet-0 and 20 of those are remote, RT records remote_ratio = 0.2. MMA uses this to relax the locality threshold: score ≥ (1.0 − 0.2), favoring larger pages for inherently shared structures.

Where it breaks: A thermometer is passive; RT actively counts page-walk requests. It's on the page-walk path (GMMU), not the data-access path—so it's cheap (2-cycle lookup) but only tracks translation, not every memory load. That's enough to estimate remote ratio (95.3% similarity to actual).
No migration · One-shot mapping

Assemble-once furniture

You buy a bookshelf. Option A: assemble it, realize it's in the wrong room, disassemble, move, reassemble (migration overhead—TLB shootdown, cache flush). Option B: measure the space first, then assemble it in the right room (CLAP). CLAP profiles during PMM (20%), decides the page-group size via MMA, then maps the remaining 80% in that size—no reassembly needed.

Where it breaks: If usage changes mid-execution (rare—GPU threads don't migrate), the one-shot mapping may be suboptimal. CLAP+migration (Section 5.2) adds selective migration for multi-kernel workloads where a structure's access pattern flips, but the baseline avoids migration entirely.
The full explanation

How it actually works

Step by step, with the paper's key mechanisms rebuilt as diagrams.

1 · The problem — page size is a Goldilocks dilemma

In a 4-chiplet MCM GPU, each thread can access any memory partition, but remote accesses cost more. The GPU driver decides two things for each page fault: (1) where to place the page (which chiplet's DRAM), and (2) how big the page is. Prior work focuses on (1)—first-touch, static analysis, remote caching—but largely assumes a fixed page size for (2). This leaves a gap.

With small pages (64KB), each page can be placed precisely in the chiplet that uses it. But 256 SMs generating faults → heavy TLB pressure. Address translation latency rises. With large pages (2MB), one TLB entry covers 2MB → fewer TLB misses. But the driver must place a contiguous 2MB virtual region into a single chiplet. If different 64KB sub-regions within that 2MB are accessed by different chiplets, some threads now make remote accesses.

📐
Concretely (Fig. 1 in paper): STE (stencil) with 2MB pages: remote access ratio → 75% (vs. 17% with 64KB). Performance drops 0.5×. With 64KB pages: remote ratio → 17%, but TLB MPKI → 1.38 (vs. 0.55 with 2MB). Slight perf gain but not maximal. The paper finds STE's sweet spot is 256KB pages—not a system-supported size. CLAP provides it.

2 · Chiplet-locality — GPU workloads naturally cluster

The paper observes that GPU data structures exhibit chiplet-locality: groups of virtually contiguous pages are primarily accessed by threads on the same chiplet, and the group size (e.g., 4×64KB = 256KB) tends to be consistent within a structure. Why? GPU programming models (CUDA, HIP) assign adjacent threads (within a threadblock) to work on adjacent data regions. When threadblocks are scheduled contiguously within a chiplet (FT policy), adjacent pages naturally cluster per chiplet.

Measurement (Section 3.4): Using 64KB pages and FT placement, the paper profiles 15 workloads. For each data structure ≥2MB, it measures what fraction of the address range exhibits chiplet-locality (i.e., groups of pages accessed by the same chiplet). Result: 93.5% average across all structures. This is the key insight: if you know the group size, you can map pages in that granularity and preserve locality.

3 · Partial Memory Mapping (PMM) — the lightweight profiling step

At kernel launch, the driver begins mapping data structures but stops after a predefined fraction (default: 20%) has been faulted in. During this phase, the driver uses 64KB pages and the first-touch policy: each faulting page is placed in the chiplet where the faulting thread resides. This creates a sample mapping that naturally exposes chiplet-locality.

Simultaneously, the driver registers each data structure with the Remote Tracker (RT), a small hardware table (32 entries, 288B) in each chiplet's GMMU. RT logs access_count and remote_count for each structure during PMM. When the page walker retrieves a PTE, RT extracts the allocation ID (stored in the PTE's unused bits) and checks if the page is mapped to the requester's chiplet or a remote one. If remote, RT increments remote_count.

💡
Why 20%? Empirically, 15% suffices for the evaluated workloads, but the paper adds a 5% margin for robustness. The paper shows performance is insensitive to the threshold: increasing to 30% causes only 1.3% average degradation. The key is that chiplet-locality is stable—so a small sample predicts the whole structure.

Opportunistic Large Paging (OLP): If all 64KB pages in a 2MB VA block land in the same chiplet during PMM, the driver promotes that block to a true 2MB page. If pages land in different chiplets, the driver releases the 2MB reservation. This recovers large-page benefits when locality is naturally strong, without waiting for MMA.

4 · Memory Mapping Analysis (MMA) — tree-based scoring

Once PMM completes (20% mapped), the driver runs MMA to pick the page-group size. MMA examines VA blocks where all 64KB subpages are fully mapped and builds a tree structure per 2MB region. Leaf nodes represent 64KB pages; each node stores the chiplet ID where its page landed. Internal nodes aggregate child counts.

At each tree level (128KB, 256KB, 512KB, 1MB, 2MB), MMA computes a locality score:

score(level) = max(C₀, C₁, C₂, C₃) / #leaf_nodes_below
# max count of pages mapped to any single chiplet,
# divided by total pages at that level

A score of 1.0 means all pages at that level landed in the same chiplet (perfect locality). MMA then computes score_avg(level) across all VA blocks in the structure and picks the highest level where:

score_avg(level) ≥ threshold − remote_ratio
# threshold = 1.0 by default
# remote_ratio from RT stats

If remote_ratio is high (e.g., 0.5), the threshold is relaxed to 0.5, allowing larger page sizes for inherently shared structures. The selected level becomes the page-group size for the remaining 80% of the structure.

🍯
Example (Fig. 15 in paper): A 512KB VA block with 8×64KB pages. At 256KB level, one internal node has all four children in chiplet-0 (score = 1.0); the other node has two in chiplet-1, two in chiplet-2 (score = 0.5). score_avg(256KB) = 0.75. If threshold = 1.0, MMA picks 128KB (next level down). If remote_ratio = 0.5, effective threshold = 0.5 → MMA picks 256KB.

5 · Applying the selected size — reservation & coalescing

MMA outputs a page-group size (e.g., 256KB). The driver then maps the remaining 80% of the structure in that granularity. For each 256KB virtual region, the driver reserves a contiguous 256KB physical frame in the target chiplet (the chiplet that scored highest in MMA). When a 64KB sub-region within that 256KB faults, the driver places it into the reserved frame. This ensures both virtual and physical contiguity.

At address translation time, the page walker fetches PTEs from the in-memory page table. Because GPUs use 128B cache lines, sixteen 8-byte PTEs arrive in one fetch. The TLB controller's coalescing logic detects contiguity (consecutive VPNs → consecutive PFNs) and merges them into a single TLB entry. That entry records the base VPN/PFN and a 16-bit valid mask (one bit per 64KB subpage). One coalesced entry can cover up to 1MB (16×64KB).

🍯
Hardware cost: Coalescing logic (Verilog-synthesized) adds ∼0.0024 mm² (0.0003% of GPU die). Extended L2 TLB to store valid masks increases TLB area by 8.98%, but the performance gain (average 17.5% vs. 64KB, 19.2% vs. 2MB) far exceeds the equivalent-area TLB size bump (which yields only 2.31% gain). (Paper Section 4.6)

If MMA selects 2MB, the driver promotes the group to a true 2MB page (already supported by NVIDIA GPUs [69]). If intermediate (128KB–1MB), the driver uses coalesced 64KB entries.

6 · Handling edge cases — small structures & phase changes

Small structures (<10MB): May not have enough 2MB VA blocks fully mapped during PMM. CLAP falls back to OLP, which dynamically promotes or falls back per 2MB block. Performance delta: 1.1% vs. ideally selected size (rare case).

Phase changes: GPU schedulers don't migrate threads across cores, so access patterns are stable within a kernel. If a structure is reused across kernels with different access, CLAP's static choice may lag. The paper evaluates an extension (CLAP+migration, Section 5.2) that selectively migrates only multi-kernel structures, achieving +34.7% in a challenging GEMM-reuse scenario. Baseline CLAP avoids migration entirely.

Chiplet memory exhaustion: If a chiplet's DRAM is full, CLAP maps pages to another chiplet (reduces locality but avoids deadlock). Rare in practice—structures are accessed in balanced manner by threads across chiplets, so mappings naturally distribute. If the system is oversubscribed (UVM), CLAP migrates least-recently-mapped page groups to host memory.

Follow the numbers, step by step

Worked examples

Three traces showing how CLAP handles different data structures. Each example uses the paper's reported input sizes, page configurations, and performance results.

① STE: intermediate page size (256KB) wins

STE (stencil) has a 128MB primary data structure. The paper (Table 4) reports that static 64KB pages yield TLB MPKI = 1.38 and performance = 1.0×, while static 2MB pages yield MPKI = 0.55 but remote_ratio = 75%, degrading performance to 0.52×. CLAP's MMA selects 256KB groups, balancing both.

  1. PMM (20%): 25.6MB mapped with 64KB pages via FT policy. RT logs remote accesses; the paper reports overall remote_ratio = 17% with 64KB pages.
  2. MMA tree analysis: MMA finds chiplet-locality at the 256KB level (groups of 4 contiguous 64KB pages accessed by the same chiplet). The paper shows 256KB is the optimal size for STE.
  3. Map remaining 80%: 102.4MB = 1600 pages remaining. Driver reserves 256KB frames (4×64KB contiguous) and maps pages into them.
  4. Result (Fig. 6, Fig. 18): CLAP achieves 1.85× performance vs. static 2MB and 1.32× vs. static 64KB. Average speedup across all workloads: 17.5% vs. 64KB baseline.
🍯
Insight: 256KB is not a hardware-supported page size in GPUs, but CLAP provides it via coalesced 64KB entries (4 pages merged into 1 TLB entry). STE's access pattern naturally groups pages in fours per chiplet.

② BFS: per-structure page sizing

BFS has multiple data structures with different access patterns. The paper (Table 4) shows that static 2MB pages yield high remote_ratio for some structures, while static 64KB pages have high TLB overhead. CLAP assigns different page sizes to different structures within the same workload.

  1. PMM completes: Each structure's 20% is mapped. The paper notes BFS's three largest structures have varying locality patterns.
  2. MMA per-structure decision: The paper reports that CLAP assigns 2MB to two structures and 64KB to the third (the visited bitmap, which exhibits mixed per-chiplet access). This heterogeneous assignment is key.
  3. Result (Fig. 8, Table 4): Static 2MB yields 0.58 remote_ratio and lower performance. Static 64KB yields 0.12 remote_ratio but higher TLB MPKI. CLAP achieves 0.11 remote_ratio with TLB efficiency comparable to 2MB—delivering 1.18× speedup vs. 2MB and 1.08× vs. 64KB.
💡
Why per-structure? Different data structures in the same application can have radically different access patterns. CLAP's profiling isolates each structure and assigns the optimal size independently—no single static configuration would optimize all three.

③ GPT3: shared matrix favors large pages

GPT3's fully connected layer performs GEMM: C = A × B. The paper (Table 4) notes that matrix B (shared across all threads) is best served by 2MB pages, as its inherently shared nature means fine-grained placement cannot reduce remote accesses—but large pages reduce TLB pressure.

  1. PMM sampling: Matrix B's 20% sample shows pages accessed by threads from all chiplets (no single chiplet dominates). RT logs a high remote_ratio.
  2. MMA decision: MMA detects the high remote_ratio and relaxes the locality threshold. Since no page size can eliminate remote accesses for a globally shared structure, MMA picks 2MB (the largest size) to maximize translation efficiency.
  3. Result (Fig. 6, Table 4): CLAP assigns 2MB to matrix B, matching the performance of static 2MB for this structure while maintaining per-structure flexibility. Overall, CLAP achieves 1.14× vs. 64KB baseline.
🌐
Shared structures: For data accessed uniformly by all chiplets (broadcast tensors, global constants), no placement strategy optimizes locality. CLAP detects this via RT's high remote_ratio and correctly selects the largest page size to reduce TLB overhead—the right trade-off when locality is fundamentally unattainable.

Practice problems

Try each before opening the answer. Every one ends with the common misconception it checks.

Problem 1 · easy
A data structure has 512 pages (64KB each). During PMM (20%), 100 pages are mapped. MMA finds that at the 256KB level (4-page groups), 24 of 25 groups have all four pages in the same chiplet. What is score_avg(256KB), and does it pass a threshold of 1.0?
Show answer
score_avg(256KB) = 24/25 = 0.96. Threshold = 1.0 → fails (0.96 < 1.0). MMA would pick the next-smaller level (128KB) where presumably score_avg ≥ 1.0. However, if remote_ratio = 0.05, effective threshold = 0.95 → 256KB passes. The threshold-relaxation is critical for borderline cases.Misconception: that 96% is "good enough" for 256KB. The default threshold is strict (1.0) to ensure high locality, but remote_ratio can relax it.
Problem 2 · medium
During PMM, a structure's pages are accessed 400 times (page walks logged by RT). 100 of those are remote. MMA finds score_avg(512KB) = 0.80. Does MMA select 512KB?
Show answer
Remote_ratio = 100/400 = 0.25. Effective threshold = 1.0 − 0.25 = 0.75. Since 0.80 ≥ 0.75, MMA selects 512KB. The high remote_ratio (25%) suggests the structure has some inherent sharing, so MMA accepts a slightly lower locality score to favor larger pages (which reduce TLB pressure).Misconception: that high remote_ratio always means "shared globally, pick 2MB." RT distinguishes between "this structure is inherently shared" vs. "pages are misplaced." If score_avg is still reasonable after threshold relaxation, MMA picks an intermediate size.
Problem 3 · medium
Two data structures: (X) 200MB, chiplet-locality at 128KB granularity; (Y) 50MB, global shared (accessed by all chiplets). What page sizes should CLAP assign?
Show answer
Structure X → MMA picks 128KB. Structure Y → remote_ratio will be high (≈0.75), so MMA relaxes threshold and picks 2MB (largest size, best translation efficiency). CLAP assigns different sizes to different structures within the same application—this is the key advantage over static schemes.Misconception: that the entire application must use one page size. CLAP's per-structure profiling and MMA enable heterogeneous page sizes, tailored to each structure's access pattern.
Problem 4 · hard
A structure's pages are accessed by threads that migrate across chiplets mid-kernel (hypothetical—GPU schedulers don't do this, but assume). How would CLAP's one-shot mapping fare?
Show answer
Poorly. CLAP profiles during PMM (early execution) and maps the rest statically. If threads migrate, the post-PMM access pattern differs from the sample. CLAP's mapping would be stale. Solution: CLAP+migration (Section 5.2) can selectively migrate page groups if usage changes. But in practice, GPU schedulers don't migrate threads—they're pinned to cores—so this scenario is academic. For multi-kernel workloads where a structure is reused with a different pattern, CLAP+migration applies.Misconception: that CLAP is purely static. The baseline is one-shot (no migration), but the paper shows an extension for dynamic cases. The key is that GPU workloads are stable within a kernel, so one-shot suffices most of the time.
Problem 5 · medium
Why does coalescing require both virtual and physical contiguity? Can't the TLB just track "these 16 VPNs, wherever they land physically"?
Show answer
Coalescing exploits the fact that consecutive VPNs map to consecutive PFNs. The TLB entry stores one base VPN, one base PFN, and a valid mask. On a hit, the hardware reconstructs PFN = base_PFN + (VPN − base_VPN). This arithmetic only works if physical frames are contiguous. If frames were scattered, you'd need to store 16 separate PFNs → no space savings, just complexity. CLAP's reservation ensures physical contiguity, enabling the coalescing trick.Misconception: that coalescing is just "merging TLB entries." It's an encoding optimization that requires a specific mapping structure—both virtual and physical contiguity—to work.
Problem 6 · easy
OLP (opportunistic large paging) is described as "free" during PMM. What's the catch?
Show answer
OLP reserves a 2MB physical frame for each 2MB VA block. If pages land in different chiplets, the reservation is released. But if >5% of VA blocks release their reservations, OLP is disabled (to avoid churn). Also, OLP only helps when locality is strong (all pages in a block cluster). For structures with mixed access, OLP falls back to separate 64KB pages, and you miss large-page benefits until MMA runs and picks the right intermediate size. OLP is "free" in the sense of no extra profiling cost, but it's not a panacea—it's a heuristic that works for clear-cut cases.Misconception: that OLP alone solves the problem. OLP is an optimization for the PMM phase; MMA is needed to handle the complex cases where OLP doesn't apply.
Problem 7 · hard
The paper compares CLAP to C-NUMA (CPU NUMA paging). Why does C-NUMA's reactive approach—migrating pages based on observed remote accesses—not work well for GPUs?
Show answer
Three reasons: (1) GPU access patterns are stable (threads don't migrate), so profiling once is enough—no need to react continuously. (2) Migration overhead (TLB shootdown, cache flush) is expensive on GPUs with 256 SMs; C-NUMA's frequent migrations degrade throughput. (3) C-NUMA adjusts page size globally (one size for the whole app), so it can't handle heterogeneous structures like BFS. CLAP is proactive (profile early, decide once, map in the right size from the start) and per-structure, avoiding migration and misfits.Misconception: that GPU NUMA is the same as CPU NUMA. The parallel programming model (fixed thread-to-core mapping, predictable access patterns) fundamentally changes the design space. CLAP exploits this difference.
Problem 8 · medium
RT logs page-walk requests, not every memory load. Why is this enough to estimate remote_ratio?
Show answer
Page walks occur on TLB misses, which correlate with memory access patterns (high working set → more misses). If 25% of page walks hit remote-mapped pages, it indicates roughly 25% of the accessed pages are remote. The paper validates this: RT's estimate has 95.3% similarity to the actual remote access ratio measured at the memory-load level. Also, RT is cheap (2-cycle lookup in the GMMU)—tracking every load would require propagating allocation IDs through the entire memory hierarchy, which is costly.Misconception: that RT undercounts remote traffic. It samples page walks, not individual loads, but the sample is representative. And MMA uses RT stats only to adjust the threshold, not as an absolute count—so minor error is tolerable.
Quick answers

Frequently asked questions

Ten questions newcomers ask most. Tap any to expand.

Q1Why is page size important in MCM GPUs but not monolithic GPUs?▾
In a monolithic GPU, all memory is equally distant—no NUMA effects. Large pages reduce TLB pressure without placement penalties. In MCM designs, each chiplet has local DRAM; large pages force 2MB regions into a single chiplet, even if sub-regions are used by different chiplets. This exile-to-remote-chiplet cost didn't exist before. CLAP addresses the new trade-off.
Q2Does CLAP require OS changes or is it purely a GPU driver mechanism?▾
CLAP's mechanisms (PMM, MMA, reservation, RT) are implemented in the GPU driver and hardware (GMMU). The paper describes CLAP's memory manager as residing in the GPU driver, with RT as a hardware extension. The paper does not explicitly state whether OS/kernel integration is required for a complete system deployment, but the core design operates at the driver level.
Q3Can CLAP coexist with remote caching schemes like NUBA or SAC?▾
Yes—orthogonal. CLAP reduces remote traffic via better placement; remote caching caches remote data locally. When combined, CLAP lowers the input to the cache (fewer remote accesses to begin with), so the cache is more effective. Figure 21 in the paper: CLAP+NUBA achieves +23.9% vs. NUBA alone (+4.8%). CLAP+SAC achieves +24.1% vs. SAC alone (+4.3%). The gains compound.
Q4What happens if PMM's 20% sample is unrepresentative of the remaining 80%?▾
The paper shows this is rare. GPU workloads have stable access patterns (threads are pinned to cores, adjacent threads access adjacent data). Chiplet-locality analysis across 15 workloads finds 93.5% of address ranges exhibit consistent grouping. If the sample is bad (high variance), Chebyshev bound (used in DSS sampling) flags it. In practice, 20% is conservative; 15% suffices, and the paper adds 5% margin. Edge case: if PMM fails, CLAP falls back to OLP (Section 4.5).
Q5Why not just use 4KB pages everywhere for maximum placement flexibility?▾
TLB pressure. A 1GB working set with 4KB pages = 262,144 entries. Even a 1024-entry L2 TLB covers only 4MB. With 64KB pages, same working set = 16,384 entries → 64MB coverage. With 2MB pages, 512 entries → 1GB coverage. Address translation latency dominates execution time when TLB miss rates are high. The paper shows 4KB pages degrade performance by up to 2× vs. 64KB. Fine-grained placement isn't worth it if you spend all your time translating addresses.
Q6Does CLAP support page sizes larger than 2MB (e.g., 512MB, 1GB)?▾
The baseline excludes them as too coarse-grained (a 512MB page in a 4-chiplet system would span multiple chiplets even with perfect locality). But CLAP's design is scalable (Section 4.7): by increasing VA/PF block size to 1GB and using 2MB as the base page, you could support 2MB–1GB groups. The tree-based algorithm and coalescing mechanism generalize. The paper doesn't evaluate this because no workload in their suite benefits from such large sizes.
Q7How does CLAP handle memory fragmentation?▾
Block-based management (Section 4.1): physical frames are carved from 2MB PF blocks. All frames in a block belong to the same structure. When the structure is deallocated, the entire block is reclaimed—no internal fragmentation. External fragmentation (PF blocks can't be re-split) is mitigated by per-structure dedicated free lists. The paper measures memory overhead: CLAP increases consumption by 0.57% (vs. 64KB pages) and 1.27% (vs. 2MB pages) on average—negligible.
Q8What if all chiplets' memory is full and a new page must be placed?▾
Two cases: (1) If the system is oversubscribed (UVM), CLAP migrates least-recently-mapped page groups to host memory, freeing space. (2) If not oversubscribed, CLAP maps the page to the chiplet with the most free space (reduces locality but avoids deadlock). The paper notes this is rare—GPU workloads access data in balanced manner across chiplets, so mappings naturally distribute (no one chiplet exhausts first). Section 4.7 discusses this.
Q9Can CLAP be applied to static-analysis–based threadblock/data placement (SA policy) instead of first-touch (FT)?▾
Yes. Section 5.2 evaluates CLAP-SA: replaces FT with LASP+SUV (static analysis). MMA still runs on the predicted placement. Result: +18.8% vs. SA-64KB, +16.1% vs. SA-2MB. CLAP-SA++ (hybrid: static analysis for regular workloads, runtime profiling for irregular) achieves +23.7%. The key is that MMA's tree-based scoring works on any mapping—whether from FT, SA, or another policy.
Q10Is CLAP applicable to CXL-attached memory or other tiered-memory systems?▾
The core principle—measuring access locality at different granularities and selecting page-group sizes accordingly—could apply to any system with NUMA-like tiers and page-level placement control. However, the paper evaluates CLAP exclusively on MCM GPUs. Extending to CXL or other tiered-memory architectures would require additional validation beyond the scope of the original work.

Glossary & further reading

Every term in one place, then a path to go deeper.

Core terms
MCM GPU (Multi-Chip Module GPU)A GPU design integrating multiple chiplets in a single package. Each chiplet has local memory partitions; threads can access remote partitions at higher latency.
Chiplet-localityThe tendency for groups of virtually contiguous pages to be primarily accessed by threads on the same chiplet. Group size (e.g., 256KB) tends to be consistent per data structure.
CLAP (Chiplet-Locality Aware Page placement)The paper's mechanism: profile chiplet-locality during PMM, decide page-group size via MMA, map pages in that granularity, coalesce PTEs for large-page-like TLB efficiency.
PMM (Partial Memory Mapping)Profiling phase: map 20% of each data structure with 64KB pages via FT policy. Creates sample mapping; RT tracks remote accesses. Endpoint triggers MMA.
MMA (Memory Mapping Analysis)Decision phase: tree-based algorithm examines sample mapping, computes locality scores at each page-group size, picks the largest size where score ≥ threshold − remote_ratio.
RT (Remote Tracker)Hardware table (32 entries, 288B) in each GMMU. Logs access_count and remote_count per data structure during PMM. Provides remote_ratio to refine MMA's threshold.
TLB coalescingHardware mechanism: merge up to 16 contiguous PTEs (fetched in one 128B cache line) into one TLB entry covering up to 1MB. Requires both virtual and physical contiguity (ensured by CLAP's reservation).
OLP (Opportunistic Large Paging)PMM optimization: if all 64KB pages in a 2MB VA block land in the same chiplet, promote to 2MB page. If not, release reservation. Recovers large-page benefits for clear-cut cases.
FT policy (First-Touch)Data placement heuristic: place each faulting page in the chiplet where the first accessing thread resides. Combined with contiguous threadblock scheduling, naturally exposes chiplet-locality.
Locality scoreMMA metric: at tree level L, score(L) = max(C₀,C₁,C₂,C₃) / #leaf_nodes_below. Fraction of pages at that granularity mapped to the intended chiplet. Score_avg(L) averaged across VA blocks.
Evaluation metrics (from the paper)
Remote access ratioFraction of memory instructions (or page walks) that hit pages mapped to a remote chiplet. Lower is better. CLAP targets reduction vs. static large pages.
TLB MPKITLB misses per 1000 warp instructions. Measures translation overhead. Large pages (and CLAP's coalescing) reduce MPKI.
Speedup (normalized perf.)Paper reports performance normalized to 64KB baseline. Examples: STE 1.85×, BFS 1.18×. Geometric mean across 15 workloads: average 17.5% speedup vs. 64KB, average 19.2% vs. 2MB.
Hardware overheadRT: 0.0124 mm² per chiplet. Coalescing logic: 0.0024 mm². Extended L2 TLB area increase: 8.98%. Total hardware overhead under 1% of GPU die area.

Understanding the baseline

MCM GPU architecture (Arunkumar+ ISCA'17) easy

Foundational · 20 min read

The original MCM GPU design paper. Characterizes NUMA effects, proposes FT-based placement. CLAP builds on this baseline (same FT policy, same interleaving—MSBs of channel bits outside page offset).

GPU virtual memory (UVM) primer easy

NVIDIA docs · 15 min

Unified Virtual Memory lets CPU and GPU share a single address space. CLAP integrates with UVM's demand-paging layer. Search "CUDA Unified Memory Programming" for official docs.

Related work

C-NUMA (Dashti+ ASPLOS'13) medium

CPU NUMA paging · reactive migration

Page-size adaptation via runtime migration on NUMA CPUs. CLAP compares favorably (Section 5.1): CLAP is proactive (profile once) and per-structure, avoiding migration overhead that GPU workloads can't amortize.

NUBA (Zhao+ ASPLOS'23) & SAC (Zhang+ ISCA'23) medium

Remote caching for MCM GPUs

NUBA caches remote data in local L2; SAC uses sharing-aware policies. Both orthogonal to CLAP. When combined, CLAP reduces input to caching → amplifies gains (Fig. 21).

LASP & SUV (Khairy+ MICRO'20, Pratheek+ MICRO'24) hard

Static analysis for threadblock/data placement

LASP uses code-level variables to predict locality; SUV leverages LLVM IR. CLAP-SA (Section 5.2) integrates these for static profiling, then applies MMA. Hybrid CLAP-SA++ combines static + runtime for irregular workloads.

TLB coalescing (Pham+ MICRO'12) easy

CoLT: original coalescing idea for CPUs

Merge contiguous PTEs into one TLB entry. CLAP adapts this for GPUs: 128B cache lines fetch 16 PTEs at once; CLAP's deliberate contiguity enables coalescing without large-page OS support for intermediate sizes.
