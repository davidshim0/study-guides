Moving computation into the cache, not data to the compute

Sparse matrix multiplication bogs down on memory traffic. C³ache fixes it by restructuring GPGPU caches into in-place computing units—multiplying in L1, merging in L2—to slash memory accesses 80% and beat Tensor Cores 2.4×.

📄 "C³ache" · Li et al. (Sun Yat-sen U.) 🎓 GPGPU & sparse matrix basics assumed ⏱ ~25 min read
🗄️

Move compute to memory hierarchy levels, not data through them: multiply sparse rows in L1 cache-turned-PIM, merge partial products in L2 cache-turned-PIM—memory stays put, computation happens in-situ.

Before you begin

Seven concepts C³ache builds on

Each card unpacks one prerequisite. Read in order the first time, or jump to gaps.

L1 PIM (multiplication) L2 PIM (merge) hybrid dataflow processor-centric baseline
1Sparse matrix multiplication (SPMM) and why it's hard▾

SPMM is C = A × B where most of A's entries are zero. Graph analytics, neural nets (pruned weights), and scientific codes all do this. The challenge: zeros scatter irregularly, so fetching one nonzero per cache line wastes 90% of each memory transaction, and each row has different numbers of nonzeros, breaking load balance.

Traditional GPGPU approaches (even Sparse Tensor Cores) struggle at high sparsity (>90%) because their processor-centric designs can't hide the long memory latencies—computation finishes fast, then stalls waiting for the next fetch.

Now you're ready: C³ache solves SPMM's memory bottleneck by moving computation into the cache hierarchy itself, so data never travels to distant compute units.
2GPGPU cache hierarchy: L1 (per-SM) and L2 (shared)▾

Modern GPGPUs (NVIDIA A100, etc.) have clustered Streaming Multiprocessors (SMs), each with a private L1 cache (~200KB) and sharing a large unified L2 cache (40MB). Threads in a thread-block share L1; all SMs share L2. Each cache line is 128 bytes.

In the conventional processor-centric model, data flows: global memory → L2 → L1 → compute units (CUDA cores) → L1 → L2 → global memory. Every round trip costs hundreds of cycles.

Now you're ready: C³ache re-architects these caches into processing-in-memory (PIM) units so computation happens where data already sits, slashing round-trip overhead.
3Three SPMM dataflows: inner product, outer product, Gustavson▾

Inner product computes each output element as a dot-product of a row from A and column from B—low memory footprint, poor data reuse. Outer product computes rank-1 partial matrices (column of A × row of B) and sums them—maximal reuse, but many partial matrices flood memory. Gustavson (row-wise) multiplies each row of A by corresponding rows of B—middle ground, row-aligned with cachelines.

The paper defines η (MACs per memory access) as the arithmetic intensity metric. Inner product has low η (suboptimal reuse), outer product has high η but large storage overhead, Gustavson balances both—its row-wise pattern matches cacheline-grained fetches.

Now you're ready: C³ache's hybrid dataflow combines outer-product reuse with Gustavson's cacheline alignment, decoupling SPMM into multiplication (in L1) and merge (in L2) phases.
4Processing-in-Memory (PIM) and in-cache computing▾

PIM integrates compute logic directly into memory arrays, eliminating the von Neumann data shuffle. In-cache PIM specifically augments SRAM cache arrays with compute circuitry (e.g., bitline multiply-accumulate). Prior work did single-level in-cache (L1-only or LLC-only); C³ache is the first hierarchical dual-level design.

The key trade-off: L1-side PIM enables fine-grained parallel streams (one per SM) but lacks global visibility for merging partial results. LLC-side PIM has global visibility but loses instruction-level control. C³ache resolves this by mapping different SPMM phases to different cache levels.

Now you're ready: C³ache partitions L1 cache into data ways (normal cache) and PIM ways (compute-enabled), and adds a local-adder layer to L2 for in-situ accumulation.
5Sparse encoding formats: CSR, COO, and why they fail C³ache▾

CSR (Compressed Sparse Row) stores row pointers + column indices + values—efficient for row-wise ops but one row per cacheline wastes space if the row is short. COO (Coordinate) stores (row, col, val) triplets—simple but huge overhead. ELLPACK pads rows to fixed length—good for vectorization, terrible for variable-length rows.

None of these align with cacheline granularity or detect slack: a 10-element row in BF16 with column indices occupies more space than values alone. C³ache needs a format that packs multiple short rows into one cacheline to maximize memory efficiency.

Now you're ready: The paper's C3SR format coalesces sparse rows at cacheline granularity, uses 1-byte relative column indices (6 bits for offset + 2 flag bits), and reduces memory size to 60% of COO, 71% of CSR.
6Cache partitioning and dual-mode operation▾

C³ache-L1 supports dual-mode operation: some ways are normal cache (data ways), others are PIM-enabled (Pways). A 1-bit PIM flag per way signals mode. During PIM execution, Pways are locked (excluded from LRU replacement), reducing cache associativity temporarily (e.g., from the cache's total ways).

After computation, a PIM_Release instruction clears flags and restores full associativity. The hardware constraint: Pways ≤ 50% of L1 to limit performance degradation of normal memory accesses. Performance impact on non-SPMM kernels when C³ache is not actively invoked is measured in the paper.

Now you're ready: Cache partitioning lets C³ache act as a normal cache when idle and a compute substrate when invoked via PIM instructions, preserving GPGPU general-purpose capability.
7Why processor-centric optimizations fail at high sparsity▾

Figure 1 in the paper shows that as sparsity increases, memory access overhead dominates: the paper reports cache hit rate, memory throughput, and IPC degradation trends on A100/cuSPARSE. Sparse Tensor Cores (STCs) add distribution networks and reconfigurable accumulation buffers, improving moderate-sparsity cases (~50–90%), but still rely on fetching data to processor-side units.

At ultrahigh sparsity (>90%, e.g., scientific matrices at 0.1% density), even STCs underperform baseline CUDA cores because nonzeros are too scattered to fill SIMD lanes. The fundamental bottleneck: data movement across the memory hierarchy, not compute throughput. Processor-centric designs optimize the wrong thing.

Now you're ready: C³ache's cache-centric approach eliminates most memory hierarchy traversals—multiplication happens in L1 (data already cached), merge happens in L2 (partial products accumulate in-place)—yielding 80%+ memory traffic reduction (Figure 9).
The whole story in one place

C³ache in a nutshell

The problem, the two-phase solution, and how it all connects.

Sparse matrix multiplication on GPGPUs is memory-bound: irregular access patterns cause cache thrashing, redundant fetches of the same input rows, and low arithmetic intensity. Processor-centric optimizations (Tensor Cores, Sparse Tensor Cores) attack the compute side but leave the memory bottleneck untouched—memory access overhead grows with sparsity (Figure 1), and at 90%+ sparsity even STCs lose to baseline CUDA cores.

C³ache's fix is hierarchical in-cache computing with hybrid dataflow. First, the hybrid dataflow decouples SPMM into two phases: multiplication (each sparse row of A × corresponding block of B → partial product vector) and merge (accumulate partial vectors at destination addresses in C). The outer-product component maximizes data reuse; the Gustavson component aligns with cacheline-grained memory transactions.

Second, C³ache restructures the cache hierarchy: L1 cache becomes large-scale data-parallel PIM units (multiply A-vectors by B-matrices stored in Pways), L2 cache adds a local-adder layer for in-situ merging (partial vectors accumulate directly in L2 via atomic add, no writeback to global memory until final). A Stream-aware Request Management Unit (SRMU) in L1 tracks PIM request readiness and schedules computation dynamically.

Third, the C3SR compressed format packs multiple short sparse rows into single cachelines (1-byte relative column indices, row-start flags), reducing memory footprint to 60% of COO and synchronizing fetch granularity with the dataflow's cacheline tiling.

Fourth, an efficient MSPM (multi-precision SRAM PIM macro) implements the L1 PIM: 9T SRAM for A-operands with in-memory Booth encoding, 7T SRAM for B-matrix, exponent-mantissa pipelining (overlap gather, exponent add, mantissa multiply) for BF16/FP32/INT8. Energy efficiency: 2.28 TFLOPS/W (BF16), 5.59 TOPS/W (INT8) at 40nm.

The payoff: C³ache achieves 7.22× speedup over CUDA cores and 2.4× over Sparse Tensor Cores on average across 22 workloads (low-sparsity DNNs + high-sparsity scientific matrices), with 42.4× and 22.9× EDP reduction. Memory accesses drop 80%+ (91.9% L1, 91.2% L2, 87.5% DRAM). Hardware overhead: <2.5% area (mostly MSPM macros + SRMU tables). Workload balance: 5.6% execution-time imbalance, 0.46% memory-access imbalance across SMs despite 85.6% NNZ-per-row variance.

🍯 Key takeaways

  • Memory bottleneck: at high sparsity, memory access overhead dominates compute—processor-centric designs optimize the wrong thing.
  • Hybrid dataflow: combine outer-product (data reuse) + Gustavson (cacheline alignment) → decouple into multiplication (L1) and merge (L2) phases.
  • Hierarchical PIM: L1 cache → data-parallel multiplication units; L2 cache → in-situ merge via local adder layer. Different phases, different cache levels.
  • C3SR format: pack multiple short rows per cacheline, 1-byte column indices → 60% of COO size, 80%+ memory traffic reduction.
  • MSPM macro: 9T/7T SRAM, in-memory Booth encoding, exponent-mantissa pipelining → 2.28 TFLOPS/W (BF16), 5.59 TOPS/W (INT8).
  • Results: 7.22× (vs CUDA), 2.4× (vs STC), 42.4× / 22.9× EDP reduction, <2.5% area overhead.
Six everyday pictures

How to think about C³ache

Each picture stands alone—read in any order. Every card ends with an honest note about where the analogy stops being true.

The core idea · Hierarchical PIM

Factory assembly line

Traditional GPGPUs: parts (data) shuttle from warehouse (DRAM) → staging area (L2) → workbenches (L1) → tools (ALUs) → back. Each round trip costs time and energy. C³ache: install tools at the staging areas—L1 benches do multiplication, L2 staging does final assembly—no shuttling, parts stay put. The factory runs faster because workers don't waste time fetching.

Where it breaks: a real factory's tools are specialized per step; C³ache's PIM units (MSPM, adder layer) are general-purpose compute, so the analogy oversimplifies the dual-mode cache partitioning—L1 must still act as a cache when not in PIM mode.
Hybrid dataflow · Two phases

Restaurant kitchen

SPMM is a two-step meal: multiplication = line cooks (SMs) prep ingredients (A-rows) using stock (B-matrix in Pways) → individual plates (C-vectors). Merge = head chef (L2 adder) combines plates from all cooks into the final dish (matrix C). Outer-product dataflow = each cook shares stock efficiently (high reuse). Gustavson = plates align with serving trays (cacheline-grained).

Where it breaks: a real kitchen has fixed roles; C³ache's SMs are identical, and the "head chef" (L2) isn't a bottleneck because it operates in parallel across cache banks. The analogy also hides that partial plates (C-vectors) accumulate in-place at L2, not physically moved.
C3SR format · Cacheline packing

Moving boxes in a truck

Traditional sparse formats (CSR): one item per box, truck half-empty every trip—wasteful. C3SR: detect box slack, pack 3–4 small items (short sparse rows) per box (cacheline). Driver (memory controller) makes fewer trips, fuel (bandwidth) drops 40%. The 1-byte column index = item label inside the box—just enough info to unpack later.

Where it breaks: a truck's slack is measured in volume; cacheline slack is measured in bytes and constrained by alignment (you can't split an element across cachelines). Also, the analogy hides that C3SR uses relative column indices (0–63 within a group), not absolute positions, to fit in 6 bits.
MSPM macro · In-memory compute

Calculator inside a filing cabinet

Normal SRAM = filing cabinet storing numbers. MSPM = cabinet with a calculator glued to each drawer—open drawer A (9T cell with Booth-encoded mantissa bits), read drawer B (7T cell holding B-matrix row), calculator (local logic) multiplies them while data sits in drawers, result stays local. No photocopying numbers to a desk (ALU) across the room. Energy saved = 2.28 TFLOPS/W (BF16).

Where it breaks: a real calculator is external; MSPM's "calculator" is bitline logic (sense amps, local adders) physically part of the SRAM array. Also, the 9T/7T cells aren't fundamentally different—just extra transistors for compute wordlines (CWL, VWL). The analogy also hides exponent-mantissa pipelining: exponent ops overlap with mantissa gathering.
L2 merge · In-situ accumulation

Shared whiteboard

Traditional: each worker (SM) writes partial sum on a sticky note, mails it to a central office (global memory), clerk rewrites the board—slow, expensive postage. C³ache L2-PIM: workers walk to a shared whiteboard (L2 cache) and add their number directly to the running total in place—no postage, no rewrite. The local adder = marker pen attached to the board.

Where it breaks: a whiteboard has no concurrency control; L2's adder layer handles atomic accumulation via cache coherence (write-back policy). Also, the analogy hides that partial sums arrive via PIM write requests from L1, not physical worker movement—it's a memory transaction, not a person.
Cache partitioning · Dual-mode operation

Hotel with conference rooms

L1 cache = hotel. Normal guests (data) rent rooms (ways). SPMM workload = conference (PIM instructions). Hotel manager (PIM flag) converts half the rooms into conference space (Pways), locks them (exclude from LRU), guests share the remaining half. Conference ends (PIM_Release), rooms revert to normal—hotel back to full capacity. Performance impact on normal guests when the hotel operates in normal mode is measured in the paper.

Where it breaks: a hotel's rooms are physically identical; L1 ways are identical in function but some have extra circuitry (MSPM) to enable PIM. The 50% cap isn't architectural—it's a design constraint to limit cache associativity loss. Also, "locking" is via LRU exclusion, not a physical lock—concurrent non-PIM accesses still work.
The full explanation

How it actually works

Section by section, with the paper's key mechanisms rebuilt as step-by-step walkthroughs.

1 · The problem — memory access dominates

Figure 1 in the paper shows that as sparsity increases (10% → 90%+), memory access overhead grows while compute stalls. The paper reports cache hit rate, memory throughput, and IPC degradation trends on A100/cuSPARSE. For ultrasparse matrices (<1% density), even Sparse Tensor Cores underperform baseline CUDA cores—the bottleneck isn't compute throughput, it's data movement.

🌱
Concretely: Figure 1b shows how cache performance degrades with increasing sparsity on NVIDIA A100 running cuSPARSE. Processor-centric designs (add more ALUs, bigger accumulators) don't fix this—the memory hierarchy is the chokepoint.

2 · Hybrid dataflow — decoupling multiplication and merge

The paper's hybrid dataflow combines outer-product (maximize data reuse) and Gustavson (cacheline alignment). It tiles matrix A into column-groups of size N = cacheline_size / data_size (e.g., 128B / 2B = 64 elements for BF16), and matrix B into N×N submatrices. Each SM loads one B-submatrix into Pways. A's column-groups are broadcast sequentially (cacheline-grained) to all SMs in a zigzag pattern.

# Pseudocode: hybrid dataflow
for col_group in A.column_groups:
    # Phase 1: Multiplication (in L1-PIM)
    for row in col_group.rows:
        C_vec = row × B_submatrix  # outer-product step
        PIM_Compute(dst_addr=C[row_idx], src=C_vec)
    # Phase 2: Merge (in L2-PIM)
    # L2 adder layer accumulates C_vecs at dst_addr in-place

Data reuse: each B-submatrix is shared across all rows in a column-group, achieving high arithmetic intensity. Memory alignment: column-groups = cacheline-sized, so fetching A is sequential (matches cacheline bursts). Workload balance: each SM processes equal numbers of column-groups (zigzag ensures uniform distribution even with variable NNZ per row).

💡
Why decouple phases? Multiplication needs fine-grained parallelism (one stream per SM, instruction-level control) → L1-side PIM. Merge needs global visibility (accumulate partial C from all SMs) and tolerates coarse-grained ops (address-indexed add) → L2-side PIM. Single-level PIM can't satisfy both.

3 · L1 PIM architecture — multiplication units

C³ache-L1 partitions each cache into data ways (normal cache) and Pways (PIM-enabled, ≤50% to limit associativity loss). A 1-bit PIM flag per way signals mode. The MSPM macro (Section 3.5 in paper, Figure 7) implements vector × matrix multiplication:

  1. Load B-submatrix: PIM_Load instruction fetches B from global memory into Pways (7T SRAM arrays BMPIM, BEPIM for mantissa and exponent). Pways are locked (excluded from LRU).
  2. Compute: For each sparse A-row (in C3SR format), the sparse decoder extracts column indices, the 9T SRAM AMPIM stores A-mantissa and performs in-memory Booth encoding (3-bit serial, CWL + VWL activation → NEG, ZERO, ONE signals), and the local logic pipelines exponent add (AEXP + BEXP - CEXP → offset) with mantissa multiply-accumulate (Booth-shifted partial products).
  3. Schedule: The SRMU (Stream-aware Request Management Unit) tracks PIM requests per warp, monitors data readiness (source address mapped to Pway index), and dispatches ready requests to MSPM. This decouples request issue from computation, avoiding pipeline stalls.
  4. Output: Result C-vector is sent via PIM_Compute to the destination address in C (triggers PIM write request to L2).
🍯
Key innovation: MSPM's Booth encoding happens in memory (9T cell's CWL/VWL gates compute AND/NOR on bitlines) → no explicit Booth decoder, 2× fewer MAC cycles. Exponent-mantissa pipelining overlaps gather (next row's column indices) with current row's compute. Energy efficiency: 2.28 TFLOPS/W (BF16), 5.59 TOPS/W (INT8) at 40nm (Table 5).

4 · L2 PIM architecture — in-situ merge

C³ache-L2 adds a local adder layer at each cache bank (minimally invasive: peripheral logic, no changes to core SRAM array). When a PIM write request arrives from L1:

  1. Tag lookup: Check if destination address (C[row_idx]) is in L2. If hit, fetch current C-vector from cache.
  2. Accumulate: Local adder performs C_new = C_old + C_partial (element-wise add of BF16 vectors). Result written back to same cache location (in-place update).
  3. On miss: Fetch C_old from global memory into L2, then proceed with accumulation. (First access to a C-row is a miss; subsequent accesses from other SMs are hits.)

Why L2, not L1? Partial C-vectors from all SMs must merge at the same destination address. L1 is per-SM (no cross-SM visibility); L2 is shared (global scope). The adder layer ensures atomic accumulation via cache coherence (write-back policy). Result: No writeback of partial matrices to global memory—only final C at end of kernel.

⚠
Trade-off: L2 adder adds overhead per cache bank (Table 6: local adder layer area). But it eliminates K (number of column-groups) global-memory writes of partial matrices—massive bandwidth saving. Figure 9 shows 91.2% L2 access reduction, 87.5% DRAM reduction.

5 · C3SR compressed format — cacheline packing

Traditional CSR wastes cacheline space when rows are short. C3SR fixes this by packing multiple short rows into one cacheline:

  1. Row compression per column-group: For each column-group, extract nonzeros, couple them with 1-byte relative column indices (6 bits for offset 0–63, 1 bit for row-start flag, 1 bit for valid flag). Store (col_idx_byte, value) pairs.
  2. Detect slack: Compare cacheline slack (remaining bytes) with next row's size. If fits, append row to current cacheline; else start new cacheline. Update row_offset[CL] (cumulative row count per cacheline).
  3. Indexing: Three arrays: Cache_Line_Offset[col_group] (starting cacheline per column-group), row_offset[CL] (row boundaries within cacheline), Row_Indices[row] (actual row number). Algorithm 1 in paper.

Compression ratio: C3SR reduces memory size to 60.1% of COO, 71.4% of CSR (Figure 14). Decoding overhead: Sparse decoder in MSPM reads 1-byte indices (row-start flag = new row, valid flag = skip if 0), extracts 6-bit offset, gathers B-matrix row via column index. Negligible latency (single-cycle index extraction).

🍯
Why it works: C3SR synchronizes fetch granularity (cacheline = 128B) with dataflow tiling (column-groups = cacheline-sized). This alignment maximizes valid bytes per memory transaction—no wasted bandwidth. Combined with hybrid dataflow's workload balance, C3SR ensures uniform memory traffic across SMs (Figure 15: 0.46% memory-access imbalance).

6 · PIM request execution flow — the complete picture

Bringing it all together (Figure 5bcd in paper):

  1. Issue: SIMT pipeline decodes PIM_Compute instruction → LSU generates PIM request (RequestType = PIM, src_addr = A-row, dst_addr = C[row_idx]).
  2. L1 tag lookup: Cache controller checks SRMU table. If src_addr maps to a ready Pway (data already loaded), proceed; else enqueue request and wait for PIM_Load to complete.
  3. Schedule: SRMU picks ready request, dispatches to MSPM. MSPM computes C_vec = A_row × B_submatrix (pipeline: gather → Booth encode → mantissa MAC → exponent align → local accumulate).
  4. L1 writeback: Result C_vec sent to cache controller → triggers PIM write to dst_addr. SRMU clears request entry.
  5. L2 tag lookup: L2 cache controller checks if dst_addr in L2. On hit, fetch C_old; on miss, fetch from DRAM.
  6. Merge: Local adder computes C_new = C_old + C_vec → write back to L2 (in-place update). Send response to L1 → forward to LSU (PIM instruction complete).
  7. Release: After all PIM instructions in kernel, PIM_Release clears PIM flags → Pways revert to data ways, full associativity restored.
💡
Why SRMU? Without it, PIM requests would stall the SIMT pipeline (waiting for data readiness). SRMU decouples issue from execution: pipeline continues issuing (requests per warp), SRMU schedules asynchronously based on Pway readiness. This hides latency and maximizes throughput.
Follow the numbers, step by step

Worked examples

Four traces, each carried all the way through—computing C-vectors, merging in L2, measuring memory savings, and verifying hardware cost.

① Computing a partial product vector in L1-PIM

Sparse row A[i] = {col[2]=3.5, col[5]=1.2, col[7]=2.8} (3 nonzeros, BF16). B-submatrix (8×8) loaded in Pways. How does MSPM compute C_vec = A[i] × B?

  1. Sparse decoder extracts column indices: C3SR format → col[2], col[5], col[7] (1-byte indices with row-start flag set on first element). Decoder outputs index list [2, 5, 7].
  2. Gather B-rows: Activate wordlines WL[2], WL[5], WL[7] in BMPIM (mantissa) and BEPIM (exponent). Read B[2,:], B[5,:], B[7,:] (3 rows × 8 elements = 24 BF16 values) into local logic buffers.
  3. Booth encode A-mantissa: 9T AMPIM stores A-mantissa bits. For A[2]=3.5 (BF16: sign=0, exp=128, mantissa=110...), CWL/VWL activate 3-bit windows serially → NEG=0, ONE=1, ZERO=0 (1×M) for first partial product. Repeat for all mantissa bits (8 cycles for BF16).
  4. Exponent pipeline: E local logic computes AEXP[2] - 127 + BEXP[2] - CEXP[dst] → offset (e.g., 128 - 127 + 130 - 0 = 131 → offset=131). Pass offset to M local logic for alignment.
  5. Mantissa MAC: M local logic shifts B[2,j] by Booth code (×1 for ONE), accumulates across 8 elements → partial product PP[2]. Repeat for cols 5, 7. Align all PPs by offset, sum → C_vec (8 elements, BF16).
  6. Result: C_vec written to SRMU → triggers PIM write to dst_addr = C[i]. Total latency: ~40 cycles (3 gathers × 2 cycles + 8 MAC cycles + 10 alignment cycles).
🍯
Key insight: In-memory Booth encoding (step 3) eliminates explicit decoder—9T cell's CWL/VWL gates compute AND/NOR on bitlines, saving 2× MAC cycles vs. traditional. Exponent-mantissa pipelining (step 4) overlaps gather of next row with current row's MAC (step 5).

② Merging partial products in L2-PIM

Three SMs (SM0, SM1, SM2) each compute partial C-vectors for the same destination row C[42]. How does L2 adder layer merge them?

  1. First arrival (SM0): PIM write request dst_addr=C[42], src=C_vec0. L2 tag lookup → miss (C[42] not in L2 yet). Fetch C[42] from DRAM (all zeros, first access). Local adder: C_new = 0 + C_vec0 = C_vec0. Write to L2 cache at C[42]. Send response to SM0.
  2. Second arrival (SM1): PIM write dst_addr=C[42], src=C_vec1. L2 tag lookup → hit (C[42] now in L2 from step 1). Fetch C_old = C_vec0. Local adder: C_new = C_vec0 + C_vec1. Write back to L2 (in-place update). Send response to SM1.
  3. Third arrival (SM2): PIM write dst_addr=C[42], src=C_vec2. Hit. Fetch C_old = (C_vec0 + C_vec1). Local adder: C_new = (C_vec0 + C_vec1) + C_vec2. Write back. Send response to SM2.
  4. Final state: C[42] in L2 holds full merged result (sum of all partial vectors from SMs). At kernel end, L2 writes C[42] to global memory (write-back policy). Total global-memory writes: 1 (final) vs. 3 (if each SM wrote partial directly).
💡
Why L2 hit rate is high: After first arrival (miss), subsequent arrivals hit because C[42] stays in L2 (LRU keeps it hot). Paper reports 91.2% L2 access reduction (Figure 9)—most merges resolve in-cache, not global memory. The local adder's overhead (per bank) is amortized across thousands of C-rows.

③ Measuring memory traffic reduction

How does C³ache reduce memory transactions compared to baseline approaches?

  1. Baseline CUDA (processor-centric): Each nonzero in A triggers memory accesses for both sparse matrix elements and dense matrix B. Irregular access patterns cause cache thrashing. Redundant fetches of B-rows occur due to poor spatial locality. Many memory transactions are needed.
  2. C³ache (cache-centric): Fetch A column-groups using C3SR packing (multiple rows per cacheline). Fetch B-submatrix per SM once (broadcast, shared across rows). Merge in L2: first access of each C-row = miss from DRAM, subsequent merges = L2 hits (in-situ accumulation). Final write C to memory only once per row.
  3. Reduction: Paper reports 87.5% DRAM reduction, 91.2% L2 access reduction, 91.9% L1 access reduction on average (Figure 9). The 80%+ reduction is the combined effect of C3SR packing, broadcast B-submatrix sharing, and L2 in-situ merge.
🍯
Breakdown of savings: (a) C3SR packing → fewer A fetches (multiple rows per cacheline). (b) Broadcast B-submatrix → shared across SMs, not fetched per row. (c) L2 in-situ merge → no writeback of partial matrices, only final C. The combination yields dramatic memory traffic reduction across all hierarchy levels.

  3. L2 local adder layer: Table 6 shows C3ache-L2 area: 46,528 μm² for the local adder layer components. This is the overhead added to L2 cache.
  4. Total C³ache overhead: Sum of all MSPM macros (L1), all SRMUs, and L2 adder layer. Paper states "less than 2.5%" of baseline GPGPU area. Most area is MSPM macros (custom SRAM); SRMU and L2 adder are smaller (logic components).
  5. Verification: Paper states "less than 2.5%" (Section 5.2). The calculation accounts for all added components while noting that dual-mode cache partitioning means PIM area is shared with normal cache operation.
💡
Why so efficient? C³ache reuses existing cache SRAM arrays—only adds compute logic (9T/7T cells, local logic, adders). No separate dedicated accelerator units required. The dual-mode cache partitioning means PIM area serves dual purpose—not idle when not in PIM mode, available for normal caching.

Problem 1 · easy
Why does C³ache partition L1 cache into data ways and Pways instead of converting the entire L1 into PIM?
Show answer
Dual-mode operation preserves GPGPU general-purpose capability. If the entire L1 were PIM-only, non-SPMM kernels (e.g., dense matrix multiply, vector add) would lose L1 caching, degrading performance. By capping Pways at ≤50%, C³ache ensures normal workloads retain sufficient L1 associativity. After PIM kernel completes, PIM_Release restores full associativity—cache behaves normally again. The paper measures performance impact on non-SPMM kernels.Misconception: that PIM requires dedicated hardware. C³ache's insight is that cache SRAM can do both—normal caching and compute—via mode flags and partitioning.
Problem 2 · medium
A sparse matrix has rows of highly variable length (1 to 100 nonzeros per row). How does C3SR handle this without wasting cacheline space?
Show answer
C3SR detects cacheline slack (remaining bytes after packing a row) and appends the next short row if it fits (Algorithm 1, line 5–13). Example: row A has 10 elements (requires storage for values + column indices), leaving slack. Row B with 30 elements may fit. Both can be packed in cachelines based on available space. The row_offset[CL] array records row counts per cacheline, and Row_Indices stores row numbers. Decoding: read row_offset to find row boundaries, use row-start flag (1st bit of 1-byte col index) to parse. Variable-length rows become uniform-length cachelines—memory transactions balanced across SMs (Figure 15: 0.46% imbalance).Misconception: that variable-length rows inherently cause load imbalance. C3SR's cacheline-grained packing averages out variance—some cachelines pack fewer long rows, others pack more short rows, but total bytes per cacheline are maximized.
Problem 3 · medium
Why does C³ache's hybrid dataflow use outer-product (not inner-product) for the multiplication phase, despite outer-product's traditional higher memory footprint?
Show answer
Outer-product maximizes data reuse of matrix B: each B-row is shared across all A-elements in a column-group, yielding high arithmetic intensity. Inner-product reuses neither A nor B effectively (each element used once per output). The "high memory footprint" trap doesn't apply here because C³ache decouples the outer-product into two phases: (1) multiply in L1 (partial C-vectors stay in registers, not written to memory), (2) merge in L2 (accumulate in-place, not stored as separate matrices). Traditional outer-product stores partial matrices in DRAM—expensive. C³ache's hybrid dataflow stores 0 intermediate matrices in DRAM—only final C. The memory footprint is lower than inner-product because fewer global-memory transactions (Figure 9: 87.5% DRAM reduction).Misconception: that outer-product always means high memory traffic. True for processor-centric designs (store partial matrices). False for C³ache's in-cache merge (partial results never leave L2).
Problem 4 · hard
The MSPM uses 9T SRAM for A-operands and 7T SRAM for B-operands. Why the asymmetry? Why not use 9T for both?
Show answer
9T SRAM adds compute wordlines (CWL, VWL) for in-memory Booth encoding—needed for A-operands because A-mantissa bits are serially processed (3-bit windows, 8 cycles for BF16). The extra transistors enable bitline-level AND/NOR operations (Figure 7: NMBE circuit). 7T SRAM has one extra transistor for separate read port (SBL), allowing simultaneous read of B-row while writing back result—needed for B-operands because B-matrix is static during a column-group's computation (loaded once, read many times). Using 9T for B would waste area (CWL/VWL unused) and power (more bitlines to precharge). Using 7T for A would break Booth encoding (no CWL/VWL gates). The asymmetry reflects access patterns: A is serial-processed, B is random-accessed.Misconception: that SRAM bit-cells are interchangeable. The paper's custom 9T/7T designs are co-optimized with the dataflow: 9T for A's compute-in-memory needs, 7T for B's concurrent read/write needs. Standard 6T would require external Booth encoder + muxes, increasing latency and area.
Problem 5 · medium
Explain why C³ache's L2 merge happens in-situ (via local adder layer) instead of using a dedicated accumulation buffer (like Sparse Tensor Cores).
Show answer
STCs use a separate accumulation buffer (outside cache) to collect partial products, then write final C to memory. Problem: (1) buffer is limited size (e.g., 16KB)—doesn't scale to large C matrices; (2) buffer-to-memory writeback adds latency; (3) partial products evicted from buffer trash cache when reloaded. C³ache's in-situ merge keeps partial C in L2 cache: (1) scales to L2 size (40MB, holds entire C for most workloads); (2) no writeback until final (write-back policy); (3) partial C stays hot in L2 (high hit rate for subsequent merges, Figure 9: 91.2% L2 reduction). The local adder layer is a peripheral addition—doesn't occupy cache capacity, just augments write path. Result: in-situ merge is both faster (fewer global-memory accesses) and more scalable (C size ≤ L2 size, not buffer size).Misconception: that accumulation buffers are "free." STCs' buffers add area and create a capacity bottleneck. C³ache leverages existing L2 cache capacity for accumulation—no new buffer needed, just adder logic on write path.
Problem 6 · hard
C³ache claims 7.22× speedup over CUDA cores but only 2.4× over Sparse Tensor Cores. Why is the STC gap smaller, and under what conditions would STC beat C³ache?
Show answer
STCs exploit structured sparsity (e.g., 2:4 patterns: 2 nonzeros per 4 elements) with distribution networks and reconfigurable accumulators—efficient for moderate sparsity (50–90%) with regular patterns (e.g., pruned neural nets). C³ache beats STCs on (a) ultrasparse matrices (>95%, e.g., scientific codes at 0.1% density), where STCs can't fill SIMD lanes and underperform CUDA (Figure 8: STC ≈ CUDA for high-sparsity workloads); (b) irregular patterns (variable NNZ per row), where STC's structured assumptions break. STC would beat C³ache if: (1) sparsity is moderate (~70%) and structured (2:4), maximizing STC's SIMD utilization; (2) matrix fits in STC's accumulation buffer (no capacity bottleneck); (3) no memory-bound bottleneck (e.g., matrix in L2). C³ache's 2.4× average advantage means it loses on some STC-friendly workloads but wins big on memory-bound/ultrasparse cases (up to 6×+ on scientific matrices, Figure 8).Misconception: that one architecture dominates all sparsity regimes. The paper shows processor-centric (STC) vs. cache-centric (C³ache) is a trade-off: STC wins on compute-bound structured sparsity, C³ache wins on memory-bound irregular sparsity. The 2.4× average reflects C³ache's robustness across varying sparsity.
Problem 7 · easy
Why does the SRMU track PIM requests per warp instead of all in-flight requests from a thread block?
Show answer
GPGPUs execute in warps (32 threads SIMT). The SIMT pipeline issues instructions warp-by-warp. The SRMU is designed to track and schedule requests based on the GPGPU's warp-level execution model. The SRMU's job is to decouple issue from execution within the warp granularity (hide latency by scheduling ready requests while others wait for data). The paper specifies SRMU has 32 entries aligned with warp configuration. Measured result: no pipeline stalls due to SRMU capacity (requests dispatch asynchronously).Misconception: that SRMU is a global request queue. It's designed to work with the GPGPU's warp-based execution model, tracking and scheduling PIM requests efficiently at the warp granularity.
Problem 8 · medium
C³ache's exponent-mantissa pipelining overlaps gather (next row) with compute (current row). Explain how this pipelining avoids data hazards when the next row's column indices depend on the current row's result.
Show answer
There's no dependency—the hybrid dataflow ensures each A-row is independent (outer-product decoupling). Specifically: (1) column indices of A-row i+1 are stored in C3SR format (pre-compressed), not computed from row i's result; (2) the B-submatrix is static (loaded once per column-group), so gathering B-rows for A[i+1] doesn't depend on C[i]. The pipelining works as: Stage 1 (row i): gather B-rows via col indices → compute C[i]. Stage 2 (row i+1, overlapped): sparse decoder reads col indices from C3SR → gather B-rows (while Stage 1 is in MAC). No hazard because gather(i+1) reads from memory (C3SR + BMPIM), not from C[i]. The only constraint: row i must finish before row i+1 starts MAC (enforced by SRMU's sequential scheduling of ready requests). Measured latency achieves near-peak throughput (Table 5).Misconception: that pipelining always introduces hazards. True for data-dependent operations (e.g., recursive algorithms). False here because A-rows are independent and C3SR pre-encodes all column indices—no runtime dependency between rows.
Quick answers

Frequently asked questions

Ten questions newcomers ask most. Tap any to expand.

Q1What does "cache-centric" mean, and how is it different from "processor-centric"?▾
Processor-centric: data moves to compute units (ALUs/Tensor Cores)—memory is passive storage. Cache-centric: compute happens in memory (caches become PIM units)—data stays put. C³ache restructures L1/L2 caches into execution backends (MSPM, adder layer), eliminating most memory hierarchy traversals. The shift: from "bring data to compute" to "bring compute to data."
Q2Why does C³ache use two levels of PIM (L1 and L2) instead of just one?▾
SPMM has two incompatible needs: (1) multiplication needs fine-grained parallelism (one stream per SM, instruction control) → L1-side PIM (per-SM, SIMT-driven); (2) merge needs global visibility (accumulate partial C from all SMs) → L2-side PIM (shared, address-indexed). Single-level PIM can't satisfy both—L1-only lacks cross-SM merge, L2-only lacks per-SM control. Hierarchical PIM maps each phase to the right cache level.
Q3How does C³ache handle cache coherence when multiple SMs write to the same C-row in L2?▾
L2 uses write-back policy with atomic accumulation via local adder layer. When SM writes PIM request to C[i]: (1) tag lookup checks if C[i] in L2; (2) if hit, fetch C_old, adder computes C_new = C_old + C_partial, write back to same location (in-place update); (3) if miss, fetch from DRAM first, then accumulate. Cache coherence protocol ensures only one writer at a time per cacheline (via MSHR locks). Result: no data races, no explicit synchronization needed in software.
Q4What happens if the matrix C is too large to fit in L2 cache?▾
C³ache tiles C into blocks that fit in L2 (hybrid dataflow already does this—each SM processes a B-submatrix → partial C-block). If C-block > L2, the OS/runtime splits it further (software tiling). Worst case: C-row evicted from L2 before all SMs merge → next SM's write triggers L2 miss → fetch from DRAM, accumulate, write back. Performance degrades gracefully (more L2 misses, more DRAM traffic) but doesn't break. Measured result: most workloads fit (40MB L2 holds substantial matrices), and even on overflow C³ache beats baseline (80%+ memory reduction from avoiding multiple partial-matrix writes).
Q5Why does C3SR use 1-byte column indices instead of 2-byte (more range)?▾
C3SR uses relative indices within a column-group (not global column number). Column-group size = cacheline_size / data_size = 128B / 2B = 64 elements for BF16 → max offset = 63, fits in 6 bits. Add 1 bit for row-start flag, 1 bit for valid flag → 8 bits = 1 byte. 2-byte indices would waste space (128 global range unused) and reduce packing efficiency (fewer rows per cacheline). Trade-off: 1-byte limits column-group to 64 elements, but that's exactly cacheline granularity—perfect alignment.
Q6How does C³ache handle very dense matrices (e.g., 90% nonzeros)?▾
C³ache's hybrid dataflow and PIM architecture work for any sparsity level, but efficiency peaks at high sparsity. For dense matrices: (1) C3SR packing gains shrink (most rows fill a cacheline, no slack to pack multiples); (2) memory traffic reduction shrinks (fewer redundant fetches because most elements are nonzero); (3) Pway locking overhead increases (more data in Pways, less cache for normal ops). The paper targets sparse workloads (10–99% sparsity), where C³ache's 7.22× speedup applies. For dense, use traditional dense GEMM (Tensor Cores).
Q7What are the stability/noise margins of the 9T/7T SRAM cells, and do they degrade reliability?▾
Figure 11 shows measured static noise margin (SNM) values for the custom SRAM cells. The paper reports SNM measurements and argues for robustness of the design. Compute SNM is lower than traditional 6T read SNM but sufficient for digital bitline logic (signals are binary, not analog). The extra transistors in 9T/7T separate compute path from read/write path, so compute operations don't disturb stored data (no read-upset). The paper discusses reliability considerations.Misconception: that custom SRAM cells are inherently unreliable. The paper's design includes careful SNM analysis and optimization.
Q8How does C³ache scale to many-core GPUs (e.g., 100+ SMs)?▾
Each SM's L1-PIM is independent (parallel multiplication), so L1 scales linearly with SMs (100 SMs → 100× L1 throughput). L2-PIM scales via bank-level parallelism: each bank's adder layer handles merges independently (no cross-bank dependencies). Bottleneck: L2 interconnect bandwidth (all SMs write to L2). The paper discusses scaling considerations and shows results at different SM counts (Figure 12). For larger configurations, architectural parameters like L2 banks can be adjusted to maintain per-bank throughput.
Q9Can C³ache run SPMM and dense GEMM simultaneously on the same GPGPU?▾
Yes—Figure 13 (right) shows concurrent SPMM + SGEMM fusion: SPMM kernel uses Pways in L1-PIM, SGEMM kernel uses data ways + CUDA cores. Measured result: fused execution takes 24% less time than sequential (1.3× speedup). How it works: (1) SPMM locks ≤50% L1 ways as Pways (via PIM flag); (2) SGEMM uses remaining 50% as normal cache + CUDA cores for compute; (3) both kernels share L2 (no conflict because SPMM writes via adder layer, SGEMM writes via normal cache path). Trade-off: SGEMM sees reduced L1 associativity (50% → slight perf loss), but overlap gains > associativity loss. Key: dual-mode cache partitioning enables coexistence of PIM and non-PIM workloads.
Q10What are the main limitations or failure cases of C³ache?▾
(1) Dense matrices: C³ache's optimizations (C3SR packing, outer-product reuse, L2 merge) are tailored for sparsity—dense matrices see less benefit. (2) Very small matrices: if matrix C fits in L1, L2 merge is unnecessary overhead. (3) Fast-changing sparsity patterns: if A's sparsity shifts rapidly (streaming data), C3SR compression overhead (per kernel invocation) may dominate. (4) Unsupported data types: MSPM supports BF16/FP32/INT8; FP64 or custom types require redesign. (5) Non-SPMM workloads: C³ache is specialized for SPMM—general sparse ops (SpMV, SpGEMM) need adaptation. The paper acknowledges these—C³ache is a domain-specific optimization, not a universal accelerator.

C³acheHierarchical Cache-Centric Computing architecture for SPMM on GPGPUs, restructuring L1/L2 caches into PIM units.
SPMM (Sparse Matrix Multiplication)C = A × B where A is sparse (most elements zero). Bottlenecked by memory access, not compute.
Hybrid dataflowCombines outer-product (high reuse) + Gustavson (cacheline-aligned) → decouples SPMM into multiplication (L1) + merge (L2) phases.
L1 PIM (multiplication phase)C³ache-L1 partitions cache into data ways + Pways (MSPM macros), computes A-row × B-submatrix → C-vector.
L2 PIM (merge phase)C³ache-L2 adds local adder layer for in-situ accumulation of partial C-vectors at destination addresses.
MSPM (Multi-precision SRAM PIM Macro)Custom 9T/7T SRAM with in-memory Booth encoding, exponent-mantissa pipelining → 2.28 TFLOPS/W (BF16).
C3SR (Cache-Centric Compressed Sparse Row)Sparse format that packs multiple short rows per cacheline, 1-byte relative col indices → 60% of COO size.
SRMU (Stream-aware Request Management Unit)Per-L1 scheduler tracking PIM requests, monitors data readiness, dispatches to MSPM asynchronously.
Pway (PIM way)Cache way with PIM flag set, excluded from LRU, holds B-submatrix for MSPM compute. ≤50% of L1 associativity.
Dual-mode cacheCache partitioned into data ways (normal) + Pways (PIM), mode controlled by PIM flag per way.
Supporting terms
GPGPU (General-Purpose GPU)Parallel processor with SMs (Streaming Multiprocessors), each with L1 cache, sharing L2 cache. SIMT execution model.
SM (Streaming Multiprocessor)Cluster of CUDA cores + shared L1 cache. C³ache: one SM = one L1-PIM unit.
Cacheline128-byte memory transaction unit in modern GPGPUs. C³ache tiles at cacheline granularity.
Column-groupPartition of sparse matrix A into cacheline-sized chunks (e.g., 64 elements for BF16). Broadcast to all SMs.
Outer-product dataflowComputes rank-1 partial matrices (col of A × row of B), maximizes reuse but traditionally requires high memory.
Gustavson dataflowRow-wise product (A-row × B-rows), moderate reuse, cacheline-aligned. C³ache uses for memory alignment.
Inner-product dataflowDot-product (A-row · B-col), low reuse, low memory footprint. C³ache avoids (poor reuse).
PIM (Processing-In-Memory)Compute integrated into memory arrays, eliminates von Neumann data movement. C³ache = in-cache PIM.
In-situ mergeAccumulate partial results directly at destination address in cache (no separate buffer). C³ache-L2 via adder layer.
Booth encodingReduce multiply-accumulate cycles by encoding multiplier in {-2M, -M, 0, M, 2M}. MSPM does it in memory (9T cell).
Exponent-mantissa pipeliningOverlap exponent ops (add/subtract) with mantissa ops (multiply-accumulate). MSPM gains 2× throughput.
Sparse Tensor Cores (STC)NVIDIA's processor-centric sparse accelerator (distribution network + reconfigurable accumulators). C³ache beats 2.4× on avg.
CSR (Compressed Sparse Row)Traditional sparse format: row pointers + col indices + values. C³ache improves via C3SR cacheline packing.
9T/7T SRAMCustom bitcells with extra transistors for compute wordlines (9T) or separate read port (7T). Enables MSPM.
