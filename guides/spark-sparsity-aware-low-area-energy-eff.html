Solving integer programs by computing inside the cache

Network routing, traffic scheduling, map routing — all integer linear programs (ILPs) that demand timely decisions but take tens of hours on CPUs/GPUs. This is the plain-English story of SPARK, an architecture that reconfigures the L1 cache to accelerate ILP near-memory.

Before you begin

Ten ideas SPARK assumes you know

Each card covers one concept the paper builds on. Read in order or jump to whatever looks unfamiliar.

near-memory acceleration sparse ILP constraints SPARK engines CPU/GPU baseline
1What Integer Linear Programming (ILP) is▾

Linear programming (LP) optimizes objectives under linear constraints: maximize/minimize F(X) = Σ Aj·Xj subject to C·X ≤ D, X ≥ 0. Integer LP (ILP) adds Xj ∈ ℤ, restricting solutions to integers — this small change makes the problem exponentially harder.

Real-world ILPs: network routing (which edges to use?), traffic scheduling (assign buses to routes?), map routing (optimal path selection), stock portfolios (how many shares?).

Now you're ready: ILPs are everywhere that need discrete choices under constraints. SPARK accelerates solving them when decisions must be timely (seconds, not hours).
2Why ILPs are slow: sparse, branch-heavy▾

Real-life ILP constraint matrices (MIPLIB benchmarks) are 65–99% sparse (most entries zero). CPUs use software sparsity optimizations (Gurobi) but still spend cycles on data movement. GPUs underutilize cores due to sparsity and thread divergence in branching.

Koch et al. showed even 4000–8000 cores take tens of hours for many ILPs. Decision thresholds (seconds to minutes) are missed.

Now you're ready: SPARK's sparsity-aware, near-memory compute skips zeros in hardware and co-locates compute with data, fixing both bottlenecks.
3Iterative solvers: Jacobi for SLE, Branch & Bound for ILP▾

Jacobi method solves systems of linear equations (SLE) iteratively: compute new X values from old X, C, D using MAC (multiply-accumulate), subtraction, division; repeat until convergence (L1 norm small).

Branch & Bound (B&B) rounds Jacobi's non-integer solution to integers: branch on fractional variables, add constraints, re-solve SLE at each node, prune when bounds match or solutions violate constraints.

Now you're ready: SPARK accelerates the SLE dot-products in-cache and reuses the same logic for B&B's constraint checking, avoiding dedicated B&B hardware.
4L1 cache as compute substrate (8T SRAM)▾

Modern CPUs use 8T SRAM bitcells in L1 with decoupled read/write ports (separate RBL/RWL for reads, WBL/WWL for writes). This lets you read after write in 2 cycles (vs 3 for 6T). SPARK reuses the read port for in-memory dot-product compute.

Why L1? It's close to the CPU (low latency), fits 65% of MIPLIB benchmarks, and prefetching hides misses for larger workloads — 20–25× better energy efficiency than using LLC.

Now you're ready: SPARK stores constraint matrix C in L1, maps variables X onto column lines (RBL), performs dot-product by precharge, reads output from row buffers — no array modification, just peripheral logic.
5Processing-in-memory (PIM) — compute where data lives▾

Traditional compute: fetch operands from memory to CPU ALU, compute, write back (high data movement cost). PIM: perform operations directly in/near the memory array, outputting only the result — e.g., dot-product by RBL discharge in SRAM.

Why it helps for ILP: constraint matrices are large and reused across iterations. Moving them to CPU for every MAC wastes energy; computing in-place slashes movement overhead.

Now you're ready: SPARK's "in-L1" dot-product is PIM. The near-L1 logic (shift-add, subtraction, division) is coarse-grain acceleration for steps PIM can't do alone, still local to cache.
6Sparsity: zeros dominate, but vary by position▾

A matrix with 99% zeros still has structure: some constraints have exactly one non-zero (Xi ≤ Di), others have scattered non-zeros. Software detects sparsity offline; hardware can detect it on the fly per constraint row.

SPARK's FC engine counts non-zeros per row: if exactly n rows have the form Xi ≤ Di (cardinality-constrained), the ILP is deemed sparse and the SA engine activates; else SLE engine handles it densely.

Now you're ready: Hardware sparsity detection is fast and parallel. SPARK skips useless computes (zero coefficients) and turns off engines not needed for sparse vs dense cases.
7Why dedicated accelerators aren't enough▾

TPUs: great for dense tensor ops, but slow on branching/sparsity (CloudTPU docs confirm). CGRAs: reconfigurable but rely on compilers, poor on conditionals. EIE/SparTen: sparse DNN accelerators lacking control-flow hardware for L1-norm or B&B.

All suffer from host↔accelerator data movement overhead and can't handle the "3C" criterion: C1 (dot-product), C2 (sparsity), C3 (control-flow). SPARK satisfies all three by integrating into CPU pipeline.

Now you're ready: SPARK is tightly coupled — it reconfigures existing L1, adding minimal near-cache logic. No separate chip, no costly data shuttling, easy compiler support via ISA extensions.
8MPKI and energy as scorecards▾

MPKI = Misses Per 1000 Instructions (cache miss rate — doesn't apply directly here, but conceptually "operations avoided"). Energy = power × time. For ILP, we care about solution time (performance) and total Joules to converge.

MIPLIB on CPU: large energy, tens of hours. SPARK: significant energy reduction and speedup (sparse ILP) by cutting data movement and insignificant compute.

Now you're ready: When you see speedup and energy improvement claims, that's SPARK vs Zen3 CPU with Gurobi, measured end-to-end on MIPLIB benchmarks.
9Reuse-awareness: sharing logic across engines▾

B&B adds sparse constraints to an initially dense ILP at each branch level. Instead of dedicated B&B hardware, SPARK reuses the SLE engine to solve the updated constraints, then verifies new sparse constraints with near-memory queues and simple MAC.

This "reuse-aware" design keeps area low (~1.4% of CPU) by sharing shift-add/adder-reduction/subtractors/dividers among FC/SA/SLE/B&B engines.

Now you're ready: SPARK's four engines (FC, SA, SLE, B&B) aren't four separate blocks — they share the same near-L1 arithmetic units, reconfigured per mode.
10Why L1 over L2/L3: latency vs throughput tradeoff▾

Larger caches (L2/L3) offer more capacity and throughput but sit farther from the core (higher latency). For 65% of MIPLIB benchmarks fitting in 128KB L1, the latency win dominates. For 35% that overflow, prefetching L2→L1 hides most miss penalty.

Energy efficiency (throughput/energy): L1 is 20–25× better than LLC because data movement L2↔L1 is shorter, and compute happens closer to the core.

Now you're ready: SPARK targets L1 for best balance. Near-L2 compute is left for future work; current design with L1 prefetching already beats CPU/GPU by orders of magnitude.

The whole story in one place

SPARK in a nutshell

The problem, the sparsity-aware near-memory solution, and how it connects to your tiering/CXL work.

ILP solvers like Gurobi use software-based sparsity optimizations on CPUs, but still hit Von Neumann bottlenecks: limited throughput, high data movement. Even 4000–8000 cores take tens of hours for decision-critical ILPs. GPUs struggle with 65–99% sparsity and thread divergence in B&B, underutilizing streaming engines. Existing accelerators (TPUs, CGRAs, sparse-DNN chips) handle at most two of the "3C" needs — dot-product (C1), sparsity (C2), control-flow (C3) — but not all three, and incur costly host↔accelerator data transfers.

SPARK reconfigures the CPU's L1 cache (8T SRAM with decoupled read ports) for in-memory dot-product compute, storing constraint matrix C in memory and mapping variables X onto column lines (RBL). A hardware FC engine detects sparsity by counting cardinality-constrained rows; if sparse, the SA engine executes a sparsity-aware algorithm (identify potential solutions via parallel PIM MACs, skip zero coefficients), else the SLE engine runs Jacobi iteration using PIM + near-L1 shift-add/subtractors/dividers. The B&B engine reuses SLE logic for branch solving, adding only small near-memory queues for new sparse constraints — no dedicated B&B hardware.

Area: ~1.4% of CPU (0.37mm² near-L1 logic per core). Throughput: 32 16-bit MACs/cycle from one core for dense ILPs. Energy: 0.001pJ extra precharge per column, orders of magnitude less than CPU↔memory transfers. Performance and energy improvements vary by workload sparsity and size, as reported in the paper's evaluation on MIPLIB benchmarks.

⚡ Key takeaways

  • Reconfigure L1 cache for in-memory dot-product (PIM) — store C, map X to RBL, read result from row buffers.
  • Hardware sparsity detection (FC engine) counts cardinality-constrained rows; SA engine skips zeros in parallel, SLE engine handles dense cases.
  • Reuse-aware: SLE engine logic is shared for B&B solving; only add small queues for new sparse constraints (no dedicated B&B chip).
  • Tightly integrated: ISA extensions (VFC, VSASLE, VBB) map to existing CPU pipeline; no separate accelerator, no costly data shuttling.
  • L1 prefetcher: for 35% of benchmarks overflowing 128KB L1, sequential prefetch hides L2→L1 latency (deterministic top-to-bottom access).
  • Speedup and energy improvements vary by workload characteristics (sparsity, size) as evaluated on MIPLIB benchmarks.
Six everyday pictures

How to think about SPARK

Each analogy stands alone. Every card ends with where it breaks.

Core idea · In-memory dot product

Computing on the shelf, not at the checkout

A grocery store analogy: traditional CPU = fetch every item from the shelf to the checkout counter (ALU), scan, return. PIM = a scanner attached to each shelf — compute the total on the shelf itself, send only the result to checkout. SPARK stores constraint rows (C) on the "shelf" (L1 SRAM array), maps variables (X) onto shelf-scanners (RBL precharge), reads the dot-product from shelf-outputs (row buffers). No fetching individual coefficients to CPU registers.

Where it breaks: Real shelves are passive; SRAM bitcells actively discharge RBL based on stored value. The analogy simplifies the active sensing circuitry. Also, not all operations happen on the shelf — subtraction/division still need near-shelf logic (counters at the end of the aisle).
Sparsity detection · FC engine

Skipping blank pages in a book

Imagine reading a 1000-page book where 990 pages are blank. You could read every page sequentially (CPU approach with software checks), or flip through and count blank pages up front (hardware sparsity detection). SPARK's FC engine does the latter: it scans constraint rows, counts how many are exactly "Xi ≤ Di" (cardinality-constrained), and if there are exactly n such rows for n variables, declares the problem sparse and activates the SA engine. Parallel counters check all rows at once — far faster than software loops.

Where it breaks: A book's blank pages don't change; an ILP's sparsity pattern is fixed per problem. The analogy doesn't capture that FC runs once per ILP setup, not continuously. Also, sparse doesn't mean "all blanks" — it means a specific structure (CC array vs C array).
Reuse-awareness · Sharing logic across engines

A Swiss Army knife for compute

Instead of four separate tools (saw, knife, screwdriver, corkscrew), a Swiss Army knife shares the handle and folds tools in/out. SPARK's four engines (FC, SA, SLE, B&B) share the same near-L1 arithmetic units (shift-add, adder-reduction, subtractors, dividers). When solving a sparse ILP, SA engine "unfolds" the shift-add for parallel MACs between C and CC arrays; for dense ILP, SLE engine unfolds the same shift-add for Jacobi iteration. B&B reuses SLE's logic plus adds small queues for new constraints — no dedicated B&B "tool."

Where it breaks: Real Swiss Army knives have mechanical linkages; SPARK's "unfolding" is ISA-driven (VFC/VSASLE/VBB instructions select which engine activates). Also, the analogy oversimplifies the control logic — some engines (FC, SA) can be gated off entirely when not needed, unlike a physical tool that's always there.
Prefetching · Hiding L2→L1 latency

Restocking the fridge before you're hungry

You don't wait until the fridge is empty to go grocery shopping — you refill it proactively. SPARK's prefetcher does the same for ILP constraints that overflow L1 (35% of MIPLIB benchmarks): it starts sequential L2→L1 fetches early, filling the next batch of constraint rows into L1 while the current batch is still being computed. Because ILP iterations access constraints top-to-bottom deterministically, the prefetcher knows exactly what to fetch next. Fill latency is amortized (hidden by ongoing compute).

Where it breaks: A fridge refill is discrete; L1 cache can handle simultaneous read (compute mode) and write (fill mode) to different indices via decoupled ports. The analogy doesn't capture that SPARK evicts the least-recently-computed constraint (LRU-style) to make room for new data. Also, prefetching helps only for sequential access patterns — random access would thrash.
Integration · Tightly coupled vs dedicated accelerator

Built-in calculator vs separate device

A smartphone has a built-in calculator app (tightly coupled — same OS, instant access). A separate physical calculator is a dedicated device (needs picking up, powering on, data entry). SPARK is the built-in app: it reconfigures existing L1 cache, adds ISA extensions (VFC/VSASLE/VBB) that map to the CPU pipeline, shares the same memory hierarchy. No separate chip, no PCIe/CXL link overhead, no distinct programming model. A dedicated ILP accelerator would be the external calculator — higher peak throughput maybe, but data transfer cost and integration overhead kill it for small/medium ILPs.

Where it breaks: A calculator app is software; SPARK is hardware (near-L1 logic gates). The analogy oversimplifies that SPARK does require new silicon (shift-add, counters), just minimal (~1.4% CPU area). Also, the built-in app can be updated; SPARK's logic is fixed post-tapeout (though ISA extensions are programmable).
Energy efficiency · Avoiding data movement

Local farmer vs shipped produce

Growing vegetables in your backyard (local) vs importing them from another country (shipped) — same nutrition, vastly different energy cost. SPARK's in/near-memory compute is the backyard: constraint matrix C lives in L1, dot-product happens there, only the result (updated X) moves to near-L1 logic. Energy cost: ~0.001pJ precharge per column (PIM) + small shifter/adder power. CPU/GPU baseline is shipped produce: fetch C and X from L1 to registers, compute in ALU, write back — higher data movement cost across iterations. Orders of magnitude more energy for large, sparse ILPs.

Where it breaks: Farming has seasons; SPARK's energy cost is constant per operation (no "harvest time"). The analogy doesn't capture that for very small ILPs (few constraints), the CPU's superior clock speed might outweigh data movement — but MIPLIB benchmarks are large enough that movement dominates. Also, "local" isn't zero-cost; PIM still needs RBL discharge energy, just far less than long-wire transfers.
The full explanation

How it actually works

Section by section, grounded in the paper's numbers and diagrams.

1 · The bottleneck — CPU/GPU aren't built for sparse, control-heavy ILP

CPUs (Zen3 + Gurobi) use software sparsity optimizations but still hit Von Neumann limits: limited ALU throughput (even with AVX), high latency fetching sparse coefficients from L1/L2. Multi-threading helps but saturates beyond ~16 threads (Koch et al. [40] show no scaling past 8000 cores). GPUs (Tesla V100 + cuSparse) underutilize streaming engines due to 65–99% sparsity (warp divergence on zero-skip checks) and thread divergence in B&B branching (periodic host↔GPU synchronization). Result: ILPs take tens of hours, consuming substantial energy on CPU.

🔬
Measured on MIPLIB 2017: The paper reports speedups and energy improvements that vary by benchmark characteristics. For highly sparse benchmarks, SPARK shows significant advantages over CPU and GPU baselines in both performance and energy efficiency. (Paper Fig. 15, 16)

2 · Sparsity-aware algorithm — skipping insignificant compute

For an ILP with m constraints and n variables (m ≥ n), SPARK's algorithm partitions constraints into cardinality-constrained (CC) array (rows of form Xi ≤ Di, exactly one non-zero) and general constraints (C) array (other rows). If CC array has exactly n rows, the ILP is sparse. The SA engine then finds potential solutions by:

  1. Dot-product between C and CC arrays: For each general constraint row j, compute Σ Cji · (Di from CC array row i). This uses PIM's parallel MAC — 32 operations/cycle per core for 16-bit coefficients (1 shift-add per 16 columns per bank).
  2. Subtraction and division: Near-L1 subtractors (1 per bank) compute Dj − (dot-product); dividers (1 per bank) find potential solutions by solving for each variable using the remaining n-1 CC values: for variable k, compute (Dj − Σ_{i≠k} Cji·Di) / Cjk. These are potential solutions.
  3. Cost calculation: Near-L1 MAC computes cost F(X) = Σ Rj·Xj for each potential solution. Enqueue into potential-cost (PC) array. Max/min logic (shared across L1 cache) identifies optimal cost.
🍯
Why it's fast: Step 1's PIM MACs run in parallel across columns. Steps 2–3 are coarse-grain (1 per bank) but still near-memory, avoiding CPU roundtrip. The paper demonstrates significant speedups for sparse ILPs through this approach. (Paper §V-A, Fig. 9)

3 · In-memory dot-product — how PIM works in 8T SRAM

SPARK stores constraint coefficients Cij in 8T SRAM bitcells (storage node SN). Variables Xj are mapped onto column read-bit-lines (RBL) via data-dependent precharge: if Xj='1', RBL precharged to Vcc; if Xj='0', RBL precharged to Vcc/2. When read-word-line (RWL) fires:

  • If Cij='1' (stored in bitcell) and Xj='1' (RBL at Vcc), RBL discharges through read-port transistors below Vcc/2 → sensed as '1' (dot-product bit = 1).
  • If Cij='0' or Xj='0', RBL stays ≥ Vcc/2 → sensed as '0' (dot-product bit = 0).

Row-buffer (flip-flops at column outputs) captures the bit-wise dot-product. Near-L1 shift-and-add (s-a) logic (1 per 16 columns per bank) combines partial products: for 16-bit coefficients, replicate C/X across 2 banks, compute 0th and 1st bit products in parallel, shift 1st product left 1 bit, add. Adder-reduction (AR) (1 per bank) sums shift-add outputs across all columns to get final MAC result. Latency: 2ns SRAM read + 1 cycle shift-add + 1 cycle AR ≈ 6ns for a 16-bit MAC (vs ~10 cycles CPU latency including fetch).

💡
Why no array modification? SPARK reuses the existing 8T decoupled read port (RBL/RWL). The only addition is a 2:1 mux on periphery to select Vcc vs Vcc/2 precharge (0.005mm² per core). Write port (WBL/WWL) unchanged. Sense-amplifier logic repurposed to detect RBL voltage. No bitcell redesign, no extra rows/columns. (Paper §IV-F, Fig. 8)

4 · SLE engine — Jacobi iteration with near-L1 arithmetic

For dense ILPs (or after SA finds initial solution for sparse ILPs), the SLE engine executes Jacobi iterative method to converge X values. Algorithm (from Fig. 2.b in paper):

# Step 1: compute new X[i] = (D[i] − Σ_{j≠i} C[i,j]·X[j]) / C[i,i]
for i in constraints:
    MAC_result = PIM_dot_product(C[i,:], X[:])  # in-L1, 32 MACs/cycle
    temp = near_L1_subtract(D[i], MAC_result)   # 1 subtractor per bank
    X_new[i] = near_L1_divide(temp, C[i,i])      # 1 divider per bank

# Step 2: check convergence (L1 norm)
norm = near_L1_abs_sum(X_new − X_old)           # adder-reduction + abs
if norm < epsilon: break

The PIM dot-product runs in parallel across all variables (banks); subtraction and division are per-bank sequential but still near-memory. L1-norm is coarse-grain (one adder-reduction tree shared across banks). The paper reports significant speedups for Jacobi iteration using this approach.

🍯
Regularizing divider trick: Division is area-intensive. SPARK uses an approximate divider (subtraction of mantissa first m bits) with a 64B lookup table (0.02mm² per core) to correct error, achieving 0.2% average error on MIPLIB benchmarks and 0.5ns latency (single-cycle). This helps regularization (avoiding local minima) while keeping area low. (Paper §VIII-C)

5 · B&B engine — reusing SLE logic for integer rounding

After SLE converges to a non-integer solution X, B&B rounds it to integers. Standard B&B (Fig. 2.c) branches on fractional variables, adding constraints like X[k] ≤ floor(X[k]) or X[k] ≥ ceil(X[k]), re-solving SLE at each node. SPARK's reuse-aware approach:

  1. Compute global upper/lower bounds: Global UB = F(X) when X = ceil(solution from SLE). Local LB = ceil(Σ Aj·Xj). Uses near-L1 MAC + ceil logic (shared).
  2. Identify branching variable: Max logic (shared) finds variable with least fractional part. Queues (1024-entry arrays near-L1) store branch value/variable/parent-node-index.
  3. Re-solve with updated constraints: For each branch, add the new constraint (e.g., X[k] ≤ 5) to constraint matrix C. Reuse SLE engine to solve updated system — no dedicated B&B compute hardware. If the new constraint is sparse (Xi ≤ const), store it in a near-memory queue; verify it via simple MAC (1 per bank) without replicating full SLE engine.
  4. Prune: If local LB = global UB, or solution is integer, or infeasible, stop branching. Queues track which child nodes to prune based on parent.
💡
Why reuse matters for area: Without reuse, a dedicated B&B engine would need its own shift-add/AR/sub/div logic (~0.3mm² per core), doubling near-L1 area. SPARK instead shares the SLE engine's logic and adds only small queues (0.11mm² total for control/queues). The paper reports energy and area benefits from this reuse-aware design. (Paper §V-B, Fig. 10)

6 · ISA extensions and execution flow

SPARK adds three instructions (Fig. 11): VFC (invoke FC engine for sparsity detect), VSASLE (invoke SA or SLE engine based on sparsity), VBB (invoke B&B engine). Registers: VS (sparse bit), VX (X vector, maps to iteration queues), VC (cost), plus control register to enable L1 compute mode. Execution flow (Fig. 13, 14):

# Dense ILP example
VFC      # FC engine: load C,D from L1, count CC rows → VS=0 (dense)
VSASLE   # SLE engine: PIM MACs, near-L1 sub/div, iterate till L1-norm < ε
VBB      # B&B engine: compute bounds, branch, reuse SLE for child nodes

# Sparse ILP example
VFC      # FC engine: count CC rows → VS=1 (sparse)
VSASLE   # SA engine: PIM MAC(C,CC), near-L1 sub/div for potentials, max cost
VBB      # NOP (B&B gated off for sparse; SA already gave integer solution)

Compiler support: existing CPU programming models (sequential, multithreading) work. Gurobi/CPLEX APIs remain, but backend compiler targets VFC/VSASLE/VBB instead of x86 loops. Syscall interface for kernel-mode ILP solving (e.g., tiering decisions).

Follow the numbers

Worked examples

Four traces using the paper's real numbers, step by step.

① Sparsity detection for a 3-variable investment ILP

Investment problem: 3 variables (X1, X2, X3), constraints X1 ≤ 10, X2 ≤ 20, X3 ≤ 15 (cardinality-constrained), plus 2X1 + 3X2 + X3 ≤ 50 (general). How does FC engine detect sparsity?

  1. Load constraints into L1 cache: First 3 rows (Xi ≤ Di) stored as CC array candidates. Fourth row (2X1 + 3X2 + X3 ≤ 50) stored in C array.
  2. FC engine scans each row: Parallel counters (one per bank) count non-zero coefficients per row. Rows 1–3 each have exactly 1 non-zero → pushed to CC array. Row 4 has 3 non-zeros → pushed to C array.
  3. Sparsity verdict: CC array has 3 rows = n (number of variables). ✓ Sparse. FC sets VS register = 1.
  4. Next step: VSASLE instruction activates SA engine (not SLE engine), because VS=1.
🍯
Hardware parallelism: The FC engine uses parallel counters to quickly classify constraints. The paper demonstrates this sparsity detection enables significant performance improvements for appropriate workloads. (Paper §V-F, step 3 in Fig. 13)

② SA engine potential-solution compute (sparse ILP)

Continuing the investment ILP: CC array = {X1≤10, X2≤20, X3≤15}, C array = {2X1+3X2+X3≤50}. How does SA engine find potential solutions and optimal cost?

  1. PIM dot-product between C and CC: For the general constraint, compute 2·(10) + 3·(20) + 1·(15) = 20 + 60 + 15 = 95 using PIM MAC (C coefficients in memory, CC Di values mapped to RBL). 3 MACs in parallel (1 bank), result in row buffer.
  2. Near-L1 subtraction: D (=50) − 95 = −45. Subtractor (1 per bank) outputs −45.
  3. Near-L1 division for potential solutions: For each variable k, compute potential Xk by solving for that variable using the remaining n-1 CC values. The algorithm substitutes the n-1 CC variables to isolate the nth variable. For X1: substitute X2=20 and X3=15, giving 2·X1 + 3·(20) + 1·(15) = 50, solving to X1 = (50 − 60 − 15)/2 = −12.5 (infeasible, negative). Similarly for X2 and X3.
  4. Cost calculation: For feasible potential solutions (in this case none, but hypothetically if X_pot = [5, 10, 8]), cost F(X) = Σ Rj·Xj using near-L1 MAC. Max logic finds optimal.
  5. Result: SA engine identifies potential solutions efficiently through parallel PIM operations.
💡
This example shows the SA engine's approach to finding potential solutions for sparse ILPs through parallel PIM MACs and near-memory arithmetic. (Paper §V-F, step 4 in Fig. 13)

③ SLE Jacobi iteration (dense ILP, 3 variables)

Dense ILP: 3 constraints, 3 variables. Constraint matrix C = [[4, 1, 0], [2, 5, 1], [1, 2, 6]], D = [7, 10, 12]. Initial X = [0, 0, 0]. Compute one Jacobi iteration.

  1. PIM MAC for first variable (X1): Compute 1·X2 + 0·X3 = 1·0 + 0·0 = 0 (excluding diagonal C[0,0]=4). PIM reads C[0,1], C[0,2] from memory, maps X[1], X[2] to RBL, outputs 0 in row buffer.
  2. Near-L1 subtraction: D[0] − MAC = 7 − 0 = 7. Subtractor (1 per bank) outputs 7.
  3. Near-L1 division: X1_new = 7 / C[0,0] = 7 / 4 = 1.75. Divider (1 per bank) outputs 1.75. Store in iteration queue (VX register).
  4. Repeat for X2, X3: X2_new = (10 − 2·1.75 − 1·0) / 5 = (10 − 3.5) / 5 = 1.3. X3_new = (12 − 1·1.75 − 2·1.3) / 6 = (12 − 1.75 − 2.6) / 6 ≈ 1.275.
  5. L1 norm check: |X_new − X_old| = |1.75 + 1.3 + 1.275 − 0| = 4.325. If threshold ε=0.01, not converged → iterate again. Adder-reduction (1 per bank) computes sum.
  6. Latency: The paper reports significant speedups for Jacobi iteration using SPARK's PIM and near-memory arithmetic compared to CPU baselines.
🍯
The SLE engine demonstrates how SPARK accelerates dense ILP solving through in-memory MACs and near-cache arithmetic operations. The paper reports performance and energy benefits from this approach. (Paper Fig. 14, step 4; Fig. 18.a,b)

④ B&B reuse-aware approach (3 variables, 1 branch level)

SLE converged to X_sol = [1.8, 2.3, 1.6]. B&B rounds to integers. How does reuse-aware B&B work for one branch level?

  1. Compute bounds (stage 1): Global UB = F(X_ceil) = cost when X = [2, 3, 2]. Near-L1 MAC computes UB. Local LB = ceil(Σ Aj·Xj). Store in global-LB register and local-UB array (1024 entries).
  2. Identify branching variable (stage 2): Fractional parts: X1=0.8, X2=0.3, X3=0.6. Max logic (shared) finds X1 (largest fractional part). Store in branch-variable queue. Branch value = floor(1.8)=1 or ceil(1.8)=2 → two child nodes.
  3. Re-solve with new constraint (stage 3): Add X1 ≤ 1 to constraint matrix C. This is a sparse constraint (form Xi ≤ const). Reuse SLE engine to solve the updated 4-constraint system (original 3 dense + 1 new sparse). PIM MAC handles dense part; near-memory queue + simple MAC verifies X1 ≤ 1 without replicating full SLE engine.
  4. Prune (stage 5): If new solution is integer or local LB = global UB, prune. Queues track parent-node-index to invalidate child if parent pruned.
  5. Performance: The reuse-aware design enables efficient B&B execution by sharing SLE logic rather than requiring dedicated hardware.
💡
The B&B engine demonstrates SPARK's reuse-aware design, where the SLE engine's logic is shared for branch solving with only small additional queues needed. The paper reports area and energy benefits from this approach. (Paper §V-G, Fig. 14 step 5; Fig. 10.b)

Practice problems

Try each before opening the answer. Every one ends with the misconception it checks.

Problem 1 · easy
A 1000-variable ILP has 1000 constraints of the form Xi ≤ Di (cardinality-constrained). Is it sparse or dense according to SPARK's FC engine? Which engine executes next?
Show answer
Sparse. FC engine counts 1000 rows with exactly 1 non-zero = n (number of variables). CC array has n rows → VS=1. Next: VSASLE activates SA engine, not SLE. SA computes potential solutions via PIM MAC between C (empty in this case, all constraints are CC) and CC arrays. For problems where all constraints are cardinality-constrained, the solution approach is simplified.Misconception: that "sparse" means "some zeros." SPARK's definition is stricter: exactly n cardinality-constrained rows. A matrix with 50% zeros but no CC rows is dense to SPARK.
Problem 2 · medium
Why does SPARK use an approximate divider with a lookup table instead of an exact hardware divider? What's the tradeoff?
Show answer
Exact dividers are area-intensive (large gate count, slow). SPARK's approximate divider (subtraction of mantissa first m bits) + 64B lookup table (0.02mm² per core) achieves 0.5ns latency (single-cycle) vs ~5ns for exact divider, and lower energy cost. Tradeoff: 0.2% average error on MIPLIB benchmarks, but this helps regularization (avoiding local minima in Jacobi/B&B), so it's actually beneficial for convergence. Area saved: ~0.1mm² per core. (Paper §VIII-C)Misconception: that approximation always hurts accuracy. Here, slight error aids algorithmic regularization (like annealing) while being cheaper and faster.
Problem 3 · hard
SPARK achieves speedups over Zen3 CPU for sparse ILPs. Break down the speedup sources: data movement reduction, parallel PIM compute, and sparsity-aware execution. Which contributes most?
Show answer
The paper reports that multiple factors contribute to SPARK's performance: i) Data movement reduction (L1 prefetching + in/near-memory compute) helps for workloads that overflow L1. ii) Parallel PIM compute (32 MACs/cycle vs CPU ~2 MACs/cycle after fetch) provides throughput benefits across workloads. iii) Sparsity-aware execution (SA engine skipping zeros, turning off unused engines) contributes for highly sparse workloads. The relative contribution varies by workload characteristics. Parallel PIM throughput is a key foundation that enables the other benefits. (Paper §VII, Fig. 15)Misconception: that sparsity-awareness is the main gain. While important for sparse workloads, PIM's throughput advantage is fundamental — without it, sparsity-aware optimizations would have less impact.
Problem 4 · medium
For a 10,000-variable dense ILP that doesn't fit in 128KB L1, SPARK uses prefetching. Describe the prefetch strategy and why it doesn't hurt performance.
Show answer
SPARK's prefetcher: i) Sequential access order: constraints accessed top-to-bottom (deterministic in Jacobi iteration). ii) Early initiation: prefetch next batch of constraint rows from L2→L1 while current batch is being computed in PIM. iii) Timeliness: fill latency (L2→L1) is amortized because compute on current batch takes longer than the fetch. iv) Eviction policy: replace least-recently-computed constraint (LRU-style). Why it works: Jacobi convergence is independent of variable update order, so fetching rows while computing others doesn't stall. 8T decoupled ports allow simultaneous read (compute) and write (fill) to different indices. The paper reports that energy efficiency remains significantly better than using LLC. (Paper §IV-I)Misconception: that prefetching requires complex prediction logic. SPARK's is straightforward (sequential) because ILP access pattern is deterministic.
Problem 5 · easy
SPARK's near-L1 logic occupies 0.37mm² per core (~1.4% of a CPU). List the five components and their area breakdown.
Show answer
From paper §VII-H: i) Sparsity detection counters (FC engine): 0.03mm² / 0.1%. ii) Subtractors/dividers: 0.11mm² / 0.4%. iii) Shift-add/adder-reduction: 0.1mm² / 0.4%. iv) Comparators (max/ceil logic): 0.02mm² / 0.1%. v) Control/queues (B&B, iteration tracking): 0.11mm² / 0.4%. Total: 0.37mm². Reduced area is due to resource sharing across FC/SA/SLE/B&B engines. Peripheral logic (2:1 mux for precharge): 0.005mm². Memory array (8T SRAM): 0.08mm² (unchanged).Misconception: that near-memory logic is "free." It's small (~1.4% CPU area) but real silicon. The win is that it's shared across engines, not replicated four times.
Problem 6 · hard
SPARK's energy improvement over CPU varies between sparse and dense ILP. Explain the difference using data movement and compute breakdown.
Show answer
Sparse ILP (e.g., highly sparse benchmarks): i) CPU wastes energy fetching/checking many zero coefficients (software loop overhead). ii) SPARK's SA engine skips zeros entirely in hardware (FC detects sparsity, SA only computes on non-zeros). iii) Data movement: CPU fetches full constraint matrix from L1→registers repeatedly across iterations. SPARK keeps it in L1, only moves updated X (small vector). iv) Result: larger energy improvement = data movement savings + sparse-compute savings. Dense ILP: i) No zeros to skip, so SA engine doesn't activate. SLE engine handles all constraints. ii) Data movement savings remain, but no sparse-compute savings. iii) Result: energy improvement primarily from data movement reduction. Sparsity-awareness adds energy benefits on top of PIM's movement savings for appropriate workloads. (Paper Fig. 16, Fig. 18)Misconception: that sparse problems are always easier. They benefit more when you can skip zeros (SPARK does), but CPU/GPU software loops still iterate over entries, wasting energy.
Problem 7 · medium
Why does SPARK show different speedups for sparse vs dense ILP compared to GPU? What changes?
Show answer
Sparse ILP: i) GPU (V100 + cuSparse) suffers from warp divergence (threads skip different zero-entries, under-utilizing SIMD width). ii) B&B branching causes thread divergence (different threads explore different nodes, periodic host-GPU sync). iii) cuSparse library overhead (dynamic dispatch, memory indirection). iv) SPARK's SA engine + hardware sparsity detection avoid these issues. Dense ILP: i) GPU's streaming engines are better utilized (no warp divergence on non-zeros). ii) B&B still causes thread divergence, but it's a smaller fraction of total time. iii) Data movement: host↔GPU transfer for constraint matrix (one-time cost) + updated X (every iteration). SPARK's in-L1 compute avoids this. Sparsity magnifies SPARK's advantage. (Paper §VII-B, Fig. 15.c)Misconception: that GPUs always win on dense linear algebra. They excel for pure matrix operations (GEMM), but ILP's iterative + branching nature reduces GPU efficiency even when dense.
Problem 8 · easy
SPARK adds three ISA instructions (VFC, VSASLE, VBB). Where in the CPU pipeline do they execute, and what state do they modify?
Show answer
i) VFC: decoded in front-end (like any instruction), signals back-end to activate FC engine (near-L1 logic). Modifies VS register (sparse bit). ii) VSASLE: signals SA or SLE engine based on VS. Modifies VX register (updated X values, mapped to iteration queues) and VC register (cost). iii) VBB: signals B&B engine. Reads VX, modifies branch queues (near-L1). All three bypass traditional ALU pipeline — they go directly to near-L1 logic, which returns results to registers. Control register (written via MSR-like instruction) enables/disables L1 compute mode. (Paper §V-C, §V-D)Misconception: that these instructions require a new execution unit in the core's back-end. They don't — the execution happens in near-L1 logic (off the critical path). The core's pipeline just dispatches them and waits for completion.
Quick answers

Frequently asked questions

Ten questions newcomers ask most. Tap any to expand.

Q1Why reconfigure L1 cache instead of building a dedicated accelerator?▾
Tight integration. A dedicated chip requires PCIe/CXL links (data transfer cost), separate memory (coherence overhead), distinct programming model (compiler support). For ILPs, 65% fit in 128KB L1, and even 35% that overflow benefit from L1 prefetching (20–25× better energy than LLC). SPARK's near-L1 logic is tiny (~1.4% CPU area), reuses existing ISA framework (just 3 new instructions), and stays on-die (no off-chip BW). For small/medium ILPs, tight coupling wins; for huge ILPs (100K+ variables), a dedicated chip might be worth it, but those are rare in real-time decision-making.
Q2How does SPARK's sparsity detection differ from cuSparse (GPU library)?▾
cuSparse detects sparsity in software (host CPU pre-processes matrix, converts to CSR/COO format, uploads to GPU). SPARK detects sparsity in hardware (FC engine parallel counters scan constraint rows on the fly, classify into CC/C arrays). The paper shows FC engine provides fast sparsity classification. Also, cuSparse still suffers from warp divergence on irregular sparsity; SPARK's SA engine skips zeros entirely (no divergence). Energy: cuSparse incurs host↔GPU transfer overhead; SPARK's detection is near-memory (low energy per operation).
Q3Does SPARK work for non-ILP problems (e.g., linear regression, neural nets)?▾
Partially. SPARK's PIM MAC + near-L1 arithmetic can accelerate any dot-product-heavy workload. Linear regression (solve normal equations A^T A x = A^T b) benefits from SLE engine. Sparse neural nets (activation sparsity) could use SA engine if sparsity pattern matches CC-array structure. However, SPARK is optimized for iterative sparse linear systems with control flow (Jacobi + B&B). Dense CNNs wouldn't benefit much (TPU/GPU better). Irregular sparse patterns (e.g., GNNs) might not fit CC/C array classification. SPARK is a domain-specific accelerator for ILP/LP, not a general-purpose PIM.
Q4What happens to SPARK's logic when the CPU runs normal (non-ILP) workloads?▾
It's gated off (clock/power gated). The control register that enables L1 compute mode is cleared, SPARK instructions (VFC/VSASLE/VBB) become NOPs or trap. L1 cache operates normally (8T SRAM read/write via standard ports, no PIM). Near-L1 logic (shift-add, counters, etc.) is idle, consuming zero dynamic power. Critical path unchanged: no extra gate depth on normal reads (the 2:1 mux is on periphery, delay absorbed). Area overhead is permanent (~1.4% CPU), but power overhead is zero when not in use. This is key for integration — SPARK doesn't slow down the CPU on non-ILP tasks.
Q5How does SPARK handle cache coherence when multiple cores solve ILPs in parallel?▾
SPARK assumes MESI protocol (or similar). Constraint matrices C, D, R are read-only → can be replicated across cores' L1 caches (shared state). Updated X vectors are core-private (each core solves a different ILP or different branch of same ILP). If X is shared (e.g., multi-threaded ILP solver), writes to X trigger coherence invalidations (same as normal CPU). SPARK's ISA extensions don't bypass coherence — they just invoke near-L1 logic, which reads/writes L1 via standard cache interface. For tiering decisions (kernel use case), each core solves independent page-placement ILP, so no sharing.
Q6Why 8T SRAM instead of 6T? What's the benefit for SPARK?▾
8T has decoupled read/write ports (separate RBL/RWL for reads, WBL/WWL for writes). Benefit: i) Read-after-write in 2 cycles (vs 3 for 6T) — important for Jacobi iteration (read old X, write new X in tight loop). ii) Simultaneous read/write to different indices — enables L1 prefetching (write new constraints via fill port) while computing (read via PIM port). iii) Reuse read port for compute — SPARK's PIM uses RBL/RWL without modifying write port or bitcell. Cost: 8T is ~30% larger area than 6T, but modern CPUs (Zen3, Intel) already use 8T for L1 due to performance benefit. SPARK gets PIM "for free" on top of existing 8T.
Q7How does SPARK compare to Ising/Boltzmann accelerators for combinatorial optimization?▾
Ising accelerators (e.g., coherent Ising machines, quantum annealers) map problems onto spin-glass models (minimize Hamiltonian energy). Limitations for ILP: i) Binary variables only — Ising spins are ±1, limiting to binary ILP (0/1 variables). Real ILPs have integer ranges (Xi ∈ [0, 100]). ii) Quadratic Hamiltonian — existing accelerators assume energy = Σ Sj·ICj (product of spin and interaction coefficient). General ILPs require different formulation. iii) Annealing overhead — slow convergence for large problems. SPARK handles general integer ILPs, converges via deterministic Jacobi+B&B, and is tightly integrated (no separate chip). Ising accelerators are complementary (better for binary combinatorial problems like MAX-SAT), not a replacement.
Q8What contributes to SPARK's speedup: PIM throughput, data movement reduction, sparsity-awareness, or B&B reuse?▾
The paper shows multiple contributing factors: i) PIM throughput: 32 MACs/cycle vs CPU's lower effective throughput (after fetch overhead). ii) Data movement reduction: L1 prefetching + in-memory compute reduces transfers. iii) Sparsity-aware (SA engine): skipping zeros + early engine gating helps for highly sparse workloads. iv) B&B reuse: sharing SLE logic + near-memory queues saves area and energy. The relative contribution varies by workload characteristics. PIM throughput is foundational, enabling the other benefits. (Paper Fig. 15)
Q9Can SPARK accelerate LP (non-integer) problems, and how does it differ from ILP?▾
Yes. LP is ILP minus the integer constraint (no B&B step). SPARK's flow for LP: i) VFC detects sparsity → SA or SLE engine. ii) VSASLE solves SLE (Jacobi iteration) → converges to non-integer X. iii) VBB is NOP (B&B engine gated off, saving power). The paper reports speedup and energy improvements for LP workloads. LP execution benefits from gating off the B&B engine while using the same core PIM and near-memory arithmetic. SPARK is broadly applicable to linear systems, not just ILP. (Paper §VII-E, §VII-G, Fig. 17, 18)
Q10What are SPARK's limitations or cases where CPU/GPU might still win?▾
i) Very small ILPs (few variables/constraints): CPU's low setup overhead + high single-thread clock speed might win. SPARK has fixed overhead for FC engine and ISA dispatch. ii) Irregular sparsity: if sparsity pattern doesn't fit CC-array structure (e.g., random scattered zeros), SA engine can't activate, falling back to SLE (no sparsity benefit). iii) Huge ILPs (100K+ variables): may overflow even with prefetching. iv) Non-iterative algorithms: if problem uses Simplex (non-iterative pivoting), SPARK's Jacobi-focused design doesn't help. v) GPU-friendly workloads: dense ILPs with no branching (pure SLE) might benefit from GPU's higher peak throughput on newer GPUs, though data movement cost remains. SPARK targets the sweet spot: medium-to-large sparse ILPs with branching, where decision thresholds matter.

Supporting terms
Jacobi methodIterative solver for SLE: compute X_new[i] = (D[i] − Σ C[i,j]·X_old[j]) / C[i,i], repeat till L1-norm < ε.
Branch & Bound (B&B)Integer rounding algorithm: branch on fractional variables, add constraints, re-solve, prune when bounds match.
L1 normConvergence check for Jacobi: Σ |X_new[i] − X_old[i]| < ε. Computed via near-L1 adder-reduction + abs.
Shift-and-addNear-L1 logic (1 per 16 cols/bank) combining partial products for multi-bit MAC (e.g., 16-bit coefficients).
Adder-reduction (AR)Near-L1 tree adder (1 per bank) summing shift-add outputs across columns for final MAC result.
Regularizing dividerApproximate divider (mantissa subtraction + 64B LUT) achieving 0.5ns latency, 0.2% error, aids convergence.
PrefetcherSequential L2→L1 fetch hiding miss latency for ILPs overflowing 128KB L1 (35% of MIPLIB benchmarks).
VFC/VSASLE/VBBISA extensions invoking FC/SA-or-SLE/B&B engines. Decoded in front-end, execute in near-L1 logic.
VS registerSparse bit (1=sparse, 0=dense) set by FC engine, read by VSASLE to select SA or SLE.
VX registerVariable vector X, mapped to iteration queues (256 entries near-L1), updated by VSASLE.
VC registerCost value F(X), updated by SA/SLE engines, read by VBB for bound computation.
Gurobi / CPLEXState-of-the-art ILP solvers (software) using branch-and-cut, heuristics, multi-threading on CPUs.
cuSparseNvidia's sparse linear algebra library for GPUs. Preprocesses sparsity in software (CSR/COO format).
Connections to tiering/CXL/memory management

Page placement as ILP conceptual

OS kernel · tiering decision

Formulate DRAM/NVM page placement as ILP: variables = page locations (binary: DRAM=1, NVM=0), constraints = DRAM capacity ≤ C1, NVM BW ≤ C2, access-frequency thresholds. Objective = minimize weighted latency. SPARK's near-L1 solve could potentially enable faster tiering decisions, though specific kernel integration would require additional research.

CXL memory pool allocation system design

CXL.mem · distributed memory

CXL disaggregated memory: multiple hosts share remote memory pools. ILP formulation: variables = memory chunk assignments (which host gets which chunk), constraints = per-host capacity, CXL link BW, QoS guarantees. SPARK integrated into each host's CPU could solve local placement problems, though coordination mechanisms would need development.

Bandwidth-aware page migration OS / runtime

Heterogeneous memory · DRAM + NVM + CXL

Given per-page access counts (profiling), decide which pages to migrate DRAM↔NVM to maximize throughput under BW limits. ILP: variables = migration decisions (binary), constraints = total migrations ≤ M (migration BW), post-migration BW usage ≤ B_nvm, B_dram. SPARK's SA engine for sparse problems could potentially help, though real-world deployment would require validation.

Network routing in data centers datacenter networking

SDN · traffic scheduling

Route packets through a fat-tree topology under link-capacity constraints. ILP: variables = flow assignments to paths, constraints = link BW ≤ C_link, flow conservation, QoS deadlines. SPARK could potentially accelerate control-plane solvers, though integration with SDN controllers would need research.

Related work in memory systems

UPMEM — Processing-In-Memory for DRAM medium

Commercial PIM · 2019–present

First commercial PIM product: DRAM modules with embedded RISC-V cores (DRAM Processing Units, DPUs). Targets bulk-data operations (filter, join, sort). Comparison to SPARK: UPMEM is coarse-grain (CPU-like DPUs), SPARK is fine-grain (bitline-level PIM). UPMEM requires explicit data partitioning; SPARK is transparent (ISA extensions). UPMEM for big-data; SPARK for real-time optimization.

Samsung's HBM-PIM (HBM2-Aquabolt) hard

Industry PIM · 2021

High-Bandwidth Memory with in-memory MAC for AI/ML (GEMM acceleration). Uses bank-level MAC units (coarser than SPARK's column-level). Energy-efficient for dense matrix multiply. SPARK vs HBM-PIM: SPARK targets sparse, control-heavy ILP (not just GEMM), uses L1 cache (closer to CPU) vs HBM (farther), and handles both sparse/dense via engine reconfiguration (HBM-PIM is dense-focused).

TCAM for page-table walks easy

Memory management · TLB acceleration

Ternary CAM for fast address translation. Conceptual parallel: SPARK's near-L1 counters/comparators (FC engine) are a form of content-addressable logic (count cardinality constraints). Both bypass software loops via parallel hardware search. Difference: TCAM is exact-match; SPARK is threshold-based (count ≥ n).
