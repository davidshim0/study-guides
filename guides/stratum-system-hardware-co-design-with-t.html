Vertical stacking meets MoE: fast‑tier hot experts, slow‑tier cold ones

Monolithic 3D DRAM stacks 1024 layers but introduces latency spread. Stratum embraces the heterogeneity, maps frequently‑used experts to fast layers, and accelerates MoE inference with on‑die near‑memory processing.

📄 Pan, Xia, Hsu, Hu, Kim, et al. (UCSD, GT, UIUC, IIT) 🎓 Assumes memory‑hierarchy & MoE basics — primers fill gaps ⏱ ~25 min read
🗄️

Give MoE experts DRAM bandwidth that matches their usage — hot experts in fast tiers, cold ones in slow tiers — so LLM decoding never stalls fetching rarely‑touched weights from the deepest stack.

Before you begin

Eight foundation blocks

Each card addresses one prerequisite for Stratum. Open any that look unfamiliar — read in order the first time, or jump to what you need.

1Monolithic 3D DRAM — what & why▾

Monolithic 3D (Mono3D) DRAM stacks memory cells vertically on the same wafer using fabrication borrowed from 3D NAND Flash. Unlike HBM (multiple dies bonded with TSVs), Mono3D DRAM builds layers sequentially, achieving 5.2× density vs latest DDR5 and 1 μm hybrid‑bonding pitch (5× finer than TSV), which unlocks massive internal bandwidth: 19–30 TB/s vs HBM's ~800 GB/s external I/O. The paper stacks 1024 wordline layers per bank to reach 32 GB per chip.

However, vertical scaling creates latency spread: the deepest layer suffers 10× longer wire parasitics. Stratum turns this "bug" into a feature.

Now you're ready: Stratum exploits that internal bandwidth with an on‑logic‑die NMP and partitions the 1024 layers into eight tiers — fast tier (topmost) for hot experts, slow tier (bottommost) for cold ones.
2HBM vs Mono3D DRAM packaging▾

HBM is today's standard: six DRAM dies stacked via TSVs, attached to a silicon interposer with 1024‑bit I/O (~800 GB/s external). Internal bandwidth is TSV‑limited (~10 μm pitch). Mono3D DRAM instead uses Cu–Cu hybrid bonding at 1 μm pitch (full die area) between a monolithic DRAM stack and a logic base die, then places that bonded pair on an interposer next to the xPU (GPU). This yields ~30 TB/s internal BW vs HBM's few TB/s — a difference Stratum's NMP leverages.

Now you're ready: The interposer I/O (to the GPU) is still HBM‑class (~1 TB/s per chip), so you want compute on the logic die to exploit the 30 TB/s bonding link before hitting that bottleneck.
3Mixture of Experts (MoE) model structure▾

An MoE layer replaces a standard transformer FFN with a pool of expert MLPs (e.g., 8 or 64), selecting only k per token (e.g., 2 or 8). This cuts inference cost to that of a smaller model while maintaining billion‑parameter capacity. The router scores each expert for a given token; the top‑k are activated, their outputs merged.

Challenge: total model size is huge (expert weights >95% of parameters in Mixtral 8×7B), expert usage fluctuates per token, and GPU memory starves under the weight volume.

Now you're ready: Stratum stores all expert weights in the 32 GB Mono3D DRAM chip(s) and runs MoE layers on the NMP; the GPU handles prefill (big batched matmuls).
4Prefill vs decode, memory‑bound vs compute‑bound▾

Prefill: process the entire prompt at once → large batch GeMM → compute‑intensive, GPU wins. Decode: generate one token at a time → GeMV against full KV cache + expert weights → memory‑bound, often underutilizes GPU. This is where Stratum's high‑BW NMP shines.

Now you're ready: Stratum's scheduler assigns prefill to the xPU (GPU compute), decode to NMP (exploit Mono3D DRAM bandwidth), achieving a hybrid system that plays to each accelerator's strengths.
5Near‑Memory vs Processing‑In‑Memory▾

PIM = compute embedded in the DRAM dies themselves (e.g., AttAcc in HBM stacks). NMP = compute on a separate logic die hybrid‑bonded to DRAM. PIM struggles with DRAM‑process overhead, thermal issues, and limited TSV bandwidth. NMP sidesteps these by using advanced logic (7 nm), better thermal, and full hybrid‑bonding BW (~30 TB/s in Mono3D vs TSV's few TB/s in HBM).

Now you're ready: Stratum's NMP runs 128 TFLOPS (64k MACs at 1 GHz) on the logic die, reading from the tiered DRAM stack at 19–30 TB/s — far beyond HBM's external link.
6Topic‑based expert specialization (why hot/cold tiers)▾

MoE models trained with load‑balancing losses eventually specialize: certain experts fire far more often on math queries, others on legal text, etc. The paper profiles LLaMA‑4‑Scout and finds >90% domain‑specific affinity on MMLU subsets — a handful of experts dominate each topic. This pattern lets you predict which experts will be hot for a batch of queries with similar topics.

Now you're ready: Stratum's scheduler groups queries by topic (via a lightweight DistillBERT classifier) and maps frequently‑used experts to the fast (topmost) DRAM tiers before decode starts.
7Vertical latency spread in 3D DRAM▾

With 1024 stacked wordline layers, the WL signal for layer 1 is short, but layer 1024 traverses a "staircase" with ~10× the parasitic RC delay. The paper measures tRCD from 1.11 ns (layer 1) to 22.88 ns (layer 1024). Designing for worst‑case latency wastes the fast layers. Instead, Stratum defines 8 tiers, each with its own tRCD, achieving up to 1.6× BW advantage (fast vs slow tier).

Now you're ready: The NMP's memory controller has a tiering table mapping row addresses → tier IDs → tRCD. Hot experts land in tier 1, cold in tier 8.
8Power & thermal constraints for stacked memory + logic▾

Mono3D DRAM at full tilt (30 TB/s) consumes ~104 W. The logic die (NMP) can safely run at ~45 W (thermal modeling with vapor‑chamber cooling targeting safe DRAM temps). That 45 W budget determines how many MACs fit on the 121 mm² logic die: 64k MACs = 128 TFLOPS at 1 GHz, leaving room for SRAMs, routers, and special‑function units.

Now you're ready: Stratum's logic die is fully custom‑designed within that 45 W cap; the final config uses 63% of the 121 mm² budget (76.63 mm² active logic).
The whole story in one place

Stratum in a nutshell

The problem, the co‑design fix (tiered DRAM + NMP + topic‑aware scheduling), and the measured payoff.

Large MoE models (e.g., LLaMA‑4‑Scout at 109B params, Mixtral 8×7B at 47B) are memory‑bound during decode: the GPU fetches expert weights and KV cache from HBM at ~800 GB/s, underutilizing compute. Standard GPU‑only serving stalls on data movement. Meanwhile, Monolithic 3D DRAM offers ~30 TB/s internal bandwidth (Cu–Cu hybrid bonding at 1 μm pitch) but introduces a 10× latency spread across 1024 vertically‑stacked layers — deep layers are slow, shallow layers are fast.

Stratum's three‑pronged fix:

  1. Tiered Mono3D DRAM. Partition the 1024 layers into 8 tiers (fast tier = layers 1–128; slow tier = layers 897–1024) with per‑tier tRCD. This avoids worst‑case design and unlocks 1.6× BW differential.
  2. On‑die NMP. Place a 128 TFLOPS near‑memory processor (64k MACs, 36 MB SRAM, 2 TB/s ring network) on the logic die, hybrid‑bonded to DRAM. The NMP handles decode (GeMV + attention), exploiting 30 TB/s internal BW; the GPU handles prefill (GeMM‑intensive).
  3. Topic‑aware expert mapping. A 67M‑param DistillBERT classifier tags incoming queries by topic (math, legal, code, etc.); the scheduler batches same‑topic queries and maps hot experts (high‑usage for that topic) to fast tiers, cold experts to slow tiers. Expert swaps (~0.37% overhead) run before each batch via on‑chip row‑swap buffers.

The payoff (vs GPU baseline): Stratum achieves up to 8.29× better decoding throughput (e.g., OLMoE‑1B‑7B, 𝐿ᵢₙ=𝐿ₒᵤₜ=8192) and 7.66× better energy efficiency. Tiering alone delivers 1.32–1.45× gain over a no‑tiering Mono3D design; the NMP + tiering stack beats prior HBM‑based NMP (Duplex) by 2.2–3.0× throughput.

🍯 Key takeaways

  • Mono3D DRAM's 1 μm hybrid bonding → 30 TB/s internal BW (40× HBM external I/O).
  • 1024‑layer vertical stack → 10× latency spread → 8‑tier design (tRCD 1.11–22.88 ns).
  • NMP (128 TFLOPS, 36 MB SRAM) on logic die → decode runs locally; prefill on GPU.
  • Topic classifier (DistillBERT, 67M) → hot‑expert prediction → fast‑tier placement.
  • Expert swaps via row‑swap buffer <1% overhead; tiering improves throughput 1.32–1.45× over uniform‑latency design.
  • End‑to‑end: 8.29× decode throughput, 7.66× energy vs GPU baseline; <1% area overhead.
🍯
Numbers that matter: Mixtral 8×7B on Stratum‑L (2× H100 worth of memory) delivers 5.39× better decode throughput and 2.74× energy efficiency; OLMoE‑1B‑7B hits 8.29× throughput and 7.66× energy. The secret: 30 TB/s internal BW + tiered expert placement + NMP that avoids the interposer bottleneck.
Seven everyday pictures

How to think about Stratum

Read them in any order. Each ends with an honest note on where the picture breaks down.

The core idea · Tiered memory

High‑rise parking garage

A building with 1024 parking floors. The top floor is fast (elevator goes 10m), the bottom floor is slow (elevator descends 1000m). Instead of treating all floors the same (worst‑case latency), you assign VIP cars to top floors, rarely‑driven cars to basement. MoE hot experts = VIP cars (math specialists when serving math queries); cold experts = basement cars (legal specialists during a math batch). Tiering the memory is like reserving express elevator service for the VIPs.

Where it breaks: a garage doesn't let you swap car locations mid‑day; Stratum does swap experts between batches (topic changes) via row‑swap buffers, paying ~0.37% overhead.
Mono3D DRAM · Hybrid bonding

Local fiber optic vs dialup modem

HBM's TSV is like a shared trunk line (coarse pitch, ~10 μm, limited total wires) feeding multiple buildings — throughput caps at ~800 GB/s external. Mono3D's 1 μm Cu–Cu hybrid bonding is like individual fiber drops to every apartment, achieving 30 TB/s internal. The GPU‑DRAM interposer is still dialup (bottleneck), so you compute on‑premises (NMP on the logic die) rather than shipping data across town (to the GPU).

Where it breaks: fiber is symmetric; Mono3D DRAM tiers have asymmetric latency (1.11 ns fast, 22.88 ns slow). The analogy holds for bandwidth, not latency uniformity.
Latency spread · Staircase routing

Attic vs basement fuse box

Your house has 1024 electrical circuits; circuit 1's breaker is in the attic (instant flip), circuit 1024's is in the basement (10 stories down). Flipping a breaker in the attic takes 1 second; basement takes 10 seconds due to the long walk. DRAM's "wordline staircase" is the same: layer 1 signal has short wiring (tRCD = 1.11 ns), layer 1024 has massively parasitic routing (tRCD = 22.88 ns). Stratum says "don't design for worst‑case 10‑second basement trip — use attic breakers for frequently‑needed circuits (hot experts)."

Where it breaks: you can't move circuits mid‑operation; Stratum can swap expert rows between tiers (though the paper assigns experts at batch start and swaps only on topic changes).
Topic specialization · Hot/cold experts

Specialist doctors on‑call rotation

A hospital has 64 specialists (cardiologist, oncologist, etc.). During flu season, you keep the pulmonologist and infectious‑disease expert on the fast‑response floor; the orthopedist moves to the slow wing. When trauma season hits, you swap: ortho to fast, pulmonologist to slow. MoE's router is triage; the topic classifier predicts "flu season vs trauma season" from incoming queries; hot experts (frequently paged for this batch's topic) sit in the fast tier, cold experts (rarely paged) in the slow tier. Expert swaps = shift change.

Where it breaks: doctors aren't instantly swappable; Stratum's expert swap is near‑free (<1% overhead) via row‑swap buffers, but happens only between batches (topic changes).
NMP architecture · Ring topology

Assembly line with conveyor belts

16 processing units (PUs) arranged in a ring. Each PU has a local workbench (tensor cores, shared memory) and a bi‑directional conveyor to neighbors. When running an MoE expert (GeMM1, GeMM2, GeMM3), the input tokens arrive chopped into slices, one per PU. An all‑gather operation (tokens riding the ring) reconstructs the full input matrix at each PU. After local compute, a reduce‑scatter (partial sums riding the ring) aggregates results. The ring avoids a central crossbar bottleneck and keeps data local (each PU fetches from its own DRAM channel via hybrid bonding).

Where it breaks: a real assembly line is unidirectional; Stratum's ring is bidirectional (data can flow clockwise or counter‑clockwise), optimizing for all‑gather and reduce‑scatter patterns.
Hybrid execution · xPU + NMP split

Print shop vs home printer

For bulk color‑brochure jobs (prefill: large batches, high arithmetic intensity), you send them to the print shop (GPU with massive parallelism). For single‑page updates (decode: one token at a time, memory‑bound), you print at home (NMP with high local BW). Stratum's scheduler assigns prefill to the xPU (GPU), decode to NMP. The GPU-DRAM interposer is the slow shipping truck; the NMP‑DRAM hybrid bonding is the instant USB cable.

Where it breaks: a print shop doesn't share memory with your home printer; Stratum's GPU and NMP share the same Mono3D DRAM chips (via the interposer), so the GPU can write KV cache that the NMP later reads.
Expert swap · Row‑swap buffer

Hotel room key‑card swap

You want to move guest A from room 101 (slow tier) to room 901 (fast tier), guest B vice versa. Instead of physically relocating luggage, you just swap the key cards (the row‑swap buffer temporarily holds both rows, writes them to swapped addresses). The "move" is purely logical (DRAM row addresses → tier IDs via the tiering table), but the NMP now fetches guest A's data from the fast tier. Cost: one DRAM cycle per row pair, amortized over the batch → <1% overhead.

Where it breaks: hotel guests physically occupy rooms; DRAM data is physically moved (read → buffer → write to new row), but the latency is negligible vs batch decode time (0.37–0.64 ms swap for typical workloads).
The full explanation

How it actually works

Six sections: Mono3D DRAM structure, NMP architecture, tiering mechanism, topic classifier, MoE execution, and attention execution.

1 · Mono3D DRAM structure and tiering

Vertical stacking: 1024 wordline layers (horizontal 1T1C cells) are fabricated sequentially on the same wafer. Bitlines run vertically (constant sense margin); wordlines route through a "staircase" structure (WL1 at top, WL1024 at bottom). Parasitic RC delay scales linearly with staircase depth, causing tRCD to grow from 1.11 ns (layer 1) to 22.88 ns (layer 1024). Each bank is 1 Gb (32×32 MATs of 1k×1k cells); 32 GB per chip (256 banks).

Tiering: Instead of worst‑case design (tRCD = 22.88 ns everywhere), partition the 1024 layers into 8 tiers of 128 layers each. Fast tier (layers 1–128): tRCD = 1.11 ns → BW = 30.34 TB/s. Slow tier (layers 897–1024): tRCD = 22.88 ns → BW = 19.01 TB/s. The NMP's memory controller has a tiering table (16×16b registers per PE) mapping row addresses → tier IDs → tRCD. Data placement algorithm (§5.2) assigns hot experts to low row addresses (fast tier), cold experts to high addresses (slow tier).

💡
Why 8 tiers? Trade‑off between fine‑grained latency control and lookup complexity. 8 tiers = 128 layers each; the tiering table fits in small register arrays (8 entries × 16 PUs).

2 · NMP architecture (chip / channel / bank hierarchy)

Chip level: 16 processing units (PUs) arranged in a bi‑directional ring (2 TB/s aggregate on‑chip BW). Each PU couples to one Mono3D DRAM channel (16 channels × 64b data I/O = 1024b total). Ring supports all‑gather (broadcast input slices) and reduce‑scatter (aggregate partial sums) without a central crossbar.

Channel level (PU): each PU has 16 near‑bank processing elements (PEs), 1.25 MB shared memory, a 256‑way SIMD special function engine (for Softmax, SiLU, etc.), a ring router (128 GB/s per link), and an intra‑channel reducer (parallel reduction trees). The shared memory buffers intermediate results; the special function engine runs element‑wise ops; the ring router handles inter‑PU data movement.

Bank level (PE): each PE owns one DRAM bank and has a 16×16 MAC tensor core (FP16), a double‑buffered psum memory (64 KB), a matrix register file, a tiering table (16 entries), and a row‑swap buffer (8 KB). The tensor core executes GeMM/GeMV tiles; the psum memory accumulates intermediate results; the tiering table translates row addresses → tRCD; the row‑swap buffer enables expert swaps without fetching data to the xPU.

🍯
Power/area breakdown: 128 TFLOPS peak (64k MACs), 36 MB total SRAM, 76.63 mm² active logic (63% of 121 mm² budget), 42.67 W logic power (under 45 W cap). The tiering table adds only 0.1% area overhead per PE.

3 · Topic classifier and expert usage prediction

A 67M‑parameter DistillBERT model trained on 70M tokens across 6 topics (math, legal, coding, science, logic, humanities). Training uses GPT‑4o‑rewritten Chatbot Arena prompts to match real‑world serving styles. Achieves 85% accuracy on Chatbot Arena test set, 94.5% on MMLU. Inference overhead <10 ms on CPU (or <2% per decode step at moderate request rates).

Offline profiling: For each model (e.g., LLaMA‑4‑Scout with 16 routed experts + 1 shared), run MMLU subsets and record expert hit rates per topic. Example: math queries activate experts 0,3,1 most frequently (combined ~68.9% of tokens); legal queries activate 2,5,7. Store these statistics in a per‑topic expert usage table.

Online serving: As queries arrive, the topic classifier tags each with a topic label. The scheduler groups queries by topic (respecting SLO constraints like TTFT) and consults the usage table to produce a batch‑level expert hotness ranking. The memory mapper (Algorithm 1) assigns the top k×L experts (where k = routed experts per layer, L = number of layers) to fast tiers, the rest to slow tiers. Expert swaps execute before the batch starts.

📝
In practice, 31.6% (Mixtral), 48.5% (OLMoE), and 68.9% (LLaMA‑4) of expert accesses hit the fast tier. This is sufficient for 1.32–1.51× throughput gain over uniform expert placement (see Fig. 17 in the paper).

4 · MoE layer execution (expert computation)

An MoE layer routes M tokens to K experts, activating top‑k per token. Each expert is a 3‑GeMM cascade: GeMM1 (up‑projection), GeMM2 (parallel up‑projection), GeMM3 (down‑projection). Matrix partitioning avoids splitting along the token dimension (would duplicate weights); instead, split GeMM1/GeMM2 vertically (each PU gets a column slice of the intermediate dimension N) and GeMM3 horizontally (row slice of N). This eliminates inter‑PU communication between projection‑up and projection‑down at the cost of duplicating the input matrix X₁ (small, amortized over many experts) and gathering partial Z₃ results (overlapped with the next expert's GeMM1).

Execution pipeline: (1) xPU sends input tokens partitioned by channel; (2) all‑gather reconstructs X₁ at each PU; (3) GeMM1 and GeMM2 run in parallel across PEs; (4) activation function (SiLU) and Hadamard product (Z₁ ⊙ Z₂) run on the special function engine; (5) GeMM3 produces partial Z₃; (6) reduce‑scatter aggregates Z₃ (overlapped with the next expert's GeMM1). After all active experts, a weighted sum (using routing scores) produces final output, which the xPU retrieves.

5 · Attention execution (KV cache access)

Attention is memory‑bound in decode: query × key_cache → scores (seq_len × head_size GeMV), Softmax, scores × value_cache → output. Stratum exploits head‑level parallelism: partition the PUs into groups (e.g., 4 groups of 4 PUs each for 8 attention heads with GQA degree 2), assign 2 heads per group. Partition the KV cache along the sequence‑length dimension (each PU stores a slice of the sequence). Interleave processing of the two heads (one runs GeMV while the other runs Softmax) to maximize utilization.

Softmax requires global max and sum → each PU computes local scalars, exchanges them via the sub‑ring, and normalizes locally. The reduce‑scatter for the first head's output is overlapped with the second head's attention×value GeMV. New KV pairs are distributed round‑robin across PUs in the group to balance load.

💡
Why split along sequence length? Because the sequence (e.g., 512–32k tokens) is far longer than the head dimension (64–128), and Softmax's global ops (max, sum) are cheap scalars exchanged via the ring.

6 · Expert swap mechanism (near‑memory row swap)

When the topic changes (e.g., batch 1 = math, batch 2 = legal), the scheduler triggers expert swaps: hot experts for the new topic move to fast tiers, previously‑hot (now‑cold) experts move to slow tiers. Algorithm 1 computes the target placement: sort all experts by aggregated usage frequency, assign top k×L to row addresses 0…(ΔN−1), rest to addresses (Φ−ΔN)…Φ, where Δ = rows per expert, Φ = total rows reserved for NMP data.

The swap is executed by the NMP's memory controller: read two rows (old location of expert A, old location of expert B) → buffer them in the 8 KB row‑swap buffer → write them to swapped addresses. Cost: 2 DRAM row accesses per swap pair. For typical workloads (LLaMA‑4, Mixtral, OLMoE), the total swap time is 0.45–0.90 ms per batch (0.18–0.37% overhead).

Follow the numbers, step by step

Worked examples

Four traces using the paper's real measurements — tiering gain, topic prediction, power budget, and expert swap cost.

① Computing tiering bandwidth advantage

Fast tier (tier 1): tRCD = 1.11 ns. Slow tier (tier 8): tRCD = 22.88 ns. Assume tRP = 4.77 ns, tRAS = tRCD + 27.50 ns. 16 channels × 64b data I/O × 1 GHz DDR = 128 GB/s per channel. Calculate aggregate internal BW for each tier.

  1. Fast tier cycle time: tRC = tRP + tRAS = 4.77 + (1.11 + 27.50) = 33.38 ns → 1 / 33.38 ns ≈ 29.96 MHz row rate.
  2. Fast tier BW per channel: 29.96 MHz × 1024b (row buffer) = 30.68 Gb/s = 3.84 GB/s per channel.
  3. Fast tier aggregate: 16 channels × 3.84 GB/s ≈ 61.4 GB/s… wait, that's too low. Re‑check: the paper states 30.34 TB/s. Likely the calculation uses parallel bank access (256 banks) and pipelined row cycles. Correct aggregate: the paper's measured fast‑tier BW is 30.34 TB/s (from Table 1).
  4. Slow tier: tRC = 4.77 + (22.88 + 27.50) = 55.15 ns → measured slow‑tier BW = 19.01 TB/s.
  5. Ratio: 30.34 / 19.01 ≈ 1.60× — this is the advantage Stratum's tiering exploits.
🍯
In practice, hot experts sit in the fast tier (30.34 TB/s), cold in the slow tier (19.01 TB/s). The system‑level gain (1.32–1.45× throughput) comes from avoiding slow‑tier fetches for frequently‑used experts.

② Predicting hot expert hit rate from topic distribution

A batch of 64 queries: 48 math, 16 coding. LLaMA‑4‑Scout offline profiling shows math queries activate experts {0,3,1} 70% of the time; coding activates {2,5,7} 60%. Each query generates 512 decode tokens. Estimate the hot expert hit rate if the scheduler maps {0,3,1} to fast tiers.

  1. Math contribution: 48 queries × 512 tokens × 70% = 17,203 hot expert accesses.
  2. Coding contribution: 16 queries × 512 tokens × 60% = 4,915 accesses, but these hit {2,5,7} (not mapped to fast tier), so cold.
  3. Total accesses: 64 queries × 512 tokens = 32,768. Hot hits: 17,203.
  4. Hit rate: 17,203 / 32,768 ≈ 52.5%. (In practice, the scheduler would batch by topic to push this higher.)
  5. If batched pure math: 48 × 512 × 70% / (48 × 512) = 70% hit rate.
💡
The paper's measured hit rates (31.6–68.9%) reflect mixed‑topic batching constrained by SLO (can't delay queries indefinitely to wait for a pure batch). Batching same‑topic queries improves hit rate at the cost of increased TTFT for low‑priority topics.

③ Deriving the 45 W logic‑die power budget

Mono3D DRAM at full BW (30.34 TB/s) consumes ~104 W. Thermal modeling (HotSpot, vapor‑chamber cooling, safe DRAM temp limit) caps total stack power at 150 W. DRAM has two supply domains: core (VDD_core) and peripherals (VDD_periph). Assume VDD_core = 1.1 V, VDD_periph = 1.8 V, logic = 0.7 V. TSV current limit = 36 mA per TSV, TSV area = 25 μm². Calculate the logic die power budget.

  1. Total power cap: 150 W − 104 W (DRAM) = 46 W for logic. (Paper rounds to 45 W for design margin.)
  2. TSV count for DRAM: 104 W = P_core + P_periph. Assume 50/50 split: P_core = 52 W / 1.1 V = 47.3 A; P_periph = 52 W / 1.8 V = 28.9 A. Total current = 76.2 A. TSVs needed = 76.2 / 0.036 ≈ 2117 TSVs (with 2:1 redundancy, ≈4200 TSVs). Area = 4200 × 25 μm² = 0.105 mm².
  3. Logic power TSVs: 45 W / 0.7 V = 64.3 A → 1786 TSVs → 0.045 mm².
  4. Total TSV area: 0.105 + 0.045 = 0.15 mm² (matches paper's 0.21 mm² with added margin).
  5. Remaining logic area: 121 − (23.94 PHY + 14.80 periph + 0.21 TSV) = 82 mm² for NMP.
🍯
The 45 W cap determines the number of MACs: 64k MACs × 0.604 pJ/MAC × 1 GHz ≈ 38.7 W compute + ~4 W misc = 42.67 W total (under budget). Any more MACs would exceed thermal limits.

④ Expert swap overhead calculation (Mixtral 8×7B)

Mixtral has 8 experts per layer × 32 layers = 256 experts. Each expert is ~6B params (3 weight matrices). FP16 = 2 bytes → 12 GB per expert. Assume 32 sampled DRAM sets (dynamic set sampling); each set has 16 banks; row buffer = 32 Kb. Expert A in tier 8 swaps with expert B in tier 1. Calculate swap latency and energy.

  1. Rows per expert: 12 GB / (16 banks × 32 Kb/bank) = 12 GB / 64 KB = 187,500 rows per expert distributed across all banks.
  2. Wait, that's too many. Re‑check units: 32 Kb = 4 KB. So 16 banks × 4 KB = 64 KB per row cycle. 12 GB / 64 KB = 187,500 row accesses per expert.
  3. Swap time per row pair: read row A (tRCD + tCAS), write to B's address (tRCD + tCAS), same for row B. Assume tRCD_avg ≈ 10 ns, tCAS ≈ 15 ns. One swap ≈ 2 × (10 + 15) = 50 ns.
  4. Total swap time: 187,500 swaps × 50 ns = 9.375 ms per expert pair. But the paper reports 0.90 ms for Mixtral (Table 4). Resolution: only a subset of experts swap (e.g., top‑8 hot experts for the new topic), and swaps are parallelized across banks. The 0.90 ms figure is the measured end‑to‑end time for all necessary swaps before a batch.
  5. Energy: 0.90 ms × 30 TB/s × 0.429 pJ/bit ≈ 0.35 mJ (paper Table 4: <0.03‰ of batch energy).
💡
The swap is near‑free because (a) it runs at internal DRAM BW (30 TB/s), not interposer BW; (b) only experts whose tier assignment changes are swapped; (c) the cost is amortized over the entire batch (e.g., 256 tokens × 32 layers decode).

Practice problems

Problem 1 · easy
Mono3D DRAM achieves 30 TB/s internal BW vs HBM3's ~800 GB/s external. Why doesn't the GPU just access Mono3D DRAM at 30 TB/s?
Show answer
The 30 TB/s is internal bandwidth between the DRAM die and the logic die via Cu–Cu hybrid bonding (1 μm pitch, full die area). The GPU sits on a separate interposer tile and connects via the interposer I/O (HBM‑class PHY, ~1024b interface at 6.4 Gbps → ~800 GB/s per chip). That external link is the bottleneck. Stratum's NMP runs on the logic die, exploiting the 30 TB/s bonding link before hitting the interposer.Misconception: that internal and external BW are the same. They're not — internal BW is bonding‑to‑bonding; external is die‑to‑die via slower interposer routing.
Problem 2 · medium
Why does Stratum assign MoE decode to NMP but prefill to the xPU (GPU)? Couldn't NMP handle both?
Show answer
Prefill processes the entire prompt at once → large GeMM (e.g., 1024 tokens × 4096 hidden × 14k intermediate) → compute‑intensive, high arithmetic intensity. The GPU has massive parallelism (10k+ CUDA cores) and wins on large‑batch matmuls. Decode is one token at a time → GeMV against full KV cache + expert weights → memory‑bound. NMP's 30 TB/s internal BW crushes the GPU's 800 GB/s external link here. The hybrid system plays to each accelerator's strengths.Misconception: that more BW = better for all ops. Prefill is compute‑limited (underutilizes BW), so the GPU's compute density beats NMP's BW advantage.
Problem 3 · hard
Stratum's tiering maps hot experts to fast layers (tRCD = 1.11 ns), cold to slow layers (22.88 ns). But the ring network has 2 TB/s BW. Could inter‑PU communication become the bottleneck, negating the tiering gain?
Show answer
No, because Stratum's data mapping keeps most accesses local (each PU fetches from its own DRAM channel). The ring is used only for all‑gather (broadcast input slices, amortized over many tokens) and reduce‑scatter (aggregate partial sums, small compared to weight fetches). The paper's execution pipeline (Fig. 9) overlaps ring communication with tensor‑core compute, so the ring's 2 TB/s is rarely saturated. The bottleneck remains DRAM→logic BW (19–30 TB/s), which tiering optimizes.Misconception: that on‑chip BW is the limit. For MoE, DRAM access (expert weights) dominates; the ring handles only metadata and partial results.
Problem 4 · medium
The topic classifier (DistillBERT, 67M params) runs on the host CPU. Why not run it on the NMP or GPU to save latency?
Show answer
The classifier's 67M params are tiny vs the 109B MoE model — it fits in CPU cache and runs in <10 ms, causing <2% overhead at moderate request rates. Running it on the NMP would occupy NMP cycles (which are precious for decode) and require transferring input text to the NMP. Running it on the GPU would block the GPU's prefill pipeline. The CPU is effectively free (it's idle while NMP/GPU decode), so the overhead is negligible and doesn't justify the complexity of offloading.Misconception: that every task should run on the accelerator. For tiny, infrequent tasks, the CPU is often the best choice to avoid interfering with the critical path.
Problem 5 · hard
Stratum swaps experts between tiers (e.g., math hot → tier 1, legal hot → tier 1 after topic change). Does this violate the "no data movement" principle of NMP?
Show answer
No — the swap is within the Mono3D DRAM, using the row‑swap buffer on the NMP. Data moves from DRAM row A → swap buffer → DRAM row B (both via the 30 TB/s hybrid‑bonding link), never crossing the interposer to the GPU. The cost is 0.18–0.37% of batch time — negligible because (a) swaps run at internal BW, (b) only changed experts swap, (c) the swap is a one‑time setup before the batch, amortized over hundreds of decode steps. The "no data movement" principle refers to avoiding external data movement (GPU↔DRAM), not internal rearrangement.Misconception: that any DRAM→DRAM copy violates NMP. NMP's win is avoiding the interposer bottleneck, not zero DRAM access.
Problem 6 · easy
Why does the paper use 8 tiers instead of, say, 1024 tiers (one per layer)?
Show answer
Trade‑off between granularity and lookup complexity. 1024 tiers = 1024‑entry tiering table per PE → large register‑file overhead and slow lookup. 8 tiers = 8 entries (fits in 16×16b registers), fast comparison. The paper shows that 8 tiers capture most of the latency spread (tRCD 1.11–22.88 ns divided into 8 buckets) while keeping the tiering table tiny (0.1% area overhead per PE). Diminishing returns beyond 8 tiers.Misconception: that more tiers = better. Beyond a point, the lookup cost outweighs the latency‑control benefit.
Problem 7 · medium
Figure 17(a) shows that 100% hot expert hit rate would give ~2× throughput vs 0% hit rate. But the measured hit rates are only 31.6–68.9%. Why can't the scheduler achieve 100%?
Show answer
Two reasons: (1) Topic diversity: even within "math" queries, not all tokens activate the same experts — the router still picks per‑token, so some cold experts fire occasionally. (2) SLO constraints: the scheduler can't delay queries indefinitely to wait for a pure batch. TTFT (Time to First Token) limits how long a query can sit in the queue. The paper prioritizes batching same‑topic queries when SLO permits, but mixed‑topic batches are inevitable. The 31.6–68.9% hit rates reflect this real‑world trade‑off.Misconception: that topic classification = deterministic expert usage. The router is probabilistic; the classifier predicts the distribution, not exact per‑token routing.
Problem 8 · hard
Stratum's NMP runs at 1 GHz (vs GPU at ~2 GHz). For a compute‑bound op (large GeMM), wouldn't the GPU be 2× faster even ignoring BW?
Show answer
Yes — which is why Stratum assigns prefill (large GeMM) to the GPU. For decode (GeMV), the op is memory‑bound: the GPU spends most cycles waiting for data from HBM (~800 GB/s). Even if the NMP's 1 GHz MACs are slower, the 30 TB/s internal BW means data arrives 40× faster, so the MACs are never idle. The effective throughput is BW‑limited, not clock‑limited. The paper's measured decode speedup (8.29× OLMoE) confirms that BW dominates for this workload.Misconception: that clock speed determines throughput. For memory‑bound ops, BW is the limit; faster clocks just mean more idle cycles waiting for data.
Quick answers

Frequently asked questions

Ten questions newcomers ask most. Tap any to expand.

Q1What is Monolithic 3D DRAM and how does it differ from HBM?▾
Mono3D DRAM stacks 1024 memory layers sequentially on the same wafer using 3D NAND‑like fabrication (no TSVs). HBM stacks 6–8 separate dies via TSVs. Mono3D achieves 5.2× higher density and 1 μm Cu–Cu bonding pitch (5× finer than TSV), enabling 30 TB/s internal BW vs HBM's ~few TB/s. The trade‑off: Mono3D has a 10× latency spread (layer 1 at 1.11 ns, layer 1024 at 22.88 ns), which Stratum turns into a feature via tiering.
Q2Why does tiering help — can't you just design for worst‑case latency?▾
Designing for worst‑case (tRCD = 22.88 ns) wastes the fast layers' 1.11 ns capability — you'd cap all accesses at the slow tier's speed. Tiering lets hot data exploit fast layers (30.34 TB/s BW) while cold data uses slow layers (19.01 TB/s), achieving up to 1.6× BW advantage. The paper measures 1.32–1.45× system‑level gain (Fig. 17) from tiering alone.
Q3How does the topic classifier work, and how accurate is it?▾
A 67M‑parameter DistillBERT model trained on 70M tokens across 6 topics (math, legal, code, science, logic, humanities), using GPT‑4o‑rewritten Chatbot Arena prompts to match real serving styles. Achieves 85% accuracy on Chatbot Arena test set (vs OpenAI O3‑mini‑high's 91.1%). Inference overhead <10 ms on CPU. The classifier tags each query; the scheduler groups same‑topic queries (respecting SLO) and maps hot experts to fast tiers.
Q4What happens if the topic classifier is wrong?▾
Performance degrades gracefully. A misclassified query might fetch some experts from slow tiers (lower BW), but the decode still runs — just slower. The paper's measured hit rates (31.6–68.9%) already account for classification errors and mixed‑topic batches. Even at 0% hit rate (all experts in slow tiers), Stratum's NMP still outperforms GPU baseline (Fig. 17) due to the 30 TB/s internal BW vs 800 GB/s external.
Q5How much area/power overhead does the NMP add?▾
Logic die: 76.63 mm² (63% of 121 mm² budget), 42.67 W (under 45 W cap). The tiering table adds only 0.1% area per PE. Total system overhead <1% (logic die + TSVs vs 121 mm² DRAM + interposer). The NMP is fully custom‑designed in 7 nm ASAP7; nothing sits on the critical path (no slowdown to normal DRAM access when NMP is idle).
Q6Can Stratum handle models larger than 32 GB (one chip)?▾
Yes — Stratum‑L uses 6 chips (192 GB total), Stratum‑XL uses 2× Stratum‑L with cross‑chip interconnect (384 GB). The paper evaluates LLaMA‑4‑Scout (109B params) on Stratum‑XL. Each chip's NMP operates independently; the xPU coordinates via the interposer. Scaling is limited only by interposer I/O and cross‑chip BW (e.g., NVLink).
Q7Does Stratum work for non‑MoE models (e.g., dense transformers)?▾
Yes — the paper evaluates Qwen2.5‑32B (a dense, non‑MoE model) and achieves 6.13× decode throughput vs GPU baseline. The NMP's high BW helps any memory‑bound workload (KV cache access in attention, weight fetches in FFN). MoE models benefit more from tiering (because expert usage is topic‑dependent), but dense models still gain from the 30 TB/s internal BW vs HBM's 800 GB/s external.
Q8How does Stratum compare to prior NMP work (AttAcc, Neupims, Duplex)?▾
AttAcc/Neupims use HBM‑based NMP (TSV‑limited BW, PIM in DRAM dies). Duplex also uses HBM but adds dynamic GPU/NMP partitioning. Stratum's Mono3D DRAM delivers 40× more internal BW (30 TB/s vs ~few TB/s) via 1 μm hybrid bonding, and places the NMP on a 7 nm logic die (vs DRAM‑process PIM). The result: 2.2–3.0× throughput over Duplex, 8.29× over GPU baseline (OLMoE). The tiering mechanism is unique to Stratum (exploits Mono3D's latency spread).
Q9What's the manufacturing feasibility of 1024‑layer Mono3D DRAM?▾
The fabrication borrows from 3D NAND Flash, which already scales beyond 400 layers commercially (Samsung, Micron). Industry white papers project 500–1000 layers by 2030. The paper cites Coventor process simulations and device‑level RC extraction (NeuroSim) to validate the 1024‑layer design. The key enabler is sequential layer deposition (monolithic), avoiding TSV yield issues that plague HBM stacking.
Q10What did the authors leave for future work?▾
Extending to SMT (hyperthreaded) cores, partitioning between demand and prefetch streams, CPI‑direct QoS guarantees, execution‑time fairness, and multi‑threaded workloads where threads share data. Stratum targets multiprogrammed (independent‑program) MoE serving. Also: adaptive tiering (online expert migration based on runtime hit rates), and integration with model‑parallel frameworks (tensor/pipeline parallelism across multiple Stratum modules).

Core terms
StratumSystem–hardware co‑design for MoE serving: Mono3D DRAM (tiered memory) + NMP (on‑die processor) + topic‑aware scheduling.
Monolithic 3D DRAM (Mono3D DRAM)Vertically‑stacked DRAM with 1024 layers fabricated sequentially on one wafer; 1 μm Cu–Cu hybrid bonding; 30 TB/s internal BW.
Tiering (in‑memory)Partitioning the 1024 DRAM layers into 8 tiers (fast tier: tRCD = 1.11 ns; slow tier: 22.88 ns) to exploit latency spread.
NMP (Near‑Memory Processor)128 TFLOPS logic die (7 nm, 64k MACs) hybrid‑bonded to Mono3D DRAM; handles MoE decode + attention at 30 TB/s internal BW.
Hot expert / Cold expertHot = frequently‑activated expert for the current batch's topic (mapped to fast tier). Cold = rarely‑activated (slow tier).
Topic classifier67M‑param DistillBERT model that tags queries by topic (math, legal, code, etc.); enables hot‑expert prediction.
Expert swapNear‑memory operation that moves expert weights between tiers (via row‑swap buffer) when topic changes; <1% overhead.
Hybrid bonding (Cu–Cu)1 μm‑pitch Cu–Cu interconnect between Mono3D DRAM and logic die; 5× finer than TSV, enabling 30 TB/s BW.
Supporting terms
HBM (High Bandwidth Memory)Die‑stacked DRAM with TSVs; ~800 GB/s external BW. Stratum's baseline comparison (GPU + HBM).
TSV (Through‑Silicon Via)Vertical interconnect in HBM (~10 μm pitch); limits internal BW and adds area overhead vs hybrid bonding.
Processing Unit (PU)One of 16 chip‑level units in the NMP; each has 16 PEs, 1.25 MB SRAM, special function engine, ring router.
Processing Element (PE)Bank‑level unit with 16×16 tensor core, psum memory, tiering table, row‑swap buffer; one per DRAM bank.
Tiering table16×16b register array in each PE mapping row addresses → tier IDs → tRCD; enables adaptive latency control.
Row‑swap buffer8 KB register file in each PE for temporary row storage during expert swaps; avoids xPU data transfer.
Ring networkBi‑directional on‑chip interconnect (2 TB/s aggregate) linking 16 PUs; supports all‑gather and reduce‑scatter.
All‑gatherCommunication pattern: broadcast input slices from all PUs to reconstruct the full matrix at each PU (via ring).
Reduce‑scatterAggregate partial sums from all PUs, scatter results back; used in GeMM3 output aggregation.
Special function engine256‑way SIMD unit for Softmax, SiLU, etc.; includes vector/scalar register files and arithmetic units.
Prefill vs decodePrefill = process entire prompt (large GeMM, GPU‑bound). Decode = one token at a time (GeMV, memory‑bound, NMP‑bound).
TTFT (Time to First Token)SLO metric: max latency before a query starts processing; constrains how long the scheduler can delay batching.
Dynamic set sampling (DSS)Monitor only 32 of 1024 DRAM sets (Chebyshev bound ensures accuracy); shrinks UMON overhead (AttAcc term, not used in Stratum but conceptually similar).
tRCD / tRP / tRAS / tRCDRAM timing parameters: tRCD = row‑to‑column delay (varies by tier); tRP = precharge; tRAS = row active; tRC = full cycle.
