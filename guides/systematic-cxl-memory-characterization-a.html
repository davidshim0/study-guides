1CXL for memory expansion: what and why▾

DRAM scaling is stuck; memory demand is exploding. CXL (PCIe-based) lets you plug in memory expanders (~128GB–1TB) beyond socket-local DRAM. Unlike NUMA (cross-socket via QPI/UPI at ~140-190ns), CXL devices sit on PCIe lanes and use third-party memory controllers, yielding ~200-400ns latency and 18-52GB/s bandwidth per device.

CXL 1.1 type-3 devices expose DRAM via the CXL.mem protocol, appearing as a NUMA node. For your tiering work: CXL is the "slow tier" (~2-4× local latency), but it's physically present and cheaper than local DRAM at scale.

Now you're ready: Melody measures how that 200-400ns (and worse, µs-level tail latencies) affects real workloads, and introduces Spa to pinpoint why slowdowns happen.
2Average latency vs. tail latency▾

Conventional memory profiling reports average latency (e.g., Intel MLC). Local DRAM: ~100ns; NUMA: ~140-190ns; CXL: 214-394ns (paper's 4 devices). But average hides variability. Tail latency = p99, p99.9, p99.99 — the worst-case delays that occur even under low load.

The paper finds CXL devices show p99.9-to-p50 differences of 50-160ns (some devices reaching 1µs at higher percentiles), while local/NUMA stay within 45-61ns. PCIe's non-deterministic nature + third-party MC scheduling = unpredictable queuing delays. For latency-sensitive apps (Redis, VoltDB), tail matters more than average.

Now you're ready: Melody is the first to disclose and quantify CXL tail latencies across devices, showing they propagate to application-level slowdowns (e.g., Redis YCSB-C: 100→200µs at p99).
3CPU backend: loads, prefetches, stores, RFOs▾

When a CPU miss (L1→L2→L3) reaches CXL memory, it's either a Demand load (CPU needs it now), a Prefetch (L1/L2 prefetcher guessing ahead), a Store (queued in store buffer, triggers RFO for coherence), or a Write (eviction). Each takes CXL's 200-400ns round-trip.

Intel CPUs track stalls via counters like BOUND_ON_LOADS, STALLS_L3_MISS, etc. Longer CXL latency means more cycles stalled waiting for data. The breakdown — which stalls — is what Spa uses to pinpoint bottlenecks.

Now you're ready: Spa reads 9 CPU counters (BOUND_ON_LOADS, STALLS_L1D_MISS, etc.) to compute differential stalls (Δs = s_CXL − s_DRAM) and attribute slowdowns to L1/L2/L3/DRAM/Store.
4Workload slowdown: how to measure it▾

Run a workload on local DRAM, measure performance (𝑃_DRAM). Repeat on CXL (𝑃_CXL). Slowdown 𝑆 = (𝑃_DRAM / 𝑃_CXL − 1) × 100%. MPKI (Misses Per 1k Instructions) and CPI (Cycles Per Instruction) are microbenchmark proxies.

The paper finds: 60% of workloads see <10% slowdown on best CXL device (CXL-D); 7% see 150-580% on worst (bandwidth-bound + high tail latency). Many workloads tolerate 200-400ns; some don't. Understanding why requires Spa.

Now you're ready: Melody measures 265 workloads (cloud/graph/ML/SPEC) × 7 latency configs (140-410ns) to map workload sensitivity to sub-µs CXL.
5Bandwidth vs. latency sensitivity▾

Some workloads are latency-sensitive (pointer-chase, Redis): every miss stalls the CPU, so 200ns→400ns doubles wait time. Others are bandwidth-bound (streaming SPEC): they saturate CXL's 18-24GB/s, causing queuing delays that dwarf base latency. A third group is both: misses hurt, and saturation amplifies tail latencies.

CXL devices vary: CXL-D (52GB/s, 239ns) handles bandwidth-bound workloads better; CXL-A (24GB/s, 214ns) has lower latency but less headroom. The paper finds 4 SPEC workloads need high bandwidth → 50-580% slowdown on CXL-A/B, but only 80-90% on CXL-D.

Now you're ready: Melody's large-scale study (265 workloads) reveals which types tolerate CXL and which don't — critical for tiering decisions in your OS work.
6CPU stalls and the pipeline▾

CPU cycles = useful execution + stalls. Stalls happen when the CPU waits (frontend: fetch/decode; backend: execution — ALU, memory). For CXL analysis, backend memory stalls dominate. Intel CPUs expose counters for stalls at each cache level (L1/L2/L3) and DRAM, plus store buffer stalls.

Key insight: the difference in stalls (Δs = s_CXL − s_DRAM) directly correlates with slowdown. If a workload sees +1M extra L3 stall cycles on CXL, and total cycles = 10M, slowdown ≈ 10%. Spa uses this to break down where the pain is.

Now you're ready: Spa's bottom-up approach: measure Δs via 9 counters, split into Δs_L1 + Δs_L2 + Δs_L3 + Δs_DRAM + Δs_Store, and you've dissected the slowdown (95% accuracy, <5% error across 265 workloads).
7Prefetcher efficiency under long latency▾

CPU hardware prefetchers (L1PF, L2PF) try to fetch data before the CPU needs it. If a prefetch arrives after the demand load, it's useless. CXL's 200-400ns (vs. 100ns local) delays prefetches, reducing timeliness and coverage (fraction of demand hits a prefetch serviced).

The paper finds: L2 prefetch coverage drops 2-38% on CXL; L1 prefetches that would've hit L2 now miss to CXL (L1PF-L3-miss increases as L2PF-L3-miss decreases, almost 1:1). Result: cache slowdowns (up to 15% of total slowdown for some workloads) despite prefetchers being on. Disabling prefetchers shifts the slowdown to DRAM stalls (worse).

Now you're ready: Spa quantifies cache slowdown (Δs_L1 + Δs_L2 + Δs_L3) and shows it's from reduced prefetch efficiency — a key finding for future CPU/CXL co-design.
The whole paper at a glance

Melody + Spa in a nutshell

The problem (CXL's unpredictable sub-µs latencies), the solution (systematic characterization + Spa for root-cause analysis), and the payoff.

CXL memory expands capacity beyond socket-local DRAM, but at a cost: 200-400ns latency (vs. ~100ns local) and 18-52GB/s bandwidth (vs. 218GB/s local). Prior work measured average latency and ran a handful of HPC/cloud benchmarks. Three gaps remain:

(1) Tail latencies: No one had disclosed CXL's p99/p99.9 behavior. The paper finds CXL devices exhibit higher tail latencies even under low load (with p99.9-to-p50 differences reaching ~160ns for CXL-B/C, and some devices showing latencies up to 1µs at higher percentiles), and these propagate to application latencies (e.g., Redis).

(2) Workload scale: Existing studies cover <20 workloads. Melody evaluates 265 (cloud, graph, ML, SPEC, PARSEC) across 4 real CXL devices, 7 latency configs (140-410ns via NUMA emulation), and 5 CPU platforms. Result: many workloads tolerate 200-400ns (<10% slowdown), but 7% see 150-580% due to bandwidth saturation + tail latency.

(3) Root-cause analysis: Why do slowdowns happen? Existing methods (TMA, heuristic LLC miss correlations) lack precision. Melody introduces Spa (Stall-based Performance Analysis): repurpose 9 CPU stall counters to compute differential stalls (Δs = s_CXL − s_DRAM), split into L1/L2/L3/DRAM/Store, and directly correlate to slowdowns (95% accuracy, <5% error). Spa reveals CPU prefetcher inefficiency under CXL: L2 coverage drops 2-38%, causing 15%+ cache slowdowns.

The payoff: (a) First disclosure of CXL tail latencies and their impact. (b) Large-scale workload taxonomy showing CPU tolerance to 140-410ns. (c) Spa: a lightweight, accurate tool for pinpointing CXL bottlenecks at workload- and phase-level granularity. (d) Open-sourced datasets + tools at https://github.com/MoatLab/Melody.

🍯 Core takeaways

  • Not all CXL are equal: 4 devices, 214-394ns avg, with varying tail latencies (p99.9-to-p50 differences: ~75ns for CXL-D vs. ~160ns for CXL-B/C).
  • Tail latencies matter: CXL shows spikes even at <50% bandwidth; local/NUMA stay stable to 90%.
  • Workload tolerance: 60% of 265 workloads see <10% slowdown on best CXL; 7% see 150-580% on worst (bandwidth-bound + tail).
  • Spa: 9 CPU counters: Measure Δs = s_CXL − s_DRAM, break into Δs_L1 + Δs_L2 + Δs_L3 + Δs_DRAM + Δs_Store → 95% accuracy, <5% error.
  • Prefetcher inefficiency: CXL's 200-400ns delays L2 prefetches → coverage drops 2-38% → cache slowdowns up to 15%.
  • CXL+NUMA danger: One hop adds 161-227ns; the CXL+NUMA tail starts around p98 and can reach up to 800ns for some workloads (e.g., 520.omnetpp shows 2.9× slowdown).
  • Period-based analysis: Spa tracks slowdowns per 1B instructions → reveals bursty phases (e.g., 602.gcc: first two-thirds exceed 30% slowdown).
Seven mental models

How to think about CXL and Spa

Each analogy stands alone. Every card ends with where the picture breaks.

Core insight · Tail vs. average

Rush hour vs. average traffic

Your commute averages 30 minutes, but twice a week it's 90 minutes (tail). CXL is the same: average latency is 214-394ns, but tail latencies can be significantly higher due to PCIe congestion, MC scheduling, or thermal throttling. Local DRAM is like a private road — stable at 100ns even under load. CXL is a shared highway with unpredictable jams.

Where it breaks: Traffic jams have visible causes (accidents); CXL tail latencies are opaque (black-box MC). The paper suspects CXL protocol queuing + suboptimal MC scheduling, but lacks white-box access to confirm.
Spa method · Differential stalls

Nested measuring cups

Intel CPUs expose nested stall counters: P1 counts all memory stalls (L1–DRAM), P3 counts L1 misses (L2–DRAM), P5 counts L3 misses (DRAM only). Subtracting them (P1−P3, P3−P4, …) isolates each level. Spa measures these on DRAM and CXL, computes Δ (the difference), and — magic — Δs directly correlates to slowdown (r² = 0.95). One run on each setup, 9 counters, done.

Where it breaks: The nesting assumes no counter aliasing (Intel architecture-specific). Other CPUs (ARM, AMD) may need different counters. But the idea — differential stalls reveal root causes — generalizes.
Prefetcher · Timeliness

Ordering pizza before you're hungry

CPU prefetchers guess you'll need data and fetch it early. If it arrives before the demand load, you eat instantly (hit). If it arrives after, you wait anyway (miss). CXL's 200-400ns delays prefetches → many arrive late → L2 coverage drops 2-38%. Result: demand loads that would've hit L2 now miss to CXL. Prefetchers are on, but ineffective.

Where it breaks: Real pizza delivery has traffic/weather; prefetchers have stride/pattern predictability. Some workloads (irregular access) were already unpredictable — CXL just makes it worse. The paper shows cache slowdowns (Δs_L1 + Δs_L2) for 15% of workloads.
Device diversity · Bandwidth

Water pipes of varying width

CXL-A/B/C (8× PCIe lanes, 1 channel) deliver 18-24GB/s. CXL-D (16× lanes, 2 channels) delivers 52GB/s — nearly matching NUMA's 97GB/s. Bandwidth-bound workloads saturate 24GB/s → queuing delays → tail latency spikes. CXL-D doesn't saturate → tail stays lower → slowdown is primarily from the latency increase. Lesson: for bandwidth-bound workloads, CXL device choice matters more than base latency. Your tiering: map bandwidth-heavy pages to high-BW CXL or DRAM.

Where it breaks: Pipes are passive; CXL MCs are active (scheduling, throttling). The paper finds CXL-C (FPGA) can't exploit CXL's bidirectional data transfer — peak bandwidth at read-only, unlike ASIC devices. Vendor optimizations vary wildly.
Phase behavior · Period-based analysis

Fever spikes vs. average temperature

Workload 602.gcc has average slowdown around 20%, but the first two-thirds of execution sees higher slowdown while the last third sees lower. Averaging hides this. Spa's period-based analysis (every 1B instructions) tracks slowdown over time → reveals critical phases. For tiering: pin hot phases to DRAM, relax during cold phases. One-time average can't do this.

Where it breaks: Fever spikes are symptomatic (viral load); workload phases may be arbitrary (algorithm stages). The paper samples every 1ms (time-based) and aligns to instruction intervals (proportional adjustment) — assumes smooth counter progression within 1ms. Works for coarse phases (>10ms); finer granularity may introduce error.
Tail impact · Propagation

One slow checkout line stalls the whole queue

Redis YCSB-C (read-only) on CXL: application p99 latencies increase significantly because device-level tail latencies propagate. Even at <50% bandwidth, CXL shows spiky latencies → app latency spreads. Local/NUMA don't — their tails stay lower. For QoS-sensitive cloud apps, tail matters more than average.

Where it breaks: Checkout lines have visible bottlenecks (one cashier); CXL tail causes are opaque (MC black-box). The paper suspects CXL link congestion + MC scheduling but can't pinpoint without CPMU (CXL 3.0 performance counters, not yet available).
Spa breakdown · Sources

Itemized bill vs. total charge

Workload slowdowns are a "total charge" (e.g., 20%). Spa itemizes it: 605.mcf pays 90% to DRAM stalls (demand-load-bound), 508.namd pays 50% to cache stalls (prefetcher delay), 519.lbm pays 60% to store buffer stalls (RFO-heavy). Same total, different causes. For optimization: 605.mcf needs lower CXL latency or tiering; 508.namd needs prefetch tuning; 519.lbm needs write buffer sizing.

Where it breaks: A bill shows line items but not causality (did you really need dessert?); Spa shows stalls but not why the workload accessed those addresses. Pairing Spa with binary instrumentation (Intel Pin, as in §5.7) can map stalls to source code.
The full technical story

How Melody and Spa work

Section by section, with key results from the paper's 265-workload evaluation.

1 · CXL device characterization (§3)

The paper tests 4 CXL devices (CXL-A/B/C/D) on 5 CPU platforms (SPR, EMR, SKX). CXL-A (ASIC, DDR4, 8× lanes): 214ns, 24GB/s. CXL-B (ASIC, DDR5, 8× lanes): 271ns, 22GB/s. CXL-C (FPGA, DDR4, 8× lanes): 394ns, 18GB/s. CXL-D (ASIC, DDR5, 16× lanes): 239ns, 52GB/s. Local DRAM (SPR/EMR): 111-117ns, 218-246GB/s. NUMA remote: 191-410ns (SKX8S for 410ns).

🌱
Key finding (Fig. 3): CXL tail latencies exceed local/NUMA. At 32 threads, pointer-chase: the p99.9-to-p50 difference is ~75ns for CXL-D and ~160ns for CXL-B/C, compared to 45-61ns for local/NUMA. Enabling prefetchers (Fig. 6) helps average but tail latencies remain elevated.

Loaded latency (Fig. 3a): CXL devices spike at 50-86% bandwidth utilization, while local/NUMA stay stable to 90-95%. Mixed read/write (Fig. 5): CXL-A/D peak at 3:1 ratio (bidirectional PCIe), CXL-C at read-only (FPGA can't exploit bidirectional). Local DRAM peaks at read-only (DDR unidirectional).

Metrics: slowdown 𝑆 = (𝑃_DRAM / 𝑃_CXL − 1) × 100%, MPKI, CPI. Setups: 7 latency configs (SKX 140ns/190ns/410ns, EMR NUMA 193ns, EMR CXL-A 214ns / CXL-B 271ns / CXL-D 239ns).

3 · Spa: the 9-counter method (§5)

Spa uses 9 Intel CPU counters (Table 2): P1 = BOUND_ON_LOADS, P2 = BOUND_ON_STORES, P3 = STALLS_L1D_MISS, P4 = STALLS_L2_MISS, P5 = STALLS_L3_MISS, P6 = RETIRED.STALLS, P7/P8 = 1/2_PORTS_UTIL, P9 = STALLS.SCOREBD. Run workload on DRAM and CXL, compute Δ𝑃𝑖 = 𝑃′𝑖 − 𝑃𝑖. Then:

# Stall breakdown (Fig. 10, Eq. 7)
Δs = Δs_L1 + Δs_L2 + Δs_L3 + Δs_DRAM + Δs_Store
where,
  Δs_Store = ΔP2
  Δs_L1 = ΔP1 − ΔP3
  Δs_L2 = ΔP3 − ΔP4
  Δs_L3 = ΔP4 − ΔP5
  Δs_DRAM = ΔP5

# Slowdown estimation (Eq. 5, 8)
S ≈ Δs / c_DRAM ≈ (Δs_L1 + Δs_L2 + Δs_L3 + Δs_DRAM + Δs_Store) / c_DRAM

Validation (Fig. 11): For 265 workloads across NUMA/CXL-A/CXL-B, |S_actual − S_estimated| < 5% for 95% of workloads. Average error < 2%. Backend stalls (Δs_Memory = ΔP1 + ΔP2) alone capture 96% of workloads within 5%, confirming frontend stalls are negligible (CXL doesn't affect instruction fetch).

4 · Prefetcher inefficiency (§5.4, Finding #4)

Why do cache slowdowns happen? Disable L1/L2 prefetchers → Δs_L1 = Δs_L2 = Δs_L3 = 0 (all slowdown shifts to DRAM). Enabling them: L2 prefetch coverage drops 2-38% on CXL (measured via L2PF-L3-miss decrease, Fig. 12b). Simultaneously, L1PF-L3-miss increases almost 1:1 (Fig. 12a, Pearson r = 0.99), indicating L2 prefetches are delayed → L1 prefetches miss L2 → hit pending L2 prefetch or miss to CXL → delayed L1 hits → cache slowdowns.

💡
Mechanistically: CXL's 200-400ns delays L2 prefetches. L1 prefetchers, unaware of the delay, issue requests that would've hit L2 but now miss. Those misses either wait on L2's pending prefetch (stall at L2) or fetch from LLC/CXL (stall at L3/DRAM). Result: 15% of workloads see >5% cache slowdown (Fig. 15).

5 · Period-based analysis (§5.6)

Spa tracks slowdowns over time (every 1B instructions) by converting time-based performance-counter samples (1ms intervals) to instruction-interval breakdowns (proportional adjustment). Example: 602.gcc (Fig. 16a) shows varying slowdown across phases, with the overall average around 20%. 605.mcf (Fig. 16b) has relatively stable DRAM slowdown throughout, while 631.deepsjeng (Fig. 16c) sees bursty cache slowdowns.

🍯
Use case: For memory tiering, Spa's period-based breakdown enables phase-aware tiering without offline profiling — just sample counters live and adjust placement dynamically.
The paper's numbers, step by step

Worked examples

Four traces using Melody's real data — tail latency impact, Spa breakdown, prefetcher coverage, and period-based analysis.

① Computing slowdown from Spa counters (508.namd, CXL-A)

Paper data: On local DRAM, 508.namd runs 10B cycles; on CXL-A, 12B cycles (20% slowdown). Spa counters (simplified): DRAM: P1=2B, P3=1.5B, P5=0.5B. CXL-A: P1=3B, P3=2.5B, P5=0.6B. Compute Δs breakdown.

  1. Total extra stalls: Δs = 12B − 10B = 2B cycles (20% of 10B).
  2. Store stalls: ΔP2 ≈ 0 (not shown, assume negligible). Δs_Store = 0.
  3. L1 stalls: Δs_L1 = (P1_CXL − P1_DRAM) − (P3_CXL − P3_DRAM) = (3B − 2B) − (2.5B − 1.5B) = 1B − 1B = 0.
  4. L2 stalls: Δs_L2 = (P3_CXL − P3_DRAM) − (P5_CXL − P5_DRAM) = (2.5B − 1.5B) − (0.6B − 0.5B) = 1B − 0.1B = 0.9B (45% of Δs).
  5. DRAM stalls: Δs_DRAM = P5_CXL − P5_DRAM = 0.6B − 0.5B = 0.1B (5%).
  6. Remaining: 2B − (0 + 0.9B + 0.1B) = 1B → L3 or other (50%).
🍯
Interpretation: 508.namd's 20% slowdown comes 50% from cache stalls (L2/L3, prefetch delay), 5% from DRAM, rest from backend pipeline. This matches Fig. 14 (paper) showing 508.namd as cache-dominated. Spa nails it with 9 counters.

② Tail latency impact on 520.omnetpp (CXL+NUMA, Fig. 8c&d)

520.omnetpp: discrete event simulation, ~1GB/s bandwidth (light load). On CXL-A: 5% slowdown. On CXL-A+NUMA (375ns avg): 290% slowdown. Why?

  1. CXL-A alone: Relatively low slowdown (5%) under light load.
  2. CXL+NUMA: One extra hop adds 161ns (214→375ns avg). The CXL+NUMA tail starts around p98 and reaches up to 800ns (Fig. 8d).
  3. Why? Paper suspects tail latencies amplify across hops: CXL's own tail + NUMA's variance → compounding effect.
  4. Validation: Reduce workload intensity: tail drops, slowdown decreases.
  5. Contrast: 2-hop-NUMA (SKX8S, 410ns avg) shows moderate slowdown — worse latency but more stable tail.
⚠
Lesson: CXL+NUMA is dangerous for tail-sensitive apps. Avoid CXL+NUMA unless you can guarantee low bandwidth utilization (the paper's threshold for stable tails).

③ Prefetcher coverage drop and cache slowdown

The paper reports qualitative trends and aggregate metrics for prefetch efficiency. L2 prefetch coverage drops 2-38% on CXL (Fig. 12b), with L1PF-L3-miss and L2PF-L3-miss showing nearly 1:1 correlation (Pearson r = 0.99). Disabling prefetchers can result in ~10% performance drop.

  1. Measure L2PF coverage: The paper shows L2PF-L3-miss reduction of 2-38% across workloads.
  2. Measure L1PF-L3-miss: Increases correlate almost 1:1 with L2PF reduction (Pearson r = 0.99).
  3. Interpretation: L2 prefetches delayed → L1 prefetches miss L2 → fetch from CXL.
  4. Coverage metric: Fraction of demand loads serviced by prefetches drops on CXL.
  5. Impact: Cache slowdowns result from reduced prefetch efficiency.
💡
Why it matters: Prefetchers are on but ineffective. Disabling them would shift cache slowdown to DRAM (worse). The solution isn't to disable prefetchers — it's to tolerate delayed prefetches or reduce CXL latency (CPU/CXL co-design, paper's Implication #2).

④ Period-based slowdown for 602.gcc (Fig. 16a)

602.gcc: C compiler benchmark, 20B total instructions. Overall average slowdown on CXL-A is around 20%, but Spa's period-based analysis (every 1B instructions) reveals that the first two-thirds of execution exceed 30% slowdown. Walk through the analysis.

  1. Early phases: First two-thirds show slowdown exceeding 30%, with varying store/DRAM/cache contributions.
  2. Later phases: Last third shows lower slowdown as memory intensity decreases.
  3. Total average: Overall ~20% slowdown, but the distribution matters: early phases are critical.
  4. Tiering decision: Pin 602.gcc's hot phases to DRAM. Relax cold phases to CXL.
🍯
Generalization: Spa's period-based breakdown enables phase-aware tiering without offline profiling. For your OS work: sample Spa counters every 100ms, compute Δs breakdown per phase, and adjust page placement dynamically. The paper shows this reduces slowdown for bursty workloads (§5.7).
Self-test

Practice problems

Try before opening answers. Each ends with the misconception it checks.

Problem 1 · easy
A workload runs 10B cycles on local DRAM and 11B on CXL-A (214ns). Compute the slowdown using T_CXL / T_DRAM - 1.
Show answer
Slowdown = (T_CXL / T_DRAM − 1) × 100% = (11B / 10B − 1) × 100% = 10%. The paper defines slowdown using performance ratio P_DRAM / P_CXL - 1; the runtime equivalent is T_CXL / T_DRAM - 1. Both give 10% slowdown.Misconception: confusing slowdown formula direction. Always normalize correctly.
Problem 2 · medium
CXL-A (214ns, 24GB/s) and CXL-B (271ns, 22GB/s) both slow down bandwidth-bound workloads significantly. Which device characteristic is the primary bottleneck: latency or bandwidth? Show your reasoning.
Show answer
Bandwidth. Bandwidth-bound workloads saturate the available bandwidth → device saturates → queuing delays dominate. Evidence: (a) CXL-D (52GB/s) shows lower slowdown, confirming bandwidth is the bottleneck. (b) Spa breakdown shows high DRAM stalls, consistent with saturation. (c) At high bandwidth utilization, CXL latency spikes due to tail latency amplification under saturation.Misconception: that higher latency causes worse slowdown. Here, bandwidth saturation is the real culprit.
Problem 3 · hard
Spa measures P5 (STALLS_L3_MISS) = 500M on DRAM, 600M on CXL-A. Total cycles: DRAM 10B, CXL 11B. Estimate the DRAM-only slowdown contribution. Then explain why this underestimates the true impact if prefetchers are enabled.
Show answer
DRAM slowdown = Δs_DRAM / c_DRAM = (600M − 500M) / 10B = 1%. But this only counts demand loads that miss L3 → go to DRAM. With prefetchers enabled, some L2 prefetches also miss L3 → stall at L3 (measured by P4−P5, not P5 alone). If P4 increases on CXL due to delayed L2 prefetches, then Δs_L3 captures additional cache slowdown. Total impact = DRAM + cache stalls. Why underestimate? Because P5 counts only demand L3 misses; prefetch L3 misses hide in P4−P5. Spa accounts for this by splitting Δs_L3 separately.Misconception: that P5 alone measures all DRAM stalls. It misses prefetch-induced L3 stalls, which Spa captures via (P4−P5).
Problem 4 · medium
520.omnetpp sees 5% slowdown on CXL-A but 290% on CXL-A+NUMA. Average latency increases 214→375ns (75% increase), not 290%. Reconcile using tail latencies.
Show answer
Average latency ≠ performance impact. At light load, 520.omnetpp is latency-sensitive. On CXL-A, relatively stable tail → 5% slowdown. On CXL-A+NUMA, tail starts around p98 and reaches up to 800ns (Fig. 8d) — a significant increase. The tail dominates performance for tail-sensitive workloads. Reducing workload intensity reduces tail and slowdown, confirming tail is the culprit.Misconception: that average latency predicts slowdown linearly. For tail-sensitive workloads, tail matters more than average.
Problem 5 · easy
Paper says disabling L1/L2 prefetchers causes Δs_L1 = Δs_L2 = Δs_L3 = 0 (all slowdown shifts to DRAM). Why doesn't this improve performance if prefetchers are inefficient under CXL?
Show answer
Because demand loads are slower than delayed prefetches. With prefetchers on, some data arrives late (delayed L2 prefetch → L1 cache slowdown). But it still arrives before the demand load needs it (prefetchers predict ahead). Disabling them means all accesses become demand loads → miss to DRAM → 200-400ns wait every time. Result: Δs_DRAM increases more than Δs_cache decreases. Prefetchers are inefficient but still better than nothing. The paper shows cache slowdowns are 5-15%, but DRAM slowdowns would be higher without prefetching.Misconception: that inefficient = useless. Prefetchers reduce average latency even if they don't eliminate tail latencies.
Problem 6 · hard
Spa claims 95% accuracy (<5% error) for 265 workloads (Fig. 11c). Construct a scenario where Spa fails (error >10%) and explain why the 9-counter model breaks down.
Show answer
Scenario: workload with heavy non-memory backend stalls (e.g., serializing operations, scoreboard contention). Spa assumes Δs ≈ Δs_Memory, i.e., backend stalls come from L1/L2/L3/DRAM/Store. But if a workload has significant Δs_Core (data dependency stalls from OoO execution under CXL's long latency), Spa underestimates slowdown. CXL's 200-400ns delays propagate through dependency chains, causing serialization stalls that aren't memory-bound but are induced by memory latency. Spa's model assumes frontend and non-memory backend stalls are constant, which holds for 95% of workloads but fails for OoO-heavy, dependent-load workloads.Misconception: that Spa is always accurate. It's accurate for memory-bound workloads but fails for compute-bound or dependency-bound workloads.
Problem 7 · medium
Paper says CXL-C (FPGA, 394ns, 18GB/s) can't exploit bidirectional PCIe, peaking at read-only (Fig. 5). But CXL protocol is bidirectional by design. What's the likely hardware bottleneck?
Show answer
FPGA CXL IP doesn't implement full duplex. The paper speculates that CXL-C's FPGA-based controller uses an unoptimized CXL IP that handles reads and writes sequentially (half-duplex). Evidence: CXL-A/B/D (ASIC) peak at mixed ratios (bidirectional), while CXL-C peaks at read-only. Mixed workloads degrade CXL-C bandwidth. Likely cause: the FPGA's PCIe PHY supports bidirectional lanes, but the soft CXL controller logic doesn't pipeline reads/writes simultaneously. ASIC controllers have dedicated queues and can saturate both directions in parallel.Misconception: that PCIe bidirectional = CXL bidirectional. The link is bidirectional, but the controller must exploit it.
Problem 8 · easy
Why does the paper use 7 latency configs (140/190/193/214/239/271/410ns) instead of testing all possible CXL devices? What's the methodological justification?
Show answer
To map the full sub-μs latency spectrum. Testing one device only reveals that device's quirks. By using NUMA emulation + 4 real CXL devices across the 140-410ns range, the paper characterizes workload behavior across the latency spectrum. Result: insights generalize beyond specific CXL vendors. The 7-point spectrum lets you understand behavior at arbitrary latencies without testing every device. The paper's spectrum study (Fig. 9) shows this continuity.Misconception: that testing more devices = better science. Melody's strength is spectrum generalization, not vendor enumeration.
Quick answers

Frequently asked questions

Twelve common questions. Tap to expand.

Q1What's the difference between CXL and NUMA memory?▾
NUMA is cross-socket memory via QPI/UPI (on-die interconnect), ~140-190ns, integrated MC. CXL is PCIe-based memory expansion via external devices, ~200-400ns, third-party MC. Key diff: NUMA is deterministic (CPU-controlled scheduling), CXL is non-deterministic (MC black-box + PCIe congestion). Both are "remote" memory but CXL has higher and more variable tail latencies.
Q2Why do tail latencies matter more than average for CXL?▾
For QoS-sensitive apps (Redis, latency-bound cloud services), high percentile latencies can violate SLO. CXL tail latencies propagate: device-level high percentiles → elevated app-level latencies (Redis YCSB-C, Fig. 7c). Local/NUMA have lower tail variability. Tail matters for: (a) long-running requests (straggler effect), (b) dependent-load chains (one stall cascades), (c) real-time systems (deadline misses).
Q3Can Spa work on AMD/ARM CPUs?▾
In principle, yes — the idea (differential stalls) generalizes. But Intel's nested counters are architecture-specific. AMD/ARM may lack equivalent events or have different nesting. You'd need to map Spa's 9 counters to each vendor's PMU events. The paper's validation is Intel-only (SKX/SPR/EMR). For your work: check AMD's PPR or ARM's PMU docs for equivalent stall counters.
Q4Why doesn't the paper test CXL 2.0 in native mode?▾
The CPUs (SPR/EMR) run CXL devices in CXL 1.1 mode. The devices are CXL 2.0-capable but hosts use 1.1 compatibility. The paper focuses on characterizing what's available today in production systems.
Q5How does Spa compare to Intel VTune or AMD uProf?▾
VTune/uProf collect the same counters but don't compute differential stalls or attribute slowdowns to L1/L2/L3/DRAM. They report absolute stall cycles, which is less useful for CXL analysis — you need to know how much worse CXL is than DRAM. Spa's value: (a) differential analysis (Δs = s_CXL − s_DRAM), (b) breakdown formula, (c) validation (95% accuracy vs. actual slowdown). You could run VTune on DRAM + CXL and post-process with Spa's formulas.
Q6What causes CXL tail latencies? Can they be fixed?▾
Paper suspects: (a) CXL protocol queuing, (b) MC scheduling, (c) thermal throttling. Can't pinpoint without white-box MC access. Fixes: (a) CXL 3.0 CPMU for diagnostics, (b) better MC scheduling, (c) CPU-side tolerance (deeper OoO windows, more MSHRs), (d) software: avoid CXL+NUMA (tail amplifies), pin tail-sensitive workloads to DRAM.
Q7Why do bandwidth-bound workloads see higher slowdown on CXL-A/B but lower on CXL-D?▾
CXL-A/B provide 22-24GB/s, CXL-D provides 52GB/s. Bandwidth-bound workloads saturate CXL-A/B → queuing delays → tail latency spikes. CXL-D doesn't saturate → tail stays lower → slowdown is primarily from latency increase. Lesson: for bandwidth-bound workloads, CXL device choice matters. Your tiering: map bandwidth-heavy pages to high-BW CXL or DRAM.
Q8Can Spa track slowdowns at finer granularity (e.g., per-function)?▾
Yes, with binary instrumentation (Intel Pin, §5.7). Pin inserts callbacks → sample Spa counters → compute Δs per function. The paper demos this for 605.mcf. Limitation: Pin adds 10-100× overhead → use for offline profiling, not production. For production: use perf's per-thread sampling + addr2line to map stalls to source code (lower overhead but coarser).
Q9What's the overhead of running Spa counters live?▾
Negligible. Reading 9 CPU counters takes minimal cycles. Sample every 100ms → very low overhead. The paper samples every 1ms for period-based analysis. Contrast: VTune's full profiling is 5-20% overhead. Spa's 9 counters are safe for production use.
Q10Why doesn't the paper test CXL with NVMe SSDs (CXL.cache)?▾
CXL.cache is for device-side caching, not memory expansion. CXL.mem (type-3 devices) is what the paper tests — pure DRAM capacity. CXL.cache would add another latency tier → orthogonal problem (storage tiering, not memory tiering). For your work, CXL.mem is the relevant protocol. CXL.cache is for accelerator coherence, not capacity.
Q11How do I choose between CXL devices for a workload?▾
Use Spa + bandwidth profile. (1) Run workload on DRAM, sample Spa counters → compute latency/bandwidth sensitivity. (2) If latency-dominated → pick low-latency CXL. If bandwidth-dominated → pick high-BW CXL. If tail-sensitive → avoid devices with high tail variability. (3) Validate on target CXL, re-run Spa → if predicted ≈ actual, you're good.
Q12What's next for CXL research after Melody?▾
Paper's open questions: (a) CPU tolerance to higher latencies (future CPUs need deeper OoO, more MSHRs), (b) prefetcher redesign for long latency, (c) CXL MC white-box analysis (CPMU in CXL 3.0), (d) software tiering with Spa (live phase detection, page migration). Your work fits here: use Spa for online profiling → predict which pages to migrate DRAM↔CXL → minimize slowdown. Melody provides baseline numbers to validate against.

Glossary & further reading

Every term in one place, plus paths to go deeper.

Core terms
CXL (Compute Express Link)PCIe-based memory fabric (CXL.mem protocol) for capacity expansion beyond socket-local DRAM.
MelodyThe paper's framework: 265 workloads × 4 CXL devices × 7 latency configs × 5 CPU platforms → first large-scale CXL characterization.
Spa (Stall-based Performance Analysis)9-counter method to break down CXL slowdowns into L1/L2/L3/DRAM/Store sources (95% accuracy).
Tail latencyHigh percentile latencies (p99/p99.9/p99.99). CXL shows elevated tails compared to local/NUMA, propagating to app-level latencies.
Differential stalls (Δs)Extra CPU stall cycles on CXL vs. DRAM (Δs = s_CXL − s_DRAM). Spa's core insight: Δs directly correlates to slowdown.
Prefetcher coverageFraction of demand loads serviced by prefetches. Drops 2-38% on CXL due to delayed L2 prefetches.
Period-based analysisTrack slowdowns per 1B instructions (time-to-instruction conversion) → reveals bursty phases.
Supporting terms
CXL.memCXL protocol for memory expansion (type-3 devices). Contrasts with CXL.cache (device caching) and CXL.io (I/O).
MPKI / CPIMisses Per 1k Instructions / Cycles Per Instruction — microbenchmark proxies for slowdown (higher CPI = slower).
BOUND_ON_LOADS (P1)Intel CPU counter: # cycles stalled while memory subsystem has ≥1 outstanding load (includes L1–DRAM).
STALLS_L3_MISS (P5)Intel counter: # cycles stalled while L3 miss is outstanding (demand loads only, excludes prefetch).
RFO (Read-For-Ownership)Coherence request triggered by stores: fetch cache line exclusively before writing (adds latency to stores).
L1PF-L3-miss / L2PF-L3-missPrefetch requests (L1/L2 prefetcher) that miss L3 → fetch from DRAM. Spa uses these to measure prefetcher efficiency.
MLC (Intel Memory Latency Checker)Microbenchmark tool for loaded latency / bandwidth (what the paper uses to characterize CXL devices).
MIOPaper's custom tool: pointer-chase microbenchmark that logs per-request latencies (validates MLC, reveals CXL tail latencies).
CXL MC (Memory Controller)Third-party controller on CXL device (vs. CPU's integrated MC). Paper suspects MC scheduling + thermal throttling cause tail latencies.
Bidirectional PCIeCXL link supports simultaneous read/write (unlike DDR's unidirectional). CXL-A/B/D exploit this; CXL-C (FPGA) doesn't.
Key results at a glance

📊 Melody's headline numbers

  • 265 workloads (SPEC/GAPBS/cloud/ML) × 4 CXL devices × 7 latency configs (140-410ns).
  • CXL tail latencies: p99.9-to-p50 differences of ~75ns (CXL-D) to ~160ns (CXL-B/C), vs. 45-61ns (local/NUMA). Some devices show latencies reaching 1µs at higher percentiles.
  • Workload tolerance: 60% of workloads <10% slowdown on best CXL (CXL-D); 7% see 150-580% on worst (bandwidth-bound).
  • Spa accuracy: 95% of 265 workloads within 5% error.
  • Prefetcher drop: L2 coverage falls 2-38% on CXL → 5-15% cache slowdown for affected workloads.
  • CXL+NUMA danger: 520.omnetpp tail starts around p98 and reaches up to 800ns → 290% slowdown vs. 5% on CXL alone.
  • Period-based: 602.gcc: first two-thirds exceed 30% slowdown, overall average ~20% (hides bursty behavior).
