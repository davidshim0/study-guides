Making your DRAM think, so your GPU doesn't have to wait

Running a 70B LLM on a consumer GPU is expensive—unless you turn memory into a co-processor. This is the story of Hermes, a system that uses near-data processing in commodity DIMMs to slash data movement and make billion-parameter inference affordable.

📄 "Hermes: Augmenting GPU Memory with NDP-DIMM" · Liu et al. 🎓 Assumes basic cache/DRAM familiarity ⏱ ~25 min read
🧠

Compute infrequently-activated neurons where the data already lives—in DRAM—not by hauling everything across PCIe.
Measure which neurons are hot, map cold ones to NDP-DIMMs, and let memory do its own math.

Before you begin

Eight concepts Hermes builds on

Each primer unpacks one idea. Read them in order the first time, or jump to what's unfamiliar.

hot neurons (20% of params, 80% of work) cold neurons (80% of params, 20% of work) GPU (computation-rich) baseline offloading (PCIe-bound)
1LLM inference bottleneck: memory, not compute▾

A 70B LLM has 140 GB of parameters (FP16). A consumer GPU like the RTX 4090 has 24 GB of memory but 330 TOPS of compute. During token generation (the slow phase), you load huge weight matrices but perform relatively little arithmetic per byte—so you're memory-bandwidth-bound, not compute-bound.

Offloading parameters to host DRAM solves the capacity problem but creates a new one: PCIe bandwidth (~64 GB/s) is 15× slower than GPU memory (~936 GB/s). So 99% of runtime is spent waiting for data.

Now you're ready: Hermes solves this by computing in DRAM (NDP-DIMMs) instead of shipping data across PCIe.
2Activation sparsity: most neurons do nothing▾

LLMs use non-linear activations like ReLU, which zeros out many intermediate values. When an activation is zero, the corresponding weight parameters don't need to be loaded or computed—they're inactive. Across typical inputs, 70–90% of neurons are inactive.

Prior work exploits this by predicting which neurons will activate and loading only those. But even loading 10–30% of a 70B model still saturates PCIe.

Now you're ready: Hermes goes further: it partitions neurons into hot (frequently active) and cold (rarely active), then computes cold neurons in-place on NDP-DIMMs.
3The 80/20 rule of neuron activity▾

The paper's key observation: activation sparsity follows a power-law distribution. About 20% of neurons account for 80% of the computation, while 80% of neurons handle only 20%. This means hot neurons have 16× higher computation intensity than cold ones.

Hot neurons fit in GPU memory and justify GPU compute; cold neurons are too bulky for GPU memory but too infrequently used to justify hauling them across PCIe every time.

Now you're ready: Map hot neurons to GPU, cold neurons to NDP-DIMMs. GPU does heavy lifting; DIMMs do light lifting in-place.
4What is NDP (near-data processing)?▾

Instead of moving data to a far-away processor, NDP adds small processing units inside or next to the memory itself. For DRAM DIMMs, this means a lightweight compute core that can read data from the DRAM chips at full internal bandwidth (~1.6 TB/s per DIMM), do simple math, and write results back—without crossing a narrow bus.

NDP-DIMMs trade off raw compute power (hundreds of GFLOPS vs. GPU's hundreds of TFLOPS) for bandwidth and energy efficiency. They're ideal for memory-bandwidth-bound tasks with low arithmetic intensity.

Now you're ready: Hermes uses center-buffer NDP-DIMMs with 256 multipliers per DIMM to handle cold neuron computation at ~1 TB/s internal bandwidth.
5Token-wise similarity & layer-wise correlation▾

Token-wise similarity: adjacent tokens in a sequence often activate similar neurons (>90% overlap). Tokens 10+ positions apart drop to ~70% similarity, so a window of ~25 tokens captures most of the pattern.

Layer-wise correlation: if certain neurons in layer i−1 activate, neurons in layer i activate with high probability (>90% for correlated pairs). This lets you predict layer i activation from layer i−1 results.

Now you're ready: Hermes uses these patterns to build a lightweight predictor (under 1 MB) instead of expensive MLP-based predictors (GBs, 10–25% overhead).
6Why existing offloading solutions are slow▾

Systems like FlexGen and Deja Vu treat host DRAM as augmented storage: they load parameters from DRAM to GPU over PCIe, compute on GPU, and write results back. Even with activation sparsity (loading only 10–30% of weights), PCIe bandwidth (~64 GB/s) is the bottleneck—data transfer accounts for 90–99% of runtime.

Offloading to the host CPU instead (PowerInfer) helps slightly (CPU DRAM bandwidth ~90 GB/s), but CPU compute is weak and bandwidth is still limiting.

Now you're ready: Hermes avoids PCIe entirely for cold neurons by computing them in NDP-DIMMs, which have 15× higher internal bandwidth than PCIe.
7Prompting vs. token generation▾

LLM inference has two stages. Prompting (prefill): process the entire input sequence once to generate keys/values (KV cache). This is done once per inference and is compute-heavy enough to run efficiently on GPU.

Token generation: generate one token at a time, updating KV cache incrementally. This is repeated for each output token and accounts for >90% of total runtime. Each generation loads all weights but processes only one token, so it's memory-bandwidth-bound.

Now you're ready: Hermes focuses on token generation, where bandwidth matters most. Prompting runs entirely on GPU.
8What's cheap vs. expensive in memory systems▾

Cheap: commodity DDR4/DDR5 DIMMs (~$100/32 GB). Internal DRAM bandwidth within a DIMM is high (~1.6 TB/s). Adding a small NDP core costs <1% area overhead (~$2 per DIMM).

Expensive: HBM-PIM (processing-in-HBM) and CXL-attached AiM accelerators are high-performance but cost 10–50× more than DDR DIMMs. Server-grade GPUs (A100, H100) with large HBM are $10k–$50k.

Now you're ready: Hermes targets budget-friendly local deployment: one consumer GPU (~$1,600) + commodity NDP-DIMMs (~$900 total) = ~$2,500 vs. $50k for 5× A100s.

Hermes in a nutshell

The problem, the cheap measurement that fixes it, and how it all connects.

Running a 70B LLM on a consumer GPU (24 GB memory) requires offloading parameters to host DRAM. Existing systems treat DRAM as storage, loading data across PCIe (~64 GB/s). Even using activation sparsity to load only 10–30% of weights, 90–99% of runtime is PCIe transfer. The GPU sits idle.

Hermes observes that activation sparsity is power-law: 20% of neurons (hot) handle 80% of computation; 80% of neurons (cold) handle 20%. Hot neurons have 16× higher compute intensity. So: map hot neurons to the GPU (computation-rich, storage-limited) and compute cold neurons in NDP-DIMMs (storage-rich, computation-limited) at ~1 TB/s internal bandwidth. Only a few KB of results cross PCIe, not gigabytes of weights.

To partition hot/cold optimally, Hermes uses a two-step approach: (1) formulate an ILP problem and solve it offline using profiled data; (2) deploy a lightweight online predictor (under 1 MB, exploiting token-wise similarity and layer-wise correlation) to adjust the partition in real-time as neuron activity shifts. A window-based scheduler redistributes cold neurons across 8 DIMMs to balance load.

The payoff: Hermes achieves 13.75 tokens/s on LLaMA2-70B (single RTX 4090 + 8× NDP-DIMMs, total cost ~$2,500) vs. 0.04 tokens/s for HuggingFace Accelerate. That's 148× faster than FlexGen, 75× faster than Deja Vu, with <1% hardware overhead and nothing on the critical path.

🔑 Key takeaways

  • Activation sparsity is power-law: 20% hot neurons do 80% of work; 80% cold neurons do 20%.
  • Map hot neurons to GPU (high compute intensity), cold neurons to NDP-DIMMs (low compute intensity, high bandwidth).
  • NDP-DIMMs offer ~1 TB/s internal bandwidth (15× PCIe), so computing in-place beats loading across PCIe.
  • Lightweight predictor (<1 MB) uses token-wise similarity and layer-wise correlation to adjust hot/cold partition online.
  • Window-based scheduler redistributes cold neurons across DIMMs via DIMM-link to balance load.
  • Result: 13.75 tokens/s on LLaMA2-70B for ~$2,500 (vs. $50k for 5× A100s), 75–148× faster than prior offloading systems.
Seven everyday pictures

How to think about Hermes

Each picture stands alone—read them in any order. Every card ends with where the analogy breaks.

The core idea · Hot vs. cold neurons

A busy restaurant's kitchen

A restaurant has two areas: a fast stovetop (GPU) and a slow oven (NDP-DIMM). The chef puts high-frequency dishes (hot neurons) on the stovetop—quick access, fast cook. Low-frequency dishes (cold neurons) go in the oven—slower but happens in the background. The waiter doesn't carry dishes from the oven to the stovetop; the oven does its own cooking. Only finished plates cross the counter (PCIe).

Where it breaks: A real oven can't rebalance load dynamically; Hermes uses DIMM-link to shuffle cold neurons between DIMMs to avoid bottlenecks.
Bandwidth · Why compute in-place wins

Water pipes vs. garden hoses

PCIe is a garden hose (64 GB/s). NDP-DIMM internal bandwidth is a fire hydrant (1 TB/s). Moving 1 GB of data from DIMM to GPU via PCIe takes 16 ms; computing it in-place on NDP takes 1 ms. Even if GPU compute is 10× faster, you save 15 ms by avoiding the hose.

Where it breaks: The analogy assumes uniform compute time. In reality, GPU is 100–1000× faster than NDP for dense compute, so hot neurons (high compute intensity) still justify the transfer. Cold neurons don't.
Power-law distribution · The 80/20 rule

Pareto principle at scale

In a city, 20% of roads carry 80% of traffic. In LLMs, 20% of neurons handle 80% of activations. Those neurons (hot) have 16× higher "traffic density" (compute intensity) than the other 80% (cold). You optimize the busy roads with fast lanes (GPU); the quiet roads can be slower (NDP-DIMM) without hurting overall throughput.

Where it breaks: City traffic is static; LLM neuron activity is input-dependent. Hermes needs a real-time predictor to track which neurons are hot/cold as the input changes.
Lightweight predictor · FSM instead of MLP

Branch prediction vs. neural network

Classic MLP-based predictors (Deja Vu) are like running a full neural network to predict the next word—expensive (GBs of storage, 10–25% overhead). Hermes uses a finite state machine (4-bit counter per neuron): "this neuron activated last time → increment state; didn't activate → decrement." Under 1 MB, negligible overhead. It's branch prediction for neurons.

Where it breaks: FSMs assume temporal locality. If input context switches abruptly, the predictor lags until the state table updates. Layer-wise correlation mitigates this by cross-checking adjacent layers.
Load balancing · Window-based scheduling

Assembly line rebalancing

Eight workers (DIMMs) assemble parts. Worker #1 gets flooded with tasks; worker #8 idles. Instead of rebalancing after every task (expensive), Hermes waits for a window of 5 tasks, measures who's overloaded, then shifts tasks from the busiest to the idlest worker via a conveyor belt (DIMM-link, 25 GB/s). Overhead drops from 5.3% to 0.2%.

Where it breaks: Assembly lines can pause; LLM inference can't. Hermes hides the rebalancing latency by doing it during projection computation (when DIMMs are idle).
Offline partition · ILP optimization

City planning vs. real-time traffic

Before opening the roads, city planners optimize the layout (ILP solver: 110 seconds, profiling on C4/Pile datasets). During rush hour, sensors adjust traffic lights in real-time (lightweight predictor: <1 ms per token). The static plan handles 80% of cases; dynamic adjustments catch the rest.

Where it breaks: City planning assumes predictable patterns; LLMs are input-dependent. The offline partition captures "typical" workloads but needs online adjustment for atypical inputs.
Token-wise similarity · Temporal locality

Weather forecasting

Tomorrow's weather is likely similar to today's (temporal locality). The next token's activated neurons are >90% similar to the previous token's. Hermes caches the activation pattern in a 4-bit state table and updates it incrementally. No need to recompute from scratch every token—just adjust the deltas.

Where it breaks: Weather patterns shift seasonally; LLM context shifts mid-inference. When the window size exceeds ~25 tokens, similarity drops to 70%. Layer-wise correlation bridges the gap.
The full explanation

How it actually works

Section by section, with the paper's key mechanisms rebuilt as visuals.

1 · The problem—offloading saturates PCIe

Running LLaMA2-70B (140 GB FP16) on a consumer GPU (24 GB memory) requires offloading 116 GB to host DRAM. Existing systems (FlexGen, Deja Vu) load parameters over PCIe (~64 GB/s). Even using activation sparsity (loading 10–30% of weights), 90–99% of runtime is PCIe transfer. For OPT-66B, Deja Vu spends 89% on communication; FlexGen spends 99%. The GPU sits idle waiting for data.

🌱
Concretely: FlexGen achieves 0.16 tokens/s on OPT-66B (batch=1). Hermes achieves 20.37 tokens/s—a 127× speedup. The difference: Hermes computes cold neurons in NDP-DIMMs at ~1 TB/s internal bandwidth instead of loading them across PCIe at 64 GB/s.

2 · Power-law activation sparsity—the 80/20 rule

Activation sparsity in LLMs follows a power-law distribution. The paper measures activation frequency across neurons using ReLU (for OPT) or ReLU-substituted models (LLaMA2, Falcon). Key finding:

  • ~20% of neurons are activated frequently ("hot neurons") and account for ~80% of total computation.
  • ~80% of neurons are activated infrequently ("cold neurons") and account for ~20% of computation.

This means hot neurons have 16× higher compute intensity than cold neurons. Hot neurons justify GPU compute (high FLOPS, high bandwidth); cold neurons don't—they're better suited to NDP-DIMMs (lower FLOPS, but compute in-place at ~1 TB/s).

🍯
The key insight: map hot neurons to GPU (20% of params, 80% of work); map cold neurons to NDP-DIMMs (80% of params, 20% of work). GPU does heavy lifting; DIMMs do light lifting in-place. Only results (few KB) cross PCIe.

3 · NDP-DIMM architecture—computing in memory

Hermes uses a center-buffer NDP-DIMM design: each DIMM has a lightweight NDP core between the DRAM chips and the memory controller. The core includes:

  • GEMV unit: 256 multipliers (128-bit each, bit-serial) + accumulator + 256 KB buffer. Reads weights from DRAM, multiplies with activations, accumulates results. ~1 TB/s internal bandwidth.
  • Activation unit: 256 FP16 exponentiation/addition/multiplication units + comparator/adder trees + divider. Handles non-linear functions (ReLU, softmax).
  • DIMM-link: 25 GB/s inter-DIMM communication fabric (8 lanes, 25 Gb/s each) for redistributing neurons to balance load.

The NDP core operates at 1 GHz, costs ~1.23 mm² in 7nm TSMC, and adds <1% area overhead per DIMM. It complies with standard DRAM access protocols, so GPU can still load/store data normally.

💡
Why center-buffer design? It allows the NDP core to access all data in the DIMM at full internal bandwidth, not just a subset. This is critical because cold neurons are randomly activated—any neuron might be needed.

4 · Offline partition—ILP solver for initial hot/cold split

Hermes starts with an offline optimization to find the initial hot/cold partition. The problem is formalized as integer linear programming (ILP):

# Objective: minimize total inference latency
minimize Σ_l max(T_GPU_l, T_DIMM_l)
subject to:
  Σ_i M_i · x_GPU_i ≤ S_GPU       # GPU memory constraint
  Σ_i M_i · x_DIMM_j_i ≤ S_DIMM_j # DIMM memory constraint

# where:
T_GPU_l = T_GPU · Σ_i f_i · x_GPU_i + 2·T_sync
T_DIMM_l = max_j(T_DIMM · Σ_i f_i · x_DIMM_j_i)

Variables: x_GPU_i, x_DIMM_j_i are binary (1 if neuron i is placed on GPU or DIMM j). Parameters: f_i = activation frequency of neuron i (profiled offline on C4/Pile with 128 samples); M_i = memory size of neuron i; T_GPU, T_DIMM = time to compute one neuron on GPU/DIMM; T_sync = synchronization overhead (fetching activations from DIMM, sending results back).

The solver (PulP) takes ~110 seconds to find the optimal partition for LLaMA2-70B. This is done once, offline, before deployment.

🍯
Why ILP? The neuron mapping space is huge (2^1000+ for LLaMA-7B). ILP efficiently finds a near-optimal solution by leveraging convex optimization. The offline partition captures "typical" workloads; online adjustment handles input-specific variations.

5 · Online predictor—lightweight FSM for real-time adjustment

The offline partition is static, but neuron activity is input-dependent. The paper finds that ~52% of initialized hot neurons change activity during inference. So Hermes deploys a lightweight online predictor to adjust the hot/cold partition in real-time.

The predictor uses two mechanisms:

  1. Token-wise prediction (finite state machine): Each neuron has a 4-bit state (0–15). When a neuron activates, state += 4; when it doesn't, state -= 1. State > 10 → hot neuron (load to GPU); state ≤ 10 → cold neuron (keep on DIMM). State is initialized based on activation frequency during prompting. Storage: 4 bits × ~14k neurons/layer × 40 layers = 290 KB for LLaMA-13B.
  2. Layer-wise prediction (correlation table): For each neuron in layer i, store the top-2 correlated neurons from layer i−1 (sampled offline). If those neurons activated, boost the current neuron's predicted activation probability by λ·s2 (λ=6, s2=number of correlated neurons that activated). Prediction criterion: s1 + λ·s2 > 15 → activate.

Prediction accuracy: 98%. Storage: <1 MB. Runtime overhead: <0.1%. Compare to MLP-based predictors (Deja Vu): 2 GB storage, 10–25% overhead.

💡
Why FSM works: Token-wise similarity means adjacent tokens activate >90% of the same neurons. FSM tracks "momentum" (state) and adjusts incrementally. Layer-wise correlation catches fluctuations when context switches abruptly.

6 · Window-based scheduling—balancing load across DIMMs

Even with optimal hot/cold partition, load imbalance across 8 DIMMs can cause bottlenecks. The slowest DIMM determines overall latency. The paper observes that due to token-wise similarity, optimal neuron mapping within a window of 5 tokens has <5% variance. So Hermes groups every 5 tokens into a window, measures neuron activity, and redistributes cold neurons across DIMMs to balance load.

Algorithm:

# Window-based online scheduling (every 5 tokens)
Z_j = Σ_i C_j,i · A_i    # count activated neurons per DIMM
Sort DIMMs by Z_j (descending)
for id in [0, J/2):
  while Z_id ≤ Z_(J-id):  # pair busiest with idlest
    h = most activated neuron in DIMM_id
    C_id,h = 0; C_(J-id),h = 1  # remap h to DIMM_(J-id)

Redistribution uses DIMM-link (25 GB/s). Overhead: <0.2% of total runtime (down from 5.3% without DIMM-link). The window size (5 tokens) balances redistribution cost vs. load balance: too small → excessive migration; too large → imbalance persists.

💡
Why window-based? Token-wise similarity means adjacent tokens activate similar neurons. Redistributing every 5 tokens captures the pattern without over-reacting to single-token outliers. Fixed inter-DIMM traffic routes to different bridges to avoid congestion.

7 · Workflow—prompting and token generation

Hermes divides LLM inference into two stages:

  • Prompting (prefill): process the entire input sequence (e.g., 128 tokens) to generate KV cache. This is compute-heavy (high arithmetic intensity) and runs entirely on GPU. The predictor records neuron activity to initialize the state table and correlation table.
  • Token generation: generate one token at a time, updating KV cache incrementally. For each transformer layer:
    1. QKV generation: predict activated neurons; map hot neurons to GPU, cold neurons to NDP-DIMMs. GPU computes hot neurons; DIMMs compute cold neurons in-place. Results merge in DIMMs (few KB across PCIe).
    2. Attention: offload to NDP-DIMMs (memory-bandwidth-bound). KV cache stays in DIMMs to save GPU memory.
    3. Projection: run on GPU (no activation sparsity). During projection, scheduler adjusts hot/cold partition and redistributes cold neurons across DIMMs (hidden by GPU compute).
    4. MLP: same as QKV—predict, partition, compute (GPU+DIMMs), merge.
🌱
Why prompting on GPU? Prompting has high arithmetic intensity (process 128 tokens in one shot), so GPU's high FLOPS justifies the data transfer. Token generation has low arithmetic intensity (process 1 token per step), so bandwidth dominates—NDP-DIMMs win.

Worked examples

Four traces using the paper's real data—computing speedup, profiling overhead, predictor accuracy, and partition decisions.

① Computing Hermes speedup over FlexGen (OPT-66B, batch=1)

FlexGen achieves 0.16 tokens/s. Hermes achieves 20.37 tokens/s. What's the speedup, and why?

  1. Speedup = 20.37 / 0.16 = 127.3×. That's the end-to-end result from the paper (Table in Results).
  2. Why so large? FlexGen spends 99% of runtime on PCIe transfer (loading weights from host DRAM at 64 GB/s). OPT-66B has ~130 GB of weights; even loading 10% (with activation sparsity) takes 13 GB ÷ 64 GB/s ≈ 200 ms per layer. With 64 layers, that's 12.8 seconds per token → 0.078 tokens/s (close to measured 0.16).
  3. Hermes avoids PCIe: hot neurons (20% of params, 80% of work) stay in GPU memory (24 GB). Cold neurons (80% of params, 20% of work) compute in NDP-DIMMs at ~1 TB/s internal bandwidth. Only results (few KB) cross PCIe → negligible transfer cost.
  4. Compute time dominates: the paper reports Hermes achieves 20.37 tokens/s on OPT-66B. The speedup comes from eliminating PCIe data transfer by computing cold neurons in-place on NDP-DIMMs.
🍯
Insight: The 127× speedup comes from eliminating PCIe transfer (99% of FlexGen's runtime) by computing cold neurons in-place on NDP-DIMMs.

② Profiling the offline partition (LLaMA2-70B)

The ILP solver takes 110 seconds to find the offline partition. What parameters does it use, and how accurate is the result?

  1. Input parameters: activation frequency f_i for each neuron (profiled on C4/Pile datasets, 128 samples). LLaMA2-70B has 80 layers × ~14k neurons/layer = 1.12M neurons. The profiler records how often each neuron activates during the 128 samples.
  2. Constraints: GPU memory = 24 GB (RTX 4090); DIMM memory = 32 GB × 8 = 256 GB. Neuron size M_i varies (QKV projections: 4k×8k FP16 = 64 MB; MLP: 8k×28k FP16 = 448 MB).
  3. Solver output: binary variables x_GPU_i, x_DIMM_j_i indicating neuron placement. For LLaMA2-70B, ~15% of parameters fit in GPU (hot neurons); 85% go to DIMMs (cold neurons).
  4. Accuracy check: the paper compares the offline partition (fixed) to an oracle (theoretically optimal for each input). Offline partition achieves 1.63× slower than oracle on average. Adding online adjustment (predictor) closes the gap to <1.1× slower than oracle.
💡
The 110-second solve time is acceptable because it's done once, offline, before deployment. Online, the predictor adjusts the partition in <1 ms per token.

③ Predictor accuracy and overhead (LLaMA-13B)

The predictor uses a 4-bit FSM + correlation table. What's the storage cost, accuracy, and runtime overhead?

  1. Storage (state table): LLaMA-13B has 40 layers. Self-attention: 4k neurons/layer (QKV). MLP: 10.5k neurons/layer (FC1, FC2). Total: (4k + 10.5k) × 40 = 580k neurons. Storage: 580k × 4 bits ≈ 290 KB.
  2. Storage (correlation table): The predictor stores correlation data. The paper states total predictor storage is <1 MB including both tables.
  3. Accuracy: the paper measures prediction accuracy on COPA, WikiText2, PIQA datasets. Token-wise similarity alone: ~85% accuracy. Token-wise + layer-wise: 98% accuracy. This means 98% of neurons are correctly classified as hot/cold.
  4. Runtime overhead: updating the state table takes <0.1% of total runtime (incrementing/decrementing 4-bit counters). Compare to Deja Vu's MLP-based predictor: 10–25% overhead (running a small MLP per layer).
🍯
The predictor is 100–250× cheaper than MLP-based predictors while achieving 98% accuracy. The key: exploit token-wise similarity (FSM) and layer-wise correlation (correlation table) instead of learning from scratch.

④ Window-based scheduling overhead (OPT-66B)

Redistributing cold neurons across DIMMs costs data transfer. How much, and why is DIMM-link critical?

  1. Without DIMM-link: neuron redistribution goes through the host (CPU). Transfer cold neurons from DIMM_i → host → DIMM_j via PCIe (64 GB/s). The paper reports overhead reduction from 5.3% to <0.2% with DIMM-link.
  2. With DIMM-link: direct inter-DIMM communication at 25 GB/s. Overhead: <0.2% of total runtime (hidden during projection computation).
  3. Why window size = 5? Token-wise similarity drops to ~70% beyond 25 tokens, so a 5-token window captures local patterns. Smaller windows (e.g., 1 token) would cause excessive redistribution overhead; larger windows (e.g., 10 tokens) would let imbalance persist longer.
💡
DIMM-link reduces redistribution overhead by 25× (5.3% → 0.2%). The 25 GB/s bandwidth is sufficient because redistribution is infrequent (every 5 tokens) and hidden during GPU projection computation.

Problem 1 · easy
A neuron in LLaMA2-70B activates 5% of the time during prompting. Should it be mapped to GPU or NDP-DIMM in the offline partition, and why?
Show answer
Cold neuron → NDP-DIMM. A 5% activation frequency means it's activated infrequently (below the ~20% hot neuron threshold). Its compute intensity is low, so offloading it to GPU would waste bandwidth (loading it across PCIe when it activates). Instead, map it to NDP-DIMM, where it can compute in-place at ~1 TB/s internal bandwidth when needed.Misconception: that any neuron that activates at all should go to GPU. GPU is for frequently activated neurons (high compute intensity). Infrequent activation → NDP-DIMM.
Problem 2 · medium
The predictor uses a 4-bit FSM state. If a neuron's state is 10 and it activates in the next token, what is the new state? If it doesn't activate, what is the new state?
Show answer
Activates: state = 10 + 4 = 14. Doesn't activate: state = 10 − 1 = 9. The FSM increments by 4 on activation (strong signal) and decrements by 1 on non-activation (weak signal). This asymmetry captures the fact that activation is a stronger indicator of "hotness" than non-activation is of "coldness."Misconception: that increment and decrement should be symmetric (e.g., ±1). The asymmetry (±4 vs. ±1) reflects the prior that once a neuron starts activating, it's likely to keep activating (temporal locality).
Problem 3 · medium
A DIMM has 256 multipliers running at 1 GHz. Each multiplier processes 8 FP16 values simultaneously. What is the peak FLOPS of the DIMM, and why is it much lower than a GPU?
Show answer
256 multipliers × 8 FP16/multiplier × 2 operations/FP16 (multiply + accumulate) × 1 GHz = 4.096 TFLOPS. A GPU (RTX 4090) delivers 330 TOPS (FP16) = 330 TFLOPS → the GPU is ~80× faster. The DIMM trades FLOPS for bandwidth: it has ~1 TB/s internal bandwidth (15× PCIe), ideal for memory-bandwidth-bound tasks like cold neuron computation.Misconception: that NDP-DIMMs should match GPU FLOPS. NDP is designed for bandwidth-bound tasks, not compute-bound. Cold neurons have low compute intensity, so bandwidth matters more than FLOPS.
Problem 4 · hard
The paper finds that ~52% of initialized hot neurons change activity during inference. Construct a scenario where the offline partition alone (no online adjustment) causes a 1.63× slowdown, and explain how the online predictor fixes it.
Show answer
Scenario: offline partition maps neurons {1, 2, 3} to GPU (hot) and {4, 5, 6, 7, 8} to DIMM (cold) based on profiling. During inference, input context switches: neurons {1, 2, 3} become inactive, and neurons {4, 5} become highly active. Without online adjustment, GPU sits idle (1, 2, 3 inactive) while DIMM_0 is overloaded (4, 5 active). Slowdown: 1.63×. With online predictor: state table detects that {1, 2, 3} have low state (≤10) and {4, 5} have high state (>10). Predictor swaps {4, 5} to GPU and {1, 2, 3} to DIMM. GPU and DIMMs rebalanced → slowdown <1.1×.Misconception: that offline profiling captures all patterns. Problem 5 · medium
Why does Hermes use a window size of 5 tokens for neuron redistribution? What happens if the window size is 1 or 25?
Show answer
Window size = 5 balances redistribution overhead vs. load balance. If window = 1: redistribute after every token → excessive DIMM-link traffic (26 redistributions per 128 tokens). Overhead: 5.3% → 10%+. If window = 25: token-wise similarity drops to ~70% at 25-token distance, so optimal neuron mapping varies within the window. Load imbalance persists → slowest DIMM bottlenecks for 25 tokens. Window = 5: token-wise similarity >90%, so one redistribution per 5 tokens captures the pattern without over-reacting. Overhead: 0.2%.Misconception: that smaller windows are always better (more reactive). Too small → redistribution overhead dominates. The window size exploits token-wise similarity to minimize redistribution while maintaining balance.
Problem 6 · hard
The paper offloads attention computation to NDP-DIMMs. Why not compute attention on GPU (which is 80× faster), and only use DIMMs for MLP?
Show answer
Attention is memory-bandwidth-bound, not compute-bound. For self-attention, you load KV cache (batch × seq_len × hidden_dim) and compute Q·K^T + softmax + V. The arithmetic intensity (FLOPs per byte) is low because you read large KV cache but do relatively little compute per element. GPU would spend most time transferring KV cache from DRAM → GPU memory via PCIe (64 GB/s bottleneck). NDP-DIMMs compute attention in-place at ~1 TB/s internal bandwidth → 15× faster. Additionally, keeping KV cache in DIMMs saves GPU memory (critical for consumer GPUs with 24 GB).Misconception: that higher FLOPS always wins. For bandwidth-bound tasks (attention, GEMV), bandwidth matters more than FLOPS. GPU's 80× FLOPS advantage is wasted if 99% of time is spent waiting for data.
Problem 7 · easy
Hermes uses center-buffer NDP-DIMM design. Why not use rank-level or bank-level NDP (cheaper)?
Show answer
Center-buffer NDP allows the NDP core to access all data in the DIMM at full internal bandwidth. Rank-level or bank-level NDP can only access data in one rank/bank, so if a cold neuron is in a different rank/bank, the NDP core can't access it without expensive inter-rank/bank transfers. Since cold neurons are randomly activated (input-dependent), any neuron might be needed → center-buffer design is necessary to ensure full coverage at high bandwidth.Misconception: that cheaper NDP designs (rank-level, bank-level) are sufficient. Random activation patterns require full DIMM access, which only center-buffer NDP provides.
Problem 8 · medium
The paper claims Hermes achieves 79.1% of TensorRT-LLM's performance (batch=1) at ~5% of the cost. How is this possible, and where does the gap come from?
Show answer
TensorRT-LLM uses 5× A100-40GB GPUs (~$50k) with 200 GB HBM. Hermes uses 1× RTX 4090 (~$1,600) + 8× NDP-DIMMs (~$900) with 24 GB GPU memory + 256 GB DIMM memory. TensorRT-LLM fits all weights in HBM (no offloading) → zero transfer overhead. Hermes offloads 85% of weights to DIMMs → 20% of compute runs on slower NDP cores (80× slower than GPU). The performance gap exists because NDP cores have lower compute throughput than GPU. At batch=1, Hermes achieves 79.1% because hot neurons (80% of work) run on GPU at full speed.Misconception: that Hermes should match TensorRT-LLM performance. Hermes trades ~20% performance for 95% cost savings by using cheaper hardware (NDP-DIMMs) for low-compute-intensity tasks.
Quick answers

Frequently asked questions

Ten questions newcomers ask most. Tap any to expand.

Q1What does "near-data processing" (NDP) actually mean?▾
NDP means adding lightweight processing units inside or next to the memory (DRAM chips), so data can be processed in-place at high internal bandwidth (~1 TB/s) instead of being moved to a distant processor (e.g., GPU) across a slow bus (PCIe, 64 GB/s). For Hermes, NDP-DIMMs have small compute cores (256 multipliers per DIMM) between the DRAM chips and the memory controller.
Q2Why not just use a bigger GPU instead of NDP-DIMMs?▾
A GPU with enough memory for LLaMA2-70B (140 GB) would need 5× A100-40GB GPUs (~$50k) or 2× H100-80GB GPUs (~$60k). Hermes uses 1× RTX 4090 ($1,600) + 8× NDP-DIMMs (~$900) = ~$2,500. That's 95% cheaper. The trade-off: Hermes runs 20% slower (79.1% of TensorRT-LLM performance at batch=1) because NDP cores are slower than GPU. For local deployment (small batch), 79% performance at 5% cost is a win.
Q3Why not use the host CPU instead of NDP-DIMMs?▾
PowerInfer does this: offload cold neurons to CPU DRAM (~90 GB/s bandwidth). The problem: CPU memory bandwidth is only 1.4× higher than PCIe, so data transfer is still slow. Additionally, CPU compute is weak (100–1000× slower than GPU for dense compute). Hermes achieves 7.17× speedup over CPU-based offloading (Hermes-host) because NDP-DIMMs have 15× higher internal bandwidth than PCIe and compute in-place (no CPU bottleneck).
Q4What is the 80/20 rule, and why does it matter?▾
Activation sparsity in LLMs follows a power-law: ~20% of neurons are activated frequently (hot) and account for 80% of total computation; ~80% of neurons are activated infrequently (cold) and account for 20% of computation. This means hot neurons have 16× higher compute intensity than cold neurons. Hot neurons justify GPU compute (high FLOPS); cold neurons don't (better suited to NDP-DIMMs with lower FLOPS but high bandwidth).
Q5How does the predictor achieve 98% accuracy with <1 MB storage?▾
The predictor exploits two patterns: (1) token-wise similarity (adjacent tokens activate >90% of the same neurons) → use a 4-bit FSM per neuron to track "momentum"; (2) layer-wise correlation (if neuron X in layer i-1 activates, neuron Y in layer i activates with >90% probability) → store top-2 correlated neurons per neuron. Combined: 98% accuracy, <1 MB storage. Compare to MLP-based predictors: 2 GB storage, 10–25% overhead.
Q6Why does Hermes beat Deja Vu (75× speedup)?▾
Deja Vu uses activation sparsity to load only 10–30% of weights, but those weights still cross PCIe (64 GB/s) from host DRAM → GPU. PCIe transfer accounts for 89% of runtime. Hermes computes cold neurons (80% of params, 20% of work) in NDP-DIMMs at ~1 TB/s internal bandwidth → no PCIe transfer. Only results (few KB) cross PCIe. Additionally, Deja Vu's MLP-based predictor costs 10–25% overhead; Hermes' FSM predictor costs <0.1%.
Q7What is DIMM-link, and why is it critical?▾
DIMM-link is an inter-DIMM communication fabric (25 GB/s per link, 8 lanes) that allows direct data transfer between DIMMs without going through the host. It's critical for redistributing cold neurons across DIMMs to balance load. Without DIMM-link, redistribution goes through host (PCIe, 64 GB/s) → 5.3% overhead. With DIMM-link (25 GB/s, direct) → 0.2% overhead. The 25× speedup comes from bypassing the host bottleneck.
Q8Does Hermes work for batch sizes >1?▾
Yes. Hermes achieves average speedups of 148.98× over FlexGen and 75.24× over Deja Vu across batch sizes 1–16. At batch=1, Hermes is bandwidth-bound (NDP internal bandwidth is the win). At batch=16, Hermes is compute-bound (NDP cores saturate), but still outperforms FlexGen/Deja Vu because PCIe transfer dominates in those systems. TensorRT-LLM (5× A100s) wins at batch=16 because all weights fit in HBM.
Q9Why does the offline partition use ILP instead of greedy heuristics?▾
Greedy heuristics (e.g., "map the most frequently activated neurons to GPU first") fail when neurons have correlated activation patterns or non-convex utility curves. ILP formulates the problem as: minimize max(T_GPU, T_DIMM) subject to memory constraints. The solver (PulP) finds a near-optimal solution in 110 seconds (offline, acceptable). Greedy would be faster (seconds) but suboptimal (1.5–2× slower in practice).
Q10What are the limitations and future directions?▾
Hermes is a research prototype demonstrating NDP-DIMM feasibility for LLM inference. Current limitations include single-GPU support, specialized hardware requirements (NDP-DIMMs), and performance trade-offs for very large batch sizes. The paper demonstrates the core technique without explicitly flagging specific future work items.

Glossary & further reading

Core terms
HermesThe paper's system: a single consumer-grade GPU augmented with 8× NDP-DIMMs to achieve affordable LLM inference by computing cold neurons in-place.
NDP-DIMMNear-Data Processing DIMM: a memory module with a lightweight compute core (256 multipliers, activation units, DIMM-link) that processes data at ~1 TB/s internal bandwidth.
Hot neuronA neuron activated frequently (~20% of all neurons, 80% of compute). Stored in GPU memory. Compute intensity: 16× higher than cold neurons.
Cold neuronA neuron activated infrequently (~80% of all neurons, 20% of compute). Offloaded to NDP-DIMMs. Compute intensity: 1/16 of hot neurons.
Activation sparsityThe property that 70–90% of neurons are inactive (zeroed by ReLU) in LLM inference. Follows power-law distribution: 20% hot, 80% cold.
Token-wise similarityAdjacent tokens activate >90% of the same neurons (temporal locality). Similarity drops to ~70% at 25-token distance.
Layer-wise correlationIf neuron X in layer i-1 activates, neuron Y in layer i activates with >90% probability. Exploited by the correlation table.
Lightweight predictorA 4-bit FSM (finite state machine) per neuron + correlation table. Storage: <1 MB. Accuracy: 98%. Overhead: <0.1%.
ILP (offline partition)Integer Linear Programming problem solved offline to find the initial hot/cold partition. Takes 110 seconds using PulP solver.
DIMM-linkInter-DIMM communication fabric (25 GB/s) for redistributing neurons to balance load. Reduces overhead from 5.3% to 0.2%.
Supporting terms
PCIe bandwidth~64 GB/s for PCIe 4.0 ×16. The bottleneck in offloading-based systems (FlexGen, Deja Vu). Hermes avoids it by computing in NDP-DIMMs.
Internal DRAM bandwidth~1.6 TB/s per DIMM (DDR4-3200). 15× higher than PCIe. Enables NDP cores to process data in-place at high speed.
Center-buffer NDPNDP design where the compute core sits between DRAM chips and memory controller, allowing access to all data in the DIMM at full bandwidth.
GEMV unitGeneral Matrix-Vector multiplication unit in NDP core. 256 multipliers, 128-bit each, bit-serial. Reads weights from DRAM, multiplies with activations, accumulates.
Activation unitNDP unit for non-linear functions (ReLU, softmax). 256 FP16 exponentiation/addition/multiplication units + comparator/adder trees + divider.
Prompting (prefill)First stage of LLM inference: process entire input sequence to generate KV cache. Compute-heavy, runs on GPU.
Token generationSecond stage: generate one token at a time, updating KV cache. Memory-bandwidth-bound, benefits from NDP.
FlexGenBaseline offloading system (zig-zag scheduling). Achieves 0.16 tokens/s on OPT-66B. 99% of runtime is PCIe transfer.
Deja VuActivation sparsity-based offloading system (MLP predictor). Achieves ~0.27 tokens/s on OPT-66B. 89% of runtime is PCIe transfer.
TensorRT-LLMHigh-performance inference system (5× A100s, $50k). Achieves ~17 tokens/s on LLaMA2-70B (batch=1). Hermes achieves 79.1% of this at 5% cost.
