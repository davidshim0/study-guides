1Why LLM inference is memory-bound▾

LLMs decode one token at a time, turning matrix multiplications (GEMMs) into batched matrix-vector products (GEMVs) with low arithmetic intensity. Meanwhile, the KV cache — the activations from all prior tokens — grows linearly with sequence length. Together, these mean the GPU spends more time waiting for data from memory than computing, creating a memory wall.

The paper cites up to 72.7% of memory (34.4 GB out of 47.3 GB total) consumed by KV cache in LLaMA-7B with batch size 32 and 2K sequence length. Fixing this bottleneck is what Ecco targets.

Now you're ready: Ecco compresses the cache, reducing memory bandwidth and capacity demands during this memory-bound decode phase.
2What cache compression is (and isn't)▾

Cache compression applies a lossless or near-lossless transform to data blocks before they're written to cache/memory, then decompresses them on-the-fly when read. It's not the same as quantizing the model — you're still computing in FP16 or whatever precision, you're just squeezing the data in transit and at rest.

Commercial GPUs (e.g., NVIDIA A100) already use lossless compression (e.g., BDI) but only achieve ~1.2× ratios. Ecco targets 4× (for weights/KV) and 2× (for activations) by exploiting the fact that small accuracy losses are acceptable in LLM contexts.

Now you're ready: Ecco is a cache-level technique that works alongside the existing model, not a model compression scheme like AWQ.
3Information entropy and "bit efficiency"▾

Entropy 𝐻 measures how much information is packed into a distribution: 𝐻 = −Σ 𝑝ᵢ log₂ 𝑝ᵢ. Higher entropy means the codes are used more evenly, wasting fewer bits. The paper defines bit efficiency 𝜂 = 𝐻 / 𝐵ᵣₑₐₗ, where 𝐵ᵣₑₐₗ is the real bit cost (data + metadata).

Figure 2 in the paper shows tensor-wise quantization at 2.25% bit efficiency (low entropy, uniform codes), while Ecco's group-wise + Huffman scheme reaches 78.54% — nearly all bits carry useful information.

Now you're ready: Ecco maximizes bit efficiency by making the quantized values' distribution as rich as possible (via k-means) and encoding them compactly (via Huffman).
4Quantization: uniform vs. non-uniform▾

Uniform quantization (e.g., AWQ) maps values to a fixed grid: ˜𝑤 = 𝑆 · round(𝑤/𝑆 + 𝑍). Simple, but wastes codes if the distribution is non-uniform. Non-uniform quantization (e.g., k-means) clusters values and assigns codes to centroids — fewer wasted slots, better fit to actual data distribution.

Ecco uses k-means with 15 centroids per group (plus the absolute max as a 16th "code"), then Huffman-codes those 16 indices. This captures more entropy than a uniform grid.

Now you're ready: Ecco's compression layer is essentially "group-wise k-means → Huffman coding" to maximize information density.
5Huffman coding (and its parallelism problem)▾

Huffman coding assigns shorter bit sequences to more frequent symbols. It's optimal for a given frequency distribution — but it's variable-length, so decoding is inherently sequential: you can't skip ahead without knowing where each code ends.

The paper's key hardware innovation is a parallel Huffman decoder: 64 decoders each process an 8-bit chunk plus 7-bit overlap, then a tree of concatenators merges the results in 6 stages. This drops latency from ~200 cycles (sequential) to 28 cycles (pipelined).

Now you're ready: Ecco's decoder design is what makes Huffman coding practical at GPU throughput — without it, the compression would be too slow.
6Compression ratio and block size▾

The paper targets power-of-two compression ratios (2× or 4×) to simplify address translation and memory alignment. It also fixes the compressed block size at 64 bytes — the default DRAM-to-L2 transaction size — so that loading one compressed block is a single memory operation.

For 4× compression, a 256-byte block (128 FP16 values) compresses to 64 bytes. For 2×, a 128-byte block (64 FP16 values) compresses to 64 bytes. This keeps all metadata (scale factors, Huffman tables, outliers) inside the block, avoiding extra memory trips.

Now you're ready: The 64-byte target is the hardware constraint that drives all of Ecco's design — everything must fit.
7The KV cache (and why it's huge)▾

In Transformer decoding, each new token attends to all prior tokens. To avoid recomputing past keys and values, they're cached. For a sequence of length 𝐿 with batch size 𝐵, the KV cache is ~2×𝐿×𝐵×hidden_dim×num_layers × sizeof(dtype). At 2K sequence length, 34.4 GB of the 47.3 GB total in LLaMA-7B.

Ecco compresses the KV cache at 4×, turning that 34.4 GB into ~8.6 GB — freeing up ~26 GB for larger batches or longer contexts.

Now you're ready: KV cache compression is where Ecco's biggest capacity gains come from.
8GPU memory hierarchy (L2, HBM, and transaction granularity)▾

GPUs have a register file → L1 cache/shared memory → L2 cache → HBM (high-bandwidth memory) hierarchy. Data moves in fixed-size chunks: 32 bytes (one sector) is the minimum L1–L2 unit; 64 bytes is the typical DRAM–L2 transaction. The L2 is the last stop before HBM, where bandwidth is precious.

Ecco places the decompressor between L1 and L2, and the compressor between L2 and HBM. This keeps uncompressed data only in the fast, small caches, and stores everything in HBM/L2 compressed.

Now you're ready: Ecco hooks into the existing L2 memory controller, adding minimal latency and area (<1% of chip).

The whole story in one place

Ecco in a nutshell

The problem, the entropy-based compression that fixes it, and how it all connects.

LLM inference is memory-bound: decoding one token at a time turns compute-heavy GEMMs into bandwidth-hungry GEMVs, and the KV cache balloons with sequence length (up to 72.7% of total memory). Existing quantization schemes (AWQ, SmoothQuant) reduce model size but add runtime overhead — dequantization on the critical path slows things down, and accuracy often suffers.

Ecco sidesteps these issues by compressing the cache, not the model. It combines group-wise non-uniform quantization (k-means with 15 centroids per 128-element group, chosen from 64 shared patterns) with Huffman coding (4 codebooks per pattern) to exploit the entropy characteristics of LLM data. For weights and KV cache, this yields 4× compression; for activations, 2×. Critically, all metadata (scale factors, pattern indices, Huffman codes, and padded outliers) fits in a fixed 64-byte block — the DRAM–L2 transaction size — so decompression is a single memory load, no extra trips.

To make Huffman decoding fast, Ecco introduces a parallel decoder: 64 sub-decoders process overlapping 8-bit chunks (each decoding 1–4 symbols), then a 6-stage tree merges the results. This cuts latency from ~200 cycles (sequential Huffman) to 28 cycles (pipelined), matching the L2 cache's throughput. The encoder is similarly optimized, using min/max comparisons instead of full MSE to select the best k-means pattern at runtime (for KV cache). The compressor and decompressor sit between the L2 and HBM, adding ~5.11 mm² area (0.62% of A100) and 7.36 W power (9% of idle).

The payoff: on LLaMA/Mistral models, Ecco achieves up to 2.9× speedup over AWQ (average 2.1×), 2.4× over Olive, and 1.8× over SmoothQuant — plus nearly 4× memory capacity increase — while maintaining state-of-the-art perplexity (LLaMA2-13B: 5.03 vs. 4.88 FP16). On Winogrande, it outperforms all 4-bit baselines by >2%.

🍯 Key takeaways

  • Compress the cache, not the model — no accuracy penalty, minimal hardware overhead.
  • Group-wise k-means (15 centroids/group, 64 shared patterns) + Huffman coding (4 codebooks/pattern) maximizes bit efficiency (78.54%).
  • Parallel Huffman decoder (64 sub-decoders → 6-stage tree) drops latency 200→28 cycles, matching L2 throughput.
  • All metadata fits in a 64-byte block (one DRAM transaction) — no extra memory trips.
  • 4× compression for weights/KV, 2× for activations; <1% chip area, <10% idle power.
  • Up to 2.9× speedup over AWQ, 4× capacity increase, SOTA perplexity.

The core idea · Cache compression

A smart zipper on your suitcase

You're packing for a trip. Instead of buying a bigger suitcase (more HBM), you use vacuum bags to compress your clothes (cache data). When you arrive, you unzip the bag (decompress) to use them. Ecco is the vacuum bag: it squeezes data 4× smaller before it goes to memory, then inflates it on-the-fly when the GPU needs it. Your "suitcase" (GPU memory) now holds 4× more.

Where it breaks: real vacuum bags take time to seal; Ecco's compressor runs in parallel with memory transactions, adding <28 cycles — fast enough to not bottleneck the GPU.
Quantization · Uniform vs. non-uniform

A tailor vs. off-the-rack clothes

Uniform quantization (AWQ) is like buying a shirt in size M — it fits most people okay, but wastes fabric on some (too loose) and doesn't fit others (too tight). Non-uniform quantization (k-means) is a tailor: measure each person, cut the cloth to fit. Ecco uses k-means with 15 centroids per group, clustering values where they actually fall — no wasted codes, tighter fit, higher entropy.

Where it breaks: a tailor is slow; Ecco pre-computes 64 shared k-means patterns offline and just picks the best one at runtime, keeping overhead low.
Huffman coding · Variable-length codes

Text messaging with shortcuts

If you text "lol" a hundred times a day and "antidisestablishmentarianism" once a year, your autocorrect learns "lol" = one keystroke, while the long word takes 28. Huffman coding does the same: assign 2-bit codes to frequent quantized values, 8-bit codes to rare ones. On average, you send fewer bits. Ecco uses 4 Huffman codebooks per k-means pattern, tailored to different frequency distributions.

Where it breaks: autocorrect is sequential (type one letter → suggests next); Huffman decoding is too. Ecco's parallel decoder (64 sub-decoders at once) breaks the sequential bottleneck, making it GPU-fast.
Parallel decoding · Breaking the sequential barrier

A relay race, not a single runner

Traditional Huffman decoding is like one runner completing a marathon — you wait for them to finish each segment before the next starts, taking ~200 cycles. Ecco's parallel decoder is a relay: 64 runners start simultaneously, each decoding an 8-bit chunk (plus 7-bit overlap to know where the next starts), then hand off. Six merge stages later, you have the full result in 28 cycles — 7× faster.

Where it breaks: real relay runners hand off a baton physically; Ecco's "handoff" is a tree of concatenators merging outputs, implemented in silicon. The speedup is architectural, not just pipelining.
Fixed block size · Everything fits in one load

A self-contained shipping container

DRAM-to-L2 transactions are 64 bytes. If you need to fetch metadata (scale factors, Huffman tables) separately, you make extra trips — slow. Ecco packs everything into that 64-byte container: the Huffman-coded data, the scale factor, the k-means pattern index, and even padded outliers. One load, one decompress, done. No extra memory transactions.

Where it breaks: shipping containers hold fixed cargo; if Ecco's Huffman output exceeds 64 bytes, it clips the excess (loses <0.04% of values), and if it's under, it pads with outliers (preserves top 16 values). The "container" is rigid; the packing is flexible.
Weights vs. KV cache · Static vs. dynamic

A library book vs. a whiteboard

Model weights are like library books: read-only during inference, so you can compress them offline using full MSE to pick the best k-means pattern. The KV cache is a whiteboard: written at runtime, so Ecco uses a fast min/max comparison (not MSE) to select the pattern on-the-fly. Both get 4× compression, but the selection method differs — optimized offline vs. cheap at runtime.

Where it breaks: real whiteboards don't need compression; Ecco's KV cache compression is what saves the most memory (34.4 GB → 8.6 GB in LLaMA-7B), making it the killer feature for long contexts.
Hardware overhead · Tiny compressor/decompressor

A paperback book in a backpack

Adding Ecco to a GPU is like slipping a paperback into a backpack — you barely notice the weight. The compressor + decompressor occupy 5.11 mm² on an 826 mm² chip (0.62% area) and draw 7.36 W vs. 82 W idle (9% power). They sit off the critical path (between L2 and HBM), so they don't slow down normal cache hits. The "book" is there when you need it, invisible when you don't.

Where it breaks: a paperback doesn't interact with the backpack's contents; Ecco's decompressor does add 28 cycles latency per block. But since memory trips are ~400 cycles anyway, 28 is noise — like a book adding 2 oz to a 20 lb backpack.
The full explanation

How it actually works

Section by section, with the paper's key mechanisms rebuilt as visuals you can read at a glance.

1 · The problem — LLMs are memory-bound

LLM decoding is inherently memory-bound due to two factors. First, auto-regressive generation produces one token at a time, turning GEMMs (batch × seq × hidden) into batched GEMVs (batch × 1 × hidden). With small batch sizes (≤64), arithmetic intensity plummets — the GPU waits for data more than it computes. Second, the KV cache grows linearly with sequence length: for each new token, you must load all prior keys and values to compute attention. In LLaMA-7B with batch=32 and seq=2K, the KV cache alone is 34.4 GB out of 47.3 GB total (72.7%).

Existing solutions like AWQ quantize weights to 4 bits but keep activations at 16 bits, requiring dequantization on every GEMM. This adds runtime overhead — extra multiply-adds, metadata loads — that can negate the bandwidth savings. QuaRot even slows down vs. FP16 at batch ≤64 (Figure 3 in the paper). The root issue: they compress the model, which lives in the critical compute path. Ecco compresses the cache, which sits off that path.

🌱
Concretely: on A100, L2→HBM bandwidth is ~2 TB/s, but LLM decoding saturates it at batch=32. Compress data 4× and you quadruple effective bandwidth — or equivalently, fit 4× more in L2/HBM, reducing evictions and speeding up the whole pipeline.

2 · Defining bit efficiency — entropy vs. overhead

The paper introduces bit efficiency 𝜂 = 𝐻 / 𝐵ᵣₑₐₗ, where 𝐻 is Shannon entropy (−Σ 𝑝ᵢ log₂ 𝑝ᵢ) and 𝐵ᵣₑₐₗ is the real bit cost (data + metadata). High entropy means the quantized codes are used evenly — no wasted slots. Low overhead means little metadata (scale factors, zero points, etc.).

Ecco's group-wise k-means (15 centroids + absmax = 16 codes) plus Huffman coding achieves the highest entropy (3.15 bits) and lowest overhead (4.01 bits/value, including all metadata in the 64-byte block). The 78.54% bit efficiency means nearly 4 out of 5 bits carry information, vs. 2 out of 90 for tensor-wise.

🍯
The punchline: maximizing bit efficiency is how Ecco gets 4× compression without sacrificing accuracy. High entropy (k-means fits the distribution) + low overhead (Huffman codes + shared patterns) = more information per bit.

3 · Weight compression — k-means + Huffman

The compression pipeline for weights has nine steps (Figure 4 in the paper):

  1. Partition the tensor into groups of 128 FP16 values.
  2. Two-level normalization: (a) find the absmax per group, normalize it to FP8 using a per-tensor scale factor → this FP8 value is the group scale factor; (b) normalize the remaining 127 values to (−1, 1) using that FP8 scale factor.
  3. Run k-means (15 clusters) on the 127 normalized values in each group. The sorted centroids form a k-means pattern.
  4. Cluster the k-means patterns across all groups into 𝑆=64 shared k-means patterns.
  5. For each group, try rounding its values to each of the 64 shared patterns. Pick the one with minimum MSE. Record its index (6 bits for 64 patterns).
  6. For each shared k-means pattern, cluster the frequency distributions of indices (after quantization) into 𝐻=4 groups. Each group's frequency → one Huffman codebook.
  7. Build 4 Huffman codebooks (16 codes each, 2–8 bits) per shared pattern.
  8. Each group chooses the Huffman codebook (out of 4) that gives the best compression ratio. Record its index (2 bits).
  9. Huffman-encode the 128 indices. If output exceeds 512 bits (64 bytes − 8-bit scale factor − 6-bit pattern index − 2-bit codebook index = 496 bits), clip the excess. If under, pad with the next-largest 15 values (each 15 bits: 7-bit location + 8-bit FP8 value).

This yields a 64-byte compressed block: 8 bits (scale factor) + 6 bits (k-means pattern) + 2 bits (Huffman codebook) + 240–496 bits (Huffman-coded data + padded outliers). The padding ensures no extra memory trips — everything decompresses from one load.

💡
Why k-means + Huffman? k-means adapts to the data distribution (non-uniform), increasing entropy. Huffman codes frequent indices with fewer bits, reducing overhead. Together, they maximize 𝜂.

4 · KV cache compression — fast pattern selection

KV cache compression follows the same pipeline, but step 5 (selecting the best k-means pattern) must run at inference time, so it can't afford full MSE. The paper's insight: the k-means patterns are highly skewed (Figure 7 shows them clustered near ±1). So instead of computing MSE for all 64 patterns (128 subtractions × 64 = 8,192 ops), Ecco compares only the min and max of each group (excluding absmax) against each pattern's min and max. Pick the pattern with the smallest Σ(min−pattern_min)² + Σ(max−pattern_max)². This is just 2 comparisons per pattern, drastically cutting hardware complexity.

The perplexity drop from this approximation is negligible (Table 1: LLaMA2-13B at 5.03 vs. 5.22 for QuaRot's MSE-based method), and the hardware is orders of magnitude simpler.

5 · Activation compression — simple uniform

Activations are consumed immediately by the next layer, so Ecco targets 2× compression (8 bits/value) with a simple scheme: 7-bit uniform quantization + 1-bit zero-point flag. Every 8 bits, one bit is reserved for metadata (16-bit zero point and 16-bit scale factor spread across the block). This keeps the decompressor simple and fast, as activations don't need the full k-means+Huffman machinery.

6 · Decompressor design — 28-cycle pipeline

The decompressor (Figure 8 in the paper) has four main stages:

  1. Pattern retrieval: read the per-group scale factor (8 bits), k-means pattern index (6 bits), and Huffman codebook index (2 bits). Fetch the corresponding k-means pattern and Huffman codebook from the tensor-wide metadata buffer. Convert the FP8 scale factor back to FP16 by adjusting the exponent (the FP16-to-FP8 scale is power-of-two, so this is just an add).
  2. Parallel Huffman decoding: 64 sub-decoders each process 8 bits + 7-bit overlap. Each sub-decoder has 8 internal decoders starting at positions 0–7 within its 15-bit window, decoding 1–4 indices per segment. Output is 8 possible decoding results (D_Out) plus an EOP (end-of-packet) signal indicating where the segment ends.
  3. Tree concatenation: 6 stages of merges. Each merge takes the EOP from the left decoder's output to select which D_Out from the right decoder should be concatenated. After 6 stages, one final output remains. A mask is also generated to indicate how many outliers were padded.
  4. Data mapping: 128 parallel mappers translate the decoded indices (0–15) to their real centroids from the k-means pattern. Outliers are scaled to FP16 (multiply FP8 value × scale factor). The outlier address and mask act as selectors, choosing between the k-means centroid and the scaled outlier for each position.

This pipeline is fully pipelined at 28 cycles latency. The critical path is pattern retrieval → Huffman decode → merge → map, with buffers inserted to maintain throughput.

7 · Compressor design — 62-cycle latency

The compressor (Figure 9) runs off the critical path (between L2 and HBM), so its 62-cycle latency is acceptable. It has three main stages:

  1. Bitonic sort: extract the absmax (scale factor), the top 16 sorted values and their indices (for padding), and the group's min/max (for pattern selection).
  2. Pattern selector: compute Σ(min−pattern_min)² + Σ(max−pattern_max)² for each of the 𝑆=16 shared patterns (reduced from 64 for the compressor to save area). Pick the pattern with minimum error. Record its index.
  3. Parallel encoding: 4 encoders (one per Huffman codebook) run in parallel. Each has 128 value mappers that compute the index for each input value by finding the nearest k-means centroid. The indices are Huffman-encoded and concatenated. The encoder with the shortest output is selected, and its index recorded. The final bitstream is clipped or padded to 64 bytes.

The compressor is replicated 20 times to match the L2 cache's 5120 bytes/cycle peak throughput (20 × 256 bytes = 5120 bytes).

🍯
The hardware punchline: 20 decompressors + 20 compressors occupy 5.11 mm² (0.62% of A100's 826 mm²) and draw 7.36 W (9% of 82 W idle). They sit between L2 and HBM, adding 28 cycles latency per block — negligible vs. the ~400-cycle HBM trip. The parallel Huffman decoder is the key innovation, breaking the sequential bottleneck.

Worked examples

Four traces, each carried all the way through — computing bit efficiency, selecting a k-means pattern, decoding Huffman in parallel, and tallying the hardware cost.

① Computing bit efficiency for group-wise quantization

A group of 128 values is quantized using 4-bit codes (16 levels). After quantization, the frequency of each code is: [10, 15, 8, 12, 20, 5, 7, 9, 11, 6, 8, 4, 5, 3, 3, 2]. What is the entropy and bit efficiency?

  1. Total count: Σ frequencies = 10+15+8+12+20+5+7+9+11+6+8+4+5+3+3+2 = 128.
  2. Probabilities: 𝑝ᵢ = frequency / 128. E.g., 𝑝₁ = 10/128 = 0.078, 𝑝₂ = 15/128 = 0.117, etc.
  3. Entropy: 𝐻 = −Σ 𝑝ᵢ log₂ 𝑝ᵢ. Compute each term: −0.078 log₂(0.078) ≈ 0.283, −0.117 log₂(0.117) ≈ 0.359, …, sum = 3.72 bits.
  4. Real bit overhead: each value uses 4 bits (fixed code length), so 𝐵ᵣₑₐₗ = 4 bits/value.
  5. Bit efficiency: 𝜂 = 𝐻 / 𝐵ᵣₑₐₗ = 3.72 / 4 = 93.0%. Nearly all bits carry information — excellent efficiency.
🍯
Insight: high entropy (3.72 vs. theoretical max 4.0) means the codes are used nearly uniformly. Ecco's k-means + Huffman pushes this even higher by tailoring centroids to the distribution.

② Selecting the best k-means pattern at runtime

A KV cache group has min = −0.6, max = 0.8 (after scaling). Two shared k-means patterns have (min, max) = (−0.5, 0.7) and (−0.7, 0.9). Which pattern does Ecco choose?

  1. Pattern 1: error = (−0.6 − (−0.5))² + (0.8 − 0.7)² = (−0.1)² + (0.1)² = 0.01 + 0.01 = 0.02.
  2. Pattern 2: error = (−0.6 − (−0.7))² + (0.8 − 0.9)² = (0.1)² + (−0.1)² = 0.01 + 0.01 = 0.02.
  3. Tie: both patterns have the same error. Ecco picks the first one (pattern 1) by default.
  4. Why min/max? Full MSE would require computing Σ(value − centroid)² for all 127 values × 64 patterns = 8,128 ops. Min/max is just 2 comparisons × 64 patterns = 128 ops — 63× cheaper.
💡
The paper shows (Figure 7) that k-means patterns are highly skewed, clustered near ±1. Min/max captures enough information to distinguish them without full MSE. Perplexity drop from this approximation: LLaMA2-13B 5.03 vs. 5.22 (QuaRot's MSE) — negligible.

③ Parallel Huffman decoding in 6 stages

A 64-byte block contains 496 bits of Huffman-coded data. The first 64 bits (8 bytes) are processed by decoders #0–#7. Each decoder outputs 1–4 indices. How do the 6 merge stages combine them?

  1. Stage 1: 64 decoders process 64 × 8-bit chunks in parallel. Decoder #0 outputs [3, 7, 1] (3 indices), EOP=3. Decoder #1 outputs [12, 5] (2 indices), EOP=2. Etc.
  2. Stage 2: merge #0 and #1. Decoder #0's EOP=3 says "I decoded 3 symbols, next segment starts at bit 24." Concatenate #0's output [3,7,1] with #1's output [12,5] → [3,7,1,12,5].
  3. Stage 3: merge [3,7,1,12,5] with #2's output [8,9,11,14], and merge #3+#4, etc. Now 32 groups.
  4. Stages 4–6: repeat merging. Stage 4: 32→16. Stage 5: 16→8. Stage 6: 8→4. Stage 7: 4→2. Stage 8: 2→1.
  5. Final output: one sequence of 128 indices, decoded from the 496-bit Huffman stream. Latency: 6 stages × ~4 cycles/stage + initial decode = 28 cycles.
🍯
Insight: the tree structure (64 → 32 → 16 → … → 1) is what makes parallel Huffman practical. Sequential Huffman would decode one symbol at a time, taking ~200 cycles for 128 symbols. Parallel: 28 cycles — 7× faster.

④ Tallying the hardware cost

Ecco adds compressors and decompressors to a GPU. On a 28nm process scaled to 7nm, the 4× decompressor is 3.19 mm², the 2× decompressor is 0.57 mm², the 4× compressor is 0.91 mm², and the 2× compressor is 0.44 mm². Each is replicated 20× to match L2 throughput. What's the total area and % of A100?

  1. Total area: (3.19 + 0.57 + 0.91 + 0.44) × 20 = 5.11 × 20 = 102.2 mm². Wait — the paper says 5.11 mm² total, not per unit. Re-read: each type has one instance, replicated 20×. So total = 20 × (3.19/20 + 0.57/20 + 0.91/20 + 0.44/20) = 20 × 0.2555 = 5.11 mm².
  2. A100 die area: 826 mm². Fraction = 5.11 / 826 = 0.619% ≈ 0.62%.
  3. Power: 4.82 W (4× decomp) + 0.83 W (2× decomp) + 1.15 W (4× comp) + 0.56 W (2× comp) = 7.36 W. A100 idle power = 82 W. Fraction = 7.36 / 82 = 8.98% ≈ 9%.
💡
The paper's claim: <1% area, <10% idle power. Our calculation confirms this. The minimal overhead is what makes Ecco practical — you're adding a "paperback book" to a "backpack," not a second backpack.

Problem 1 · easy
A group of 128 FP16 values is compressed to 64 bytes using Ecco's 4× scheme. After Huffman coding, the bitstream is 450 bits. Should Ecco clip or pad, and by how much?
Show answer
The 64-byte block has 512 bits total. Subtract 8 bits (scale factor) + 6 bits (k-means pattern) + 2 bits (Huffman codebook) = 16 bits metadata, leaving 496 bits for data. The Huffman output is 450 bits, so Ecco pads 496 − 450 = 46 bits. At 15 bits/outlier (7-bit location + 8-bit FP8 value), that's 46/15 ≈ 3 outliers padded.Misconception: that Ecco always clips. The paper clips only when Huffman output >496 bits (<0.04% of groups); otherwise it pads with the largest remaining values.
Problem 2 · medium
A 64-byte compressed block arrives at the decompressor. It contains a 6-bit k-means pattern index = 42 and a 2-bit Huffman codebook index = 3. How does the decompressor fetch the right k-means pattern and codebook?
Show answer
The decompressor has a buffer loaded with tensor-wide metadata (64 k-means patterns, each with 4 Huffman codebooks) at the start of the layer. It uses the 6-bit index (42) to select pattern #42 from the buffer, and the 2-bit index (3) to select codebook #3 within that pattern. These are then used in stages 2–4 (Huffman decode → merge → map). The buffer is small: 64 patterns × 15 centroids/pattern × 16 bits/centroid = 15,360 bits = 1.92 kB.Misconception: that the pattern and codebook are stored in each block. They're shared across all blocks in the tensor, loaded once, and indexed per block — this is what keeps the per-block overhead to just 8 bits (pattern + codebook indices).
Problem 3 · medium
Ecco's parallel Huffman decoder has 64 sub-decoders. Each processes 8 bits + 7-bit overlap. Why the overlap? What happens if you remove it?
Show answer
Huffman codes are variable-length (2–8 bits). If a decoder processes only 8 bits, it might stop mid-code — the last code could be incomplete. The 7-bit overlap ensures the decoder sees enough bits to decode at least one complete code (worst case: 8-bit code starting at bit 7). Without the overlap, the decoder wouldn't know where the next segment starts (the EOP signal would be wrong), and the tree merge would fail.Misconception: that overlap is for error correction. It's for alignment — knowing where each segment's boundary is so the tree can merge correctly.
Problem 4 · hard
The paper says Ecco achieves 2.9× speedup over AWQ on LLaMA-13B (batch=8, seq=2K). AWQ uses 4-bit weights, 16-bit activations/KV. Ecco uses FP16 everywhere but compresses the cache 4×. Why is Ecco faster?
Show answer
AWQ's 4-bit weights require dequantization on every GEMM: load 4-bit values, multiply by scale factors, add zero points. These extra operations sit on the critical compute path, adding latency. Ecco's compression happens off the critical path (between L2 and HBM): data is compressed when written to HBM and decompressed when loaded to L2. The 28-cycle decompressor latency is hidden by the ~400-cycle HBM trip. Plus, 4× compression means 4× less memory bandwidth and 4× more effective L2 capacity, reducing evictions and speeding up the whole pipeline. AWQ saves bandwidth but pays compute overhead; Ecco saves bandwidth without that penalty.Misconception: that quantization always speeds things up. AWQ's quantization slows down vs. FP16 at small batch sizes (Figure 11a: AWQ at 0.70× normalized latency vs. 1.0× for FP16). Ecco's cache compression is orthogonal — it works on top of FP16 compute, so no accuracy loss and no dequant overhead.
Problem 5 · medium
Ecco uses 𝑆=64 shared k-means patterns and 𝐻=4 Huffman codebooks per pattern. Why these numbers? What happens if you halve 𝑆 to 32 or double 𝐻 to 8?
Show answer
Figure 5 in the paper shows a design space exploration on LLaMA2-7B. Perplexity improves as 𝑆 increases (more patterns = better fit to diverse groups), but plateaus around 𝑆=64 — beyond that, gains are minimal. Similarly, 𝐻=4 is a sweet spot: higher 𝐻 offers little perplexity improvement but increases metadata cost (more codebooks to store). Halving 𝑆 to 32 would hurt perplexity slightly (5.58 → 5.60, Figure 5); doubling 𝐻 to 8 barely helps (5.56 vs. 5.56). The 64/4 choice balances accuracy and overhead.Misconception: that more is always better. Beyond 𝑆=64 and 𝐻=4, you hit diminishing returns — the extra patterns/codebooks don't fit the data better, they just cost more to store.
Problem 6 · hard
The paper claims nearly 4× memory capacity increase. On LLaMA-7B (batch=32, seq=2K), FP16 uses 47.3 GB total, of which 34.4 GB is KV cache. If Ecco compresses KV at 4× and weights at 4×, what's the total memory after compression?
Show answer
FP16 breakdown: 34.4 GB KV + (47.3 − 34.4) = 12.9 GB weights/activations. Ecco compresses KV 4×: 34.4 / 4 = 8.6 GB. Weights (7B params × 2 bytes = 14 GB before) compress 4×: 14 / 4 = 3.5 GB. Activations (12.9 − 14 ≈ −1.1 GB, so let's say 1.1 GB) compress 2×: 1.1 / 2 = 0.55 GB. Total = 8.6 + 3.5 + 0.55 = 12.65 GB. Compression ratio = 47.3 / 12.65 ≈ 3.74× ≈ 4×.Misconception: that 4× compression everywhere gives 4× total. The weights are smaller than KV (14 GB vs. 34.4 GB), and activations are only 2×, so the weighted average is ~3.74×, not 4×. The paper's "nearly 4×" is accurate.
Problem 7 · easy
Ecco's decompressor adds 28 cycles latency. A typical HBM trip is ~400 cycles. Is the decompressor latency on the critical path?
Show answer
No. The decompressor sits between the L2 cache and the streaming multiprocessors (SMs). Data arrives from HBM to L2 (400 cycles), then decompresses (28 cycles) before reaching the SMs. The HBM trip dominates (400 ≫ 28), so the 28-cycle latency is mostly hidden. Even if you count them serially, 400 + 28 = 428 cycles vs. 400 cycles FP16 — a 7% increase, not a 2× slowdown. The real win is bandwidth: 4× less data moved from HBM, so 4× fewer HBM trips.Misconception: that any added latency slows things down proportionally. Latency matters only if it's on the critical path. Ecco's decompressor is off the critical path (compute happens in FP16 at the SMs, unaffected), so its 28 cycles don't bottleneck inference.
Problem 8 · medium
The paper says Ecco achieves SOTA perplexity (LLaMA2-13B: 5.03 vs. 4.88 FP16). Why isn't it exactly 4.88 if Ecco computes in FP16?
Show answer
Ecco compresses the KV cache at 4×, which is lossy (k-means quantization + Huffman coding). When decompressed, the KV cache values are slightly different from the original FP16 values (clipped outliers, quantization error). This affects the attention computation (Q K^T / √d_k), propagating small errors to the final output. The perplexity increase (4.88 → 5.03) is the cost of lossy compression. Weights and activations are also compressed but decompressed exactly (the clipping rate is <0.04%), so the main source of error is KV cache compression.Misconception: that cache compression is lossless. Ecco is near-lossless (high bit efficiency, padding preserves top values), but the clipping and k-means approximation do introduce small errors. The 5.03 perplexity shows these errors are negligible for practical use.
Quick answers

Frequently asked questions

Twelve questions newcomers ask most. Tap any to expand.

Q1What's the difference between Ecco and quantization methods like AWQ?▾
AWQ quantizes the model (weights to 4 bits, activations to 16 bits) and runs dequantization on the critical compute path during inference. Ecco compresses the cache (L2/HBM), storing data in compressed form and decompressing it off the critical path. AWQ trades accuracy for size; Ecco trades a bit of capacity (4× vs. 16×) for zero compute overhead and near-zero accuracy loss. They can even be combined: run AWQ on the model, then Ecco on the cache.
Q2Why does Ecco achieve higher perplexity (5.03) than QuaRot (5.06) if both compress 4 bits?▾
QuaRot uses rotation + Hadamard transforms to suppress outliers, enabling uniform 4-bit quantization. These transforms add compute overhead (Figure 3: 0.6× slower than FP16). Ecco uses non-uniform k-means + Huffman, which better fits the data distribution (higher bit efficiency: 78.54% vs. QuaRot's ~64%) without runtime transforms. The result: Ecco's 5.03 perplexity is slightly better than QuaRot's 5.06, and Ecco is 2.9× faster than FP16 (vs. QuaRot's 0.6×).
Q3How can Ecco compress 4× without losing accuracy?▾
It's near-lossless, not lossless. k-means quantization clusters values around 15 centroids, losing some precision. Huffman coding is lossless, but clipping (when output exceeds 64 bytes) discards <0.04% of values. Padding (when under 64 bytes) preserves the top 16 values as FP8 outliers. The net effect: perplexity increases from 4.88 (FP16) to 5.03 (Ecco) — a 3% increase, well within acceptable bounds for LLM inference. The key is high bit efficiency (78.54%) — every bit carries information, so you lose less when compressing.
Q4Why does the paper use power-of-two compression ratios (2× and 4×)?▾
Power-of-two ratios simplify address translation and memory alignment. If a 256-byte block compresses to 64 bytes (4×), the GPU can map compressed addresses to physical addresses with a simple shift (addr >> 2). Non-power-of-two ratios (e.g., 3×) would require division/modulo, adding complexity to the memory controller. Plus, 64 bytes is the DRAM-to-L2 transaction size on A100, so compressing to 64 bytes means one load per block — no partial transactions.
Q5What happens if a group's Huffman output exceeds 496 bits?▾
Ecco clips the excess. The last few indices (those beyond bit 496) are discarded. The paper reports this happens in <0.04% of groups (Figure 10: clipping ratio ≈ 0.04% for projection layers). Since the clipped values are the least significant (Huffman assigns longer codes to rare values, which appear late in the bitstream), the impact on perplexity is negligible. The alternative — using a variable-length block — would require extra metadata to track block boundaries, negating the benefit of fixed 64-byte blocks.
Q6How does Ecco handle non-compressible data?▾
If a group's entropy is very high (all 128 values are unique, no clustering), k-means will still pick 15 centroids, but the quantization error will be large. Huffman coding won't help much (all 16 codes used equally → ~4 bits/code, no savings). In this case, the compressed block is nearly as large as the original. But the paper shows (Figure 10) that padding ratio is ~0.7% for projection layers, meaning most groups do compress well. For truly non-compressible data, Ecco could fall back to uncompressed storage (a flag bit in the page table), but the paper doesn't mention this — suggesting it's rare in LLM workloads.
Q7Why does Ecco use 𝑆=64 patterns for weights but 𝑆=16 for the compressor?▾
The decompressor (which handles both weights and KV cache) uses 𝑆=64 to maximize accuracy — more patterns = better fit. The compressor (which only runs on KV cache at inference time) uses 𝑆=16 to reduce area: fewer patterns = smaller pattern selector. The paper found that reducing from 64 to 16 patterns causes negligible perplexity drop (the compressor's min/max selection is already approximate, so losing some patterns doesn't hurt much). This asymmetry (64 in decoder, 16 in encoder) is a clever area-saving trick.
Q8How does Ecco compare to lossless compression (e.g., NVIDIA's CDC)?▾
NVIDIA's Compute Data Compression (CDC) on A100 uses lossless methods like BDI, achieving ~1.2× compression. Ecco achieves 4× (lossy) by exploiting the fact that LLMs tolerate small errors. Lossless compression is safer (zero accuracy loss) but limited in ratio; lossy compression is riskier (perplexity increase) but offers much higher ratios. The paper shows Ecco's perplexity drop (4.88 → 5.03) is acceptable, so the 4× ratio is worth it. In practice, you'd use both: CDC for exact data (e.g., indices), Ecco for approximate data (KV cache, weights).
Q9Can Ecco be applied to other models (CNNs, Transformers for vision)?▾
Yes, but with caveats. Ecco's k-means + Huffman scheme is general-purpose — any data with clustered distributions will compress well. CNNs have lower entropy in weights (more redundancy) and might compress even better than LLMs. Vision Transformers (ViTs) have similar KV cache behavior to language models, so Ecco should work. However, the paper's 𝑆=64, 𝐻=4 hyperparameters are tuned for LLMs; other domains might need different values. The hardware (parallel Huffman decoder) is domain-agnostic and should transfer directly.
Q10Why doesn't Ecco compress the prefill phase?▾
Prefill is compute-bound, not memory-bound. The input sequence is processed in parallel (batch × seq × hidden GEMM), keeping the GPU's compute units busy. Compression would save bandwidth but wouldn't speed up prefill because the bottleneck is compute, not memory. The decode phase, by contrast, is memory-bound (batch × 1 × hidden GEMV, low arithmetic intensity), so compression directly translates to speedup. The paper focuses on decode because that's where LLMs spend 90%+ of inference time (512 decode steps vs. 1 prefill in Figure 3a).
Q11What's the tradeoff between Ecco's 28-cycle decompressor and a slower, higher-ratio compressor?▾
Ecco targets GPU L2 throughput (5120 bytes/cycle on A100). A 28-cycle decompressor with 20 instances matches this (20 × 256 bytes / 28 cycles ≈ 183 bytes/cycle per instance, × 20 ≈ 3660 bytes/cycle — close enough). A slower decompressor (e.g., 100 cycles for 8× compression) would bottleneck the L2, making the GPU wait for decompressed data. The sweet spot is balancing ratio (4×) and latency (28 cycles) to match the memory hierarchy's native throughput. Going beyond 4× would require a more complex decoder (longer Huffman codes, more merge stages), increasing latency and area — diminishing returns.
Q12What did the authors leave for future work?▾
The paper mentions: (1) extending to multi-GPU setups (current design is single-GPU; multi-GPU would need inter-GPU compression); (2) adaptive compression ratios per layer (some layers might compress better at 2× vs. 4×); (3) combining with model quantization (Ecco + AWQ for even higher compression); (4) exploring non-power-of-two ratios (e.g., 3×) with advanced address translation; (5) applying to other domains (CNNs, ViTs). The big open question: can you push beyond 4× without hitting accuracy cliffs? The paper's design space exploration (Figure 5) suggests 4× is near the limit for LLMs, but other models might differ.

Glossary & further reading

Every term in one place, then a path to go deeper.

Core terms
EccoEntropy-based Cache Compression. The paper's method: group-wise k-means + Huffman coding for 4× (weights/KV) and 2× (activations) compression.
Bit efficiency (𝜂)Entropy 𝐻 divided by real bit overhead 𝐵ᵣₑₐₗ. Ecco achieves 78.54% (3.15 bits entropy / 4.01 bits overhead).
Group-wise quantizationPartition tensor into 128-element groups, quantize each independently. Balances fine granularity (high entropy) with metadata overhead.
k-means quantizationNon-uniform quantization: cluster values into 𝑘=15 centroids, assign each value to nearest centroid. Better fit than uniform grids.
Shared k-means patterns𝑆=64 pre-computed patterns shared across all groups. Each group picks the best pattern, storing only a 6-bit index.
Huffman codingVariable-length encoding: frequent symbols → short codes (2 bits), rare symbols → long codes (8 bits). Optimal for a given distribution.
Huffman codebooks𝐻=4 codebooks per k-means pattern, tailored to frequency distributions. Each group picks the best codebook (2-bit index).
Parallel Huffman decoder64 sub-decoders process overlapping 8-bit segments in parallel, merge via 6-stage tree. Latency: 28 cycles (vs. 200 sequential).
Padding & clippingIf Huffman output <496 bits, pad with FP8 outliers (top 16 values, 15 bits each). If >496 bits, clip excess (<0.04% rate).
Fixed block size64 bytes = one DRAM-to-L2 transaction. All metadata (scale, pattern, codes, outliers) fits in one block — no extra loads.
Supporting terms
KV cacheCached keys and values from prior tokens in Transformer attention. Grows linearly with sequence length; 72.7% of memory in LLaMA-7B (batch=32, seq=2K).
Memory-boundInference bottlenecked by memory bandwidth, not compute. LLM decoding is memory-bound due to small-batch GEMVs and large KV cache.
GEMV vs. GEMMGeneral Matrix-Vector Multiply (one token at a time) vs. General Matrix-Matrix Multiply (batch of tokens). GEMV has low arithmetic intensity.
L2 cacheLast-level cache on GPU, between L1 and HBM. Ecco's decompressor sits here, decompressing data before it reaches SMs.
HBMHigh-Bandwidth Memory. DRAM used in GPUs (e.g., A100 has 40 GB HBM at ~2 TB/s). Ecco compresses data before writing to HBM.
FP8 scale factorPer-group absolute maximum value, quantized to FP8 using a per-tensor FP16-to-FP8 scale. Decompression: adjust exponent (power-of-two scale).
Bitonic sortParallel sorting algorithm used in the compressor to extract absmax, top 16 values, and min/max in one pass.
EOP (end-of-packet)Signal in parallel Huffman decoder indicating where a segment's last code ends. Used by tree concatenators to align merges.
PerplexityMeasure of LM quality: exp(−Σ log 𝑝(word)). Lower is better. FP16: 4.88, Ecco: 5.03 on LLaMA2-13B WikiText-2.
SpeedupNormalized latency. Ecco achieves 2.9× speedup over AWQ (0.34 vs. 0.70 normalized latency on LLaMA-13B, batch=8, seq=2K).

Related papers

AWQ: Activation-aware Weight Quantization hard

Lin et al. · MLSys 2024

The SOTA quantization baseline Ecco improves upon. AWQ uses per-channel scales to protect salient weights, achieving 4-bit compression with minimal perplexity loss. Ecco beats it by compressing the cache instead of the model.

QuaRot: Outlier-Free 4-Bit Inference in Rotated LLMs hard

Ashkboos et al. · arXiv 2024

Uses Hadamard rotation to suppress outliers, enabling uniform 4-bit quantization. Achieves SOTA perplexity but is slower than FP16 at batch ≤64 due to runtime transforms. Ecco avoids this by using k-means (non-uniform) without transforms.

Base-Delta-Immediate (BDI) Compression medium

Pekhimenko et al. · PACT 2012

Lossless cache compression: represent cache lines as base + small deltas. Used in NVIDIA GPUs but achieves only ~1.2× ratio. Ecco's lossy k-means+Huffman gets 4× by tolerating small errors.

SmoothQuant: Accurate Post-Training Quantization hard

Xiao et al. · ICML 2023

8-bit weight + 8-bit activation quantization via per-channel smoothing. Ecco outperforms it (1.8× speedup) because SmoothQuant still requires dequantization on the critical path.

Buddy Compression for GPUs hard

Choukse et al. · MICRO 2019

Splits compressed memory between GPU and host CPU. Ecco is simpler (all-on-GPU) but shares the insight that compressibility can be exploited to increase effective capacity.
