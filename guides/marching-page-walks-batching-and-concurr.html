HPCA 2025 · A Field Guide

One page walker, walking many tables at once

Thousands of GPU threads miss the TLB in bursts, but each page-table walker serves one walk at a time. The queue explodes. This is the plain-English story of Marching Page Walks — a small change that lets each walker batch and fire many walks together.

📄 "Marching Page Walks" · Lee, Ko, Yoon, Jeong, Oh, Ro (Yonsei/Ewha/Korea Univ.) 🎓 GPU + virtual-memory basics assumed — the primers cover them ⏱ ~35 min read
🚶‍♂️🚶‍♀️🚶

Let each walker BATCH pending walks to the same page-table level and issue all their loads at once — instead of draining the queue one walk at a time.
Widen the throughput of the walkers you already have, don't add more.

Before you begin

Ten ideas the paper assumes you know

Each card clears up one concept MPW builds on. They're collapsed — tap any open. Read in order the first time, or jump to whatever looks unfamiliar. If you already live in the Linux memory stack, most of this maps directly onto what you know from the CPU side.

1Virtual memory & the page table walk▾

Programs use virtual addresses; hardware must translate each to a physical address before touching memory. The map lives in a page table. On a translation miss, the hardware performs a page table walk (PTW): it chases pointers down the table until it finds the leaf entry that holds the physical frame.

This is exactly the mechanism you know from x86-64 and the Linux MM — just running on a GPU, under vastly more thread pressure.

Now you're ready: the whole paper is about making page table walks faster on GPUs — not by caching more, but by servicing many walks in parallel.
2A multi-level (radix) page table: L4→L3→L2→L1▾

The paper assumes a 4-level radix tree, naming the levels L4 (root), L3, L2, and L1 (leaf). Each page-table page holds 512 entries of 8 bytes = 4 KB. A walk starts at L4 and issues one memory access per level, following pointers down until the L1 leaf gives the final 4 KB-page mapping.

(Note the naming is inverted from x86's PML4→…→PTE convention: here L1 is the leaf, L4 is the root. Same tree, opposite labels.)

Now you're ready: "walking L1" means fetching the leaf PTE — and the paper finds that ~89–92% of all walk work is exactly these L1 leaf accesses.
3TLBs and page walk caches (PWCs)▾

A TLB caches finished translations so most accesses skip the walk entirely. GPUs use a two-level TLB: a private L1 TLB per streaming multiprocessor (SM) and a shared L2 TLB. When the last-level TLB misses, a walk is dispatched.

To speed the walk itself, GPUs add page walk caches (PWCs) that store partial translations from the upper levels (L4, L3, L2). In this paper's setup the upper-level PWCs almost always hit, so upper levels are cheap — the cost is dominated by L1 leaf walks that PWCs cannot cover.

Now you're ready: MPW leaves TLBs and PWCs untouched; it only changes the walkers that handle the leftover L1-heavy walk traffic.
4Why GPUs pressure the walkers so hard▾

A GPU runs thousands of threads concurrently. When they miss the TLB — often in a burst — they generate a flood of walk requests. But each hardware walker is a finite-state machine that handles exactly one walk at a time. The mismatch is the entire problem.

The paper measures it: contention raises average PTW latency 3.0× (almost all of it queueing delay) and drags GPU performance down by 1.6× versus a walker-unconstrained ideal.

Now you're ready: the bottleneck isn't the walk itself — it's the waiting in line for a free walker. That queue is what MPW drains.
5The page walk subsystem: PW queue + walkers + PWCs▾

Requests land in a page walk queue (PW queue); a pool of walkers pull from it. The baseline here uses 32 concurrent walkers and a 256-entry PW queue, with requests distributed round-robin into per-walker queues.

To track outstanding memory requests, the subsystem needs a CAM (content-addressable memory) that it searches associatively — and CAMs get expensive fast as you add walkers or ports.

Now you're ready: MPW's changes live entirely inside this subsystem — it extends the PW queue, adds a small status table, and adds a bloom filter per walker.
6Why not just add more walkers?▾

The obvious fix. The paper shows peak performance needs about 192 walkers (6× the baseline's 32) — and up to 512 on a big V100-class chip. But every walker needs associative CAM lookups when it starts and finishes a walk.

Scaling to 192 walkers demands an 18-port CAM; going 4→18 ports blows CAM area up 9.1× (≈3.9% of a whole TU106 die). Adding walkers trades the queueing problem for an area/energy problem.

Now you're ready: MPW's goal is to get 192-walkers-worth of throughput out of just 32 walkers, without the CAM explosion.
7The prior art: neighborhood-aware coalescing▾

An earlier scheme, neighborhood-aware translation, coalesces walk requests that fall within a single memory access. A 32 B cache sector holds four 8-byte PTEs, so it can merge up to 4 walks — one memory read serves all four.

That helps (≈1.52× walker throughput, +16.6% performance) but is capped by the sector: it can only merge walks whose PTEs are physically adjacent. GPU threads scatter far wider than that.

Now you're ready: MPW's key move is batching beyond a single memory access — merging walks to any page-table page at the same level, not just adjacent ones.
8Batch size = concurrency = throughput▾

If a walker can issue several independent memory loads at once (non-blocking) instead of one, it services several walks per "cycle" of work. The batch size — how many walks it groups together — directly sets the effective walker throughput.

The paper models a "batched PTW" walker that handles up to 8 requests at a time, and shows the achievable batch size grows as you widen the scope of what may be batched together.

Now you're ready: the question the paper answers is which walks should be allowed into a batch — and the answer is "any at the same level," which needs the widest scope.
9A CAM, and why associative search is costly▾

A CAM answers "which entry holds this value?" by comparing against every entry in parallel. It's fast but power-hungry, and both area and energy grow super-linearly with size and port count.

MPW's naïve version would need CAM searches over a big monolithic PW queue every time it batches a walk, and over every walker's records every time a PTE returns. That energy cost is what the bloom filters and partitioned queues exist to tame.

Now you're ready: half of MPW's design is about shrinking those associative searches, not just enabling batching.
10Bloom filters — cheap "definitely-not" tests▾

A bloom filter is a tiny hash-indexed structure that answers "have I seen this?" It can produce false positives (says maybe-yes when the answer is no) but never false negatives. A counter-based bloom filter also supports removals by incrementing/decrementing counters.

MPW puts one 128-entry, 3-bit-counter bloom filter per walker so that when a PTE returns, it can skip the CAM search on walkers that definitely didn't request it — cutting associative searches from 32 down to ~1.84.

Now you're ready: the bloom filter is the energy-saving twist that makes wide batching affordable in hardware.
The whole story in one place

MPW in a nutshell

The bottleneck, the batching insight that fixes it, and how the pieces connect.

GPUs run thousands of threads that miss the TLB in bursts, but each page-table walker is a state machine that services one walk at a time. Requests pile up in the page walk queue. The paper measures the damage: contention inflates page-walk latency by 3.0× — almost entirely queueing delay — and cuts GPU performance by 1.6×. You could reach peak throughput by scaling to ~192 walkers, but the associative CAM hardware that tracks in-flight walks would balloon (an 18-port CAM, ≈9.1× the area), so brute force is out.

The fix, Marching Page Walks (MPW), keeps 32 walkers but lets each one batch multiple pending walks and fire all their memory loads concurrently. The critical design choice is the batching scope: the prior scheme (neighborhood-aware) could only merge walks inside one 32 B memory access — at most 4 adjacent PTEs. MPW instead batches any walks heading to the same page-table level, regardless of how far apart their pages are, because GPU workloads scatter accesses across tens to hundreds of MB. Widening the scope from "single page-table page" to "any page-table page" lifts walker throughput from 3.10× to 3.90×.

To keep the extra associative searches cheap, MPW partitions the monolithic PW queue (one small split queue per walker) so batching scans only a few entries, and adds a counter-based bloom filter per walker so returning PTEs skip CAM searches on walkers that didn't request them — dropping average searches from 32 to 1.84. A small walker status table (WST) tracks which walks each walker is handling.

The payoff: MPW reduces the queueing latency of page walks by 86.7% and improves GPU performance by 55.6% over the baseline — landing within 1.0% of an idealized GPU with infinite walkers — for just 3.02 KB of extra storage, 0.11% of a TU106 die, and 0.004% of its power budget.

🍯 Key takeaways

  • The bottleneck is queueing for a free walker, not the walk itself: contention → 3.0× PTW latency, 1.6× slowdown.
  • Adding walkers works (peak ≈ 192) but needs a huge multi-ported CAM (18 ports, ~9.1× area) — impractical.
  • MPW keeps 32 walkers but lets each batch up to 8 walks and issue their loads concurrently.
  • Batching scope is everything: any page-table page at the same level beats single/adjacent scopes (3.90× vs 3.10×/3.37×).
  • Wins come from L1 leaf walks (~89–92% of all walks); upper levels are covered by PWCs.
  • Partitioned PW queues + per-walker bloom filters shrink associative searches (32 → 1.84) for energy efficiency.
  • Result: −86.7% queueing latency, +55.6% performance, within 1.0% of infinite-walker ideal, for 3.02 KB / 0.11% area.
🔌
Why this should interest a tiering / CXL person: MPW is a queueing-theory fix — it attacks the service rate of the translation path, not its cache hit rate. That framing transfers directly to CPU-side page-walk pressure under CXL-attached and tiered memory, where fragmentation and huge working sets similarly flood the walker. See the tiering callouts throughout "How It Works."
Seven everyday pictures

How to think about MPW

Each picture stands alone — read them in any order. Every card ends with an honest note about where the picture stops being true.

The core idea · Batching

A ferry, not a rowboat

A crowd waits to cross a river. A rowboat carries one person per trip — everyone else queues on the bank (that's the baseline walker draining its queue one walk at a time). A ferry loads many passengers and crosses once. MPW turns each walker into a ferry: it scans the queue, gathers every walk headed the same way (same page-table level), and issues all their memory loads in one crossing.

Where it misleads: a ferry departs on a schedule; an MPW walker batches opportunistically — only whatever same-level walks happen to be waiting when it becomes free.
The key move · Batching scope

Carpool: neighbors vs anyone

The prior scheme carpools only immediate neighbors — riders whose PTEs sit in the same 32 B cache sector (at most 4). But GPU threads live all over town: their target pages scatter across tens to hundreds of MB. MPW's carpool takes anyone going to the same level, no matter where their page lives — so it fills far more seats and lifts throughput from 3.10× to 3.90×.

Where it breaks: a real carpool must detour to each rider's house; MPW pays no such penalty because all batched loads are independent and issued in parallel to the memory system.
Cutting cost · Partitioned queue

Splitting one big mailroom

Searching one giant mailroom for every parcel is slow and power-hungry (a big monolithic CAM search). MPW gives each walker its own small mailroom — a partitioned slice of the PW queue. Now a walker scans only its own few entries to find batchable walks, so each associative search is tiny.

Where it breaks: splitting means a walker can only batch walks in its own slice — it can't grab a batchable walk sitting in a neighbor's queue, trading a little batching potential for much cheaper search.
Cutting cost · Bloom filter

A guest list at the door

When a parcel (a returned PTE) arrives, someone must find which walker ordered it — checking every walker's mailroom is wasteful. Each walker keeps a guest list (a bloom filter) of what it ordered. The doorman checks the lists first and only searches the mailrooms whose list says "maybe mine." That drops associative searches from 32 to just 1.84.

Where it breaks: a bloom filter can say "maybe" when the answer is no (a false positive, ~2.6% here) — costing an extra wasted search — but it never misses a real match, so correctness is safe.
Why not brute force · Scaling walkers

Hiring more toll booths

Traffic jams at the toll plaza? Just build more booths (more walkers). Peak flow needs about 192 booths — but every booth needs a live wire into one shared control board (the CAM), and wiring 192 booths means an 18-lane board whose cost explodes 9.1×. MPW instead makes each of the 32 existing booths process a whole busload at once.

Where it breaks: toll booths are independent; walkers share the CAM and PW queue, which is exactly why simply "adding booths" is so much costlier than it looks.
Why it works · Scattered accesses

Errands clustered vs all over town

Some workloads (ST) run errands close together — a tight cluster of pages. Others (BH) scatter errands across the whole city (a 128 MB range). If your carpool only accepts near-neighbors, the scattered runner gets no help. Because MPW batches walks to any same-level page, it's precisely the scattered, irregular workloads — the ones that pressure walkers most — that benefit the most.

Where it breaks: "scattered" here means scattered page indices, but all the batched walks still touch the same page-table level — the batch's common thread is level, not locality.
Where the cost lives · Leaf-heavy walks

The last mile is the whole trip

A delivery's early legs (airport to depot) are fast and pre-arranged — that's the upper page-table levels, which the PWCs almost always hit. The cost is the last mile to each doorstep: the L1 leaf lookups, which are 89–92% of all walk work and can't be cached in advance. MPW pours its batching effort exactly there, into concurrent L1 leaf walks.

Where it breaks: the last mile isn't all of it — a small slice (≈6.5–11%) still hits L2, and MPW batches those too, just with less payoff.
The full explanation

How it actually works

Section by section, with the paper's key structures rebuilt as visuals you can read at a glance. Tiering/CXL connections are flagged in green callouts.

1 · The problem — one walk per walker, thousands of threads

On a GPU the translation path looks like the CPU's: L1 TLB per SM, a shared L2 TLB, then a page walk subsystem of a PW queue plus hardware walkers. The mismatch is quantitative: thousands of threads miss the TLB in bursts, but each walker services one walk at a time. Requests back up in the queue.

The paper isolates the cost by sweeping the walker count. With a 256-entry PW queue, 256 walkers eliminate queueing entirely, so any latency above that is pure waiting. The baseline's 32 walkers see 3.0× higher PTW latency — the excess is queueing delay — and that translates to a 1.6× GPU slowdown versus the walker-unconstrained ideal (192 walkers reach peak). A real-hardware check on an RTX 3090 with 4 KB vs 2 MB pages shows the gap growing super-linearly with working-set size, confirming that a real GPU can only service a limited number of concurrent walks.

📉
Concretely: the applications that pressure walkers hardest (DR, BIC, BF, GUP) each need ~192 walkers to reach peak — 6× the baseline. That 6× gap is the throughput MPW has to recover from just 32 walkers.
🔌
Tiering link: this is a service-rate bottleneck, not a hit-rate one. In tiered/CXL memory, walker pressure rises the same way — smaller effective pages, fragmentation, and multi-TB footprints mean more TLB misses and thus more walks queued. MPW's lesson is that widening translation throughput can matter as much as improving TLB reach.

2 · Why more walkers isn't the answer

Adding walkers works on paper — but each walker performs associative CAM searches both when it starts and finishes a walk, to track in-flight memory requests. Scaling the CAM to 192 walkers means either serializing on a single port (a synthesized single-ported CAM costs 10 cycles at 1,365 MHz, re-inflating the queueing delay) or adding ports.

The trap: brute-forcing walker count converts a queueing problem into an area/energy problem. MPW's whole premise is to raise per-walker throughput instead — keep 32 walkers, make each do the work of many.

3 · The insight — batch beyond a single memory access

The prior neighborhood-aware scheme merges only walks inside one memory access: a 32 B sector holds four 8-byte PTEs, so it coalesces up to 4, giving 1.52× throughput. But that's capped by physical adjacency. The paper models a "batched PTW" walker (up to 8 requests) and evaluates three batching scopes:

Batching scopeWhat can be batched togetherWalker throughput
Single PT pagewalks into the same 512-entry page-table page (up to 512 adjacent PTEs)3.10×
Adjacent 4 PT pageswalks into any of 4 neighboring page-table pages (inter-page locality)3.37×
Any PT pageany walks heading to the same page-table level, however far apart3.90×

Wider is strictly better because GPU workloads scatter accesses across tens to hundreds of MB. And the throughput comes overwhelmingly from L1 leaf walks — 89.0–92.5% of all walks target L1 — because the upper-level PWCs almost always hit. (As batching widens, the L2 share of walks rises modestly, from 7.5% to 11.0%, since L1 pressure clears faster.)

🍯
The load-bearing observation: the batch's common thread is the page-table level, not spatial locality. That decouples batching from address contiguity — the reason MPW beats neighborhood-aware coalescing on exactly the scattered, irregular workloads (like BH's 128 MB spread) that hurt walkers most.

4 · The mechanism — PW queue + WST

MPW extends the page walk subsystem and nothing else — TLBs and the memory hierarchy are untouched. Two structures do the work:

  • Extended PW queue. Each entry keeps the target VPN and a valid bit (baseline), plus MPW's additions to carry a partial translation: a 37-bit PFN (next physical frame of the target page table), a 7-bit offset (cache-sector index), a 2-bit PL (target page-table level), and a 1-bit W flag (this walk is being serviced). Carrying partial state lets a walker serve a request one level at a time.
  • Walker Status Table (WST). One entry per walker: a 2-bit status (idle / batching / issuing / waiting), a 2-bit PL (the level this walker is currently batching), and a PTW-request bit vector marking which queue entries this walker is handling.

5 · Making it cheap — partitioned queues + bloom filters

Batching over a big monolithic PW queue would need large associative searches on every batch (when selecting) and on every returning PTE (to find the owner walker) — a serious energy cost. MPW attacks both:

# 1) Partition the PW queue — one small split queue per walker
batch(walker w):
    scan only w's split queue            # small CAM search, not the whole 256 entries
    group entries whose PL == w.PL      # same page-table level → batchable

# 2) On a returning PTE, avoid searching every walker's queue
on_return(pte):
    for each walker's bloom filter:
        if filter[hash(pte.addr)] != 0:  # "maybe requested this"
            CAM-search that walker's split queue
    # bloom filters cut avg searches from 32 → 1.84 (2.6% false positives)

The bloom filter is counter-based (128 entries, 3-bit counters per walker): issuing a load increments the hashed entry; matching a return decrements it. False positives cost only an extra wasted search, never a wrong result. This design skips on average 30.2 redundant CAM searches for just 1.5 KB of storage.

6 · The bill — storage, area, power

Everything MPW adds, summed:

# PW queue extension: 37 PFN + 7 offset + 2 PL + 1 W = 47 bits/entry
PW queue  = 47 * 256 / 8      = 1504 B
# WST: 2 status + 2 PL + 8-bit PTW vector = 12 bits/walker
WST       = 12 * 32  / 8      =   48 B
# bloom: 128 entries * 3-bit counter = 48 B/walker
bloom     = 128 * 3 * 32 / 8  = 1536 B
                        total  = 3088 B = 3.02 KB

Via CACTI 7.0, that's an added 0.473 mm² — 0.11% of a TU106 die (445 mm²) — and 6.6 mW, or 0.004% of its 160 W TDP. And the batching compare logic (an 8-entry split queue) synthesizes to 0.33 ns, within one 1,365 MHz cycle; even a pessimistic +2-cycle batching penalty costs only 0.06% performance.

💡
Why so cheap? MPW doesn't add walkers or a big CAM — it reuses the 32 walkers and adds a few bits per queue entry plus tiny per-walker structures. All of it sits off the memory-access critical path, so normal walks aren't slowed.

7 · Correctness & corner cases

MPW handles only reads (page-table loads), so it doesn't touch memory consistency. When a page migration updates page tables, the GPU issues a TLB shootdown; MPW simply flushes the affected walk requests from its PW queues and WST entries, and the flush latency hides easily under the GPU's long (~4.2 µs) shootdown. Two second-order risks are measured and found minor: (i) firing many loads at once barely pressures the memory system (page-walk traffic is a small share of DRAM), and (ii) a walk that finishes earlier than its batch-mates can wait for a free walker — this hits 9.7% of walks, raises their queueing 27%, and costs only 1.6% overall, because 87.3% of walks finish at the leaf and return individually.

🔌
Tiering link: the "read-only, so no consistency issue" and "shootdown just flushes queued walks" arguments are exactly the concerns you hit when page tables change under a live tiering/migration daemon. MPW shows a batched walker composes cleanly with page migration — a useful precedent for CXL/tiered systems where migrations are frequent.

8 · What the evaluation shows

On AccelSim (RTX 2060-like, 30 SMs, 32 walkers, 4-level page tables), across 12 benchmarks:

  • +55.6% GPU performance vs baseline; neighborhood-aware gets only +16.6%. MPW lands within 1.0% of an infinite-walker ideal.
  • −86.7% queueing latency, −62.4% average walk latency; memory latency improves 1.7×. Neighborhood-aware still has 5.1× MPW's queueing latency.
  • MPW concurrently services 3.1 walks on average (vs 1.5 for neighborhood-aware; up to 5.6), and keeps 72.1% of queued walks active at once.
  • Robust across page sizes: with 2 MB superpages MPW still adds +29.1% on large-footprint workloads.
  • Design-space: 32 walkers suffice (within 2.3% of 192); a 128-entry PW queue is enough; 128-entry bloom filters are the sweet spot; larger TLBs and PWCs help but MPW is complementary (iso-storage, MPW beats a 384-entry L2 PWC).
Follow the numbers, step by step

Worked examples

Five traces, each carried through with the paper's real numbers — the walker gap, the coalescing cap, the storage bill, the bloom-filter win, and the iso-storage PWC comparison.

① Why 32 walkers fall 6× short

The baseline has 32 walkers; peak needs ~192. Where does "6×" come from, and what does MPW have to deliver?

  1. 256 walkers = no queueing. The PW queue holds 256 entries, so 256 walkers can service every queued walk in parallel — that's the zero-queueing reference.
  2. 32 walkers → 3.0× PTW latency. Measured average across benchmarks; since walk time itself is fixed, the extra 2.0× is pure queueing delay.
  3. Peak throughput needs 192 walkers (e.g. DR, BIC), which is 192 / 32 = 6× the baseline's walker count.
  4. So MPW must raise per-walker throughput ~6× to match 192 walkers with 32. Its measured average is 3.1 concurrent L1 walks, up to 5.6 — enough to land within 1.0% of infinite walkers.
🍯
Insight: MPW doesn't literally hit 6× on average, but it doesn't need to — most of the queueing damage clears well before the theoretical ceiling, which is why 3.1 average batching recovers nearly all of the infinite-walker headroom.

② The coalescing cap: why neighborhood-aware tops out at 4

Neighborhood-aware translation merges walks within one memory access. Compute its hard ceiling and its realized throughput.

  1. A memory access is a 32 B sector. That's the baseline's data-access granularity (sectored cache, 128 B lines).
  2. A PTE is 8 B. So a 32 B sector holds 32 / 8 = 4 PTEs — the most walks that can share one memory read.
  3. Realized throughput ≈ 1.52× (not 4×), because walks rarely line up four-to-a-sector. With wider 64 B / 128 B accesses it rises only to 1.62× / 1.83×.
  4. MPW isn't sector-bound: it batches any same-level walks, realizing 3.1× on L1 — because it merges across page-table pages, not within a sector.
💡
The cap is structural: coalescing rides spatial adjacency, which GPU threads violate by scattering across tens–hundreds of MB. MPW rides level commonality instead, which scatter doesn't break.

③ Tallying MPW's storage — down to the bit

Is MPW actually cheap? Add it up for 256 PW-queue entries and 32 walkers (49-bit physical addresses).

  1. One PW-queue entry extension = 37 PFN + 7 offset + 2 PL + 1 W = 47 bits.
  2. Whole PW queue: 47 bits × 256 entries ÷ 8 = 1504 bytes.
  3. One WST entry = 2 status + 2 PL + 8-bit PTW-request vector = 12 bits; × 32 walkers ÷ 8 = 48 bytes.
  4. Bloom filters: 128 entries × 3-bit counter = 48 B/walker; × 32 = 1536 bytes.
  5. Total: 1504 + 48 + 1536 = 3088 bytes = 3.02 KB → 0.473 mm² (0.11% of TU106) and 6.6 mW (0.004% of TDP).
🍯
Insight: the punchline is the ratio — a +55.6% performance gain for ~three kilobytes and one-tenth of one percent of the die. The bloom filters are half the storage, and they exist purely to save energy, not to enable batching.

④ The bloom filter's payoff — 32 searches down to 1.84

When a PTE returns, which walker asked for it? Without help you'd search all 32 split queues. Trace what the filter buys.

  1. Naïve cost: a returning PTE must be matched to its owner walker → one CAM search per walker = 32 searches.
  2. Each walker keeps a bloom filter (128 entries, 3-bit counters). On issue, it increments the hashed entry for the load's address.
  3. On return, hash the PTE address and check each walker's filter. Search only walkers whose counter is non-zero ("maybe mine").
  4. Result: average searches drop from 32 to 1.84, with a 2.6% false-positive rate — i.e. the filter skips ~30.2 useless searches per return.
  5. Cost of a false positive: just one extra wasted search; never a wrong walk result, since bloom filters have no false negatives.
💡
Bigger filters push false positives lower but cost storage; 128 entries is the knee. This is a classic bloom-filter energy/accuracy trade — the same tool you'd reach for to prune directory or coherence lookups in a tiered system.

⑤ Iso-storage: MPW's 3 KB vs a bigger PWC

A fair critic asks: spend MPW's 3,088 bytes on a bigger page walk cache instead. How many PWC entries, and does it win?

  1. A PWC entry costs 9.25 bytes (assuming 49-bit virtual/physical addresses).
  2. Convert the budget: 3088 ÷ 9.25 ≈ 334 entries (the paper rounds to ~338; the design-space sweep uses a 384-entry L2 PWC for simplicity).
  3. Bigger PWCs only cache upper-level (L4/L3/L2) partial translations — but ~89–92% of the walk cost is at the L1 leaf, which PWCs can't hold.
  4. So under equal storage, MPW beats the enlarged PWC: PWCs shave the cheap part; MPW attacks the leaf-heavy part. (Even a 512-entry PWC only partly relieves walker pressure.)
🍯
Insight: the iso-storage test is the honest one — and it lands because MPW and PWCs target different parts of the walk. They're complementary, but for a fixed budget the throughput fix (MPW) beats the caching fix (bigger PWC).

The cap is structural; actual gains are far below it because GPU accesses scatter, which is exactly why MPW batches across pages instead.
Problem 3 · medium
The paper evaluates three batching scopes with throughputs 3.10× (single PT page), 3.37× (adjacent 4), and 3.90× (any PT page). Why does widening the scope help, and what workload property makes "any page" the clear winner?
Show answer
A wider scope lets a walker pull more batchable walks into one group, raising batch size and thus throughput. "Any page" wins because GPU workloads concurrently access pages across tens to hundreds of MB (e.g. BH spreads over a 128 MB range). With a narrow scope, most of those scattered walks can't be batched together even though they all target the same page-table level; "any page" removes the spatial restriction and captures them.Misconception: that batching needs spatial locality. MPW's batch is unified by page-table level, not address proximity — so scatter, which kills coalescing, doesn't hurt MPW.
Problem 4 · medium
Roughly 89–92% of page walks target L1 (leaf) page tables, and upper-level PWCs almost always hit. Explain why MPW's benefit comes overwhelmingly from batching L1 walks, and why bigger PWCs can't substitute.
Show answer
Upper levels (L4/L3/L2) are cheap: the PWCs cache their partial translations and hit ~100% except cold misses, so they rarely queue. The expensive, un-cacheable work is the L1 leaf lookup, which is the vast majority of walks and the source of contention. MPW batches those leaf walks concurrently, attacking the bottleneck directly. A bigger PWC only stores upper-level translations, so it shaves the already-cheap part and barely touches the leaf pressure — under iso-storage MPW wins.Misconception: that caching more of the page table relieves walker pressure. PWCs can't cache leaf entries, so they miss the 89–92% that actually matters.
Problem 5 · medium
MPW partitions the monolithic PW queue into per-walker split queues and adds a per-walker bloom filter. Each addresses a different associative-search cost. Name both and match each mechanism to the cost it cuts.
Show answer
Two CAM costs: (1) selecting/batching a new walk requires scanning the queue for same-level entries; (2) on a returning PTE, the subsystem must find which walker requested it by searching walkers' records. Partitioning cuts cost (1): a walker scans only its small split queue, not all 256 entries. Bloom filters cut cost (2): each returning PTE checks cheap filters first and only CAM-searches walkers whose filter says "maybe," dropping average searches from 32 to 1.84.Misconception: that the two mechanisms are redundant. They target opposite ends of the walk — batching-time selection vs return-time matching — so both are needed.
Problem 6 · hard
A walker frees only after every walk in its batch finishes. A walk that hits L2 cache returns before a batch-mate that goes to DRAM. Why doesn't this "straggler" effect hurt much (the paper measures ~1.6% loss)?
Show answer
Two reasons. First, 87.3% of walks finish at the L1 leaf and their responses are handled individually, so most walks aren't stuck behind a slower batch-mate. Second, the affected fraction is small: only 9.7% of walks experience differing batch latencies, and even those see just a 27% queueing increase. Net effect ≈ 1.6% — negligible against the 86.7% queueing-latency reduction MPW delivers.Misconception: that batching forces every walk to wait for the slowest in its group. Only the minority with mixed memory latencies are exposed, and leaf walks — the bulk — complete and return on their own.
Problem 7 · hard
You have MPW's 3,088-byte budget. A colleague proposes spending it on a bigger page walk cache instead (~334–338 entries). Argue which is better under equal storage, and why the comparison is fair.
Show answer
It's fair because both are on-chip storage attacking translation overhead. But they target different parts of the walk: a PWC caches upper-level partial translations (L4/L3/L2), while ~89–92% of the cost is at the L1 leaf, which PWCs cannot hold. So the bigger PWC relieves the cheap levels and barely dents the leaf-driven queueing, whereas MPW raises throughput on exactly the leaf walks. Result: iso-storage, MPW beats the enlarged PWC (even a 512-entry PWC only partly helps). They're complementary, but for a fixed budget the throughput fix wins.Misconception: that more caching always beats more concurrency. When the bottleneck is un-cacheable leaf walks, adding cache to upper levels can't touch it.
Problem 8 · medium
Does MPW's batching of page-table loads create a memory-consistency problem? And when page tables change under a migration, how does MPW stay correct?
Show answer
No consistency problem: MPW's batched operations are all reads (page-table loads), and consistency issues arise from load/store ordering to the same address — MPW issues no stores. When a page migration updates page tables, the GPU performs a TLB shootdown; MPW cooperates by flushing the affected walk requests from its PW queues and WST entries during the shootdown. The flush latency hides under the GPU's long shootdown (~4.2 µs), so correctness holds without a performance hit.Misconception: that issuing many concurrent memory requests risks consistency. Reads alone are safe; the only correctness concern is stale mappings during migration, which the existing shootdown mechanism already handles.
Quick answers

Frequently asked questions

Twelve questions newcomers ask most. Tap any to expand.

Q1In one sentence, what does MPW do?▾
It lets each GPU page-table walker batch multiple pending walks that target the same page-table level and issue all their memory loads concurrently, so the walk queue drains far faster without adding more walkers.
Q2What exactly is the bottleneck — the walk, or the wait?▾
The wait. Thousands of threads miss the TLB in bursts, but each walker services one walk at a time, so requests queue. That queueing inflates page-walk latency 3.0× and cuts GPU performance 1.6×. MPW cuts the queueing (−86.7%), not the per-walk memory latency.
Q3Why not just add more walkers?▾
Peak throughput needs ~192 walkers (up to 512 on a V100-class chip), but each walker needs associative CAM lookups when starting/finishing a walk. Scaling the CAM to 192 walkers requires an 18-port design whose area explodes 9.1× (≈3.9% of a TU106 die). MPW gets the throughput of many walkers from just 32, at ~0.1% area.
Q4How is this different from neighborhood-aware coalescing?▾
Neighborhood-aware merges only walks inside one 32 B memory access — at most 4 adjacent PTEs (≈1.52× throughput). MPW batches walks to any page-table page at the same level, no matter how far apart. That's why MPW gets 3.90× (any-page scope) and 3.1× realized on L1, and +55.6% vs the prior scheme's +16.6%.
Q5Why does the batch group by page-table level?▾
Because walks at the same level need the same kind of memory access (an L1 leaf fetch, say), they can be issued and completed together. Grouping by level — not by address proximity — decouples batching from locality, which is what lets MPW help scattered, irregular workloads that coalescing can't.
Q6What are the WST and the W flag for?▾
The Walker Status Table tracks each walker's state (idle/batching/issuing/waiting), the level it's batching, and a bit vector of which queue entries it's handling. The W flag in each PW-queue entry marks that a walk is being serviced, so no two walkers grab the same walk. Together they let a walker safely gather and fire a batch.
Q7Why partition the PW queue and add bloom filters?▾
Both save associative-search energy. Partitioning means a walker scans only its small split queue when batching, not all 256 entries. Bloom filters mean a returning PTE only searches walkers that "maybe" requested it, cutting average CAM searches from 32 to 1.84 (2.6% false positives). Together they make wide batching affordable in hardware.
Q8How much does MPW cost?▾
About 3.02 KB of storage (1504 B PW-queue extension + 48 B WST + 1536 B bloom filters), which via CACTI is 0.473 mm² (0.11% of a TU106 die) and 6.6 mW (0.004% of its 160 W TDP). None of it sits on the memory-access critical path.
Q9Does MPW help with big (2 MB) pages too?▾
Yes. With 2 MB superpages on large-footprint workloads (3.1–6.7 GB working sets), MPW still adds +29.1%, because even with bigger pages the walker contention persists once footprints exceed TLB coverage. The batching win is largely page-size-independent.
Q10Is there a straggler problem when a batch's walks finish at different times?▾
A small one. A walker frees only after its whole batch completes, so a walk that returns early can wait if all walkers are busy. But 87.3% of walks finish at the leaf and return individually; only 9.7% see mixed latencies (27% more queueing), for a net ~1.6% loss — negligible against the 86.7% queueing reduction.
Q11Where's the ceiling — how close is MPW to ideal?▾
Very close. MPW lands within 1.0% of an idealized GPU with infinite walkers, and 32 MPW walkers come within 2.3% of 192 walkers. Its average of 3.1 concurrent L1 walks (up to 5.6) recovers nearly all the infinite-walker headroom, so more walkers or a bigger PW queue barely help.
Q12Does MPW compose with better TLBs, PWCs, or speculation?▾
Yes — those are orthogonal. Larger TLBs and PWCs reduce walk volume; MPW raises walk throughput, and the paper shows they stack (e.g. bigger L2 TLBs plus MPW beat either alone). Speculative translation (AVATAR) is limited by compression-ratio requirements some sectors miss, so MPW is a more general fix. The authors position MPW as a component that cooperates with, not replaces, TLB/PWC improvements.

Glossary, results & source notes

Every term in one place, the headline numbers, a path to go deeper — and an honest ledger of what's from the paper vs my framing.

Core terms
MPW (Marching Page Walks)The paper's mechanism: let each page-table walker batch multiple same-level walks and issue their memory loads concurrently, raising walker throughput without adding walkers.
Page table walk (PTW)The hardware traversal of a multi-level page table to resolve a virtual→physical mapping after a TLB miss.
Page walk subsystemPW queue + hardware walkers + page walk caches. The only part MPW modifies.
Batching scopeHow far apart batchable walks may be: single PT page (3.10×), adjacent 4 pages (3.37×), or any PT page at the same level (3.90×). Wider is better.
PW queue (partitioned)The queue of pending walks; MPW splits it into a small per-walker slice so batching scans few entries. 256 entries baseline, 8 per walker.
WST (Walker Status Table)Per-walker table tracking status (idle/batching/issuing/waiting), the level being batched (PL), and a bit vector of the walks in the current batch.
Bloom filter (counter-based)Per-walker 128-entry, 3-bit-counter filter that, on a returning PTE, skips CAM searches on walkers that didn't request it — 32 searches → 1.84.
PL / W fieldsPL = target page-table level of a walk/walker; W = "this walk is being serviced," so two walkers never grab the same walk.
Supporting terms
L4→L1 page table4-level radix tree: L4 root … L1 leaf. 512 × 8 B entries = 4 KB per page. (Note: inverted vs x86's PML4→PTE labels.)
PWC (page walk cache)Caches upper-level (L4/L3/L2) partial translations; almost always hits. Cannot cache L1 leaf entries — the bulk of walk cost.
TLB (L1/L2)Private L1 TLB per SM, shared L2 TLB. A last-level TLB miss triggers a walk.
CAMContent-addressable memory: searches all entries in parallel. Tracks in-flight walk requests; area/energy grow super-linearly with ports.
Neighborhood-aware translationPrior art: coalesces walks within one 32 B memory access (≤4 PTEs, ~1.52×, +16.6%). The baseline MPW improves on.
Queueing latencyTime a walk waits in the PW queue for a free walker. The dominant cost (2.0× of the 3.0× PTW latency); MPW cuts it 86.7%.
SM (streaming multiprocessor)A GPU core cluster; each has a private L1 TLB. The eval GPU has 30 SMs at 1,365 MHz.
PFN / offsetPartial-translation fields MPW adds to PW-queue entries: 37-bit next physical frame number and 7-bit cache-sector index, concatenated to form the next load address.
ShootdownInvalidation when page tables change (e.g. migration); MPW flushes affected queued walks. Hidden under the ~4.2 µs GPU shootdown.
AVATARA prior speculative-translation scheme for GPUs; limited by a compression-ratio (1.45) requirement not all sectors meet.
