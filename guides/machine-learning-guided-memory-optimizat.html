Teaching memory where the next terabyte will be needed

DLRM inferences touch billions of embedding vectors across GPU and CPU memory. LRU caching misses half the accesses. This is the story of RecMG, a pair of ML models that learn what to cache and what to prefetch — cutting latency up to 43%.

1DLRM architecture: embeddings + interaction▾

A DLRM takes categorical features (e.g., "user interested in sports") and continuous features (e.g., "age 25") as input, predicts click-through rate. Categorical features are sparse (one-hot/multi-hot vectors); each activates indices in huge embedding tables (EMBs) to gather dense embedding vectors. These vectors are pooled, combined with continuous features via MLPs, then fed to an interaction layer.

EMBs dominate memory: a single table can be 100s of GB; total can be multi-TB. GPU memory is the bottleneck — hence tiered memory (GPU + CPU). Inference differs from training: batches access diverse items simultaneously (higher pooling factor), and no predetermined access pattern for easy prefetch planning.

Now you're ready: RecMG targets the GPU buffer for EMBs during inference, where LRU caching alone leaves up to 58.7% of time waiting on CPU fetches (Table I in the paper).
2Tiered memory: GPU fast/small, CPU slow/big▾

DLRM's EMBs exceed GPU capacity (40 GB on an A100). Solution: keep a small portion of frequently accessed vectors in GPU memory (the GPU buffer), the rest in CPU DRAM or SSD. On-demand fetches from CPU cost O(10 µs) per vector; at inference scale (millions of vectors per batch), this becomes a big fraction of total time.

PyTorch's TorchRec library maintains a software-managed cache on GPU using LRU. But LRU only considers recency, not benefit from caching — and Table I shows even with caching, embedding-vector accesses can take 58.7% of total inference time when buffer size is tight.

Now you're ready: RecMG replaces LRU with learned policies to minimize CPU fetches, achieving up to 43% end-to-end speedup (Figure 16).
3Power-law distribution + long reuse distances▾

Embedding-vector accesses follow a power law: about 20% of vectors take ~80% of accesses. This suggests LRU caching should work. But Figure 3 reveals the problem: 20% of accesses have reuse distance > 2²⁰. Reuse distance = number of distinct vectors accessed between two references to the same vector. If reuse distance exceeds cache size, the second access misses.

GPU buffers in production hold O(100,000) vectors. Reuse distance 2²⁰ ≈ 1 million — far too large. LRU evicts these vectors before they're reused, causing on-demand fetches. Also, some vectors are accessed rarely but predictably (e.g., following trending news); LRU can't prefetch them.

Now you're ready: RecMG's prefetch model learns correlations between accesses to predict irregular or long-reuse-distance vectors, reducing on-demand fetches by 2.2× vs LRU (Figure 14).
4Why existing prefetchers fail for DLRM▾

Rule-based prefetchers (Bingo spatial, Domino temporal) assume spatial or temporal locality. Embedding-vector accesses have low spatial locality (indices within a table are not contiguous) and limited temporal locality (long reuse distances). Figure 9 shows their correctness < 0.3%.

ML-based prefetchers (Voyager, TransFetch) face two problems: (1) huge search space — billions of unique indices, not sparse addresses; one-hot labeling is infeasible (out of memory); (2) high overhead — Table II shows Voyager takes 1521 µs per prediction vs RecMG's 92 µs, and TransFetch 1052 µs. They can't run fast enough for production.

Now you're ready: RecMG narrows the search space via a caching model (binary labels, not billions) and uses LSTM (CPU-friendly) not transformers, keeping overhead low (Section VI-C).
5Why two separate ML models?▾

Traditional ML prefetchers use one model to predict "next address." RecMG uses two models: a caching model (decides which vectors to keep) and a prefetch model (decides which vectors to insert). Why?

Caching exploits temporal locality and can be framed as binary classification (keep or evict each vector in the input sequence) — just 2 labels, low complexity. Prefetching targets irregular accesses and must predict a sequence of vectors with no reuse yet — a regression problem in a large search space. By separating them, the caching model reduces the search space for the prefetch model (fewer vectors need prefetching), and each model can be optimized for its specific task.

Now you're ready: Figure 5 shows their architectures. Caching model: 1 LSTM stack, 37K params; prefetch model: 2 LSTM stacks, 74K params. Together they reduce on-demand fetches by 4.5× vs rule-based methods (Figure 14).
6What "utility" means (and why not "demand")▾

Demand = how much cache a vector tries to use (roughly, how many unique accesses). Utility = the benefit a vector gets from being cached, measured by misses saved. A streaming workload touching gigabytes of data once has huge demand but zero utility (no reuse). LRU rewards demand; RecMG rewards utility.

RecMG's caching model learns to approximate Belady's optimal algorithm (which knows the future and keeps vectors with farthest next access). Figure 8 shows the caching model achieves 83% of optgen's hit rate, 38% better than LRU — by predicting utility, not just recency.

Now you're ready: This is the core insight from the UCP paper (see house style reference) adapted to DLRM: give cache to vectors that benefit most, not to vectors that ask most.
7Ground truth from optgen (Belady's algorithm)▾

To train the models, RecMG needs labels. For the caching model, it runs optgen [35] on offline traces: optgen computes what Belady's MIN algorithm (which knows all future accesses) would cache. Each vector in the trace gets a binary label: "1" if optgen would keep it, "0" if evict. This is the caching trace (ground truth).

For the prefetch model, the ground truth is the sequence of vectors that miss in optgen's cache (the prefetch trace). RecMG trains the prefetch model to predict these misses in advance, so they can be prefetched. By learning from the optimal policy's mistakes, the prefetch model complements the caching model.

Now you're ready: Section VI-A details training. RecMG retrains periodically when EMBs update (content popularity shifts). Cross-entropy loss for caching (binary classification); Chamfer measure for prefetch (regression with variable-length output).
The whole story in one place

RecMG in a nutshell

The problem, the ML solution, and how it all connects.

DLRM inference accesses billions of embedding vectors across GPU and CPU memory. LRU caching on GPU rewards demand (recency of access) but misses two critical patterns: (1) vectors with long reuse distances (> 2²⁰ for 20% of accesses, larger than typical GPU buffers), which get evicted before reuse; and (2) vectors with few reuses but predictable correlations (e.g., trending content), which LRU never prefetches.

RecMG introduces two LSTM-based models to fix this. The caching model learns to predict which vectors should stay in the GPU buffer by approximating Belady's optimal policy (trained on optgen labels), achieving 38% more cache hits than LRU (Figure 8). The prefetch model learns correlations between access sequences to predict which vectors will be needed soon, even if they haven't been accessed recently — reducing on-demand fetches by 2.2× vs LRU (Figure 14).

To make ML practical, RecMG: (1) uses binary classification for caching (just 2 labels: keep or evict), avoiding the billion-label search space; (2) outputs a sequence of vectors for prefetch (not just next address), improving coverage; (3) introduces a Chamfer-based loss function that handles variable-length outputs without bias; (4) deploys on CPU (not GPU) using LSTM (CPU-friendly) and achieves 92 µs per prediction vs 1521 µs for Voyager (Table II).

The payoff: on five Meta datasets, RecMG reduces end-to-end DLRM inference time by 31% on average (up to 43%) vs LRU, for under 1% extra area. Prefetch correctness: 37% (190× better than Domino, 27× better than TransFetch; Figure 9). Coverage: 10% better than TransFetch (Figure 10).

🍯 Key takeaways

  • LRU fails on DLRM because 20% of accesses have reuse distance > 2²⁰ (larger than GPU buffer), and many vectors are rarely reused but predictably correlated.
  • RecMG uses two separate ML models: caching (binary classification, approximates Belady) and prefetch (sequence regression, learns correlations).
  • Caching model reduces search space for prefetch by 2× (Figure 14); together they cut on-demand fetches by 4.5× vs rule-based prefetchers.
  • Key innovation: Chamfer measure loss for prefetch model — handles variable-length output sequences without introducing bias (Equation 5).
  • Deployed on CPU (not GPU) using LSTM (not transformers) for low overhead: 92 µs per prediction vs 1521 µs for Voyager (Table II).
  • Result: 31% avg speedup (up to 43%), prefetch correctness 37% (vs 0.3% for Domino), < 1% area overhead.
Six everyday pictures

How to think about RecMG

Each picture stands alone — read in any order. Every card ends with where the analogy breaks.

The core idea · Caching vs prefetching

Library stacks vs delivery service

You're studying in a library with a small desk (GPU buffer). Some books you reference often — keep them on the desk (caching model predicts these). Others you'll need soon but haven't touched yet — have them delivered to your desk just before you need them (prefetch model). LRU only keeps books you've recently touched, missing both.

Where it breaks: A library has physical limits; RecMG's "delivery" (prefetch) is constrained by CPU-GPU bandwidth and latency, not just shelf space. But the dual-strategy idea holds.
Caching model · Binary classification

Sorting mail: keep or shred

You get a stack of mail (input sequence). For each item, decide: keep (utility high) or shred (utility low). The caching model does this for embedding vectors — binary labels ("1" keep, "0" evict), not billions of possible addresses. This simplicity is what makes training feasible.

Where it breaks: Real mail sorting is independent per item; the caching model uses an LSTM to learn dependencies between items in the sequence, capturing patterns like "if vec A accessed, vec B likely soon."
Prefetch model · Long reuse distance

Stocking groceries for rare events

You buy milk weekly (short reuse distance, easy to cache). But you also buy birthday candles once a year (long reuse distance > cache size). The prefetch model predicts the candles will be needed soon by learning correlations — e.g., if you bought cake mix last week, candles are coming. LRU evicts candles as soon as you use them, then fetches on-demand next year.

Where it breaks: Groceries are planned; DLRM accesses are user-driven and stochastic. But the insight — correlation across sequences predicts rare events — is exactly what RecMG's prefetch model exploits (Figure 10: 10% coverage improvement vs TransFetch).
Model architecture · Why two models?

Two tools for two jobs

You have a hammer (caching model — simple, 1 LSTM stack, 37K params) for straightforward tasks (binary keep/evict) and a multi-tool (prefetch model — 2 LSTM stacks, 74K params) for complex tasks (sequence regression). Using one tool for both wastes capacity; using two optimizes each. Table III shows the prefetch model needs 2 stacks for 50% accuracy vs 39% with 1 stack.

Where it breaks: Tools don't coordinate; the two models do — the caching model reduces the search space for the prefetch model by ~2× (Figure 14), so they're complementary, not just parallel.
Chamfer measure · Loss function

Matching party RSVPs to attendees

You invite 5 people (output sequence) to a party expecting 15 guests (evaluation window, 3× longer). Chamfer measure: for each invitee, find the closest actual attendee, and vice versa. This avoids bias (a single match satisfying all invitees) and captures both precision (your invites matched) and recall (attendees were invited). Equation 5 uses this to train the prefetch model.

Where it breaks: Party guests are independent; prefetch hits have order dependencies (vectors prefetched stay in buffer for a time window). But the loss function's unbiased, two-way matching is exactly what makes RecMG's prefetch training work (Section V-B).
Deployment · Pipeline execution

Assembly line with inspection

GPU processes DLRM batch i while CPU (running RecMG models) prepares decisions for batch i+1. If CPU is slow, GPU moves on to i+2 without waiting — some cached states lag, but inference doesn't stall. Figure 6 shows this pipeline; Section VI-C reports 10× speedup from parallelism + AVX512.

Where it breaks: An assembly line stops if inspection is too slow; RecMG's pipeline doesn't — it trades freshness of caching decisions for throughput, accepting some stale priorities to avoid GPU idle time.
The full explanation

How it actually works

Section by section, with the paper's key mechanisms explained.

1 · The problem — LRU misses irregular and long-reuse patterns

DLRM inference accesses embedding vectors in two problematic patterns. First, 20% of accesses have reuse distance > 2²⁰ (Figure 3), larger than typical GPU buffers (~10⁵ vectors). LRU evicts these before reuse, causing on-demand CPU fetches. Second, some vectors are accessed rarely but predictably (e.g., following trending news) — LRU can't prefetch them because they lack temporal locality.

Table I shows the impact: with 192 EMBs, batch size 18K, and 7% caching ratio, embedding-vector accesses take 58.7% of total inference time. Even PyTorch's UVM prefetch (which lacks guidance on which vectors to prefetch) doesn't solve this — Section VII-B shows rule-based prefetchers achieve < 1% accuracy.

🌱
Concretely: Figure 13 shows that with 20% buffer size, LRU achieves ~60% hit rate. Optgen (Belady's algorithm, which knows the future) achieves ~85%. RecMG closes that gap to ~78% hit rate — a 45% reduction in misses vs LRU.

2 · Caching model — binary classification approximating Belady

The caching model takes a sequence of k embedding-vector accesses as input (each vector = table ID + row ID) and outputs a binary sequence of the same length: "1" means high priority (keep in GPU buffer), "0" means low priority (candidate for eviction). The model is trained on optgen labels: offline, RecMG runs Belady's MIN algorithm on production traces to compute what would have been cached if we knew the future.

# Training data generation
optgen( trace, buffer_size = 80% of GPU capacity )
  → caching_trace: each vector labeled 1 (kept) or 0 (evicted)

# Caching model architecture (Figure 5a)
Input: [ (table_id₁, row_id₁), ..., (table_idₖ, row_idₖ) ]
LSTM encoder-decoder (1 stack, 37K params)
Attention mechanism (learns long-range dependencies)
Output: [ 1, 0, 1, ..., 1 ] ← binary keep/evict for each input vector

# Loss function
Cross-entropy loss (binary classification)
  → sigmoid probability distribution over {keep, evict}

Figure 8 shows the result: the caching model achieves 83% of optgen's hit rate, and 38% more hits than LRU on all five datasets. The model approximates Belady without knowing the future, by learning correlations in the input sequence.

💡
Why binary classification works: Instead of predicting "which vector will be accessed next" (billions of labels), the model predicts "should this vector stay" (2 labels). This shrinks the search space and makes training feasible (Table III: 429 min training time for caching model vs 765 min for prefetch model).

3 · Prefetch model — sequence regression with Chamfer loss

The prefetch model also takes a sequence of k vector accesses as input, but outputs a shorter sequence of vectors to prefetch (length |PO| < k). Unlike traditional prefetchers that predict the next address, RecMG predicts a sequence to improve coverage — multiple vectors are prefetched at once and stay valid in the buffer for a time window.

# Prefetch model architecture (Figure 5b)
Input: same as caching model
LSTM encoder-decoder (2 stacks, 74K params)
Attention mechanism
Output embedding layer (fully connected + projection)
  → Output: [ vec_a, vec_b, ..., vec_m ] ← sequence of m vectors to prefetch

# Ground truth: evaluation window W
W = next |W| vectors accessed after input sequence
  where |W| > |PO| (typically 3× larger, Figure 12)
  → allows prefetched vectors to stay valid longer

# Loss function: Chamfer measure (Equation 5)
dist(PO, W) = α × (1/|PO|) × Σ min|x - y| (x ∈ PO, y ∈ W)
            + (1 - α) × (1/|W|) × Σ min|x - y| (x ∈ W, y ∈ PO)
  α = 0.7 (weight toward minimizing PO → W distance)
  → differentiable, unbiased (both directions), captures locality

Figure 9 shows prefetch correctness: RecMG achieves 37% (vs Domino 0.3%, TransFetch 10%). Figure 10 shows coverage: RecMG outperforms TransFetch by 10%, Domino by 190×. The Chamfer loss avoids the "shortcut" problem where all output vectors converge to the same value (Section V-B explains why a single CM term would fail).

🍯
Key insight: The evaluation window W is 3× longer than output sequence PO (Figure 12 shows optimal ratio). This means prefetched vectors don't need to match the immediately next accesses — they can stay in buffer and hit in the near future, maximizing prefetch utility.

4 · Model deployment — CPU pipeline, priority-based eviction

RecMG runs on CPU (not GPU) to save GPU cycles for DLRM inference. Figure 6 shows the pipeline: while GPU processes batch i, CPU infers caching/prefetch decisions for batch i+1. If CPU is slow, GPU proceeds to batch i+2 without waiting — some caching states lag, but inference doesn't stall.

Algorithm 1 (load_embeddings) shows how the two models' outputs are used. For each vector in the current batch, the caching model assigns a priority (high or low). When the buffer is full, Algorithm 2 (gpu_buffer_populate) evicts the vector with the lowest priority. Prefetched vectors (from the prefetch model) are inserted with high priority and protected from immediate eviction by an eviction_speed parameter (set to 4, inspired by RRIP [38]).

# Algorithm 1: load_embeddings (simplified)
for each vec in current_batch:
    priority[vec] = caching_model_output[vec] + eviction_speed
    # higher priority → stays longer

for each vec in prefetch_model_output:
    if buffer is full:
        evict_lowest_priority_vector()  # Algorithm 2
    fetch(vec)
    priority[vec] = eviction_speed  # protect from immediate eviction

# Algorithm 2: evict_lowest_priority_vector
evict_id = argmin( priority[vec] for vec in buffer )
priority[all vecs] = max(0, priority - 1)  # decay over time
evict(buffer[evict_id])
💡
Why eviction_speed matters: It balances the two models' impact. A larger value (e.g., 8) means prefetched vectors stay longer, dominating the buffer; a smaller value (e.g., 2) means caching-model decisions dominate. The paper sets it to 4 based on RRIP's re-reference interval prediction.

5 · System optimizations — parallelism, vectorization, quantization

To keep CPU overhead low (< 92 µs per prediction, Table II), RecMG uses three techniques:

  1. Pipeline execution: GPU and CPU run in parallel. CPU doesn't block GPU even if inference takes longer than DLRM batch processing (Figure 6).
  2. Thread-level parallelism: A batch of DLRM inference requests is split into n sub-batches (n = number of idle CPU cores), each served by one thread. Figure 7 shows near-linear scaling (1 thread: 1.8K samples/s, 16 threads: 28K samples/s).
  3. Vectorization + quantization: AVX512 instructions, C++ implementation, and model quantization combine for 10× speedup vs no optimization (Section VI-C).

Table II compares overhead: RecMG 92 µs/prediction, TransFetch 1052 µs, Voyager 1521 µs. The difference comes from (1) LSTM being more CPU-friendly than transformers, and (2) RecMG's narrower search space (binary classification for caching, small output sequence for prefetch).

Follow the numbers, step by step

Worked examples

Four traces using the paper's real numbers and mechanisms.

① Computing reuse distance and why LRU fails

An access sequence: [A, B, C, D, E, F, G, A]. GPU buffer holds 5 vectors. What is A's reuse distance, and does it hit or miss?

  1. Reuse distance = number of distinct vectors between two accesses to the same vector. Between first A and second A: {B, C, D, E, F, G} = 6 distinct vectors.
  2. GPU buffer holds 5 vectors. Reuse distance 6 > buffer size 5 → A is evicted before the second access.
  3. Second access to A = cache miss. Must fetch from CPU, O(10 µs) penalty.
  4. Figure 3 shows 20% of accesses have reuse distance > 2²⁰ ≈ 1 million. Typical GPU buffers hold O(100,000) vectors → these accesses always miss in LRU.
  5. RecMG's prefetch model learns correlations (e.g., "if B, C, D, then A soon") to prefetch A before the second access, avoiding the miss.
🍯
Insight: Reuse distance quantifies how far back temporal locality extends. LRU only works if reuse distance < cache size. For 20% of DLRM accesses, this condition fails — hence the need for ML prefetching.

② Training the caching model with optgen labels

Given a trace of 10 accesses and GPU buffer size = 3, how does RecMG generate training labels for the caching model?

  1. Input trace: [A, B, C, D, A, B, E, F, C, D]. Each is a (table_id, row_id) pair in practice, simplified here to letters.
  2. Run optgen (Belady's MIN) with buffer size = 3. Optgen evicts the vector whose next access is farthest in the future. Track which vectors stay cached at each step.
  3. After access to D (step 4): Buffer = {A, B, D} (A reused next at step 5, B at step 6, D at step 10 — keep all 3).
  4. After access to A (step 5): Buffer still {A, B, D}. No eviction needed.
  5. After access to B (step 6): Need to insert E. Evict D (next access step 10, farthest). Buffer = {A, B, E}.
  6. Generate binary labels: For each access, "1" if optgen kept it in buffer at that step, "0" if evicted. Full trace labels: [1, 1, 1, 0, 1, 1, 1, 1, 0, 0].
  7. Train caching model on (input_sequence, labels) pairs. Model learns to approximate optgen's decision without knowing the future.
💡
Figure 8 shows the caching model achieves 83% of optgen's hit rate. The 17% gap comes from two sources: (1) the model sees only past accesses, not future; (2) the model generalizes across traces, while optgen is trace-specific.

③ Why Chamfer loss avoids the "shortcut" problem

Prefetch model outputs PO = [1, 2, 3], ground truth window W = [2, 6, 7, 8]. Why does a single Chamfer term d_CM(PO, W) fail, and how does Equation 5 fix it?

  1. Single Chamfer term: d_CM(PO, W) = Σ min|x - y| for x ∈ PO, y ∈ W. For each x in PO, find closest y in W.
  2. For x = 1: closest y is 2 (distance 1). For x = 2: closest y is 2 (distance 0). For x = 3: closest y is 2 (distance 1).
  3. Total distance = 1 + 0 + 1 = 2. But notice: "2" in W is chosen for all three x's. During gradient descent, the model learns "always output 2" to minimize this loss — a shortcut.
  4. The problem: Single-term CM loses locality information. If all outputs converge to the same value, the prefetch model becomes useless.
  5. Equation 5 adds a second term: d_CM(W, PO) = Σ min|x - y| for x ∈ W, y ∈ PO. Now each y in W must also find closest x in PO.
  6. For y = 6, 7, 8: closest x is 3 (distances 3, 4, 5). Total = 3 + 4 + 5 = 12. If all outputs were "2," this term would be huge (distances 4, 5, 6 = 15), penalizing the shortcut.
  7. Final loss: 0.7 × (2/3) + 0.3 × (12/4) ≈ 1.37. The two terms balance each other, preventing convergence to a single value.
🍯
Insight: Chamfer measure is differentiable (needed for gradient descent) and two-way (prevents shortcuts). The weight α = 0.7 prioritizes "did PO match W?" over "did W match PO?" because we care more about prefetch precision than perfect coverage.

④ Calculating end-to-end speedup from hit rate

Figure 18 shows a linear model: inference time = f(hit rate). If LRU achieves 60% hit rate (200 ms), and RecMG achieves 78% hit rate, what is the speedup?

  1. Linear model (from Figure 18): Inference_time = a × (100 - hit_rate) + b, where a = slope (ms per 1% miss rate), b = base latency.
  2. At 60% hit rate (LRU): T_LRU = a × 40 + b = 200 ms (given). At 78% hit rate (RecMG): T_RecMG = a × 22 + b.
  3. From Figure 18, RMSE < 3.75 ms (1.7% error). Assume linear fit: T_RecMG ≈ 200 - (18% improvement × 200 / 40) = 200 - 90 = 110 ms.
  4. Speedup = T_LRU / T_RecMG = 200 / 110 ≈ 1.82 (82% faster). But this is theoretical; Figure 16 shows actual speedup 31% avg (up to 43%) because:
  5. Other factors: (1) GPU buffer management overhead (not just hit/miss), (2) prefetch timing (some prefetches arrive too late), (3) CPU inference overhead (92 µs per prediction adds up).
  6. Figure 19 validates the model: RecMG's measured hit rate (from Figure 15) predicts 31% speedup, matching Figure 16's result (within 3.6% deviation).
💡
The linear model is a simplification (reality has non-linear effects like bandwidth saturation), but it's accurate enough (1.7% RMSE) to estimate speedup from hit rate — useful for quick evaluation of new caching/prefetch strategies.

Problem 1 · easy
Figure 3 shows 20% of accesses have reuse distance > 2²⁰. If a GPU buffer holds 100,000 vectors, what percentage of these accesses will miss in an LRU cache, and why?
Show answer
All 20% will miss. Reuse distance 2²⁰ ≈ 1 million > buffer size 100,000 → the vector is evicted before its second access. LRU has no way to predict the second access and keep the vector cached, so every long-reuse-distance access causes an on-demand fetch from CPU.Misconception: that LRU "learns" from repeated accesses. LRU only tracks recency, not frequency or future reuse. If reuse distance > cache size, the second access always misses.
Problem 2 · medium
RecMG's caching model is trained on optgen labels (Belady's MIN). Belady knows all future accesses. How can RecMG approximate it using only past accesses?
Show answer
RecMG uses an LSTM with attention to learn correlations between past accesses that predict future reuse. E.g., if the sequence [A, B, C] historically precedes A's reuse, the model learns to keep A cached when it sees [A, B, C]. The attention mechanism captures long-range dependencies, so even if A's reuse is far away, the model assigns high priority to A. Figure 8 shows this achieves 83% of optgen's hit rate — the model generalizes patterns from training traces to new traces.Misconception: that you need future knowledge to approximate Belady. RecMG uses statistical patterns from past traces to predict future reuse, not actual clairvoyance. The 17% gap vs optgen is the cost of this approximation.
Problem 3 · medium
Why does the prefetch model use 2 LSTM stacks while the caching model uses only 1? What happens if you use 1 stack for prefetch?
Show answer
Table III shows that with 1 LSTM stack, the prefetch model achieves 39% accuracy; with 2 stacks, 50% (vs caching model 80% with 1 stack, 83% with 3 stacks). The difference: task complexity. Caching is binary classification (2 labels: keep/evict) — simpler, 1 stack sufficient. Prefetch is sequence regression in a large search space (billions of possible vectors) — needs more capacity (2 stacks) to learn complex correlations. Using 1 stack for prefetch underperforms by 11% and reduces coverage (Figure 10).Misconception: that "more stacks = always better." For caching, adding stacks beyond 1 gives < 5% improvement (Table III: 80% → 83%) but increases training time by 41% (429 → 603 min). RecMG uses the minimum stacks needed for each task.
Problem 4 · hard
Equation 5 uses α = 0.7 for the Chamfer loss. What happens if α = 1? If α = 0.5? Justify RecMG's choice.
Show answer
If α = 1, the second Chamfer term (d_CM(W, PO)) disappears. The loss becomes single-sided, causing the "shortcut" problem: all outputs converge to the same value (Example 3 in worked examples). If α = 0.5, the two terms are equally weighted. This overpenalizes coverage (did W match PO?) vs precision (did PO match W?). RecMG cares more about precision — prefetch should hit what it predicts — because low-precision prefetch pollutes the buffer with useless vectors, reducing cache hits. α = 0.7 weights precision 2.3× more than coverage (0.7 / 0.3), matching RecMG's design goal: maximize prefetch utility, not just coverage.Misconception: that α is arbitrary. It's a design choice reflecting RecMG's priority: better to prefetch fewer vectors that will be used than to prefetch many vectors that might be used. Figure 9 shows RecMG's 37% correctness balances precision and coverage effectively.
Problem 5 · medium
Figure 12 shows the optimal evaluation window W is 3× the output sequence length |PO|. Why not 1×? Why not 10×?
Show answer
If W = 1× |PO|, prefetched vectors must hit immediately (in the next |PO| accesses). This is too strict — many useful prefetches arrive slightly late and miss the window, lowering accuracy. If W = 10× |PO|, the window is too long — the loss function becomes insensitive (many vectors in W will naturally be close to some PO), and the model overfits to noise. Figure 12 shows 3× is the sweet spot: accuracy 37%, coverage > 0 (vs 1× gives lower accuracy, 10× gives no coverage gain). The intuition: prefetched vectors stay valid for ~3 batches, so evaluate them over that window.Misconception: that "bigger window = more coverage." Coverage saturates beyond 3× (Figure 12) because the model's output length |PO| is the limiting factor, not the window size. The window is for evaluation, not for increasing the number of prefetches.
Problem 6 · hard
Table II shows RecMG takes 92 µs per prediction, TransFetch 1052 µs, Voyager 1521 µs. Why is RecMG 11× faster than TransFetch, and why does this matter for production deployment?
Show answer
Three reasons: (1) Model architecture — RecMG uses LSTM (CPU-friendly, sequential); TransFetch uses transformers (GPU-optimized, parallel). On CPU, LSTM is faster. (2) Search space — RecMG's caching model has binary output (2 labels), and prefetch model has small sequence output (e.g., 5 vectors); TransFetch predicts from billions of possible addresses. (3) Optimizations — RecMG uses AVX512, C++, quantization (Section VI-C: 10× speedup). The 11× difference matters because: at 1M predictions/sec, RecMG adds 92 ms overhead; TransFetch adds 1052 ms (1 sec) — too slow for production inference (SLA ~hundreds of ms, per paper intro). Figure 6 shows RecMG overlaps with GPU, so 92 µs is hidden; 1052 µs would stall the pipeline.Misconception: that "transformers are always faster because of parallelism." On GPU, yes (TransFetch is faster there). But RecMG runs on CPU (to save GPU cycles for DLRM), where LSTM's sequential nature is actually an advantage — less synchronization overhead, better cache locality.
Problem 7 · easy
Figure 14 shows RecMG reduces on-demand fetches by 4.5× vs Domino (temporal prefetcher). Why does Domino fail so badly on DLRM workloads?
Show answer
Domino assumes temporal locality (if A accessed, A will be accessed again soon). But DLRM has two problems: (1) 20% of accesses have reuse distance > 2²⁰ (Figure 3), far exceeding Domino's metadata capacity (set to 10% of unique indices, per Section VII-B); (2) accesses are user-driven and stochastic (trending content, ad-hoc queries), not deterministic program patterns. Figure 9 shows Domino's prefetch correctness < 0.3% — it predicts the wrong vectors. RecMG's 37% correctness comes from learning correlations (not just recency), so it can predict irregular accesses.Misconception: that "temporal prefetchers always work." Domino excels on regular programs with short reuse distances (e.g., array scanning). DLRM is the opposite: irregular, long reuse distances, user-driven. The wrong tool for the job.
Problem 8 · medium
Figure 17 shows that at 0.1% buffer size, the prefetch model contributes 67.5% of RecMG's benefit; at 15% buffer size, only 27.7%. Why does the relative importance flip?
Show answer
At 0.1% buffer size, the buffer is tiny — LRU can only cache a handful of high-frequency vectors. Most accesses miss, so prefetch dominates: predicting which vectors will be needed soon is more valuable than optimizing the small cache. At 15% buffer size, the buffer is large enough to cache many high-frequency vectors, so caching dominates: keeping the right vectors (via caching model approximating Belady) reduces misses more than prefetching. Figure 17 shows GPU buffer management time drops 2.91× from 0.1% → 15% with RecMG, mostly due to improved caching (2.79× with caching model alone).Misconception: that "prefetch always dominates." At small buffer sizes, yes (prefetch is crucial). At large buffer sizes, no (caching is crucial). RecMG uses both models because DLRM workloads span this range — some deployments have tiny buffers (memory-constrained), others have large buffers (latency-constrained).
Quick answers

Frequently asked questions

Ten questions newcomers ask most. Tap any to expand.

Q1What's the difference between DLRM training and inference, and why does RecMG target inference?▾
DLRM training accesses embedding vectors in a predetermined order (batched samples from a fixed dataset), making prefetch easier (e.g., RecShard [68] uses offline profiling). DLRM inference is user-driven and stochastic — batch size varies (512 to 18K, Table I), pooling factor varies (1 to hundreds, per Section III), and accesses are unpredictable. RecMG targets inference because: (1) it's 80% of AI inference cycles at Meta datacenters (paper intro), and (2) existing methods (LRU, rule-based prefetch) fail on irregular patterns (Figure 9: < 1% correctness).
Q2Why not just increase GPU memory capacity instead of using tiered memory?▾
DLRM's EMBs can be multi-TB (paper intro: "terabyte scale"). Even the largest GPUs (e.g., A100 with 80 GB HBM) can't hold this. Tiered memory (GPU + CPU DRAM + SSD) is cost-effective: GPU memory is ~10× more expensive per GB than CPU DRAM. The tradeoff: GPU memory is O(100× faster), so the challenge is what to cache. RecMG optimizes this tradeoff, achieving up to 43% speedup (Figure 16) by learning access patterns.
Q3How does RecMG handle the "cold start" problem — new users or new content with no historical access patterns?▾
RecMG learns patterns from aggregated user behavior, not individual users. E.g., if trending news causes correlated accesses across many users, the model learns "if vec A accessed, vec B likely soon" — even for new users following the same trend. For genuinely new content (e.g., a new product launch), RecMG falls back to the caching model (which approximates Belady using temporal locality) until enough traces accumulate to retrain. Section VI-A mentions retraining when EMBs update, which handles content popularity shifts.
Q4Does RecMG work for other ML workloads besides DLRM (e.g., graph neural networks, transformers)?▾
RecMG is tailored to DLRM's specific patterns: (1) power-law distribution (20% vectors take 80% accesses), (2) long reuse distances (> 2²⁰ for 20% of accesses), (3) embedding-table structure (table ID + row ID). Other workloads have different patterns. E.g., GNNs have irregular graph traversal (PrefEdge [56] uses graph structure); transformers have attention patterns (Prodigy [76] uses program semantics). RecMG's techniques (binary classification for caching, Chamfer loss for prefetch) could adapt, but the model would need retraining on new workload traces.
Q5Why use LSTM instead of transformers, given that transformers are state-of-the-art for sequence modeling?▾
Transformers are GPU-optimized (parallel attention, large batch sizes). RecMG runs on CPU (to save GPU cycles for DLRM inference, Section VI-C). On CPU, transformers are slower: Table II shows TransFetch (transformer-based) takes 1052 µs per prediction vs RecMG's 92 µs (LSTM-based). The difference: (1) LSTM is sequential, better CPU cache locality; (2) transformers need more FLOPS per token (quadratic attention vs linear LSTM). RecMG prioritizes low overhead (< 92 µs) over model expressiveness, and LSTM is sufficient (37% prefetch correctness, Figure 9).
Q6How does RecMG handle multiple DLRM models running on the same GPU (multi-tenancy)?▾
The paper doesn't explicitly address multi-tenancy, but the design is flexible. Each DLRM model would have its own caching/prefetch models (trained on its own traces), and the GPU buffer would be partitioned (e.g., 50/50 for two models). The challenge: if models have different access patterns, the buffer partition might be suboptimal. A multi-model RecMG could use a global caching model that learns to partition the buffer dynamically, similar to UCP (the house-style reference). This is future work.
Q7What's the storage overhead of RecMG's models (37K + 74K params)? How big are they on disk?▾
37K + 74K = 111K parameters total. At FP32 (4 bytes/param), that's 111K × 4 ≈ 444 KB. With quantization (e.g., INT8, 1 byte/param), it's 111 KB. This is tiny compared to DLRM's EMBs (multi-TB). The models fit in CPU cache (L3 is typically 10s of MB), enabling fast inference (92 µs per prediction, Table II). Compare to Voyager (millions of params for one-hot labeling) or TransFetch (large transformer, 1052 µs overhead) — RecMG is 3–4 orders of magnitude smaller.
Q8How often does RecMG retrain, and what's the training cost?▾
Section VI-A: "periodically retrain when the DLRM requires retraining, which typically happens when embedding tables are updated." In production, this is ~weekly to monthly (content popularity shifts). Table III shows training time: caching model 429 min (7 hrs), prefetch model 765 min (13 hrs) on CPU. Total ~20 hrs on a single CPU, parallelizable across cores (Figure 7: near-linear scaling to 16 cores → ~1.5 hrs with 16 cores). The cost is acceptable given the 31% avg speedup (Figure 16) — training once per week to get 31% faster inference 24/7 is a good tradeoff.
Q9Does RecMG require changes to the DLRM model itself, or is it purely a memory-management layer?▾
Purely a memory-management layer. RecMG sits between the DLRM (which issues embedding-vector accesses) and the GPU buffer (which serves those accesses). The DLRM is unchanged. RecMG intercepts the access sequence, runs the caching/prefetch models on CPU, and updates buffer priorities + prefetch list (Algorithm 1). This is why RecMG is practical: it's a drop-in replacement for PyTorch's LRU cache, no DLRM retraining needed.
Q10What did the authors leave for future work?▾
The paper doesn't have an explicit "future work" section, but implicit gaps: (1) multi-model (multi-tenancy) support; (2) adaptive retraining (detecting when access patterns shift, triggering retraining automatically); (3) larger-scale evaluation (paper uses 5 datasets, all from Meta — generalization to other DLRM workloads?); (4) integrating with CXL (Compute Express Link) for tiered memory — the paper mentions "CPU memory or SSD" but CXL is emerging as a middle tier; (5) extending to DLRM training (currently inference-only).

Glossary & further reading

Every term in one place, then a path to go deeper.

Core terms
RecMGMachine learning-guided memory optimization system for DLRM inference on tiered memory. Uses two LSTM models (caching + prefetch) to manage GPU buffer.
DLRM (Deep Learning Recommendation Model)ML model for click-through rate prediction. Combines continuous features (user info) and categorical features (sparse, via embedding tables).
Embedding table (EMB)Large lookup table mapping categorical feature indices to dense vectors. Can be 100s of GB per table; multi-TB total. Dominates DLRM memory usage.
Caching model (UMON)RecMG's first ML model. Outputs binary priorities (keep/evict) for each vector in input sequence. Approximates Belady's optimal policy via optgen training labels.
Prefetch modelRecMG's second ML model. Outputs a sequence of vectors to prefetch (insert into GPU buffer before they're accessed). Learns correlations for irregular/long-reuse patterns.
Tiered memoryGPU (fast, small) + CPU DRAM (slow, big) + SSD (slowest, huge). DLRM's EMBs exceed GPU capacity, so misses fetch from CPU at O(10 µs) cost.
Reuse distanceNumber of distinct vectors accessed between two references to the same vector. Quantifies temporal locality. Paper shows 20% of accesses have reuse distance > 2²⁰.
Optgen (Belady's MIN)Offline algorithm that computes optimal caching decisions by knowing all future accesses. Used to generate training labels for the caching model.
Chamfer measureDifferentiable loss function for comparing two sets of points. RecMG uses it to train the prefetch model (Equation 5: two-way matching avoids shortcuts).
Evaluation window (W)Length of future access sequence used to evaluate prefetch hits. Typically 3× the output sequence length (Figure 12: optimal ratio).
Supporting terms
LRU (Least Recently Used)Traditional cache eviction policy. Evicts the vector not accessed for the longest time. Fails on DLRM because 20% of accesses have reuse distance > cache size.
Pooling factorNumber of embedding vectors gathered per DLRM inference query. Varies widely (1 to hundreds, per Section III), making prefetch challenging.
GPU bufferSoftware-managed cache on GPU memory (e.g., 20% of unique vectors, per paper). Managed by RecMG's caching/prefetch models, not hardware cache.
LSTM (Long Short-Term Memory)Recurrent neural network architecture. RecMG uses it for sequence modeling (learns dependencies across input vectors). CPU-friendly (vs transformers).
Attention mechanismNeural network layer that learns long-range dependencies. RecMG uses it to capture correlations between distant accesses in the input sequence.
eviction_speedParameter controlling how fast priorities decay (Algorithm 2). Set to 4, inspired by RRIP [38]. Balances caching-model vs prefetch-model influence.
Binary classificationML task with two labels (0 or 1). RecMG's caching model uses it (keep/evict), avoiding billion-label search space of address prediction.
Sequence regressionML task predicting a sequence of values. RecMG's prefetch model uses it (outputs multiple vectors), improving coverage vs single-address prediction.
Prefetch correctnessPercentage of prefetched vectors that hit within the evaluation window. RecMG: 37% (Figure 9), vs Domino 0.3%, TransFetch 10%.
CoveragePercentage of unique vectors in ground truth that appear in prefetch output (Equation 2). RecMG: 10% better than TransFetch (Figure 10).
Key results at a glance

Performance (Figure 16): RecMG reduces end-to-end DLRM inference time by 31% on average (up to 43%) vs LRU, across five Meta datasets. On-demand fetches reduced by 4.5× vs Domino, 2.8× vs Bingo, 1.5× vs TransFetch (Figure 14).

Model accuracy (Figures 8, 9, 10): Caching model achieves 83% of optgen's hit rate (38% better than LRU). Prefetch model: 37% correctness (190× better than Domino, 27× better than TransFetch), 10% coverage improvement vs TransFetch.

Overhead (Table II): 92 µs per prediction (vs Voyager 1521 µs, TransFetch 1052 µs). Training: 429 min (caching), 765 min (prefetch) on CPU. Model size: 111K params ≈ 444 KB (FP32), fits in CPU cache.

Area overhead (Section VI-C): < 1% (two models + metadata for GPU buffer management). No changes to DLRM model or critical path.

Concepts to shore up first

Recommendation systems and embedding tables easy

explainer · 20–30 min

How categorical features are encoded as embeddings, and why they're so large. Search "DLRM architecture explained" (e.g., Meta's DLRM repo [27]). RecMG's caching/prefetch assume you understand what an embedding lookup is.

Tiered memory systems medium

explainer · 30–40 min

How GPU + CPU DRAM + SSD interact, and the latency hierarchy. Search "tiered memory for ML" (e.g., [33], [60], [61] from the paper's references). RecMG targets this hierarchy.

Belady's MIN algorithm (optgen) easy

classic paper · 15–20 min

The 1966 algorithm that evicts the vector whose next access is farthest in the future. RecMG uses it to generate training labels. Read Belady [14] or Jain & Lin's optgen [35] for implementation.

Related papers

Utility-Based Cache Partitioning (UCP) medium

Qureshi & Patt · MICRO 2006

The house-style reference for this field guide. UCP partitions a shared cache by utility (benefit from cache), not demand (recency). RecMG adapts this idea: caching model predicts utility, prefetch model handles irregular patterns UCP couldn't.

RecShard: Memory Optimization for DLRM Training hard

Sethi et al. · ASPLOS 2022

Offline profiling-based memory optimization for DLRM training. RecMG targets inference, where access patterns are unpredictable (no offline profiling). Complements RecMG's approach.

Voyager: Learning Memory Access Patterns hard

Shi et al. · ASPLOS 2021

ML-based prefetcher using LSTM. RecMG differs: (1) binary classification for caching (not one-hot labeling), (2) Chamfer loss for prefetch (not L2), (3) CPU deployment (Voyager targets GPU). Table II shows RecMG is 16× faster.

TransFetch: Transformer-Based Prefetching hard

Zhang et al. · CF 2022

Transformer-based prefetcher. RecMG differs: LSTM (CPU-friendly) vs transformer (GPU-optimized), and 11× faster (Table II). Figures 9, 10 show RecMG's 27× better correctness, 10% better coverage.

Domino: Temporal Data Prefetcher medium

Bakhshalipour et al. · HPCA 2018

Rule-based temporal prefetcher using multiple streams. Fails on DLRM (0.3% correctness, Figure 9) because accesses have low temporal locality (20% reuse distance > 2²⁰). RecMG's ML approach learns correlations Domino can't.

Bingo: Spatial Data Prefetcher medium

Bakhshalipour et al. · HPCA 2019

Rule-based spatial prefetcher. Fails on DLRM (< 0.1% correctness, Figure 9) because embedding-vector accesses have low spatial locality (indices within a table aren't contiguous). RecMG learns irregular patterns.
