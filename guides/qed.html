A reordering only matters if it can be seen

Verifying that a real out-of-order CPU obeys its memory-consistency model is brutal — the cases explode with every in-flight instruction and every core. QED's escape hatch: only check reorderings that could produce a value the model forbids. This is the plain-English story of how that one idea makes verification scale.

📄 "QED: Scalable Verification of Hardware Memory Consistency" · Purdue 🎓 MCM & µarch background assumed — the primers cover it ⏱ ~35 min read
🔍

An out-of-order execution violates the MCM if and only if it is OBSERVABLE — it produces a value no compliant ordering could. So only check whether a reordered pair can be RESTORED to a legal order without changing any value.
If you can restore it unobservably, it was never a bug.

1What a memory consistency model (MCM) is▾

An MCM is the contract that says which values a load is allowed to return when many threads read and write shared memory at once. The paper frames a model as exactly two things: (1) which subsets of memory instructions must keep their program order, and (2) whether writes appear atomic. Sequential Consistency (SC) is the strictest: all program order preserved, writes atomic. TSO relaxes only store→load order to different addresses. RISC-V WMO relaxes almost everything, re-imposing order only through fences and acquire/release annotations.

QED denotes program order <p and the global memory order <m. A system is compliant if a single <m total/partial order exists that respects the model's required <p edges and makes every load read the latest store in <m.

Now you're ready: QED's whole job is to check that an RTL implementation can never produce an execution that no valid <m order could explain — for SC, TSO, and RISC-V WMO.
2Out-of-order issue and the load-store queue (LSQ)▾

A modern high-performance core issues memory instructions out of program order and overlaps them. The structure that holds in-flight loads and stores and lets them execute/complete in a different order than they were fetched is the load-store queue (LSQ). Loads may issue to the cache early; a store issues to the cache only at commit (for precise interrupts), though it may prefetch coherence permissions earlier and may complete out of order under weaker models.

The paper deliberately scopes itself to the LSQ and the coherence interface — the seam between the core's local cache hierarchy and global coherence — because that is where most consistency bugs live. It assumes the pipeline front-end (register/control dependencies) and cache coherence (write serialization, atomicity) are verified separately by other work.

Now you're ready: "the implementation" QED checks is concretely the LSQ + coherence interface, not the whole CPU. That scoping is half of why it can scale.
3Instruction reordering — and why it's usually fine▾

The core executes a pair of memory instructions in a different order (<e) than program order (<p) to hide latency. Most of the time this is invisible: if you swap two loads to different addresses and nobody wrote to either in between, both loads return the same values they would have anyway. The reordering happened, but no observer could tell.

The danger is only when an intervening write from another core changes a value between the two reordered accesses, so the out-of-order load grabs a value that program order would have forbidden.

Now you're ready: QED's color scheme reserves violet for this out-of-order <e execution. The question is never "did it reorder?" but "can the reordering be undone without changing a value?"
4Observability & forbidden values▾

This is QED's central concept. A reordering is observable when the values it produces cannot be produced by any MCM-compliant ordering of the same instructions — i.e., it produced a forbidden value. If the same final values are also reachable legally, the reordering is unobservable and therefore harmless, no matter how scrambled the execution order looked.

The paper grounds this in the lifetime of a value: a value's lifetime begins when a load-miss/prefetch brings it in and ends when an invalidation (or a local store, or an eviction) overwrites it. A load may be reordered freely within a value's lifetime, but never before the store that produced the value or past the invalidation that ends it.

Now you're ready: rust in this guide always means "forbidden value / observable." Observability is the filter that throws away the vast majority of reorderings as irrelevant.
5External events & the coherence interface▾

Other cores don't appear in your LSQ as instructions — they appear as external events crossing the coherence interface: an incoming invalidation (a proxy for some other core's store), an incoming external read request, plus the local side-effects of cache misses, prefetches, and evictions. These events are how another core's activity becomes visible to yours.

QED's second key move: among external events, only their ordering relative to your instructions matters — not which core they came from, because real implementations don't look at an event's origin (an out-of-order load is squashed on a matching invalidation regardless of who sent it). So one core's-eye view captures any number of other cores.

Now you're ready: honey means "external event." Treating other cores as anonymous events is what makes QED scale in the number of cores.
6Litmus tests & the "*check" lineage▾

The dominant prior approach (PipeCheck, CCICheck, RTLCheck, TriCheck, RealityCheck… the "*check" papers) runs a suite of small litmus tests — typically 4–8 memory accesses across 2–4 threads, e.g. Intel's tests — through a microarchitecture or its model and checks every interleaving against the MCM, often via µhb (microarchitectural happens-before) graphs. This catches every bug the tests expose.

The gap the paper hammers: bugs not exposed by the chosen tests are never caught. A later work generates exhaustive yet minimal tests, but bounded by n instructions across all threads and does not scale past n = 7 in practice — far below modern instruction windows of hundreds.

Now you're ready: QED positions itself as unbounded verification of the LSQ + coherence interface for out-of-order cores — the regime prior unbounded work (PipeProof, Kami) couldn't reach because they handle only in-order pipelines.
7Coherence vs. consistency (what QED assumes done)▾

Coherence is the single-address guarantee: writes to one location are serialized, and (if the model needs it) writes appear atomic. Consistency is the cross-address ordering guarantee the MCM specifies. They interact, but coherence verification is a deeply studied field of its own.

QED assumes coherence is already verified by standard methods, and that the front-end correctly handles register/control dependencies and marks instructions in the LSQ with their ordering requirements. It then verifies only that the LSQ and coherence interface don't reorder things in a way that breaks consistency.

Now you're ready: when QED "handles write atomicity," it means it checks the coherence interface preserves local event order per the model's atomicity rules — not that it re-verifies the coherence protocol itself.
8RTL, predicates, and checking against the design▾

RTL (register-transfer level) is the actual hardware description — here, BOOM v3, an open out-of-order RISC-V core written in Chisel/Scala (18K lines of Scala → ~500K lines of generated Verilog). A predicate is a narrow yes/no question about that RTL, e.g. "does the design squash an out-of-order load when an invalidation to its address arrives before it commits?"

QED's output for each instruction pair is a decision tree of predicates. Walking the tree by answering each predicate against the RTL lands you at a leaf labeled "compliant" or "violation." Predicates become reachability goals checkable with model-checking / SMT.

Now you're ready: steel means "RTL / predicate / decision tree." QED automates tree generation; mechanically evaluating predicates against RTL is demonstrated by hand and left to future work.
The whole story in one place

QED in a nutshell

The two scalability walls, the observability insight that knocks them down, and how it all connects to real RTL.

Verifying an out-of-order core against its MCM drowns in two combinatorial explosions. First, the number of in-flight memory instructions (say 100): they may reorder almost arbitrarily, so naïvely you face up to 100! orderings. Second, the number of cores: any number of them can interleave activity with yours. Prior unbounded methods (PipeProof, Kami) sidestep neither — they only handle far simpler in-order pipelines — and bounded test-based methods stall around 7 instructions across all threads.

QED's escape is observability: a hardware reordering matters if and only if it can produce a forbidden value — one no MCM-compliant ordering could yield. That single principle licenses two drastic reductions. (1) Directly-ordered pairs. Instead of all reorderings of all instructions, you need only the transitively non-redundant instruction pairs — those connected by an edge in the transitive reduction of the MCM's partial order. Any violation among many instructions must already show up as a violation of one such directly-ordered pair. (2) Anonymous external events. Other cores enter only as external events (invalidations, external reads), and only their ordering with your instructions matters, never their origin — so one core's view covers all cores.

Combine them and the case count collapses from 100! to O(m²e²) — pairs of m instruction types crossed with e external-event types (e well under 10), "a few thousands." For each pair, QED exhaustively builds an exploration tree of all relevant interleavings, then at each leaf attempts restoration: can the out-of-order (<e) execution be transformed back to a legal <m order without changing any value? A failed restoration is a violation. Restoration logic becomes a decision tree of simple predicates to evaluate against the RTL.

What's actually delivered: QED automatically generates exploration and decision trees for SC, TSO, and RISC-V WMO (24 / 34 / 112 trees; 103 / 113 / 305 leaves). It then demonstrates RTL checking by hand — verifying one substantial predicate (same-address load→load ordering) against BOOM v3, showing the violating leaf is unreachable. Full automation of predicate evaluation is explicitly left to future work.

🍯 Key takeaways

  • Observability is the whole lever: a reordering violates the MCM iff it produces a value no compliant ordering could (a forbidden value).
  • Reduction 1 — directly-ordered pairs: consider only instruction pairs in the transitive reduction of the MCM partial order, not all in-flight instructions. Scales in instruction count.
  • Reduction 2 — anonymous external events: capture other cores as ordered external events (invalidations / reads), ignoring origin. Scales in core count.
  • Case count drops to O(m²e²) ("a few thousands") instead of n!, independent of LSQ size and core count.
  • Restoration, not equivalence: try to move an out-of-order instruction back to a legal order unobservably; a failure flags a violation. One event type per instruction, one event per type, suffice.
  • Output = decision trees of predicates checked against RTL; generation is automated, predicate evaluation demonstrated on BOOM v3 and left to future work.
  • Trees built for SC / TSO / RISC-V WMO: 24 / 34 / 112 trees, 103 / 113 / 305 leaves — easily tractable.
Six everyday pictures

How to think about QED

Each picture stands alone — read them in any order. Every card ends with an honest note about where the picture stops being true, because the details matter for a verification reader.

The core idea · Observability

Did the magician really swap the cards?

A magician shuffles two cards behind their back. You only care whether the face-up result changed — not how the hands moved. If the same two faces could have come up by an honest shuffle, the trick is "fair," however acrobatic the hidden motion. QED treats reordering the same way: it ignores the scramble and asks only whether the produced values could also arise from a legal ordering. If yes, the reordering is unobservable and fine.

Where it breaks: a magic trick has one outcome; QED must be adversarial and consider all intervening external events that could make a forbidden value appear — it proves the implementation, not one execution.
Reduction 1 · Directly-ordered pairs

Skipping the redundant arrows

If a<b, b<c, and a<c are all required, you don't separately police a<c — guard the consecutive links a<b and b<c and the long-range one follows by transitivity. QED keeps only the edges in the transitive reduction of the MCM order. Any violation among many instructions must already break one of those direct edges, so checking adjacent pairs checks everything.

Where it breaks: in relaxed models the "adjacent" pair may have unrelated instructions physically between them; the reduction is over the partial order, not program-text adjacency, and the dashed long edge can split into several direct paths.
Reduction 2 · Anonymous events

The doorbell doesn't care who rang it

Your front door reacts to the bell the same way whether the visitor came from next door or across town. Likewise the LSQ squashes an out-of-order load on a matching invalidation regardless of which core sent it — the implementation never inspects origin. So QED models any number of other cores as a stream of anonymous external events and only worries about their ordering with your instructions.

Where it breaks: origin can still be implied by ordering between events — two invalidations from one thread are <m-ordered, from different threads maybe not — so QED must enumerate ordered, unordered, and both directions among events, even while ignoring identity.
Restoration · Undo the reorder

Rewinding a chess move

A player slides a piece, then realizes they touched the wrong one. They can take it back only if nothing on the board has changed in consequence — no piece captured, no clock-bound commitment. QED's restoration slides an out-of-order instruction back toward program order, allowed to pass <e events freely but never to cross an <m edge that would change a value. If it reaches a legal order untouched, no bug; if it gets stuck fused to the other instruction, it's a violation.

Where it breaks: chess take-backs are about board state; restoration is purely about values. An instruction and its <m-ordered event chain move together, and crossing an invalidation to the same address is forbidden because the value would change.
Output · Decision tree of predicates

A medical flowchart

A triage flowchart asks one yes/no question at a time — "fever? rash? recent travel?" — and each answer narrows you to the next, ending at a diagnosis. QED compiles each instruction pair into exactly such a decision tree of predicates about the RTL: "Are these two loads ever reordered? Is the out-of-order one squashed on an invalidation? Is the other a hit?" Walk the answers; some leaves say "compliant," some say "violation."

Where it breaks: a triage nurse answers from the patient in front of them; QED's predicates must be answered about all possible executions of the RTL via model checking / SMT — and automating that answering is the part the paper leaves to future work.
Scalability · Bounded vs unbounded

Proving a property vs. spot-checking

You can confirm a bridge holds by driving a few trucks across (a bounded test), or by analyzing the structure so it holds for any load (a proof). Bounded litmus-test methods cap out around 7 instructions across all threads — a few trucks. QED reasons about instruction pairs and event types, so its case count (O(m²e²)) doesn't grow with the 100s-deep instruction window or the number of cores: an argument, not a sample.

Where it breaks: the bridge "proof" is only as good as its assumptions. QED's guarantee is contingent on coherence and the front-end being verified elsewhere, and on each RTL predicate actually being checkable — its scope is the LSQ + coherence interface, not the whole core.
The full explanation

How it actually works

Section by section, following the paper's structure (§III), with its key diagrams rebuilt as visuals you can read at a glance and the algorithms in compact pseudocode.

1 · The two scalability walls

Naïvely modeling the hardware blows up the state space two ways. Wall 1 — in-flight instructions: with n≈100 reorderable memory instructions in a core, there are up to n! orderings. Wall 2 — cores: any number of other cores can interleave arbitrary code. Prior unbounded methods (PipeProof, Kami) only scale on simpler in-order pipelines; bounded test methods stall at n = 7 across all threads. QED targets the LSQ and coherence interface of an out-of-order core specifically, assuming the front-end and coherence are verified separately.

🎯
Scope, precisely: QED checks memory ordering in the load-store queue and the local order at the coherence interface. It assumes (a) register/control-flow front-end correctness and (b) coherence — write serialization and, if the MCM needs it, write atomicity — are already verified by other work.

2 · Reduction 1 — directly-ordered instruction pairs (§III-A)

An MCM imposes a partial order over memory instructions. QED's first observation: you only need the pairs connected by an edge in the transitive reduction of that order — the directly-ordered pairs. Formally, i <p j are directly-ordered if edge (i,j) survives transitive reduction (it removes every transitively-redundant edge while preserving the transitive closure).

Why this suffices (proof sketch, paraphrased): take a path l₀=i, l₁, …, lₙ=j in the reduction. If a multi-instruction execution violates i <m j, walk to the first node l_k that violates ordering w.r.t. i. Then l_k must already violate ordering with its predecessor l_{k-1} — a directly-ordered pair — because all orders up to that point are preserved. So any violation manifests as a direct-pair violation.

💡
For SC it's just transitivity: given a <p b <p c, SC needs a<m b, b<m c, a<m c. Preserving only the consecutive pairs a<m b and b<m c guarantees a<m c for free. Relaxed models generalize via the partial order, so a directly-ordered pair may have unrelated instructions physically between its two members.

3 · Reduction 2 — scalability to any number of cores (§III-B)

Other cores reach you only as external events (incoming invalidations, external reads). The key observation: origins don't matter, because implementations don't inspect them — an out-of-order load is squashed on a matching invalidation no matter who sent it. What does matter is the ordering among the events and between events and your instructions. Two invalidations may be <m-ordered (same thread, or a chain like st B <p st A in another core) or unordered (different threads / non-atomic stores).

4 · Observability & restoration (§III-C, §III-D)

The linchpin: an out-of-order instruction violates the MCM iff it is observable — its produced values cannot be produced by any compliant ordering. From the lifetime of a value (born on a load-miss/prefetch; dies on an invalidation, a local store, or — conservatively — an eviction), within a lifetime a load may be reordered freely (same value returns), but never before the producing store or past the overwriting invalidation.

Restoration operationalizes this. Three orders are in play: program order <p, the implementation's executed order <e, and a compliant global order <m. If <e can be transformed into a valid <m without changing any value, the system is compliant — even if <e looked illegal. Only <e edges may be reordered (they carry no global commitment); <m edges are fixed because they affect some thread's values. A crucial corollary: a <e ordering between two external events is subsumed by the <m ordering between them, so only <m needs checking.

# Restoration (Algorithm 2, paraphrased): trace T = B <x e1 … en <x A
do:
    B+ = longest <m-chain starting at B   # instr fuses with its <m-ordered events
    A+ = longest <m-chain ending at A
    if B+ == A+:  valid = false; return     # fused together → cannot restore = VIOLATION
    elif (B+ <e e):  rewrite (B+ <e e) → (e <e B+)   # slide B+ down past a <e event
    elif (e <e A+):  rewrite (e <e A+) → (A+ <e e)   # slide A+ up past a <e event
while not (B <e A)
valid = true                                # reached a legal order unobservably → OK
⚠
Why you can't just compare values to "some legal execution". QED deliberately inverts the usual check. Instead of asking "could these observed values also come from a legal ordering?", it asks "can this execution be restored to a legal ordering without changing its values?" The restoration framing is what makes the search local and finite — and it's why a load can't be restored past an invalidation to its own address (the value would change).

5 · Building exploration trees (§III-E)

For each instruction pair (with same or different addresses), QED enumerates a tree of execution traces: cache hits, misses, store-load forwarding, and every relevant external event, progressively adding events and their orderings. Two pruning facts keep it tiny: (1) one event type per instruction at a time — invalidations, misses, prefetches, evictions all yield the same observation, so they're considered separately, not together; (2) one event per type — if an instruction can(not) be restored past one event of a type, the same holds for many. Prefetches are treated as misses (order after the producing store); evictions like invalidations (end of a value's lifetime).

6 · From trees to predicates — checking the RTL (§III-F)

Each violating trace becomes a path in a decision tree of predicates — narrow binary questions answered against the implementation, each generated from the answers before it. For the SC ld-ld pair the paper gives the canonical sequence:

  1. Q1 — Does the design ever reorder loads to different addresses? If no, there can be no ld-ld violation. Done.
  2. Q2 — Is an out-of-order load (ld B) squashed on receiving an invalidation to B? If yes, the violating traces are unreachable. The architect maps this to an LSQ CAM search + a squash signal in the RTL.
  3. Q3 — If ld B is not squashed: is ld A a hit? If ld A hits, it can be restored above ld B (safe). If ld A misses, the violation is unavoidable.

To check a predicate, the architect annotates which RTL signals correspond to each atomic condition; QED converts the predicate into a reachability goal and asks whether any execution can satisfy it. A correctness assumption: QED trusts the building blocks (a CAM search circuit correctly searches the CAM); it only checks that the right search is issued and the right squash is raised. Because the relevant state is bounded (e.g., 100–300 LSQ entries), the paper expects SMT encodings to scale.

🧩
Two bug classes fall out: a violation found during restoration is a high-level design bug (e.g., a missing squash on some invalidation); a violation found during predicate evaluation is a low-level implementation bug (e.g., the squash exists but doesn't actually flush the LSQ).

7 · Coherence-interface verification (§III-G) & atomicity

Beyond the LSQ, the coherence interface itself could reorder events. QED verifies it (1) preserves the local bus order — the actual application of an invalidation must be locally ordered, though the acknowledgment may be sent out of order — and (2) sequences "collect invalidation acks → write to cache → allow other threads' early reads" per the MCM's write-atomicity rules. For multi-thread causality tests, write serialization/atomicity is covered by standard coherence verification while each thread's program order is checked by one of QED's exploration trees. Non-multi-copy-atomic models (e.g., PC) are handled because writes there aren't globally <m-ordered (only <e), yet fences/atomics still impose <m where needed — so QED's adversarial <m cases remain valid.

8 · The scalability claim, stated plainly (§III-E3)

With m memory-instruction types, there are at most 2m² pairs (same/different address). Each instruction is observed by e_i event types; let e = maxᵢ(eᵢ). Considering one event type per instruction gives O(e²) trees per pair, hence O(m²e²) total — and e is well under 10. Each tree has ≤ 4 items → ≤ 4! leaves; restoration of each leaf is linear in those (constant) items. Crucially, this is independent of the number of in-flight instructions and of cores. That independence — not a faster search over the old space, but a structurally smaller space — is the paper's central scalability argument.

🍯
The realized numbers: automatic generation yields 24 / 34 / 112 exploration-and-decision trees (of which 5 / 15 / 69 are trivial) and 103 / 113 / 305 leaves for SC / TSO / RISC-V WMO — confirming "several tens of trees remains easily tractable." (Paper Table II.)
Follow the trace, step by step

Worked examples

Four traces carried all the way through — the signature SC restoration, the BOOM v3 predicate the paper actually checks by hand, the reduction's pruning, and the tree-count arithmetic.

① The SC ld-ld restoration, by hand (Paper Fig. 1 & 4)

Main thread program order: ld A <p ld B (so SC requires ld A <m ld B). The core issues them out of order: ld B <e ld A. Is this a bug?

  1. By itself, no. If nothing overwrites A or B between the two loads, both return the values program order would give. The reorder is unobservable — restorable: slide ld B back down to just before its commit. ✓
  2. Introduce an external store to B, seen as an incoming inv B (a proxy for some other thread's st B; QED never asks which thread). Now consider its ordering with the loads.
  3. Case (i): inv B <e ld A only. Different addresses, no global commitment — ld A slides past inv B, then past ld B → program order restored. Still compliant. ✓
  4. Case (ii): inv B is forced before ld A in <m, together with inv A (or a ld A miss that binds a new external st A, where st B <p st A in the other thread). Now ld B <m inv B <m ld A: the early ld B is trapped behind an invalidation to its own block.
  5. Restoration fails: ld B can't move past inv B (its value would change), and ld A can't move below it either — they fuse. A fused pair is the algorithm's failure condition: SC VIOLATION ✗, the forbidden value is observable.
  6. The fix that makes it safe: squash ld B whenever an inv B arrives between its issue and commit. Then only the un-trapped (restorable) executions ever commit. Squashing on inv B alone — even without inv A — is conservative and correct [Gharachorloo et al. 1991].
🍯
Insight: the verifier never enumerated other threads or full executions. It reasoned about one pair plus anonymous invalidations, and turned "is this a bug?" into "can I restore it without changing a value?"

② The predicate QED actually checks against BOOM v3 (§V-B)

RISC-V WMO requires two loads to the same address to stay in program order even without a fence. The checked predicate: if ld1 A <p ld2 A and ld2 A <m inv A <m ld1 A, then squash ld2 A. BOOM v3 issues such loads out of order and squashes the later one on an intervening invalidation before commit. The job: show the "violation" leaf is unreachable.

  1. Map atomic conditions to RTL signals. Program order ld1 A <p ld2 A is implicit in ldQ (loads held/searched in program order). Per-instruction fields: addr, boolean executed, boolean observed. Invalidations arrive on io.release; the global ld_xcpt_valid triggers a squash.
  2. Encode the reorder: ld2 A <e ld1 A ≡ ld1.executed==0 ∧ ld2.executed==1 (the later load ran first).
  3. Encode the invalidation: ld2 A <m inv A ≡ ld2.addr==io.release.addr ∧ ld2.executed==1, so inv A searches the ldQ with A and sets matching ld2.observed=1.
  4. Convert each predicate to a reachability goal (variables existentially quantified):
    Goal1: i<j ∧ ldQ[i]=A ∧ ldQ[j]=A ∧ ldQ[j].executed ∧ ¬ldQ[i].executed
    Goal2: Goal1 ∧ io.release.addr=A ∧ ldQ[j].observed ∧ ¬ld_xcpt_valid
    Goal3: Goal2 ∧ ldQ[i].executed ∧ ¬ld_xcpt_valid
  5. Prove Goal3 unreachable. Goal3 (the red "invalid" leaf) demands the first load completes without a squash after the invalidation was already noted — but the RTL guarantees an out-of-order load is always squashed once an invalidation has been received. So Goal3 is an unreachable state → the predicate holds → BOOM v3 is correct for this case.
💡
This is the paper's concrete RTL evidence: one substantial, real predicate verified by hand against a 500K-line generated-Verilog out-of-order core. Goals are stated invertedly — proving the violating state unreachable is what certifies the predicate.

③ Why "one event per type" is sound (§III-E pruning)

A load can be observed by invalidations, misses, prefetches, and evictions. Doesn't ignoring most of them lose cases?

  1. Same observation per type. All the event types that can observe a given instruction produce the same observation about it, so considering several together is redundant — you consider each type, but separately, never simultaneously.
  2. One event suffices per type. If an instruction can be restored past one event of a type, it can be restored past many of them; if it cannot, one already blocks it. So a single representative event per type is enough.
  3. Prefetch folds into miss. An unrestricted prefetch could hide a miss that would have caused a squash; treating a prefetch as a miss (ordered after the store that starts the value's lifetime) cleanly closes that hole.
  4. Eviction folds into invalidation. An eviction ends a value's lifetime (later invalidations aren't delivered), so it's handled like an invalidation.
  5. Net effect: a trace needs only the instruction pair + at most one event per instruction → ≤ 4 items, ≤ 4! leaves, and the O(m²e²) bound holds.
🍯
Insight: the pruning isn't a heuristic — each step is justified by observability (same value ⇒ same observation), so exhaustiveness is preserved while the tree stays tiny.

④ Tallying the trees (Paper Table II)

How many cases does QED actually produce for the three models — and is "scalable" borne out?

  1. SC considers loads, stores, and synch (synch = stores for ordering), enumerating all pairs (same/different address) and all external events → 24 trees, of which 5 trivial (e.g., ld A <p st B: the store executes in-order after the load commits, by precise interrupts), 103 leaves.
  2. TSO adds atomics (3 instruction types) and relaxes only store→load order for different addresses → 34 trees, 15 trivial, 113 leaves.
  3. RISC-V WMO has 5 instruction types (load, store, atomic, load-reserved, store-conditional) plus 4-bit fence annotations and acquire/release → 112 trees, 69 trivial, 305 leaves.
  4. Read the trend: from SC to the much richer RISC-V WMO, the count grows from tens to ~hundred — "several tens of trees, easily tractable" — not the combinatorial blow-up of n!.
  5. Why it stays flat: none of these counts depend on LSQ depth or core count. Adding instructions to the window or cores to the system adds zero trees.
💡
That's the punchline of the scalability argument made concrete: the entire SC analysis is 24 trees / 103 leaves — a finite artifact you can generate once and check against any RTL, for any window size and core count.
Test yourself

Practice problems

Try each before opening the answer. Every one ends with the common misconception it's checking for.

Problem 1 · easy
A core reorders two loads to different addresses, and no store to either address occurs anywhere in the system during their window. Is this a memory-consistency bug? State the QED reasoning.
Show answer
No. By observability, a reordering is a violation only if it produces a value no compliant ordering could. With no intervening write, both loads return the same values either way, so the execution is restorable to program order without changing any value — unobservable, hence compliant.Misconception: that any out-of-order execution that "looks like" it violates program order is a bug. QED checks whether the reorder is observable (changes a value), not whether it happened.
Problem 2 · medium
An MCM requires a <m b, b <m c, and a <m c for three program-ordered accesses. Which of these pairs are "directly-ordered," and why does QED skip the third?
Show answer
Only (a,b) and (b,c). The edge (a,c) is transitively redundant — it's implied by a<m b and b<m c — so it's removed by the transitive reduction. QED checks only reduction edges because any violation of a<m c must already manifest as a violation of a<m b or b<m c (the proof walks the path to the first violating node, which breaks its direct predecessor edge).Misconception: that you must explicitly check every required ordering edge. Transitivity means guarding the reduction's edges guards the full closure.
Problem 3 · medium
QED claims it scales to "any number of cores." But an invalidation clearly comes from some other core. How can ignoring the origin be sound — and what about other cores does QED still have to track?
Show answer
Sound because implementations don't inspect an event's origin: an out-of-order load is squashed on a matching invalidation regardless of who sent it. So one core's-eye view of anonymous events captures all cores. What QED still tracks is the ordering among events and between events and local instructions — two invalidations from the same thread are <m-ordered, from different threads possibly not — so it enumerates ordered (both directions) and unordered cases.Misconception: that "ignore the origin" means "ignore the events." It means ignore identity but exhaustively consider event ordering, which is where another core's program order leaks in.
Problem 4 · hard
Standard SC checking asks: "can the observed values be reproduced by some MCM-permitted ordering?" QED instead asks whether the execution can be "restored." What concrete advantage does inverting the question buy, and where does restoration refuse to proceed?
Show answer
Restoration replaces a search over all legal orderings (which produced these values?) with a local, finite transformation: slide the out-of-order instruction back toward program order, allowed to pass <e events freely but forbidden to cross any <m edge that would change a value. It terminates when the instruction reaches a legal position (compliant) or fuses with the other instruction (violation). It refuses to move a load past an invalidation/store to the load's own address, because that would alter the load's value — exactly the boundary of a value's lifetime.Misconception: that restoration is just "find an equivalent legal trace." It's a constrained rewrite whose only legal moves are value-preserving, which is what makes it bounded and mechanical.
Problem 5 · medium
For the SC ld-ld pair, QED emits predicates Q1 (does the design reorder loads to different addresses?), Q2 (is an out-of-order load squashed on an inv to its address?), Q3 (is the other load a hit?). A design answers Q1=yes, Q2=no, Q3 = "ld A is a miss." Compliant or buggy?
Show answer
Buggy. Q1=yes means reordering is possible; Q2=no means the out-of-order ld B is not squashed on inv B; and with ld A a miss it can bind a new external value, so the early ld B is trapped behind inv B and cannot be restored — the violating leaf. (Had Q3 been "ld A is a hit," ld A could be restored above ld B and the design would be safe even without the squash.)Misconception: that not squashing is automatically a bug. It's only a violation when the other load misses; a hit provides an alternative safe restoration — which is exactly why Q3 exists.
Problem 6 · hard
In the BOOM v3 demonstration, the verification target is Goal3, and the paper proves it unreachable rather than proving some "good" state reachable. Explain why an unreachability obligation is the right formulation here.
Show answer
Each goal encodes the conjunction of conditions along the path to a violation leaf. Goal3 = "the two same-address loads reordered, the invalidation was received and noted (observed), yet the first load completed without a squash (¬ld_xcpt_valid)." That is precisely the forbidden state. Proving it unreachable proves no execution can reach the violation — i.e., the design is correct for this predicate. The goal is stated invertedly on purpose: correctness = the bad reachability goal has no satisfying execution.Misconception: that verification means showing the good path works. For a safety property you show the bad state is unreachable; a single reachable counterexample would be the bug.
Problem 7 · medium
QED considers invalidations, misses, prefetches, and evictions as observers of a load, yet says it needs only "one event type per instruction at a time" and "a single event per type." Justify both claims without losing exhaustiveness.
Show answer
All observer types of a given instruction yield the same observation about it, so combining several in one trace is redundant — consider each type, but separately, not together. And one event per type suffices because restorability past a type is monotone: if a load can be restored past one event of that type it can past many; if it cannot, one already blocks it. Prefetches are folded into misses (order after the producing store) and evictions into invalidations (end of lifetime), closing the remaining holes.Misconception: that fewer events = weaker checking. The reductions are observability-justified, so the smaller trees are still exhaustive over distinguishable behaviors.
Problem 8 · hard
A skeptic says: "QED claims to verify memory consistency, but it assumes coherence and the front-end are already correct — so it hasn't verified the whole processor." Is the criticism fair? What exactly has QED established?
Show answer
Partly fair as scope, not as soundness. QED deliberately verifies the LSQ and coherence interface — where most consistency bugs arise — and explicitly factors out (a) write serialization/atomicity, handled by mature coherence-verification work, and (b) register/control-flow dependencies, handled by front-end verification. Given those discharged elsewhere, QED's contribution is unbounded, scalable verification of memory ordering for out-of-order cores, which prior unbounded methods (PipeProof, Kami — in-order only) couldn't do. So it's a modular guarantee, not an end-to-end whole-CPU proof.Misconception: that modular scoping is a weakness. Decomposing against well-studied coherence/front-end verification is what makes the out-of-order LSQ problem tractable at all.
Quick answers

Frequently asked questions

Twelve questions a memory-model reader asks first. Tap any to expand.

Q1What does "observability" mean, exactly?▾
An out-of-order instruction is observable when the values its execution produces cannot be produced by any MCM-compliant ordering of the same instructions — i.e., it yielded a forbidden value. QED's core theorem: a reordering violates the MCM iff it is observable. Unobservable reorderings (same values reachable legally) are harmless regardless of how scrambled the execution order looks.
Q2How is "restoration" different from just checking value-equivalence?▾
Instead of asking "could these values come from some legal ordering?", QED asks "can this execution be transformed back into a legal ordering without changing any value?" Only <e (execution-order) edges may be reordered; <m (global) edges are fixed. The transformation slides an out-of-order instruction toward program order, fusing it with its <m-ordered event chain; if it can't reach a legal order (fuses with the other instruction), that's a violation. The framing makes the search local and finite.
Q3What are "directly-ordered pairs"?▾
The instruction pairs connected by an edge in the transitive reduction of the MCM's partial order — the non-redundant orderings. You don't separately check transitively-implied edges (if a<m b and b<m c are enforced, a<m c follows). Any violation among many instructions must already break one direct edge, so checking pairs checks everything. This is QED's reduction in the number of in-flight instructions.
Q4How can ignoring an event's originating core be sound?▾
Because real implementations don't look at origins — an out-of-order load is squashed on a matching invalidation no matter which core sent it. So QED models other cores as anonymous external events and only tracks their ordering with local instructions. That makes the analysis independent of the number of cores. (Ordering between events still encodes another core's program order, so QED enumerates ordered/unordered cases.)
Q5Why is the case count O(m²e²) and not n!?▾
m instruction types give at most 2m² pairs (same/different address); each instruction is observed by e event types, giving O(e²) trees per pair → O(m²e²) total, with e<10 ("a few thousands"). Each tree has ≤4 items (2 instrs + 2 events) → ≤4! leaves. Critically, none of this depends on the LSQ depth (the n! term) or core count — it's a smaller space, not a faster search.
Q6What is a "predicate," and what's a "decision tree of predicates"?▾
A predicate is a narrow yes/no question about the RTL (e.g., "is an out-of-order load squashed on an invalidation to its address before commit?"). Each instruction pair compiles to a decision tree: answer a predicate, branch, answer the next, … until a leaf labeled compliant or violation. Predicates are generated from the answers before them, and each becomes a reachability goal for model checking / SMT.
Q7Which models did the authors actually run, and how big are the results?▾
SC, TSO, and RISC-V WMO. Automatically generated tree counts (Table II): SC = 24 trees / 5 trivial / 103 leaves; TSO = 34 / 15 / 113; RISC-V WMO = 112 / 69 / 305. The growth from tens to a few hundred — even for rich WMO with fences and acquire/release — is the tractability claim in numbers.
Q8Was anything checked against real RTL?▾
Yes, but by hand and for one predicate. The authors verified same-address load→load ordering against BOOM v3 (an open out-of-order RISC-V core; 18K lines of Chisel/Scala → ~500K lines generated Verilog), mapping conditions to signals (addr, executed, observed, io.release, ld_xcpt_valid) and proving the violation goal unreachable. Full automated predicate evaluation is left to future work.
Q9What does QED assume is already verified?▾
Two things, deliberately factored out: (1) the pipeline front-end — register and control-flow dependencies — verified by front-end methods; and (2) cache coherence — write serialization and, if the MCM requires, write atomicity — verified by the large body of coherence-verification work. QED then verifies only memory ordering in the LSQ and the local order at the coherence interface.
Q10How is QED different from the "*check" / µhb-graph line of work?▾
The "*check" papers (PipeCheck, CCICheck, RTLCheck, TriCheck, RealityCheck) verify all interleavings of a finite litmus-test suite — catching only bugs those tests expose. Exhaustive-test generation is bounded and stalls near 7 instructions. PipeProof and Kami achieve unbounded proofs but only for simpler in-order pipelines. QED targets unbounded verification of the LSQ + coherence interface for out-of-order cores, scalable in both instruction count and core count.
Q11Does QED handle relaxed and non-atomic models?▾
Yes. Under relaxed/non-atomic models, store→store order isn't forced, so between two writes only <e ordering remains — but fences (within a thread) and atomic writes (across threads) still impose <m where needed, giving valid adversarial cases QED must consider. It even handles non-multi-copy-atomic models like PC, where writes aren't globally <m-ordered. QED proves an implementation correct, so it considers all such cases, not one test.
Q12What's explicitly left for future work?▾
The second component — mechanically evaluating predicates against RTL — is automatable but not implemented here. Future steps: architects annotate which RTL variables map to each predicate's conditions; then mature techniques (predicate-abstraction model checking, dataflow analysis, SMT) discharge the reachability goals. The authors automated the first component (generating exploration and decision trees) and demonstrated the second by hand.

Glossary, provenance & further reading

Every term in one place, then an honest accounting of what comes from the paper vs. this guide's framing — including the title/version caveat — and a path to go deeper.

Core terms
QEDThe paper's method: scalable verification of an RTL implementation against an MCM, via observability, directly-ordered pairs, anonymous external events, and restoration into decision trees of predicates.
ObservabilityA reordering is observable iff its values can't be produced by any compliant ordering. QED's theorem: a reordering violates the MCM iff it is observable. The filter that discards most reorderings.
Forbidden valueA value produced by an out-of-order execution that no MCM-compliant ordering of the same instructions could yield — the concrete signature of a consistency violation.
RestorationTransforming the executed order <e back into a compliant <m order without changing any value. Success = compliant; the two instructions fusing = violation. (Algorithm 2.)
Directly-ordered pairA pair i<p j whose edge survives the transitive reduction of the MCM's partial order. Only these need checking — reduction 1 (scales in instruction count).
External eventAn incoming invalidation, external read, miss, prefetch, or eviction — how other cores become visible. Only their ordering matters, not origin — reduction 2 (scales in core count).
Exploration treePer instruction pair, the exhaustive tree of traces (hits/misses/forwarding + interleaved events). ≤4 items per trace → ≤4! leaves.
Decision tree of predicatesThe verification output per pair: narrow binary questions about the RTL, walked to a compliant/violation leaf; each predicate becomes a reachability goal.
Supporting terms
MCMMemory consistency model: the contract on which values loads may return. Defined by (1) which program-order subsets are preserved and (2) write atomicity.
<p / <e / <mProgram order / executed (implementation) order / global memory order. QED restores <e toward a legal <m; <m edges are fixed.
LSQLoad-store queue: the structure in an out-of-order core that holds and reorders in-flight loads/stores. QED's main verification target.
Coherence interfaceThe seam between the core/local cache and global coherence. QED checks it preserves local bus order and the MCM's write-atomicity sequencing.
SC / TSO / RISC-V WMOThe three models QED generates trees for: strict; store→load relaxed; nearly all order relaxed except fences/acquire-release.
Value lifetimeBegins at a load-miss/prefetch, ends at an invalidation, a local store, or (conservatively) an eviction. A load may reorder within it but not past its boundaries.
Predicate / reachability goalA yes/no RTL question, encoded as "can any execution reach this state?" — discharged by model checking / SMT. Proving the violation goal unreachable certifies the predicate.
BOOM v3SonicBOOM, an open out-of-order RISC-V core in Chisel (≈18K Scala → ≈500K Verilog lines); the RTL QED checks one predicate against.
µhb graphMicroarchitectural happens-before graph — the representation used by the "*check" lineage QED contrasts itself with.
Transitive reductionThe minimal edge set with the same transitive closure as a DAG (Aho–Garey–Ullman 1972). The formal basis for "directly-ordered."
