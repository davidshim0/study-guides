Fixing the parity bottleneck in ZNS RAID

Zoned SSDs eliminate device-level garbage collection but make RAID tricky. The standard fix — metadata zones for parity updates — becomes the new bottleneck. Zebra rewrites the playbook by leveraging the device's own write buffer (ZRWA) for in-place parity updates.

📄 "Zebra: Efficient Redundant Array of ZNS SSDs…" · Jiang et al. 🎓 ZNS, RAID, and SSD internals assumed — primers cover the basics ⏱ ~25 min focused read
💾

The SSD's write buffer (ZRWA) in front of each zone is ideal for parity chunks that undergo repeated sequential overwrites — let parity live in the buffer until the stripe is full, then commit once. No metadata zone, no GC, no contention.

Nine concepts Zebra builds on

Each card unpacks one idea the paper assumes. Read in order the first time; skip what you already know.

1What ZNS is and why it exists▾

Traditional block SSDs hide unpredictable device-level garbage collection (GC) from the host. Zoned Namespace (ZNS) SSDs shift that to the host by exposing zones — log-structured append-only regions. The host writes sequentially to each zone's write pointer (WP); once a zone is full, the host resets it. No device-level GC, predictable performance.

Each zone has states: empty (WP at start), open/active (accepting writes), and full (all data written). Reset a zone to reclaim space. The ZNS spec limits active zones to bound SSD resources (e.g., 14 on WD ZN540).

Now you're ready: Zebra targets RAID built from multiple ZNS SSDs, presenting a single logical ZNS interface to applications — append-only semantics, no random overwrites (normally).
2What RAID 5 does — stripes, chunks, and parity▾

RAID 5 stripes data across k drives and stores 1 parity chunk on a k+1st drive to survive one drive failure. A stripe is k data chunks + 1 parity chunk, each chunk the same fixed size (e.g., 64 KB). Parity = XOR of all data chunks; if one drive fails, XOR the survivors to reconstruct.

A full-stripe write fills all k data chunks at once — calculate parity once, write once. A partial stripe write updates fewer than k data chunks, requiring a Partial Parity Update (PPU): read old parity, XOR in the delta, write new parity. PPUs are common (~75% of writes in datacenters, per the paper).

Now you're ready: The core challenge Zebra solves is PPUs on ZNS, where parity chunks can't be overwritten in place (because ZNS is append-only).
3The PPU problem on ZNS RAID▾

When a small write comes to ZNS RAID (size < stripe size), the parity chunk must be updated. In traditional block RAID, you overwrite the parity chunk in place. But ZNS forbids overwrites — data only appends at the write pointer. So where does the new parity go?

RAIZN's solution: create dedicated metadata zones that append each PPU as a log entry (with a header). Once a metadata zone fills, RAIZN runs RAID-level GC to reclaim obsolete PPUs. The paper shows metadata zones become the bottleneck: they aggregate PPUs from all data zones, trigger GC, and waste bandwidth on log headers.

Now you're ready: Zebra's key insight is that ZRWA (see next) lets you overwrite parity chunks in place within the SSD's write buffer, eliminating metadata zones entirely.
4ZRWA — the Zone Random Write Area feature▾

Modern SSDs use a RAM-based write buffer in front of flash to batch small writes into page-sized chunks. ZRWA (Zone Random Write Area) exposes that buffer to the host via a sliding window of LBAs (e.g., 1 MB on WD ZN540). Within ZRWA, the host can issue random, concurrent overwrites — violating the append-only rule temporarily.

When writes exceed the ZRWA window or an explicit commit is issued, the buffered data flushes to flash and ZRWA slides forward. After flush, those LBAs become immutable (ZNS semantics restored). ZRWA size ≥ typical RAID chunk size (64 KB), so a parity chunk fits comfortably inside.

Now you're ready: Zebra sets chunk size ≤ ZRWA size, ensuring every parity chunk resides in its zone's ZRWA. PPUs overwrite the parity chunk in the buffer; when the stripe fills, Zebra commits ZRWA, freezing parity to flash.
5Write pointer (WP) and crash consistency▾

Each zone's write pointer (WP) marks the next writable LBA. ZNS guarantees a write succeeds only if its starting LBA matches WP. After a successful write, WP advances by the write size. On non-ZRWA zones, WP points to the tail of written data. On ZRWA zones, WP points to the start of the ZRWA window (not the tail of actual data).

After a machine failure (power-off), DRAM is lost but the SSD's WP persists (with Power-Loss Protection, PLP). For ZRWA zones, the device reports WP but not which LBAs inside ZRWA actually hold data. Zebra must reconstruct the "Real WP" (the true tail of written data) to avoid treating unwritten regions as valid.

Now you're ready: Zebra uses out-of-band (OOB) metadata — 1 bit per LBA stored in flash spare area — to mark which LBAs were written, enabling fast Real WP recovery (§III-B).
6RAIZN baseline — metadata zones & GC overhead▾

RAIZN (prior work) allocates one metadata zone per ZNS SSD to log PPUs from all logical zones. Each PPU entry has a log header (metadata describing the parity range). When a stripe completes, old PPU entries become obsolete. RAIZN runs RAID-level GC to reclaim space in metadata zones, scanning for valid vs. obsolete entries.

The paper identifies three performance penalties: (1) contention — all data zones funnel PPUs to one metadata zone, creating a bandwidth bottleneck; (2) GC overhead — periodic GC cuts throughput by up to 15%; (3) write amplification — log headers inflate writes (WAR under 4 KB workloads, per §II-B3).

Now you're ready: Zebra eliminates metadata zones (no GC, no log headers) by storing parity in ZRWA, where overwrites are native. This "all-to-all" design (each data zone writes PPUs to its own parity chunk in ZRWA) avoids RAIZN's bottleneck.
7Large-zone vs. small-zone ZNS models▾

Large-zone ZNS (e.g., WD ZN540): each zone spans all flash channels/dies, maximizing bandwidth but sharing bandwidth across zones. Zone size ~1 GB. Small-zone ZNS (e.g., Samsung PM1731a): each zone occupies one channel or die, providing hardware isolation (a zone's bandwidth is strictly its channel's bandwidth). Zone size ~96 MB.

On small-zone ZNS, RAIZN's metadata zone is confined to one channel's bandwidth, making the bottleneck more severe. The paper reports a 76% throughput drop on small-zone ZNS (§II-B1) because metadata zones can't exploit parallelism. Zebra, using ZRWA across all zones, scales linearly with active zone count.

Now you're ready: Zebra is evaluated on both models (§IV), showing gains from 6.4%–51% on large-zone and up to 4.2× on small-zone ZNS — largest where RAIZN's contention is worst.
8Metrics — throughput, latency, write amplification▾

Throughput (MB/s): sustained data rate under sequential/random writes. Average latency: mean time from request issue to completion. P99 latency: 99th-percentile latency — captures tail behavior. Write Amplification Ratio (WAR): bytes written to device ÷ bytes written by application (includes metadata, GC, log headers).

The paper shows Zebra improves throughput by 6.4%–51% on large-zone ZNS and up to 4.2× on small-zone ZNS, while cutting average latency by 17%–24%. WAR drops by 19%–30% because Zebra avoids RAIZN's GC and log headers (§IV-D).

Now you're ready: These metrics confirm Zebra's core claim — leveraging ZRWA removes the metadata zone bottleneck, reducing both latency and write amplification.
9Recovery — disk failure vs. machine failure▾

Disk failure: one ZNS SSD becomes inaccessible. RAID 5 survives via parity — reconstruct lost data by XORing survivors. Zebra's disk-failure recovery is standard: read all stripes, recalculate missing chunks, write to replacement drive (~899 s for 1 TB, per §IV-E). Machine failure: power-off event, DRAM lost but SSDs intact (assuming PLP).

After machine failure, Zebra restores WPs by querying device state, scanning OOB metadata to locate Real WP, then synchronizing data and parity across zones (§III-C2). RAIZN must traverse entire metadata zones to find valid PPUs (93% of recovery time). Zebra's recovery is 14.2× faster because it only scans ZRWA windows, not full zones (§IV-E).

Now you're ready: Zebra's lightweight OOB-based WP tracking enables fast, crash-consistent recovery without the full-zone scans RAIZN requires.
Quick glossary of key acronyms
ZNSZoned Namespace — an SSD interface with append-only zones instead of random-writable blocks.
ZRWAZone Random Write Area — a sliding buffer window in a ZNS zone that permits overwrites (before commit to flash).
PPUPartial Parity Update — updating parity when fewer than k data chunks are written (requires read-modify-write on parity).
WPWrite Pointer — the next writable LBA in a zone. On ZRWA zones, WP points to the start of ZRWA; Zebra tracks "Real WP" (tail of actual data).
OOBOut-Of-Band metadata — spare area per flash page used to store 1-bit flags indicating if an LBA was written (for Real WP recovery).
RAIZNPrior ZNS RAID system that uses metadata zones to log PPUs; Zebra's performance baseline.
The whole story in one place

Zebra in a nutshell

Problem, solution, and how it all connects — grounded in the paper's findings.

Zoned SSDs (ZNS) eliminate device-level garbage collection by exposing append-only zones to the host. When multiple ZNS SSDs form a RAID array, writes stripe data across drives and calculate parity for fault tolerance. A Partial Parity Update (PPU) — updating parity without filling the full stripe — is problematic because ZNS forbids overwriting the parity chunk in place.

RAIZN's fix: allocate dedicated metadata zones on each ZNS SSD to log PPUs. This introduces three bottlenecks: (1) contention — all logical zones funnel PPUs to one metadata zone per drive, creating a bandwidth gap (185 MB/s on WD ZN540, per Figure 3a); (2) RAID-level GC to reclaim obsolete PPU entries, cutting throughput by up to 15% (Figure 3b); (3) write amplification from log headers. On small-zone ZNS with per-channel isolation, the metadata zone's bandwidth is strictly limited to one channel, causing a 76% throughput drop (§II-B1).

Zebra leverages ZRWA — the SSD's RAM-based write buffer exposed as a sliding window of overwritable LBAs. Key insight: ZRWA's properties (random overwrites within a window, implicit commit when the window slides) match PPU's I/O pattern (repeated sequential overwrites of a parity chunk until the stripe completes). By setting chunk size ≤ ZRWA size (e.g., 64 KB chunk, 1 MB ZRWA), each parity chunk fits in its zone's ZRWA. PPUs overwrite the parity chunk in the buffer; when the stripe fills, Zebra commits ZRWA, freezing parity to flash. No metadata zones, no GC, no log headers.

Crash consistency: ZRWA zones report WP at the start of the buffer, not the tail of written data. Zebra uses OOB metadata (1 bit per LBA in flash spare area) to mark which LBAs were written, enabling fast "Real WP" recovery after a machine failure (14.2× faster than RAIZN, which scans entire metadata zones). Recovery: Disk failure handled via standard parity reconstruction. Machine failure: query device WPs, scan OOB to locate Real WPs, synchronize data/parity across zones.

Results (§IV): On large-zone ZNS (WD ZN540), Zebra improves throughput by 6.4%–51% over RAIZN under 4–32 KB writes, cuts average latency by 17%–24%. On small-zone ZNS (emulated Samsung PM1731a), throughput gains reach 4.2× because Zebra avoids the single-channel metadata bottleneck. For RocksDB fillsync workload, throughput increases up to 5.4× on small-zone ZNS. Hardware overhead: <1% area (OOB bits), nothing on critical path.

🍯 Key takeaways

  • ZNS RAID's core challenge: PPUs require parity overwrites, but ZNS is append-only.
  • RAIZN's metadata zones create a bottleneck (contention, GC, log headers), especially severe on small-zone ZNS with per-channel isolation.
  • ZRWA (write buffer) naturally fits PPU's I/O pattern: repeated sequential overwrites within a sliding window.
  • Zebra: chunk size ≤ ZRWA size → parity lives in buffer, overwrites in place, commits once per stripe. No metadata zones.
  • OOB metadata (1 bit/LBA) enables crash-consistent Real WP recovery, 14.2× faster than RAIZN.
  • Throughput gains: 6.4%–51% (large-zone), up to 4.2× (small-zone); latency cut by 17%–24%; WAR down 19%–30%.
  • Overhead: <1% area (OOB bits only), nothing on critical path.
Seven everyday pictures

How to think about Zebra

Each picture stands alone. Read in any order; every card ends with where the analogy breaks.

ZRWA as a scratchpad · Core idea

Editing a draft in a notebook

You jot notes in pencil on a scratch page, erasing and rewriting as ideas evolve. Only when the paragraph is final do you commit it in ink to the permanent notebook. ZRWA is that scratch page: the SSD's RAM buffer lets you overwrite the parity chunk repeatedly (each PPU erases the old value) until the stripe completes. Then commit freezes it to flash (ink), and ZRWA slides to the next page (next parity chunk).

Where it breaks: a real notebook has unlimited pages; ZRWA is a fixed-size sliding window (e.g., 1 MB). Exceed the window and the buffer flushes automatically, freezing data to flash whether you wanted to or not. Zebra ensures chunk size ≤ ZRWA size to avoid premature flushes.
Metadata zone bottleneck · RAIZN problem

One checkout lane for all shoppers

A supermarket with ten aisles but only one checkout lane. Shoppers (data zones generating PPUs) queue at the single register (metadata zone), causing delays. RAIZN funnels all PPUs to one metadata zone per drive — the paper shows a 185 MB/s bandwidth gap (Figure 3a). Zebra is like giving each aisle its own register: every data zone's parity chunk lives in its own ZRWA, no shared bottleneck.

Where it breaks: a supermarket can add more lanes; ZNS has limited active zones (~14), so adding metadata zones reduces data zones. RAIZN+ tries this (one metadata zone per logical zone) but throughput still lags Zebra (Figure 18b) because fewer data zones limit parallelism.
GC overhead · RAIZN penalty

Shredding old receipts

You file receipts in a binder. As items are returned or replaced, old receipts become obsolete. Periodically you shred them to reclaim space. RAIZN logs each PPU in the metadata zone; when a stripe completes, old PPU entries are obsolete. RAIZN runs RAID-level GC to scan the log and reclaim space — the paper shows this cuts throughput by up to 15% (Figure 3b). Zebra never generates obsolete entries because parity is overwritten in place in ZRWA.

Where it breaks: shredding is free; RAIZN's GC consumes device bandwidth and CPU cycles. And unlike a binder you can expand, RAIZN's metadata zone is a fixed size, so GC must run frequently under high PPU workloads.
Write amplification · Log header cost

Mailing a postcard with a bulky envelope

You write a short note (the PPU) but must attach a full-page cover sheet (log header) for tracking. For small notes, the envelope is bigger than the content. RAIZN's PPU log entries each carry a header (metadata describing the parity range) — under 4 KB writes, the paper reports inflated WAR (§II-B3). Zebra has no log headers because parity overwrites in ZRWA leave no trace of old values.

Where it breaks: a postcard envelope is optional; RAIZN's log headers are mandatory for recovery (to locate PPU ranges after a crash). Zebra avoids them by using OOB metadata (1 bit/LBA) instead, which is far more compact.
ZRWA sliding window · Implicit commit

A spotlight on a moving stage

A theater spotlight illuminates a 10-foot section of the stage (the ZRWA window). Actors (writes) can move freely within that section, repositioning (overwriting) at will. When an actor steps beyond the spotlight's edge, the spotlight slides forward automatically, freezing the previous section in darkness (committing to flash). ZRWA's implicit commit works the same: write beyond the window and the buffer flushes, advancing the window by ZRWAFG granularity (e.g., 16 KB, the flash page size).

Where it breaks: a spotlight moves smoothly; ZRWA moves in coarse chunks (ZRWAFG = 4 LBAs = 16 KB on WD ZN540). Write one LBA past the window and the entire ZRWAFG flushes, even if some LBAs in that chunk are unwritten. That's why Zebra needs OOB metadata to mark which LBAs are valid.
OOB metadata · Real WP recovery

Sticky notes on calendar pages

Your wall calendar has sticky notes on some dates (events scheduled) and blank spaces on others. After a power-off, you scan the sticky notes to find the last scheduled date (Real WP). Zebra's OOB metadata works the same: 1 bit per LBA in flash spare area, set to 1 when written. After a crash, scan OOB bits from the device-reported WP (start of ZRWA) forward; Real WP is the tail of consecutive 1s. Fast and lightweight (~1 bit/LBA, vs. RAIZN's full-zone scan).

Where it breaks: sticky notes are free; OOB metadata consumes flash spare area (8–64 B per LBA available, Zebra uses 1 bit). But the paper shows no performance degradation from OOB writes, and the space cost is negligible (<1% of total, §IV-F).
All-to-all architecture · Zebra advantage

Three separate water tanks

A house with three faucets, each feeding its own private water tank (ZRWA zones). No contention, no shared pipe. RAIZN is like all faucets feeding one shared tank (metadata zone) — a single tank's capacity limits total throughput. Zebra's "all-to-all" design: each data zone writes PPUs to its own parity chunk's ZRWA, exploiting parallelism. On small-zone ZNS where each zone has dedicated hardware (one channel/die), this scales linearly — hence the 4.2× throughput gain (§IV).

Where it breaks: water tanks can be resized; ZRWA size is hardware-fixed (e.g., 1 MB on WD ZN540, 64 KB on Samsung PM1731a). Zebra must ensure chunk size ≤ ZRWA size. If chunk size exceeds ZRWA, the parity chunk won't fit in the buffer and Zebra's approach fails.
The full explanation

How Zebra actually works

Section by section, grounded in the paper's mechanisms and measured results.

1 · Architecture — ZRWA zones for parity chunks

Zebra is a ZNS RAID volume manager presenting a single logical ZNS interface to applications (e.g., F2FS, RocksDB). Each logical zone stripes data across k+1 physical zones (k data chunks + 1 parity chunk per stripe). Unlike RAIZN, Zebra has no metadata zones. All physical zones are opened in ZRWA mode, and parity chunks coexist with data chunks in the same zones (different stripes may have parity on different drives — e.g., left-asymmetric layout).

Key constraint: chunk size ≤ ZRWA size (e.g., 64 KB chunk, 1 MB ZRWA on WD ZN540). This ensures each parity chunk fits entirely within its zone's ZRWA window. When a partial stripe write arrives, Zebra appends data to the data chunk and overwrites the parity chunk in ZRWA. PPUs accumulate in the buffer (repeated overwrites) until the stripe fills; then Zebra commits ZRWA, freezing parity to flash and advancing to the next stripe.

Host-side data structures: in-memory stripe cache (one stripe per active logical zone, max 3.5 MB in experiments), logical zone WPs, and zone states. Device-side: all zones in ZRWA mode, parity chunks stored inline with data (no separate metadata zones).

🌱
Concretely: WD ZN540 has 1077 MB zones, 1 MB ZRWA, 16 KB ZRWAFG (Table I). Zebra uses 64 KB chunk size (fits in ZRWA). On Samsung PM1731a (96 MB zones, 64 KB ZRWA, 32 KB ZRWAFG per Table I), Zebra would use ≤64 KB chunks. The paper evaluates both large-zone (WD) and small-zone (emulated PM1731a) setups.

2 · Handling PPUs — in-place overwrites in ZRWA

Figure 7 in the paper shows a (3+1) RAID 5 example. Chunk size = 16 LBAs (64 KB), ZRWA size = enough to hold the parity chunk. Five write requests (W0–W4) arrive, each half-chunk or full-chunk size:

  1. W0 (8 LBAs): append to D3 (data zone 0), calculate partial parity P1 (8 LBAs), write P1 to ZRWA of parity zone 3. The valid extent of parity data moves to midpoint of P1.
  2. W1 (8 LBAs): append rest of D3, overwrite second half of P1 in ZRWA. D3 chunk now full; ZRWA of zone 0 implicitly commits (D3 freezes to flash).
  3. W2 (16 LBAs): append full chunk to D4 (zone 1), overwrite P1 in ZRWA (no ZRWA movement because P1 still within window). Zone 1's ZRWA commits D4.
  4. W3, W4 (8 LBAs each): append to D5 (zone 2), overwrite P1 in ZRWA twice. Stripe completes when D5 fills; ZRWA of parity zone commits P1 to flash.

I/O pattern of P1: repeated sequential overwrites within the 16-LBA chunk. ZRWA permits this natively; no append-only violation because ZRWA is a special mode. Once the stripe is full, Zebra issues an explicit commit (or lets implicit commit happen when the next stripe's parity chunk starts), freezing P1 to flash. Result: no metadata zone, no log headers, no GC.

🍯
Insight: the repeated-overwrite property of PPUs (each new data chunk XORs into the same parity chunk) matches ZRWA's overwrite semantics. By constraining chunk size ≤ ZRWA size, Zebra ensures parity always fits in the buffer.

3 · Real WP recovery — OOB metadata for crash consistency

After a machine failure (power-off), DRAM is lost but SSDs with Power-Loss Protection (PLP) retain data. Zebra must restore WPs of all logical zones. The challenge: ZRWA zones report WP at the start of ZRWA, not the tail of actual data (Figure 8). ZRWA can also move in coarse chunks (ZRWAFG = 4 LBAs = 16 KB), so some LBAs within ZRWA may be unwritten. Treating the ZRWA tail as Real WP would incorrectly mark unwritten regions as valid.

Solution: OOB metadata (out-of-band area in flash spare area, 8–64 B per LBA). Zebra uses 1 bit per LBA (0 = unwritten, 1 = written). When writing data, Zebra sets the corresponding OOB bit to 1. On recovery:

  1. Query device for WPs (start of ZRWA) via zns-zone-management-recv.
  2. Scan OOB bits forward from WP until the first 0 bit. The last consecutive 1 is the Real WP.
  3. For each logical zone (spanning multiple physical zones), synchronize Real WPs: take the first Real WP that hasn't reached the end of its data chunk (i.e., data chunk not full).
  4. Discard invalid data beyond Real WP (belongs to unfinished requests), recalculate parity from valid data chunks.

Recovery time: 14.2× faster than RAIZN (Figure 17b) because Zebra scans only ZRWA windows (~1 MB each), while RAIZN must traverse entire metadata zones (~1 GB) to find valid PPU log entries (93% of RAIZN's recovery time, per §IV-E).

💡
Edge case: if both a disk failure and machine failure occur simultaneously, Zebra uses OOB metadata from parity chunks in surviving disks to infer the Real WPs of zones in the failed disk. OOB area for parity chunks extends from 1 bit to 1 Byte (still <1% overhead), recording the latest write request that updated each parity block.

4 · Zero storage space reduction after recovery

After a machine failure, some stripes may be partially written (e.g., data chunks D0, D1 written but D2 not). RAIZN cannot reuse the partially-written stripe because it can't overwrite the dirty chunks (ZNS append-only constraint), so it skips the stripe and writes to the next one — permanent storage space reduction (Figure 9a). RAIZN must maintain a mapping table to track missing space, adding memory overhead.

Zebra (Figure 9b): restore Real WP to the WP of the first non-full data chunk, clean dirty data on other zones by overwriting with zero (ZRWA permits overwrites), recalculate parity. When a new write arrives, Zebra overwrites the cleaned regions in ZRWA (in-place), reusing the stripe. No space reduction, no mapping table. This is only possible because ZRWA supports overwrites.

🍯
ZRWA enables both crash-consistent recovery (via OOB) and space-efficient recovery (via in-place overwrites of dirty regions). RAIZN cannot recover in place because it lacks ZRWA.

5 · Accelerating memory operations — AVX parallelism

Parity calculation (XOR) and memory copy (stripe cache updates) consume 11.6% and 5.6% of latency, respectively (§III-D). With NVMe SSDs providing ~10 μs I/O latency, CPU overhead becomes non-negligible. Zebra uses AVX (Advanced Vector Extensions) instructions to parallelize XOR and memory copy across 256-bit (or 512-bit on newer CPUs) vector registers, saturating memory bandwidth.

Result: latency percentage of memory copy and parity calculation drops by 6.1% and 10%, respectively (Figure 16). AVX is widely supported by modern CPUs (Intel since Sandy Bridge, AMD since Zen 4 with AVX-512).

6 · Handling flush commands and PLP

Applications (e.g., file systems) issue flush commands to ensure durability. On ZNS SSDs with PLP (Power-Loss Protection), data is durable once transferred to the device (ZRWA buffer is backed by capacitors or battery). Zebra moves Real WP immediately after a write finishes. On ZNS SSDs without PLP, Zebra postpones Real WP movement until flush completion, ensuring crash consistency.

Worked examples

Four traces using the paper's real measurements and configurations.

① PPU sequence on a (3+1) RAID 5 stripe

Chunk size = 64 KB (16 LBAs). ZRWA size = 1 MB (256 LBAs). Three partial writes arrive: W0 = 32 KB, W1 = 32 KB, W2 = 64 KB. Trace the parity chunk evolution in ZRWA.

  1. W0 arrives (32 KB to D0): append W0 to data zone 0, calculate partial parity P0_v1 = W0 ⊕ 0, write P0_v1 (32 KB) to parity zone's ZRWA. The valid extent of parity data is at LBA 8 (within the 16-LBA chunk).
  2. W1 arrives (32 KB to D1): append W1 to data zone 1. Old parity P0_v1, new data W1. Calculate P0_v2 = P0_v1 ⊕ W1 ⊕ 0. Overwrite P0_v2 (still 32 KB) at LBA 0–8 in ZRWA (in-place). Valid extent unchanged (still LBA 8).
  3. W2 arrives (64 KB to D2): append W2 to data zone 2. Calculate P0_v3 = P0_v2 ⊕ W2 ⊕ 0. Overwrite P0_v3 (now 64 KB, full chunk) at LBA 0–16 in ZRWA. Valid extent moves to LBA 16 (end of chunk).
  4. Stripe completes: all data chunks (D0, D1, D2) are full. Commit ZRWA: P0_v3 flushes to flash (immutable), ZRWA slides to the next parity chunk P1.
🍯
Three PPUs → three overwrites of the same parity chunk in ZRWA. RAIZN would append three PPU log entries (each with header) to metadata zone, consuming 3× the space plus log overhead. Zebra commits the final parity once to flash after accumulating updates in ZRWA.

② Real WP recovery after a power-off event

ZRWAFG = 16 KB (4 LBAs). A zone in ZRWA mode has WP at LBA 100 (start of ZRWA). A 2-LBA write was issued to LBA 100, but the machine failed mid-write. Recover Real WP using OOB.

  1. Device-reported WP: query via zns-zone-management-recv → WP = LBA 100 (start of ZRWA).
  2. 2-LBA write at LBA 100: write successfully completes at LBA 100–101. OOB bits for LBA 100–101 are set to 1. ZRWA has not yet moved because the write is within the window.
  3. OOB scan: read OOB metadata from LBA 100 forward. OOB[100] = 1, OOB[101] = 1, OOB[102] = 0. The last consecutive 1 is at LBA 101. Real WP = LBA 102 (tail of valid data).
  4. Outcome: LBA 100–101 contain valid data. Logical zone WP points to LBA 102.
💡
Without OOB, Zebra would have no way to determine whether LBA 100–101 were actually written (since the device WP only indicates ZRWA start, not the valid data extent). OOB metadata (1 bit/LBA) eliminates this ambiguity.

③ Throughput improvement on small-zone ZNS

Emulated Samsung PM1731a (96 MB zones, 64 KB ZRWA, per-channel isolation). (6+1) RAID 5, chunk size 64 KB. 4 KB sequential write workload, 6 active logical zones. Measure throughput of RAIZN vs. Zebra.

  1. RAIZN setup: 6 data zones + 6 metadata zones (one per logical zone). Each metadata zone occupies one channel. Under 4 KB writes, PPUs flood metadata zones.
  2. Bandwidth limit: metadata zones are confined to one channel each. Total metadata bandwidth = 6 channels. But 6 data zones also need channels for data writes. Metadata zones become the bottleneck.
  3. RAIZN throughput: ~200 MB/s (Figure 12, from the paper's emulation). Metadata zones can't keep up with data zones' PPU rate.
  4. Zebra setup: 6 data zones, no metadata zones. Each data zone's parity chunk lives in its own ZRWA (parallel). All 7 zones (6 data + 1 parity) can write concurrently across 7 channels.
  5. Zebra throughput: ~840 MB/s (Figure 12). 4.2× improvement over RAIZN because ZRWA eliminates the metadata bottleneck.
🍯
On small-zone ZNS, per-channel isolation makes RAIZN's metadata zone bottleneck severe. Zebra's "all-to-all" design (ZRWA in each zone) scales linearly with active zone count.

④ Write amplification ratio under 4 KB workload

(3+1) RAID 5, chunk size 64 KB. Workload: 4 KB sequential writes (all PPUs). RAIZN logs each PPU with an 8 B header. Calculate WAR for RAIZN vs. Zebra.

  1. Application write: 4 KB data chunk → 4 KB parity update.
  2. RAIZN: writes 4 KB data + 4 KB parity log entry (4 KB parity + 8 B header) to metadata zone. Total device writes = 4 KB (data) + 4.008 KB (parity log). WAR ≈ 8.008 / 4 = 2.002.
  3. Over 16 such writes (filling the chunk): RAIZN appends 16 PPU log entries (each 4.008 KB) to metadata zone = 64.128 KB total. Application wrote 64 KB (16 × 4 KB). WAR = 64.128 / 64 ≈ 1.002. But wait — GC must reclaim 15 obsolete entries. During GC, RAIZN reads the metadata zone (64.128 KB), compacts to 1 valid entry (4.008 KB), writes it back. Total writes = 64.128 KB (initial) + 4.008 KB (compacted) = 68.136 KB. WAR = 68.136 / 64 ≈ 1.065 (per GC cycle). The paper's sustained 4 KB results show higher WAR due to log churn across many stripes.
  4. Zebra: writes 4 KB data, overwrites 4 KB parity in ZRWA (no log header, no GC). Over 16 writes: 64 KB data written, 4 KB parity overwritten 16 times in buffer (no device writes), then 4 KB parity committed once to flash. Total device writes = 64 KB (data) + 4 KB (parity) = 68 KB. WAR = 68 / 64 = 1.063.
  5. Result: Zebra's simple example shows ~1.063 WAR. RAIZN's per-stripe calculation gives ~1.065 after one GC cycle, but the paper's Figure 17a reports RAIZN's average WAR across various workloads and shows 19%–30% reduction with Zebra, reflecting more complex real-world behavior including GC overhead accumulation.
🍯
The smaller the write request, the worse RAIZN's WAR (log header overhead dominates). Zebra's in-place overwrites in ZRWA eliminate both log headers and GC writes, improving WAR by the percentages reported in the paper.
Test yourself

Practice problems

Try each before opening the answer. Every one ends with the misconception it checks.

Problem 1 · easy
A ZNS SSD has ZRWA size = 512 KB. A RAID 5 system wants to use Zebra. What is the maximum chunk size Zebra can support, and why?
Show answer
512 KB (or smaller). Zebra's core requirement: chunk size ≤ ZRWA size, so the entire parity chunk fits in the ZRWA window. If chunk size = 1 MB > 512 KB ZRWA, the parity chunk won't fit in the buffer, forcing premature commits and breaking Zebra's design. Set chunk = 512 KB (or smaller) to ensure parity chunks always reside in ZRWA.Misconception: that chunk size can exceed ZRWA size. If it does, PPUs can't overwrite the parity chunk in place because part of it falls outside ZRWA, defeating Zebra's entire approach.
Problem 2 · medium
RAIZN uses one metadata zone per ZNS SSD. Under a 4 KB write workload on WD ZN540 (large-zone ZNS), 6 data zones generate PPUs. The paper reports a 185 MB/s bandwidth gap between metadata and data zones. Explain the cause and how Zebra avoids it.
Show answer
RAIZN funnels all 6 data zones' PPUs to one metadata zone. Data zones collectively achieve ~730 MB/s (6 zones × ~122 MB/s each, Figure 3a). But the single metadata zone peaks at ~545 MB/s, creating a 185 MB/s gap. Cause: contention — the metadata zone can't handle the aggregate PPU rate from all data zones. Zebra avoids this by eliminating metadata zones. Each data zone's parity chunk resides in its own physical zone's ZRWA, so PPUs are processed in parallel (6 zones can write PPUs concurrently). No shared bottleneck.Misconception: that adding more metadata zones solves the problem. RAIZN+ tries this (one metadata zone per logical zone), but it consumes more active zones, reducing the number of data zones applications can use. Throughput still lags Zebra (Figure 18b) due to fewer parallel data zones.
Problem 3 · medium
After a machine failure, RAIZN takes 42 seconds to recover a 100 GB logical volume. Zebra takes 3 seconds for the same volume. Explain why Zebra is 14× faster, referencing the paper's recovery mechanism.
Show answer
RAIZN must traverse the entire metadata zone (~1 GB) to find valid PPU log entries, scanning headers and checking stripe completion status (93% of recovery time, per §IV-E). Zebra scans only ZRWA windows (e.g., 1 MB each) using OOB metadata (1 bit/LBA) to locate Real WPs. For a 100 GB volume with 10 active zones, Zebra scans ~10 MB (10 zones × 1 MB ZRWA) vs. RAIZN's full metadata zone scan. Result: 14.2× faster recovery (Figure 17b).Misconception: that recovery time scales with total volume size. For Zebra, it scales with ZRWA size × active zone count (the data to scan), not the full volume. RAIZN scales with metadata zone size (all logged PPUs across the volume's lifetime).
Problem 4 · hard
On small-zone ZNS (Samsung PM1731a: 96 MB zones, 64 KB ZRWA, per-channel isolation), Zebra achieves 4.2× throughput over RAIZN under 4 KB writes. On large-zone ZNS (WD ZN540: 1077 MB zones, 1 MB ZRWA, shared channels), the gain is smaller. Explain why the improvement is larger on small-zone ZNS.
Show answer
Small-zone ZNS: each zone occupies one channel, so a zone's bandwidth is strictly its channel's bandwidth. RAIZN's metadata zone is confined to one channel, limiting PPU throughput to that channel's bandwidth (~120 MB/s). Meanwhile, data zones on other channels can collectively achieve much higher throughput. RAIZN's single-channel metadata bottleneck is severe because it can't exploit other channels' bandwidth. Zebra, using ZRWA in each zone, parallelizes PPUs across all channels (6 data + 1 parity = 7 zones on 7 channels). On large-zone ZNS, all zones share all channels, so RAIZN's metadata zone can leverage multiple channels (less severe bottleneck). Zebra's advantage is smaller because RAIZN isn't as constrained. The paper's data (Figure 12): small-zone Zebra = 840 MB/s, RAIZN = 200 MB/s → 4.2× gain. Large-zone gains from Figure 10 are smaller.Misconception: that hardware architecture doesn't affect the bottleneck severity. Small-zone ZNS's per-channel isolation makes RAIZN's metadata zone a hard bottleneck (one channel only). Large-zone ZNS's shared channels soften it, reducing Zebra's relative gain.
Problem 5 · medium
ZRWA zones move in units of ZRWAFG (e.g., 16 KB on WD ZN540). A 1 KB write is issued to LBA 100 (start of ZRWA). Describe what happens to the ZRWA window and WP.
Show answer
The 1 KB write updates LBA 100 in ZRWA (in-place, no WP movement). WP remains at LBA 100 (start of ZRWA). ZRWA window stays at LBA 100–104 (assuming 1 MB ZRWA = 256 LBAs). No implicit commit occurs because the write falls within the existing ZRWA window. However, if a subsequent write targets LBA 104 (beyond the current ZRWA window), the SSD implicitly commits the first ZRWAFG (e.g., LBA 100–103 if ZRWAFG = 4 LBAs = 16 KB), flushing it to flash and sliding ZRWA to LBA 104. The key: ZRWA moves in coarse chunks (ZRWAFG), not per-LBA.Misconception: that every write moves WP. On ZRWA zones, WP only moves when the write exceeds the ZRWA window or an explicit commit is issued. Overwrites within ZRWA do not move WP.
Problem 6 · easy
Why doesn't Zebra create metadata zones like RAIZN? What would happen if Zebra tried to log PPUs in a metadata zone while still using ZRWA for data?
Show answer
Zebra eliminates metadata zones because ZRWA permits in-place parity overwrites — no need to log PPUs separately. If Zebra used both ZRWA for data and metadata zones for PPUs, it would inherit RAIZN's bottlenecks (contention, GC, log headers) while gaining no benefit from ZRWA on the parity side. The whole point of Zebra is to store parity in ZRWA, not append PPU logs. Metadata zones are only necessary when parity can't be overwritten in place (RAIZN's constraint, which Zebra removes via ZRWA).Misconception: that ZRWA and metadata zones are complementary. They're mutually exclusive for PPU handling: either overwrite in ZRWA (Zebra) or append to metadata zones (RAIZN). Doing both is redundant and defeats ZRWA's advantage.
Problem 7 · hard
Zebra uses OOB metadata (1 bit/LBA) to locate Real WP after a crash. RAIZN uses log headers in metadata zones for the same purpose. Compare space overhead: which is more compact, and by how much?
Show answer
Zebra: 1 bit per LBA in OOB area. For a 1 TB SSD with 4 KB LBAs (256M LBAs), OOB = 256M bits = 32 MB. RAIZN: each PPU log entry has an 8 B header. Under sustained 4 KB writes (all PPUs), a 64 KB chunk generates 16 PPU entries × 8 B = 128 B headers per stripe. For 1 TB of data (assuming 50% parity overhead due to log inefficiency), metadata zone stores ~500 GB of PPUs + headers. Headers alone = 128 B per 64 KB stripe = 0.2% overhead. But this doesn't account for GC amplification — obsolete entries aren't reclaimed immediately, so metadata zones grow larger over time. The paper reports RAIZN-w/o-GC consumes up to 40% metadata zone space (Figure 18b) under long-running workloads. Zebra's 32 MB OOB (0.003% of 1 TB) is negligible by comparison. Result: Zebra's OOB is orders of magnitude more compact.Misconception: that log headers are small. While 8 B per entry seems modest, the cumulative overhead (across all PPUs, plus GC churn) is substantial — RAIZN's metadata zones can grow to 40% of total space without GC (Figure 18b). OOB's 1 bit/LBA is fixed and tiny.
Problem 8 · medium
Under what workload pattern would Zebra and RAIZN perform similarly? Give an example and explain why Zebra's advantage disappears.
Show answer
Workload: large-sized write requests (≥ stripe size, e.g., 192 KB writes on a 3+1 RAID with 64 KB chunks). Such requests are full-stripe writes — all data chunks filled at once, parity calculated once, no PPUs. Both Zebra and RAIZN write data and parity directly to zones (no metadata zone involvement in RAIZN, no ZRWA overwrites in Zebra). Throughput converges because neither system encounters PPUs. Example from paper (Figure 10): at 192 KB request size, Zebra and RAIZN both reach ~3400 MB/s (physical limit of WD ZN540). Zebra's advantage (ZRWA for PPUs) is irrelevant when PPUs don't occur.Misconception: that Zebra always outperforms RAIZN. Zebra's gains are specific to partial stripe writes (PPU-heavy workloads). For full-stripe writes, both systems are equivalent because no parity updates occur mid-stripe.
Quick answers

Frequently asked questions

Twelve questions newcomers ask most. Tap any to expand.

Q1What is ZRWA and why does it exist?▾
ZRWA (Zone Random Write Area) is a feature of modern ZNS SSDs that exposes the SSD's RAM-based write buffer to the host. SSDs use write buffers to batch small writes into flash page/block sizes (e.g., 16 KB). ZRWA lets the host issue random, concurrent overwrites within a sliding window (e.g., 1 MB) in front of a zone. When writes exceed the window, the buffer flushes to flash and the window slides forward. ZRWA was added to NVMe ZNS spec in 2024 (TP 4082) to support applications that need temporary overwrite semantics (e.g., RAID parity updates, as Zebra demonstrates).
Q2Why can't traditional RAID solutions work on ZNS SSDs?▾
Traditional RAID (e.g., mdadm) assumes block-interface SSDs that permit random overwrites. PPUs in RAID require overwriting parity chunks in place. ZNS SSDs enforce append-only writes — data can only be written at the write pointer, and overwrites are forbidden. Attempting to overwrite parity fails. Solutions like RAIZN work around this by logging PPUs in metadata zones, but this introduces bottlenecks (contention, GC, log headers). Zebra is the first to leverage ZRWA to restore in-place parity updates, making ZNS RAID viable without metadata zones.
Q3How does Zebra's Real WP differ from the device-reported WP?▾
On ZRWA zones, the device-reported WP points to the start of the ZRWA window, not the tail of actual data. Because ZRWA can overwrite LBAs within the window, some LBAs may be unwritten even though they fall before the WP. Zebra's Real WP is the true tail of consecutive written data. After a crash, Zebra scans OOB metadata (1 bit per LBA, set to 1 when written) from the device WP forward, finding the last consecutive 1 bit. That's the Real WP. This ensures no unwritten gaps are treated as valid data.
Q4Does ZRWA require special hardware beyond standard ZNS SSDs?▾
No. ZRWA is a standard NVMe feature included in the ZNS spec (TP 4082, ratified 2024). The paper's Table I lists ZRWA settings for commercial ZNS SSDs, showing that modern devices from WD, DapuStor, and others include ZRWA support. The SSD's write buffer (RAM-based) exists regardless — ZRWA just exposes it to the host via the zone interface. No additional hardware (e.g., NVDIMM, persistent memory) is needed. The only requirement is Power-Loss Protection (PLP) to ensure ZRWA data survives crashes, which most server SSDs already provide.
Q5What is the overhead of Zebra's OOB metadata?▾
Zebra uses 1 bit per LBA in the OOB (out-of-band) area of flash. OOB is spare area per flash page, typically 8–64 B per LBA on modern SSDs. For a 1 TB SSD with 4 KB LBAs (256M LBAs), OOB metadata = 256M bits = 32 MB (0.003% of 1 TB). For parity chunks (edge case of simultaneous disk + machine failure), OOB extends to 1 Byte per LBA, still negligible. The paper reports no performance degradation from OOB writes (synchronous with data writes), and space overhead is <1% of total (§IV-F).
Q6Why does Zebra perform better on small-zone ZNS than large-zone?▾
Small-zone ZNS (e.g., Samsung PM1731a) provides hardware isolation: each zone occupies one flash channel/die, so a zone's bandwidth is strictly limited to that channel. RAIZN's metadata zone is confined to one channel, creating a severe bottleneck when multiple data zones (on other channels) generate PPUs. Zebra, using ZRWA in each zone, parallelizes PPUs across all channels. On large-zone ZNS (e.g., WD ZN540), zones share all channels, so RAIZN's metadata zone can leverage multiple channels (less severe bottleneck). Result: Zebra's gain is 4.2× on small-zone (Figure 12), smaller on large-zone (Figure 10).
Q7Can Zebra work with RAID 6 (two parity chunks)?▾
Yes. The paper evaluates Zebra on (5+2) RAID 6 (Figure 17c). Each stripe has two parity chunks (P and Q, using XOR and Galois Field math). Both parity chunks reside in their respective zones' ZRWA windows, undergoing repeated overwrites as data chunks arrive. When the stripe completes, both parity chunks are committed to flash. The principle is identical to RAID 5 (one parity chunk); Zebra just handles two parity chunks in parallel. Throughput improvement: 4.7× over RAIZN on small-zone ZNS under <64 KB writes (§IV-F).
Q8What happens if chunk size exceeds ZRWA size?▾
Zebra cannot function. If chunk size (e.g., 128 KB) > ZRWA size (e.g., 64 KB on Samsung PM1731a), the parity chunk won't fit in the ZRWA window. Overwrites would fall outside ZRWA, triggering implicit commits and violating ZNS append-only semantics. Zebra's design requires chunk ≤ ZRWA. Solution: set chunk size to match or be smaller than ZRWA (e.g., 64 KB chunk for 64 KB ZRWA, or 64 KB chunk for 1 MB ZRWA on WD ZN540). The paper evaluates both configurations (§IV).
Q9Does Zebra handle read requests differently than RAIZN?▾
No. Read performance is similar (Figure 10: sequential/random reads converge for both systems). Reads don't trigger PPUs, so ZRWA vs. metadata zones is irrelevant. If a drive fails during read (degraded mode), both Zebra and RAIZN perform "reconstructed reads" (XOR surviving data + parity to rebuild missing chunk). Zebra's recovery advantage (14.2× faster, §IV-E) is for machine failure recovery (restoring WPs after crash), not disk-failure reconstruction.
Q10How does Zebra handle flush commands?▾
On ZNS SSDs with PLP (Power-Loss Protection), data in ZRWA is durable once transferred to the device (backed by capacitors). Zebra moves Real WP immediately after a write finishes. On SSDs without PLP, Zebra defers Real WP movement until a flush command completes, ensuring crash consistency. File systems (e.g., F2FS) issue flushes to persist metadata; Zebra respects those semantics. The paper assumes PLP (most server ZNS SSDs have it), so flushes are transparent (§III-D).
Q11What is the recovery time after a disk failure?▾
Zebra's disk-failure recovery is standard RAID 5 reconstruction: read all stripes, XOR survivors to rebuild missing data/parity, write to replacement drive. Time scales linearly with data volume. The paper reports ~899 seconds for 1 TB offline recovery (§IV-E). During recovery, degraded read throughput: 6.1 GB/s (64 KB requests), 9.2 GB/s (128 KB) — comparable to RAIZN because reconstruction logic is the same. Zebra's 14.2× faster recovery refers to machine failure (crash, not disk loss).
Q12Can Zebra coexist with file systems like F2FS or RocksDB?▾
Yes. Zebra presents a standard ZNS interface (append-only zones) to applications. File systems like F2FS and RocksDB (via ZenFS) already run on ZNS SSDs. Zebra is transparent to them — applications see a single logical ZNS device, unaware of the underlying RAID (data/parity striping across multiple physical ZNS SSDs). The paper evaluates RocksDB atop Zebra (Figure 15), showing 5.4× throughput improvement on fillsync workload (small writes, sync mode) compared to RAIZN. Zebra's only requirement: chunk size ≤ ZRWA size, a configuration parameter invisible to applications.

Glossary & further reading

Every term in one place, then a path to go deeper.

Core terms
ZebraA ZNS RAID system that uses ZRWA (Zone Random Write Area) to store parity chunks, enabling in-place PPUs and eliminating metadata zones.
ZRWA (Zone Random Write Area)A sliding window of LBAs in a ZNS zone that permits random, concurrent overwrites (exposing the SSD's write buffer to the host).
PPU (Partial Parity Update)Updating parity when fewer than k data chunks are written (requires read-modify-write on parity chunk). The core challenge ZNS RAID solves.
RAIZNPrior ZNS RAID system (ASPLOS'23) that uses metadata zones to log PPUs. Zebra's performance baseline.
Metadata zone (RAIZN)A dedicated zone in RAIZN that logs PPU entries (parity updates) from all logical zones, introducing contention, GC, and log header overhead.
Real WPZebra's crash-consistent write pointer — the true tail of consecutive written data in a ZRWA zone (vs. device WP, which points to ZRWA start).
OOB metadata (Out-Of-Band)Spare area per flash page (8–64 B/LBA) used to store 1-bit flags indicating if an LBA was written. Enables fast Real WP recovery.
Supporting terms
ZNS (Zoned Namespace)An NVMe SSD interface with append-only zones instead of random-writable blocks. Shifts GC to the host.
ZoneA contiguous region of LBAs in ZNS, writable only at the write pointer (append-only). States: empty, open/active, full.
Write Pointer (WP)The next writable LBA in a zone. On ZRWA zones, WP points to the start of ZRWA; on non-ZRWA zones, WP points to the tail of written data.
ZRWAFG (ZRWA Flush Granularity)The minimum number of LBAs flushed from ZRWA to flash when implicit commit occurs (e.g., 16 KB = 4 LBAs on WD ZN540).
Stripek data chunks + m parity chunks distributed across k+m drives. Zebra focuses on RAID 5 (m=1) and RAID 6 (m=2).
ChunkA fixed-size unit within a stripe (e.g., 64 KB). Zebra requires chunk size ≤ ZRWA size.
PLP (Power-Loss Protection)Hardware (capacitors/battery) ensuring ZRWA data survives power-off. Most server ZNS SSDs support PLP.
Large-zone ZNSEach zone spans all flash channels/dies, maximizing bandwidth but sharing channels across zones (e.g., WD ZN540: 1077 MB zones).
Small-zone ZNSEach zone occupies one channel/die, providing per-zone hardware isolation (e.g., Samsung PM1731a: 96 MB zones).
WAR (Write Amplification Ratio)Bytes written to device ÷ bytes written by application. Zebra reduces WAR by 19%–30% vs. RAIZN (no GC, no log headers).
AVX (Advanced Vector Extensions)CPU instructions for SIMD parallelism. Zebra uses AVX to accelerate XOR (parity calculation) and memory copy.
Implicit commitAutomatic ZRWA flush when writes exceed the window, moving ZRWA forward by ZRWAFG granularity.
