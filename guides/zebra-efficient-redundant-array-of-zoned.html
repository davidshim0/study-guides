Making ZNS RAID efficient by hiding parity in the write buffer

Zoned Namespace (ZNS) SSDs eliminate device-level GC but complicate RAID. Zebra exploits Zone Random Write Area (ZRWA)—an existing feature that exposes the SSD's internal RAM write buffer—to update parity in place and achieve 1.1×–4.2× faster throughput than prior ZNS RAID systems.

📄 "Zebra: Efficient Redundant Array of ZNS SSDs Enabled by ZRWA" · Jiang et al. 🎓 Assumes ZNS basics, RAID concepts, and SSD internals ⏱ ~35 min read
💾

The SSD's RAM write buffer in front of each zone (ZRWA) is a natural fit for Partial Parity Updates (PPUs), which exhibit repeated sequential overwrites within a fixed chunk—eliminating metadata zones and the RAID-level GC that plague prior ZNS RAID systems.

Quick glossary
ZNS (Zoned Namespace)SSD interface with append-only zones, eliminating device-level GC.
ZRWA (Zone Random Write Area)RAM write buffer window (e.g., 1 MB) in a zone supporting in-place overwrites.
PPU (Partial Parity Update)Updating parity when a write is smaller than a full RAID stripe.
RAID stripek+m chunks (k data + m parity) distributed across drives; one chunk per drive.
RAIZNPrior ZNS RAID storing PPUs in dedicated metadata zones (the bottleneck Zebra eliminates).
WP (Write Pointer)Marks the end of valid data in a ZNS zone.
OOB (Out-of-Band)Spare area per flash page; Zebra uses 1 bit per LBA to locate Real WP after crash.
ZRWAFGMinimum LBAs flushed from ZRWA to flash when ZRWA commits (typically 16–32 KB).
1The PPU problem in ZNS RAID▾

Traditional RAID: partial stripe writes read old parity, XOR with new data, overwrite parity in place. ZNS's no-overwrite rule forbids this. RAIZN appends PPUs to metadata zones (log-structured), requiring GC, log headers, and creating a bandwidth bottleneck (all PPUs funnel into one zone per SSD).

Now you're ready: Zebra uses ZRWA (the SSD's RAM write buffer) to overwrite parity in place within a small window, eliminating metadata zones entirely.
2ZRWA: exposing the SSD's internal write buffer▾

SSDs buffer small writes in RAM before flushing to flash. ZRWA exposes this buffer as a contiguous LBA window (e.g., 64 KB–1 MB) supporting random overwrites. Writing beyond ZRWA implicitly commits it to flash (advances WP by ZRWAFG). The paper's tested ZNS SSDs (Table I) support ZRWA ≥ 64 KB.

Now you're ready: Zebra sets chunk size ≤ ZRWA size, so parity chunks fit entirely in ZRWA and can be repeatedly overwritten until the stripe fills.
3Crash consistency via OOB metadata▾

ZRWA moves WP in coarse units (ZRWAFG, e.g., 16 KB). After a crash, WP points to ZRWA start but doesn't reveal which LBAs within ZRWA were written. Zebra writes 1 bit to OOB per LBA (marking written pages) and scans OOB to locate Real WP (tail of consecutively written data).

Now you're ready: This lightweight metadata (1 bit vs 8–64 B available) ensures no data loss and no garbage treated as valid after recovery.
4Machine-failure recovery (14.2× faster than RAIZN)▾

Three steps: (1) query zone states, (2) scan OOB on ZRWA to locate Real WPs, (3) discard data beyond Real WP and recalculate parity. Zebra avoids reading entire metadata zones (RAIZN traverses gigabytes to find valid PPUs). Zebra also reuses incomplete stripes via ZRWA overwrites (zero storage space reduction, unlike RAIZN).

Now you're ready: Fast recovery reduces downtime, which is important for production systems.
5Why Zebra gains 4.2× on small-zone ZNS▾

Small-zone ZNS (e.g., Samsung PM1731a): each zone's flash blocks are confined to one or a few channels (hardware isolation for QoS). RAIZN's metadata zone is limited to one channel's bandwidth, creating a severe bottleneck. Zebra's 'all-to-all' design (each zone has ZRWA for parity) exploits all channels in parallel. Large-zone ZNS shows smaller gains (1.1×–1.5×) because metadata zones can use all channels.

Now you're ready: Per-channel isolation amplifies Zebra's benefits; shared-channel architectures show smaller gains.
The whole story

Zebra in a nutshell

Problem, insight, mechanism, recovery, results—grounded in the paper's numbers.

Problem: ZNS's no-overwrite rule forbids in-place parity updates. RAIZN appends PPUs to metadata zones, causing three bottlenecks: (1) multi-zone aggregation (all PPUs funnel into one zone per SSD, up to 76% throughput loss on small-zone ZNS), (2) RAID-level GC to reclaim obsolete PPUs (up to 15% throughput hit), (3) log headers (WAR ~3 under 4 KB requests).

Insight: PPUs exhibit repeated sequential overwrites within a fixed chunk (as partial writes arrive in a stripe, parity is recalculated and overwritten in the same LBA range until the stripe fills). ZRWA (the SSD's RAM write buffer, e.g., 64 KB–1 MB) is a natural fit—it permits in-place overwrites within a window. Modern ZNS SSDs have ZRWA ≥ typical chunk sizes (64 KB).

Mechanism: Open all zones in ZRWA mode. Set chunk size ≤ ZRWA size. Parity chunks fit in ZRWA and are overwritten in place as partial writes arrive. When the stripe fills, ZRWA commits to flash (advancing WP). In-memory stripe cache avoids extra reads for parity calculation. Parity calculation and memory copy occupy 11.6% and 5.6% of write latency (before AVX optimization), reducing to minimal overhead after optimization.

Recovery: OOB metadata (1 bit per LBA) locates Real WP after crash. Machine-failure recovery: (1) query zone states, (2) scan OOB metadata in ZRWA regions to find Real WPs, (3) discard incomplete writes and recalculate parity. 14.2× faster than RAIZN (avoids reading metadata zones). Zero storage space reduction (Zebra overwrites incomplete stripes via ZRWA; RAIZN must skip them).

Results: Large-zone ZNS: 1.1×–1.5× faster (6.4%–51%), avg latency down 17%–24%. Small-zone ZNS: 4.2× faster (4 KB writes). RocksDB +5.4× throughput. WAR = 1 (vs ~3 in RAIZN). Metadata zones, GC, log headers eliminated. Hardware overhead: in-memory stripe cache ~3.5 MB (max active zones × stripe size), 1 bit OOB per LBA. Parity calculation and memory copy add modest latency overhead on the write path.

🍯 Key takeaways

  • ZRWA's in-place overwrite: Enables parity updates without metadata zones or GC, eliminating the bottlenecks seen in RAIZN.
  • Small-zone gains: Hardware isolation (per-channel zones) amplifies Zebra's advantage (4.2× vs 1.5×). Relevant for architectures with per-channel mapping.
  • Crash consistency: OOB-based Real WP location is lightweight (1 bit per page) and fast (only scan ZRWA regions, not entire zones).
  • Zero storage waste: Zebra's ability to overwrite incomplete stripes after crash avoids RAIZN's permanent capacity loss.
  • Eliminate log-structured metadata: ZRWA removes the need for metadata zones (GC-free, no log headers).

Quick answers

FAQ

Q1What is ZRWA and why does it matter for ZNS RAID?▾
ZRWA exposes the SSD's internal RAM write buffer as a contiguous LBA window supporting in-place overwrites. It matters because PPUs require repeated overwrites of the same parity chunk—exactly what ZRWA enables. By setting chunk size ≤ ZRWA size, Zebra processes PPUs in place without metadata zones.
Q2How does Zebra locate Real WP after a crash?▾
ZRWA moves WP in coarse units (ZRWAFG, e.g., 16 KB). Zebra writes 1 bit to OOB per LBA to mark written pages. After crash, it scans OOB metadata in ZRWA regions to find the longest consecutive prefix of written pages—Real WP is the tail of that prefix. The paper reports a 14.2× recovery-time reduction compared to RAIZN.
Q3Why is the gain larger on small-zone ZNS (4.2× vs 1.5×)?▾
Small-zone ZNS limits each zone to one or a few channels (hardware isolation). RAIZN's metadata zone is constrained to one channel's bandwidth, creating a severe bottleneck. Zebra's 'all-to-all' design (each zone has ZRWA) uses all channels in parallel. Large-zone ZNS lets metadata zones use all channels, reducing the bottleneck.
Q4Does Zebra work for RAID 6 (two parity chunks)?▾
Yes. The paper evaluates a 5+2 RAID 6 setup (Figure 17c). Both parity chunks (P and Q) are placed in ZRWA and overwritten in place as partial writes arrive. Average gain: 4.7× vs RAIZN on small-zone ZNS under <64 KB writes.
Q5What if chunk size > ZRWA size?▾
If the parity chunk exceeds ZRWA size, a PPU would fall outside ZRWA, triggering an implicit commit and moving WP prematurely. The parity would be committed to flash before the stripe fills, preventing in-place overwrite. The paper sets chunk size = 64 KB, within typical ZRWA sizes (64 KB–1 MB), ensuring PPUs stay inside the ZRWA window.
Q6How does Zebra handle disk + machine failure simultaneously?▾
Zebra extends OOB on parity chunks from 1 bit to 1 byte to record the last write request ID that updated each parity block. After crash, it traces back those requests to infer the data chunk ranges in the failed disk, restores Real WPs, and rebuilds the RAID. Overhead: 1 byte per LBA for parity chunks (vs 1 bit for data chunks), still modest.
Q7What workloads benefit most from Zebra?▾
Workloads with high PPU ratios (small request sizes, e.g., 4–16 KB). The paper shows gains diminish as request size approaches full stripe size (192 KB for 3+1 RAID 5 with 64 KB chunk). At 192 KB, PPUs vanish and Zebra ≈ RAIZN. Real-world traces (TPC-C, YCSB, VirtualDesktop) with avg request sizes 12.9–30.4 KB show 1.3×–3.8× gains.
Q8Does adding more metadata zones in RAIZN solve the bottleneck?▾
No. More metadata zones reduce the bottleneck but introduce trade-offs: (1) fewer data zones for applications (ZNS SSDs have limited active zones, e.g., 14), (2) GC and log headers persist (WAR > 1), (3) on small-zone ZNS, each metadata zone is still limited to one channel. The paper's RAIZN+ (one metadata zone per logical zone, no GC) is still slower than Zebra (Figure 18b).
Q9How much DRAM overhead does Zebra add?▾
In-memory stripe cache: one stripe per active logical zone = max_active_zones × stripe_size. The paper reports ~3.5 MB in experiments (this corresponds to the formula: max active zone count × stripe size). This is negligible for servers with 64+ GB DRAM.
Q10What is the recovery time after machine failure?▾
Zebra's recovery is 14.2× faster than RAIZN (§IV-E). The speedup comes from scanning OOB metadata in ZRWA regions (instead of reading gigabytes of metadata zones in RAIZN). The paper provides a latency distribution showing the reduction but does not state an absolute recovery time.
Q11Does Zebra support non-PLP ZNS SSDs?▾
Yes. If the ZNS SSD lacks Power-Loss Protection, data on ZRWA is volatile. Zebra postpones Real WP movement until flush completion (rather than on write completion). Applications must issue flush commands to ensure durability, similar to traditional file systems.
Q12What are potential future directions for Zebra?▾
The paper focuses on append-only ZNS RAID. Potential extensions include block-interface RAID (allowing random overwrites at the RAID level), adaptive chunk sizing based on ZRWA size variations, and integration with distributed storage systems (e.g., Ceph, MinIO). These are interpretive suggestions, not explicit future work stated in the paper.
