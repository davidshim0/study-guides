HPCA 2025 · A Field Guide

Manage pages by the object they belong to, not one page at a lonely time

Four GPUs share one virtual address space. Today they run one migration policy for every page. But a program's arrays each want a different policy — and it changes with the phase. This is the plain-English story of OASIS.

📄 "OASIS: Object-Aware Page Management" · Wang, Li, Tarek Ibn Ziad, Eeckhout, Yang, Jaleel, Tang 🎓 Some UVM/GPU basics assumed — the primers cover them ⏱ ~30 min read
🗺️

Pick the page policy from what an OBJECT is doing right now — private, read-shared, or write-shared — not from a one-size-fits-all default.

Before you begin

Ten ideas the paper assumes you know

Each card clears up one concept OASIS builds on. They're collapsed — tap any open. Read in order the first time, or jump to whatever looks unfamiliar. If you already do OS-level tiering and page management, most of this is home turf — just mapped onto GPUs.

1Multi-GPU systems and why they share memory▾

To meet growing compute demand, machines pack several GPUs (here 4, and up to 16 in a scaling study) connected by fast links — NVLink-v2 at 300 GB/s between GPUs, PCIe-v4 at 32 GB/s to the host. Each GPU has its own local memory (4 GB DRAM in the baseline) and its own page table.

A datum lives on one GPU but may be needed by another. Reaching across the link is a remote access — slower than local. Managing where pages live is the whole game.

Now you're ready: OASIS is about deciding, per data object, how pages move (or copy) among four GPUs so most accesses stay local.
2Unified Virtual Memory (UVM) — one address space▾

UVM gives the CPU and all GPUs a single shared virtual address space. You call cudaMallocManaged(), get one pointer, and any device can dereference it — no manual copies. A CPU-side UVM driver keeps a centralized page table and resolves faults.

When a GPU touches a page whose local page-table entry (PTE) is invalid, it takes a page fault, which is sent to the UVM driver on the CPU. That fault is the moment OASIS gets to act.

Now you're ready: if you know Linux demand paging, UVM is the same idea across PCIe/NVLink — faults trap to a driver that decides page placement.
3NUMA overhead, remote access, and migration▾

Because memory is spread across GPUs, this is a NUMA (Non-Uniform Memory Access) system: local reads are cheap, remote reads cost a link traversal. You reduce that cost by migrating a page to the GPU that keeps using it — but migration itself is expensive (fault, transfer, PTE updates).

So there's a tension: migrate too eagerly and you thrash ("ping-pong"); migrate too late and you pay remote latency in the meantime.

Now you're ready: this is exactly the DRAM/NVM hot-vs-cold migration tradeoff you know from tiering — same knobs (threshold, promote/demote), different fabric.
4On-touch migration — the eager default▾

On-touch always migrates a page to whichever GPU just touched it. Every later access from that GPU is then local — great when the page is private to one GPU.

The failure mode: if two GPUs both want the page, it ping-pongs back and forth, one expensive migration per bounce. In OASIS this is the default policy (PTE bits "00").

Now you're ready: on-touch is the promote-on-first-touch strategy; it's ideal for private data, ruinous for shared data.
5Access-counter migration — the patient middle▾

NVIDIA Volta+ GPUs add an access-counter policy: a remote GPU may access a page in place, and hardware counts the remote accesses. Only after a threshold (256 remote accesses for a 64 KB page group in the NVIDIA driver) does the page migrate.

This kills ping-ponging — a page only moves once it's clearly worth it — but you pay remote latency for all accesses before the threshold, plus PTE-invalidation cost.

Now you're ready: this is a hysteresis threshold, the same idea as "N accesses before promotion" in a tiering controller.
6Page duplication — copy the read-shared▾

Duplication makes a local read-only copy of a page on each GPU that reads it, so all those reads are served locally. Perfect for read-only shared data.

The catch is writes: any write to a duplicated page triggers a page-write-collapse — a protection fault that invalidates every copy to restore consistency. On read-write data this is very expensive.

Now you're ready: duplication is replication-for-reads; it's the read-heavy analog of a coherent shared cache line, and writes are its Achilles' heel.
7What an "object" is here▾

An object is a data structure allocated at runtime by one cudaMallocManaged() call — an array, a matrix, a layer's weights. It spans many contiguous pages (often thousands). Applications here have anywhere from 2 to 263 objects.

The paper's central empirical claim: pages within one object behave alike — if one page is read-only-shared, they nearly all are. So the object is the natural, cheap unit of decision.

Now you're ready: think of an object like a VMA / mmap region — one allocation, uniform intent, many pages.
8Private vs shared, read vs write▾

Two orthogonal axes classify a page over a period. Private = touched by one GPU; shared = touched by several. Read-only / write-only / rw-mix = which operations occur. Combine them: "shared-read-only," "private-write-only," "shared-rw-mix."

Each combination has a best policy: private → on-touch; shared-read-only → duplication; shared-write / shared-rw-mix → access-counter.

Now you're ready: OASIS classifies each object on these two axes, then maps the class to one of the three policies.
9Phases — behavior that changes over time▾

A phase is a stretch where objects show a stable, recognizable pattern. Explicit phases are visible to the runtime (each CUDA kernel launch); implicit phases are not announced but show up as pattern shifts (e.g. loop iterations that swap buffers).

The same object can be read-only in one phase and write-only in the next — so a policy that was right becomes wrong at the phase boundary.

Now you're ready: OASIS re-evaluates each object's policy at phase boundaries, like a tiering controller re-scoring hotness each epoch.
10Spare pointer bits (TBI / LAM / UAI)▾

A 64-bit pointer only needs ~48 bits for the address; the top bits are unused. Modern hardware lets you stash metadata there and ignore it on dereference: ARM's Top-Byte-Ignore (TBI), Intel's Linear Address Masking (LAM), AMD's Upper Address Ignore (UAI).

OASIS uses this to tag each pointer with its object ID — 4 bits for the ID plus 1 config bit — so every memory access self-identifies which object it belongs to, for free.

Now you're ready: tagged pointers are how OASIS turns "which object is this address in?" from a lookup into a bit-extract — the same trick memory-safety schemes use.
The whole story in one place

OASIS in a nutshell

The problem, the cheap object-level fix, and how it all connects.

UVM multi-GPU systems ship three page-management policies — on-touch migration, access-counter migration, and duplication — and today they pick one and apply it to every page. But there is no universal winner: on-touch is best for private data, duplication for read-only shared data, and access-counter for write-shared / rw-mix data. Force one policy everywhere and each application pays for the objects it mismatches.

The paper's insight is that the right unit of decision is the data object, not the page. Pages inside one object behave alike (only 2 of 26 studied objects were "non-uniform"), so learning per page is wasteful. OASIS tracks each object at runtime and assigns it the best policy, re-deciding at phase boundaries. It uses three tiny pieces: an Object Tracker that stamps a 4-bit object ID into spare pointer bits at allocation; an Object Table (O-Table) — just 16 entries × 12 bits = 24 bytes — that records each object's chosen policy; and an Object Policy Controller that reads the class from a single page fault (read ⇒ duplication, write ⇒ access-counter) and self-corrects when the pattern shifts.

The host page table acts as a free filter: first-touch faults resolve there with on-touch (serving private objects); only shared faults reach the O-Table. So OASIS spends metadata only where a decision actually matters.

The payoff: across 11 multi-GPU workloads, OASIS beats uniformly using on-touch, access-counter, and duplication by 64%, 35%, and 42% on average — and beats the state-of-the-art per-page scheme GRIT by 12% (with 22% fewer page faults) while using a 24-byte table instead of GRIT's 352-byte structure and 12 bits/object instead of 48 bits/page.

🍯 Key takeaways

  • No single page policy wins for all apps — data objects within an app each prefer a different one.
  • Object > page as the decision unit: pages in one object behave alike (only 2/26 objects non-uniform), so per-page learning is overkill.
  • Class → policy: private → on-touch; shared-read-only → duplication; shared-write / rw-mix → access-counter.
  • Three parts: Object Tracker (tagged pointers), O-Table (24 B), OP-Controller (decides from one fault + self-corrects).
  • Host page table filters: first-touch = private → on-touch; only shared faults hit the O-Table.
  • Phases matter: re-decide at explicit (kernel) boundaries and implicit ones (via a shared-fault reset threshold, default 8).
  • Result: +64% / +35% / +42% over the three uniform policies; +12% over GRIT; 22% fewer faults; <0.02% area.
Seven everyday pictures

How to think about OASIS

Each picture stands alone — read them in any order. Every card ends with an honest note about where the picture stops being true.

The core idea · Right policy per object

Sorting laundry by fabric

You don't wash every garment on one setting. Delicates, denim, and towels each want a different cycle — run them all on "hot heavy" and you ruin the silk. OASIS treats each data object like a fabric type: private data gets on-touch, read-shared gets duplication, write-shared gets access-counter. Uniform management is the person who washes everything on one setting and shrugs at the damage.

Where it misleads: a garment's fabric is fixed; an object's "fabric" can change between phases (read-only now, write-only next), so OASIS must re-sort at phase boundaries.
The O-Table · Coarse metadata

One recipe card per dish, not per bite

A kitchen doesn't write cooking instructions on every individual grain of rice — it keeps one recipe card per dish. OASIS keeps one 12-bit entry per object, not per page. Since a 2 GB array can be half a million pages but one object, a 16-entry, 24-byte table covers a whole workload. GRIT, by contrast, annotates every page (48 bits each) — like re-writing the recipe on every grain.

Where it breaks: the recipe card only works if the whole dish is uniform. A truly "non-uniform" object (mixed pages) is like a plate where each bite needs different handling — rare here (2 of 26 objects), but the card can't capture it.
The Object Tracker · Tagged pointers

A dye mark on every dollar

Bank a stack of bills and later you can't tell which came from which envelope — unless each was marked when it was withdrawn. OASIS marks the pointer at allocation, writing the object's ID into the unused top bits (which hardware ignores on use, via TBI/LAM/UAI). Then every memory access already carries its object's identity — no lookup needed to know "which object is this?"

Where it breaks: there are only so many spare bits (up to 15). Run out — or need those bits for something else — and you must fall back to the software shadow-map version (OASIS-InMem).
The filter · Private stays cheap

A bouncer at the door

Not everyone who shows up needs to see the manager. A bouncer waves through the regulars and only escalates the unusual cases. The host page table is that bouncer: a first-touch fault (a private page, sitting on the CPU) is resolved right there with on-touch — no O-Table visit. Only a shared fault (data already on another GPU) gets escalated to the O-Table, where a real policy decision happens.

Where it breaks: the bouncer reads "who holds the page" from physical-address ranges. Under memory oversubscription, a shared page evicted back to the CPU looks private again — a misread OASIS patches using the PTE policy bits.
Self-correction · One page speaks for all

Testing the pool with one toe

You judge the whole pool's temperature by dipping one toe — because the water is well-mixed. OASIS decides an object's policy from the read/write nature of a single page fault, trusting that all the object's pages agree (the paper's key measurement). And when the pool's temperature changes — the object flips from read to write across a phase — a stream of shared faults trips a reset threshold and OASIS re-dips.

Where it breaks: if the "water" isn't uniform (a genuinely mixed object) or flips faster than the reset threshold (default 8 faults) can follow, the single-toe reading can be stale — which is why the threshold is tunable.
State machine · Never back to on-touch

A one-way turnstile between rooms

Some doors only let you go forward. An object starts on on-touch; once it proves shared it moves to duplication (if read) or access-counter (if write), and can flip between those two as reads and writes alternate — but it never returns to on-touch. Why? Once shared data has been migrated or duplicated locally, later accesses stop faulting, so there's simply no signal (and no need) to revert.

Where it breaks: a real turnstile is about crowd control; here the "no going back" rule is an optimization assumption. If an object truly reverted to long-lived private use, on-touch would be ideal again — OASIS just accepts that this case doesn't pay to detect.
OASIS-InMem · Software fallback

A ledger when you run out of margins

Tagged pointers are like scribbling a note in a book's margin — fast, but there's only so much margin. When objects outnumber the spare bits (or those bits are reserved), OASIS keeps a separate ledger instead: a two-level shadow map plus an in-memory O-Table, cached in the mostly-idle CPU last-level cache. It costs a bit more (about 2% on average) but scales to any number of objects and touches no pointer bits.

Where it breaks: a ledger adds lookups. The first access to each shadow-map region is unavoidable overhead; OASIS-InMem leans on the CPU LLC to hide it, which works because most data lives on the GPUs, leaving that cache underused.
The full explanation

How it actually works

Section by section, with the paper's key structures rebuilt as visuals you can read at a glance.

1 · The problem — one policy can't fit an app

Multi-GPU UVM ships three policies. On-touch is best for private data; duplication for read-only shared data; access-counter for write-shared / rw-mix data. But a single application mixes these object types, and today the system applies one policy to all of them. So every app pays for the objects it mismatches — and no policy is a universal winner (the paper's Observation 1).

🌱
Concretely: in MM (matrix multiply), MM_A and MM_B are shared-read-only (80% of accesses) — duplication is best. But MM_C is private-write-only — on-touch is best. Force one policy and one of them suffers. OASIS gives each its own.

2 · Why objects, not pages

OASIS's cost advantage comes from a measured fact: pages within one object behave alike. In matrix-transpose (MT), the first ~9,000 pages (MT_Input) are entirely read-only and the next ~9,000 (MT_Output) entirely write-only. Scale MT to a 2 GB input and each object grows to 262,144 pages — still just two objects with two clean patterns, instead of over 500,000 pages to analyze individually.

⚠
Edge case — non-uniform objects. Not every object is 100% uniform: a non-uniform object has ≥1 page whose pattern differs on both axes. The paper found only 2 of 26 objects non-uniform, and just 1 of 7 single-phase apps (ST) qualifies as a non-uniform app (<5% of pages). Rare enough that one page can safely speak for the whole object.

3 · The three components

Object Tracker. A wrapper around cudaMallocManaged() stamps a 4-bit object ID (plus a 1-bit config flag) into the pointer's unused top bits at allocation. Hardware's Top-Byte-Ignore (and Intel LAM / AMD UAI) ignores those bits on dereference, so every access self-identifies its object with no lookup.

# Object Tracker: stamp Obj_ID into spare pointer bits
ADDR_BITS = 48
obj_ID_config_shifted = OBJ_ID_Config << ADDR_BITS
MASK  = (1 << ADDR_BITS) - 1        # keep the low 48 address bits
ptr_a = (ptr_a & MASK) | obj_ID_config_shifted   # tag the top bits

Object Table (O-Table). An on-chip, LRU-managed table of up to 16 entries. Each entry is 12 bits: a 4-bit Obj ID, a 1-bit policy (0 = duplication, 1 = access-counter), a 3-bit page-fault counter (PF Count), and 4 LRU bits. On-touch isn't stored — it's the default. Total: 12 × 16 = 24 bytes.

Object Policy Controller (OP-Controller). Uses the host page table as a filter and the O-Table as the decision store, described next.

4 · The decision flow — one fault decides

Default policy is on-touch (PTE "00"), chosen precisely because it faults on every remote access, giving the CPU maximum visibility into patterns. The flow:

  1. First touch of a page resolves at the host page table with on-touch — never reaching the O-Table. Private objects live entirely here.
  2. A later fault where the page is already on another GPU is, by definition, a shared fault → forwarded to the O-Table.
  3. First O-Table visit (PF Count = 000): classify from this one fault — read ⇒ duplication (policy 0), write ⇒ access-counter (policy 1). Resolve, write the 2 policy bits into the PTEs, increment PF Count.
  4. Subsequent shared faults (PF Count ≠ 000): reuse the recorded policy, increment PF Count.
# OP-Controller, per page fault
if page not yet on any GPU:          # first touch = private
    resolve with on-touch; done       # host PT only, skip O-Table
else:                                # page on another GPU = shared
    e = O_Table[obj_id]
    if e.PF_Count == 0:              # first shared fault: LEARN
        e.policy = DUP if request.is_read else CTR
    apply e.policy; write PTE bits; e.PF_Count += 1
💡
Why is one fault enough? Because pages in an object agree (Observations 2 & 4). Analyzing one page's read/write nature accurately predicts all the object's pages — removing the latency and metadata of tracking many pages. This is the crux of OASIS's low cost.

5 · Self-correction across phases

Object patterns change at phase boundaries, so a learned policy can go stale. OASIS resets in two ways:

Explicit phases (kernel launches) are visible: a modified cudaLaunchKernel resets the object's PF Count to 000, so the next fault relearns.

Implicit phases (e.g. ST's loop that swaps currData/newData each iteration) aren't announced. OASIS watches shared-fault frequency: once PF Count hits a reset threshold (default 8), it resets and relearns from the new pattern. The state machine only ever moves forward from on-touch, and flips freely between duplication and access-counter as reads/writes alternate — a shared write to a duplicated page causes a protection fault that flips it to access-counter, and vice versa.

6 · Overhead, and the software twin

Hardware OASIS costs almost nothing: the 24-byte O-Table is <0.02% of a 256 KB 16-way L2's area (per CACTI), sits on-chip so lookups are negligible, and PTE policy-bit updates piggyback on the fault resolution (which already invalidates/updates PTEs) — off the critical path.

OASIS-InMem is the software-only alternative for when objects exceed the spare pointer bits (max 15) or those bits are reserved. It keeps the O-Table in memory and uses a two-level shadow map (like a page table) to find an object's entry. For 1,024 objects the in-memory O-Table is just 1,792 bytes. The shadow map for a 64 GB footprint is ~160 MB (128 MB level-1 + 32 MB level-2) — <0.3% of memory — and both tables ride in the underused CPU LLC. Net performance loss vs hardware OASIS: about 2%.

Follow the numbers, step by step

Worked examples

Five traces, each carried all the way through — routing a fault, sizing the metadata, defeating a phase change, and comparing against GRIT — using the paper's real numbers.

① Routing three faults for object MM_A (shared-read-only)

MM_A is a matrix read by multiple GPUs. Follow it from cold to steady state.

  1. GPU0 first-touches a page of MM_A. The page is on the CPU (nowhere else) → treated as private → resolved at the host page table with on-touch. No O-Table visit.
  2. GPU1 reads the same page. Host PT shows the data is now on GPU0 (another GPU) → it's a shared fault → forwarded to the O-Table.
  3. First O-Table visit (PF Count = 000): the request is a read ⇒ set policy bit = 0 = duplication. Resolve by duplicating locally; write PTE bits "11"; PF Count → 1.
  4. GPU2, GPU3 read it: PF Count ≠ 0 → reuse duplication. Each gets a local read copy. All shared reads are now local.
  5. Result: duplication is optimal for shared-read-only — exactly the paper's finding that MM (with MM_A + MM_B = 80% of accesses) runs best under duplication.
🍯
Insight: the host page table did the private/shared triage for free (by physical-address range), and a single read fault set the policy for the whole object. No per-page learning.

② Sizing the O-Table (and why it's 24 bytes)

Add up the on-chip metadata and compare it to the L2 it lives beside.

  1. One entry = 4-bit Obj ID + 1-bit policy + 3-bit PF Count + 4-bit LRU = 12 bits.
  2. Table capacity = 16 entries (enough because most apps have <24 objects, and only a handful are live per phase).
  3. Total = 12 bits × 16 = 192 bits = 24 bytes.
  4. As a fraction of a 256 KB 16-way L2: 24 B ÷ 262,144 B ≈ 0.009% — the paper reports <0.02% via CACTI (which also counts decode/logic).
  5. Contrast GRIT: a 352-byte on-chip PA-Cache and 48 bits of in-memory metadata per page. OASIS uses 24 bytes on-chip and 12 bits per object.
💡
Because objects ≪ pages, "12 bits per object" is a rounding error against GPU memory, whereas "48 bits per page" measurably shrinks effective capacity. Coarse granularity is the whole cost story.

③ Catching an implicit phase change in ST

ST loops, reading currData and writing newData, then swaps them each iteration. So an object flips read↔write with no kernel boundary. Reset threshold = 8.

  1. Iteration k: ST_Data1 is being read-shared → OASIS learns duplication (policy 0). Reads are local. Good.
  2. Swap → iteration k+1: ST_Data1 is now written by GPUs. Writes to duplicated pages raise protection faults (write-collapses) — these are shared faults, so PF Count climbs.
  3. PF Count reaches 8 (the reset threshold) → OASIS resets the entry (PF Count → 000).
  4. Next fault relearns: it's a write ⇒ policy bit = 1 = access-counter. The costly collapses stop.
  5. Result: OASIS tracked an implicit phase using only page faults — the paper reports ST has ~20 implicit phases and OASIS follows them.
🍯
Insight: the reset threshold is a hysteresis knob. Too low (4) and it flip-flops on rw-mix objects, adding needless collapses; too high (32) and it lags. The sweet spot 8 gave +64% vs 4→+55% and 32→+56%.

④ Why 2 MB pages help less

OASIS gives +64% with 4 KB pages but only +43% with 2 MB pages. Trace why.

  1. Take an object of 10 × 4 KB pages: GPU0 privately uses pages 1–5, GPU1 privately uses pages 6–10. At 4 KB granularity this is a private-rw-mix object → on-touch reaches ideal.
  2. Consolidate into one 2 MB page: now that single page is touched by both GPU0 and GPU1 → it becomes shared-rw-mix.
  3. Consequence: on-touch reaches ideal for private-rw-mix, but no policy reaches ideal for shared-rw-mix (some remote/collapse/migration cost is unavoidable).
  4. So large pages manufacture sharing: the paper's Fig. 20 confirms a higher fraction of shared and rw-mix pages at 2 MB.
  5. Net: OASIS still wins (+43%), but the headroom shrinks because coarser pages push objects into the one class nobody can make ideal.
⚠
Reader's connection: this is exactly the false-sharing tradeoff you weigh in tiering/THP — big pages cut TLB/fault overhead but blur per-region intent. OASIS's object view helps, but can't undo sharing forced by the page size.

⑤ Where the +12% over GRIT comes from

GRIT learns per page; OASIS per object. Both use faults as signals. Account for the gap.

  1. Fewer decision units: OASIS tracks a handful of objects vs GRIT's thousands of pages — object info fits in a 24 B on-chip table, slashing metadata access latency.
  2. Cheaper resets: GRIT needs 4 faults to change one page's policy; OASIS needs a few faults to reset a whole object at once.
  3. No neighbor prediction: GRIT adds hardware to predict neighboring pages' policies; OASIS applies one policy to all of an object's pages, so it needs none of that complexity.
  4. Faults drop: these effects cut total GPU page faults by 22% vs GRIT — and each fault is a costly CPU interrupt + UVM handling.
  5. Bottom line: +12% performance over GRIT with far lower design complexity and on-chip storage (24 B vs 352 B).
💡
The punchline mirrors OASIS's whole thesis: the object is the natural granularity of intent. Working at it makes the decision both cheaper and more accurate than grinding at the page level.
Test yourself

Practice problems

Try each before opening the answer. Every one ends with the common misconception it's checking for.

Problem 1 · easy
An object is written exclusively by one GPU for its whole lifetime (private-write-only, like MM_C). Which OASIS policy does it get, and why not duplication?
Show answer
It gets on-touch (the default; the page never becomes shared, so its faults never reach the O-Table). Duplication is wrong because a duplicated page is read-only — the very first write triggers a protection fault (write-collapse) and expensive fault handling, even though the data is private. On-touch just migrates the page once to its owner and all later accesses are local.Misconception: that duplication is a safe default for "keep it local." Duplication is only free for reads; any write to a duplicated page is costly.
Problem 2 · medium
OASIS decides an object's policy from a single page's read/write behavior. What empirical property makes this safe, and roughly how often did it fail in the paper's study?
Show answer
It's safe because pages within one object behave alike (Observations 2 & 4): if one page is shared-read-only, essentially all are. The paper found only 2 of 26 objects were "non-uniform" (≥1 page differing on both private/shared and read/write axes), and just 1 of 7 single-phase apps (ST) was a non-uniform app, with non-uniform pages under 5%. So one page reliably speaks for the object.Misconception: that you must sample many pages to be accurate. The uniformity is measured, not assumed — which is exactly what lets OASIS be so cheap.
Problem 3 · medium
Why is on-touch chosen as the default policy rather than, say, access-counter? Give the observability argument.
Show answer
On-touch faults on every remote access, so the CPU/UVM driver sees every access pattern and can classify objects. Access-counter deliberately serves up to 256 remote accesses without a fault, and duplication serves read copies without faults — both hide the very signals OASIS needs to learn from. Defaulting to on-touch keeps the object patterns visible until a policy is chosen.Misconception: that the default should be the "least harmful" policy. It's chosen for information, not performance — the harm of extra faults is paid back by learning the right policy fast.
Problem 4 · hard
The policy state machine never transitions back to on-touch, even if a shared object later becomes private again. Justify this, and name the one scenario it could cost you.
Show answer
Once an object is shared and has been migrated/duplicated onto the requesting GPU(s), subsequent accesses are local and generate no page faults — so there's neither a signal to detect the revert nor a benefit to acting on it (local access is already ideal). Reverting to on-touch would require faults you no longer get. The cost case: an object that goes shared, then truly private on a different GPU long-term — on-touch would migrate it to that new owner, but OASIS keeps the (now suboptimal) shared policy. The paper judges this too rare to detect.Misconception: that "more transitions = more adaptive = better." Adding a back-edge would need extra fault machinery for a case that seldom pays off.
Problem 5 · medium
Under 150% memory oversubscription, OASIS's private/shared classification can be fooled. Explain the failure and OASIS's fix, and cite the measured result.
Show answer
OASIS infers private vs shared from the page's physical-address range (which GPU/CPU holds it). Under oversubscription, a genuinely shared page can be evicted back to CPU memory; when re-referenced it looks like it's "on the CPU" = private, causing a wrong policy and re-learning overhead. Fix: also consult the PTE policy bits — if a CPU-resident page's policy isn't the default on-touch, treat it as shared and go to the O-Table. Result: OASIS still gives +20% over on-touch under 150% oversubscription (though oversubscription cost can dominate overall).Misconception: that physical location alone identifies sharing. Location is a proxy that breaks under eviction; the PTE policy bit is the durable signal.
Problem 6 · medium
OASIS's O-Table has only 16 entries but LeNet/VGG16/ResNet18 have 115–263 objects. Why doesn't the table overflow in practice, and what happens if it does?
Show answer
The "# Objects" figure is the maximum allocated across the whole run; only a handful are live in any one phase. Since decisions matter per active phase, 16 entries suffice for the working set of objects. If entries are exceeded, the O-Table is LRU-managed and evicts the least-recently-used object; and for genuinely large object counts (or reserved pointer bits) you switch to OASIS-InMem, which stores the O-Table in memory (e.g. 1,792 bytes for 1,024 objects) via a shadow map.Misconception: that you need one table slot per object that ever exists. You need slots for objects concurrently active in a phase — far fewer.
Problem 7 · hard
Relate OASIS's mechanism to OS-managed DRAM/NVM tiering. Which OASIS design choices have direct analogs in a tiering controller, and which are GPU-specific?
Show answer
Direct analogs: (1) the access-counter threshold is hysteresis before promotion, like "N accesses before DRAM promotion"; (2) the reset threshold / epoch re-evaluation mirrors re-scoring hotness each interval; (3) on-touch = promote-on-first-touch; (4) choosing granularity above the page (object) to cut metadata is the same argument as region/huge-page-based tiering to shrink tracking cost. GPU/UVM-specific: (a) duplication (read replication with write-collapse) has no tiering equivalent — a tier stores data once; (b) private-vs-shared classification via physical-address range across GPUs; (c) tagged-pointer object IDs using TBI/LAM/UAI. The core lesson transfers: pick the migration/placement policy from measured access class at a coarse-but-meaningful granularity.Misconception: that GPU page management is a separate world. The placement/migration/threshold machinery is the same tiering toolkit; duplication and cross-GPU sharing are the genuinely new pieces.
Problem 8 · easy
OASIS-InMem performs within ~2% of hardware OASIS despite storing tables in DRAM. What hides most of that latency, and why is it available?
Show answer
The shadow map and in-memory O-Table are cached in the CPU's last-level cache (LLC). That LLC is underutilized because most program data lives on the GPUs, not the CPU — so there's spare capacity to hold these small metadata structures, making lookups fast. Only the unavoidable first accesses to each shadow-map region pay full latency, causing the small ~2% gap.Misconception: that a software table must be slow. The bottleneck is memory latency, and caching the hot metadata in an idle LLC largely removes it.
Quick answers

Frequently asked questions

Twelve questions newcomers ask most. Tap any to expand.

Q1What problem does OASIS actually solve?▾
Multi-GPU UVM systems apply one page-management policy (on-touch, access-counter, or duplication) to every page, but different data objects within an app prefer different policies — and no policy is universally best. OASIS picks the right policy per object at runtime, gaining 64% / 35% / 42% over uniformly using each of the three policies.
Q2Why decide at object granularity instead of per page?▾
Because pages within one object behave alike (only 2 of 26 studied objects were non-uniform). A 2 GB array can be 262,144 pages but one object — so deciding per object cuts the metadata from 48 bits/page (GRIT) to 12 bits/object, and lets a single page's fault set the policy for the whole object. Coarser, but not less accurate.
Q3How does OASIS know which object an address belongs to?▾
The Object Tracker wraps cudaMallocManaged() and stamps a 4-bit object ID into the pointer's unused upper bits (plus a 1-bit config flag). Hardware features — ARM Top-Byte-Ignore, Intel LAM, AMD UAI — ignore those bits on dereference, so every access self-identifies its object with no lookup.
Q4How is a policy chosen from just one page fault?▾
Only shared faults reach the O-Table (private ones resolve at the host page table with on-touch). On the first shared fault for an object, the request's read/write nature decides: read ⇒ duplication, write ⇒ access-counter. Because the object's pages agree, that single sample generalizes. The choice is written into every PTE's 2 policy bits.
Q5How does it adapt when an object's behavior changes?▾
Two ways. Explicit phase changes (kernel launches) reset the object's PF Count via a modified cudaLaunchKernel. Implicit changes (e.g. ST's buffer swap each iteration) are caught when shared faults accumulate to a reset threshold (default 8), triggering a relearn. The state machine flips between duplication and access-counter as reads/writes alternate.
Q6Why does it never go back to on-touch?▾
Once shared data has been migrated or duplicated locally, later accesses are local and stop faulting — so there's no signal to detect a revert and no benefit (local is already ideal). Adding a back-edge would cost fault machinery for a case too rare to matter.
Q7How big is the hardware overhead?▾
Tiny. The on-chip O-Table is 16 entries × 12 bits = 24 bytes, under 0.02% of a 256 KB 16-way L2 (per CACTI). It's on-chip so lookups are negligible, and PTE policy-bit updates piggyback on fault resolution (which already touches PTEs), staying off the critical path. Plus 2 policy bits per PTE, reusing existing unused bits.
Q8What is OASIS-InMem for?▾
A software-only alternative for when objects exceed the spare pointer bits (max 15) or those bits are reserved. It stores the O-Table in memory and uses a two-level shadow map to find entries. For 1,024 objects the in-memory O-Table is 1,792 bytes; the shadow map for a 64 GB footprint is ~160 MB (<0.3% of memory), cached in the CPU LLC. Costs ~2% vs hardware OASIS.
Q9How does OASIS compare to GRIT, the prior state of the art?▾
GRIT learns policies per page (48 bits/page in memory, a 352-byte on-chip PA-Cache, plus hardware to predict neighboring pages) and needs 4 faults to change a page's policy. OASIS works per object (12 bits/object, 24-byte O-Table, no neighbor prediction) and resets whole objects with a few faults. Result: +12% performance, 22% fewer page faults, far lower complexity.
Q10Does OASIS keep working at scale and with big inputs?▾
Yes. It gives +65% and +67% on 8- and 16-GPU systems, +62% with larger inputs, +57% when pages start on the GPUs, and +20% under 150% oversubscription. It still helps with 2 MB pages (+43%) but less, because large pages convert some private objects into shared-rw-mix ones that no policy can make ideal.
Q11What does the "Ideal" bar in the results mean?▾
A hypothetical, infeasible upper bound where every shared page is duplicated regardless of reads or writes, so after an initial duplication all accesses are zero-NUMA-latency. It's not a real policy — just a ceiling. OASIS reaches it for private and read-only objects, but shared-rw-mix and shared-write-only objects always leave a gap because their remote/collapse/migration cost is unavoidable.
Q12What did the authors leave for future work?▾
A full real-system software implementation of OASIS-InMem — with UVM-driver and Linux-kernel optimizations — is beyond the paper's scope and planned as future work (they validate it in simulation as a proof of concept). More broadly, handling truly non-uniform objects and faster-than-threshold phase changes remain open.

Glossary, results & source notes

Every term in one place, the headline numbers, then an honest account of what's from the paper vs framing.

Core terms
OASISThe paper's system: object-aware page management that assigns each data object the best of three UVM policies at runtime, re-deciding at phase boundaries.
ObjectA data structure allocated by one cudaMallocManaged() call, spanning many pages. OASIS's unit of decision because its pages behave alike.
Object TrackerWraps allocation to stamp a 4-bit object ID (+1 config bit) into the pointer's spare upper bits, using TBI/LAM/UAI so hardware ignores them on use.
O-Table (Object Table)On-chip, LRU-managed, up to 16 entries × 12 bits = 24 bytes. Each entry: 4-bit Obj ID, 1-bit policy, 3-bit PF Count, 4-bit LRU.
OP-ControllerObject Policy Controller: uses the host page table to filter private objects and the O-Table to classify shared objects and record their policy.
PF CountPer-object page-fault counter in the O-Table. 000 means "learn a new policy"; reaching the reset threshold triggers relearning.
OASIS-InMemSoftware-only alternative storing the O-Table in memory with a two-level shadow map; for when objects exceed spare pointer bits. ~2% slower.
Non-uniform objectAn object with ≥1 page whose pattern differs on both axes. Rare (2 of 26), which is why one page can set the whole object's policy.
The three policies & supporting terms
On-touch migrationAlways migrate a page to the touching GPU. Best for private data; ping-pongs on shared data. OASIS default (PTE "00").
Access-counter migrationServe remote accesses in place; migrate only after a threshold (256). Best for shared-write / rw-mix. PTE "01".
DuplicationMake local read-only copies of a shared page; a write triggers a collapse (invalidate all copies). Best for read-only shared. PTE "11".
UVMUnified Virtual Memory: one address space for CPU + GPUs, managed by a CPU-side driver that resolves page faults.
Private / sharedWhether a page (or object) is touched by one GPU or several in a period — the sharing axis of classification.
Read-only / write-only / rw-mixThe operation axis of classification. Combined with sharing, it names the object's class.
Explicit / implicit phaseA phase is a period of stable object patterns. Explicit = kernel launches (runtime-visible); implicit = pattern shifts within a kernel (e.g. loop swaps).
Write-collapseA write to a duplicated (read-only) page: a protection fault that invalidates all copies. The cost that makes duplication bad for writes.
Reset thresholdShared-fault count that triggers a policy relearn (default 8; also tested 4 and 32). Hysteresis for detecting implicit phases.
TBI / LAM / UAIARM Top-Byte-Ignore / Intel Linear Address Masking / AMD Upper Address Ignore — hardware that ignores metadata in a pointer's top bits.
GRITThe prior state-of-the-art: per-page dynamic placement with neighbor prediction. OASIS beats it by 12% with far less complexity.
MGPUSimThe industry-validated multi-GPU simulator used for all experiments (4-GPU baseline, 4 KB pages, 256 KB 16-way L2, NVLink-v2 / PCIe-v4).
