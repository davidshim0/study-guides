Before you begin

Seven concepts PIMnet assumes you know

Each card clears up one idea the paper builds on. Read in order or jump to what's unfamiliar.

PIM/DPU (near-bank compute) host CPU (the bottleneck) PIMnet (new interconnect)
1Processing-in-memory (PIM) — compute near the data▾

PIM places small compute units (called DPUs in UPMEM, or "function-in-memory" logic in Samsung HBM) right next to memory banks. Each DPU can read/write only its "local" bank — not remote banks on the same chip or other chips. The goal: use high internal DRAM bandwidth instead of the slow off-chip DDR interface.

Key constraint: Each DPU is isolated. To talk to another bank's DPU, data must go to the host CPU and back. That's the problem PIMnet fixes.

Now you're ready: PIMnet gives DPUs a direct link to each other so they can communicate without bouncing through the host — which is the entire bottleneck.
2Collective communication — synchronized data exchange▾

Collective communication = coordinated exchange among multiple nodes where the pattern (source, destination, size) is known ahead of time. Common patterns: AllReduce (combine partial results from all nodes and distribute the final answer), All-to-all (every node sends to every other node), Broadcast (one sends to all).

Used in HPC, DNN training (gradient averaging), recommendation models (embedding lookups). Performance depends on network bandwidth and latency.

Now you're ready: PIMnet exploits the deterministic pattern of collective ops to schedule communication statically — no hardware arbitration needed, so the router can be tiny.
3UPMEM DPU architecture — a real PIM you can buy▾

UPMEM is the only commercial PIM with a programmable general-purpose core (14-stage pipeline, 32-bit, 24 threads called "tasklets"). Each chip has 8 DPUs and 8 banks (64 MB each). A DIMM has 8 chips (64 DPUs total). PIMnet's modeled scope is 64 DPUs per rank and 4 ranks per memory channel (256 DPUs per channel).

Each DPU has 64 KB scratchpad (WRAM), 24 KB instruction memory (IRAM), and can only access its local 64 MB MRAM bank. To talk to another DPU's bank, the host must copy data.

Now you're ready: PIMnet is demonstrated on UPMEM but the idea applies to any bank-level PIM (Samsung HBM-PIM, SK Hynix GDDR-PIM) — all have the same isolation problem.
4Memory packaging hierarchy — bank → chip → rank → channel▾

DRAM is organized hierarchically: multiple banks on one chip (8 banks/chip in UPMEM), multiple chips on one rank (or DIMM, 8 chips/rank), multiple ranks on one memory channel (4 ranks/channel in the paper's setup).

Each level has different physical connectivity: banks share an internal I/O bus; chips share DQ pins to a buffer chip on the DIMM; ranks share a DDR bus to the host. PIMnet must match this hierarchy.

Now you're ready: PIMnet is a multi-tier network: inter-bank (ring using I/O bus), inter-chip (crossbar in buffer chip), inter-rank (DDR bus used as broadcast bus) — each tier exploits existing wires.
5The baseline bottleneck — all inter-PIM traffic through host▾

In baseline UPMEM (and other PIMs), if DPU A wants data from DPU B's bank: (1) host reads B's bank into CPU memory, (2) host writes to A's bank. That's two DDR trips, both serialized. For AllReduce across 256 DPUs, all communication is bottlenecked by one shared channel to the host.

Prior work (SimplePIM, PID-Comm) optimized the host overhead but didn't remove the fundamental serialization through the DDR channel.

Now you're ready: PIMnet provides a direct path between DPUs, avoiding the host entirely for inter-PIM data movement — so communication scales with the number of DPUs, not the host bandwidth.
6DRAM technology constraints — limited logic, limited wires▾

DRAM process (e.g., 1ynm) is optimized for capacitor density, not logic performance. It has only 3 metal layers (vs ~10 in logic) — so adding routing wires is expensive. Adding pins to the chip package is also expensive. Adding buffers or crossbars directly scales with area/power.

That's why a full-featured router (input buffers, virtual channels, dynamic arbitration) is prohibitively expensive. PIMnet must be domain-specific to meet these constraints.

Now you're ready: PIMnet has no input buffers, no dynamic arbitration, low radix (2–8 ports), reuses existing pins/wires, and is PIM-controlled (the DPU schedules traffic) — all to fit DRAM constraints.
7Bandwidth parallelism — local comm can happen simultaneously▾

In a hierarchical network, if DPUs 0–7 on chip 0 are doing AllReduce among themselves while DPUs 8–15 on chip 1 do the same, both local reductions can happen in parallel because they use different wires (inter-bank ring on each chip). Only the final global reduction across chips competes for shared bandwidth.

The paper calls this bandwidth parallelism. PIMnet exploits it: hierarchical AllReduce (local reduction in parallel, then global reduction) is faster than flat AllReduce (all 64 DPUs competing for one global channel).

Now you're ready: PIMnet's multi-tier topology enables parallel local communication (inter-bank and inter-chip within each rank) and reduces the amount of data sent globally (inter-rank), improving scalability.

The whole story in one place

PIMnet in a nutshell

The problem, the cheap network that fixes it, and why it scales.

Modern PIM (UPMEM, Samsung HBM-PIM, SK Hynix) gives each bank its own compute unit — but each unit can only access its local bank. To access remote data or communicate with another bank's compute, the data must go through the host CPU via the DDR memory channel. On UPMEM, that means two serialized trips: host reads the remote bank, host writes to the local bank. For collective communication (AllReduce, All-to-all) — common in many workloads (DNN training, graph analytics, recommendation models, homomorphic encryption) — this indirect route becomes the bottleneck. As the number of banks increases, performance doesn't scale because all traffic funnels through the host's DDR channel.

The paper's fix, PIMnet, is a domain-specific interconnection network that provides direct PIM-to-PIM connectivity. PIMnet is multi-tier to match the memory packaging hierarchy: inter-bank (ring, reusing the internal I/O bus wires, ~0.7 GB/s per channel), inter-chip (8×8 crossbar in the buffer chip, reusing DQ pins, ~1.05 GB/s per channel), and inter-rank (DDR bus as broadcast bus, ~16.8 GB/s half-duplex). PIMnet is PIM-controlled: the DPU "orchestrates" communication by scheduling it statically (the source, destination, and message size are known ahead of time for collective ops) — so no hardware arbitration, no input buffers, no contention. Synchronization is done via READY/START signals. The hardware overhead is tiny: 0.09% area per DPU, 1.6% power.

The key benefit is bandwidth parallelism: local AllReduce within each rank (or chip) happens in parallel because it uses separate wires — so total bandwidth scales with the number of ranks. For example, 4 ranks doing local AllReduce simultaneously achieve higher aggregate bandwidth than global AllReduce through the host. Only the final global reduction (after local) is bottlenecked by the DDR bus, but the data volume is reduced by the number of ranks.

The payoff: on 20 real workloads (BFS, CC, MLP, GEMV, NTT, embedding tables, SpMV, Join), PIMnet provides up to 85× speedup on collective communication latency and reported overall application speedup of 11.8× compared to baseline PIM (communicating through host). For collective operations, PIMnet scales weak-scaling performance linearly with the number of banks (256 DPUs), while baseline saturates beyond 64 banks. Hardware cost: <1% area, nothing on critical path.

🍯 Key takeaways

  • Baseline PIM isolates banks — remote data must go through the host CPU, serialized on the DDR channel.
  • PIMnet provides direct inter-bank connectivity via a multi-tier network (ring/crossbar/bus) that exploits existing wires in the memory hierarchy.
  • PIMnet is PIM-controlled: DPUs schedule communication statically (no contention, no buffers, no arbitration) because collective ops are deterministic.
  • Bandwidth parallelism: local AllReduce in each rank happens in parallel on separate wires, so aggregate bandwidth scales with # ranks.
  • Collective communication speedup: up to 85× latency reduction. Real application speedup: reported overall 11.8× (e.g., CC: 5.6×, NTT: 2×, BFS: 3×).
  • Hardware overhead: 0.09% area per DPU, 1.6% power, nothing on critical path. Scales to 256 DPUs per channel (4 ranks × 64 DPUs/rank).
  • Alternative approaches (DIMM-Link, NDPBridge, ideal software) still bottlenecked by host; PIMnet removes host from inter-PIM path entirely.
Six everyday pictures

How to think about PIMnet

Each picture stands alone. Every card ends with where the analogy breaks.

The core problem · Indirect communication

Three islands with one ferry

Three islands (PIM banks) want to exchange cargo (data). The only ferry goes to the mainland (host CPU) — so island A sends cargo to the mainland, which forwards it to island B. If all three islands want to share data (AllReduce), every piece of cargo goes to the mainland twice: once to collect, once to distribute. The ferry is the bottleneck. PIMnet is a direct bridge between the islands — bypass the mainland entirely.

Where it breaks: a ferry is slow because of distance; the host CPU is slow because of serialization — all traffic shares one DDR channel. But the fix is the same: direct paths.
The multi-tier network · Matching packaging

Office mail system

In a big office building, mail within a floor uses the internal mail slots (inter-bank ring — uses existing I/O bus). Mail between floors goes through the mailroom clerk (inter-chip crossbar in buffer chip — uses DQ pins). Mail between buildings uses the postal van (inter-rank DDR bus). PIMnet matches each tier to the physical infrastructure — don't add new hallways, reuse the slots/clerk/van that are already there.

Where it breaks: office mail isn't constrained by metal layers and pin count. PIMnet is — so it can't just "add a hallway." It must reuse existing wires, which limits bandwidth per tier.
PIM-controlled interconnect · Scheduled traffic

Dance choreography

In a choreographed dance, every move is planned ahead of time — dancer A steps left while B steps right, they never collide because the script says so. PIMnet schedules communication like a choreographer: the DPU knows the entire AllReduce pattern (ring algorithm: send to neighbor, receive from neighbor, N−1 steps), so it can generate the schedule at compile time. No hardware arbitration needed — no collision ever occurs because the script prevents it.

Where it breaks: dance works because all dancers follow the script. If one DPU "misses its cue" (finishes compute late), the others must wait (synchronization via READY/START). That's a small overhead (~6 cycles) but it exists.
Bandwidth parallelism · Local + global

Two phone lines

Two households (chip 0, chip 1) each have a landline (inter-bank ring). Within each house, family members (DPUs) can talk on their landline simultaneously — the two conversations don't interfere because they're on separate lines. That's bandwidth parallelism: local AllReduce in each chip happens in parallel. Only when the two houses need to talk to each other do they compete for the shared cell tower (inter-chip crossbar or DDR bus).

Where it breaks: phone lines have unlimited capacity; PIMnet's inter-bank bandwidth is ~0.7 GB/s per channel (limited by I/O bus width and DRAM constraints). But the parallelism still multiplies total bandwidth.
Hardware simplicity · No buffers/arbitration

Toll booth vs green light

A toll booth (conventional NoC router) has lanes (input buffers) where cars queue, a toll collector (arbiter) who decides which car goes next, and a gate (crossbar). A green light (PIMnet stop) has no queue, no collector — cars just go when the light is green because the schedule guarantees no collision. PIMnet is the green light: 60× smaller area than a traditional ring router because it has no buffers, no arbitration logic. The choreography (schedule) is done by the DPU, not the router.

Where it breaks: a green light works in a small intersection; a city needs dynamic traffic lights. PIMnet only works because collective communication is deterministic. For general-purpose communication (unknown pattern), you'd need buffers/arbitration again.
Scalability · Weak scaling with # DPUs

Highway lanes

A single-lane highway (host DDR channel) has fixed throughput: more cars (DPUs) means more congestion. A multi-lane highway (PIMnet with local rings in parallel) lets traffic flow simultaneously in different lanes (different chips) — throughput scales linearly with lanes. The paper shows weak scaling: as # DPUs increases from 8 to 256, baseline saturates (all traffic on one lane) while PIMnet scales (more lanes = more throughput).

Where it breaks: even PIMnet has a global bottleneck — the DDR bus for inter-rank. If communication crosses memory channels (>256 DPUs in the paper's setup), it still goes through the host. PIMnet scales within a channel, not across channels.
The full explanation

How it actually works

Section by section, with the paper's key mechanisms rebuilt as visuals.

1 · The problem — scalability hits a wall

On UPMEM, each DPU can only access its local 64 MB MRAM bank. To access another bank's data, the host CPU must (1) read the remote bank via DDR, (2) write to the local bank. Both are serialized on the same memory channel. For collective communication (AllReduce across 256 DPUs), the bottleneck is catastrophic: all 256 DPUs funnel data through one shared DDR channel.

The paper's roofline model (Fig. 2) shows that even an ideal software implementation (zero host overhead, full DDR bandwidth utilization) is ~8× slower than a dedicated PIM interconnect. Why? Because software still goes through the host — one channel, serialized. Hardware (PIMnet) provides parallel direct paths.

⚠
Prior work is limited: SimplePIM [16] provides a software API for collective ops; PID-Comm [67] optimizes host overhead. Both are software — they reduce the host's compute cost but can't remove the fundamental bandwidth bottleneck (all traffic still through DDR). PIMnet removes the host from the inter-PIM path entirely.

2 · The fix — three-tier direct network

PIMnet provides direct connectivity across the memory hierarchy. Three tiers:

TierConnectivityPhysical resource reusedBandwidth per channel
Inter-bankRing (8 DPUs per chip)Internal I/O bus (64 bits, split into 4×16-bit unidirectional)~0.7 GB/s
Inter-chip8×8 crossbar (8 chips per rank)DQ pins (8 pins, split into 4 send + 4 receive)~1.05 GB/s
Inter-rankDDR bus as broadcast (4 ranks per channel)DDR bus (64-bit, half-duplex)~16.8 GB/s

Key design principle: reuse existing pins/wires. DRAM has only 3 metal layers (vs ~10 in logic) — adding wires is expensive. PIMnet doesn't add pins; it partitions existing buses (I/O bus, DQ pins, DDR bus) into dedicated channels for PIMnet when collective communication is active. Normal memory operations (host reads/writes) don't overlap with PIMnet — the DPU kernel runs first, then collective communication, then output retrieval.

💡
Why ring for inter-bank? Low radix (2 ports per stop) keeps the router tiny. Ring also matches the I/O bus topology inside DRAM — banks are already connected in a hierarchy (bank group I/O bus → global I/O bus). PIMnet just repurposes those wires.

3 · PIM-controlled interconnect — no hardware arbitration

Unlike a conventional NoC router (which has input buffers, virtual channels, credit-based flow control, and an arbiter on the critical path), PIMnet has none of that. Why? Because collective communication is deterministic — the source, destination, and message size are known ahead of time (at compile time or kernel launch time). The DPU can schedule the communication to ensure no contention ever occurs.

For example, for Ring AllReduce: each DPU sends to its right neighbor and receives from its left neighbor, in N−1 steps (where N = number of DPUs in the ring). The schedule is:

for step in 0..(N-1):
    send chunk[(my_id + step) % N] to right neighbor
    receive from left neighbor # no collision: everyone sends simultaneously

The key: everyone sends simultaneously, but to different destinations (neighbor pairs) — so no contention on any link. The router just forwards packets; no arbitration needed. The DPU generates the schedule (addresses and timing) at compile time and stores it in a small table (~64 B per DPU for 256-DPU AllReduce).

🍯
Hardware savings: no input buffers (~1 KB in a traditional router), no arbiter (~500 gates), no virtual channels. PIMnet stop is 60× smaller area than a conventional ring router. Power overhead: only 1.6%.

4 · Synchronization — READY/START signals

Because communication is scheduled, all DPUs must start simultaneously. If one DPU is still computing when another tries to send, the schedule breaks. PIMnet uses a simple synchronization protocol: each DPU sets a READY bit when it finishes compute; the control unit (in the buffer chip) waits for all READY bits, then broadcasts START. The synchronization overhead is the worst-case propagation latency across PIMnet — approximately 15 ns or ~6 DPU cycles at 350 MHz.

For small messages (1 KB AllReduce across 256 DPUs takes >1000 cycles), the sync overhead is <2%. For very small messages, the overhead can be higher, but the paper shows (Fig. 13) that PIM-controlled scheduling still outperforms credit-based flow control (no sync, but contention) because contention on the crossbar is worse than sync overhead.

5 · Bandwidth parallelism — hierarchical AllReduce

The killer feature of PIMnet is hierarchical AllReduce. Instead of flat AllReduce (all 256 DPUs reduce together on one global ring, bottlenecked by inter-rank DDR bus), PIMnet does:

  1. Local reduction within each chip (8 DPUs per chip) using inter-bank ring — done in parallel across all 32 chips (4 ranks × 8 chips/rank).
  2. Intermediate reduction across chips within each rank (8 chips per rank) using inter-chip crossbar — done in parallel across all 4 ranks.
  3. Global reduction across ranks using inter-rank DDR bus — this is the only serialized step, but now only reduced data from each rank needs to be communicated, not the full datasets from all 256 DPUs.
  4. AllGather (reverse): broadcast global result back down the hierarchy.

The benefit: local reductions in steps 1–2 happen in parallel on separate wires, so total bandwidth = (# of parallel groups) × (bandwidth per group). For 4 ranks doing local AllReduce simultaneously, aggregate bandwidth is higher than serialized communication through the host. That's the 85× speedup in Fig. 3 (compared to baseline, which must serialize all 256 DPU communications through the host).

💡
Why hierarchical is critical: flat AllReduce has communication complexity O(N × message size) / bandwidth. Hierarchical reduces it by enabling parallel local communication and reducing the volume of data in the global phase.

6 · Address generation and scheduling

For collective communication, each DPU needs to know (1) which memory addresses to read/write (e.g., for AllReduce, which chunk of the local buffer to send at each step) and (2) when to start each step (timing). The paper shows (Algorithm 1 in the paper, simplified below) how the host generates this schedule at kernel launch time:

# inputs: scope (bank/chip/rank), phase (RS/AG), my_id, N
if phase == ReduceScatter:
    for step in 0..(N-1):
        send_addr = base + (chunk_size × ((my_id + step) % N))
        recv_addr = base + (chunk_size × ((my_id - 1 + step) % N))
        timing_offset = sync_time + step × (chunk_size / bandwidth)
        # DPU stores: {step, send_addr, recv_addr, timing_offset}

The schedule is stored in a small table in the DPU's scratchpad (WRAM). At runtime, the DPU just reads the table and executes the send/receive according to the schedule. The host is not involved during execution.

🌱
For All-to-all: the schedule is more complex (N² address pairs), but the same principle applies. The paper uses pair-wise swaps (permutation) to minimize memory overhead — e.g., step 1: DPU 0↔1, 2↔3, …; step 2: DPU 0↔2, 1↔3, …
Follow the numbers, step by step

Worked examples

Four traces using the paper's real numbers — AllReduce on 8 DPUs, bandwidth calculation, hardware overhead, and weak scaling.

① AllReduce on 8 DPUs (one chip) — ring algorithm

8 DPUs on one chip want to sum their local values. Each has 1 KB of data. Use inter-bank ring (0.7 GB/s per channel). How many steps, and how long?

  1. Divide data into N chunks (N = 8 DPUs). Each DPU divides its 1 KB into 8 chunks of 128 B. Total data = 8 KB.
  2. Reduce-Scatter phase: N−1 = 7 steps. Step 0: each DPU sends chunk 0 to right neighbor. Step 1: send chunk 1, etc. After 7 steps, each DPU has one reduced chunk (128 B).
  3. AllGather phase: another 7 steps. Each DPU broadcasts its reduced chunk. After 7 steps, all DPUs have all 8 reduced chunks.
  4. Total communication: In a ring, links operate in parallel. Per-DPU time for each phase: (N−1) steps × chunk_size / bandwidth = 7 × 128 B / 0.7 GB/s ≈ 1.3 µs per phase.
  5. Time: 2 phases × 1.3 µs ≈ 2.6 µs (plus ~6 cycles sync overhead = ~0.017 µs). Total ≈ 2.6 µs.
🍯
Insight: ring AllReduce is bandwidth-optimal (every link is used every step, no idle time). The latency is (N−1) × chunk_size / bandwidth for each phase, so 2 × (N−1) × chunk_size / bandwidth total.

② Bandwidth parallelism — 4 ranks doing local AllReduce

256 DPUs (4 ranks × 64 DPUs/rank) want to do AllReduce. Baseline (through host): serialized DDR. PIMnet: hierarchical. Compare aggregate bandwidth.

  1. Baseline: all 256 DPUs send to host, host reduces, host sends back. Measured host-PIM bandwidth: 4.74 GB/s PIM→CPU and 6.68 GB/s CPU→PIM (per rank). Communication is serialized across ranks.
  2. PIMnet local phase: each rank does local AllReduce (64 DPUs) in parallel. 4 ranks operating simultaneously. Each rank's inter-bank ring provides 0.7 GB/s per bank channel, with 4 bank channels giving 2.8 GB/s bisection bandwidth per chip. With 8 chips per rank, rank bisection bandwidth is 22.4 GB/s. For bandwidth-optimal Ring AllReduce, aggregate send/receive bandwidth per rank is 179.2 GB/s internal to the rank.
  3. PIMnet global phase: 4 ranks reduce globally via DDR bus (16.8 GB/s half-duplex), but now only reduced data from each rank, not the full datasets from all 256 DPUs. Data volume is greatly reduced compared to baseline.
  4. Speedup: local phase exploits parallelism across ranks and much higher aggregate internal bandwidth. PIMnet avoids the serialized host path and two-trip overhead (read + write). The paper shows (Fig. 3) up to 85× latency reduction because PIMnet removes host overhead and exploits parallelism.
  5. Reality check: the 85× is not just bandwidth — it's also avoiding two DDR trips (read + write) in baseline, plus host overhead. PIMnet is one direct path.
💡
The roofline model (Fig. 2) shows PIMnet's theoretical max compute throughput is ~8× higher than ideal software because PIMnet removes the serialized DDR bottleneck. The 85× in Fig. 3 is for communication latency, not compute throughput.

③ Hardware overhead calculation

What's the area and power overhead of PIMnet for 256 DPUs (4 ranks, 1 channel)?

  1. PIMnet stop (per DPU): synthesized using OpenROAD (Nangate45, 45 nm, 3 metal layers). Area = 0.09% of baseline DPU area. Power = 1.6% of DPU power. Per stop: ~0.0013 mm² (vs ~0.08 mm² for a traditional ring router → 60× smaller).
  2. Inter-chip crossbar (per rank, in buffer chip): 8×8 crossbar. Area = 0.013 mm², power = 17 mW. Per rank: one crossbar. 4 ranks → 4 crossbars.
  3. Inter-rank switch (per channel): negligible (just control logic for DDR bus arbitration, reuses existing pins). No new buffers.
  4. Total overhead for 256 DPUs: 256 stops + 4 crossbars + 1 inter-rank switch. Total area ≈ 256 × 0.0013 mm² + 4 × 0.013 mm² ≈ 0.38 mm² (compare to baseline DPU array ~40 mm² → <1% overhead). Power: 256 × (1.6% DPU power) + 4 × 17 mW → also <1% of total system power.
  5. Critical path: PIMnet is off the critical path. Normal memory operations (host read/write) don't use PIMnet. PIMnet only active during collective communication (DPU-to-DPU), which doesn't overlap with host traffic.
🍯
The paper's claim: "<1% area, nothing on critical path" is validated. The key: no input buffers (which dominate router area in traditional NoC). PIMnet's simplicity (no arbitration, no buffers) makes it feasible in DRAM technology.

④ Weak scaling — why baseline saturates at 64 DPUs

Figure 12(a) shows AllReduce weak scaling: baseline saturates at 64 DPUs, PIMnet scales linearly to 256. Why?

  1. Weak scaling: message size per DPU is constant (32 KB in the paper). Total data = N × 32 KB. As N increases, total communication volume increases linearly.
  2. Baseline: all data goes through host DDR. At N = 8, total data = 256 KB. At N = 64, total data = 2 MB. At N = 256, total data = 8 MB. The measured host-PIM bandwidth limits throughput.
  3. Why saturation at 64? 64 DPUs = 1 rank (8 chips × 8 DPUs). Beyond 64, traffic crosses ranks, so inter-rank DDR bus is fully utilized. More DPUs = more congestion on the shared bus → no more speedup.
  4. PIMnet: at N = 8 (within one chip), inter-bank ring is used (0.7 GB/s × 8 = 5.6 GB/s aggregate). At N = 64 (within one rank), inter-bank + inter-chip (crossbar) are used in parallel across 8 chips → aggregate bandwidth scales. At N = 256 (4 ranks), local AllReduce in each rank happens in parallel (4× bandwidth), then one small global reduction. So performance scales nearly linearly.
  5. Result: PIMnet at N = 256 is ~10× faster than baseline (normalized performance ~1.0 vs ~0.1 in Fig. 12(a)).
🌱
The paper's weak scaling is per-DPU message size constant. Strong scaling (total message size constant, divided among more DPUs) would show even bigger benefits for PIMnet because communication volume per DPU decreases.

Practice problems

Try each before opening the answer. Every one ends with the misconception it checks.

Problem 1 · easy
Baseline PIM (UPMEM) requires two DDR trips for inter-PIM communication: host reads remote bank, host writes local bank. If measured host-PIM bandwidth is 4.74 GB/s PIM→CPU and 6.68 GB/s CPU→PIM, what's the effective communication bandwidth for 1 KB transfer? Assume no host overhead.
Show answer
Total data moved = 1 KB read + 1 KB write = 2 KB. Time = (1 KB / 4.74 GB/s) + (1 KB / 6.68 GB/s) = 211 ns + 150 ns = 361 ns. Effective bandwidth = 1 KB / 361 ns ≈ 2.8 GB/s (much lower than either raw bandwidth because of two serialized trips). PIMnet avoids this entirely with direct paths.Misconception: that DDR bandwidth is the bottleneck. The real bottleneck is the double trip (read + write) — even if DDR were infinite, the host overhead (not modeled here) would still limit performance.
Problem 2 · medium
For hierarchical AllReduce on 256 DPUs (4 ranks, 8 chips/rank, 8 DPUs/chip), each DPU has 32 KB. How much data crosses the inter-rank DDR bus during the global reduction phase?
Show answer
After local reduction within each rank (64 DPUs per rank), each rank has reduced data. For Reduce-Scatter, the inter-rank phase communicates reduced message chunks across ranks, not just 4 scalar values. The exact volume depends on the algorithm, but it is significantly less than the baseline's 256 × 32 KB = 8 MB full transfer. Hierarchical reduction substantially reduces global traffic by first aggregating locally within each rank.Misconception: that all data crosses ranks. Hierarchical AllReduce reduces data volume at each level through local aggregation before the global phase.
Problem 3 · medium
PIMnet's inter-bank ring has 0.7 GB/s per channel. A conventional NoC with 64-bit flit, 1 GHz clock would have 64 bits × 1 GHz = 8 GB/s. Why is PIMnet 11× slower?
Show answer
PIMnet reuses the existing DRAM I/O bus (64 bits at ~350 MHz, partitioned into 4 unidirectional channels of 16 bits each). So per channel: 16 bits × 350 MHz / 8 = 0.7 GB/s. A NoC on logic technology (1 GHz, full 64-bit width) would be faster, but (1) DRAM can't run at 1 GHz, (2) adding dedicated wires in DRAM (3 metal layers) is prohibitively expensive. PIMnet's design is constrained by memory technology — it's not a "slow NoC," it's a "cheap NoC that reuses existing wires."Misconception: comparing PIMnet to a logic-process NoC. PIMnet is in DRAM technology where wires and logic are scarce. The right comparison is to no interconnect (baseline), not to an ideal NoC.
Problem 4 · hard
Why does PIM-controlled scheduling (PIMnet) outperform credit-based flow control for All-to-all (Fig. 13(b)), even though credit-based has no synchronization overhead?
Show answer
All-to-all has many simultaneous point-to-point communications (every DPU sends to every other DPU). With credit-based flow control, contention occurs at the inter-chip crossbar — multiple senders try to reach the same destination port, causing backpressure and queueing delay. PIMnet's schedule avoids this: it uses a permutation pattern (step 1: pairs {0↔1, 2↔3, …}; step 2: {0↔2, 1↔3, …}) so no two senders target the same output port in any step. Result: no contention, no queueing delay. The paper shows 18.7% latency reduction (PIMnet vs credit-based) for All-to-all because avoiding contention wins over the small sync overhead.Misconception: that synchronization is always a loss. For deterministic patterns, scheduling avoids contention, which is a bigger win than eliminating sync.
Problem 5 · medium
The paper claims 85× speedup on collective communication (Fig. 3) but only 11.8× on real applications (Fig. 10). Why the gap?
Show answer
85× is communication latency (time spent in AllReduce or All-to-all). 11.8× is reported overall application speedup (including compute time). Many real workloads (e.g., MLP, NTT) are compute-bound — communication is <20% of total time. Even if you make communication infinitely fast, speedup is capped by Amdahl's Law: if 80% is compute, max speedup = 1 / 0.8 = 1.25×. Workloads like BFS and CC (where communication is 83% of baseline time) see higher speedup (5.6× and 3×) because communication is the bottleneck. The 11.8× is the reported overall speedup across the workloads (some compute-bound, some communication-bound).Misconception: that accelerating one component (communication) speeds up the whole system by the same factor. Amdahl's Law applies — the non-communication part (compute, memory access) limits total speedup.
Problem 6 · hard
PIMnet reuses existing pins (DQ pins for inter-chip, DDR bus for inter-rank). But those pins are also used for normal memory operations (host read/write). How does PIMnet avoid conflict?
Show answer
PIMnet and normal memory operations are time-multiplexed, not simultaneous. The execution flow is: (1) host loads data into PIM banks (normal memory write), (2) DPU runs kernel (compute, no memory traffic), (3) DPU runs collective communication (PIMnet active, normal memory idle), (4) host retrieves output (normal memory read). Phases don't overlap. During phase 3, the DPU "owns" the DQ pins and DDR bus for PIMnet; the host doesn't access memory. This is enforced by the PIM SDK API (UPMEM) — the host calls PIMnet_AllReduce() and waits. No conflict.Misconception: that PIMnet and host memory traffic happen simultaneously. They don't — PIMnet is a dedicated mode during collective communication, where normal memory operations are paused.
Problem 7 · easy
The paper says PIMnet scales within a memory channel (256 DPUs) but not across channels. If a system has 4 memory channels (1024 DPUs total), what happens to AllReduce across all 1024 DPUs?
Show answer
PIMnet only provides connectivity within one channel. For inter-channel communication, data must still go through the host CPU (same as baseline). So for 1024-DPU AllReduce: (1) each channel does local AllReduce using PIMnet (256 DPUs per channel, in parallel → 4× speedup), (2) host gathers 4 reduced values (one per channel) and reduces them, (3) host broadcasts the final result back to all channels. The benefit: local reduction (step 1) is fast; only step 2 is slow. The paper's scope is one channel — extending to multi-channel would require inter-channel links (not proposed).Misconception: that PIMnet scales indefinitely. It scales within a channel; beyond that, you're back to the host bottleneck (though the amount of data is reduced by local reduction).
Quick answers

Frequently asked questions

Ten questions newcomers ask most. Tap any to expand.

Q1What's the fundamental problem PIMnet solves?▾
Baseline PIM (UPMEM, Samsung HBM-PIM, etc.) isolates each bank's compute — to access remote data, the host CPU must copy it via the DDR channel. For collective communication (AllReduce, All-to-all), this serialized two-trip (read + write) path becomes the bottleneck. PIMnet provides direct inter-bank connectivity, avoiding the host entirely for PIM-to-PIM data movement.
Q2How much bandwidth does PIMnet provide vs baseline?▾
Baseline: measured host-PIM bandwidths of 4.74 GB/s PIM→CPU and 6.68 GB/s CPU→PIM for inter-PIM communication. PIMnet: inter-bank ~0.7 GB/s per channel (but 8 channels per chip in parallel → 5.6 GB/s per chip), inter-chip ~1.05 GB/s, inter-rank ~16.8 GB/s. The key: PIMnet exploits bandwidth parallelism — local AllReduce in 4 ranks happens simultaneously, so aggregate bandwidth is higher. Baseline serializes everything through one DDR channel.
Q3Why not just use a conventional NoC router in DRAM?▾
DRAM technology (1ynm, 3 metal layers) can't support conventional NoC routers with input buffers, virtual channels, and arbitration — too much area/power. PIMnet is domain-specific: it exploits the deterministic pattern of collective communication to schedule traffic statically (no arbitration, no buffers) and reuses existing pins/wires (I/O bus, DQ pins, DDR bus). Result: 60× smaller area than a traditional ring router.
Q4What's "PIM-controlled" mean?▾
The DPU (not the router) generates the communication schedule at kernel launch time. For AllReduce, the DPU knows the ring algorithm (send to right neighbor, receive from left, N−1 steps) and precomputes the addresses and timing. At runtime, the DPU just executes the schedule — no hardware arbitration needed. The router (PIMnet stop) is a dumb switch that forwards packets according to the pre-determined route.
Q5How does hierarchical AllReduce work?▾
Instead of flat AllReduce (all 256 DPUs reduce on one global ring), PIMnet does local reduction within each chip (8 DPUs, using inter-bank ring) in parallel across all 32 chips, then intermediate reduction within each rank (8 chips, using inter-chip crossbar) in parallel across all 4 ranks, then one global reduction across ranks (using DDR bus). The benefit: local/intermediate reductions happen on separate wires → bandwidth parallelism. Global reduction moves much less data than baseline (already reduced locally).
Q6What's the synchronization overhead?▾
PIMnet uses READY/START signals: each DPU sets READY when done computing; the control unit waits for all, then broadcasts START. Worst-case propagation latency is ~15 ns (~6 cycles at 350 MHz). For typical messages (32 KB), this is <2% overhead. The paper shows (Fig. 13) that this small overhead is worth it to avoid contention (which is much worse in credit-based flow control).
Q7Does PIMnet work for other collective ops (Broadcast, Gather, Scatter)?▾
Yes. The paper demonstrates AllReduce and All-to-all in detail. PIMnet can be extended to other collective ops with known patterns. Table V shows implementations for Reduce-Scatter, AllGather, AllReduce, All-to-all, and Broadcast. Operations like Reduce or Gather can use a single DPU as the root.
Q8How does PIMnet compare to prior work (DIMM-Link, NDPBridge)?▾
DIMM-Link [89] adds point-to-point links between buffer chips on different DIMMs (ranks) — solves inter-rank, but not inter-bank or inter-chip. NDPBridge [85] proposes bridges across the DRAM hierarchy and supports All-to-all but not AllReduce (Table I places it as N/A for collective operations). PIMnet is the first to provide bank-level PIM-to-PIM connectivity across all three tiers (inter-bank, inter-chip, inter-rank) and optimizes for collective communication (AllReduce, All-to-all).
Q9What's the hardware overhead on a real system?▾
For 256 DPUs (4 ranks, 1 channel): 256 PIMnet stops + 4 inter-chip crossbars + 1 inter-rank switch. Total area <1% of system area, power <1% of system power. Critical path: PIMnet is off the critical path (doesn't affect normal memory operations). The paper synthesized PIMnet using OpenROAD (45 nm, 3 metal layers) and validated the overhead is negligible.
Q10Why doesn't PIMnet work across memory channels?▾
Each memory channel has its own DDR bus to the host — they're physically separate. PIMnet reuses the DDR bus (as a broadcast bus for inter-rank), so it can only connect DPUs within one channel. For inter-channel communication, you'd need additional wires (e.g., a crossbar between channels), which isn't proposed. The paper's scope is 256 DPUs per channel; beyond that, the host is still needed. But local reduction (within each channel) is fast, so the amount of inter-channel data is small.

Core terms
PIMnetProcessing-In-Memory interconnection network — a domain-specific network for direct PIM-to-PIM communication, avoiding the host CPU.
PIM / DPUProcessing-In-Memory / Data Processing Unit — compute logic placed near a memory bank. In UPMEM, each DPU is a 14-stage, 32-bit, 24-thread core.
Multi-tier networkPIMnet's three-level topology: inter-bank (ring), inter-chip (crossbar), inter-rank (DDR bus as broadcast) — matching memory packaging hierarchy.
PIM-controlled interconnectThe DPU schedules communication statically (no hardware arbitration, no buffers) by exploiting the deterministic pattern of collective ops.
Bandwidth parallelismLocal communication in different chips/ranks happens simultaneously on separate wires → aggregate bandwidth scales with # of parallel groups.
Hierarchical AllReduceLocal reduction within each chip (parallel) → intermediate within each rank (parallel) → one global reduction → broadcast back. Exploits bandwidth parallelism.
Collective communicationCoordinated data exchange among multiple nodes with known pattern (AllReduce, All-to-all, Broadcast, etc.). Used in HPC, DNN training, recommendation models.
AllReduceReduce data from all nodes (e.g., sum) and distribute the result to all. Consists of Reduce-Scatter + AllGather.
Supporting terms
UPMEMA commercial PIM system with programmable DPUs (24 threads, 350 MHz) — 8 DPUs per chip, 8 chips per rank, 4 ranks per channel. Used as the baseline in this paper.
Inter-bank / inter-chip / inter-rankThree tiers of PIMnet. Inter-bank: within one chip (ring, 0.7 GB/s). Inter-chip: within one rank (crossbar, 1.05 GB/s). Inter-rank: across ranks (DDR bus, 16.8 GB/s).
PIMnet stopThe "router" in PIMnet — a 2-port switch with no buffers, no arbitration. Just forwards packets according to the pre-computed route. 60× smaller than a traditional ring router.
Reduce-ScatterFirst half of AllReduce: each node gets a unique reduced chunk (e.g., node 0 gets sum of chunk 0 from all nodes).
AllGatherSecond half of AllReduce: each node broadcasts its chunk to all others, so everyone has the full reduced result.
All-to-allEvery node sends a chunk to every other node. N² communication pairs. PIMnet uses permutation (pair-wise swaps) to avoid contention.
READY/START signalsSynchronization protocol in PIMnet. Each DPU sets READY when done computing; control unit waits for all, then broadcasts START. Overhead ~6 cycles.
Weak scalingMessage size per node is constant; total data = N × message size. PIMnet scales linearly; baseline saturates because DDR channel is fixed.
DIMM-Link / NDPBridgePrior work on inter-DRAM communication. DIMM-Link: point-to-point links between ranks. NDPBridge: bridges across DRAM hierarchy for All-to-all (not AllReduce).
