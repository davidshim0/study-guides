Making the cache's hidden moves something software can prove about

In täkō, a cache miss or eviction can run your code and change your program's answer. täkōFormal writes down exactly what such a machine is allowed to do — and then proves a hardware model never breaks those rules.

The ideas this paper builds on

1What a memory consistency model (MCM) is▾

An MCM answers one question for a parallel program: given all the loads and stores running concurrently, which values is each load allowed to return? It constrains the legal outcomes. Sequential Consistency (SC) is the strictest familiar one; x86-TSO, ARMv8, and PTX are weaker, real-ISA models.

Without a precise MCM, "what should this program print?" has no defined answer, compilers can't be proven correct, and subtle reordering bugs hide for years. The paper notes a lack of careful interface reasoning has caused real vulnerabilities and spec ambiguities (Spectre, C/C++11 SC repairs, ARM read-after-read hazards).

Now you're ready: täkōFormal's central artifact is a new ISA-level MCM — but one that must also constrain values produced by cache events and callbacks, which no conventional MCM does.
2Axiomatic models: events, relations, and "forbid a cycle"▾

An axiomatic MCM represents an execution as a directed graph: nodes are memory events (reads R, writes W), and labeled edges are relations — program order (sb/po), reads-from (rf: which write a read observed), from-reads (fr), and modification order (mo/co, the total order on writes to one address). An execution is legal iff it satisfies all the model's axioms.

Many axioms have the shape "forbid a cycle." SC, for instance, is acyclic(rf ∪ fr ∪ sb ∪ mo): if those edges form a cycle, the outcome is impossible.

Now you're ready: täkōFormal's MCM (its Figure 6) is exactly this — a set of events, relations, and axioms — extended with brand-new event/relation kinds for callbacks.
3Litmus tests — tiny programs that pin down a model▾

A litmus test is a minimal multi-threaded program with a specific candidate outcome, used to ask "does the model allow this?" The canonical one is message passing (mp): under SC the outcome r1=1, r2=0 is forbidden; under weak models it may be allowed. Litmus tests are how architects compare models and how implementations are spot-checked.

täkōFormal introduces a family of täkō litmus tests — mprmw, mpcb, icbsb, wbr/wbf, phiR/phiNR, hatsR/hatsNR — each probing a callback behavior, and encodes them in Alloy.

Now you're ready: the whole "how it works" story is told by walking these tests; the extended arXiv version adds more of them to Section V.
4What a programmable memory hierarchy (PMH) is, and what täkō does▾

A programmable memory hierarchy gives software control over data movement through the caches. täkō (Schwedock et al., ISCA 2022) is one such design: a tiled chip where each tile has a core, L1, private L2, an L3 shard, and an engine that runs callbacks. Programmers register OnMiss, OnEvict, and OnWriteback (OnWB) callbacks for virtual address ranges; these run in-cache, on the engine, when the matching cache event fires.

This lets data transformation happen as part of data movement (graph traversal, scatter-updates, etc.), and caches results to avoid redundant work.

Now you're ready: the source of all the difficulty is that a hardware event (miss/eviction) now runs user code that can change program-visible values.
5Phantom addresses — data that lives only in the cache▾

A phantom address is not backed by main memory. The only way a value for it exists is if an OnMiss callback creates it in the cache. If the line is evicted, that value is gone unless a writeback callback persisted it somewhere real.

This breaks an assumption every conventional MCM makes — that a load can always fall back to memory. With phantoms, a load's value depends on whether and when an OnMiss ran, and whether an eviction intervened. The paper denotes phantom reads/writes as Rcb/Wcb ("cb" for callback) to keep them distinct from regular R/W.

Now you're ready: "a write you performed can be silently forgotten by a later read" is the canonical täkō surprise — and it's entirely about phantom addresses + evictions.
6Happens-before, reads-from, and coherence order▾

happens-before (hb) is the ordering a model derives from program order plus inter-thread synchronization (sw, "synchronizes-with"); traditionally hb = (sb ∪ sw)+. The visibility axiom (Vis) uses hb to forbid a read from observing a write that is hb-shadowed by a newer write to the same address.

reads-from links a write to the read that observed it; coherence/modification order totally orders writes per address. These are the load-bearing relations of any axiomatic model.

Now you're ready: a core contribution is augmenting hb with callback orderings (adding vf, eb, and certain cbo edges) so visibility reasoning works across callbacks.
7Soundness vs. completeness (and why this proof is soundness)▾

For a spec S and an implementation I: soundness means every behavior of I is permitted by S (the hardware never does something the spec forbids). Completeness means every behavior S allows is realizable by I. They are different directions.

täkōFormal proves soundness: for all programs, any execution of the operational (implementation) model is allowed by the ISA-level MCM. That is the safety-critical direction — it guarantees a programmer reasoning with the MCM is never surprised by the real machine.

Now you're ready: when the guide says "proven sound," it means impl ⊆ spec — every implementation execution satisfies every axiom.
8Operational models, induction, and inductive invariants▾

An operational model is a transition system: a state set S, an Init(s) predicate, and a Next(s,s′) step relation. To prove a property P holds on all (possibly infinite) executions, you use induction: show Init ⇒ P and P(s) ∧ Next(s,s′) ⇒ P(s′).

The catch: most useful P aren't preserved step-by-step from unreachable states. The fix is an inductive invariant Inv — a strengthening (Inv ⇒ P) that is closed under Next. Finding Inv is the central labor of any such proof.

Now you're ready: the soundness proof is an induction over the model's transitions; the paper reports it needed 119 invariant clauses and 61K lines of proof annotation.
9Prefix-closure — the property that makes the proof possible▾

An axiom is prefix-closed if, whenever it holds for a complete execution graph, it also holds for every partial graph along the way to building that execution. Many natural axioms are not: "every OnMiss-start has a matching OnMiss-end" is true of finished executions but fails mid-OnMiss, so it can't be proven by induction over partial graphs.

Rephrasing it prefix-closed ("between every pair of OnMiss-starts there's an OnMiss-end") preserves the intent while holding for partial executions — and so becomes inductively provable. The paper argues future MCM designers should deliberately aim for prefix-closed axioms.

Now you're ready: prefix-closure is one of the paper's two "general insights" — it's what lets an axiomatic spec be verified against a state machine at all.
10Refinement and an intermediate state machine (ISM)▾

The spec is about execution graphs; the implementation is a detailed state machine whose steps mostly shuffle data through caches and a network without touching the graph. To bridge them, the proof inserts an intermediate state machine (ISM) whose state is a partial execution graph, and a refinement argument: an abstraction function abs maps each operational state to an ISM state, and every operational transition maps to either a valid ISM transition or a graph no-op.

This decouples two concerns: refinement abstracts away cache/network internals; the inductive proof reasons only about how adding graph nodes preserves axioms. Both halves are machine-checked.

Now you're ready: the end-to-end proof = (refinement: operational → ISM) + (induction: ISM satisfies every axiom). The paper claims this is the first such fully machine-checked end-to-end proof.
The whole story in one place

täkōFormal in a nutshell

Why a programmable cache breaks intuition, the spec that tames it, and the proof that ties spec to silicon.

In täkō, a cache miss, eviction, or writeback runs a user-defined callback on an in-cache engine. That is powerful — data is transformed as it moves — but it welds previously-hidden microarchitectural details (prefetch timing, replacement policy) to program-visible values. The paper's opening example: a thread writes 1 to a phantom address [x] then reads it back; if an eviction sneaks in between, the write is silently forgotten and the read sees an OnMiss-generated value instead. Reasoning about correctness now requires reasoning about the cache.

täkōFormal's answer is a new cache-aware ISA-level MCM (the spec): an axiomatic model that adds events for callbacks (Ms/Me, Es/Ee), phantom accesses (Rcb/Wcb), and flushes (Fl); a callback-order relation cbo reflecting täkō's per-address FIFO serialization; and axioms that (i) force a phantom read to be sourced by a prior OnMiss, (ii) make callback values correspond correctly, and (iii) augment happens-before with callback orderings so visibility reasoning works. A programmer can then check, in Alloy, whether a given outcome is possible — without understanding täkō's microarchitecture.

To make sure the spec faithfully captures täkō, the authors build a detailed parameterized operational model (the implementation) in Dafny — a hierarchical MSI-coherent transition system over tiles, with environmental transitions that let misses, evictions, prefetches, and callbacks interleave arbitrarily — and prove soundness: for all programs, every execution of the implementation model is allowed by the ISA-level MCM. The proof is fully machine-checked, split into a refinement (operational → intermediate state machine that builds partial graphs) and an inductive proof (the axioms hold as the graph grows), and is unlocked by designing every axiom to be prefix-closed.

The payoff: the first cache-aware ISA-level MCM; what the authors call the first end-to-end machine-checked proof of an operational implementation against an axiomatic ISA-level MCM; a model parameterized over replacement policy, prefetcher, and NoC so architects keep design freedom; and two transferable insights (aim for prefix-closed axioms; build models that serve both architects and formalists).

🍯 Key takeaways

  • A programmable cache links hidden hardware events (eviction, prefetch, replacement) to program-visible values — so correctness now depends on the memory hierarchy.
  • Phantom addresses (no memory backing) + callbacks create outcomes like "a write you did is forgotten by a later read of the same address."
  • The ISA-level MCM is axiomatic: new events (Rcb/Wcb, Ms/Me, Es/Ee, Fl), a callback-order relation cbo, and axioms incl. an hb augmented with callback orderings (vf, eb, Me→Es, Ee→Ms).
  • A parameterized operational model in Dafny (MSI, hierarchical, environmental transitions) captures täkō across replacement / prefetch / NoC choices.
  • Soundness (impl ⊆ spec) is proven for all programs, fully machine-checked, via refinement to an intermediate state machine + induction.
  • Prefix-closed axioms are the enabling trick; the proof needed 119 invariant clauses and 61K LoC of annotation.
  • Tools: axioms + litmus tests in Alloy; proof in Dafny (Z3). Artifact public; this is the extended arXiv version with extra Section V litmus tests.
Seven everyday pictures

How to think about täkōFormal

Each picture stands alone — read them in any order. Every card ends with an honest note about where the picture stops being true.

The core surprise · Phantom + eviction

Writing on a whiteboard mid-erase

You write a number on a shared whiteboard, turn away, and a janitor (the cache) wipes it and rewrites the room's default. When you turn back to "read your number," it's the default — your write is simply gone. Phantom addresses live only in the cache; an eviction between your write and read silently drops the value, and the re-read triggers a fresh OnMiss instead. A "hidden" hardware event changed your program's answer.

Where it misleads: a janitor wipes randomly; täkō's eviction is governed by a replacement policy, and the MCM precisely captures which interleavings are possible — it's not arbitrary erasure but a constrained set the axioms enumerate.
Soundness · Spec vs implementation

A rulebook and a referee

The ISA-level MCM is the published rulebook; the operational model is a meticulous simulation of the actual game. Soundness is the claim "no play the machine can make breaks a printed rule." You prove it by walking every legal move of the machine and checking it against the book — never the other way around. That's why it's soundness (machine ⊆ rulebook), not completeness (rulebook ⊆ machine).

Where it breaks: a referee watches one game; this proof covers all programs and all parameter choices at once, by induction — closer to proving the rulebook self-consistent than refereeing a match.
Callback order · cbo & serialization

A single-file queue at one counter

For a given address, täkō serializes all its callbacks in FIFO order on the engine — like one bank teller serving customers for that account strictly in line. The MCM captures this as a callback-order relation cbo: a total order over all callback events for one address. Many axioms are just "the queue must contain an X between two Y's" — e.g. an eviction must sit between two OnMisses for the same line.

Where it breaks: the single-file picture is per address only. Callbacks for different addresses run concurrently and can race — which is exactly what the icbsb and wbr tests exploit.
Prefix-closure · Provable axioms

Checking a wall as it's built

A good building rule can be checked at every stage of construction, not only on the finished wall. "Each brick course is level" works mid-build; "the wall is exactly 10 courses tall" does not — it's false until the last brick. The proof builds the execution graph incrementally, so its axioms must be the first kind: prefix-closed, holding for every partial graph. The authors rephrase "every OnMiss-start has a matching end" into "between two starts there's an end" to make it checkable mid-build.

Where it breaks: not every property has a clean prefix-closed restatement; the paper presents it as a best practice to strive for, found to work for täkō, not a free lunch that always exists.
Modeling timing · Environmental transitions

Stagehands who move whenever

Misses, evictions, prefetches, and callbacks aren't tied to the script — like stagehands who can rearrange the set at any moment, independent of the actors' lines. The model captures this with environmental transitions: steps not triggered by any instruction. A load becomes two decoupled steps (perform-if-present; send-request-when-absent), so a fetch can fire from a prefetch instead of a load, and OnMiss-OnEvict can loop forever with no core involvement.

Where it breaks: real stagehands have limits; environmental transitions deliberately over-approximate — they admit more interleavings than any one prefetcher/replacement policy would, which is what makes the soundness proof hold for all such policies.
Synchronization · FlushRange

Flushing the line before you trust it

An OnWB can fire any time after a phantom line is written, so a core read of the published result may race the writeback. FlushRange is the fix — like flushing a pipe and waiting until it runs clear before drinking. It forces all lines in a range out of the cache, runs and blocks on their OnEvict/OnWB, and so establishes an hb edge that orders the publish before the read. The wbf test shows it eliminates the wbr race.

Where it breaks: flushing only helps if no later callback can re-introduce the racing access after the flush commits — the phiR/hatsR tests show flushes that fail exactly because a callback can still run the racing write afterward.
Connecting the layers · Refinement

A translator between two clerks

The detailed machine speaks in cache lines and network messages; the spec speaks in graph events. A refinement proof is a translator (abs) that maps every machine state to a spec state, and shows each machine step either becomes a legal spec step or a graph no-op. Most internal steps (sending a coherence message) translate to "no change to the graph," while the two meaningful steps (ending an OnWB, starting an OnMiss) add nodes.

Where it breaks: a translator can be lossy; here the abstraction must be exact for the steps that matter — the proof carries an inductive invariant ensuring cache/network internals never silently change graph-relevant behavior.
The full explanation

How it actually works

Section by section — the täkō execution model, the ISA-level MCM, the implementation model, and the soundness argument — with the paper's key diagrams rebuilt as visuals you can read at a glance.

1 · The täkō execution model — why intuition breaks

Each tile has a core, L1, private L2, an L3 shard, and an engine that runs callbacks. For a registered virtual-address range, an OnMiss runs on a cache miss (to compute the line's contents), and an OnEvict (clean) or OnWB (dirty) runs on eviction. Callbacks have thread semantics: they run on dedicated engines in parallel with core threads and can touch regular addresses too — unlike prior trap-based schemes (IMO, ECMon) where handlers have function-call semantics and can't really race the core.

The running example (paper Fig. 2/4): a thread writes 1 to phantom [x] then reads it. If [x] stays cached, the read sees 1. But if an eviction intervenes, the re-access misses, the OnMiss re-runs, and — since phantoms aren't memory-backed — the prior write is dropped entirely. The previously-hidden replacement/prefetch policy now changes the functional result.

🐙
Concretely (Fig. 4a): with an added OnWB on [x] that writes 1 to a regular address [y], the outcome r1=2, r2=0 is impossible on täkō. Seeing r1=2 means an intervening OnMiss ran; that requires an eviction; the eviction of dirty [x] must be an OnWB; per-address serialization then forces the write of [y] to complete before the read of [y] — so r2 can't be 0.

2 · The ISA-level MCM — new events and relations

The model is axiomatic (execution = graph; legal = satisfies all axioms in Fig. 6). To reason about caches and callbacks it adds:

New pieceMeaning
Rcb / Wcbphantom (callback) read / write; RMWcb for atomics. Regular accesses stay R / W.
Ms / MeOnMiss start / end events.
Es / Eeeviction start / end; a dirty bit distinguishes OnEvict (false) from OnWB (true).
Fla flush event from FlushRange.
cbocallback order: a total order on all callback events for a single address — encodes täkō's per-address FIFO serialization.
viscbwhich phantom writes / OnMiss results are visible to a phantom read or eviction callback.

The section then derives the axioms by progressively outlawing wrong execution graphs for the r1=2, r2=0 outcome (paper Fig. 7a–f), each fix adding structure:

# the chain of fixes that forbids r1=2, r2=0  (Fig. 7a → 7f)
VfWf  every Rcb/Wcb must be preceded in cbo by an Me   # a phantom read needs a prior OnMiss source
CboVal viscb ⊆ val                                   # a phantom read sees the value actually visible to it
OEInt  an eviction (Es/Ee) must sit between two OnMisses  # can't OnMiss twice with nothing evicted between
EvDirty/WbDirty dirty bit ⇔ a write happened before evict # OnWB iff line was written; else OnEvict
hb augmented add vf, eb, (Me;cbo;Es), (Ee;cbo;Ms) to hb  # callback orderings enter happens-before
Vis   a read can't observe an hb-shadowed older write   # the final axiom that forbids the outcome
💡
The key move is augmenting hb. Traditional hb = (sb ∪ sw)+ can't see callback orderings, so visibility reasoning misses the edge from the OnWB's write of [y] to the later read of [y]. Adding vf (Me → the Rcb/Wcb it populates), eb (Ee → the next Fl), and the Me→Es / Ee→Ms cbo edges restores it — and the Vis axiom then forbids r2=0.

3 · The implementation model — a parameterized Dafny state machine

To check the spec is faithful, the authors build an operational model: a transition system (S, Init, Next) in Dafny (which discharges proof obligations to Z3). It is hierarchical — each tile is its own transition system (Core+L1, Engine+L1, L2, L3 shard), with a single Memory machine and a Network machine holding in-flight messages. Caches are inclusive; coherence is a hierarchical directory-based MSI protocol (HieraGen-style proxy L2). Supported instructions: Load, Store, RMW, Flush, Branch; each commits via a PerformInst step in a Core or Engine.

Two design choices make the model both realistic and provable across many designs:

  • Parameterization over cache sizes, the program, core count, and address-to-L3-bank mapping — so proofs hold for all configurations, not one.
  • Environmental transitions (§VI-D): steps not tied to instructions. A load is split into perform-if-present and send-GetS-when-absent, so a fetch can come from a prefetch, and OnMiss-OnEvict loops can repeat with no core action. This over-approximates any prefetcher / replacement policy, keeping the proof valid across them.

4 · The soundness argument — refinement + induction, via prefix-closure

The goal (§VII): for all programs, every execution of the operational model is allowed by the ISA-level MCM. Two obstacles, two fixes.

Obstacle A — graphs vs. steps. Axioms are properties of a full execution graph; the operational model builds executions incrementally. Fix: an intermediate state machine (ISM) whose state is a partial execution graph and whose transitions are graph additions. Prove each axiom (strengthened with an inductive invariant) holds at Init and is preserved by Next on partial graphs.

Obstacle B — partial graphs falsify natural axioms. "Each Ms has a matching Me on the same thread" is true of full executions but fails right after an OnMiss starts (its Me isn't added yet) — so induction can't prove it from a reachable state. Fix: make every axiom prefix-closed. Rephrasing it to MeInt ("between every two Ms there is an Me") keeps the intent and holds for partial executions, so it's inductively provable. The paper's claim is strong: if an axiom is prefix-closed and true of the system, an inductive invariant proving it always exists.

The end-to-end proof divides into two machine-checked halves (paper Fig. 18):

I  (induction, §VII-B):   each axiom holds as the ISM grows the partial graph
R  (refinement, §VII-E):  abs() maps every operational state → an ISM state,
                          and every Next(s,s') maps to a valid ISM step
                          or a graph NoOp (internal cache/Network steps)
# Decoupling: R abstracts away data movement; I only reasons about graph growth.
# Both are machine-checked — claimed first end-to-end such proof of an
# operational impl against an axiomatic ISA-level MCM.

Coherence itself is assumed (the MSI protocol is well-studied and known correct) so the proof focuses on the consistency/täkō interface — but the authors explicitly model all protocol transient states and verify the directory-level dirty bit is preserved (so OnEvict vs OnWB fire correctly). Even granting coherence, the proof needed 119 invariant clauses and 61K LoC of annotation.

⚠
Honest scope. This is soundness (impl ⊆ spec), not completeness, and coherence is assumed rather than re-proven. Litmus-test outcomes are confirmed in Alloy; the operational↔spec correspondence is the part that is fully machine-checked in Dafny.

5 · Using the model — FlushRange as synchronization, and where it fails

§V shows programmers using the MCM via litmus tests. A recurring lesson concerns FlushRange. Because an OnWB can fire any time after a phantom write, a core read of the published result can race the writeback (the wbr test). Adding a FlushRange (the wbf test) forces the line out, blocks on its OnWB, and creates an hb edge (eb) that orders publish-before-read — forbidding r1=0.

🍯
The subtle requirement (from phiR/hatsR): a flush only kills a race if no callback can run the racing access after the flush commits. If an OnWB/OnEvict can still fire post-flush and perform the write (e.g. branching on how many updates happened, or logging an edge after traversal), the race survives. The "NR" variants fix this by making the callback side-effect-free once it can't legitimately act.
Follow the reasoning, step by step

Worked examples

Four traces carried all the way through — the running forbidden outcome, a callback-triggered reordering, a FlushRange race repair, and one refinement step of the proof.

① Why r1=2, r2=0 is impossible (the running example)

Fig. 4a: Core 0 does (i1)[x]←1; (i2)r1←[x]; (i4)r2←[y]. [x] is phantom with an OnMiss returning 2 and an OnWB writing 1 to regular [y]. Why can't a reader see r1=2 yet r2=0?

  1. r1=2 can't come from (i1). (i1) wrote 1. The only source of a 2 is an OnMiss for [x] (Me([x],2)). By VfWf, that Rcb/load must be preceded in cbo by an Me — so an OnMiss ran.
  2. An OnMiss ran after (i1). But (i1) already brought [x] into the cache. To OnMiss again, the line must first have been evicted. By OEInt, an eviction (Es/Ee) sits between the two OnMisses.
  3. That eviction is an OnWB, not OnEvict. [x] was written (dirty) by (i1)'s Wcb. By WbDirty, a dirty eviction must be an OnWB — and the OnWB's body writes 1 to [y].
  4. The write of [y] is happens-before the read of [y]. The augmented hb chains: OnWB's [y]←1 → (via callback orderings, Me→Es/Ee→Ms and vf) the Rcb of [x] that (i2) reads → (no load-load reorder on x86) the read of [y] in (i4).
  5. Vis forbids r2=0. Since [y]←1 is hb-before the read, the read can't observe the older initial 0. So r1=2 ⇒ r2=1; the pair r1=2, r2=0 is impossible.
🍯
Insight: a purely "hidden" hardware sequence (evict → OnWB → re-OnMiss) is forced into existence by the observed value r1=2, and that forced sequence has a program-visible consequence ([y]=1). The MCM makes this chain explicit instead of folklore.

② A callback-triggered reordering that IS allowed (icbsb)

Fig. 10a: two cores write phantom [x] and [y]; their OnWBs then do writes and reads to regular [a]/[b], recreating the store-buffering (sb) pattern inside callbacks. Is r1=0, r2=0 allowed?

  1. The OnWBs for [x] and [y] are for different addresses. täkō's per-address serialization (cbo) says nothing about ordering callbacks across different addresses — they can run concurrently.
  2. So the two callbacks form classic store buffering. Each writes its own location then reads the other's, with no enforced order between the two callbacks.
  3. For both reads to return 0, the instructions in at least one callback must reorder — exactly the sb condition. The MCM does not require program order between different addresses within a callback (mirroring how it doesn't for non-täkō threads).
  4. The axioms admit the fr;sb cycle among the callback instructions, so r1=0, r2=0 is allowed.
💡
Design payoff stated by the paper: by not forcing program order inside callbacks, the MCM gives architects freedom to build engines that buffer and reorder memory ops — those designs still conform. Permissiveness here is deliberate, not an oversight.

③ FlushRange repairs a race (wbr → wbf)

Fig. 11: Core 0 updates phantom [x] in (i1); its OnWB publishes to regular [y]; (i3) reads [y]. Without a flush the access to [y] races; wbf inserts (i2) FlushRange[x]. Trace the fix.

  1. wbr (no flush): a race. The OnWB of [x] can fire any time after (i1), so (i5)'s publish and (i3)'s read of [y] are unordered by hb — conflicting accesses, one a write ⇒ race.
  2. Add FlushRange[x] ((i2)). By EbWf, the flush must commit either before the OnMiss of [x] or after its OnWB.
  3. Before-OnMiss is ruled out. Committing the flush before the OnMiss creates a forbidden cbo cycle (CboWf1). So the flush commits after the OnWB.
  4. Now an eb edge appears. Ee → Fl (the eviction-end of [x] to the flush) is in eb ⊆ hb. Composed with the two program-order (sb) edges, this yields an hb edge from (i5) to (i3) — the race is gone.
  5. r1=0 becomes forbidden. The new hb + fr cycle between (i3) and (i5) violates Vis. FlushRange has been formalized as a synchronization primitive.
🍯
Insight: the flush works because it blocks on the OnWB and its position in cbo is pinned (after the OnWB), injecting a real hb edge. Contrast with phiR, where a later callback can still perform the racing write after the flush commits — there the same construct fails.

④ One refinement step of the soundness proof

Fig. 18: the operational model takes four steps s1→s5 for Fig. 4a, but only two touch the execution graph. Walk what abs() does at each.

  1. s1→s2: End OnWB of [x]. This adds an Ee (and the [y]←1 effect) to the graph. abs(s1)→abs(s2) is a valid ISM transition that grows the partial graph.
  2. s2→s3: L3 sends an OnMiss request into the Network. Pure data-movement; the execution graph is unchanged. abs(s2)→abs(s3) is a graph NoOp.
  3. s3→s4: Engine receives the OnMiss request. Still internal — another NoOp on the graph. But it's not irrelevant: it determines which address/line the next step will populate.
  4. s4→s5: Start OnMiss of [x]. Adds an Ms node — a valid ISM transition. Its content was fixed by the internal messages in steps 2–3, which the inductive invariant tracks.
  5. Compose. Every operational step is a valid ISM step or a NoOp, so the whole trace s1→s5 abstracts to an ISM trace abs(s1)→abs(s5) over partial graphs — on which the (prefix-closed) axioms hold inductively.
💡
This is the modular split in action: the refinement half only worries about "does this step change the graph, and if so how?"; the inductive half only worries about "does adding this node keep every axiom true?" Neither half re-derives coherence — that's assumed.
Test yourself

Practice problems

Try each before opening the answer. Every one ends with the common misconception it's checking for.

Problem 1 · easy
A programmer writes 1 to a phantom address, then immediately reads it back on the same thread, and is shocked to read a different value. Under täkō, what hidden event explains this, and could it ever happen for a regular (memory-backed) address?
Show answer
An eviction between the write and the read. Phantom addresses aren't backed by memory, so once the dirty line is evicted (and, here, its value not re-derivable), the re-access misses and re-runs the OnMiss, which produces its own value — the prior write is forgotten. For a regular address this can't silently happen: the value is persisted to memory, so a re-read still observes it.Misconception: that a load on the same thread always sees that thread's most recent store to the address. With phantom addresses + callbacks, an intervening cache event can drop the store entirely.
Problem 2 · medium
Why does täkōFormal prove soundness (every implementation execution is allowed by the MCM) rather than completeness? What would completeness have bought, and why is soundness the safety-critical direction for a programmer?
Show answer
Soundness guarantees the hardware never produces a behavior the spec forbids — so a programmer who reasons "the MCM says outcome O is impossible" is never blindsided by the real machine. That's the property you need to trust the spec as a contract. Completeness (every MCM-allowed behavior is realizable by this implementation) would tell you the spec isn't needlessly loose, but a too-loose spec is still safe to program against; it just permits behaviors this particular implementation happens never to exhibit.Misconception: that soundness and completeness are the same "the model matches the hardware" claim. They're opposite inclusions; täkōFormal proves impl ⊆ spec.
Problem 3 · medium
An architect wants to swap in a more aggressive prefetcher and a different cache replacement policy. Does the soundness proof still hold? Point to the specific modeling choice that makes your answer true.
Show answer
Yes. The operational model uses environmental transitions — memory-hierarchy steps (fetches, evictions, callback triggering) are decoupled from instructions and may fire at any time their preconditions hold. This deliberately over-approximates any prefetcher or replacement policy: whatever such a policy could do is already an admissible interleaving in the model. The model is also explicitly parameterized over cache sizes and NoC specifics. So a proof over this model covers all those choices at once.Misconception: that the proof is tied to one microarchitecture. The over-approximation and parameterization are exactly what generalize it — at the cost of a harder proof.
Problem 4 · hard
The axiom "every OnMiss-start Ms has a corresponding OnMiss-end Me on the same thread" is true of every complete täkō execution, yet the authors could not prove it by induction and replaced it. Explain precisely why, and what property the replacement has.
Show answer
It is not prefix-closed. Induction reasons over partial execution graphs built one event at a time. Right after a transition adds an Ms (an OnMiss starts), the matching Me has not yet been added, so the axiom is false on that partial graph — and that partial graph is reachable from a reachable state. Thus P(s) ∧ Next(s,s′) ⇒ P(s′) fails, and no inductive proof goes through. The replacement, MeInt ("between every two Ms there is an Me"), expresses the same intent but is true of every partial graph (vacuously, while an OnMiss is in flight), so it's prefix-closed and inductively provable. The paper claims prefix-closed + true ⇒ an inductive invariant always exists.Misconception: that an axiom being true of all real executions makes it inductively provable. Truth on full executions is not enough; it must survive every partial graph along the way.
Problem 5 · medium
In icbsb, two OnWB callbacks (for different phantom addresses) recreate store buffering and the outcome r1=0, r2=0 is allowed. In the running example, callback orderings instead forbid an outcome. Reconcile: when do callbacks impose ordering and when don't they?
Show answer
täkō serializes callbacks per address (the cbo total order is for a single address). So callbacks to the same address are strictly ordered, and the MCM derives strong consequences (e.g. an OnWB must complete before a later OnMiss for that address, which is what forbids r2=0 in the running example). Callbacks to different addresses have no such order — they can run concurrently — so icbsb's two OnWBs are unordered and the sb reordering is permitted. The MCM also doesn't enforce program order between different addresses within a callback.Misconception: that callbacks are globally serialized. Serialization is per-address; cross-address callbacks race, which is the source of both flexibility and bugs.
Problem 6 · hard
A FlushRange eliminates the race in wbf but the analogous flush in phiR (and hatsR) does not. State the general requirement for FlushRange to remove a race, and explain why phiR violates it.
Show answer
Requirement: after the FlushRange commits, no callback may still run an access that causes the race. In wbf, once the flush of [x] commits (after the OnWB), there is no remaining callback that writes the published location, so the publish is ordered before the read by an eb ⊆ hb edge. In phiR, the OnWB branches on how many updates were published and can write to [z] on a second firing that occurs after the other core's flush has committed — so the flush can't prevent that write from racing a read of [z]. phiNR fixes it by having the OnWB branch on the evicted value so that, once a flush commits, any write it could still do is already guaranteed complete.Misconception: that flushing always synchronizes. It only works if the flush's hb edge actually covers every racing access — callbacks that can re-fire and act post-flush defeat it.
Problem 7 · medium
The proof "assumes coherence" rather than re-proving the MSI protocol correct, yet the authors say they still explicitly model all protocol transient states. Why is that not a contradiction — and give the concrete täkō-specific fact they verify at the coherence interface.
Show answer
Assuming coherence means they don't re-derive that the well-studied MSI protocol provides single-writer/multiple-reader, etc. — that's known. But coherence-consistency interface bugs can still arise from how täkō's features interact with transient protocol states, so they model those states anyway. The concrete fact they verify: the dirty bit at the directory level is accurately preserved by the protocol, because OnEvict vs OnWB selection depends on it — get the dirty bit wrong and the wrong eviction callback fires, breaking consistency. Once phantom data populates a directory entry it's indistinguishable to the protocol from memory-fetched data, which is why assuming coherence is sound here.Misconception: that "assuming coherence" means ignoring the protocol. They assume its coherence guarantee but still model its states to catch interface issues.
Problem 8 · easy
You are reading the arXiv "extended version" of täkōFormal for a literature review. Name two things you should be careful about when citing specific claims, results, or figures from it.
Show answer
(1) It may differ from the ISCA 2026 camera-ready: the extended version explicitly adds material — more litmus tests in Section V — so section/figure numbering, the exact test set, and wording can diverge from the published paper. (2) Quantitative proof-effort figures (e.g. 119 invariant clauses, 61K LoC), tool versions (Dafny 4.11.0, Alloy 6.2.0), and the artifact DOI are implementation specifics that can change between versions and revisions; verify against the exact version you cite, and distinguish what the paper proves (soundness; coherence assumed) from what it confirms empirically (Alloy litmus outcomes).Misconception: that an arXiv extended version is interchangeable with the camera-ready. Treat them as related-but-distinct artifacts and cite the version you actually used.
Quick answers

Frequently asked questions

Twelve questions a comp-arch reader asks first. Tap any to expand.

Q1What problem is täkōFormal actually solving?▾
täkō lets cache events (miss, eviction, writeback) run user callbacks in-cache. That couples previously-hidden microarchitecture (prefetch timing, replacement policy) to program-visible values, making programs hard to reason about. täkōFormal gives a precise ISA-level memory consistency model that captures these semantics, and proves a detailed hardware model conforms to it — so programmers can reason about correctness without knowing the microarchitecture.
Q2What's a "phantom address" and why does it break conventional MCMs?▾
An address not backed by main memory; its value only exists if an OnMiss callback created it in the cache. Conventional MCMs assume a load can always fall back to memory, so they have no way to express "this value only exists because a callback ran, and an eviction could erase it." The paper adds dedicated Rcb/Wcb events and callback axioms to handle exactly this.
Q3What are the new ingredients in the ISA-level MCM?▾
New events: Rcb/Wcb (phantom access), Ms/Me (OnMiss start/end), Es/Ee (eviction start/end, with a dirty bit selecting OnEvict vs OnWB), and Fl (flush). A new relation cbo (callback order) gives a per-address total order over callback events. And hb is augmented with callback orderings (vf, eb, and Me→Es/Ee→Ms cbo edges) so the visibility axiom Vis works across callbacks.
Q4What exactly does "soundness" mean here, and is completeness proven?▾
Soundness = for all programs, every execution of the operational implementation model is allowed by the ISA-level MCM (impl ⊆ spec). Completeness (the reverse inclusion) is not proven. Soundness is the direction that lets a programmer trust the spec as a contract: the hardware never does something the model forbids.
Q5How is the implementation model built, and what does it cover?▾
A hierarchical transition system in Dafny: each tile is Core+L1, Engine+L1, L2, L3 shard; plus a single Memory machine and a Network machine for in-flight messages. Caches are inclusive; coherence is hierarchical directory-based MSI. It supports Load/Store/RMW/Flush/Branch. It's parameterized over cache sizes, the program, core count, and address-to-L3 mapping, and uses environmental transitions so the proof holds across prefetchers, replacement policies, and NoC designs.
Q6What are "environmental transitions" and why are they essential?▾
Transitions not triggered by any instruction. A load is split into perform-if-present and send-request-when-absent, so a fetch can come from a prefetch rather than that load, and OnMiss-OnEvict can loop with no core action. They over-approximate cache behavior, which is what makes a single soundness proof valid under arbitrary prefetching and replacement policies.
Q7What is prefix-closure and why does the whole proof hinge on it?▾
An axiom is prefix-closed if, holding for a full execution graph, it also holds for every partial graph built along the way. The inductive proof traverses partial graphs, so non-prefix-closed axioms (true only at the end) can't be proven. The authors rephrase such axioms (e.g. "every Ms has an Me" → "between two Ms there's an Me", MeInt). Their claim: a prefix-closed, true axiom always admits an inductive invariant — making state-machine verification of the MCM possible.
Q8How is the end-to-end proof structured?▾
Two machine-checked halves. (I) Induction: an intermediate state machine (ISM) builds partial execution graphs, and each (prefix-closed) axiom is shown to hold as the graph grows. (R) Refinement: an abstraction function maps each operational state to an ISM state, and every operational transition maps to a valid ISM step or a graph no-op. The decoupling lets refinement ignore axioms and induction ignore cache/network internals.
Q9Why assume coherence instead of proving it?▾
The MSI protocol used is well-studied and known to be coherent, and coherence is tangential to täkō's novelty: once phantom data populates a directory entry, it's indistinguishable from memory-fetched data to the protocol. Assuming it avoids re-proving coherence while focusing effort on the consistency/täkō interface. The authors still model all transient protocol states and verify täkō-relevant facts (e.g. directory-level dirty-bit preservation).
Q10How does this connect to the Check/Trippel-Manerkar lineage?▾
It's squarely in that tradition of formal MCM verification at the HW/SW interface — TriCheck, PipeCheck/CCICheck, RTLCheck, PipeProof, ILA-MCM. Those tools verify implementations (often RTL or μarch) against memory models, sometimes via bounded methods or intermediate machines. täkōFormal pushes further on two fronts: a brand-new cache-aware ISA-level MCM, and a fully machine-checked, all-programs, end-to-end soundness proof of an operational model against an axiomatic ISA MCM (which it claims is a first), using environmental transitions inspired by Wickerson et al.
Q11What tools were used, and is there an artifact?▾
The MCM axioms and litmus tests are encoded in Alloy (6.2.0); the operational model and soundness proof are in Dafny (4.11.0), which discharges obligations to the Z3 SMT solver. There's a public, MIT-licensed artifact (GitHub, archived on Zenodo) with two scripts: one runs the Alloy litmus tests, one verifies the Dafny end-to-end proof.
Q12This is the arXiv "extended version" — how does that matter?▾
It's the extended version of the ISCA 2026 paper and explicitly adds material — additional litmus tests in Section V — so the test set, section/figure numbering, and some wording may differ from the camera-ready. Treat quantitative figures, tool versions, and the exact litmus set as version-specific, and cite the version you actually read.

Core terms (täkō & the MCM)
täkōA programmable memory hierarchy (Schwedock et al., ISCA 2022) where engines run user callbacks on cache events. The subject system; not introduced by this paper.
Callback (OnMiss / OnEvict / OnWB)User code run by the tile's engine on a cache miss, clean eviction, or dirty writeback for a registered address range. Has thread semantics (runs in parallel with cores).
Phantom addressAn address with no main-memory backing; its value exists only if an OnMiss created it in the cache. Denoted with Rcb/Wcb events.
ISA-level MCMThis paper's contribution: an axiomatic memory consistency model that constrains values produced by loads, callbacks, and cache events at the ISA level.
cbo (callback order)A per-address total order over all callback events (Rcb, Wcb, RMWcb, Ms, Me, Es, Ee), reflecting täkō's per-address FIFO serialization.
Ms / Me, Es / EeOnMiss start/end and eviction start/end events. A dirty bit on Es/Ee distinguishes OnEvict (clean, false) from OnWB (dirty, true).
Augmented hbhappens-before extended with callback orderings — vf (Me→the Rcb/Wcb it populates), eb (Ee→next Fl), and Me→Es/Ee→Ms cbo edges — so the Vis axiom reasons across callbacks.
FlushRangeA synchronization primitive that evicts all lines in a range, runs and blocks on their OnEvict/OnWB, and thereby injects an hb edge usable to eliminate races.
Core terms (the proof)
Operational / implementation modelA Dafny transition system (Init, Next) modeling täkō hardware: tiles, MSI coherence, a Network of messages, parameterized over sizes/policies/NoC.
Soundnessimpl ⊆ spec: every execution of the operational model is allowed by the ISA-level MCM, for all programs. The direction proven here.
Environmental transitionA transition not triggered by an instruction (prefetch fetch, eviction, callback firing). Over-approximates cache timing so the proof spans all prefetch/replacement policies.
Prefix-closureAn axiom property: if it holds for a full execution graph it holds for every partial graph along the way. Required for inductive provability; a stated best practice.
Intermediate state machine (ISM)A simplified state machine whose state is a partial execution graph; the bridge between the operational model and the axioms.
RefinementProof that each operational state abstracts (via abs) to an ISM state and each operational step to a valid ISM step or a graph no-op.
Inductive invariantA strengthening Inv ⇒ P of a target property that is preserved by every transition, enabling an inductive proof. Here: 119 clauses.
Vis / viscbVis is the visibility axiom forbidding a read from observing an hb-shadowed older write; viscb defines which phantom writes/OnMiss results are visible to phantom reads and eviction callbacks.
The täkō litmus tests (from §V & the artifact table)
paper-ex (Fig. 4a)r1=2, r2=0 — forbidden. The running example tying an OnWB's regular-address write to a phantom re-read.
mp / mprmw / mpcbMessage passing. Plain mp: r1=1,r2=0 allowed. With RMW synchronization (mprmw, regular) or a phantom OnMiss (mpcb): the same outcome is forbidden.
icbsbStore buffering recreated inside two cross-address OnWBs. r1=0, r2=0 allowed — callbacks don't enforce program order between different addresses.
wbr / wbfWriteback race / writeback flush. wbr (no flush): racy. wbf (with FlushRange): no race, r1=0 forbidden.
phiR / phiNRMulticore scatter-update publish. phiR (OnWB v1): racy despite flushes. phiNR (OnWB v2, branches on evicted value): no race, r1=0,r2=0 forbidden.
hatsR / hatsNRHATS graph-traversal case study. hatsR (OnEvict logs post-traversal): racy. hatsNR (OnEvict logs only valid edges): no race, r1=r2 forbidden.
