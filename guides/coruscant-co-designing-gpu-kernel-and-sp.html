MICRO 2025 · A Field Guide

Compressing irregular sparsity in LLM weights without sacrificing performance

Pruning LLM weights in arbitrary patterns saves memory and preserves accuracy, but GPUs won't accelerate it. This is the plain-English story of Coruscant, a sparse format plus GPU kernel plus tensor core extension that makes unstructured sparsity (30–70%) both compact and fast.

📄 "Coruscant" · Joo, Hosseini, Hadidi, Asgari 🎓 Assumes GPU basics — primers recap them ⏱ ~30 min read
🗜️

Represent arbitrary sparsity patterns with a bitmap instead of indices, so the compressed matrix stays smaller than the dense one — and add a tiny hardware unit to decode the bitmap inside the tensor core, avoiding decompression overhead.
One format, two execution modes, zero structural restrictions.

Before you begin

Eight concepts Coruscant builds on

Each card expands one idea the paper assumes. Read them in order the first time.

efficient compression memory-bound SpMM Coruscant machinery baseline (dense / prior sparse)
1Unstructured vs structured sparsity▾

Structured sparsity imposes a fixed pattern — e.g. "zero out every 4th channel," "prune entire attention heads." Hardware maps these patterns efficiently, but the pruning algorithm can't choose freely. Unstructured sparsity lets you zero any weight based on its importance, keeping accuracy better at higher sparsity, but GPUs struggle to exploit the irregular pattern.

NVIDIA's 2:4 format (two zeros in every four consecutive weights) is a middle ground — slightly more flexible than coarse structure, but still locked to 50% sparsity and a rigid micro-pattern. Coruscant targets fully arbitrary patterns with no fixed structure.

Now you're ready: Coruscant is designed for unstructured sparsity where no two rows have the same zero positions — the hardest case to compress and accelerate.
2LLM decode phase is memory-bound SpMM▾

In LLM inference, the decode phase dominates total runtime (Figure 1a in the paper). Each decode step computes weight projections: sparse matrix (pruned weights) × dense matrix (activations), where the batch size N is small (8–32 tokens). These "skinny" matrix multiplications have M and K dimensions far larger than N, making them memory-bound — performance limited by memory bandwidth, not compute.

Reducing the data transferred from GPU global memory to on-chip processors is the primary speedup lever. Computation skipping (doing fewer MACs) helps less, because the bottleneck is already memory.

Now you're ready: Coruscant's bitmap compression cuts memory traffic; the hardware decompressor avoids the cost of unpacking tiles in shared memory.
3Why prior sparse formats exceed 100% at moderate sparsity▾

CSR, COO, and Flash-LLM's Tiled-CSL all store indices (16-bit pointers) for each non-zero. At 50% sparsity in a float16 matrix, you have one 2-byte value and one 2-byte index per non-zero — which already equals the original dense size, before counting offset pointers. The compressed representation is larger than the dense matrix.

Figure 3 in the paper shows CSR and Tiled-CSL exceed 100% compression ratio at 50% sparsity for a 9216×9216 matrix. COO fares worse (90% even at 70% sparsity). This makes them useless for the 30–70% sparsity range where LLM pruning operates.

Now you're ready: a bitmap-based format (one bit per potential element) compresses better because it doesn't pay the per-non-zero index cost.
4GPU memory hierarchy and data movement▾

A GPU has global memory (VRAM, large but slow), shared memory inside each streaming multiprocessor (SM, small but faster), and registers (tiny, fastest). Data flows: global → registers (or shared) → compute → registers → global writeback. Each hop has a bandwidth and latency cost.

In memory-bound kernels, the dominant cost is the global-to-SM transfer. Flash-LLM reduces this by loading compressed tiles into registers, then decompressing into shared memory before tensor core execution. Coruscant goes one step further: decompress inside the tensor core, eliminating the shared memory write and its stalls.

Now you're ready: Coruscant's bitmap decoder sits between the register file and the tensor core's MAC units, so no intermediate decompression stage pollutes shared memory bandwidth.
5Tensor cores and MMA instructions▾

NVIDIA tensor cores are specialized matrix-multiply units inside each SM. They operate on small tiles — e.g. 16×16 × 16×8 — and are programmed via MMA (matrix-multiply-accumulate) instructions. Volta introduced dense MMA; Ampere added 2:4 sparse MMA, which selects two out of four operands using a 2-bit index, effectively doubling K dimension throughput.

Standard tensor cores require either a dense tile or the 2:4 compressed format. Arbitrary sparsity must be decompressed to dense before execution, incurring overhead. Coruscant extends the tensor core to accept a bitmap-compressed tile directly.

Now you're ready: the Coruscant Sparse Tensor Core is a minimal extension — a bitmap decoder plus selector logic — that fits between the register file and the existing MAC array.
6Shared memory bank conflicts▾

Shared memory in an SM is divided into 32 banks, each 4 bytes wide. If multiple threads in a warp access the same bank simultaneously (a bank conflict), those accesses serialize, stalling the warp. Conflict-free access requires each thread to touch a different bank.

Coruscant uses column-wise tiling: each thread decompresses two 64-element columns. Because columns are laid out contiguously, successive threads write to successive banks in round-robin, avoiding conflicts. Row-wise or rectangular tiling would cause conflicts (Figure 9 in the paper).

Now you're ready: the tile shape (64 columns per warp) is chosen to eliminate bank conflicts during decompression, a design decision that speeds up the software kernel.
7Prefill vs decode, and batch size▾

Prefill processes the input prompt (N = batch × sequence length, often large). Decode generates one token at a time (N = batch, typically 8–32). Prefill is a one-time cost; decode iterates hundreds of times. Figure 1a shows decode dominates total latency for Llama 2.

Small N makes decode SpMM memory-bound. Larger N (batch 64+) shifts the workload toward compute-bound, where dense cuBLAS's higher occupancy can outweigh Coruscant's compression gains (Figure 24b). But long-context LLMs with large KV caches constrain batch size, keeping N small in practice — Coruscant's sweet spot.

Now you're ready: Coruscant is optimized for decode (small N) where memory bandwidth is the bottleneck, not prefill (large N).
8The roofline model and operational intensity▾

A roofline plot shows peak FLOPs (compute ceiling) and memory bandwidth (memory wall). Operational intensity = FLOPs / bytes transferred. If intensity is low (memory-bound), you're limited by bandwidth; if high (compute-bound), by FLOPs.

Figure 1b plots prefill (high intensity, near compute roof) and decode (low intensity, bandwidth-limited). Coruscant's bitmap compression raises operational intensity by reducing bytes transferred without changing FLOPs, pushing the kernel higher on the roofline (Figure 11).

Now you're ready: Coruscant's two-stage improvement — compression (raise intensity) + hardware decompression (remove instruction overhead) — shifts decode SpMM from memory-wall to near-optimal.

The whole story in one place

Coruscant in a nutshell

The problem, the bitmap-based solution, and how the kernel + hardware extension connect.

LLM pruning at 30–70% sparsity preserves accuracy better than structured or 2:4 patterns, but existing sparse formats (CSR, COO, Tiled-CSL) store per-non-zero indices that bloat the representation beyond the dense matrix size at moderate sparsity. GPUs can't compress it effectively, so they can't reduce memory traffic, leaving unstructured sparsity slow.

Coruscant fixes this with three components: (1) a bitmap-based sparse format that represents each 64-element tile with a 64-bit bitmap plus the non-zero values (no per-element indices), compressing a 50% sparse matrix to 56% of dense size; (2) a GPU kernel that loads compressed tiles into registers, decompresses them into shared memory using CLZ (count leading zeros) to iterate the bitmap, then executes dense MMA on tensor cores — this alone achieves up to 2× speedup over cuBLAS on commercial GPUs; (3) a Sparse Tensor Core extension that adds a bitmap decoder inside the tensor core to select non-zeros directly from registers without the shared memory decompression step, reaching 2.75× speedup over cuBLAS with under 0.02% area overhead.

On Llama 2 7B, Coruscant's software kernel gives an additional 135 tokens/sec, and the hardware extension an additional 206 tokens/sec, at 50% sparsity, reaching up to 847 tokens/sec total. The compressed format also frees GPU memory for larger batch sizes and longer contexts.

🍯 Key takeaways

  • Bitmaps compress better than indices at moderate sparsity (30–70%): a 64-bit bitmap per 64-element tile avoids per-non-zero pointer overhead.
  • Column-wise tiling (each thread decompresses two columns) eliminates shared memory bank conflicts during decompression.
  • CLZ-based iteration extracts non-zero positions from the bitmap without inner loops, keeping the non-zero vector in registers.
  • The GPU kernel alone (load-as-compressed, compute-as-dense) achieves up to 2× over cuBLAS and 1.48× over Flash-LLM at 70% sparsity.
  • The Sparse Tensor Core adds a bitmap decoder (combinational logic + selectors) to skip decompression, reaching 2.75× speedup with cumulative GPU-level added area of 0.51–1.44 mm² (0.006–0.018% of GPU).
  • Speedup grows with sparsity: 30% sparsity is near break-even vs cuBLAS (compression overhead vs bandwidth savings); 70% sparsity maximizes gains.
  • End-to-end benefit: up to 40% higher token throughput on Llama 2 7B; frees memory for batch size 32 on Llama 2 13B (impossible with dense weights).
Six everyday pictures

How to think about Coruscant

Each picture stands alone — read them in any order. Every card ends with an honest note about where the picture stops being true.

The core idea · Bitmap vs indices

A checkbox grid instead of a catalog

To record which of 64 items are present, you could write down each item's ID number (CSR's index array), or you could check 64 boxes — one bit per slot. If half the items are present, the catalog needs 32 IDs × 2 bytes = 64 bytes; the checkboxes need 64 bits = 8 bytes. The bitmap wins by 8×. That's exactly what Coruscant does: each 64-element tile gets a 64-bit bitmap instead of a variable-length index list.

Where it breaks: at very high sparsity (>90%), indices become shorter than the bitmap because there are so few non-zeros. The crossover is around 94% sparsity (4 non-zeros/tile → 4 indices × 2 bytes = 8 bytes, same as the bitmap). But LLM pruning rarely goes above 70%.
Decompression · CLZ iteration

Scanning a checklist from top to bottom

Given the bitmap 1010 (positions 0 and 2 are present), you scan from the left: count leading zeros (CLZ) gives position 0 → fetch the first non-zero; flip that bit to 0 to get 0010; CLZ again gives position 2 → fetch the second non-zero. Each non-zero is located without an inner loop. This is Algorithm 1 in the paper, running at most 64 iterations (tile size) even if only a few elements are non-zero, which keeps the compiler from spilling registers.

Where it breaks: CLZ is a single instruction, but it's not free — especially at low sparsity where you call it many times. Flash-LLM's direct index lookup is simpler. Coruscant's decompression overhead is why it lags Flash-LLM slightly at 70% sparsity (where both compress well). The hardware decoder eliminates this by doing CLZ in parallel combinational logic.
Bank conflicts · Column tiling

Parallel loading docks with one-way lanes

Shared memory has 32 banks, like 32 loading docks. If two workers try to use the same dock at the same time, one waits. Column-wise tiling assigns each thread to specific columns, so thread 0 always writes to bank 0, thread 1 to bank 1, etc. — no collisions. Row-wise or rectangular tiling would have threads stepping on each other's docks.

Where it breaks: this only works if the tile width is a multiple of 32 and each thread writes to consecutive addresses within its columns. If the access pattern were irregular, bank conflicts could still occur. Coruscant's 64-column tiles (two per thread) are carefully chosen to keep the round-robin mapping clean.
Hardware decompression · Bitmap decoder

A vending machine that unpacks snacks on-site

Normally you'd buy a compressed snack pack, take it home, unwrap it, then eat. The decoder is like a vending machine that unwraps the snacks inside and hands you only the ones you wanted — no need to carry the wrapper home or waste counter space unwrapping. Coruscant's bitmap decoder sits inside the tensor core, selecting non-zeros from the bitmap as the MAC units need them, skipping the shared memory decompression step entirely.

Where it breaks: vending machines are big and expensive. The decoder is tiny (cumulative GPU-level added area 0.51–1.44 mm²) because it's just combinational logic (no buffers, no state), but it still adds area and power (~26–75 mW cumulative). The trade-off is worthwhile because it removes a bottleneck (shared memory stalls), but it's not free.
Memory-bound roofline · Operational intensity

A highway where the speed limit isn't the car, it's the traffic

If the road is congested, a faster car (more compute) doesn't help — you're stuck in traffic (memory bandwidth). Coruscant's compression is like carpooling: fewer vehicles (bytes) for the same number of passengers (FLOPs), so the highway flows faster. The roofline plot in Figure 11 shows the kernel shifting upward as operational intensity (FLOPs/byte) increases.

Where it breaks: at very large batch sizes (N ≥ 64), the highway clears up (more compute per byte loaded), and the car's horsepower (cuBLAS's higher occupancy) matters more than carpooling. Coruscant's advantage fades because the bottleneck has moved from memory to compute. But LLM decode typically has N < 32.
Sparsity range · 30–70% sweet spot

Goldilocks and the three pruning levels

Too little sparsity (<30%) and compression gains are too small to justify the decompression overhead (the porridge is too cold). Too much sparsity (>70%) and the model's accuracy degrades exponentially (too hot). The 30–70% range is "just right" — enough sparsity to compress meaningfully without destroying the model. Figure 2 in the paper shows perplexity spiking beyond 70% across multiple pruning methods.

Where it breaks: the 30–70% range is empirical, not a hard boundary. Some models tolerate 80% sparsity with careful layer-wise tuning; others degrade at 60%. The paper targets the range where most pruning methods yield usable models, but it's not universal.
The full explanation

How it actually works

Section by section, with key mechanisms illustrated.

1 · The bitmap-based sparse format

Coruscant partitions the matrix into column tiles of size 64. For each tile, it stores: (1) a 64-bit bitmap (one bit per potential element), (2) the non-zero values in compressed order, and (3) a tile offset array pointing to the start of each tile's non-zero list. Figure 7 in the paper shows the layout.

At 50% sparsity in a float16 matrix, a tile has ~32 non-zeros. Storage: 8 bytes (bitmap) + 64 bytes (32 × 2-byte values) + 2 bytes (offset pointer) = 74 bytes, vs 128 bytes for the dense tile — a 56% compression ratio. CSR/Tiled-CSL would store 32 indices × 2 bytes + 32 values × 2 bytes + offset = 130 bytes, exceeding the dense size.

🌱
Why 64-element tiles? Larger tiles amortize the bitmap overhead better but require more registers per thread. 64 fits comfortably in the register file (each thread holds two compressed tiles + bitmaps) while keeping the bitmap small enough to process efficiently with CLZ.

2 · The GPU kernel pipeline

Coruscant adapts Flash-LLM's load-as-sparse, compute-as-dense pipeline (Figure 4 in the paper). Each iteration has two phases: memory (load compressed tile, decompress to shared memory) and compute (tensor core MMA on the decompressed tile). These phases overlap via pipelining.

A thread block has four warps, each mapped to a GPU sub-partition (SMSP). Within a warp, each thread handles two 64-element columns. The warp computes a 64×64 tile of matrix A (sparse) × a 64×N slice of matrix B (dense) → a 64×N output tile. The kernel iterates through the K dimension, accumulating results.

3 · Decompression algorithm

Algorithm 1 in the paper: each thread iterates up to 64 times (tile size). On each iteration: (1) use CLZ to find the position of the leftmost 1 in the bitmap, (2) fetch the corresponding non-zero from the register-held compressed vector, (3) compute the output index in shared memory, (4) write the non-zero to that position, (5) clear the processed bit from the bitmap. If the bitmap is exhausted (all non-zeros placed), break early.

The fixed-length loop (64 iterations, not dynamic) is critical: it lets the compiler allocate the non-zero vector in registers. A dynamic loop would force the vector into stack memory, incurring expensive global memory transfers.

💡
Why CLZ instead of iterating the bitmap? CLZ is a single instruction that directly yields the bit position. Iterating would require an inner loop to scan each bit, creating more branches and stalling the pipeline. CLZ is faster despite the bit-manipulation overhead.

4 · Padding for coalesced memory access

Each tile's non-zero vector is padded to a multiple of 8 elements, so every thread's load is serviced by a single LDG.128 instruction (128-bit load). Without padding, unaligned accesses would require multiple memory requests, wasting bandwidth. The padding adds a small overhead (up to 14 bytes per tile at 50% sparsity) but ensures full memory coalescing.

5 · The Sparse Tensor Core extension

The Coruscant Sparse Tensor Core (Figure 12) adds a Bitmap Decoder between the register file and the tensor core's MAC units. The decoder has one detector per thread (32 total). Each cycle, the detector: (1) reads the next 4 bits of its thread's bitmap, (2) computes the register indices of the corresponding non-zeros via a chain of adders (Figure 14), (3) emits those indices to the register file, which fetches the values and pads unused slots with zeros, (4) broadcasts four dense-matrix elements to all threads.

The tensor core performs 4 MACs per thread per cycle, just as in dense mode. The only difference is that the sparse operands are selected from a variable-position register file instead of consecutive registers. No changes to the MAC array itself.

🍯
Register bandwidth check: at most 164 operands (128 non-zeros + 32 accumulators + 4 dense elements) + 16 bytes of bitmap per cycle per warp = 344 bytes/cycle, well under Volta's 384 bytes/cycle register bandwidth. The decoder doesn't create a new bottleneck.

6 · Area and power overhead

The paper synthesizes the bitmap decoder in 45nm and scales to 7nm using Stillmaker-Baas equations. Cumulative GPU-level added area (after considering tensor-core counts): 0.51 mm² (Volta), 0.68 mm² (Ampere), 1.44 mm² (Hopper), with power 26–75 mW. As a fraction of the GPU die, this is 0.006–0.018% — negligible. The area is dominated by the adder chains (for computing register indices) and the multiplexers (for selecting non-zeros). No buffers or SRAM are needed, keeping the overhead minimal.

🍯
Comparison to prior work: DSTC and RM-STC require large accumulation buffers (87% and 60% of their area, respectively) to handle scatter-gather for dual-sparse operands. Coruscant avoids this by targeting SpMM (only one operand sparse) and reusing the tensor core's existing accumulation path. Result: 96% less area than DSTC, 95% less than RM-STC (Table 3).
Follow the numbers, step by step

Worked examples

Four traces using the paper's real data.

① Computing compression ratio at 50% sparsity

A 9216×9216 float16 matrix, 50% sparsity. How big is the Coruscant representation vs dense?

  1. Dense size: 9216 × 9216 × 2 bytes = 169,869,312 bytes (~162 MB).
  2. Tile count: partition into 64-element column tiles → (9216 rows × 9216 cols) ÷ 64 = 1,327,104 tiles.
  3. Per-tile storage at 50%: 64-bit bitmap (8 bytes) + ~32 non-zeros × 2 bytes (64 bytes, plus 6 bytes padding to multiple of 8) = 78 bytes.
  4. Total non-zero storage: 1,327,104 tiles × 78 bytes = 103,514,112 bytes.
  5. Offset pointers: 1,327,104 pointers × 2 bytes = 2,654,208 bytes.
  6. Total Coruscant size: 103,514,112 + 2,654,208 = 106,168,320 bytes (~101 MB).
  7. Compression ratio: 101 / 162 ≈ 62% (slightly higher than the 56% in Figure 3 due to padding overhead).
🍯
Why CSR/Tiled-CSL fail: at 50%, each tile has 32 non-zeros. CSR stores 32 indices × 2 bytes + 32 values × 2 bytes + 2-byte offset = 130 bytes per tile → 172 MB total, larger than dense. Coruscant's bitmap avoids per-element indices, saving 64 bytes per tile.

② Tracing the decompression of one tile

A 64-element tile, bitmap = 0xA800000000000000 (positions 0, 2, 4 have non-zeros). Walk through Algorithm 1.

  1. Iteration 0: CLZ(bitmap) = 0 (leftmost bit is 1) → position 0. Fetch non-zero[0] → write to shared_mem[tile_base + 0]. Clear bit 0: bitmap ← 0x2800000000000000.
  2. Iteration 1: CLZ(0x2800000000000000) = 2 → position 2. Fetch non-zero[1] → write to shared_mem[tile_base + 2]. Clear bit 2: bitmap ← 0x0800000000000000.
  3. Iteration 2: CLZ(0x0800000000000000) = 4 → position 4. Fetch non-zero[2] → write to shared_mem[tile_base + 4]. Clear bit 4: bitmap ← 0x0.
  4. Iteration 3: bitmap is now zero, and we've placed 3 non-zeros (the tile's NNZ count). Break.
  5. Result: shared memory positions 0, 2, 4 hold the non-zeros; all other positions in the tile remain zero (from the initial memset).
💡
The loop always runs ≤64 iterations (tile size), not the number of non-zeros, so the compiler can unroll and keep the non-zero vector in registers. If the loop were for i in 0..NNZ, the compiler would spill it to the stack.

③ Speedup breakdown: kernel vs hardware

OPT-30B, M=K=7168, N=8, 50% sparsity (Figure 17 shows cuBLAS vs Coruscant kernel at these parameters). Where do the gains come from?

  1. Memory reduction: Dense FP16 weights for this tile: M × K × 2 bytes = 7168 × 7168 × 2 ≈ 98 MiB. At 50% sparsity, Coruscant compresses to ~56% → ~55 MiB transferred from global memory.
  2. cuBLAS to Coruscant kernel: Compression reduces memory-transfer time proportional to the compression ratio. Figure 18a shows reduced memory stall cycles. Speedup observed is up to 1.5–2× at 50% sparsity (Figure 16, 17).
  3. Decompression overhead: ~36% of kernel time at 50% sparsity, N=8 (from Figure 10). CLZ iterations, bit manipulation, and shared memory writes account for this.
  4. Coruscant kernel to Coruscant-STC: Hardware decoder removes decompression overhead (36% → near zero) and eliminates shared memory stalls. Figure 19b shows shared memory stall cycles drop from ~4,800 to ~200. Tensor core utilization rises from 18% to 42% (Figure 19a). Additional speedup: ~1.3–1.5× on top of kernel-alone gains.
  5. Total observed speedup: ~2× over cuBLAS at 50% sparsity (kernel alone), up to ~2.75× with hardware extension at 70% sparsity (Figures 16, 19c).
🍯
At 70% sparsity, compression ratio improves further (~44% of dense), so kernel-alone speedup grows. Decompression overhead also increases (more CLZ iterations for fewer non-zeros), making the hardware extension's contribution larger — observed as ~1.5× additional gain, yielding ~2.7× total.

④ Offloading latency reduction (70% sparse OPT-66B GeMM)

Figure 23 shows offloaded GeMM for 70% sparse OPT-66B. How does Coruscant reduce transfer time?

  1. Baseline (dense): SSD transfer ~70 ms + compute ~0.27 ms; CPU transfer ~16 ms + compute ~0.27 ms. Total SSD-offloaded latency ~70.27 ms; CPU-offloaded ~16.27 ms.
  2. Coruscant (70% sparse): SSD transfer drops to ~37.9 ms (roughly proportional to 30% compression ratio from Figure 3); compute ~0.14 ms (slightly faster due to reduced data). CPU transfer ~7.8 ms; compute ~0.14 ms. Total SSD-offloaded ~38.04 ms; CPU-offloaded ~7.94 ms.
  3. Speedup: SSD-offloaded: 70.27 / 38.04 ≈ 1.85×. CPU-offloaded: 16.27 / 7.94 ≈ 2.05×. The compression reduces PCIe/SSD bandwidth pressure proportional to the compression ratio.
  4. Key insight: The ~37.9 ms value is the SSD transfer time for the compressed weights in Coruscant, not the compute time. The compression cuts the dominant transfer latency by roughly half, critical for multi-GPU or weight-swapping scenarios.
💡
Figure 23 is for 70% sparsity offloaded GeMM in OPT-66B, not the 50% case. At 70% sparsity, Coruscant achieves ~44% compression ratio (Figure 3), so transfer time drops proportionally. The compute time is much smaller than transfer time, so the speedup is dominated by memory bandwidth savings.

Practice problems

Try each before opening the answer. Every one ends with the common misconception it's checking for.

Problem 1 · easy
A tile has 10 non-zeros out of 64 elements. How many bytes does Coruscant store for this tile (bitmap + values + padding to multiple of 8)?
Show answer
Bitmap: 8 bytes (always 64 bits). Non-zeros: 10 values × 2 bytes = 20 bytes. Padding to multiple of 8: next multiple is 16 values → 6 padding zeros × 2 bytes = 12 bytes. Total: 8 + 20 + 12 = 40 bytes. (Dense tile: 64 × 2 = 128 bytes, so compression ratio = 40/128 = 31%.)Misconception: forgetting the padding. Without it, unaligned loads would waste memory bandwidth. The padding ensures each tile is serviced by a single LDG.128 instruction.
Problem 2 · medium
Why does Coruscant's kernel underperform cuBLAS at very large batch sizes (N ≥ 64), even though compression reduces memory traffic?
Show answer
At large N, the workload shifts from memory-bound to compute-bound (more FLOPs per byte loaded). cuBLAS uses a "buffered tile" approach with higher occupancy (2 warps active per SM vs Coruscant's 1), so it can issue more MMA instructions in flight and saturate the tensor cores. Coruscant's register pressure (holding compressed tiles + bitmaps) limits occupancy, so it can't fully exploit the compute throughput at large N. The compression benefit is overshadowed by the occupancy deficit.Misconception: that compression always helps. It helps when memory bandwidth is the bottleneck, but at large N the bottleneck is compute scheduling, where cuBLAS's higher occupancy wins.
Problem 3 · medium
The paper says CLZ-based decompression is faster than iterating the bitmap bit-by-bit. Explain why, given that both eventually process the same number of bits.
Show answer
Iterating bit-by-bit requires an inner loop inside the outer 64-iteration loop, creating nested branches and more pipeline stalls. CLZ is a single instruction that directly yields the position of the next 1-bit, eliminating the inner loop. Even though bit manipulation (clearing the processed bit) still occurs, the branch overhead is lower. Additionally, the compiler can better optimize a fixed-64-iteration outer loop than a dynamic inner loop.Misconception: that instruction count is the only metric. CLZ reduces branch divergence and loop complexity, which matters more on GPUs than raw instruction count.
Problem 4 · hard
Coruscant's bitmap decoder adds 0.006–0.018% area to the GPU. DSTC and RM-STC add 96× and 20× more area. Where does the difference come from?
Show answer
DSTC and RM-STC support dual-sparse operands (both A and B can be sparse), which requires scatter-gather logic and large intermediate buffers to accumulate partial results across a wide output region. These buffers dominate their area (87% in DSTC, 60% in RM-STC). Coruscant targets SpMM (only A is sparse), so it leverages the tensor core's existing per-lane accumulation — no scatter-gather, no extra buffers. The bitmap decoder is pure combinational logic (adders + muxes), much smaller than SRAM buffers.Misconception: that all sparse tensor cores are similar. Dual-sparse designs pay a large area cost for generality. Coruscant sacrifices generality (SpMM only) to minimize area.
Problem 5 · medium
Why does column-wise tiling avoid bank conflicts, but row-wise tiling causes them?
Show answer
Shared memory has 32 banks, each 4 bytes wide. Column-wise tiling assigns each thread to specific columns, so thread 0 writes to addresses 0, 64, 128, … (all in bank 0); thread 1 writes to 1, 65, 129, … (all in bank 1); etc. Threads touch different banks, no conflict. Row-wise tiling assigns threads to rows, so thread 0 writes to addresses 0, 1, 2, 3 (banks 0, 1, 2, 3), and thread 1 writes to addresses 64, 65, 66, 67 (also banks 0, 1, 2, 3) — simultaneous writes to the same banks serialize, causing conflicts.Misconception: that tiling direction is arbitrary. It's chosen to match the memory layout so threads naturally hit different banks in round-robin.
Problem 6 · hard
Coruscant's format uses a 64-bit bitmap per 64 elements. At what sparsity does this break even with storing indices, and why doesn't the paper target higher sparsity?
Show answer
Break-even when bitmap overhead = index overhead: 8 bytes (bitmap) vs N_nz × 2 bytes (indices). Solve: 8 = N_nz × 2 → N_nz = 4. If a tile has ≤4 non-zeros, indices are smaller. For 64 elements, 4 non-zeros = 94% sparsity. Below that, the bitmap wins. The paper doesn't target >70% because (a) model accuracy degrades exponentially (Figure 2), and (b) at very high sparsity, the number of non-zeros is so small that even indices compress well, so the bitmap's advantage shrinks. The sweet spot is 30–70%, where bitmaps dominate and models remain usable.Misconception: that bitmaps are always better. They're optimal at moderate sparsity; at extreme sparsity, indices can be competitive (though models are usually too degraded to care).
Problem 7 · easy
Why does Coruscant's prefill phase run slower than cuBLAS, and why does this not doom the overall speedup?
Show answer
Prefill has large N (batch × sequence length), making it compute-bound. Coruscant's decompression overhead and lower occupancy (due to register pressure) slow it down vs cuBLAS's higher-occupancy pipeline. However, prefill is a one-time cost: it runs once at the start. Decode runs hundreds of times (one per generated token), and Coruscant accelerates decode significantly. The decode speedup amortizes the prefill penalty. Figure 25 shows that in short-generation tasks (summarization), the prefill penalty dominates, but in long-generation tasks (reasoning), the decode gains dominate.Misconception: that a kernel must be faster on all workloads. Coruscant is optimized for decode (small N); prefill slowdown is an acceptable trade-off because decode is the bottleneck in token generation.
Problem 8 · medium
Coruscant compresses a 70% sparse OPT-66B GeMM (Figure 23). How does this affect CPU/SSD offloading latency, and what are the actual transfer times?
Show answer
Figure 23 shows 70% sparsity offloaded GeMM. SSD transfer: dense ~70 ms → Coruscant ~37.9 ms (the paper's reported value for SSD transfer time with compression). CPU transfer: dense ~16 ms → Coruscant ~7.8 ms. Compute time is much smaller (~0.27 ms dense, ~0.14 ms Coruscant). Total SSD-offloaded: ~70 ms (dense) → ~38 ms (Coruscant); CPU-offloaded: ~16 ms → ~8 ms. The ~37.9 ms is the SSD transfer time for the compressed weights, not the compute time. Compression reduces PCIe/SSD bandwidth pressure proportional to the ~44% compression ratio at 70% sparsity (Figure 3). This is critical for multi-GPU setups or weight swapping where transfer latency dominates.Misconception: that compression only helps GPU-internal memory. It helps any data transfer — global-to-SM, host-to-GPU, SSD-to-GPU, or GPU-to-GPU. Also, the ~37.9 ms is transfer time, not total latency.
Quick answers

Frequently asked questions

Tap any to expand.

Q1What exactly is "unstructured" sparsity?▾
Sparsity with no fixed pattern — the zero positions can be anywhere in the matrix, chosen freely by the pruning algorithm based on weight importance. Contrast: structured (e.g. prune entire channels), semi-structured (2:4: two zeros in every four elements). Unstructured preserves accuracy best but is hardest to exploit on hardware.
Q2Why 30–70% sparsity — why not higher?▾
Figure 2 shows perplexity spiking beyond 70% across multiple pruning methods (Wanda, SparseGPT, DSnoT). At 80–90%, models become unusable (perplexity > 100 on some benchmarks). Below 30%, compression gains are too small to justify overhead. 30–70% is the empirical sweet spot where models retain quality and compression is meaningful.
Q3How does Coruscant compare to NVIDIA's 2:4 sparse tensor core?▾
At 50% sparsity, Coruscant kernel is slightly slower than cuSPARSELt (2:4 kernel) because it still has decompression overhead. With the hardware extension, Coruscant-STC nearly matches 2:4 at 50% and surpasses it at 70% (where 2:4 is stuck at 50%). Crucially, Coruscant supports arbitrary patterns, not just 2:4, so it enables pruning methods to reach optimal accuracy at any sparsity 30–70%, while 2:4 is locked to exactly 50% with a rigid micro-pattern (Table 1, Figure 22).
Q4Why not just use Flash-LLM's Tiled-CSL format?▾
Tiled-CSL stores per-tile indices for each non-zero. At 50% sparsity, this exceeds 100% of the dense matrix size (Figure 3). Flash-LLM only achieves compression at 70%+ sparsity, so it can't help at 30–50%. Coruscant's bitmap stays under 60% across the entire 30–70% range, enabling speedups at lower sparsity where Flash-LLM is slower than cuBLAS (Figure 16).
Q5How much does the Sparse Tensor Core cost in area and power?▾
Cumulative GPU-level added area: 0.51–1.44 mm² (0.006–0.018% of GPU die), 26–75 mW power (Table 2). This is 96% less area than DSTC, 95% less than RM-STC (Table 3), because Coruscant reuses the tensor core's accumulation path instead of adding large buffers for scatter-gather. It's a minimal extension.
Q6Does Coruscant skip computation like 2:4 sparse tensor core?▾
No. The bitmap decoder selects non-zeros from registers, but the tensor core still performs 4 MACs per thread per cycle (padding unused slots with zeros). There's no computation skipping because Coruscant targets memory-bound SpMM, where reducing data transfer is more important than reducing MACs. Computation skipping would require scatter-gather logic (like DSTC/RM-STC) with high area cost; Coruscant avoids this trade-off.
Q7Why column-wise tiling instead of row-wise?▾
To avoid shared memory bank conflicts (Figure 9). Column-wise ensures each thread writes to a different bank in round-robin. Row-wise causes multiple threads to hit the same bank simultaneously, serializing writes and stalling warps. This is critical during decompression, which writes ~64 values per tile into shared memory.
Q8Can Coruscant handle sparsity outside 30–70%?▾
Technically yes, but it's not optimal. Below 30%, compression is too small to amortize decompression overhead (Figure 24a: Coruscant kernel slower than cuBLAS at 10–20% sparsity). Above 70%, Flash-LLM's simpler index-based decompression is faster despite lower compression (Figure 16), and model accuracy degrades anyway (Figure 2). Coruscant is designed for the 30–70% range where both compression and accuracy matter.
Q9Why does Coruscant lag Flash-LLM at 70% sparsity in the kernel-only comparison?▾
Because decompression overhead grows with the number of non-zeros to process — at very high sparsity there are fewer non-zeros, but Coruscant's bitmap-based CLZ iteration is more complex than Flash-LLM's direct index lookup. While the memory-transfer advantage of compression shrinks at very high sparsity (fewer non-zeros means both formats compress similarly), the bit-level position derivation adds overhead. However, with the hardware extension, Coruscant-STC eliminates decompression overhead and outperforms Flash-LLM across all sparsity (Figure 19c).
Q10Does Coruscant work with quantization (e.g. 4-bit weights)?▾
In principle, yes — quantization reduces each weight's size (float16 → int4), and Coruscant's bitmap compresses the sparsity pattern. However, the paper does not evaluate stacking Coruscant with int4 quantization. The compression ratio would not simply be 56% × 25% = 14% because the bitmap and pointer overhead (which is fixed in bytes) becomes much larger relative to 4-bit values, reducing the compound benefit. The format is agnostic to data type, but the actual compression gain would require empirical measurement.
Q11What extensions or future directions does the paper discuss?▾
The paper's conclusion mentions exploring integration with other memory optimization techniques (quantization, KV cache compression), multi-GPU weight distribution, and adapting the kernel for different GPU generations. The paper does not explicitly present non-SpMM ops, dynamic sparsity, larger tile sizes, or Ada/Blackwell optimization as planned future work in a dedicated section.
Q12How does Coruscant relate to your LtRAM/CXL tiering work?▾
Coruscant reduces weight memory footprint, freeing DRAM/NVM for KV cache and activations. In a tiered system, smaller weight footprint means less pressure on DRAM tier, allowing more KV cache to stay hot. Also, Coruscant's compression reduces PCIe/CXL traffic when offloading weights from SSD or remote memory (Figure 23), which directly benefits tiered architectures where inter-tier bandwidth is a bottleneck.

Glossary & further reading

Every term in one place, then a path to go deeper.

Core terms
CoruscantThe paper's system: a bitmap-based sparse format + GPU kernel + sparse tensor core extension for 30–70% unstructured sparsity in LLMs.
Bitmap-based sparse formatEach 64-element tile: a 64-bit bitmap (1 = present, 0 = absent) + compressed non-zero values + tile offset pointer. No per-element indices.
Unstructured sparsityArbitrary zero positions — no fixed pattern. Preserves accuracy better than structured/semi-structured, but harder to compress and accelerate.
SpMMSparse matrix × dense matrix multiplication — the core decode-phase operation (weight projections) in LLM inference.
Memory-boundPerformance limited by memory bandwidth, not compute throughput. Typical of LLM decode with small batch sizes (N < 32).
Bitmap decoderCombinational logic inside the tensor core that extracts non-zero positions from the bitmap and selects values from registers, avoiding decompression to shared memory.
CLZ (count leading zeros)Single instruction that finds the position of the leftmost 1-bit in a bitmap. Used to iterate non-zeros without inner loops.
Column-wise tilingPartitioning the matrix into vertical slices (columns) to avoid shared memory bank conflicts during decompression.
Supporting terms
CSR / COO / Tiled-CSLIndex-based sparse formats (Compressed Sparse Row / Coordinate List / Flash-LLM's tiled variant). Store per-non-zero indices; exceed dense size at 50% sparsity.
2:4 sparsitySemi-structured format: exactly 2 zeros in every 4 consecutive elements (50% sparsity, fixed pattern). Supported by NVIDIA Ampere+ sparse tensor cores.
Tensor CoreSpecialized matrix-multiply unit in NVIDIA GPUs. Executes MMA (matrix-multiply-accumulate) instructions on small tiles (e.g. 16×16).
MMAMatrix-multiply-accumulate instruction. Coruscant reformulates as M=32, N=1, K=64 to align with the 64-element tile size.
Shared memoryFast on-chip memory inside each SM, divided into 32 banks. Coruscant decompresses tiles here (software kernel) or skips this step (hardware).
Bank conflictWhen multiple threads access the same shared memory bank simultaneously, accesses serialize. Avoided by column-wise tiling.
Prefill / DecodePrefill: process input prompt (large N). Decode: generate tokens one-by-one (small N, memory-bound). Decode dominates LLM inference latency.
Roofline modelPerformance plot with compute ceiling (FLOPs) and memory wall (bandwidth). Operational intensity = FLOPs/byte determines which limit applies.
OccupancyNumber of warps active per SM. Higher occupancy → more instructions in flight → better pipeline utilization. Coruscant's register pressure limits occupancy to 1 warp/SM; cuBLAS achieves 2.
DSTC / RM-STCPrior sparse tensor core designs (Dual-Side Sparse Tensor Core / Row-Merge Sparse Tensor Core) supporting unstructured sparsity in both operands. High area overhead (buffers for scatter-gather).
Wanda / SparseGPT / DSnoTLLM pruning methods targeting unstructured sparsity. Restricted to 2:4 format for hardware efficiency, despite accuracy loss.
PerplexityLanguage model quality metric — lower is better. Spikes beyond 70% sparsity (Figure 2).
