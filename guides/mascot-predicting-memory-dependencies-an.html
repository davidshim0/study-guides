Predicting when a load doesn't depend — so it can run early

A load can't safely issue until it knows no older store will feed it. Guess right and you unlock parallelism; guess wrong and you squash. This is the plain-English story of MASCOT, one TAGE-like table that learns both dependence and non-dependence.

📄 "MASCOT" · Mose, Kim, Ros, Jones & Mullins 🎓 Some OoO / TAGE basics assumed — the primers cover them ⏱ ~30 min read

Before you begin

Ten ideas the paper assumes you know

Each card clears up one concept MASCOT builds on. They're collapsed — tap any open. Read in order the first time, or jump to whatever looks unfamiliar. If you already live in the memory-ordering / speculation world, most of this is review — skim for the MDP-vs-SMB framing.

MDP / memory dependence SMB / bypassing TAGE / MASCOT machinery mispredict / squash (the cost)
1Out-of-order loads and the store queue▾

An out-of-order superscalar core executes instructions as soon as their inputs are ready, not in program order. Loads are the prize: getting a load's value early lets everything downstream proceed. But a load might read an address that an older store (still in flight, address not yet computed) is about to write.

The store queue (SQ) holds in-flight stores; a load that issues early must eventually check it for an aliasing older store — a store-to-load forward if found. The problem: at the moment you'd like to issue the load, some older stores may not have addresses yet.

Now you're ready: MASCOT predicts, before addresses are known, whether an older store will alias — so the load can either wait, or run and forward.
2Memory-dependence prediction (MDP)▾

MDP answers one yes/no question per load: will this load alias some older in-flight store? If the predictor says "no dependence," the load issues as soon as its own address is ready, rather than waiting for every older store's address. If "yes," the load is delayed until the offending store resolves.

The classic scheme is Store Sets (Chrysos & Emer, 1998): each load remembers the set of store PCs it has aliased before, and stalls if any are in flight.

Now you're ready: MDP is a gate — issue now, or wait. It changes when a load runs, never where its value comes from.
3Speculative memory bypassing (SMB)▾

SMB goes further. It predicts the exact older store a load depends on and forwards that store's value straight to the load — before either address is computed. This short-circuits the address-computation dependency chain entirely, so the load's value is available much earlier.

For SMB the meaning of "dependence" is stricter: the store must supply the load's full value (same size, aligned, no offset). A partial overlap is a memory dependence but not a bypass opportunity (paper, Fig. 1).

Now you're ready: SMB is aggressive value forwarding. MASCOT does both MDP and SMB from one predictor.
4Squashes — the cost of a wrong guess▾

Speculation is only worthwhile if wrong guesses are rare and cheap. When a speculatively-issued load turns out to have violated a real dependence (or forwarded the wrong value), the load and every instruction that consumed its value must be squashed and re-executed. That's expensive — pipeline flush, re-fetch, lost work.

So the whole game is a lopsided bet: you want the rare mispredictions to be the cheap kind.

Now you're ready: MASCOT's contribution is driving both kinds of misprediction down at once, which prior predictors couldn't do.
5False positives vs false negatives (and why they're not symmetric)▾

A false negative = predicting "no dependence" when there is one (a missed dependence). A false positive = predicting a dependence that doesn't exist (a false dependence).

For MDP, a false negative is costly — it issues the load wrongly and forces a squash — while a false positive only delays the load. So MDP predictors are skewed toward low false negatives (over-predict dependence). For SMB it flips: a false positive means you forwarded a bogus value and must squash, so SMB wants low false positives.

Now you're ready: the two mechanisms want opposite skews — which is exactly why prior work built separate predictors, and why MASCOT unifying them is hard.
6The TAGE predictor (borrowed from branches)▾

TAGE (TAgged GEometric-history) is the dominant branch-direction predictor. It keeps several tables indexed by the PC hashed with geometrically increasing lengths of global branch history. On a lookup, all tables are probed in parallel and the longest-history matching entry wins. On a misprediction it allocates a new entry in a longer-history table — capturing the extra context that would have gotten it right.

TAGE has been repurposed for MDP (MDP-TAGE, PHAST) and for SMB (Perais et al.'s IDist) — separately.

Now you're ready: MASCOT is "TAGE, but for load-store dependence," with a smarter allocation rule.
7Global branch history as context▾

The taken/not-taken outcomes of recent branches form a global history vector. Hashing the load PC together with more history bits distinguishes different dynamic instances of the same static load — the same instruction reached via different control-flow paths.

Why it matters here: whether a load aliases an older store often depends on which path the program took. One control path stores to the address, another doesn't. History is the context that separates those cases.

Now you're ready: MASCOT's whole insight is that non-dependence is also context-sensitive, so it deserves history-indexed entries too.
8Saturating confidence counters▾

Predictors gate risky actions behind small saturating counters: increment on a correct outcome, decrement (or reset) on a wrong one, clamp at the ends. Only when a counter is saturated do you trust it enough to act.

MASCOT gives each entry two: a 3-bit usefulness counter (confidence for MDP) and a 2-bit bypass counter (confidence for SMB). SMB only fires when both are saturated — aggressive about dependence, conservative about bypassing.

Now you're ready: two counters per entry are how one table serves two mechanisms with opposite risk profiles.
9What "store distance" means▾

Instead of storing a full store PC, TAGE-style MDP encodes the dependence as a distance: how many stores back (in program order) the aliasing store sits. MASCOT uses a 7-bit distance field. A distance of 0 is the special code for "non-dependent — depends on no prior store." Any non-zero value points at a specific slot in the store queue.

This is compact and lets the predictor say "wait for exactly this store," which is what SMB needs to forward a value.

Now you're ready: distance = 0 encoding non-dependence is the hook MASCOT hangs its whole idea on.
10How we score it: IPC, MPKI, and area▾

Three scorecards. IPC = Instructions Per Cycle (higher = faster; the bottom line). MPKI-style misprediction counts measure how often the predictor is wrong. And area — the KiB of SRAM the predictor's tables consume — because a smarter predictor that's huge is a bad trade.

MASCOT reports gains on all three: more IPC, far fewer mispredictions, and (in its optimised form) less area than the prior state of the art.

Now you're ready: when you see "+1.9% IPC" or "10.1 KiB," those are the three axes the paper is trading on.
The whole story in one place

MASCOT in a nutshell

The problem, the one-line insight that fixes it, and how the pieces connect.

Two speculation opportunities live in the load-store pipeline. MDP asks "can this load skip waiting for older stores?" and SMB asks "can I forward an older store's value straight into this load?" They pull in opposite directions: MDP fears missed dependencies (false negatives), SMB fears false dependencies (false positives). Because of that tension, prior work built two separate predictors — wasting area and leaving IPC on the table.

The paper's fix, MASCOT (Memory-dependence And Short-Circuit Optimising TAGE), is a single TAGE-like table that reaches state-of-the-art accuracy on both error types at once. The key innovation is its allocation policy. A normal TAGE-MDP predictor only ever records dependencies; when it falsely predicts a dependence for a load that actually depends on nothing, it has nowhere obvious to allocate and just slowly decrements a confidence counter. MASCOT instead allocates a new entry, in a longer-history table, that explicitly encodes "non-dependent in this context" (distance = 0). Now the next time that control path recurs, the predictor confidently says "no dependence" — and the load runs early.

Learning context-sensitive non-dependence drastically cuts false dependencies, which frees the predictor to be aggressive about true ones (few false negatives) and accurate enough to drive SMB (few false positives). Each entry carries a 7-bit distance, a 16-bit tag, a 3-bit usefulness counter (MDP confidence), and a 2-bit bypass counter (SMB confidence) — 28 bits total. SMB fires only when both counters saturate.

The payoff: on SPEC CPU2017, MASCOT reduces mispredictions by 70% on average vs the prior best MDP predictor PHAST. As an MDP-only predictor it edges PHAST by +0.4% IPC at the same size; doing MDP and SMB it gains +1.9% IPC on average, with peaks of ~26%. A size-tuned variant, MASCOT-OPT, keeps essentially all of that while shrinking to 10.1 KiB — versus 14.5 KiB for PHAST.

🍯 Key takeaways

  • MDP and SMB have opposite error costs (MDP dreads false negatives; SMB dreads false positives), so prior work split them into two predictors.
  • MASCOT is one TAGE-like table that serves both, at state-of-the-art accuracy on both error types.
  • The trick: on a false dependence, allocate a longer-history entry encoding non-dependence (distance = 0) — learning context, not just forgetting.
  • Two counters per entry: 3-bit usefulness (MDP) + 2-bit bypass (SMB); bypass fires only when both saturate — aggressive on dependence, conservative on bypassing.
  • 70% fewer mispredictions vs PHAST; false dependencies down 91%, speculative errors down 39%.
  • IPC: +0.4% (MDP-only) or +1.9% (MDP+SMB) over PHAST; MASCOT-OPT delivers ~this at just 10.1 KiB.
  • Off the critical path in the frontend; predictions made at decode, used later; entries updated at commit.
Seven everyday pictures

How to think about MASCOT

Each picture stands alone — read them in any order. Every card ends with an honest note about where the picture stops being true.

The core problem · Ordering hazard

Passing a note down a line

You're the 5th person in a phone-tree, waiting to copy a number from an earlier person. If you knew nobody ahead of you is writing your number, you'd stop waiting and act now. If you knew exactly who holds it, you could grab it the instant they wrote it. MDP is "nobody has your number, go"; SMB is "person #2 has it, take it directly." Both need a confident guess before the line has finished sorting itself out.

Where it misleads: a phone-tree is ordered by position; real load-store aliasing depends on addresses, which aren't known yet — that's the whole reason you must predict rather than check.
The gap MASCOT fills · Non-dependence

An address book with no "moved away" page

Imagine a contacts app that can only record where someone lives — it has no way to note "this person no longer lives anywhere on your route." So every time you pass their old street you slow down expecting them. Old TAGE-MDP predictors are like this: they store dependencies but have no clean slot for "no dependence here." MASCOT adds the missing page — a context-tagged "nobody's home" entry — so it stops braking for ghosts.

Where it breaks: a contacts app records facts you enter; MASCOT only allocates a non-dependence entry after it gets burned by a false dependence — it learns from squashes, not from foreknowledge.
The structure · Geometric history

Nested captions on a photo

Label a face "a person," then "a person at a wedding," then "the groom's uncle at the 2019 wedding." Each longer caption disambiguates cases the short one blurs together. TAGE-style tables do this with global branch history: table 1 keys on just the load PC, table 8 on the PC plus 128 branches of context. When a short caption keeps being wrong, MASCOT writes a longer one in the next table.

Where it breaks: captions are free-form; MASCOT's "captions" are fixed-length hashes into 4-way sets, so two very different paths can still collide onto one entry — hence tags and usefulness counters to arbitrate.
Two counters · Asymmetric risk

A gas pedal and a nitro button

Driving fast (MDP) you keep your foot down whenever you're fairly sure the road is clear — a small confidence bar is enough, because braking late only costs a moment. But the nitro button (SMB) is irreversible-ish: hit it wrongly and you spin out (squash). So you only press nitro when both your confidence gauges are pegged. MASCOT's 3-bit usefulness is the pedal; the 2-bit bypass counter is the nitro arming switch.

Where it breaks: a real nitro is one gauge; MASCOT gates SMB on two saturated counters (usefulness==7 AND bypass==2), because it must be confident both that there's a dependence and that the full value can be forwarded.
The allocation policy · Learn vs forget

A student who writes the rule down

Two students both get a question wrong. One just lowers their confidence and hopes to guess better next time — that's the old approach, "decrement the counter" (which the paper computes could take ~1,625 tries to flip a big counter). The other writes down the specific exception with the context that caused it, and nails it next time. MASCOT is the second student: on a false dependence it allocates a context-tagged non-dependence entry instead of waiting to forget.

Where it breaks: a student has unlimited notebook pages; MASCOT's tables are finite, so aggressive allocation is paired with frequent usefulness-decrements and evictions to keep stale notes from piling up.
Space efficiency · Try-again allocation

Circling the parking lot

You want a spot in row N₁, but it's full of cars that are still "in use." Rather than double-park, you drive to N₂, then N₃, taking the first free spot — and on the way you dock a demerit from everyone you passed who was hogging a space. MASCOT allocates to the next-longer-history table; if that set is full of useful entries it tries the one after, decrementing the passed-over entries' usefulness so stale ones clear out. This is why periodic global decay barely helped — the try-again passes already do it.

Where it breaks: a driver eventually gives up; MASCOT's try-again cascade is bounded by the number of tables (8), and unlike a lot, its "spots" are 4-way sets that tolerate a few collisions before overflowing.
Right-sizing · F1-guided tuning

Packing a suitcase by usefulness

Before a trip you score each item by how often you'd actually reach for it, then keep the high-scorers and shrink or drop the rest. MASCOT ranks every table entry by its F1 score (accuracy balanced against how often it's used), and finds table 1 deserves to be bigger while tables 5–8 can be halved or quartered. Re-sizing this way yields MASCOT-OPT: 16% smaller tables, then trimmed tags → 10.1 KiB for a mere ~0.1% IPC loss.

Where it breaks: a suitcase is packed once; MASCOT's phases shift, so F1 is averaged over many intervals — and the try-again policy cushions a too-small table by spilling into longer-history ones.
The full explanation

How it actually works

Section by section, with the paper's key diagrams rebuilt as visuals you can read at a glance.

1 · The problem — two mechanisms, opposite fears

MDP lets a load skip the wait for older stores' addresses; SMB forwards a specific store's value straight into the load. Both speculate on dependence, but they weight errors oppositely. For MDP a false negative (missed dependence) forces a squash while a false positive only delays — so MDP predictors historically skew to over-predict dependence. For SMB a false positive forwards a wrong value and squashes, while a false negative barely hurts (a good MDP still stalls the load) — so SMB skews the other way. Because the skews conflict, prior work built separate predictors, paying twice in area and missing joint opportunities.

🧭
Where this lands for you: if you think in terms of memory-ordering hazards and the load/store queues, MDP is a speculative reordering gate and SMB is speculative value forwarding — both are load-speculation you must be able to recover from. MASCOT is the predictor that makes both recoveries rare.

2 · Why plain TAGE can't do MDP to its best

In branch prediction, TAGE always has an obvious move after a misprediction: allocate a longer-history entry with the correct direction. In MDP there's a case with no obvious move — a load predicted dependent that actually depends on nothing. You can't allocate "the right dependence," because there is none, and the longer history that would have revealed the non-dependence isn't recorded at the moment of misprediction. So prior TAGE-MDP predictors only ever store dependencies and handle this case by decrementing usefulness — which is slow and weak:

Failure modeWhat goes wrongPaper's number
Prediction persistenceload is always either right-as-dependent or fully non-dependent, so there's no natural allocation trigger—
Slow confidence adjustmenta 3-bit counter started at max takes forever to reach 0~1,625 predictions
Eviction riskonce the counter hits 0 the entry is evictable, so the pattern is lost before it can grow—
⚠
The asymmetry that motivates it all. Reducing false dependencies is what lets a predictor be aggressive about true ones (low false negatives) and trustworthy for bypassing (low false positives). Prior predictors couldn't get both — MASCOT can.

3 · The fix — allocate context-sensitive non-dependence

MASCOT's key move: on a false dependence, allocate a new entry in the next-longer-history table whose distance field is 0 — an explicit "non-dependent in this context" record. Because the index folds in the recent branch history (including the latest not-taken bit), the entry keys precisely on the control path that made the load non-dependent this time. Next time that path recurs, MASCOT hits this longer-history entry and confidently predicts "no dependence," so the load issues early. Meanwhile the shorter-history dependence entry still fires on the other path.

4 · The entry, and how the two counters gate the two mechanisms

Each entry is 28 bits: a 16-bit tag, a 7-bit distance, a 3-bit usefulness counter (MDP confidence), and a 2-bit bypass counter (SMB confidence). Distance = 0 means non-dependent; any non-zero value is an offset into the store queue (distance 1 = the nearest prior store). A memory-dependence prediction fires whenever distance ≠ 0, regardless of usefulness — aggressive. SMB fires only when usefulness is saturated (7) and bypass is saturated (2) — conservative. Updates happen at commit:

# on commit, update the predicting entry
if memory dependence predicted correctly:  usefulness += 1
if bypass predicted correctly:             bypass    += 1
if memory dependence was wrong:            usefulness -= 1
if bypass was wrong:                       bypass     = 0    # hard reset

# allocation confidences (paper §IV-C)
new dependent entry     : usefulness = 6      # start near-confident
new non-dependent entry : usefulness = 2      # start cautious
new bypass-capable entry: bypass     = 1      # else 0; needs 2 to fire
# only entries with usefulness == 0 may be evicted
🍯
Why the reset-on-wrong for bypass but decrement for usefulness: a wrong bypass always squashes, so its confidence must collapse instantly; a wrong MDP is cheaper, so its confidence merely erodes. Same table, two risk profiles.

5 · The structure and where it sits in the pipeline

MASCOT has 8 tables, each 4-way associative with 512 entries, indexed by folding the load PC with geometrically growing history lengths of [0, 2, 4, 8, 16, 32, 64, 128] branches (1 bit per conditional branch; indirect targets folded to 5 bits). All tables are probed in parallel at decode; the longest-history match wins, and if nothing matches the base predictor defaults to "no dependence." The prediction is ready by the time the load leaves the frontend, so MASCOT is off the critical path. Dependencies are resolved as usual by snooping the load/store queues; at commit the outcome flows back to update entries. Default size (tables only): 14 KiB.

6 · Try-again allocation keeps the tables lean

On a misprediction from a table Nᵢ₋₁, MASCOT tries to allocate in the next table Nᵢ. If that set is full of still-useful entries (all usefulness > 0), it tries Nᵢ₊₁, then Nᵢ₊₂, and so on. Crucially, whenever an allocation into Nᵢ₊₁ was attempted, all four entries in Nᵢ have their usefulness decremented (but not the deeper tables'). This "try-again" cascade both places the new entry and continuously erodes stale entries — which is why the paper found periodic global decay didn't help: the cascade already does the decay.

def allocate(from_table):
    i = from_table + 1
    while i < NUM_TABLES:
        if a slot in set[i] has usefulness == 0:
            place new entry there; return
        decrement usefulness of all 4 entries in set[i-1]   # passed-over erosion
        i += 1
    # cascade bounded by 8 tables → cheap, self-cleaning

7 · Shrinking it — MASCOT-OPT

An F1-guided study (score every entry by accuracy-balanced-against-usage, averaged over 1M-cycle intervals) shows table 1 is the most valuable and could be larger, while tables 5–8 hold low-value entries and can shrink. MASCOT-OPT re-sizes the tables to [1024, 512, 512, 512, 256, 256, 256, 128] with tags [15, 16, 16, 16, 17, 17, 17, 18] (sized to keep conflict probability constant), a 16% size cut for only −0.09% IPC. Trimming tags a further 4 bits reaches 10.1 KiB (a 27.7% area saving) at just −0.13% IPC, despite +17.4% mispredictions — the try-again policy absorbs the loss by spilling into longer-history tables.

💡
Why the try-again policy makes shrinking safe: if one table is undersized, its overflow simply allocates into a longer-history table — so MASCOT keeps most of the benefit in good phases and pays little in bad ones. PHAST, which pins each load-store pair to one specific table, would pay a bigger penalty for the same trim.
Follow the numbers, step by step

Worked examples

Five traces, each carried all the way through — the slow-forget math, the gate logic, the misprediction breakdown, the size budget, and where the IPC actually comes from.

① Why "just decrement the counter" is hopeless

A load is dependent when the last branch was taken (70%) and non-dependent when not-taken (30%). A dependence entry sits in table 1 with a 3-bit usefulness counter at max. The old fix: wait for that counter to hit 0 so the entry can be replaced. Why is that the wrong tool?

  1. The entry is right 70% of the time (taken → dependent), wrong 30% (not-taken → false dependence). Each correct use increments, each wrong use decrements.
  2. Net drift toward 0 is tiny: +0.70 −0.30 = +0.40 per prediction on average pushes the counter up, not down. It rarely reaches 0 at all under this mix.
  3. The paper's Markov-chain analysis puts the expected number of predictions to reach confidence 0 at ~1,625 — and even then the entry only becomes evictable, not corrected.
  4. MASCOT's alternative: the first false dependence immediately allocates a distance-0 entry in a longer-history table keyed on the not-taken path. Learned in one shot, not 1,625 tries.
🍯
Insight: a mixed-behavior entry can be individually "accurate enough" yet globally harmful. Forgetting is far too slow to fix it; explicit context-learning fixes it on the first squash.

② Walking an entry through the gate

A load repeatedly aliases the same prior store, and every time the full value could be forwarded. Trace what MASCOT is allowed to do as confidence builds (usefulness starts at 6 for a fresh dependent entry; bypass starts at 1 since it's a bypass-capable conflict).

  1. Start (u=6, b=1): distance ≠ 0, so MDP fires — the load waits for the predicted store. SMB is locked (needs u=7 AND b=2).
  2. One correct MDP + one correct bypass-would-have-worked: u→7, b→2. Both saturated.
  3. Now SMB is unlocked: next time, MASCOT forwards the store's value directly — the load's value is ready without waiting for addresses.
  4. A wrong bypass later: bypass resets to 0 immediately (b=0), SMB re-locks, but MDP keeps firing (usefulness only decremented). Aggressive on dependence, conservative on bypass.
💡
Use the interactive on the "How It Works" tab: set u=7, b=2 to see SMB light up, then knock bypass to 0 and watch SMB re-lock while MDP stays on.

③ Reading the misprediction breakdown vs PHAST

The paper reports MASCOT cuts total mispredictions by 70% vs PHAST, decomposed into two error types. Given "false dependencies −91%" and "speculative errors −39%," check that the story is consistent and see which error dominated.

  1. Two error buckets: false dependencies (predicted a dep that wasn't there — MDP's false positive) and speculative errors (a bad speculation that squashes — includes SMB false positives / missed deps).
  2. False dependencies fall 91% — that's the direct payoff of allocating non-dependence entries; it's the big lever.
  3. Speculative errors fall 39% — a smaller but real drop, since fewer false deps also means more accurate true-dependence prediction.
  4. Because false dependencies were the larger bucket, a 91% cut there drives the blended ~70% total reduction (vs NoSQ the total cut is 98%).
  5. Ablation confirms causation: a TAGE-like predictor without non-dependence entries has >12× more false dependencies than MASCOT.
🍯
Insight: the headline "70% fewer mispredictions" is almost entirely the false-dependence collapse — precisely the error the non-dependence entries were designed to kill.

④ Tallying the entry and the area budget

Is one unified predictor actually smaller than two specialized ones? Add up MASCOT's storage and compare.

  1. One entry = 16 tag + 7 distance + 3 usefulness + 2 bypass = 28 bits.
  2. Default MASCOT: 8 tables × 512 entries × 28 bits = 114,688 bits ≈ 14,336 bytes ≈ 14 KiB (the paper's stated tables-only size).
  3. Compare the prior art: PHAST needs 14.5 KiB; NoSQ+StoreSets combined ~18.5–19 KiB for the same job across two structures.
  4. MASCOT-OPT re-sizes tables (16% smaller) and trims 4 tag bits → 10.1 KiB, a 27.7% cut from default, for −0.13% IPC.
  5. Net: one table doing both MDP and SMB is ~30% smaller than the prior single-purpose MDP predictor — while being more accurate.
💡
That's the unification dividend: merging two mechanisms into one high-accuracy table removes the redundant second structure entirely, so "do more" and "use less" happen together.

⑤ Where the perlbench2 speedup actually comes from

MASCOT's average IPC gain is +1.9%, but perlbench2 sees ~26% over PHAST. The paper measured why. Follow the mechanism.

  1. Instrument the issue stage: for instructions depending on one-or-more loads, measure average cycles spent waiting in issue for dependencies to resolve.
  2. Without bypassing: that wait averages 38.7 cycles in perlbench2 — its dependent instructions sit idle a long time.
  3. Enable SMB: values arrive early via bypass; the wait drops to 15.7 cycles — a 60% reduction in issue-stage stall.
  4. Contrast lbm: also a high-bypass benchmark, but enabling bypass cuts its wait only 1.9% — its loads weren't the bottleneck.
  5. Conclusion: SMB pays off precisely where load-value latency gates downstream work; perlbench2 is that case, lbm is not — hence "highly application specific."
🍯
Insight: the same accurate predictor yields 26% on one benchmark and ~0% on another. The gain isn't from the predictor alone — it's from unlocking loads that were on the critical path of dependent instructions.

Practice problems

Try each before opening the answer. Every one ends with the common misconception it's checking for.

Problem 1 · easy
A memory-dependence predictor predicts "no dependence," the load issues early, and it turns out an older store did alias it. Is that a false positive or a false negative, and why is it the expensive one for MDP?
Show answer
It's a false negative — a missed dependence. It's expensive because the load already issued (and fed dependents) on a wrong assumption, so the load and everything downstream must be squashed and re-executed. A false positive, by contrast, only stalls a load that could have gone early — a delay, not a flush. That asymmetry is why MDP predictors skew toward over-predicting dependence.Misconception: that "false positive" always means the costly error. The costly error depends on the mechanism — for MDP it's the false negative; for SMB it flips to the false positive.
Problem 2 · medium
A load is dependent on the taken path and non-dependent on the not-taken path. A plain TAGE-MDP predictor keeps one dependence entry in table 1 and, on false dependencies, just decrements its usefulness. Why does this barely help, and what does MASCOT do instead?
Show answer
If the taken path is common (say 70%), the entry is correct most of the time, so it's incremented more often than decremented — its usefulness drifts up, not down, and the paper estimates ~1,625 predictions just to reach 0 (after which it's merely evictable, still not corrected). MASCOT instead allocates a distance-0 (non-dependence) entry in a longer-history table keyed on the not-taken path, so the very next time that path recurs it predicts "no dependence" correctly. Learning beats forgetting.Misconception: that lowering a confidence counter is equivalent to learning the exception. It only makes the entry replaceable eventually; it never records the context that distinguishes the two paths.
Problem 3 · medium
MASCOT gates SMB on usefulness == 7 AND bypass == 2, but fires MDP whenever distance ≠ 0 regardless of usefulness. Explain the design logic behind these two different thresholds.
Show answer
The thresholds match the error costs. A wrong MDP false positive only delays a load (cheap), so MASCOT can afford to predict a dependence even from a low-confidence entry — being aggressive keeps false negatives (the costly MDP error) low. A wrong SMB always squashes (it forwarded a bad value), so SMB must be conservative: it demands the MDP confidence be fully saturated and a separate bypass confidence be fully saturated before it forwards. Two saturated counters = "I'm sure there's a dependence AND sure the full value can be forwarded."Misconception: that one confidence counter could serve both. The two mechanisms have opposite risk profiles, so they need independently-gated confidences — that's why the entry carries two counters.
Problem 4 · hard
The paper ablates MASCOT by building a TAGE-like predictor that does not allocate non-dependence entries (it decrements on false deps, like prior work). It shows >12× more false dependencies. Trace the causal chain from "no non-dependence entries" to "worse SMB," not just worse MDP.
Show answer
Without non-dependence entries, false dependencies can only be handled by decrementing the confidence of the dependent entry. So repeated false deps drag that entry's usefulness down. But SMB requires usefulness to be saturated — so an entry that's constantly being decremented by false deps can never reach the confidence needed to bypass. The false dependencies therefore poison the very confidence SMB depends on. MASCOT's non-dependence entries divert those false deps into their own longer-history slots, leaving the true-dependence entry free to saturate — so SMB can actually fire. Hence the ablation is "worse as MDP, much worse as MDP+SMB."Misconception: that non-dependence entries only help the "no dependence" case. They also protect the dependence entries' confidence, which is what unlocks bypassing.
Problem 5 · medium
On a misprediction from table Nᵢ₋₁, MASCOT tries to allocate in Nᵢ; if the set is full of useful entries it tries Nᵢ₊₁, decrementing all four entries in Nᵢ as it passes. The paper says periodic global usefulness decay "did not meaningfully change performance." Why not?
Show answer
Because the try-again allocation already is a decay mechanism. Every failed allocation attempt decrements four entries in the passed-over table, and with 4-way sets and frequent mispredictions this happens often — stale entries get eroded to usefulness 0 (and become evictable) as a side effect of ordinary allocation traffic. Adding a separate periodic decay is redundant: the churn already clears out stale state. The authors attribute this to the combination of 4-way associativity and the aggressive allocation policy.Misconception: that TAGE-style predictors always need explicit periodic decay. They need some way to retire stale entries; MASCOT's allocation policy supplies it implicitly.
Problem 6 · medium
MASCOT-OPT trims tag bits and shrinks tables to reach 10.1 KiB. One variant increases mispredictions by 17.4% yet loses only 0.13% IPC. How can a 17% jump in mispredictions cost almost no performance?
Show answer
Two reasons. First, the absolute misprediction rate is already very low (MASCOT cut it ~70% below PHAST), so a 17.4% relative increase is a small absolute number of extra squashes. Second, the try-again allocation policy means an undersized/under-tagged table just spills its overflow into longer-history tables rather than failing outright — most of the useful predictions survive. So the extra mispredictions are few and mostly non-critical, and IPC barely moves. (Reducing tags also risks aliasing — the extra mispredictions are partly tag collisions — but they're cheap here.)Misconception: that misprediction count maps linearly to IPC loss. What matters is how many extra squashes land on the critical path — and when the baseline rate is tiny, a big relative bump is a small absolute one.
Problem 7 · hard
On future, wider cores (Golden Cove → Lion Cove), the paper reports the ceiling — a perfect MDP+SMB predictor over a perfect MDP-only one — rises from 2.1% to 2.8%, and MASCOT's realized gain rises from 1.0% to 1.3%. Why should SMB's value grow as the core gets wider, in terms of your memory-ordering / ILP intuition?
Show answer
A wider core has more ROB/IQ/LQ/SB entries and more execution ports, so it can keep more instructions in flight and expose more ILP — if their inputs are available. Load-value latency becomes the binding constraint more often: the machine has slots ready to execute dependents but is stalled waiting on load values. SMB attacks exactly that by delivering load values early (short-circuiting address computation), so the bigger the window of instructions waiting on those values, the more work SMB unblocks. MDP alone only removes a stall; SMB removes a latency on the value itself, which scales with window size. Hence the ceiling and MASCOT's realized gain both rise on Lion Cove.Misconception: that a predictor's benefit is fixed by its accuracy. Benefit is accuracy × opportunity, and wider cores create more opportunity for early load values to matter.
Problem 8 · easy
MASCOT makes its prediction at decode and updates entries at commit, and the paper stresses it is "off the critical path." Why does the timing of prediction-vs-use matter for whether the predictor slows the core?
Show answer
A predictor consulted in the time-critical issue/execute loop would add latency to every load and could lower clock frequency. MASCOT makes its prediction in the frontend at decode; because there are several pipeline stages before the load actually needs the answer (issue), the multi-table lookup has time to complete and its result is simply ready when needed. Updates at commit are likewise off the hot path. So the predictor buys accuracy without lengthening the critical path — a smart predictor that slowed hits would be a bad trade.Misconception: that adding any predictor structure inherently slows the pipeline. It only does if it sits on the critical path; placed in the frontend with slack before use, it's effectively free in latency terms.
Quick answers

Frequently asked questions

Twelve questions newcomers ask most. Tap any to expand.

Q1What's the one-sentence difference between MDP and SMB?▾
MDP decides when a load may issue (does an older store alias it, so should it wait?); SMB decides where the load's value comes from (forward a specific store's value directly, before addresses are known). MDP changes timing; SMB changes the data path.
Q2What does MASCOT stand for, and what is it based on?▾
Memory-dependence And Short-Circuit Optimising TAGE. It's a TAGE-like predictor (multiple tables indexed by the load PC hashed with geometrically-increasing global branch history), repurposed from branch prediction to load-store dependence, with a new allocation policy.
Q3What is the single key innovation?▾
On a false dependence, MASCOT allocates a new entry — in a longer-history table — that explicitly encodes non-dependence (distance = 0), keyed on the control-flow context that caused the mistake. Prior TAGE-MDP predictors only ever stored dependencies and handled this by slowly decrementing a counter. Learning non-dependence, rather than forgetting a dependence, is the whole idea.
Q4Why couldn't prior work just use one predictor for both?▾
Because MDP and SMB fear opposite errors. MDP dreads false negatives (missed deps → squash) and skews to over-predict dependence; SMB dreads false positives (bad value forwarded → squash) and skews the other way. A predictor tuned for one is mistuned for the other. MASCOT gets both error rates low simultaneously — mainly by killing false dependencies — so one table can serve both.
Q5What's in a MASCOT entry?▾
28 bits: a 16-bit tag, a 7-bit distance (0 = non-dependent; else store-queue offset), a 3-bit usefulness counter (MDP confidence), and a 2-bit bypass counter (SMB confidence). MDP fires when distance ≠ 0; SMB fires only when usefulness == 7 AND bypass == 2.
Q6How big is it, and how does that compare?▾
Default MASCOT is 14 KiB of tables (8 tables × 512 entries × 28 bits). PHAST, the prior state-of-the-art MDP predictor, is 14.5 KiB; a NoSQ+Store-Sets combo is ~18.5–19 KiB. MASCOT-OPT shrinks to 10.1 KiB — about 30% smaller than PHAST — while doing both MDP and SMB.
Q7How much faster is it, really?▾
On SPEC CPU2017: as an MDP-only predictor, +0.4% IPC over PHAST (and +6.2% over Store Sets) at the same size. Doing MDP+SMB, +1.9% IPC over PHAST on average, +4.9% over NoSQ, and even +1.0% over a perfect MDP-only predictor. Peaks reach ~26% (perlbench2). MASCOT-OPT keeps ~1.8% over PHAST at 10.1 KiB.
Q8Where do the accuracy gains come from?▾
Almost entirely from crushing false dependencies. Vs PHAST, MASCOT reduces false dependencies by 91% and speculative errors by 39%, for ~70% fewer total mispredictions (98% fewer than NoSQ). The ablation without non-dependence entries has >12× more false dependencies — direct evidence the non-dependence allocation is the cause.
Q9Is MASCOT on the critical path? Does it slow the core?▾
No. Predictions are made at decode and are ready by the time the load leaves the frontend (before it's needed at issue); updates happen at commit. The multi-table parallel lookup has pipeline slack to complete, so the predictor adds accuracy without lengthening the timing-critical loop.
Q10Why two saturating counters instead of one?▾
Because the two mechanisms need independently-gated confidence. The 3-bit usefulness counter governs MDP and merely decrements on a wrong dependence (cheap error). The 2-bit bypass counter governs SMB and resets to 0 on a wrong bypass (expensive error, so confidence must collapse instantly). SMB requires both saturated — sure there's a dependence AND sure the full value can be forwarded.
Q11Does SMB work with misaligned or partial-value loads?▾
MASCOT's default fires SMB only for same-address forwards where the store fully supplies the load's value (the load may be smaller than the store, but no address offset). Fig. 2 shows that's the vast majority of bypass opportunities. It can be extended to offset cases by adding a shifting field, at extra cost — the paper notes NoSQ handles partial-word bypassing, MASCOT keeps the common case cheap.
Q12What did the authors leave open?▾
They point at richer bypassing (offset/partial-word via a shift field), applying the idea to store-queue-free designs (DMDP-style predication), load-load bypassing (Moshovos & Sohi), and note the gains grow on wider future cores (Lion Cove), suggesting more headroom as windows enlarge. The evaluation also disables vectorization in the simulator, so SIMD interaction is left for future modeling.

Core terms

MDP (memory-dependence prediction)Predicting whether a load aliases an older in-flight store, to decide if it may issue early. A false negative (missed dep) forces a squash.
SMB (speculative memory bypassing)Forwarding a predicted older store's value directly into a dependent load before addresses are known. A false positive forwards a bad value → squash.
Non-dependence entryMASCOT's innovation: an entry with distance = 0, allocated in a longer-history table after a false dependence, encoding "no dependence in this context."
Distance field7 bits. 0 = non-dependent; a non-zero value is the offset into the store queue of the aliasing store (1 = nearest prior store).
Usefulness counter3-bit saturating confidence for MDP. Incremented on a correct dependence, decremented on a wrong one; only usefulness == 0 entries may be evicted.
Bypass counter2-bit saturating confidence for SMB. Incremented on a correct bypass, reset to 0 on a wrong one. SMB fires only when this is 2 and usefulness is 7.
Try-again allocationOn a full target set, allocate in the next-longer-history table, decrementing the passed-over table's entries — a self-cleaning policy that made periodic decay unnecessary.
