When small LLMs think harder, edge GPUs choke

Test-time compute lets a 1B-parameter model beat 405B by reasoning across multiple steps. But the messy control flow and memory fragmentation from iterative search bring edge GPUs to their knees. This is the story of ORCHES, a GPU–PIM system designed to fix it.

📄 "ORCHES: Orchestrated Test-Time-Compute-based LLM Reasoning" · Li et al., MICRO'25 🎓 Some LLM & memory-system basics assumed — the primers cover them ⏱ ~25 min read
⚡

Iterative LLM reasoning isn't just more inference — it's variable-parallelism, branch-dependent, memory-fragmenting inference. Accelerating it demands heterogeneous GPU–PIM collaboration, speculative pipelining, and memory reorganization tuned to the reasoning workload's unique shape.

Before you begin

Background you'll need

Eight concepts this paper builds on. Expand each to see how it connects to ORCHES.

generation (policy model) verification (PRM) prediction / pipelining memory / PIM
1LLM prefilling vs decoding▾

Prefilling processes an entire input prompt in parallel, computing KV caches for all tokens at once — high arithmetic intensity, compute-bound. Decoding generates one token at a time autoregressively, reusing the KV cache — low arithmetic intensity, memory-bound on edge devices.

In standard LLM inference, you prefill once, then decode a whole sequence. In test-time compute (TTC), each reasoning step alternates between generating (decoding) and verifying (prefilling), blending the two workloads unpredictably.

Now you're ready: ORCHES must handle both compute-bound prefill in PRMs and memory-bound decode in policy models, across varying batch sizes per step.
2Test-time compute (TTC)▾

Instead of outputting a single answer, TTC-based reasoning generates multiple candidate solutions at each step, scores them with a verification model (the Process Reward Model, or PRM), keeps the best few, and repeats — building a search tree with width = candidates per step and depth = reasoning steps.

This lets a 1B model rival or beat 405B by "thinking harder" at inference time. But the search involves irregular control flow, dynamic branching, and sequential dependencies between steps — none of which standard LLM accelerators expect.

Now you're ready: ORCHES targets exactly this iterative multi-branch reasoning, addressing the unique bottlenecks it creates.
3Processing-in-Memory (PIM)▾

GPUs have enormous compute but limited effective memory bandwidth — data must travel from memory banks through a narrow off-chip interface. PIM places lightweight compute units directly inside memory banks, exposing far higher internal bandwidth without moving data off-chip.

For memory-bound LLM decoding (batch size 1), PIM can be much faster than a GPU — but it has lower peak compute, so compute-bound workloads (large batch prefills) still favor the GPU.

Now you're ready: ORCHES uses a heterogeneous GPU–PIM system to assign compute-bound ops to GPU, memory-bound to PIM, dynamically as the workload shifts.
4Shared vs unique KV caches in beam search▾

All reasoning branches share the same input prompt and early trajectory, so their KV caches for that portion are identical. As branches diverge in later steps, each grows its own unique KV segment. When querying attention, the shared portion can be processed once and reused across all branches (higher parallelism, compute-bound), while unique segments must be processed per branch (lower parallelism, memory-bound).

The ratio of shared to unique KV shifts as the search deepens — shared grows larger with each step, while unique is reset for new candidates.

Now you're ready: ORCHES must dynamically balance workload assignment as this ratio changes, which a static GPU-only or PIM-only approach can't do efficiently.
5Arithmetic intensity▾

Arithmetic intensity = FLOPs ÷ bytes accessed. High intensity means lots of reuse (compute-bound); low means little reuse (memory-bound). For linear layers, intensity = batch size × hidden dimension ÷ parameter bytes. For attention, it depends on whether KV is shared (higher) or unique (lower).

A GPU's peak arithmetic intensity threshold (>500) is far above what small-batch edge workloads typically hit, so memory bandwidth becomes the limiting factor. PIM's internal bandwidth sidesteps this bottleneck.

Now you're ready: ORCHES uses roofline-style models of intensity to decide GPU vs PIM assignment per operator, per step.
6Memory fragmentation from branch pruning▾

At each TTC step, unselected candidates are discarded. Their KV caches remain in memory as "holes" until compacted, creating irregular access patterns. On PIM, where memory is distributed across many banks, fragmented access means more pointer-chasing and lower effective throughput.

Candidate lengths vary widely (10–1000 tokens per candidate in MATH dataset), so naive static allocation wastes memory, while dynamic allocation fragments over time.

Now you're ready: ORCHES introduces memory reorganization (when fragmentation ratio β nears 1) and a controller-side buffer to smooth out access patterns.
7Pipelining and inter-step dependencies▾

In standard decoding, each token depends on the last (auto-regressive), blocking token-level parallelism. In TTC, each step's generation depends on the previous step's verification results — you can't start generating step N+1 until you know which candidates from step N survived. This strictly enforced sequence limits pipelining.

Yet verification (PRM) can take as long or longer than generation (especially with a large PRM model), leaving the PIM idle during verification if there's no overlap.

Now you're ready: ORCHES uses speculative prediction to overlap generation and verification, breaking the dependency with rollback on misprediction (like CPU branch prediction).
8Edge device constraints▾

Edge platforms (e.g., NVIDIA AGX Orin) have limited off-chip memory bandwidth (~204.8 GB/s) and smaller compute capability than server GPUs. This makes memory-bound LLM decoding even slower, and exacerbates the stalls caused by TTC's irregular workload patterns.

Existing GPU-PIM systems target server settings with high throughput; ORCHES is designed specifically for the low-batch, high-irregularity edge TTC regime.

Now you're ready: ORCHES achieves 4.16× (text) and 3.10× (vision) speedup over SOTA GPU on edge, with no accuracy loss.

The whole story in one place

ORCHES in a nutshell

The problem, the three-technique solution, and why it matters for your OS-tiering work.

Standard LLM inference on edge devices is memory-bound and well-studied — GPU-PIM hybrid systems like AttAcc already accelerate it. But test-time compute (TTC) changes everything: a 1B model generating multiple reasoning branches per step, verified by an 8B PRM model, with inter-step dependencies and unpredictable control flow, creates a workload that's part compute-bound (PRM prefill), part memory-bound (policy decoding), and structurally irregular (branch pruning, dynamic batch sizes).

Existing approaches fall short because they assume uniform memory-bound behavior, static assignment, and no sequential dependencies. The paper identifies three concrete barriers: C1: Variable Parallelism (shared KV grows over time; operators shift between compute/memory-bound as branch count and search depth change); C2: Branch Dependencies (verification must complete before next-step generation begins, and vice versa, stalling the GPU or PIM in turn); C3: Branch Pruning Fragmentation (discarded candidates leave memory holes, degrading PIM access efficiency).

ORCHES addresses each with a targeted technique: T1 (Adaptive Assignment) uses an offline roofline-based partitioning of linear/attention operators across GPU and PIM based on batch size, plus online runtime compensation as the shared-KV ratio increases. T2 (Branch Prediction) introduces a lightweight PRM predictor (the first 10 layers of the 8B PRM) that speculatively selects candidates, enabling generation to proceed in parallel with verification, with rollback on misprediction (53% → 78% accuracy after history alignment). T3 (Memory Structuring) deploys an address cache in the controller die for fast pointer lookup, dynamic memory reorganization when fragmentation ratio β nears 1, and a shared-KV buffer to optimize GPU access and smooth out compaction.

Together, ORCHES achieves 4.16× speedup over GPU on text tasks (MATH500, Llama3.2-1B + Llama3.1-8B PRM) and 3.10× on vision tasks (MATHVista, Llama3.2-11B-Vision), with 2.45× energy efficiency and zero accuracy loss. On coding tasks (LiveCodeBench), speedup reaches 4.24×. The system incurs <1% area overhead (address cache + buffer) and adds no latency to the critical path.

🔑 Key takeaways

  • TTC reasoning mixes compute-bound (PRM prefill) and memory-bound (policy decode) workloads with dynamic batch sizes → naive GPU-only or PIM-only fails.
  • Shared KV cache grows with search depth while unique KV stays constant → offline assignment + online compensation needed.
  • Verification blocks generation and vice versa → speculative prediction + pipelining breaks the dependency.
  • Branch pruning fragments memory → address cache + reorganization (when β near 1) + controller buffer restore efficiency.
  • ORCHES: heterogeneous GPU–PIM + branch predictor + memory structuring = 4.16× (text) / 3.10× (vision) speedup, <1% area.
  • Connects to your tiering work: same tension between capacity (like PIM) and performance (like GPU), same need for adaptive workload assignment under irregular access patterns.

Seven everyday pictures

How to think about ORCHES

Each analogy stands alone. Every card ends with where the picture breaks.

The core idea · Heterogeneous assignment

Conveyor belt with two workers

One worker (GPU) is fast at big batches but slow at small ones. The other (PIM) is steady at small batches but can't match the GPU on large ones. Items arrive in varying sizes (batch sizes = candidates per step). A smart dispatcher (ORCHES) sends big batches to the GPU, small to PIM, and dynamically rebalances as the mix shifts — instead of locking each worker to one task regardless of load.

Where it breaks: a real dispatcher knows the batch size in advance; ORCHES must predict the compute/memory bound transition from roofline models and adapt online as shared KV grows.
Variable parallelism · Shared KV growth

Expanding shared prefix

A group of hikers follows the same trail for the first mile (shared KV), then splits into separate paths (unique KV per candidate). At each checkpoint (reasoning step), the shared trail extends by another segment as the group reunites, appending new shared context. The ratio of shared-trail to individual-path time increases with each checkpoint, changing the workload balance (more parallelism = more compute-bound).

Where it breaks: hikers walk at constant speeds; ORCHES's shared-KV workload shifts from memory-bound to compute-bound as batch size (number of hikers) and shared length both grow, requiring continuous rebalancing.
Branch dependencies · Sequential stalls

Assembly line with a quality check

Parts (candidates) move from fabrication (generation) to inspection (verification) to next-stage fabrication (generation step N+1). But inspection can't start until fabrication finishes, and next-stage fabrication can't start until inspection picks the passing parts. The two stations (GPU for inspection, PIM for fabrication) alternate idling, waiting for the other.

Where it breaks: a real assembly line might buffer parts; ORCHES can't — each step's generation depends on the previous step's verification outcome. The branch predictor effectively "guesses" which parts will pass and pre-starts next-stage fabrication speculatively.
Memory fragmentation · Branch pruning

Parking lot after a rush

Cars (candidates) fill the lot (memory banks), then half leave (pruned branches), leaving scattered empty spots. New arrivals (step N+1 candidates) must search for contiguous spaces, and the valet (PIM controller) spends extra time finding them. When fragmentation ratio β (empty spots ÷ total spots) hits a threshold, the valet compacts the remaining cars into a contiguous block, clearing one clean section for new arrivals.

Where it breaks: a valet moves cars physically; ORCHES's memory reorganization is a copy-and-invalidate operation (read KV from fragmented banks, write to contiguous region, update address cache). The controller's shared-KV buffer speeds this by caching frequently accessed data.
Branch prediction · Speculative execution

Draft manuscript before peer review

A small review panel (10 layers of the 8B PRM) quickly scans a paper draft and guesses which sections will pass full review. The author (generation) starts revising the predicted-pass sections immediately while the full panel (8B PRM) conducts the slow, thorough review in parallel. If the guess is right, the revision finishes earlier. If wrong, the author rolls back to the full panel's decision and re-revises — but the time lost is small if the guess is usually correct (78% after history alignment).

Where it breaks: a review panel's opinion is independent; ORCHES's small PRM depends on historical scores from the large PRM (history alignment replaces small-model scores with large-model scores from prior steps to improve accuracy from 53% to 78%).
Memory structuring · Address cache

Phone directory in the lobby

A building (memory banks) has scattered offices (KV cache fragments). Instead of searching floor by floor, visitors check a quick-lookup directory (address cache) in the lobby (controller die) that maps names (candidate IDs) to office numbers (physical addresses). Updating the directory is cheap (SRAM access), and it avoids slow sequential reads (DRAM traversal). When the directory gets stale after reorganization, it's updated once in batch.

Where it breaks: a phone directory is a fixed-size list; ORCHES's address cache is shared across all banks and stores only the start address + length per candidate (typically <1000 entries for common benchmarks, so overhead is minimal).
Memory structuring · Controller buffer

Warehouse loading dock

Incoming shipments (KV segments) are aggregated at a loading dock (controller die buffer) before being stacked in the warehouse (memory banks). When the GPU (a delivery truck) needs a shipment, it picks it up from the dock instead of driving into the warehouse aisles (memory banks), avoiding traffic jams with the warehouse's internal forklifts (PIM compute units). The dock also simplifies reorganization — move shipments from dock to contiguous shelves in one batch.

Where it breaks: a real loading dock has infinite capacity; ORCHES's buffer is small and caches only shared KV segments (frequently accessed across all candidates). Unique KV stays in the banks. The buffer adds 12% area overhead but cuts reorganization runtime overhead to 0.12%.
The full explanation

How it actually works

Section by section, with the key numbers from the paper.

1 · The hardware architecture — GPU + controller + memory dies

ORCHES integrates three silicon dies: Host GPU (e.g., NVIDIA AGX Orin) handles compute-intensive tasks (prefill, large-batch linear ops); Memory Controller Die sits between GPU and memory, performing aggregation (Accum Units = parallel adders for accumulation, Softmax Units = pipelined fixed-function for normalization) and caching frequently accessed data in a Shared KV Buffer; Memory Dies (32 GB total capacity, 2048 banks, each with 16 GEMV units = multiplier-adder trees) execute near-bank compute for memory-bound ops. The controller also hosts an Address Cache (SRAM) mapping candidate IDs to physical addresses, avoiding slow DRAM pointer-chasing.

2 · Technique 1 (Adaptive Assignment) — offline partition + online compensation

Offline partitioning uses a roofline model to decide GPU vs PIM for each operator based on batch size W. For linear operators and shared-KV attention, arithmetic intensity = (W × D²) ÷ (2WD + D²) for linear, where D = hidden dimension. When W is large (e.g., W = sequence length L > 100 in PRM prefill), intensity is high → compute-bound → assign to GPU. When W is small (e.g., W = 2–4 candidates in policy decoding), intensity is low → memory-bound → assign to PIM. For unique-KV attention, W = 1 always (batch size per candidate), so intensity is fixed at 2 (assuming standard attention; for Grouped Query Attention it depends on group size) — always memory-bound, always PIM.

Equations guide the decision: for a linear operator, latency on GPU is T_GPU = (WD²)/CC_GPU + (2WD + D²)/BW_GPU; on PIM it's T_PIM = (WD²)/CC_PIM + (2WD)/BW_PIM_IO + D²/BW_PIM, where BW_PIM_IO is controller-die IO bandwidth and BW_PIM is internal bank bandwidth. Compare T_GPU and T_PIM; if the workload is borderline, co-process: send fraction α of the output dimension to GPU, (1−α) to PIM, solving T_PIM(α) = T_GPU(α) to minimize max(T_PIM(α), T_GPU(α)). Data movement overhead is WD(2−α), which is typically small (W ≪ D), so D² dominates and co-processing adds ~8.3% runtime on average across benchmarks.

Online compensation handles the growing shared-KV workload. At each step, the system computes α_i per KV fragment KV_i (length L_i, batch W_i). Start with all layers on GPU (α_i = 1), then iteratively reassign layers to PIM (α_i = 0) from lowest W_i to highest until T_PIM ≥ T_GPU. For the critical layer t where the balance shifts, solve T_PIM = T_GPU for α_t while fixing other α_i.

🍯
The numbers: On a 3B policy + 7B PRM setup, T1 alone (without T2) achieves 3× speedup over GPU baseline, and GPU utilization is 97.9% (vs 62.2% for T2 alone), while PIM is 43.6% (vs 66.7% for T2 alone). Only T1+T2 together reach high utilization for both (GPU 93.21%, PIM 61.0%).

3 · Technique 2 (Branch Prediction) — small PRM + history alignment + pipelined pre-verification

Candidate verification predictor: Use the first 10 layers of the 8B PRM as a lightweight "small PRM" to predict which candidates will be selected, enabling generation step N+1 to start speculatively while the full 8B PRM verifies step N. If the prediction is correct, generation finishes 3×–5× earlier (because small-PRM prefill is 3×–5× faster than large-PRM prefill). If wrong, roll back and regenerate from the correct selection. To improve accuracy, history alignment replaces the small PRM's own historical scores (from steps 1 to N−1) with the large PRM's scores from those same steps, enriching the context with more reliable data. This boosts prediction accuracy from 53% (naive small model) to 78% (with alignment).

Pipelined pre-verification: Instead of waiting for all candidates to finish generating, start small-PRM verification as soon as a portion (enough tokens to reach sufficient arithmetic intensity for GPU utilization) is generated. This overlaps verification with generation, reducing inter-step latency. Pre-verification runs only when the GPU is idle (generation has priority), so it doesn't interfere with critical-path generation tasks.

💡
Case study (from Fig. 13): With a 1B policy + 8B PRM, the small PRM's correct prediction enables overlapping 60% of verification with generation, saving 100% of the sequential wait time. With a 3B policy + 8B PRM, pipelined pre-verification further overlaps 60% of small-PRM execution concurrently, reducing remaining wait by 40%.

4 · Technique 3 (Memory Structuring) — address cache + reorganization + controller buffer

Address cache (in controller die, shared across banks): Maps logical candidate IDs to physical memory locations (start address + length). Avoids storing pointers in DRAM cells (which incur high read latency) by using SRAM, transforming two sequential DRAM accesses (pointer fetch + data fetch) into SRAM + DRAM (1–2 orders of magnitude faster for the pointer). Since only sequence starts and lengths are cached (<1000 entries for typical benchmarks like MATH), area overhead is minimal.

Dynamic memory reorganization: Track fragmentation ratio β = (total memory holes) ÷ (memory for reasoning). When β approaches 1, compact valid KV blocks into contiguous space by reading from fragmented banks and writing to a contiguous region, then updating the address cache. The controller's shared-KV buffer speeds this: after QKV accumulation at the controller die, KV segments are stored in the buffer, reducing read operations during reorganization (write buffer contents back to banks to eliminate fragmentation). On average across all settings, memory reorganization saves 65% of context memory footprint (Tab. 5), while adding only 0.12% runtime overhead.

Controller buffer for GPU access: When decoding is fully offloaded to PIM, the GPU must fetch the latest KV cache. Instead of reading from scattered memory banks (interfering with PIM compute units), the GPU reads from the controller's shared-KV buffer, which caches frequently accessed shared KV segments. This avoids PIM-host data transfers during reorganization and allows background compaction without blocking GPU reads. Buffer adds 12% area overhead under the same hardware implementation settings (Sec. 5.5).

🌱
The memory-tiering analogy: ORCHES's address cache is like your extended page tables for tier pointers; reorganization is like your background DRAM-to-NVM migration when hot-page fragmentation hits a threshold. Both trade rare compaction overhead for sustained fast access.
Follow the numbers, step by step

Worked examples

Four traces using the paper's real numbers.

① Computing arithmetic intensity for a linear operator

A 3B model (hidden dimension D = 3072) decoding with batch size W = 4 (four candidates). How many FLOPs and bytes accessed?

  1. FLOPs for linear operator: output = input × weights, so FLOPs = W × D² = 4 × 3072² ≈ 37.7M FLOPs.
  2. Bytes accessed: input is W × D (4 × 3072 floats = 12,288 FP16 = 24,576 bytes), weights are D² (3072² FP16 = 18.9 MB). Output is W × D (24,576 bytes). Total ≈ 2WD + D² in bytes, ignoring constants = 24,576 + 18.9M ≈ 18.9M bytes.
  3. Arithmetic intensity: 37.7M FLOPs ÷ 18.9M bytes ≈ 2.0 FLOPs/byte.
  4. Compare to GPU threshold: AGX Orin peak intensity ~500+ FLOPs/byte. 2.0 ≪ 500 → memory-bound → assign to PIM.
  5. If W = 100 (PRM prefill): FLOPs = 100 × 3072² ≈ 942M, bytes ≈ 18.9M (weights dominate), intensity ≈ 50 FLOPs/byte. Still <500, but closer — borderline, might co-process.
🍯
Insight: Batch size W directly controls whether the workload is compute- or memory-bound. Small W (4) → PIM. Large W (100+) → GPU or co-process. This is the core of T1's offline partition.

② Deciding GPU vs PIM assignment for shared-KV attention

Shared KV cache length L = 200 tokens, batch size W = 4 candidates, D = 3072. Should Q·K^T (the shared-KV query) go to GPU or PIM?

  1. FLOPs for Q·K^T: Each candidate's query (W × D) multiplies the shared K (L × D). Total = W × L × D ≈ 4 × 200 × 3072 ≈ 2.46M FLOPs.
  2. Bytes accessed: Q is W × D (4 × 3072 FP16 = 24,576 bytes), K is L × D (200 × 3072 FP16 = 1.23 MB). Total ≈ W×D + L×D ≈ 1.25 MB.
  3. Arithmetic intensity: 2.46M FLOPs ÷ 1.25M bytes ≈ 2.0 FLOPs/byte → memory-bound → PIM.
  4. If W = 16 (wider search): FLOPs = 16 × 200 × 3072 ≈ 9.8M, bytes ≈ 1.26 MB (K dominates), intensity ≈ 7.8 FLOPs/byte. Higher, but still <500 → PIM, though approaching co-process threshold.
  5. As L grows to 500 (later steps): FLOPs = 4 × 500 × 3072 ≈ 6.14M, bytes ≈ 3.1 MB, intensity ≈ 2.0 still. W matters more than L for intensity here, because L affects both numerator and denominator linearly.
💡
Online compensation kicks in: If W and L both grow (e.g., W=16, L=500), FLOPs ≈ 24.6M, bytes ≈ 3.1MB, intensity ≈ 7.9. Now borderline — T1 recomputes α dynamically and might co-process or shift to GPU for that layer.

③ Branch prediction accuracy with history alignment

A small PRM (Qwen2.5-1.5B, first 10 layers) predicts which of 4 candidates the full PRM (Qwen2.5-7B) will select. Step N is Level 3 difficulty (MATH500). What's the prediction accuracy without and with history alignment?

  1. Without history alignment: The small PRM uses its own scores from steps 1 to N−1 to compute aggregated scores for step N. From Tab. 4, Qwen2.5-1.5B policy + Level 3 → 53.2% accuracy (naive).
  2. With history alignment: Replace the small PRM's historical scores with the large PRM's scores from steps 1 to N−1. Same row in Tab. 4 → 82.2% accuracy.
  3. Accuracy gain: 82.2% − 53.2% = 29 percentage points improvement.
  4. Impact on speedup: At 82.2% accuracy, 17.8% of predictions are wrong → rollback cost = regenerating step N+1 from the correct candidate. But the 82.2% correct cases overlap verification and generation, saving 3×–5× time. Net: substantial speedup (see Fig. 13 case studies).
  5. Why history alignment helps: The small model's own early predictions may be inaccurate, polluting its context. Using the large model's accurate historical scores enriches the context, so the small model's final prediction for step N is much more reliable.
🍯
The numbers: History alignment boosts accuracy from ~52% (avg across all configs) to ~78% (avg), a 26-point gain. This is what makes T2 practical — without it, rollback overhead would negate the speedup.

④ Memory footprint savings from reorganization (T3)

A TTC run starts with 8 candidates (step 1), prunes to 4 (step 2), then 2 (step 3). Each candidate's unique KV is ~100 tokens (FP16, D=3072). How much memory is wasted before reorganization, and how much is saved after?

  1. Initial allocation (step 1): 8 candidates × 100 tokens × 3072 × 2 bytes (FP16) = 8 × 100 × 3072 × 2 = 4.92 MB total.
  2. After pruning to 4 (step 2): 4 candidates remain (2.46 MB live data), but the memory for the pruned 4 is still allocated → 2.46 MB wasted. Fragmentation ratio β = 2.46 MB holes ÷ 4.92 MB total ≈ 0.5.
  3. After pruning to 2 (step 3): 2 candidates remain (1.23 MB live), 2 more pruned → 2.46 MB holes + 1.23 MB from step 2 = 3.69 MB holes total. β = 3.69 ÷ 4.92 ≈ 0.75 (approaching 1).
  4. Reorganization triggers (β near 1): Compact the 1.23 MB live data into contiguous space. Free the 3.69 MB holes. New allocation for step 4 uses the freed space, avoiding further fragmentation.
  5. Savings (from Tab. 5): On average across all policy/PRM combinations, T3 saves 65% of context memory footprint. In this example, without reorg you'd allocate 4.92 MB but use 1.23 MB (74.9% waste). After reorg, you allocate 1.23 MB (0% waste) → 65% of the 4.92 MB saved.
💡
The reorganization cost: Reading 1.23 MB from fragmented banks + writing to contiguous region + updating address cache. The controller's shared-KV buffer caches frequently accessed segments, reducing read ops. Average runtime overhead: 0.12% (Sec. 5.5). Buffer area overhead: 12%.

Problem 1 · easy
A policy model decodes with batch size W=8 (eight candidates). The linear operator has arithmetic intensity = 2W (simplified). The GPU's threshold for compute-bound is intensity ≥ 50. Should ORCHES assign this operator to GPU or PIM?
Show answer
Intensity = 2 × 8 = 16, which is < 50 → memory-bound → assign to PIM. Even though W=8 is relatively large for edge, the threshold is much higher, so the operator remains memory-bound and benefits from PIM's higher internal bandwidth.Misconception: that any multi-candidate batch is "large enough" for GPU. Problem 2 · medium
A TTC pipeline alternates generation (PIM, 5 ms) and verification (GPU, 8 ms). Without T2 (branch prediction), what is the total time for 3 reasoning steps? With T2 (78% accuracy, rollback cost = regenerating step N+1 at 5 ms), what is the expected time?
Show answer
Without T2: each step is sequential gen (5 ms) + verify (8 ms) = 13 ms. 3 steps = 39 ms total. With T2: overlap gen and verify. Step 1: gen 5 ms, verify 8 ms (parallel) = max(5,8) = 8 ms. Step 2 speculative gen starts at t=8, finishes at t=13. Step 2 verify (8 ms) overlaps, finishes at t=16. If prediction correct (78%), step 3 gen starts at t=8 (speculative), finishes at t=13, verify at t=21. If wrong (22%), rollback at t=16, regen 5 ms → t=21, verify 8 ms → t=29. Expected per step after step 1: 0.78×(13−8) + 0.22×(21−8) = 3.9 + 2.86 = 6.76 ms. Total ≈ 8 (step 1) + 6.76×2 ≈ 21.5 ms (vs 39 ms) → 1.8× speedup.Misconception: that perfect prediction is required for speedup. Even 78% accuracy yields substantial gain because the overlap benefit (3×–5×) far exceeds the occasional rollback cost.
Problem 3 · medium
Why does ORCHES use the first 10 layers of the 8B PRM as the "small PRM" instead of training a separate lightweight model?
Show answer
Using the first 10 layers of the same model ensures (1) no additional training cost, (2) the small PRM is a strict subset of the large PRM, so if the large PRM's full execution is partitioned (small PRM → remaining layers), there's no computational redundancy — the small PRM's work contributes directly to the final result, (3) easier integration into the pipelined verification (pre-execute small PRM, then finish with remaining layers). A separate model would require separate prefill and wouldn't naturally pipeline into the large PRM's execution.Misconception: that the small PRM is a separately trained distilled model. It's simply the first N layers of the full PRM, run speculatively and then continued with the remaining layers if needed.
Problem 4 · hard
ORCHES's address cache maps candidate IDs to physical addresses. Why not store pointers directly in DRAM alongside each KV segment?
Show answer
Storing pointers in DRAM means: (1) to access KV for candidate i, you must first read the pointer from DRAM (one DRAM access, ~50 ns latency), then read the actual KV data (another DRAM access). These two accesses have a data dependency, so they're sequential → total latency = 2× DRAM latency. (2) The address cache in the controller die (SRAM, ~1–2 ns latency) breaks this: read pointer from SRAM (fast), then read KV from DRAM (one access). The SRAM read is 1–2 orders of magnitude faster, so the total latency is dominated by the single DRAM access, not the pointer fetch. (3) Since the address cache is shared across all banks and only stores start+length (<1000 entries for typical benchmarks), the area overhead is minimal compared to the latency gain.Misconception: that DRAM-stored pointers are "free" because they're just metadata. The latency cost of the sequential pointer→data dependency is significant on PIM, where every DRAM access incurs bank activation overhead. The address cache avoids this.
Problem 5 · medium
ORCHES triggers memory reorganization when fragmentation ratio β (holes ÷ total) nears 1. Why not reorganize after every pruning step to keep memory perfectly contiguous?
Show answer
Reorganization has overhead: reading live KV from fragmented banks, writing to contiguous region, updating address cache. Doing this every step would add constant overhead (even when fragmentation is low). By waiting until β nears 1, ORCHES amortizes the reorganization cost across multiple steps — only paying it when fragmentation is severe enough to degrade access efficiency. The paper shows average runtime overhead is 0.12% with threshold-based reorg; reorganizing every step would likely be higher (not quantified, but implied by the design choice).Misconception: that perfect contiguity is always optimal. The cost of maintaining it (frequent reorg) can exceed the benefit if fragmentation isn't severe. ORCHES trades some fragmentation tolerance for lower overhead, triggering reorg only when necessary.
Problem 6 · hard
ORCHES reports 4.16× speedup on text tasks (MATH500) and 3.10× on vision tasks (MATHVista). Why is the vision speedup lower?
Show answer
Vision tasks (using Llama-3.2-11B-Vision for both policy and PRM, Tab. 3) have: (1) the same model for policy and PRM, so the PRM isn't substantially larger → less compute-bound advantage for GPU in verification, (2) search tree width is smaller (2 or 4 vs up to 8 in text tasks) → lower batch sizes → less parallelism to exploit, (3) question lengths vary more widely (short/medium/long QA), and the speedup depends on how much shared KV accumulates (longer questions → more shared KV → more benefit from T1's online compensation). The 3.10× average reflects these factors. On long QA with width=2, speedup reaches 4.85× (Tab. 3), matching text tasks, but shorter QA and smaller widths pull the average down to 3.10×.Misconception: that the techniques apply uniformly across tasks. The speedup depends heavily on workload characteristics (batch size, model size ratio, shared-KV accumulation rate). Vision tasks' smaller batch sizes and same-size policy/PRM reduce the heterogeneity that ORCHES exploits.
Problem 7 · easy
ORCHES's T1 uses co-processing (splitting an operator α to GPU, 1−α to PIM). What happens if α=0 or α=1?
Show answer
α=0 → entire operator on PIM (no GPU participation). α=1 → entire operator on GPU (no PIM participation). These are the "pure" assignments. Co-processing (0 < α < 1) happens only when neither pure assignment is clearly optimal — specifically, when splitting the workload and incurring data movement overhead (WD(2−α) bytes) is faster than running entirely on the slower device. The equations (Sec. 4.2.1) solve for α that minimizes max(T_PIM(α), T_GPU(α)), which can be 0, 1, or between 0 and 1 depending on W, D, and bandwidth/compute parameters.Misconception: that co-processing always happens. It's an optimization for borderline cases; many operators are clearly GPU-bound or PIM-bound and get pure assignment (α=0 or α=1).
Problem 8 · medium
ORCHES reports average GPU utilization of 93.21% and PIM utilization of 61.0% (with T1+T2). Why isn't PIM also at ~90%+?
Show answer
PIM handles memory-bound ops (policy decoding, unique-KV query), which are inherently less parallel than GPU's compute-bound ops (PRM prefill, large-batch linear). Even with perfect assignment, PIM's utilization is bounded by: (1) decoding is auto-regressive (sequential token generation), limiting intra-step parallelism, (2) unique-KV queries are per-candidate (batch size 1), offering no batch-level parallelism, (3) during verification (GPU-bound phase), PIM runs speculative generation (T2), which is lower-priority and may idle if GPU needs resources. The 61% reflects these structural limits, not inefficiency in ORCHES. For comparison, T1 alone (without T2) achieves 43.6% PIM utilization, so T2 boosts it by ~17 points — but 100% isn't achievable given the workload's memory-bound, sequential nature.Misconception: that high utilization is always possible. Memory-bound, sequential workloads inherently have lower parallelism, so PIM's 61% is strong given the constraints. The key is that GPU and PIM are both productively busy (not idle waiting on each other), which T1+T2 achieve.
Quick answers

Frequently asked questions

Tap any to expand.

Q1What exactly is "test-time compute" (TTC)?▾
Instead of generating a single answer, TTC generates multiple candidate solutions at each reasoning step, scores them with a verification model (Process Reward Model, PRM), keeps the best, and repeats. This builds a search tree with width = candidates per step and depth = reasoning steps. It lets a 1B model rival 405B by "thinking harder" at inference time, trading more compute for better quality.
Q2Why can't you just use a bigger GPU for edge TTC?▾
Edge devices (e.g., NVIDIA AGX Orin) have limited power, cost, and form factor — you can't fit a server GPU. Even on the available GPU, TTC's small-batch decoding (W=2–8 candidates) is memory-bound, so the GPU's compute sits idle while waiting for data. PIM accelerates memory-bound ops by bringing compute to memory, but PIM alone can't handle the compute-bound PRM prefill. You need both, dynamically assigned.
Q3How does ORCHES decide "GPU or PIM" for each operator?▾
ORCHES uses a roofline-based analytical model: compute arithmetic intensity = FLOPs ÷ bytes accessed as a function of batch size W. If intensity exceeds the GPU's threshold (~500 FLOPs/byte for real ops on AGX Orin), the operator is compute-bound → GPU. Otherwise, memory-bound → PIM. For borderline cases, co-process by splitting the operator (fraction α to GPU, 1−α to PIM) to balance latency. This decision is made offline based on expected batch sizes, then adjusted online as shared-KV grows.
Q4What is "shared KV cache" and why does it grow?▾
All reasoning branches share the same input prompt and early trajectory, so their KV caches for that portion are identical (shared KV). As the search deepens, selected candidates are appended to the prompt → shared KV grows. Each candidate also has unique KV for its divergent portion, which resets each step. The growing shared KV increases parallelism (all candidates query the same shared data → higher batch size → more compute-bound), shifting the optimal GPU/PIM assignment over time.
Q5How does the branch predictor work?▾
ORCHES uses the first 10 layers of the 8B PRM as a "small PRM" to predict which candidates will survive verification. This small PRM is 3×–5× faster than the full PRM (because prefilling 10 layers is much faster than 32 layers). The predictor runs speculatively during verification of step N, allowing generation of step N+1 to start early. If the prediction is correct (78% accuracy after history alignment), step N+1 finishes much earlier. If wrong, roll back and regenerate from the correct candidate. Net: substantial speedup despite occasional rollbacks.
Q6What is "history alignment" and why is it needed?▾
The PRM's decision at step N depends on its scores from steps 1 to N−1 (historical context). If the small PRM uses its own (less accurate) historical scores, its prediction for step N is unreliable (53% accuracy). History alignment replaces the small PRM's historical scores with the large PRM's scores from the same steps, enriching the context with accurate data. This boosts accuracy to 78%, making the predictor practical.
Q7Why does branch pruning cause memory fragmentation?▾
At each step, unselected candidates are discarded, but their KV caches remain in memory as "holes" until explicitly freed or compacted. With candidate lengths varying widely (10–1000 tokens in MATH), the holes are irregular, creating fragmented access patterns. On PIM, where memory is distributed across 2048 banks, fragmented access means more pointer-chasing and bank conflicts → lower throughput. ORCHES fixes this with reorganization (when fragmentation ratio β nears 1) and an address cache for fast pointer lookup.
Q8What does the controller's shared-KV buffer do?▾
After computing QKV in the controller die's accumulation units, the resulting KV segments are stored in a buffer before being written to memory banks. This buffer serves three purposes: (1) cache frequently accessed shared-KV segments so the GPU can read them without activating memory banks (avoiding interference with PIM compute), (2) reduce read operations during memory reorganization (buffer contents are already in the controller, so compaction writes them directly to contiguous banks), (3) smooth out GPU access patterns, as the buffer provides a single contiguous read point instead of scattered bank accesses. Area overhead: 12%. Runtime overhead of reorganization: 0.12%.
Q9Does ORCHES always beat GPU-only?▾
On the tested benchmarks (MATH500, LiveCodeBench, MATHVista), ORCHES achieves 3.10×–4.24× average speedup across varying batch sizes, model sizes, and question lengths. The speedup is lowest when batch sizes are very large (approaching compute-bound even for policy decoding) or when the PRM is the same size as the policy model (reducing GPU/PIM heterogeneity), but it never falls below 1× (no slowdown). The techniques degrade gracefully — if TTC becomes more uniform, ORCHES reduces to the baseline GPU or PIM-only assignment, preserving correctness.
Q10How does this connect to OS memory tiering?▾
ORCHES's core challenge — dynamically assigning workloads between heterogeneous resources (GPU = fast/expensive, PIM = slow-per-op but high-bandwidth/cheap) based on runtime access patterns — mirrors your LtRAM work's challenge: assigning pages between DRAM (fast/expensive) and NVM (slow/cheap) based on access frequency. ORCHES's T1 (adaptive assignment) = your promotion/demotion policy. ORCHES's T3 (address cache + reorganization) = your extended page tables + compaction to handle tier-migration fragmentation. Both systems trade rare reorganization overhead for sustained fast access to the right-tier data.
Q11What's the area overhead of ORCHES?▾
The address cache and shared-KV buffer together add <1% area overhead relative to the baseline GPU-PIM system (which already includes 2048 PIM banks with GEMV units). Specifically, the address cache is shared across all banks and stores <1000 entries (start address + length per candidate, ~10 KB total). The buffer is in the controller die and sized to cache shared-KV segments (~100 KB). Compared to the 32 GB memory capacity and the GPU's die area, this is negligible. The paper reports 12% buffer overhead under the same hardware implementation settings (Sec. 5.5), but this is relative to the controller die alone, not the total system.
Q12Can ORCHES handle non-TTC workloads?▾
Yes. For standard single-step LLM inference (no TTC), ORCHES reduces to prior work's GPU-PIM assignment (T1 with static batch size = 1 → PIM for decoding, GPU for prefill). T2 (branch prediction) doesn't apply (no iterative search), and T3 (memory structuring) isn't needed (no branch pruning). The system degrades gracefully to the baseline AttAcc-style behavior, so it's strictly better (or equal) on any LLM workload, not just TTC.

Core terms
ORCHESOrchestrated GPU–PIM system for test-time-compute LLM reasoning acceleration. Addresses variable parallelism, branch dependencies, memory fragmentation via three techniques.
TTC (Test-Time Compute)Multi-step iterative reasoning: generate multiple candidates per step, verify with PRM, select best, repeat. Increases inference "width" (candidates) and "depth" (steps).
PRM (Process Reward Model)Verification function scoring candidate solutions at each reasoning step. Typically a large model (8B) running prefill (compute-bound on GPU).
PIM (Processing-in-Memory)Compute units inside memory banks, exposing high internal bandwidth without off-chip data movement. Ideal for memory-bound ops like small-batch decoding.
Shared KV cacheKV cache segments common to all reasoning branches (from shared prompt and trajectory). Grows with search depth, increasing parallelism and shifting compute/memory bound.
Unique KV cacheKV cache unique to each candidate's divergent portion. Resets each step, remains small, always memory-bound (batch size 1 per candidate).
Arithmetic intensityFLOPs ÷ bytes accessed. High intensity (≥500 on AGX Orin) → compute-bound → GPU. Low intensity → memory-bound → PIM. Depends on batch size W.
Branch prediction (T2)Speculative execution: small PRM (first 10 layers of 8B PRM) predicts candidate selection, enabling overlapped generation and verification. 78% accuracy after history alignment.
Supporting terms
Prefilling vs decodingPrefilling: process entire prompt in parallel (compute-bound). Decoding: generate one token at a time auto-regressively (memory-bound on edge).
Policy modelThe generator in TTC (e.g., Llama3.2-1B). Runs decoding to produce candidates. Memory-bound on edge → PIM.
Batch size WNumber of input tokens processed in parallel. In TTC: W = number of candidates (2–8 typical). Determines arithmetic intensity and GPU vs PIM assignment.
Roofline modelAnalytical framework comparing arithmetic intensity to hardware's peak intensity threshold to predict compute- vs memory-bound behavior. Used in T1 for offline partitioning.
Co-processingSplitting an operator between GPU (fraction α) and PIM (fraction 1−α) when neither pure assignment is optimal. Adds data movement overhead WD(2−α) bytes.
History alignmentReplacing small PRM's historical scores with large PRM's scores from prior steps to improve prediction accuracy (53% → 78%).
Fragmentation ratio β(Total memory holes) ÷ (memory for reasoning). Triggers reorganization when β nears 1. Typical pruning creates β ≈ 0.65 after several steps.
Address cacheSRAM in controller die mapping candidate IDs to physical addresses (start + length). Avoids slow DRAM pointer-chasing (<1000 entries, ~10 KB).
Controller dieInterface between GPU and memory dies. Performs accumulation (Accum Units), softmax (Softmax Units), caches shared KV (buffer), and hosts address cache.
GEMV unitGeneral Matrix–Vector multiplication unit in memory banks (multiplier-adder tree). 16 per bank, 2048 banks total → high parallelism for memory-bound ops.
