Moving the compute to the flash, not the flash to the channel

Running LLMs on your phone is bottlenecked by SSD bandwidth. AiF puts GEMV accelerators inside flash chips, leveraging 102.4 GB/s internal bandwidth — a 14.6× speedup and 1.4× faster than in-memory inference.

Before you begin

Eight ideas the paper assumes you know

Each card clarifies one concept AiF builds on. Read in order or jump to anything unfamiliar.

internal flash bandwidth (AiF) external channel bandwidth (bottleneck) GEMV accelerator / ECC baseline SSD offloading
1On-device LLM memory bottleneck▾

LLaMA-3 8B needs 16 GB of memory (INT8). High-end phones have 8–12 GB. Laptops average 16 GB. So the model won't fit, and inference engines offload parameters to the SSD. But SSDs deliver only 4–8 GB/s external bandwidth, while DRAM gives 80–100 GB/s.

LLM inference is memory-bound (1–2 ops/byte), so token generation rate is capped by bandwidth / model_size. A 40B model on an 8 GB/s SSD maxes out at 0.2 tokens/s — far below the 3 tokens/s needed for real-time chat.

Now you're ready: AiF targets this exact bottleneck by accessing parameters at 102.4 GB/s internally, bypassing the slow external channels entirely.
2Flash chip vs SSD controller layout▾

A 1 TB SSD has ~16 flash chips, each connected to the SSD controller through 8 flash channels (multi-drop bus). Only one chip per channel can transfer data at a time, limiting external bandwidth to 12.8–19.2 GB/s (8 channels × 1.6–2.4 GB/s).

But internally, all 16 chips can read in parallel, delivering 25.6 GB/s aggregate bandwidth. Existing in-storage processing (ISP) still moves data over the channels, so ISP is channel-bound. In-flash processing (IFP) computes inside the chips, leveraging the higher internal bandwidth directly.

Now you're ready: AiF is an IFP solution — it performs GEMV inside each flash chip and sends only the small output vector over the channels, not the entire model matrix.
3NAND flash pages: LSB, CSB, MSB▾

A TLC cell stores 3 bits by encoding 8 distinct voltage (VTH) states. Reading a cell requires comparing VTH against reference voltages (VREF). To read all 3 bits, you need multiple sensing operations — typically 2, 3, and 2 for LSB, CSB, and MSB pages respectively (called "2,3,2 coding").

More sensing operations = longer read latency (tR). Also, the middle states have narrower margins, so CSB/MSB pages have higher error rates than LSB pages. The paper exploits this by reconfiguring to "1,3,3 coding" — LSB becomes as fast and reliable as SLC (single sensing), while CSB/MSB slow down.

Now you're ready: AiF stores LLM parameters only on LSB pages, achieving SLC-level speed and reliability while leaving CSB/MSB for general data.
4NAND flash read sequence: precharge, eval, discharge▾

A conventional flash read has three phases: (1) precharge — charge wordlines (WLs) and bitlines (BLs) to VPASS/VDD; (2) evaluation — sense whether the cell turns on by detecting current on the BL; (3) discharge — return WLs and BLs to 0 V before the next operation.

Precharge and discharge are slow (high RC delay), consuming ~72% of read energy. They exist to reset the block to a known state, but if you're reading successive WLs in the same block, most voltages can be reused from the previous read.

Now you're ready: AiF's charge-recycling read (cr-read) skips precharge/discharge between successive WL reads, cutting tR by 64% and energy by 72%.
5Error correction: on-chip vs off-chip ECC▾

Flash has raw bit error rates (RBER) > 10⁻³. LLMs are error-sensitive — even 10⁻⁷ RBER drops LLaMA-3 8B accuracy by 60%. So robust error correction codes (ECC) are mandatory. Conventional SSDs use a powerful LDPC decoder at the controller (off-chip), which handles data after it crosses the channels.

For IFP, the ECC must run inside each chip before computation. But decoding throughput must match the SSD's internal bandwidth (102.4 GB/s for 16 chips), which would require enormous power and area if each chip had a full LDPC decoder.

Now you're ready: AiF uses bias-error encoding to make LSB pages 87.5% more reliable, enabling a compact on-chip BCH decoder (ECCLITE) instead of LDPC — cutting area and power by ~15×.
6GEMV (matrix-vector multiplication) dominance in LLM inference▾

LLM decode phase is autoregressive: generate one token at a time, reading the entire model for each token. Most operations are GEMVs — multiply a large weight matrix (e.g., 175 MiB) by a small token embedding vector (e.g., 16 KiB). The result is another small vector.

This data reduction property is ideal for IFP: read a big matrix internally at high bandwidth, compute the small output vector inside the chip, and send only that small vector over the slow channel. Compare to offloading the matrix over the channel first — that wastes bandwidth on data you'll throw away.

Now you're ready: AiF integrates GEMV product elements (PEs) into flash chips, so the matrix never leaves the chip — only the output vector crosses the channel.
7Sampling vs monitoring all sets▾

A flash chip has ~1000 sets. Monitoring all sets (for e.g., utility curves) is expensive. But if sets behave similarly (low variance), watching a small random sample — say, 32 of 1024 — accurately estimates the whole chip. A Chebyshev bound says the error shrinks with sample size: P(off by ≥1 way) ≤ σ²/n.

Real workloads have σ² < 3, so 32 sets suffice. This technique — dynamic set sampling (DSS) — is what lets UCP (the reference paper) shrink its monitor to under 2 KiB. AiF applies the same principle to keep on-chip structures compact.

Now you're ready: AiF's on-chip ECC and GEMV logic are per-chip, but by storing parameters sequentially in LSB pages of known blocks, overhead stays minimal — no need to monitor every set.
8Head-level and tensor-level parallelism in LLM decode▾

A decoder block has three stages: QKV generation (GEMV to produce Q, K, V vectors for each attention head), multi-head attention (MHA, which accesses the KV cache in memory), and feed-forward networks (FFN, more GEMVs). Each attention head is independent, so you can start MHA on head 0 while AiF is still generating Q,K,V for head 1 — that's head-level parallelism.

Tensor-level parallelism splits a projection matrix into submatrices: host loads one submatrix into memory, AiF handles the rest on-chip, and both compute in parallel. The degree of parallelism depends on available host memory.

Now you're ready: AiF overlaps host and AiFSSD execution (Figure 15c in the paper), achieving 1.4× higher throughput than in-memory inference despite 8 GB memory vs 128 GB.
The whole story in one place

AiF in a nutshell

The bottleneck, the two read optimizations that fix it, and why it beats in-memory.

On-device LLMs require reading gigabytes of parameters per token. Offloading to an SSD should help, but external SSD bandwidth (4–8 GB/s) is 10–20× slower than DRAM, capping a 40B model at 0.2 tokens/s — far below the 3 tokens/s needed for usable chat. The root cause is the flash channel bottleneck: even though 16 flash chips can read at 25.6 GB/s aggregate internally, they share 8 channels delivering only 12.8 GB/s externally.

AiF's fix is to move GEMV into the flash chips, not the data out of them. Each chip holds a submatrix of model weights and performs GEMV locally using integrated product elements (PEs). The host sends a small input vector (8–32 KiB) to all chips; each computes its sub-result; the SSD controller aggregates. This leverages the full 102.4 GB/s internal bandwidth (16 chips × 6.4 GB/s) while sending only the tiny output vector over the slow channels.

Two novel flash read techniques unlock this: charge-recycling read (cr-read) skips the precharge and discharge phases when reading successive wordlines in a block (models are stored sequentially), cutting read latency (tR) by 64% and energy by 72%. Bias-error encoding (be-enc) reconfigures TLC pages from (2,3,2) to (1,3,3) coding, making LSB pages as fast and reliable as SLC (single sensing, 87.5% fewer errors) while CSB/MSB pages slow down. AiF stores LLM parameters only on LSB pages, so they read fast and need only a compact on-chip BCH decoder (ECCLITE, 10-bit correction per 1 KiB) instead of a power-hungry LDPC.

The payoff: 14.6× throughput vs baseline SSD offloading; 1.4× throughput vs in-memory inference (8 GB vs 128 GB memory); 5.74 tokens/s for 20B models (2.7 tokens/s for 40B models); under 1% area overhead; 72% energy reduction per read. All with no changes to the flash chip fabrication — only VTH state encoding and a few hundred bytes of on-chip logic.

🍯 Key takeaways

  • External channels are the bottleneck (12.8 GB/s) — internal flash bandwidth is 2× higher (25.6 GB/s) but unused in baseline SSD.
  • AiF puts GEMV inside flash chips (in-flash processing, IFP) to bypass channels entirely — only the small output vector crosses.
  • Cr-read recycles voltage settings between successive WL reads (models stored sequentially) — 64% faster tR, 72% lower energy.
  • Be-enc makes LSB pages SLC-like (1 sensing, 87.5% fewer errors) — enables compact on-chip ECC (ECCLITE) vs power-hungry LDPC.
  • LLM parameters stored only on LSB pages; general data on CSB/MSB pages (slight latency increase, but existing ECC margin absorbs it).
  • Head-level and tensor-level parallelism overlap host and AiFSSD execution — 1.4× faster than in-memory despite 16× less memory.
  • Result: 14.6× throughput vs baseline offloading; 5.74 tokens/s for 20B models; <1% area overhead; 0.209 mm² per chip.
Seven everyday pictures

How to think about AiF

Each picture stands alone. Every card ends with an honest note about where it breaks.

The core idea · Channel bottleneck

Factory with a loading dock bottleneck

16 warehouses (flash chips) can each pack boxes at 6.4 GB/s, but they share 8 loading docks (channels) that each move 1.6 GB/s → only 12.8 GB/s leaves the factory. Baseline SSD ships raw materials through the docks (slow). AiF assembles products inside each warehouse and ships only the finished goods — tiny boxes that fit through the docks easily. Same factory, no new loading docks, 8× more throughput.

Where it misleads: factories can expand docks; SSDs are pin-limited and can't just add channels without redesigning the package. The analogy also hides that cr-read speeds up packing within warehouses, not dock speed.
Cr-read · Voltage recycling

Shower temperature knob

Reading a wordline is like setting the shower temperature: turn hot knob (precharge WLs), test water (evaluation), turn cold (discharge). If you're reading successive wordlines in the same block (models stored sequentially), you can reuse the existing temperature — just nudge the knob slightly for the next WL instead of resetting to cold every time. Cr-read saves the precharge/discharge steps, cutting tR from 28 µs to 9.7 µs (64% faster).

Where it breaks: a shower has thermal inertia; flash has capacitive discharge that's actually faster than the analogy suggests. Also, cr-read only works for successive WLs in the same block — random reads still need full precharge.
Be-enc · State encoding bias

Express lane vs regular checkout

A TLC cell stores 3 bits on three pages (LSB, CSB, MSB). Conventional (2,3,2) coding treats them equally — 2, 3, 2 sensing operations. Be-enc reconfigures to (1,3,3), making LSB the express lane (1 sensing, SLC-speed, 87.5% fewer errors) while CSB/MSB become regular checkout (3 sensing each, slower). AiF puts LLM parameters in the express lane (LSB pages) and general data in regular lanes. Same chip, just rearranged the voltage thresholds.

Where it breaks: express lanes don't change the total checkout capacity; be-enc does — LSB pages get faster and CSB/MSB slow down. Also, the reliability boost comes from LSB using only the middle VTH states (P3/P4), which have the widest margins — not from intrinsic page properties.
ECCLITE · Compact on-chip ECC

Industrial vs home air filter

Conventional off-chip ECC is like an industrial air filter — handles heavy pollution (RBER > 10⁻³), costs a lot in power/area. Be-enc makes LSB pages 87.5% cleaner (RBER ≈ 10⁻⁴), so AiF can use a home filter (BCH decoder, 10-bit correction per 1 KiB) inside each chip. Same clean air, 15× less power and area (0.167 mm² vs 2.5 mm² for LDPC). The key: you don't need a sledgehammer when you've reduced the dirt.

Where it breaks: air filters have static capacity; ECC decoding throughput must match internal bandwidth (6.4 GB/s per chip). Also, CSB/MSB pages still exist and need the off-chip LDPC for general data — ECCLITE only handles LSB pages (LLM parameters).
GEMV data reduction · IFP sweet spot

Juicer vs shipping oranges

GEMV is like juicing oranges: read a crate of oranges (175 MiB matrix), squeeze out juice (16 KiB output vector). Baseline SSD ships the whole crate over the channel (slow). AiF puts the juicer inside the chip — only the juice crosses the channel (10,000× less data). The channel bandwidth is irrelevant when you've reduced the payload by orders of magnitude. That's why GEMV is the IFP sweet spot — extreme data reduction.

Where it misleads: juicing is lossy; GEMV is lossless (the output vector contains all the information needed for the next layer). Also, the host must still send the input vector to all chips — that's overhead, but tiny (16 KiB vs 175 MiB matrix).
Parallel execution · Overlapping host/SSD

Two chefs in a kitchen

A decoder block has three stages: QKV generation (GEMV), MHA (memory-bound), FFN (GEMV). Baseline runs them sequentially — AiFSSD waits while host does MHA. AiF's parallel execution splits the work: as soon as AiFSSD finishes Q,K,V for head 0, host starts MHA on head 0 while AiFSSD generates Q,K,V for head 1. Both chefs work simultaneously, no idle time. That's how AiF beats in-memory (1.4×) despite 16× less memory — utilization, not just bandwidth.

Where it breaks: chefs share the kitchen (memory/PCIe bandwidth); AiF's host and SSD share PCIe for result transfers. The speedup depends on available host memory for tensor-level parallelism — 8 GB limits degree to ~2 for large models.
IFP vs non-IFP pages · Block partitioning

Reserved parking vs general parking

Each TLC wordline has 3 pages (LSB, CSB, MSB). AiF reserves LSB pages for LLM parameters (fast, reliable, IFP-accelerated), leaves CSB/MSB for general data (slower, but conventional off-chip ECC handles them). Only blocks containing LLM parameters get be-enc applied. General-only blocks stay (2,3,2) coding. It's like a parking lot with reserved spots (LSB) and general spots (CSB/MSB) — same lot, different rules per zone.

Where it breaks: parking spots don't share space; LSB/CSB/MSB pages share the same wordline (you can't program LSB without eventually programming CSB/MSB). AiF statically pre-allocates IFP blocks when posting the model, but if the model is updated, it must be rewritten to maintain sequential LSB layout.
The full explanation

How it actually works

Section by section, with key mechanisms rebuilt as step-by-step visuals.

1 · The bottleneck — why baseline SSD offloading fails

On-device LLM decode is autoregressive: generate one token, read the entire model, repeat. Token rate is capped by bandwidth / model_size. A 40B model (40 GB INT8) on an 8 GB/s SSD maxes out at 0.2 tokens/s — below the 3 tokens/s threshold for usable real-time chat.

The root cause is the flash channel bottleneck. A 1 TB SSD has 16 flash chips reading at 1.6 GB/s each → 25.6 GB/s aggregate internal bandwidth. But they share 8 channels (multi-drop bus), each 2.0 GB/s → only 16 GB/s external bandwidth. Existing in-storage processing (ISP) moves data over channels, so ISP is still channel-bound (12.8 GB/s). In-flash processing (IFP) computes inside chips and sends only the small result, bypassing the bottleneck.

🌱
Concretely: reading a 175 MiB matrix at 16 GB/s takes 11 ms. With IFP, read the matrix internally at 25.6 GB/s (6.8 ms), compute GEMV (negligible), send the 16 KiB output vector over channels (1 µs) → 6.8 ms total, 1.6× faster. With AiF's 102.4 GB/s (cr-read + be-enc), it's 1.7 ms — 6.5× faster than baseline.

2 · Charge-recycling read (cr-read) — 64% faster tR

Conventional flash read has three phases: (1) precharge — charge WLs from 0 V to VPASS (6 V), BLs to VDD; (2) evaluation — apply VREF to target WL, sense current; (3) discharge — return WLs and BLs to 0 V. Precharge and discharge are slow (high RC delay, full-range voltage swing), consuming tPRE = 18 µs and tDISCH = 10 µs out of tR = 37 µs total.

Cr-read exploits a key property: when reading successive WLs in the same block (LLM parameters are stored sequentially), most voltages can be reused. After reading WL_i, only WL_(i+1) needs to discharge from VPASS to VREF, and WL_i charges from VREF to VPASS — all others hold VPASS. BLs need only slight recharge (voltage drop during eval is minimal). The recycling phase tRECY = 6.04 µs → tR = tRECY + tEVAL = 9.7 µs (64% reduction).

# conventional read sequence for n WLs
for i in 1..n:
    precharge(WLs, BLs)         # tPRE = 18 µs
    evaluate(WL_i)              # tEVAL = 9 µs
    discharge(WLs, BLs)         # tDISCH = 10 µs
# tR = 37 µs per WL

# cr-read sequence (successive WLs in same block)
precharge(WLs, BLs)             # first WL only
for i in 1..n:
    recycle(WL_i → WL_(i+1))    # tRECY = 6.04 µs (partial swing)
    evaluate(WL_(i+1))          # tEVAL = 9 µs
discharge(WLs, BLs)             # last WL only
# tR = 9.7 µs per WL (after first)
🍯
Implementation: cr-read requires only minor firmware changes — the X-decoder (WL voltage control) and timer code in the flash scheduler already exist. Effective bandwidth per chip goes from 2.3 GB/s (conventional TLC LSB read, tR=28 µs single-sensing after be-enc) to 6.4 GB/s (cr-read, tR=9.7 µs). Energy drops 72% (18.278 → 5.098 pJ/bit).

3 · Bias-error encoding (be-enc) — 87.5% fewer LSB errors

TLC pages (LSB, CSB, MSB) use different sensing operations to resolve 8 VTH states. Conventional (2,3,2) coding: LSB needs 2 sensings (V1, V5), CSB needs 3 (V2, V4, V6), MSB needs 2 (V3, V7). Be-enc reconfigures to (1,3,3): LSB needs only 1 sensing (V4 — middle states P3/P4), CSB/MSB need 3 each. This makes LSB as fast as SLC (single sensing) and 87.5% more reliable (only middle states, widest margin).

Why it works: LSB in (1,3,3) only distinguishes P3 (110) vs P4 (100) with V4. These states are in the most stable voltage range — fewest VTH shifts from retention/wear. CSB/MSB still cover the full range, so their error rates rise slightly, but existing off-chip LDPC at the controller has a large ECC margin (Figure 11b: 4K P/E cycles + 1 year retention needs only 17 errors/1 KiB, LDPC hard-decision corrects 30).

Tradeoff: be-enc is applied per-block. Blocks with LLM parameters (IFP blocks) use (1,3,3) — LSB pages get fast reads, CSB/MSB store general data with slightly slower reads. Non-IFP blocks keep (2,3,2). The overhead is small: sequential reads unaffected (external bandwidth bottleneck); random reads show 6.8% IOPS drop, 9.3% latency increase (Figure 18) when all blocks are IFP. Tolerable for the reliability gain.

4 · ECCLITE — compact on-chip BCH decoder

LLMs are error-sensitive: RBER 10⁻⁷ drops LLaMA-3 8B accuracy by 60% (Figure 4b). So on-chip ECC is mandatory. But a full LDPC decoder for 6.4 GB/s throughput would consume 10.694 W and 40.12 mm² across 16 chips (Figure 5) — exceeds the SSD power budget (6–8 W).

Be-enc makes LSB pages 87.5% more reliable (9 errors/1 KiB vs 48 errors for CSB, at 4K P/E + 1 year retention). ECCLITE is a BCH decoder that corrects 10-bit errors per 1 KiB, tuned to LSB pages only. BCH is simpler than LDPC (no iterative decoding), so 0.167 mm² and 45.1 mW per chip → 2.672 mm² and 0.722 W total for 16 chips (Figure 5 shows ECCLITE at 6.4 GB/s is ~15× smaller/lower-power than LDPC at same throughput).

💡
Why LSB-only ECC is safe: CSB/MSB pages (general data) cross the channel to the controller, where the existing off-chip LDPC decoder corrects them. Only LSB pages (LLM parameters) need on-chip correction for IFP. The ECC margin at the controller is so large (Figure 11b) that the slight CSB/MSB error increase from be-enc is absorbed — no new off-chip decoder needed.

5 · On-chip GEMV and computation flow

Each AiFChip integrates product elements (PEs): INT8 multipliers and an adder tree. The SSD controller partitions the model matrix into equal-sized submatrices, stores them sequentially in IFP blocks across chips (one submatrix per chip). During inference, the host sends an aif_gemv command with: input vector (8–32 KiB), matrix LBA range, output vector address.

  1. Controller sends GEMV command to all chips, along with the input vector. Each chip stores the vector in its page buffer (PB) — reusing the existing cache/data latches (TLC needs only 1 latch for LSB reads, leaving 2 free for the vector).
  2. Each chip reads its submatrix sequentially from LSB pages (cr-read applies). Matrix segments pass through ECCLITE (10-bit BCH correction), then to PEs for multiply-accumulate.
  3. PEs compute sub-vector (e.g., chip 0 computes rows 0–499 of the output). Results accumulate in an output FIFO.
  4. Controller polls chips in round-robin, retrieves FIFO contents, aggregates sub-vectors into the final output vector, sends to host memory via PCIe.

Parallelism: all 16 chips execute steps 1–3 simultaneously (102.4 GB/s aggregate). The controller aggregates in step 4, which is fast since each sub-vector is tiny (e.g., 1 KiB). Total overhead: 0.209 mm² per chip (0.167 mm² ECCLITE + 0.026 mm² PEs + 0.016 mm² others), 51.68 mW (45.1 mW ECC + 3.98 mW PEs + 2.6 mW others).

6 · Parallel host-AiFSSD execution

A decoder block has QKV generation (GEMV), MHA (memory KV cache), FFN (GEMV). Baseline runs them sequentially — AiFSSD idle during MHA. AiF overlaps via head-level and tensor-level parallelism:

  • Head-level: as soon as AiFSSD finishes Q,K,V for head 0, host starts MHA on head 0 while AiFSSD generates Q,K,V for head 1. Independent attention heads enable pipelining.
  • Tensor-level: split FFN projection matrix into host-submatrix (loaded in memory during prefill) and AiFSSD-submatrix. Host and AiFSSD compute their pieces in parallel, then aggregate. Degree limited by host memory (8 GB → ~2-way split for large models).

Result: despite 8 GB memory vs in-memory's 128 GB, AiF achieves 1.4× throughput (Figure 16) — not by raw bandwidth, but by utilization. The host's 86.5 GB/s DRAM bandwidth + AiFSSD's 102.4 GB/s internal bandwidth work concurrently, not sequentially.

Follow the numbers, step by step

Worked examples

Four calculations carried all the way through — from bandwidth limits to error budgets to speedup arithmetic.

① Token rate ceiling from bandwidth

A 40B model (40 GB INT8) runs on an SSD with 8 GB/s external bandwidth. What's the maximum token generation rate?

  1. Each token requires reading the entire model (autoregressive decode). 40 GB per token.
  2. Read time per token = 40 GB ÷ 8 GB/s = 5 seconds.
  3. Token rate = 1 token / 5 s = 0.2 tokens/s.
  4. Real-time chat threshold is 3 tokens/s (Section 1) → 15× too slow. Even 8 GB/s is insufficient.
  5. With AiF (102.4 GB/s): 40 GB ÷ 102.4 GB/s = 0.39 s/token → 2.56 tokens/s. Close to 3, and overlapped execution (parallel host/SSD) pushes above 3 for 20–40B models (Figure 16: 2.7–5.74 tokens/s).
🍯
Insight: the equation max tokens/s ≤ bandwidth / model_size is a hard limit in the decode phase. Baseline SSD (8 GB/s) can't sustain 3 tokens/s for models > 2.7 GB. AiF's 102.4 GB/s supports 3 tokens/s for models up to 34 GB (and parallel execution extends this further).

② Cr-read latency improvement

Conventional TLC LSB read (after be-enc, so single sensing) takes tR = 28 µs (tPRE=14 µs, tEVAL=9 µs, tDISCH=5 µs). Cr-read replaces precharge/discharge with tRECY=6.04 µs. What's the new tR and bandwidth?

  1. Conventional tR = tPRE + tEVAL + tDISCH = 14 + 9 + 5 = 28 µs.
  2. Cr-read tR (after first WL) = tRECY + tEVAL = 6.04 + 9 = 15.04 µs — but the paper reports 9.7 µs (presumably optimized tRECY further, or overlapped eval). Use 9.7 µs.
  3. Bandwidth per chip (page size 16 KiB): conventional = 16 KiB / 28 µs = 0.57 GB/s; cr-read = 16 KiB / 9.7 µs = 1.65 GB/s — but the paper says 6.4 GB/s. The discrepancy is from plane-level parallelism: each chip has 4 planes, all reading in parallel. So 1.65 GB/s × 4 planes ≈ 6.6 GB/s (rounds to 6.4 GB/s in paper).
  4. Speedup = 9.7 µs vs 28 µs → 2.89× faster (64% reduction matches (28 - 9.7)/28 = 65%).
  5. 16 chips total: 6.4 GB/s × 16 = 102.4 GB/s aggregate internal bandwidth.
💡
The key is that cr-read only works for successive WL reads in the same block. Since AiF stores each model matrix sequentially in a block, all parameter reads during inference benefit. Random reads (e.g., general data) still need conventional precharge/discharge.

③ On-chip ECC error budget

At 4K P/E cycles + 1 year retention, LSB pages (1,3,3 coding) have 9 errors per 1 KiB. ECCLITE is a BCH decoder that corrects 10-bit errors per 1 KiB. Is the margin safe?

  1. Raw errors (RBER from characterization, Figure 13b): 9 errors / 1 KiB for LSB pages under worst-case conditions.
  2. ECCLITE capability: 10-bit correction / 1 KiB (BCH decoder).
  3. Margin = 10 - 9 = 1-bit headroom. Tight, but safe — worst-case is rare (median P/E cycles in consumer SSDs ≈ 500–1000, not 4K; retention ≪ 1 year for active data).
  4. Compare to CSB/MSB (general data): 66–69 errors/1 KiB → needs 70-bit correction. Off-chip LDPC soft-decision corrects 40/KiB (hard 30/KiB), so CSB/MSB need LDPC, not ECCLITE. But LDPC is at the controller, not on-chip, so no area/power burden on the chip.
  5. Why 10-bit BCH is sufficient: be-enc makes LSB pages 87.5% more reliable than conventional (2,3,2) LSB (49 → 9 errors). That's what allows the compact BCH instead of LDPC.
🍯
Implementation detail: ECCLITE is 0.167 mm² (Table 1), vs 2.5 mm² for a 50-bit LDPC at same throughput (inferred from Figure 5 scaling). That's 15× smaller. Power: 45.1 mW vs ~670 mW (15× lower). The reliability boost from be-enc is what makes this possible.

④ Throughput speedup: baseline vs AiF

GPT-NeoX-20B (20.4 GiB weights) runs on baseline SSD offloading (8 GB/s) vs AiF (102.4 GB/s + parallel execution). What's the speedup?

  1. Baseline (Memory+SSD): 8 GB memory, offload rest to SSD. Read time per token ≈ 20.4 GiB ÷ 8 GB/s = 2.55 s/token → 0.39 tokens/s (ignoring host compute, which is small). Actually Figure 16e shows 0.23 tokens/s — includes I/O overhead (PCIe, NVMe protocol).
  2. AiF (no parallel): 102.4 GB/s → 20.4 GiB ÷ 102.4 GB/s = 0.2 s/token → 5 tokens/s (theoretical). But Figure 16e shows 1.7 tokens/s for AiF−− (25.6 GB/s baseline IFP) and 5.7 tokens/s for AiF. The difference is cr-read+be-enc (102.4 GB/s) vs plain IFP (25.6 GB/s).
  3. AiF (with parallel): head-level and tensor-level parallelism overlap host MHA and AiFSSD GEMV. Figure 16e shows 5.7 tokens/s for AiF vs 0.23 for baseline.
  4. Speedup = 5.7 / 0.23 = 24.8×. Paper reports 14.6× average across all models (Figure 16) — some models benefit less (e.g., LLaMA-3 70B is 1.6/0.06 = 26.7×, but smaller models already fit in memory so baseline does better).
  5. vs In-Memory (Figure 16e): in-memory = 3.4 tokens/s (host CPU, 86.5 GB/s DRAM). AiF = 5.7 tokens/s → 1.68× faster. Average across all models is 1.4× (some models like LLaMA-3 8B see less benefit because tensor-level parallelism is limited by 8 GB host memory).
💡
The baseline speedup varies by model size because: (1) small models fit in memory → baseline ≈ in-memory, AiF still faster due to parallel execution; (2) large models bottlenecked by SSD → AiF wins big; (3) MoE models (Mixtral) read only 14.7 GiB per token despite 46.2 GiB total → effective bandwidth requirement is lower, so AiF vs baseline gap is smaller (but AiF still 18.8× faster, Figure 16g).

Practice problems

Try each before opening the answer. Every one ends with the common misconception it checks.

Problem 1 · easy
An SSD has 16 flash chips, each reading at 1.6 GB/s internally, connected via 8 channels at 2.0 GB/s each. What's the internal vs external bandwidth, and which is the bottleneck for baseline ISP?
Show answer
Internal bandwidth = 16 chips × 1.6 GB/s = 25.6 GB/s. External (channel) bandwidth = 8 channels × 2.0 GB/s = 16 GB/s. But channels are multi-drop: only one chip per channel transfers at a time → effective external = 8 × 1.6 GB/s = 12.8 GB/s. Baseline ISP moves data over channels, so it's channel-bound (12.8 GB/s), not internal-bound. IFP computes in-chip, sending only small results over channels → leverages the 25.6 GB/s internal bandwidth.Misconception: that "8 channels × 2.0 GB/s" gives the full external bandwidth. The multi-drop topology limits each channel to one chip's speed (1.6 GB/s), and only 8 chips can transfer simultaneously (one per channel).
Problem 2 · medium
Cr-read works by recycling voltages between successive WL reads in the same block. If a 175 MiB matrix is split across 16 chips and each chip stores its 11 MiB submatrix sequentially in one block, what fraction of reads benefit from cr-read during inference?
Show answer
Each submatrix is stored sequentially in one block (AiF's layout optimization). Reading the submatrix involves reading all its WLs in order. The first WL read needs conventional precharge; all subsequent WLs in that block use cr-read. For a large submatrix (11 MiB ≈ 704 WLs at 16 KiB/page), fraction benefiting ≈ (704 - 1) / 704 = 99.86%. So nearly all reads use cr-read, achieving 6.4 GB/s per chip instead of 2.3 GB/s.Misconception: that cr-read applies to all reads. It only works for successive WLs in the same block. Random reads (e.g., CSB/MSB general data, or accessing different blocks) still need precharge/discharge. But LLM parameters are large and stored sequentially, so they're ideal for cr-read.
Problem 3 · medium
Be-enc makes LSB pages 87.5% more reliable by switching from (2,3,2) to (1,3,3) coding. Explain why LSB reliability improves while CSB/MSB reliability degrades, using VTH state encoding.
Show answer
In (2,3,2), LSB distinguishes ER/P7 from P1–P6 (sensing V1) and P1/P2/P3 from P4/P5/P6 (sensing V5). These span the full VTH range → many states, narrow margins, high error rates. In (1,3,3), LSB only distinguishes P3 from P4 (sensing V4), the middle states with the widest margin (most stable voltage range). So LSB errors drop from 49 to 9 per 1 KiB (87.5%). CSB/MSB now cover the full range (edge states), so their errors rise from 48 to 66/69 (Figure 13). The key: LSB (1,3,3) avoids edge states; CSB/MSB absorb them.Misconception: that (1,3,3) coding is "better" overall. It's a trade — LSB becomes SLC-like at the cost of CSB/MSB becoming worse. AiF exploits this by storing LLM parameters (error-sensitive) on LSB and general data (ECC-tolerant) on CSB/MSB.
Problem 4 · hard
AiF's on-chip ECCLITE corrects 10-bit errors per 1 KiB. Figure 4b shows LLaMA-3 8B drops 60% accuracy at RBER 10⁻⁷. Convert 9 errors/1 KiB to RBER and verify ECCLITE is sufficient.
Show answer
1 KiB = 1024 bytes = 8192 bits. 9 errors / 8192 bits = RBER ≈ 1.1 × 10⁻³. That's far above 10⁻⁷, so without ECC, inference would fail. ECCLITE corrects 10-bit errors → residual errors after ECC = 9 - 10 = 0 (when ≤10 errors occur). If 11 errors occur (rare, worst-case), 1 bit remains uncorrected → RBER ≈ 1/8192 = 1.2 × 10⁻⁴. Still above 10⁻⁷, but: (1) 11 errors/KiB is tail-case (median is lower); (2) bitflips in certain positions (e.g., low-order weight bits) have minimal impact on accuracy. The paper's characterization (Figure 13b) shows 9 errors is the max at 4K P/E + 1 year — typical retention/cycles are far lower, so residual RBER after ECCLITE is likely < 10⁻⁷.Misconception: confusing raw RBER with post-ECC RBER. The 10⁻⁷ threshold in Figure 4b is for uncorrected errors. ECCLITE reduces raw 10⁻³ to effective < 10⁻⁷ by correcting up to 10 bits/KiB, which covers the 9-error worst-case with margin.
Problem 5 · medium
AiF achieves 1.4× throughput vs in-memory inference despite 8 GB memory vs 128 GB. How does parallel host-AiFSSD execution overcome the memory capacity gap?
Show answer
Key: AiF overlaps host and AiFSSD compute. In-memory runs MHA and FFN sequentially on the host. AiF runs MHA on host (accessing 8 GB KV cache) while AiFSSD runs GEMV for the next head's QKV or FFN (accessing 102.4 GB/s flash). Total effective bandwidth ≈ host DRAM (86.5 GB/s) + AiFSSD internal (102.4 GB/s) = 188.9 GB/s when overlapped. But overlap isn't perfect — head-level parallelism is limited by number of heads (e.g., 32 heads → max 32 overlaps), and tensor-level parallelism is limited by 8 GB host memory (can only load ~2 submatrices for large models). So observed speedup is 1.4× (average, Figure 16), not 188.9/86.5 = 2.2×. Smaller models benefit more (more tensor parallelism fits in 8 GB); larger models less (tensor parallelism ≈ 1, only head parallelism).Misconception: that AiF "has more bandwidth" than in-memory. Raw bandwidth: in-memory's DRAM is 86.5 GB/s (all for model weights); AiF's internal is 102.4 GB/s but host memory is only 8 GB (so most weights are on flash). The speedup comes from concurrent use of host memory + flash bandwidth, not raw bandwidth superiority.
Problem 6 · medium
Why does AiF store LLM parameters only on LSB pages, leaving CSB/MSB for general data? What breaks if you used all three pages for parameters?
Show answer
Be-enc makes LSB pages fast (1 sensing, tR=28 µs → 9.7 µs with cr-read) and reliable (9 errors/KiB), enabling compact on-chip ECCLITE (10-bit BCH). CSB/MSB pages are slower (3 sensings, tR=46 µs) and less reliable (66–69 errors/KiB), requiring off-chip LDPC. If you stored parameters on CSB/MSB too: (1) slower reads → lower bandwidth (e.g., 46 µs vs 9.7 µs → 2.1 GB/s vs 6.4 GB/s per chip); (2) on-chip ECC would need 70-bit correction → LDPC, not BCH → 15× more area/power (defeats the point of compact on-chip ECC). So LSB-only is optimal: fast + reliable → compact decoder. CSB/MSB stay available for general data (slight latency increase, but off-chip LDPC absorbs the error rate).Misconception: that TLC pages are "wasted" if only LSB is used for parameters. CSB/MSB still store general data (e.g., OS files, KV cache if swapped). The trade is intentional: sacrifice CSB/MSB speed/reliability to make LSB SLC-like for the high-bandwidth, error-sensitive workload (LLM inference).
Problem 7 · hard
Figure 16 shows AiF−− (plain IFP, 25.6 GB/s) achieves 1.7 tokens/s for GPT-NeoX-20B, while AiF (102.4 GB/s with cr-read+be-enc) achieves 5.7 tokens/s. But 102.4/25.6 = 4× bandwidth → why only 5.7/1.7 = 3.35× speedup?
Show answer
Amdahl's Law: LLM inference has (1) GEMV (bandwidth-bound, benefits from 4× bandwidth) and (2) vector ops (host-bound, e.g., MHA, activations, layer norm). Even with infinite bandwidth, vector ops remain. From Figure 7, a typical decoder block is ~60% GEMV (offloaded to AiF) and ~40% vector ops (host). So speedup ≈ 1 / (0.4 + 0.6/4) = 1 / 0.55 = 1.82× — but parallel execution (head/tensor parallelism) overlaps host and AiFSSD, boosting further. Observed 3.35× is between 1.82× (no overlap) and 4× (perfect overlap). The gap is (1) imperfect overlap (finite heads, limited tensor parallelism), (2) control overhead (NVMe protocol, PCIe DMA), (3) some operations can't overlap (e.g., final aggregation). Smaller models see less speedup (already fast), larger models closer to 4× (more GEMV-dominant).Misconception: that 4× bandwidth → 4× end-to-end speedup. The non-accelerated portions (host vector ops) limit the speedup. AiF's parallel execution mitigates this by overlapping, but doesn't eliminate it.
Problem 8 · easy
AiF uses dynamic set sampling (DSS) to keep overhead low. But the paper doesn't detail a sampling-based UMON like UCP. Why not, and what does AiF sample instead?
Show answer
AiF doesn't need utility monitors (UMONs) because it's not partitioning cache dynamically. UCP samples sets to estimate utility curves; AiF statically pre-allocates IFP blocks when posting the model (Section 5.2). The "sampling" in AiF's context is minimal: (1) monitor which blocks are IFP vs non-IFP (a per-block bitmap); (2) apply be-enc only to IFP blocks. No per-set tracking needed — the LLM parameter layout is known at model load time. The only runtime "sampling" is error detection in ECCLITE (BCH decodes the received codeword), but that's not set-sampling — it's syndrome calculation on every read. So AiF's overhead is per-chip (ECCLITE + PEs = 0.209 mm²), not per-set.Misconception: that all IFP solutions need sampling. Sampling is useful when you're learning a dynamic property (e.g., utility curves in UCP). AiF's property (which blocks have LLM parameters) is static once the model is posted, so no sampling needed.
Quick answers

Frequently asked questions

Ten questions readers ask most. Tap any to expand.

Q1What's the key difference between ISP and IFP?▾
ISP (in-storage processing) integrates accelerators at the SSD controller — data from flash chips crosses the flash channels to reach the controller, so ISP is still channel-bound (12.8–19.2 GB/s for 8-channel SSDs). IFP (in-flash processing) integrates accelerators inside each flash chip — data never leaves the chip, leveraging the higher internal bandwidth (25.6–102.4 GB/s aggregate). AiF is IFP: GEMV happens in-chip, only the tiny output vector crosses the channel.
Q2Why is 3 tokens/s the threshold for usable LLM chat?▾
Section 1 cites reading speed studies: humans read ~3–4 words/s comfortably when skimming. LLMs generate ~0.75 words/token on average (for English), so 3 tokens/s ≈ 2.25 words/s — close to natural reading pace. Below this, the output feels sluggish and disrupts conversation flow. Above 3 tokens/s, most users perceive the response as instantaneous. (The paper cites Brysbaert 2019 and reading-rate meta-analyses.)
Q3Does cr-read work on all flash types (SLC/MLC/TLC)?▾
Yes, in principle. Cr-read exploits voltage recycling between successive WL reads in the same block — that's a physical property of NAND flash, not specific to multi-level encoding. But the benefit varies: SLC already has single sensing (low tR), so cr-read saves less (e.g., 37 µs → 15 µs, 59% reduction). TLC with be-enc (LSB pages, single sensing after recoding) achieves 28 µs → 9.7 µs (65% reduction). MLC would fall in between. The energy savings (72%) apply to all types since precharge/discharge dominate energy regardless of sensing count.
Q4What happens to CSB/MSB pages in IFP blocks after be-enc is applied?▾
They store general data (e.g., OS files, KV cache, non-LLM app data). Be-enc makes CSB/MSB slightly slower (46 µs vs 37/46 µs in conventional 2,3,2) and less reliable (66–69 errors/KiB vs 48). But: (1) sequential reads are still external-bandwidth-limited (no latency impact at the user level, Figure 18a); (2) random reads show 6.8% IOPS drop, 9.3% latency increase (Figure 18b/c) — tolerable for non-critical data; (3) off-chip LDPC at the controller has a huge ECC margin (Figure 11b), so the error rate increase is absorbed. The trade is intentional: sacrifice CSB/MSB to make LSB ideal for LLM parameters.
Q5How does AiF handle model updates (e.g., fine-tuning)?▾
Model updates require rewriting the updated matrices to IFP blocks (LSB pages, sequential layout). The aif_post command re-partitions the matrix and stores it across chips. Since flash is write-once (program/erase granularity is block-level), updating a small portion of the model still requires erasing and rewriting the entire IFP block. For frequent updates (e.g., online fine-tuning), this could increase write amplification and wear. The paper doesn't address this — AiF targets inference-only workloads where parameters are static. Future work could explore incremental updates or hybrid schemes (cache small deltas in memory, batch-write to flash).
Q6Why doesn't AiF just use more flash channels to increase external bandwidth?▾
Flash channels are pin-limited — adding more channels requires more physical pins on the SSD package, which increases cost and power (each channel needs drivers, receivers, termination). Consumer SSDs are constrained by form factor (M.2, U.2) and power budget (6–8 W). Enterprise SSDs can have 16+ channels, but then cost scales non-linearly. IFP bypasses this by leveraging the existing internal bandwidth without adding pins — you get the benefit of 16 chips reading in parallel (102.4 GB/s) while sending only small results over the existing 8 channels. It's a "bandwidth multiplier" without hardware changes to the package.
Q7What's the area overhead of AiF per SSD (not per chip)?▾
Per chip: 0.209 mm² (Table 1). 16 chips total: 0.209 × 16 = 3.344 mm². A typical flash chip die is ~100–150 mm² (depending on density), so 3.344 mm² across 16 chips is ~0.2% of total die area. The SSD controller is unchanged (no new ECC for LSB pages, since ECCLITE is on-chip). So total SSD area overhead is <1% (matching the paper's claim). Power: 51.68 mW/chip × 16 = 0.827 W — well within the 6–8 W SSD budget.
Q8Can AiF accelerate other workloads besides LLM inference (e.g., training, or other ML models)?▾
Any workload with (1) large GEMV operations, (2) sequential parameter access, (3) high data reduction (big matrix → small vector), and (4) error sensitivity (so ECCLITE is valuable) could benefit. Examples: (1) Transformer-based models beyond LLMs (e.g., vision transformers, speech models); (2) recommendation models with large embedding tables (GEMV to fetch embeddings); (3) graph neural networks (sparse GEMV if adjacency matrices are stored sequentially). Training is trickier because gradients flow backward (requires transpose operations) and weights update frequently (write amplification). AiF's current design optimizes read-only GEMV — extending to training would need bidirectional PEs (for transposed GEMV) and wear-aware update schemes.
Q9How does AiF compare to CXL-attached memory for on-device LLMs?▾
CXL.mem expands DRAM capacity over CXL (e.g., 128 GB CXL-attached DRAM). Bandwidth: CXL 2.0 is ~64 GB/s (8 lanes PCIe 5.0); CXL 3.0 is ~128 GB/s. AiF's 102.4 GB/s is competitive with CXL 2.0, and AiF's parallel execution can push effective bandwidth higher (overlapping host DRAM + AiF flash). Cost: CXL DRAM is expensive (DRAM pricing + CXL controller); AiF uses cheap TLC flash (50× higher bit density). Latency: CXL is lower latency (~200 ns) vs AiF (~10 µs page read); but LLM decode is throughput-bound, not latency-bound (large batches of sequential reads). So AiF trades latency for cost/capacity. For on-device (cost-sensitive), AiF wins; for datacenter (latency-sensitive), CXL may be better. Hybrid: CXL for KV cache (small, latency-critical) + AiF for model weights (large, throughput-bound) could be optimal.
Q10What did the authors leave for future work?▾
The paper flags: (1) extending AiF to support LLM training (bidirectional GEMV for backprop, weight updates with wear-leveling); (2) scaling to multi-chip packages (stacking AiFChips vertically for higher density); (3) integrating AiF with heterogeneous memory systems (CXL + AiF hybrid); (4) supporting dynamic quantization (switching between INT8/INT4 on the fly); (5) handling multi-tenant inference (partitioning AiFChips across multiple models/users). Also: (6) optimizing for non-convex utility curves (like UCP's Lookahead, but for IFP — different models have different access patterns, so dynamic allocation could help).

Glossary & further reading

Every term in one place, then a path to go deeper.

Core terms
AiF (Accelerator-in-Flash)The paper's IFP solution: GEMV accelerators inside flash chips, leveraging 102.4 GB/s internal bandwidth via cr-read + be-enc.
IFP (In-Flash Processing)Computing inside flash chips (vs ISP at the SSD controller). Bypasses channel bottleneck by sending only small results externally.
ISP (In-Storage Processing)Computing at the SSD controller. Still channel-bound (12.8–19.2 GB/s) because data must cross flash channels from chips to controller.
Cr-read (Charge-Recycling Read)AiF's technique to skip precharge/discharge between successive WL reads by reusing voltages. 64% faster tR, 72% energy cut.
Be-enc (Bias-Error Encoding)Reconfiguring TLC from (2,3,2) to (1,3,3) coding — LSB pages become SLC-like (1 sensing, 87.5% fewer errors), CSB/MSB slow down.
ECCLITEAiF's compact on-chip BCH ECC decoder (10-bit correction per 1 KiB) for LSB pages. 0.167 mm², 45.1 mW — 15× smaller/lower-power than LDPC.
GEMV (Matrix-Vector Multiplication)The dominant operation in LLM decode: multiply a large weight matrix by a small token vector. High data reduction (175 MiB → 16 KiB) ideal for IFP.
Channel bottleneckFlash channels (multi-drop bus) limit external bandwidth to 12.8 GB/s (8 channels × 1.6 GB/s) despite 25.6 GB/s internal bandwidth (16 chips × 1.6 GB/s).
Supporting terms
LSB / CSB / MSB pagesThree pages per TLC wordline (storing 3 bits/cell via 8 VTH states). LSB = least significant bit, CSB = center, MSB = most significant.
VTH (Threshold Voltage)The voltage level stored in a flash cell's charge trap layer. Different VTH levels encode different bit values (e.g., 8 states for TLC).
Sensing operationComparing a cell's VTH against a reference voltage (VREF) to determine its stored bit. More sensings = longer tR. Be-enc reduces LSB to 1 sensing.
tR (Read Latency)Time to read one page from flash. Conventional TLC LSB (2,3,2) = 37 µs; be-enc LSB (1,3,3) = 28 µs; cr-read LSB = 9.7 µs.
RBER (Raw Bit Error Rate)Fraction of bits read incorrectly before ECC. Flash RBER > 10⁻³; LLMs need < 10⁻⁷. ECC bridges the gap.
BCH / LDPCError correction codes. BCH is simpler (low power/area) but weaker (10–50 bit correction). LDPC is powerful (soft-decision 100+ bits) but complex.
P/E cyclesProgram/Erase cycles — number of times a block has been written/erased. Flash wears out (higher errors) with more P/E cycles. Typical consumer SSD: 500–1K cycles.
Head-level parallelismLLM attention has multiple heads computed independently. AiF starts MHA on head 0 while generating Q,K,V for head 1 → overlap host + SSD.
Tensor-level parallelismSplitting a projection matrix into host-submatrix (in memory) + AiFSSD-submatrix (in flash). Host and AiFSSD compute in parallel, then aggregate.
Autoregressive decodeLLM generates one token at a time, feeding output back as input for the next token. Requires reading the full model per token → bandwidth-bound.
IFP blockA flash block where LLM parameters are stored (LSB pages only, with be-enc applied). Sequential layout enables cr-read for all parameter reads.
Page buffer (PB)Temporary storage in each flash plane (multiple page latches). AiF reuses PBs to hold input vectors (freeing 2 latches since LSB needs only 1 for sensing).

ISP for ML inference at the SSD controller. Limited by channel bandwidth (12.8 GB/s). AiF's IFP bypasses this by computing in-chip (102.4 GB/s internal).
