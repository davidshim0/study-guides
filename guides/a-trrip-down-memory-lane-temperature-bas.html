Keep the hottest code in cache — let the cold code go

Profile-guided optimization already tells the compiler which code runs most. TRRIP uses that information at runtime to prioritize hot instruction cache lines, cutting L2 misses 26.5% and speeding up simulated proxy benchmarks 3.9% geomean — for free.

Before you begin

Six ideas the paper assumes you know

Each card clears up one concept TRRIP builds on. Read in order the first time.

hot code (most executed) warm code cold code (rarely executed) hardware mechanism
1CPU frontend bottleneck — instruction cache misses stall everything▾

Modern CPUs are pipelined: fetch instructions, decode them, execute, retire. A miss in the instruction cache stalls the frontend (fetch/decode), starving the rest of the pipeline. Backend stages sit idle, wasting cycles. Mobile system software (interpreters, UI frameworks, graphics libs) shows 20%+ frontend stalls even after compiler optimization.

Why? Complex layered code, deep call stacks, huge instruction footprints — up to 2.7× growth per year. Hardware can't keep up.

Now you're ready: TRRIP targets instruction cache misses that cause frontend stalls, especially in the unified L2 where miss penalties are high (hundreds of cycles to DRAM).
2PGO — profile-guided optimization tells the compiler what's hot▾

Profile-Guided Optimization (PGO): compile the app, run it with representative workloads, count how often each basic block executes, then re-compile using those counts. The compiler knows which functions/loops run most (hot), which rarely run (cold), and can optimize code layout accordingly — placing hot code together to improve spatial locality.

PGO already reduces frontend stalls significantly. But the paper shows it's not enough: hot code has a substantial tail in reuse distance, with many accesses occurring after 9+ intervening cache lines.

Now you're ready: TRRIP leverages PGO's temperature classification (hot/warm/cold) but extends it to runtime cache replacement, not just compile-time layout.
3RRIP — re-reference interval prediction (the baseline replacement policy)▾

RRIP (Re-Reference Interval Prediction) is a cache replacement policy that assumes new cache lines will be re-accessed distantly in the future — pessimistic insertion. It inserts lines at low priority (RRPV=2 in a 2-bit scheme). Only hits upgrade a line to high priority (RRPV=0). This prevents one-time streams from polluting the cache.

RRIP outperforms LRU on modern workloads. But it's purely hardware, blind to which program code is hot or cold — it only sees recency.

Now you're ready: TRRIP extends RRIP with software hints (code temperature) so hot instruction lines get priority from the start, not just after a hit.
4Reuse distance — why hot code still gets evicted▾

Reuse distance = the number of unique cache lines accessed between two uses of the same line. Short reuse distance = good temporal locality. Long reuse distance = the line gets evicted before its next use.

Paper's key finding (Fig. 3): about 60% of hot code accesses have reuse distance 0–4, but a substantial tail extends to 9+ cache ways — far beyond an 8-way L2. Cause: intervening accesses to warm/cold code and data lines evict hot code prematurely.

Now you're ready: TRRIP prioritizes hot lines to keep them in cache longer despite high reuse distance, preventing evictions by less-important code.
5Page table entries (PTEs) — how software talks to hardware transparently▾

The OS maps virtual addresses to physical addresses via page tables. Each PTE (Page Table Entry) stores the physical address plus metadata: permissions (read/write), caching attributes, etc. Commercial ARM CPUs already reserve implementation-defined bits in PTEs (e.g., PBHA — Page-Based Hardware Attributes) for custom use.

TRRIP reuses at most two of these existing implementation-defined bits to store code temperature. No ISA changes, no new instructions — just populate the existing PTE bits at load time.

Now you're ready: TRRIP's software-to-hardware interface is via PTEs. The MMU reads the temperature bits and passes them with memory requests to the cache.
6ELF sections — how the compiler organizes code by temperature▾

The ELF (Executable and Linkable Format) binary has sections: .text for code, .data for initialized data, etc. TRRIP's PGO-enabled compiler creates three code sections: .text.hot, .text.warm, .text.cold. Each section is contiguous in the binary and mapped to pages at load time.

The loader reads the ELF headers, sees which section is hot, and marks the corresponding pages' PTEs with the hot temperature bit. Simple.

Now you're ready: TRRIP's compiler changes are minimal — just partition code into temperature-labeled sections. The loader does the rest.

The whole story in one place

TRRIP in a nutshell

The problem, the co-design fix, and the payoff.

Modern mobile apps are layered, deep-stacked, and instruction-hungry. Even after PGO optimizes code layout, the CPU frontend still stalls 20%+ on instruction cache misses. Why? High reuse distance: about 60% of hot code accesses have reuse distance 0–4, but a substantial tail extends to 9+ unique cache lines — far beyond an 8-way L2's capacity. Warm and cold code, plus data accesses, evict hot lines before their next use.

Pure hardware solutions (like RRIP) are blind to code temperature. They only see recency and can't distinguish hot from cold. Adding hardware predictors (SHiP, Emissary) or bigger tables costs power/area — unsustainable as code grows 2.7× per year.

TRRIP's fix: leverage PGO's existing temperature classification. The compiler places hot/warm/cold code in separate ELF sections. The OS loader marks pages with temperature bits (at most 2 bits in existing PTEs, e.g., ARM PBHA). The MMU passes those bits with memory requests. The cache replacement policy (a lightweight RRIP extension) inserts hot lines at Immediate priority (RRPV=0) instead of Intermediate (RRPV=2), keeping them in cache longer. Warm lines get Near priority (RRPV=1); cold lines follow default RRIP.

The payoff: On 10 mobile-proxy benchmarks already PGO'd and evaluated in the Sniper simulator, TRRIP cuts L2 instruction MPKI by 26.5% (TRRIP-1, hot-only) to 27.3% (TRRIP-2, hot+warm+cold), yielding 3.9% geomean speedup over RRIP. Hardware overhead: ~0% area and power (reuses existing PTE bits). No ISA changes. No new instructions. Easy to toggle on/off.

🔥 Key takeaways

  • PGO alone can't fix the substantial tail of high reuse distance in hot code — it needs runtime help.
  • TRRIP extends RRIP with software hints (code temperature via PTEs) to prioritize hot instruction lines.
  • Compiler splits code into .text.hot/.warm/.cold sections; loader marks PTEs; MMU passes bits to cache.
  • Hot lines inserted at RRPV=0 (immediate), warm at RRPV=1, cold at default RRPV=2.
  • Result: 26.5% L2 I-MPKI reduction, 3.9% speedup in simulation, negligible overhead.
  • TRRIP-1 (hot-only) is simpler; TRRIP-2 (hot+warm+cold) marginally better.
  • TRRIP requires modified compiler/loader/OS behavior and a modified cache replacement policy. Evaluated in Sniper, not on commercial silicon.
Six everyday pictures

How to think about TRRIP

Each picture stands alone. Every card ends with where the picture breaks down.

The core idea · Hot vs cold priority

Emergency room triage

An ER prioritizes patients by severity, not arrival order. A heart attack goes straight to the front (hot code → RRPV=0). A sprained ankle waits (cold code → RRPV=2). Both get treated, but urgency dictates order. TRRIP does the same for cache lines: hot code gets immediate priority because frontend stalls on its misses are most costly.

Where it breaks: triage is per-patient; TRRIP is per-page (4kB/16kB), so a page spanning hot and warm code gets one temperature. The paper's listed mitigations are padding sections, disabling temperature marking on overlapping pages, or using smaller code pages.
Compiler output · ELF sections

Sorting laundry by color

You sort clothes (hot/warm/cold code) into separate bins before washing. The compiler does the same: after PGO, it places hot basic blocks in .text.hot, warm in .text.warm, cold in .text.cold. Each bin is contiguous in the binary. When the OS loads the binary, it sees the section labels and marks pages accordingly.

Where it breaks: laundry stays sorted; code sections can overlap pages if they're adjacent. TRRIP handles this by padding sections or letting such pages default to RRIP.
OS interface · Page table entries

Passport stamps

A passport has official stamps recording where you've been. A PTE is similar: it records the physical address of a page plus metadata (permissions, caching). TRRIP reuses at most 2 existing implementation-defined bits (ARM PBHA) to stamp the page as hot/warm/cold. No new fields — just repurpose spare bits already in commercial CPUs.

Where it breaks: stamps are permanent until you renew; PTE bits persist only while the page is mapped. TRRIP is fine with that — code sections are stable at runtime.
Cache replacement · RRIP extension

Priority boarding zones

Airlines board by zone: Zone 1 (first class) boards first, Zone 2 next, Zone 3 last. RRIP's RRPV is similar: 0=board first (keep in cache), 3=board last (evict soonest). TRRIP sets hot to RRPV=0 at insertion (Zone 1), warm to RRPV=1 (Zone 2), cold to RRPV=2 (Zone 3). Data lines follow default RRIP (Intermediate = RRPV=2).

Where it breaks: boarding is one-time; RRIP's RRPV can change on hits (upgrade to 0). TRRIP only sets insertion priority — hits follow RRIP's rules.
Result · Fewer evictions of hot code

Reserved parking for VIPs

A parking lot reserves front spots for VIPs (hot code), middle for regulars (warm), back for visitors (cold). When the lot fills, visitors are towed first (RRIP's eviction scans for RRPV=3). VIPs keep their spots longer. TRRIP does the same: hot lines start at RRPV=0, so they survive many eviction rounds before reaching RRPV=3.

Where it breaks: parking is per-lot; TRRIP is per-set, uniform across all sets. A program's hot code may map to different sets — TRRIP protects it everywhere, not just in one "lot."
Why co-design · Software knows what hardware can't see

Map vs compass

A compass (hardware) shows direction now but not the terrain ahead. A map (PGO) shows the whole route. TRRIP combines both: PGO's global view of execution (which code runs most over millions of cycles) guides hardware's local decisions (which line to evict). Pure hardware (RRIP, SHiP) only sees recent history; TRRIP sees the whole program.

Where it breaks: a map can be outdated; PGO profiles can differ from deployment workload. TRRIP relies on representative profiling — if the workload changes drastically, temperature hints may be stale.
The full explanation

How it actually works

Section by section, with the paper's key steps rebuilt as visuals.

1 · The problem — PGO isn't enough

Mobile system software (interpreters, UI libs, graphics) shows 20%+ frontend stalls even after PGO optimizes code layout (Fig. 1, 2). Why? High reuse distance: about 60% of hot code accesses have reuse distance 0–4, but a substantial tail extends to 9–16+ unique cache lines (Fig. 3). An 8-way L2 can't hold them all. Intervening warm/cold code and data evict hot lines before their next use.

🌱
Concretely: in sqlite, gcc, clang, about 60% of hot code accesses have reuse distance 0–4 (fits in 4-way cache), but a substantial tail extends to 9+ (evicted in 8-way cache). PGO improves spatial locality but can't fix temporal locality at runtime — that's a cache replacement job.

2 · The co-design interface — PTEs as the bridge

TRRIP needs to pass temperature from software (compiler knows hot/warm/cold) to hardware (cache replacement). Three requirements: (1) no ISA changes, (2) minimal hardware cost, (3) transparent to the program. Solution: reuse existing PTE bits.

Commercial ARM CPUs have PBHA (Page-Based Hardware Attributes) — 4 implementation-defined bits per PTE already transferred with memory requests. TRRIP uses at most 2 bits to encode temperature. The loader reads ELF section headers (which mark .text.hot, .text.warm, .text.cold) and sets the PTE bits accordingly. The MMU passes them with instruction fetch requests. No new storage in cache — temperature is ephemeral, only valid during the fetch.

3 · The cache replacement policy — TRRIP-1 and TRRIP-2

TRRIP extends RRIP with temperature-aware insertion and promotion. Two variants:

VariantHot linesWarm linesCold/data lines
TRRIP-1Insert at RRPV=0 (Immediate); hits → RRPV=0Default RRIP (RRPV=2)Default RRIP (RRPV=2)
TRRIP-2Insert at RRPV=0; hits → RRPV=0Insert at RRPV=1 (Near); hits → decrement (not jump to 0)Default RRIP (RRPV=2)

Key insight: hot lines get Immediate priority from the start (RRPV=0), bypassing RRIP's pessimistic Intermediate insertion (RRPV=2). This keeps hot lines in cache longer despite high reuse distance. Warm lines (TRRIP-2 only) get Near priority (RRPV=1), above data but below hot, and promote conservatively (decrement, don't jump) to avoid displacing hot lines.

# TRRIP-1 insertion (Algorithm 1, lines 15-18)
if temp == hot:
    LineRRPV = 0  # Immediate re-reference
else:
    LineRRPV = 2  # default RRIP Intermediate

# TRRIP-2 adds (lines 19-21)
elif temp == warm:
    LineRRPV = 1  # Near re-reference
🔥
Eviction unchanged: TRRIP doesn't modify RRIP's eviction logic. It only changes insertion priority. Eviction still scans for RRPV=3 (or increments all RRPVs until one reaches 3).

4 · Coverage and limitations — what TRRIP can't see

TRRIP only marks code in the compiled binary. It can't see: (1) code in external libraries (unless they're also PGO'd), (2) PLT (Procedure Linkage Table) stubs for dynamic linking, (3) JIT-generated code. Fig. 7a shows hot code coverage of costly instruction misses (misses that cause decode starvation + retire useful instructions): abseil, clang, gcc, python show 80–90% coverage; bullet, clamscan, omnetpp, rapidjson show 40–70%.

Why lower? Many costly misses land in PLTs or external libraries. Fig. 7b excludes external code → coverage jumps to 90%+. Pure hardware techniques (CLIP, Emissary) see all code, hence higher coverage on bullet/clamscan. But they require extra storage (Emissary: 2 bits per cache line; SHiP: 64kB predictor).

⚠
Tradeoff: TRRIP's near-zero overhead comes at the cost of code coverage. Compiling system libraries with TRRIP's PGO (common in mobile platforms) raises coverage to ~90%. For code beyond TRRIP's reach, default RRIP applies — no worse than baseline.

5 · Hardware cost — negligible

Table 4: TRRIP's overhead is ~0.0% static power and area (relative to SRRIP baseline). Why? It reuses existing PTE bits (ARM PBHA) and adds no on-chip storage. Comparison:

  • SHiP: 64kB predictor table → 1.7% power, 3.0% area.
  • Emissary: 2 bits per L1+L2 cache line → 0.5% power, 0.7% area.
  • CLIP: set-dueling (32 sampling sets) → ~0.0% overhead (same as TRRIP).

TRRIP matches CLIP's hardware simplicity but outperforms it by being selective (hot-only vs all-instructions).

Follow the numbers, step by step

Worked examples

Four traces from the paper's data, carried all the way through.

① Classifying code temperature in PGO

The compiler has profile data: total execution count Ctotal = 1,000,000. Percentilehot = 99% (default). Basic blocks sorted by count: BB1=500k, BB2=300k, BB3=150k, BB4=40k, BB5=9k, BB6=1k. Which BBs are hot?

  1. Calculate threshold: Cthreshold = Ctotal × 0.99 = 990,000.
  2. Sum counts until threshold exceeded: BB1=500k, BB1+BB2=800k, BB1+BB2+BB3=950k (still < 990k), BB1+BB2+BB3+BB4=990k (exactly reached).
  3. Cn (cutoff count) = 40k (BB4's count, the last one included before exceeding threshold).
  4. Hot BBs: all BBs with count ≥ 40k → BB1, BB2, BB3, BB4 are hot. BB5, BB6 are warm/cold.
  5. Result: hot code = top 4 BBs (99% of execution). Compiler places them in .text.hot section.
🔥
Insight: Percentilehot=99% means "mark code that covers 99% of total execution as hot." Higher threshold = more code marked hot (closer to CLIP). Default 99% balances selectivity and coverage (Fig. 8).

② Marking PTEs at load time

The loader reads ELF headers: .text.hot spans vAddr 0x10000–0x14000 (16kB), .text.warm 0x14000–0x18000, .text.cold 0x18000–0x1C000. Page size = 4kB. How many pages are marked hot/warm/cold?

  1. Hot section: 0x10000–0x14000 = 16kB → 16kB ÷ 4kB = 4 pages marked hot (PTE bits set accordingly).
  2. Warm section: 0x14000–0x18000 = 16kB → 4 pages marked warm (PTE bits set accordingly).
  3. Cold section: 0x18000–0x1C000 = 16kB → 4 pages marked cold (PTE bits set accordingly).
  4. Edge case: if .text.hot ends at 0x13800 (14kB), the 4th page (0x13000–0x14000) spans hot and warm. The paper's listed mitigations are padding sections to page boundaries, disabling temperature marking on overlapping pages, or using smaller code pages.
  5. Result: 4 hot pages, 4 warm, 4 cold. Total overhead: at most 2 bits × 12 pages = 24 bits (3 bytes) in PTEs — negligible.
💡
Larger pages (16kB) reduce TLB pressure but increase risk of mixed-temperature pages. Table 5 shows most benchmarks need <200 hot/warm pages even at 4kB granularity — TLB span is not a bottleneck.

③ Insertion priority in action

Cache set with 8 ways, initially empty. Sequence: (1) hot line A, (2) data line B, (3) warm line C, (4) hot line D, (5) hit on A, (6) data line E. What are the RRPVs after step 6?

  1. Step 1: Insert hot A at RRPV=0. Set: [A:0, -, -, -, -, -, -, -].
  2. Step 2: Insert data B at RRPV=2 (default RRIP). Set: [A:0, B:2, -, -, -, -, -, -].
  3. Step 3: Insert warm C at RRPV=1 (TRRIP-2) or RRPV=2 (TRRIP-1). Using TRRIP-2: [A:0, B:2, C:1, -, -, -, -, -].
  4. Step 4: Insert hot D at RRPV=0. Set: [A:0, B:2, C:1, D:0, -, -, -, -].
  5. Step 5: Hit on A → stays RRPV=0 (already Immediate). Set unchanged.
  6. Step 6: Insert data E at RRPV=2. Set: [A:0, B:2, C:1, D:0, E:2, -, -, -]. Hot lines (A, D) are at highest priority.
🔥
Observation: hot lines stay at RRPV=0 longer than RRIP's Intermediate (RRPV=2). When the set fills and eviction scans for RRPV=3, hot lines survive more increments (0→1→2→3) than data lines (2→3).

④ Computing MPKI reduction

Benchmark sqlite under SRRIP: L2 instruction MPKI = 4.08, data MPKI = 6.99 (Table 3). Under TRRIP-1: instruction MPKI reduced by 20.58%, data MPKI increased by 8.73%. What are the new MPKI values?

  1. Instruction MPKI reduction: 20.58% of 4.08 = 0.84. New I-MPKI = 4.08 − 0.84 = 3.24.
  2. Data MPKI increase: 8.73% of 6.99 = 0.61. New D-MPKI = 6.99 + 0.61 = 7.60.
  3. Total MPKI: SRRIP = 4.08 + 6.99 = 11.07. TRRIP-1 = 3.24 + 7.60 = 10.84. Reduction = 11.07 − 10.84 = 0.23 (2.1% total).
  4. Speedup interpretation: In the paper's simulator config, L2 access is 8/12 cycles, SLC is 10/30 cycles, and DRAM is 400 cycles. An L2 miss may go to SLC or DRAM. 0.84 fewer instruction misses per 1k instructions translates to reduced stalls. Data misses are masked by OoO execution, so net gain ≈ 2–3% speedup (sqlite shows 2.0%, Fig. 6).
  5. Result: TRRIP trades slight data MPKI increase for large instruction MPKI reduction — profitable because instruction misses stall the frontend.
💡
Why does data MPKI increase? Hot instruction lines occupy more cache space (higher priority), displacing some data lines. But modern cores tolerate data misses better (OoO execution, LSQ buffering) than instruction misses (frontend starvation).
Test yourself

Practice problems

Try each before opening the answer. Every one ends with the common misconception it checks.

Problem 1 · easy
TRRIP inserts hot instruction lines at RRPV=0 (Immediate). Why not also promote warm lines to RRPV=0 on hits, like RRIP does?
Show answer
TRRIP-2 conservatively decrements warm lines on hits (RRPV=1 → 0 after one hit, or 2 → 1) instead of jumping to RRPV=0 immediately. Reason: protect hot lines. If warm lines jump to RRPV=0 on every hit, they compete equally with hot lines for priority, potentially displacing hot code that's more frequently executed. Conservative promotion keeps warm below hot in the priority hierarchy.Misconception: that all instruction lines should be treated equally. TRRIP's whole idea is selectivity — hot code runs far more often than warm, so it deserves higher priority.
Problem 2 · medium
Fig. 3 shows hot code has 60% of accesses with reuse distance 0–4. An 8-way cache should keep these without misses. Why does TRRIP still help?
Show answer
The 0–4 reuse distance counts all unique lines (instruction + data) between accesses. Even if hot code has short reuse within itself, intervening data accesses and warm/cold code push the true reuse distance higher. Fig. 3's "optimistic" measurement (counting only hot unique lines) shows better fit, but reality is base measurement. An 8-way cache fills with hot + data + warm → hot gets evicted. TRRIP prioritizes hot at RRPV=0, keeping it alive despite data/warm competition.Misconception: that reuse distance is measured only within one memory type. The cache sees a unified stream of instruction and data accesses — reuse distance is across both.
Problem 3 · medium
TRRIP uses at most 2 PTE bits to encode temperature states. Why not use 1 bit (hot/non-hot) and save even more space?
Show answer
TRRIP-1 (hot-only) effectively uses 1 bit (hot vs default). TRRIP-2 adds warm and cold to enable finer-grained priority: warm gets RRPV=1 (higher than data's RRPV=2 but lower than hot's RRPV=0). Cold could be inserted at RRPV=3 (lowest priority) to prevent polluting the cache. The paper's TRRIP-2 shows marginally better performance (27.3% vs 26.5% MPKI reduction) by distinguishing warm. But the gain is small — TRRIP-1 is simpler and sufficient for most cases.Misconception: that more bits = always better. TRRIP-1's 1-bit-equivalent scheme (hot vs not) captures most of the benefit. The 2-bit encoding is for future extensibility and to match existing PBHA hardware (4 bits available, using at most 2).
Problem 4 · hard
Fig. 7a shows omnetpp has ~50% coverage of costly instruction misses in hot code. Why doesn't TRRIP still help omnetpp (Fig. 6 shows 1.8% speedup)?
Show answer
Coverage measures % of costly misses in hot code, but TRRIP helps by reducing total instruction MPKI (14.3% reduction for omnetpp, Table 3). Even if only 50% of costly misses are in hot code, reducing those 50% still cuts frontend stalls. The other 50% (in warm/external code) are untouched by TRRIP but handled by default RRIP — no worse than baseline. Also, omnetpp's raw instruction MPKI is 4.71, relatively high, so a 14.3% reduction (0.67 fewer misses/1k instr) translates to noticeable speedup. Coverage is not the only metric — raw MPKI reduction matters more.Misconception: that low coverage means no benefit. TRRIP only needs to reduce misses on some hot code to improve overall performance. 50% coverage is still better than RRIP's 0% coverage (blind to temperature).
Problem 5 · medium
TRRIP reuses ARM PBHA bits (implementation-defined PTE bits). What if the OS already uses those bits for another purpose?
Show answer
PBHA provides 4 bits, and most OSes don't use all of them. TRRIP uses at most 2 bits. If the OS uses the other 2, coexistence is possible (e.g., bits 0-1 for TRRIP, bits 2-3 for OS). If all 4 are taken, TRRIP can't be deployed without OS changes. Alternative: use x86's AVL (available) bits in PTEs (Intel/AMD reserve 3 bits for software use). The key is TRRIP's flexibility — it's a co-design technique, not hardware-only, so OS cooperation is required but minimal.Misconception: that TRRIP is zero-cost to deploy. It requires OS loader changes to populate PTE bits — not a hardware-only drop-in. But the OS changes are localized (loader + mmap syscall) and don't affect the ISA or kernel critical path.
Problem 6 · hard
CLIP (a pure-hardware baseline) treats all instruction lines as high priority. Why does TRRIP outperform it (Fig. 6: 3.9% vs 1.6%)?
Show answer
CLIP inserts all instruction lines at Immediate priority (RRPV=0), competing equally for cache space. When hot and warm code both arrive at RRPV=0, CLIP's eviction can't distinguish which to keep — both are treated the same. TRRIP is selective: only hot lines get RRPV=0; warm gets RRPV=1 (or default RRPV=2 in TRRIP-1). This hierarchy ensures hot code survives longer when the cache is full. Result: TRRIP reduces instruction MPKI more than CLIP (26.5% vs 13.6% geomean, Table 3). CLIP's all-or-nothing approach dilutes priority across too much code.Misconception: that prioritizing all instructions is better than prioritizing just hot instructions. Selectivity is key — flooding the cache with high-priority lines defeats the purpose of priority. TRRIP's software hints enable that selectivity.
Problem 7 · easy
Does TRRIP require recompiling all code in the system, or just the target application?
Show answer
Only code compiled with TRRIP's PGO gets temperature markings. External libraries not recompiled are treated as default (no temperature bits, RRIP applies). This is fine — TRRIP degrades gracefully. Ideally, system libraries (UI, graphics, runtime) are also PGO'd (common in mobile platforms) to maximize coverage. But user apps can benefit from TRRIP even if libraries aren't, since the app's own hot code gets prioritized.Misconception: that TRRIP requires whole-system adoption. It's incremental — any binary compiled with TRRIP's PGO gains benefit, and non-TRRIP binaries coexist safely (default RRIP behavior).
Problem 8 · medium
Fig. 9a shows TRRIP's speedup decreases as L2 size grows (128kB → 512kB). Why?
Show answer
Larger caches have lower MPKI (fewer misses) because more code/data fits. TRRIP's benefit comes from reducing instruction misses — if the baseline MPKI is already low (large cache), there's less room for improvement. At 512kB, many hot+warm+cold lines fit simultaneously, so TRRIP's prioritization matters less. CLIP and Emissary also see reduced gains but less so, because they cover all code (no coverage limitation). TRRIP's relative drop is larger because its coverage is limited to compiled binaries. Key: TRRIP shines when cache is under pressure (mobile: 128kB L2 per core is common).Misconception: that TRRIP is equally useful at all cache sizes. It's most valuable when the cache is too small to hold hot+warm+data — exactly the mobile constraint.
Quick answers

Frequently asked questions

Ten questions newcomers ask most. Tap any to expand.

Q1What's the one-sentence idea of TRRIP?▾
Pass PGO's knowledge of hot/warm/cold code to the cache via existing PTE bits, so the cache replacement policy can prioritize hot instruction lines (insert at RRPV=0) and reduce frontend stalls — for free.
Q2Why does PGO alone not fix the frontend bottleneck?▾
PGO optimizes code layout (spatial locality: place hot code together) but can't fix the substantial tail of high reuse distance in hot code (temporal locality: many hot accesses occur after 9+ intervening lines). Reuse distance is a runtime cache replacement problem, not a compile-time layout problem. TRRIP extends PGO's benefits to runtime.
Q3How is TRRIP different from RRIP or CLIP?▾
RRIP is blind to code temperature (treats all instruction/data lines by recency only). CLIP treats all instruction lines as high priority (no selectivity). TRRIP uses software hints (PGO) to be selective: hot lines get highest priority, warm/cold/data get lower. Result: TRRIP outperforms both (3.9% vs RRIP's baseline, 2.3% over CLIP, Fig. 6).
Q4What hardware changes does TRRIP require?▾
Zero new storage. It reuses existing implementation-defined PTE bits (ARM PBHA: 4 bits, TRRIP uses at most 2). The cache replacement policy adds a few conditionals (if temp==hot → RRPV=0) — negligible logic. No ISA changes, no new instructions. Total overhead: ~0% area and power (Table 4).
Q5Does TRRIP work on x86 or only ARM?▾
TRRIP is architecture-agnostic in principle. It needs implementation-defined PTE bits to pass temperature. x86 PTEs have AVL (available) bits (3 bits reserved for software use) — TRRIP can use 2 of those. The paper evaluates on an ARM-like simulator because mobile platforms (target domain) are predominantly ARM, but x86 is feasible.
Q6What if the PGO profile is stale (workload changes)?▾
Missing temperature hints degrade to default RRIP for lines without markings. Stale or wrong hints can cause issues: stale hot markings may over-prioritize no-longer-hot code, and TRRIP-2's different warm/cold hit behavior doesn't automatically degrade to RRIP. The paper explicitly discusses possible regressions and disabling TRRIP if profiles become inaccurate. Best practice: update PGO profiles periodically (standard in mobile dev).
Q7Why does TRRIP increase data MPKI (Table 3)?▾
Hot instruction lines occupy more cache space (higher priority), displacing some data lines. But modern OoO cores tolerate data misses (LSQ, prefetching, execution continues) far better than instruction misses (frontend starvation, pipeline bubble). The tradeoff is profitable: 26.5% instruction MPKI reduction outweighs 4.89% data MPKI increase (geomean).
Q8Can TRRIP be extended to data caching?▾
Yes, but it requires PGO to classify data temperature (e.g., frequently accessed globals, loop-carried arrays). The paper focuses on instruction caching because (1) mobile apps are frontend-bound, (2) instruction temperature is easier to measure (basic block execution counts), (3) data access patterns are more dynamic. The paper mentions data-cache extension as future work (§6).
Q9How does TRRIP compare to hardware prefetching?▾
Orthogonal. Prefetching fetches lines before they're needed (hides latency). TRRIP keeps lines in cache longer once fetched (reduces misses). Both target frontend stalls but via different mechanisms. TRRIP's advantage: no prefetch misprediction (pollution), no extra bandwidth. Prefetching's advantage: proactive, works without PGO. They can be combined (paper uses stride prefetchers + TRRIP).
Q10What did the authors leave for future work?▾
Extending TRRIP to: (1) data caching (needs data temperature analysis), (2) BTB and branch prediction, (3) TLB optimization, (4) larger caches (L3, SLC) for server workloads, (5) dynamic temperature learning (adapt at runtime without recompiling), (6) JIT/AOT compilers (profile + optimize on-the-fly). The paper does not mention heterogeneous cores or per-core thresholds as future work.

Glossary & further reading

Every term in one place, then a path to go deeper.

Core terms
TRRIPTemperature-based Re-Reference Interval Prediction — the paper's co-designed cache replacement policy (MICRO 2025).
Hot codeInstructions that contribute top 99% of execution (default PGO threshold). Placed in .text.hot ELF section.
Warm/cold codeIntermediate or rarely executed instructions. TRRIP-2 inserts warm at RRPV=1, cold at RRPV=2.
PGO (Profile-Guided Optimization)Compile → run → collect basic block counts → recompile using profile. TRRIP reuses PGO's temperature classification.
RRIP (Re-Reference Interval Prediction)Cache replacement policy that inserts new lines at Intermediate priority (RRPV=2), upgrades to Immediate (RRPV=0) on hits. TRRIP extends RRIP.
RRPV (Re-Reference Prediction Value)Per-line priority in RRIP. 2-bit: 0=Immediate (keep), 1=Near, 2=Intermediate, 3=Distant (evict). Lower = higher priority.
Reuse distanceNumber of unique cache lines accessed between two uses of the same line. Paper shows about 60% of hot code has reuse distance 0–4, but a substantial tail extends to 9+.
MPKI (Misses Per Kilo-Instruction)Key metric. TRRIP cuts L2 instruction MPKI by 26.5% geomean (Table 3).
PTE (Page Table Entry)OS data structure mapping virtual→physical address + metadata. TRRIP stores temperature in at most 2 existing PTE bits.
PBHA (Page-Based Hardware Attributes)ARM feature: 4 implementation-defined bits in PTEs, passed with memory requests. TRRIP uses at most 2 bits for temperature.
Frontend bottleneckCPU pipeline stall due to instruction fetch/decode delays (e.g., I-cache misses). Starves backend, wastes cycles.
ELF (Executable and Linkable Format)Binary format on Linux/Android. TRRIP's compiler creates .text.hot/.warm/.cold sections.
Baseline policies (related work)
LRU (Least Recently Used)Classic replacement: evict the oldest line. Struggles with scans/thrashing. RRIP outperforms LRU on mobile workloads.
SRRIP (Static RRIP)Insert at RRPV=2 (Intermediate), promote to RRPV=0 on hits. Resists scan pollution. Paper's baseline.
BRRIP (Bimodal RRIP)Occasionally insert at RRPV=0 (ε=1/32) to detect thrashing. Complementary to SRRIP.
DRRIP (Dynamic RRIP)Set-dueling: some sets use SRRIP, some BRRIP, choose winner via saturating counter. Adapts to workload mix.
CLIP (Code Line Preservation)RRIP variant: all instruction lines inserted at RRPV=0. Outperforms RRIP (1.6% over SRRIP, Fig. 6) but less selective than TRRIP.
SHiP (Signature-based Hit Predictor)Adds 64kB predictor table (PC signatures) to predict Distant re-reference. High overhead (1.7% power, 3.0% area, Table 4).
EmissaryPrioritizes instruction lines causing decode starvation via way-locking. Requires 2 bits per cache line (0.5% power, 0.7% area).

Related papers

Utility-Based Cache Partitioning (UCP) adjacent

Qureshi & Patt · MICRO 2006

UCP partitions shared cache between programs by utility (misses saved per way), not demand. TRRIP is analogous but for code temperature within one program: prioritize hot code, not high-demand code.

RRIP: Re-Reference Interval Prediction foundation

Jaleel et al. · ISCA 2010

The baseline policy TRRIP extends. Introduces RRPV, Intermediate insertion, and the idea of predicting re-reference intervals. TRRIP adds software hints to improve prediction accuracy.

AutoFDO: Automatic Feedback-Directed Optimization related

Chen, Li, Moseley · CGO 2016

Sampling-based PGO for warehouse-scale apps. TRRIP uses instrumentation PGO (more accurate) but the temperature classification idea applies to sampling PGO too.

AsmDB: Understanding Front-End Stalls in Warehouse-Scale Computers motivation

Ayers et al. · ISCA 2019

Characterizes frontend bottlenecks in datacenter workloads. Shows instruction cache misses are a major stall source. TRRIP targets the same problem on mobile platforms.

Emissary: Enhanced Miss Awareness Replacement Policy comparison

Nagendra et al. · ISCA 2023

Pure hardware approach: prioritize instruction lines causing decode starvation via way-locking. TRRIP achieves similar goal via software hints, lower overhead but limited code coverage.

I-SPY: Context-Driven Conditional Instruction Prefetching orthogonal

Khan et al. · MICRO 2020

PGO-guided instruction prefetching using new ISA hints. TRRIP avoids ISA changes and targets cache replacement, not prefetching. Both reduce frontend stalls via different mechanisms.
