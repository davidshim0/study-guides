MICRO 2025 · A Field Guide

Balancing memory traffic fast, by partitioning the cache instead of migrating pages

CXL and host DRAM offer massive aggregate bandwidth, but migration-only policies saturate near-memory and waste far-memory. This is the plain-English story of TierTune, which balances latency in milliseconds via LLC partitioning, backed by selective page migration.

📄 "Beyond Page Migration" · Lee et al. 🎓 Assumes cache basics, page migration, CXL ⏱ ~30 min read
🗄️

Partition the LLC between near and far memory nodes to balance L1 miss latency — far faster than page migration — then migrate pages only when cache management alone can't resolve persistent imbalances.

Before you begin

Ten ideas the paper assumes you know

Each card clears up one concept TierTune builds on. Read in order the first time, or jump to what's unfamiliar.

near memory (host DRAM) far memory (CXL DRAM) LLC partitioning page migration
1Tiered memory: near (host) vs far (CXL) DRAM▾

Near memory is host-attached DRAM over DDR5; far memory is CXL-attached DRAM over PCIe. CXL adds ~50–100 ns latency but delivers huge bandwidth via multiple PCIe lanes (e.g., 35 GB/s per CXL device, scalable with more devices). Combined, near+far memory offers aggregate bandwidth far exceeding near memory alone — if you use it.

The paper focuses on a baseline with 8 channels of near memory and 8 of far memory, both DDR5-3200. Far memory has +100 ns overhead (CXL-100), or +200/+400 ns in other configurations.

Now you're ready: TierTune's goal is to utilize the full aggregate bandwidth by balancing traffic, not just migrating hot pages to near memory.
2Page migration and why it's slow▾

To place hot data in near memory, the OS or a runtime migrates pages: copy 4 KB from one memory node to another, update the page table, flush TLBs. Each migration takes time and bandwidth. Migration bandwidths measured in the paper vary by policy: TPP+Colloid ~0.2 GB/s, HeMem/Memtis+Colloid ~0.4 GB/s. At these rates, migrating 10 GB (as measured in GUPS) takes 25–250 seconds depending on the policy.

Policies like TPP, HeMem, and Memtis track page hotness (via page-fault counters or Intel PEBS sampling) and migrate hot pages to near memory, cold to far memory. But migration is inherently coarse-grained and slow.

Now you're ready: TierTune's key innovation is using LLC partitioning to balance latency without migrating pages, achieving sub-second response.
3Hotness-based migration saturates near memory▾

Policies like Memtis prioritize moving frequently accessed pages to near memory, under the assumption that near memory is always lower latency. But when near memory's bandwidth is saturated, its loaded latency exceeds far memory's unloaded latency. The paper shows that Memtis achieves only 67% of the aggregate bandwidth in memory-intensive workloads like LULESH and miniFE, saturating near memory while underutilizing far memory.

Result: 39% and 33% slowdowns vs. simple interleaving in those workloads. Demand ≠ benefit.

Now you're ready: TierTune measures latency, not hotness, and partitions the cache to distribute traffic.
4What "latency" means: L1 miss latency vs LLC miss latency▾

The paper distinguishes two metrics. LLC miss latency measures time from an LLC miss to data arrival from memory; prior work (Colloid) uses it. L1 miss latency measures time from an L1 miss to when the core gets data — from L2, LLC, or memory. L1 miss latency inherently captures the effects of caching and prefetching; LLC miss latency does not.

Example: in mcf, aggressive prefetching raises LLC miss latency (more traffic) but IPC improves because prefetches hide the latency. Using LLC miss latency as a balancing metric would mislead the policy. TierTune uses L1 miss latency.

Now you're ready: TierTune monitors per-node L1 miss latency via MSHRs and uses it to drive cache partitioning.
5LLC ways and way-based partitioning (Intel CAT/AMD CAE)▾

An LLC is split into sets, each with ways. A 20-way LLC means each set has 20 slots. Intel CAT and AMD CAE allow software to partition ways among applications or VMs via a bit-mask register. For example, app A gets ways 0–9, app B gets ways 10–19.

TierTune extends this: instead of partitioning among apps, it partitions among memory nodes. Near-memory lines use ways 0–N, far-memory lines use ways N+1–19. The partition size shifts dynamically based on latency.

Now you're ready: TierTune adds destination bits to MSHR entries to tag each line's memory node, enabling per-node way allocation.
6Why cache management is faster than migration▾

Reallocating one LLC way is a register write (a bit-mask update). Data already in cache doesn't move; newly inserted lines use the new partition. No memory copying, no TLB flush, no page-table update. Response time: 50 ms intervals in the paper. Compare to page migration: copying data at 0.2–0.4 GB/s measured bandwidth for TPP+Colloid and HeMem/Memtis+Colloid respectively, plus overhead.

The trade-off: cache capacity is small (~2.5 MB per core) vs. memory capacity. So cache management handles short-term traffic shifts; page migration handles long-term imbalances.

Now you're ready: TierTune uses cache management for rapid response, migration for sustained imbalances the cache can't fix.
7MSHR (Miss State Holding Register)▾

An MSHR entry tracks an outstanding cache miss: address, tag, timestamp, etc. Intel calls it a Line Fill Buffer. When an L1 miss occurs, an MSHR entry is allocated; when data arrives, the entry is freed. Latency = time between allocation and free.

TierTune adds destination bits to each MSHR entry (by comparing the physical address to each node's address range) to distinguish near vs. far memory. This minor hardware extension lets the latency monitor measure per-node L1 miss latency.

Now you're ready: TierTune's latency monitor sits in the MSHR, reading per-node latency every 50 ms with negligible overhead.
8Why Colloid (migration-based balancing) is slow▾

Colloid, integrated with Memtis, measures per-tier LLC miss latency (via Intel CHA counters) and migrates pages from high-latency to low-latency nodes when the difference exceeds a threshold. It uses a binary-search approach to find the right fraction of pages to move, recomputes every 500 ms.

Result: in GUPS, Colloid takes 25 seconds to balance latency, migrating 10 GB. In DLRM (a dynamic workload), it achieves latency balance only 20% of the time. TierTune achieves balance in sub-second intervals via cache partitioning.

Now you're ready: TierTune is designed to do what Colloid does, but orders of magnitude faster, by managing cache instead of migrating pages.
9Performance metrics▾

Three performance metrics. Weighted speedup = Σ (IPCᵢ / singleIPCᵢ), capturing execution-time benefit. Throughput = Σ IPCᵢ, raw aggregate IPC. Fairness = harmonic mean of normalized IPCs, balancing across programs.

The paper reports normalized performance as the primary metric in Figure 6, showing TierTune+ achieves consistent improvement across all three metrics.

Now you're ready: TierTune's gains come from better aggregate bandwidth utilization and fewer migration overheads.
10Quick glossary of key acronyms▾

CXL = Compute Express Link (memory interconnect over PCIe). LLC = Last-Level Cache. MSHR = Miss State Holding Register (L1↔L2 tracking). CAT = Cache Allocation Technology (Intel's way-partitioning). MPKI = Misses Per Kilo Instruction. PEBS = Precise Event-Based Sampling (Intel's hardware profiling). TPP, HeMem, Memtis = prior hotness-based migration policies. Colloid = prior latency-balancing policy via migration.

Now you're ready: TierTune integrates cache management (fast) with page migration (selective) to balance latency in tiered memory.
The whole story in one place

TierTune in a nutshell

The problem, the fast fix via cache, the selective migration backup, and the payoff.

When near and far memory share the same system, traditional hotness-based migration (TPP, Memtis) concentrates traffic on near memory, saturating its bandwidth and leaving far memory underutilized. For example, Memtis routes 96% of traffic to near memory in bc, saturating the 67.6 GB/s limit and achieving 33% less aggregate bandwidth than simple interleaving in miniFE. This leads to 39% and 33% slowdowns in memory-intensive workloads (LULESH, miniFE).

Prior work Colloid attempts to balance latency by migrating pages from high-latency to low-latency nodes based on LLC miss latency measurements. But migration is slow: in GUPS, achieving balance requires moving 10 GB and takes 25–250 seconds depending on the policy (25 s for Memtis+Colloid at 0.4 GB/s, 250 s for TPP+Colloid at 0.2 GB/s). In dynamic workloads like DLRM, Colloid achieves balance only 20% of the time. Moreover, LLC miss latency is a misleading metric — it ignores caching and prefetching effects (e.g., in mcf, IPC improves despite high LLC miss latency due to aggressive prefetching).

TierTune's fix: dynamically partition the LLC between near and far memory nodes, allocating ways to balance L1 miss latency (which inherently captures caching effects). TierTune monitors per-node latency every 50 ms via MSHR-based counters with added destination bits, reallocating LLC ways in sub-second intervals. This handles short-term traffic shifts far faster than page migration. For sustained imbalances the cache can't resolve (e.g., heavily skewed workloads), TierTune integrates a selective page migration policy that moves pages from high-latency to low-latency nodes only when justified by the latency difference and bandwidth conditions.

The payoff: on 18 memory-intensive workloads (SPECCPU2017, GAPBS, HPC, AI), TierTune (cache-only) improves normalized performance by 8.5% over Memtis+Colloid; TierTune+ (cache + migration) improves by 19.6% on average (up to 39.6% improvement versus Memtis+Colloid in cc_sv, or 48.6% versus Memtis alone). It achieves 84.4% aggregate bandwidth utilization vs. 70.4% for Memtis+Colloid and performs 9.76× fewer migrations. Hardware cost: 12 bits per core for MSHR destination bits plus 1 comparator, 32 bits for LLC bit-masks (in a 16-way configuration), no critical path impact.

🌱 Key takeaways

  • Hotness-based migration saturates near memory, underutilizes far memory → up to 39% slowdown.
  • Migration-based balancing (Colloid) is slow: 25–250 s in GUPS, 20% success in DLRM.
  • LLC miss latency is a poor metric — ignores caching/prefetching; L1 miss latency is accurate.
  • TierTune partitions LLC by near/far memory nodes, rebalances in ~50 ms via way reallocation.
  • Extended with selective page migration for sustained imbalances beyond cache capacity.
  • Hardware: destination bits in MSHR, bit-mask in LLC; ~12 bits/core, 32 bits/LLC (16-way), no critical path.
  • Result: +19.6% avg normalized performance, 9.76× fewer migrations, 84.4% aggregate bandwidth usage.
Seven everyday pictures

How to think about TierTune

Each picture stands alone. Every card ends with an honest note about where the picture stops being true.

The core idea · Latency balance vs. hotness

Two faucets feeding two buckets

You have two faucets (near and far memory channels) filling two buckets (applications). Hotness-based policies open the near faucet wide for the "thirsty" app, saturating it, while the far faucet drips unused. TierTune monitors which bucket is emptying faster (higher latency) and adjusts the faucets to balance the fill rates. Cache partitioning is like a splitter valve on the faucets — instant adjustment. Migration is like scooping water between buckets — slow but moves more volume.

Where it misleads: faucets have fixed flow rates; real memory bandwidth varies with load. TierTune adapts to loaded latency, not static capacity.
LLC partitioning · Way-based allocation

Nested parking garages

The LLC is a multi-story parking garage. Each way is a floor. TierTune assigns floors to near-memory cars (blue) and far-memory cars (red). If near-memory cars are circling (high latency), TierTune gives them more floors by updating a registry (bit-mask) — no cars move, just the sign at the entrance changes. Compare to migration: physically driving cars between two garages across town.

Where it breaks: in a real garage, cars stay parked; in cache, lines are evicted and refilled constantly. TierTune's partition affects new insertions, not existing data.
Latency monitoring · MSHR destination bits

Two stopwatches, one per route

Every L1 miss starts two stopwatches — one for near-memory misses, one for far. When data arrives, the stopwatch stops. TierTune averages the stopwatch times every 50 ms and compares: if far memory's average is higher, it gives far memory more LLC ways to reduce future misses. The MSHR destination bits are like colored tags on the stopwatches, distinguishing near vs. far.

Where it breaks: a stopwatch measures one event; the MSHR tracks many concurrent misses. The analogy simplifies the pipelining and aggregation TierTune does.
Cache vs. migration · Speed vs. capacity

Adjusting a thermostat vs. opening windows

You're in a room that's too hot. Turning the thermostat (cache partitioning) changes the temperature in seconds — fast but limited by the HVAC capacity. Opening windows (page migration) takes longer, moves more air, but requires physical effort and time. TierTune does both: thermostat first for immediate comfort, windows only if the thermostat can't keep up.

Where it breaks: a thermostat is continuous; LLC partitioning is discrete (one way at a time). And windows don't cost bandwidth — page migration does.
Why L1 miss latency, not LLC miss latency · Capturing caching effects

Measuring travel time from home, not from the last bus stop

You want to know how long it takes to get to work. Measuring from your front door (L1 miss) includes the car ride, the bus, and the walk — the full journey, including shortcuts you take (L2/LLC hits, prefetches). Measuring from the last bus stop (LLC miss) only tracks the final leg and misses the shortcuts. TierTune measures from the front door, so it sees the real latency including all caching effects.

Where it breaks: in mcf, aggressive prefetching increases LLC miss latency (more bus trips) but improves IPC (you arrive faster overall). L1 miss latency captures the net effect; LLC miss latency does not.
Why migration is slow · Copying and overhead

Rearranging furniture vs. repositioning the lights

You want to brighten one side of the room. Dragging heavy furniture (page migration: 10 GB copied at 0.4 GB/s = 25 seconds for Memtis+Colloid in GUPS) takes time and effort. Swiveling the ceiling lights (LLC partitioning: way reallocation via bit-mask update = ~50 ms) is instant. TierTune swivels the lights first, drags furniture only if the room stays dark.

Where it breaks: lights have limited reach (cache capacity 2.5 MB per core); furniture is permanent (memory capacity GB). Both are needed for different timescales.
Integrated design · Cache signals migration

A thermostat that signals when to open windows

TierTune's cache allocator monitors latency continuously. When the latency difference drops below a threshold, it sends a hold signal to the OS, pausing page migration (no need to open windows). When the difference exceeds the threshold despite cache adjustments, it sends a resume signal, triggering selective migration. This coordination prevents the thermostat and windows from fighting each other.

Where it breaks: a thermostat is closed-loop; TierTune's cache and migration operate at different timescales (50 ms vs. 500 ms), introducing lag between signal and action.
The full explanation

How it actually works

Section by section, with the paper's key mechanisms rebuilt as visuals.

1 · The problem — hotness saturates near memory

Policies like Memtis track page hotness via Intel PEBS (sampling LLC misses or memory loads once per 1000 events) and migrate hot pages to near memory. The assumption: near memory is always lower latency. But when near memory's bandwidth is saturated, its loaded latency exceeds far memory's. In LULESH and miniFE, Memtis saturates near memory at 67.6 GB/s (measured via Intel PCM), using only 67% of the 100 GB/s aggregate bandwidth. Result: 39% and 33% slowdowns vs. interleaving.

📊
Concretely: in bc (GAPBS), Memtis routes 96% of traffic to near memory, saturating bandwidth. Far memory sits at 4% utilization. TierTune+ balances to 47% near / 53% far, achieving 84.4% aggregate bandwidth utilization.

2 · Why migration is slow: Colloid's 25-second lag

Colloid (integrated with Memtis) measures per-tier LLC miss latency via Intel CHA counters (queue occupancy and request rates) and migrates pages when the latency difference exceeds 5% (default threshold). It uses binary search to find the fraction of pages to move, adjusts every 500 ms, enforces a dynamic migration limit. In GUPS (with a hotset shift), achieving latency balance requires moving ~10 GB. The paper measures effective migration bandwidths: TPP+Colloid at 0.2 GB/s (250 seconds to migrate 10 GB), HeMem+Colloid and Memtis+Colloid at 0.4 GB/s (25 seconds). In DLRM, Colloid achieves balance only 20% of the time at the 5% threshold, 84% at 50% (Figure 2b).

⚠
Dynamic workloads: DLRM's working set shifts constantly (Figure 2a). Migration lags these shifts, leaving latency imbalances unresolved. TierTune's 50 ms cache rebalancing responds orders of magnitude faster.

3 · The metric: L1 miss latency vs. LLC miss latency

Colloid uses LLC miss latency. But LLC miss latency ignores caching and prefetching. Example: in mcf (SPECCPU2017), LLC miss latency spikes sharply (Figure 3a), prompting Colloid to migrate pages. But IPC improves during these spikes (Figure 3b) because aggressive L1D prefetching (Figure 3c: 5M+ prefetch requests) hides the latency. LLC miss latency misleads the policy; L1 miss latency (measured from L1 miss to data arrival, whether from L2, LLC, or memory) captures the true application-visible latency.

TierTune measures L1 miss latency via MSHR-based counters. Each L1 miss allocates an MSHR entry with a timestamp and destination bits (near vs. far, determined by comparing the physical address to each node's address range). When data arrives, the latency is computed and added to a per-node running average. Every 50 ms, TierTune reads the average L1 miss latency for near and far memory.

4 · LLC partitioning: way-based allocation

TierTune partitions the LLC between near and far memory nodes using way-based allocation (Intel CAT / AMD CAE). Each node gets a subset of ways. Initially, 50/50 (e.g., 10 ways near, 10 ways far in a 20-way LLC). Every 50 ms, TierTune reads the per-node L1 miss latency. If far memory latency > near memory latency, reallocate one way from near to far (e.g., 9 near, 11 far). Repeat until latency difference < threshold or until one node reaches the minimum (2 ways to prevent thrashing).

Implementation: TierTune uses per-node bit masks to specify which ways each node can use, consistent with Intel CAT/AMD CAE. On eviction, the replacement policy respects the partition quota for each node. Changing the partition = updating bit-mask registers in hardware; no data movement.

5 · Integrating page migration: hold and resume signals

Cache management handles short-term imbalances (sub-second response), but the cache is small (~2.5 MB per core in the paper's setup). For sustained imbalances (e.g., heavily skewed workloads where one node's demand far exceeds the other), TierTune integrates a page migration policy. The cache allocator sends a hold signal to the OS when latency difference < threshold, pausing migration. When latency difference > threshold despite cache adjustments, it sends a resume signal, triggering selective migration.

Migration policy: each memory node maintains a promotion list (hot pages) and demotion list (cold pages), tracked via page-table scan (similar to TPP/Memtis). When far memory latency is high, promote hot pages from far to near; if near is full, demote cold pages from near to far. When near memory latency is high, demote hot pages from near to far. Migrations run asynchronously via kswapd, up to 0.5 GB per 500 ms window.

6 · Hardware cost and multi-node support

Per-core cost: 12 MSHR entries × 1 destination bit = 12 bits, + 1 address comparator. LLC cost: for a 16-way LLC partitioned between 2 nodes, TierTune uses 32 bits for the bit-masks (16 bits per node to specify which ways it can use). The paper states total hardware cost is "12 + 32 = 44" bits in a representative configuration. Overhead is negligible relative to cache area, with no critical path impact (address comparison is parallel to tag matching).

Multi-node extension: for N memory nodes (e.g., host DRAM + 2 CXL devices), use N sets of destination bits and N bit-masks. Balancing algorithm: diffusion-based, pairwise compare adjacent nodes (node1 vs. node2, then node2 vs. node3), reallocate ways iteratively. Hardware cost scales linearly: +1 bit per MSHR entry, +16 bits per LLC per added node (for a 16-way configuration).

Follow the numbers, step by step

Worked examples

Four traces using the paper's real data, carried all the way through.

① GUPS: 25 seconds vs. sub-second balancing

GUPS (random memory updates) shifts its hotset dynamically. The paper reports that achieving latency balance requires moving ~10 GB. Effective migration bandwidths measured: TPP+Colloid 0.2 GB/s (250 seconds), HeMem+Colloid and Memtis+Colloid 0.4 GB/s (25 seconds). How does TierTune compare?

  1. Initial state: hotset in far memory, far latency spikes to 200 ns, near latency at 110 ns.
  2. TierTune (cache-only): detects latency gap, reallocates ways from near to far. Response time: 50 ms (one monitoring interval). After multiple adjustments, latency difference drops below threshold in sub-second time.
  3. If hotset is very large: cache capacity insufficient. TierTune sends resume signal, triggers page migration (~0.5 GB per 500 ms window). Achieves balance faster than Colloid's 25–250 seconds.
  4. Result: TierTune+ (cache + migration) achieves 28.5% speedup in GUPS; Memtis+Colloid achieves similar speedup but takes significantly longer to converge.
🌱
Insight: cache partitioning handles the initial spike instantly; migration backs it up for sustained load. Colloid's migration-only approach is orders of magnitude slower.

② LULESH: bandwidth saturation fixed by cache

LULESH saturates near memory at 67.6 GB/s under Memtis, while far memory sits at 4% utilization. Execution time is 39% higher than interleaving. How does TierTune restore balance?

  1. Memtis behavior: migrates hot pages to near memory. Near memory bandwidth saturates at 67.6 GB/s (measured via Intel PCM). Aggregate bandwidth: 70 GB/s of 100 GB/s available (70% utilization).
  2. TierTune (cache-only): monitors L1 miss latency. Near latency rises (loaded), far latency remains lower (unloaded). TierTune reallocates ways from near to far, reducing near-memory LLC misses, shifting traffic to far memory.
  3. Result: after cache adjustments, latencies converge. Aggregate bandwidth utilization: 84 GB/s (84%). Execution time improves by 37.3% over Memtis+Colloid.
💡
Cache partitioning is sufficient here because LULESH's working set benefits from the cache rebalancing. No page migration needed.

③ DLRM: dynamic phases, 20% Colloid success rate

DLRM's memory access pattern shifts constantly (Figure 2a). HeMem+Colloid achieves latency balance only 20% of the time at the default 5% threshold. How does TierTune adapt?

  1. Phase 1: near latency 110 ns, far latency 120 ns. TierTune allocates 11 ways to far, 9 to near. Balance achieved quickly.
  2. Phase 2: hotset shifts; near latency spikes to 140 ns, far latency drops to 110 ns. TierTune reallocates ways from far to near (8 ways far, 12 ways near). Balance restored in sub-second time.
  3. Phase 3: another shift; near latency 130 ns, far 150 ns. TierTune re-adjusts to 10 ways near, 10 ways far. Balance achieved quickly.
  4. Colloid behavior: lags each shift by 500 ms (monitoring interval) + migration time (seconds per phase). Achieves balance in only 6 of 30 phases (20% success at 5% threshold, 25 of 30 at 50% threshold).
  5. Result: TierTune+ achieves 19.6% avg speedup over Memtis+Colloid by responding to every phase shift.
⚠
Edge case: if phases flip faster than 50 ms (TierTune's monitoring interval), even cache management lags. But the paper finds no workloads with sub-50 ms phase changes in their suite.

④ Hardware cost: 12 bits per core, 32 bits per LLC

TierTune adds destination bits to MSHR entries and bit-masks to the LLC. How much does it cost?

  1. Per-core MSHR cost: 12 MSHR entries × 1 destination bit = 12 bits. One address comparator (compares physical address to near/far address ranges, parallel to tag matching, no added latency).
  2. LLC bit-mask cost: For a 16-way LLC partitioned between 2 nodes: 32 bits (16 bits per node for bit-masks). Stored in registers, updated every 50 ms.
  3. Total overhead: 12 bits + 32 bits = 44 bits per core-LLC pair as stated in the paper. This is negligible compared to cache area (e.g., 2.5 MB LLC per core).
  4. Multi-node: for 3 nodes (host + 2 CXL), add 1 bit/MSHR + 16 bits/LLC per node → 13 bits/core + 48 bits/LLC (for 16-way). Overhead remains negligible.
🌱
Insight: negligible hardware cost, no critical path impact. The paper emphasizes that existing CAT/CAE infrastructure does 90% of the work; TierTune just adds per-node tagging.

Practice problems

Try each before opening the answer. Every one ends with the misconception it's checking for.

Problem 1 · easy
A workload saturates near memory bandwidth at 67.6 GB/s while far memory is at 10% utilization. TierTune's latency monitor shows near latency at 150 ns, far at 110 ns. What does TierTune do first, and why?
Show answer
Reallocate LLC ways from near to far memory (e.g., shift from 10/10 to 9/11). This reduces near-memory LLC misses, shifting traffic to far memory without migrating any pages. Response time: ~50 ms (one monitoring interval). Page migration is not needed yet because the imbalance is traffic-driven, not placement-driven.Misconception: that page migration is always the first response. TierTune's cache partitioning is faster and sufficient for traffic imbalances.
Problem 2 · medium
Colloid measures LLC miss latency; TierTune measures L1 miss latency. In mcf, aggressive prefetching raises LLC miss latency from 100 ns to 150 ns, but IPC improves from 0.8 to 1.0. Which metric correctly captures the performance impact, and why?
Show answer
L1 miss latency correctly captures the performance impact. Prefetching increases LLC miss traffic (raising LLC miss latency) but hides the latency by fetching data before the core needs it, improving IPC. L1 miss latency measures the time from L1 miss to data arrival at the core, inherently including the prefetch benefit. LLC miss latency only measures the final memory access, missing the caching effect. Colloid would see the 150 ns spike and wrongly migrate pages; TierTune would see the improved L1 miss latency and avoid unnecessary migration.Misconception: that LLC miss latency always correlates with performance. It doesn't capture the full cache hierarchy and prefetching effects.
Problem 3 · hard
In GUPS, Memtis+Colloid takes 25 seconds to balance latency by migrating 10 GB at 0.4 GB/s. TierTune achieves balance in sub-second time via cache partitioning. But the LLC is only 2.5 MB per core (16 cores = 40 MB total). How can 40 MB of cache rebalance 10 GB of working-set traffic?
Show answer
Cache partitioning doesn't move data; it controls which new cache misses go where. After TierTune reallocates ways from near to far, newly inserted far-memory lines use the expanded far-memory partition, reducing far-memory misses. The near-memory partition shrinks, so some near-memory lines are evicted to make room, but this happens naturally via LRU, not by copying data. The 40 MB cache acts as a traffic valve, steering future accesses without touching the 10 GB working set. Over time, the cache's influence propagates to memory traffic. If the imbalance persists beyond the cache's reach, TierTune triggers page migration as a backup.Misconception: that cache partitioning must "hold" the working set. It doesn't; it controls the flow of misses, indirectly balancing traffic far beyond its capacity.
Problem 4 · medium
TierTune adds 12 bits per core (MSHR destination bits) and 32 bits per LLC (bit-masks for 16-way configuration). For a 128-core system with a 20-way LLC (2.5 MB per core), what is the total hardware overhead in bytes?
Show answer
Per core: 12 bits (MSHR) + 32 bits (LLC bit-masks for 16-way; the paper states the configuration cost as representative) = 44 bits = 5.5 bytes. 128 cores × 5.5 B = 704 B total. This is negligible compared to 128 cores × 2.5 MB = 320 MB of LLC. (Note: the paper also adds 1 address comparator per core, a small combinational logic block, not counted here.)Misconception: that per-core hardware scales prohibitively. The overhead is tiny even at 128 cores because the extension is minimal — one bit per MSHR entry, bit-masks per LLC.
Problem 5 · easy
TierTune sends a "hold signal" to the OS to pause page migration when latency difference < threshold. Why not just disable migration entirely once latency is balanced?
Show answer
Because latency can become imbalanced again due to phase changes in the workload (e.g., hotset shifts, as in DLRM). TierTune continuously monitors latency every 50 ms. If the difference exceeds the threshold again, it sends a "resume signal" to re-enable migration. Disabling migration permanently would leave TierTune unable to address imbalances that exceed the cache's capacity.Misconception: that once balanced, latency stays balanced. Dynamic workloads shift constantly; TierTune adapts by toggling migration on and off as needed.
Problem 6 · hard
Colloid achieves latency balance only 20% of the time in DLRM at the default 5% threshold (84% at 50% threshold). TierTune achieves balance in every phase. Construct a scenario where TierTune would also fail to balance latency.
Show answer
Scenario: a workload with phase changes faster than 50 ms (TierTune's monitoring interval). For example, a microbenchmark that flips its hotset every 10 ms between near and far memory. TierTune reads latency every 50 ms; by the time it adjusts the partition (e.g., shift from 10/10 to 9/11), the phase has flipped 5 times, and the adjustment is already stale. The cache partition thrashes, never converging. The paper finds no such workloads in SPECCPU2017, GAPBS, HPC, or AI suites, so this is a theoretical edge case.Misconception: that TierTune always balances latency. It's bounded by its monitoring granularity (50 ms); sub-50 ms phases can break it, though they're rare in practice.
Problem 7 · medium
TierTune uses way-based LLC partitioning. Suppose the LLC is 16-way. TierTune guarantees each node at least 2 ways. Why 2, not 1?
Show answer
Allocating only 1 way to a node causes severe thrashing due to conflict misses. In a 16-way set-associative cache, 1 way means only 1 slot per set is available for that node's lines. High-traffic workloads would constantly evict and refill that single slot, achieving near-zero hit rate. 2 ways is the minimum to maintain a basic LRU replacement policy (MRU vs. LRU) and avoid pathological thrashing. The paper notes that performance degrades sharply when a node has 4 ways or fewer (Figure 11a: 4-way partition shows 10–15% slowdown), so 2 ways is a conservative lower bound.Misconception: that any partition is better than none. A 1-way partition is functionally equivalent to no cache, so TierTune enforces a 2-way minimum.
Problem 8 · easy
TierTune+ (cache + migration) performs 9.76× fewer migrations than Memtis+Colloid. Does this mean TierTune+ migrates 10.2% as many pages (1 / 9.76 ≈ 0.102), or something else?
Show answer
Yes, TierTune+ migrates approximately 10.2% as many pages as Memtis+Colloid, because cache partitioning resolves most imbalances without migration. TierTune+ only migrates when the latency difference persists despite cache adjustments, targeting sustained imbalances. Quick answers

Frequently asked questions

Twelve questions newcomers ask most. Tap any to expand.

Q1Why is cache partitioning faster than page migration?▾
Cache partitioning is a register write (bit-mask update) that changes which ways new cache lines use. No memory copying, no TLB flush, no page-table update. Response time: 50 ms intervals in the paper. Page migration copies 4 KB pages between memory nodes (measured at 0.2–0.4 GB/s effective bandwidth for TPP+Colloid and Memtis+Colloid), updates page tables, flushes TLBs. Response time: seconds to tens of seconds depending on the data volume. Cache management is orders of magnitude faster.
Q2What if the cache is too small to balance latency?▾
If the latency difference persists despite cache adjustments (i.e., the imbalance is due to page placement, not just traffic), TierTune sends a "resume signal" to the OS, triggering page migration. The cache handles short-term traffic shifts; migration handles long-term placement imbalances. TierTune+ (cache + migration) combines both for comprehensive coverage.
Q3Does TierTune work with existing hotness-based policies like Memtis?▾
Yes. TierTune is orthogonal to hotness-based page placement. Memtis decides which pages are hot and migrates them; TierTune decides how to balance traffic via cache partitioning and selective migration. The paper evaluates TierTune integrated with Memtis (TierTune+), showing 19.6% speedup over Memtis+Colloid. TierTune replaces Colloid's migration-based balancing with cache-based balancing, using migration as a backup.
Q4Why measure L1 miss latency instead of LLC miss latency?▾
L1 miss latency measures the time from L1 miss to when the core gets data — from L2, LLC, or memory. It inherently captures the effects of caching (L2/LLC hits reduce latency) and prefetching (fetched data arrives before the core stalls). LLC miss latency only measures memory access time, ignoring cache hierarchy effects. In mcf, aggressive prefetching raises LLC miss latency but improves IPC; L1 miss latency correctly reflects this. Using LLC miss latency would mislead the balancing policy.
Q5How does TierTune handle multi-tenant environments (multiple VMs)?▾
Each VM or application is allocated a subset of LLC ways by the hypervisor or OS (standard practice in Intel CAT / AMD CAE). TierTune operates within each VM's allocated ways, partitioning them between near and far memory. The latency monitor is per-core, so it measures latency only for the cores assigned to that VM. Page migration in VMs requires coordination between the guest OS and hypervisor (guest-physical to host-physical mapping), but TierTune's cache management is entirely local to the VM's LLC partition.
Q6What's the hardware cost, really?▾
Per core: 12 MSHR entries × 1 destination bit = 12 bits, + 1 address comparator. LLC: 32 bits for bit-masks (in a 16-way configuration for 2 nodes). Total: ~44 bits per core-LLC pair as stated in the paper. For a 2.5 MB LLC per core, overhead is negligible. No critical path impact (address comparison is parallel to tag matching). The paper emphasizes that Intel CAT/AMD CAE already support way-based partitioning; TierTune just adds per-node tagging.
Q7Does TierTune support more than two memory nodes (e.g., host + 2 CXL devices)?▾
Yes. For N nodes, add N sets of destination bits to MSHR entries and N bit-masks to the LLC. Balancing algorithm: diffusion-based, pairwise compare adjacent nodes (node1 vs. node2, then node2 vs. node3), reallocate ways iteratively until latencies converge. Hardware cost scales linearly: +1 bit per MSHR entry, +16 bits per LLC per added node (for 16-way). The paper evaluates 2-node and 4-node CXL configurations, showing TierTune+ achieves 15.4% and 37.2% speedup over Memtis in 2-node and 4-node setups, respectively (Figure 12a).
Q8Why does TierTune+ perform 9.76× fewer migrations than Memtis+Colloid?▾
Because cache partitioning resolves most traffic imbalances without migrating pages. TierTune+ only migrates when the latency difference persists despite cache adjustments, targeting sustained placement imbalances. Memtis+Colloid has no fast-response mechanism, so it migrates aggressively to balance latency, incurring high overhead. Fewer migrations = less bandwidth waste, fewer TLB flushes, faster adaptation to phase changes.
Q9What if memory bandwidth is very high (e.g., 8× the baseline)?▾
As bandwidth scales, the benefit of TierTune diminishes because bandwidth congestion is less severe, reducing the latency gap between near and far memory. The paper evaluates 2×, 4×, and 8× bandwidth (Figure 12b): at 2×, TierTune+ achieves 19.8% speedup; at 8×, only 3.2%. But the paper notes that real systems are pin-constrained (Intel Sierra Forest: 144 cores, 8 channels = 18 cores/channel; Clearwater Forest: 288 cores, 12 channels), so bandwidth scaling is limited. TierTune is most valuable in current and near-future many-core systems where bandwidth saturation is common.
Q10Does TierTune always beat Memtis+Colloid?▾
TierTune+ (cache + migration) beats Memtis+Colloid by 19.6% on average across 18 workloads, with gains up to 39.6% versus Memtis+Colloid (cc_sv). However, the paper explicitly notes that cache-only TierTune performs worse than Memtis/Memtis+Colloid on low-bandwidth workloads such as pr and pr_spmv; only TierTune+ (which includes page migration) addresses this limitation. TierTune is designed for memory-intensive workloads where traffic balancing provides substantial benefit.
Q11How does TierTune compare to simple interleaving?▾
Interleaving distributes pages 50/50 across near and far memory, balancing traffic but ignoring hotness. In bandwidth-intensive workloads (e.g., LULESH, miniFE), interleaving outperforms Memtis by 39% and 33% because it utilizes aggregate bandwidth better. TierTune improves over interleaving by 15% on average (Figure 14) by adapting the traffic split to actual latency instead of a static 50/50 ratio. Weighted interleaving (allocating by bandwidth ratio, e.g., 6:4) improves by 8%, but TierTune's dynamic approach beats both.
Q12What did the authors leave for future work?▾
The paper explicitly states that they plan to extend TierTune to multithreaded applications sharing data structures. They also note opportunities for integration with heterogeneous-bandwidth CXL devices and optimizing the monitoring interval. The paper does not discuss prefetch-policy integration, memory-only CXL devices beyond bandwidth heterogeneity, or dynamic tuning of the 50 ms interval as specific future work items.

Glossary & further reading

Core terms
TierTuneA dynamic memory management framework that partitions the LLC between near and far memory nodes to balance latency, backed by selective page migration.
TierTune+TierTune extended with page migration policy for sustained imbalances beyond cache capacity.
Near memoryHost-attached DRAM over DDR5, typically lower latency (~110 ns baseline).
Far memoryCXL-attached DRAM over PCIe, higher latency (+100–400 ns overhead) but scalable bandwidth.
L1 miss latencyTime from L1 cache miss to data arrival at the core (from L2, LLC, or memory). Captures caching and prefetching effects.
LLC miss latencyTime from LLC cache miss to data arrival from memory. Does NOT capture L2/LLC hits or prefetching.
LLC partitioningAllocating LLC ways to different entities (apps, VMs, or memory nodes). Intel CAT / AMD CAE support way-based partitioning via bit-mask registers.
MSHR (Miss State Holding Register)Hardware structure tracking outstanding cache misses. TierTune adds destination bits to distinguish near vs. far memory.
Destination bitsPer-MSHR-entry tag indicating which memory node (near or far) a miss targets, determined by address comparison.
Hold / resume signalsSignals from TierTune's cache allocator to the OS: hold pauses page migration when latency is balanced; resume triggers migration when cache management alone is insufficient.
Supporting terms
Hotness-based migrationPolicies (TPP, HeMem, Memtis) that track page access frequency and migrate hot pages to near memory, cold to far.
ColloidA prior latency-balancing policy (SOSP'24) that migrates pages from high-latency to low-latency nodes based on LLC miss latency measurements via Intel CHA counters.
Intel PEBSPrecise Event-Based Sampling, hardware profiling that samples LLC misses or memory loads to track page hotness.
Intel CAT / AMD CAECache Allocation Technology / Cache Allocation Enforcement, hardware support for way-based LLC partitioning among apps or VMs.
Aggregate bandwidthTotal bandwidth available from near + far memory. TierTune+ achieves 84.4% utilization vs. 70.4% for Memtis+Colloid.
Normalized performancePerformance metric reported in Figure 6, comparing different approaches across workloads.
Way-based allocationAssigning subsets of LLC ways to different entities, controlled by bit-mask registers.
Migration bandwidthRate at which pages are copied between memory nodes; measured in the paper as 0.2 GB/s (TPP+Colloid) and 0.4 GB/s (HeMem+Colloid, Memtis+Colloid).
CXL-100 / CXL-200 / CXL-400CXL configurations with +100 ns, +200 ns, +400 ns latency overhead vs. host DRAM.
