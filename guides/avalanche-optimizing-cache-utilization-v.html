ISCA 2025 · A Field Guide

Cache the right things, evict the dead ones

Sparse matrix multiplication fills on-chip memory with half-finished products that sit idle. Avalanche reorders computations to finish them faster, kicks out dead products immediately, and caches the next tile's matrix B instead — achieving 1.97× over the state-of-the-art.

📄 "Avalanche" · Byeon et al., Sungkyunkwan University 🎓 Sparse formats background helps — primers cover the rest ⏱ ~25 min read
🗄️

Reorder columns of A to complete products early, evict them the moment they're done, and fill the freed cache space with reusable B rows — not with idle half-sums waiting for the last tile.

Before you begin

Eight concepts the paper assumes

Each clears up one idea. Read in order the first time, or jump to unfamiliar ones.

live products / matrix B reused next tile Avalanche machinery dead products / baseline tiling
1Sparse matrices and CSC/CSR formats▾

A sparse matrix is mostly zeros. Storing every entry wastes memory, so we use compressed formats. CSC (Compressed Sparse Column) stores non-zero values column by column with three arrays: a pointer (ptr) to where each column starts, an index (idx) of each non-zero's row, and the data (data) itself. CSR does the same row-wise. Example: column 1 of A has 2 non-zeros at rows 0 and 3; CSC stores ptrA[1]=1, idxA[1..2]={0,3}, dataA[1..2]={B,C}.

Now you're ready: Avalanche reads A in CSC and B in CSR, exploiting the formats' sorted order to match indices cheaply.
2Outer-product (OutP) dataflow▾

To multiply C = A × B, the outer-product dataflow takes one column of A and one row of B, multiplies them (a scalar-vector product), and accumulates the result into C. Repeat for every column-row pair. Each pair generates a partial matrix; the merger sums them all into C. The key win: A and B are each read once. The loss: C sits in cache as an intermediate matrix through many merges, which can bloat on-chip memory.

Now you're ready: Avalanche is OutP-based; it optimizes the merge to reduce intermediate C's footprint and make space for B.
3Matrix tiling to fit intermediate C on-chip▾

With large A, the intermediate C can exceed cache. Horizontal tiling splits A row-wise into tiles: tile 1 computes rows 0–k of C, tile 2 rows k+1–m, etc. Each tile's C fits on-chip. But now B must be re-read for every tile — if A splits into 32 tiles, measured B traffic increases 5.9× on average (up to 14× for some matrices). Tiling fixes C bloat but breaks B locality.

Now you're ready: Avalanche's goal is to cache the B rows needed across tiles, cutting repeated DRAM reads.
4Live vs dead intermediate products▾

An element of C is live if it still needs more partial sums (more column-row pairs will update it). It's dead once all its contributions are done — no further merges will touch it. In dense multiplication every C element needs the same # of merges, but sparse patterns differ: some C[i,j] finish after 1 merge, others after 50. The paper found that in baseline approaches, a significant portion of on-chip intermediate products are already dead (Fig. 11) — taking space but never reused.

Now you're ready: If you can spot dead products and evict them immediately, you free cache for something useful (like B).
5Why plain LRU fails here▾

A standard LRU (Least Recently Used) cache evicts the oldest-touched line. That's fine for temporal locality within one tile, but it treats live C and B identically — both get evicted if not recently hit. Evicting a live C element costs 3 memory ops: write it out, read it back for the next merge, write again when dead. Evicting a B row costs 1 op: re-read next tile. So live C is 3× more expensive to evict. A simple LRU-based cache can't tell them apart, causing contention.

Now you're ready: Avalanche's Unified Cache adds two bits (iMat, RMC) to flag live C and reusable B, preventing costly evictions.
6Reuse distance for matrix B▾

Reuse distance = # of tiles between when B row k is used and when it's needed again. Distance 1 means the row is reused in the very next tile — highest locality. On average 40% of B has distance 1 (Fig. 15), and for some matrices (filter3D, wiki-Vote) it's >90%. If those rows stay on-chip, the next tile pays no DRAM fetch. Avalanche's RM-Caching identifies distance-1 rows at compile time (by analyzing the tiling plan) and marks them to keep.

Now you're ready: RM-Caching is a static analysis that sets a bit in the job; the cache obeys it to retain B across tiles.
7Job conflicts and the Job Allocator▾

Avalanche has 16 processing lanes (each a multiplier + merger). A job is one scalar-vector multiply (column element of A × row of B → partial vector). If two lanes process jobs for the same C row simultaneously, their mergers both read/write the same cache line — a RAW (Read After Write) hazard, called a job conflict. The Job Allocator prevents this by ensuring no two active lanes work on the same row index (AIDX) at once.

Now you're ready: The Job Allocator is a simple conflict-avoidance scheduler; the insight is that reordering A (next primer) makes conflicts more common, so handling them is critical.
8Insertion-sort-based merger▾

Each merger takes a scalar partial product (one element) from the multiplier and inserts it into the sorted intermediate row (a cache line holding, say, 4 non-zeros in column order). Insertion sort is O(N) when data is already sorted — which it is, since CSC/CSR store indices in order. The merger uses a linear array of comparators: each cycle it compares the new element to the cached row, shifts larger elements right, inserts the new one. Scales linearly with comparators (Avalanche uses 8).

Now you're ready: The merger is cheap (2.76% chip area, Table 4) but handles variable-width streams (1 new element per cycle vs a cache line of 4) efficiently.

The whole story

Avalanche in a nutshell

Problem, measurement, three techniques, and the 1.97× speedup.

Sparse matrix multiplication (SpMM, C = A × B) with the outer-product dataflow and horizontal tiling should reuse A and B perfectly, but tiling breaks B's access pattern — B gets fetched from DRAM once per tile. On 32 tiles, B traffic rises 5.9× on average (paper Fig. 7). Prior work caches B, but cache fills with intermediate C products that are half-done or already dead, leaving no space for B. On-chip utilization averages only 48% (Fig. 9).

Avalanche solves this with three techniques. Mat-Reorder (§4.1) dynamically reorders A's columns for each tile into a lower-triangular form, clustering each row's non-zeros so C products finish sequentially from top to bottom — improving temporal locality and making dead products easy to spot. DP-Evict (§4.2) detects dead products (the last column-row pair for a given row sets a bit) and immediately evicts them to DRAM, freeing cache lines. RM-Caching (§4.3) statically identifies B rows reused in the next tile (reuse distance 1, ~40% of B on average, Fig. 15) and marks them to stay on-chip via a priority-based replacement policy in the Unified Cache.

The Unified Cache (§5.5) uses two bits per line: iMat (1 = live C, highest priority to keep) and RMC (1 = reusable B, medium priority). Dead C is evicted explicitly; reusable B is protected from LRU churn. Live C costs 3× more to evict (write, re-read, write again) than B (1 re-read), so the policy minimizes total traffic. Implementation is a 2 MB, 16-bank cache (6.15 mm² total, 91% cache, Table 4) with a Job Allocator to prevent RAW conflicts and insertion-sort mergers (8 comparators, 2.76% area).

Result: Avalanche averages 1.97× speedup over SPAGHETTI (state-of-the-art OutP) and 1.2× over GAMMA (row-wise inner-product dataflow), with total off-chip memory traffic cut to 0.47× of SPAGHETTI (Fig. 22). The techniques also apply to the row-wise dataflow, yielding 11% gain there when Mat-Reorder and DP-Evict are applied (§6.7).

🍯 Key takeaways

  • Tiling fixes C bloat but breaks B locality: B is re-fetched every tile (up to 5.9× average traffic increase, max 14× on 32 tiles).
  • On-chip memory is half-empty: 48% average utilization (Fig. 9); significant dead product storage in baseline approaches.
  • Mat-Reorder clusters A's non-zeros dynamically for each tile → products finish early, sequentially top-to-bottom.
  • DP-Evict writes dead products to DRAM immediately → frees cache space.
  • RM-Caching keeps B rows with reuse distance 1 on-chip → next tile pays no DRAM fetch.
  • Unified Cache priority policy: high (live C), medium (reusable B), low (everything else) — prevents costly evictions.
  • 1.97× over SPAGHETTI, 1.2× over GAMMA, 0.47× total traffic, <1% area overhead.
Seven everyday pictures

How to think about Avalanche

Each stands alone. Every card ends with where the analogy breaks.

The core idea · Mat-Reorder

Finishing homework in order

You have 10 assignments, each needing 3 steps. Jumping around (step 1 of all, then step 2 of all) leaves 10 half-done papers on your desk. Completing them in order (all 3 steps of #1, then #2, etc.) lets you file each as done and clear the desk. Mat-Reorder clusters A's non-zeros by row, so C products finish sequentially top-to-bottom — dead products appear in a predictable order, easy to evict.

Where it breaks: real homework steps are independent; SpMM's partial sums must be accumulated (merged), and the sparse pattern limits how much clustering is possible.
Eviction · DP-Evict

Tossing dishes as you finish

A conveyor brings dirty dishes; you wash one at a time. Instead of stacking clean ones on the counter until the shift ends, you toss each into the bin the moment it's dry. DP-Evict does this: the instant a C product is dead (last partial sum merged), write it to DRAM and free the cache line. Don't wait for LRU to pick a victim later — clean space proactively.

Where it breaks: dishes don't reappear; in multi-tile SpMM, if you evict too aggressively (a live product you'll need next cycle), you pay a re-fetch. Mat-Reorder ensures only truly dead products are flagged.
Caching strategy · RM-Caching

Keeping tools you'll use next

You're assembling 10 widgets in sequence. Tool A is needed for widgets 1 and 2 (reuse distance 1); tool B for 1 and 6 (distance 5). Keep A on your bench between steps; stow B in the toolbox — you won't need it for a while. RM-Caching statically analyzes the tiling plan, identifies B rows used in adjacent tiles (distance 1, ~40% of B), and marks them to stay on-chip. The rest can be evicted.

Where it breaks: tool analogy is deterministic; SpMM's sparsity means some rows are reused unpredictably. RM-Caching only handles distance 1 (next tile) for simplicity, missing longer-range reuse.
Priority policy · Unified Cache

Triage in an ER

Patients are tagged: critical (live C, 3× cost if delayed), urgent (reusable B, 1× cost), stable (everything else). Treat critical first, urgent second, stable last. The Unified Cache does this with iMat (1 = live C, high priority) and RMC (1 = reusable B, medium priority). On eviction, pick low-priority lines first. Prevents evicting a live C product (which costs 3 memory ops: write, re-read, write) when a dead one (1 write) is available.

Where it breaks: ER triage is continuous; cache replacement happens discretely on a miss. If all lines are high-priority, the policy degrades to random — but the paper shows this is <2% of cases.
Merger · Insertion sort

Slotting a card into a sorted hand

You hold 4 cards sorted low-to-high; a dealer gives you one new card. Scan from left, find where it fits, slide bigger cards right, insert. That's insertion sort — O(N) if the hand is already sorted, which it is (CSC/CSR store indices in order). Avalanche's merger does this: 1 new partial product per cycle, merge into a cache line of (say) 4 elements. 8 comparators handle it in one pass. Scales linearly, low complexity.

Where it breaks: a card hand is fixed-width; the merger handles variable-length intermediate rows (some C rows have 1 non-zero, some 100). It processes them in cache-line chunks (4 elements), chaining merges across lines.
Job Allocator · Conflict avoidance

Assembly line with no overlap

Four workers, each assembles one widget. If two workers try to paint the same widget simultaneously, the paint smears (RAW hazard). The foreman (Job Allocator) ensures each worker gets a different widget — no two lanes process the same C row (AIDX) at once. Simple rule: skip jobs in the queue until you find one with a unique row index. Mat-Reorder clusters rows, making conflicts more common, so this is critical.

Where it breaks: assembly line is spatial; cache conflicts are temporal (read after write). The analogy captures the avoidance logic but not the memory coherence issue underneath.
ROD (Reorder Operand Descriptor) · Mat-Reorder mechanics

A dynamically updated playlist

Instead of reading A's columns in storage order (0,1,2,...), the Matrix Fetcher reads them from a reordered playlist (ROD bucket). Each ROD entry is {ptrA_start, ptrA_end, ptrB_start, ptrB_end} — the pointers to fetch one column-row pair. The playlist is computed by software to form a lower-triangular shape and updated for each tile. Hardware reads the list sequentially.

Where it breaks: the ROD bucket is dynamically updated between tiles (write-back for the next tile's sequence), adding ~4% memory traffic. The analogy of a static playlist misses that the reordering adapts per tile.
The full explanation

How it actually works

Section by section, with the paper's key mechanisms.

1 · The problem — tiling breaks B locality

Outer-product SpMM without tiling generates huge intermediate C (memory bloating). Horizontal tiling splits A into row-wise tiles (e.g., rows 0–10, 11–20, ...) so each tile's C fits on-chip. But B must be re-read for every tile. With 32 tiles, B traffic rises 5.9× on average (Fig. 7) — some matrices (poisson3Da, wiki-Vote) see 14× increase. Prior work uses tiling but doesn't cache B effectively, so DRAM bandwidth becomes the bottleneck.

🌱
Concretely: a 1 MB cache holds ~250k floats. If intermediate C for one tile is 150k elements and B is 200k, they don't both fit. A simple cache evicts randomly → live C or needed-B rows get kicked out, causing re-fetches. Result: on-chip memory sits 48% utilized (Fig. 9) — half empty.

2 · Mat-Reorder — finish products early, sequentially

In standard OutP, columns of A are processed in storage order (0,1,2,...). Each column's non-zeros are scattered across rows, so C products accumulate unpredictably — some finish early, some late, hard to tell which. Mat-Reorder dynamically reorders A's columns for each tile into a lower-triangular form: longest column first, lengths decreasing left-to-right. This clusters each row's non-zeros toward the front of the sequence.

Example (Fig. 12): if row 0 of A has non-zeros in columns {1, 3, 5}, reorder to process those columns consecutively. Row 0's C product finishes after column 5 is done — predictably. The paper proves that in a perfect lower-triangular matrix, C products become dead sequentially from top row to bottom row (Fig. 12). Real sparse matrices can't be perfectly triangularized, but the reordering still improves temporal locality significantly (Fig. 14: live elements decreased by an average of 34% after Mat-Reorder + DP-Evict).

💡
Dynamic per-tile reordering: Section 5.2 explicitly calls this "dynamic Mat-Reorder for each tile." The ROD bucket is software-controlled, stored off-chip, read sequentially, and updated for subsequent tiles. Hardware doesn't perform reordering logic, but the reordering is not a static offline preprocessing step—it adapts for each tile's computation.

3 · DP-Evict — detect & evict dead products immediately

Thanks to Mat-Reorder, products become dead in a known sequence. Detection: the Matrix Fetcher tracks which column-row pair (ROD entry) is the last one for a given row. It sets a DP-Evict bit in the job for that pair. When the merger finishes that job, it signals the cache to evict the row. Eviction: the cache writes the dead product to DRAM and frees the line immediately — no waiting for LRU.

Impact: Fig. 14 shows live elements decreased by an average of 34% after applying Mat-Reorder + DP-Evict. The freed space is available for B. Crucially, evicting a dead product costs 1 DRAM write; evicting a live product costs 3 ops (write, re-read, write when dead). DP-Evict avoids the 3× penalty.

🍯
The bit overhead: one DP-Evict bit per job entry. Table 2 specifies the job queue is 256 depth, so 256 bits total, negligible. The ROD update (writing back entries for the next tile) adds ~4% memory traffic, but that's dwarfed by the savings from not re-fetching B.

4 · RM-Caching — keep reusable B on-chip

With dead products evicted, cache space is available. RM-Caching identifies which B rows are reused in the next tile (reuse distance 1) and marks them to stay on-chip. How? Static analysis: the tiling plan (decided offline) knows which rows of A belong to tile k and tile k+1. If a column-row pair in tile k references B row j, and a pair in tile k+1 also references j, then j has distance 1. The Matrix Fetcher sets an RMC bit in the job for such rows.

The Unified Cache obeys the bit: when replacing a line, prefer evicting lines with RMC=0 (not reused next tile) over RMC=1 (reused). On average 40% of B has distance 1 (Fig. 15); for some matrices (filter3D, wiki-Vote) it's >90%. Keeping them on-chip reduces total off-chip memory traffic to 0.47× of baseline (Fig. 22).

💡
Why only distance 1? Tracking longer distances (e.g., reuse in tile k+2, k+3) is feasible but adds complexity. Distance 1 captures the bulk of reuse (>40%) with minimal overhead (one bit per job). Diminishing returns for k+2 (the paper doesn't quantify it, but implied by the 40% coverage).

5 · Unified Cache — priority-based replacement

The Unified Cache stores both B and intermediate C. Each tag entry has two control bits: iMat (1 = intermediate C) and RMC (1 = reusable B). Replacement priority:

  • High: iMat=1 (live C) — evicting costs 3 memory ops, so keep.
  • Medium: iMat=0, RMC=1 (reusable B) — evicting costs 1 re-read next tile, so prefer to keep.
  • Low: iMat=0, RMC=0 (non-reusable B or dead C already evicted) — safe to evict.

On a cache miss, the replacement engine picks a low-priority line first. If none exist, pick medium (happens <5% of accesses). If still none, pick high randomly (happens <2%). This policy minimizes total memory traffic by protecting the most expensive-to-evict data.

# Unified Cache replacement logic (simplified)
if any line has (iMat=0, RMC=0):  # low priority
    evict one of them  # LRU within this set
elif any line has (iMat=0, RMC=1):  # medium
    evict one (random if multiple)
else:  # all high (iMat=1)
    evict random  # rare case (<2%)
🌱
The policy is local per set (16-way set-associative cache) — it doesn't do global replacement across all banks. That would require a CAM (content-addressable memory) or a global victim buffer, adding area. The 16-way associativity is enough to handle most conflicts.

6 · Job Allocator — prevent RAW conflicts

Avalanche has 16 processing lanes (multiplier + merger pairs). Each job is one scalar-vector multiply (AIDX = row index of A, ADATA = value, SPB/EPB = B row pointers). If two lanes process jobs with the same AIDX simultaneously, their mergers access the same C row → RAW hazard. The Job Allocator ensures each active lane has a unique AIDX by skipping jobs in the queue until it finds one with no conflict.

Example (Fig. 18): job queue has {(0,B), (0,E), (1,A), (2,D)}. Lane 0 takes (0,B). Lane 1 sees (0,E) — conflict! Skip it, take (1,A) instead. Lane 2 takes (2,D). Now all three lanes run in parallel, no conflict. The skipped (0,E) waits for lane 0 to finish (0,B) before being assigned.

⚠
Mat-Reorder increases conflicts: by clustering rows' non-zeros, multiple jobs for the same row appear close together in the queue. The Job Allocator's skipping handles this but can stall if the queue is short. The paper uses a 256-deep queue to mitigate stalls.

7 · Insertion-sort merger

The merger takes a partial product (one element, COO format: row, col, value) from the multiplier each cycle and inserts it into the intermediate row (a cache line holding, say, 4 non-zeros in column order). Algorithm: read the cache line, use 8 comparators to find where the new element fits (by column index), shift larger elements right, insert. If the line overflows, write the oldest elements back to cache and fetch the next line. Complexity is O(N) where N = # elements in the cache line (4–8), because both input streams are sorted (CSC/CSR guarantee).

The paper's merger uses 8 comparators (Table 2) — enough to handle a cache line of 4–8 elements in one cycle. Larger cache lines (16+) show diminishing returns (Fig. 24c convergence at 32 comparators). The merger accounts for 2.76% of chip area (Table 4), efficient and scalable.

Step-by-step walkthroughs

Worked examples

Four traces using the paper's real numbers.

① Reordering A into lower-triangular form

Given a 4×4 sparse A with non-zeros at (1,0)=A, (0,1)=B, (3,1)=C, (2,2)=D, (0,3)=E, (2,3)=F. Reorder columns to maximize clustering per row.

  1. Count non-zeros per column: col 0: 1, col 1: 2, col 2: 1, col 3: 2. Lengths: {1,2,1,2}.
  2. Sort columns by length descending: col 1 (len 2), col 3 (len 2), col 0 (len 1), col 2 (len 1). New order: {1,3,0,2}.
  3. Check clustering: row 0 has non-zeros in cols {1,3} (now adjacent). Row 2 has {2,3} (now {3,2}, reversed but clustered). Row 1 and 3 each have 1 non-zero (trivially clustered).
  4. ROD bucket: store column-row pairs in reordered sequence. For row 0, pairs from cols 1 and 3 are stored at indices 0. For row 2, pairs from cols 3 and 2 at indices 2. Matrix Fetcher reads them top-to-bottom.
  5. Result: products for row 0 complete after processing col 3 (the last in row 0's sequence). DP-Evict sets the bit for the job from (col 3, row 0) pair.
🍯
Insight: lower-triangular isn't geometric — it's a property of the column ordering. The fetcher reads columns in {1,3,0,2} order, so row 0's non-zeros appear first (cols 1,3), finishing its product early.

② Detecting dead products with the DP-Evict bit

Processing the reordered A from example ① with 2 tiles: tile 1 = rows 0–1, tile 2 = rows 2–3. When does row 0 become dead in tile 1?

  1. Tile 1 processes rows 0–1. The Matrix Fetcher reads ROD entries for row 0 (cols 1, 3) and row 1 (col 0).
  2. Column 1: A(0,1)=B × B_row_1 generates partial products for C_row_0. Merger accumulates them into the intermediate row.
  3. Column 3: A(0,3)=E × B_row_3 adds more partials to C_row_0. This is the last column-row pair for row 0 in tile 1 (the ROD for row 0 has only 2 entries).
  4. DP-Evict bit set: the fetcher marks the job for (col 3, row 0) with D=1. When the merger finishes, it signals the cache.
  5. Cache eviction: Unified Cache writes C_row_0 to DRAM and frees the line. Space is now available for B_row_3 (if marked RMC=1) or other data.
💡
The fetcher knows it's the last entry because the ROD bucket has a structure: each row's entries are stored consecutively, and the fetcher reads sequentially. When it hits the end of row 0's entries, it sets D=1 for that row's last job.

③ Computing reuse distance for B

Matrix A splits into 3 tiles: {rows 0–2}, {rows 3–5}, {rows 6–8}. B has 9 rows. Tile 1 uses B_rows {0,2,4}, tile 2 uses {2,4,6}, tile 3 uses {6,7,8}. Which B rows have reuse distance 1?

  1. Tile 1 → tile 2: B_rows {2,4} are used in both. Distance 1 for rows 2 and 4.
  2. Tile 2 → tile 3: B_row {6} is used in both. Distance 1 for row 6.
  3. Mark RMC bits: when processing tile 1, the fetcher checks tile 2's B usage (precomputed). Rows 2 and 4 → RMC=1. Row 0 is not reused → RMC=0.
  4. Cache behavior: after tile 1, if the cache must evict a line, it prefers B_row_0 (RMC=0) over rows 2 or 4 (RMC=1). Rows 2 and 4 stay on-chip for tile 2 → no DRAM fetch.
  5. Savings: 2 out of 3 B_rows reused (67%) → 2 DRAM reads avoided. Total B reads drop from 9 (3 tiles × 3 rows avg) to 7 (initial 3 + 2 re-reads for non-reused + 2 evicted).
🍯
Insight: the 40% average reuse (Fig. 15) means roughly 2–3 out of every 5–8 B_rows are distance 1. For matrices with high spatial locality (filter3D, >90%), almost all B stays on-chip across tiles.

④ Hardware overhead breakdown

Avalanche uses 16 processing lanes, 2 MB Unified Cache, 8 comparators per merger. What's the total chip area and how does it compare?

  1. Multipliers: 16 × (scalar-vector unit) = 0.13 mm² (2.11%, Table 4).
  2. Mergers: 16 × (8-comparator insertion sort) = 0.17 mm² (2.76%).
  3. Fetcher + Job Allocator: 0.25 + 0.002 mm² = 0.252 mm² (4.09%).
  4. Unified Cache: 2 MB, 16-bank, 16-way set-assoc, 32 B lines. CACTI 7.0 estimates 5.6 mm² (91.05%) at 28 nm. Includes tag array (iMat, RMC bits add negligible overhead: 2 bits × 32k lines = 8 kB, <0.4% of cache).
  5. Total: 0.13 + 0.17 + 0.252 + 5.6 = 6.15 mm². Power: 3508 mW (97.78% from cache, Table 4).
  6. Comparison: the linear insertion-sort merger is area-efficient (2.76%) compared to complex comparator structures in prior work.
🌱
The 6.15 mm² is tiny for a 2 MB cache accelerator. For comparison, a 4 MB L2 cache in a CPU is ~10–15 mm² at 28 nm. Avalanche's logic overhead (multipliers, mergers, fetcher) is only 9%, showing excellent area efficiency.

Practice problems

Try each before opening the answer. Every one ends with the misconception it checks.

Problem 1 · easy
In Fig. 9, on-chip memory utilization averages 48%. If you naively double the cache size to 4 MB, will utilization drop to 24%? Why or why not?
Show answer
No. Utilization depends on the ratio of (actual intermediate C size) to (cache capacity). Doubling cache doubles capacity, but actual C size is fixed by the sparsity pattern and tiling plan. If the tiling algorithm re-runs with 4 MB, it makes larger tiles (more rows per tile), so C grows proportionally. Utilization stays roughly constant (~40–50%) because the tiling tries to fill the cache. What does improve is the number of tiles (fewer tiles → less B re-fetching), which is why Fig. 24a shows traffic dropping with larger cache.Misconception: that cache utilization is inversely proportional to cache size. It's actually driven by the tiling algorithm trying to maximize tile size within cache constraints.
Problem 2 · medium
A 4-row A splits into 2 tiles: {0–1}, {2–3}. Row 0 has non-zeros in columns {1,3}, row 1 in {0,2}, row 2 in {2,3}, row 3 in {1}. After Mat-Reorder for tile 1, what is the column processing order?
Show answer
Tile 1 includes rows 0–1. Their non-zeros: row 0 cols {1,3}, row 1 cols {0,2}. Combined: {0,1,2,3}. Sort by # of non-zeros descending: col 1 (1 nz), col 3 (1 nz), col 0 (1 nz), col 2 (1 nz) — all tied at 1. Break ties by row coverage: col 1 touches row 0, col 0 touches row 1. Prefer columns that finish rows earlier → process col 1 first (finishes row 0 partially), then col 3 (finishes row 0 fully), then col 0, then col 2 (finishes row 1). Order: {1, 3, 0, 2}. Row 0 is dead after col 3; row 1 dead after col 2.Misconception: that Mat-Reorder is global. It's per-tile — each tile gets its own reordered sequence based on which rows belong to it.
Problem 3 · medium
Why does evicting a live intermediate C product cost 3 memory ops (write, read, write), while evicting matrix B costs only 1 (read)?
Show answer
A live C product is dirty (modified) and will be accessed again. Evicting it: (1) write it to DRAM (dirty eviction), (2) later, when the next partial sum arrives, re-read it from DRAM to merge, (3) eventually it becomes dead and gets written again (final write-back). Total: 2 writes + 1 read = 3 ops. Matrix B is read-only — never modified. Evicting it: just free the line. Later, if needed again, (1) re-read from DRAM. Total: 1 read. Dead C has already been written (via DP-Evict), so evicting it is free (0 ops).Misconception: that all evictions cost the same. The cost depends on whether the data is dirty (modified) and whether it will be re-accessed. Avalanche's priority policy exploits this difference.
Problem 4 · hard
Fig. 15 shows 40% of B has reuse distance 1 on average, but filter3D is >90%. Construct a sparsity pattern where reuse is 0% (no B row is used in adjacent tiles).
Show answer
Make A block-diagonal with non-overlapping blocks. Example: A is 8×8, split into 2 tiles {rows 0–3}, {rows 4–7}. Each tile's non-zeros reference disjoint B_row ranges. Tile 1: rows 0–3 have non-zeros in cols {0,1,2,3} → uses B_rows {0,1,2,3}. Tile 2: rows 4–7 have non-zeros in cols {4,5,6,7} → uses B_rows {4,5,6,7}. No overlap → reuse distance is ∞ for all B_rows. Result: 0% reuse distance 1. Real matrices with this pattern: roadNet-CA (low reuse, Fig. 15) — road networks are spatially partitioned, tiles touch different geographic regions.Misconception: that high sparsity = high reuse. Reuse depends on spatial locality (how non-zeros cluster) not just density. Diagonal or block-structured matrices can have zero inter-tile reuse despite high sparsity.
Problem 5 · medium
The Job Allocator prevents RAW conflicts by ensuring no two lanes process the same row (AIDX) simultaneously. Why doesn't this cause deadlock if the queue is full of jobs for the same row?
Show answer
Because lanes process jobs sequentially and eventually free up. Example: queue has 256 jobs, all for row 0. Lane 0 takes job 1 (row 0). Lanes 1–15 see conflict → stall. Lane 0 finishes job 1, takes job 2 (row 0). Lanes 1–15 still stalled. Eventually lane 0 finishes all 256 jobs for row 0. No deadlock — work progresses, just serialized. Mat-Reorder increases this serialization (clusters rows), so Avalanche uses a 256-deep queue to buffer enough jobs for parallel dispatch. If all 256 are for the same row, throughput drops to 1/16th (only 1 lane active), but Fig. 24b shows performance scales linearly → conflicts are rare in practice.Misconception: that conflict avoidance = deadlock risk. Deadlock requires circular wait; here there's no circle — jobs are processed in queue order, and each eventually completes.
Problem 6 · hard
Applying Mat-Reorder and DP-Evict to GAMMA (row-wise dataflow) achieved 11% speedup (Fig. 25). Why is the gain smaller than Avalanche's 1.97× over SPAGHETTI?
Show answer
The 11% speedup from applying Mat-Reorder + DP-Evict to GAMMA's row-wise dataflow is a different comparison than Avalanche-MDR's 1.97× over SPAGHETTI. Avalanche-MDR is the full outer-product system with all three techniques (Mat-Reorder, DP-Evict, RM-Caching). The 11% gain is from adding Mat-Reorder + DP-Evict to the existing GAMMA row-wise architecture, which already has different bottlenecks. Row-wise has moderate C reuse (rows accumulated incrementally), so reducing C footprint helps less than in outer-product where C bloat is severe. The primary benefit in GAMMA comes from reduced B memory traffic through better A reordering, while Avalanche's outer-product gets larger gains by freeing cache space for B through aggressive dead-product eviction combined with RM-Caching.Misconception: conflating two different comparisons. The 11% is GAMMA vs GAMMA-MD (same dataflow, adding techniques). The 1.97× is Avalanche-MDR vs SPAGHETTI (different baseline architectures and dataflows).
Problem 7 · easy
Why does the paper use insertion sort for merging instead of a parallel merge network (e.g., bitonic sort)?
Show answer
Insertion sort is O(N) when input is already sorted, which it is (CSC/CSR store indices in order). Bitonic sort is O(N log N) always, better for random data but overkill here. Also, insertion sort scales linearly with comparators (8 comparators → 8-element merge per cycle), while bitonic requires a tree (log N stages → higher latency). Area cost: insertion sort is 2.76% (Table 4); bitonic would require more complex structures. Finally, insertion sort handles variable-width streams (1 new element + 4-element cache line) naturally; bitonic needs fixed-width inputs.Misconception: that more sophisticated algorithms are always better. Insertion sort's simplicity and O(N) on sorted data make it optimal for this workload.
Problem 8 · medium
The ROD bucket update (writing back entries for the next tile) adds ~4% memory traffic. Could you eliminate this by storing all tiles' RODs upfront?
Show answer
Yes, but storage explodes. Example: 32 tiles, each tile has ~10k column-row pairs (ROD entries), each ROD is 16 B (4 pointers × 4 B). Total: 32 × 10k × 16 B = 5.12 MB just for RODs. For comparison, the entire Unified Cache is 2 MB. Precomputing all RODs and storing them in DRAM would work, but fetching them adds traffic (5 MB read once vs 4% of ~20 MB total traffic = 0.8 MB written across tiles). The write-back approach trades one-time storage (small ROD writes per tile) for lower total traffic. Also, ROD updates allow adaptive reordering (e.g., if tile sizes change dynamically based on runtime feedback), which precomputation can't do.Misconception: that eliminating a cost is always beneficial. The 4% ROD write traffic is dwarfed by the overall traffic savings (total off-chip memory traffic reduced to 0.47×, Fig. 22), so it's a good trade. Precomputation would save the 4% but cost more in upfront storage and fetch traffic.
Quick answers

Frequently asked questions

Twelve questions newcomers ask most.

Q1What does "outer-product dataflow" actually mean?▾
Take one column of A (a vector) and one row of B (a vector), compute their outer product (matrix), and add it to C. Repeat for all column-row pairs. The key: A and B are each read once (perfect reuse), but C accumulates partial sums from many pairs (low reuse, memory bloat). Contrast with inner-product (dot each row of A with each column of B — poor A/B reuse) or row-wise (each row of A with all rows of B — moderate C reuse, poor B reuse).
Q2How does tiling "break" B locality?▾
Without tiling, you process all columns of A once → each row of B is read once (perfect). With tiling, you process tile 1's columns (rows 0–k of A), then tile 2's columns (rows k+1–m), etc. Each tile multiplies its A rows by all rows of B. So B is re-read once per tile. If A splits into 32 tiles, B traffic increases significantly. The paper's Fig. 7 shows this traffic rising 5.9× on average with max 14× for some matrices.
Q3Why can't LRU handle this on its own?▾
LRU treats all cache lines equally — it evicts the least-recently-used one, regardless of whether it's live C (will be re-accessed, costly to evict) or dead C (never re-accessed, free to evict) or B (reusable next tile, medium cost to evict). With all three types mixed in cache, LRU causes contention: it might evict a live C product or reusable B row just because it wasn't hit in the last N accesses, even though keeping it would save 3 memory ops. Avalanche's priority policy explicitly differentiates them.
Q4What is "reuse distance" and why only track distance 1?▾
Reuse distance = # of tiles between when B row k is used and when it's next used. Distance 1 means the row is reused in the immediately following tile (highest locality). The paper found ~40% of B has distance 1 (Fig. 15). Why not track distance 2, 3, ...? Diminishing returns: distance 2 might add another 10–15% coverage, but requires 2 bits per job (or a more complex scheme). Distance 1 captures the bulk of the benefit with minimal overhead (1 bit).
Q5How much overhead does the ROD bucket add?▾
ROD reads are free — they replace the ptr array reads in standard CSC/CSR (same data structure size). ROD writes (updating the bucket between tiles) add ~4% of total memory traffic (paper §5.2). For a workload with 20 MB total traffic, that's 0.8 MB. Compare to the overall traffic reduction to 0.47× baseline (10+ MB saved, Fig. 22) — the trade is heavily favorable. The ROD bucket itself occupies a small reserved region in DRAM (a few MB for large matrices), negligible vs the matrices themselves (100s of MB).
Q6Does Mat-Reorder work for all sparse patterns?▾
Not perfectly. A perfect lower-triangular matrix (all non-zeros below the diagonal) ensures products finish strictly top-to-bottom. Real sparse matrices are irregular — Mat-Reorder approximates by sorting columns by length and clustering. For some matrices (e.g., roadNet-CA, which is spatially partitioned), clustering is weak → less benefit. For others (ca-CondMat, 2cubes_sphere), clustering is strong → products finish more sequentially (Fig. 14 shows an average 34% reduction in live elements). The gain varies but is positive on average.
Q7Why does Avalanche beat GAMMA (row-wise) by only 1.2× vs 1.97× over SPAGHETTI?▾
GAMMA already has moderate C reuse (row-wise accumulates rows incrementally), so Mat-Reorder + DP-Evict add less value than for OutP (where C bloat is severe). Also, GAMMA's bottleneck is B (poor reuse in row-wise), and Avalanche's RM-Caching is designed for OutP's tiling — it doesn't map cleanly to row-wise. The 1.2× comes from the overall superior outer-product architecture with all three techniques. The 11% gain (Fig. 25) from applying Mat-Reorder + DP-Evict to GAMMA's row-wise shows the techniques are dataflow-dependent and primarily reduce B memory traffic through better A reordering in that context.
Q8What happens if the cache is too small to hold any B rows?▾
RM-Caching degrades gracefully. If intermediate C for a tile fills the entire cache, there's no space for B → all B is fetched from DRAM every tile (same as baseline). The priority policy still helps by preventing live C from being evicted (reducing the 3× cost), but the B traffic savings are zero. Fig. 24a shows this: at 256 kB cache, performance is poor because C doesn't fit → thrashing. At 2 MB, C + some B fit → big win. At 4 MB, almost all B fits → diminishing returns (converges).
Q9Can Mat-Reorder be computed at runtime?▾
Section 5.2 describes Mat-Reorder as "dynamic for each tile" — the ROD bucket is software-controlled, stored off-chip, read sequentially by hardware, and updated for subsequent tiles. While the reordering logic itself is not performed by the hardware during execution, the approach is adaptive per tile rather than a single static offline preprocessing. The software component analyzes the sparsity pattern and generates the per-tile reordering, which hardware then follows.
Q10Why does the paper use 16 processing lanes and 8 comparators?▾
Design space exploration (Fig. 24b–c). Performance scales linearly with # lanes up to ~32–64 (Fig. 24b), beyond which memory bandwidth saturates. 16 is a sweet spot: 9% of chip area (Table 4), good parallelism, minimal conflicts with a 256-deep job queue. For comparators, 8 merges a cache line of 4–8 elements in 1–2 cycles; more comparators (16, 32) show diminishing returns (Fig. 24c converges at 32). 8 is cost-effective.
Q11Does Avalanche require changes to software?▾
Yes — the ROD bucket must be generated. A software tool analyzes A's sparsity, determines the tiling plan, reorders columns per tile, and writes the ROD bucket to DRAM. At runtime, the accelerator reads A, B, and the ROD bucket from DRAM — no further software involvement during execution. This is preprocessing per matrix (or per tiling configuration if A's size changes), with the ROD bucket updated dynamically between tiles as described in Section 5.2.
Q12What are potential future directions?▾
Possible extensions include: extending RM-Caching to track reuse distance >1 for broader B coverage; adaptive dataflow selection based on sparsity patterns; applying the techniques to other sparse operations like graph processing; multi-tile parallelism with multiple accelerators; and supporting dynamic sparsity where matrices change at runtime.

Glossary & further reading

Every term in one place, then paths to go deeper.

Core terms
AvalancheThe paper's SpMM accelerator using Mat-Reorder, DP-Evict, and RM-Caching to optimize on-chip memory for matrices B and C.
OutP (outer-product)Dataflow: multiply column of A by row of B → partial matrix, merge into C. High A/B reuse, low C reuse.
Tiling (horizontal)Splitting A into row-wise tiles so each tile's intermediate C fits on-chip. Fixes C bloat but re-fetches B every tile.
Live productAn intermediate C element still awaiting more partial sums from future column-row pairs. Costly to evict (3 memory ops).
Dead productA C element fully computed; no more merges will touch it. Safe to evict immediately (1 write).
Mat-ReorderDynamically reordering A's columns for each tile into a lower-triangular form to cluster each row's non-zeros → products finish early, sequentially.
DP-EvictDead-Product Early Eviction: detecting dead products (last column-row pair for a row) and evicting them immediately to DRAM.
RM-CachingReuse Distance-Aware Matrix Caching: marking B rows with reuse distance 1 (used next tile) to keep on-chip.
Unified CacheA 2 MB cache storing both B and intermediate C, with priority bits (iMat, RMC) to prevent evicting live/reusable data.
ROD bucketReorder Operand Descriptor: a DRAM structure storing column-row pairs in Mat-Reordered sequence, read by Matrix Fetcher and updated for each tile.
Supporting terms
CSC / CSRCompressed Sparse Column/Row formats: store non-zeros with ptr (column/row starts), idx (row/column indices), data (values).
JobOne scalar-vector multiply: AIDX (row of A), ADATA (value), SPB/EPB (B row pointers), D/R bits.
Job AllocatorDistributes jobs to processing lanes while preventing RAW conflicts (two lanes accessing same C row simultaneously).
Processing LaneA multiplier + merger pair; Avalanche has 16, each handling one scalar-vector multiply per cycle.
Insertion sort mergerMerges one new partial product into a sorted cache line using linear comparators (O(N) when input is sorted).
iMat bit1 = cache line holds live intermediate C (high priority to keep); 0 = holds B or dead C.
RMC bit1 = cache line holds B row reused next tile (medium priority); 0 = non-reusable B or dead C.
Reuse distance# of tiles between when B row k is used and next used. Distance 1 = used in adjacent tiles (~40% of B on avg).
Matrix FetcherReads ROD bucket sequentially, fetches A columns and B row pointers, generates jobs for Job Allocator.
RoW (row-wise inner-product)Dataflow: each row of A dotted with all rows of B. Moderate C reuse, poor B reuse (GAMMA uses this).

SPAGHETTI: Streaming Accelerators for SpGEMM on FPGAs hard

Hojabr et al. · HPCA 2021

The baseline OutP accelerator Avalanche improves on. Introduces horizontal tiling for OutP but doesn't optimize on-chip memory for B. Reading this first helps appreciate Avalanche's delta (Mat-Reorder, DP-Evict, RM-Caching).

Zhang et al. · ASPLOS 2021

The baseline row-wise accelerator. Uses FiberCache for B. Avalanche applies Mat-Reorder + DP-Evict to GAMMA in §6.7, achieving 11% gain — shows the techniques generalize across dataflows.

OuterSPACE: An Outer Product SpMM Accelerator medium

Pal et al. · HPCA 2018

Early OutP accelerator, identifies memory bloating problem but no tiling. Good introduction to why OutP is attractive (high A/B reuse) and why C bloat is the challenge.

SpArch: Efficient Architecture for Sparse Matrix Multiplication hard

Zhang et al. · HPCA 2020

OutP with "matrix condensing" (merge partial matrices eagerly to reduce bloat). Avalanche's DP-Evict is similar in spirit but more aggressive (evict dead products immediately, not just merge them).

Kim et al. · MICRO 2023

Introduces the ROD (Runtime Operand Descriptor) concept Avalanche reuses for Mat-Reorder. HARP uses ROD for dynamic scheduling; Avalanche uses it for static reordering. Understanding HARP's ROD design clarifies Avalanche's §5.2.
