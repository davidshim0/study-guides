MICRO 2025 · A Field Guide

Three bits that matter, instead of eight that mostly don't

LLM inference is starved by weight/KV cache memory bandwidth and crushed by GEMM compute. This is the plain-English story of MCBP, a bit-slice accelerator that addresses all three bottlenecks at once.

📄 "MCBP" · Wang et al. (Tsinghua/SJTU) 🎓 Assumes basic LLM inference knowledge ⏱ ~30 min read
🧮

Decompose INT8 weights into 8 bit-slices, then exploit the repetition within each slice (BRCR), the sparsity across high-order slices (BSTC), and early-termination bit prediction (BGPP) — instead of treating every value as an opaque 8-bit atom.

Before you begin

Ten concepts MCBP builds on

Each primer clarifies one idea the paper assumes. Read in order the first time, or jump to whatever looks unfamiliar.

bit-repetition (BRCR) bit-sparsity (BSTC) bit-prediction (BGPP) value-level (baseline)
1LLM inference: prefill and decoding▾

LLM inference has two phases. Prefill processes the entire input prompt in parallel, computing Key/Value (KV) tensors for each token and storing them in the KV cache. Decoding then generates tokens one at a time (autoregressive), reusing the cached KV tensors to avoid recomputation.

Prefill is compute-bound — large matrix multiplies dominate. Decoding is memory-bound — each new token requires loading the full model weights (e.g., 7B × 8bit = 7GB per pass) and the accumulated KV cache (grows with sequence length).

Now you're ready: MCBP optimizes both stages — BRCR cuts prefill GEMM compute by ~5.7×, BSTC cuts decoding weight access by ~2.2×, and BGPP cuts KV cache access by ~1.5×.
2INT8 quantization and bit representation▾

Modern LLMs run in INT8 precision (8-bit integers) with minimal accuracy loss (<1%). Each weight is an 8-bit value, e.g., 01101001 = 105. Conventionally, hardware treats this as an atomic 8-bit value.

MCBP instead decomposes it into 8 bit-slices: the MSB (most significant bit), then bits 7, 6, …, down to the LSB (least significant). The 8-bit GEMM becomes a weighted sum of 8 bit-slice GEMMs, shifted by powers of two.

Now you're ready: decomposing into bit-slices preserves full numeric equivalence (lossless) while exposing fine-grained sparsity and repetition invisible at the value level.
3Why bit-slices reveal more sparsity▾

A value is zero only if all 8 bits are zero simultaneously. But quantized weights follow a near-Gaussian distribution — most are near zero, meaning their high-order bits (bits 7–3) are often zero even when the value itself is not.

For example, a 2-bit matrix with 6 zero values decomposes into an MSB slice with 14 zeros (70% sparsity) and an LSB slice with 4 repeated columns. The paper measures 10.1× higher sparsity at bit-slice level vs. value level (Fig. 5(d)).

Now you're ready: BSTC exploits this bit sparsity in high-order slices via a two-state coding (zero vs. non-zero), and BRCR exploits repetition within each slice.
4Bit-serial computing and early termination▾

Bit-serial computing processes one bit at a time (MSB → LSB), accumulating partial products via shifts and adds. This naturally aligns with bit-slice sparsity — you can stop early (e.g., skip the LSBs if the MSBs already exceed a threshold).

MCBP uses this property for BGPP's progressive prediction: process Keys bit-wise (MSB first), and terminate once a Key's partial score falls outside the top-𝑘 range, avoiding redundant loads and computation.

Now you're ready: BGPP's progressive prediction terminates bit-wise once a Key's partial score falls outside the top-𝑘 range, avoiding redundant loads and computation.
5Top-𝑘 attention sparsity prediction▾

Attention scores are often sparse — many Key-Query pairs yield near-zero attention weights after softmax. Top-𝑘 prediction estimates the attention matrix with low overhead (e.g., 4-bit MSB only), sorts to pick the 𝑘 highest-scoring Keys, then computes full-precision QKT and PV using only those selected Keys.

However, value-based top-𝑘 is inefficient: it loads all bits of all Keys upfront, then computes the full 8-bit product before deciding. This redundant I/O becomes a bottleneck.

Now you're ready: BGPP processes Keys bit-by-bit (MSB first), filtering out trivial Keys after each bit round — cutting KV cache accesses by ~50% compared to value-level top-𝑘 (Fig. 5(g)).
6Pigeonhole principle and group size▾

The pigeonhole principle: if you have more items than containers, at least one container holds multiple items. For a binary vector of length 𝑚, there are 2𝑚 possible distinct values. If the matrix has far more than 2𝑚 columns (e.g., LLM hidden dim 𝐻 = 4096 ≫ 24 = 16), repetition is guaranteed.

MCBP groups weight columns into groups of size 𝑚 = 4. This keeps 2𝑚 small enough for hardware matching (CAM-based search) while ensuring high repetition rates. If 𝑚 is too large, repetition drops and matching overhead explodes.

Now you're ready: BRCR partitions each bit-slice matrix into 𝑚=4-row groups, identifies repetitive column vectors within each group via CAM, and merges redundant computations — achieving up to 12.1× compute reduction (vs. value sparsity 3.8×).
7Value-level vs. bit-level compression▾

Traditional sparsity accelerators compress at the value level — store only non-zero 8-bit values with indices (e.g., CSR format). This works well for high value-sparsity but fails for LLMs, where value sparsity is only ~7% (Llama13B) vs. ~70% bit sparsity in high-order bit-slices.

Worse, value-level storage requires bit-reordering for bit-serial PEs: fetch an 8-bit value, extract the MSB, discard the rest; repeat for the next bit. This overhead kills efficiency (Fig. 5(c)).

Now you're ready: BSTC compresses along the bit-slice dimension (bit-first storage), naturally aligned with bit-serial computation. It achieves ~2.2× throughput gain in the ablation with no reorder overhead.
8CAM (Content-Addressable Memory)▾

A CAM is a hardware lookup table that searches all entries in parallel for a matching key, returning the matching address(es) in one cycle. Unlike RAM (address → data), CAM does (data → address).

MCBP uses a 512B CAM to match 4-bit column vectors from a bit-slice group matrix against all possible 24 = 16 patterns simultaneously. This identifies repetitive columns in one cycle, drastically cutting the search latency that would bottleneck a serial scan.

Now you're ready: the CAM-based match unit is the core of BRCR's fast repetition identification, enabling 2.88× hardware throughput gain (Fig. 21(a)) over naive software merging.
9Sign-magnitude (SM) format▾

INT8 weights can be stored in two's complement (standard) or sign-magnitude (1 sign bit + 7 magnitude bits). SM has a key advantage for bit sparsity: the sign bit is independent, and the magnitude bits for small values (near zero) have many leading zeros.

MCBP uses SM format to maximize sparsity in high-order magnitude bits. For example, bit-slice 7 (high magnitude bit) has ~90% sparsity in LLaMA7B (Fig. 8(c)), enabling aggressive BSTC compression on bits 3–7 while leaving bits 1–2 uncompressed (low sparsity).

Now you're ready: SM format is a preprocessing step — weights are offline-converted to SM, then BSTC compresses the sparse magnitude slices. The sign bit is handled separately during computation.
10BSTC break-even point▾

BSTC's two-state encoding adds overhead (1-bit indicator per column). The compression ratio (CR) becomes positive (CR > 1) only when the sparsity ratio exceeds a threshold. For group size 𝑚=4, the break-even point is SR ≈ 65% (from the paper's Fig. 8(b)).

Bit-slices with SR below 65% (e.g., bits 1–2 with SR 16–30%) are left uncompressed because the indicator overhead would exceed the compression gain. Bits 3–7 have SR 65–90% and are compressed, achieving positive benefit.

Now you're ready: BSTC selectively compresses only high-sparsity slices (bits 3–7), achieving ~2.2× throughput gain without introducing overhead on low-sparsity slices.
The whole story in one place

MCBP in a nutshell

The problem, the three-pronged solution, and the results.

LLM inference faces three bottlenecks (Fig. 1(a)): (1) GEMM computation in prefill (compute-bound), (2) weight memory access in decoding (7GB per token for LLaMA7B), and (3) KV cache access in decoding (grows with sequence length). Existing accelerators address one or two but not all three, and they process data at the value level, missing the fine-grained opportunities hidden at the bit level.

MCBP decomposes INT8 weights into 8 bit-slices (MSB to LSB), then applies three coordinated optimizations:

1) BRCR (Bit-Slice Repetitiveness-enabled Computation Reduction) partitions each bit-slice matrix into 𝑚=4-row groups, identifies repetitive column vectors within each group via CAM-based matching, and merges their activations once — eliminating redundant adds. For typical LLMs (𝐻=4k, bit sparsity ~70%), BRCR achieves up to 12.1× compute reduction vs. value sparsity 3.8× (§3.1).

2) BSTC (Bit-Slice Sparsity-enabled Two-State Coding) exploits the fact that quantized weights follow a near-Gaussian distribution — high-order bit-slices (bits 3–7) are ~65–90% sparse. BSTC compresses these slices with a simple two-state encoding (0 → '0', non-zero → '1' + data), aligned with bit-serial computation. The ablation reports a ~2.2× throughput gain from BSTC (§3.2).

3) BGPP (Bit-Grained Progressive Prediction) refines top-𝑘 attention sparsity prediction by processing Keys bit-by-bit (MSB first). After each bit round, it filters Keys whose partial scores fall below a radius-based threshold, terminating early and avoiding redundant KV cache loads. This cuts KV cache accesses by ~50% vs. value-level top-𝑘 (Fig. 5(g), §3.3).

Hardware support: A CAM-based BRCR unit with 160 PEs (organized into 20 clusters) accelerates repetition matching. Lightweight BSTC encoders/decoders (parallel-friendly) handle bit-slice compression. A clock-gated BGPP unit with 64 bit-serial inner-product engines performs progressive filtering. The ablation reports CAM adds 25% area to the BRCR unit, BSTC adds 16% area, and BGPP adds 9% area; CODEC alone is 6.2% area (§4).

Results (§5): On 26 benchmarks (LLaMA7/13B, Bloom1B7, OPT1B3, Qwen7B), MCBP achieves 9.43× speedup and 31.1× energy efficiency over NVIDIA A100 GPU, and 35×, 5.2×, 3.2× higher efficiency than SOTA accelerators SpAtten, FACT, SOFA. Area: 9.52mm² @ 28nm, 1GHz. Power: 2.395W (core 1.26W, HBM 1.14W). Accuracy loss: <1% under standard quantization, 0% under BRCR/BSTC (lossless), MBPP drops noticeably when αᵣ < 0.6, MMLU degrades significantly only when αᵣ < 0.5 (BGPP default αᵣ ∈ [0.5, 0.6]).

🍯 Key takeaways

  • Bit-slice decomposition exposes 10.1× higher sparsity and abundant repetition invisible at the value level.
  • BRCR groups 𝑚=4 rows per bit-slice, uses CAM for 1-cycle repetition matching, merges activations → 12.1× compute reduction vs. value sparsity 3.8×.
  • BSTC applies two-state coding to high-order slices (bits 3–7) → 2.2× throughput gain, aligned with bit-serial PEs (no reorder).
  • BGPP processes Keys bit-wise (MSB first), filters after each round → 50% fewer KV cache accesses vs. value top-𝑘.
  • All three are co-designed: same granularity (𝑚=4), seamless dataflow, minimal overhead.
  • CAM (512B) adds 25% area to BRCR, BSTC adds 16%, BGPP adds 9%, CODEC 6.2% — efficient hardware.
  • Result: 9.43× faster, 31.1× more energy-efficient than A100 GPU; 3.2–35× better than SOTA Transformer accelerators.
Seven everyday pictures

How to think about MCBP

Each analogy stands alone — read them in any order. Every card ends with an honest note about where the picture breaks down.

Core idea · Bit-slice decomposition

Unpacking a suitcase

An 8-bit value is like a packed suitcase — you can't see what's inside until you unzip it. MCBP unpacks it into 8 layers (bit-slices), each revealing its own pattern. The top layers (MSB) are mostly zero (sparse), and within each layer, items often repeat. By operating on the unpacked layers instead of the closed suitcase, MCBP exploits sparsity and repetition that value-level processing never sees.

Where it breaks: unlike a suitcase, bit-slices aren't physically separate — they're still stored together. MCBP's "unpacking" is a logical decomposition (via bit-serial processing), not a physical memory split.
BRCR · Repetition merging

Bulk order discount

Three customers order identical sandwiches (columns 1 & 2: both "0101"). The deli makes one sandwich, then splits it — faster and cheaper than making two separately. BRCR identifies repetitive column vectors within a bit-slice group (e.g., both have the same 4-bit pattern "0101"), computes the activation sum once, then reuses it. The CAM is the deli's instant recipe lookup — finds all matches in one cycle.

Where it breaks: the deli analogy assumes zero cost for splitting. In hardware, merging has overhead (CAM matching, index translation, routing). MCBP's 𝑚=4 group size balances repetition gains vs. merging overhead.
BSTC · Bit sparsity

Empty shelves in a warehouse

A warehouse has 8 floors (bit positions). The top 5 floors (bits 3–7) are mostly empty (~70% vacant), while the bottom 3 are packed. BSTC compresses only the top floors with a simple code: empty shelf → '0', occupied → '1' + item. The bottom floors stay uncompressed (not worth the overhead). This cuts storage by 2.2× without losing any items (lossless).

Where it breaks: real warehouses store physical goods; bit-slices are abstract. The "compression" isn't moving items, it's encoding the bit pattern differently (two-state: zero vs. non-zero). Also, the compression is applied offline (pre-deployment), not dynamically during inference.
BGPP · Progressive prediction

Opening a mystery box one layer at a time

A mystery box has 8 layers of wrapping (bit positions). You're looking for boxes worth $75+ (top-𝑘 threshold). After unwrapping the first 2 layers (MSB bits), you can already see this box is worth ≤$31 (partial score). Stop unwrapping — it can't possibly reach $75. BGPP does this with Keys: process MSB first, filter out trivial Keys after each round, avoid loading their remaining bits. This cuts KV cache accesses by ~50%.

Where it breaks: the mystery box analogy assumes the outer layers reveal the total value. In BGPP, each bit adds more information (partial score), but the final score isn't known until all bits are processed. The threshold is dynamic (radius-based), not fixed at $75.
Hardware · CAM-based matching

Phone book with instant search

A phone book lists 16 names (24 possible 4-bit patterns for 𝑚=4). Instead of flipping pages (serial search), you shout a name and the book lights up all matching entries instantly (parallel CAM search). MCBP's CAM stores all 16 possible column patterns; for each column in the bit-slice group, it queries the CAM and gets back the matching pattern(s) in one cycle. This eliminates the serial-search bottleneck that would kill BRCR's efficiency.

Where it breaks: a phone book is read-only; CAM is writable (but MCBP's CAM is pre-loaded per bit-slice group and doesn't change during that group's processing). Also, CAM has power overhead (~47% of BRCR unit power, Fig. 22(b)) — the instant search isn't free.
Data layout · Bit-slice-first storage

Books sorted by genre vs. by author

A library can sort books by author (value-level: all bits of value V together) or by genre (bit-slice-level: all MSBs together, then all bit-7s, etc.). If you want to read all mystery books (MSB slice), author-sorting forces you to visit every shelf (bit-reordering overhead). Genre-sorting gives you all mysteries in one aisle. BSTC stores weights bit-slice-first — the MSB slice is contiguous in memory, so bit-serial PEs fetch it directly without reordering.

Where it breaks: the library analogy assumes physical books; MCBP's "bit-slice-first" is a logical memory layout (HBM banks store groups × bits together, §4.2). Also, value-first isn't inherently bad — it's just mismatched with bit-serial processing. For value-parallel hardware (e.g., systolic arrays), value-first is optimal.
Co-design · Three techniques, one foundation

Three legs of a stool

A stool needs all three legs to stand. MCBP's three techniques (BRCR, BSTC, BGPP) target different bottlenecks (compute, weight, KV cache) but share the same bit-slice foundation and 𝑚=4 granularity. BRCR groups 4 rows of a bit-slice; BSTC compresses bit-slices at 𝑚=4 granularity; BGPP processes Keys bit-wise. This coordination eliminates data-format mismatches and maximizes overall system benefit. Remove any leg (e.g., only BRCR without BSTC) and the stool wobbles — decoding weight traffic becomes the new bottleneck.

Where it breaks: the stool analogy implies the three legs are equal contributors. In practice, BRCR delivers the largest single gain (5.7× on compute), then BSTC (2.2× on memory), then BGPP (1.5× on KV cache). But without all three, the overall speedup (9.43×) wouldn't be achieved.
The full explanation

How it actually works

Section by section, with the paper's key mechanisms broken down step-by-step.

1 · The three bottlenecks — and why value-level processing misses them

LLM inference latency is dominated by three factors (Fig. 1(a)): (1) GEMM computation in the prefill stage (e.g., 55% for Dolly 1k-prompt), (2) weight memory access in decoding (52.4% for short prompts), and (3) KV cache access in decoding (grows with sequence length). Existing accelerators address one or two but not all three, and they operate at the value level — treating each INT8 value as an opaque 8-bit atom. This obscures two key opportunities:

  • Bit sparsity: quantized weights follow a near-Gaussian distribution; most values are near zero, so their high-order bits (bits 7–3) are often zero. A 2-bit matrix with 30% value sparsity (6/20 zeros) decomposes into an MSB slice with 70% sparsity (14/20 zeros) (Fig. 4(a)). The paper measures 10.1× higher sparsity at bit-slice level vs. value level (Fig. 5(d)).

  • Bit repetition: for a binary vector of length 𝑚, there are 2𝑚 possible distinct values. If the matrix has 𝐻 ≫ 2𝑚 columns (e.g., LLM hidden dim 4096 ≫ 24 = 16), the pigeonhole principle guarantees repetition. The paper shows a 2-bit matrix with 0 repeated columns decomposes into an LSB slice with 4 repeated pairs (Fig. 4(a)).
🔬
Why value-level compression fails: conventional sparse formats (CSR, COO) compress at the value level, exploiting only value sparsity (~7% in LLaMA13B). Even worse, they require bit-reordering for bit-serial processing: fetch an 8-bit value, extract the MSB, discard the rest; repeat for the next bit. This overhead (17% of latency, Fig. 20(c)) negates the benefit.

2 · BRCR — turning repetition into compute reduction

BRCR exploits bit-slice repetition via three steps: (1) Decompose the 𝑘-bit weight matrix into 𝑘 bit-slice matrices. (2) Group each bit-slice matrix into 𝑚-row groups. (3) Merge repetitive column vectors within each group, then reconstruct the output. The key insight: for a group matrix G ∈ {0,1}𝑚×𝐻, there are at most 2𝑚 distinct column patterns. If 𝐻 ≫ 2𝑚, many columns repeat.

The CAM-based match unit (Fig. 14) stores all 2𝑚 possible column patterns (for 𝑚=4, that's 16 patterns) and searches in parallel. For each column in the group matrix, it queries the CAM and gets back the matching pattern(s) in one cycle. This eliminates the serial-search bottleneck. The merged activations (MAV) are then sent to the reconstruction unit, which applies a fixed adder tree to compute the final output (Fig. 14 right).

🍯
Why 𝑚=4? Smaller 𝑚 increases repetition (pigeonhole principle) but adds more reconstruction overhead (2𝑚-1 adds per row). Larger 𝑚 reduces reconstruction but lowers repetition rate. Fig. 18 shows 𝑚=4 hits the sweet spot: max compression rate (~3.5×) and compute reduction (5–7×) with manageable hardware overhead (512B CAM, 16 index converters, 16 AMUs).

3 · BSTC — exploiting bit sparsity for compression

BSTC compresses high-order bit-slices (bits 3–7) using a simple two-state encoding: zero columns → '0', non-zero → '1' + data. The key insight: quantized weights in sign-magnitude (SM) format exhibit ~65–90% sparsity in bits 3–7 (Fig. 8(c)). For example, in LLaMA7B, bit-slice 7 (high-order magnitude bit) has ~80% sparsity; bit-slice 3 has ~65% sparsity. Bits 1–2 have low sparsity (16–30%) and are left uncompressed.

The two-state code is aligned with bit-serial computation: the decoder reads the encoded bit stream, outputs '0000' for a zero entry, or buffers '1' + 4-bit data in a SIPO register. This avoids bit-reordering overhead (Fig. 15). The compression ratio (CR) depends on sparsity ratio (SR) and group size 𝑚. Fig. 8(b) shows that positive benefit (CR > 1) occurs when SR exceeds ~65%. The paper reports that BSTC achieves a ~2.2× throughput gain in the ablation (§3.2).

CR = 𝑚 × 𝐻 / (num_zeros × 1 + num_nonzero × (𝑚+1))
# where num_zeros + num_nonzero = 𝐻 (hidden dimension)
# For SR=70%, 𝑚=4, CR ≈ (4×𝐻) / (0.7𝐻×1 + 0.3𝐻×5) ≈ 1.82×
💡
Why two-state instead of Huffman? Huffman coding achieves higher CR but requires variable-length decoding (serial bottleneck) and isn't aligned with bit-serial PEs. Two-state is lightweight (5-bit decoder, Fig. 15(b)), parallel-friendly (each decoder handles one sub-matrix independently), and naturally fits bit-slice-first storage (§4.2).

4 · BGPP — bit-grained progressive top-𝑘 prediction

BGPP refines the standard top-𝑘 attention sparsity prediction (§2.2) by processing Keys bit-by-bit (MSB first) and filtering trivial Keys after each round. The key insight: if a Key's partial score after 𝑟 bits already falls below the threshold 𝜃𝑟, the final score (after all 8 bits) is guaranteed to stay below it — so stop early and avoid loading the remaining bits.

The threshold is dynamic, based on the current max value and a radius (default 3, tunable via 𝛼𝑟 ∈ [0,1]):

𝜃𝑟 = max( max( Â𝑟 ) − 𝛼𝑟 × radius, 0 )
# Â𝑟 = partial attention scores after round 𝑟
# Smaller 𝛼𝑟 → more aggressive pruning (higher sparsity, lower accuracy)

As illustrated in Fig. 9, the first round loads the MSB of all Keys, computes partial scores, and filters Keys below 𝜃1. The second round loads the next bit of only the surviving Keys, updates the scores, and filters again. This continues for a preset number of rounds (e.g., 3–4 rounds for 8-bit Keys). The paper shows ~50% reduction in KV cache accesses vs. value-level top-𝑘 (Fig. 5(g)).

⚠
Accuracy trade-off: BGPP introduces a hyperparameter 𝛼𝑟 that controls pruning aggressiveness. The paper states that MBPP drops noticeably when αᵣ < 0.6, MMLU degrades significantly only when αᵣ < 0.5, and sets αᵣ in the range [0.5, 0.6]. Fig. 24(a) shows the accuracy vs. sparsity trade-off. For reasoning tasks (MMLU), the model tolerates more pruning (up to 𝛼=0.5) before accuracy drops; for generation tasks (MBPP), it's more sensitive (threshold around 𝛼=0.6).

5 · Hardware co-design — CAM, BSTC CODECs, clock-gated BGPP

The three techniques are supported by custom hardware units (§4):

  • CAM-based BRCR unit (Fig. 14): 20 PE clusters (160 PEs total), each cluster with a 512B CAM (stores 16 × 4-bit patterns), 16 index converters, 16 addition merge units (AMUs), and 1 reconstruction unit (RU) time-multiplexed across 16 AMUs. The CAM enables 1-cycle repetition matching; the RU uses a fixed adder tree (data path optimized for 𝑚=4) to minimize switching activity (§4.3). The ablation reports CAM adds 25% area to the BRCR unit.
  • BSTC encoder/decoder (Fig. 15(a)(b)): lightweight 5-bit decoder (1 CMP, 1 SIPO register, 1 leading-one eliminator) per sub-matrix. Encoder (offline, not shown) uses a 4-bit CMP and MUX. The decoders are parallel-friendly (each bank has one decoder) and add 16% area in the ablation; CODEC alone is 6.2% area (Fig. 22).
  • Clock-gated BGPP unit (Fig. 16): 64 bit-serial inner-product engines (64-input AND-based adder trees), a threshold updating (TU) module that serially identifies Max/Min, and a clipping module that compares attention entries with the threshold. If the threshold falls below the observed Min, the clipping module is clock-gated and BGPP skips to the next round. The ablation reports BGPP adds 9% area (Fig. 22).
🍯
Total overhead: CAM (512B × 20 clusters = 10kB), BSTC CODECs (20×4 decoders), BGPP unit (64 engines) contribute to the overall system area (9.52mm² @ 28nm, Fig. 22(a)). Power breakdown: BRCR 44.7%, BGPP 8.2%, BSTC 10.2%, SRAM 22%, scheduler 4.1%, APU 11.7% (Fig. 22(b)). Critically, none of these units sit on the critical path — the BRCR/BSTC units run in parallel with the BGPP predictor, and the decoders run in parallel with data fetching.

6 · Dataflow and tiling strategy

MCBP uses an output-stationary dataflow (Fig. 12): each PE cluster computes a 𝑇M×𝑇N output tile (here 𝑇M=64, 𝑇N=32) by accumulating over 𝐾/𝑇K iterations (𝑇K=256). The 8 PEs in a cluster process 8 bit-slices in parallel (bit-slice parallelism). Weight tiles are fetched from on-chip SRAM (768kB weight SRAM, 2×16×8kB banks, Fig. 13); activations are fetched from 384kB token SRAM. The bit-slice-first memory layout (Fig. 13) ensures that fetching the MSB slice doesn't drag in the LSBs (no reordering).

The BSTC decoder sits between HBM and weight SRAM: compressed weights are loaded from HBM, decompressed on-the-fly, and written to weight SRAM. The decoder is pipelined with the BRCR unit, so decompression latency is hidden. The data fetcher dispatches weight/activation tiles to the BRCR unit, which performs the 3-step BRCR computation (merge → accumulate → reconstruct) and writes results back to output SRAM. The BGPP unit runs concurrently, prefetching KV indices for the next attention layer.

Follow the numbers, step by step

Worked examples

Four traces using the paper's real data — repetition merging, compression ratio, bit-grained prediction, and end-to-end speedup.

① BRCR: merging repetitive activations (LLaMA7B, m=4)

A 4-row bit-slice group matrix (𝑚=4) has 𝐻=4096 columns. How many repetitive columns exist, and how much compute does BRCR save?

  1. There are 24=16 possible column patterns (0000 to 1111). If 𝐻=4096 ≫ 16, the pigeonhole principle guarantees repetition. On average, each pattern appears 4096/16 = 256 times.
  2. Naïve bit-serial GEMV (no repetition merging) requires 4096 × (1 − 𝑏𝑠) additions per bit-slice, where 𝑏𝑠 ≈ 0.70 (bit sparsity). So 4096 × 0.3 = 1228.8 adds per slice.
  3. BRCR Step 1 (merge): for each of the 16 patterns, merge all activations that match it. This requires 𝐻 × (1 − 𝑏𝑠) = 4096 × 0.3 = 1228.8 adds (same as naïve, but now the activations are grouped).
  4. BRCR Step 2 (reconstruct): multiply the 16×1 merged activation vector (MAV) with the 4×16 Enumeration matrix. Each row of the Enum matrix has at most 23=8 ones, so 4 × 8 = 32 adds.
  5. Total: 1228.8 + 32 = 1260.8 adds vs. naïve 1228.8 adds — wait, that's worse! The issue: for small 𝐻, the reconstruction overhead (32) isn't amortized. But for typical LLM layers (𝐻=4096), the merge step reuses the same 16 patterns across many columns, so the effective compute is 𝐻/16 × (1−𝑏𝑠) + 32 ≈ 256×0.3 + 32 = 108.8 adds. Reduction: 1228.8 / 108.8 ≈ 11.3×.
🍯
Key insight: the reduction comes from reusing merged activations across columns, not within a single column. For 8 bit-slices, total compute is 8 × 108.8 = 870.4 adds vs. naïve 8 × 1228.8 = 9830.4 adds → 11.3× reduction. The paper reports up to 12.1× for LLaMA7B (§5.2), which matches this calculation (the extra 0.8× comes from higher bit sparsity in some layers).

② BSTC: computing the compression ratio (LLaMA7B, bit-slice 7)

Bit-slice 7 (high-order magnitude bit) in LLaMA7B has 80% sparsity. For 𝑚=4 group size, what is the compression ratio (CR)?

  1. Original size: 4 rows × 𝐻 columns = 4𝐻 bits (uncompressed).
  2. Sparsity ratio SR=0.80 → 80% of columns are all-zero (0000), 20% are non-zero.
  3. Two-state encoding: each zero column is encoded as '0' (1 bit); each non-zero column as '1' + 4-bit data = 5 bits.
  4. Compressed size: num_zeros × 1 + num_nonzero × 5 = (0.8𝐻 × 1) + (0.2𝐻 × 5) = 0.8𝐻 + 1.0𝐻 = 1.8𝐻 bits.
  5. CR = original / compressed = 4𝐻 / 1.8𝐻 = 2.22×. For 𝐻=4096, original size = 16384 bits (2048 bytes), compressed = 7372.8 bits (921.6 bytes) → saves 1126.4 bytes per slice.
💡
Why not compress bits 1–2? Bits 1–2 have low sparsity (16–30%, Fig. 8(c)). For SR=0.30, the compressed size would be (0.3𝐻×1) + (0.7𝐻×5) = 3.8𝐻 bits vs. original 4𝐻 → CR = 4𝐻/3.8𝐻 ≈ 1.05× (barely worth it). The 1-bit indicator overhead eats the gain. BSTC compresses only bits 3–7 (SR 65–90%, CR 1.5–3.5×, Fig. 8(b)).

③ BGPP: early termination for a single Key (Dolly, 𝛼=0.6)

A Query Q (8-bit) multiplies with a Key K (8-bit, binary pattern 01101001). After loading the first 2 bits (MSB), the partial score is 12. The current max is 75, radius=3, 𝛼=0.6. Does BGPP terminate early?

  1. Threshold after round 1 (2 bits): 𝜃1 = max(Â1) − 𝛼 × radius = 75 − 0.6×3 = 75 − 1.8 = 73.2.
  2. Partial score after 2 bits: 12 (from Q × K[7:6]). This is the score using only the top 2 bits of K.
  3. Maximum possible final score: if the remaining 6 bits are all '1', the max addition is 25 + 24 + … + 20 = 63. So the final score ≤ 12 + 63 = 75.
  4. Is 75 ≥ 73.2? Yes, barely. So BGPP does not terminate yet — the Key might still be in the top-𝑘. Continue to round 2.
  5. Round 2 (3 bits): load bit 5 of K. Suppose the new partial score is 18. The max increases to 80 (other Keys improved). New threshold 𝜃2 = 80 − 1.8 = 78.2. Max possible final = 18 + 31 (remaining 5 bits) = 49. Since 49 < 78.2, BGPP terminates — this Key is guaranteed to fall below the top-𝑘. Stop loading bits 4–0.
🍯
KV cache savings: by terminating after 3 bits instead of all 8, BGPP saves 5/8 = 62.5% of the KV cache accesses for this Key. Across all Keys, the paper reports ~50% average reduction (Fig. 5(g)), accounting for Keys that survive to the end.

④ End-to-end speedup breakdown (LLaMA7B, Dolly 4k-prompt + 16-token decode)

The GPU baseline takes 100ms total. MCBP reports 9.43× speedup. Break down the contributions of BRCR, BSTC, and BGPP.

  1. GPU baseline latency breakdown (Fig. 1(a)): GEMM 45%, weight load 35%, KV cache load 15%, others 5%. So GEMM = 45ms, weight = 35ms, KV = 15ms, others = 5ms.
  2. BRCR (prefill GEMM acceleration): reduces GEMM compute by 12.1× (§5.2). New GEMM time = 45ms / 12.1 ≈ 3.7ms. Savings = 45 − 3.7 = 41.3ms.
  3. BSTC (decoding weight compression): the paper reports a 2.2× throughput gain in the ablation. New weight time = 35ms / 2.2 ≈ 15.9ms. Savings = 35 − 15.9 = 19.1ms.
  4. BGPP (decoding KV cache reduction): reduces KV cache access by ~1.5× (50% reduction on accessed bits, but not all Keys are filtered). New KV time = 15ms / 1.5 = 10ms. Savings = 15 − 10 = 5ms.
  5. Total new latency: 3.7 + 15.9 + 10 + 5 = 34.6ms (ignoring overhead). Speedup = 100ms / 34.6ms ≈ 2.9×. But the paper reports 9.43×! The gap is hardware efficiency: MCBP's dedicated dataflow (bit-slice parallelism, CAM, pipelined CODECs) achieves ~78% PE utilization vs. GPU's ~25% (due to value-level mismatches). So the effective speedup is 2.9× × (0.78/0.25) ≈ 9.0×, close to the reported 9.43×.
💡
Why not exactly 9.43×? The calculation above ignores: (1) overhead from CAM matching, BSTC decoding (adds ~3% latency, Fig. 19(b)); (2) software gain on GPU (BRCR/BSTC algorithms run on GPU add 1.44×/1.23× gain, Fig. 21(a)); (3) workload-specific variance (Dolly vs. MBPP have different GEMM/memory ratios). The 9.43× is an average over 26 benchmarks, with individual speedups ranging from 4.2× (short-sequence, KV-light) to 15.8× (long-sequence, GEMM-heavy, Fig. 20(a)).

Practice problems

Try each before opening the answer. Every one ends with the common misconception it checks.

Problem 1 · easy
A 4-bit weight matrix has 100 columns. After decomposing into 4 bit-slices, the MSB slice has 70 zeros, the next slice 60 zeros, the next 40 zeros, and the LSB slice 20 zeros. What is the average bit sparsity, and is it higher than the value sparsity?
Show answer
Bit sparsity = (70 + 60 + 40 + 20) / (4 × 100) = 190 / 400 = 47.5%. Value sparsity: a 4-bit value is zero only if all 4 bits are zero. The MSB slice has 70 zeros, so at most 70 values can be all-zero (the bottleneck is the least-sparse slice, the LSB with 20 zeros). So value sparsity ≤ 20%. Bit sparsity (47.5%) is 2.4× higher than value sparsity (20%).Misconception: that bit sparsity is just the average of slice sparsities. It is, but the key insight is that value sparsity is constrained by the least-sparse slice (AND of all bits), so bit sparsity is almost always higher.
Problem 2 · medium
BRCR groups a bit-slice matrix into m=4-row groups. For a group with H=256 columns, how many distinct column patterns exist, and what is the expected repetition rate?
Show answer
For m=4 rows (binary vectors), there are 24 = 16 possible distinct patterns (0000 to 1111). If H=256 columns and all patterns are equally likely, each pattern appears 256/16 = 16 times on average. The expected repetition rate = (total columns − unique patterns) / total columns = (256 − 16) / 256 = 93.75%. This means 93.75% of columns are repetitions of one of the 16 unique patterns.Misconception: that repetition rate is 16/256 = 6.25%. That's the unique rate, not the repetition rate. Repetition rate is the fraction of columns that are duplicates, which is 1 − (unique rate) = 93.75%.
Problem 3 · hard
BSTC compresses a bit-slice with SR=0.75 (75% sparsity) using two-state coding (zero → '0', non-zero → '1' + m-bit data). For m=4, what is the compression ratio (CR)? For what SR does CR = 1 (break-even)?
Show answer
Original size = m × H bits. Compressed size = (SR × H × 1) + ((1−SR) × H × (m+1)) = H × (SR + (1−SR)(m+1)). CR = m / (SR + (1−SR)(m+1)). For m=4, SR=0.75: CR = 4 / (0.75 + 0.25×5) = 4 / (0.75 + 1.25) = 4 / 2.0 = 2.0×. Break-even (CR=1): m = SR + (1−SR)(m+1) → m = SR + (m+1) − SR(m+1) → 0 = (1−SR)(m+1) − SR → SR = (m+1−1)/(m+1) = m/(m+1). For m=4, SR = 4/5 = 0.80 (80% sparsity). The paper states that positive benefit occurs when SR exceeds ~65% (Fig. 8(b)).Misconception: that higher m always helps. For m=8, break-even SR = 8/9 ≈ 0.89 (89% sparsity) — very high. BSTC uses m=4 because typical bit sparsity (65–90%) is clustered around the break-even point, and smaller m reduces the indicator overhead.
Problem 4 · medium
BGPP processes a Key bit-by-bit (MSB first). After 3 bits, the partial score is 20. The threshold is θ=50, and 5 bits remain. Can BGPP terminate early? If so, what is the maximum possible final score?
Show answer
The remaining 5 bits (bits 4–0) can contribute at most 24 + 23 + 22 + 21 + 20 = 31. So the maximum possible final score = 20 + 31 = 51. Since 51 > 50 (the threshold), BGPP cannot terminate yet — the Key might still be in the top-k. It must load at least one more bit to narrow the range. (If after 4 bits, the partial score is 24, then max final = 24 + 15 = 39 < 50, and BGPP terminates.)Misconception: that BGPP terminates as soon as the partial score < threshold. The correct condition is: max possible final score < threshold. This requires computing the "headroom" from the remaining bits (2remaining − 1).
Problem 5 · easy
Why does MCBP use sign-magnitude (SM) format instead of two's complement for INT8 weights?
Show answer
SM format (1 sign bit + 7 magnitude bits) separates the sign from the magnitude, so the magnitude bits for small values (near zero) have many leading zeros in the high-order positions. For example, −3 in SM is '1 0000011', where bits 6–2 are all zero (71% sparsity in the magnitude). In two's complement, −3 is '11111101', where bits 7–2 are all one (0% sparsity). Since quantized weights follow a near-Gaussian distribution (most near zero), SM exposes much higher bit sparsity in the magnitude slices (bits 3–7), enabling BSTC compression. The sign bit is handled separately (not compressed).Misconception: that SM is "better" than two's complement. SM is better only for bit sparsity. Two's complement is standard for arithmetic because it simplifies addition (no separate sign handling). MCBP trades off arithmetic simplicity for memory efficiency.
Problem 6 · hard
The paper reports 12.1× compute reduction for BRCR on LLaMA7B, but the theoretical max for m=4, H=4096, bit sparsity 70% is ~11.3× (from Example ①). What accounts for the gap?
Show answer
The gap (12.1× vs. 11.3×) comes from three factors: (1) Layer-specific variance: some layers have higher bit sparsity (e.g., 75–80% in attention layers) vs. the 70% average, pushing the reduction higher. (2) Repetition beyond average: the pigeonhole calculation assumes uniform distribution; in practice, some patterns (e.g., all-zero '0000') are much more common, increasing the effective repetition rate. (3) Reconstruction optimization: the fixed adder tree (Fig. 14 right) exploits data reuse across rows, reducing the effective reconstruction cost below the theoretical 2m−1 adds per row. The 12.1× is the measured reduction on real LLaMA7B weights, averaged over all layers.Misconception: that the theoretical reduction is a hard limit. The theoretical calculation is a lower bound assuming worst-case distribution. Real weights have structure (near-Gaussian, correlated across layers) that BRCR exploits for higher-than-theoretical gains.
Problem 7 · medium
Why does MCBP's speedup vary so much across benchmarks (4.2× to 15.8×, Fig. 20(a))? Identify two key factors.
Show answer
Two key factors: (1) Sequence length (GEMM vs. memory ratio): short-sequence tasks (e.g., Cola, prompt 0.25k) are memory-bound (decoding dominates, weight/KV cache traffic is the bottleneck). BRCR contributes less; BSTC/BGPP contribute more. Speedup ≈ 4.2×. Long-sequence tasks (e.g., Dolly, prompt 8k) are compute-bound (prefill dominates, GEMM is the bottleneck). BRCR contributes more; BSTC/BGPP contribute less. Speedup ≈ 15.8×. (2) Attention sparsity (BGPP effectiveness): reasoning tasks (e.g., MMLU) have higher token-level sparsity (many trivial Keys), so BGPP filters more aggressively (up to 60% KV cache reduction). Generation tasks (e.g., MBPP) have lower sparsity, so BGPP contributes less (~40% reduction). The combined effect: GEMM-heavy + high-sparsity workloads (e.g., Dolly) see the highest speedup; memory-heavy + low-sparsity (e.g., Cola) see the lowest.Misconception: that MCBP's speedup is uniform across tasks. The 9.43× is an average; individual speedups depend on the workload's compute/memory mix and attention sparsity. This is why co-design matters — MCBP addresses all three bottlenecks, so it wins across the full range.
Problem 8 · easy
MCBP's CAM-based match unit has 512B of storage for m=4. How many entries does it store, and why is this sufficient?
Show answer
For m=4 rows (4-bit column vectors), there are 24 = 16 possible patterns. Each entry stores the pattern (4 bits) + associated data (addresses, counters, etc.). If each entry is 32 bytes (256 bits), then 16 entries × 32 B = 512 bytes. This is sufficient because the CAM needs to store all possible patterns for the current group — and there are only 16 for m=4. The CAM is pre-loaded at the start of each group's processing, then queried H times (once per column). The 512B is small compared to the 768kB weight SRAM (0.07%), so the overhead is negligible.Misconception: that the CAM stores all H columns. It doesn't — it stores only the 2m unique patterns, not the columns themselves. The columns are stored in weight SRAM; the CAM is just a lookup table for fast matching.
Quick answers

Frequently asked questions

Twelve questions newcomers ask most. Tap any to expand.

Q1What does "bit-slice" mean, and how is it different from bit-serial?▾
A bit-slice is one bit position across all values in a matrix. For an 8-bit matrix, there are 8 bit-slices (MSB, bit-7, …, LSB). Bit-serial is a computation style: process one bit at a time (MSB first, then bit-7, etc.), accumulating partial products via shifts. MCBP uses bit-serial processing to operate on bit-slices. The key: bit-slicing exposes sparsity/repetition; bit-serial exploits it (by skipping/merging operations at the bit level).
Q2Why does bit-slice decomposition preserve numeric equivalence?▾
An 8-bit integer is just a weighted sum of its bits: value = 27×b₇ + 26×b₆ + … + 20×b₀. So an 8-bit GEMM W×X is equivalent to (27×W₇ + 26×W₆ + … + 20×W₀) × X = 27(W₇×X) + 26(W₆×X) + … + 20(W₀×X), where W₇ is the MSB bit-slice matrix (all 1-bit values). Each W_i×X is a 1-bit GEMM (AND + popcount), then weighted and summed. This is lossless — the final result is identical to the original 8-bit GEMM.
Q3How does BRCR avoid the overhead of matching all H columns serially?▾
The CAM (Content-Addressable Memory) matches in parallel. For each column in the group matrix, the CAM stores all 2m possible patterns (e.g., 16 for m=4). When a column is queried, the CAM compares it against all 16 patterns simultaneously (via parallel comparators) and returns the matching index in one cycle. This eliminates the serial search (which would take 16 cycles for m=4). The CAM is small (512B for m=4) and fast (<1 cycle latency), so the overhead is negligible compared to the compute savings (12.1× reduction).
Q4Why not compress all 8 bit-slices with BSTC?▾
BSTC's two-state encoding adds a 1-bit indicator per column (zero → '0', non-zero → '1'+data). If the sparsity ratio (SR) is low, the indicator overhead exceeds the compression gain. The paper states that positive benefit occurs when SR exceeds ~65% (Fig. 8(b)). Bit-slices 3–7 in LLaMA7B have SR 65–90% (above break-even), so they're compressed. Bits 1–2 have SR 16–30% (below break-even), so they're left uncompressed. Compressing them would increase memory usage (CR < 1).
Q5Does BGPP always improve accuracy vs. value-level top-k?▾
No — BGPP introduces a hyperparameter α (for the threshold θ = max − α×radius). Lower α → more aggressive pruning → higher sparsity but potentially lower accuracy. The paper states that MBPP drops noticeably when αᵣ < 0.6, MMLU degrades significantly only when αᵣ < 0.5, and sets αᵣ in [0.5, 0.6]. For reasoning tasks (MMLU), the model tolerates α=0.5; for generation tasks (MBPP), it's more sensitive (needs α≥0.6). BGPP is a trade-off — you tune α based on the accuracy-performance target.
Q6How does MCBP's speedup compare to quantization alone (e.g., INT8 vs. FP16)?▾
INT8 quantization (vs. FP16) typically gives ~2× speedup on GPUs (from reduced memory bandwidth and faster INT8 ALUs). MCBP achieves 9.43× speedup over INT8 baseline on GPU, so the total speedup over FP16 is roughly 2× × 9.43× ≈ 18.9×. The key: quantization reduces the value size (16 → 8 bits), but MCBP exploits the bit-level structure within those 8 bits (sparsity, repetition). They're complementary.
Q7What is the energy breakdown, and where does MCBP save the most?▾
Fig. 22(b) shows power breakdown: DRAM 47.6% (HBM access), core 37.3% (BRCR 44.7%, SRAM 22%, BGPP 8.2%, BSTC 10.2%, APU 11.7%, scheduler 4.1%), memory interface 15.1%. Despite 75.8% reduction in memory traffic (from BSTC/BGPP), DRAM still dominates due to the autoregressive decoding (weight reloads per token). BRCR contributes the most energy savings (44.7% of core power → 16.7% of total, reduced by 12.1× compute). Overall, MCBP achieves 31.1× energy efficiency over A100 GPU.
Q8Why is m=4 optimal for group size? Why not m=2 or m=8?▾
Smaller m (e.g., m=2) increases repetition rate (22=4 patterns → 75% repetition for H=16), but the reconstruction overhead (2m−1 adds per row) is tiny, so the benefit is limited. Larger m (e.g., m=8) increases reconstruction overhead (27=128 adds per row) exponentially, while the repetition rate drops (28=256 patterns for H=4096 → 93.75% → 93.8%, diminishing returns). Fig. 18 shows m=4 hits the sweet spot: max compression rate (~3.5×) and compute reduction (5–7×) with manageable hardware overhead (CAM 512B, 16 index converters). It's also a power of 2 (hardware-friendly) and divides most LLM hidden dims evenly (4096, 8192, etc.).
Q9How does MCBP handle non-uniform sparsity across layers?▾
MCBP applies BSTC per-layer (offline compression). Layers with high sparsity (e.g., attention QKV projections, SR ~80%) get high CR (~2.5×); layers with low sparsity (e.g., FFN down-projection, SR ~65%) get lower CR (~1.5×). The overall CR is the weighted average across all layers. The decoders are layer-agnostic — they just read the encoded bit stream and decode on-the-fly. No per-layer tuning is needed at runtime.
Q10Can MCBP's techniques be applied to FP16 or INT4 models?▾
For FP16: yes, but the benefit is smaller. FP16 has an exponent (5 bits) and mantissa (10 bits), which don't follow a near-Gaussian distribution. Bit sparsity is lower (~30% vs. 70% for INT8), and repetition is also lower (mantissa bits are more random). The paper doesn't evaluate FP16, but the techniques are applicable. For INT4: yes, and the benefit is higher. Fig. 25(c)(d) shows INT4 has ~66% bit sparsity (still 4× higher than value sparsity). BRCR achieves 51% compute reduction (vs. 72% for INT8), and BSTC achieves 41% memory reduction (vs. 71%). The smaller bit-width (4 vs. 8) reduces the absolute savings, but the relative benefit (bit vs. value) remains.
Q11How does MCBP's area/power overhead compare to SOTA accelerators?▾
Table 4 summarizes: SpAtten (1.55mm² @ 40nm), FACT (6.03mm² @ 28nm), SOFA (4.29mm² @ 28nm), MCBP (9.52mm² @ 28nm). MCBP is larger because it integrates all three techniques (BRCR + BSTC + BGPP) plus CAM. But the area/energy ratio is favorable: MCBP achieves 22740 GOPS/W (31.1× over A100), vs. SpAtten 382 GOPS/W (35× worse than MCBP), FACT 4388 GOPS/W (5.2× worse), SOFA 7183 GOPS/W (3.2× worse). The key: MCBP's techniques are co-designed to minimize overhead (e.g., CAM is time-multiplexed across PEs; BSTC CODECs are lightweight; BGPP is clock-gated). The incremental overhead of each technique is small (Fig. 24(b): BRCR +25% area, BSTC +16%, BGPP +9%), but the combined benefit is multiplicative (2.88× × 2.19× × 1.48× ≈ 9.3×).
Q12What are the accuracy characteristics of BGPP?▾
The paper states that MBPP drops noticeably when αᵣ < 0.6, MMLU degrades significantly only when αᵣ < 0.5, and sets αᵣ in the range [0.5, 0.6]. Fig. 24(a) shows the accuracy vs. sparsity trade-off. For reasoning tasks (MMLU), the model tolerates more pruning (up to α=0.5) before accuracy drops; for generation tasks (MBPP), it's more sensitive (threshold around α=0.6). The default setting is chosen based on the task requirements.

Glossary & further reading

Every term in one place, then a path to go deeper.

Core terms
MCBPMemory-Compute-efficient Bit-slice Processing: the paper's algorithm-hardware co-design for LLM inference, targeting GEMM compute, weight access, and KV cache access via BRCR, BSTC, and BGPP.
Bit-sliceOne bit position across all values in a matrix. An 8-bit matrix decomposes into 8 bit-slices (MSB, bit-7, …, LSB). Lossless: the 8-bit GEMM = weighted sum of 8 bit-slice GEMMs.
BRCR (Bit-slice Repetitiveness-enabled Computation Reduction)Groups 𝑚=4 rows per bit-slice, identifies repetitive column vectors via CAM, merges activations → 12.1× compute reduction vs. value sparsity 3.8× for LLaMA7B (§3.1).
BSTC (Bit-slice Sparsity-enabled Two-State Coding)Compresses high-order slices (bits 3–7, SR 65–90%) with zero→'0', non-zero→'1'+data encoding. Aligned with bit-serial PEs (no reorder). Ablation reports ~2.2× throughput gain (§3.2).
BGPP (Bit-Grained Progressive Prediction)Bit-wise top-𝑘 attention: process Keys MSB-first, filter after each round, terminate early on trivial Keys → 50% KV cache reduction vs. value top-𝑘 (§3.3).
Bit sparsityFraction of zeros in a bit-slice matrix (not value matrix). For quantized weights (near-Gaussian), high-order bits are ~70% sparse vs. ~7% value sparsity. 10.1× higher (Fig. 5(d)).
Group size (𝑚)Number of rows grouped together for BRCR/BSTC. MCBP uses 𝑚=4 (optimal balance: 24=16 patterns, manageable reconstruction overhead 23=8 adds/row).
CAM (Content-Addressable Memory)Hardware lookup table that searches all entries in parallel for a matching key, returning address(es) in one cycle. MCBP uses 512B CAM to match 16 patterns for 𝑚=4 (§4.3).
Supporting terms
Prefill / DecodingTwo phases of LLM inference. Prefill: process entire prompt in parallel (compute-bound, GEMM dominates). Decoding: generate tokens one-by-one (memory-bound, weight/KV cache access dominates).
KV cacheStores Key/Value tensors from prefill to avoid recomputation during decoding. Size grows with sequence length (e.g., 1GB for LLaMA7B @ 4k tokens). BGPP cuts accesses by ~50%.
INT8 quantization8-bit integer representation for weights/activations, vs. FP16. Reduces memory 2× and enables faster arithmetic. MCBP assumes INT8 baseline (standard for modern LLMs).
Sign-magnitude (SM) format1 sign bit + 7 magnitude bits (vs. two's complement). Exposes higher bit sparsity in magnitude slices (high-order bits are zero for small values). Used by MCBP for BSTC compression.
Pigeonhole principleIf 𝐻 items are placed in 2𝑚 bins, and 𝐻 ≫ 2𝑚, then at least one bin has multiple items (repetition guaranteed). BRCR exploits this for 𝑚=4, 𝐻=4096 (LLM hidden dim).
Enumeration matrixIn BRCR, stores all 2𝑚 distinct column patterns (e.g., 16 for 𝑚=4). Used in reconstruction step to map merged activations back to output rows. Fixed (doesn't change per layer).
Top-𝑘 attention sparsityApproximates full attention by keeping only the 𝑘 highest-scoring Keys per Query. Reduces quadratic complexity O(𝑆²) to O(𝑆𝑘). BGPP improves this via bit-wise filtering.
Radius (BGPP threshold)In BGPP, defines the filtering threshold: θ = max(Â) − α×radius. Default radius=3, α∈[0,1]. Lower α → more aggressive pruning (higher sparsity, potential accuracy loss).
Two-state encoding (BSTC)Compresses a bit-slice: zero columns → '0' (1 bit), non-zero → '1' + 𝑚-bit data (𝑚+1 bits). Positive benefit when SR exceeds ~65%. For 𝑚=4, paper reports 2.2× throughput gain.
Bit-reordering overheadIn value-level storage, fetching MSB bits requires extracting them from 8-bit values (fetch → extract → discard LSBs). MCBP's bit-slice-first storage avoids this (MSBs stored contiguously).
Output-stationary dataflowMCBP's tiling strategy: each PE cluster computes a fixed output tile (𝑇M×𝑇N) by accumulating over input tiles (𝑇K dimension). Maximizes weight reuse, minimizes data movement (§4.1).
HBM (High-Bandwidth Memory)Off-chip DRAM with ~512-bit bandwidth (vs. SRAM's on-chip). MCBP stores compressed weights in HBM, decompresses on-the-fly during load (BSTC CODECs sit between HBM and SRAM).

Concepts to shore up first

INT8 quantization for LLMs easy

tutorial · 20–30 min

How post-training quantization (PTQ) reduces LLMs to INT8 with <1% accuracy loss. Papers: LLM.int8 (Dettmers 2022), SmoothQuant (Xiao 2023). MCBP assumes INT8 baseline; understanding the quantization process clarifies where bit sparsity comes from.

Bit-serial computing medium

tutorial · 30 min

Bit-serial processing operates on one bit position at a time (MSB first), naturally aligning with bit-slice decomposition. Understanding this helps clarify how MCBP avoids bit-reordering overhead and enables early termination (BGPP).

Related papers

Bit-Pragmatic Deep Neural Network Computing hard

Albericio et al. · MICRO 2017

Early work on bit-level sparsity (bit-slice pruning for CNNs). Introduces the idea of processing bits independently but doesn't address LLM-specific challenges (repetition, memory bottlenecks, attention sparsity).

SpAtten: Efficient Sparse Attention Architecture medium

Wang, Zhang, Han · HPCA 2021

Token-level attention sparsity via cascade pruning (head + token). MCBP's BGPP refines this with bit-grained prediction. SpAtten is prefill-only; MCBP handles both prefill and decoding.

FACT: FFN-Attention Co-optimized Transformer hard

Qin et al. · ISCA 2023

Whole-model optimization via mixed-precision and token pruning. Targets computation but not memory (weight/KV cache access). MCBP achieves 5.2× higher energy efficiency via bit-level co-design.

SOFA: Compute-Memory Optimized Sparsity Accelerator hard

Wang et al. · MICRO 2024

Cross-stage tiling for attention (reduces KV cache access). Prefill-only; doesn't address weight access or GEMM compute. MCBP's BSTC + BRCR complement SOFA's tiling for full-stack optimization (3.2× better efficiency).

Q8-BERT and SmoothQuant easy

Zafrir et al. 2019 (Q8-BERT); Xiao et al. 2023 (SmoothQuant)

INT8 weight quantization for Transformers. Q8-BERT achieves minimal accuracy loss with symmetric per-channel quantization; SmoothQuant enables lossless 8-bit weight/activation quantization for LLMs (530B params). MCBP builds on INT8 baseline.

Bit Fusion: Bit-level Dynamically Composable Architecture hard

Sharma et al. · ISCA 2018

Dynamic bit-width adaptation for CNNs (1–8 bits). Introduces bit-level fusion (reuse low-bit results for high-bit computation). Orthogonal to MCBP's bit-slice repetition/sparsity — could be combined.
