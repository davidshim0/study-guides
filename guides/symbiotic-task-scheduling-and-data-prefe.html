MICRO 2025 · A Field Guide

Feeding fine-grain tasks before they starve the cores

Hardware task-parallel systems schedule hundreds of tiny 100-cycle tasks across 256 cores — too short for conventional prefetchers to help. This is the plain-English story of TSP + MRS, a prefetcher that learns from task arguments and a scheduler that exploits what it fetches.

📄 "Symbiotic Task Scheduling and Data Prefetching" · Posluns & Jeffrey 🎓 Assumes familiarity with caches & task parallelism ⏱ ~25 min read
⚙️

Prefetchers need a seed to predict addresses — and task schedulers hold that seed in idle task descriptors. Use it before tasks run, then schedule tasks whose data arrived first.

Before you begin

Ten concepts the paper builds on

Each card covers one foundational idea. Read in order the first time; revisit any that look unfamiliar.

TSP (Task-Seeded Prefetcher) MRS (Memory Response Task Scheduler) Baseline (no prefetching) Task metadata
1Hardware task-parallel systems▾

Instead of coarse threads, programs spawn fine-grain tasks (10s–100s of instructions) into hardware queues. A hardware scheduler (not software) picks tasks and dispatches them to idle cores. Hardware scheduling is >100× faster than software [Hive, ISCA'22], enabling tiny tasks that maximize parallelism for irregular applications like graph analytics.

The paper targets architectures like Swarm [MICRO'15], which provides out-of-order speculative execution of tasks with priority ordering, and systems with simpler non-speculative schedulers.

Now you're ready: TSP and MRS integrate directly with hardware task queues — software never sees the prefetching happen.
2Task descriptors and arguments▾

A task descriptor is a small record held in hardware: a function pointer, a few arguments (e.g., pointers to graph vertices), and metadata like priority. When a core enqueues a task, the descriptor goes into a hardware queue. When the scheduler dequeues a task, it loads the descriptor into the core and starts executing the function with those arguments.

Crucially, the descriptor sits idle in the queue between enqueue and dequeue — often for hundreds of cycles — and its arguments are visible to hardware even before the core runs the task.

Now you're ready: TSP reads those idle arguments to predict memory addresses before the core ever executes a load.
3Why conventional prefetchers fail▾

Conventional prefetchers (stride, stream, indirect [IMP]) detect an access pattern in a running thread and issue prefetches. But they need a trigger — typically observing several loads in sequence. A 100-cycle task is too short: by the time a prefetcher detects the pattern and issues the request, the task is already done and the core is stalled waiting for data.

Hardware task schedulers make the problem worse: the sequence of tasks dispatched to a core is unpredictable (e.g., priority-ordered), so there's no "stride" across task pops to detect.

Now you're ready: The prefetcher must trigger before a task starts — which means it must use something other than program execution to seed it.
4Memory latency vs. bandwidth▾

Latency is the time one access takes (e.g., 200 cycles to DRAM); bandwidth is how many bytes/sec the system can move. A core running a 100-cycle task with a single load stalls on latency, but uses almost none of the available bandwidth (it's not issuing enough requests to keep memory busy).

The paper's baseline systems use 10% or less of DRAM BW for most benchmarks, with only sc-r and kcore-r exceeding 25%. An ideal system (zero latency beyond L2) would double BW use but still not saturate memory — which tells you these programs are latency-bound, not bandwidth-bound.

Now you're ready: TSP's prefetches won't exhaust memory BW, so the only risk is cache pollution — and TSP learns which prefetches are useful.
5Predictable vs. unpredictable loads▾

A load is predictable if its address can be computed as scale · [ref] + offset, where ref is either (i) a task argument or (ii) a value returned by a prior predictable load. Example: v->dist is predictable from argument v; e->v->dist is predictable after you've prefetched v->nbegin, which returns e.

The paper measures that 79–99.9% of loads (covering 79–99% of cache lines accessed) are predictable by this formula across its benchmarks. Loads like a[hash(x)] or complex control flow are not predictable.

Now you're ready: TSP targets the 80–99% that are predictable — enough to hide most memory latency without requiring runahead or complex hardware.
6Argument-based prefetching▾

The simplest predictable load: v->dist is just v + 8 if dist is at offset 8 in the vertex struct. Given a task descriptor with argument v = 0x1000, TSP can issue a prefetch for 0x1000 + 0x08 = 0x1008 immediately, before the task runs.

TSP learns these argument→address mappings from past task executions by watching the (task function, load PC) → address stream and fitting scale and offset parameters.

Now you're ready: Argument-based prefetching handles direct field accesses — the low-hanging fruit of task-parallel prefetching.
7Indirect (multi-level) prefetching▾

Loads like e->v->dist depend on a prior load: you must first load e->v, then use its returned value to compute e->v->dist. TSP chains these: (i) it issues an argument-based prefetch for v->nbegin → e; (ii) when that prefetch returns, it reads the loaded value and issues a second prefetch for e->v; (iii) when that returns, it prefetches e->v->dist.

This is recursive: TSP can handle arbitrary dependence chains, limited only by available prefetch slots (PSHRs).

Now you're ready: Indirect prefetching is what catches pointer-chasing traversals — the hallmark of irregular graph workloads.
8Inner-loop prefetching▾

Consider for (e = v->nbegin; e < v->nend; e++) reading e->dist each iteration. The load strides by the size of an edge struct. TSP learns (i) the base address v->nbegin, (ii) the stride (e.g., 16 bytes), and (iii) the loop bound v->nend — all before the task runs — then issues all the loop's prefetches simultaneously.

Most benchmarks see limited benefit from loop prefetching (compared to argument/indirect) because loop bodies are short and fit in one cache line, but a few (cf, kcore) have longer lists that benefit.

Now you're ready: Loop prefetching extends TSP to small arrays or edgelists accessed within a single task.
9Prefetch-aware scheduling (MRS)▾

TSP tells the scheduler when a task's prefetches have filled. MRS (Memory Response Task Scheduler) adjusts the baseline task dispatch policy to prioritize tasks whose data is ready over those still waiting. For example, if two tasks are both high-priority but one has its prefetches filled and the other doesn't, MRS dequeues the ready one first — keeping the core busy while the other waits for memory.

MRS is a soft preference: if no ready tasks exist, it falls back to the baseline policy to avoid idling cores.

Now you're ready: MRS is the "symbiotic" half — it exploits what TSP fetches to hide latency further.
10Training TSP online▾

TSP's prefetch table (PT) holds rows keyed by (task function, load PC) pairs. Each row has entries storing dependence, offset, scale, and a state (INITIAL, TRANSIENT, STABLE, or UNSTABLE). As tasks run, TSP observes actual addresses and fits the formula. A finite-state machine promotes entries to STABLE when predictions confirm across multiple tasks, or demotes to UNSTABLE if they break.

A usefulness counter per row tracks prediction success: +1 on correct predictions, −128 on mispredictions. If a row's counter goes negative, TSP stops prefetching from it, avoiding cache pollution.

Now you're ready: TSP learns from program behavior at runtime — no programmer annotations or compiler passes required.
The whole story in one place

TSP + MRS in a nutshell

The problem hardware task schedulers face, how TSP seeds prefetching from idle task descriptors, and how MRS adjusts the schedule to exploit filled prefetches.

Hardware task-parallel systems like Swarm schedule fine-grain tasks (10s–100s of instructions) to hundreds of cores, achieving 3–60× speedups over software task schedulers. But they expose memory latency: a 100-cycle task that waits 200 cycles for DRAM data is mostly idle, and conventional prefetchers can't help — they need a consistent access pattern across many accesses, but each task is unique and too short.

The paper's insight: the task scheduler already holds the seed conventional prefetchers lack. When a task is enqueued but idle in hardware queues, its descriptor (function pointer + arguments + metadata) is visible to hardware. Those arguments — typically pointers to graph vertices or other data — determine most of the task's memory accesses via affine arithmetic: address = scale · [ref] + offset. Specialized accelerators use task metadata for prefetching; general-purpose hardware can do the same.

TSP (Task-Seeded Prefetcher) learns, for each (task function, load PC) pair, the mapping from task arguments and prior load values to addresses. When the scheduler seeds TSP with an idle task descriptor, TSP computes addresses by substituting argument values into learned formulas, issues prefetches, and chains them recursively for indirect loads. A 64-entry PSHR (Prefetch Status Handling Register) tracks outstanding prefetches; when they fill, TSP notifies the scheduler. Training happens online: TSP observes actual addresses as tasks run, fits scale/offset parameters, and promotes predictions that repeatedly confirm to STABLE state. A usefulness counter (+1 per correct prediction, −128 per misprediction) penalizes wrong predictions to avoid pollution.

MRS (Memory Response Task Scheduler) augments the baseline dispatch policy by maintaining a Prefetch Queue (PQ) — a duplicate of the task queue's data structure. As TSP signals that tasks' prefetches have filled, MRS prioritizes ready tasks over pending ones when dispatching to idle cores. This keeps cores working on cache-resident data while slower tasks' memory accesses trickle in. MRS is a soft hint: if no ready tasks exist, it falls back to the baseline policy.

Together, TSP and MRS deliver speedups of up to 3.1× (gmeans up to 1.4×) across 13 benchmarks on 256-core systems with four distinct scheduling policies (speculative/non-speculative × random/spatial mapping). The hardware overhead is 13.5% of a 1MB per-tile L2 cache, or ~2% of total chip area. Performance often exceeds that of a 16× larger cache with no prefetching, showing that prefetching is a better use of area than passive storage for task-parallel workloads.

🍯 Key takeaways

  • Hardware task schedulers expose memory latency because 100-cycle tasks are too short for conventional prefetchers.
  • Task descriptors (held idle in queues) provide the seed — arguments that determine addresses via affine arithmetic.
  • TSP learns scale · [ref] + offset for each (function, load PC), chains indirect prefetches recursively, and trains online.
  • MRS prioritizes tasks whose prefetches filled, hiding latency by keeping cores on cache-resident data.
  • Speedups: up to 3.1× (gmean 1.4×) over baselines that were already 3–60× faster than software.
  • Cost: ~2% total chip area, 13.5% of per-tile L2 cache.
  • Trained TSP outperforms programmed TSP — online learning beats static analysis by finding library accesses and data-dependent patterns.
Seven everyday pictures

How to think about TSP and MRS

Each stands alone — read in any order. Every card ends with an honest note about where the picture stops being true.

The core insight · Idle tasks are seeds

Reading the recipe before cooking

You wouldn't start boiling water only after you start cooking — you'd read the recipe first and boil it in advance. TSP reads task descriptors while they're idle in the queue (the "recipe") and starts fetching data (the "water") before the core begins execution. By the time the core runs the task, the data is hot and ready.

Where it misleads: a recipe is static; TSP learns the "recipe" (address formulas) dynamically by watching past executions. It's more like learning what ingredients you always need for spaghetti after making it ten times.
Predictability · Affine arithmetic

Measuring with a ruler

If you know the edge of a table is at position X, you can measure 8 inches from there without touching it. TSP does the same: given argument v = 0x1000, it calculates v->dist is at 0x1000 + 8 without running the load. The "ruler" is the learned scale/offset — once fit, it can predict addresses for tasks calling that function with that load PC.

Where it breaks: a ruler assumes a flat surface; some address generation is nonlinear (hash functions, bit-twiddling). TSP only handles affine (linear) patterns — but 80–99% of loads are affine in practice.
Indirect prefetching · Chaining loads

Following a treasure map

A treasure map says "walk to tree A, dig for clue to tree B, dig there for the treasure." You can't skip ahead — each step reveals the next. TSP does this: it prefetches v->nbegin (tree A), waits for the value, uses it to prefetch e->v (tree B), waits, then prefetches e->v->dist (treasure). The chain can be arbitrarily deep, limited only by available PSHRs.

Where it breaks: a treasure map is serial; TSP can issue many chains in parallel (one per idle task). Also, if the chain is too long and the task starts before it completes, the core stalls anyway — TSP just hides what it can.
MRS scheduling · Data-ready prioritization

Serving hot food first

A restaurant kitchen doesn't serve half-cooked dishes while hot ones sit under lamps. MRS does the same: it prioritizes tasks whose prefetches filled (hot food) over those still waiting on DRAM (still cooking). If no ready tasks exist, it serves what's available to keep the waiter (core) working — better a slightly delayed dish than an idle waiter.

Where it breaks: a kitchen worries about customer wait time; MRS only cares about core utilization. Reordering tasks can cause priority inversions (a high-priority task delayed while a lower one runs), which can hurt work efficiency if the application is sensitive to order (e.g., sssp on some inputs).
Training · Online learning

Learning to ride a bike

You don't learn to ride a bike by reading a manual — you get on, wobble, adjust, and eventually it clicks. TSP does the same: it starts with no knowledge (INITIAL state), observes addresses as tasks run, fits scale/offset, and if predictions repeatedly confirm (5 hits without a miss), promotes the entry to STABLE. If a prediction breaks, it demotes to UNSTABLE and stops prefetching. Over time, it "learns the bike."

Where it misleads: learning a bike is one-time; TSP keeps learning. If a program changes phase (different data structure), TSP detects the mispredictions and relearns. But there's a lag — the first tasks in a new phase miss, just like your first wobbly pedals.
Usefulness tracking · Avoiding pollution

Not watering dead plants

If you keep watering a dead plant, you're wasting water. TSP tracks a usefulness counter per (function, load PC). Every confirmed prediction increments it; every misprediction decrements by 128. If the counter goes negative, TSP stops prefetching that load — it's "dead." This penalizes loads with repeated wrong predictions to avoid polluting the cache.

Where it breaks: a dead plant stays dead; a load can become useful again if the program phase changes. TSP's counter can recover (mispredictions eventually stop, confirmations accumulate), but there's hysteresis — once penalized, it takes many successes to re-enable.
Hardware cost · Small overhead

A GPS unit vs. a bigger gas tank

Adding a GPS to your car costs a little space and power but lets you avoid traffic, saving fuel. TSP+MRS is the GPS: 13.5% of per-tile L2 cache (PT 60 KB, PQ 4 KB, PSHRs 3 KB per tile), ~2% of total chip area, that avoids memory latency "traffic." The paper shows TSP+MRS with a 1/16th cache often beats a 16× larger cache with no prefetching — the small GPS saves more "fuel" than a giant tank would.

Where it breaks: a GPS never slows you down; aggressive prefetching can hurt by saturating memory BW or causing cache thrashing. TSP mitigates this (usefulness counters, throttling PSHRs) but it's not free — a few benchmarks see slowdowns due to over-prefetching or exacerbating abort/priority-drift pathologies.
The full explanation

How it actually works

Section by section, with the paper's key mechanisms rebuilt as diagrams you can read at a glance.

1 · The problem — short tasks expose latency

Table 1 in the paper reports that 8/13 benchmarks have average task lengths <200 cycles (measured on an idealized system that always hits in the L2). DRAM latency is >200 cycles. By the time a core dequeues a task and executes its first load, it's already too late for a conventional prefetcher to issue a timely prefetch — the task will finish (or stall waiting) before the prefetch arrives.

Worse, hardware task schedulers hide the task sequence from cores. A conventional prefetcher triggers on a thread's load stream (e.g., "this thread accessed A, B, C → predict D"). But the core only sees task pops (a special instruction that returns the next descriptor), not a predictable sequence of addresses. The task queue is a black box — its contents are invisible to both the core and conventional prefetchers.

📊
Measured impact: Fig. 1a shows IPC ≤ 0.6 for most benchmarks on the baseline system. An ideal system (zero latency beyond L2) grants >2× IPC for 5/13 benchmarks, yet uses 10% or less of DRAM BW for most benchmarks (Fig. 1b), with only sc-r and kcore-r exceeding 25%. These programs are latency-bound, not bandwidth-bound — hiding latency is the bottleneck.

2 · TSP: seed, predict, prefetch

TSP's Prefetch Table (PT) holds one row per (task function, load PC) pair. Each row has 8 entries (configurable), each storing a dependence (which task argument or prior load this address depends on), offset and scale for Eq. 1, and a state (INITIAL, TRANSIENT, STABLE, or UNSTABLE). When the scheduler seeds TSP with a task descriptor, TSP:

  1. Searches all PT rows matching the descriptor's task function.
  2. For each STABLE entry depending on an argument, calculates the prefetch address as scale · arg + offset.
  3. Allocates a PSHR (Prefetch Status Handling Register) and issues the prefetch to the memory hierarchy.
  4. For STABLE entries depending on a prior load, allocates a dependent PSHR but waits for the parent PSHR to fill before computing the address.
  5. When a prefetch fills, recursively issues any dependent prefetches, and for loop loads, issues all stride iterations simultaneously.
  6. When all PSHRs for a task are freed, notifies MRS that the task is ready.
💡
Why PSHRs, not cache lines? A task may access dozens of addresses; tracking "is this task ready" requires counting outstanding prefetches, not scanning cache tags. PSHRs are cheap (47 bytes each) and capture the dependency DAG (parent/child/sibling pointers) to enable recursive indirect prefetching.

3 · Training: fitting scale and offset

As a task runs, TSP maintains an Executing Table (ET) — one entry per load PC, holding (address, value, count). On the first occurrence of a load PC within a task, TSP updates all PT entries for that load using the difference between the new and old addresses/values:

# Equations 2 & 3 from the paper
newScale = (oldLastAddr − newLastAddr) / (oldLastVal − newLastVal)
newOffset = newAddr − newScale · newLastVal

This fits a line through two points: the last time this load was seen and now. If the new scale/offset match the old, the FSM promotes INITIAL→TRANSIENT→STABLE (after 5 confirmations). If they differ, the FSM resets or demotes to UNSTABLE. For loop loads, TSP trains the stride similarly, and trains End/Count entries when the task finishes.

If a row's entries keep changing dependence (U→U transitions), TSP randomly selects a new dependence from the task arguments or ET entries and returns to INITIAL. This lets it search the space of possible dependences with finite storage.

🍯
Key insight: Training is automatic and adapts to program phase. The programmer writes zero prefetch annotations — TSP observes and learns. This is why trained TSP outperforms programmed TSP (Sec. 5.2).

4 · MRS: scheduling with prefetch awareness

MRS adds a Prefetch Queue (PQ) to each task unit — a duplicate of the task scheduler's ordering structure (e.g., priority queue). When a task is enqueued, MRS inserts it into both the TQ (Task Queue) and PQ. The PQ's head is seeded to TSP proactively. When TSP finishes prefetching a task, it marks it as ready.

When a core executes a dequeue instruction, MRS checks if the highest-priority task in the TQ is ready. If not, MRS scans forward in the TQ for the next ready task and dispatches it instead. If no ready tasks exist, MRS falls back to the baseline policy (dispatch the highest-priority task anyway) to avoid idling the core.

⚠
Trade-off: prioritizing ready tasks can cause priority inversions — a high-priority task delayed while lower-priority ones run. For work-efficient algorithms (e.g., sssp), this can increase total task count and hurt performance (Sec. 5.2). MRS is a performance hint, not a correctness requirement.

5 · Mitigating harmful prefetches

Two risks: (i) mispredictions that pollute the cache with useless data, and (ii) bandwidth saturation if aggressive prefetching floods memory. TSP mitigates the first with usefulness counters per PT row: +1 on every S→S confirmation (correct prediction), −128 on every S→U misprediction. If the counter goes negative, TSP stops prefetching that row. This penalizes loads that repeatedly mispredict (e.g., hash-based accesses) without permanently disabling them.

For bandwidth, TSP monitors prefetch fill latency. If prefetches start taking much longer than memory latency, TSP disables some PSHRs and slowly re-enables them. The paper reports that only 1/13 benchmarks (tri-r) saturates BW with TSP+MRS; most remain under 50% utilization even with prefetching (Fig. 7).

Follow the numbers, step by step

Worked examples

Four traces through the paper's real numbers and mechanisms.

① Fitting scale and offset from two accesses

Suppose TSP observed v->dist at address 0x1008 when v = 0x1000 (task 1), then at address 0x2008 when v = 0x2000 (task 2). Derive the scale and offset.

  1. oldLastAddr = 0x1008, newLastAddr = 0x2008, oldLastVal = 0x1000, newLastVal = 0x2000.
  2. newScale = (0x1008 − 0x2008) / (0x1000 − 0x2000) = (−0x1000) / (−0x1000) = 1.
  3. newOffset = 0x2008 − 1 · 0x2000 = 0x08.
  4. Formula: address = 1 · v + 0x08 — i.e., v->dist is 8 bytes into the struct.
  5. On task 3 with v = 0x3000, TSP predicts 0x3000 + 0x08 = 0x3008 before the core runs.
🍯
Insight: Two observations suffice to fit the line. After 5 confirmations (tasks 3–7 hit the predicted address), the entry promotes to STABLE and TSP trusts it for prefetching.

② Chaining indirect prefetches for e->v->dist

A task has v = 0x1000. TSP already learned v->nbegin is at v + 0x08 and e->v is at e + 0x00 and e->v->dist is at e->v + 0x10. Walk through the prefetch chain.

  1. TSP sees task descriptor with v = 0x1000. PT row for v->nbegin is STABLE, depends on arg0.
  2. Compute: 0x1000 + 0x08 = 0x1008. Allocate PSHR₁, issue prefetch for 0x1008.
  3. PT row for e->v is STABLE, depends on v->nbegin. Allocate PSHR₂ as child of PSHR₁, but don't issue yet.
  4. Prefetch for 0x1008 fills, returning e = 0x2000. TSP reads the value from PSHR₁.
  5. Compute: 0x2000 + 0x00 = 0x2000. Issue prefetch for 0x2000 using PSHR₂.
  6. PT row for e->v->dist is STABLE, depends on e->v. Allocate PSHR₃ as child of PSHR₂.
  7. Prefetch for 0x2000 fills, returning vertex* = 0x5000. Compute: 0x5000 + 0x10 = 0x5010. Issue prefetch using PSHR₃.
  8. All PSHRs free → task marked ready. Core runs and hits in cache for all three loads.
💡
The chain depth is 3 (arg → load → load → load). TSP handles arbitrary depth, limited only by available PSHRs (64 per tile = room for ~10 such chains in parallel).

③ Why trained TSP beats programmed TSP

The paper reports trained TSP grants 3.3× speedup over programmed TSP on color-r. Why? Consider enqueuer trees (a library that spawns tasks in batches).

  1. Programmed TSP: a human writes prefetch hints for application-level task functions (e.g., ColorVertex).
  2. The enqueuer-tree library calls internal functions like TreeNode::enqueue, which access both tree metadata and application vertex data.
  3. Programmed TSP cannot hint these internal functions — the library is generic, so it doesn't know the app's data layout.
  4. Trained TSP observes actual addresses when TreeNode::enqueue runs, fits scale/offset for its loads.
  5. Trained TSP prefetches tree metadata and vertex data for tree nodes, even though the programmer never hinted them.
  6. For workloads with many small tasks (color has 47-cycle tasks on average), these "hidden" library accesses dominate. Trained TSP's extra coverage yields 3.3× over programmed.
🍯
Insight: Static analysis misses (i) library internals with opaque ABIs and (ii) data-dependent loop bounds. Online learning catches both — which is why trained TSP almost always ≥ programmed TSP (Fig. 9).

④ Tallying the hardware cost

The paper reports TSP+MRS adds 13.5% of per-tile L2 cache area, or ~2% of total chip area. Add it up for one tile (256-core system = 64 tiles).

  1. PT (Prefetch Table): 256 rows. Each row = 22 B (PC, lastAddr, stride, etc.) + 8 entries × 26 B each = 230 B/row. Total: 256 × 230 = 57.5 KB.
  2. PT CAM (for matching function pointers): 256 entries × 10 B = 2.5 KB.
  3. PSHRs: 64 entries × 47 B = 3 KB.
  4. Executing Table (ET): 4 tasks × 128 load PCs × 25 B = 12.5 KB (shared across running tasks).
  5. PQ (Prefetch Queue): 256 task descriptors × 2 × 8 B = 4 KB TCAM.
  6. Total per tile: 57.5 + 2.5 + 3 + 12.5 + 4 = 79.5 KB (some shared, amortized).
  7. Compare to baseline: 1 MB L2/tile. The paper reports this is 13.5% of a 1MB per-tile L2 cache.
  8. Full chip: The paper reports TSP+MRS add ~2% of total chip area (assuming cache dominates chip area).
💡
Per-tile overhead is 13.5% of L2, but the benefit is worth it: Fig. 6 shows TSP+MRS with 1/16th cache often matches or beats a 16× larger cache without prefetching — an "iso-performance area benefit" that makes TSP+MRS a better use of transistors.
Test yourself

Practice problems

Try each before opening the answer. Every answer ends with the common misconception it checks.

Problem 1 · easy
A benchmark has 200-cycle tasks on average. DRAM latency is 200 cycles. A conventional L2 stride prefetcher triggers after observing 3 misses. Will it help these tasks? Why or why not?
Show answer
No. By the time the core has executed 3 misses (≥600 cycles total), the task is long finished (200 cycles). The prefetcher would issue prefetches for data the next task will use — but hardware task schedulers don't guarantee which task pops next, so the prefetch is likely useless. This is why Fig. 6 reports that adding IMP (I) grants minimal speedup for most benchmarks (except bfs-o on NonspecRand).Misconception: that conventional prefetchers "just work" if you add them to the system. They need a consistent access pattern and enough time to trigger — neither exists for fine-grain tasks.
Problem 2 · medium
TSP observes e->dist at address 0x2000 when e = 0x2000 (task 1), then at 0x2008 when e = 0x2008 (task 2). What are scale and offset? Is this a loop stride or an argument dependence?
Show answer
newScale = (0x2000 − 0x2008) / (0x2000 − 0x2008) = 1. newOffset = 0x2008 − 1 · 0x2008 = 0. Formula: address = 1 · e + 0. This is a loop stride (or an inner-loop access where the base address itself strides). Specifically, if e is incrementing by 8 bytes per iteration, the formula predicts e->dist = e + 0 each time. TSP's stride field would also capture the +8 increment and issue all iterations' prefetches simultaneously.Misconception: that scale=1, offset=0 implies "do nothing." It means the address equals the dependence value — a valid pattern for pointer-chasing or array-of-structs iteration.
Problem 3 · medium
MRS prioritizes tasks whose prefetches filled. For sssp on road graphs (r), this grants a 2.2× speedup. On social-media graphs (o), it grants only 1.0× (no speedup) or even a 1.3× slowdown on one config. Explain using "temporal locality" and "task fanout."
Show answer
Road graphs have low-degree vertices (avg. degree 4) → small task fanout, and each vertex is visited ~once → low temporal locality. Prefetching vertex data is critical because it's rarely in cache. Social-media graphs (power-law) have high-degree hubs → huge task fanout (thousands of tasks enqueued at once, saturating hardware structures) and high temporal locality (hubs accessed repeatedly). Prefetching helps less because vertex data is already cached, and MRS's priority inversions (delaying high-priority hub tasks to run ready low-priority ones) cause work inefficiency — sssp enqueues 2.1× more tasks with MRS on sssp-r NonspecSpat 1/16 cache, offsetting the 1.5× per-task speedup.Misconception: that prefetching always helps irregular workloads. It helps when (i) data isn't cached and (ii) reordering doesn't break work efficiency. Power-law graphs violate both.
Problem 4 · hard
TSP's usefulness counter increments by +1 on each S→S confirmation, decrements by −128 on each S→U misprediction. After 5 confirmations, the counter is +5. One misprediction later, it's −123. How many confirmations are needed to recover to positive?
Show answer
From −123, need 124 confirmations to reach +1. This asymmetry is deliberate: a few mispredictions severely penalize a load (to quickly disable pollution), but recovery is slow (to avoid thrashing). The paper shows Fig. 7 that 12/22 benchmark/input pairs see extra DRAM accesses from TSP (over-prefetching), but only tri-r saturates BW — the counter prevents most harm.Misconception: that mispredictions should penalize by −1 (symmetric). The −128 penalty biases TSP toward conservatism: once it decides a load is unpredictable, it takes many successes to convince it otherwise. This trades coverage for accuracy.
Problem 5 · medium
The paper reports that color sees a 3.1× speedup with TSP+MRS on SpecSpat. Inspecting Table 1, color-r's tasks average 154 cycles, access 7 loads (4 cache lines), 85% predictable lines. Compute: if the baseline task stalls for 200 cycles on 1 L2 miss, what's the speedup if TSP eliminates 85% of misses?
Show answer
Baseline: 154 cycles execution + (4 lines × 1 miss/line × 200 cycles/miss) = 154 + 800 = 954 cycles/task. TSP eliminates 85% of 4 = 3.4 misses → 0.6 misses remain. TSP: 154 + (0.6 × 200) = 154 + 120 = 274 cycles/task. Speedup: 954 ÷ 274 ≈ 3.5×. Close to the measured 3.1× (the difference is MRS overhead, cache sharing, NoC contention).Misconception: that predictable loads (85%) equals predictable lines. The paper separates these (Table 1 bottom two rows) because multiple loads can hit the same line. Also, "85% predictable" doesn't mean "85% hit" — it means TSP issues prefetches for 85%; some arrive late or are evicted.
Problem 6 · hard
Fig. 6 shows msf-o SpecRand sees a 1.2× slowdown with TSP+MRS (M) vs. baseline (B), despite committed cycles dropping. The breakdown shows "Abort" cycles increased more than "Commit" decreased. Explain using "prefetch visibility" and "speculation."
Show answer
msf (union-find) has large tasks that access many addresses → high conflict probability under speculation (Swarm aborts tasks that violate data dependence order). TSP issues prefetches for all those addresses before the task runs. Those prefetches can trigger conflicts (e.g., TSP prefetches a line that a concurrent task is writing) → the task aborts before it even starts. Worse, MRS delays high-priority tasks until their prefetches fill, allowing lower-priority tasks to run and create more conflicts. The paper reports msf-o has 50% aborted cycles with MRS vs. 30% baseline — the aggressive prefetching amplifies a bad scheduler's pathology.Misconception: that prefetching is "free" because it's off the critical path. Prefetches have side effects — they touch the coherence protocol and can trigger aborts or evict useful data. For workloads with high abort rates, aggressive prefetching can make things worse.
Problem 7 · easy
TSP's PT has 256 rows with 8 entries each. A benchmark has 30 static task functions, each with ~10 loads. How many (function, load PC) pairs exist? Will all of them fit in the PT?
Show answer
30 functions × 10 loads/function = 300 pairs. PT holds 256 rows. No, they won't all fit. TSP uses a CAM (content-addressable memory) for the PT, so it evicts the least-recently-used row when full. For most benchmarks (Table 1 reports 3–16 static task functions), 256 rows suffice. For tri (30 functions) or kmeans (16), some rows evict and must re-train if accessed again — but the paper shows this doesn't hurt performance much because the hot functions dominate execution.Misconception: that TSP must learn every load. It only needs to learn the frequently executed ones. A few cold functions missing from the PT barely affect performance.
Problem 8 · medium
The paper compares TSP+MRS with a 1/16th cache to baseline with full cache. For bfs-o SpecSpat, both run in ~27M cycles (Fig. 6). Compute the "iso-performance area benefit": if TSP+MRS costs 2% of total chip area, what's the net area saving?
Show answer
Full cache = 100% cache area. Baseline uses 100%, runs in 27M cycles. TSP+MRS uses 6.25% cache (1/16th) + 2% overhead (of total chip area) = 8.25% total cache area, runs in 27M cycles. Same performance, 91.75% less cache area. The "area benefit" is that you can either (i) shrink the cache 12× and maintain speed, or (ii) use the saved area for more cores, more LLC, etc.Misconception: that the 2% overhead is "free" because it's small. It's 2% of total chip area. The key is that prefetching with 8.25% area beats no prefetching with 100% area — showing prefetching is a better use of transistors.
Quick answers

Frequently asked questions

Twelve questions newcomers ask most. Tap any to expand.

Q1Why can't conventional prefetchers help hardware task schedulers?▾
Conventional prefetchers trigger on a running thread's load stream — they detect a pattern (e.g., "accessed A, B, C → predict D") and issue prefetches. But (i) tasks are too short (~100 cycles) for a prefetcher to detect a pattern and issue a timely prefetch, and (ii) hardware schedulers dispatch tasks unpredictably (e.g., priority-ordered), so there's no "stride" across task pops. By the time a prefetcher triggers, the task is done and the core is idle. That's why Fig. 6 shows adding IMP (I) grants minimal speedup except for bfs-o on NonspecRand.
Q2What does it mean that 80–99% of loads are "predictable"?▾
A load is predictable if its address can be computed as scale · [ref] + offset, where ref is a task argument or a prior load's return value. Table 1 shows that 79–99.9% of dynamic load instructions are predictable, covering 79–99% of cache lines accessed. The unpredictable loads are hash functions, computed indices, or complex control flow. TSP targets the 80–99%; future work could use runahead for the rest.
Q3How does TSP know which task function a descriptor belongs to?▾
Task descriptors include a function pointer (part of the standard task-parallel programming model). TSP's PT is indexed by function pointer — when a task is seeded, TSP searches all PT rows matching that pointer. This is why TSP can learn per-function patterns (e.g., SSSPTask always accesses v->dist at offset 8) without knowing anything about the source code.
Q4Why 64 PSHRs per tile? Why not more or fewer?▾
Fig. 10 sweeps PSHR count from 16 to 65536. Performance saturates at 64 for most benchmarks; rbp needs 64 to avoid running out. More than 64 doesn't help (tasks finish before prefetches fill) and costs area. Fewer hurts rbp, which has deep dependence chains. The paper chooses 64 as the sweet spot — enough for 10–15 independent chains in parallel, or 1–2 deep chains per active task.
Q5What is a "moot task" and why does it cause over-prefetching?▾
A moot task exits early without doing work (e.g., Listing 1 line 2: if (dist > v->dist) return). Prior work [Hive, ISCA'22] found 52–94% of sssp tasks are moot. TSP learns the full task's access pattern (all loads in the function) and prefetches them when seeded — but moot tasks never execute those loads, so the prefetches are wasted. Fig. 7 shows 12/22 benchmarks see extra DRAM accesses; com-Orkut inputs (high moot rate) see 2–4× more traffic. TSP's usefulness counter mitigates this by penalizing loads with repeated wrong predictions, but it's not perfect.
Q6Does MRS break program correctness by reordering tasks?▾
No. MRS reorders tasks within the constraints of the baseline scheduler. Consecutively scheduled tasks are likely independent, and for speculative systems (Swarm), the speculation hardware enforces data dependences — if MRS dispatches a task out of order and it conflicts with a higher-priority task, it aborts. For non-speculative systems, the application must synchronize (locks, atomics) — MRS doesn't change the synchronization semantics, just the dispatch timing. However, MRS can hurt work efficiency by causing priority drift (Sec. 5.2), which is a performance issue, not a correctness bug.
Q7Why does trained TSP beat programmed TSP?▾
Two reasons: (i) library internals — enqueuer trees or other runtime functions access data structures the programmer can't annotate (opaque ABI), and (ii) data-dependent patterns — loop bounds or which fields are accessed depend on graph structure (e.g., power-law graphs have most vertices with single-child trees, so the tree-building code accesses 1 edge 90% of the time). Trained TSP observes actual addresses at runtime and learns these patterns; programmed TSP relies on static analysis, which misses them. Fig. 6 shows trained (M) almost always ≥ programmed (P), by up to 3.3× (color-r).
Q8How does TSP differ from IMP (Indirect Memory Prefetcher)?▾
IMP [MICRO'15] is a conventional prefetcher for multithreaded programs. It (i) detects a striding access pattern in a thread, (ii) identifies if the loaded values are used as pointers (address generation), (iii) issues prefetches for the indirect loads. TSP borrows the "affine address generation" idea but differs in the trigger: IMP triggers on a running thread's access stream; TSP triggers on idle task descriptors. Also, IMP requires a consistent stride to start; TSP seeds with task arguments immediately. Fig. 6 shows IMP (I) grants minimal speedup on most benchmarks because tasks are too short to establish a stride.
Q9Can TSP work with other hardware task schedulers (not just Swarm)?▾
Yes. TSP requires only that (i) tasks have descriptors with function pointers and arguments, and (ii) descriptors are held in hardware queues before dispatch. The paper evaluates four schedulers: SpecRand, SpecSpat (Swarm-like), NonspecRand (RELD-like [HD-CPS, HPCA'22]), NonspecSpat (Dalorex-like [HPCA'23]). TSP+MRS grant speedups on all four. The paper mentions TSP is "applicable to any hardware task scheduler" — the key is that the scheduler must expose task descriptors to TSP before dispatch.
Q10Does TSP help bandwidth-bound workloads?▾
No — TSP is designed for latency-bound workloads. Fig. 1b shows the baseline uses 10% or less of DRAM BW for most benchmarks, with only sc-r and kcore-r exceeding 25%; an ideal system would use 20–74% (still not saturated). TSP adds prefetch traffic (Fig. 7), but only tri-r reaches 100% BW utilization — the rest stay under 50%. For bandwidth-bound workloads (e.g., dense linear algebra), prefetching can't help much because memory is already fully utilized; you'd need bandwidth optimizations (compression, better NoC routing, etc.).
Q11Why does TSP+MRS sometimes hurt performance (e.g., msf-o)?▾
Three failure modes: (i) over-prefetching → wasted BW and cache pollution (sssp-o); (ii) exacerbating aborts → prefetches trigger conflicts under speculation, and MRS delays high-priority tasks, increasing abort rate (msf-o); (iii) priority inversions → MRS reordering causes work inefficiency for priority-sensitive algorithms (sssp-r on NonspecSpat 1/16 cache). The paper is honest about these pathologies — prefetching isn't a silver bullet, and bad schedulers get worse with aggressive prefetching.
Q12What did the authors leave for future work?▾
The paper mentions applying MRS to Hive-style schedulers that eliminate moot tasks before running them (would fix over-prefetching), and prefetching for non-affine address generation (hash functions, bit-twiddling) via runahead or other techniques. Balancing MRS's reordering with speculation state and work efficiency (e.g., limit priority inversions to ≤ some threshold) is also unaddressed.

Glossary & further reading

Every term in one place, then a path to go deeper.

Core terms
TSP (Task-Seeded Prefetcher)The paper's prefetcher: learns address formulas for each (task function, load PC), seeded with idle task descriptors, issues prefetches before tasks run.
MRS (Memory Response Task Scheduler)The paper's scheduling policy: prioritizes tasks whose prefetches filled over those still waiting, exploiting TSP to hide latency.
Task descriptorA small record held in hardware: function pointer, arguments (e.g., pointers to graph vertices), metadata (priority, timestamp). Visible to TSP while idle in queues.
PSHR (Prefetch Status Handling Register)Tracks one outstanding prefetch: task ID, PC, address, parent/child/sibling pointers for dependence chains. 64 per tile, 47 bytes each.
Prefetch Table (PT)Holds learned formulas: one row per (task function, load PC), 8 entries per row, each with dependence/offset/scale/state. 256 rows, ~60 KB per tile.
Prefetch Queue (PQ)MRS's duplicate of the task scheduler's queue — used to seed TSP proactively and track which tasks are ready for dispatch. 256 entries, ~4 KB per tile.
Predictable loadA load whose address = scale · [ref] + offset, where ref is a task argument or prior load. 79–99% of loads are predictable (Table 1).
Affine address generationComputing an address as a linear function of another value — the formula TSP learns. Covers direct field accesses, pointer chasing, and striding loops.
Supporting terms
Hardware task schedulerA chip component that holds task queues and dispatches tasks to cores — eliminates software scheduling overhead (>100× faster [Hive, ISCA'22]).
Fine-grain taskA task with 10s–100s of instructions (~100 cycles). Enabled by hardware scheduling; too small for software runtimes to manage efficiently.
Enqueue / dequeueInstructions that add/remove task descriptors from hardware queues. The core "pops" tasks via dequeue; the sequence is unpredictable (e.g., priority-ordered).
Argument-based prefetchingTSP's simplest mode: compute address from task arguments (e.g., v->dist = v + 8). Handles direct field accesses.
Indirect prefetchingTSP chaining: prefetch A, wait for value, use it to compute B's address, prefetch B, repeat. Handles pointer-chasing traversals.
Inner-loop prefetchingTSP learns base address, stride, and loop bound; issues all iterations' prefetches simultaneously. Handles short loops within tasks.
Executing Table (ET)Per running task: maps load PC → (address, value, count). Used to train PT entries by observing actual addresses.
Usefulness counterPer PT row: +1 on correct prediction, −128 on misprediction. If negative, TSP stops prefetching that row. Avoids cache pollution.
Moot taskA task that exits early without doing work (e.g., guarded by an early-return check). TSP over-prefetches moot tasks because it learns the full function.
Priority inversionMRS dispatching a lower-priority ready task before a higher-priority pending one. Can hurt work efficiency for priority-sensitive algorithms.
Speculative executionRunning tasks out of order and aborting those that violate data dependences (Swarm). TSP can trigger aborts by prefetching conflicting addresses.
Latency-bound vs. bandwidth-boundLatency-bound: core stalls waiting for individual accesses (TSP helps). BW-bound: memory fully utilized (TSP can't help much).

Concepts to shore up first

Hardware task parallelism and Swarm easy

Jeffrey et al. · MICRO 2015

The baseline architecture TSP targets. Swarm provides speculative out-of-order task execution with priority ordering — the "SpecRand/SpecSpat" systems in the paper. Read this to understand what a task descriptor is and how dequeue works.

Caches, associativity, and LRU easy

explainer · 30 min

TSP prefetches into the L2 cache. Understanding cache structure (sets, ways, replacement) clarifies why "prefetching" ≠ "always helps" — aggressive prefetching can evict useful data. Search "cache associativity explained" (Hennessy & Patterson Ch. 2).

Related papers

Hive: Queuing Theory for Task Schedulers hard

Posluns, Zhu, Zhang, Jeffrey · ISCA 2022

Swarm successor that eliminates moot tasks via a "moot queue" — would fix TSP's over-prefetching problem. Also quantifies that software task schedulers are 3–60× slower than hardware.

IMP: Indirect Memory Prefetcher medium

Yu, Hughes, Satish, Devadas · MICRO 2015

The conventional prefetcher TSP adapts. IMP detects striding accesses and prefetches indirect loads; TSP does the same but seeds with task arguments. The paper shows IMP (I) grants minimal speedup because it can't trigger on short tasks.

Spatial Hints and Swarm Locality medium

Jeffrey et al. · MICRO 2016

The "SpecSpat" baseline in the paper. Spatial hints map tasks to tiles to localize data accesses — orthogonal to prefetching (TSP helps both with and without hints).

HD-CPS: Concurrent Priority Scheduling hard

Shan, Khan · HPCA 2022

The "NonspecRand" baseline. A non-speculative hardware scheduler that uses relaxed priority dispatch to reduce contention — TSP helps here too, but priority drift from MRS can hurt work efficiency.

Tesseract: Message-Triggered Prefetching medium

Ahn, Hong, Yoo, Mutlu, Choi · ISCA 2015

A processing-in-memory accelerator for graph analytics. Augments RPC descriptors with one prefetch hint per message — the "T" baseline in the paper (programmed TSP with 1 address). TSP generalizes this to arbitrary addresses and learns them automatically.

Runahead Execution and DVR hard

Mutlu, Stark, Wilkerson, Patt · HPCA 2003; Naithani et al. · ISCA 2023

An alternative approach: speculatively execute past stalling loads to generate future addresses. DVR (Decoupled Vector Runahead) is triggered by a stride prefetcher — could be combined with TSP to handle non-affine address generation (future work).
