ISCA 2026 · A Field Guide

When does a byte of data earn its rent in DRAM?

In 1987 Jim Gray answered "every five minutes." This paper redoes the arithmetic from first principles for the AI era — and the answer collapses to a few seconds, turning NAND flash from cold storage into an active memory tier.

📄 "Five-Minute Rule 40 Years Later" · Zhang et al. · ScaleFlux / NVIDIA / Stanford 🎓 A little storage economics assumed — the primers cover it ⏱ ~30 min read
⏱️

Keep a page in DRAM when its reuse interval is shorter than the break-even time; otherwise serve it from flash.
Break-even = per-access storage cost ÷ per-page DRAM rent. In 1987 that was minutes. In 2025, seconds.

Before you begin

Nine ideas the paper assumes you know

Each card clears up one concept this revisit builds on. They're collapsed — tap any open. Read in order the first time, or jump to whatever looks unfamiliar.

DRAM — the fast tier (the "rent") flash / SSD / NAND — the active tier host & feasibility limits cost / economics / break-even
1The original five-minute rule (Gray & Putzolu, 1987)▾

In the 1980s, databases lived on two tiers: fast, expensive DRAM and slow, cheap hard disk. The question Gray asked: when is it cheaper to keep a record in DRAM than to keep fetching it from disk?

His answer was a break-even reuse interval. A 1 KB record accessed more often than every five minutes should live in DRAM; less often, leave it on disk. The number was the point where DRAM "rent" equals disk fetch cost — pure economics, no physics.

Now you're ready: this paper keeps Gray's break-even idea but re-derives the number with modern costs, host overheads, and device physics — and watches it shrink from minutes to seconds.
2The memory hierarchy & "tiers"▾

Computers stack storage by speed and price: registers → caches → DRAM → flash/SSD → disk/tape. Each tier down is bigger and cheaper per byte, but slower. Data placement = deciding which tier each piece of data lives in.

The classic adage the paper quotes: "Tape is dead, disk is tape, flash is disk." Each tier slides down a rung. This paper argues flash is now sliding up to sit beside DRAM — an active tier, not just capacity.

Now you're ready: the whole paper is about one boundary — DRAM ↔ flash — and where to draw the line between them.
3IOPS, latency, and bandwidth▾

Three different storage scorecards. IOPS = I/O operations per second (how many separate small reads/writes you can do). Latency = time for one operation to finish. Bandwidth = bytes per second moved.

For small, random accesses (the AI-era pattern), IOPS is king: you want millions of tiny independent reads. A drive can have huge bandwidth yet poor IOPS, or vice-versa. This paper lives in the small-block, high-IOPS regime.

Now you're ready: "Storage-Next" SSDs are defined by scaling small-block IOPS — e.g. 50M IOPS at 512 B vs 10M at 4 KB — which is what collapses the break-even time.
4How an SSD actually works (NAND, channels, planes)▾

An SSD is a controller + some internal DRAM (for its address map) + a NAND subsystem. NAND is organized into channels (parallel buses), each with several dies, each die holding several planes that can operate in parallel.

Key timings: τsense (read a page), τprog (write a page), and τCMD (bus occupancy per command). Peak IOPS comes from sensing/programming latency × how many planes and channels run in parallel.

Now you're ready: the paper's "first-principles SSD model" computes peak IOPS straight from these timings and counts — not from a vendor datasheet.
5Write amplification & the read:write ratio▾

NAND can't overwrite in place — it must erase whole blocks first. Garbage collection shuffles live data around, so one host write can cause several physical writes. That multiplier is write amplification (ΦWA ≥ 1).

Workloads also have a read:write ratio (ΓRW). AI workloads are read-heavy; the paper fixes ΓRW = 90:10 and conservatively sets ΦWA = 3. Writes (and their amplification) steal IOPS from reads.

Now you're ready: these two knobs scale the achievable IOPS in the device model — more writes and more amplification mean fewer usable read IOPS.
6Amortized capital cost (CapEx) and "rent"▾

The rule is an economic argument, so everything is a cost. Amortized capital cost spreads a component's purchase price over its useful life. Holding a page in DRAM has a recurring "rent" — the slice of DRAM cost it occupies over time.

Against that rent you weigh the per-access cost of fetching from storage instead. Cache the page only if the rent is cheaper than the repeated fetches. (All costs here are CapEx, matching Gray's original.)

Now you're ready: break-even is literally "rent = saved fetch cost." Solve that equation for the reuse interval and you get the rule.
7Host overhead: the CPU/GPU is not free▾

Issuing an I/O costs the host processor real work: building a request, ringing a doorbell, handling the completion interrupt, moving bytes. At 100 IOPS (an old HDD) that's negligible. At 50 million IOPS it dominates.

So each core (or GPU SM) can only sustain so many IOPS — about 1M IOPS/core for a CPU, 4M IOPS/SM for a Hopper GPU. The host becomes a feasibility limit, not just a cost.

Now you're ready: the paper's headline twist over older revisits is charging this host cost — and it's why GPUs (more IOPS capacity) beat CPUs here.
8Queueing & tail latency (M/D/1)▾

If you push a device near 100% utilization, requests queue and latency explodes. A simple model is the M/D/1 queue: random (Poisson) arrivals, Deterministic service time, 1 server (one NAND channel).

Utilization ρ (0–1) is how busy the channel is. As ρ → 1, both mean and tail latency (e.g. the 99th percentile) blow up. To hit a latency target you must cap ρ below 1 — which caps the usable IOPS.

Now you're ready: the paper uses M/D/1 to convert a latency target into a maximum safe utilization ρmax, then multiplies peak IOPS by it.
9Why this matters for memory tiering (your area)▾

OS-managed DRAM/NVM tiering ("LtRAM"-style) asks exactly Gray's question at runtime: which pages stay in fast memory and which get demoted to a slower tier? The cost of being wrong — and how aggressive demotion should be — depends on the break-even interval.

If break-even is minutes, you keep almost everything hot. If it collapses to seconds, far more pages "deserve" demotion to flash, and the tiering policy must churn faster and tolerate flash in the hot path.

Now you're ready: read this paper as the economic justification for aggressive tiering — it tells you where the demotion threshold should sit and why it's moving.
The whole story in one place

The five-minute rule, recomputed

The original heuristic, the three things this revisit adds, and why the number collapses from minutes to seconds.

In 1987, Gray and Putzolu balanced DRAM rent against disk fetch cost and got a break-even reuse interval of five minutes for a 1 KB page. Access more often than that → keep it in DRAM; less often → leave it on storage. Three later revisits (1997, 2007, 2019) updated the constants but kept the view purely economic — vendor price tags only — and still landed at the minute scale.

This paper argues that economics-only view is now both unrealistic and non-actionable. It rebuilds the rule from first principles, adding three ingredients the old form ignored: (1) host cost — the CPU/GPU cycles and DRAM bandwidth every I/O consumes; (2) device physics — SSD IOPS and cost derived from NAND timing, parallelism, and write amplification instead of datasheets; and (3) feasibility limits — finite host IOPS and a tail-latency target that bound the usable (not peak) SSD IOPS.

The driver of the collapse is the rise of "Storage-Next" SSDs, engineered for fine-grained random access. From the paper's own device model, a low-latency SLC NAND configuration delivers roughly 57M IOPS at 512 B but only 11M at 4 KB — small-block IOPS scales up sharply. Because break-even depends on IOPS-per-dollar, that 10×-class jump in small-block IOPS/$ drags the break-even interval down with it.

The payoff: under realistic configurations the DRAM↔flash break-even collapses from minutes to seconds. On a CPU+DDR host with SLC Storage-Next it's ~34 s at 512 B, ~10 s at 4 KB; pair the same SSD with a GPU+GDDR host and 512 B falls to ~5 s — a roughly 7× reduction, driven by the GPU's far larger IOPS capacity and better IOPS/$. Flash is promoted from passive storage to an active tier of the memory hierarchy.

🍯 Key takeaways

  • The break-even interval is just per-access storage cost ÷ per-page DRAM rent. Cache a page only if its reuse interval is shorter.
  • Old revisits were economics-only; this one adds host cost, device physics, and feasibility limits (host IOPS + tail-latency).
  • Peak SSD IOPS is derived from NAND timing/parallelism: ~57M @ 512 B vs ~11M @ 4 KB for SLC — small blocks win big.
  • Because break-even tracks IOPS/$, that small-block IOPS surge collapses the interval from minutes → seconds.
  • GPUs (≈4M IOPS/SM) beat CPUs (≈1M IOPS/core) as I/O engines: 512 B break-even ~34 s (CPU) → ~5 s (GPU).
  • Host IOPS capacity dominates the gains; tightening the tail-latency target moves the needle only slightly.
  • A workload-aware framework turns this into provisioning guidance: how much DRAM to buy for viability vs. economic optimality.
Seven everyday pictures

How to think about the rule

Each picture stands alone — read them in any order. Every card ends with an honest note about where the picture stops being true.

The core idea · Break-even

Rent vs. ride-share

Do you rent an apartment downtown (pay every month, but it's always there) or take a cab in each time you visit? If you visit often, the rent is cheaper; if rarely, the cab wins. DRAM is the apartment (recurring "rent"), an SSD fetch is the cab fare. Break-even is the visit frequency where they cost the same — visit more often and DRAM pays off.

Where it misleads: rent and fare are fixed prices, but the paper's cab fare (SSD per-access cost) is itself derived from device physics and shrinks as small-block IOPS rise — which is what moves the break-even point.
Storage-Next · Small-block IOPS

Many small deliveries vs. one big truck

A courier can drop off thousands of small envelopes an hour, or far fewer big crates. Storage-Next SSDs are built for envelopes: 57M tiny 512 B reads per second vs. only 11M at 4 KB. Conventional SSDs are crate-only — flat at ≤4 KB because their ECC and controller are built around a 4 KB page. Shrinking the parcel raises total throughput, which is what cheapens each access.

Where it breaks: envelopes aren't free to address — each still needs a logical-to-physical lookup and a host doorbell. Past a point the courier's dispatcher (host IOPS) saturates, not the trucks, so smaller isn't endlessly better.
Feasibility · Host IOPS ceiling

Checkout lanes, not shelf stock

A supermarket can stock unlimited groceries (SSD bandwidth), but customers leave only as fast as the checkout lanes can scan them. Each CPU core is one lane at ~1M IOPS; a GPU SM is a faster lane at ~4M. Add SSDs all you like — if the host can't issue the I/Os, the extra device IOPS is stranded. That's why the paper charges the host and treats its capacity as a hard ceiling.

Where it breaks: real lanes can be sped up with self-checkout (leaner I/O stacks, hardware offload) — the paper flags exactly this as future work, so the ceiling isn't truly fixed.
Latency · M/D/1 queueing

The highway near rush hour

A road flows fine at 70% capacity; push it to 99% and a single extra car triggers stop-and-go. NAND channels are the same: run them near full utilization and tail latency explodes. To honor a 99th-percentile target you must keep utilization ρ below a cap (e.g. ρ ≤ 90% for a 13 µs target at 512 B), which means using only part of the SSD's peak IOPS.

Where it breaks: the paper finds tail latency is a minor dial — relaxing 512 B GPU tail from 7 µs to 85 µs buys only ~1.5 s. The host IOPS ceiling, not latency, is the real bottleneck. The highway is rarely the binding constraint here.
Provisioning · The cached set S(T)

The fridge and the pantry

You keep what you reach for hourly in the fridge (small, premium DRAM) and the rest in the pantry (big, cheap flash). Pick a threshold T: everything reused more often than T goes in the fridge — that's the set S(T). The paper sizes the fridge three ways: enough to not starve bandwidth (TB), enough to not overload the pantry runs (TS), and what actually fits (TC).

Where it breaks: a real fridge has fixed shelves; here the "optimal fridge size" is set by economics (τbreak-even) and can demand caching nearly the entire dataset at 512 B — the analogy understates how big the cost-optimal cache can get.
Methodology · First principles

Cooking from the recipe, not the box

A frozen dinner lists one cook time on the box (the vendor datasheet). But if you know the oven temperature and the dish's mass, you can compute the time for any portion. The paper derives SSD IOPS and cost from NAND timings (τsense, τprog, τCMD), plane/channel counts, and write amplification — so it can reason across block sizes and configs no datasheet covers.

Where it breaks: a model is only as good as its parameters. The authors pick mature-2025 NAND values and a conservative ΦWA=3; different process nodes shift the absolute numbers (though they argue the trend is robust).
Validation · MQSim-Next

A flight simulator before the flight

Before betting on a design you fly it in a simulator. MQSim-Next is a calibrated SSD simulator that models NAND timing, multi-plane reads, ECC, and channel scheduling. When its simulated IOPS lands right next to the closed-form model's prediction, you trust the model — and can then run "what-if" sweeps (channel bandwidth, read:write mix, ECC failure rate) cheaply.

Where it breaks: the simulator and the analytic model share assumptions (e.g. a non-limiting controller), so agreement validates internal consistency more than real silicon — there's no hardware measurement in the loop.
The full derivation

How the rule is rebuilt

Section by section, following the paper's four research questions, with the key formulas and figures rebuilt as visuals you can read at a glance.

1 · The classical rule, in unit-consistent form

The original rule (Gray & Putzolu, 1987) balances DRAM rent against the cost of repeatedly fetching a page from storage. The paper restates it cleanly. Let C_page^DRAM be the amortized cost of holding one page in DRAM, and C_IO^SSD the amortized cost per storage access (SSD cost ÷ peak IOPS). Break-even is when, over a reuse interval T, the rent equals the avoided fetch cost:

# rent over interval T  =  one avoided storage access
T · C_page^DRAM = C_IO^SSD
T_break-even = C_IO^SSD / C_page^DRAM   # per-access storage cost ÷ per-page DRAM rent

Under 1987 HDD-era parameters this ratio came out to minutes (the famous "five minutes" for a 1 KB record). The paper's full calibrated formula reduces to exactly this when host and bandwidth costs are ignored and full peak SSD IOPS is assumed.

⚠
Two gaps in the economics-only view. (A) Insufficient realism: it treats host resources as free — fine at 100–200 HDD IOPS, wrong at tens of millions of SSD IOPS — and trusts vendor peak specs. (B) Missing feasibility: minimizing $/GB or $/IOPS can't guarantee the config is actually deployable under host-IOPS, latency, and DRAM bandwidth/capacity limits.

2 · RQ1 — Calibrated economics (add the host cost)

The paper charges every resource an I/O touches. For a block of size l_blk reused every τ, caching it in DRAM avoids a recurring cost with three parts: host processor, host DRAM bandwidth, and the SSD access itself. The "rent" is the DRAM capacity it occupies. Setting rent = saving and solving for τ gives the calibrated break-even (Eq. 1):

# Eq. 1 — calibrated break-even interval
τ_break-even =
  ( $CORE/IOPS_CORE          # host processor cost per I/O
  + l_blk·$H_DRAM/B_H_DRAM     # host DRAM bandwidth cost per I/O
  + $SSD/IOPS_SSD )           # SSD access cost per I/O
  ·  C_H_DRAM / (l_blk·$H_DRAM)  # ÷ per-page DRAM rent

The numerator has units of $/IO; the denominator is the inverse of $/MB-of-DRAM, so τ comes out in time, as expected. The two new ideas: it explicitly charges host resources, and it replaces datasheet peaks with an IOPS_SSD and $SSD derived from device physics (next section).

3 · First-principles SSD IOPS (where 57M comes from)

Peak SSD IOPS is the minimum of four limits: NAND device parallelism, channel bandwidth, controller address-translation bandwidth, and PCIe packet rate. For the small-block regime the first two bind. With read fraction R_r and write fraction R_w (set by ΓRW and ΦWA):

# per-die NAND peak (sensing for reads, programming for writes)
IOPS_NAND = R_r·(N_Plane/τ_sense) + R_w·(N_Plane·l_PG)/(τ_prog·l_blk)
# per-channel peak (τ_R = τ_CMD + l_blk/B_CH)
IOPS_CH   = R_r·1/τ_R + R_w·(l_PG/l_blk)/τ_W
# device-limited SSD IOPS over N_CH channels
IOPS_dev  = (Γ+1)/(Γ+2Φ−1) · N_CH · min(N_NAND·IOPS_NAND, IOPS_CH)
IOPS_SSD  = min(IOPS_dev, IOPS_xlat, IOPS_pcie)

Plugging in the Table I SLC parameters (τsense=5µs, τprog=50µs, NPlane=6, NCH=20, NNAND=4, BCH=3.6 GB/s, τCMD=150 ns) with ΓRW=90:10 and ΦWA=3 yields ~57M IOPS at 512 B and ~11M at 4 KB — reproduced exactly in Worked Example ①.

4 · RQ2 — Constraint-aware break-even (usable, not peak)

Gray (and RQ1) assumed the host can fully use the SSD's peak IOPS. Reality imposes two ceilings. First, a tail-latency target: model each NAND channel as an M/D/1 queue and the p-th percentile latency is

# M/D/1 tail latency; ρ = channel utilization, p = percentile
τ_p(ρ) = (N_CH/IOPS_SSD^peak) · ρ/(2(1−ρ)) · ln(1/(1−p)) + τ_sense

Given a target, solve for the largest admissible ρmax; usable IOPS = ρmax · peak. Second, a host IOPS ceiling IOPS_proc^peak (≈1M/core CPU, ≈4M/SM GPU). The usable SSD IOPS is the smaller of the two:

IOPS_SSD = min( ρ_max·IOPS_SSD^peak ,  IOPS_proc^peak / N_SSD )
⚖️
Which constraint bites? The paper's sweeps show host IOPS capacity dominates: raising a CPU budget from 40M→100M IOPS cuts the 512 B interval from 83 s → 47 s, while relaxing the 512 B GPU tail-latency target from 7 µs → 85 µs buys only ~1.5 s. GPUs, with ~4× the IOPS capacity, run almost entirely device-limited and stay below ~7 s across block sizes.

5 · RQ3 — Workload-aware provisioning (three thresholds)

Now fold in the workload. Order blocks by access interval; let S(T) be the set reused at least every T (the blocks worth caching). As T grows, more blocks are cached, cached throughput Ψc rises and uncached Ψd falls. The host-DRAM bandwidth demand under a zero-copy stack is B_DRAM^use(T) = Ψ_c(T) + 2·Ψ_d(T) (a miss costs one DMA in + one read out). Three thresholds isolate the binding resource:

ThresholdMeaningSet by
T_Bsmallest T where DRAM traffic ≤ DRAM bandwidthhost DRAM bandwidth BDRAM
T_Ssmallest T where uncached throughput ≤ usable SSD bandwidthSSD IOPS × NSSD (from RQ2)
T_Clargest T whose cached set still fits in DRAMDRAM capacity CDRAM

The decision rules fall straight out: the platform is viable if max(T_B, T_S) ≤ T_C, and it operates at the economic optimum if τ_break-even ∈ [max(T_B,T_S), T_C]. If not, the violated inequality tells you exactly what to upgrade — more DRAM bandwidth, more SSD throughput, or more DRAM capacity.

# provisioning diagnosis (Section V-A)
T_v = max(T_B, T_S)                 # viability threshold
if T_v ≤ T_C:            viable
if τ_break-even ∈ [T_v, T_C]: economically optimal
elif T_B > T_C ≥ T_S:   raise DRAM bandwidth
elif T_S > T_C ≥ T_B:   raise SSD throughput / host IOPS
else:                  add DRAM capacity (or upgrade bandwidth)
💡
The provisioning payoff. Pick CDRAM so that TC = max(TB, TS) for minimum-cost viability. On CPU+DDR the break-even interval is so long that the cost-optimal cache holds nearly the whole 512 GB–1 TB dataset; on GPU+GDDR with Storage-Next both TB and TS drop below 5 s, so viability needs far less DRAM.

6 · Validation & the software rethink (MQSim-Next, RQ4)

MQSim-Next extends the MQSim simulator with three modern features — SCA command timing, independent multi-plane reads, and transfer/sense overlap — plus a configurable two-layer (BCH+LDPC) ECC model so 512 B reads needn't pay a full 4 KB decode. Its simulated IOPS lands right next to the analytic model (slightly higher, since the model conservatively assumes ΦWA=3), and reproduces the expected sensitivities: IOPS drops 82M (read-only) → 68M (90:10) → 52M (70:30) → 34M (50:50) as writes/GC compete, and rises 68M → 85M as channel bandwidth grows 3.6 → 5.6 GB/s.

The collapse to seconds blurs the memory/storage line, so the paper closes with two SSD-resident case studies. A blocked-Cuckoo KV store with no DRAM-resident index (all DRAM caches hot pairs) sustains 100+ Mops/s on GPU+Storage-Next — in-memory-class throughput from flash. A two-stage progressive ANN search fetches 512 B reduced vectors first (small-block, IOPS-bound) and only re-ranks a small promoted fraction with full vectors, sustaining >98% recall and a 2–3× throughput edge over normal SSDs.

🍯
Both case studies follow the same two design principles the seconds-scale regime rewards: (1) re-architect for ultra-high small-block random IOPS (fine-grained access, concurrency), and (2) spend cheap flash $/GB freely — sparse or over-provisioned structures — to buy speed and simplicity.
Follow the numbers, step by step

Worked examples

Four calculations carried all the way through with the paper's actual parameters — every arithmetic step recomputed here, not copied.

① Deriving 57M IOPS at 512 B (SLC, from Table I)

Use τsense=5µs, τprog=50µs, lPG=4 KB, NPlane=6, NCH=20, NNAND=4, BCH=3.6 GB/s, τCMD=150 ns, ΓRW=90:10 (so Γ=9), ΦWA=3.

  1. Read/write fractions. Rr = (Γ+Φ−1)/(Γ+2Φ−1) = (9+3−1)/(9+6−1) = 11/14 ≈ 0.786; Rw = Φ/(Γ+2Φ−1) = 3/14 ≈ 0.214.
  2. Per-die NAND peak. Rr·(6/5µs) + Rw·(6·4096)/(50µs·512) = 0.786·1.2M + 0.214·0.96M ≈ 1.15M IOPS/die. ×NNAND=4 → 4.6M/channel-of-dies.
  3. Per-channel limit. τR=150ns+512/3.6e9≈292ns; τW=150ns+4096/3.6e9≈1.29µs. IOPSCH = 0.786/292ns + 0.214·(4096/512)/1.29µs ≈ 2.69M + 1.33M ≈ 4.02M/channel.
  4. Take the min, scale by channels and the front factor. min(4.6M, 4.02M)=4.02M. Front factor (Γ+1)/(Γ+2Φ−1)=10/14≈0.714. IOPS = 0.714·20·4.02M ≈ 57.4M.
  5. Repeat at 4 KB: the channel term collapses (τR≈1.29µs), giving ≈ 11.1M. ✓ matches the paper's 57M / 11M and Table II's 57.4M / 11.1M baseline.
🍯
Insight: the ~5× small-block advantage isn't a datasheet claim — it falls out of NAND timing. At 512 B the channel binds and scales as BCH/lblk; at 4 KB the channel term shrinks ~8× and the device side takes over.

② Why the break-even is "minutes → seconds" in spirit

The structural reason, using τ = C_IO^SSD / C_page^DRAM and the paper's stated outputs.

  1. Break-even scales as per-access SSD cost. C_IO^SSD = $SSD / IOPS_SSD. Hold $SSD roughly fixed and raise IOPS_SSD ~5× (4 KB→512 B), and the per-access cost — hence τ — drops with it.
  2. 1987 anchor: HDD at ~100 IOPS and minute-scale break-even (the original "5 minutes" for 1 KB).
  3. 2025 SLC on CPU+DDR: the paper reports ~34 s at 512 B falling to ~10 s at 4 KB. (Larger blocks → higher DRAM "rent" per page → shorter τ.)
  4. Swap CPU→GPU host: better IOPS/$ and bandwidth drop 512 B from ~34 s to ~5 s. Check the ratio: 34/5 ≈ 6.8×, matching the paper's stated "≈7× reduction."
  5. Conclusion: from minutes (1987) to ~34 s (CPU) to ~5 s (GPU) — the same equation, with IOPS/$ raised by orders of magnitude.
💡
The interactive on the "How It Works" tab walks these endpoints. Note the direction: bigger blocks give shorter break-even (more DRAM rent), while faster small-block IOPS also shortens it — two different levers.

③ Host IOPS vs. latency — which constraint actually moves the needle?

From the RQ2 sweeps (Section IV), using the paper's reported numbers.

  1. Host IOPS sweep (512 B, CPU+DDR): raising the CPU budget 40M → 100M IOPS shortens the interval 83 s → 47 s — a 36 s swing.
  2. Latency sweep (512 B, GPU+GDDR): relaxing the 99th-percentile target 7 µs → 85 µs shortens the interval by only ~1.5 s.
  3. Ratio of leverage: 36 s vs 1.5 s ≈ 24× more impact from the host-IOPS dial than the latency dial.
  4. Regime check: at 4 KB the CPU 512 B→4 KB interval stays near 10 s regardless of CPU budget — there the device limits, not the host, so raising the host budget does nothing.
  5. Takeaway the paper draws: GPUs (≈4M IOPS/SM) move the system out of the host-limited regime; that, not latency tuning, is what drives the few-seconds result.
🍯
Insight: "feasibility-aware" doesn't mean "latency-bound." The dominant feasibility limit here is how many I/Os the host can issue — a software/architecture problem, which is why host-side I/O optimization tops the future-work list.

④ Recomputing a tail-latency tier (Table IV, ρ=0.90)

Check the 512 B, 99th-percentile entry with the M/D/1 formula: τp = (NCH/IOPSSSD)·ρ/(2(1−ρ))·ln(1/(1−p)) + τsense.

  1. Per-channel service time. NCH/IOPSSSD = 20 / 57.4M ≈ 0.348 µs.
  2. Queue factor at ρ=0.90. ρ/(2(1−ρ)) = 0.90/(2·0.10) = 4.5.
  3. Percentile factor at p=0.99. ln(1/(1−0.99)) = ln(100) ≈ 4.605.
  4. Combine + sensing. 0.348 µs · 4.5 · 4.605 + 5 µs ≈ 7.22 µs + 5 µs ≈ 12.2 µs.
  5. Compare to the paper. Table IV lists 13 µs for the 90% tier at 512 B. ✓ (The small gap is rounding / their exact peak-IOPS value.) The 0.70 / 0.80 / 0.99 tiers recompute to ≈6.9 / 8.2 / 84 µs vs the paper's 7 / 9 / 85 µs.
💡
The ln(1/(1−p)) term is the heavy-traffic (Kingman) approximation: tail latency grows linearly in the percentile's log and blows up as ρ→1. That's the queueing knee that caps usable IOPS below peak.

Peak IOPS = min(device/NAND limit, channel limit). The device limit (set by τsense/τprog and plane count) barely changes with block size. The channel limit scales as ≈ BCH/lblk, so it's huge for small blocks and shrinks ~8× from 512 B to 4 KB. At 512 B the channel limit is the binding (smaller) term but still high → ~57M; at 4 KB the channel term has dropped below the device term → ~11M. Conventional SSDs stay flat ≤4 KB because their ECC/controller decode a full 4 KB page regardless.Misconception: that smaller blocks are slower. For random access on Storage-Next NAND, smaller blocks yield more IOPS, because the channel sheds bytes faster than the device sheds operations.
Problem 4 · hard
For 512 B SLC Storage-Next (NCH=20, peak ≈57.4M IOPS, τsense=5µs), use the M/D/1 tail formula to find the 99th-percentile latency at ρ=0.80, and say whether that's tighter or looser than the 13 µs / ρ=0.90 tier.
Show answer
Service time = 20/57.4M ≈ 0.348 µs. Queue factor at ρ=0.80 = 0.80/(2·0.20) = 2.0. Percentile factor = ln(100) ≈ 4.605. τ99 ≈ 0.348·2.0·4.605 + 5 = 3.21 + 5 ≈ 8.2 µs (paper's Table IV lists 9 µs for the 80% tier). That is a tighter latency target than the 13 µs / ρ=0.90 tier — a stricter SLO admits a lower ρmax and therefore less usable IOPS.Misconception: that a smaller ρ gives worse latency. Lower utilization gives better (smaller) tail latency; the cost is leaving more peak IOPS on the table.
Problem 5 · medium
On a CPU+DDR host at 512 B, raising the CPU IOPS budget 40M → 100M cut the break-even from 83 s to 47 s, but at 4 KB it stayed near 10 s no matter the budget. Why the difference?
Show answer
At 512 B the system is host-limited: the CPU can't issue enough I/Os to saturate the SSD's high small-block IOPS, so giving it more budget directly serves more accesses and shortens the interval. At 4 KB the SSD's own peak is only ~11M IOPS, so the system is device-limited — the host budget is already more than enough, and adding more does nothing. The crossover from host- to device-limited depends on both block size and host budget.Misconception: that "more host IOPS always helps." It only helps while the host is the bottleneck; once the device is the limit, host upgrades are wasted.
Problem 6 · medium
Define TB, TS, and TC. Give the viability condition and the economic-optimality condition in terms of them and τbreak-even.
Show answer
TB = smallest cache-threshold T at which I/O-induced DRAM traffic ≤ host DRAM bandwidth; TS = smallest T at which uncached throughput ≤ usable SSD bandwidth; TC = largest T whose cached set S(T) still fits in DRAM capacity. Viable iff max(TB,TS) ≤ TC. Economically optimal iff τbreak-even ∈ [max(TB,TS), TC]. If a condition fails, the violated inequality names the resource to upgrade.Misconception: that viability and optimality are the same. A system can be viable (meets throughput/latency) yet off the economic optimum — you'd then add DRAM up to where τbreak-even sits.
Problem 7 · hard
The paper notes a DRAM cache miss costs DRAM bandwidth 2Ψd while a hit costs Ψc, giving B_DRAM^use = Ψc + 2Ψd. Why is a miss twice as expensive on the DRAM bus, and why does total DRAM bandwidth demand fall as the cache grows?
Show answer
Under a zero-copy stack, a hit is one DRAM read by the processor (1× the data over the bus). A miss is two DRAM transfers: the SSD→DRAM DMA that brings the block in, then the processor's read of it — hence 2×. As the cache grows (T rises), S(T) expands, so Ψc grows and Ψd shrinks while their sum is fixed; replacing 2Ψd terms with 1Ψc terms strictly lowers B_DRAM^use. More caching converts expensive misses into cheaper hits.Misconception: that caching always raises DRAM bandwidth pressure. For I/O-induced traffic it's the opposite — a bigger cache means fewer DMAs, so less bus traffic, not more.
Problem 8 · easy
You manage OS-level DRAM/NVM tiering. The break-even interval just dropped from "minutes" to "a few seconds." What should change about your demotion policy, and what new risk appears?
Show answer
Far more pages now fail the break-even test, so the policy should demote more aggressively — a page idle for a few seconds no longer earns its DRAM rent. The new risk is churn / flash in the hot path: with a seconds-scale threshold the tiering engine must re-evaluate and migrate pages much faster, tolerate flash latency on more accesses, and avoid thrashing pages that hover near the threshold. The paper's host-IOPS finding also warns that the issuing cost of all that I/O can become the bottleneck.Misconception: that a shorter break-even means "use less DRAM, simpler win." It also means more migrations and more reliance on flash being fast — the policy gets harder, not easier. (This connection is the guide's framing, grounded in the paper's economics.)
Quick answers

Frequently asked questions

Twelve questions a reader asks first. Tap any to expand.

Q1What exactly is the "five-minute rule"?▾
Jim Gray and Gianfranco Putzolu's 1987 heuristic for data placement: a 1 KB record accessed more often than every five minutes should live in DRAM; less often, leave it on disk. The number is the reuse interval where DRAM "rent" equals the cost of re-fetching from storage. This paper keeps the idea but recomputes the number for 2025 hardware.
Q2What's actually new here versus the 1997 / 2007 / 2019 revisits?▾
The earlier revisits updated the price constants but stayed economics-only — vendor specs, no host costs, no feasibility limits. This paper adds three things: host I/O cost (CPU/GPU cycles + DRAM bandwidth per I/O), a first-principles SSD model (IOPS and cost from NAND physics, not datasheets), and feasibility constraints (host-IOPS ceiling + tail-latency-bounded usable IOPS). It then wraps these in a workload-aware provisioning framework.
Q3Why does the break-even collapse from minutes to seconds?▾
Break-even ≈ per-access SSD cost ÷ per-page DRAM rent, and per-access SSD cost = $SSD ÷ IOPS_SSD. Storage-Next SSDs raise small-block IOPS by ~10× (e.g. ~50M at 512 B vs ~10M at 4 KB), so IOPS/$ jumps and the per-access cost — and thus the interval — falls. On a GPU host with high IOPS capacity it reaches ~5 s at 512 B.
Q4What is a "Storage-Next" SSD?▾
An SSD engineered for fine-grained random access that scales small-block IOPS — delivering far higher IOPS per dollar as access granularity shrinks (the paper's model gives ~57M IOPS at 512 B for low-latency SLC NAND). It contrasts with conventional SSDs whose IOPS is flat at ≤4 KB because their ECC and controller are built around a 4 KB page. The term ties to NVIDIA's Storage-Next initiative and SSD-vendor roadmaps.
Q5Where do the IOPS numbers come from — are they vendor projections?▾
No. They're derived from a first-principles device model using representative NAND timing/architecture parameters (Table I): sensing/programming latencies, plane and channel counts, channel bandwidth, command timing, plus a read:write ratio and write amplification. The 57M-at-512 B figure for SLC is a direct calculation (reproduced in Worked Example ①), not a datasheet number, and a sensitivity sweep shows it stays in the tens-of-millions across pessimistic/optimistic settings.
Q6Why do GPUs beat CPUs as I/O engines here?▾
Issuing I/Os costs host work, and each engine has an IOPS ceiling: ~1M IOPS per CPU core vs ~4M per GPU SM (the paper follows NVIDIA's SCADA numbers on Hopper). GPUs also pair with higher-bandwidth GDDR and have better IOPS/$. So a GPU host stays out of the host-limited regime and drives the 512 B break-even from ~34 s (CPU) to ~5 s.
Q7Does tail latency matter for the result?▾
Surprisingly little. The paper models each NAND channel as an M/D/1 queue and converts a tail-latency target into a max utilization ρmax. But tightening the 512 B GPU tail from 7 µs to 85 µs changes the break-even by only ~1.5 s, whereas host IOPS capacity swings it by tens of seconds. Host capacity dominates; latency is a minor dial — unless the host budget is already the limiter.
Q8What are TB, TS, and TC for?▾
They turn the rule into provisioning advice. TB, TS, TC are cache-threshold values bounded by DRAM bandwidth, usable SSD bandwidth, and DRAM capacity respectively. The platform is viable if max(TB,TS) ≤ TC, and economically optimal if τbreak-even falls in [max(TB,TS), TC]. A failed condition names exactly which resource to upgrade.
Q9What is MQSim-Next and why build it?▾
A calibrated SSD simulator extending MQSim with modern device features (SCA command timing, independent multi-plane reads, transfer/sense overlap, a configurable BCH+LDPC ECC model, many I/O queues). It validates the analytic model — simulated IOPS tracks the model closely — and lets the authors run sensitivity sweeps (read:write mix, channel bandwidth, ECC failure rate) that would be impractical analytically.
Q10What are the two case studies showing?▾
That seconds-scale caching enables new SSD-resident software. (1) A blocked-Cuckoo KV store with no DRAM-resident index — all DRAM caches hot pairs — sustains 100+ Mops/s on GPU+Storage-Next, rivaling in-memory stores like FASTER. (2) A two-stage progressive ANN search fetches small 512 B reduced vectors first and re-ranks only a small promoted fraction with full vectors, keeping >98% recall and a 2–3× edge over normal SSDs. Both exploit ultra-high small-block IOPS plus cheap flash $/GB.
Q11What does this mean for OS-managed DRAM/NVM memory tiering?▾
It's the economic case for aggressive tiering. If break-even is seconds, a page idle for a few seconds no longer "deserves" DRAM, so far more pages should be demoted to flash — and the tiering engine must churn faster and tolerate flash in the hot path. The paper's host-IOPS finding is a caution: the cost of issuing all that migration I/O can itself become the bottleneck. (This tiering framing is the guide's connection, grounded in the paper's economics — the paper itself targets data-placement/provisioning, not OS page tiering specifically.)
Q12What did the authors flag as limitations / future work?▾
Cost/device modeling uses normalized 2025-era parameters (no process-scaling or TCO/OpEx yet, e.g. power); no device endurance/retention modeling; workloads are read-dominant, large-footprint, single-tenant (no write-heavy, transactional, or multi-tenant cases); the analysis assumes local PCIe/NVMe paths (no disaggregated fabrics or CXL tiers); and host-side I/O optimization plus the broad SSD-resident algorithm design space remain open. They argue the qualitative trend — the IOPS-driven collapse — is robust even as absolute numbers shift.

Glossary, source notes & further reading

Core terms
Five-minute ruleGray & Putzolu's 1987 heuristic: cache a page in DRAM if its reuse interval is shorter than the break-even interval (then ≈5 min for a 1 KB record).
Break-even interval (τ)The reuse interval where DRAM rent equals avoided storage-fetch cost. τ = per-access storage cost ÷ per-page DRAM rent.
DRAM "rent"The recurring amortized capital cost of the DRAM capacity a cached page occupies: (l_blk/C_H_DRAM)·$H_DRAM.
Storage-Next SSDAn SSD engineered for fine-grained random access whose small-block IOPS scales up sharply (e.g. ~50M at 512 B vs ~10M at 4 KB); ties to NVIDIA's Storage-Next initiative.
IOPS / IOPS_SSD^peakI/O operations per second. Peak SSD IOPS = min(device, channel, translation, PCIe limits), derived from NAND physics here, not datasheets.
Usable IOPSPeak IOPS scaled down by feasibility: ρ_max (latency-bounded) and the host-IOPS ceiling. min(ρ_max·peak, IOPS_proc/N_SSD).
Host IOPS ceilingMax I/Os the processor can issue: ≈1M IOPS/CPU-core, ≈4M IOPS/GPU-SM. Often the binding feasibility limit.
T_B / T_S / T_CCache-threshold values bounded by DRAM bandwidth, usable SSD bandwidth, and DRAM capacity. Viable iff max(T_B,T_S) ≤ T_C.
Supporting terms
τ_sense / τ_prog / τ_CMDNAND per-plane read (sense), per-plane write (program), and per-command bus-occupancy latencies. SCA cuts τ_CMD from ~1.2 µs to 100–200 ns.
Write amplification (Φ_WA)Extra physical writes per host write caused by garbage collection (≥1). The paper conservatively sets Φ_WA=3.
Read:write ratio (Γ_RW)Fraction of reads vs writes in the workload. Fixed at 90:10 (read-heavy AI) for the headline study.
M/D/1 queueMarkov arrivals, Deterministic service, 1 server. Models a NAND channel to derive mean and tail latency vs utilization ρ.
ρ_maxLargest channel utilization that still meets the latency target; usable IOPS = ρ_max · peak IOPS.
Zero-copy I/OI/O path avoiding kernel↔user copies; a DRAM miss then costs one SSD→DRAM DMA + one processor read (the "2Ψ_d" term).
MQSim-NextThe paper's calibrated SSD simulator (extends MQSim) with SCA timing, independent multi-plane reads, transfer/sense overlap, and a BCH+LDPC ECC model.
SLC / pSLC / TLCNAND cell types: 1 bit/cell (fastest, costliest/GB), pseudo-SLC (TLC run as SLC), 3 bits/cell (slowest, cheapest/GB).
Blocked-Cuckoo KV storeCase study 1: an SSD-resident hash table with no DRAM index; each key maps to two SSD buckets (≈1.5 reads/lookup).
Two-stage progressive ANNCase study 2: fetch small reduced-dim vectors (512 B) to prune, re-rank a small promoted fraction with full vectors; >98% recall.
HNSWHierarchical Navigable Small World graph — the ANN index used; upper layers are hot and DRAM-cache-friendly.
