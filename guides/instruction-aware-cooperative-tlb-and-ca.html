Stop starving instructions when the pipeline is waiting

Server workloads have million-instruction footprints. The TLB shares space between instruction and data translations — but instruction misses stall the pipeline while data miss latency can be partially hidden. This is the plain-English story of iTP+xPTP, two tiny policies that fix it.

📄 "Instruction-Aware Cooperative TLB and Cache Replacement" · Chasapis+ ASPLOS'25 🎓 TLB/cache basics assumed — primers cover them ⏱ ~25 min read
🔄

In the STLB, keep instruction translations over data translations — even if it increases data page walks — then speed those walks by keeping their PTEs in L2. Instruction misses stall; data misses (mostly) don't.

Before you begin

Eight ideas the paper assumes you know

Tap any card to expand. Read in order the first time, or jump to whatever looks unfamiliar.

iTP (STLB policy) xPTP (L2C policy) instruction translations data translations
1TLBs and why they exist▾

Every memory access requires translating a virtual address to physical. The page table stores all mappings, but walking it takes hundreds of cycles. The TLB (Translation Lookaside Buffer) caches recent translations. CPU designs have small L1 TLBs (ITLB for instructions, DTLB for data) and a large last-level TLB (STLB) shared between instruction and data.

On a TLB miss, a hardware page table walker traverses the multi-level page table — each level requires a memory access. Frequent STLB misses devastate performance.

Now you're ready: iTP+xPTP is about making the STLB and L2 cache cooperate to minimize the cost of instruction translation misses, which stall the pipeline.
2Instruction vs data: which hurts more▾

Instruction references are on the critical path of pipeline execution — if the next instruction isn't translated, the frontend stalls immediately. Data references, by contrast, can often hide their latency via out-of-order execution: while one load waits, the core keeps working on independent instructions.

Both share the same STLB. Letting data translations crowd out instruction translations is a particularly bad trade, because you pay a pipeline stall for every instruction miss while data misses are partially hidden.

Now you're ready: iTP's entire strategy is to sacrifice data translations in the STLB to keep instruction translations, because instruction misses hurt more.
3LRU and why it ignores critical path▾

STLB replacement policies (typically LRU or variants) evict the least-recently-used entry. This is sensible for temporal locality, but it's oblivious to whether an entry is for an instruction or data translation. A data translation that was just touched evicts an instruction translation that's about to be needed, causing a pipeline stall.

LRU treats all translations as equal. The paper's insight: they're not — instruction translations deserve priority.

Now you're ready: iTP replaces LRU with a policy that knows the difference and acts on it.
4Page table entries stored in the cache▾

Page walks fetch page table entries (PTEs) from memory. These PTEs are cached in the L2 and LLC like any data. A miss on a PTE means the walk goes to DRAM, multiplying the cost of a TLB miss. A page walk serving an instruction translation fetches instruction PTEs; a walk serving data fetches data PTEs.

Previous translation-aware cache policies (TDRRIP, PTP) prioritize all PTEs equally. They don't distinguish instruction PTEs from data PTEs.

Now you're ready: xPTP is the first L2 replacement policy that specifically prioritizes data PTEs — to compensate for iTP increasing data page walks.
5The code footprint problem▾

Server/datacenter applications (unlike HPC codes) have huge instruction footprints — millions of instructions spread across many pages. A 64-entry ITLB covers at most 256 KB; server apps routinely exceed that by 10×, spilling instruction translations into the STLB.

The paper shows that Qualcomm Server workloads spend up to 20% of their cycles serving instruction address translation, while SPEC workloads spend nearly zero (their code fits in the ITLB). The problem is real and growing: instruction footprints increase 30% yearly.

Now you're ready: iTP+xPTP targets exactly these big-code workloads where instruction translations dominate STLB pressure.
6Insertion vs promotion vs eviction policies▾

A replacement policy has three knobs: insertion (where a new entry lands in the recency stack), promotion (where a hit entry moves to), and eviction (which entry to throw out). LRU inserts at MRU, promotes hits to MRU, evicts at LRU. You can keep LRU's eviction logic but change insertion/promotion to bias the stack.

iTP uses novel insertion and promotion rules based on whether an entry is instruction or data, plus a frequency counter. xPTP modifies only the eviction logic, favoring data PTEs.

Now you're ready: iTP's design is about smart insertion and promotion; xPTP's is about smart eviction.
7The STLB-L2 feedback loop▾

If you prioritize instruction translations in the STLB, data translations get evicted more often, triggering more data page walks. Each walk fetches data PTEs into the L2 cache. If those PTEs also get evicted quickly (because the L2 replacement policy doesn't know they matter), each data page walk becomes even slower, erasing the gains from keeping instruction translations.

This is why iTP alone delivers modest gains (2.2% single-thread) but iTP+xPTP together delivers 18.9%: xPTP prevents the L2 from undoing iTP's work.

Now you're ready: The two policies are cooperative — xPTP exists to amplify iTP, not to work alone.
8Phase adaptivity — when to enable xPTP▾

xPTP helps when STLB pressure is high, but it can hurt during phases with low STLB pressure (where keeping PTEs in L2 wastes space). So iTP+xPTP monitors STLB MPKI (misses per thousand instructions) and enables xPTP only when MPKI exceeds a threshold T1. Otherwise, plain LRU runs at the L2.

The paper's implementation uses two counters (miss count, instruction count) and a 1-bit status register to decide each 1000 instructions whether to enable xPTP.

Now you're ready: iTP+xPTP is adaptive — it automatically detects phases where cooperation helps and disables xPTP when it doesn't.

The whole story in one place

iTP+xPTP in a nutshell

The problem, the two policies that fix it, and how they work together.

Modern server workloads have million-instruction footprints. The 64-entry ITLB can't hold them all, so instruction translations spill into the shared STLB (1536 entries). The STLB uses LRU — it doesn't distinguish instruction translations from data translations, so a recently-touched data translation can evict an instruction translation the pipeline is about to need. Instruction TLB misses stall the pipeline; data TLB miss latency can be partially hidden in out-of-order execution. This asymmetry is the problem.

The paper proposes two cooperative replacement policies. iTP is an STLB policy that keeps instruction translations at the top of the recency stack and inserts data translations at the bottom. It uses a 3-bit frequency counter per entry to identify frequently-hit instruction translations and promote them to MRU; infrequent or new instruction translations land a few positions below MRU. Data translations land at LRU (highest eviction priority) unless hit, in which case they're promoted slightly. This trades data page walks for instruction page walks — exactly what we want, since instruction misses are more expensive.

But increasing data page walks increases cache misses on data PTEs, harming performance. xPTP is an L2 replacement policy that fixes this. On an L2 miss, xPTP finds two candidates: the standard LRU victim, and an alternative victim that's the oldest non-data-PTE block. If the LRU victim is a data PTE and the alternative is far enough down the stack (position ≥ LRU + K), xPTP evicts the alternative; otherwise it evicts the LRU victim. This preferentially keeps data PTEs in L2, speeding data page walks and compensating for iTP's trade-off. The combined scheme, iTP+xPTP, monitors STLB MPKI and enables xPTP only when STLB pressure is high.

The payoff: iTP alone improves single-core performance by 2.2% over LRU; iTP+xPTP improves it by 18.9%. Under two-thread SMT, the gains are 0.3% (iTP) and 11.4% (iTP+xPTP). Hardware overhead is 4 bits per STLB entry (768 B for 1536 entries) plus 1 bit per L2 block — under 1% of STLB area, nothing on the critical path.

🍯 Key takeaways

  • Server apps have huge instruction footprints (millions of instructions) that spill into the shared STLB, causing frequent instruction translation misses.
  • Instruction TLB misses stall the pipeline; data TLB miss latency can be partially hidden in OoO execution. LRU treats them equally — wrong.
  • iTP (STLB policy) keeps instruction translations at the top of the recency stack via smart insertion + a 3-bit frequency counter, evicting data translations first.
  • This increases data page walks. xPTP (L2 policy) compensates by favoring data PTEs in L2, speeding those walks.
  • iTP+xPTP monitors STLB MPKI and enables xPTP only when pressure is high (adaptive).
  • Gains: 18.9% single-core, 11.4% two-thread over LRU. Hardware: ~768 B STLB overhead + 1 bit/L2 block.
  • Outperforms state-of-the-art (CHiRP, PTP, TDRRIP) on big-code workloads.
Six everyday pictures

How to think about iTP+xPTP

Each picture stands alone. Every card ends with where the analogy breaks.

The core idea · Critical path asymmetry

Emergency room triage

Two patients arrive: one with a heart attack (instruction miss) that requires immediate treatment or they die, one with a broken arm (data miss) that can wait in the queue. Standard protocol treats them in arrival order — but smart triage sees the difference and rushes the heart attack to the front. iTP does triage: it keeps instruction translations (the heart attacks) in the STLB at the expense of evicting data translations (the broken arms), because instruction misses stall the pipeline immediately while data misses can often hide.

Where it breaks: in a real ER, both patients eventually get treated. In the STLB, data translations evicted by iTP trigger page walks every time they're needed — that's the cost xPTP mitigates.
iTP's insertion policy · Stack positioning

VIP vs regular boarding

An airline boards VIPs before regular passengers, but not all VIPs get the front row — frequent flyers sit in row 1, new VIPs in row 3. Regular passengers board last. iTP does this: instruction translations are VIPs (insert near MRU), but only frequently-hit instruction translations (Freq counter saturated) reach MRU; new or infrequent instructions land a few positions below (MRU−N). Data translations are regular passengers (insert at LRU). The Freq counter is the frequent-flyer status.

Where it breaks: VIP status in airlines is permanent; iTP's Freq counter resets when an entry reaches saturation and is promoted to MRU, allowing the policy to adapt.
xPTP's eviction policy · Favoring PTEs

Spare tire in the trunk

You keep a spare tire in the trunk because you might need it in an emergency, even though it takes up space. When packing for a trip, you remove luggage before you remove the spare. xPTP is the same: it finds two eviction candidates — the LRU victim (normal luggage) and an alternative victim that's the oldest non-data-PTE block (more luggage). If the LRU victim is a data PTE and the alternative is far enough down the stack (≥ LRU + K positions), xPTP evicts the alternative, keeping the data PTE (the spare tire) for the next data page walk.

Where it breaks: a spare tire is rarely used; data PTEs are frequently accessed during the page walks triggered by iTP evicting data translations from the STLB. The "spare" is actually critical.
Cooperative design · Amplification

Two-stage rocket

A rocket has two stages: the first burns to get you partway to orbit; the second ignites only after the first finishes, using the momentum to reach full velocity. Firing the second stage alone (without the first) gets you nowhere. iTP is the first stage — it reduces instruction TLB misses by evicting data translations, providing a modest (2.2%) gain. xPTP is the second stage — it only helps because iTP increased data page walks; it keeps data PTEs in L2 to speed those walks. Together, the gain is 18.9% (single-core). Neither works well alone; the synergy is the whole point.

Where it breaks: rocket stages are sequential (first finishes, then second starts). iTP and xPTP operate concurrently — xPTP continuously compensates for iTP's ongoing evictions.
Phase adaptivity · Conditional activation

Thermostat with setpoint

A thermostat monitors temperature and turns on the heater only when it drops below the setpoint. If the room is already warm, running the heater wastes energy. iTP+xPTP does this for STLB pressure: it monitors STLB MPKI (the "temperature") and enables xPTP only when MPKI exceeds threshold T1 (the "setpoint"). During low-pressure phases (small working set, few STLB misses), xPTP is disabled and plain LRU runs at L2 — no need to favor PTEs if page walks are rare. The system adapts to the workload's behavior.

Where it breaks: a thermostat has hysteresis (turn-on and turn-off thresholds differ to prevent oscillation). iTP+xPTP uses a single threshold and re-decides every 1000 instructions — simpler, but could oscillate on borderline workloads.
Hardware overhead · Minimalism

Post-it note on the fridge

You don't need a whiteboard to track your grocery list — a small sticky note (a few bytes) does the job. iTP adds 4 bits per STLB entry (1 Type bit, 3 Freq bits) — for a 1536-entry STLB that's 768 bytes, about 2.1% of the STLB area. xPTP adds 1 bit per L2 block (Type) — for a 512 KB L2 with 64 B blocks, that's 8192 bits = 1024 bytes, negligible. Total overhead is under 1% of cache/TLB area, nothing on the critical path. Minimal hardware, large performance gain.

Where it breaks: a post-it is disposable; iTP/xPTP's metadata is persistent and must be kept in sync with the TLB/cache state (requires careful implementation).
The full explanation

How it actually works

Section by section, with the paper's key mechanisms.

1 · The problem quantified

The paper measures address translation overhead across SPEC and Qualcomm Server workloads with varying ITLB sizes. For a realistic 64-entry ITLB, SPEC workloads spend on average 0.03% of cycles on instruction address translation (their code fits in the ITLB), while Qualcomm Server workloads spend 12.52% (their code doesn't fit). With a 1536-entry STLB, Qualcomm Server workloads have an average instruction STLB MPKI of 0.4 (up to 0.9 for some workloads), while SPEC workloads have near-zero instruction STLB MPKI.

The baseline STLB uses LRU, which treats instruction and data translations identically. The paper tests a modified LRU that probabilistically favors keeping instruction translations (parameter P = probability of evicting a data entry). For P=0.8, Qualcomm Server workloads improve by up to 5% over standard LRU; for lower P values (favoring data), performance degrades. SPEC workloads show no change (their instruction translations fit in ITLB). This motivates iTP.

🍯
Finding 1: Applications with large code footprints spend significant cycles on instruction address translation. Finding 2: Prioritizing instructions over data in the STLB can provide large gains for these workloads.

2 · iTP: the STLB replacement policy

iTP augments each STLB entry with two fields: a 1-bit Type (0 = instruction, 1 = data) and a 3-bit Freq counter. It uses smart insertion and promotion policies:

# Insertion (end of page walk, new entry inserted)
if Type == 0 (instruction):
    insert at MRUpos − N      # N=4 (paper default)
    Freq = 0                        # new entry, zero frequency
else (data):
    insert at LRUpos             # highest eviction priority

# Promotion (upon STLB hit)
if Type == 0 (instruction):
    if Freq < MAX:
        promote to MRUpos − N
        Freq += 1                   # increment frequency
    else:
        promote to MRUpos         # saturated = frequently used
        Freq = 0                    # reset
else (data):
    promote to LRUpos + M        # M=8 (paper default)

Eviction uses standard LRU: evict the entry at LRUpos. The key insight: instruction entries cluster near MRU (protected), data entries cluster near LRU (vulnerable). Frequently-hit instruction entries reach MRUpos; useless instruction entries eventually drift to LRU and get evicted. This trades data page walks for instruction page walks.

💡
Why N positions below MRU? To reserve MRUpos for proven frequently-used instruction translations. New or infrequent ones land slightly lower and must "earn" MRU by hitting repeatedly (Freq counter).

3 · xPTP: the L2 cache replacement policy

xPTP augments each L2 block with a 1-bit Type field (1 = data PTE, 0 = other). Insertion and promotion follow LRU. The novel part is the eviction policy:

# On L2 miss, find victim
LRU_victim = block at LRUpos
ALT_victim = oldest non-data-PTE block  # (Type=0)

if LRU_victim is data PTE and ALT_victim.position ≥ LRUpos + K:  # K=8
    evict ALT_victim                      # keep the data PTE
else:
    evict LRU_victim                      # evict normal LRU

If the LRU victim is a data PTE and an alternative victim (oldest non-PTE) is far enough down the stack (≥ LRU + K positions), xPTP evicts the alternative, preserving the data PTE. This preferentially keeps data PTEs in L2, accelerating the data page walks that iTP's evictions trigger. If the LRU victim is not a data PTE or no suitable alternative exists, xPTP falls back to standard LRU.

💡
Why only data PTEs? Because iTP prioritizes instruction translations in the STLB, it increases data page walks. Those walks fetch data PTEs into L2. xPTP keeps those data PTEs to speed the walks. Instruction PTEs aren't prioritized — instruction page walks are already reduced by iTP.

4 · iTP+xPTP: the adaptive combination

xPTP helps when STLB pressure is high (many page walks), but it can hurt during low-pressure phases (favoring PTEs wastes L2 capacity). So the combined scheme, iTP+xPTP, monitors STLB MPKI and enables xPTP only when MPKI exceeds threshold T1. Otherwise, plain LRU runs at the L2.

The implementation uses two counters (STLB miss count, instruction count) and a 1-bit status register. Every 1000 instructions, compute MPKI = misses ÷ instructions. If MPKI > T1, set status = xPTP; else status = LRU. Reset counters and repeat.

🍯
Hardware overhead: iTP adds 4 bits/STLB entry (768 B for 1536 entries). xPTP adds 1 bit/L2 block (~1024 B for 512 KB L2). Total < 1% of TLB/cache area, nothing on critical path. Phase detection adds two small counters + 1-bit register.
Follow the numbers, step by step

Worked examples

Three traces using the paper's real data.

① Reading the instruction translation overhead

A Qualcomm Server workload spends X% of its cycles on instruction address translation with a 64-entry ITLB and a 1536-entry STLB. What's happening?

  1. The paper (Figure 1) measures translation overhead. For a 64-entry ITLB, Qualcomm Server workloads spend on average 12.52% of cycles on instruction translation, vs 0.03% for SPEC. This is because server apps have huge code footprints (millions of instructions) that don't fit in the ITLB.
  2. Instruction translations spill into the STLB. Figure 2 shows instruction STLB MPKI is up to 0.9 for Qualcomm workloads (average 0.4), vs near-zero for SPEC. Each STLB miss triggers a multi-level page walk (potentially 4-5 memory accesses per miss), costing hundreds of cycles.
  3. Instruction misses stall the pipeline. Unlike data misses (which can hide in OoO), instruction misses block frontend fetch, idling the entire pipeline. This is why the overhead is so large despite "only" ~0.5 MPKI.
  4. The opportunity. If we can keep instruction translations in the STLB longer (reducing instruction MPKI), we save those critical-path stalls — even if it increases data MPKI, because data misses are less critical.
🍯
Insight: The paper shows that big-code workloads face a severe instruction translation bottleneck — and that trading data translations for instruction translations is the right move.

② Tracing iTP's insertion and promotion

A new instruction translation arrives (page walk completes). Walk through iTP's insertion and later promotion when it's hit.

  1. Insertion (Type=0, instruction). iTP inserts the new entry at position MRUpos − N (N=4). Its Freq counter is set to 0. This places it near the top of the stack, but not at MRU (reserved for proven frequently-used entries).
  2. First hit (Freq=0, max is 7). On the first hit, Freq is not saturated, so iTP promotes the entry back to MRUpos − N and increments Freq to 1. It stays at the same position but gains frequency credit.
  3. Subsequent hits (Freq increments). Each hit increments Freq (up to 7) and keeps the entry at MRUpos − N. Once Freq saturates (reaches 7), the entry is frequently used.
  4. Promotion to MRU (Freq saturated). On the next hit after saturation, iTP promotes the entry to MRUpos (the very top) and resets Freq to 0. This entry is now maximally protected — it's unlikely to be evicted unless it stops being hit.
  5. If never hit again. A useless instruction entry inserted at MRUpos − N will eventually drift down the stack as other entries get promoted. Once it reaches LRUpos, it gets evicted. The Freq counter filters out useless entries.
💡
Why the Freq counter? Without it, every instruction entry would reach MRU on first hit, wasting space on one-time accesses. The counter ensures only frequently-used instruction translations get the top spot.

③ xPTP's victim selection

An L2 miss occurs. The LRU victim is a data PTE (Type=1). Walk through xPTP's decision.

  1. Identify LRU victim. Standard LRU would evict the entry at LRUpos. In this case, that entry is a data PTE (Type=1) — a block containing a page table entry for a data translation.
  2. Find alternative victim. xPTP scans the recency stack for the oldest non-data-PTE block (Type=0). Suppose it finds one at position 6 (in an 8-way L2, LRUpos=7).
  3. Evaluate inequality (K=8). xPTP checks: is ALT_victim.position ≥ LRUpos + K? Here, position 6 is not ≥ 7 + 8 = 15 (impossible in an 8-way cache). So the condition is: is the alternative at least at position 7 (LRU)? No (it's at 6). The inequality fails.
  4. Evict LRU victim. Since the alternative is not far enough down the stack, xPTP falls back to standard LRU and evicts the data PTE at LRUpos.
  5. Contrast: suppose ALT_victim is at LRUpos (position 7). Now the alternative is the LRU non-PTE. If K=8 means "alternative must be LRU or beyond", then position 7 ≥ 7, so xPTP evicts the alternative (the non-PTE at LRU), keeping the data PTE at position 6. The exact semantics depend on the paper's definition; the key idea is xPTP prefers keeping data PTEs when a suitable non-PTE victim exists.
🍯
Key insight: xPTP doesn't blindly keep all PTEs — it only keeps them when there's a non-PTE victim available that's close enough to LRU. This balances PTE retention with cache efficiency.
Test yourself

Practice problems

Try each before opening the answer. Every one ends with the misconception it checks for.

Problem 1 · easy
A workload has tiny instruction footprint (code fits in ITLB) but huge data footprint. Will iTP+xPTP help? Why or why not?
Show answer
No. If instruction translations fit in the ITLB, the STLB holds mostly data translations. iTP's logic (prioritizing instruction translations) has little impact — there are few instruction entries to protect. The paper shows this: SPEC workloads (small code footprints) see no benefit from iTP+xPTP because their instruction STLB MPKI is near-zero. iTP+xPTP only helps workloads with large code footprints that spill into the STLB.Misconception: that iTP+xPTP helps all workloads. It specifically targets big-code apps; for small-code apps it degenerates to near-LRU behavior.
Problem 2 · medium
iTP inserts instruction translations at MRUpos − N (N=4) instead of MRUpos. What happens if you set N=0 (insert at MRU directly)? Why is this suboptimal?
Show answer
If N=0, every new instruction translation lands at MRU immediately, regardless of whether it will be reused. One-time instruction accesses (e.g., initialization code) would occupy the top of the stack, crowding out frequently-used instruction translations that should be there. The Freq counter can't help because the entry is already at MRU on first insertion. Setting N=4 reserves MRU for proven frequently-used entries (those whose Freq counter saturates), filtering out useless entries. The paper's sensitivity analysis (not shown in detail) indicates N=4 performs best.Misconception: that inserting at MRU is always optimal. For entries with unknown reuse (new translations), inserting slightly below MRU + using promotion to reach MRU is more robust.
Problem 3 · medium
xPTP favors keeping data PTEs in L2. Why not also keep instruction PTEs — wouldn't that speed instruction page walks too?
Show answer
Because iTP already reduces instruction page walks by keeping instruction translations in the STLB. The bottleneck xPTP addresses is the increase in data page walks caused by iTP evicting data translations from the STLB. Instruction page walks are less frequent (iTP handles that), so instruction PTEs don't need special protection in L2. Favoring both instruction and data PTEs would dilute L2 capacity for regular data/code blocks without addressing the actual problem (data page walk overhead).Misconception: that all PTEs are equally important. xPTP is specifically designed to compensate for iTP's trade-off (more data page walks), not to generically favor all PTEs.
Problem 4 · hard
The paper's phase-adaptive mechanism enables xPTP when STLB MPKI > T1. Construct a scenario where enabling xPTP during low MPKI would hurt performance.
Show answer
Suppose a phase has very small working set (both code and data fit in STLB) → STLB MPKI ≈ 0, very few page walks. If xPTP is enabled, it prioritizes data PTEs in L2 — but there are almost no PTEs to cache (page walks are rare), so L2 is now wasting capacity on keeping a few PTEs at the expense of evicting useful data/instruction payload blocks. Performance degrades because L2 misses on regular blocks increase, while the "benefit" (faster page walks) is negligible since page walks are rare. The paper's adaptive mechanism disables xPTP in this case (MPKI < T1), letting LRU run and maximizing L2 hit rate for the working set.Misconception: that favoring PTEs is always good. It's only beneficial when page walks are frequent enough that the PTE hit rate improvement outweighs the payload eviction cost.
Problem 5 · medium
Why does iTP use a 3-bit Freq counter (max 7) instead of a 1-bit "has been hit" flag? What does the multi-bit counter buy you?
Show answer
A 1-bit flag can only distinguish "never hit" vs "hit at least once". A 3-bit counter (0-7) can distinguish frequency of hits: an entry hit once vs an entry hit seven times. This filters out low-frequency instruction accesses (e.g., initialization code hit once and never again) from high-frequency accesses (hot loop instructions hit repeatedly). Only entries that hit frequently enough to saturate the counter (Freq=7) get promoted to MRUpos. A 1-bit flag would promote all hit entries to MRU immediately, polluting the top of the stack with infrequent accesses. The 3-bit counter provides a middle ground: entries that are hit but not frequently stay at MRUpos − N, eventually drifting to LRU if they stop being used.Misconception: that a single hit proves an entry is useful. Many instruction accesses are one-time (cold misses). The Freq counter's job is to identify sustained reuse.
Problem 6 · hard
The paper shows iTP alone improves performance by 2.2%, but iTP+xPTP improves it by 18.9% (single-core). Where does the huge gap come from? Why isn't iTP's gain closer to the combined gain?
Show answer
iTP reduces instruction STLB misses (good) but increases data STLB misses (bad). Each additional data STLB miss triggers a page walk that fetches data PTEs into L2. If those data PTEs are quickly evicted from L2 (because the L2 replacement policy doesn't know they matter), the page walk must fetch them from LLC or DRAM — taking hundreds of cycles. The net effect: iTP's instruction-miss reduction is partially undone by the slowdown of data page walks. xPTP fixes this by keeping data PTEs in L2, so data page walks hit in L2 instead of going to LLC/DRAM. The paper's Figure 9 shows that iTP+xPTP reduces average L2 miss latency by 47.5% over LRU (single-core) because LLC MPKI drops significantly. This latency reduction is what unlocks the 18.9% gain — iTP+xPTP doesn't just reduce misses, it dramatically speeds the remaining misses.Misconception: that iTP's gains are additive with xPTP's. They're synergistic — xPTP amplifies iTP by compensating for its downside (data page walk slowdown).
Problem 7 · easy
What is the hardware overhead of iTP+xPTP for a system with a 1536-entry STLB and a 512 KB L2? Express it as a fraction of STLB/L2 area.
Show answer
iTP: 4 bits/STLB entry (1 Type + 3 Freq). 1536 entries × 4 bits = 6144 bits = 768 bytes. Assuming STLB is ~24 bytes/entry (tag + data + control) → 1536 × 24 = 36864 B. Overhead = 768 / 36864 ≈ 2.1% of STLB area. xPTP: 1 bit/L2 block. 512 KB ÷ 64 B/block = 8192 blocks × 1 bit = 8192 bits = 1024 bytes. L2 tag array is ~(24 bits tag + ~6 bits state) × 8192 = ~30 KB. Overhead = 1024 B / 30 KB ≈ 3.3% of L2 tag area (negligible relative to 512 KB L2 data array). Total: under 1% of combined TLB/cache area. (Paper §4.1.3, §4.2)Misconception: that the overhead is large. It's tiny — a few hundred bytes per structure, nothing on the critical path.

Frequently asked questions

Ten questions newcomers ask most. Tap any to expand.

Q1Why prioritize instructions over data in the STLB if data accesses are more frequent?▾
Because instruction TLB misses stall the pipeline (frontend can't fetch the next instruction), while data TLB miss latency can be partially hidden in out-of-order execution. Even if data accesses are more frequent, instruction misses are more harmful to performance. The paper shows that trading data page walks for instruction page walks — even if it increases total page walks — improves performance because you're saving critical-path stalls. Frequency isn't the only metric; criticality matters too.
Q2Does iTP+xPTP help desktop/HPC workloads, or only server workloads?▾
The paper evaluates primarily Qualcomm Server workloads where instruction footprints are large. SPEC CPU 2006/2017 workloads (desktop/HPC) spend near-zero cycles on instruction address translation because their code fits in the ITLB. The paper's motivation focuses on server workloads with million-instruction footprints. Results for SPEC are not extensively reported, so claims about performance on those workloads should be limited to what the paper demonstrates: the technique targets big-code server applications.
Q3What happens to iTP's Freq counter during phase changes?▾
The paper's description shows that the Freq counter is incremented on hits and reset on saturation (when an entry reaches MRU). The paper does not describe a periodic decay or halving mechanism for the Freq counter. The counter adapts through natural replacement: as old entries drift down the recency stack and are evicted, new entries with fresh Freq values are inserted. Phase adaptation for xPTP is handled separately through MPKI monitoring.
Q4Why does xPTP only favor data PTEs, not all PTEs?▾
Because iTP's effect is asymmetric: it increases data page walks (by evicting data translations from STLB to protect instruction translations) but decreases instruction page walks (by keeping instruction translations in STLB). So the bottleneck xPTP needs to address is specifically the slowdown of data page walks. Instruction page walks are already reduced by iTP; instruction PTEs don't need special protection in L2. Favoring all PTEs would waste L2 capacity without targeting the actual problem.
Q5What is the threshold T1 for enabling xPTP, and how was it chosen?▾
The paper describes a threshold T1 based on STLB MPKI and uses a 1000-instruction counter window for the decision. The paper does not specify the exact T1 value used in the evaluation or describe a sweep of T1 values. The adaptive mechanism monitors STLB MPKI every 1000 instructions and enables xPTP when MPKI exceeds the threshold, disabling it otherwise. The exact threshold and its selection process are implementation details not fully detailed in the paper.
Q6Does iTP+xPTP work with other LLC replacement policies (e.g., SHiP, Mockingjay)?▾
Yes. The paper (Figure 11) evaluates iTP+xPTP with LRU, SHiP, and Mockingjay at LLC. iTP+xPTP provides consistent gains with LRU and SHiP (18.9% and 15.8% single-core, respectively) but smaller gains with Mockingjay (1.6%). The Mockingjay result comes from Mockingjay itself being less effective for server workloads (it was designed for data-intensive workloads with different access patterns). iTP (STLB) and xPTP (L2) are orthogonal to LLC replacement — the gains are independent of LLC policy, though the magnitude depends on how well the LLC policy handles the workload.
Q7How does iTP+xPTP compare to state-of-the-art (CHiRP, PTP, TDRRIP)?▾
iTP+xPTP significantly outperforms all three. Single-core results (Figure 8a): iTP+xPTP 18.9%, TDRRIP 9.3%, PTP 7.1%, CHiRP ~0% (similar to LRU). Two-thread results (Figure 8b): iTP+xPTP 11.4%, TDRRIP 8.5%, PTP ~0%, CHiRP ~0%. CHiRP fails because it doesn't differentiate instruction vs data translations (its prediction mechanism treats them equally). PTP and TDRRIP favor all PTEs equally (both instruction and data), missing the asymmetry iTP+xPTP exploits. iTP+xPTP is the first to specifically prioritize instruction translations in STLB and compensate with data-PTE prioritization in L2 — that synergy is what delivers the large gains.
Q8What happens in a split STLB design (separate instruction and data STLBs)?▾
The paper (Figure 14) compares unified STLB (1536 entries shared) vs split STLB (768 instruction + 768 data). For equal total entries (1536), the split STLB performs slightly worse than unified STLB + iTP+xPTP, because the split design wastes capacity when one type (e.g., data) needs more than its fixed share. Doubling the split STLB (1536 inst + 1536 data = 3072 total) matches the performance of 1536 unified + iTP+xPTP — but at 2× the storage cost. The unified design with iTP+xPTP is more area-efficient. Vendors typically prefer unified STLBs for this reason; iTP+xPTP makes them even more effective.
Q9Does iTP+xPTP help under SMT (two hardware threads)?▾
Yes, but the gains are smaller than single-core (11.4% vs 18.9%). Under SMT, the STLB and L2 are shared between two threads. iTP still prioritizes instruction translations, but now it must balance both threads' instruction vs data needs. The per-thread benefit is diluted, and contention increases L2 miss rates. xPTP still helps (it keeps data PTEs for both threads), but the net gain is lower. The paper evaluates 75 two-thread workload pairs (Intense/Medium/Relaxed load) and shows consistent but reduced gains over single-core. iTP+xPTP is still beneficial under thread co-location, just not as dramatic.
Q10What about huge pages (2MB pages)? Does iTP+xPTP still help?▾
Yes, but the gains shrink as more of the footprint is mapped to huge pages. The paper (Figure 13) evaluates multi-size page scenarios (0%, 10%, 50%, 100% of code+data on 2MB pages). When 0% uses huge pages (all 4KB), iTP+xPTP delivers 18.9% (single-core). At 10% huge pages, it's 10.1%. At 50%, it's ~5%. At 100%, it's ~1% (two-thread; single-core is near-zero). This makes sense: huge pages drastically reduce TLB pressure (one 2MB page covers 512 4KB pages), so the STLB bottleneck diminishes. Even at 100% huge pages, iTP+xPTP provides small gains because not all accesses hit (some still miss and trigger page walks), but the effect is minor. The technique is most valuable for systems using predominantly 4KB pages, which is common in practice (huge pages require contiguity and incur fragmentation/defragmentation overhead).

Glossary & further reading

Every term in one place, then a path to go deeper.

Core terms
iTP (Instruction Translation Prioritization)STLB replacement policy that keeps instruction translations near MRU (protected) and data translations near LRU (vulnerable), using a 3-bit Freq counter to filter infrequent accesses.
xPTP (extended Page Table Prioritization)L2 cache replacement policy that favors keeping data PTEs to speed data page walks (triggered by iTP's evictions). Finds oldest non-PTE victim when possible.
iTP+xPTPThe combined adaptive scheme: iTP at STLB, xPTP at L2 (enabled when STLB MPKI > T1). Delivers 18.9% single-core, 11.4% two-thread gains over LRU.
STLB (Shared Last-Level TLB)Large TLB (e.g., 1536 entries, 12-way) shared between instruction and data translations. Accessed after ITLB/DTLB miss.
Type field1-bit metadata per TLB/cache entry: 0 = instruction (or non-PTE), 1 = data (or data PTE). Used by iTP and xPTP to drive replacement.
Freq counter3-bit frequency counter per STLB entry (0-7). Tracks how many times an instruction translation has been hit. Saturated entries (7) get promoted to MRU.
Phase adaptivityMechanism that monitors STLB MPKI (every 1000 instructions) and enables xPTP only when MPKI > T1, disabling it during low-pressure phases.
Supporting terms
ITLB / DTLBFirst-level instruction/data TLBs (e.g., 64 entries, 4-way). Small, fast, accessed first on every memory reference.
Page walkHardware traversal of the multi-level page table (4-5 levels on x86) to find a translation after TLB miss. Each level may require a memory/cache access.
PTE (Page Table Entry)A memory block containing translation information (virtual-to-physical mapping). Can be intermediate (PML4/PDP/PD) or leaf (PT). Stored in L2/LLC/DRAM.
Instruction PTEPTE fetched by a page walk serving an instruction translation request (ITLB → STLB miss). Stored in L2/LLC.
Data PTEPTE fetched by a page walk serving a data translation request (DTLB → STLB miss). xPTP prioritizes these in L2.
MPKI (Misses Per Kilo-Instructions)Metric for TLB/cache miss rate. STLB MPKI is the number of STLB misses per 1000 instructions. Used to trigger xPTP.
LRU (Least Recently Used)Standard replacement policy: evict the entry that was used longest ago. Treats all entries equally — the baseline iTP+xPTP improves on.
MRUpos / LRUposTop/bottom of the LRU recency stack. MRU = most recently used (protected); LRU = least recently used (evicted next).
Insertion / Promotion / EvictionThree knobs of a replacement policy. Insertion = where a new entry lands. Promotion = where a hit entry moves. Eviction = which entry to throw out.

Related work

CHiRP: Control-Flow History Reuse Prediction hard

Mirbagher-Ajorpaz+ MICRO'20

State-of-the-art STLB replacement policy. Uses control-flow signatures to predict reuse. Does not differentiate instruction vs data — paper shows it performs similar to LRU for big-code workloads.

Page Table Prioritization (PTP) medium

Park+ ASPLOS'22

Translation-aware L2/LLC replacement policy. Prioritizes all PTEs (instruction + data) equally. iTP+xPTP improves on this by (i) prioritizing instruction translations in STLB and (ii) only prioritizing data PTEs in L2.

TDRRIP and T-SHiP medium

Vasudha & Panda ISPASS'22

Translation-aware variants of DRRIP/SHiP. Prioritize PTEs and deprioritize demand blocks that missed in STLB. Paper shows iTP+xPTP outperforms TDRRIP (9.3% vs 18.9% single-core).

Mockingjay: Effective Mimicry of Belady's MIN hard

Shah+ HPCA'22

State-of-the-art translation-oblivious LLC replacement. Uses long reuse-distance histories. iTP+xPTP delivers smaller gains with Mockingjay LLC (1.6% vs 18.9% with LRU LLC) because Mockingjay itself is less effective for server workloads.

Morrigan: Composite Instruction TLB Prefetcher hard

Vavouliotis+ MICRO'21

STLB prefetcher targeting instruction translations. Orthogonal to iTP+xPTP — prefetcher reduces misses, iTP+xPTP optimizes replacement given the miss stream. Combining them is future work.
