Training only the right prefetcher, not every prefetcher

Three prefetchers share one demand stream. The standard approach trains all three on every request — polluting their tables. This is the plain-English story of Alecto, a tiny circuit that routes each request to the prefetcher that can actually use it.

📄 "Integrating Prefetcher Selection with Dynamic Request Allocation" · Li, Zhang, Ren, Xie 🎓 Prefetching, cache architecture, memory hierarchy ⏱ ~25 min read

Before you begin

Eight ideas the paper assumes you know

Each card clears up one concept Alecto builds on. Read in order the first time or jump to what's unfamiliar.

1What cache prefetchers do and why there are many▾

A prefetcher predicts what data you'll need soon and fetches it into cache before you ask. The prediction depends on recognizing a pattern in the stream of memory addresses you've accessed. Different programs exhibit different patterns (streaming, fixed-stride, irregular-spatial, temporal), so commercial processors hybrid several specialized prefetchers.

The paper uses three: a stream prefetcher (detects sequential access), a stride prefetcher (fixed delta), and a spatial prefetcher (region-based pattern).

Now you're ready: Alecto is about scheduling these three so they don't step on each other's tables or generate redundant prefetch requests.
2Demand requests, prefetch requests, and training▾

A demand request is a real access the CPU core needs right now. Prefetchers watch these demand requests to learn patterns. That's called training. After training, the prefetcher issues prefetch requests — speculative fetches of data it predicts will be needed soon.

Training is not free: every demand request updates tables inside the prefetcher, consuming storage and evicting older entries. If a prefetcher trains on requests it can't predict, it wastes table space and pollutes its metadata.

Now you're ready: Alecto's core contribution is selecting which demand requests each prefetcher sees, preventing pollution.
3Prefetcher tables — IP table, Pattern table, metadata▾

Each prefetcher maintains small on-chip tables — an IP table keyed by the instruction address (PC), a pattern table or history, and bookkeeping metadata (LRU state, confidence counters). When a demand request arrives, the prefetcher looks up the PC, updates the pattern, and possibly issues prefetch requests.

A table miss happens when the PC or pattern isn't found, forcing the prefetcher to evict an old entry or ignore the request. The paper's Figure 1 shows: prior approaches see ~300k prefetcher table misses; Alecto sees ~100k because it only sends requests the prefetcher can handle.

Now you're ready: Alecto's dynamic demand request allocation dramatically reduces table misses by ensuring each prefetcher trains only on its designated patterns.
4Accuracy, coverage, timeliness — the three metrics▾

Accuracy = fraction of issued prefetches that hit a future demand. Coverage = fraction of demand misses eliminated by prefetching. Timeliness = did the prefetch arrive in time, or was the demand stalled anyway?

A good prefetcher needs all three. High accuracy but low coverage helps little; high coverage but low accuracy pollutes the cache; good accuracy and coverage but late prefetches still leave the CPU stalled.

Now you're ready: Alecto balances all three by fine-tuning which prefetcher handles each request and adjusting prefetching aggressiveness dynamically.
5Static vs dynamic selection — what prior work tried▾

Static selection (DOL, IPCP) uses fixed priority rules at design time: always try the stream prefetcher first, then stride, then spatial. Dynamic selection (Bandit, RL-based) adjusts at runtime using reinforcement learning to turn prefetchers on/off or change their degree based on recent performance.

Neither solves the core problem: IPCP/RL train all prefetchers on all requests (polluting tables), while DOL sequentially passes requests and stops when one handles the data (inflexible static priority). Both suffer from coarse-grained rules that can't distinguish one instruction's pattern from another's.

Now you're ready: Alecto's innovation is fine-grained dynamic allocation — routing each PC's demand requests only to its suitable prefetchers.
6Per-PC vs global — why one size doesn't fit all▾

Different memory access instructions (identified by their PC) exhibit different patterns. One PC might stream (+1, +1, +1...); another might stride by 4 (+4, +4, +4...); another might hop irregularly. A global selection rule applies the same prefetcher choice to all PCs. A per-PC rule tailors the choice to each instruction individually.

The paper's Figure 2 (459.GemsFDTD) shows two interleaved patterns: PC=0x30b00 is spatial, PC=0x30aca is stride/stream. Global rules conflate them; per-PC rules separate them.

Now you're ready: Alecto maintains a per-PC state for each prefetcher in its Allocation Table, enabling fine-grained identification and allocation.
7Prefetch degree and aggressiveness▾

The prefetch degree is how many lines the prefetcher fetches on each trigger. A conservative degree (e.g., 2) reduces pollution but may miss coverage; an aggressive degree (e.g., 6) boosts coverage but risks accuracy. Aggressiveness is a related concept: how eagerly the prefetcher acts.

Alecto starts conservative (degree 3) and incrementally raises the degree (up to 3+M+1) for prefetchers in the IA_m state, if accuracy stays above the Proficiency Boundary (75%).

Now you're ready: Alecto's state machine (UI, IA_m, IB_n) adjusts degree dynamically based on runtime accuracy, balancing coverage and pollution.
8Storage overhead and scalability▾

Adding hardware costs die area and power. The storage overhead for a prefetcher selection algorithm is the extra tables it needs (e.g., counters, state, RL arms). RL-based schemes like Bandit scale exponentially: storage = 8 × (#actions)^(#prefetchers) bytes. With 3 prefetchers and 8 actions each, that's 4 kB.

Alecto's overhead is linear: 5312 + 1792×P bits (≈1.3 kB for P=3), and excluding the Sandbox Table (which doubles as a prefetch filter anyway), only 760 bytes. That's 5.4× smaller than Bandit for the same configuration.

Now you're ready: Alecto's lightweight design makes it practical even for many prefetchers, while prior RL schemes explode in storage.

The whole story in one place

Alecto in a nutshell

The problem, the routing fix, and how it all connects.

When three prefetchers (stream, stride, spatial) share a cache, the standard approach trains all three on every demand request. But each prefetcher only benefits from requests matching its pattern. Sending every request to every prefetcher pollutes their tables: irrelevant PCs crowd out useful ones, table misses spike, and prefetching accuracy degrades. Prior selection schemes (IPCP, DOL, Bandit) either apply static priorities or control outputs, not inputs — so the pollution remains.

The paper's fix, Alecto, integrates prefetcher selection with Dynamic Demand Request Allocation (DDRA). Each demand request enters an Allocation Table keyed by its PC. The table tracks per-PC, per-prefetcher states (UI = unknown, IA_m = accurate, IB_n = blocked) based on historical accuracy. Alecto only routes the request to prefetchers in UI or IA_m states, preventing blocked prefetchers from training on patterns they can't handle. A Sample Table and Sandbox Table collect accuracy feedback (issued vs confirmed prefetches) every 100 demand accesses, triggering state transitions that promote high-accuracy prefetchers, block low-accuracy ones, and incrementally adjust prefetching degree for IA prefetchers up to 3+M+1 lines.

To shrink overhead, the Sandbox Table doubles as a prefetch filter, eliminating redundant requests from multiple prefetchers. Total storage: ~1.3 kB (or 760 bytes excluding the filter), about 0.2% of a 1 MB cache and 5.4× smaller than Bandit for the same setup.

The payoff: On SPEC CPU2006/2017, Alecto improves IPC by 2.76% on average over Bandit in single-core, 7.56% in eight-core, and 5.25% on memory-intensive workloads. It cuts prefetcher table training events by 48% (reducing prefetcher-level energy by 48%, system-level memory hierarchy energy by 7%). Prefetcher table misses drop from ~300k to ~100k. All for ~1.3 kB of hardware.

🍯 Key takeaways

  • IPCP/RL train all prefetchers on all demand requests; DOL passes requests sequentially with static priority. Alecto routes each request only to suitable prefetchers based on per-PC runtime accuracy.
  • Fine-grained per-PC tracking (not global rules) pinpoints which prefetcher fits which instruction's pattern.
  • State machine (UI/IA_m/IB_n) uses runtime accuracy feedback to promote, block, or adjust prefetchers every 100 demands.
  • Allocation Table (per-PC states) + Sample Table (accuracy) + Sandbox Table (filter + feedback) = ~1.3 kB.
  • Sandbox Table doubles as prefetch filter, eliminating duplicate requests from multiple prefetchers at no extra cost.
  • Result: +2.76% single-core, +7.56% eight-core, +5.25% memory-intensive; 48% fewer table updates; 7% memory energy reduction.
Six everyday pictures

How to think about Alecto

Each picture stands alone. Every card ends with where the analogy breaks.

The core idea · Selective routing

Three specialist chefs, one prep line

A sushi chef, a taco chef, and a pasta chef share one conveyor of ingredients. The old system sends every ingredient to all three — the sushi chef gets cornmeal (useless), the taco chef gets seaweed (irrelevant), each station clutters with junk. Alecto is a dispatcher at the front: recognizing that each order (PC) has a pattern, it routes sushi orders only to the sushi chef, taco orders only to the taco chef. Each station stays focused, tables stay clean.

Where it misleads: a real kitchen dispatcher knows orders upfront; Alecto learns patterns at runtime via accuracy feedback, and patterns can shift (phase changes).
State machine · UI / IA_m / IB_n

Trial period, promotion, and timeout

A new employee (prefetcher for a PC) starts on trial (UI): given a chance with conservative resources (degree 3) while the company gathers data. If they perform well (accuracy ≥75%), they're promoted (IA_m) and given more resources (degree 3+m+1). If they underperform (accuracy <5%), they're put on leave (IB_n) for N periods to cool down. After the timeout, they may be re-evaluated. The company tracks performance every 100 tasks (demand accesses) and adjusts accordingly.

Where it breaks: real promotions are subjective; Alecto's transitions are mechanical (accuracy thresholds), so a prefetcher that almost works can linger in UI or get blocked unfairly if variance is high.
Sample Table + Sandbox Table · Accuracy feedback

Tasting the soup plus a guest book

The Sample Table is the chef tasting the soup periodically: it counts how many prefetches each prefetcher issued (IssuedByP1) and how many were confirmed by later demand hits (ConfirmedP1). Accuracy = Confirmed ÷ Issued. The Sandbox Table is a guest book logging recent visitors (issued prefetch addresses): when a demand request matches an entry, Alecto knows that prefetch was useful and increments the confirmed counter. Together they provide the accuracy signal that drives state transitions.

Where it breaks: a guest book only records who came, not why they stayed; the Sandbox Table doesn't distinguish how quickly the prefetch was used (timeliness), just that it was used.
Sandbox Table as prefetch filter · Dual role

A bouncer who also takes notes

The Sandbox Table serves two jobs at once. As a bouncer, it checks every arriving prefetch request against its log of recently issued addresses: if it's a duplicate (tag hit), the request is dropped. As a note-taker, it records which prefetcher issued which address, so when a demand request hits that address later, Alecto knows to credit the correct prefetcher. This dual role eliminates the need for a separate prefetch filter, saving storage.

Where it breaks: a real bouncer has memory of faces; the Sandbox Table has limited entries (512 in the paper's config) and can evict recent prefetches, letting a duplicate slip through if it arrives after eviction.
Per-PC tracking · Fine-grained identification

A tutor for each student, not one lecture for all

A classroom has 50 students: some excel at math, some at reading, some at art. A global rule would assign one tutor to everyone regardless of subject. A per-student rule pairs each student with the tutors that fit their strengths. Alecto does the latter: each PC (student) has its own row in the Allocation Table, recording which prefetchers (tutors) work for it. When PC1's demand request arrives, Alecto routes it only to prefetchers suitable for PC1's pattern, not to prefetchers that happen to fit PC2 or PC3.

Where it breaks: a real tutor relationship evolves slowly; Alecto's per-PC state can change every 100 demands (a few microseconds), so it adapts faster but can also thrash if the pattern shifts rapidly.
Storage overhead · Linear vs exponential

A filing cabinet vs a warehouse

RL-based schemes (Bandit) store one "arm" for every combination of actions. With 3 prefetchers and 8 possible degrees each, that's 8³ = 512 arms × 8 bytes = 4 kB — a warehouse. Alecto stores one state entry per PC per prefetcher (plus counters), scaling linearly with the number of prefetchers: 5312 + 1792×P bits ≈ 1.3 kB for P=3 (or 760 bytes excluding the filter). That's a small filing cabinet. If you add a fourth prefetcher, Bandit's warehouse grows to 4096 arms; Alecto's cabinet just adds one more drawer.

Where it breaks: a real filing cabinet has physical drawers; Alecto's "linear" overhead still depends on the number of tracked PCs (64 entries in the Allocation Table), so it's not truly O(1) — but it's vastly better than exponential.
The full explanation

How it actually works

Section by section, with the paper's key mechanisms rebuilt as visuals.

1 · The problem — table pollution and conflicts

Commercial processors hybrid multiple prefetchers (stream, stride, spatial) at the same cache level. Each prefetcher maintains on-chip tables indexed by PC: an IP table, pattern history, confidence counters. The standard approach trains all prefetchers on every demand request. For a request with PC=X that exhibits a stride pattern, the stride prefetcher updates its tables correctly — but the stream and spatial prefetchers also update their tables, storing metadata for a pattern they can't predict. This pollutes their tables: useful entries evict, table miss rates spike (Figure 1 of the paper: ~300k misses vs Alecto's ~100k), and overall prefetching accuracy drops.

🌱
Concretely: Figure 2 shows 459.GemsFDTD with two interleaved PCs. PC=0x30b00 is spatial (handled by PMP); PC=0x30aca is stream. IPCP/RL train all prefetchers on both PCs. Alecto identifies that PMP should see PC=0x30b00 and the stream prefetcher should see PC=0x30aca, keeping their tables clean.

2 · The Allocation Table — per-PC, per-prefetcher state

Alecto's core data structure is the Allocation Table, indexed by PC (64 entries). Each entry stores the state of every prefetcher for that PC. Three states:

StateMeaningAction
UIUn-Identified (unknown suitability)Send request with conservative degree (3)
IA_mIdentified Aggressive (accurate, m=0..M)Send request with degree 3+m+1 (more aggressive as m rises)
IB_nIdentified Blocked (unsuitable, n=-N..0)Do NOT send request; blocked for N epochs

When a demand request (PC, address) arrives, Alecto looks up the PC in the Allocation Table. For each prefetcher in UI or IA_m state, it generates an identifier (prefetcher ID, degree) and routes the request to that prefetcher. Prefetchers in IB_n state are skipped entirely.

⚠
Why block instead of just reducing degree? A prefetcher in IB_n isn't just "cautious" — it's wrong for this PC's pattern. Training it on these requests wastes table space and energy with no benefit. Blocking saves both.

3 · State transitions — accuracy thresholds and epochs

States change based on runtime accuracy feedback. Two thresholds:

  • Proficiency Boundary (PB) = 75%: accuracy ≥ PB → promote/stay; accuracy < PB → demote/reevaluate.
  • Deficiency Boundary (DB) = 5%: accuracy < DB → block (IB_-N).

Transitions happen every 100 demand accesses for that PC (one "epoch"). The state machine (Figure 5 of paper):

  1. UI → IA_0 or IB_0: After the first epoch, if a prefetcher's accuracy ≥ PB, promote to IA_0. Others drop to IB_0 (blocked for 0 epochs, i.e., waiting for next decision).
  2. IA_0 → UI: If accuracy falls below PB, demote to UI. If no prefetcher remains in IA, also transition IB_0 → UI (reevaluate).
  3. UI → IB_-N: If accuracy < DB, block for N epochs (N=8 in the paper). Each epoch, IB_n → IB_{n+1} until IB_0, then reevaluate.
  4. IA_m → IA_{m+1} or IA_{m-1}: If accuracy ≥ PB and m < M, increase aggressiveness (m+1). If accuracy < PB and m > 0, decrease (m-1). Max degree = 3 + M + 1 (M=5 → max 9 lines).
🍯
Insight: the state machine is a PID controller for prefetcher aggressiveness. It reacts to accuracy drops (demote/block), amplifies success (promote/raise degree), and has a cooling-off period (IB_n countdown) to avoid oscillation.

4 · Sample Table + Sandbox Table — accuracy measurement

The Allocation Table needs accuracy = (confirmed prefetches) ÷ (issued prefetches). Two helper tables provide this:

Sample Table (indexed by PC): For each PC and each prefetcher, track IssuedByP1, ConfirmedP1, plus a DemandCounter (triggers epoch at 100) and DeadCounter (detects deadlock if no prefetches issue for 150 demands → reset states to UI).

Sandbox Table (indexed by address, 512 entries): Logs recently issued prefetch addresses. Each entry stores the address tag, a valid bit per prefetcher, and a hashed PC. When a demand request hits an entry's tag, Alecto checks if the demand's PC matches the recorded PC. If yes, increment ConfirmedP1 in the Sample Table. The hash function XORs PC segments to compress the PC address to log₂(512)=9 bits.

💡
Why hash the PC? Storing full PC addresses (40 bits) in 512 entries = 2560 bytes just for PCs. Hashing to 9 bits = 576 bits (72 bytes), cutting overhead 97%. The tradeoff: hash collisions may occasionally misattribute a prefetch, but collisions are rare and don't break correctness.

5 · Prefetch filter — eliminating duplicates

Multiple prefetchers in IA_m state may issue overlapping prefetch requests. The Sandbox Table doubles as a prefetch filter: when a prefetcher issues a request for address A, Alecto checks if A already exists in the Sandbox Table (tag hit). If yes, drop the request (duplicate). If no, add it to the table and forward to the prefetch queue. This prevents redundant prefetches with no extra storage beyond the Sandbox Table itself.

6 · Temporal prefetching — filtering for metadata efficiency

Temporal prefetchers (e.g., Triangel) track correlations between memory accesses over time, requiring large metadata tables (>17 kB). Alecto dramatically improves their efficiency by filtering which demand requests train the temporal prefetcher. Three categories to filter:

  1. Non-temporal patterns (stream, stride, spatial) — already handled by non-temporal prefetchers.
  2. Rare recurrence — PC appears so infrequently that metadata evicts before the next occurrence.
  3. Simultaneous coverage — temporal pattern that a non-temporal prefetcher also covers (e.g., stride of +1).

Without modification, Alecto filters all three. Event ① (promote non-temporal to IA_0, block temporal) handles category 1 and 3. Event ③ (block if accuracy < DB) handles category 2. Result: the temporal prefetcher's metadata table stays focused on patterns only it can handle, dramatically reducing table misses. The paper shows Alecto outperforms Bandit by 8.39% and Triangel by 2.18% on temporal benchmarks (Figure 13).

🍯
Insight: Alecto is the first prefetcher selection algorithm to efficiently allocate demand requests for temporal prefetching, cutting metadata table pressure without custom logic — just the existing state machine.
Follow the numbers, step by step

Worked examples

Four traces using the paper's real numbers.

① State transition: UI → IA_0 (promotion)

PC=0x30b00 starts in UI state for all three prefetchers (stream, stride, spatial). After 100 demand accesses, Sample Table shows: stream 10 issued / 4 confirmed (40% accuracy), stride 15 issued / 8 confirmed (53%), spatial 20 issued / 16 confirmed (80%). What happens?

  1. Check PB (75%): Only spatial prefetcher exceeds 75%. Promote spatial to IA_0.
  2. Block others: Stream and stride drop to IB_0 (accuracy below PB but not yet DB).
  3. Next epoch: PC=0x30b00 demand requests route only to spatial (IA_0 state) with conservative degree 3. Stream and stride receive nothing (IB_0 → IB_1 → ... → IB_0 → reevaluate if no IA prefetcher exists).
  4. Result: Spatial's table stays clean; stream and stride don't pollute with irrelevant spatial pattern.
🍯
Insight: promotion is selective — only prefetchers meeting the 75% bar advance. This ensures only accurate prefetchers train on this PC, maximizing table utilization.

② State transition: IA_m → IA_{m+1} (degree increase)

PC=0x30aca is in IA_2 state for stream prefetcher (degree = 3+2+1 = 6). After the next 100 demands, stream issues 60 prefetches, 50 confirmed (83% accuracy). M=5, so max IA_5. What happens?

  1. Check PB: 83% > 75% → accuracy is high.
  2. Check m < M: m=2 < 5 → room to increase aggressiveness.
  3. Transition: IA_2 → IA_3. New degree = 3+3+1 = 7 lines.
  4. Next epoch: stream prefetcher issues 7 prefetches per trigger, boosting coverage while maintaining accuracy. If accuracy stays ≥75%, m can increment again (IA_4, IA_5) up to degree 9. If accuracy drops below 75%, transition to IA_2 (back off).
💡
Incremental degree increases let Alecto find the sweet spot: aggressive enough for coverage, conservative enough to avoid pollution. The IA_m substates form a fine-grained aggressiveness ladder.

③ State transition: UI → IB_-N (blocking)

PC=0x40a88 is in UI for stride prefetcher. After 100 demands, stride issues 25 prefetches, 1 confirmed (4% accuracy). DB=5%. What happens?

  1. Check DB: 4% < 5% → accuracy is abysmal.
  2. Transition: UI → IB_-N (N=8 in the paper). Stride is blocked for 8 epochs.
  3. Next epoch: PC=0x40a88 demand requests do NOT route to stride. Stride's table stops updating for this PC, preventing further pollution. IB_-8 → IB_-7 → ... → IB_0 over 8 epochs.
  4. After 8 epochs: If no other prefetcher is in IA state for this PC, stride transitions IB_0 → UI (reevaluate). If pattern has shifted, stride gets another chance; if not, it blocks again.
🍯
Insight: the N-epoch cooldown (IB_-N) prevents oscillation. A prefetcher that's truly unsuitable stays blocked long enough to avoid wasting energy, but eventually reevaluates in case the pattern changed (phase shift).

④ Storage overhead calculation

Calculate Alecto's total storage for P=3 prefetchers (stream, stride, spatial). Use the paper's Table III.

  1. Allocation Table (64 entries): 1 valid + 9 tag + 4×3 state bits = 22 bits/entry. 64×22 = 1408 bits = 176 bytes. (Formula: 640 + 256×P = 640 + 768 = 1408 bits.)
  2. Sample Table (64 entries): 1 valid + 9 tag + 8×3 issued + 8×3 confirmed + 7 dead + 8 demand = 73 bits/entry. 64×73 = 4672 bits = 584 bytes. (Formula: 1600 + 1024×P = 1600 + 3072 = 4672 bits.)
  3. Sandbox Table (512 entries): 6 tag + 3 valid_per_prefetcher = 9 bits/entry. 512×9 = 4608 bits = 576 bytes. (Formula: 3072 + 512×P = 3072 + 1536 = 4608 bits.)
  4. Total: 1408 + 4672 + 4608 = 10688 bits ≈ 1.34 kB. Excluding Sandbox (which doubles as prefetch filter): 1408 + 4672 = 6080 bits ≈ 760 bytes.
🍯
Compare to Bandit: Bandit with 8 actions per prefetcher = 8×8³ bytes = 4096 bytes = 4 kB. Alecto is 5.4× smaller — and scales linearly (add one prefetcher → +1792 bits), not exponentially.

Problem 1 · easy
A PC has three prefetchers: stream (accuracy 85%), stride (accuracy 70%), spatial (accuracy 40%). Proficiency Boundary = 75%. Which prefetchers does Alecto promote to IA_0 for this PC?
Show answer
Only stream (85% ≥ 75%). Stride (70%) and spatial (40%) both fall below PB and are demoted to IB_0. Alecto promotes prefetchers meeting the threshold, not just the highest one — the goal is to balance accuracy and coverage by allowing multiple high-accuracy prefetchers.Misconception: that Alecto always picks the single highest-accuracy prefetcher. Multiple prefetchers can be in IA_m state simultaneously if they all meet PB.
Problem 2 · medium
Why does Alecto use per-PC state instead of a global prefetcher selection? Give an example where global fails and per-PC succeeds.
Show answer
Different PCs exhibit different patterns. Example: PC1 streams (+1,+1,+1) → best served by stream prefetcher. PC2 strides by 4 (+4,+4,+4) → best served by stride prefetcher. A global rule (e.g., "always prefer stream") sends PC2's requests to stream, which generates inaccurate +1 prefetches, while stride sits idle. Per-PC state lets Alecto route PC1 to stream and PC2 to stride, maximizing accuracy.Misconception: that one prefetcher selection rule fits all. Real programs interleave multiple patterns from different instructions, requiring fine-grained per-PC tracking.
Problem 3 · medium
Alecto's Sandbox Table serves two roles. Name them and explain why this dual role saves storage.
Show answer
(1) Prefetch filter: checks incoming prefetch requests against logged addresses, drops duplicates. (2) Feedback tracker: logs which prefetcher issued which address, so when a demand hits that address, Alecto credits the correct prefetcher's confirmed counter. Without the dual role, you'd need a separate prefetch filter (another 512-entry table), doubling storage. By reusing the Sandbox Table for both, Alecto eliminates that overhead.Misconception: that the Sandbox Table is "just" a filter. Its feedback role is equally critical — it's the link between issued prefetches and confirmed hits, driving the accuracy measurement.
Problem 4 · hard
A prefetcher is in IA_4 state (degree = 3+4+1 = 8). Over the next epoch, accuracy drops to 72% (below PB=75%). M=5. What is the next state and degree?
Show answer
Transition: IA_4 → IA_3 (back off one level). New degree = 3+3+1 = 7. The state machine decrements m when accuracy falls below PB, reducing aggressiveness to recover accuracy. If accuracy continues to fall, m keeps decrementing (IA_2, IA_1, IA_0, then eventually UI if it stays below PB at IA_0).Misconception: that dropping below PB immediately blocks the prefetcher. The IA_m substates provide a gradual backoff ladder — only sustained low accuracy or falling below DB (5%) triggers blocking (IB_-N).
Problem 5 · medium
Alecto's Sample Table uses a DeadCounter. What problem does it detect, and what is the recovery action?
Show answer
The DeadCounter detects deadlock: a PC is in IA_m state but the prefetcher stops issuing prefetch requests for 150+ demands (e.g., pattern shifted, no longer matches). This wastes the IA_m slot and prevents other prefetchers from being tried. Recovery: reset all prefetcher states for this PC to UI, forcing a fresh identification round. The DeadCounter is a saturation counter (increments on no-prefetch, decrements otherwise), triggering reset at a threshold (e.g., 150).Misconception: that state transitions only happen every 100 demands. The DeadCounter is an exception — it can reset states mid-epoch to escape deadlock, ensuring Alecto doesn't get stuck with a silent prefetcher.
Problem 6 · hard
Compare Alecto's storage overhead to Bandit's for a system with 4 prefetchers and 10 actions per prefetcher. Show the calculation for both.
Show answer
Alecto: 5312 + 1792×P = 5312 + 1792×4 = 12480 bits = 1560 bytes (or 920 bytes excluding Sandbox). Bandit: 8 × (#actions)^(#prefetchers) = 8 × 10^4 = 80000 bytes = 80 kB. Alecto is 51× smaller. Bandit's exponential scaling makes it impractical beyond a few prefetchers; Alecto's linear scaling remains manageable.Misconception: that RL-based schemes are "lightweight." Even Bandit (the smallest RL approach) explodes exponentially with prefetchers or actions. Alecto's linear overhead is fundamentally more scalable.
Problem 7 · medium
Alecto filters demand requests for temporal prefetchers. Name the three categories of requests Alecto filters and explain why each should be filtered.
Show answer
(1) Non-temporal patterns (stream, stride, spatial): already handled by cheaper non-temporal prefetchers; no need for temporal metadata. (2) Rare recurrence: PC appears so infrequently that temporal metadata evicts before the next occurrence — storing it wastes table space. (3) Simultaneous coverage: temporal pattern that a non-temporal prefetcher also covers (e.g., stride +1) — redundant to track in both. Filtering all three keeps the temporal prefetcher's metadata table focused on patterns only it can predict, dramatically reducing table misses.Misconception: that temporal prefetchers should train on all accesses. Temporal prefetching is expensive (large metadata); filtering non-temporal and rare patterns is essential for efficiency.
Problem 8 · easy
Why does Alecto hash the PC before storing it in the Sandbox Table? What is the tradeoff?
Show answer
A full PC is 40 bits; storing 512 PCs = 2560 bytes. Hashing to log₂(512)=9 bits = 576 bits = 72 bytes, cutting overhead 97%. Tradeoff: hash collisions may occasionally misattribute a prefetch (e.g., PC1 and PC2 hash to the same value, so a demand from PC2 is credited to PC1's prefetcher). This is rare and doesn't break correctness — worst case, one prefetcher's accuracy is slightly overestimated, another slightly underestimated. The massive storage savings justify the small risk.Misconception: that hash collisions make Alecto unreliable. Collisions are infrequent (hash function spreads PCs evenly) and the error is benign — state transitions are gradual, so one misattributed prefetch doesn't derail the system.
Quick answers

Frequently asked questions

Ten questions newcomers ask most. Tap any to expand.

Q1What does "dynamic demand request allocation" actually mean?▾
Concretely: each demand request (PC, address) is routed only to prefetchers that are suitable for that PC's pattern, rather than training all prefetchers on all requests. "Dynamic" = the routing decision changes at runtime based on observed accuracy. This prevents table pollution and reduces conflicts.
Q2Why do prior approaches (IPCP, Bandit) pollute prefetcher tables?▾
They either (a) train all prefetchers on all demand requests (IPCP, Bandit) or (b) sequentially pass requests through prefetchers without blocking irrelevant ones (DOL). Result: a stride-pattern request updates the spatial prefetcher's table, a spatial-pattern request updates the stride prefetcher's table, etc. — useful entries evict, table miss rates spike, accuracy drops.
Q3How does Alecto know which prefetcher is "suitable" for a given PC?▾
It learns at runtime. Initially, all prefetchers start in UI (unknown). After 100 demand accesses, Alecto calculates each prefetcher's accuracy for that PC (issued vs confirmed prefetches). Accuracy ≥75% → promote to IA_0 (suitable); accuracy <5% → block (IB_-N, unsuitable); otherwise stay UI. The state updates every 100 demands based on fresh accuracy feedback.
Q4What are the three states (UI, IA_m, IB_n) and why have substates?▾
UI = unknown suitability, send requests conservatively (degree 3). IA_m = suitable and accurate, send requests with degree 3+m+1 (m=0..M, higher = more aggressive). IB_n = unsuitable, do NOT send requests, blocked for N epochs (n=-N..0, counts up each epoch until reevaluation). Substates let Alecto fine-tune aggressiveness (IA_m) and implement a gradual cooldown (IB_n) rather than abrupt on/off.
Q5How do the Sample Table and Sandbox Table work together?▾
Sample Table (indexed by PC): tracks issued and confirmed prefetches per PC per prefetcher. Sandbox Table (indexed by address): logs recently issued prefetch addresses and which prefetcher issued them. When a demand hits an address in Sandbox, Alecto increments the confirmed counter in Sample Table. Sample Table then computes accuracy = confirmed ÷ issued and feeds this to Allocation Table for state transitions.
Q6Why does the Sandbox Table double as a prefetch filter?▾
Because it already logs issued prefetch addresses. When a new prefetch request arrives, Alecto checks if the address exists in Sandbox (tag hit). If yes, it's a duplicate → drop. If no, add it. This eliminates redundant prefetches from multiple prefetchers with no extra storage beyond the Sandbox Table itself, which is needed anyway for accuracy feedback.
Q7How does Alecto scale to many prefetchers vs RL-based schemes?▾
Alecto's storage = 5312 + 1792×P bits (linear in P). Bandit's storage = 8 × (#actions)^P bytes (exponential in P). For P=3, Alecto ≈1.3 kB vs Bandit ≈4 kB. For P=4, Alecto ≈1.6 kB vs Bandit ≈32 kB. Alecto's linear scaling makes it practical even for 5+ prefetchers; Bandit's exponential explosion doesn't.
Q8Does Alecto always outperform Bandit?▾
Not always. On a few benchmarks (mcf, omnetpp), Bandit's aggressive degree (6) wins because the pattern is very regular and high coverage matters more than accuracy. Alecto's moderate approach (max degree 3+M+1=9, but starts at 3) is less aggressive. However, on average across SPEC CPU2006/2017, Alecto wins +2.76% single-core, +7.56% eight-core, and +5.25% on memory-intensive workloads.
Q9Why does Alecto help temporal prefetching so much?▾
Temporal prefetchers have large metadata tables (>17 kB) that track correlations. If non-temporal patterns (stream, stride) train the temporal prefetcher, they waste metadata space. Alecto filters three categories: (1) non-temporal patterns (handled by cheaper prefetchers), (2) rare recurrence (evicts before next use), (3) simultaneous coverage (e.g., stride +1, redundant with stream). Result: temporal metadata table stays focused on patterns only it can handle, reducing table misses dramatically.
Q10Can Alecto adapt to phase changes (when a PC's pattern shifts)?▾
Yes, but with a lag. States update every 100 demand accesses (~few microseconds). If a PC's pattern shifts, the prefetcher in IA_m state will see accuracy drop → eventually demote to UI or block (IB_n). After the cooldown (N epochs), it reevaluates. The DeadCounter also detects if a prefetcher stops issuing requests (deadlock) and forces a reset to UI. So Alecto adapts, but not instantly — very fast phase changes (< 100 accesses) can lag.

Glossary & further reading

Every term in one place, then a path to go deeper.

Core terms
DDRA (Dynamic Demand Request Allocation)Alecto's principle: route each demand request only to prefetchers that can benefit from it, based on per-PC runtime accuracy feedback.
Allocation TablePer-PC state for each prefetcher (UI, IA_m, IB_n). Indexed by PC (64 entries). Decides which prefetchers receive each demand request.
Sample TableTracks issued/confirmed prefetches per PC per prefetcher. Calculates accuracy every 100 demands, triggers state transitions in Allocation Table.
Sandbox TableIndexed by address (512 entries). Dual role: (1) prefetch filter (drop duplicates), (2) feedback tracker (log which prefetcher issued which address).
UI (Un-Identified)Prefetcher suitability unknown. Alecto sends requests with conservative degree (3) while gathering accuracy data.
IA_m (Identified Aggressive)Prefetcher is suitable (accuracy ≥75%). Degree = 3+m+1. Higher m = more aggressive (up to M+1, M=5 in paper).
IB_n (Identified Blocked)Prefetcher is unsuitable (accuracy <5%). Blocked for N epochs. No requests sent. Counts up (IB_-N → IB_0) each epoch.
Prefetch degreeNumber of cache lines prefetched per trigger. Conservative (e.g., 3) reduces pollution; aggressive (e.g., 9) boosts coverage but risks accuracy.
Supporting terms
PB (Proficiency Boundary)Accuracy threshold (75%). Prefetcher must exceed PB to promote to IA_0 or stay in IA_m.
DB (Deficiency Boundary)Accuracy threshold (5%). Fall below and prefetcher blocks (IB_-N).
Epoch100 demand accesses for a given PC. State transitions happen at epoch boundaries.
DeadCounterSaturation counter in Sample Table. Detects deadlock (prefetcher in IA_m but not issuing requests). Threshold (e.g., 150) resets states to UI.
Prefetcher table pollutionIrrelevant demand requests update a prefetcher's tables, evicting useful entries. Caused by training all prefetchers on all requests.
Table missPrefetcher's IP table or pattern table lacks an entry for the incoming PC or pattern. Requires eviction or miss handling. Alecto cuts table misses significantly.
AccuracyFraction of issued prefetches that hit a future demand. Confirmed ÷ Issued. Alecto's primary metric for state transitions.
CoverageFraction of demand misses eliminated by prefetching. Alecto balances accuracy and coverage by promoting multiple high-accuracy prefetchers.
TimelinessDid the prefetch arrive before the demand stalled? Alecto improves timeliness by raising degree for IA_m prefetchers.
Stream / stride / spatial prefetchersThe three prefetchers Alecto schedules. Stream: sequential access (+1). Stride: fixed delta. Spatial: region-based pattern.
Temporal prefetcherTracks correlations over time (e.g., A→B→C). Requires large metadata. Alecto filters demand requests for temporal prefetchers to reduce metadata pressure.
Prefetch filterDrops duplicate prefetch requests. Alecto's Sandbox Table serves this role at no extra cost.
