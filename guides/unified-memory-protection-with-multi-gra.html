One security engine, many access patterns — unified memory protection

Heterogeneous SoCs run CPU, GPU, and NPU workloads through a single off-chip memory, each accessing data in wildly different chunk sizes. This is how multi-granular MAC&tree protects all of them with the same hardware.

📄 Lee et al., "Unified Memory Protection" · ISCA '25 🎓 Assumes counter-mode encryption basics, tiering/CXL context ⏱ ~25 min read
🔐

Measure each memory region's actual access granularity (64B fine-grained vs 32KB coarse-grained) dynamically, then store counters and MACs at that granularity — cutting metadata by up to 21% execution-time improvement.

Before you begin

Seven prerequisites

Each card explains a concept the paper builds on. Read in order the first time, or skip to what's unfamiliar.

Quick glossary of key acronyms
MACMessage Authentication Code — an 8B hash per 64B block to detect tampering
CTR (counter)Version number per block for replay-attack protection; stored in an integrity tree
SoCSystem-on-a-Chip — CPU + GPU + NPU sharing one off-chip DRAM
GranularityThe chunk size (64B, 512B, 4KB, 32KB) at which counters and MACs protect data; coarser granularity means one counter or MAC covers multiple blocks, but switching granularity requires re-encryption and tree updates
1Counter-mode encryption and replay attacks▾

Off-chip memory is untrusted. To guarantee confidentiality, each 64B block is encrypted with a one-time-pad (OTP) derived from (secret_key, address, counter). The counter increments on dirty eviction (when a modified block is written back), so the pad never repeats. An attacker who replays an old ciphertext is defeated because the counter has changed and decryption yields garbage.

The counter itself lives in off-chip memory, so we verify its freshness with an integrity tree rooted on-chip.

Now you're ready: The paper reduces overhead by using one counter for multiple blocks when they're written together — a coarse-grained counter.
2Message Authentication Codes (MACs) for integrity▾

Even if data is encrypted, an attacker can flip bits. A MAC is a keyed hash of the plaintext, stored alongside the ciphertext. On a read, you decrypt, recompute the MAC, and compare. Mismatch = tampering detected, abort.

Standard approach: one 8B MAC per 64B block. For a 4GB system that's 512MB of MACs — massive traffic overhead.

Now you're ready: The paper merges multiple fine-grained MACs into one coarse-grained MAC for regions written in bulk, compacting that 512MB significantly.
3The integrity tree of counters▾

Counters live off-chip too — an attacker could replay an old counter. To prevent that, counters form a Merkle-like tree: each non-leaf node is a hash of its 8 children. The root is stored on-chip in tamper-proof storage. Verifying a counter means checking the path from leaf to root — typically 3–4 levels for GB-scale memory.

Latency = tree_height × memory_access_latency. Reducing tree height or caching intermediate nodes is critical.

Now you're ready: The multi-granular tree prunes child nodes for coarse-grained regions, shortening the tree.
4Heterogeneous SoCs — CPU, GPU, NPU sharing one DRAM▾

Modern edge devices (Orin, Apple M1) integrate a general-purpose CPU, a GPU for parallel workloads, and an NPU (deep-learning accelerator with a systolic array) on one chip, all accessing a shared off-chip LPDDR4/5. Each unit has different access patterns: CPU often fine-grained (64B), GPU mixes, NPU often coarse-grained (32KB tensors).

A single memory-protection engine sees a jumbled stream of requests. Prior work optimized for one device or one domain (e.g., ML tensors) but not all three together.

Now you're ready: The paper's challenge is a unified mechanism that detects the right granularity per region, adapting to whatever workload each unit runs.
5Access patterns — streaming vs fine-grained▾

A stream chunk means all blocks in a chunk (e.g., 512B) are accessed within a short window (16K cycles). NPU tensor loads are highly streaming (up to 74% of accesses in 32KB chunks). CPU pointer-chasing or hash-table lookups are fine-grained (64B). GPU sits in between, depending on the kernel.

The paper's measurement: 64B stream chunks dominate CPU (up to 95%), 32KB chunks dominate NPU (64.5% on average), GPU varies widely.

Now you're ready: A per-device static granularity mispredicts up to 50% of accesses. Dynamic per-512B-partition granularity is the solution.
6Prior dual-granular approaches and their limits▾

Common Counters (GPU) shares a single counter for a 32KB segment if all counters are equal, but it only supports 16 shared counters and requires a scanning step at kernel end [35]. Adaptive (GPU) uses 4KB dual-granular MACs but doesn't optimize counters [56]. NPU studies (TNPU, GuardNN, MGX) use software-detected tensor granularity with on-chip version numbers — works for ML, not general workloads.

None supports multi-granularity (more than two levels) for both counters and MACs in a general, device-independent way.

Now you're ready: This paper proposes 4 granularities (64B, 512B, 4KB, 32KB) for counters + MACs + tree, detected dynamically.
7Subtree optimization techniques (BMF, PENGLAI)▾

Bonsai Merkle Forests caches roots of subtrees for hot regions [17]; PENGLAI prunes unused memory [16]. Both reduce tree height by decomposing one tall tree into multiple shorter ones. Orthogonal to granularity — the paper combines them for further gains.

Subtrees reduce traversal depth; multi-granular counters reduce the number of leaf counters. Together they compound: fewer nodes and shorter paths.

Now you're ready: The paper's final result uses both: multi-granular MAC&tree + subtrees → 21.1% execution-time improvement vs conventional fixed-granular MAC&tree.
The whole story in one place

Multi-granular MAC&tree in a nutshell

The problem, the detection mechanism, and the unified tree that solves it.

Conventional memory protection assigns one 64B-granular counter and one 8B MAC per cacheline, ballooning metadata traffic and cache misses in heterogeneous SoCs where GPU and NPU workloads access data in multi-KB chunks. Prior work optimized either counters or MACs, not both, and only for dual granularity (64B vs one coarse size).

This paper proposes a unified multi-granular scheme supporting four granularities (64B, 512B, 4KB, 32KB) for both counters and MACs. An access tracker monitors accesses per 32KB chunk; when all blocks in a 512B partition are touched within 16K cycles, that partition is promoted to coarse-grained. The promotion merges fine-grained MACs into one coarse-grained MAC (via nested hashing) and prunes child counter-tree nodes, shortening the integrity tree. A granularity table stored in a secure memory region records the current granularity per chunk; an address-compute engine resolves counter/MAC addresses from (address, granularity).

Key optimization: lazy switching. When granularity changes, the mechanism doesn't immediately re-encrypt all data — instead it marks both old and new granularity in the table and handles mismatches on-demand. 73.5% of accesses correctly predict the granularity and require no switching overhead; mispredictions are handled case-by-case (scale-down, scale-up RAR/WAR/WAW), with total switching overhead reduced to ~8% of access requests (8.8% for RAR counter fetches, 2.8% for non-read-only MAC scale-down cases).

Result: On 250 heterogeneous scenarios (CPU+GPU+2×NPU), multi-granular MAC&tree improves execution time by 14.2% over conventional 64B-only. Combined with subtree optimization (BMF&Unused), it reaches 21.1% improvement. Hardware overhead: 850B on-chip + 2MB granularity table (~0.029% area, 0.71% power on an Orin-class SoC).

🔑 Key takeaways

  • Conventional 64B-fixed protection causes 33.8% overhead (CPU+GPU+NPU); multi-granular MAC&tree cuts it to 19.6%.
  • Four granularities (64B / 512B / 4KB / 32KB) chosen as powers-of-8 to align with 8-arity integrity tree.
  • Access tracker monitors 32KB chunks; granularity detection marks 512B partitions as stream or not.
  • Coarse-grained MACs: merge via nested hashing (the existing protection engine is 64B-cacheline based, so the coarse MAC is computed from fine-grained MACs as Hash(Hash(MAC1), Hash(MAC2), ...)); counters: prune child nodes in the tree.
  • Lazy switching defers re-encryption until a mismatch occurs; 73.5% of accesses correctly predict granularity and incur zero switching cost, with remaining cases handled on-demand (total switching overhead ~8% of requests).
  • Subtrees (BMF) + multi-granular tree compound: the paper reports that BMF&Unused+Ours improves by 7.4% over standalone BMF&Unused, contributing to the 21.1% total improvement.
  • Unified = one mechanism for CPU, GPU, NPU — no per-device logic, scalable to future accelerators.
Six everyday pictures

How to think about multi-granular MAC&tree

Each picture stands alone. Every card ends with an honest note about where the analogy breaks.

The core idea · Unified vs per-device

One luggage scale, three travelers

Three travelers (CPU, GPU, NPU) bring bags of different sizes to an airport security checkpoint. A conventional scale measures each bag individually (64B fixed granularity) even if one traveler (NPU) always packs a giant suitcase. A smart scale detects each traveler's typical bag size and adjusts its measurement granularity — one reading for the giant suitcase instead of 512 individual scans. That's multi-granular MAC&tree: one security engine adapts to each unit's pattern.

Where it breaks: real bags are weighed once; memory protection checks every access. The "scale" (access tracker) runs continuously, updating its granularity map as patterns shift.
Access tracker · One-hot vector

Bingo card for memory chunks

The access tracker is a bingo card where each row represents a 32KB chunk of memory and each square a 64B block. When a block is accessed, you mark its square. The card is evicted when (1) the tracker entry's lifetime exceeds 16K cycles, (2) the number of chunk accesses exceeds 512, or (3) LRU replacement is needed. On eviction, you check: if every square in an 8×8 region (512B partition) is marked, that partition is a stream and gets promoted to coarse granularity. The card itself is tiny (12 rows × 512 bits = 842B on-chip).

Where it breaks: bingo waits until the end of the game; the tracker evicts entries continuously (LRU policy) and runs granularity detection on eviction, not at a fixed deadline.
Merged MACs · Nested hashing

Sealing envelopes inside envelopes

Fine-grained: each 64B block has its own tamper-evident seal (MAC). Coarse-grained: you seal 8 blocks together in one big envelope, then seal that envelope in another layer. The outer seal (coarse-grained MAC) is computed as Hash(Hash(MAC1), Hash(MAC2), ...). To verify, you open the outer envelope, recompute the nested hash from the 8 inner seals, and compare. Same security, one outer seal instead of 8 separate ones in the metadata cache.

Where it breaks: real envelopes can't be unsealed without destroying them; MACs are recomputed on every read. The nested hash is deterministic — no destruction, just re-computation.
Multi-granular tree · Pruning child nodes

Cutting off tree branches

A conventional integrity tree has a leaf counter for every 64B block (8 per parent node). When a region goes coarse-grained (512B = 8×64B), you prune all 8 child nodes and promote their responsibility to the parent. The parent counter becomes the leaf counter for that region. The tree is now one level shorter for coarse regions — fewer hops to verify, fewer metadata-cache misses. It's like cutting off branches of a tree: fewer nodes = faster traversal.

Where it breaks: real tree pruning is permanent; the multi-granular tree can "regrow" branches if granularity scales down (a region becomes fine-grained again). The promotion is reversible.
Lazy switching · Deferring re-encryption

Double-booking during a move

You're moving apartments but haven't finished packing. For a week you have keys to both the old place and the new one — double-booked. When someone visits, you give them whichever address you've moved their stuff to. Lazy switching stores both the old granularity and the new granularity in the granularity table. If the next access matches the new granularity, you skip re-encryption. If it misses, you handle it on-demand. Result: 73.5% of requests correctly predict the granularity and pay zero switching cost.

Where it breaks: apartments don't change their addresses on the fly; memory regions do (the address-compute engine recalculates CTR/MAC addresses based on granularity). The "move" is a logical re-mapping, not a physical relocation.
Granularity table · Trusted storage

The vault that stores the rulebook

The granularity table (2MB for 4GB memory) lives in a secure memory region protected by a separate fixed-64B integrity tree. It's like a vault that stores the rulebook for which memory regions use which granularity. An attacker can't tamper with the table without breaking the vault's own integrity checks. The table is too big for on-chip storage (2MB vs a few KB of cache), so it's off-chip but under the umbrella of a TEE (Intel SGX, Arm TrustZone, etc.). The address-compute engine reads it on every access to decide how to interpret addresses.

Where it breaks: a vault is read rarely; the granularity table is read on every memory access. Fortunately it has high locality (16B entry per 32KB of data) and benefits from the existing counter-cache infrastructure — only 0.3% extra overhead.
The full explanation

How it actually works

Section by section, with the paper's key mechanisms rebuilt as step-by-step flows.

1 · The problem — fixed 64B granularity on heterogeneous SoCs

A heterogeneous SoC runs CPU (general-purpose), GPU (parallel), and NPU (ML accelerator) workloads through a single memory-protection engine. CPU often accesses fine-grained (64B pointer-chasing), GPU mixes, NPU streams coarse-grained (32KB tensors). Conventional counter-mode protection assigns one 8B MAC and one counter per 64B block.

The overhead: For a 4GB system with 64B granularity, that's 512MB of MACs + counter tree traffic. On the paper's heterogeneous system, this causes 33.8% execution-time degradation vs unsecured (14.3% from MACs, 19.5% from counters+tree). The metadata cache (8KB) and MAC cache (4KB) thrash on coarse-grained NPU accesses.

⚠
Why prior work doesn't solve it: Common Counters (GPU) only does counters, supports 16 shared counters, requires scanning [35]. Adaptive (GPU) does MACs only, dual-granular (64B/4KB) [56]. NPU studies (TNPU, MGX) use software-detected tensor granularity — not general, not dynamic for mixed workloads.

2 · Access tracking — detecting stream patterns per 32KB chunk

The access tracker monitors memory accesses to detect coarse-grained patterns. It has 12 entries (one per processing unit on average), each tracking a 32KB chunk with a 512-bit one-hot vector (one bit per 64B block). When block i in chunk j is accessed, set tracker[j].bits[i] = 1.

# on memory access
chunk_idx = addr[63:15]     # upper 49 bits
block_idx = addr[14:6]      # bits 14:6 = offset within 32KB
entry = find_or_allocate(chunk_idx)
entry.bits[block_idx] = 1      # mark accessed

An entry is evicted when: (1) the entry's lifetime exceeds 16K cycles, (2) the number of chunk accesses exceeds 512, or (3) LRU replacement (12 entries full). On eviction, run granularity detection.

🌱
Design choice: 32KB chunks because that's the coarsest granularity (32KB = 8³×64B in an 8-arity tree). 512 bits = 64B of on-chip storage per entry. 12 entries = 842B total.

3 · Granularity detection — scanning partitions for stream patterns

When an access-tracker entry is evicted, the granularity detection engine scans its 512-bit vector in chunks of 8 bits (each 8×64B = 512B partition). If all 8 bits in a partition are set, mark that partition as stream in the streampart bitmap (64 bits per 32KB chunk, one bit per 512B partition).

# Algorithm 1 in paper (simplified)
streampart = 0
for i in 0..63:          # 64 partitions of 512B
    partition = access_bits[i*8 : (i+1)*8]
    if all_bits_set(partition):
        streampart |= (1 << i)
return streampart

Special cases: if 8 consecutive partitions are all stream (bits 0–7, or 8–15, etc.), the granularity is 4KB. If all 64 partitions are stream (streampart = 0xFFFF_FFFF_FFFF_FFFF), the granularity is 32KB. Otherwise it's a mix of 512B and 64B.

💡
Why 512B is the "partition" unit: The granularity table stores one streampart bitmap per 32KB chunk. 64 bits = one bit per 512B = the second-finest granularity. Finer than that (64B) doesn't need a bit — it's the default.

4 · Storing granularity — the granularity table in secure memory

The detected streampart bitmap is written to the granularity table, a per-chunk record stored in a secure memory region (Intel SGX enclave, or similar TEE). Each entry is 16B: 8B for the current granularity, 8B for the next granularity (lazy switching).

For a 4GB memory, that's 4GB / 32KB × 16B = 2MB of table. Too big for on-chip, so it lives off-chip but protected by a separate fixed-64B integrity tree (the granularity table itself is encrypted and MAC'd). The on-chip overhead is just the address-compute engine (a few ALUs) and the 12 access-tracker entries (842B).

🍯
Storage breakdown: 2MB table = 0.5% of 4GB memory. Locality is high (16B entry per 32KB of data), so counter-cache hit rate is good. The paper measures 0.3% extra overhead vs data access overhead.

5 · Promotion — merging MACs and pruning tree nodes

When a region's granularity is promoted (e.g., 64B → 512B), two things happen, and both require re-encryption of the data chunk to update counter values and MAC-recalculation steps:

  1. Merge MACs: The 8 fine-grained MACs (one per 64B block) are combined into a single coarse-grained MAC via nested hashing: MAC_coarse = Hash(Hash(MAC1), Hash(MAC2), ..., Hash(MAC8)). This matches how MACs are verified: decrypt all 8 blocks, recompute their MACs, hash the MACs, and compare to the stored coarse MAC. The nested-hash approach is used because the existing protection engine operates on 64B cachelines, so the coarse MAC is constructed from fine-grained MACs rather than as a monolithic hash of the entire chunk.
  2. Prune tree nodes: The 8 child counters (leaf nodes) are removed from the integrity tree, and the parent counter is updated to MAX(CTR1..8) + 1 (upon scale-up) to ensure no replay attacks. The parent becomes the new leaf node for that region. The tree is now one level shorter. Reverse (demotion, coarse → fine): the parent counter value is copied to all new child counters upon scale-down.

Reverse (demotion, 512B → 64B): split the coarse MAC into 8 fine MACs (recompute from decrypted data), and "regrow" the 8 child counters by copying the parent counter value (same counter for all children, since they were all encrypted with the same coarse-grained counter).

⚠
Switching overhead: Promotion requires re-encryption of the data chunk (to update counters), fetching tree nodes up to the root (for counter update), and MAC recalculation. The paper uses lazy switching to defer this cost until the next access.

6 · Address computation — resolving CTR/MAC addresses from granularity

Because counters and MACs are merged/promoted, their off-chip addresses shift. The address-compute engine calculates the new addresses from (address, granularity).

For MACs (Equation 1 in paper):

Idx_MAC = (addr / granul) # index within MAC array
Addr_MAC = Base_MAC + Idx_MAC × 8B

For counters (Equations 2–4 in paper): counters are promoted up the tree, so we need to find the ancestor node at the right level. Example: 512B granularity = 1 level up from leaf, so Parents = log₈(512/64) = 1. Walk the tree index up by 1 parent.

Parents = log₈(granul / 64B)
Idx_CTR = Ancestor(Idx_CTR_leaf, Parents)   # walk tree
Addr_CTR = Base_CTR + (Idx_CTR / 8) × 64B  # 8-arity
🌱
Example: 32KB granularity = log₈(32768/64) = 3 parents. The leaf counter at level 3 is promoted to level 0 (the root). The address is 3 levels higher in the tree than the leaf address.

7 · Lazy switching — deferring re-encryption overhead

Granularity switching (64B ↔ 512B ↔ 4KB ↔ 32KB) requires: (1) fetching the data chunk, (2) re-encrypting with the new counter, (3) recomputing MACs, (4) updating tree nodes up to the root. Expensive.

Lazy switching stores both old and new granularity in the granularity table. On the next access, if the request matches the new granularity, proceed normally (no switching cost). If it misses (requests the old granularity), handle it as a special case: fetch the data chunk, perform the switching steps, and update the table to mark the switch complete.

CaseFrequencyExtra Overhead
Correct prediction (fine→fine, coarse→coarse)73.5%Zero
Scale-down (coarse→fine), all cases4.4%Zero (leaf node stays cached)
Scale-up (fine→coarse), WAR/WAW8.1%Zero (next write fetches tree anyway)
Scale-up (fine→coarse), RAR8.8%Fetch parent→root (metadata cache hit likely)
Scale-up (fine→coarse), RAW5.2%Negligible (prior write cached tree)

Result: 73.5% of requests correctly predict the granularity and pay zero switching cost. Scale-down cases (4.4%) and WAR/WAW scale-up (8.1%) also incur zero overhead. Only RAR (8.8%) pays moderate cost (parent-to-root fetch). Overall switching overhead is approximately 8% of access requests (8.8% RAR counter cases + 2.8% non-read-only MAC scale-down cases from Table 2), vs a naive approach that would trigger re-encryption on every misprediction (26.5% total misprediction rate).

8 · Combining with subtrees (BMF & PENGLAI) — orthogonal optimization

Bonsai Merkle Forests (BMF) and PENGLAI prune the integrity tree by caching roots of subtrees for hot regions [16,17]. This is orthogonal to multi-granular counters: subtrees reduce tree height by dividing one tall tree into many short ones; multi-granular counters reduce the number of leaf nodes by merging them.

The paper combines both: for a 32KB chunk with 512B granularity, the multi-granular tree has 64 leaf counters instead of 512. If that chunk is hot, BMF caches its subtree root on-chip, further reducing traversal depth. The two optimizations compound: fewer nodes and shorter paths.

Result: Multi-granular MAC&tree alone = 14.2% improvement. Multi-granular MAC&tree + subtrees (BMF&Unused) = 21.1% improvement vs conventional. The paper reports that BMF&Unused+Ours improves by 7.4% compared to standalone BMF&Unused, indicating the compounding benefit of multi-granular counters atop subtree pruning.

Follow the numbers, step by step

Worked examples

Four traces, each carried through the full mechanism — from access pattern to final performance gain.

① Detecting a 512B stream partition

A GPU kernel accesses blocks 0, 1, 2, ..., 7 within a 32KB chunk (addresses 0x1000, 0x1040, 0x1080, ..., 0x11C0) in 5000 cycles. What granularity is detected?

  1. Chunk index: 0x1000 >> 15 = 0x0 (upper 49 bits). All 8 accesses map to chunk 0.
  2. Block indices: 0x1000[14:6] = 0, 0x1040[14:6] = 1, ..., 0x11C0[14:6] = 7. Access tracker entry for chunk 0 has bits 0–7 set.
  3. Eviction: After 5000 cycles (< 16K, but assume eviction for demonstration), granularity detection scans partition 0 (bits 0–7). All 8 bits set → streampart |= (1 << 0).
  4. Result: streampart = 0x0000_0000_0000_0001. Bit 0 set → partition 0 is 512B granularity. Remaining partitions (bits 1–63 unset) are 64B granularity.
  5. MAC compaction: The 8 fine-grained MACs (8B each = 64B total) are merged into 1 coarse MAC (8B). Saved: 56B of MAC traffic.
  6. Tree pruning: The 8 leaf counters are pruned; the parent counter is promoted. Tree height for this partition: 2 levels instead of 3.
🍯
Impact: For this 512B partition, memory traffic for MACs drops 87.5% (56B saved per 512B), and tree traversal is one level shorter (saving one memory round-trip per access to this partition).

② Computing the MAC address for a promoted region

A 32KB chunk (base address 0x10000) has partition 0 promoted to 512B. What is the off-chip MAC address for block 0x10080?

  1. Granularity lookup: Read granularity table entry for chunk 0x10000 >> 15 = 0x2. streampart = 0x0001 (bit 0 set). Partition 0 is 512B.
  2. Determine partition: Block 0x10080[14:6] = 2 → partition 2 / 8 = 0. Granularity = 512B.
  3. MAC index: In conventional scheme, MAC index = 0x10080 / 64B = 0x402. But partition 0 is coarse: the 8 fine MACs (indices 0x400–0x407) are merged into 1 coarse MAC at index 0x400 / 8 = 0x80.
  4. MAC address: Base_MAC + 0x80 × 8B = Base_MAC + 0x400.
  5. Check: Blocks 0x10000, 0x10040, ..., 0x101C0 all map to MAC index 0x80 — they share one coarse MAC.
💡
Why divide by 8: The 8 fine MACs are merged into 1. The merged MAC is placed at the first index of the group (index 0x400) in the conceptual fine-MAC array, but physically it's stored at index 0x400 / 8 in the merged-MAC array to eliminate fragmentation (the paper's "compaction").

③ Lazy switching — handling a granularity mismatch

A 32KB chunk is promoted from 64B to 512B (partition 0). The next access is to block 0x10040 (still within partition 0). What happens?

  1. First access (promotion): Granularity table updated: current = 0x0000 (all 64B), next = 0x0001 (partition 0 is 512B). Lazy switching: no re-encryption yet.
  2. Second access (0x10040): Read granularity table: current = 0x0000, next = 0x0001. Block 0x10040 belongs to partition 0, which has next = 512B.
  3. Address compute: Use next granularity (512B). MAC address = Base_MAC + 0x80 × 8B (as in example ②).
  4. Fetch: Load the coarse MAC (8B) and the 8 blocks (512B). Decrypt all 8 blocks using the parent counter, recompute the nested MAC, verify. Success → mark current = next = 0x0001 in the table (promotion complete).
  5. Overhead: One-time fetch of 512B + 8B MAC + parent counter + tree path to root. Subsequent accesses to partition 0 pay no switching cost (granularity is now stable).
🌱
Contrast with eager switching: If the mechanism re-encrypted immediately on promotion, it would fetch+decrypt+re-encrypt all 512B even if the next access were to a different partition. Lazy switching defers that work until the data is actually needed, reducing wasted effort.

④ Performance breakdown — a coarse-grained NPU workload (alex)

The paper's "alex" (AlexNet CNN) NPU workload shows 74.1% of accesses in 32KB stream chunks. What's the measured gain?

  1. Baseline (conventional): 1.0× execution time (normalized to unsecured). All counters/MACs are 64B-granular. Metadata traffic = 512MB MACs + counter tree for 4GB memory.
  2. Multi-granular MAC&tree: 74.1% of 32KB chunks → 32KB granularity. Each chunk has 1 coarse counter (instead of 512) and 1 coarse MAC (instead of 512). Tree height: 0 levels (root is the counter) instead of 3.
  3. MAC traffic reduction: For a 32KB chunk, 512 MACs × 8B = 4KB → 1 MAC × 8B = 8B. Saved: 4088B per 32KB accessed. Over the workload, MAC traffic drops ~99.8%.
  4. Counter cache misses: 512 leaf counters → 1 root counter. Misses drop ~99.8%. Tree traversal: 0 hops instead of 3 per access.
  5. Result: Execution time improves from 1.21× (conventional) to 1.05× (multi-granular), a 15.6% speedup. Combined with BMF subtrees: 1.05× → 0.95× (another 10% gain).
🍯
Why alex benefits most: It's the coarsest workload in the paper's suite (74.1% 32KB). The multi-granular tree perfectly matches its access pattern, cutting metadata to near-zero for those chunks. Compare to "sfrnn" (RNN, 50% 32KB) or "ncf" (recommendation, 30% 32KB) — they still benefit, but less dramatically.

Problem 1 · easy
A 32KB chunk has streampart = 0x0000_0000_0000_0003 (bits 0 and 1 set). What are the granularities of partitions 0, 1, and 2?
Show answer
Partition 0: 512B (bit 0 set). Partition 1: 512B (bit 1 set). Partition 2: 64B (bit 2 unset — default fine-grained). Each bit represents one 512B partition; set = coarse, unset = fine.Misconception: that bit position encodes granularity level (e.g., bit 0 = 64B, bit 1 = 512B). No — the bit just marks stream (1) or not (0); the granularity is always the second-finest (512B) unless the entire chunk is stream (then it's 4KB or 32KB).
Problem 2 · medium
An 8-arity tree has a 512B-granular region (8×64B blocks). How many tree levels are saved compared to 64B granularity, and why?
Show answer
One level saved. 64B granularity: 8 leaf counters (level 3), 1 parent (level 2), up to root (level 0). 512B granularity: the parent counter becomes the leaf counter (level 2), directly under the grandparent (level 1), up to root (level 0). Tree height: 2 levels instead of 3. Formula: Parents = log₈(512/64) = 1.Misconception: that the tree is "deleted" entirely for coarse regions. No — the tree still exists, it's just shorter because child nodes are pruned and the parent is promoted.
Problem 3 · medium
Why does lazy switching store both current and next granularity in the granularity table, instead of just next?
Show answer
To handle accesses that arrive before the switch is complete. If the table only stored next = 512B but the data is still encrypted with the old 64B counters, the address-compute engine would compute the wrong MAC/counter addresses, leading to a mismatch. Storing current allows the engine to check "does this access match current or next?" and handle both gracefully. Once the switch is confirmed (data re-encrypted), current = next.Misconception: that granularity changes are atomic. They're not — switching takes multiple accesses (fetch, decrypt, re-encrypt, update tree), and during that window both old and new granularity coexist.
Problem 4 · hard
A region is promoted 64B → 512B. The paper says "granularity scale-up for read-after-read (RAR) requires fetching parent-to-root" but "write-after-read (WAR) requires zero extra overhead." Why the difference?
Show answer
WAR = a read happened, then a write to the same region. The write must update the counter, which means fetching all tree nodes from leaf to root anyway (to verify and update the hash path). So the promotion's "fetch parent-to-root" cost is subsumed by the write's normal cost. RAR = a read, then another read. The first read didn't fetch to the root (reads only verify, don't update), so the promotion must explicitly fetch parent-to-root to compute the new parent counter (MAX(children)+1). That's extra work the normal read didn't do.Misconception: that reads never touch the tree. Reads do verify the tree path, but only up to the root if there's a miss. Writes always update the path to the root (to change the parent hash), so they pay the full tree-traversal cost regardless.
Problem 5 · medium
The paper uses an 8-arity tree and proposes granularities 64B, 512B (8×64B), 4KB (8×512B), 32KB (8×4KB). Could you use a 4-arity tree? What granularities would you choose?
Show answer
Yes. For a 4-arity tree, each parent has 4 children. Granularities should be powers of 4: 64B (base), 256B (4×64B), 1KB (4×256B), 4KB (4×1KB), 16KB (4×4KB), etc. The partition unit (second-finest) would be 256B, so streampart would have 32KB / 256B = 128 bits per chunk (twice as many as the 8-arity design). The address-compute engine would use log₄ instead of log₈.Misconception: that the granularities are fixed to the paper's values. They're not — they depend on the tree's arity. The key constraint is that each coarse granularity = arity × fine granularity, so tree pruning removes exactly one level.
Problem 6 · medium
The paper says the granularity table is "too big for on-chip" (2MB for 4GB memory). Why not use a smaller table by tracking only non-default granularities in a hash table?
Show answer
That's called a "sparse" encoding, and it's possible — but it adds complexity. A hash table needs collision handling, eviction policy, and a fallback for misses (assume default 64B). The dense table is simpler: one entry per 32KB chunk, indexed by address bits, no collisions. The 2MB cost is acceptable because: (1) it's 0.5% of 4GB memory, (2) it benefits from counter-cache locality (16B entry per 32KB data = high hit rate), (3) it's protected by a separate fixed-64B tree (amortized overhead is low). A hash table might save space but adds latency and design complexity.Misconception: that 2MB off-chip is "unacceptable." It's not — the paper measures 0.3% extra overhead vs data access, meaning the table is accessed efficiently (cached). Sparse encoding would save space but hurt latency.
Problem 7 · hard
The paper combines multi-granular MAC&tree with BMF (subtree caching). Why do these two techniques compound instead of interfering?
Show answer
They're orthogonal optimizations targeting different bottlenecks. Multi-granular counters reduce the number of leaf nodes (fewer counters = fewer tree entries = less metadata traffic + cache pressure). BMF reduces the height of the tree for hot regions (cache subtree roots = shorter traversal paths). Together: (1) multi-granular prunes nodes → fewer total entries, (2) BMF caches roots of the pruned tree → even shorter paths for hot regions. They don't interfere because BMF doesn't care what the leaves are (fine or coarse) — it just caches the root of whatever tree structure exists. Result: fewer nodes and shorter paths = compounding benefit (14.2% → 21.1%).Misconception: that caching roots of a multi-granular tree is "invalid" because the tree structure changes dynamically. It's not — BMF's caching is based on hotness, and the cached root is invalidated/updated when the tree structure changes (granularity promotion/demotion). The paper's lazy switching ensures most accesses hit the stable granularity, so cache invalidations are rare.
Problem 8 · easy
A CPU workload shows 95% fine-grained (64B) accesses, 5% coarse (512B). Does multi-granular MAC&tree help?
Show answer
Minimally. The 5% coarse accesses benefit (fewer MACs, shorter tree for those regions), but 95% stay 64B-granular and pay no penalty (the mechanism detects fine-grained and leaves it as-is). The gain is ~5% × (MAC_reduction + tree_reduction) ≈ 2–3% execution-time improvement. Compare to a coarse-grained NPU workload (74% coarse) where the gain is 15–20%. The mechanism adapts to the pattern: if most accesses are fine, it stays fine; if most are coarse, it promotes. No harm, small gain.Misconception: that multi-granular MAC&tree penalizes fine-grained workloads. It doesn't — fine-grained regions are detected as such and remain 64B-granular. The only overhead is the access tracker (842B on-chip) and granularity-table reads (0.3% extra traffic), which is negligible.
Quick answers

Frequently asked questions

Ten questions newcomers ask most. Tap any to expand.

Q1Why "unified" — what's wrong with per-device protection?▾
Per-device protection means separate security engines for CPU, GPU, NPU — each with its own metadata caches, integrity trees, and logic. Problems: (1) under-utilization (each engine sits idle when its device isn't active), (2) complexity (3× the design/verification effort), (3) inflexibility (adding a new accelerator means a new engine). A unified engine shares hardware across all devices, adapting granularity dynamically to each device's workload. The paper shows per-device static granularity mispredicts up to 50% of accesses; dynamic per-partition granularity fixes this.
Q2How does the access tracker know when to evict an entry?▾
Three triggers: (1) the entry's lifetime exceeds 16K cycles (timeout), (2) the number of chunk accesses exceeds 512, or (3) a new access needs an entry but all 12 entries are full (LRU eviction). On eviction, the granularity-detection engine scans the 512-bit vector and writes the streampart bitmap to the granularity table.
Q3What happens if an attacker tampers with the granularity table?▾
The granularity table lives in a secure memory region protected by a separate fixed-64B integrity tree (the table itself is encrypted and MAC'd). If an attacker flips a bit in streampart, the MAC verification on the next read of that table entry will fail, and the system aborts. The attacker can't forge a valid MAC without the secret key. The granularity table is part of the trusted computing base (TCB) — it's secured by the same counter-mode + integrity-tree mechanism as user data.
Q4Why 12 access-tracker entries — why not more or fewer?▾
The paper sizes the tracker to match the on-chip memory requirement of a prior study [56] (96 entries covering 4KB each = 384KB tracked). With 32KB chunks, that's 384KB / 32KB = 12 entries. Each entry is 561 bits (512 + 49 for the chunk index), so 12 entries = 842B on-chip. More entries would track more chunks simultaneously (better for diverse workloads with many hot regions), but at higher on-chip cost. Fewer entries would save space but cause more LRU evictions (premature detection).
Q5Can you mix granularities within a single 512B partition?▾
No. The granularity table stores one bit per 512B partition: set = coarse (512B or higher), unset = fine (64B). Within a partition, all 8 blocks share the same granularity. If you want finer control (e.g., blocks 0–3 coarse, 4–7 fine), you'd need a denser encoding (2 bits per 256B sub-partition), doubling the table size. The paper chose 512B as the partition unit to balance table size (2MB for 4GB) vs granularity flexibility. Most workloads exhibit locality at 512B or larger, so within-partition mixing isn't needed.
Q6How does lazy switching know which granularity to use on the next access?▾
The address-compute engine reads both current and next from the granularity table. (Note: The paper describes storing current and next granularity and handling mismatches on-demand, but does not specify the exact decision logic for inferring which granularity an access implies. The following is an inference based on the mechanism's behavior.) The engine likely checks: does the access pattern (e.g., which blocks are requested together) suggest the requester expects next granularity? If yes, use next (the switch is in progress or complete). If no, use current (the switch hasn't started yet for this access). If using next and the data is still encrypted with current counters, trigger the switching steps (fetch, re-encrypt, update tree). Once done, mark current = next.
Q7Why does the paper use nested hashing for coarse MACs instead of a single hash of the entire chunk?▾
The paper states the nested MAC formula is roughly MACcoarse = Hash(Hash(Hash(MACfine1), MACfine2), ...) and explains this is because the existing protection engine is 64B-cacheline based. Data is encrypted/decrypted in 64B units. To verify a coarse-grained MAC, you: (1) decrypt all 8×64B blocks using their shared counter, (2) compute the fine-grained MAC for each block (Hash(block)), (3) compute the coarse MAC by hashing the fine MACs together. This design maintains compatibility with 64B-granular encryption engines and avoids the need to re-architect the baseline encryption mechanism.
Q8Does multi-granular MAC&tree work with other tree designs (e.g., Bonsai)?▾
Yes — it's orthogonal. The paper's baseline is an 8-arity counter tree (standard Intel SGX-like design), but the multi-granular counter promotion works with any arity: just choose granularities as powers of the arity (4-arity → 64B, 256B, 1KB; 16-arity → 64B, 1KB, 16KB). The paper explicitly combines with Bonsai Merkle Forests (BMF) — BMF caches subtree roots, multi-granular counters prune child nodes. They compound: the paper reports that BMF&Unused+Ours improves by 7.4% compared to standalone BMF&Unused (Table 3), contributing to the 21.1% total improvement.
Q9What about memory allocation — does granularity change when malloc/free happen?▾
(Note: The paper does not discuss malloc/free behavior, TEE zeroing of freed pages, or granularity-table reinitialization on first post-reallocation access. The following is an inference.) Granularity is purely runtime-access-pattern-based, not tied to allocation. A 32KB malloc'd buffer might be accessed fine-grained (64B) if the program reads it sparsely, or coarse-grained (32KB) if it's written in bulk. The access tracker detects the pattern after allocation, as accesses occur. Free'ing a region doesn't trigger granularity updates — the granularity table entry would be overwritten when new data is allocated to that address (standard TEE memory management likely zeros freed pages, and the granularity table would be re-initialized by the first access post-reallocation).
Q10Could this work for CXL-attached memory or tiered DRAM/NVM?▾
Yes, conceptually. CXL memory is off-chip and untrusted (same threat model as DRAM), so it needs the same counter+MAC+tree protection. The access tracker would monitor CXL accesses just like local DRAM accesses. The granularity table would cover the CXL address space. The paper doesn't simulate CXL, but the mechanism is device-independent — it only cares about access patterns, not the physical memory technology. For tiered DRAM/NVM (your research area), coarse-granular protection might benefit NVM (which has higher latency) by reducing metadata round-trips. The multi-granular tree could adapt: DRAM regions stay fine-grained (low latency), NVM regions go coarse (amortize the high-latency tree traversal over larger chunks).

Key papers to read next

Common Counters (GPU) · Na et al. · HPCA '21

dual-granular counters (64B/32KB) · 16 shared counters

Detects uniform counter values within 32KB segments via scanning. Multi-granular tree improves on this by: (1) supporting 4 granularities (not 2), (2) pruning tree nodes (Common Counters doesn't modify the tree), (3) optimizing MACs too, (4) no scanning step (dynamic per-partition detection).

Adaptive (GPU) · Yuan et al. · HPCA '22

dual-granular MACs (64B/4KB) · access tracker

Uses an access tracker similar to this paper's, but only for MACs (counters stay 64B). Multi-granular tree extends to 4 granularities and counters, plus the tree modification.

MGX (NPU) · Hua et al. · ISCA '22

tensor-granular counters + MACs · software-detected

Leverages compiler-known tensor boundaries for coarse granularity. Tree-less (on-chip version numbers). Not general — requires ML workloads. Multi-granular tree is hardware-detected, works for any workload.

Bonsai Merkle Forests (BMF) · Freij et al. · MICRO '21

subtree root caching for hot regions

Prunes the tree by caching roots of subtrees. Orthogonal to multi-granular counters — the paper combines both for 21.1% total improvement.

PENGLAI · Feng et al. · OSDI '21

subtree pruning for unused memory

Prunes tree nodes for memory regions not yet allocated. Also orthogonal — multi-granular tree can be combined with PENGLAI's pruning.

Intel SGX Explained · Costan & Devadas · 2016

foundational TEE + counter-mode encryption

The baseline memory-protection architecture (counter tree, MACs). This paper optimizes it for heterogeneous SoCs.
