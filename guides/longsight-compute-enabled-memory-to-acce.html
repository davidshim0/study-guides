Sparse attention meets cheap DRAM — LongSight's million-token window

As LLM context windows reach hundreds of thousands of tokens, GPU memory runs out and attention slows to a crawl. LongSight solves both: a compute-enabled CXL memory device stores the KV cache and accelerates sparse attention, turning low-cost LPDDR into the functional equivalent of scarce HBM.

📄 "LongSight" · Quinn et al., MICRO '25 🎓 Assumes basic transformer & memory tiering knowledge ⏱ ~30 min read
🔍

Attention in transformers is dominated by a small subset of past tokens with high dot-product similarity — so offload the KV cache to CXL memory, filter with sign-bit PIM, retrieve top-k, and let the GPU finish. Result: 1M tokens on a single GPU + one DReX unit.

Before you begin

Six concepts LongSight builds on

If you're already working on LLM inference or CXL memory tiering, you know most of these — tap any to review.

GPU (local HBM) DReX (CXL memory + PIM) Dense attention (baseline) Hybrid sparse (LongSight)
1Transformer attention and the KV cache▾

Each decoder layer computes Query, Key, Value tensors. Attention scores come from Q·Kᵀ (dot-product), passed through softmax, then weighted-sum the Values. In autoregressive generation, we cache past Keys and Values (the KV cache) to avoid recomputing them for every new token.

Cost: for each new token, compute its query's dot-product with all past keys — so a 100k-token context means 100k dot-products per head. At large contexts, attention dominates runtime and memory use.

Now you're ready: LongSight offloads the bulk of that KV cache to a separate CXL memory device and computes only the top-k most relevant keys there, not all 100k.
2Why GPU HBM runs out at long context▾

A single H100 has 80 GB HBM. Llama-3-8B weights take ~16 GB; the rest is for activations and the KV cache. For 32 layers × 8 KV heads × BF16 (2 bytes/dim) × 128 dims × 2 (K + V), each token needs 32 × 8 × 2 × 128 × 2 = 131 kB. At 128k tokens that's ~16 GB just for KV; at 1M tokens it's ~128 GB — far beyond one GPU.

Existing fix: use 2 GPUs, or compress the cache with quantization. LongSight instead uses a CXL memory expander with 512 GB of low-cost LPDDR to hold the KV cache.

Now you're ready: LongSight treats GPU HBM as a small working set (sliding window) and DReX as a large, accelerated KV store.
3CXL memory expanders and byte-addressability▾

CXL Type-3 devices attach DDR-based memory to the host over PCIe, making it byte-addressable — the GPU can load/store into it with regular instructions (no separate DMA API). LongSight leverages NVIDIA's SCADA to let GPU threads read/write DReX memory directly.

CXL bandwidth is lower than HBM (PCIe 5.0 × 16 ≈ 64 GB/s vs. HBM's 3+ TB/s), so naive access would bottleneck. LongSight's compute-enabled DReX does filtering and scoring near the memory, shipping back only top-k results — not the full KV cache.

Now you're ready: LongSight's GPU writes Keys/Values to DReX once during prefill, then sends query vectors at decode time for in-/near-memory retrieval.
4Sign-concordance filtering (SCF)▾

To find relevant keys quickly, LongSight uses one-bit quantization: for each dimension of a Key or Query vector, store only its sign bit. A key passes the filter if it shares ≥ TH sign-matches with the query (XOR the sign bits, count zeros). This is cheap to compute in-DRAM: PIM units process 128 bits (128 keys × 1 sign bit per dimension) in parallel.

Why it works: vectors with high dot-product similarity tend to have matching signs in most dimensions. The threshold is tuned so filtering ratio ≈ 20× (only 5% of keys survive to full scoring) while keeping perplexity within 5% of dense attention.

Now you're ready: PIM filtering is stage 1 of LongSight retrieval; it prunes distant keys without fetching full vectors from DRAM.
5Processing-in-memory (PIM) and near-memory accelerators▾

DReX integrates PIM filtering units inside each LPDDR bank (lightweight logic on the DRAM die) and near-memory accelerators (NMAs) in separate chips next to the LPDDR packages. PIM does sign-bit filtering; NMA does full-precision dot-product scoring and top-k selection.

PIM filters 128 keys per cycle using just XOR + accumulate (low area overhead: 6.7% of DRAM die). NMA has SRAM scratchpads and FP16 MACs, consuming ~1 W per chip. Eight NMAs cover all eight LPDDR packages in DReX, enabling 8-way head-level parallelism.

Now you're ready: LongSight's three-stage pipeline (PIM filter → NMA score → NMA top-k) happens entirely in DReX; the GPU only sees the final top-k keys and values.
6Hybrid dense–sparse attention▾

LongSight keeps a sliding window of the W most recent KV pairs in GPU HBM (default W = 1,024) and performs dense attention over them. Older keys are in DReX and retrieved via sparse attention (filtering + top-k). The GPU combines both: softmax over (window + top-k), then weighted-sum of the corresponding values.

Why hybrid? (1) Recent tokens are often the most relevant, so dense local attention improves accuracy. (2) The window batches KV updates to DReX, removing them from the per-token critical path. (3) GPU can overlap window attention with DReX offload.

Now you're ready: LongSight's algorithm is dense short-range + sparse long-range; the system is GPU (window) + DReX (history).

LongSight in a nutshell

The problem, the algorithmic fix, and the system integration that makes it fast.

Large-context LLMs need to attend to hundreds of thousands of past tokens. With standard dense attention, that means loading all past Keys from memory to compute Q·Kᵀ for every new token — which quickly exceeds GPU HBM capacity and bandwidth. For a 1M-token context in Llama-3-8B, the KV cache alone is ~128 GB; a single H100 has 80 GB HBM total.

Prior work tries sparse attention in software (e.g., sliding windows, LSH, static masks), but either sacrifices accuracy or requires task-specific fine-tuning. Alternatively, PIM approaches (NeuPIMs, AttAcc) accelerate dense attention, but still pay the cost of loading all keys. DynaX quantizes and prunes at block granularity, but must still load ~9.4% of the KV cache even with 4-bit quantization.

LongSight co-designs algorithm and hardware: it recognizes that attention in transformers is dominated by a small subset of past tokens whose keys have high dot-product similarity with the current query. So it offloads the KV cache to a CXL memory expander (DReX) and implements a three-stage retrieval pipeline:

  1. PIM filtering. In-DRAM PIM units evaluate sign-concordance (matching sign bits) between the quantized query and all keys in parallel. Keys below the threshold are pruned; only ~5% survive (20× filter ratio).
  2. NMA scoring. Near-memory accelerators fetch surviving keys from DRAM across all channels (balanced load), compute full-precision Q·K dot-products, and rank them.
  3. Top-k selection. NMAs select the top k = 1,024 keys (plus their corresponding values) and ship them back to the GPU over CXL.

Meanwhile, the GPU performs dense attention over a sliding window (W = 1,024 recent tokens in HBM), overlapping with DReX offload. Once DReX returns top-k, the GPU combines them: softmax over (window + top-k), then weighted-sum of values.

The payoff: LongSight supports up to 1 million tokens on a single GPU + one DReX unit — impossible with GPU-only systems. At the maximum context length a single GPU can handle (128k for Llama-3-8B), LongSight achieves 8.1–9.6× higher throughput and 3.6–11.9× lower per-token latency. Perplexity stays within 5% of dense attention; the entire DReX unit (512 GB LPDDR + PIM + 8 NMAs) consumes ~158 W peak and occupies under 121 mm² logic area (8 × 15.1 mm² NMAs).

🍯 Key takeaways

  • Attention at long context is bottlenecked by memory capacity (KV cache exceeds HBM) and bandwidth (loading all keys is slow).
  • LongSight offloads KV cache to a CXL memory expander (DReX) with 512 GB LPDDR — 6.4× the capacity of one H100.
  • Sign-bit PIM filtering prunes ~95% of keys in-DRAM; near-memory accelerators score survivors and return top-k over CXL.
  • Hybrid dense (GPU, recent window) + sparse (DReX, history) attention maintains accuracy and hides CXL latency.
  • Iterative Quantization (ITQ) rotates key/query vectors to balance sign-bit distributions, boosting filter ratio 6.4–46×.
  • Result: 1M-token context on 1 GPU + 1 DReX; up to 9.6× faster than 1-GPU dense at max single-GPU context (128k).

Five everyday pictures

How to think about LongSight

Each analogy stands alone. Every card ends with where the picture breaks.

The core idea · Sparse retrieval

Library search without reading every book

You enter a million-book library with a question. Dense attention = skim every book's title to rank relevance. LongSight = use the card catalog's index (sign-bit filtering) to prune 95% instantly, then read only the top 1,000 titles in detail. Same result, 20× less work. The "card catalog" is PIM; "reading titles" is NMA scoring.

Where it breaks: real card catalogs use manually curated keywords; sign-concordance is an automatic, learned approximation that can miss edge cases (hence the 5% perplexity gap).
Memory capacity · Tiering

Desk (HBM) vs. filing cabinet (DReX)

Your desk holds 80 sheets of paper (GPU HBM, fast but small). The filing cabinet holds 512 sheets (DReX, slower access but huge). You keep the current page + recent notes on the desk (dense window), and file older notes in the cabinet. When you need an old fact, you don't pull every drawer — you use the index sticker (sign-bits) to grab just the relevant folder (top-k).

Where it breaks: filing cabinets are static; DReX is constantly updated during generation (every new token adds a KV pair). The batching window is what makes updates off-critical-path.
Sign-concordance filtering · One-bit quantization

Matching compass bearings

Each dimension of a vector is like a compass direction: positive (+) or negative (−). Two vectors with 4 out of 5 matching directions are probably aligned; if they only match 2 out of 5, they're probably opposite. Sign-concordance counts matches: set threshold TH = 4; keys with ≥ 4 matches pass, others are pruned. It's a coarse but fast test, done entirely with XOR operations in-DRAM.

Where it breaks: real vectors have magnitudes, not just signs; two vectors with matching signs but tiny values aren't truly aligned. That's why LongSight follows up with full-precision scoring on survivors.
Hybrid attention · Dense + sparse

Fresh news vs. archive search

You track today's headlines closely (dense, local window in GPU HBM) but only occasionally search last year's archives (sparse, DReX). When you need old context, you don't re-read every past article — you keyword-search (sign-filter) and pull the top matches. Combining both gives you recency (today) plus historical depth (archived top-k).

Where it breaks: news archives are immutable; LongSight's "archive" (DReX) grows every token. The window acts as a staging buffer so updates don't block generation.
Iterative Quantization · Balancing sign bits

Rotating a photo to level the horizon

If all your keys cluster in the positive quadrant, their sign-bits are unbalanced (all +), making filtering ineffective. ITQ learns a rotation matrix that "levels" the distribution — spreading keys evenly across + and − in each dimension, so sign-matching becomes more discriminative. Takes <1 minute to train, no task data needed, and runs once per head.

Where it breaks: a photo rotation is 2D; ITQ is a high-dimensional linear transform (128×128 for Llama-3). Also, ITQ can't be fused into the projection layer because positional embeddings break distance invariance — it's applied at inference time after pos-embed, adding ~3% overhead (negligible vs. the 6.4–46× filter gain).
The full explanation

How it actually works

Algorithm, data layout, and hardware operation, with the paper's key measurements.

1 · The problem: attention cost grows with context

In transformer decode, each new token's query Q computes dot-product with all past keys K in the cache. For a context of L tokens, that's L dot-products per head. At L = 100k, Llama-3-8B (32 layers × 32 Q heads × 8 KV heads) does 32 × 32 × 100k ≈ 102M dot-products per token. Each dot-product is 128 FP16 MACs (head dimension), so ~13 GFLOPs per token just for Q·K.

Worse: loading the KV cache from memory dominates. Each key is 128 × 2 bytes = 256 B; 100k keys = 25.6 MB per head. With 8 KV heads across 32 layers, that's 32 × 8 × 25.6 MB = 6.6 GB loaded per token. At 100 tokens/sec (modest throughput), that's 660 GB/s — 20% of H100 HBM bandwidth, crowding out other ops.

🌱
Concretely: Llama-3-8B at 128k context fills ~16 GB HBM with KV cache alone (32 layers × 8 heads × 128k tokens × 2 (K+V) × 128 dims × 2 bytes BF16 = 16 GB). Add 16 GB weights, activations → GPU is full. At 1M tokens, KV cache is ~128 GB — you need two H100s.

2 · Algorithmic foundation: sign-concordance filtering (SCF)

LongSight leverages the observation that attention scores correlate with dot-product similarity, and dot-product similarity correlates with sign-bit concordance. For two vectors Q and K of dimension D, define:

SCF(Q, K, TH) = (count of matching sign bits) ≥ TH
                  = Σᵢ (SQ[i] ⊕ SK[i]) ≤ (D − TH)   # XOR=0 if match

For Llama-3 (D = 128), setting TH ≈ 80–90 (tuned per KV head) gives a 20× filter ratio (only 5% of keys survive) while keeping perplexity within 5% of dense. The key is that sign-bits are free: just test the sign bit of each FP16/BF16 value, no full precision needed.

Critically, SCF operates at per-token granularity (not blocks), so it's exact on the sparse set — unlike DynaX's block-level pruning, which forces coarser sparsity. And it's data-dependent (threshold varies per head), unlike static masks in Longformer.

💡
Why this works better than LSH: LSH (Reformer) has per-token overhead with linear time complexity and requires multiple rounds + recomputation. SCF is embarrassingly parallel (one XOR + accumulate per key, all in parallel across DRAM banks) with zero recomputation. Plus SCF doesn't assume Q = K, so it supports grouped-query attention (8 KV heads, 32 Q heads) used in Llama-3.

3 · Iterative Quantization (ITQ): making sign-bits informative

Raw sign-bits from Llama-3 keys are poorly distributed — they cluster (e.g., 80% positive in some dimensions), reducing SCF's discriminative power. Iterative Quantization learns a rotation matrix R [D×D] that minimizes quantization error when mapping vectors to sign-bits. Training: sample 1k tokens, compute Keys and Queries, learn R via procrustes optimization (takes <1 minute).

At inference: after QKV projection and positional embedding, multiply Q and K by R. This costs D² MACs per token, but D = 128 → 16k MACs, which is <3% of the cost of computing Q (which is D × model_dim matrix-vector multiply). The gain: filter ratio improves 6.4× for Llama-3-1B, 46× for Llama-3-8B (Fig. 3c in the paper).

# pseudocode for ITQ-enhanced query
Q_proj = linear(x, W_q)           # standard projection
Q_pos  = Q_proj + pos_embed(t)    # add positional embedding
Q_rot  = matmul(Q_pos, R_itq)       # ITQ rotation (D×D)
SQ     = sign(Q_rot)              # extract sign bits for SCF
🍯
Key insight: Without ITQ, baseline SCF filter ratio is ~3× (Fig. 3a). With hybrid attention (dense window), it's ~7× (Fig. 3b). Add ITQ: up to 1,564× for Llama-3-8B at 128k context (Fig. 3c). ITQ is what makes sign-bit filtering competitive with ANNS while staying cheap.

4 · Hybrid dense–sparse attention

LongSight keeps the most recent W = 1,024 KV pairs in GPU HBM and performs full dense attention over them. Older keys (beyond the window) are stored in DReX and accessed via sparse retrieval (SCF filter → full-precision scoring → top-k). The GPU combines both:

  1. Dense phase (GPU, overlapped): Compute Q·K for window (1,024 keys), get attention scores S_dense.
  2. Sparse phase (DReX offload): DReX filters, scores, returns top-k = 1,024 keys and their scores S_sparse.
  3. Combine: Softmax over (S_dense ∪ S_sparse), then weighted-sum of (V_dense ∪ V_sparse).

Why 1,024? (1) Recent tokens are often most relevant (Fig. 4 shows sliding window alone can hit 95–98% accuracy on some configs). (2) Batching KV updates: when the window fills, ship 1,024 KV pairs to DReX at once (off critical path) rather than one per token. (3) Overlap: while DReX filters, the GPU computes the dense window, hiding CXL latency.

🌱
Trade-off: Larger windows → higher accuracy (more dense coverage) but less room for batch in HBM and less filtering gain (DReX handles fewer keys). The paper finds W = 1,024, k = 1,024 is a sweet spot (Fig. 4).

5 · DReX architecture: PIM + NMA

DReX integrates PIM Filtering Units (PFUs) in each of 8,192 LPDDR5X banks (8 packages × 8 channels × 128 banks) and Near-Memory Accelerators (NMAs) in 8 chips (one per package). Each PFU processes 128 bits/cycle; each NMA has FP16 MACs, SRAM scratchpads, and a top-k sorter.

Pipeline for one attention request (single user, layer, head):

  1. PIM filtering (in-DRAM): GPU writes sign-quantized keys (SK) to DRAM such that each bank holds 128 keys. On offload, the memory controller broadcasts the query sign-bits (SQ) to all PFUs in parallel. Each PFU XORs SQ with its 128 SK vectors (dimension-wise), accumulates mismatches, and outputs a 128-bit bitmap (1 = pass, 0 = fail). This happens in parallel across all banks covering the context slice. Time: D × 1.25 ns (for D = 128, that's 160 ns per epoch). Bitmap is read into NMA in 120.4 ns.
  2. NMA scoring (near-DRAM): NMA fetches surviving keys from DRAM (interleaved across all 8 channels to balance bandwidth), computes full-precision Q·K dot-products. Each dot-product is 128 FP16 MACs; NMA has enough compute to sustain ~1 TFLOP/s. Keys are pre-striped so channel utilization is balanced.
  3. Top-k selection (NMA): NMA maintains a min-heap of size k = 1,024. As scores arrive, it updates the heap. Once all survivors are scored, the heap contains the top-k. NMA packages these keys + their values + scores into a response descriptor and writes it to a GPU-visible response buffer via CXL.
⚠
Capacity limit: Each LPDDR package can hold 64 GB; 8 packages = 512 GB total. A "Context Slice" (one layer, one head, one user) can hold up to 128 banks × 1,024 keys/bank = 131k keys. For Llama-3-8B (32 layers, 8 heads), a full 1M-token context requires 32 × 8 = 256 context slices. With 8 packages, that's 256 / 8 = 32 users max at 1M tokens, or more users at shorter contexts.

6 · Data layout and address mapping

Keys are laid out in Key Blocks of 128 keys per bank. Sign-bits are stored column-major: each 128-bit DRAM column holds one sign-bit dimension across all 128 keys. Full-precision keys are interleaved across all 8 channels within a package (so fetching them saturates bandwidth). A Context Slice = multiple Key Blocks across banks, storing one head × one layer × one user. A Multi-Layer Context Slice concatenates slices for the same head across layers. A User Partition = 8 Multi-Layer slices (one per head), each in a different package, enabling head-level parallelism.

Address translation: DReX uses simple physical mapping (columns → rows → banks → channels → packages). The GPU writes to DReX via CXL load/store; the DReX CXL Controller (DCC) translates addresses and routes them. To ensure balanced channel utilization during NMA reads, keys are placed at strided addresses (skip columns to hit all channels).

7 · End-to-end flow: prefill and decode

Prefill stage: GPU accumulates KV tensors in HBM. Once the window (1,024 tokens) fills, GPU prepares Key Sign Objects (sign-bits), Key Objects (full-precision), and Value Objects in groups of 128, and writes them to DReX via CXL load/store. This happens off the critical path (separate kernel). If context > 1,024, the GPU continues filling DReX until the entire input is cached.

Decode stage: For each token, at each layer, the GPU constructs a Request Descriptor (User ID, layer, Q vectors) and writes it to DReX's MMIO Request Queue. In parallel, the GPU performs dense attention over the local window (1,024 recent KV pairs in HBM). Meanwhile, DReX processes the request (PIM filter → NMA score → top-k), writes results to a Response Buffer, and sets a bit in the Polling Register. GPU polls the register; when the bit is set, it reads the top-k keys/values, combines them with the window results, applies softmax and weighted-sum, and proceeds to the next layer or token.

🍯
Latency budget: At 100 tokens/sec with 32 layers, each layer has ~312 µs. LongSight's DReX offload (PIM + NMA + CXL round-trip) is ~150–200 µs (dominated by NMA scoring + CXL transfer of top-k values), overlapped with ~100 µs of GPU dense attention. Net latency ≈ max(DReX offload, GPU dense) + softmax (~20 µs). Stays within SLO.
Follow the numbers, step by step

Worked examples

Four traces using the paper's real measurements.

① Computing KV cache size for Llama-3-8B at 128k context

How much memory does the KV cache occupy? Use the paper's model: 32 layers, 8 KV heads, BF16 (2 bytes), head dimension 128.

  1. Per token: each token has a Key and a Value, each of dimension 128. Per head: 128 × 2 bytes (K) + 128 × 2 bytes (V) = 512 bytes. With 8 KV heads: 8 × 512 = 4,096 bytes/token/layer.
  2. Per layer: 128k tokens × 4,096 bytes/token = 524 MB.
  3. All layers: 32 layers × 524 MB = 16.8 GB.
  4. Total GPU usage: Add 16 GB weights (8B params × 2 bytes) + activations (~4 GB) → ~37 GB of 80 GB HBM. The GPU can handle 128k, but not much more.
  5. At 1M tokens: scale linearly: 1M / 128k = 7.8×; KV cache = 16.8 GB × 7.8 ≈ 131 GB — exceeds one GPU, needs two or offload to DReX.
🍯
Insight: KV cache grows linearly with context length and number of layers. At large context, it dominates memory usage. DReX's 512 GB can hold ~3.8× more context than one H100 (512 / 131 ≈ 3.9M tokens for Llama-3-8B if no batching).

② Sign-concordance filtering: filter ratio and threshold

For Llama-3-8B with ITQ, the paper reports a 1,564× filter ratio at 128k context on Wiki2 (Fig. 3c). What does that mean, and how is the threshold set?

  1. Filter ratio: 1,564× means only 1 / 1,564 ≈ 0.064% of keys survive SCF. For 128k keys, survivors ≈ 128k / 1,564 ≈ 82 keys (out of 128k!) before top-k.
  2. Threshold TH: for D = 128 dimensions, TH ≈ 80–90 (tuned per KV head). If a key matches ≥ 80 sign bits with the query, it passes; else pruned.
  3. Why so few survive? ITQ rotates vectors to spread sign-bits evenly. After rotation, distant keys (low dot-product) have ~50% sign-match by chance; only truly similar keys exceed 80/128 = 62.5% threshold.
  4. Top-k step: After SCF, the 82 survivors are scored with full-precision Q·K. Of those, top k = 1,024 are selected. But wait — 82 < 1,024. In practice, with k = 1,024 and such high filtering, all survivors are kept (no further pruning), and the sparse set has < k entries (filled with zeros or masked in softmax).
  5. Accuracy check: Perplexity is within 5% of dense (Table in supplement), confirming the filter isn't too aggressive.
💡
The paper tunes thresholds iteratively: start with TH = 0 (no filtering), increase until perplexity exceeds 5%, then back off. This is done per KV head because each head has different score distributions. At high context, the threshold can be very strict (high TH) because there are so many keys that even a tiny fraction yields enough for top-k.

③ DReX offload latency breakdown (single user, 128k context)

From Fig. 8 (latency breakdown), estimate the time for one DReX offload at 128k context, Llama-3-8B. Assume one user, one head, one layer.

  1. PIM filtering: 128k keys across 128 banks (each bank holds 1,024 keys in 8 epochs of 128). Per epoch: 128 dims × 1.25 ns = 160 ns (sign XOR). Bitmap read: 120.4 ns. Total per epoch ≈ 280 ns. With 8 epochs: 8 × 280 ns ≈ 2.24 µs. Negligible (Fig. 8 shows "DP-Bound" dominates).
  2. NMA scoring: With 20× filter ratio, 128k / 20 = 6,400 keys survive. Each key is 128 FP16 MACs for Q·K. Total: 6,400 × 128 = 819k MACs. NMA has ~1 TFLOP/s → 819k / 1e12 ≈ 0.8 µs for compute. But memory bandwidth limits: 6,400 keys × 256 bytes/key = 1.64 MB; LPDDR5X per package ≈ 137 GB/s (8 channels × ~17 GB/s). Load time: 1.64 MB / 137 GB/s ≈ 12 µs. So scoring is memory-bound at ~12 µs.
  3. Top-k selection: Heap update for 6,400 keys, k = 1,024. Using a binary heap, insert is O(log k) per key; total ≈ 6,400 × log₂(1,024) ≈ 64k ops. At ~GHz clock, this is a few µs (overlapped with scoring in pipelined design).
  4. Value loading + CXL transfer: Top-k = 1,024. Each value is 256 bytes → 1,024 × 256 = 256 kB. CXL bandwidth ≈ 64 GB/s (PCIe 5.0 × 16). Transfer time: 256 kB / 64 GB/s ≈ 4 µs. But Fig. 8 shows "CXL-Bound" is much larger (~50% of latency at high user count) — this includes round-trip overhead, DCC queueing, and possible contention. Realistic: ~50–100 µs for value transfer when DReX is busy.
  5. Total (single user): PIM (~2 µs) + NMA scoring (~12 µs) + value CXL (~50 µs) ≈ 64 µs. At max users (496 for 128k, Fig. 7), DReX is saturated and value loading pipelined, so per-user latency rises to ~150 µs (Fig. 8 bottom shows "CXL+DP" ≈ 50% each, totaling ~150 µs).
🍯
Key observation: At low load (one user), DReX offload is fast (~64 µs). As users increase, value loading over CXL becomes the bottleneck because it's a fixed per-user cost that can't be batched. But GPU can overlap its dense window attention (~100 µs) with DReX, so end-to-end latency ≈ max(GPU, DReX) + softmax.

④ Throughput gain: LongSight vs. 1-GPU dense at 128k

Fig. 7 shows LongSight achieves ~9× higher throughput than 1-GPU dense for Llama-3-8B at 128k context with max users. Explain where the gain comes from.

  1. 1-GPU dense bottleneck: At 128k context, KV cache is ~16.8 GB. Loading it for every token: 16.8 GB × tokens/sec. H100 HBM bandwidth ≈ 3.35 TB/s. Max tokens/sec ≈ 3.35 TB/s / 16.8 GB ≈ 200 tokens/sec (if fully memory-bound, ignoring compute). With batching (62 users, Fig. 7), throughput ≈ 62 × 200 / (some overhead) — but in practice, Fig. 7 shows ~400 tokens/sec total (per-user latency ~155 ms → 1 / 0.155 ≈ 6.5 tokens/sec/user × 62 ≈ 400).
  2. LongSight sparse: With 20× filter, only 128k / 20 = 6,400 keys are scored. Loading: 6,400 × 256 bytes = 1.64 MB per head per layer. Plus top-k values: 1,024 × 256 bytes = 256 kB. Total per layer ≈ 1.9 MB × 8 heads = 15.2 MB (vs. 16.8 GB / 32 layers ≈ 525 MB per layer in dense). Reduction: 525 / 15.2 ≈ 34×. But CXL bandwidth is 64 GB/s (vs. 3.35 TB/s HBM), so 3,350 / 64 ≈ 52× slower per byte. Net: 34× less data but 52× slower link → slightly worse per-access, but DReX has 8 NMAs running in parallel (head-level parallelism), and the GPU's local window overlaps. Effective speedup ≈ 9× (empirical, Fig. 7).
  3. Why 9× not 34×? (a) CXL is slower than HBM (52× bandwidth gap). (b) Filtering and scoring overhead in DReX (though small). (c) Hybrid approach means some keys are still loaded from GPU HBM (the window). (d) Value loading over CXL is a fixed cost. Net result: LongSight trades HBM capacity (freed up) for CXL bandwidth, and the ~9× gain comes from the combination of sparsity + near-memory acceleration.
💡
Key trade-off: LongSight is "capacity-bound" (can fit huge contexts) rather than "bandwidth-bound" (can serve many users at moderate context). At very high user counts, CXL value transfer saturates, but at lower user counts (where GPU is the bottleneck), LongSight still wins by freeing HBM for more users or larger batches.

Practice problems

Problem 1 · easy
Why does LongSight use a sliding window of recent tokens in GPU HBM instead of offloading everything to DReX?
Show answer
Three reasons: (1) Accuracy — recent tokens are often the most relevant (temporal locality); dense attention over them improves perplexity (Fig. 4). (2) Latency hiding — the GPU can compute the window in parallel with DReX offload, overlapping work. (3) Update batching — the window acts as a staging buffer; when it fills (1,024 tokens), KV pairs are shipped to DReX in bulk, removing per-token write latency from the critical path. Without the window, every new token would require an immediate DReX write, adding ~10–20 µs overhead per token.Misconception: that offloading everything to DReX would be simpler or faster. The hybrid design is essential for both accuracy and performance.
Problem 2 · medium
LongSight uses Iterative Quantization (ITQ) to rotate key/query vectors. Why can't this rotation be fused into the linear projection layer (like in some quantization schemes)?
Show answer
Because positional embeddings are added after the linear projection. ITQ learns a rotation that balances sign-bit distributions in the final key/query space — which includes positional information. If you fused ITQ into the projection (W_q or W_k), the positional embedding would be added afterward, breaking the distance invariance that ITQ relies on. Therefore, ITQ must be applied at inference time, after positional embeddings: Q_rot = matmul(Q_proj + pos_embed, R_itq). This adds ~3% overhead (16k MACs for D=128), but the 6.4–46× filtering gain far outweighs it.Misconception: that ITQ is just a static linear transform that can be pre-applied. The interaction with positional embeddings forces it to be a per-token runtime operation.
Problem 3 · medium
The paper tunes one threshold per KV head (not per query head or per layer). Why that granularity, and what would go wrong if you used a single global threshold for all heads?
Show answer
Each KV head attends to different semantic subspaces (e.g., one head might track syntax, another sentiment). The distribution of attention scores — and hence the sign-bit concordance — varies per head. Setting TH per KV head allows fine-grained tuning: heads with tighter score distributions can use stricter thresholds (higher filtering), while heads with broader distributions need looser thresholds to avoid pruning relevant keys. A global threshold would either be too loose (low filtering on tight heads) or too strict (false negatives on broad heads). The paper tried per-query-head tuning but found it unstable (overfitting to individual queries); per-KV-head is the sweet spot. Llama-3 has 8 KV heads, so 8 thresholds is manageable and effective.Misconception: that a single threshold is simpler and "good enough." The per-head tuning is what enables the 20× filter ratio while staying within 5% perplexity — a global threshold can't match that.
Problem 4 · hard
At 1M-token context, DReX can hold ~32 concurrent users for Llama-3-8B. Compute this capacity and explain the bottleneck that limits it.
Show answer
Llama-3-8B has 32 layers × 8 KV heads = 256 context slices per user. Each context slice can hold up to 128 banks × 1,024 keys/bank = 131,072 keys. For 1M tokens, each head needs 1M / 131k ≈ 7.6 → round up to 8 context slices per head. Total slices per user: 32 layers × 8 heads × 8 = 2,048 slices. DReX has 8 packages × 128 banks = 1,024 banks total; each package can hold 128 banks × 8 slices (across layers) = 1,024 slices. With 8 packages: 8 × 128 = 1,024 banks → 1,024 / 8 (slices per head per user) = 128 users if we could use all banks. But data layout spreads one user's 8 heads across 8 packages (one head per package for parallelism). So each user occupies 8 packages × (8 slices per head × 32 layers / 128 banks) — working backwards: 8 packages, 32 layers × 8 heads = 256 slices, each slice ≤ 131k keys. For 1M tokens, we need ~8 "tiers" of slices per head. The 8 packages can each hold 128 banks / 8 tiers = 16 users worth of slices per package. But slices are partitioned across packages by head, so total users = 1,024 banks / (256 slices per user / 8 packages) = 1,024 / 32 = 32 users. Bottleneck: the number of banks (1,024) and the need to spread heads across packages for parallelism.Misconception: that DReX's 512 GB capacity directly translates to # users. The constraint is bank count and layout (one head per package) — not raw bytes. At shorter contexts (128k), you can fit 256 users because each user needs fewer slices.
Problem 5 · medium
Why does LongSight interleave full-precision key vectors across all 8 memory channels within a package, and what would go wrong if keys were stored in a single channel?
Show answer
During the NMA scoring phase, the NMA must load all surviving keys from DRAM (after PIM filtering prunes 95%). If keys were in a single channel, that channel's bandwidth (~17 GB/s for LPDDR5X) would be the bottleneck — the other 7 channels sit idle. By interleaving keys (dimension-wise or block-wise) across all 8 channels, the NMA can issue parallel reads and saturate the full package bandwidth (8 × 17 GB/s ≈ 136 GB/s). The paper stripes keys at column granularity (each channel holds a different portion of each key), so fetching one key hits all channels. Result: 8× bandwidth vs. single-channel. This is why the paper says "balanced channel utilization" (§7.3.2) is critical — without it, NMA scoring would stall on memory, negating the benefit of PIM filtering.Misconception: that DRAM layout doesn't matter as long as data fits. The physical layout determines whether you can exploit parallelism; interleaving is what turns multiple channels into usable bandwidth.
Problem 6 · hard
LongSight uses k = 1,024 for top-k selection. The paper also tests k = 128. Under what conditions would a smaller k (like 128) be better, and when would it hurt?
Show answer
Smaller k helps when: (1) CXL bandwidth is the bottleneck (high user count). With k = 128, you ship 128 × 256 bytes = 32 kB of values per head (vs. 256 kB for k = 1,024), reducing CXL transfer time by 8×. (2) The accuracy target is loose (low perplexity requirement). Fig. 4 shows that at 95% accuracy (vs. dense), k = 128 can suffice with aggressive filtering. Smaller k hurts when: (1) Filtering is weak (low filter ratio). If SCF only prunes 50% of keys, you're left with 64k survivors; selecting top-128 from 64k means you discard 63.9k keys that might include relevant ones → perplexity degrades. (2) The accuracy target is strict (>98%). Fig. 4 shows k = 1,024 is needed to stay within 2% of dense. Trade-off: the paper finds k = 1,024 + W = 1,024 is the sweet spot for most configs (Fig. 4 "All Configs" pareto frontier). At very high user counts where CXL saturates, you'd tune down k to reduce transfer cost; at low user counts where GPU is the bottleneck, you'd keep k = 1,024 to maximize accuracy.Misconception: that larger k is always better. There's a bandwidth vs. accuracy trade-off; the optimal k depends on system load and perplexity tolerance.
Problem 7 · easy
LongSight claims 1M-token support on 1 GPU + 1 DReX. Why can't you do this with just 2 GPUs (which together have 160 GB HBM)?
Show answer
You can fit 1M tokens in 2 GPUs — the KV cache is ~128 GB (computed in Example ①), leaving ~32 GB for weights/activations. But throughput is the issue. With dense attention, both GPUs must load the full KV cache for every token → 128 GB × tokens/sec. Even with 2 × 3.35 TB/s = 6.7 TB/s HBM bandwidth, max tokens/sec ≈ 6.7 TB/s / 128 GB ≈ 52 tokens/sec (if fully memory-bound). LongSight, with 20× filtering, loads only ~6.4 GB equivalent per token (128 GB / 20), so 64 GB/s CXL bandwidth → 64 GB/s / 6.4 GB ≈ 10 tokens/sec per user, but with batching and parallelism (8 NMAs), it scales better. Fig. 7 shows LongSight at 1M tokens serves 3 users at ~30 total tokens/sec — competitive with 2-GPU dense at lower context. The real win is cost: one DReX unit (~$1k for LPDDR + logic) vs. a second H100 (~$30k).Misconception: that memory capacity is the only constraint. Bandwidth and cost matter too — and LongSight trades expensive HBM for cheap LPDDR + near-memory acceleration.
Problem 8 · medium
The paper reports that LongSight's perplexity is within 5% of dense attention. Is this acceptable for production LLM serving, and in what scenarios might it not be?
Show answer
Context: Perplexity is an intrinsic metric; 5% increase means the model is slightly less confident in its predictions (e.g., perplexity 10 → 10.5). For many tasks (e.g., RAG, summarization), this is acceptable — the model still retrieves the right context and generates coherent text; the small degradation is imperceptible. The paper shows (Fig. 10) that LongSight's accuracy vs. throughput pareto frontier is competitive with sliding-window baselines. Where it might not be acceptable: (1) Safety-critical tasks (e.g., medical diagnosis, legal advice) where even 5% confidence loss could change the output. (2) Tasks requiring exact recall (e.g., copying a long document verbatim) — if the model misses key tokens due to aggressive filtering, outputs could be incorrect. (3) Regulated environments that require bit-exact reproducibility with the baseline model. Mitigation: tune thresholds more conservatively (lower filter ratio) to reduce perplexity gap to <1%, at the cost of lower throughput. The 5% is a tunable trade-off, not a hard constraint.Misconception: that sparse attention always degrades quality unacceptably. For most LLM applications, 5% perplexity increase is negligible — and users can tune the accuracy-speed trade-off per use case.
Quick answers

Frequently asked questions

Fourteen questions about LongSight. Tap to expand.

Q1How does LongSight compare to other sparse attention schemes like Reformer or Longformer?▾
Reformer uses LSH with per-token overhead (linear time) and multiple rounds; LongSight uses sign-bit filtering (constant time per key, embarrassingly parallel). Longformer uses static sliding windows + global tokens; LongSight uses dynamic, data-dependent filtering (sign-concordance) plus top-k. Result: LongSight achieves higher sparsity (20× filter ratio vs. Longformer's ~4× for comparable accuracy) with lower overhead, and doesn't require task-specific fine-tuning like Longformer's attention masks.
Q2What is the area and power overhead of DReX?▾
PIM filtering units add 6.7% area overhead to DRAM dies (scaled to 7nm process). Each NMA chip is 15.1 mm² in 16nm, consuming ~1 W peak. Total: 8 NMAs = 121 mm² logic + 8 W. LPDDR5X packages consume ~18.7 W each (8 × 18.7 = 150 W). Total system: ~158 W peak for the entire DReX unit (512 GB + 8 NMAs). For comparison, an H100 GPU is ~700 W — so DReX is ~23% of one GPU's power, while providing 6.4× the memory capacity.
Q3Can LongSight support models other than Llama-3 (e.g., GPT-4, Gemini)?▾
Yes, as long as the model uses standard multi-head or grouped-query attention. LongSight's algorithm (sign-concordance filtering + top-k) is model-agnostic; you'd just re-tune ITQ rotation matrices and thresholds for the new model's key/query distributions. The paper demonstrates Llama-3-1B and Llama-3-8B; extending to larger models (e.g., 70B) would require more DReX capacity or shorter per-user contexts (to fit more users). For models with exotic attention (e.g., sparse transformers with learned masks), you'd need to adapt the algorithm — but the hardware (PIM + NMA) is general-purpose for dot-product + top-k.
Q4How does LongSight handle multi-turn conversations where context keeps growing?▾
Each turn appends new tokens to the KV cache in DReX. The sliding window in GPU HBM always holds the most recent 1,024 tokens (including the latest turn), and older turns are in DReX. As long as the total context fits in DReX's 512 GB, LongSight can handle arbitrarily long conversations. When the cache fills, you'd either evict old turns (LRU-style), compress them (e.g., summarize), or expand to more DReX units (multiple CXL memory expanders). The paper doesn't address cache eviction policies — that's future work.
Q5Why use CXL instead of NVLink or Infinity Fabric for GPU-DReX communication?▾
CXL provides byte-addressability — the GPU can load/store into DReX memory with regular instructions (via SCADA API), without DMA or OS involvement. NVLink is GPU-to-GPU only (not host-attached memory); Infinity Fabric is AMD-specific. CXL is an industry standard, supports cache coherence, and is available on existing PCIe infrastructure. The trade-off: CXL bandwidth (~64 GB/s on PCIe 5.0 × 16) is lower than NVLink (~900 GB/s for NVLink 4.0), but LongSight mitigates this with near-memory acceleration (only top-k values cross CXL, not the full cache).
Q6Can LongSight support batching across multiple users, and how does it scale?▾
Yes — the paper evaluates batches up to 496 users (Fig. 7). Each user's KV cache is stored in a separate "User Partition" in DReX. DReX processes attention requests sequentially (FIFO queue), assigning NMAs to different users' heads in parallel. Scaling: at low batch sizes (few users), GPU is the bottleneck (underutilized while waiting for DReX). At high batch sizes, DReX saturates (value loading over CXL becomes the bottleneck). Optimal batch size depends on context length and user SLOs; the paper finds ~62–124 users at 128k context gives good GPU utilization without overwhelming DReX.
Q7What happens if a user's context exceeds DReX's capacity?▾
DReX can hold ~32 users at 1M tokens each for Llama-3-8B (computed in Practice Problem 4). If a single user exceeds 1M tokens, you'd either (a) use multiple DReX units (chained via CXL), (b) evict old KV pairs (e.g., keep only the most recent 1M), or (c) compress the cache (quantize, prune less-relevant layers). The paper doesn't address these scenarios — it focuses on demonstrating 1M-token support with one DReX. For extreme contexts (>1M), you'd likely need a tiered KV cache: DReX for recent history, SSD for very old history (with even coarser filtering).
Q8How does LongSight's latency compare to GPU-only dense attention at short contexts?▾
At short contexts (<32k tokens), GPU-only dense is faster than LongSight. Why? Because the KV cache fits entirely in HBM, so there's no capacity bottleneck, and HBM bandwidth (3.35 TB/s) dwarfs CXL (64 GB/s). LongSight adds overhead: CXL round-trip (~20 µs), PIM filtering (~2 µs), NMA scoring (~12 µs). At 32k context, Fig. 7 shows LongSight is competitive (slightly worse throughput for single-user, better at high batch). The crossover is around 64k–128k context, where GPU HBM starts to fill and LongSight's sparse retrieval + large capacity win. Moral: LongSight is for large-context scenarios; stick with GPU-only for short contexts.
Q9Can LongSight be used for training, or only inference?▾
The paper focuses on inference (decode phase). Training has different characteristics: (a) gradients must flow through the full attention mask (can't just keep top-k), (b) you need to store activations for backward pass (doubles memory), (c) training workloads often use smaller batch sizes and can fit in GPU memory. That said, LongSight's algorithm (sign-bit filtering) could be adapted for training — e.g., as a form of gradient checkpointing or sparse backprop. But the system integration (CXL memory, PIM) is optimized for read-heavy inference; training is write-heavy (updating weights), which CXL handles poorly (high latency for small writes). Future work could explore LongSight for training, but it's not the current focus.
Q10Why does LongSight use BF16 instead of lower-bit quantization (e.g., INT8, INT4)?▾
The paper uses BF16 because it's the standard precision for Llama-3 models; the goal is to match the baseline's quality. LongSight could be combined with quantization — e.g., store keys in INT8 in DReX, then dequantize for scoring. This would reduce memory footprint (2× for INT8, 4× for INT4) and transfer bandwidth. However, quantization adds complexity (calibration, potential accuracy loss) and is orthogonal to the sparse retrieval mechanism. The paper leaves this as future work. Note that sign-bit filtering is already a form of 1-bit quantization for the filtering stage; the full-precision scoring ensures accuracy.
Q11How does LongSight compare to recent work like FlexGen or DeepSpeed Infinity?▾
FlexGen and DeepSpeed Infinity offload model weights and activations to CPU memory or SSD to enable large-model inference on limited GPU memory. LongSight offloads the KV cache to CXL memory and uses PIM/NMA to accelerate sparse attention. The two approaches are complementary: you could combine LongSight (for KV cache) with FlexGen (for weights/activations). Key difference: LongSight uses compute-enabled memory (PIM + NMA do filtering/scoring in-place), while FlexGen treats CPU memory as passive storage (GPU pulls data). Result: LongSight is faster for attention-heavy workloads, FlexGen is more general-purpose for weight-limited scenarios.
Q12What are the failure modes of sign-concordance filtering?▾
SCF can false-negative (prune a relevant key) if its sign-bit concordance with the query is below the threshold — even if the actual dot-product is high. This happens when vectors are nearly orthogonal in some dimensions (sign mismatch) but aligned in others (high magnitude). ITQ mitigates this by rotating to balance signs, but it's not perfect. The paper tunes thresholds to ensure <5% perplexity increase, which bounds the false-negative rate. A second failure mode: false-positive (keep an irrelevant key). This is less critical because the top-k step filters these out, but it wastes NMA scoring bandwidth. The paper's high filter ratio (20×) suggests false-positives are rare.
Q13How does LongSight handle models with very large head dimensions (e.g., 256 instead of 128)?▾
Larger head dimension (D) increases PIM filtering cost (D × 1.25 ns), NMA scoring cost (D MACs per key), and memory footprint (D × 2 bytes per key). For D = 256, PIM filtering time doubles (320 ns vs. 160 ns), but this is still negligible (<5 µs total for 128k context). NMA scoring doubles in FLOPs but is memory-bound anyway. The real impact: memory footprint doubles → fewer tokens fit in DReX (512 GB / 2 = 256 GB effective). So for D = 256, you'd halve the supported context length (500k tokens instead of 1M) or use more DReX units. The algorithm itself is dimension-agnostic; only the capacity/performance scale changes.
Q14Can LongSight be implemented on existing CXL Type-3 devices without custom hardware?▾
No — LongSight requires compute-enabled CXL memory (DReX), which integrates PIM filtering units in DRAM and near-memory accelerators (NMAs). Standard CXL Type-3 devices are passive (just memory). You could implement sign-bit filtering in software on the host CPU, but then you'd have to transfer all keys over CXL for scoring → CXL bandwidth becomes the bottleneck, negating the benefit. The paper's whole point is to do filtering in-place (in DRAM) and scoring near DRAM (NMA), so only top-k results cross CXL. Without custom hardware, LongSight's throughput gains would evaporate. That said, DReX is a real, implementable design (the paper cites RTL synthesis results) — it's not science fiction.

Related work

DReX: Dense Retrieval Accelerator prerequisite

Quinn et al., ISCA '25

The original DReX paper (reference [34] in LongSight) describes the compute-enabled CXL memory architecture. LongSight repurposes DReX for attention by treating the KV cache as a vector database of Keys.

Reformer: LSH-based sparse attention alternative

Kitaev, Kaiser, Levskaya, ICLR '20

Uses locality-sensitive hashing for probabilistic filtering. LongSight contrasts this (§3.1): LSH has per-token overhead and requires multiple rounds; SCF is constant-time per key and embarrassingly parallel.

Longformer: Sliding window + global attention alternative

Beltagy, Peters, Cohan, ACL '20

Static sliding window + manually configured global tokens. LongSight's hybrid is similar (dense window) but adds data-dependent sparse retrieval (SCF + top-k) for the rest of context.

NeuPIMs & AttAcc: PIM for dense attention related

Heo et al., ASPLOS '24 (NeuPIMs); Park et al., ASPLOS '24 (AttAcc)

Offload dense attention to PIM units. LongSight differs: it implements sparse attention (sign-bit filtering + top-k) and uses CXL memory, not on-chip HBM-PIM.

DynaX: Block-level sparse attention with pruning alternative

Xiong et al., ASPLOS '25

Block-level sparsity + 4/6-bit quantization for approximate attention scores. LongSight achieves finer-grained (per-token) sparsity with sign-bit filtering, avoiding the quantization overhead on full keys during filtering.
