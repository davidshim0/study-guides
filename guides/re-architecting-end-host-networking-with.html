When the host and NIC speak the same cache language

For decades, PCIe forced NICs to talk to CPUs through slow DMA and MMIO. CXL replaces that with cache-coherent load/store semantics, cutting NIC-to-CPU latency by half and unlocking coherent on-device memory. This is the story of how.

1What a NIC is and why the CPU-NIC link matters▾

A Network Interface Controller (NIC) connects a host machine to the network. It receives packets from the wire, transfers them to CPU memory, and transmits packets the CPU has prepared. Historically NICs connect via PCIe, a high-bandwidth but non-coherent bus that requires explicit DMA setup for data movement and MMIO for control signaling.

At 100+ Gbps line rates, millions of packets arrive per second. The CPU-NIC link's latency and throughput directly determine networking performance. PCIe's DMA/MMIO overhead becomes a measurable bottleneck.

Now you're ready: CXL-NIC replaces PCIe's non-coherent primitives with cache-coherent load/store, eliminating that overhead.
2Rx and Tx datapaths: the packet journey▾

Rx (receive): packets arrive from the network. The NIC DMA-writes them into host memory buffers referenced by descriptors in a ring buffer, updates the descriptor to signal arrival, then optionally interrupts the CPU. The CPU polls descriptors to detect ready packets and processes them.

Tx (transmit): the CPU writes a packet into a buffer, creates a descriptor pointing to it, writes the descriptor to a ring buffer, and triggers a doorbell (MMIO write) to notify the NIC. The NIC DMA-reads the descriptor and the packet, then transmits it.

Now you're ready: CXL-NIC replaces every DMA read/write and MMIO operation in these flows with CXL.cache requests, reducing latency at each step.
3PCIe's two bottlenecks: DMA and MMIO▾

DMA (Direct Memory Access) enables bulk data transfer between NIC and host memory. While it achieves high throughput for large transfers, its setup overhead (memory pinning, descriptor preparation) makes it inefficient for small transfers like 64B descriptors. MMIO (Memory-Mapped I/O) allows the CPU to write to device registers via store instructions, but those writes traverse the PCIe bus with >8μs latency and <0.3 GB/s bandwidth for small writes.

Worse, PCIe provides no cache coherence. Software must manage cache consistency when sharing buffers between CPU and NIC, and the NIC must poll host memory to detect updates.

Now you're ready: the paper quantifies these bottlenecks (§3.1) and shows that PCIe operations account for 45–52% of end-to-end packet latency even with optimized DPDK stacks.
4CXL protocols: .io, .cache, and .mem▾

CXL.io is backward-compatible with PCIe for device initialization and configuration. CXL.cache lets devices access host memory with cache-coherent load/store semantics (Device-to-Host), eliminating software cache management. CXL.mem lets the CPU access device memory via load/store (Host-to-Device), exposing it as a remote NUMA node.

CXL Type-1 devices support .io and .cache; Type-2 devices add .mem and expose on-device DRAM. Both are built atop the PCIe physical layer, reusing existing signaling while adding new transaction and link layers.

Now you're ready: CXL-NIC uses Type-1 for coherent datapath (§4) and Type-2 for coherent on-NIC memory (§5).
5CXL.cache request types: explicit cache line state control▾

CXL.cache lets the device specify four cache line states: NC (non-cacheable) bypasses the device cache; NC-P (push) writes to device cache and pushes to host LLC; CS (cacheable shared) caches in Shared state; CO (cacheable owned) acquires exclusive ownership. This explicit control is unique to CXL — neither PCIe nor even Intel's UPI offer it.

Why it matters: by choosing the right request type, CXL-NIC avoids device cache pollution, reduces coherence traffic, and eliminates software cache flushes that CC-NIC [48] requires.

Now you're ready: CXL-NIC uses NC for data movement (to avoid cache-to-cache transfers), CO for event-driven synchronization, and NC-P to push RxDs into host LLC (§4.3).
6Descriptor rings: how the CPU and NIC coordinate▾

A descriptor is a small metadata structure (~64B) that references a packet buffer and carries status flags. Descriptors live in a circular ring buffer in host memory. The CPU and NIC each maintain head/tail pointers: for Rx, the CPU writes the head to tell the NIC how many blank descriptors are ready; for Tx, the CPU writes the tail to signal new packets ready to transmit.

In PCIe NICs, these pointers live in device registers updated via MMIO. In CXL-NIC, they live in host memory, and the NIC polls them with cache-coherent CO-read requests.

Now you're ready: CXL-NIC's event-driven Tx (§4.2) replaces MMIO doorbell writes with CO-read polling of a host-resident tail pointer, reducing latency by 29%.
7Device-bias vs host-bias mode▾

CXL.mem supports two coherence modes for device-local accesses. Host-bias mode enforces full coherence: on every device memory access, CXL hardware checks the host cache for modified copies and fetches them if present. Device-bias mode skips host checks, allowing fast local access but requiring software to manage coherence explicitly.

Device-bias is a win when the device repeatedly accesses local data the host rarely touches. Host-bias is necessary when both host and device access the same memory region.

Now you're ready: CXL-NIC's KVS offload (§5.4) uses device-bias for cache hits in NIC memory and switches to host-bias for misses that fetch from host memory.
8Why DPDK and why it's still limited▾

DPDK (Data Plane Development Kit) is a user-space networking framework that bypasses the kernel to reduce latency. It uses polling instead of interrupts, avoids context switches, and batches operations. Despite these optimizations, DPDK still relies on PCIe DMA and MMIO, and it cannot eliminate cache flushes or software-managed coherence when sharing buffers between CPU and NIC.

CXL-NIC implements a CXL-aware DPDK stack (§4.1) that retains DPDK's user-space polling model but replaces PCIe operations with CXL.cache requests.

Now you're ready: the paper's evaluation (§7) compares CXL-NIC running CXL-DPDK against a commodity NIC running standard DPDK, isolating the benefit of CXL protocols.

The whole story in one place

CXL-NIC in a nutshell

The problem, how CXL fixes it, and the performance gains.

PCIe NICs suffer from two fundamental limitations. First, they rely on DMA for data movement, which incurs setup overhead that makes small transfers (e.g., 64B descriptors) inefficient, and MMIO for control signaling, which is slow (>8μs) and bandwidth-limited (<0.3 GB/s). Second, PCIe provides no cache coherence, forcing software to manage cache consistency and requiring the NIC to poll host memory to detect updates. At 100+ Gbps line rates, these bottlenecks contribute 45–52% of end-to-end packet latency (§3.1).

CXL-NIC replaces these non-coherent primitives with cache-coherent CXL.cache requests. For data movement, it uses NC-read/NC-write instead of DMA, achieving 69–81% lower latency for small transfers (§3.2). For synchronization, it uses CO-read to poll host-resident pointers (replacing MMIO doorbells) and NC-write to update descriptors, eliminating cache flushes. CXL's unique ability to specify cache line states (NC, NC-P, CS, CO) enables fine-grained optimizations: NC avoids device cache pollution; CO enables event-driven polling; NC-P pushes RxDs into host LLC for fast CPU access.

Building on Type-1 CXL-NIC's coherence benefits, Type-2 CXL-NIC adds CXL.mem to expose gigabyte-scale on-NIC DRAM as a coherent remote NUMA node. This allows flexible placement of descriptor rings and packet buffers across host and NIC memory, with device-bias mode enabling fast local access when the host doesn't need coherence checks. As a case study, the paper implements a Key-Value Store (KVS) on Type-2 CXL-NIC that hosts frequently-accessed data in NIC memory (device-bias) and falls back to host memory (host-bias) on misses, all with hardware-managed coherence (§5.4).

The payoff: on an FPGA-based prototype, CXL-NIC reduces network packet processing latency by 49% vs. PCIe NICs (37% vs. CC-NIC) and KVS request tail latency by 39% vs. SmartNIC-based designs.

🍯 Key takeaways

  • PCIe's DMA/MMIO account for 45–52% of packet latency; CXL.cache requests cut that by 69–81% (§3).
  • CXL.cache's explicit cache line state control (NC, NC-P, CS, CO) enables optimizations impossible with PCIe or UPI (§4.3).
  • Type-1 CXL-NIC replaces DMA/MMIO with cache-coherent requests; Type-2 adds coherent on-NIC memory via CXL.mem (§4, §5).
  • Device-bias mode allows fast local NIC memory access without host coherence checks; host-bias enforces coherence when needed (§5.1).
  • CXL-NIC reduces packet processing latency by 49% (vs. PCIe NIC) and 37% (vs. CC-NIC) on real hardware (§7).
  • Application-layer co-acceleration (KVS on NIC memory) cuts tail latency by 39% vs. RDMA-based SmartNIC (§7.2).
Six everyday pictures

How to think about CXL-NIC

Each analogy stands alone. Every one ends with where it breaks.

The core idea · DMA/MMIO vs load/store

Sending a letter vs talking

PCIe is like mailing a letter: write it, address the envelope, drop it in a mailbox (DMA setup), wait for delivery, get a receipt (completion). For one sentence, that's absurd. CXL.cache is talking directly: issue a load/store (one CPU instruction), hardware fetches the data coherently, done. Same latency whether you're asking for a word or a paragraph. CXL-NIC uses this for descriptor updates and pointer polling.

Where it breaks: for gigabyte-scale bulk transfers, DMA's pipelining and batching still win. CXL.cache shines on small, frequent transfers (descriptors, control structures) where DMA setup overhead dominates.
Coherence · Hardware-managed consistency

Shared whiteboard with instant sync

PCIe is two people with separate whiteboards trying to stay in sync by shouting updates across the room. One writes, the other must manually erase and rewrite. Errors creep in. CXL.cache is a single shared whiteboard: write on your side, hardware instantly mirrors it to the other side. Neither party ever sees stale data. CXL-NIC uses this to eliminate cache flushes when recycling packet buffers.

Where it breaks: coherence isn't free — it adds latency when both sides fight for ownership. CXL-NIC mitigates this with explicit cache line state control (NC, CO) to avoid unnecessary coherence traffic.
Cache line state control · Explicit ownership

Library checkout policies

Most caches are like libraries: check out a book (fetch to cache), library tracks who has it, return triggers automatic updates. CXL.cache adds explicit policies: NC = read-only room visit (no checkout); NC-P = copy to someone else's shelf; CS = shared checkout (multiple readers); CO = exclusive checkout (only you can write). CXL-NIC uses CO to poll Tx tail pointers — once acquired, local reads cost nothing until the host updates it.

Where it breaks: choosing the wrong policy hurts. CO on read-only data wastes coherence traffic acquiring exclusive ownership. NC on frequently-reused data misses caching benefits. The paper optimizes each datapath step.
CXL.mem · Coherent device memory

A guest room in your house

PCIe device memory is a locked shed in the backyard: you can peek through MMIO (slow), but to move anything in/out requires hauling boxes (DMA). CXL.mem makes it a guest room in your house: walk in, grab what you need, coherence ensures you always see the latest version. CXL-NIC's Type-2 uses this to host descriptor rings and packet buffers in NIC memory, accessible to both CPU and NIC with ld/st.

Where it breaks: the guest room is remote — accessing NIC memory from the CPU is slower than local DRAM. Device-bias mode mitigates this by letting the NIC skip coherence checks when the host isn't watching.
Bias modes · Coherence on demand

Trusting your neighbor

Device-bias = you assume your neighbor won't touch your stuff, so you don't lock the door (no coherence checks). Fast when you're right. Host-bias = you always check if they moved something before using it (coherence enforced). Slower but always correct. CXL-NIC's KVS uses device-bias for NIC-resident cache hits (neighbor won't touch it) and switches to host-bias for misses that fetch from host memory.

Where it breaks: device-bias requires software discipline. If the host accesses device-bias memory without coordination, coherence breaks. The paper uses device-bias only for NIC-local data that the host never touches during normal operation.
Application offload · KVS on NIC memory

Store inventory at the loading dock

Traditional setup: warehouse (host memory) stores all inventory; loading dock (NIC) just moves boxes in/out. Every shipment requires a warehouse trip. CXL-NIC: keep popular items at the dock (NIC memory). Shipping requests check the dock first (device-bias, fast); if not found, fetch from warehouse (host-bias, slower but rare). Both are in the same address space, so the forklift (coherence) moves items between them automatically.

Where it breaks: the loading dock is small. CXL-NIC's KVS only hosts "hot" keys in NIC memory; "cold" keys stay in host memory. An LRU policy swaps entries between tiers, but thrashing can hurt performance.
The full explanation

How it actually works

Section by section, with diagrams you can read at a glance.

1 · The PCIe bottleneck — quantifying the overhead

The paper begins by measuring PCIe's latency contribution in a loopback test: packets arrive at the NIC, get transferred to host memory (Rx), the CPU copies them to Tx buffers, and the NIC transmits them. By timestamping at four points (packet arrival at NIC, availability in host memory, Tx initiation, packet departure), the authors calculate that PCIe operations (DMA reads/writes + MMIO doorbells) account for 52% of total latency for 64B packets and 45% for 1500B packets at 64 Gbps (Fig. 3).

Why does PCIe struggle? (1) DMA setup overhead (memory pinning, descriptor preparation, completion signaling) dominates small transfers; (2) MMIO writes to device registers (doorbells) have >8μs latency; (3) non-coherence forces cache management in software. As line rates exceed 100 Gbps, even microsecond delays become bottlenecks.

🌱
Concrete numbers: a 64B DMA read takes ~3.4μs on NVIDIA BlueField-3; CXL NC-read takes ~1.1μs (69% reduction). MMIO doorbell: ~3.0μs; CXL CO-read poll: ~2.1μs (29% reduction). (Paper Fig. 4)

2 · Type-1 CXL-NIC architecture — coherent datapath

Type-1 CXL-NIC replaces PCIe operations with CXL.cache requests while keeping all networking data in host memory. Figure 5 shows the hardware: a Descriptor Management Unit (DMU) fetches and updates descriptors; a Packet Forwarding Unit (PFU) moves packets between network interface and host memory; both connect to the CXL Device COHerence engine (DCOH) to issue D2H requests. A 1 MB on-chip SRAM buffers packets during bursty traffic.

Configuration happens via CXL.io (backward-compatible with PCIe MMIO) during initialization: the host CPU writes descriptor ring buffer addresses and request type selections to CSR/MMIO space. At runtime, the host updates ring pointers and descriptors via ld/st, and CXL hardware ensures coherence.

3 · Rx datapath — coherent packet reception

The Rx flow in Type-1 CXL-NIC:

  1. Initialization: host CPU allocates Rx descriptor ring and packet buffers in host memory; writes ring base address to DMU via CXL.io.
  2. Prefetch: DMU issues D2H CS-read to prefetch available RxDs into device cache (HMC) in Shared state.
  3. Packet arrival: DMU fetches RxD from HMC via CO-read (hit in cache, 3.3× faster than host memory access), extracts packet buffer address, instructs PFU to transfer packet.
  4. Packet transfer: PFU issues D2H NC-write to write packet to host memory buffer, bypassing device cache to avoid pollution.
  5. Completion: DMU issues D2H NC-write to update RxD status field, invalidating cached copy.
  6. Polling: host CPU polls RxD status (no interrupt), processes packet, recycles buffer.
💡
Why CS-read for prefetch? CS caches RxDs in Shared state, enabling fast CO-read hits when packets arrive while allowing hardware to invalidate prefetched descriptors if the host updates them (e.g., reclaiming buffers).

4 · Tx datapath — event-driven transmission

The Tx flow in Type-1 CXL-NIC:

  1. Initialization: host CPU allocates Tx descriptor ring and packet buffers in host memory; writes ring base address and tail pointer address to DMU. Unlike PCIe NICs where the tail lives in a device register, CXL-NIC stores it in host memory.
  2. Packet preparation: host CPU writes packet to Tx buffer, creates TxD pointing to buffer, inserts TxD into ring, updates tail pointer via st.
  3. Polling: DMU periodically issues D2H CO-read to poll tail pointer. CO caches it in Owned state in HMC; subsequent polls hit locally (no CXL traffic) until the host updates the tail.
  4. Update detected: when host updates tail, CXL coherence invalidates DMU's cached copy, next poll fetches new value.
  5. Descriptor fetch: DMU issues D2H NC-read to fetch new TxDs (inline signaling: TxD itself contains a status flag checked by CO-read).
  6. Packet fetch: PFU issues D2H NC-read to fetch packet from host memory buffer.
  7. Completion: DMU issues D2H NC-write to update TxD status; host CPU polls descriptor to reclaim buffer.
🍯
Event-driven Tx: CO-read on the tail pointer creates a hardware-managed event notification. The NIC polls locally (in HMC) until the host updates the pointer, triggering a coherence invalidation that wakes the poll. Zero MMIO writes. (Paper §4.2)

5 · Optimizations with cache line state control

CXL.cache's explicit cache line state control enables fine-grained optimizations impossible with PCIe or even Intel UPI:

OperationRequest typeRationale
RxD prefetchCS-readCaches RxDs in Shared state; hardware auto-invalidates on host update
Tx tail pollCO-readCaches tail in Owned state; local polls until host invalidates it
Packet transferNC-read/writeBypasses device cache to avoid pollution + cache-to-cache transfers
Descriptor updateNC-writeWrites to host memory, invalidates device cache copy (no flush needed)

The paper shows (§4.3) that naïve choices hurt: using CO-write for Rx packet transfer (instead of NC-write) increases latency by 15–19% because it forces exclusive ownership acquisition and pushes packets into device cache, requiring cache-to-cache transfer when the host reads them. Using CS-read for Tx packets (instead of NC-read) similarly adds 13–17% latency.

⚠
CC-NIC's limitation: CC-NIC [48], built on Intel UPI, lacks explicit cache line state control. It must use software cache flushes (CLFLUSH) to invalidate Tx packet buffers in device cache before the host can reuse them, adding software overhead that CXL-NIC avoids with NC-read.

6 · Type-2 CXL-NIC — coherent on-NIC memory

Type-2 CXL-NIC extends Type-1 by adding CXL.mem, exposing gigabyte-scale on-NIC DRAM as a coherent remote NUMA node. The host CPU can allocate descriptor rings and packet buffers in either host or NIC memory via standard NUMA-aware allocation (e.g., numactl), and CXL hardware ensures coherence.

Key architectural choice: CXL.mem supports two bias modes. Host-bias enforces full coherence — on every D2D access to NIC memory, DCOH checks the host cache for modified copies and fetches them if present. Device-bias skips host checks, enabling fast local access but requiring software to coordinate when both host and NIC touch the same data.

The paper evaluates four memory layouts (both Rx/Tx buffers in host; Rx in NIC, Tx in host; etc.) and finds that naïve NIC-resident buffers hurt latency by 23–71% (§7.2, Fig. 13) due to (1) slower remote memory access over CXL and (2) coherence overhead in host-bias mode. The optimal layout places the Rx descriptor ring, Rx packet buffer, and Tx packet buffer in NIC memory, while the Tx descriptor ring stays in host memory, with careful request selection (CS-read for RxD prefetch, NC-P for Rx completion, device-bias NC-read for Tx packets).

7 · NC-P: intelligent cache injection

CXL introduces a unique write request: NC-P (non-cacheable push). On NC-P, DCOH writes the cache line to device memory and pushes a copy into the host LLC, then invalidates the device cache copy. This reduces host access latency by pre-populating the LLC, avoiding a remote fetch from NIC memory.

CXL-NIC uses NC-P for Rx descriptor completion: after writing the packet to NIC memory, DMU issues NC-P to update the RxD. The host CPU's poll hits the LLC (fast) instead of fetching from remote NIC memory (slow). However, unrestricted NC-P risks overwhelming the LLC ("leaky DMA" problem). The paper mitigates this with two mechanisms: (1) adaptive push-write gating — a runtime-configurable flag lets the host CPU disable NC-P when LLC utilization is high, reverting to NC-write; (2) post-push write-back — after NC-P, issue a delayed NC-write to the same address, evicting the cache line from LLC after 𝑁cycles (tunable via device register).

🍯
Why this matters: NC-P is unique to CXL — neither PCIe nor UPI provide equivalent primitives. DDIO can inject I/O data into LLC, but it lacks fine-grained control and coherence guarantees. (Paper §5.3)

8 · Application-layer co-acceleration: KVS on NIC memory

Type-2 CXL-NIC enables cross-layer optimization. The paper implements a Key-Value Store (KVS) on NIC memory with a request handler in the user design region (Fig. 9). The KVS uses a hash table with cuckoo hashing; frequently-accessed ("hot") key-value pairs live in NIC memory (device-bias mode for fast local access), while "cold" pairs reside in host memory.

Request flow: (1) packet arrives with KVS GET/PUT request; (2) KVS handler extracts key, computes hash, probes hash table in NIC memory via D2D CS-read (device-bias); (3) on hit, serve locally via D2D NC-read (device-bias); (4) on miss, switch to host-bias mode, issue D2H CO-read to fetch from host memory, serve request, asynchronously swap into NIC memory (LRU eviction).

The coherence guarantee: the host CPU can update the KVS database at any time via ld/st, and CXL hardware ensures the NIC sees the latest value (in host-bias mode) or the update invalidates device-bias entries (software-managed). No manual cache flushes, no RDMA setup. (Paper §5.4)

Follow the numbers, step by step

Worked examples

Four traces using the paper's real measurements.

① Computing PCIe latency contribution from timestamps

The paper measures 64B packet loopback latency at 64 Gbps. Four timestamps: 𝑇₀ = packet arrival at NIC; 𝑇₁ = packet in host memory; 𝑇₂ = Tx initiation by CPU; 𝑇₃ = packet back at NIC. Total latency = 𝑇₃ − 𝑇₀ = 5.1 μs. Host-side processing = 𝑇₂ − 𝑇₁ = 2.4 μs. What's PCIe's contribution?

  1. Total latency: 𝑇₃ − 𝑇₀ = 5.1 μs (packet arrival to departure).
  2. Host-side processing: 𝑇₂ − 𝑇₁ = 2.4 μs (copy packet, prepare descriptor, etc.).
  3. PCIe latency: everything except host processing = 5.1 − 2.4 = 2.7 μs.
  4. PCIe fraction: 2.7 / 5.1 = 52%. Over half the latency is PCIe DMA/MMIO.
  5. CXL reduction: CXL-NIC measures 2.6 μs total latency for the same workload. PCIe contribution drops to ~0.6 μs (25%), a 78% reduction in interconnect overhead. (Paper Fig. 3)
🍯
Insight: even with DPDK polling and batching optimizations, PCIe still accounts for over half the latency. CXL.cache cuts that in half, freeing CPU cycles for packet processing.

② DMA vs CXL.cache latency for small transfers

Measure 64B descriptor read latency: NVIDIA BlueField-3 DOCA-DMA vs. Intel Agilex-7 CXL NC-read. Why is DMA so much slower?

  1. DOCA-DMA setup: ARM cores on BF-3 prepare a DMA descriptor (source/dest addresses, length), write it to DMA engine registers, trigger DMA, poll completion status. Measured latency: 3.4 μs.
  2. CXL NC-read: user-defined logic on Agilex-7 issues D2H NC-read request to DCOH, which fetches 64B from host memory via CXL.cache. Measured latency: 1.05 μs.
  3. Reduction: (3.4 − 1.05) / 3.4 = 69% lower. (Paper Fig. 4)
  4. Why DMA loses: DMA setup (descriptor write, doorbell, completion poll) dominates 64B transfer time. CXL.cache bypasses setup — one request, one response, done.
  5. Larger packets: at 9 KB, DOCA-DMA latency grows to 4.4 μs (1.3× increase), CXL NC-read to 1.25 μs (1.2× increase). CXL still wins, but the gap narrows as transfer time dominates setup overhead.
💡
DMA is optimized for bulk transfers where setup amortizes over many kilobytes. For descriptor-sized transfers, CXL.cache's single-request model is a better match.

③ Event-driven Tx with CO-read polling

Walk through Tx tail pointer polling using CO-read. Why does it reduce CXL traffic?

  1. Initialization: host CPU allocates Tx ring in host memory, writes tail pointer address (also in host memory) to DMU via CXL.io.
  2. First poll: DMU issues D2H CO-read to fetch tail pointer. DCOH fetches cache line from host memory, stores it in device cache (HMC) in Owned state, returns value to DMU. Tail = 0 (no new descriptors).
  3. Subsequent polls: DMU issues CO-read again. DCOH hits HMC (cache line still in Owned state), returns value without CXL traffic. Tail = 0 (no update).
  4. Host update: host CPU st to tail pointer (tail = 8, indicating 8 new TxDs). CXL coherence invalidates the cached line in HMC.
  5. Next poll: DMU issues CO-read, HMC miss, DCOH fetches updated tail from host memory. Tail = 8 → DMU fetches 8 TxDs and processes them.
  6. Comparison: NC-read would generate CXL traffic on every poll (no caching). CO-read caches locally and only fetches on host update (event-driven). The paper reports that CO-read reduces bandwidth consumed by polling traffic; at 100 Gbps with frequent Tx updates, this avoids significant CXL bus utilization. (Paper §4.3)
🍯
Insight: CO-read's exclusive ownership (Owned state) enables local polling without CXL traffic until the host invalidates it. This is the "hardware-managed event notification" that replaces PCIe MMIO doorbells.

④ KVS request processing with device-bias/host-bias

A KVS GET request for key "foo". Hash maps to bucket 42 in NIC memory. Walk through the request flow and compare latency to RDMA-based SmartNIC.

  1. Request arrival: packet with "GET foo" arrives at network interface, KVS handler extracts key.
  2. Hash lookup: compute hash(foo) = 42. Bucket 42 resides in NIC memory (device-bias region). KVS handler issues D2D CS-read (device-bias) to probe bucket 42. Latency: ~200 ns (local DRAM access, no coherence check with host).
  3. Hit: "foo" found in bucket 42. KVS handler issues D2D NC-read (device-bias) to fetch value, constructs response packet, transmits it. Total latency (median): the paper reports aggregate latency bars showing CXL-NIC outperforms the baseline. (Paper Fig. 15)
  4. Miss case: "bar" not in NIC memory. KVS handler switches to host-bias mode, issues D2H CO-read to fetch "bar" from host memory. Handler serves request, asynchronously swaps "bar" into NIC memory (LRU eviction of cold entry).
  5. Comparison to RDMA: NVIDIA BlueField-3 SmartNIC with KVS processing on ARM cores. On cache miss, the ARM cores use RDMA following RAMBDA methodology to retrieve the missing key-value entry from host memory. Measured tail latency improvement: CXL-NIC reduces p99 latency by ~39% compared to the SmartNIC baseline. (Paper §7.2)
🍯
Insight: device-bias mode enables nanosecond-scale local access for cache hits, avoiding coherence overhead. Host-bias enforces coherence for misses, but CXL.cache's cache-coherent load/store is faster than setting up RDMA operations for remote data retrieval.

Problem 1 · easy
Why does CXL-NIC use NC-write (not CO-write) for Rx packet transfers? What's the latency impact?
Show answer
NC-write writes the packet directly to host memory, bypassing the device cache. CO-write would cache the packet in device cache in Owned state, forcing a cache-to-cache transfer when the host CPU reads it (CPU fetches from HMC across CXL instead of local DRAM). The paper measures 15% higher latency with CO-write for 64B packets due to (1) ownership acquisition overhead and (2) cross-CXL cache read. NC-write avoids both.Misconception: that caching is always faster. For write-once-read-once data (like packets), caching in the device adds latency by forcing cross-interconnect reads instead of local DRAM access.
Problem 2 · medium
Type-1 CXL-NIC stores the Tx tail pointer in host memory, polled by DMU via CO-read. Why not store it in a device register (like PCIe NICs) and have the host write to it via CXL.io?
Show answer
CXL.io (backward-compatible with PCIe MMIO) has high latency for small writes (>3 μs for a tail pointer update). Storing the tail in host memory and polling with CO-read has two benefits: (1) host CPU updates via fast ld/st (~100 ns); (2) CO-read caches the tail in device cache — subsequent polls hit locally (no CXL traffic) until the host updates it, at which point coherence invalidates the cached copy and the next poll fetches the new value. This creates event-driven signaling without MMIO overhead.Misconception: that device registers are the "right" place for control state. CXL.cache makes coherent host memory a faster option for frequently-updated state like ring buffer pointers.
Problem 3 · medium
The paper evaluates four memory layouts (L1–L4) for Type-2 CXL-NIC, placing Rx/Tx packet buffers in host or NIC memory. L4 (both in NIC memory) shows 71% higher latency than L1 (both in host). Why, given that NIC memory should be "closer" to the NIC?
Show answer
Two factors: (1) the CPU must access NIC memory over CXL.mem, which is slower than local DRAM (~5× higher latency for ld, §5.1); (2) D2D accesses in host-bias mode require coherence checks with the host cache on every access, nullifying the spatial-locality benefit. The optimal layout (Fig. 14) places the Rx descriptor ring, Rx packet buffer, and Tx packet buffer in NIC memory but uses device-bias mode for Tx packet reads (host never accesses them during normal operation) and NC-P for Rx completion (pushes RxD into host LLC), avoiding coherence overhead where possible.Misconception: that "on-device memory" is automatically faster. Without careful request selection and bias mode tuning, coherence overhead and remote access latency can make NIC-resident data slower than host-resident data.
Problem 4 · hard
CC-NIC [48] requires software cache flushes (CLFLUSH) to invalidate Tx packet buffers in device cache before the host CPU can reuse them. CXL-NIC avoids this with NC-read. Explain the difference and quantify the benefit.
Show answer
CC-NIC uses UPI, which lacks explicit cache line state control. When the device reads a Tx packet, it caches it. To reuse the buffer, the host must CLFLUSH to invalidate the device cache copy, then st new data. CLFLUSH is expensive (~300 ns per cache line on Intel Xeon). CXL-NIC uses NC-read for Tx packet transfer, which bypasses the device cache entirely — no copy is cached, so no flush is needed. The host can immediately st new data. CC-NIC mitigates flush overhead with a buffer recycling allocator, but it forces Rx/Tx to share a single global pool, limiting scalability. CXL-NIC's NC-read enables independent per-queue buffer pools with no flush overhead. Measured benefit: CXL-NIC achieves 37% lower latency than CC-NIC (Fig. 14).Misconception: that cache-coherent interconnects automatically eliminate software cache management. UPI provides coherence but not explicit cache line state control, leaving software responsible for cache flushes. CXL.cache's NC/CO/CS/NC-P primitives enable hardware-only coherence management.
Problem 5 · medium
CXL-NIC's KVS uses device-bias mode for NIC-resident data and host-bias for misses. What happens if the host CPU updates a key in NIC memory while device-bias is active? How does the paper handle this?
Show answer
Device-bias mode skips coherence checks, so the NIC won't see the host's update — it will keep reading stale data from local NIC memory. The paper's design ensures this doesn't happen: the KVS request handler owns NIC-resident data (exclusive writer), and the host CPU only updates cold data in host memory. When the handler swaps a key from host to NIC memory (LRU replacement), it switches that region to device-bias, ensuring exclusive access. If the application requires host CPU writes to NIC memory (e.g., admin updates), the handler would need to (1) switch to host-bias mode, or (2) explicitly invalidate NIC-side copies before the host writes (software-managed coherence).Misconception: that device-bias is "automatic optimization." It's a contract: software promises the host won't touch device-bias regions, and the NIC skips coherence checks. Breaking that contract breaks correctness.
Problem 6 · hard
The paper claims CXL-NIC reduces packet latency by 49% vs. PCIe NIC, but uses an FPGA running at 400 MHz vs. a 2.5 GHz NVIDIA BlueField-3 SmartNIC. Is this a fair comparison? Defend your answer.
Show answer
It's fair for interconnect comparison, not absolute performance. The 49% reduction isolates the benefit of CXL.cache vs. PCIe DMA/MMIO by measuring the same datapath operations (descriptor read/write, packet transfer) on a CXL-NIC prototype. The comparison is CXL-NIC on Agilex-7 versus a BF-3 SmartNIC baseline using a custom packet generator methodology, not a direct FPGA-as-PCIe vs. FPGA-as-CXL comparison on identical hardware (the paper states an ideal same-FPGA PCIe baseline was not possible). The FPGA's lower clock frequency increases absolute latency equally for the operations being measured, but the relative reduction reflects interconnect protocol differences. For production comparison, the paper also compares CXL-NIC to CC-NIC [48] (37% reduction), which uses coherent UPI on the same Intel Xeon platform, validating that CXL.cache's explicit state control beats UPI. BF-3's ARM cores and RDMA engines are orthogonal to the interconnect — CXL-NIC could also integrate accelerators.Misconception: that clock frequency alone determines latency. Interconnect latency (DMA setup, MMIO round-trip, coherence protocol) dominates clock cycles for small transfers. CXL.cache's protocol efficiency is the win, not raw hardware speed.
Problem 7 · easy
Why does the paper use NC-P (not just NC-write) for Rx descriptor completion in Type-2 CXL-NIC? What's the latency benefit?
Show answer
After updating the RxD in NIC memory, NC-P pushes a copy into the host LLC, so the host CPU's poll hits the LLC (fast) instead of fetching from remote NIC memory (slow, ~5× higher latency per §5.1). Without NC-P, the host would issue a remote ld to NIC memory on every RxD poll. With NC-P, the first access hits LLC. However, NC-P risks overwhelming the LLC under high load ("leaky DMA"), so the paper adds adaptive gating: the host disables NC-P when LLC utilization is high, reverting to NC-write.Misconception: that NC-P is always better. It reduces host access latency but consumes LLC capacity. The paper's gating mechanism balances latency and cache pressure dynamically.
Problem 8 · medium
The paper's KVS offload (§5.4) achieves 39% lower tail latency than RDMA-based SmartNIC. Break down the latency sources and explain where CXL-NIC wins.
Show answer
RDMA-based design (BF-3): ARM cores process KVS request, issue RDMA READ to host memory on cache miss following RAMBDA methodology. Latency components include request parsing, RDMA setup, and retrieving the missing entry from host. The paper reports aggregate median/p99 latency results showing the SmartNIC baseline. CXL-NIC: hardware request handler parses request, issues D2H CO-read to host memory (host-bias) for cache miss, constructs response. The paper reports that CXL-NIC achieves ~39% lower p99 tail latency. CXL wins by using cache-coherent load/store for host memory access, avoiding RDMA setup overhead and simplifying the data retrieval path compared to the network-based RDMA approach.Misconception: that RDMA is always "low-latency I/O." RDMA is optimized for bandwidth over networks. CXL.cache's cache-coherent load/store provides a more direct path for small, frequent accesses to host memory in the same system.
Quick answers

Frequently asked questions

Ten questions newcomers ask most.

Q1What's the key difference between CXL.cache and PCIe DMA?▾
CXL.cache uses cache-coherent load/store semantics with explicit cache line state control (NC, CO, CS, NC-P), bypassing DMA setup overhead and software cache management. PCIe DMA requires descriptor preparation, memory pinning, and software cache flushes/invalidations. For small transfers (e.g., 64B descriptors), CXL.cache is 69–81% faster (§3.2).
Q2Why does PCIe account for 45–52% of packet latency if it's "high bandwidth"?▾
PCIe's bandwidth (PCIe 5.0 ×16 = 128 GB/s aggregate bidirectional, ~64 GB/s per direction) helps bulk transfers, but latency-per-operation matters more for networking. DMA setup (descriptor write, doorbell, completion poll) adds ~1–3 μs per descriptor; MMIO doorbells add >3 μs. At 100 Gbps, packets arrive every few hundred nanoseconds — microsecond-scale operations become the bottleneck.
Q3How does CO-read enable "event-driven" Tx without interrupts?▾
CO-read caches the Tx tail pointer in device cache in Owned state. Subsequent polls hit locally (no CXL traffic) until the host CPU updates the tail via st. CXL coherence invalidates the cached copy, and the next poll fetches the new value from host memory. This creates hardware-managed event notification: the NIC "sleeps" locally until the host "wakes" it via a coherence invalidation. Zero interrupts, zero MMIO. (§4.2)
Q4What's the advantage of CXL.cache over Intel UPI (used by CC-NIC)?▾
CXL.cache provides explicit cache line state control (NC, CO, CS, NC-P), enabling fine-grained optimizations like bypassing cache for write-once data (NC) or pushing data into host LLC (NC-P). UPI lacks this — it only supports coherent ld/st, forcing software cache flushes (CLFLUSH) to manage device cache pollution. CXL-NIC achieves 37% lower latency than CC-NIC by eliminating flushes with NC-read. (§7.2, Fig. 14)
Q5Why does Type-2 CXL-NIC place some data in NIC memory if host memory is faster?▾
Host memory is faster for CPU access, but NIC-local access to NIC memory (device-bias mode) is fastest for the NIC (~200 ns vs. ~800 ns for host memory). The paper co-optimizes placement: Rx/Tx buffers in NIC memory with device-bias for local hits; descriptor rings in host memory to avoid cross-CXL CPU access. Device-bias enables fast NIC-local processing; CXL.mem enables seamless host access on misses. (§5.2)
Q6What's NC-P and why is it unique to CXL?▾
NC-P (non-cacheable push) writes a cache line to device memory and pushes a copy into the host LLC, then invalidates the device cache copy. This pre-populates the LLC for fast host access. Neither PCIe nor UPI provide equivalent primitives — DDIO can inject I/O data into LLC, but it's DMA-based (non-coherent) and lacks fine-grained control. CXL-NIC uses NC-P for Rx completion, cutting host access latency by avoiding remote fetches from NIC memory. (§5.3)
Q7How does device-bias mode work, and when is it safe?▾
Device-bias mode skips host cache coherence checks on D2D accesses to NIC memory, enabling fast local access (~200 ns vs. ~800 ns in host-bias). It's safe when the host CPU doesn't access that memory region. CXL-NIC's KVS uses device-bias for NIC-resident cache hits (NIC is the exclusive writer) and switches to host-bias for misses that fetch from host memory. Breaking the contract (host writes to device-bias regions) breaks correctness — software must coordinate. (§5.1, §5.4)
Q8Can CXL-NIC support multiple queues or just one Rx/Tx pair?▾
CXL-NIC supports multiple queues by replicating DMU-PFU pairs, each with unique descriptor ring addresses. The paper's FPGA prototype implements dual queues functionally. Throughput doesn't scale linearly due to FPGA clock (400 MHz) and CXL IP bandwidth limits, but ASIC implementations or higher-frequency designs would scale further. Unlike CC-NIC's shared buffer pool, CXL-NIC enables independent per-queue pools with no cache flush overhead. (§7.1)
Q9Why does the paper compare to NVIDIA BlueField-3 if it's an FPGA prototype?▾
BF-3 is a production PCIe-based SmartNIC, providing a realistic baseline for interconnect comparison (PCIe DMA/MMIO vs. CXL.cache). The FPGA's lower clock frequency (400 MHz) affects absolute latency equally for both PCIe and CXL measurements, so the relative reduction (49%) isolates interconnect benefits. The paper also compares to CC-NIC [48] (Intel Xeon + UPI) for coherent interconnect comparison, showing 37% reduction. (§7)
Q10What are the limitations or open challenges for CXL-NIC?▾
Hardware: current CXL 1.1 devices lack atomic operations (CAS), limiting MPMC queue support. Security: CXL devices can access host memory via HPA, requiring IOMMU-equivalent protection mechanisms (CXL 2.0 HPA filters or MMU-like schemes). Virtualization: SR-IOV requires per-VF memory isolation and coherence-aware hypervisor management to prevent side channels. Performance: optimal data placement and bias mode tuning are workload-dependent — future work could explore dynamic placement policies. (§8)

Core terms
CXL-NICNetwork Interface Controller using CXL.cache (Type-1) or CXL.cache + CXL.mem (Type-2) instead of PCIe DMA/MMIO.
CXL.cacheCXL protocol for cache-coherent Device-to-Host memory access with explicit cache line state control (NC, CO, CS, NC-P).
CXL.memCXL protocol for Host-to-Device memory access with ld/st semantics; exposes device DRAM as coherent remote NUMA node.
DMU (Descriptor Management Unit)CXL-NIC hardware component that fetches, updates, and manages Rx/Tx descriptors.
PFU (Packet Forwarding Unit)CXL-NIC hardware component that moves packets between network interface and host/NIC memory.
DCOH (Device COHerence engine)CXL hardware that manages cache coherence, routes D2H/D2D requests, and maintains device cache (HMC + DMC).
NC / NC-P / CS / COCXL.cache request types: Non-Cacheable (bypass cache), NC-Push (push to LLC), Cacheable Shared (Shared state), Cacheable Owned (exclusive ownership).
Device-bias / Host-biasCXL.mem coherence modes: device-bias skips host cache checks (fast local access); host-bias enforces full coherence.
Supporting terms
DMA (Direct Memory Access)PCIe's bulk data transfer primitive; requires setup overhead (descriptor write, memory pinning, completion poll).
MMIO (Memory-Mapped I/O)PCIe's control signaling primitive; CPU st to device registers with >8 μs latency for small writes.
HMC / DMCHost Memory Cache / Device Memory Cache — partitions of CXL device cache storing data from host/device memory.
D2H / H2D / D2DDevice-to-Host (CXL.cache), Host-to-Device (CXL.mem), Device-to-Device (local NIC memory access) request directions.
DPDK (Data Plane Development Kit)User-space networking framework using polling instead of interrupts; bypasses kernel for low latency.
RxD / TxDRx/Tx Descriptor — metadata structure (~64 B) referencing packet buffer, carrying status flags and DMA addresses.
Ring bufferCircular queue of descriptors in host/NIC memory; CPU and NIC maintain head/tail pointers to coordinate.
Inline signalingMerging descriptor status check into descriptor read (one CXL request instead of two); borrowed from CC-NIC [48].
CLFLUSHx86 instruction to flush cache line; required by CC-NIC to invalidate device-cached Tx buffers before reuse; not needed with CXL NC-read.
RDMA (Remote Direct Memory Access)Network protocol for direct host-to-host memory access; SmartNIC baseline using RoCE over Ethernet.
KVS (Key-Value Store)Application-layer database mapping keys to values; paper implements on-NIC KVS with device-bias cache hits, host-bias misses.
