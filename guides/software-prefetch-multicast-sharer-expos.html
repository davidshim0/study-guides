MICRO 2025 · A Field Guide

When hundreds of cores share one cache, don't send the same data hundreds of times

As cores scale, naive coherence floods the network-on-chip with duplicate copies of shared data. This is the story of Software Prefetch Multicast (SPM), a software-hardware co-design that exposes sharers to send once and deliver to all.

📄 "Software Prefetch Multicast" · Chen et al., MICRO'25 🎓 Assumes familiarity with cache coherence, NoC basics ⏱ ~25 min read
📡

When software knows who will share data, tell the hardware — let one prefetch request trigger one multicast response to all sharers, not N unicast copies flooding the network.

Before you begin

Eight concepts the paper builds on

Each card explains one building block. Read in order first time, or jump to what looks unfamiliar.

shared data / multicast SPM machinery redundant traffic / problems baseline / status quo
1Shared data in multi-threaded programs▾

Multi-threaded programs often have threads reading the same data (e.g., an input matrix in matrix multiplication). Each thread requests its own copy from the cache hierarchy. Under standard MESI coherence, the LLC sends N separate responses—same address, same data, N times through the network-on-chip (NoC).

The paper focuses on 16-core and 64-core systems where this redundant traffic becomes a bottleneck as core count grows.

Now you're ready: SPM targets exactly this scenario—multiple cores requesting identical data—and collapses N responses into one multicast.
2Network-on-chip (NoC) bandwidth pressure▾

In a manycore processor, all coherence messages traverse a mesh NoC connecting tiles (each tile: core + private L1/L2 + LLC slice). As core count scales, coherence traffic scales—and shared data accesses create redundant copies traveling the same paths.

The paper shows that doubling cores (16→64) nearly doubles average load latency (Fig. 1), and that wider links help but don't eliminate congestion.

Now you're ready: SPM's payoff is bandwidth savings—42% average in 16-core, 50% in 64-core systems—by eliminating redundant responses.
3Request coalescing vs response multicast▾

Coalescing combines multiple pending requests for the same address into one LLC fetch. Multicast sends one response to multiple destinations. Both save bandwidth, but coalescing requires requests to arrive close in time—hard when threads vary by thousands of cycles (thread variation, Fig. 3).

Earlier work (Stream Floating, Push Multicast) extends the observation window, but can't reliably capture all sharers.

Now you're ready: SPM sidesteps the timing problem by having software tell hardware who the sharers are, enabling multicast even when requests arrive far apart.
4Thread variation and the moving leader▾

Even when threads execute the same loop, their progress differs—one may lag hundreds of thousands of cycles behind another (Fig. 3). The leading thread (first to issue a request) changes over time.

This variation makes it impossible for one thread's request to predictably trigger a multicast that serves all others—some may not have reached that request yet.

Now you're ready: SPM uses prefetching (decouples data delivery from demand) and a dynamic leader-switching algorithm to handle this variation.
5Software prefetching basics▾

A software prefetch instruction hints the hardware to fetch data before a demand load needs it, hiding latency. The compiler or programmer inserts prefetches ahead of actual accesses (at a chosen prefetch distance).

Standard software prefetch is per-thread and unaware of sharing—16 threads prefetch the same shared block 16 times.

Now you're ready: SPM extends software prefetch with a sharer group ID, letting the hardware know "these threads all need this block" and triggering multicast instead of N separate fetches.
6The LLC directory and coherence state▾

The LLC keeps a directory tracking which cores hold copies of each cached block and in what state (MESI). When a request arrives, the LLC consults the directory to send the right coherence messages.

Push Multicast (prior work) uses directory sharers to predict multicast destinations, but loses this info on eviction (unless silent eviction is supported) and can't multicast on first access.

Now you're ready: SPM stores sharer groups in a separate structure (the Share Map) configured by software, not reliant on directory state—so it works on first access and with any eviction protocol.
7Prefetch distance and timeliness▾

A prefetch issued too late arrives after the demand load (no benefit). Too early risks cache pollution or eviction before use. The prefetch distance (how many loop iterations ahead) must balance these.

SPM's leader/followers model has followers wait for multicast data—so even if the leader prefetches early, followers don't demand-miss; they just wait for the multicast to arrive.

Now you're ready: SPM picks a prefetch distance large enough to install data before demand, but not so large it pollutes the cache—typically a few iterations ahead.
8Group IDs and OS management▾

SPM assigns each set of sharers a Group ID (allocated by the OS). At the start of a parallel region, the OS supplies Group IDs for sharer groups, and threads attach the Group ID to their sharer-exposed prefetch instructions.

The OS limits the number of concurrent groups (8 in 16-core, 32 in 64-core systems). When IDs are exhausted, the system falls back to normal software prefetching—no functional correctness impact, just no multicast benefit.

Now you're ready: The Group ID is the software-hardware interface that communicates "these threads share this data" to the LLC and private caches.

The whole story in one place

SPM in a nutshell

The problem, the software-hardware interface that fixes it, and the measured impact.

When K threads share data (e.g., an input matrix in matrix multiplication), standard coherence sends K separate responses from the LLC—same address, same data, K paths through the NoC. As core count scales (16→64), this redundant traffic creates severe bandwidth pressure, nearly doubling load latency (Fig. 1).

The paper's fix, Software Prefetch Multicast (SPM), is a software-hardware co-design that exposes sharers to enable bandwidth-efficient multicast. Software identifies shared data in the source code, forms sharer groups, and inserts sharer-exposed prefetch instructions carrying a Group ID. The hardware extends the LLC with a Share Map (storing Group ID → leader + followers) and private caches with a leader/followers lookup and follower waiting table.

At runtime, a configuration stage broadcasts sharer groups to the LLC, which assigns one core as leader (closest to LLC by default). The leader's prefetch request triggers a multicast response to all recorded sharers; followers block redundant prefetch requests and wait. A timeout mechanism (512 cycles for 16 cores, 1024 for 64) handles lagging leaders—if no multicast arrives in time, the follower sends its own request (unicast response). A leader-switching algorithm detects persistent timeout patterns (cycle counter exceeds 16384) and reassigns the leader to a faster follower.

The payoff: on 7 multi-threaded workloads (cachebw, multilevel, backprop, conv3d, mlp, mv, particlefilter), SPM achieves an average of 42% NoC bandwidth saving in 16 cores (50% in 64 cores), a geomean of 1.28× speedup (1.38× in 64 cores), with a maximum of 1.46× (1.86× in 64 cores). Hardware overhead: ~1048 bytes per tile/core in the 64-core configuration (0.1% of 1 MB LLC per core), with no critical-path changes.

Key takeaways

  • Shared data accesses create redundant traffic: N requests → N responses for the same data.
  • Software has precise sharing info that hardware can't easily predict—SPM exposes it via Group IDs.
  • Leader/followers strategy: one core sends the request, others wait for multicast—avoiding redundant requests.
  • Timeout + leader switching handle thread variation: if leader lags, followers timeout and a new leader is chosen.
  • 42% NoC bandwidth saving (16 cores), 50% (64 cores); geomean 1.28× speedup (1.38× in 64 cores).
  • Beats Push Multicast (prior speculative multicast) by working on first access and avoiding in-network filtering.
  • Lightweight: ~1048 B per tile in 64-core, <0.2% LLC area.
Seven everyday pictures

How to think about SPM

Each analogy stands alone—read in any order. Every card ends with where the picture breaks.

The core idea · Multicast vs unicast

One announcement, many listeners

A school principal makes an announcement over the PA system—one message, all classrooms hear it. Without multicast, the principal would walk to each classroom individually and repeat the announcement. SPM is the PA system for shared data: the LLC sends one multicast response to all sharers (cores in the same group) instead of walking the NoC to each core separately with a unicast copy.

Where it breaks: a PA system broadcasts to everyone; SPM multicasts only to the recorded sharers in the Share Map, not all cores—it's a targeted multicast, not a global broadcast.
Software-hardware interface · Group ID

Mailing list subscription

You subscribe to a mailing list by giving the server your email address. When someone posts, the server sends one email to everyone on the list. SPM's Group ID is the subscription: software tells the hardware "these cores {0,1,8,15} are on the list for shared matrix X." When the leader prefetches X, the LLC sends one multicast to all subscribers instead of N separate messages.

Where it breaks: a mailing list is static once subscribed; SPM's sharer groups are configured per parallel region and can change as threads fork/join or context-switch. The OS manages Group ID allocation and deallocation dynamically.
Leader/followers · Representative request

One person orders pizza for the group

Four friends want the same pizza. Instead of all four calling the restaurant (clogging the phone line), they pick one person to place the order and deliver to everyone. SPM's leader core is that person: it sends the prefetch request on behalf of all sharers in the group. The followers don't call (block their requests) and just wait for the pizza (multicast data) to arrive.

Where it breaks: the pizza orderer is chosen once; SPM's leader can change during execution (leader switching) if the current leader is too slow—like switching who orders if the first person falls asleep before calling.
Timeout mechanism · Fallback plan

Wait for the bus, but not forever

You're waiting for a friend to pick you up, but if they don't arrive in 15 minutes, you call a taxi yourself. SPM's timeout is that 15-minute limit: a follower waits up to 512 cycles (16 cores) for the leader's multicast. If it doesn't arrive (leader is lagging), the follower sends its own request and gets a unicast response. Not ideal, but better than waiting indefinitely and stalling.

Where it breaks: calling a taxi doesn't help the friend; SPM's timeout can trigger leader switching (if timeouts repeat)—the LLC detects the pattern and reassigns the leader to a faster follower, so future requests succeed.
Thread variation · Uneven runners

Marathon runners at different paces

Five runners start a marathon together, but one pulls ahead while another lags behind. At mile 10, they're spread across thousands of meters. Thread variation is this spread: even executing the same loop, one thread may issue a request thousands of cycles before another (Fig. 3). SPM handles this by using prefetching (fetch data ahead of demand, so the fast thread's prefetch serves slow threads) and leader switching (if the leader is the slow runner, promote a faster one).

Where it breaks: marathon runners don't wait for each other; SPM's followers do wait (up to timeout) for the leader's multicast. The waiting acts as a soft synchronization, trading some latency for bandwidth efficiency.
Configuration stage · Seating chart

Setting up a classroom seating chart

Before class starts, the teacher assigns students to groups and writes the group roster on the board. SPM's configuration stage is that setup: before the parallel region executes, each sharer broadcasts a config_req to all LLC slices. The LLC records the group (Share Map) and assigns roles (leader/followers), then multicasts config_rsp back. Now everyone knows who's in which group and who's the representative—like students knowing their group assignments before group work begins.

Where it breaks: a seating chart is permanent for the semester; SPM's configuration happens per parallel region and can reconfigure on context switch or when a process forks—it's dynamic, not set-once.
Why expose sharers · Inside knowledge

The guest list vs guessing who'll show up

Planning a party: you either have a guest list (software knows exactly who accepted) or you guess based on who showed up last time (hardware predicts from directory). SPM is the guest list—software analyzes the code (e.g., "all threads in this loop read matrix X") and tells hardware the precise sharers. Prior work (Push Multicast) guesses from directory history, which fails on first access and loses info on eviction. SPM's list is complete and works from the start.

Where it breaks: a guest list is known upfront; SPM requires predictable sharing patterns (e.g., input matrix accessed by all threads)—it works for regular parallel loops, not for unpredictable, input-dependent sharing where only profiling could reveal sharers.
The full mechanism

How it actually works

Section by section, with diagrams from the paper.

1 · The problem — shared data floods the network

In a manycore processor with hierarchical cache (private L1/L2, shared LLC slices on a mesh NoC), coherence messages between private caches and LLC traverse the network. As core count scales (16→64), shared data accesses create redundant traffic: K threads request the same block → K separate responses from LLC, same data traveling K paths through the NoC.

The paper shows (Fig. 1) that doubling cores (16→64) nearly doubles average load latency (e.g., cachebw: ~100 cycles → ~200 cycles with 128-bit links). Wider links (512-bit) help but don't eliminate congestion. The root cause: repeated bandwidth consumption for identical data. In benchmarks like cachebw and multilevel, 10–80% of LLC traffic is shared data.

🔍
Concretely: mlp (matrix multiplication) has 16 threads all reading the same input matrix. Standard coherence sends 16 copies of each cache line from LLC → 16 private caches. SPM sends 1 multicast. On a 64-core system, this scales to 64 copies vs 1—hence the 50% bandwidth saving.

2 · Sharer-exposed prefetch instruction

SPM introduces a new instruction: prefetch_shared(address, group_id). The Group ID is allocated by the OS (one ID per sharer group in the parallel region). When executed, it generates a load request carrying the Group ID, sent to the cache hierarchy.

Software inserts these instructions by identifying shared loads (address invariant across threads within a parallel region, e.g., load(X + inner) where inner is the loop index but X is shared). All threads accessing the same shared data get the same Group ID. The paper manually inserts them; future work could automate via compiler (Section 6).

🍯
Key insight: software has precise sharing info that hardware can't reliably predict. Exposing it via Group ID eliminates prediction overhead and enables multicast on first access (unlike Push Multicast, which needs directory history).

3 · Configuration stage — setting up sharers

Before the parallel region executes, each thread (sharer) broadcasts a config_req with its Group ID to all LLC slices. The LLC allocates a Share Map entry: Group ID → leader core, follower cores (bit map), and a demand counter (number of expected config_req). When the LLC receives config_req from all sharers (counter reaches total), it determines the leader (by default, the core physically closest to the LLC, to minimize latency) and multicasts config_rsp with the leader ID back to all sharers.

Private caches store the leader/followers roles in a Leader/Followers Lookup Map (one entry per LLC slice, per Group ID). After configuration, the system knows: for Group ID 0, Core 8 is the leader for LLC_0, and Cores {0,1,15} are followers.

4 · Leader/followers strategy — avoiding redundant multicast

When a sharer-exposed prefetch misses in L2, the private cache checks its Leader/Followers Lookup Map: am I the leader for this LLC slice? If yes, send the request. If no (follower), block the request and insert it into the Follower Waiting Table with a cycle counter reset to 0. The follower waits for the multicast response.

The LLC, upon receiving the leader's prefetch request, looks up the Share Map, finds all sharers for that Group ID, and sends one multicast response (data in shared state) to all recorded cores. This avoids N separate requests and N unicast responses.

💡
Why not just assign prefetch to one thread in software? Because the leading thread changes (Fig. 3). A fixed assignment would miss fast threads or lag slow ones. SPM's dynamic leader (closest to LLC by default, switches on timeout) adapts to thread variation.

5 · Timeout mechanism — handling lagging leaders

If the leader is slow, followers wait indefinitely—bad for performance. SPM sets a timeout threshold (512 cycles for 16 cores, 1024 for 64 cores, tuned empirically). Each follower increments a cycle counter for waiting requests. When counter exceeds timeout, the follower pops the request from the Follower Waiting Table and sends it to the LLC as a normal request, triggering a unicast response only to that follower.

Timeout events are not failures—they're a fallback to avoid stalling. Fig. 15 shows that SPM satisfies most requests without timeout (except backprop and mlp, which have high thread variation). Frequent timeouts indicate the leader is lagging, triggering the next mechanism.

6 · Leader switching — adapting to moving leaders

When an LLC receives a timeout request, it checks the Share Map's cycle counter for that group. If the counter exceeds the switch threshold (16384 cycles, ~8 µs at 2 GHz), the LLC triggers a leader switch: (1) multicast the data to all sharers (fulfilling the request), (2) reassign the leader to the timeout requester (the faster follower), (3) send switch_req to both old and new leaders. Both update their Leader/Followers Lookup Maps.

The new leader checks its Follower Waiting Table—if a pending request matches the LLC that sent switch_req, it marks it "Issue Now" and sends the request on behalf of all followers. Future prefetches now come from the faster thread.

🍯
Why switch? Fig. 3 shows the leading thread moves (Core 1 → Core 5 across iterations). A static leader (Core 8, chosen by distance) may not stay fastest. Switching tracks the actual leading thread, keeping multicast timely.

7 · Hardware overhead

SPM's hardware modifications are lightweight. Per LLC slice in the 64-core configuration: Share Map (32 entries, each ~76 bits) = 304 B. Per L2 private cache: Leader/Followers Lookup Map (32 entries, ~70 bits each) = 280 B, plus Follower Waiting Table (64 entries, 58 bits each) = 464 B. Total per tile/core: 304 B + 280 B + 464 B = 1048 B (~0.1% of 1 MB LLC per tile). A 64-core system spends <0.2% of total LLC area. The paper notes light hardware overhead with no critical path additions—configuration is off the main access path, and comparators for timeout/switching logic don't slow normal cache hits.

Follow the numbers

Worked examples

Four scenarios from the paper, walked through step by step.

① Configuration stage for a 4-core group

Threads {0,1,8,15} share data X in a 16-core system. Walk through the configuration stage.

  1. Each thread executes a configuration instruction with Group ID 0 (allocated by OS). Their L2 private caches broadcast config_req(requester_id, group_id=0, total_sharers=4) to all LLC slices.
  2. LLC_0 receives the first config_req from Core 0. It allocates a Share Map entry: group_id=0, leader=0 (tentative), followers={}, demand_config=4.
  3. LLC_0 receives config_req from Cores 1, 8, 15. For each, it checks distance: Core 8 is closer to LLC_0 (2 hops) than Core 0 (4 hops). LLC updates: leader=8, followers={0,1,15}. Demand counter reaches 4.
  4. LLC_0 multicasts config_rsp(leader=8) to {0,1,8,15}. Each private cache sets its Leader/Followers Lookup Map: Core 8 marks itself leader for LLC_0; Cores 0,1,15 mark themselves followers.
  5. Result: Core 8 will send prefetch requests for Group 0 data stored in LLC_0; Cores 0,1,15 will block and wait.
🍯
Insight: the LLC picks the physically closest core as default leader to minimize request latency. This is just the initial assignment—leader can switch if Core 8 turns out to be slow.

② Leader prefetch triggers multicast

Core 8 (leader) executes prefetch_shared(X, group_id=0). Core 0 (follower) executes the same prefetch 1000 cycles later. What happens?

  1. Core 8's prefetch: misses in L2, checks Leader/Followers Map → "I'm leader for LLC_0." Sends request to LLC_0 with Group ID 0.
  2. LLC_0 receives request: looks up Share Map for Group 0 → finds {leader=8, followers={0,1,15}}. Fetches data from memory (or lower cache), sends multicast response (5 flits, one data packet) to all 4 cores.
  3. Core 0's prefetch: arrives 1000 cycles later. L2 checks Map → "I'm follower." Inserts request into Follower Waiting Table, resets cycle counter to 0. Blocks the request from going to LLC.
  4. Core 0 receives multicast: the data arrives (from step 2), matches the waiting request. Core 0 pops the request from the table, installs data in L2. No need to send its own request.
  5. Result: 1 request + 1 multicast instead of 2 requests + 2 unicasts. If timeout (512 cycles) had occurred before multicast arrived, Core 0 would've sent its own request (unicast response), but here the leader was fast enough.
💡
Why the follower waits: the leader's multicast is already en route. Sending a redundant request would waste NoC bandwidth. The Follower Waiting Table holds the request until the multicast arrives or timeout triggers.

③ Timeout and unicast fallback

Same setup, but Core 8 (leader) is lagging—it hasn't issued the prefetch yet. Core 0 (follower) executes the prefetch and waits. What happens after 512 cycles?

  1. Core 0's prefetch: misses L2, checks Map → follower. Inserts into Follower Waiting Table, cycle counter = 0.
  2. Cycles elapse: no multicast arrives (Core 8 hasn't sent the request yet). Cycle counter increments every cycle.
  3. Cycle 512: counter exceeds timeout threshold. Core 0 pops the request from the table, generates a timeout request with a timeout flag, sends it to LLC_0.
  4. LLC_0 receives timeout request: checks Share Map cycle counter for Group 0. If counter < switch threshold (16384), LLC sends a unicast response only to Core 0 (not a multicast). Increments the group's cycle counter.
  5. Core 0 receives data: installs it in L2, proceeds. Other followers (Cores 1, 15) are still waiting—they may also timeout and trigger unicasts, or Core 8 may eventually send the leader request, triggering a multicast for them.
  6. Result: fallback to unicast to avoid indefinite stalling, but bandwidth efficiency is lost for this request. If timeouts repeat (counter > 16384), leader switching kicks in (next example).
🔍
Fig. 15 shows timeout breakdown: in backprop and mlp, ~20–30% of requests timeout (high thread variation). In cachebw, multilevel, conv3d, <5% timeout—leader is keeping up.

④ Leader switching reassigns the leader

Core 8 (leader) is consistently slow; Core 0 (follower) has timed out multiple times. The LLC's cycle counter for Group 0 exceeds 16384 cycles. Core 0 times out again. What happens now?

  1. Core 0 sends timeout request: (as in example 3). LLC receives it, checks Share Map cycle counter for Group 0 → counter = 17000 (exceeds switch threshold 16384). Switch event triggered.
  2. LLC switches leader: updates Share Map: old_leader=8, new_leader=0, followers={1,8,15}. Multicasts two messages: (a) data response to all sharers (fulfilling Core 0's request), (b) switch_req(new_leader=0) to Cores 0 and 8.
  3. Core 0 receives switch_req: updates Leader/Followers Lookup Map → "I'm now leader for LLC_0." Checks Follower Waiting Table—if a pending request matches LLC_0, marks it "Issue Now" and sends the request on behalf of all followers.
  4. Core 8 receives switch_req: updates Map → "I'm now follower for LLC_0." Future prefetch requests will be blocked and wait.
  5. Future prefetches: Core 0 (new leader) sends requests; Cores 1, 8, 15 (followers) block and wait for Core 0's multicast. Core 8 is no longer burdened with sending requests it can't keep up with.
  6. Result: the leader role moves to the faster thread (Core 0), reducing future timeouts and improving multicast efficiency.
🍯
Why 16384 cycles? Sensitivity study (Fig. 20) shows performance stable across 4096–65536 cycles—16384 is a middle ground that avoids premature switching (thrashing) while catching persistent lags. The key is the timeout threshold (512/1024) does the heavy lifting; switch threshold just confirms the leader is consistently slow.
Test yourself

Practice problems

Try each before opening the answer. Every answer ends with the misconception it checks.

Problem 1 · easy
A 16-core system has threads {0,1,8,15} sharing data X (Group ID 0). Core 8 is designated leader for LLC_0. Why does Core 8 send the prefetch request, not all four cores?
Show answer
To avoid redundant multicast traffic. If all four cores sent requests, the LLC would receive 4 requests and send 4 multicast responses (16 total deliveries for the same data). The leader/followers strategy ensures only one request triggers one multicast, while followers block their requests and wait. This collapses 4 requests + 4 unicasts (or 4 multicasts) into 1 request + 1 multicast—the core bandwidth savings.Misconception: that all sharers send requests and the LLC coalesces them. Coalescing requires requests to arrive simultaneously (hard with thread variation). SPM avoids that timing dependency by having followers not send requests at all—they wait for the leader's multicast.
Problem 2 · medium
Core 0 (follower) executes a sharer-exposed prefetch, finds no entry in its Leader/Followers Lookup Map for Group ID 0. What happens, and why might this occur?
Show answer
If the Lookup Map has no entry for Group 0, the prefetch instruction behaves as a normal software prefetch—it sends a request to the LLC, which responds with a unicast. No multicast, no blocking. This can occur if: (1) the OS couldn't allocate a Group ID (table full), (2) the configuration stage hasn't completed yet (race condition), or (3) a context switch wiped the Lookup Map and the new thread hasn't reconfigured yet. In all cases, functional correctness is preserved (data still arrives), but the multicast optimization is lost for that access.Misconception: that missing configuration crashes or causes errors. SPM gracefully falls back to normal prefetch—Group ID absence or missing entries just mean "don't multicast," not "fail."
Problem 3 · medium
The LLC chooses the leader by picking the core physically closest to the LLC. For Group 0, Core 8 (2 hops) is chosen over Core 0 (4 hops). Why distance, and what happens if Core 8 is actually the slowest thread?
Show answer
Distance minimizes the request latency—a closer core's request reaches the LLC faster, triggering multicast sooner. But distance is just the initial heuristic; it doesn't predict thread speed. If Core 8 turns out to be slow (lags the loop), followers will timeout waiting for its prefetch. The leader switching mechanism detects this (cycle counter > 16384 after repeated timeouts) and reassigns the leader to a faster follower (e.g., Core 0). So the distance-based choice is a starting point that adapts to actual thread behavior.Misconception: that the leader is fixed. The leader role is dynamic—it switches to track the actual leading thread, which moves over time (Fig. 3).
Problem 4 · hard
SPM's timeout threshold is 512 cycles (16 cores) or 1024 cycles (64 cores). Too short and followers timeout unnecessarily; too long and they stall. Explain the tradeoff and why 512/1024 is chosen.
Show answer
The timeout is the max wait for the leader's multicast. If set too short (<512), the multicast may still be in-flight when timeout triggers—the follower sends a redundant request, gets a unicast, wastes bandwidth (defeats the purpose). If set too long (>1024), a lagging leader causes followers to stall for many cycles, hurting performance. The chosen values (512/1024) are empirically tuned (Fig. 19 sensitivity study): they're large enough to let the multicast arrive from most of the network (recall: mesh NoC, max ~10–20 hops + LLC latency + memory latency if miss), but not so large that a lagging leader causes significant stalls. The 64-core threshold (1024) is higher because the network diameter is larger (8×8 mesh vs 4×4), so multicast takes longer to reach distant cores.Misconception: that timeout is a failure. Timeout is a safety valve—a fallback when the leader is too slow. The goal is to set it so timeouts are rare (leader is usually fast enough), but they exist to avoid indefinite waiting. Fig. 15 shows SPM satisfies (no timeout) the majority of requests in most benchmarks.
Problem 5 · medium
SPM inserts prefetch instructions a fixed prefetch distance ahead (e.g., 8 iterations). Why not make the leader prefetch immediately when entering the loop, getting all data upfront?
Show answer
Prefetching too far ahead risks cache pollution and eviction before use. If the leader prefetches the entire shared array upfront, (1) it may not fit in the L2 cache (e.g., 256 KB private L2 vs multi-MB array), causing evictions, and (2) by the time a follower's demand load arrives, the prefetched data may have been evicted by other accesses, triggering a cache miss anyway—negating the prefetch benefit. A modest prefetch distance (e.g., 8 iterations) fetches data just-in-time: early enough to hide latency, late enough to avoid pollution. The paper picks distances large enough for multicast data to install before demand, but not so large the cache is polluted.Misconception: that more prefetching is always better. Over-prefetching wastes cache capacity and bandwidth. The optimal distance balances latency hiding and cache pollution—too aggressive can harm performance.
Problem 6 · hard
Push Multicast (prior work) multicasts data to sharers recorded in the directory, but requires silent eviction and loses sharer info on capacity misses. SPM stores sharers in a separate Share Map. Explain why this enables multicast on first access and with any eviction protocol.
Show answer
Push Multicast relies on the directory (which tracks coherence state) to know sharers. The directory is populated when cores request data—so on first access (compulsory miss), the directory is empty, and Push Multicast can't predict sharers (no multicast). It must wait for the data to be re-referenced after an eviction to rebuild sharers. Also, if a core's L2 evicts a block (capacity miss), a silent eviction protocol is needed to keep the directory sharer bit set; otherwise, the directory loses that sharer, and the next multicast misses it. SPM's Share Map is configured by software before execution—it knows the sharers from the source code, independent of runtime directory state. So on first access, the LLC already has the complete sharer list (via configuration stage), and multicast works immediately. And since the Share Map isn't tied to directory state, any eviction protocol (silent or not) is fine—sharers are preserved in the Share Map regardless of private cache evictions.Misconception: that hardware prediction (from directory) is as good as software exposure. Hardware can only observe past behavior (re-reference); software knows future intent (all threads will read this matrix). SPM's software-exposed sharers are complete and predictive, not reactive.
Problem 7 · medium
The Share Map in the LLC and the Leader/Followers Lookup Map in private caches together consume ~1048 bytes per tile/core in a 64-core system. Is this overhead per cache line or per sharer group?
Show answer
Per sharer group, not per cache line. The Share Map has 32 entries (in 64-core), each storing one Group ID and its sharers (~76 bits). The Leader/Followers Map also has 32 entries per Group ID (~70 bits). So ~1048 B supports up to 32 concurrent sharer groups (the OS limit). A group can cover many cache lines—e.g., in mlp, one group covers the entire input matrix (thousands of lines). The overhead scales with the number of distinct sharing patterns (groups), not the data size. That's why it's so cheap: in practice, most parallel regions have a small number of shared data structures (e.g., one input matrix, one weight matrix), so 32 groups is plenty.Misconception: that SPM adds overhead per cache line, like a sharer bit per line. The Group ID is carried in the request, not stored per line—the Share Map is a small lookup table indexed by Group ID. This is what keeps overhead under 0.2% of LLC.
Problem 8 · easy
SPM achieves 42% NoC bandwidth saving in 16 cores, 50% in 64 cores. Why does the saving increase with more cores?
Show answer
Because the number of redundant responses scales with core count. In a 16-core system with 16 threads sharing data, baseline coherence sends 16 unicast responses; SPM sends 1 multicast—saving 15 copies. In a 64-core system, baseline sends 64 unicasts; SPM sends 1 multicast—saving 63 copies. The ratio of saved traffic increases (15/16 = 93.75% vs 63/64 = 98.4%), but the measured NoC bandwidth saving (42% vs 50%) is lower because not all traffic is shared data—there's also non-shared data, writebacks, coherence control messages. The 50% saving in 64 cores reflects that a larger fraction of total traffic is shared data in these benchmarks when more threads are active, and SPM eliminates more of it.Misconception: that bandwidth saving should be the same regardless of core count. The saving depends on the fraction of traffic that's shared data and the number of sharers per group. More cores → more sharers → more redundant copies eliminated → higher bandwidth saving.
Quick answers

Frequently asked questions

Ten questions about SPM. Tap any to expand.

Q1What's the difference between SPM and Push Multicast?▾
Push Multicast (prior work, MICRO'25) predicts sharers from the directory (past requests) and multicasts on a demand load. SPM exposes sharers from software (via Group ID) and multicasts on a prefetch. Key differences: (1) SPM works on first access (no directory history needed), (2) SPM works with any eviction protocol (not just silent eviction), (3) SPM uses followers blocking requests instead of in-network filtering to avoid redundant multicast, (4) SPM uses prefetching to decouple multicast timing from demand, handling thread variation better. Fig. 14 shows SPM outperforms Push Multicast when data size > LLC (Push Multicast loses directory sharers on eviction; SPM doesn't).
Q2How does SPM handle workloads where sharing patterns aren't known statically?▾
SPM targets regular, predictable sharing—e.g., all threads read the same input matrix, common in AI/HPC. For irregular, input-dependent sharing (e.g., graph traversal where edges determine sharing), software can't statically identify sharers. Future work (Section 6) discusses profile-guided SPM: use runtime profiling to observe which threads access which data, form dynamic groups, and expose them. Alternatively, for unpredictable sharing, SPM gracefully falls back—missing Group ID configuration just uses normal prefetch, no multicast. Functional correctness is preserved; only the optimization is lost.
Q3Why does SPM use prefetching instead of just multicasting on demand loads?▾
Prefetching decouples data delivery from demand timing. With demand loads, the first thread to miss triggers the multicast—but if that thread is slow (thread variation), other fast threads have already missed and sent their own requests before the multicast arrives (no benefit). Prefetching lets the leader (not necessarily the first to demand) fetch data ahead of time, so by the time followers reach their demand loads, the data is already in cache (from the multicast). The Follower Waiting Table holds prefetch requests (not demand loads) while waiting for multicast—this soft synchronization aligns data delivery with all threads' needs, despite variation.
Q4What happens during a context switch or when a process forks?▾
On context switch: the outgoing thread sends a de-allocation message to its L2 and LLC, wiping the Leader/Followers Lookup Map and Share Map entries for its Group IDs. The incoming thread, if it has sharer groups, sends config_req to re-configure. If it's the same process, re-configuration is fast (the LLC may already have the Share Map from other threads). For hyperthreaded cores, each thread manages its own Group IDs independently. For fork: the OS allocates new Group IDs for the child process's parallel regions; configuration happens at the start of each parallel region. Section 3.7 details this—key point: de-allocation and re-configuration are designed to be cheap (<0.1 µs per config stage).
Q5Can SPM be combined with hardware prefetchers (e.g., Bingo, stride)?▾
Yes—SPM and hardware prefetchers are orthogonal. SPM targets shared data (exposed via Group ID); hardware prefetchers target per-thread access patterns (shared or not). Section 6 discusses combining them: SPM multicasts shared data (e.g., input matrix), hardware prefetchers handle non-shared data (e.g., thread-local weights). One open question: should a hardware prefetch request (with Group ID) trigger multicast? Possibly, but inaccurate prefetches could waste bandwidth—throttling would be needed. Fig. 13 shows L1Bingo-L2Stride outperforms SPM in some benchmarks (conv3d, mlp) because it prefetches both shared and non-shared data; combining them could capture both benefits.
Q6How does the OS allocate Group IDs, and what if it runs out?▾
The OS supplies Group IDs from a limited pool (8 in 16-core, 32 in 64-core—matching the Share Map capacity). When a process starts a parallel region and its sharer groups need IDs, the OS allocates available IDs. If the pool is exhausted, the system falls back to normal software prefetching for that parallel region—no configuration, no multicast, just regular L2 requests. Functional correctness is preserved; the optimization is lost. In practice, the OS limit (32 groups in 64-core) is rarely exhausted because most programs have a small number of distinct sharing patterns (e.g., 1–4 shared arrays per parallel region). The paper discusses OS management and ID release policies in Section 3.7.
Q7Why 16384 cycles for the switch threshold?▾
The switch threshold (16384 cycles, ~8 µs at 2 GHz) is chosen to avoid premature switching (thrashing) while catching persistent leader lag. Fig. 20 shows performance is stable across 4096–65536 cycles—the exact value isn't critical. The key is to let the leader have multiple chances (timeout events) before switching: one timeout might be a transient hiccup; repeated timeouts (cycle counter accumulating to 16384) indicate the leader is consistently slow. The threshold is high enough to avoid switching on noise, low enough to switch within the parallel region's lifetime (milliseconds to seconds).
Q8Does SPM work with non-inclusive or exclusive LLCs?▾
The paper's evaluation uses an inclusive LLC (data in L2 is also in LLC), which is common in Intel designs. The paper does not prove correctness or evaluate SPM's behavior with non-inclusive or exclusive LLCs. In principle, the Share Map and multicast logic might be adaptable (they are independent of inclusivity), but any required coherence protocol changes are not discussed. For exclusive LLCs (data in L2 is not in LLC), SPM would need to ensure the LLC can still coordinate multicast without redundantly caching data—an open question the paper leaves for future work.
Q9What about false sharing—threads writing to different words in the same cache line?▾
SPM targets read-shared data (e.g., input matrix where all threads read, none write). For false sharing (threads writing to different words in the same line, e.g., adjacent array elements), SPM doesn't apply—each write invalidates other cores' copies, and multicast would deliver stale data. The paper's workloads (mlp, conv3d, mv) have producer-consumer patterns (one thread writes, others read) or pure read-sharing. For false sharing, the solution is typically padding or privatization (each thread gets its own cache line), not multicast. SPM's sharer-exposed prefetch is for shared reads, not shared writes.
Q10Can this be automated by a compiler instead of manual insertion?▾
Yes—Section 6 discusses extending LLVM to automatically identify shared loads (by analyzing loop indices and thread IDs), form sharer groups, and insert sharer-exposed prefetch instructions. The compiler would: (1) identify loads where the address is invariant across threads (shared), (2) group threads by access pattern (all threads in the parallel region → one group), (3) insert prefetch_shared with Group ID and prefetch distance. The paper manually inserts to demonstrate feasibility; compiler automation would enable broader adoption. The challenge is choosing the right prefetch distance (requires profiling or heuristics) and handling complex sharing patterns (nested loops, conditionals)—active research direction.

Glossary & further reading

Every term in one place, then pointers to dig deeper.

Core terms
SPM (Software Prefetch Multicast)The paper's mechanism: software-exposed sharer groups + prefetch instructions trigger one multicast response to all sharers, saving NoC and LLC bandwidth.
Sharer-exposed prefetchA prefetch instruction carrying a Group ID (e.g., prefetch_shared(X, group_id=0)), telling hardware which cores need this data.
Group IDAn OS-allocated identifier (1–32) for a set of threads sharing data. The paper shows Group ID as a configuration parameter; its use as a sentinel (if any) is an implementation detail not fully specified.
Share MapLLC structure storing sharer groups: Group ID → leader core ID, follower cores (bit map), cycle counter, demand config count. 8 entries (16-core) or 32 (64-core).
Leader/Followers Lookup MapL2 private cache structure storing roles: Group ID → leader/follower bit per LLC slice. Tells the cache "am I leader or follower for this group on this LLC?"
Follower Waiting TableL2 private cache buffer (64 entries) holding blocked prefetch requests with cycle counters. When counter exceeds timeout, sends timeout request.
Timeout mechanismFollowers wait up to 512 cycles (16-core) or 1024 (64-core) for leader's multicast. On timeout, send request → unicast response.
Leader switchingLLC detects repeated timeouts (cycle counter > 16384), reassigns leader to faster follower, sends switch_req to both old/new leaders.
Supporting terms
Thread variationThreads executing the same code progress at different rates—one may lag thousands of cycles behind another (Fig. 3).
NoC (Network-on-Chip)Mesh interconnect connecting tiles (core + L1/L2 + LLC slice). 4×4 mesh (16 cores) or 8×8 (64 cores), 128-bit links, 2-stage routers.
LLC (Last-Level Cache)Shared, sliced, 1 MB per tile, 16-way set-associative, inclusive (paper's config). Handles coherence, multicasts data to sharers.
config_req / config_rspConfiguration messages: sharers broadcast config_req to LLC; LLC responds with config_rsp (leader ID) to configure roles.
Multicast responseOne data packet sent by LLC to multiple destinations (all sharers in Share Map), saving bandwidth vs N unicast responses.
Prefetch distanceNumber of loop iterations ahead to prefetch (e.g., prefetch X+inner+8 in iteration inner). Balances latency hiding and cache pollution.
Push MulticastPrior work (MICRO'25, Huang et al.): predicts sharers from directory, multicasts on demand load. Needs silent eviction; fails on first access.
Request coalescingCombining multiple pending requests for the same address into one LLC fetch. Hard with thread variation (requests arrive far apart).
Silent evictionA coherence protocol where evicting a block from L2 doesn't notify the LLC directory (sharer bit stays set). Needed for Push Multicast's directory-based prediction.
MESI protocolCache coherence protocol: Modified, Exclusive, Shared, Invalid states. SPM multicasts data in Shared state to all sharers.

Hardware overhead (Table 3, Section 4)

  • Share Map (LLC): 32 entries × 76 bits = 304 B per LLC slice (64-core).
  • Leader/Followers Lookup Map (L2): 32 entries × 70 bits = 280 B per L2 cache.
  • Follower Waiting Table (L2): 64 entries × 58 bits = 464 B per L2 cache.
  • Total per tile/core: 304 B + 280 B + 464 B = 1048 B (~0.1% of 1 MB LLC per tile). 64-core system: <0.2% total LLC area.
  • Critical path: The paper lists light hardware overhead and comparator logic; no explicit critical-path analysis is provided.

Thresholds (Table 1, Figs. 19–20)

  • Timeout threshold: 512 cycles (16-core), 1024 cycles (64-core). Sensitivity: 128–2048 cycles tested; 512/1024 optimal.
  • Switch threshold: 16384 cycles (~8 µs at 2 GHz). Sensitivity: 4096–65536 cycles tested; performance stable, 16384 chosen.
  • Configuration overhead: ~200 cycles round-trip latency (~0.1 µs) per parallel region.

Related work pointers

Multicast and coalescing

NYU Ultracomputer (Gottlieb et al., 1982), GPU packet coalescing (Kim et al., ICS'17), Stream Floating (Wang et al., HPCA'21)—coalesce requests or multicast responses. SPM differs by using software-exposed sharers, not hardware observation/prediction.

Coherence prediction

Push Multicast (Huang et al., HPCA'25): predicts sharers from directory, multicasts on demand. Kaxiras & Goodman (HPCA'99), Mukherjee & Hill (ISCA'98): predict coherence messages to accelerate protocol. SPM avoids prediction by exposing sharers from software.

Software prefetching

Callahan et al. (ACM SIGARCH'91): original software prefetch. Ainsworth & Jones (CGO'17): prefetch for indirect accesses. Jamilan et al. (EuroSys'22), Zhang et al. (ASPLOS'24): profile-guided timely prefetch. SPM extends software prefetch with Group ID for multicast.

Hardware prefetchers

Bingo (Bakhshalipour et al., HPCA'19): spatial data prefetcher. Stride, IMP (Yu et al., MICRO'15): indirect memory prefetcher. These are per-core and unaware of sharing—SPM targets shared data that hardware prefetchers duplicate.
