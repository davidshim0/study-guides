Inference on a budget: serving huge MoEs with tiny GPUs

Your 16 GB GPU can't hold Mixtral 8x7B's ~47 billion parameters. MoE-Lightning streams expert weights from CPU, overlaps everything, and reaches 10.3× higher throughput than existing systems. Here's how.

📄 Cao, Liu, et al. · ASPLOS '25 🎓 Assumes basic LLM + tiering background ⏱ ~25 min read
⚡

Pipeline GPU compute, CPU attention, and I/O so nothing waits for anything else — then choose the batch size that balances GPU memory bandwidth and CPU↔GPU I/O to maximize throughput.
CGOPipe + HRM deliver up to 10× the throughput of naive offloading.

Before you begin

Eight concepts the paper assumes

Open any card for a quick primer. Read them in order the first time, or jump to what you need.

GPU computation CPU computation MoE-Lightning's machinery Baseline systems
1Mixture of Experts (MoE)▾

An MoE model replaces the single dense feed-forward network in each transformer layer with a router and multiple expert FFNs. For each token, the router picks the top-k experts (e.g., 2 of 8) to run. This activates fewer parameters per token than a dense model of the same total size, cutting inference FLOPs while keeping capacity high.

Example: Mixtral 8x7B has 8 experts per layer; each token activates 2, so active parameters ≈ 13B while total parameters ≈ 47B. Much cheaper than a dense 47B model but needs far more memory than a dense 13B.

Now you're ready: MoE-Lightning targets serving these big sparse models when GPU memory is far smaller than total model size.
2Prefill vs decode stages▾

Prefill: process the initial prompt tokens (say, 512 tokens) in parallel to produce the first output token and cache all their key-value pairs. Typically compute-bound (lots of matmuls on many tokens at once). Decode: generate subsequent tokens one-at-a-time, each time using all cached KV pairs. Decode is memory-bound — each token does relatively few FLOPs but reads large weights and the entire KV cache.

For throughput-oriented batch inference, the decode stage dominates wall time because you generate many tokens (e.g., 64–256) sequentially.

Now you're ready: MoE-Lightning's optimizations focus on decode, where memory bandwidth and I/O are the bottleneck.
3Weight offloading — the standard hack▾

If the model doesn't fit in GPU memory, store weights in CPU DRAM (or disk). For each layer, stream the weights to GPU, compute, stream intermediate results back, repeat. This is slow — a T4 GPU has ~300 GB/s on-chip bandwidth but only ~32 GB/s CPU↔GPU PCIe — but it's the only way to run a 47B parameter model on a 16 GB GPU.

Naive offloading wastes time: the GPU idles waiting for weights, or the I/O path idles waiting for computation. Overlapping them efficiently is non-trivial.

Now you're ready: MoE-Lightning is about how to offload efficiently, not whether to offload (offloading is unavoidable in memory-constrained settings).
4Batch size and micro-batch size▾

Batch size N: total tokens in the batch (in MoE-Lightning's notation, N is measured in tokens, not requests). Larger N amortizes weight-transfer overhead, raising throughput — but also increases CPU memory (to store all KV caches) and GPU peak memory (during prefill). Micro-batch size μ: the chunk of N processed by one GPU kernel (e.g., μ = 128). Larger μ improves GPU utilization but raises GPU memory.

Choosing (N, μ) is a trade-off: too small and you underuse resources; too large and you run out of memory or saturate a different bottleneck (e.g., CPU attention latency).

Now you're ready: MoE-Lightning's optimizer searches for the (N, μ) that balances GPU memory bandwidth and CPU↔GPU I/O so both saturate together.
5KV cache and why it lives in memory▾

During decode, each new token's attention reads all previous tokens' key-value vectors. Recomputing them from scratch is wasteful, so they're cached. For Mixtral-style models with GQA, the KV cache per token is roughly 128 KB in fp16 (2 bytes × 32 layers × 8 KV heads × 128 head_dim × 2 for K and V). Across a batch with long generation (256 tokens each), the KV cache can dwarf the model weights themselves.

On a memory-constrained GPU, offloading KV cache to CPU is standard. The question is whether to transfer it to GPU for each attention or perform attention directly on CPU.

Now you're ready: MoE-Lightning does attention on the CPU, avoiding KV cache transfers and freeing I/O bandwidth for weight transfers (§3.3).
6The Roofline Model — compute vs memory bound▾

The Roofline Model plots achievable performance (FLOPs/s) against operational intensity I = FLOPs/byte. Two roofs: compute roof (peak FLOPs, horizontal line) and memory roof (bandwidth × I, diagonal). If I is high (lots of compute per byte), you hit the compute roof; if low (few FLOPs, many bytes), you hit the memory roof. The intersection is the "ridge point."

For decode-stage MoE FFN with batch size N and micro-batch μ, I increases with both (more tokens = more reuse of weights). The paper extends this to Hierarchical Roofline with CPU, GPU, and CPU↔GPU bandwidth, revealing multiple turning points.

Now you're ready: MoE-Lightning's performance model (HRM) uses these turning points to decide where computation should run (GPU vs CPU) and what (N, μ) to use.
7Pipeline bubbles and why overlap is hard▾

A pipeline "bubble" is idle time: the GPU waits for data, or the I/O path waits for the GPU to finish. In traditional DNN training pipelines, only activations transfer between stages. Here, both weights and intermediate results transfer, in opposite directions, with vastly different sizes (weights for one layer: ~GB; intermediate results for one micro-batch: ~MB).

Naive scheduling: transfer all weights for layer L, then compute all micro-batches. Result: the first micro-batch of layer L+1 waits a long time for the second weight transfer to complete, creating a bubble. MoE-Lightning's CGOPipe pages the weights and interleaves transfers to eliminate most bubbles.

Now you're ready: CGOPipe is the pipeline schedule (§4.1) that keeps GPU, CPU, and I/O busy simultaneously.
8Why existing systems fall short▾

FlexGen: prefetches KV cache to GPU and does attention on GPU, wasting I/O bandwidth that could transfer weights; its policy search doesn't account for shifting bottlenecks as you scale batch size. DeepSpeed-ZeRO: uses tiny batch sizes (often 1–2), so weight-transfer overhead dominates and throughput is low. FastDecode: overlaps CPU attention with GPU FFN but doesn't handle weight offloading (targets latency, not throughput).

None of them systematically model how GPU memory capacity, CPU↔GPU bandwidth, and CPU compute interact to define performance regions. MoE-Lightning does, via HRM.

Now you're ready: MoE-Lightning outperforms them by better pipelining (CGOPipe) and smarter policy search (HRM).

CGOPipeCPU-GPU-I/O pipeline: the fine-grained schedule that overlaps weight transfers (paged into chunks), intermediate-result transfers, GPU FFN, and CPU attention.
HRMHierarchical Roofline Model: extends the classical Roofline to multiple memory tiers (GPU HBM, CPU DRAM, CPU↔GPU link) and processors (GPU, CPU), identifying turning points where the bottleneck shifts.

The whole story in one place

MoE-Lightning in a nutshell

The problem, the two-part solution, and the payoff.

The problem: Serving a Mixture-of-Experts model like Mixtral 8x7B (~47B parameters) on a T4 GPU (16 GB) requires offloading expert weights to CPU. Existing systems (FlexGen, DeepSpeed-ZeRO) leave resources underutilized: the GPU idles waiting for weights, or the I/O path idles waiting for computation, or CPU attention becomes the bottleneck. As a result, throughput is far below what the hardware could deliver if everything overlapped perfectly.

The two-part fix: (1) CGOPipe, a pipeline schedule that pages weight transfers into small chunks and interleaves them with GPU computation and CPU attention so nothing waits. For example, while GPU runs the MoE FFN for micro-batch 1 of layer L, CPU runs attention for micro-batch 3 of layer L, and the I/O path transfers one page of weights for layer L+1 — all at once, no bubbles. (2) HRM, a Hierarchical Roofline Model that models GPU compute, CPU compute, GPU memory bandwidth, CPU↔GPU bandwidth, and CPU memory bandwidth as separate roofs. HRM identifies "turning points" (e.g., when batch size is so large that CPU attention latency dominates, or when micro-batch is so large that GPU memory bandwidth saturates) and searches for the (N, μ, device assignment) policy that balances GPU memory bandwidth and CPU↔GPU bandwidth for the modeled computation.

The payoff: On a single T4 GPU, MoE-Lightning achieves up to 10.3× higher throughput than FlexGen for Mixtral 8x7B (MTBench, gen_len=128, no padding). With padding (fixed-length batches), still 3.5× higher. On 4×T4 GPUs, super-linear scaling: 3.38× throughput gain over 2×T4 for Mixtral 8x22B. MoE-Lightning also uses 2–3× less CPU memory than FlexGen to reach the same throughput upper bound, because better I/O utilization means less need to over-batch.

🍯 Key takeaways

  • CGOPipe pages weights into n chunks (n = # micro-batches) and overlaps weight-page transfers with GPU FFN, CPU attention, and intermediate-result transfers — eliminating pipeline bubbles.
  • HRM extends the Roofline Model to heterogeneous systems: separate compute/memory roofs for GPU and CPU, plus a CPU↔GPU I/O roof. Finds the policy that balances GPU memory bandwidth and CPU↔GPU I/O for the modeled computation.
  • CPU attention is faster than transferring KV cache from CPU to GPU (3–4× lower latency on the paper's hardware), freeing I/O bandwidth for weights.
  • Turning points: HRM identifies when the bottleneck shifts (e.g., from I/O to GPU memory bandwidth, or to CPU attention latency) as you scale (N, μ).
  • Balance point: the (N, μ) where GPU memory bandwidth and CPU↔GPU I/O saturate for the modeled computation — HRM's optimizer targets this point.
  • Results: 10.3× (no padding) or 3.5× (with padding) vs FlexGen on 1×T4; super-linear scaling on multi-GPU; 2–3× less CPU memory for the same throughput.
  • Overhead: Performance model has near-zero runtime cost (vs hours/days of fitting for FlexGen); reported in Appendix B.2.
Six everyday pictures

How to think about MoE-Lightning

Each analogy stands alone. Every card ends with where the picture breaks down.

CGOPipe · Pipelining

Three chefs and a conveyor belt

Chef A (GPU) makes the main course, Chef B (CPU) prepares the salad, and a conveyor belt (I/O) delivers ingredients. Naive schedule: wait for all ingredients for dish 1, then cook dish 1, then wait for all ingredients for dish 2. Result: chefs idle while the belt moves. CGOPipe's fix: page the ingredients into small batches and overlap: while A cooks dish 1's main, B prepares dish 3's salad, and the belt delivers dish 2's next ingredient page. No one waits.

Where it breaks: Real chefs can't interleave tasks; the analogy ignores dependencies (you can't serve dish 2 until dish 1 finishes). CGOPipe respects data dependencies via careful synchronization (§A.1).
HRM · Balance point

Filling a bathtub with two taps

Tap A (GPU memory bandwidth) and Tap B (CPU↔GPU I/O) fill a tub. If you open Tap A too wide, the tub fills before Tap B delivers enough water — B is underutilized. If you open B too wide, you wait for A to catch up. The balance point is where both taps run full-blast and the tub fills fastest. HRM models both taps' capacity and finds the setting (N, μ) that hits the balance point.

Where it breaks: Real taps don't change flow rate based on tub size; in MoE-Lightning, "tap capacity" (operational intensity) depends on batch size and micro-batch size, so the analogy oversimplifies the interdependencies HRM models.
Weight paging · I/O overlap

Slicing a loaf while the toaster runs

A whole loaf (layer's weights) takes 10 seconds to transfer. If you wait for the entire loaf before toasting, the toaster idles for 9 seconds. Instead, slice the loaf into 3 pieces (pages) and hand them over every 3 seconds while the toaster runs on the previous slice. Now the toaster never waits — the transfer is hidden behind useful work. That's weight paging: chunk the transfer to match the computation cadence of micro-batches.

Where it breaks: Slicing takes negligible time; paging weights requires careful memory management (pinned buffers, page tables — see §4.1, §A.1). The analogy also assumes transfers and toasting take exactly the same time per slice, which isn't always true.
CPU attention · Device assignment

Washing dishes by hand vs dishwasher

The dishwasher (GPU) is fast for heavy-duty jobs (FFN matmuls) but takes forever to load/unload delicate items (KV cache). Hand-washing (CPU attention) is slower per dish but skips the load/unload time. For decode-stage attention (small per-token compute, huge KV cache to transfer), hand-washing wins: 3–4× lower latency than transferring KV to GPU, and it frees the dishwasher's loading dock (I/O) for weights. HRM computes this trade-off and assigns attention to CPU when I/O is tight.

Where it breaks: A dishwasher doesn't get slower as you load more dishes, but GPU attention does get slower as context length grows (and KV cache transfer time grows linearly). The analogy also ignores that CPU attention latency becomes the bottleneck at very large batch sizes (Fig. 9, §6.2).
Turning points · Shifting bottleneck

A highway with two lane-reduction zones

You enter a 4-lane highway (high I/O headroom). At mile 30, it drops to 2 lanes (GPU memory bandwidth limit). At mile 60, it drops to 1 lane (CPU attention latency limit). Your max speed changes at each reduction — those are turning points. HRM plots these as intersections of different roofs (I/O, GPU mem bandwidth, CPU compute). The policy search identifies which "zone" you're in for a given (N, μ) and picks the setting that moves you to the fastest zone without hitting a wall (out-of-memory).

Where it breaks: Real lane reductions are fixed in space; in MoE-Lightning, the "location" of turning points depends on model config (number of experts, hidden size) and hardware (bandwidth, FLOPs). The analogy also treats bottlenecks as binary switches, whereas HRM uses continuous roofline equations.
Policy search · Why FlexGen's policy is suboptimal

Two hikers choosing a route

FlexGen picks a trail (batch size) based on yesterday's bottleneck (e.g., "I/O was slow, so increase N to amortize it"). But today the bottleneck might be different (CPU memory ran out, or GPU mem bandwidth saturated). FlexGen doesn't re-check, so it keeps hiking the wrong trail. MoE-Lightning uses HRM to model all bottlenecks at once for the current hardware/workload and picks the route (N, μ) that avoids hitting any single bottleneck too hard — reaching the balance point where GPU memory bandwidth and CPU↔GPU I/O both saturate for the modeled computation.

Where it breaks: A hiker can see the terrain in real-time and adapt; FlexGen's policy is computed once per workload (offline), as is MoE-Lightning's. The difference is HRM's model is more accurate, so the chosen route is better. The analogy also oversimplifies the multi-dimensional search space (N, μ, device assignments, r_w, r_c).
The full explanation

How it actually works

Section by section, with the paper's analysis and diagrams rebuilt.

1 · The memory-throughput trade-off

To run Mixtral 8x7B (~47B parameters) on a T4 GPU (16 GB), you must offload weights to CPU and stream them layer-by-layer. Larger batch size N amortizes this I/O overhead, raising throughput — but also increases CPU memory (for KV caches) and GPU peak memory (during prefill). Existing systems like FlexGen over-batch to compensate for poor I/O utilization, requiring 2–3× more CPU memory to reach peak throughput than MoE-Lightning does with efficient overlap.

🌱
Concretely (paper Fig. 1): At N=504 (the batch size MoE-Lightning uses), FlexGen achieves ~17 tokens/s on MTBench (S1, gen_len=128). MoE-Lightning achieves ~30 tokens/s at the same N and uses ~100 GB CPU memory. FlexGen needs N≈1116 and ~200 GB CPU memory to reach ~30 tokens/s — 2× the CPU memory for the same throughput.

2 · Hierarchical Roofline Model (HRM)

The classical Roofline plots performance (FLOPs/s) vs operational intensity I (FLOPs/byte), with two roofs: compute (horizontal, at peak FLOPs) and memory (diagonal, slope = bandwidth). For a heterogeneous CPU-GPU system with offloading, we need multiple roofs:

  • GPU compute roof: P_gpu_peak (e.g., 242 TFLOPs for L4)
  • GPU memory roof: B_gpu × I_gpu (e.g., 300 GB/s × I)
  • CPU compute roof: P_cpu_peak (e.g., 1.3 TFLOPs)
  • CPU memory roof: B_cpu × I_cpu (e.g., 100 GB/s × I)
  • CPU↔GPU I/O roof: B_cpu_gpu × I_cpu (e.g., 32 GB/s × I)

For a computation x running on GPU with data from CPU, the achievable performance is P = min(P_gpu_peak, B_gpu × I_gpu, B_cpu_gpu × I_cpu). The turning points are where the min switches from one roof to another, e.g., when B_cpu_gpu × I_cpu = B_gpu × I_gpu (the "balance point" for GPU memory bandwidth and CPU↔GPU I/O).

🍯
Key insight: HRM extends the Roofline to model GPU, CPU, and I/O simultaneously, revealing regions where different resources are the bottleneck. MoE-Lightning's optimizer searches for (N, μ) that balances GPU memory bandwidth and CPU↔GPU I/O for the modeled computation.

3 · CGOPipe — the fine-grained pipeline

Naive offloading transfers all weights for layer L, then runs all micro-batches, then transfers all weights for L+1. Result: the first micro-batch of L+1 waits a long time (the entire weight transfer for L+1) while the GPU idles. CGOPipe fixes this by paging weights into n chunks (n = number of micro-batches) and overlapping:

  1. GPU: runs post-attention (O proj + MoE FFN) for micro-batch j of layer L, then pre-attention (layer norm + QKV proj) for micro-batch j+1.
  2. CPU: runs attention (softmax) for micro-batch j+2, using KV cache already on CPU.
  3. I/O: simultaneously transfers (a) intermediate results (QKV for micro-batch j+1) from GPU→CPU (small, ~MB), (b) hidden states for micro-batch j from CPU→GPU (small), and (c) one page of weights for layer L+1 from CPU→pinned mem (large, ~GB/n). Transfers in opposite directions happen concurrently; same-direction transfers are queued.

By the time GPU finishes micro-batch j, the next micro-batch's inputs and one weight page for the next layer are ready. No bubbles.

💡
Why paging? Transferring the entire next layer's weights as one chunk would block all other same-direction transfers (hidden states, QKV) until complete, creating a bubble. Paging into n chunks lets you interleave weight pages with smaller transfers, keeping the I/O path busy. (Algorithm 1, Fig. 6)

4 · The performance model and policy search

Given hardware H (GPU mem, CPU mem, bandwidths, FLOPs), model M (layer count, hidden dims, # experts), and workload W (avg prompt length, gen length), MoE-Lightning searches for policy P = (N, μ, A_g, F_g, r_w, r_c) that minimizes per-layer decode latency T(M,H,W,P) subject to memory constraints. A_g, F_g are binary indicators (do attention/FFN on GPU?); r_w, r_c are ratios of weights/KV cache statically on GPU. For the paper's settings, A_g=0 (CPU attn), F_g=1 (GPU FFN), r_w chosen by optimizer, r_c=0 (all KV on CPU).

The model estimates T as max(comm_cpu_to_gpu, T_cpu, T_gpu), where T_gpu = T_gpu_attn + T_gpu_ffn, T_cpu = T_cpu_attn + T_cpu_ffn. Each term uses the HRM roofline equations (Eq. 7–8, §3.2) to compute T_x = max(comm_x, comp_x) based on theoretical FLOPs and bytes for the operation, plus profiled peak bandwidth/FLOPs for the hardware.

🍯
Why this works: The model is analytical (no hours of trial-and-error fitting like FlexGen's), yet accurate enough to rank policies correctly. It runs in <1 minute offline via mixed-integer linear programming (MILP) and has near-zero overhead at runtime. (§4.2, Appendix B.2)

5 · Case study — Mixtral 8x7B on L4

L4 GPU: 24 GB memory, 242 TFLOPs, 300 GB/s on-chip BW. CPU: 192 GB, 100 GB/s BW, 1.3 TFLOPs. CPU↔GPU: 32 GB/s PCIe. For MoE FFN in decode (context=512), HRM identifies turning points in the performance model. The optimal policy found by the optimizer lands near the balance point where GPU memory bandwidth and CPU↔GPU I/O saturate for the modeled computation.

For attention with KV cache on CPU, operational intensity is very low — below the first turning point — so MoE-Lightning assigns attention to CPU, skipping the KV transfer entirely and freeing I/O for weights. (§3.3, §6.2)

Follow the numbers, step by step

Worked examples

Four traces using the paper's real measurements.

① Computing the turning point P1

L4 instance: B_cpu_gpu = 32 GB/s, P_cpu_peak = 1.3 TFLOPs = 1300 GFLOPs. Mixtral 8x7B MoE FFN with top-2 routing, μ=128 achieves P_cpu = 50 GFLOPs/s on CPU. At what operational intensity I_cpu does transferring to GPU stop being beneficial?

  1. The CPU can achieve P_cpu = 50 GFLOPs/s (measured, or min(P_cpu_peak, B_cpu × I_cpu) if we model it).
  2. Transferring to GPU incurs I/O roof: P_gpu ≤ B_cpu_gpu × I_cpu = 32 GB/s × I_cpu.
  3. When P_cpu ≥ B_cpu_gpu × I_cpu, running on CPU is faster. So the turning point P1 is at I_cpu = P_cpu / B_cpu_gpu = 50 GFLOPs/s / 32 GB/s = 1.56 FLOPs/byte (paper §3.3 uses this logic; the exact number varies by μ and kernel tuning).
  4. For μ=128, the paper's measured I_cpu ≈ 10 FLOPs/byte (depends on hidden dims, experts, etc.), so MoE FFN is above P1 → beneficial to use GPU. For attention, I_cpu ≈ 2 (very memory-bound) → below P1 → keep on CPU.
🍯
Insight: P1 is where "running on CPU + not transferring" beats "transferring to GPU + running on GPU." Below P1, the I/O cost dominates the GPU speedup. (Eq. 9 in paper, Fig. 5)

② Computing the balance point P2

Same L4 instance. MoE FFN with μ=128 achieves P_gpu = 120 GFLOPs/s on GPU (limited by GPU mem BW). At what I_cpu does the CPU↔GPU I/O roof equal the GPU mem BW roof?

  1. GPU mem BW roof: P ≤ B_gpu × I_gpu. For this kernel, I_gpu ≈ 0.4 FLOPs/byte (measured or calculated from matrix dims), so B_gpu × I_gpu ≈ 300 GB/s × 0.4 = 120 GFLOPs/s (matches profiled GPU performance).
  2. CPU↔GPU I/O roof: P ≤ B_cpu_gpu × I_cpu = 32 GB/s × I_cpu.
  3. Balance point P2: 32 × I_cpu = 120 → I_cpu = 3.75 FLOPs/byte (approximately; paper's Fig. 5 shows P2 around I≈5–10 depending on μ and N).
  4. To reach P2, increase batch size N (more tokens = more FLOPs per weight byte → higher I_cpu) or increase μ (better GPU kernel utilization → higher I_gpu, shifting the balance). The optimizer searches for (N, μ) that hits P2 without violating memory limits.
🍯
Insight: P2 is where both I/O and GPU mem BW saturate together for the modeled computation — the "sweet spot." Below P2, you're I/O-bound (increase N to amortize). Above P2, you're GPU mem-bound (can't help by increasing N). (Eq. 10, 11 in paper)

③ Why FlexGen's policy is suboptimal

MTBench, S1 (T4 GPU, 16 GB; CPU 192 GB), gen_len=128. FlexGen uses N=1112, μ=8, achieving 9.5 tokens/s. MoE-Lightning uses N=504, μ=36, achieving 30.1 tokens/s. Why the 3.17× gap?

  1. FlexGen's policy (N=1112, μ=8) was chosen to maximize batch size (amortize I/O) but uses tiny μ (constrained by GPU mem during prefill). Small μ → poor GPU utilization → low throughput.
  2. FlexGen also does GPU attention + KV cache prefetch, which consumes I/O bandwidth (32 GB/s × fraction of time spent transferring KV) that could be used for weights. Result: weights transfer slower → more time per layer → lower throughput.
  3. MoE-Lightning's policy (N=504, μ=36): smaller N (fits in 192 GB CPU mem with KV caches), larger μ (better GPU kernel efficiency). CPU attention frees all I/O for weights. CGOPipe overlaps everything → no bubbles → higher effective I/O utilization.
  4. Applying MoE-Lightning's policy to FlexGen (same N=504, μ=36) raises FlexGen's throughput to ~17 tokens/s (Table 5), but it's still bottlenecked by KV cache transfers. Only with CGOPipe + CPU attn do you reach 30.1 tokens/s.
🍯
Insight: FlexGen's policy search doesn't model the shifting bottleneck (from I/O to GPU mem BW to CPU attn) as N, μ change. It also doesn't account for the I/O cost of KV cache transfers. MoE-Lightning's HRM does both, yielding a better policy. (§5.2, §6.1)

④ Tensor parallelism and super-linear scaling

Mixtral 8x22B on 2×T4 (32 GB total GPU memory) vs 4×T4 (64 GB). MTBench, gen_len=128. MoE-Lightning achieves 7.75 tokens/s on 2×T4 and 26.2 tokens/s on 4×T4 — a 3.38× speedup from 2× more GPUs. Explain the super-linear scaling.

  1. With 2×T4, GPU memory capacity = 32 GB. The HRM balance point for this hardware/workload is at a certain N and μ, achieving ~7.75 tokens/s (bottlenecked by GPU mem capacity — can't increase N or μ without OOM).
  2. With 4×T4, GPU memory capacity = 64 GB, and aggregate GPU memory bandwidth = 2× (two memory channels active), and CPU↔GPU I/O bandwidth = 2× (each GPU has its own PCIe link). The new balance point shifts to higher N and μ, enabling ~26.2 tokens/s.
  3. Why super-linear? Because the bottleneck shifted: 2×T4 was GPU mem capacity-bound (flat performance past a certain N); 4×T4 is no longer capacity-bound, so it can exploit the higher bandwidth. The throughput gain is proportional to the bandwidth gain (~2×) times the batch-size gain (~2×) = ~4× potential, realized as 3.38× after accounting for overheads.
  4. This is not possible with pipeline parallelism (FlexGen's approach): adding more GPUs via pipeline doesn't increase per-layer bandwidth, only depth. Hence FlexGen sees diminishing returns (paper §5.3).
🍯
Insight: Tensor parallelism increases both capacity and bandwidth, unlocking a new balance point. MoE-Lightning's HRM recalculates the optimal (N, μ) for the new hardware and exploits it. (Fig. 7, 8; §5.3)

Problem 1 · easy
Why does MoE-Lightning do attention on the CPU instead of the GPU, even though the GPU is much faster at compute?
Show answer
Because transferring the KV cache from CPU to GPU is slower than just running attention on the CPU. For context length 512 and batch size N (in tokens), the KV cache is large (roughly 128 KB/token in fp16 for Mixtral-style GQA). At 32 GB/s PCIe, that transfer takes longer than the CPU attention kernel (which runs at ~50 GFLOPs/s but doesn't transfer anything). By doing attention on CPU, you free the I/O bandwidth for weight transfers, which are unavoidable. (HRM analysis in §3.3 shows attention's operational intensity is below P1, so GPU isn't beneficial.)Misconception: that GPU is always faster for any computation. The total time (transfer + compute) matters, not just compute. When I/O cost dominates, keeping data on CPU wins.
Problem 2 · medium
FlexGen achieves ~9.5 tokens/s with N=1112, μ=8 on MTBench (S1, gen_len=128). If you apply MoE-Lightning's policy (N=504, μ=36) to FlexGen, its throughput rises to ~17 tokens/s. Yet MoE-Lightning itself achieves ~30 tokens/s. What accounts for the remaining 1.76× gap?
Show answer
CGOPipe + CPU attention. Even with the better (N, μ), FlexGen still does GPU attention + KV cache prefetch, which (a) consumes I/O bandwidth that could be used for weights, and (b) creates pipeline bubbles (weights for the next layer must wait for KV transfer to finish). MoE-Lightning's CPU attention eliminates the KV transfer, and CGOPipe's paged weights + fine-grained overlap eliminate the bubbles. Together, these raise I/O utilization and GPU utilization, closing the gap to 30 tokens/s. (Table 5, §6.1)Misconception: that policy (N, μ) is the only thing that matters. Even with the optimal policy, poor pipeline scheduling and device assignment can leave throughput far below the hardware's capability.
Problem 3 · hard
The paper claims MoE-Lightning can reach the throughput upper bound with 2–3× less CPU memory than FlexGen. Derive a back-of-the-envelope formula for CPU memory consumption as a function of batch size N (in tokens), average prompt length s, generation length l, and explain why lower I/O utilization forces higher N (and thus more CPU memory).
Show answer
CPU memory ≈ (model weights on CPU) + (KV cache for all requests). For Mixtral 8x7B, weights ≈ 40 GB (assuming some experts on GPU). KV cache per token ≈ 128 KB/token in fp16 for Mixtral-style GQA. For a batch with total N tokens across all requests, KV cache ≈ N × 128 KB. If I/O utilization is low (e.g., 40% instead of 80%), you need 2× the batch size to amortize the same I/O overhead, doubling KV cache memory. Hence FlexGen's poor I/O utilization (due to KV transfers + lack of pipelining) forces large N to reach peak, requiring more CPU memory, while MoE-Lightning reaches peak at lower N with less CPU memory. (Paper §1, Fig. 1)Misconception: that CPU memory is only a function of model size. It's actually dominated by KV cache, which scales with N. Low I/O utilization → need high N to compensate → high CPU memory.
Problem 4 · medium
In the HRM plot for MoE FFN (Fig. 5), the operational intensity I_cpu depends on batch size N and micro-batch size μ. Explain qualitatively how increasing N and μ each affect I_cpu and which one you'd increase first if you're currently I/O-bound (below P2).
Show answer
Operational intensity I_cpu = FLOPs / bytes_accessed_from_CPU. For MoE FFN, FLOPs ≈ N × 2 × (activated experts' params) × micro_batch_size (roughly; ignoring router). Bytes accessed from CPU = weights for the activated experts (fixed per layer, independent of N or μ) + intermediate results (scales with N). So I_cpu ≈ (N × μ × const) / (weights + N × intermediate). As N grows, both numerator and denominator grow, but FLOPs grow faster (quadratic-ish in N), so I_cpu increases. As μ grows (for fixed N), FLOPs grow (more tokens per kernel call → better reuse) but bytes stay roughly the same (weights are still loaded once per micro-batch), so I_cpu increases. If you're I/O-bound (below P2), increase N first: it directly raises the amortization of weight transfers across more tokens. Increase μ second to improve GPU kernel efficiency (higher I_gpu), which raises the GPU mem BW roof and shifts P2 to the right, letting you absorb the higher I_cpu. (§3.3, Eq. 6–11)Misconception: that μ and N are interchangeable. They affect different things: N affects I/O amortization (I_cpu for weights from CPU), μ affects GPU kernel efficiency (I_gpu on GPU). When I/O-bound, prioritize N.
Problem 5 · hard
The paper shows super-linear scaling (3.38× throughput from 2× GPUs) with tensor parallelism. Under what conditions would you NOT see super-linear scaling, and how does this relate to the balance point?
Show answer
Super-linear scaling happens when doubling GPUs lifts a capacity bottleneck and also doubles bandwidth, enabling a new balance point at higher (N, μ). If you're NOT capacity-bound (e.g., 1×A100-80G for Mixtral 8x7B, plenty of GPU memory), adding a second GPU via tensor parallelism doubles bandwidth but doesn't unlock a higher N or μ (already at the balance point). Result: ~2× scaling (linear), not super-linear. Similarly, if you're CPU memory-bound (not enough CPU DRAM to hold the larger N's KV cache for 2× GPUs), you can't exploit the extra GPU capacity → sub-linear scaling. Super-linear requires: (1) the original system was GPU mem capacity-bound, and (2) adding GPUs lifts that constraint, and (3) CPU memory is sufficient for the new, larger N. (§5.3, §6.3)Misconception: that adding more GPUs always gives super-linear scaling. It only happens when the bottleneck shifts favorably (e.g., from capacity to bandwidth). If the bottleneck is elsewhere (CPU mem, CPU compute), adding GPUs won't help as much.
Problem 6 · medium
CGOPipe pages weights into n chunks (n = number of micro-batches). Why not just split into 2 chunks, or 100 chunks? What determines the optimal n?
Show answer
Too few chunks (n=2): each chunk is large, so transferring one chunk takes a long time, potentially longer than the GPU's computation for one micro-batch. Result: the next micro-batch still waits for the chunk to finish → bubble. Too many chunks (n=100): each chunk is tiny, so the overhead of launching 100 small transfers (each with its own latency and API call) dominates. Also, if n > number of micro-batches, you're transferring more often than necessary. Optimal n = number of micro-batches in the pipeline, so each micro-batch's GPU work overlaps with exactly one weight-page transfer. This minimizes both bubble time (chunk size small enough to hide) and transfer overhead (not too many small transfers). (§4.1, Algorithm 1)Misconception: that more chunks is always better (finer granularity). There's a sweet spot where chunk size matches the computation cadence, avoiding both bubbles and overhead.
Problem 7 · easy
FlexGen uses pipeline parallelism across GPUs; MoE-Lightning uses tensor parallelism. Why does MoE-Lightning's approach scale better for MoE models in memory-constrained settings?
Show answer
Pipeline parallelism splits the model by layers (e.g., layers 0–15 on GPU0, 16–31 on GPU1). This doubles the number of layers you can fit but doesn't increase per-layer bandwidth or capacity — each GPU still has the same memory and I/O. For MoE, where throughput is often GPU memory capacity-bound or I/O-bound (not depth-bound), pipeline parallelism doesn't help much. Tensor parallelism splits each layer across GPUs (e.g., half the experts on GPU0, half on GPU1), so you double the per-layer memory capacity and aggregate memory bandwidth (2× the channels, 2× the PCIe links). This directly raises the throughput upper bound by lifting the capacity/bandwidth bottleneck. (§5.3, Fig. 7–8)Misconception: that pipeline parallelism is always the right way to scale. It's great for deep models on fast interconnects, but for memory-capacity-bound or I/O-bound workloads, tensor parallelism wins.
Problem 8 · medium
The paper's performance model (HRM) uses theoretical FLOPs and bytes plus profiled peak bandwidth/FLOPs, with no hours of trial-and-error fitting. Why does this simple model work well enough to rank policies correctly, whereas FlexGen's model requires extensive fitting?
Show answer
HRM is based on the Roofline Model, which captures the fundamental trade-off: computation vs memory bandwidth. For LLM inference, this trade-off is the dominant factor — kernels are either compute-bound (hit the FLOPs roof) or memory-bound (hit the bandwidth roof), with little in between. By modeling both roofs (GPU, CPU, I/O), HRM accurately predicts which resource is the bottleneck for a given (N, μ). The profiled peak values (e.g., 242 TFLOPs, 300 GB/s) are hardware constants, not workload-dependent, so they generalize. FlexGen's model, in contrast, tries to fit kernel latency directly as a function of batch size, which requires measuring many (N, μ) combinations — hence hours of fitting. HRM's analytical approach is faster and more general. (§4.2, §B.2)Misconception: that a performance model must be empirical (fit to measurements). For LLM inference, the roofline abstraction is powerful enough that a simple analytical model suffices, as long as you profile the hardware constants once.
Quick answers

Frequently asked questions

Twelve common questions. Tap any to expand.

Q1Why is MoE-Lightning's throughput improvement so large (10.3×) without padding but smaller (3.5×) with padding?▾
Without padding, FlexGen must pad all requests to the max prompt length in the batch (e.g., 418 tokens for MTBench), wasting memory and compute on padding tokens. MoE-Lightning supports variable-length requests (§A.2), so no waste. With padding (artificial workload where all prompts are the same length), FlexGen's overhead shrinks, reducing MoE-Lightning's relative gain. But even with padding, MoE-Lightning still wins 3.5× due to better pipelining (CGOPipe) and device assignment (CPU attn). (§5.2, Fig. 7)
Q2What is the hardware overhead of MoE-Lightning's components (weight paging, performance model, etc.)?▾
The optimizer overhead is reported in Appendix B.2. Weight paging uses page tables (a few kB of metadata per layer) and pinned buffers (already needed for efficient CPU↔GPU transfers). The performance model runs offline in <1 minute via MILP (§4.2, Appendix B.2) — no runtime overhead. The paper focuses on software scheduling, not custom hardware. (§1, §5)
Q3Can MoE-Lightning handle dense models like LLaMA-70B, or is it MoE-specific?▾
The core techniques (CGOPipe, HRM, CPU attention) apply to any LLM with weight offloading. For dense models, the FFN is a single large matmul (not routed experts), so the paging is simpler (fewer chunks). Dense models benefit from CGOPipe + HRM, but the relative gain over baselines is smaller because dense models' memory-to-FLOPs ratio is lower (less I/O overhead to hide). MoE-Lightning is most impactful for models where I/O is the bottleneck — which MoEs are, due to their high parameter count. (§B.1)
Q4How does MoE-Lightning compare to quantization (e.g., int4 weights) as a way to fit models in GPU memory?▾
Quantization reduces model size (e.g., ~47B parameters in fp16 to int4), potentially fitting Mixtral 8x7B in a smaller GPU. But quantization also reduces quality (perplexity, accuracy), whereas offloading preserves full fp16/bf16 precision. MoE-Lightning and quantization are orthogonal: you can quantize and offload (e.g., int4 weights in CPU DRAM → GPU). The paper uses fp16 to isolate the system's contribution, but quantization could stack on top for even smaller memory. (§5 uses fp16; quantization not evaluated but compatible)
Q5Why does the paper use Mixtral 8x7B, 8x22B, and DBRX? Are these representative of real-world MoE deployments?▾
These are popular open-weight MoE models (as of 2024–2025) with diverse sizes (~47B, 141B, 132B parameters) and expert counts (8, 8, 16), representative of the class of models users want to serve but can't fit on consumer/low-end GPUs. Mixtral 8x7B is the canonical "large MoE that doesn't fit on a 16 GB GPU" example. DBRX and 8x22B test scaling to even larger models. (§5.1)
Q6The paper focuses on batch inference (offline evaluation, data generation). What about online serving (chat, API)?▾
MoE-Lightning targets throughput-oriented batch workloads where latency per request is less critical than total completion time. For online serving, latency matters, and offloading adds significant latency (10–100× slower than on-device inference). Techniques like speculative decoding, quantization, or buying a bigger GPU are more appropriate there. However, for low-QPS services on a budget (e.g., internal tools), MoE-Lightning's high throughput per $ could still be useful. (§1, "batch-processing workloads")
Q7How does MoE-Lightning handle variable-length prompts in a batch (e.g., MTBench with prompts 10–418 tokens)?▾
Algorithm 2 (Appendix A.2) describes the batching strategy: sort requests by input length descending, then greedily assign each to the micro-batch with the fewest tokens so far. This balances token distribution across micro-batches, avoiding one micro-batch being much larger (which would create a bubble). Within each micro-batch, padding is still needed (GPU kernels require fixed shapes), but it's minimized because requests are similar length. FlexGen, by contrast, pads all requests to the global max, wasting far more memory/compute. (§A.2, §5.2)
Q8The paper uses T4 and L4 GPUs (mid-range). What about high-end GPUs like A100 or H100?▾
On high-end GPUs with 80 GB memory, Mixtral 8x7B fits entirely in GPU memory (no offloading needed), so MoE-Lightning's techniques are moot for that model. But for larger models (e.g., Mixtral 8x22B: 141B parameters, or hypothetical 1T MoEs), even A100s need offloading, and MoE-Lightning's approach applies. The paper targets "memory-constrained" settings — which are more common on mid-range GPUs but can occur on high-end GPUs for sufficiently large models. (§1, §5.1)
Q9How does MoE-Lightning handle multi-node deployments (e.g., 8× GPUs across 2 nodes)?▾
The paper focuses on single-node tensor parallelism (up to 4× GPUs, §5.3). For multi-node, tensor parallelism requires fast interconnects (InfiniBand, NVLink) to avoid communication bottlenecks. Pipeline parallelism (which FlexGen uses) scales better across slow inter-node links but doesn't exploit per-layer bandwidth gains. MoE-Lightning could combine tensor parallelism within a node and pipeline parallelism across nodes, but the paper doesn't evaluate this. (§C, "multi-node communication")
Q10The performance model uses profiled peak bandwidth/FLOPs. How sensitive are the results to measurement errors?▾
HRM is used to rank policies, not predict absolute latency. As long as the roofline curves' relative positions are correct (e.g., "I/O roof is below GPU mem roof"), the optimizer picks the right policy even if the absolute numbers are off by 10–20%. The paper's results (§5) validate this: the chosen policies achieve near-optimal throughput. For new hardware, you'd re-profile once (takes minutes) and re-run the optimizer. (§4.2, §B.2)
Q11What about disk offloading (e.g., model on SSD, swap to CPU DRAM)? Does MoE-Lightning support it?▾
The paper assumes sufficient CPU DRAM to hold the model (e.g., 192 GB for Mixtral 8x7B). Disk offloading is mentioned as future work (§C). Disk bandwidth (~5 GB/s for NVMe SSD) is 5–10× slower than CPU↔GPU PCIe, so the bottleneck would shift entirely to disk I/O. CGOPipe's pipelining would still help, but the balance point would be at much lower (N, μ) due to the I/O constraint. (§4, §C)
Q12How does MoE-Lightning compare to other recent MoE inference systems (e.g., MegaBlocks, Tutel, FastMoE)?▾
MegaBlocks, Tutel, FastMoE focus on training or inference with sufficient GPU memory (no offloading). They optimize expert routing, load balancing, and kernel efficiency but assume the model fits on-device. MoE-Lightning targets the orthogonal problem: inference when the model doesn't fit, via offloading. The techniques are complementary — you could combine MegaBlocks' expert kernels with MoE-Lightning's offloading pipeline. The paper compares against FlexGen and DeepSpeed-ZeRO, the main offloading baselines. (§7, Related Work)

Glossary & further reading

Every term in one place, then a path to go deeper.

Core terms
MoE (Mixture of Experts)An architecture where each transformer layer has multiple expert FFNs; a router selects top-k experts per token, activating fewer parameters than a dense model.
CGOPipeCPU-GPU-I/O Pipeline: MoE-Lightning's fine-grained schedule that pages weights, overlaps CPU attention + GPU FFN + I/O transfers, eliminating bubbles.
HRMHierarchical Roofline Model: extends the classical Roofline to model GPU, CPU, and I/O roofs simultaneously, identifying turning points and balance points.
Balance pointThe (N, μ) where GPU memory bandwidth and CPU↔GPU I/O bandwidth both saturate for the modeled computation, maximizing throughput (Eq. 11).
Turning pointWhere the bottleneck shifts (e.g., from I/O to GPU mem BW, or to CPU attn latency) as you scale (N, μ). P1 and P2 are key turning points in the paper.
Weight pagingChunking a layer's weights into n pages (n = # micro-batches) so each page transfers while the previous micro-batch's GPU work runs, hiding I/O latency.
Operational intensity IFLOPs / bytes_accessed. Higher I = more compute per byte → more likely to hit compute roof (less memory-bound).
Batch size NTotal tokens in the batch (in MoE-Lightning's notation, N is measured in tokens). Larger N amortizes I/O overhead but increases memory (KV cache).
Micro-batch size μTokens processed by one GPU kernel invocation. Larger μ improves GPU utilization (higher I_gpu) but increases GPU peak memory.
Supporting terms
Prefill stageProcessing the initial prompt tokens in parallel to generate the first output token and cache all KV pairs. Typically compute-bound.
Decode stageGenerating subsequent tokens one-at-a-time, using cached KV. Memory-bound — few FLOPs, many bytes read (weights + KV cache).
KV cacheStored key-value vectors from all previous tokens, needed for attention. For Mixtral-style models, roughly 128 KB/token in fp16.
CPU attentionRunning the attention softmax on CPU (using KV cache in CPU DRAM) instead of transferring KV to GPU. Faster when KV transfer latency > CPU compute latency.
Pipeline bubbleIdle time where one resource (GPU, CPU, I/O) waits for another. CGOPipe eliminates bubbles by overlapping everything.
Tensor parallelismSplitting each layer's weights across multiple GPUs (e.g., half the experts on GPU0, half on GPU1). Increases per-layer capacity and bandwidth.
Pipeline parallelismSplitting the model by layers (e.g., layers 0–15 on GPU0, 16–31 on GPU1). Increases depth but not per-layer bandwidth.
FlexGenPrior work (Sheng et al., ICML 2023) on offloading-based LLM inference. Does GPU attention + KV prefetch; uses a fitted performance model.
DeepSpeed-ZeROMicrosoft's framework for training/inference with weight offloading. Uses very small batch sizes (1–2), so weight-transfer overhead dominates.
Roofline ModelClassical performance model (Williams et al., 2009) plotting achievable FLOPs/s vs operational intensity, with compute and memory roofs.
