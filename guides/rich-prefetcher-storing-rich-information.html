MICRO 2025 · A Field Guide

Storing rich information in memory to trade bandwidth for latency hiding

Future memory is fast and vast but slow. Conventional prefetchers hoard small on-chip metadata and miss huge opportunities. RICH flips that: spend off-chip capacity and bandwidth to predict more patterns—and hide the latency that matters most.

📄 "RICH Prefetcher" · Ai et al., MICRO '25 🎓 Assumes basic cache/prefetch knowledge ⏱ ~25 min read
🗄️

Use multiple spatial region sizes (2/4/16 KB) with tiered on-chip/off-chip storage — large regions trigger only after multi-offset validation, small regions trigger aggressively — to maximize coverage and timeliness while keeping accuracy high.
Measure rich metadata cheaply, choose regions wisely, hide latency by prefetching far ahead.

Before you begin

Eight ideas the paper assumes you know

Each card clears up one concept RICH builds on. Read in order the first time, or jump to whatever looks unfamiliar.

2 KB region 4 KB region 16 KB region LRU/conventional
1Cache prefetching—predicting future accesses▾

A prefetcher guesses which memory blocks the processor will need soon and fetches them before the processor asks, turning slow cache misses into fast hits. The challenge: predict accurately without wasting bandwidth on useless prefetches.

Prefetchers store metadata—learned patterns, training state, control info. Conventional designs minimize that metadata to save on-chip area, but RICH accepts more metadata to capture richer patterns.

Now you're ready: RICH is about using abundant off-chip memory to store more metadata than traditional prefetchers dare, converting capacity into latency hiding.
2Spatial locality—nearby data often accessed together▾

Programs exhibit spatial locality: accessing address X often means you'll soon access X+64, X+128, etc., because data structures lay out related fields near each other. A spatial prefetcher learns which offsets within a region are frequently visited and prefetches them as a group.

Conventional spatial prefetchers (SMS with fixed 2 KB regions, Bingo) track patterns within small fixed-size regions. RICH extends this to 2 KB, 4 KB, and 16 KB regions because different programs have different "scales" of locality.

Now you're ready: RICH captures multi-scale spatial locality by supporting three region sizes and choosing the best one dynamically.
3Accuracy, Coverage, Timeliness—the three metrics▾

Accuracy = fraction of prefetches that are used (not evicted unused). Coverage = fraction of cache misses eliminated by prefetching. Timeliness = fraction of useful prefetches already in cache (vs still in-flight) when the processor needs them.

High accuracy avoids wasting bandwidth; high coverage eliminates more misses; high timeliness means prefetches arrive early enough. RICH optimizes all three by region arbitration and multi-offset triggers.

Now you're ready: RICH's design decisions—when to trigger, which region to prefetch—are all about balancing these three metrics.
4Memory latency trends—higher capacity = higher latency▾

DDR5 offers 3× the bandwidth of DDR3 but ~40% higher latency (14–18 ns row-hit vs DDR3's 10–15 ns). Emerging NVM (phase-change, resistive memory) and CXL-based memory pooling add even more latency. Future memory is fast and vast but inherently slower to access.

This trend motivates prefetching that exploits high bandwidth to issue long prefetch streams, hiding the latency penalty by fetching far ahead.

Now you're ready: RICH is designed for the high-latency future—its gains grow as memory latency increases (e.g., +8.3% over Bingo at +120 ns).
5Trigger events—when to issue a prefetch▾

A prefetcher triggers when it sees a specific event—e.g., a load PC plus an offset within a region. Bingo uses (PC, offset) or (PC, address); SMS uses (PC, offset). The trigger identifies which learned pattern to apply.

RICH extends this to multi-offset triggers: wait until you see (PC, 5 offsets) for 16 KB regions, (PC, 3 offsets) for 4 KB. This validates the access stream is continuing before committing to a large prefetch, raising accuracy.

Now you're ready: The multi-offset trigger is RICH's key innovation for large regions—it trades a bit of coverage for much higher accuracy.
6Bit-vector patterns—encoding spatial footprints▾

A spatial prefetcher encodes a region's access pattern as a bit-vector: 64 bits for a 4 KB region (64 cache lines), where bit i=1 means "cacheline at offset i is frequently accessed." When triggered, prefetch all the '1' bits.

RICH uses 32 bits for 2 KB regions, 64 bits for 4 KB, 256 bits for 16 KB. A 16 KB region with a 256-bit pattern represents up to 256 individual cachelines that could be prefetched—the actual number fetched depends on how many bits are set to '1'. Larger bit-vectors consume more space—which is why RICH offloads the 16 KB patterns to main memory.

Now you're ready: The Pattern History Table (PHT) stores these bit-vectors; RICH keeps 2 KB and 4 KB PHTs on-chip, 16 KB PHT mostly off-chip.
7Off-chip metadata—using main memory for prefetch state▾

Temporal prefetchers like ISB, MISB, STMS already store long miss sequences in main memory. But they require exact address-sequence repetition, limiting applicability. RICH offloads spatial patterns to memory—a broader class of metadata that helps many more workloads.

The trick: 16 KB-region patterns are latency-tolerant (a single fetch can yield many prefetches from the '1' bits in the 256-bit pattern, providing plenty of runway) and infrequent (most accesses hit the on-chip PHT cache). So off-chip is cheap and rarely delays prefetching.

Now you're ready: RICH uses 128 KB of main memory per core for 16 KB patterns—tiny in modern systems, huge gain in pattern storage.
8Region arbitration—choosing which pattern to prefetch▾

When multiple regions trigger at once (e.g., a 2 KB pattern and a 16 KB pattern both match), RICH must choose. It uses a fixed priority: 2 KB by (PC, address) > 16 KB > 4 KB > 2 KB by (PC, offset), reflecting accuracy × performance-per-correct-prefetch.

Plus deduplication: if a higher-priority prefetch is in-flight, don't issue overlapping lower-priority prefetches unless the new request has higher priority than the in-flight one. This maximizes useful prefetches while minimizing wasted bandwidth.

Now you're ready: Arbitration + deduplication = maximizing useful prefetches, minimizing wasted bandwidth.

The whole story in one place

RICH in a nutshell

The problem, the multi-region solution, the hierarchical storage trick, and how it all connects.

Conventional spatial prefetchers (SMS, Bingo, PMP) capture patterns within small fixed-size regions and store all metadata on-chip to minimize area. But future memory systems offer high bandwidth and capacity at the cost of higher latency—exactly the environment where a prefetcher should spend capacity and bandwidth to predict more patterns and issue longer prefetch streams. Traditional designs leave that opportunity untapped.

RICH fixes this by supporting three spatial region sizes (2 KB, 4 KB, 16 KB) and using hierarchical on-chip/off-chip storage. Large 16 KB regions can prefetch long streams to hide latency, but trigger only after observing (PC, 5 offsets) to maintain ~90% accuracy. Small 2 KB and 4 KB regions trigger aggressively (1 or 3 offsets) for broad coverage. A region arbitration mechanism picks the best pattern when multiple regions trigger, deduplicates overlaps, and maximizes useful prefetches.

To keep costs low, RICH analyzes metadata along three axes—access frequency, latency tolerance, area overheads—and partitions it: on-chip (2/4 KB PHTs, training tables, control) for low-latency frequent accesses; off-chip (16 KB PHT, 128 KB in main memory) for infrequent, latency-tolerant patterns. A 256-entry on-chip cache of 16 KB patterns handles most requests; misses fetch from memory asynchronously, and the long prefetch streams (with many '1' bits in the 256-bit pattern) absorb the fetch latency.

The payoff: On 100+ traces (SPEC06/17, Ligra, Parsec), RICH achieves 39% average speedup over no prefetching, outperforming Bingo by 3.4% in conventional systems. With +120 ns memory latency (simulating future NVM/CXL), RICH's advantage grows to 8.3% over Bingo and 6.2% over PMP—demonstrating its suitability for high-latency memory. On-chip area is 47.3 KB (vs Bingo's 127.8 KB), plus 128 KB off-chip per core.

🌱 Key takeaways

  • Future memory = high bandwidth/capacity, high latency. Conventional prefetchers underutilize it.
  • RICH supports 2 KB / 4 KB / 16 KB regions—multi-scale spatial locality; ~half of workloads peak above 4 KB.
  • Large regions use multi-offset triggers (PC, 5 offsets) for 90% accuracy; small regions trigger aggressively for coverage.
  • Region arbitration picks best pattern by accuracy-first priority; deduplication blocks overlapping lower-priority prefetches when a higher-priority one is already in-flight.
  • Hierarchical storage: 2/4 KB on-chip; 16 KB mostly off-chip (128 KB main memory, 256-entry on-chip cache).
  • 16 KB patterns are latency-tolerant (many prefetches per pattern from '1' bits) and infrequent (200 entries cover most requests).
  • Result: 3.4% faster than Bingo (conventional latency), 8.3% faster at +120 ns; 47 KB on-chip vs Bingo's 128 KB.
Six everyday pictures

How to think about RICH

Each picture stands alone—read them in any order. Every card ends with an honest note about where the picture stops being true.

The core idea · Multi-scale locality

Fishing with three nets

A program's data structures aren't all the same size. Tiny objects fit in a 2 KB net (high coverage, cheap misprediction), medium ones in 4 KB, but large arrays need a 16 KB net to capture the whole stream. Conventional prefetchers use only one net size—missing half the fish. RICH uses all three, dynamically picking the right one.

Where it breaks: real nets don't validate before casting; RICH validates large regions with multi-offset triggers first. And nets don't deduplicate—RICH blocks overlapping small-region prefetches when a large one already covers the space.
Multi-offset triggers · Delayed commitment

Checkpoint hiking

You don't commit to a 16-mile hike after seeing the first trailhead sign—you walk to checkpoint 1, then 2, then 3, confirming the trail continues before committing. RICH does this with (PC, 5 offsets) for 16 KB regions: wait until you've seen five distinct accesses confirming the pattern, then prefetch based on the learned 256-bit pattern. This raises accuracy to 90%+, avoiding massive misprediction waste.

Where it misleads: checkpoints are fixed in space; RICH's offsets can appear in any order. And real hikers don't get a second path option—RICH can fall back to a 2 KB prefetch if the 16 KB trigger never completes.
Region arbitration · Priority-based choice

Triage at the ER

When three patients arrive at once, the ER uses a priority system: life-threatening first, urgent second, stable last. RICH does this with prefetch regions: 2 KB by (PC, address) has highest accuracy (like life-threatening), 16 KB next (urgent), then 4 KB and 2 KB by (PC, offset) (stable). If a high-priority prefetch is already in-flight, lower-priority overlapping ones are blocked unless the new request has higher priority—no duplicate work.

Where it breaks: the ER priority is about medical need; RICH's is about accuracy × performance-per-correct-prefetch. And patients don't overlap in space—RICH's deduplication is unique to spatial prefetchers.
Hierarchical storage · On-chip cache + off-chip PHT

Desk drawer + filing cabinet

Your desk drawer holds 200 frequently-used files (on-chip 16 KB PHT cache); the filing cabinet in the hallway holds 4000 infrequent ones (off-chip 128 KB). Most accesses hit the drawer—fast. A rare miss fetches from the cabinet—slow, but you're fetching a pattern that can yield many items at once (from the '1' bits in the 256-bit vector), so the walk is amortized. RICH's 256-entry on-chip cache covers most 16 KB requests; off-chip handles the long tail.

Where it breaks: real filing cabinets don't compete for bandwidth with your desk work; off-chip metadata fetches do share the memory bus. RICH mitigates this by only writing patterns with ≥30 prefetch counts (the "prefetch count threshold").
Latency tolerance · Long prefetch distance

Launching a rocket early

A 16 KB pattern can trigger many cacheline prefetches (from the '1' bits in the 256-bit vector)—like a rocket with a long flight time. Even if the metadata fetch takes 50 ns (the "launch delay"), you still have many cacheline fetches in-flight by the time the processor catches up. The long distance absorbs the latency. In contrast, a 2 KB prefetch (32 cachelines maximum) can't tolerate that delay—it must launch immediately.

Where it breaks: rockets don't validate their trajectory mid-flight; RICH tracks late prefetches (2 KB sub-regions already accessed during the off-chip fetch) and trims them from the final prefetch stream to avoid useless work.
Coverage diversity · No one size fits all

Wardrobe for all seasons

You don't wear only T-shirts all year—winter needs a coat, spring a jacket, summer a T-shirt. Programs are the same: 46% peak with large regions (winter coat), 27% with medium (jacket), 27% with small (T-shirt). RICH keeps all three "wardrobes" and picks the right one per access pattern. Bingo wears a fixed-size jacket year-round—comfortable some of the time, suboptimal the rest.

Where it breaks: you choose clothes before leaving the house; RICH chooses regions during execution, dynamically per trigger event. And you don't wear three layers at once—RICH can trigger multiple regions but arbitrates to prefetch only one.
The full explanation

How it actually works

Section by section, with the paper's key mechanisms rebuilt as visuals you can read at a glance.

1 · The problem—conventional prefetchers underutilize rich memory

Future memory systems (DDR5, NVM, CXL) offer high bandwidth and capacity but higher latency (e.g., DDR5 ~18 ns row-hit vs DDR3's ~12 ns; CXL adds ~100+ ns). Conventional spatial prefetchers like SMS/Bingo/PMP are designed to minimize on-chip area—they capture patterns within small fixed-size regions and store all metadata on-chip (Bingo: 127.8 KB). This leaves two opportunities on the table: (1) programs exhibit spatial locality at multiple scales (2 KB, 4 KB, 16 KB), and (2) high-bandwidth memory can serve more metadata cheaply if we're willing to use it.

🌱
Concretely: Figure 1 of the paper shows that 46% of traces peak in performance when region size is set above 4 KB. But no existing spatial prefetcher supports large regions because (a) accuracy drops without validation, and (b) storing 256-bit patterns on-chip is expensive.

2 · Multi-region patterns—capturing diverse spatial scales

RICH supports three region sizes: 2 KB, 4 KB, and 16 KB. Each has a separate PHT storing bit-vector patterns (32 bits for 2 KB, 64 bits for 4 KB, 256 bits for 16 KB). The training process follows SMS/Bingo: a Filter Table (FT) tracks regions with few accesses to filter noise; an Accumulation Table (AT) builds the bit-vector as accesses occur; upon eviction or AT overflow, the pattern is promoted to the PHT.

The key innovation: each region uses a different trigger mechanism tuned to its scale:

RegionTriggerOffsets requiredStrategyAccuracy (Fig. 3)
2 KB(PC, address)1Coverage-first~80%
2 KB(PC, offset)1Coverage-first~60%
4 KB(PC, 3 offsets)3Balanced~75%
16 KB(PC, 5 offsets)5Accuracy-first~90%+
⚠
Why multi-offset matters: Figure 2 of the paper shows "common-origin, later-divergent" streams—two access patterns share the first offset but diverge later. A single-offset trigger can't distinguish them (low accuracy). Waiting for (PC, 5 offsets) validates the stream is continuing before committing to the prefetches from the 256-bit pattern.

3 · Region arbitration—choosing the best pattern

When multiple regions trigger simultaneously (e.g., both 2 KB and 16 KB PHTs hit), RICH uses a fixed priority to pick one:

  1. Highest: 2 KB by (PC, address) — ~80% accuracy, temporal+spatial.
  2. 16 KB — ~90% accuracy, large performance gain per correct prefetch.
  3. 4 KB — ~75% accuracy, medium gain.
  4. Lowest: 2 KB by (PC, offset) — ~60% accuracy, small gain.

Plus deduplication: the Inflight Prefetch Table tracks regions currently being prefetched. If a new trigger overlaps with an in-flight region, the new trigger is blocked unless it has higher priority than the in-flight one. This prevents lower-priority prefetches from issuing when a higher-priority prefetch already covers that space.

4 · Hierarchical storage—on-chip cache + off-chip PHT

To keep costs low, RICH analyzes metadata along three axes and partitions it:

MetadataSizeAccess frequencyLatency toleranceRICH's decision
2 KB / 4 KB PHTs~30 KBHighLowOn-chip
Training (FT/AT)~6 KBHighLowOn-chip
Control metadata~10 KBHighLowOn-chip
16 KB PHT>100 KBMixed (concentrated)HighOn-chip cache (256-entry) + off-chip (128 KB)

The 16 KB PHT is the only off-chip component. Key insights:

  • Latency tolerance: A single 16 KB pattern can trigger many cacheline prefetches from the '1' bits in the 256-bit vector. Even with 50 ns off-chip fetch latency, prefetch opportunities degrade <15% (Figure 5).
  • Access concentration: Table 3 shows 200 on-chip patterns cover 66–100% of requests for traces that benefit from 16 KB prefetching.
  • Prefetch count threshold: Only patterns with ≥30 prefetch counts (≥30 '1' bits) are written to off-chip, avoiding excessive off-chip traffic for sparse patterns.
🌱
Storage breakdown: 47.3 KB on-chip (2/4 KB PHTs, 16 KB cache, training, control) + 128 KB off-chip per core. The paper reports total on-chip storage of 47.3 KB. Bingo: 127.8 KB all on-chip. RICH uses 63% less on-chip area.

5 · The off-chip prediction process

When a (PC, 5 offsets) trigger for 16 KB misses in the on-chip PHT cache:

  1. Check Valid Map: a 4096-bit array indexed by 12-bit trigger ID. If bit = 0, no pattern exists; abort. If bit = 1, pattern is in off-chip PHT.
  2. Issue metadata read: off-chip PHT is an array (one-to-one mapping: trigger ID → pattern). Fetch the 256-bit pattern asynchronously.
  3. Track late prefetches: during the fetch, the processor may access some 2 KB sub-regions within the 16 KB space. Mark these at 2 KB granularity (8 bits for a 16 KB region).
  4. Trim and arbitrate: when the pattern returns, trim the marked 2 KB sub-regions (avoid useless late prefetches) and send to region arbitration. If no overlap, generate prefetch stream.
💡
Why async + late-prefetch tracking? The off-chip fetch can take time, during which the processor advances. Tracking which 2 KB sub-regions are already accessed prevents wasting bandwidth on data the processor already has. The pattern's many potential prefetches absorb the latency—most prefetches still arrive early.

Worked examples

Four traces, each carried all the way through—region selection, off-chip fetch, performance gains.

① Why 459.GemsFDTD-765B needs multi-region

Dense accesses occur within 2 KB sub-regions, but inter-page spatial correlations span 16 KB. Bingo (fixed region sizes) achieves only 0.54 coverage. How does RICH improve?

  1. RICH triggers both 4 KB and 16 KB regions during execution. The 16 KB trigger (PC, 5 offsets) validates the long stream.
  2. Region arbitration picks 16 KB (higher priority than 4 KB). Prefetches based on the learned 256-bit pattern—long stream issued early.
  3. Coverage rises to 0.78 (45% more useful prefetches). Accuracy for 4 KB and 16 KB regions: 0.92.
  4. Performance improves from Bingo's 182% to RICH's 199% — a 17% relative gain (from 1.82× to 1.99× speedup).
  5. Insight: Multi-region captures both small-object and large-array locality; arbitration ensures the right scale wins.
🌱
Key metric: coverage (fraction of misses eliminated) jumps from 0.54 to 0.78 because RICH can prefetch across larger boundaries—fixed-size prefetchers can't.

② Timeliness gains on 603.bwaves-891B

Large amounts of cross-page spatial locality. Bingo triggers small-region prefetches (timeliness: ~60%). How does RICH's 16 KB region improve timeliness?

  1. RICH's 16 KB trigger (PC, 5 offsets) fires after observing five distinct accesses in the region.
  2. A single trigger can yield many cacheline prefetches from the learned 256-bit pattern—long prefetch stream issued in one shot, leveraging high bandwidth.
  3. Timeliness improves by 35% (from ~60% to ~95%)—most prefetches arrive before the processor needs them.
  4. Performance gain: bwaves-891B speedup rises from Bingo's ~2.0× to RICH's ~2.3× (15% relative improvement).
  5. Insight: Large regions exploit high bandwidth to prefetch far ahead, hiding memory latency.
💡
Timeliness = fraction of useful prefetches already in cache when accessed. Many cachelines issued early from the pattern means most arrive before the processor catches up, eliminating stalls.

③ Off-chip fetch latency absorption (arbitrary trace)

A 16 KB trigger misses in the on-chip PHT cache. The off-chip fetch takes time. Does this delay kill performance?

  1. Valid Map check: bit = 1, pattern is in off-chip PHT. Issue async read to main memory.
  2. During the fetch: processor continues executing. Suppose it accesses 2 KB sub-regions 0 and 1 within the 16 KB space. Mark them (8-bit mask: 0b00000011).
  3. Pattern returns (256 bits). Trim sub-regions 0 and 1 (some cachelines → fewer cachelines remain).
  4. Send to region arbitration. No overlap with higher-priority in-flight prefetches. Generate prefetch stream from remaining '1' bits.
  5. Prefetch opportunity loss <15% (Figure 5 of the paper). The remaining prefetches are still enough to hide latency.
🌱
Key insight: The potential for many prefetches from the 256-bit pattern is the latency-tolerance mechanism. Even losing some prefetches to late accesses still leaves many—far more than a 2 KB region (32 cachelines maximum).

④ Storage overhead breakdown

Practice problems

Problem 1 · easy
A workload exhibits spatial locality within 8 KB regions but Bingo achieves only 60% coverage. Which RICH region size(s) would help, and why?
Show answer
The 16 KB region would help most. Although the locality is within 8 KB, RICH doesn't have an 8 KB region size—it has 2/4/16 KB. The 16 KB region (triggered by (PC, 5 offsets) for high accuracy) can capture the 8 KB pattern and prefetch it as a long stream. The 4 KB region would also help but wouldn't capture cross-region correlations within the 8 KB space.Misconception: that RICH needs an exact region size match. It uses the next-larger size (16 KB) and relies on the bit-vector to encode which cachelines are actually accessed within that region.
Problem 2 · medium
RICH's 16 KB trigger uses (PC, 5 offsets). Why not (PC, 10 offsets) for even higher accuracy?
Show answer
Figure 3a shows accuracy saturates around 5 offsets (~90%+) while coverage drops sharply beyond 5. Requiring 10 offsets makes the trigger too strict—most access streams never accumulate 10 distinct offsets before the region is evicted or the program moves to a different region. The lost coverage (missed prefetch opportunities) outweighs the marginal accuracy gain. RICH chooses 5 offsets as the sweet spot.Misconception: that more offsets always means better accuracy. True, but the coverage loss from waiting too long can hurt performance more than the accuracy gain helps.
Problem 3 · medium
The paper says "200 on-chip 16 KB patterns cover most requests" (Table 3). Why not make the on-chip cache 4096 entries (entire off-chip PHT) and skip off-chip entirely?
Show answer
Area cost. A 4096-entry on-chip PHT for 16 KB patterns would require massive on-chip storage—much more than RICH's entire on-chip budget (47.3 KB). The 200-entry coverage is "most" (66–100% of requests for traces that benefit from 16 KB), but not all. The off-chip PHT handles the long tail cheaply. The latency tolerance of 16 KB regions (many potential prefetches from the pattern) makes the rare off-chip fetch acceptable.Misconception: that on-chip is always better. It's better only if you can afford the area. RICH trades a small on-chip cache + rare off-chip fetches for massive area savings.
Problem 4 · hard
RICH's region arbitration uses fixed priority: 2KB(PC,addr) > 16KB > 4KB > 2KB(PC,off). Suppose a program has a 16 KB pattern with 95% accuracy and a 2 KB (PC,addr) pattern with 80% accuracy, both triggered. Which wins, and is this optimal?
Show answer
The 2 KB (PC,addr) wins (highest priority). Is this optimal? Not always. The 16 KB pattern has higher accuracy (95% vs 80%) and larger gain per correct prefetch (many cachelines from the pattern vs 32 maximum). But RICH uses a static priority, not dynamic—it can't know per-trigger accuracy at runtime. The fixed priority reflects average behavior across workloads: 2 KB (PC,addr) has high accuracy (temporal+spatial) and low misprediction cost, so it's always safe. In this specific case, dynamically choosing 16 KB would be better, but that would require per-trigger confidence tracking (too expensive). RICH trades static simplicity for near-optimal average-case performance.Misconception: that fixed priority is always right. It's a heuristic that works well on average but can be suboptimal for specific triggers. A fully dynamic system would need confidence counters and more complex arbitration logic.
Problem 5 · medium
The "prefetch count threshold" for 16 KB patterns is 30. Why not set it to 10 (more patterns written to off-chip) or 50 (fewer patterns, less bandwidth)?
Show answer
The threshold balances two costs: (1) off-chip write/read bandwidth, and (2) coverage loss from not storing patterns. A threshold of 10 would waste bandwidth on sparse patterns (e.g., 10 '1' bits in 256-bit vector = only 10 prefetches, not worth the off-chip access). A threshold of 50 would miss medium-density patterns that still provide useful prefetches. The paper chose 30 experimentally—enough prefetches to justify the off-chip access, not so high that useful patterns are lost.Misconception: that all patterns should be stored. Sparse patterns (few '1' bits) aren't worth the off-chip access cost in future high-latency memory.
Problem 6 · hard
RICH improves by 8.3% over Bingo at +120 ns memory latency but only 3.4% at baseline latency. Explain the trend using coverage, timeliness, and accuracy.
Show answer
RICH's advantage grows with latency because of timeliness. At baseline latency, Bingo's smaller-region prefetches arrive reasonably early. But at +120 ns, Bingo's prefetches can't get far enough ahead, and timeliness drops. RICH's 16 KB regions issue many-cacheline streams from learned patterns, prefetching far ahead—timeliness stays high even at +120 ns. The coverage advantage also helps, but the timeliness gap is what amplifies with latency. Higher latency = more stall cycles per late prefetch, so RICH's better timeliness translates to bigger performance gains.Misconception: that coverage alone explains the latency sensitivity. Coverage determines how many misses are eliminated; timeliness determines how early prefetches arrive. At high latency, timeliness matters more.
Problem 7 · easy
RICH uses virtual addresses for 16 KB regions (crossing page boundaries). What's the downside, and how might it be mitigated?
Show answer
Downside: prefetch addresses must be translated via TLB before issuing to memory. If TLB misses, the prefetch is delayed. The paper mentions this requires translation and that page-size-aware physical prefetching is future work (a recent study shows modern systems heavily use 2 MB pages). For 2 MB pages, a 16 KB region fits within one page, so physical addressing becomes more feasible. RICH's current design uses virtual addresses; future work could explore physical addressing with large pages.Misconception: that virtual addressing is free. It requires TLB lookups, which can bottleneck if TLB is small or heavily contended. RICH leaves this as future optimization.
Problem 8 · medium
RICH's deduplication mechanism blocks lower-priority prefetches when a higher-priority prefetch is in-flight. Does this ever hurt performance?
Show answer
Yes, if the higher-priority prediction is wrong. If a 16 KB or 2 KB (PC,addr) region triggers with high priority (90% or 80% accuracy, but still some misprediction), it blocks overlapping lower-priority prefetches. If the high-priority prediction was a misprediction, those blocked lower-priority prefetches would have been useful. The cost: bandwidth wasted on useless high-priority prefetches + missed opportunities for correct lower-priority prefetches. RICH mitigates this by the high accuracy of its triggers—mispredictions are rare enough that deduplication is a net win.Misconception: that deduplication is always correct. It's a heuristic: prioritize high-accuracy regions, accept rare losses when they mispredict.
Quick answers

Frequently asked questions

Ten questions newcomers ask most. Tap any to expand.

Q1Why not just scale up Bingo to 256-entry PHT and keep it all on-chip?▾
Area cost. Bingo already uses 127.8 KB for smaller fixed regions. Scaling to support 16 KB regions (256-bit patterns) would require much more on-chip space—too expensive. RICH gets the same coverage with 47.3 KB on-chip + 128 KB off-chip, using the latency tolerance of large regions to make off-chip cheap.
Q2What happens if two 16 KB triggers arrive in quick succession?▾
The on-chip PHT cache is set-associative (LRU replacement). If both triggers map to the same set and both miss in the cache, two off-chip fetches are issued. The Inflight Prefetch Table tracks both. If they overlap in memory space, the arbitration logic handles priority. If they don't overlap, both prefetch streams are generated. The off-chip PHT has enough bandwidth to handle a few concurrent fetches (128 KB is small; modern memory can serve it quickly).
Q3Does RICH work for non-spatial patterns (e.g., pointer-chasing)?▾
No. RICH is a spatial prefetcher—it assumes spatial locality (nearby addresses accessed together). Pointer-chasing exhibits temporal locality (same sequence of arbitrary addresses repeated) with no spatial correlation. For that, use temporal prefetchers like ISB/MISB/STMS, which store exact address sequences. RICH and temporal prefetchers are complementary—RICH handles spatial, temporal handles irregular.
Q4How does RICH compare to PMP (another recent spatial prefetcher)?▾
PMP merges similar patterns to save storage (4.3 KB on-chip). It achieves high coverage by pattern sharing, but lower accuracy (~66% vs RICH's 76.3%). RICH outperforms PMP by 4.4% at baseline, 6.2% at +120 ns latency. PMP's small on-chip area is impressive, but its accuracy/timeliness gaps hurt performance. RICH accepts more on-chip area (47.3 KB vs 4.3 KB) for better coverage and timeliness.
Q5What if a program's phases change rapidly (every 1M cycles)?▾
RICH re-trains patterns continuously (FT/AT updated on demand misses or hits on prefetches), but the PHT reflects learned patterns from the recent past. If phases flip every 1M cycles, the PHT may lag. The paper doesn't explicitly handle this—it's a known limitation of spatial prefetchers (SMS/Bingo have the same issue). Possible mitigation: per-PC pattern versioning or confidence counters, but that's future work.
Q6Why not use multiple region sizes per trigger (e.g., both 4 KB and 16 KB)?▾
Bandwidth and cache pollution. Prefetching both would issue overlapping cachelines (the 4 KB space is a subset of the 16 KB space). Deduplication prevents this—RICH picks the best region size via arbitration. Issuing both would increase prefetch bandwidth and potentially evict useful data from cache (pollution). The arbitration mechanism is designed to maximize useful prefetches per unit bandwidth.
Q7Does RICH support multi-core (interference between cores)?▾
Yes. Figure 19 shows 4-core results: RICH outperforms Bingo by 2.8% (slightly lower than single-core's 3.4% due to bandwidth contention). Each core has its own RICH instance (separate PHTs, training tables). Off-chip metadata fetches share the memory bus with demand accesses and other cores' prefetches, so bandwidth can be a bottleneck. The paper notes this as a tradeoff of RICH's aggressive prefetching.
Q8What's the latency of an off-chip metadata fetch?▾
The paper evaluates adding a 50 ns delay to model off-chip fetch latency in Figure 5. Actual latency depends on whether the metadata hits in row buffers, experiences bank conflicts, etc. Figure 5 shows prefetch opportunity loss <15% even with this additional delay, because the potential for many prefetches from the 256-bit pattern absorbs the latency.
Q9Could RICH's off-chip metadata affect the cache?▾
The off-chip 16 KB PHT is stored in a dedicated region of main memory. The paper mentions that evicted PHT-cache entries may be written to main memory through the cache hierarchy. This means metadata traffic could interact with the cache system, though the design aims to minimize this impact through the on-chip cache and threshold filtering.
Q10What did the authors leave for future work?▾
Several items: (1) Page-size awareness (use 2 MB pages to enable physical addressing for 16 KB regions). (2) Feedback-based tuning (dynamically adjust prefetch aggressiveness). (3) Multi-threaded workloads (current evaluation is single-threaded per core). (4) Integration with temporal prefetchers (RICH handles spatial, temporal handles irregular). (5) Power/energy analysis (not covered in the paper).

Core terms
RICHThe proposed prefetcher: multi-scale spatial patterns (2/4/16 KB regions) with hierarchical on-chip/off-chip storage.
Multi-offset triggerWaiting for (PC, N offsets) before triggering a prefetch—validates access stream continuity for high accuracy.
Region arbitrationFixed-priority mechanism that picks the best pattern when multiple regions trigger; deduplicates overlapping prefetches.
Hierarchical storageOn-chip cache (256-entry) for frequent 16 KB patterns; off-chip PHT (128 KB) for infrequent ones.
Latency tolerancePotential for many cacheline prefetches from learned patterns (many '1' bits in 256-bit vector for 16 KB regions) absorbs off-chip fetch latency without losing prefetch opportunities.
CoverageFraction of cache misses eliminated by prefetching. RICH achieves 75% (vs Bingo's 64%).
TimelinessFraction of useful prefetches already in cache when accessed. RICH's baseline aggregate metric is ~91%.
AccuracyFraction of prefetches that are used (not evicted unused). RICH achieves 76.3% (vs Bingo's 76.6%).
Supporting terms
PHT (Pattern History Table)Stores learned bit-vector patterns; RICH has separate PHTs for 2/4/16 KB regions.
FT (Filter Table)Tracks regions with few accesses to filter noise before learning a pattern.
AT (Accumulation Table)Accumulates the bit-vector as accesses occur before promoting to PHT.
Inflight Prefetch TableTracks in-flight prefetch regions to enforce deduplication (block overlapping lower-priority prefetches).
Valid Map4096-bit array indicating which 16 KB patterns exist in off-chip PHT (1 = pattern present, 0 = absent).
Prefetch count thresholdMinimum number of '1' bits in a 16 KB pattern (≥30) required to justify off-chip write/read (avoids sparse patterns).
Late prefetch trackingMarks 2 KB sub-regions accessed during off-chip fetch (8 bits per 16 KB region) to trim useless prefetches.
AMAT (Average Memory Access Time)Weighted sum of hit/miss latencies. RICH cuts AMAT by up to 30% vs Bingo (Figure 12b).
BingoState-of-the-art spatial prefetcher (HPCA '19): (PC,offset) and (PC,address) triggers, 127.8 KB on-chip.
PMPRecent lightweight spatial prefetcher (MICRO '22): pattern merging, 4.3 KB on-chip, lower accuracy than RICH.
SMSEarly spatial prefetcher (ISCA '06): fixed 2 KB regions, (PC,offset) trigger, set-associative PHT.

Lightweight spatial prefetcher that merges similar patterns to reduce storage (4.3 KB on-chip). Lower accuracy than RICH but impressive area efficiency.
