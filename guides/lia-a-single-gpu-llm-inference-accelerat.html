ISCA 2025 · A Field Guide

Running huge LLMs on a single GPU — by asking the CPU to actually help

GPT-scale models don't fit in one GPU. The old fix: use more GPUs (expensive). The new fix: LIA offloads computation to modern CPUs equipped with Advanced Matrix Extensions (AMX) and uses cheap CXL memory to store what doesn't fit.

📄 "LIA: A Single-GPU LLM Inference Acceleration" · Kim et al. 🎓 Assumes basic LLM inference + memory hierarchy knowledge ⏱ ~25 min read
🧮

Let the AMX-powered CPU handle layers where it can match GPU throughput — and store parameters in cheap CXL memory — not slow PCIe transfers or multi-GPU overhead.

Before you begin

Eight ideas LIA builds on

Each card clears up one concept. They're collapsed — tap any to open. Read in order the first time.

AMX-powered CPU GPU computation CXL memory Baselines (FlexGen, IPEX)
1LLM inference's two bottlenecks: memory capacity and PCIe bandwidth▾

Models like OPT-175B (350 GB parameters) don't fit in a single GPU's memory (e.g., H100 has 94 GB). The standard fix: memory-offloading — store parameters in CPU memory and transfer them to the GPU over PCIe as needed. But PCIe is slow (64 GB/s on H100 PCIe 5.0), so transferring 350 GB of parameters takes ~5 seconds per inference, dominating latency.

Batch size (B) and input length (L) also demand memory for intermediate values. For OPT-175B at B=256, L=1024, total memory requirement rises from 330 GB to 1.6 TB — far beyond any single GPU or even typical DDR capacity.

Now you're ready: LIA's two fixes are (1) offload compute to the CPU so less data crosses PCIe, and (2) store parameters in cheap CXL memory to handle huge batches.
2Prefill vs. decoding stages: different operations/byte▾

Prefill: processes the entire input token sequence (L tokens) in one shot to generate the first output token and populate the KV cache. Compute scales with B×L, so it's heavily compute-bound for long sequences. Decoding: generates one output token at a time, reusing the existing KV cache. Compute scales only with B (not L), so it's more memory-bound.

The operations/byte of sublayers varies wildly — e.g., the FC1 sublayer does 50,000 ops/byte during prefill for OPT-175B (B=180, L=512), while attention scoring (Q×K^T) does only 1 op/byte during decoding. This range is LIA's opportunity: high operations/byte sublayers can justify GPU use or CPU offload depending on transfer costs; low operations/byte sublayers may benefit from CPU offloading to avoid transferring large KV cache over PCIe.

Now you're ready: LIA picks which sublayers to offload by checking each one's operations/byte for the given B and L — if CPU throughput + memory bandwidth can handle it and avoid expensive transfers, offload; else use GPU.
3Why prior CPU offloading was weak: AVX couldn't compete▾

FlexGen and earlier frameworks offloaded only the least compute-intensive sublayer (attention scoring during decoding) to the CPU, because older CPUs with AVX512 had throughput <1% of a GPU's. For example, SPR with AVX512: 0.18 TFLOPS vs. A100's 19.5 TFLOPS. That 100× gap made CPU compute a last resort.

The paper shows that even with AVX's weak throughput, compute-offloading saves some PCIe transfer — but the CPU becomes a new bottleneck, sometimes increasing latency compared to memory-offloading alone (Figure 4: ~10 s CPU compute vs. 0.4 s KV cache transfer).

Now you're ready: LIA's entire premise rests on AMX being fast enough to justify offloading more sublayers. §4's microbenchmarks prove it is.
4Intel AMX: a 2D matrix accelerator inside the CPU▾

Starting with 4th-gen Xeon (Sapphire Rapids, 2023), Intel CPUs include Advanced Matrix Extensions (AMX): on-chip tiles (2D register arrays) and a Tile Matrix Multiply Unit (TMUL) that performs BF16 or INT8 fused multiply-add operations. Unlike 1D vector engines (AVX), AMX exploits 2D parallelism for much higher throughput per core, and performance scales with core count.

The paper benchmarks SPR (40-core) and GNR (Granite Rapids, 128-core). SPR-AMX hits 20 TFLOPS BF16 (theoretical 90.1 TFLOPS); GNR-AMX hits 40 TFLOPS. That's 4.5× and 9× higher than AVX512, respectively, and 11% and 22% of A100's throughput for GEMM workloads (Figure 5).

Now you're ready: With AMX, the CPU is no longer a toy accelerator — it's a viable co-processor for matrix-heavy LLM layers, especially during prefill.
5CXL memory: cheap capacity, but slower and lower bandwidth▾

Compute Express Link (CXL) is a memory interface built on PCIe that lets you connect DDR4 DRAM modules (repurposed from retired servers) to the CPU as a capacity expander. It requires 3× fewer pins than DDR5 and costs less per GB. But CXL adds 140–170 ns latency and provides only ~50% of DDR bandwidth (e.g., two CXL expanders interleaved = 34 GB/s vs. DDR5's 260 GB/s on SPR).

The paper shows that for parameters (which are streamed once per layer), CXL doesn't hurt GPU throughput — PCIe (64 GB/s) is the real bottleneck, not CXL. But for KV cache (reused many times during decoding), storing it in CXL degrades CPU throughput by up to 82% (Figure 8b). So LIA keeps parameters in CXL, KV cache and activations in DDR.

Now you're ready: CXL lets LIA support batch sizes up to 1.76× larger (e.g., B=900→1580) and deliver 1.45× higher throughput by offloading parameters to cheaper CXL without hurting performance.
6FlexGen's mini-batch overlapping: good for prefill, bad for decoding▾

FlexGen splits a batch into mini-batches and overlaps parameter transfers with computation to hide PCIe latency. For prefill, where compute time scales with B×L, this works well — pipeline bubbles are small. For decoding, where compute scales only with B, mini-batching actually hurts: the overhead of splitting reduces per-mini-batch compute time, so pipeline bubbles grow (the paper reports 1.1–1.3× latency increase for OPT-175B at B=900).

LIA overlaps transfers with full-batch computation during decoding, avoiding the mini-batch overhead. During prefill, it keeps mini-batch overlapping to hide GPU idle time.

Now you're ready: LIA's execution back-end (§5.2, Optimization-2) uses different overlapping strategies for prefill vs. decoding to maximize utilization without the mini-batch penalty.
7Why GPU memory goes mostly unused for small batches▾

At B=1, an entire layer of OPT-30B (parameters + activations) fits in ~1.2 GB. On an A100 with 40 GB, that leaves >90% of GPU memory idle. FlexGen stores parameters for all layers but only the sublayers that fit, so it can skip PCIe for those sublayers across layers. LIA does the opposite: store entire decoder layers (all sublayers) for as many layers as fit. This granular use of GPU memory is more efficient — e.g., LIA stores 62% of OPT-30B's layers vs. FlexGen storing 58% of sublayers (§5.2, Optimization-1).

Result: LIA skips PCIe transfers for entire layers, not just individual sublayers scattered across layers, reducing transfer overhead.

Now you're ready: LIA's policy treats GPU memory as a cache for whole layers, not a pool for sublayer fragments, which reduces CPU-GPU transfers and simplifies scheduling.
8Operations per byte: the key to offloading decisions▾

A sublayer with high operations/byte (e.g., FC1 during prefill: 50,000) is compute-bound — the GPU or CPU spends most time computing, not fetching data. A sublayer with low operations/byte (e.g., Q×K^T during decoding: 1) is memory-bound — most time is waiting on memory. The ratio changes with B and L: e.g., FC1 at B=1, L=64 has low operations/byte, but at B=900, L=1024 it's high.

LIA's offloading algorithm (Equation 1–9) computes for each sublayer whether the CPU's lower throughput but higher memory capacity is faster than the GPU's higher throughput but the cost of PCIe transfer. The decision depends on B, L, the sublayer's operations/byte, and the system's throughput + bandwidth parameters.

Now you're ready: LIA doesn't use a fixed offloading policy — it solves an optimization problem for every (B, L) pair to pick the offloading vector p = (p₁, ..., p₆) that minimizes latency.

The whole story in one place

LIA in a nutshell

The problem, the measurements that justify the fix, and how the pieces fit together.

Single-GPU LLM inference hits two walls: (1) memory capacity — models like OPT-175B (350 GB) don't fit in GPU memory, and (2) PCIe bandwidth — transferring parameters from CPU memory to the GPU takes seconds per inference. The standard fix, memory-offloading (FlexGen), stores everything in CPU memory and transfers on demand, but PCIe is so slow that transfers dominate latency (>98% for small batches, Figure 3). Adding compute-offloading (sending some layers to the CPU) helps only slightly because older CPUs with AVX512 are 100× slower than GPUs, making the CPU a new bottleneck.

LIA's insight: modern CPUs with AMX (Advanced Matrix Extensions) are fast enough to handle real LLM computation. Microbenchmarks (§4) show SPR-AMX delivers 20 TFLOPS (11% of A100) and GNR-AMX delivers 40 TFLOPS (22% of A100) for GEMM workloads — 4.5× and 9× higher than AVX512. That's competitive with some GPUs (P100: 9.3 TFLOPS, V100: 15.7 TFLOPS). For memory-bound GEMV, AMX achieves 38–44% of A100/H100 throughput.

Building on these results, LIA formulates an optimization problem (Equation 1–9) to systematically decide which sublayers to offload to the CPU for a given batch size B and input length L. The algorithm considers each sublayer's operations/byte, the volume of CPU-GPU transfer, and the compute throughput + memory bandwidth of both CPU and GPU. For example, for OPT-175B at B=900, L=1024, LIA offloads attention scoring (Q×K^T, S×V) to the CPU because it's memory-bound and the GPU-to-CPU transfer of KV cache (145 GB) would cost more than the CPU compute time; all other sublayers run on the GPU. For small B and short L, prefill may send all sublayers to CPU; for long input sequences at B=1, prefill can use the GPU for compute-bound layers while decoding offloads to CPU.

To handle batch sizes beyond DDR capacity (e.g., B=900 for OPT-175B needs 1.4 TB), LIA stores all parameters in cheap CXL memory while keeping KV cache and activations in DDR. CXL's bandwidth (34 GB/s with two expanders) is lower than DDR (260 GB/s), but it doesn't bottleneck GPU transfers (PCIe: 64 GB/s). The paper shows (Figure 8a) that GPU throughput is unaffected by CXL-offloaded parameters, while storing KV cache in CXL would degrade CPU throughput by 10–82% — so LIA keeps KV cache in DDR. Result: for OPT-30B at B=900, parameters offloaded to CXL range from 14.4% to 43.1% of memory depending on output length, with <1% throughput loss, or 1.76× larger batch sizes enabled, delivering 1.45× higher throughput.

The payoff: On SPR-A100 (SPR-H100), LIA achieves 1.8–7.3× (4.0–7.0×) lower latency for online inference and 1.2–6.0× (1.2–10×) higher throughput for offline inference compared to FlexGen. Against IPEX (CPU-only), LIA delivers 1.1–2.1× (1.1–2.5×) lower latency and 1.1–6.1× (1.2–10×) higher throughput. On GNR-A100, latency drops by 10–24× over FlexGen and throughput increases by 1.5–9.4×. Energy efficiency improves 1.1–5.8× over IPEX and 1.6–10.3× over FlexGen.

🍯 Key takeaways

  • AMX-enabled CPUs deliver 20–40 TFLOPS (4.5–9× higher than AVX), making them viable co-processors for LLM inference.
  • LIA solves an optimization problem to pick which sublayers to offload to the CPU based on operations/byte, transfer cost, and device throughput.
  • Offloading policy changes with batch size (B) and input length (L) — e.g., for OPT-175B, small B×L may send all sublayers to CPU during prefill, while large B sends attention to CPU during decoding.
  • CXL memory stores parameters (14–43% of memory for OPT-30B depending on output length) without hurting throughput; KV cache stays in DDR to avoid CPU performance degradation.
  • Two optimizations: (1) store entire decoder layers in unused GPU memory, and (2) different overlapping strategies for prefill vs. decoding.
  • Result: 1.8–24× lower latency, 1.2–10× higher throughput, 1.6–10× better energy efficiency vs. FlexGen; 1.1–2.5× lower latency, 1.1–10× higher throughput vs. IPEX.
  • GNR-A100 system costs 10% of multi-GPU DGX-A100 ($22K vs. $200K) while delivering 1.4–1.8× higher per-GPU throughput for B=1.
Seven everyday pictures

How to think about LIA

Each picture stands alone. Every card ends with an honest note about where the analogy breaks.

The core idea · Cooperative compute

Two chefs in one kitchen

The GPU is a fast chef who can chop onions at lightning speed but only has a tiny prep station. The CPU (with AMX) is slower but has a huge workspace. Instead of forcing the fast chef to run back and forth to the pantry (PCIe) for every ingredient, LIA hands some tasks to the big-workspace chef — like mixing batter (attention scoring) while the fast chef sears the steak (FC1). Both work at once, and the meal finishes faster.

Where it breaks: Real chefs don't share ingredients mid-recipe; LIA must coordinate which data (activations, parameters) crosses the PCIe "counter" between CPU and GPU, adding scheduling complexity the analogy hides.
AMX architecture · 2D parallelism

A spreadsheet vs. a list

AVX engines process vectors (1D lists) one element at a time, like adding a column in Excel. AMX's TMUL is a 2D array of multiply-add units — like selecting an entire spreadsheet range and applying a formula to every cell at once. The 2D structure delivers more operations per cycle because it exploits both row and column parallelism, not just one direction. That's why AMX hits 4.5–9× higher throughput than AVX for the same matrix sizes.

Where it breaks: A spreadsheet is infinitely large; AMX tiles are fixed-size (e.g., 16×64 bytes for BF16). Large matrices must be tiled and iterated, and the tiling overhead isn't captured by the spreadsheet analogy.
Offloading policy · Operations/byte

A tollbooth on a highway

Every byte of data that crosses PCIe pays a "toll" (transfer time). If a sublayer does 50,000 operations per byte (compute-intensive), paying the toll is worth it — the GPU speeds through those operations fast enough to justify the crossing. If a sublayer does only 1 operation per byte (memory-bound), the toll is too expensive relative to the work done; offload it to the CPU (where there's no toll) even if the CPU is slower. LIA's algorithm (Eq. 1–9) calculates the "toll vs. speedup" trade-off for each sublayer.

Where it breaks: Real tolls are fixed; PCIe latency varies with congestion and size. LIA models it as bandwidth × size, which is an average — bursty transfers or small packets add overhead the tollbooth analogy ignores.
CXL memory · Capacity vs. bandwidth

A storage unit vs. a closet

DDR memory is like a closet in your bedroom — small capacity but you can grab things instantly. CXL is a storage unit across town — huge capacity but slower access. For parameters (read once per layer), the storage unit works fine because you load them in bulk and don't revisit. For KV cache (read many times during decoding), using the storage unit would mean constant trips, slowing you down. LIA keeps parameters in CXL, KV cache in DDR — bulk storage for rarely-accessed data, fast closet for hot data.

Where it breaks: You can't combine a closet and storage unit into one address space; CXL and DDR are both exposed as NUMA memory, so the OS can page between them. LIA's policy ensures hot data stays in DDR to avoid involuntary paging.
Prefill vs. decoding · Different policies

Rush hour vs. night shift

During prefill (rush hour), there's tons of traffic (B×L tokens to process), so it's worth calling in the GPU (the express lane). During decoding (night shift), traffic is light (only B tokens), so the CPU can handle it without the overhead of coordinating with the GPU. LIA uses different offloading policies for the two stages: e.g., for OPT-175B at B=900, prefill uses p=(0,0,0,0,0,0) (all GPU), decoding uses p=(0,1,1,0,0,0) (attention on CPU, rest on GPU). One size doesn't fit both.

Where it breaks: Rush hour doesn't change rules mid-trip; LIA switches policies between prefill and decoding, which adds a one-time overhead of moving activations across PCIe if the policies differ.
Optimization-1 · Layer-granular caching

Parking entire cars vs. scattered parts

At B=1, GPU memory is 90% empty. FlexGen stores "a wheel from car 1, a door from car 2, …" (sublayers from all layers). LIA stores "entire car 1, entire car 2, …" (all sublayers of some layers). When you need a car (layer), it's faster to grab a complete one from the lot (GPU memory) than to run around collecting scattered parts from the warehouse (CPU memory). LIA's granular caching reduces PCIe transfers by storing coherent units.

Where it breaks: Cars don't share parts; LLM layers share attention patterns (KV cache), so "storing an entire car" still leaves some shared components (KV cache) in CPU memory. The analogy oversimplifies shared state.
Optimization-2 · Overlapping transfers

Loading the dishwasher while it runs

You don't wait for the dishwasher to finish before starting to hand-wash the next batch — you overlap. LIA overlaps PCIe transfers (loading parameters for the next layer) with computation (GPU/CPU working on the current layer). For prefill, it uses mini-batches to keep both devices busy. For decoding, it skips mini-batching (which adds overhead) and overlaps full-batch computation with transfers. Result: GPU and CPU idle time shrinks, throughput rises.

Where it breaks: A dishwasher has fixed capacity; GPU memory is dynamic (layers vary in size). LIA must carefully schedule transfers to avoid overfilling GPU memory mid-layer, adding complexity the dishwasher analogy hides.
The full explanation

How it actually works

Section by section, with the paper's key results and diagrams.

1 · The bottleneck — PCIe transfers dominate

Figure 3 (paper) shows that for OPT-175B at B=1, L=64, 98% of latency is spent transferring parameters from CPU to GPU over PCIe (31 s out of 31.6 s total). Even for long sequences (L=1024), transfers contribute 87% of prefill latency. For B=32, where KV cache and activations must also be offloaded (145 GB total), transfers still dominate. The root cause: PCIe 5.0 bandwidth (64 GB/s on H100) is 4× slower than DDR5 (260 GB/s on SPR) and 70× slower than HBM3 (3.35 TB/s on H100).

🌱
Concretely: Transferring OPT-175B's 350 GB parameters once takes 5.5 s. At B=1, L=64, the prefill stage alone processes 64 tokens → 64 PCIe transfers across 96 layers → 350 GB transferred 96 times during inference. That's the entire problem in one number.

2 · Compute-offloading's old problem — AVX is too slow

FlexGen offloads only attention scoring (Q×K^T, S×V) to the CPU to avoid transferring KV cache (145 GB at B=32, L=1024). But Figure 4 (paper) shows that the CPU compute time with AVX512 (1 s) is often longer than the KV cache transfer time (0.4 s), so offloading sometimes increases latency. The paper's microbenchmarks (§4) explain why: SPR with AVX512 delivers only 0.18 TFLOPS vs. A100's 19.5 TFLOPS — a 100× gap. Offloading is pointless when the CPU is 100× slower.

3 · AMX's throughput — competitive with some GPUs

The paper benchmarks SPR (40-core) and GNR (128-core) AMX using GEMM and GEMV microbenchmarks derived from OPT-175B sublayers. Key results (Figure 5):

SPR-AMX is 4.5× faster than AVX512 and exceeds P100's GEMM throughput. GNR-AMX doubles SPR-AMX, reaching 40 TFLOPS (22% of A100). For GEMV (memory-bound), AMX matches AVX512 (both limited by DDR bandwidth) but achieves 38% of A100's throughput — higher than P100. The key insight: AMX is fast enough to justify offloading compute-intensive sublayers, not just attention scoring.

💡
Why does GEMV match AVX512? GEMV is memory-bound, so throughput is limited by DDR bandwidth (260 GB/s on SPR), which both AMX and AVX512 share. AMX's 2D parallelism helps only for compute-bound tasks (GEMM).

4 · LIA's offloading algorithm — minimize latency

LIA formulates an optimization problem: for a given batch size B and input length L, pick the offloading vector p = (p₁, p₂, ..., p₆) (where p_i ∈ {0,1} indicates CPU=1 or GPU=0 for sublayer i) that minimizes the latency of computing one decoder layer. The latency is:

# Equation 2 from the paper
T(p) = Σi=1..6 [ Ti,load(p) + Ti,comp(p) + Ti,store(p) ]
     = load time + compute time + store time for each sublayer

# Load time (Eq. 4–6): if p_i ≠ p_{i-1}, transfer activation X_i over PCIe
Ti,load,X(p) = DX_i / BWPCIe  if CPU→GPU or GPU→CPU switch
Ti,load,Y(p) = DY_i / BWPCIe  if p_i = 1 (CPU computes, needs parameter Y_i)

# Compute time (Eq. 8): memory + compute latency
Ti,comp(p) = (DX_i + DY_i) / BW + C_i / TH
           where BW/TH = GPU's bandwidth/throughput if p_i=0, else CPU's

# Store time (Eq. 9): only for KV cache in sublayer 1
Ti,store(p) = DKV / BWPCIe  if i=1 and p_i=1 (CPU→GPU KV transfer)

The algorithm tries all 64 possible vectors p (2^6 combinations) and picks the one with minimum T(p). Table 1 (paper) lists D_X, D_Y, and C for each sublayer. For example, for OPT-175B (d_model=12,288) at B=180, L=512, the FC1 sublayer has D_X = 2BL·d_m = 4.4 MB, D_Y = 8d_m² = 2.4 GB, and C = 8BL·d_m² = 1.1 × 10^15 FLOPs. Operations/byte = C / (D_X + D_Y) ≈ 458, highly compute-bound.

🍯
The key: operations/byte changes dramatically with B and L. At B=1, L=64, FC1's operations/byte drops to 16 (memory-bound) → offload to CPU. At B=900, L=1024, it's 50,000 (compute-bound) → use GPU. LIA's policy adapts.

5 · Optimal policies by batch size

Figure 9 (paper) shows the optimal offloading policies for OPT-175B across B and L. Three main policies emerge:

  • p = (1,1,1,1,1,1): Full CPU offloading. Used for small B and L where compute is minimal and PCIe transfers dominate anyway. Prefill depends on B×L (transition around 850 for OPT-175B); for long inputs at B=1, prefill can be GPU-heavy. All sublayers run on CPU; no GPU used for small B×L.
  • p = (0,1,1,0,0,0): Partial CPU offloading. Used for medium-large B (e.g., B=64–900) during decoding where attention scoring (sublayers 2, 3) is memory-bound and benefits from CPU's large memory, while other sublayers are compute-bound and use the GPU.
  • p = (0,0,0,0,0,0): Full GPU compute. Used for very large B×L (e.g., B=256, L=1024 during prefill) where all sublayers become compute-bound and the GPU's high throughput dominates.

The transition between policies happens at B×L ≈ 850 for prefill and B < 858 for decoding (for OPT-175B on SPR-A100). These thresholds shift with CPU/GPU capabilities: on SPR-H100, the GPU's higher throughput pushes the transition to higher B×L values (more aggressive GPU use).

6 · CXL offloading — cheap capacity without throughput loss

For B=900, OPT-175B needs 1.4 TB total memory (350 GB parameters + 1.05 TB KV cache/activations). DDR capacity is limited by channel count (8 channels on SPR → 512 GB max), so LIA uses CXL memory to store parameters. Figure 8a (paper) shows that CXL-to-GPU transfer bandwidth (34 GB/s with two expanders interleaved) approaches DDR-to-GPU bandwidth for large transfers (>300 MB per sublayer), because both are bottlenecked by PCIe (64 GB/s on H100). So storing parameters in CXL doesn't hurt GPU throughput.

But Figure 8b shows that storing KV cache in CXL degrades CPU throughput by 10–82% because the CPU repeatedly accesses KV cache during attention scoring (low operations/byte), making it memory-bandwidth-sensitive. LIA's policy: parameters in CXL, KV cache in DDR. Result (Table 3): for OPT-30B at B=900 with varying output lengths, offloaded percentage ranges from 14.4% to 43.1% depending on output length, with <1% throughput loss, or enables 1.76× larger batch sizes (B=900→1580) for 1.45× higher throughput.

⚠
Edge case: If B is so large that even KV cache doesn't fit in DDR, LIA must page KV cache to CXL, accepting the throughput degradation. The paper evaluates up to B=1600, where this threshold is reached for OPT-175B.

7 · Two performance optimizations

Optimization-1: Layer-granular GPU memory use. FlexGen stores sublayers of all layers in unused GPU memory; LIA stores all sublayers of some layers. For OPT-30B at B=1, LIA stores 62% of layers (35 GB) vs. FlexGen storing 58% of sublayers (32 GB). This reduces PCIe transfers because entire layers can be computed on GPU without any transfers. Table 4 shows this optimization reduces latency by 2× at B=1 (10.09 s → 5.05 s).

Optimization-2: Different overlapping for prefill vs. decoding. During prefill, LIA uses mini-batch overlapping (like FlexGen) to hide PCIe latency. During decoding, it overlaps full-batch computation with parameter transfers (unlike FlexGen) to avoid the mini-batch overhead that increases latency by 1.1–1.3× (§7.2). Table 4 shows overlapping reduces latency by 1.5× at B=900 (443.61 s → 291.02 s).

Follow the numbers, step by step

Worked examples

Four traces from the paper, carried all the way through.

① Computing operations/byte for FC1 sublayer

OPT-175B (d_model=12,288) at B=180, L=512. FC1 multiplies (B×L, d_m) by (d_m, 4×d_m). What's the operations/byte?

  1. Data size: D_X = 2BL·d_m bytes (activation), D_Y = 8d_m² bytes (parameter). For BF16, that's 2·180·512·12,288 = 2.25 GB and 8·12,288² = 2.42 GB. Total = 4.67 GB.
  2. Compute: C = 8BL·d_m² FLOPs (matrix multiply). That's 8·180·512·12,288² = 1.1 × 10^15 FLOPs.
  3. Operations/byte: C / (D_X + D_Y) = 1.1e15 / 4.67e9 ≈ 235 ops/byte. Highly compute-bound.
  4. Compare to PCIe transfer time: Transferring 4.67 GB over PCIe 5.0 (64 GB/s) takes 73 ms. Computing 1.1e15 FLOPs on A100 (194 TFLOPS) takes 5.7 s. Compute dominates → use GPU.
  5. Change the parameters: At B=1, L=64, operations/byte drops to 16 (memory-bound) → offload to CPU.
🍯
Insight: Operations/byte is the key signal. Above ~100, use GPU; below ~10, use CPU. LIA's algorithm (Eq. 1–9) formalizes this trade-off for all sublayers and (B, L) pairs.

② Why FlexGen's mini-batch overlapping hurts decoding

OPT-175B at B=900, L=1024. FlexGen splits into mini-batches (e.g., 4×225). How does this affect latency?

  1. Prefill stage: Compute time scales with B×L. Mini-batch 1 (225×1024) takes T1 = C / TH. While GPU computes mini-batch 1, CPU transfers parameters for the next layer. Pipeline is balanced.
  2. Decoding stage: Compute time scales only with B (generating one token per batch entry). Mini-batch 1 (225) takes T1 = C / TH. But now T1 is 4× smaller (225 vs. 900), so the parameter transfer (fixed cost) becomes a larger fraction of total time.
  3. The overhead: Pipeline bubbles grow because the GPU finishes mini-batch 1 before the parameter transfer completes. The paper reports 1.1–1.3× latency increase for OPT-175B at B=900 (decoding) with mini-batching.
  4. LIA's fix: Overlap full-batch (B=900) computation with parameter transfers during decoding. The longer compute time hides the transfer latency without the mini-batch overhead.
  5. Result: Table 5 shows LIA's latency at B=900 is 291 s vs. FlexGen's ~450 s (estimated from Figure 11) — 1.5× lower.
💡
Mini-batch overlapping is an optimization for compute-bound stages (prefill) but becomes an anti-optimization for memory-bound stages (decoding) where compute time doesn't scale linearly with batch size.

③ Computing CXL's memory savings

OPT-30B at B=900, L_in=32, L_out=32. How much memory can LIA offload to CXL?

  1. Total memory: Parameters = 60 GB (OPT-30B is half of OPT-66B), KV cache = B×L×d_m×layers×2 ≈ 900×32×5120×48×2 ≈ 28 GB, activations ≈ 5 GB. Total = 93 GB.
  2. LIA's policy: Store parameters (60 GB) in CXL, keep KV cache (28 GB) and activations (5 GB) in DDR. DDR usage = 33 GB.
  3. Percentage offloaded: Table 3 shows offloaded percentage varies from 14.4% to 43.1% depending on output length (L_out=32 to 128). For L_out=32, the offloaded percentage is lower because KV cache and activations are a larger fraction of total memory.
  4. Throughput impact: Table 3 shows throughput = 280.38 tokens/s without CXL, 280.50 tokens/s with CXL (<1% difference). CXL doesn't hurt performance because parameters are streamed once per layer, and PCIe is the bottleneck, not CXL bandwidth.
  5. Enabling larger batches: With CXL, LIA can increase B from 900 to 1580 (1.76×) within the same DDR capacity, delivering 406.88 tokens/s (1.45× higher throughput).
🍯
Insight: CXL is a capacity expander, not a bandwidth bottleneck, when used correctly. LIA's policy ensures hot data (KV cache) stays in DDR while cold data (parameters) goes to CXL.

④ Scaling from SPR to GNR — throughput doubles

SPR-AMX (40-core) vs. GNR-AMX (128-core). How much faster is GNR for LLM inference?

  1. Core count: GNR has 128 cores vs. SPR's 40 → 3.2× more cores. But AMX performance doesn't scale perfectly linearly due to memory bandwidth limits and inter-core communication overhead.
  2. GEMM throughput: SPR-AMX = 20 TFLOPS, GNR-AMX = 40 TFLOPS (Figure 5) → 2× throughput. This is the compute-bound limit.
  3. GEMV throughput: SPR-AMX = 199 GFLOPS (limited by 8×DDR5-4800 = 260 GB/s), GNR-AMX = 338 GFLOPS (12×DDR5-5600 = 442 GB/s) → 1.7× throughput. Memory bandwidth doesn't scale 3.2× because DDR channel count only increases 1.5×.
  4. End-to-end latency: Table 6 shows GNR-A100 achieves 1.1–1.2× lower latency vs. SPR-A100 for OPT-175B online inference (B=1). The modest gain is because at B=1, most sublayers are offloaded to GPU (policy p=(0,0,0,0,0,0)), so CPU throughput doesn't dominate.
  5. Offline throughput: Table 6 shows GNR-A100 achieves 1.1–4.1× higher throughput vs. SPR-A100 for OPT-175B offline inference (B=64–900). The larger gain at high B is because more sublayers are offloaded to CPU (policy p=(0,1,1,0,0,0)), so AMX's 2× throughput directly improves end-to-end performance.
💡
GNR's benefit depends on the offloading policy. For GPU-heavy policies (small B), GNR helps only slightly. For CPU-heavy policies (large B), GNR's 2× AMX throughput delivers near-2× speedup.

Practice problems

Try each before opening the answer. Every one ends with the common misconception it's checking for.

Problem 1 · easy
LIA stores parameters in CXL memory and KV cache in DDR. Why not store both in CXL to maximize cost savings?
Show answer
KV cache is accessed repeatedly during decoding (every attention scoring operation), making it memory-bandwidth-sensitive. Storing it in CXL degrades CPU throughput by 10–82% (Figure 8b) because CXL bandwidth (~34 GB/s) is only 13% of DDR (~260 GB/s). Parameters are streamed once per layer, so CXL's lower bandwidth doesn't bottleneck GPU throughput (PCIe is the limit). LIA keeps KV cache in DDR to avoid the throughput hit.Misconception: that CXL is "just slower DDR" and can replace it uniformly. CXL works only for cold data (parameters), not hot data (KV cache).
Problem 2 · medium
For OPT-175B at B=900, L=1024, LIA uses policy p=(0,1,1,0,0,0) during decoding. Which sublayers run on the CPU, and why?
Show answer
Sublayers 2 and 3 (Q×K^T and S×V, attention scoring) run on the CPU. These sublayers have operations/byte = 1 (memory-bound) because they repeatedly access the KV cache (size = B×L×d_m). Transferring the KV cache (145 GB) over PCIe would take longer than the CPU compute time, so offloading to CPU avoids the transfer. Sublayers 1, 4, 5, 6 (QKV mapping, output projection, FC1, FC2) have high operations/byte and run on the GPU.Misconception: that LIA always offloads the same sublayers. The offloading policy changes with B and L — e.g., at B=1, all sublayers may go to CPU for small B×L.
Problem 3 · medium
AMX delivers 20 TFLOPS on SPR (40-core). A100 delivers 194 TFLOPS. Why does LIA achieve competitive end-to-end performance despite AMX being 10× slower?
Show answer
LIA avoids PCIe transfers by offloading compute to the CPU. For example, at B=1, transferring parameters (350 GB) over PCIe (64 GB/s) takes 5.5 s. Computing the same layers on the CPU with AMX takes ~10 s, but the total latency is 10 s (CPU-only) vs. 31 s (GPU with PCIe transfers). The 5.5 s transfer overhead is eliminated, making the 10× slower CPU faster end-to-end. For larger B, LIA offloads only memory-bound sublayers to CPU, using the GPU for compute-bound ones, so the 10× gap matters less.Misconception: that raw throughput (TFLOPS) determines end-to-end performance. LIA's advantage comes from eliminating bottlenecks (PCIe), not from matching GPU throughput.
Problem 4 · hard
FlexGen stores 58% of sublayers in GPU memory; LIA stores 62% of layers. For OPT-30B at B=1, which approach reduces PCIe transfers more, and why?
Show answer
LIA reduces transfers more because it stores entire layers (all 6 sublayers), not scattered sublayers across all layers. When computing layer i, if it's stored in GPU memory, no PCIe transfer is needed for any sublayer. FlexGen stores sublayers of all layers, so even if sublayer j is in GPU memory, the other 5 sublayers of layer i must still be transferred. The paper shows LIA uses 35 GB of GPU memory (62% of 48 layers = 30 layers × 1.2 GB/layer) vs. FlexGen's 32 GB (58% of 6 sublayers × 48 layers ≈ 168 sublayers). LIA's layer-granular caching reduces transfers by 2× (Table 4: 10.09 s → 5.05 s).Misconception: that storing more sublayers (58% vs. 62%) always reduces transfers more. It's the granularity of storage (whole layers vs. scattered sublayers) that matters, not just the percentage.
Problem 5 · medium
LIA's offloading algorithm (Eq. 1–9) tries all 64 possible policies (2^6). For a 128-layer model, why doesn't this explode to 2^768 evaluations?
Show answer
The offloading policy applies uniformly to all layers — the same vector p = (p₁, ..., p₆) is used for every decoder layer. So the search space is 2^6 = 64, not 2^(6×128). The paper's assumption is that all layers have identical structure (same sublayers, same parameters), so the optimal policy for one layer is optimal for all. The exception is the embedding/encoding layer and LM head, which have different structures, but these are computed separately outside the optimization.Misconception: that LIA optimizes each layer independently. It applies one policy to all decoder layers, which is justified by the fact that all decoder layers share the same structure in Transformer models.
Problem 6 · hard
Table 6 shows GNR-A100 delivers 10–24× lower latency vs. FlexGen for OPT-175B. Is this gain from AMX (GNR vs. SPR) or from the offloading algorithm (LIA vs. FlexGen)?
Show answer
Mostly from the offloading algorithm. SPR-A100 already delivers 8.5–12× lower latency vs. FlexGen (Figure 10). The GNR upgrade (2× AMX throughput) adds another 1.1–1.2× gain (Table 6: GNR-A100 = 1.1–1.2× SPR-A100), so the total is 8.5×1.2 = 10–24×. The algorithm's gain (8.5–12×) dominates because it eliminates PCIe transfers by offloading more sublayers to the CPU — a structural advantage that holds even with slower CPUs. GNR's 2× throughput helps, but it's a multiplier on top of the algorithm's benefit, not the primary source.Misconception: that newer hardware (GNR) is the main driver. LIA's offloading algorithm is the 10× improvement; GNR adds another 1.2× on top.
Problem 7 · easy
LIA uses mini-batch overlapping during prefill but not during decoding. What's the difference that makes one work and the other fail?
Show answer
Prefill compute scales with B×L (all input tokens processed at once), so splitting into mini-batches (e.g., 4×225) reduces per-mini-batch compute time proportionally (4×). The PCIe transfer time (fixed cost) can be overlapped with the 4× shorter compute, hiding most of the latency. Decoding compute scales only with B (one output token per batch entry), so splitting into mini-batches reduces compute time by 4× but doesn't reduce the transfer cost — the pipeline bubble grows, increasing latency by 1.1–1.3×. LIA avoids mini-batching during decoding to prevent this overhead.Misconception: that overlapping always helps. It helps when compute time dominates (prefill), but hurts when transfer time dominates (decoding).
Problem 8 · medium
Figure 14 shows GNR-A100 delivers 1.4–1.8× higher per-GPU throughput vs. 8×A100 DGX for OPT-175B at B=1. Why does a single GPU beat multi-GPU?
Show answer
At B=1, the multi-GPU system uses 8-way tensor parallelism, splitting each layer across 8 GPUs. This introduces inter-GPU communication overhead (AllReduce after each layer) that doesn't exist in the single-GPU setup. The paper shows LIA on GNR-A100 has 30–33% lower per-GPU throughput than DGX-A100 at B=64 (Figure 14), which suggests communication overhead and different workload characteristics. For B=1, LIA avoids this overhead by running entirely on one GPU (with CPU offloading for memory-bound sublayers), so there's no inter-GPU communication. The 1.4–1.8× advantage is from eliminating the multi-GPU overhead, not from AMX being faster than A100.Misconception: that more GPUs always mean higher throughput. For small batches, multi-GPU overhead can dominate, making a single well-utilized GPU faster.
Quick answers

Frequently asked questions

Twelve questions newcomers ask most. Tap any to expand.

Q1What is AMX and why does it matter for LLM inference?▾
AMX (Advanced Matrix Extensions) is a built-in matrix-multiplication accelerator in Intel CPUs starting with 4th-gen Xeon (Sapphire Rapids, 2023). It uses 2D tile registers and a Tile Matrix Multiply Unit (TMUL) to deliver 20–40 TFLOPS BF16 — 4.5–9× higher than AVX512. This makes CPUs viable co-processors for LLM inference, not just data movers. LIA leverages AMX to offload compute-intensive sublayers to the CPU, reducing PCIe transfers and end-to-end latency.
Q2Why does LIA beat FlexGen by up to 24× despite using the same GPU?▾
FlexGen uses AVX512 (0.18 TFLOPS) for CPU offloading, so it can only offload the least compute-intensive sublayer (attention scoring) without becoming a bottleneck. LIA uses AMX (20–40 TFLOPS), which is 100× faster, allowing it to offload more sublayers and eliminate more PCIe transfers. For example, at small B×L, LIA offloads all sublayers to CPU, avoiding 98% of PCIe transfers that dominate FlexGen's latency. The 24× gain on GNR-A100 is from both AMX's throughput and LIA's offloading algorithm.
Q3How does LIA decide which sublayers to offload to the CPU?▾
LIA solves an optimization problem (Eq. 1–9) to pick the offloading vector p = (p₁, ..., p₆) that minimizes latency for a given batch size B and input length L. The algorithm considers each sublayer's operations/byte, the volume of CPU-GPU transfer, and the compute throughput + memory bandwidth of both CPU and GPU. It tries all 64 possible policies (2^6 combinations) and picks the one with minimum latency. The optimal policy changes with B and L — e.g., for OPT-175B, small B×L may use p=(1,1,1,1,1,1) at B=1 during prefill, p=(0,1,1,0,0,0) at B=900 during decoding.
Q4What is CXL memory and how does LIA use it?▾
CXL (Compute Express Link) is a memory interface built on PCIe that connects DDR4 DRAM modules (repurposed from retired servers) to the CPU as a capacity expander. It's cheaper than DDR5 but has higher latency (140–170 ns) and lower bandwidth (~34 GB/s vs. 260 GB/s for DDR). LIA stores all parameters in CXL (because they're streamed once per layer) and keeps KV cache in DDR (because it's accessed repeatedly). This offloads 14–43% of memory to CXL depending on output length with no throughput loss, or enables 1.76× larger batch sizes for 1.45× higher throughput (Table 3).
Q5Why does LIA use different overlapping strategies for prefill vs. decoding?▾
Prefill compute scales with B×L (all input tokens processed at once), so mini-batch overlapping reduces per-mini-batch compute time and hides PCIe transfer latency. Decoding compute scales only with B (one output token per batch entry), so mini-batching reduces compute time but doesn't reduce transfer cost — the pipeline bubble grows, increasing latency by 1.1–1.3×. LIA uses mini-batch overlapping for prefill and full-batch overlapping for decoding to avoid the mini-batch overhead.
Q6How much hardware overhead does LIA add?▾
None — LIA uses existing hardware (AMX-enabled CPU, NVIDIA GPU, optional CXL memory). The software overhead is minimal: the offloading algorithm runs once per (B, L) pair and takes <1 ms to evaluate 64 policies. The execution back-end (modified Intel IPEX + HuggingFace Transformers) adds no runtime overhead beyond the coordination of CPU-GPU data transfers, which are already present in FlexGen. LIA's advantage is algorithmic, not hardware.
Q7Does LIA work for models other than OPT?▾
Yes — LIA generalizes to any decoder-only Transformer model with multi-head attention, such as GPT, Llama2, and Gemini. The paper evaluates Llama2-70B, Chinchilla-70B, and Bloom-176B using an analytical model and shows consistent gains: 6.1–8.4× (1.4–7.6×), 7.4–10× (1.3–5.0×), and 7.6–11× (1.2–4.9×) lower latency (higher throughput) vs. FlexGen for the three models, respectively. For models like Mixture of Experts (MoE), the diversity of offloading policies increases (e.g., p=(0,1,1,0,1,1) instead of (0,1,1,0,0,0)), but the algorithm still applies.
Q8Why doesn't LIA just use a faster GPU instead of offloading to the CPU?▾
Faster GPUs (e.g., H100 instead of A100) don't solve the memory capacity problem — OPT-175B (350 GB) still doesn't fit in H100's 94 GB memory. Multi-GPU deployments (e.g., 5×H100 = $150K) solve capacity but introduce inter-GPU communication overhead and are cost-prohibitive for most users. LIA's approach (single GPU + AMX CPU + CXL memory = $22K) is 7× cheaper and delivers competitive performance by eliminating PCIe transfers through compute offloading. Figure 14 shows GNR-A100 achieves 1.4–1.8× higher per-GPU throughput vs. 8×A100 DGX for B=1.
Q9How does LIA compare to Grace-Hopper systems?▾
Grace-Hopper has 900 GB/s CPU-GPU bandwidth (14× higher than PCIe 5.0) but lower CPU compute throughput (6.91 TFLOPS vs. GNR's 40 TFLOPS). The paper's analytical model shows that LIA on Grace-Hopper achieves 1.8–2.3× lower latency and 3.0–4.1× higher throughput than GNR-H100 (not GNR-A100) for OPT-175B. This suggests that improving CPU-GPU bandwidth may be more effective than increasing CPU compute power for this specific comparison, though the paper does not evaluate Grace-Hopper experimentally.
Q10What's the energy efficiency of LIA compared to baselines?▾
LIA achieves 1.1–5.8× higher energy efficiency (energy/token) vs. IPEX and 1.6–10.3× vs. FlexGen (Figure 12). The advantage comes from two sources: (1) shorter inference latency reduces static power consumption (e.g., 5.05 s vs. 31 s → 6× less idle power), and (2) GPUs are more energy-efficient than CPUs for compute-intensive tasks (e.g., A100 = 400W TDP for 194 TFLOPS; SPR = 350W TDP for 20 TFLOPS → 10× higher TFLOPS/W). LIA capitalizes on this by offloading compute-intensive sublayers to the GPU and keeping memory-bound ones on the CPU.
Q11Can LIA scale to multi-GPU systems?▾
The paper's discussion section mentions extending LIA to multi-GPU systems with Tensor Parallelism as future work. As GPU compute throughput and CPU-GPU bandwidth scale with GPU count, GPUs will handle computation more frequently than in a single-GPU setup. Figure 14 shows that LIA on GNR-A100 has 30–33% lower per-GPU throughput than DGX-A100 at B=64, which the paper attributes to different workload characteristics and potential communication overhead in multi-GPU setups. For cost-efficiency, single-GPU LIA systems outperform multi-GPU for small batches (B=1), while multi-GPU is better for very large batches (B>1000).
Q12What did the authors leave for future work?▾
The paper's discussion section (§8) identifies extending LIA to multi-GPU systems with Tensor Parallelism and co-designing CPU and GPU architectures for better CPU-GPU bandwidth as future work. The current work focuses on BF16 inference without compression. Other potential extensions like quantization integration or alternative memory technologies are not explicitly mentioned in the paper's future work section.

Glossary & further reading

Core terms
LIA (LLM Inference Acceleration)The paper's framework for single-GPU LLM inference using AMX-enabled CPU-GPU cooperative computing and CXL memory offloading.
AMX (Advanced Matrix Extensions)Intel CPU accelerator with 2D tile registers and TMUL unit, delivering 20–40 TFLOPS BF16 (4.5–9× higher than AVX512).
CXL (Compute Express Link)Memory interface built on PCIe that connects DDR4 DRAM to CPU as capacity expander; cheaper but slower than DDR5.
SPR (Sapphire Rapids)Intel 4th-gen Xeon Scalable Processor (40-core), first with AMX support; delivers 20 TFLOPS BF16.
GNR (Granite Rapids)Intel 6th-gen Xeon Scalable Processor (128-core), 3.2× core count vs. SPR; delivers 40 TFLOPS BF16.
Operations/byteRatio of compute (FLOPs) to data transferred (bytes) for a sublayer; high ops/byte → compute-bound, low → memory-bound.
Offloading policy (p)Vector p = (p₁, ..., p₆) where p_i ∈ {0,1} indicates CPU=1 or GPU=0 for sublayer i; LIA optimizes p for each (B, L) pair.
Memory-offloadingStoring parameters and intermediate values in CPU memory and transferring to GPU on demand; baseline approach of FlexGen.
Compute-offloadingSending computation to the CPU to avoid PCIe transfers; LIA extends this beyond FlexGen's single-sublayer offloading.
Supporting terms
Prefill stageProcessing the entire input token sequence (L tokens) to generate the first output token and populate KV cache; compute scales with B×L.
Decoding stageGenerating one output token at a time, reusing existing KV cache; compute scales with B, not L.
KV cacheKey-Value matrices stored during prefill and reused during decoding for attention scoring; grows with B×L and is batch-specific.
GEMM (General Matrix Multiply)Compute-bound operation (e.g., FC1 sublayer); high operations/byte, benefits from GPU's high TFLOPS.
GEMV (General Matrix-Vector multiply)Memory-bound operation (e.g., Q×K^T during decoding); low operations/byte, limited by memory bandwidth.
TMUL (Tile Matrix Multiply Unit)2D array of multiply-add units in AMX; exploits row and column parallelism for higher ops/cycle than 1D AVX.
Mini-batch overlappingSplitting a batch into smaller mini-batches to overlap PCIe transfers with computation; helps for prefill, hurts for decoding.
Layer-granular cachingStoring entire decoder layers (all sublayers) in GPU memory, not scattered sublayers across all layers; reduces PCIe transfers.
PCIe bandwidth64 GB/s on H100 (PCIe 5.0), 32 GB/s on A100 (PCIe 4.0); the bottleneck for memory-offloading frameworks.
DDR5 bandwidth260 GB/s on SPR (8 channels × DDR5-4800), 442 GB/s on GNR (12 channels × DDR5-5600).
FlexGenLatest GPU+AVX offloading framework; baseline for comparison; offloads only attention scoring to CPU.
IPEX (Intel Extension for PyTorch)CPU-only framework using AMX; baseline for comparison; no GPU offloading.
