Quick Glossary — Key Acronyms & Terms

SU-LLM State Update-based LLM—post-transformer models (Mamba-2, RetNet, GLA, HGRN2) that share a "state update" operation.
PIM Processing-in-Memory—compute logic inside each DRAM bank to avoid data transfer bottlenecks.
SPU State-update Processing Unit—Pimba's accelerator shared across two banks via access interleaving.
MX8 Microsoft's block floating-point format (8-bit avg per value)—Pareto-optimal for area-efficiency & accuracy.
UMON Utility Monitor (from prior UCP work)—here repurposed concept; Pimba monitors state via bank counters.
KV cache Key-Value cache in transformers—grows linearly with sequence length; state in SU-LLMs is constant.
1Why transformers are expensive (and why post-transformers exist)▾

Transformer LLMs use attention to model dependencies between tokens. During generation, they store a growing KV cache (one entry per past token) to avoid recomputation. Memory usage and latency both scale linearly with sequence length—fine for short contexts, prohibitive for millions of tokens. Recent apps (test-time scaling, RAG, multimodal) demand long contexts, forcing hyperscalers to deploy hundreds of thousands of GPUs at $30k+ each.

Post-transformer LLMs (SSMs like Mamba-2, linear attention like RetNet/GLA, RNNs like HGRN2) replace attention with a constant-size state that's updated each token. No growing cache → constant memory and compute regardless of length. Figure 1(a) in the paper shows Mamba-2 uses 2.3× less memory and delivers 2.6× higher throughput than a same-size transformer, with 4.5% better accuracy.

Now you're ready: Pimba targets this class of post-transformer models, accelerating their core "state update" operation—plus attention for hybrid models like Zamba2.
2The shared "state update" pattern across post-transformers▾

Despite algorithmic differences, many post-transformers (Mamba-2, RetNet, GLA, HGRN2) share a common computational structure. At each token step, they (1) decay the previous state matrix by an element-wise multiplication with a vector 𝑑ₜ, (2) compute an outer product of vectors 𝑘ₜ and 𝑣ₜ to capture the new token's info, (3) add that outer product to the decayed state, and (4) dot-product the updated state with a query vector 𝑞ₜ to produce output.

This unified operation is called state update. It's memory-bound (low arithmetic intensity) and not reused across user requests—each request reads, updates, and writes its own state. Hence it dominates latency at large batch sizes (Figure 3: RetNet spends 73.8% of time on state updates at batch 128).

Now you're ready: Pimba's architecture is designed around this shared pattern, not a single model—so it generalizes across SU-LLMs.
3Processing-in-Memory (PIM) and why it helps bandwidth-bound ops▾

DRAM is organized hierarchically: channels → ranks → bank groups → banks. Each bank stores data in a matrix. Normally, to compute on data you (1) activate a row into the bank's sense amplifier (row buffer), (2) select a column, and (3) send the data over the DRAM channel to the GPU. Only one bank can use the channel at a time, so channel bandwidth is the bottleneck.

PIM places small compute logic inside each bank. Operations execute during steps 1–2, reading from/writing to the row buffer without crossing the channel. Because DRAM has many banks, the internal aggregate bandwidth far exceeds the channel bandwidth. For memory-bound tasks (like state updates), PIM can deliver huge speedups by bypassing the channel bottleneck.

Now you're ready: Pimba offloads state updates and attention to PIM, exploiting this internal bandwidth. The GPU handles other ops (FFN, etc.).
4Why state update is not just another GEMV—and what that means for PIM design▾

Prior LLM-focused PIMs (like HBM-PIM, NeuPIMs) target GEMV (matrix-vector multiplication), which is the building block of attention. They implement GEMV units and call it done. State update, however, involves element-wise multiply, element-wise add, outer product, and dot product. Implementing each as a separate specialized unit would blow up area (the paper shows a "pipelined" design incurs 32.4% area overhead—far above the ~25% practical limit for PIM logic).

Solution: resource sharing. Pimba's SPE (State-update Processing Engine) uses the same multipliers/adders for both state update and attention, plus it shares one SPU across two banks via access interleaving (reading from one bank while writing to the other in a pipelined fashion). This cuts the number of processing units in half while maintaining throughput.

Now you're ready: Pimba's core innovation is maximizing hardware reuse to stay under area constraints while accelerating both state updates and attention.
5Low-precision formats—and why MX8 is Pareto-optimal for PIM▾

Quantizing the state reduces memory bandwidth and compute cost. The paper tests several 8-bit formats: int8 (integer with scaling factor), fp8 variants (e4m3, e5m2), and MX8 (Microsoft's block floating-point: 16 values share an 8-bit exponent, pairs share a 1-bit microexponent). It also tests stochastic rounding (SR), which probabilistically preserves small values.

Key findings: (1) fp8 formats (e4m3, e5m2) catastrophically fail for SU-LLMs due to the "swamping effect"—continuous state updates accumulate rounding errors in low-mantissa formats, causing perplexity to explode (e.g., GLA's perplexity jumps to 8,114 with e4m3). (2) int8 is accurate but area-expensive: element-wise addition requires dequantization (multiply by scaling factor) and re-quantization (normalize), needing extra comparators. (3) MX8 is accurate and area-efficient: shared exponents allow direct addition via simple shifting, no dequantization needed. Stochastic rounding adds minimal area (just an LFSR + adder) and improves accuracy. Figure 6 shows MX8 achieves the best accuracy-area tradeoff.

Now you're ready: Pimba's SPE implements custom MX8 multipliers and adders, enabling efficient state update and attention with negligible accuracy loss.
6Access interleaving—how one SPU serves two banks without throughput loss▾

A naïve per-bank PIM design wastes cycles: during a write operation, the processing unit sits idle because the row buffer can't simultaneously read new data. Pimba solves this with access interleaving. One SPU is shared between two banks (called "upper" and "bottom"). In iteration i, the SPU reads a sub-chunk from the upper bank to start a new computation, while the bottom bank writes the result of a computation started several iterations earlier. In iteration i+1, roles swap: the bottom bank reads, the upper bank writes.

Because reads and writes happen on different banks, there's no structural hazard—the SPU receives input every iteration, maintaining full throughput with half the processing units. The four-stage pipeline (fetch state → decay+outer-product → add → dot-product+writeback) is continuously fed by this interleaving.

Now you're ready: Access interleaving is why Pimba achieves the throughput of a per-bank design (like NeuPIMs) while keeping area overhead at 13.4% instead of 32%+.
7Custom DRAM commands and the GPU+PIM interface▾

Pimba extends standard DRAM with five custom commands: ACT4 (activate four banks in parallel, respecting tFAW timing), REG_WRITE (load operands into SPU registers), COMP (execute state update/attention across all banks), RESULT_READ (retrieve results to GPU), and PRECHARGES (precharge all banks, writing updated state back to DRAM cells). Command scheduling overlaps REG_WRITE during ACT4 idle time and RESULT_READ during PRECHARGES to hide transfer overhead.

The GPU remains in control: it issues PIM commands, waits for deterministic completion, then retrieves results. For operations with dependencies (e.g., attention's "score" phase must finish before "attend"), GPU and PIM execute in blocked fashion. For large models, multiple Pimba devices communicate via NVLink using pipeline/tensor parallelism.

Now you're ready: Pimba is a "drop-in replacement" for existing GPU memory—same interface philosophy as HBM-PIM, extended to support state updates and hybrid models.
The whole story in one place

Pimba in a nutshell

The problem, the measurement-driven solution, and how the pieces fit together.

Transformer LLMs' KV cache grows linearly with sequence length, consuming massive memory and compute for long contexts. Post-transformer models (Mamba-2, RetNet, GLA, HGRN2) replace attention with a constant-size state updated each token. This solves the growth problem but creates a new bottleneck: state update operations are memory-bound (low arithmetic intensity) and lack parameter reuse across requests. At large batch sizes, state updates dominate latency (e.g., 73.8% for RetNet at batch 128)—despite constant compute per token.

Pimba's fix: offload state updates (and attention for hybrid models) to PIM, exploiting DRAM's internal aggregate bandwidth. Two design principles govern the architecture: (1) Maximize hardware resource sharing—instead of per-bank specialized units (32%+ area overhead), Pimba shares one SPU across two banks via access interleaving, sustaining full throughput at 13.4% overhead. (2) Use Pareto-optimal quantization—MX8 (Microsoft's block floating-point) achieves transformer-like accuracy while enabling area-efficient element-wise addition (no dequantization needed), unlike int8 or fp8 formats that either blow up area or destroy accuracy due to swamping.

The resulting SPU (State-update Processing Unit) includes an MX8-based SPE (State-update Processing Engine) with custom multipliers/adders that handle both state update primitives (decay, outer product, addition, dot product) and attention score/attend phases. Custom DRAM commands (ACT4, REG_WRITE, COMP, RESULT_READ, PRECHARGES) orchestrate GPU-PIM collaboration. For models beyond single-GPU capacity, Pimba scales via pipeline/tensor parallelism over NVLink.

Payoff: On 2.7B–7B SU-LLMs (RetNet, GLA, HGRN2, Mamba-2, Zamba2) plus OPT (transformer baseline), Pimba delivers 14.6× and 6.9× lower state update latency vs GPU and GPU+PIM, resulting in up to 4.1× and 2.1× higher throughput, with negligible accuracy loss and <1% area overhead on memory devices.

🍯 Key takeaways

  • Post-transformer LLMs use constant-size state instead of growing KV cache—memory usage and compute are fixed, not linear in sequence length.
  • State update operations are memory-bound (low arithmetic intensity) and dominate latency at large batch sizes (up to 73.8% of time).
  • Pimba offloads state updates (and attention) to PIM, exploiting DRAM's internal aggregate bandwidth to bypass the channel bottleneck.
  • Access interleaving—one SPU serves two banks by reading from one while writing to the other—halves processing units while maintaining throughput.
  • MX8 quantization is Pareto-optimal: 8-bit average per value, area-efficient (shared exponent enables direct addition), and accurate (avoids swamping effect).
  • Custom DRAM commands (ACT4, REG_WRITE, COMP, RESULT_READ, PRECHARGES) orchestrate GPU+PIM execution with overlapped transfers.
  • Result: up to 4.1× throughput improvement over GPU, 2.1× over GPU+PIM, with 13.4% area overhead and negligible accuracy loss.
Seven everyday pictures

How to think about Pimba

Each analogy stands alone. Every card ends with an honest note about where the picture stops being true.

The core idea · Constant vs growing state

Fixed-size toolbox vs expanding file cabinet

Transformers store every past token's key-value pair in a growing KV cache—like a file cabinet that adds a folder for each new document, eventually filling the room. Post-transformers keep a constant-size state matrix—a fixed toolbox that's updated with each new piece of info, compressing history without growth. No matter how many tokens arrive, the toolbox stays the same size. Hence memory and compute are constant, not linear in sequence length.

Where it breaks: the "toolbox" isn't static—it's actively rewritten each token via decay + outer product + addition. That update is memory-bound, hence the bottleneck Pimba targets.
PIM's advantage · Internal bandwidth

Highway vs side streets

DRAM's channel is a single highway—only one bank can use it at a time, creating a bottleneck. Inside the chip, each bank has its own "side street" (row buffer) with far higher aggregate bandwidth across all banks. PIM is like building shops along the side streets: you process data in the neighborhood (inside the bank) instead of driving everything to the highway (GPU). For memory-bound tasks, this bypasses the bottleneck and delivers huge speedups.

Where it misleads: not all operations benefit—compute-bound tasks (e.g., large GEMM with reuse) still prefer the GPU's higher compute density. PIM shines on bandwidth-bound ops like state updates.
Access interleaving · Shared processing unit

Ambidextrous worker serving two stations

Imagine a worker operating two assembly stations. While the left hand reads parts from station A, the right hand assembles and writes to station B—then roles swap. This worker never idles: there's always input from one station. Access interleaving is the same: one SPU alternates between two banks, reading from one while writing to the other. Because read and write happen on different banks, no conflict—full throughput with half the processing units compared to a per-bank design.

Where it breaks: the analogy assumes perfect pipelining. In practice, timing constraints (tCCD_L, etc.) and data dependencies can introduce bubbles, but Pimba's four-stage pipeline minimizes them.
Quantization · MX8 vs alternatives

Measuring cups with different precisions

Imagine three measuring systems: (1) fp8—coarse notches that lose small amounts each pour (swamping: tiny values get rounded away during repeated additions). (2) int8—fine notches but requires converting units every time you mix ingredients (dequantization + re-quantization = area overhead). (3) MX8—shared scale per group of cups, letting you mix directly with simple shifting. For state updates (continuous accumulation), fp8 catastrophically loses info, int8 needs extra logic, and MX8 is both accurate and area-efficient.

Where it breaks: the "shared scale" in MX is actually a shared exponent plus microexponents—more nuanced than a simple unit conversion, but the analogy captures why it enables direct addition.
Why PIM helps · Arithmetic intensity

Low-return errands

State updates are like running to the store for milk (huge trip, tiny purchase)—low arithmetic intensity (FLOPs per byte transferred). You spend most time on the road (memory transfer) rather than shopping (compute). PIM puts the store next door (inside the bank), so the trip is trivial and the actual shopping (compute) becomes the main activity. GPU excels at high-intensity tasks (buying a week's groceries in one trip = GEMM), but state updates need many tiny trips—PIM's domain.

Where it breaks: some ops (like causal conv in Mamba-2) have even lower intensity but are small enough to run on GPU without killing throughput. PIM targets the ops that are both low-intensity and dominate latency.
Resource sharing · SPE versatility

Swiss army knife vs single-purpose tools

State update needs element-wise multiply (decay), outer product, element-wise add (update), and dot product (output). Building a specialized unit for each would be like carrying four tools—heavy. Pimba's SPE is a Swiss army knife: the same MX8 multipliers and adders handle all primitives, plus attention's score/attend phases. By reusing hardware across operations, Pimba stays under the ~25% PIM area budget while delivering full functionality.

Where it breaks: some operations (e.g., outer product) need different dataflows than dot product, requiring mux/demux logic. The "knife" has multiple blades, not one blade doing everything—but it's still one compact tool.
GPU+PIM orchestration · Blocked execution

Relay race with mandatory handoffs

State updates and attention have data dependencies (e.g., attention's "score" must finish before "attend" can use the scores). So GPU and PIM execute in a relay race: GPU issues PIM commands, waits for deterministic completion, retrieves results, then resumes. For hybrid models like Zamba2, the baton passes back and forth (PIM handles state updates in Mamba layers, attention in attention layers, GPU handles FFN/projection). Custom DRAM commands (ACT4, COMP, RESULT_READ, etc.) are the handoff protocol.

Where it breaks: future work (like NeuPIMs' dual row buffers + sub-batch interleaving) could overlap GPU and PIM execution across different sub-batches, eliminating some bubbles. Pimba's blocked model is simpler but leaves room for improvement.
The full explanation

How it actually works

Section by section, with the paper's key mechanisms rebuilt as visuals you can read at a glance.

1 · The shared state update pattern

Mamba-2, RetNet, GLA, HGRN2 all share a common structure during generation. For each head at time step t, they: (1) decay the previous state matrix 𝑆ₜ₋₁ by element-wise multiplication with a vector 𝑑ₜ (broadcast to match state dimensions), (2) compute an outer product of vectors 𝑘ₜ and 𝑣ₜ (capturing new token info), (3) add that outer product to the decayed state to produce 𝑆ₜ, and (4) dot-product the updated state (transposed) with 𝑞ₜ to get output 𝑦ₜ.

This is the state update operation. It's memory-bound (each step needs to read/write the state matrix, with FLOPs proportional to state size → low arithmetic intensity) and lacks parameter reuse across requests (each user's state is independent). Hence it dominates latency at large batch sizes—e.g., 73.8% for RetNet at batch 128 (Figure 3).

🌱
Concretely: for a 2.7B model with 𝑑ᵢₘₕₑₐ𝒹 = 128 and 𝑑ᵢₘₛₜₐₜₑ = 256, each head's state is a 128×256 matrix (32 kB in fp16). With 32 heads and batch 128, that's 128 MB of state read+write per token. At GPU memory bandwidth (~2 TB/s), that's 64 μs just for transfers—plus compute. PIM's internal bandwidth (~10× higher aggregate) cuts transfer time proportionally.

2 · Why naive PIM designs fail—and the two principles

Principle 1: Maximize hardware resource sharing. A per-bank "pipelined" PIM that implements the full state update sequence (decay, outer product, add, dot product) in dedicated hardware achieves 4.3× throughput over GPU—but incurs 32.4% area overhead (Figure 5b), far above the ~25% PIM logic budget. A "time-multiplexed" design with just basic mul/add units has only 17.8% overhead but delivers a modest 2.8× speedup. Neither is Pareto-optimal.

Principle 2: Pareto-optimal quantization. Figure 6 shows the accuracy-area tradeoff for various 8-bit formats. fp8 (e4m3, e5m2) catastrophically fails (perplexity explodes to 6,964 for GLA with e4m3) due to the swamping effect—continuous state updates accumulate rounding errors in low-mantissa formats. int8 is accurate but area-expensive (needs dequantization + re-quantization for addition → extra comparators). MX8 is accurate (perplexity 11.51 vs 11.46 for fp16 Mamba-2) and area-efficient (shared exponent enables direct addition via shifting). With stochastic rounding, MX8 is the Pareto frontier.

3 · Access interleaving—solving the structural hazard

A per-bank pipelined design wastes cycles during writes (row buffer can't read new data while writing old results). Pimba solves this with access interleaving: one SPU serves two banks. In iteration i, the SPU reads a sub-chunk from the "upper" bank to start a new computation, while the "bottom" bank writes the result of a computation started several iterations earlier. In iteration i+1, roles swap—bottom reads, upper writes. Because read and write are on different banks, no conflict. The four-stage pipeline (fetch state → decay+outer-product → add → dot-product+writeback) is continuously fed, maintaining full throughput with half the processing units.

4 · MX8 microarchitecture—element-wise ops without dequantization

MX8 groups 16 values that share an 8-bit exponent, with pairs sharing a 1-bit microexponent. For multiplication: add exponents, add microexponents (if sum ≥ 2, set result microexp = 1 and right-shift mantissas by 1), multiply mantissas as integers. For addition: compare group exponents to get max (result exponent), right-shift the smaller group's mantissas by the exponent difference, further right-shift by microexponents, add mantissas as integers, result microexp = 0. No dequantization → simple shifters + integer ALUs, very area-efficient.

Stochastic rounding adds minimal area (LFSR for random bits + adder to inject into mantissa LSB). Combined, this enables Pimba's SPE to handle state update and attention primitives in MX8 with negligible accuracy loss and low overhead.

5 · Attention support—reusing the same SPE

Attention in hybrid models (e.g., Zamba2) has two phases: (1) score—dot products between query and each key, sent to GPU for softmax; (2) attend—multiply each attention score with its value vector, accumulate. Pimba reuses its multipliers, adders, and dot product unit for both phases. KV cache is laid out similarly to state (sub-chunks in DRAM rows), and the same access interleaving applies. No dedicated attention hardware needed—SPE is versatile.

6 · Custom DRAM commands and GPU orchestration

Pimba extends DRAM with five commands: ACT4 (activate four banks in parallel, respecting tFAW timing), REG_WRITE (load operands 𝑑ₜ, 𝑞ₜ, 𝑘ₜ, 𝑣ₜ into SPU registers), COMP (execute state update/attention across all banks, observing tCCD_L between consecutive column reads), RESULT_READ (retrieve partial sums to GPU), PRECHARGES (precharge all banks, writing updated state back to DRAM cells). Command scheduling overlaps REG_WRITE during ACT4 idle and RESULT_READ during PRECHARGES to hide transfer overhead (Figure 11).

GPU issues PIM commands, waits for deterministic completion (no complex sync logic—all commands have fixed timing), retrieves results. For operations with dependencies (e.g., attention score → softmax → attend), GPU and PIM alternate in blocked fashion.

Follow the numbers, step by step

Worked examples

Four traces using the paper's real values—computing state update latency, quantization impact, area overhead, and throughput gains.

① Computing state update latency on GPU vs Pimba

RetNet 2.7B, batch 128, 𝑑ᵢₘₕₑₐ𝒹 = 128, 𝑑ᵢₘₛₜₐₜₑ = 256, 32 heads. How much data movement, and how does Pimba cut it?

  1. State size per head: 128 × 256 = 32,768 elements. In fp16, 32,768 × 2 bytes = 64 kB/head.
  2. Total state for batch 128: 128 users × 32 heads × 64 kB = 256 MB (read) + 256 MB (write) = 512 MB per token.
  3. GPU memory bandwidth: A100 has ~2 TB/s. Transfer time = 512 MB ÷ 2000 GB/s ≈ 0.256 ms. Plus compute (decay, outer product, add, dot product).
  4. Pimba with MX8: state is 8-bit avg → 256 MB total (half of fp16). Internal bandwidth ~10× channel → effective BW ~20 TB/s. Transfer ≈ 256 MB ÷ 20,000 GB/s ≈ 0.0128 ms. Plus MX8 compute (area-efficient, same throughput).
  5. Speedup: 0.256 ms ÷ 0.0128 ms = 20× on transfer alone. Actual measured latency reduction is 14.6× (Figure 13) due to compute + overhead. Still massive.
🍯
Insight: State updates are bandwidth-bound. PIM's internal bandwidth + MX8 quantization both slash transfer time, yielding order-of-magnitude latency cuts.

② Quantization accuracy impact—why fp8 fails and MX8 succeeds

Mamba-2 2.7B on WikiText-2. Why does e4m3 (fp8) get perplexity 8,114 while MX8 gets 11.51 (vs 11.46 for fp16)?

  1. Swamping effect: State update continuously adds outer products to the state. With e4m3 (3-bit mantissa), small values (e.g., 0.001 in one element, 100 in another) align their exponents for addition. The small value's mantissa is right-shifted so far it vanishes (rounds to zero).
  2. Accumulation over time: Each token's update compounds the error. After thousands of tokens, the state has lost most small-magnitude information—perplexity explodes.
  3. int8 avoids swamping: 7-bit mantissa (after sign) is enough to preserve small values during shifts. But adding requires dequantization (multiply by scaling factor), re-quantization (normalize by new max) → extra comparators for finding max, extra multipliers → area-heavy.
  4. MX8's solution: 6-bit mantissa + shared exponent per group. Addition aligns exponents via shifting (no dequantization), adds integer mantissas, result has microexp = 0. Enough mantissa bits to avoid swamping, no area-heavy dequant logic.
  5. Stochastic rounding: Adds random noise to LSB, probabilistically preserving small values that would otherwise round down. Minimal area (LFSR + adder), significant accuracy gain (e.g., e5m2 perplexity drops from 62 to 11.9 with SR, Figure 4).
💡
MX8 with stochastic rounding achieves perplexity 11.51 vs fp16's 11.46—negligible degradation—while cutting area by >50% vs int8 (Figure 6). That's the Pareto frontier.

③ Area overhead calculation—why Pimba stays under the PIM budget

One SPU shared between two banks. FreePDK 45nm synthesis, scaled to 10nm. What's the overhead per bank?

  1. Compute logic (SPE): 0.053 mm² per SPU. Includes MX8 multipliers, adders, dot product unit, control.
  2. Buffers (registers): 0.039 mm² per SPU (CACTI7, 22nm → 10nm scaling).
  3. Total per SPU: 0.053 + 0.039 = 0.092 mm².
  4. Per bank: One SPU serves two banks → 0.092 mm² ÷ 2 = 0.046 mm² per bank. Bank area ≈ 0.343 mm² (Table 3 implies 13.4% overhead → 0.046 mm² ÷ 0.343 mm² ≈ 0.134). Overhead = 13.4%.
  5. Comparison: Per-bank pipelined design (no interleaving) would be 0.092 mm² per bank → 26.8% overhead (2× Pimba's). HBM-PIM (time-multiplexed, fp16) is 0.081 mm² per SPU shared between two banks → 11.8% overhead, but delivers only 1.4× throughput vs Pimba's 2.1×.
🍯
Insight: Access interleaving halves the number of SPUs, cutting area from 26.8% to 13.4%—comfortably under the ~25% PIM logic budget, while MX8 keeps throughput high.

④ End-to-end throughput gain—RetNet 70B at batch 128

Large scale model, tensor parallelism across 8 GPUs. Prefill (2,048 tokens), generation (2,048 tokens). What's the speedup?

  1. Baseline (GPU): State updates at batch 128 spend 74% of latency (Figure 13, RetNet large scale). Total latency = 100% (normalized).
  2. GPU+PIM (HBM-PIM): State update latency cuts by ~4.7× (Figure 13 implies Pimba is 14.6× vs GPU, HBM-PIM is ~3.1× vs GPU, so ratio ≈ 4.7). New state update latency = 74% ÷ 4.7 ≈ 15.7%. Other ops (26%) unchanged. Total = 15.7% + 26% = 41.7%. Speedup = 100 ÷ 41.7 ≈ 2.4×.
  3. Pimba: State update latency cuts by 14.6×. New state update = 74% ÷ 14.6 ≈ 5.1%. Total = 5.1% + 26% = 31.1%. Speedup = 100 ÷ 31.1 ≈ 3.2×.
  4. Check against Figure 12: Paper reports up to 4.1× for RetNet (small scale, batch 128). Large scale has more communication overhead (NVLink), so 3.2× is plausible. Actual measured is ~3.2× (Figure 12, RetNet 70B, batch 128).
  5. Attention-dominated hybrid (Zamba2): Attention also accelerates by ~6.3× vs GPU, 2.1× vs GPU+PIM (Figure 13). Combined state update + attention acceleration yields similar 2–3× gains.
💡
The throughput gain tracks the fraction of time spent on state updates/attention. For SU-LLMs at large batch, that's 70%+, so even 10–15× latency cuts on those ops translate to 3–4× end-to-end speedup.

Problem 1 · easy
Why can't you just use the same per-bank PIM design as HBM-PIM (focused on GEMV for transformers) for state updates in Mamba-2?
Show answer
HBM-PIM implements fp16 multipliers/adders optimized for GEMV (matrix-vector multiply), which is the building block of attention. State updates, however, need element-wise multiply (decay), outer product, element-wise add (update), and dot product. Implementing each as a dedicated unit would blow up area (32.4% overhead for a pipelined design, Figure 5b). Moreover, state updates incur both read and write, creating structural hazards if you have one processing unit per bank (idle during writes). Pimba solves both via (1) resource sharing—same MX8 mul/add handles all primitives, and (2) access interleaving—one SPU serves two banks, overlapping read/write to sustain throughput. Result: 13.4% area overhead, 4.3× throughput.Misconception: that GEMV acceleration is sufficient. State updates have fundamentally different primitives and access patterns, requiring distinct design choices.
Problem 2 · medium
A colleague suggests using e5m2 (fp8 with 5-bit exponent, 2-bit mantissa) for state quantization because it has a wider dynamic range than int8. Why does this fail catastrophically for SU-LLMs despite working fine for transformer KV cache quantization?
Show answer
The 2-bit mantissa in e5m2 causes swamping: when adding two numbers with different magnitudes (e.g., 0.001 and 100), you align exponents by right-shifting the smaller number's mantissa. With only 2 mantissa bits, the smaller value often shifts entirely off the register (rounds to zero). Transformer KV cache is mostly read (attention score/attend), so swamping doesn't accumulate. SU-LLMs update the state every token—hundreds of additions compound the error, and small-magnitude info is lost. Figure 4 shows GLA's perplexity explodes to 6,964 with e4m3, 62 with e5m2. Stochastic rounding helps (drops e5m2 to 11.9) but int8/MX8 with 6–7 mantissa bits avoid swamping entirely.Misconception: that dynamic range (exponent bits) is more important than precision (mantissa bits) for accumulation. Continuous updates need high mantissa precision to preserve small values.
Problem 3 · hard
Access interleaving lets one SPU serve two banks by reading from one while writing to the other. Why doesn't this create a data dependency hazard where the write from iteration i hasn't finished before iteration i+1 tries to read from the same bank?
Show answer
The write at iteration i+2 (to the bottom bank) is writing the result of a computation that started at iteration i−2 (due to the four-stage pipeline: fetch → decay+outer-product → add → dot-product+writeback). Meanwhile, the read at iteration i+2 (from the upper bank) is fetching a new sub-chunk for a new computation. The two operations touch different rows of their respective banks (state is partitioned into chunks across banks, and chunks are processed sequentially). Hence no data dependency—the write is finalizing an old computation on one bank while the read is starting a new computation on the other bank. The pipeline depth (4 stages) ensures enough separation that by the time you circle back to the same bank, the prior write is complete.Misconception: that read/write to the same bank in adjacent iterations creates a hazard. The key is that they're on different banks and operate on different chunks, with pipeline latency providing separation.
Problem 4 · medium
Why does Pimba still offload attention to PIM in hybrid models like Zamba2, even though attention has higher arithmetic intensity than state updates (Figure 1b shows attention is 4× less bandwidth-bound)?
Show answer
Even though attention has 4× higher arithmetic intensity than state updates (meaning it's less bandwidth-bound), it's still significantly bandwidth-bound compared to compute-heavy ops like FFN. The roofline analysis (Figure 1b) shows attention is well below the "compute bound" region. Moreover, in hybrid models like Zamba2, attention layers handle long sequences (e.g., 2,048 tokens), so the KV cache is large and the score/attend phases involve substantial data movement. Pimba's PIM acceleration cuts attention latency by 6.3× vs GPU, 2.1× vs GPU+PIM (Figure 13)—not as dramatic as state updates (14.6× / 6.9×) but still significant. Since Zamba2 spends 65.5% of time on attention at batch 128 (despite having 6× more Mamba-2 layers), offloading it to PIM is critical for overall speedup.Misconception: that only the most bandwidth-bound operation benefits from PIM. Attention is bandwidth-bound enough that PIM helps, and it dominates latency in hybrid models due to sequence-length scaling.
Problem 5 · easy
A system architect proposes using Pimba's SPU design for accelerating large GEMM operations (e.g., FFN layers) by partitioning the matrix across banks. Why is this inefficient?
Show answer
Large GEMMs have high arithmetic intensity (many FLOPs per byte of data due to parameter reuse across batch/output dimensions). They're compute-bound, not memory-bound, so the bottleneck is GPU FLOPS, not DRAM bandwidth. Offloading to PIM would (1) underutilize the GPU's massive FLOP capacity, (2) incur transfer overhead (moving partial results between PIM and GPU for accumulation), and (3) not leverage PIM's main advantage (internal bandwidth) since the op doesn't need it. Pimba targets memory-bound ops (state updates, attention) where bandwidth is the bottleneck. FFN stays on GPU where it belongs.Misconception: that PIM is universally better. PIM shines on bandwidth-bound ops; compute-bound ops should stay on the GPU's high-FLOP cores.
Problem 6 · hard
Why does MX8 addition not require dequantization (unlike int8), and why does the result always have microexponent = 0?
Show answer
In int8, each element has its own scaling factor (or a group shares one). To add two elements with different scaling factors, you must dequantize (multiply by scale → fp16), add in fp16, then re-quantize (find new max, compute new scale, divide all elements). This needs comparators to find max and multipliers for scaling—area-heavy. In MX8, a group of 16 values shares an 8-bit exponent, and pairs share a 1-bit microexponent. For addition: (1) compare group exponents, take max as result exponent; (2) right-shift the smaller group's mantissas by the exponent difference; (3) further right-shift each mantissa by its microexponent (0 or 1); (4) add the aligned integer mantissas. No multiply, no dequantization—just shifts + integer add. The result has microexp = 0 because after shifting by microexps, the sum is already in "standard" form (no additional scaling beyond the group exponent).Misconception: that all quantization formats need dequantization for addition. Block floating-point (MX) enables direct addition via exponent alignment, which is why it's area-efficient.
Problem 7 · medium
Pimba achieves up to 4.1× throughput improvement over GPU baseline. Why isn't the speedup closer to 14.6× (the latency reduction for state updates)?
Show answer
The 14.6× is the speedup on state update operations alone. End-to-end throughput depends on the fraction of time spent on state updates. For RetNet at batch 128, state updates are 73.8% of latency (Figure 3). So even if you cut state update latency to zero, max speedup = 1 / (1 − 0.738) ≈ 3.8× (Amdahl's law). In practice, state updates don't drop to zero—they drop by 14.6×, so new fraction = 73.8% ÷ 14.6 ≈ 5%. Other ops (GEMM, discretization, etc.) still take 26%. Total latency = 5% + 26% = 31%, speedup = 100 / 31 ≈ 3.2×. Figure 12 shows RetNet achieves ~3.2× at large scale (70B, batch 128). At small scale (2.7B), less communication overhead → closer to 4.1×. For models with less state update dominance (e.g., Zamba2 has attention too), speedup is lower (~2×). The 14.6× is the potential; Amdahl limits the realized speedup based on workload composition.Misconception: that accelerating one operation by 𝑁× gives 𝑁× end-to-end speedup. Amdahl's law says speedup is capped by the fraction of time spent on the accelerated op.
Problem 8 · easy
Why does Pimba use stochastic rounding instead of deterministic round-to-nearest for MX8 quantization?
Show answer
Stochastic rounding probabilistically rounds up or down based on the distance from representable values (e.g., if a value is 0.7 of the way to the next representable point, it rounds up with 70% probability). This preserves small values on average during repeated accumulation—critical for state updates where you add hundreds of outer products to the state. Deterministic round-to-nearest would always round small values down, causing them to vanish (swamping). Stochastic rounding adds random noise to the LSB, so small values survive some fraction of the time. Figure 4 shows stochastic rounding cuts Mamba-2's perplexity from 62 to 11.9 for e5m2 format. The area cost is minimal (LFSR for random bits + adder to inject into mantissa LSB).Misconception: that deterministic rounding is sufficient. For continuous accumulation (state updates), stochastic rounding significantly improves accuracy at negligible area cost.
Quick answers

Frequently asked questions

Twelve questions newcomers ask most. Tap any to expand.

Q1What exactly is a "state update" and why is it common across post-transformers?▾
State update is the pattern shared by Mamba-2, RetNet, GLA, HGRN2: (1) decay the previous state matrix by element-wise multiplying with a vector, (2) compute an outer product of new token vectors, (3) add that outer product to the decayed state, (4) dot-product the updated state with a query vector to produce output. Despite algorithmic differences (SSM vs linear attention vs RNN), they all follow this structure during generation. It's the "common denominator" Pimba targets for acceleration.
Q2Why is state update memory-bound if the state is constant-size (unlike KV cache)?▾
Constant-size means the state doesn't grow with sequence length—it's fixed (e.g., 128×256 per head). But each token's update still requires reading the entire state, performing decay/outer-product/add/dot-product (FLOPs proportional to state size), then writing it back. With low arithmetic intensity (few FLOPs per byte), memory bandwidth becomes the bottleneck. Moreover, each user's state is independent (no parameter reuse across batch), so latency scales linearly with batch size—hence state updates dominate at large batch (73.8% for RetNet at batch 128, Figure 3).
Q3How does PIM achieve higher bandwidth than the GPU's memory interface?▾
DRAM has many banks (e.g., 64 banks across 8 channels in an A100's HBM). The channel bandwidth is ~2 TB/s total, but the internal aggregate bandwidth (sum of all banks' row buffer throughput) is ~10× higher. PIM places compute logic inside each bank, so operations read/write directly from/to the row buffer without crossing the channel. For bandwidth-bound ops, this 10× internal bandwidth multiplier is the source of PIM's speedup. The channel is only used for initial operand transfer (REG_WRITE) and final result retrieval (RESULT_READ), both of which Pimba overlaps with other commands to hide overhead.
Q4Why can't you just use int8 quantization like transformers do for KV cache?▾
int8 works for KV cache because attention mostly reads it (score/attend phases). State updates accumulate—hundreds of additions of outer products to the state. int8's scaling-factor approach requires dequantization (multiply by scale → fp16) before addition, then re-quantization (find new max, compute new scale) after—needing extra comparators and multipliers, hence area-heavy (Figure 6 shows int8 has high area overhead). MX8 avoids this by using shared exponents, enabling direct addition via simple shifting. The paper shows int8 is accurate but impractical for PIM's area constraints.
Q5What is the "swamping effect" and why does it only hurt SU-LLMs?▾
Swamping occurs when adding two floating-point numbers with vastly different magnitudes: you align exponents by right-shifting the smaller number's mantissa, and if the shift is large enough, the mantissa bits fall off the register (round to zero). With fp8 formats (e4m3, e5m2) having only 2–3 mantissa bits, even modest magnitude differences cause loss. Transformer KV cache is mostly read, so swamping doesn't compound. SU-LLMs update the state every token—hundreds of additions where small values repeatedly get swamped and lost, causing perplexity to explode (e.g., 8,114 for GLA with e4m3, Figure 4). MX8's 6-bit mantissa + stochastic rounding preserve small values.
Q6How does access interleaving avoid the structural hazard during writes?▾
A per-bank design wastes cycles: during a write (precharge + writeback to DRAM cells), the row buffer can't read new data, so the processing unit idles. Access interleaving assigns one SPU to two banks (upper and bottom). In iteration i, SPU reads from upper to start a new computation, while bottom writes the result of a computation started several iterations earlier (due to pipeline latency). In iteration i+1, roles swap—bottom reads, upper writes. Because read and write are on different banks, no conflict. The four-stage pipeline ensures continuous input to the SPU every iteration, maintaining full throughput with half the processing units.
Q7Why doesn't Pimba suffer from the same utilization issues as prior PIMs (like HBM-PIM)?▾
HBM-PIM uses a time-multiplexed design: basic fp16 mul/add units execute state update operations sequentially (multiply for decay, add for update, etc.), underutilizing internal bandwidth because each step is a separate command. Pimba's SPU is pipelined—decay, outer product, add, and dot product happen in parallel across pipeline stages, and access interleaving keeps the pipeline fed every iteration. This sustains throughput close to the internal bandwidth limit. Figure 5 shows GPU+PIM (HBM-PIM) achieves only 1.4× vs GPU, while Pimba achieves 2.1× vs GPU+PIM (i.e., 2.9× vs GPU) due to better utilization.
Q8Does Pimba work for models beyond the four SU-LLMs in the paper?▾
Yes—any post-transformer that uses the state update pattern (decay, outer product, add, dot product) can leverage Pimba. The paper evaluates RetNet, GLA, HGRN2, Mamba-2 (all 2.7B) plus Zamba2 (7B hybrid). Newer models like Samba, Nemotron-H, or future SSMs/linear-attention variants would also benefit, as long as their core operation matches the structure. Pimba's SPE is general-purpose (MX8 mul/add handles any element-wise op), so it's not tied to a specific algorithm—just the shared computational pattern.
Q9How does Pimba handle hybrid models like Zamba2 with both Mamba-2 and attention layers?▾
Pimba's SPU accelerates both state updates (for Mamba-2 layers) and attention (for attention layers) using the same MX8 multipliers/adders. Attention has two phases: (1) score—dot products between query and keys (uses SPU's dot product unit), sent to GPU for softmax; (2) attend—multiply attention scores with values, accumulate (uses SPU's mul/add). KV cache is laid out similarly to state (sub-chunks in DRAM rows). The same access interleaving and custom DRAM commands apply. Figure 13 shows Zamba2 spends 65.5% of time on attention at batch 128 (despite 6× more Mamba-2 layers), so accelerating both is critical.
Q10What are the custom DRAM commands and why are they needed?▾
Pimba extends DRAM with five commands: ACT4 (activate four banks in parallel, respecting tFAW timing), REG_WRITE (load operands into SPU registers—overlapped during ACT4 idle time), COMP (execute state update/attention across all banks, observing tCCD_L between consecutive column reads), RESULT_READ (retrieve partial sums to GPU—overlapped during PRECHARGES), PRECHARGES (precharge all banks, writing updated state back to DRAM cells). These commands orchestrate GPU-PIM collaboration, hiding transfer overhead via overlapping and maintaining deterministic timing so the GPU doesn't need complex synchronization logic.
Q11Why is the area overhead 13.4% instead of the 11.8% for HBM-PIM?▾
Pimba's SPE includes more logic than HBM-PIM's basic fp16 mul/add: MX8 multipliers/adders (with exponent alignment, microexponent handling, stochastic rounding), plus a dot product unit for the final step. Despite this, the 1.6% additional overhead is justified by 2.1× throughput improvement over HBM-PIM (which itself only achieves 1.4× vs GPU). The key is access interleaving—halving the number of SPUs keeps total area under the ~25% PIM logic budget. Table 3 shows Pimba's 13.4% is still comfortably below the practical limit.
Q12What are the limitations—when does Pimba not help?▾
Pimba doesn't help for: (1) Small batch sizes—state updates are fast enough on GPU that PIM overhead isn't justified. (2) Compute-bound ops (e.g., FFN layers)—these should stay on GPU where they leverage high FLOP capacity. (3) Models with very small state (e.g., if 𝑑ᵢₘₛₜₐₜₑ is tiny, state updates aren't the bottleneck). (4) Fast-phasing workloads (like some in Figure 12, e.g., twolf-vpr in UCP parlance)—if useful/useless behavior flips faster than the 5M-cycle re-decision interval, Pimba's global quota lags, and GPU's per-access adaptivity wins. The paper is honest about this: Pimba targets batch-heavy, state-update-dominated scenarios, not all LLM serving regimes.
