Teaching page tables to learn where translations actually live

Your PhD is in OS-managed memory tiering for CXL. The radix page table wastes up to five sequential memory accesses per translation, poisoning your careful placement. This is the plain-English story of LVM, a learned index that predicts PTE locations and walks in one access.

Quick glossary

LVM – Learned Virtual Memory (this paper)
GPT – Gapped Page Table (LVM's leaf tables)
LWC – LVM Walk Cache (caches learned models)
ECPT – Elastic Cuckoo Page Tables (hashed baseline)
PWC – Page Walk Cache (radix tree's cache)
MPKI – Misses Per Kilo Instructions
1Radix page tables — and why five accesses▾

A five-level radix tree (x86-64 LA57) has a 57-bit virtual address: 45 VPN bits split into five 9-bit chunks, plus a 12-bit offset. Each chunk indexes a 512-entry table. To translate, the MMU chases pointers: fetch L1 entry → get L2 base → fetch L2 entry → … → fetch PTE at L5. In the worst case, that's five sequential memory accesses. Page walk caches (PWCs) cache upper levels, but coverage shrinks fast: L4 covers only 2 MB per entry. For memory-intensive workloads (graph analytics, KV stores), MPKI remains high even with large PWCs.

Now you're ready: LVM replaces the five-level pointer chase with a learned function that outputs the PTE's physical address in one lookup.
2Hashed page tables — and why parallel still isn't one▾

Hashed page tables (e.g., ECPT) hash the VPN into a table slot. Collisions force a "𝑑-way cuckoo" design: probe three candidate locations in parallel. On a hit, latency is low—but you fetch three cache lines and discard two, so memory traffic spikes 3×. ECPT also requires separate tables per page size (4KB, 2MB), adding auxiliary lookups. For your tiering work, that extra traffic congests CXL links and pollutes the cache.

Now you're ready: LVM learns the VPN distribution so collisions are rare (0.6%), and fetches only one cache line 99.4% of the time.
3Learned indexes — predicting position from key distribution▾

In databases, a learned index replaces a hash function with a machine learning model that learns the cumulative distribution function (CDF) of keys. Given a key x, the model outputs an index y (position in an array). If keys are regular (e.g., sequential IDs), a single linear model y = ax + b can predict perfectly. Hierarchies of models handle less-regular distributions. The payoff: fewer collisions, faster lookups, but only if keys are somewhat regular.

Now you're ready: LVM applies this idea to VPNs. The paper first validates that virtual address spaces are regular (78–100% sequential gaps), then builds a hierarchy of cheap linear models.
4Why virtual address spaces are regular▾

The paper studies graph analytics, bioinformatics, KV stores (memcached), databases (MongoDB), web servers (hhvm, Finagle), and four Meta production workloads. It measures gap coverage = the fraction of VPNs that are sequential (VPNnext − VPNcurrent = 1). Result: 78–100% of gaps = 1. Why? Userspace allocators (jemalloc, tcmalloc) pack allocations to minimize holes. Large objects (arrays) must be contiguous. The OS coalesces mappings to reduce page table size. Unlike physical memory, virtual address spaces don't fragment—they're per-process and applications expand them contiguously.

Now you're ready: this regularity is the foundation. LVM's linear models exploit it to achieve near-zero collisions.
5Physical memory fragmentation — and LVM's adaptive leaf tables▾

Your CXL tiering work knows this: Meta's datacenters show that contiguous 2 MB+ regions are scarce (Figure 3 in the paper). Existing learned indexes assume a large contiguous table for keys; that's impractical. LVM solves this by allocating a separate gapped page table (GPT) per leaf node. Each GPT can be as small as 4 KB (or 64 KB on ARM). The OS queries the buddy allocator for available contiguity and adjusts the number of leaf nodes accordingly. So LVM adapts to fragmentation levels—even at FMFI = 0.9, it maintains performance.

Now you're ready: LVM's per-leaf GPTs are tiny and scattered, so they coexist with your DRAM/NVM tiering without requiring huge slabs of contiguous DRAM.
6Multiple page sizes — encoding via slope▾

A 4 KB page at VPN 142 occupies one slot. A 2 MB page at VPN 1024 covers VPNs [1024, 1536). LVM trains on the first 4 KB sub-page's VPN (1024). During lookup, the hardware derives the 4 KB VPN from the VA, looks it up, and the linear model rounds down to the 2 MB PTE. The PTE itself uses two bits to encode size (4 KB / 2 MB / 1 GB). Crucially, different page sizes appear as lines with different slopes in the CDF: 4 KB = steep (many PTEs per VPN range), 2 MB = shallow (fewer PTEs). So one learned index handles all page sizes in current PTE encoding. The paper envisions support for arbitrary page sizes as future work.

Now you're ready: this contrasts with ECPT's separate tables per page size. LVM supports existing page sizes without separate structures.
7Fixed-point arithmetic — no floating point in the page walker▾

Database learned indexes use float operations. The Linux kernel forbids FP. LVM hardware would waste area and power on FP units. So LVM quantizes: each value is 44 bits integer + 20 bits fractional, stored in 8 bytes. Each model (slope a, intercept b) = 16 bytes. The page walker has one adder and one multiplier. The entire LWC is 16 entries × 16 bytes = 256 bytes (plus tags). This is 3× smaller in bytes than radix PWCs and uses simple fixed-point ops that fit in 2 cycles at 2 GHz.

Now you're ready: LVM's models are tiny and cheap to compute, so they cache well and don't bloat the MMU pipeline.
The whole story in one place

LVM in a nutshell

The problem, the learned index solution, how it trains and walks, and the end-to-end payoff.

Radix page tables incur up to five sequential memory accesses per translation—a killer for memory-intensive workloads. Hashed page tables (ECPT) reduce latency via parallel lookups but send 2–3× more traffic to memory and require separate tables per page size. Neither is acceptable for CXL tiering, where every extra memory reference adds RTT and congests the fabric.

LVM replaces the hash function with a learned index: a hierarchy of lightweight linear models (y = ax + b) that predict the physical address of the PTE from the VPN. Each model describes a contiguous chunk of the virtual address space; internal nodes zoom in, leaf nodes output PAPTE. Because virtual address spaces are 78–100% sequential, a few linear functions capture the distribution with near-zero collisions (0.6% on average). LVM stores PTEs in gapped page tables (GPTs)—small arrays with empty slots for insertions—allocated at different physical addresses, so no large contiguity is required.

A cost model balances index depth, width, and collision rate to keep the index tiny (<200 bytes steady-state, peak 570 bytes) and highly cacheable. The OS trains the index on page fault, using linear regression over (VPN, position) pairs, scales by a gap factor (1.3), and enforces an error bound (Cerr = 3) so mispredictions require at most 3 extra accesses. Insertions are handled by rescaling (for out-of-bounds near the edge) or local retraining (for within-bounds), avoiding full rebuilds except when necessary. Retraining occurs at most 3 times in the OS prototype workloads.

The hardware MMU adds an LVM Walk Cache (LWC)—a 16-entry fully associative cache of 16-byte model nodes—and a page walker with one adder and one multiplier. On a TLB miss, the walker traverses the learned index (up to 3 internal nodes + 1 leaf), fetches the PTE in one memory access 99.4% of the time (based on PTE-location collision rate), and validates the VPN tag. Context switches are handled via ASID tags in the LWC (no flush).

The payoff: LVM reduces page walk cycles by 39% for 4KB pages and 29% for THP on average; page walk traffic is reduced by 43% for 4KB pages and 34% for THP, with THP cycle reduction reaching 44%. The system achieves a 2–27% speedup (average 7% for THP, 14% for 4KB) over radix, outperforming ECPT by 3–5%. It's within 1% of an ideal single-access page table. The LWC is 3× smaller in bytes than radix PWCs (1.5× in area). Hardware area: 0.00364 mm² (1.5× reduction), power: 0.588 mW leakage (1.9× reduction). The learned index itself averages 162 bytes for THP, 112 bytes for 4KB—scalable to terabyte footprints without growing, unlike radix PWCs.

🍯 Key takeaways

  • LVM replaces radix's 5-level pointer chase with a learned function: VPN → PAPTE in one access (99.4% PTE-location collision rate).
  • Virtual address spaces are 78–100% sequential (gap = 1), so linear models achieve 0.2–0.6% collision rates vs. 19–22% for hash tables.
  • Gapped page tables (GPTs) are small (KB-scale), non-contiguous arrays with empty slots for insertions. The cost model keeps the learned index <200 bytes.
  • Insertions use rescaling (for out-of-bounds near edge) or local retraining (within-bounds), avoiding full rebuilds in typical use.
  • The LWC caches 16 models in 256 bytes; the page walker uses one adder + one multiplier. 3× smaller in bytes, 1.5× smaller in area than radix PWCs, 1.9× lower power.
  • Multiple page sizes (4KB/2MB/1GB) are encoded via slope variations in one index using current PTE encoding. No separate structures, unlike ECPT.
  • End-to-end: 39% MMU overhead reduction for 4KB, 29% for THP; 43% less page walk traffic for 4KB, 34% for THP. 2–27% speedup. Within 1% of ideal.
Six everyday pictures

How to think about LVM

Each picture stands alone. Every card ends with an honest note about where the picture breaks.

The core idea · Learning the distribution

Fitting a ruler to a staircase

You have 1000 books sorted by title on a shelf. Instead of a catalog (hash table), you notice: "A" starts at position 20, "Z" ends at 980. Fit a line: position ≈ 960×(letter/26) + 20. Given a title starting with "M", predict position ≈ 480. The pages are your VPNs; the positions are PTE indices. Because virtual address spaces are mostly sequential, LVM's linear models fit tightly with few outliers.

Where it breaks: if the shelf has a huge gap (e.g., no books for "Q–S"), one line mispredicts. LVM handles this with a hierarchy: internal nodes subdivide the address space, so each leaf model describes a smaller, more regular chunk.
Radix vs. learned · Pointer chase vs. arithmetic

A phone tree vs. a formula

Radix: "Call extension 1 → they say call 3 → they say call 7 → …" (5 calls). LVM: "Extension = 10×floor + 7×room. Done." The radix tree serializes lookups; LVM computes the address in one shot. The cost: you must learn the formula from the building layout (the virtual address space). Once learned, it's fast and cheap.

Where it breaks: if the building is rearranged (insertions far from existing ranges), the formula may need retraining. LVM handles common expansions (contiguous allocations near the edge) via rescaling, but scattered random inserts may force a rebuild. In the OS prototype workloads, retraining occurs at most 3 times per application.
Hashed vs. learned · Collisions vs. prediction

Guessing three mailboxes vs. knowing one

ECPT hashes your VPN and checks three mailboxes (3-way cuckoo). One has the letter, two are empty—but you opened all three (3× memory traffic). LVM's learned model predicts the one mailbox with 99.4% accuracy. On a rare misprediction, it checks up to 3 nearby (bounded by Cerr), but that's 0.6% of the time vs. ECPT's 100%.

Where it breaks: if the address space were completely random (no regularity), LVM's models would fail and collisions would spike. The paper validates that real workloads have 78–100% sequential gaps, so this doesn't happen in practice.
Gapped page tables · Insertions without rebuild

A row of seats with empty chairs

You fill a lecture hall leaving every third seat empty. Latecomers can sit in the gaps without rearranging. LVM's gapped page tables (GPTs) pre-allocate slots (gap factor 1.3 = 30% empty). When a new VPN arrives in-bounds, it inserts into the predicted gap. For out-of-bounds expansions near the edge (common), LVM rescales the GPT (expand the array, keep the model unchanged). Scattered random inserts may force a local retrain.

Where it breaks: if every gap fills and the insertion is random (not near the edge), LVM must retrain the leaf model or rebuild. The paper shows this occurs at most 3 times in the prototype workloads because applications expand their address space contiguously.
Physical contiguity · Per-leaf tables

A jigsaw puzzle scattered across shelves

You store puzzle pieces in many small boxes (KB-scale), each box on a different shelf. You don't need one giant box (MB-scale contiguous slab). LVM allocates a separate GPT per leaf node, querying the buddy allocator for available contiguity. Even in highly fragmented servers (FMFI = 0.9), hundreds of KB are abundant. So LVM adapts: more leaf nodes if contiguity is scarce, fewer if it's plentiful.

Where it breaks: if the OS cannot allocate even 4 KB (base page size), LVM fails. But that's a system-wide OOM condition, not an LVM-specific issue. The paper shows LVM works even when 2 MB+ contiguity is unavailable, unlike prior schemes (FPT).
Multiple page sizes · Slope encoding

Two staircases with different step heights

A 4 KB page: 1 VPN → 1 PTE (steep staircase, many steps). A 2 MB page: 512 VPNs → 1 PTE (shallow staircase, few steps). LVM trains on the first 4 KB sub-page's VPN for huge pages. During lookup, the model rounds down to the 2 MB PTE. Different page sizes appear as lines with different slopes in the CDF, all in one learned index using current PTE encoding.

Where it breaks: if a huge page is partially unmapped (sub-pages individually freed), LVM's model may misdirect. The paper assumes standard THP behavior (all-or-nothing), which holds in Linux.
The full explanation

How it actually works

Section by section: training, walking, updates, cost model, hardware.

1 · Training the learned index

The OS trains LVM on the first page fault. Starting at the root, the cost model (Eq. 1) greedily picks the optimal number of child nodes by computing the number of spline points (segments where a linear model fits), then evaluating C(n) = x₁·d + x₂·s + x₃·cr·ma for n ± 2 candidates. Here d = depth, s = size in bytes, cr = collision rate, ma = average extra accesses per collision. Weights: x₁ = 10, x₂ = 5, x₃ = 200. The cost model enforces d_limit = 3 (max depth) and requires that each node's coverage per byte matches or exceeds radix's upper levels.

For internal nodes, LVM computes a linear function y = ax + b that maps VPNs to child indices (evenly dividing the key space). For leaf nodes, LVM queries the buddy allocator for available contiguity, adapts the number of sibling leaves accordingly, then learns y = ax + b via linear regression over (VPN, position) pairs. It scales by ga_scale = 1.3 to create gaps, allocates a GPT of size num_VPNs × 1.3, and inserts PTEs at predicted positions. If a slot is occupied, exponential search finds the next gap. The model enforces C_err = 3: if the error bound isn't met, the cost model boosts x₃ (collision penalty) and subdivides further.

💡
Concretely: for memcached (124 GB footprint), the steady-state index is 112 bytes (7 models × 16 bytes). Peak during init: 570 bytes. Training time: <1.9 ms per rebuild. Retraining occurs at most 3 times in the OS prototype workloads. 99.4% of the time, the model predicts the exact PTE location.

2 · Page walk in hardware

On an L2 TLB miss, the LVM page walker starts at the root node (base PA stored in a control register, like x86's CR3). It fetches the root model from the LWC (or memory if a miss), computes y = floor(ax + b) with x = VPN, and follows the child pointer (offset into the internal-node array at the next level). This recurses up to 3 levels. At the leaf, y = floor(ax + b) yields PA_PTE. The walker fetches the PTE (one 64-byte cache line, which clusters 8 PTEs for spatial locality), checks the VPN tag, and returns the PPN. If the tag misses, binary search within [y - C_err, y + C_err] (up to 3 extra accesses).

💡
Why one adder + one multiplier? Fixed-point: 44-bit integer + 20-bit fractional. Each model = 8 bytes slope + 8 bytes intercept = 16 bytes. The LWC is 16 entries × 16 bytes = 256 bytes (plus tags). Compare to radix PWC: 3 levels × 32 entries × 8 bytes = 768 bytes. LVM is 3× smaller in bytes.

3 · Handling insertions

LVM handles two insertion classes: (i) within-bounds (the new VPN falls inside an existing leaf's range), and (ii) out-of-bounds (the new VPN extends the range). For within-bounds, LVM queries the leaf model, attempts to insert at the predicted position, and if occupied, performs exponential search for an empty gap. If the leaf's GPT is full, it retrains only that leaf (local retrain). This is rare because the gap factor (1.3) leaves 30% empty slots.

For out-of-bounds insertions near the edge (the common case, as applications expand contiguously), LVM uses two techniques: (1) minimum insertion distance (default 64 MB): always expand the range by at least this amount, absorbing temporary non-contiguous inserts and amortizing fixed costs. (2) Rescaling: expand the GPT size without modifying the model. For example, if the leaf's range is [500, 1000] and a VPN 1030 arrives, LVM expands the range to [500, 1050], resizes the GPT by 1.3× (adding 65 slots), and inserts VPNs 1001–1050 at positions predicted by the unchanged model. No retraining, no re-insertion of existing keys.

For out-of-bounds insertions far from the edge (less common), LVM may rebuild the parent node or the entire index. The paper measures this: retraining occurs at most 3 times in the OS prototype workloads, mostly during initialization. Training time: <1.9 ms.

🍯
Insight: because applications expand their address space contiguously (mmap, brk), LVM's rescaling avoids retraining 99.9% of the time. Radix and ECPT also handle insertions (allocate new leaf tables), but LVM's adaptability is what makes learned indexes practical for virtual memory.

4 · The cost model

LVM's cost model (Eq. 1) balances three competing goals: (i) minimize depth (fewer indirections), (ii) keep the index small (better cacheability), (iii) reduce collisions (fewer mispredictions). The weights x₁ = 10, x₂ = 5, x₃ = 200 heavily penalize collisions, lightly penalize size, and moderately penalize depth. The depth limit (d_limit = 3) ensures LVM never exceeds radix's worst-case indirections. The coverage constraint ensures each node maps enough address space per byte to match or exceed radix's upper levels (e.g., L2 in radix covers 512 GB in 4 KB).

During training, the cost model computes the number of spline points (segments where a piecewise-linear model fits tightly) for the keys in a node. This gives a rough estimate of how many child nodes are needed. LVM evaluates C(n) for n ± 2 around that estimate, picks the minimum cost, and recurses. If a leaf node violates C_err = 3, the cost model backtracks to the parent and boosts x₃ to force more subdivisions.

🌱
Concretely: for graph workloads (75 GB footprint), the learned index is 112 bytes (7 models). For memcached (124 GB), also 112 bytes. For MUMmer (20 GB), 128 bytes. The index size is independent of memory footprint—unlike radix PWCs, which grow linearly. This is LVM's scalability: a single linear function can describe an arbitrarily large contiguous region.

5 · Hardware synthesis and area

The authors implemented the LVM page walker in RTL (22nm PDK) and synthesized the LWC with CACTI. Results: the page walker (one adder, one multiplier, control logic) requires 0.000637 mm². The LWC (16 entries, 256 bytes data + tags) requires 0.00364 mm² and 0.588 mW leakage. Compare to radix PWC (3 levels × 32 entries): 0.0055 mm², 1.1 mW. LVM is 1.5× smaller in area, 1.9× lower power. The page walk pipeline completes model lookup in 2 cycles at 2 GHz. The entire MMU (L1 TLB, L2 TLB, LWC, page walker) fits in <0.01 mm²—well within the processor's uncore budget.

🍯
Insight: fixed-point arithmetic (44-bit integer + 20-bit fractional) is cheap. One multiply-add per internal node; no FP units, no division, no expensive ops. The learned index's compactness (16 bytes per model) is what makes the LWC tiny and the pipeline fast.
Follow the numbers, step by step

Worked examples

Four traces from the paper, each carried all the way through—training a leaf model, handling a misprediction, computing the cost model, and tallying the hardware area.

① Training a leaf model for a 50-page region

You have 50 VPNs in the range [100, 149] to be mapped by a leaf node. The OS allocates a GPT starting at PA 0x2000. Walk through the training.

  1. Sort the VPNs: [100, 101, …, 149]. These are the keys.
  2. Assign positions: VPN 100 → position 0, VPN 101 → position 1, …, VPN 149 → position 49.
  3. Linear regression: fit y = ax + b to (VPN, position). Since the VPNs are sequential, the fit is perfect: a = 1, b = -100. So y = 1×x - 100.
  4. Scale by gap factor (1.3): multiply slope by 1.3. New model: y = 1.3×(x - 100) = 1.3x - 130.
  5. Allocate GPT: size = 50 × 1.3 = 65 slots, each 8 bytes (PTE size) = 520 bytes. PA range: 0x2000–0x2208.
  6. Add base PA to model: final model: y = 1.3x - 130, outputting slot index. PTE physical address = 0x2000 + (slot_index × 8).
  7. Insert PTEs: for VPN 100, slot index = 1.3×100 - 130 = 0, so PA = 0x2000. For VPN 101, slot index ≈ 1.3, so PA ≈ 0x2000 + 10 = 0x200A (slot 1). Gaps appear naturally due to the 1.3 scaling.
🍯
Insight: the 1.3 scaling creates 15 empty slots (30% of 50). When new VPNs arrive in-bounds, they insert into these gaps without retraining. The model remains y = 1.3x - 130.

② Handling a collision (misprediction)

A page walk for VPN 120 predicts a slot index, but the VPN tag at the predicted PTE location is 121 (collision). How does LVM recover?

  1. Leaf model outputs: slot index = 1.3×120 - 130 = 26.
  2. Compute PTE address: PA = 0x2000 + (26 × 8) = 0x20D0.
  3. Fetch PTE at 0x20D0: the VPN tag is 121 (not 120). Collision detected.
  4. Binary search within error bound: LVM enforces C_err = 3, so search slots [26-3, 26+3] = [23, 29]. Addresses checked: 0x20D0 (miss), 0x20B8 (miss), 0x20E8 (hit at slot 29, VPN 120).
  5. Total accesses: 1 (initial fetch) + 2 (binary search) = 3 accesses. Within the bound.
💡
Collisions occur 0.6% of the time on average. When they do, LVM's bounded search (Cerr = 3) guarantees ≤4 total accesses (1 initial + 3 search). Radix always takes 5; ECPT sends 3 parallel probes (3× traffic) even on hits.

③ Cost model: choosing the number of child nodes

An internal node covers VPNs [0, 1000]. The cost model computes the number of spline points = 4. Should it create 4, 5, or 6 children?

  1. Evaluate C(n) = x₁·d + x₂·s + x₃·cr·ma for n = 4, 5, 6. Assume depth = 2, size per child = 16 bytes (one model), collision rate = 1% for n=4, 0.5% for n=5, 0.3% for n=6, and avg extra accesses = 2.
  2. n = 4: d = 2, s = 4×16 = 64, cr = 0.01, ma = 2. C(4) = 10×2 + 5×64 + 200×0.01×2 = 20 + 320 + 4 = 344.
  3. n = 5: C(5) = 10×2 + 5×80 + 200×0.005×2 = 20 + 400 + 2 = 422.
  4. n = 6: C(6) = 10×2 + 5×96 + 200×0.003×2 = 20 + 480 + 1.2 = 501.2.
  5. Pick minimum: C(4) = 344 is lowest. Create 4 children.
💡
The cost model heavily penalizes collisions (x₃ = 200), so it prefers more subdivisions if collisions are high. But it also penalizes size (x₂ = 5) and depth (x₁ = 10), so it won't subdivide unnecessarily. This balance keeps the index small and accurate.

④ Tallying the hardware area

Compare the area of LVM's LWC to radix's PWC.

  1. LVM LWC: 16 entries, each entry = 16 bytes data (slope + intercept) + 16 bits tag + 2 bits level + 14 bits offset = 16 bytes + 4 bytes overhead ≈ 20 bytes per entry. Total: 16×20 = 320 bytes. CACTI synthesis: 0.00364 mm².
  2. Radix PWC: 3 levels, each level = 32 entries × 8 bytes (PA pointer) = 256 bytes per level. Total: 3×256 = 768 bytes. CACTI synthesis: 0.0055 mm².
  3. Ratio: 0.0055 / 0.00364 = 1.5×. LVM is 1.5× smaller in area.
  4. Power: LVM = 0.588 mW leakage. Radix = 1.1 mW. LVM is 1.9× lower.
  5. Page walker: LVM = one adder (16×16 fixed-point multiply) + one adder = 0.000637 mm². Radix = control logic for sequential fetches, similar area.
🍯
Insight: the learned index's compactness (112–200 bytes steady-state) is what makes the LWC tiny. Radix PWCs cache upper-level page table entries (8 bytes each), but even with 32 entries per level, the cache is larger. LVM's models are more space-efficient because one model describes many PTEs.

Practice problems

Try each before opening the answer. Every one ends with the common misconception it's checking for.

Problem 1 · easy
An application allocates 1 TB of virtual address space (all anonymous, no file-backed). Its VPNs are 100% sequential. How many bytes does LVM's learned index require, and how does this compare to radix PWCs?
Show answer
LVM's index is ~112–200 bytes regardless of footprint. A single linear function can describe the entire 1 TB if the VPNs are perfectly sequential. The cost model may subdivide into 2–3 internal nodes for better error bounds, but the total remains <200 bytes. Radix PWCs, in contrast, grow linearly: at L2 (512 GB coverage per entry), 1 TB requires 2 entries; at L3 (1 GB coverage), 1024 entries; at L4 (2 MB coverage), 524,288 entries. Even caching only L2 and L3, the PWC is orders of magnitude larger. LVM's scalability is its key advantage.Misconception: that learned indexes grow with footprint. They don't—they grow with distribution complexity. A sequential address space is maximally simple, so LVM's index stays tiny.
Problem 2 · medium
A leaf model is trained on 100 VPNs [500, 599] with gap factor 1.3. The GPT has 130 slots. Later, the app frees VPNs [550, 559] (10 pages). What happens to the learned index, and how does LVM handle future lookups for those VPNs?
Show answer
LVM clears the PTEs in the GPT for VPNs [550, 559], flushes the TLB, and releases the physical pages. The learned index (the model y = ax + b) is not modified. The 10 empty slots remain in the GPT. Future lookups for VPNs [550, 559] will compute the correct slot index, fetch the PTE, find it invalid (present bit = 0), and trigger a page fault. The OS handles the fault as usual (allocate a new physical page, re-insert the PTE). Because the app's allocator tends to reuse holes in the virtual address space, LVM bets that these slots will be reused later, avoiding retraining. If the app's memory usage peaks and then drops significantly, the OS can optionally rebuild the index to reclaim space, but this is rare.Misconception: that freeing pages requires retraining. LVM's design treats the learned index as a map of the virtual address space, not a reflection of active pages. Inactive PTEs simply have invalid entries in the GPT.
Problem 3 · medium
LVM's paper reports collision rates of 0.2% for 4KB pages and 0.6% for THP (2MB pages). Why is the collision rate higher for THP?
Show answer
THP uses fewer PTEs for the same memory footprint (one 2MB PTE covers 512 4KB PTEs). With fewer keys (VPNs), the learned index has less data to fit, and the linear regression may have slightly higher error. Additionally, THP allocations can introduce small irregularities (e.g., a 2MB page starting at an odd offset) that increase mispredictions. However, 0.6% is still very low—99.4% of lookups hit on the first try. The paper also notes that the cost model enforces C_err = 3, so even when collisions occur, they require ≤3 extra accesses.Misconception: that THP should have lower collision rates because there are fewer keys. In reality, THP's larger granularity can introduce misalignment in the learned model, especially if applications mix 4KB and 2MB pages. LVM handles this by encoding page size in the PTE and rounding down lookups.
Problem 4 · hard
LVM's cost model enforces d_limit = 3 (max depth). Construct a pathological virtual address space where LVM's index depth exceeds 3, and explain how the cost model prevents this in practice.
Show answer
A pathological case: VPNs are clustered in many tiny, far-apart regions (e.g., VPNs [0, 10], [1000, 1010], [2000, 2010], …, [100000, 100010]). The cost model computes many spline points (high complexity) and attempts to subdivide recursively. However, the cost model's coverage constraint kicks in: if a node doesn't map enough address space per byte (relative to radix's upper levels), the cost model forbids further subdivisions. So the index bottoms out at depth 3, and the leaf nodes have higher collision rates (violating C_err = 3). At that point, the cost model boosts x₃ (collision penalty) and backtracks to the parent, forcing more children at depth 2 instead. The result: a wider tree at depth 2–3, not a deeper tree at depth 4+. This prevents LVM from degenerating into a deep hierarchy worse than radix.Misconception: that LVM will always find a perfect model. The cost model is a heuristic that balances depth, size, and accuracy. In pathological cases, LVM may sacrifice some accuracy (higher collision rate) to maintain shallow depth and good cacheability. The paper shows that even with d_limit = 3, LVM outperforms radix and ECPT on real workloads.
Problem 5 · medium
LVM uses dynamic set sampling (DSS) in database learned indexes to reduce training time. But the paper doesn't mention DSS for LVM. Why? And how does LVM keep training fast (<1.9 ms)?
Show answer
DSS is a technique where only a subset of keys is used for training, with the assumption that the sample represents the full dataset. LVM doesn't use DSS because virtual address spaces are already small relative to database workloads. Even a 1 TB footprint with 4KB pages = 256 million VPNs. Training a linear model via least-squares regression on 256M points is fast (<1 ms) with modern CPUs. Database learned indexes deal with billions of keys, so DSS matters there. For LVM, the bottleneck is not regression but the cost model's evaluation of candidate tree structures. The paper limits the cost model to ±2 candidates around the spline-point estimate, which keeps the search space small. Result: training time <1.9 ms even for large address spaces.Misconception: that LVM's training is expensive. It's not—linear regression is O(n) for n keys, and LVM's hierarchical structure subdivides the keys, so each node trains on a small subset. The total training time is dominated by the cost model, not the regression itself.
Problem 6 · hard
LVM's rescaling technique handles out-of-bounds insertions near the edge without retraining. Explain why rescaling works for a learned linear model but wouldn't work for a learned neural network model.
Show answer
Rescaling works because a linear model y = ax + b has a simple property: if you expand the key range [k_min, k_max] to [k_min, k_max + Δ], the new keys follow the same linear relationship. Specifically, for VPN x > k_max, y = ax + b still predicts the correct position (modulo the gap factor). So you just expand the GPT to accommodate the new range and insert new keys at positions predicted by the unchanged model. A neural network, in contrast, is a black box: its predictions depend on learned weights that are tuned to the training data [k_min, k_max]. Extrapolating beyond that range (predicting for x > k_max) is unreliable—the network may output nonsense. So rescaling requires retraining the network on the expanded range. This is why LVM uses linear models: they're explainable and extrapolate predictably.Misconception: that any learned model can be rescaled. Only models with simple, predictable extrapolation (like linear models) support rescaling. Neural networks and decision trees don't have this property, which is one reason prior learned indexes struggled with insertions.
Problem 7 · easy
LVM's paper reports that the learned index is "independent of memory footprint." A skeptic says: "But if the footprint doubles, the number of PTEs doubles, so the GPTs must grow." Is the skeptic right? Clarify what "independent of footprint" means.
Show answer
The skeptic is right that the GPTs grow (they hold the PTEs, which scale linearly with footprint). But the learned index (the hierarchy of models) does not grow. If the address space expands contiguously (e.g., from 500 GB to 1 TB), LVM rescales the leaf GPTs (expand their size) without adding new models. The models themselves remain the same: a single linear function y = ax + b can describe an arbitrarily large contiguous region. So the index size (in bytes) is constant, even as the GPTs grow. This is the key scalability property: radix PWCs must cache more upper-level entries as footprint grows (e.g., L2 covers 512 GB, so 1 TB needs 2 L2 entries), but LVM's learned index stays tiny. "Independent of footprint" refers to the index, not the GPTs.Misconception: that the entire page table structure (models + GPTs) is independent of footprint. Only the models are. The GPTs scale linearly, but they're stored in main memory, not cached in the MMU. The LWC caches only the models, which stay tiny.
Problem 8 · medium
LVM supports multiple page sizes (4KB, 2MB, 1GB) by encoding them in the PTE and training on the first 4KB sub-page's VPN for huge pages. A colleague asks: "Can LVM support 64KB base pages (common on ARM)?" Walk through what changes.
Show answer
Yes, LVM supports arbitrary page sizes. For 64KB base pages: (1) the PTE size remains 8 bytes, but the VPN offset shrinks (log₂(64KB) = 16 bits instead of 12 for 4KB). (2) The learned model trains on VPNs at 64KB granularity. For a 2MB huge page (32× 64KB), the model trains on the first 64KB sub-page's VPN and the PTE encodes "2MB page." During lookup, the hardware derives the 64KB VPN from the VA, the model rounds down to the 2MB PTE. (3) The GPT slot size (PTE size) stays 8 bytes, so the gap factor and collision bounds remain the same. No modifications to the cost model or learned index logic. The only change is the VPN/offset split in the VA and the page size encoding in the PTE. This is why the paper envisions LVM can support any number of page sizes as future work.Misconception: that different page sizes require separate learned indexes or hardware modifications. LVM's linearity and slope encoding allow existing page sizes in one unified structure using current PTE encoding. Support for arbitrary page sizes without OS changes is mentioned as future work.
Quick answers

Frequently asked questions

Ten questions your fellow PhD students ask most. Tap any to expand.

Q1What's the key difference between LVM and database learned indexes?▾
Database learned indexes assume a static dataset and use complex models (neural networks, recursive model indexes). LVM is tailored for virtual memory: (i) uses only simple linear models (y = ax + b) for fast, fixed-point arithmetic, (ii) handles dynamic insertions via rescaling and local retraining, (iii) enforces a shallow hierarchy (d_limit = 3) for low latency, (iv) adapts GPT allocation to physical memory fragmentation. Database indexes aim for accuracy; LVM balances accuracy with hardware constraints (cacheability, latency, area).
Q2Why doesn't LVM use neural networks? Wouldn't they be more accurate?▾
Neural networks have three dealbreakers for page walks: (1) inference cost—tens to hundreds of ops per prediction, vs. one multiply-add for a linear model. The page walker would need an expensive NN accelerator, bloating area and power. (2) no extrapolation—NNs can't reliably predict beyond their training data, so rescaling wouldn't work. (3) floating point—NNs rely on FP32/FP16, which the kernel forbids and hardware avoids. LVM's linear models achieve 99.4% accuracy (0.6% collisions) on real workloads, which is sufficient. The 0.6% difference vs. a hypothetical perfect model isn't worth the complexity.
Q3How does LVM handle TLB shootdowns and context switches?▾
TLB shootdowns occur only when PTEs are modified (e.g., swapping, permission changes). LVM's insertions and rescaling don't modify existing PTEs—they only add new ones—so no shootdown. On retraining (rare, occurring at most 3 times in prototype workloads), if a model node changes, the OS flushes that entry from the LWC (similar to flushing a TLB entry). Context switches: the LWC uses ASID tags (like TLBs) to disambiguate per-process entries, so no flush is needed. The OS updates the control registers (base PAs of the learned index levels) on switch, just like updating CR3 in radix.
Q4What if the virtual address space is pathologically fragmented (e.g., ASLR with huge gaps)?▾
ASLR partitions the address space into distinct regions (text, data, heap, stack, mmaps) that are far apart. LVM handles this by creating separate leaf nodes for each ASLR region. Because each region is internally contiguous, the per-region models are accurate. The paper validates this on real ASLR-enabled workloads (Meta production) and shows 0.6% collision rates. If a pathological app creates truly random scattered mappings, the cost model's depth limit and coverage constraint bound the index size, and LVM accepts higher collision rates while still meeting the Cerr = 3 bound.
Q5How does LVM's learned index compare to a hash table in terms of collision rates?▾
The paper measures this directly. A hash table (Blake2 hash, load factor 0.6) achieves 22% collision rate for 4KB pages, 19% for THP. LVM achieves 0.2% for 4KB, 0.6% for THP. That's a 37× reduction. Why? Because LVM learns the distribution of VPNs, while a hash function treats all keys as uniform-random. Real virtual address spaces are 78–100% sequential, so LVM's linear models fit tightly. Hash tables can't exploit this regularity—they're designed for worst-case adversarial inputs.
Q6Can LVM support nested page tables (virtualization)?▾
Yes. The paper briefly mentions this. In nested paging, the hypervisor maintains LVM page tables for GPA→HPA (guest physical to host physical), and each guest OS maintains LVM page tables for GVA→GPA (guest virtual to guest physical). A nested walk traverses both: (1) walk the guest's LVM tree (GVA→GPA), (2) for each internal node and leaf fetch, walk the hypervisor's LVM tree (GPA→HPA). The worst-case depth is 2× (guest tree depth + host tree depth). Because LVM's tree is shallow (≤3 levels), nested walks are ≤6 accesses—still better than radix's 2D page walks (up to 25 accesses in x86-64). The authors leave detailed nested evaluation for future work, but the design supports it.
Q7What happens if an application uses huge pages opportunistically (e.g., Linux THP on some regions, 4KB on others)?▾
LVM handles this seamlessly because the learned index doesn't care about page size—it maps VPNs to PTEs. A 2MB page at VPN 1024 covers VPNs [1024, 1536). The model trains on VPN 1024; lookups for any VPN in [1024, 1536) round down to 1024 and fetch the 2MB PTE. The PTE itself encodes "2MB page" in 2 bits. If the app later splits that huge page into 512 4KB pages, the OS retrains the leaf node (local retrain) to insert 512 new PTEs at finer granularity. The cost: <1.9 ms per retrain, occurring at most 3 times in prototype workloads (mostly at init). In practice, THP usage is stable after init, so this is rare.
Q8How does LVM perform on NUMA systems with interleaved memory?▾
LVM is orthogonal to NUMA. The learned index maps VPNs to PTEs; the PTEs contain PPNs, which the hardware then translates to NUMA node and channel via the memory controller's address mapping (typically low-order bit interleaving). LVM doesn't change this. However, LVM's low latency (1 access common case) reduces page walk stalls, which is especially beneficial on NUMA: in radix, if a page walk fetches from a remote NUMA node (5× latency penalty), the stall is long. LVM minimizes the number of fetches, so remote-node penalties are less frequent. The paper doesn't evaluate NUMA explicitly, but the traffic reduction (43% for 4KB, 34% for THP) should help.
Q9Can LVM's learned index be precomputed offline (e.g., at compile time for static binaries)?▾
Theoretically, yes—if the virtual address space were fully known at compile time. But in practice, (i) the OS assigns ASLR base addresses at load time (randomized), (ii) shared libraries are mmapped at runtime, (iii) dynamic allocations (brk, mmap) expand the heap at runtime. So the learned index must be trained at runtime. However, LVM's training is fast (<1.9 ms) and happens only during init, so the overhead is negligible. For deeply embedded systems with static memory maps (no OS, no ASLR), precomputing the learned index could eliminate even that overhead, but this is outside the paper's scope.
Q10What did the authors leave for future work?▾
The paper mentions: (i) evaluating nested page tables (virtualization) in detail, (ii) extending LVM to physically-indexed caches (e.g., learning the PPN→cache-set mapping to reduce conflict misses), (iii) supporting arbitrary page sizes without OS or hardware changes as an envisioned extension, (iv) applying LVM's techniques to other hardware indexing structures (TLBs, BTBs). They also note that LVM's cost model could be extended with additional parameters to optimize for tiered memory systems.

Glossary & further reading

Every term in one place, then a path to go deeper.

Core terms
LVM (Learned Virtual Memory)The paper's page table design: a hierarchy of learned linear models that map VPNs to PTEs in one access (99.4% PTE-location collision rate).
VPN (Virtual Page Number)The key: the upper bits of the virtual address (VA) after removing the page offset. The learned index maps VPN → PAPTE.
PTE (Page Table Entry)The value: an 8-byte entry mapping a VPN to a PPN, plus permission bits, accessed/dirty bits, and page size encoding.
Learned indexA hierarchy of models (y = ax + b) that predict the position of a value (PTE) in a table from its key (VPN). Learns the CDF of keys.
GPT (Gapped Page Table)A small array (KB-scale) holding PTEs for a leaf node. Contains empty slots (gaps) for insertions. Non-contiguous physical allocation.
Cost modelA function C(n) = x₁·d + x₂·s + x₃·cr·ma that balances depth, size, and collision rate to pick the optimal number of child nodes.
LWC (LVM Walk Cache)A 16-entry fully associative cache (256 bytes) that stores learned models. 3× smaller in bytes than radix PWCs, 1.5× in area, 1.9× lower power.
RescalingA technique for handling out-of-bounds insertions near the edge: expand the GPT without modifying the model. Avoids retraining.
Cerr (error bound)The maximum number of additional memory accesses allowed for collision resolution (default: 3). Enforced during training.
d_limit (depth limit)The maximum depth of the learned index hierarchy (default: 3). Ensures LVM never exceeds radix's worst-case indirections.
Supporting terms
Radix page tableThe baseline: a 5-level tree (x86-64) requiring up to 5 sequential memory accesses per translation.
ECPT (Elastic Cuckoo Page Tables)A hashed page table design (3-way cuckoo) that sends 3 parallel probes (3× traffic) to reduce latency.
PWC (Page Walk Cache)A structure in radix MMUs that caches upper-level page table entries (L1, L2, L3) to reduce sequential fetches.
Gap coverageThe fraction of VPNs that are sequential (VPNnext − VPNcurrent = 1). LVM's evaluation shows 78–100% gap = 1 for real workloads.
Linear modelA function y = ax + b. LVM's building block. Fast to compute (one multiply, one add), easy to train (linear regression), and extrapolates predictably.
Spline pointsSegments where a piecewise-linear model fits the data. The cost model uses the number of spline points to estimate the optimal number of children.
ga_scale (gap factor)The scaling factor for gapped arrays (default: 1.3). Determines the percentage of empty slots (30%) reserved for insertions.
Minimum insertion distanceThe smallest VPN range expansion (default: 64 MB). Amortizes fixed costs and absorbs temporary non-contiguous inserts.
FMFI (Free Memory Fragmentation Index)A metric (0–1) for physical memory fragmentation. 0 = no fragmentation, 1 = maximum. The paper tests LVM at FMFI = 0.9.
Fixed-point arithmeticRepresenting values as 44-bit integer + 20-bit fractional. Avoids floating point in hardware. Each model = 16 bytes (slope + intercept).
THP (Transparent Huge Pages)Linux's automatic 2MB page allocation. LVM supports THP by encoding page size in the PTE and training on the first 4KB sub-page's VPN.
ASLR (Address Space Layout Randomization)Randomizes base addresses of text/data/heap/stack. LVM handles this by creating separate leaf nodes per ASLR region.

Prefetches page table entries during radix walks. Requires large physical contiguity (hundreds of MB). LVM adapts to fragmentation (KB-scale GPTs) and outperforms ASAP by 8%.

Park et al., ASPLOS'22

Folds adjacent radix levels using 2MB pages. Requires 2MB physical contiguity (scarce in datacenters). LVM works with 4KB contiguity and supports all page sizes.
