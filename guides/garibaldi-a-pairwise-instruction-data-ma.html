Keeping instructions in the LLC when their data is hot

In server workloads, cold instructions are evicted to cache hot data — then the pipeline starves waiting for those instructions, even though their data is right there. Garibaldi pairs instruction lines with the hot data they trigger, selectively protecting the instructions and prefetching paired data on unprotected misses.

📄 Garibaldi · Kwon et al. 🎓 Assumes LLC basics ⏱ ~25 min read
🔗

If an instruction line triggers hot data, protect the instruction — its miss costs you every hot data access it would have made. If it's cold and unprotected, prefetch its paired data during the instruction miss to overlap latencies.

Before you begin

What Garibaldi assumes you know

Seven quick primers on shared LLCs, frontend stalls, and why server workloads differ from SPEC. Tap any to expand; read in order if you're new to the domain.

hot instruction (protected) hot data (cached) Garibaldi machinery baseline / victim
1The shared LLC in a many-core server chip▾

Modern server CPUs (40–144 cores) share one big last-level cache (LLC) among all cores. It's the final on-chip stop before DRAM. In the paper's 40-core config, the LLC is 30MB (0.75MB/core), 12-way associative. All instruction and data lines from all cores contend for the same sets.

Speculative extrapolation (not demonstrated in the paper): If you're working with CXL-attached memory or tiering, the LLC sits before that tier — if the LLC stalls, your tiering can't help. The LLC is where the instruction victim problem lives.

Now you're ready: Garibaldi targets exactly this shared LLC, deciding which instruction lines to protect from eviction by tracking the hot data they trigger.
2Instruction vs data accesses, and the ifetch stall▾

A CPU does two kinds of memory access: instruction fetch (ifetch) to bring code into the pipeline, and data access to load/store operands. An ifetch miss stalls the frontend — the decoder can't feed the backend until the instruction arrives, no matter how much data is cached.

In traditional (SPEC) workloads, ifetch is negligible. In server workloads (the paper's verilator, kafka, etc.), ifetch stalls are the main bottleneck — up to 42% faster if instructions always hit (Fig. 3d).

Now you're ready: One instruction miss can delay many hot data accesses, making the instruction miss far costlier than a single data miss.
3LRU and reuse-distance–based LLC policies▾

Baseline LRU evicts the least-recently-used line. Modern schemes (Hawkeye, Mockingjay) predict reuse distance and prioritize caching lines with near-term reuse, achieving hit-status similarity of 90–99% to the optimal hot set under modern LLC policies.

These policies excel at caching hot data but treat instructions and data equally. They don't account for the asymmetry: a missed instruction delays every hot data access it triggers.

Now you're ready: Garibaldi integrates with these policies (e.g., Mockingjay), propagating data hotness to instructions without rewriting the replacement logic.
4Why server workloads have huge instruction footprints▾

Server apps (kernel ops, virtualization, networking) have deep software stacks. The instruction footprint (unique code touched) exceeds the L1I and L2 capacity, so a substantial fraction of instruction accesses reach the LLC. The paper measures 13.4% of LLC accesses as instructions in server workloads vs 0.26% in SPEC.

Even with advanced branch prediction (TAGE/ITTAGE) and instruction prefetching (I-SPY), ifetch misses dominate: 95.9% instruction miss rate in the LLC for servers vs 42.11% data miss rate.

Now you're ready: The LLC is instruction-miss–heavy in servers, unlike SPEC where instructions rarely reach it.
5The many-to-few vs few-to-many access pattern▾

SPEC: a few hot instructions (avg 24.7 accesses/line) trigger accesses to many dispersed data lines (avg 1.0 accesses/line). Few-to-many. Server: many cold instructions (avg 13.7 accesses/line) trigger a few hot data lines (avg 4.8 accesses/line). Many-to-few.

In many-to-few, the hotness of data doesn't naturally transfer to the instruction lines that access it. Caching hot data evicts the cold instructions — then the pipeline stalls waiting for those instructions, even though the data is present.

Now you're ready: Garibaldi explicitly propagates data hotness to instructions in the many-to-few pattern, protecting instruction victims.
6Opportunity cost of an instruction miss▾

When an instruction line I misses and its subsequent data accesses D₁, D₂ hit in the LLC, the frontend stalls waiting for I, delaying D₁, D₂. The opportunity cost is all the cycles lost waiting for I instead of immediately using the cached data.

The paper shows: if I triggers hot data (high LLC hit rate), I's miss is much costlier than if I triggered cold data (misses anyway). In verilator, 90.5% of instructions leading to data hits are themselves missed, vs 67.6% for instructions leading to data misses (Fig. 4c).

Now you're ready: Garibaldi quantifies this cost via a miss_cost counter that increments when paired data hits, decrements when it misses.
7Why not partition the LLC for instructions?▾

Partitioning (reserving N ways for instructions) worked in private L2 caches (Emissary). But shared LLCs with 40+ cores and only 12–16 ways can't afford a fixed partition — dedicating 2 ways to instructions reduces data capacity enough to hurt, and more than 2 ways degrades below baseline (Fig. 14d).

Garibaldi uses query-based selective protection instead: instructions are protected only if their miss cost exceeds a dynamic threshold, with no fixed partition reducing associativity.

Now you're ready: Selective protection avoids the capacity/associativity trade-off of static partitioning.

The whole story in one place

Garibaldi in a nutshell

The problem, the measurement, the solution, and the payoff — all grounded in the paper's numbers.

Modern LLC replacement policies (Mockingjay, Hawkeye) excel at caching hot data by predicting reuse distance. But in server workloads with a many-to-few access pattern — many cold instructions triggering a few hot data lines — these policies create a trap: they cache the hot data, evicting the cold instructions, then the pipeline stalls waiting for those instructions, wasting the benefit of cached data.

The paper calls this the instruction victim problem. In a 40-core server, instruction accesses are 13.4% of LLC traffic (vs 0.26% in SPEC), with a 95.9% miss rate. The opportunity cost is high: 90.5% of instructions triggering data hits are themselves missed, vs 67.6% for instructions triggering data misses (verilator). An oracle that makes instructions always hit after first access improves performance 42.5% over LRU vs Mockingjay's 6.3%.

Garibaldi propagates data hotness to instructions via a pair table that tracks (instruction P.A. → [miss_cost, data P.A.s]). Each data LLC access updates the paired instruction's miss_cost: increment if the data hit (hot), decrement if miss (cold). A selective protection mechanism queries the table on eviction: if an instruction's miss_cost exceeds a dynamic threshold, it's protected. For unprotected instruction misses, pair-wise prefetch brings the paired data lines into the LLC, overlapping data miss latency with instruction fetch.

Storage: 193.9KB for 40 cores (0.6% of 30MB LLC) — 16K-entry pair table (k=1 data line per instruction), 8K-entry D_PPN table, 128-entry/core helper table, plus 1-bit I/D indicator per cacheline (total 0.8%). Performance: on 16 server workloads, Garibaldi+Mockingjay achieves 13.2% speedup over LRU (vs Mockingjay's 6.1%), 18% ifetch stall reduction, 10.4% energy savings.

🍯 Key takeaways

  • Modern LLC schemes cache hot data but ignore that instruction misses delay access to that data.
  • Server workloads: many-to-few pattern (many cold instructions → few hot data) + 13.4% instruction LLC accesses + 95.9% ifetch miss rate.
  • Instruction victims: 90.5% of instructions triggering data hits are themselves missed (verilator).
  • Pair table: maps IL_P.A. → (miss_cost, DL_P.A.s); miss_cost += on data hit, -= on miss.
  • Selective protection: query-based, no partition; protect if miss_cost > dynamic threshold.
  • Pair-wise prefetch: on unprotected instruction miss, prefetch paired cold data lines.
  • 13.2% speedup (vs 6.1% Mockingjay), 18% ifetch reduction, 0.8% LLC storage overhead.

The main dish (instruction) is delayed. Instead of sitting idle, the waiter brings appetizers (paired data lines) you'll need anyway. The full explanation

How it actually works

Pair table tracking, dynamic threshold, selective protection, pair-wise prefetch, and the storage optimizations. All numbers from the paper.

1 · Tracking instruction-data pairs

Garibaldi needs to map (instruction P.A. → data P.A.s). The challenge: the LLC sees physical addresses, but the PC (instruction virtual address) is the signature passed with each access. The solution is a helper table (128 entries/core, 4-way) that acts like an ITLB, translating PC → instruction P.A. at page granularity.

  1. Instruction fetch: The core accesses LLC with (PC, IL_P.A.). The helper table records (PC_VPN → I_PPN) = (PC's page frame → instruction page frame).
  2. Data access: The core accesses LLC with (PC, DL_P.A.). The PC is the instruction that triggered the data access. Look up PC_VPN in the helper table to get I_PPN, combine with PC's page offset → deduce IL_P.A.
  3. Main pair table update: Use IL_P.A. to index the pair table (direct-mapped, 16K entries). Update miss_cost (increment if data hit, decrement if miss) and record DL_P.A. in the DL_P.A. field.
🌱
Why not use ITLB directly? To avoid pipeline interference. The helper table is decoupled from the core's ITLB, accessed only during LLC operations, so it doesn't stall instruction fetch.

2 · The pair table structure

Each entry (indexed by IL_P.A.) contains:

  • IL_P.A. tag (24 bits) — identifies the instruction line
  • miss_cost (6 bits, saturating) — the cost metric
  • color (3 bits) — last update time for aging
  • k DL_P.A. fields (k=1 in the eval) — each field stores (D_PPO 6b + D_PPN_idx 13b + old_bit 1b + sctr 3b = 23b)

The D_PPN table (8K entries, decoupled storage) stores the page frame number (D_PPN, 19 bits) shared by multiple DL_P.A. fields, exploiting spatial locality. Each DL_P.A. field keeps only the page offset (D_PPO, 6 bits) and a 3-bit index into the D_PPN table.

DL_P.A. field management: Each field has an old_bit set on color update or instruction miss. Data accesses clear the old_bit and update the 3-bit sctr (saturating counter): if the DL_P.A. matches, increment sctr and clear old_bit; if no match and old_bit is set, decrement sctr and replace the field if sctr < threshold (e.g., 4).

3 · Dynamic threshold adjustment via coloring

The protection threshold determines which instructions are protected. It's adjusted every N LLC accesses (e.g., N=100K) using a 3-bit timer (8 colors, cycling 0→7). At each color transition:

  1. Track P(D_miss | I_miss): Maintain a list of the most recent 10 instruction-miss PCs per thread. On each data access, if the PC matches an instruction miss in the list, increment a hit/miss counter and a total counter. At the end of the color, compute P(D_miss | I_miss) = miss_count / total_count.
  2. Compare to overall LLC miss rate: If P(D_miss | I_miss) < overall miss rate → many data accesses are hits despite instruction misses → decrease threshold (protect more instructions). If P(D_miss | I_miss) > overall miss rate → instructions are over-protected → increase threshold (protect fewer).
  3. Advance color: Update the timer, reset counters, repeat.
💡
Why dynamic? Workloads vary. Fixed thresholds (paper's sensitivity study, Fig. 14b) achieve 5.2–7.4% speedup; the dynamic threshold achieves 10.1%. Adapting to conditional miss rates extracts more benefit.

4 · Selective protection (query-based)

When the LLC replacement policy (e.g., Mockingjay) selects an instruction line for eviction:

  1. Query the pair table: Use the instruction line's P.A. to look up its entry. Retrieve miss_cost and color.
  2. Compute aged_cost: aged_cost = miss_cost − ((current_color − entry_color) mod 8). This decays the cost by the number of color transitions since the entry was last updated.
  3. Threshold check: If aged_cost > threshold → protect (reset the eviction priority to lowest, try the next victim). Else → evict as normal.
  4. Query limit: QBS_MAX_ATTEMPTS=2 caps the worst-case overhead. Pair table lookup is 0.331 ns (reported by the paper), which is 1 cycle at the 3 GHz clock.
🍯
No fixed partition: Unlike static way-partitioning (which reserves N ways for instructions, reducing data capacity), selective protection dynamically protects only high-cost instructions, leaving all ways available for data when not needed.

5 · Pair-wise prefetch on instruction miss

When an unprotected instruction line misses (not present in the LLC and its pair table entry exists but was not protected):

  1. Query the pair table: Use the missed instruction's P.A. to look up the entry.
  2. Check for paired data: If the entry contains k DL_P.A. fields with valid bits set, issue prefetch requests for those data lines.
  3. Overlap latencies: The prefetch happens during the instruction miss handling (fetching from DRAM). By the time the instruction arrives, the paired data lines are in-flight or cached, reducing the delay for subsequent data accesses.
🌱
Why only unprotected misses? Protected instructions have high miss_cost (paired with hot data). If they miss, the data is likely already cached — no need to prefetch. Unprotected misses have low miss_cost (paired with cold data), so prefetching is safe and beneficial.
Follow the numbers, step by step

Worked examples

Four traces using the paper's real data: miss_cost calculation, aging, protection decision, and storage overhead.

① Computing miss_cost from data accesses

Instruction I at IL_P.A. 0xABCD triggers 5 data accesses: D₁ (hit), D₂ (hit), D₃ (miss), D₄ (hit), D₅ (miss). What is the final miss_cost?

  1. Start: Pair table entry for 0xABCD has miss_cost = 0 (or saturated value from previous accesses; assume 0 for simplicity).
  2. D₁ hit: miss_cost++ → 1.
  3. D₂ hit: miss_cost++ → 2.
  4. D₃ miss: miss_cost−− → 1.
  5. D₄ hit: miss_cost++ → 2.
  6. D₅ miss: miss_cost−− → 1. Final miss_cost = 1.
🍯
Insight: The miss_cost is a net balance of hits and misses. If I triggers mostly hot data (high hit rate), miss_cost grows; if cold data, it shrinks. This is how data hotness propagates to the instruction.

② Aging miss_cost with the color timer

An entry has miss_cost = 25, last updated at color 5. The system is now at color 0. The threshold is 23. Should the instruction be protected?

  1. Compute color transitions: From color 5 to color 0 (wrapping around 8-color cycle) = 3 transitions (5→6, 6→7, 7→0).
  2. Aged cost: aged_cost = 25 − 3 = 22.
  3. Compare to threshold: 22 < 23 → not protected. The entry is evictable.
  4. Reason: The entry hasn't been updated for 3 color periods (~300K LLC accesses if N=100K). Its miss_cost has decayed below the threshold, indicating the instruction-data pair is stale.
💡
Without aging, high-miss-cost entries would persist indefinitely even if the program phase shifted. The 3-bit color timer (8 states) provides coarse-grained but effective aging — enough to adapt to phases while keeping overhead low (3 bits per entry).

③ Selective protection in action

Mockingjay selects instruction line I₁ for eviction. The pair table shows I₁ has miss_cost = 30, last color = 0 (current color also 0), threshold = 23. Is I₁ protected?

  1. Query pair table: Entry for I₁ is retrieved.
  2. Aged cost: aged_cost = 30 − ((0 − 0) mod 8) = 30 − 0 = 30.
  3. Threshold check: 30 > 23 → protected. I₁'s eviction priority is reset to the lowest (e.g., LRU position reset).
  4. Next victim: Mockingjay tries the next candidate (say, a data line or another instruction with lower miss_cost). Suppose the next is data line D with RRPV=7 (farthest reuse). D is evicted instead.
🍯
Key point: I₁ is protected without reserving a way. All 12 ways remain available; the protection is dynamic and query-based. QBS_MAX_ATTEMPTS=2 means if the next two candidates are also protected, the third is evicted regardless, capping the overhead at 2 extra queries (2 cycles).

④ Storage overhead calculation

For a 40-core system with 30MB LLC (12-way, 0.75MB/core), compute Garibaldi's storage overhead.

  1. Main pair table: 16K entries (2¹⁴). Each entry: IL_P.A. tag 24b + miss_cost 6b + color 3b + valid 1b = 34b. Plus k=1 DL_P.A. field: D_PPO 6b + D_PPN_idx 13b + old_bit 1b + sctr 3b = 23b. Total per entry = 57b. 16K × 57b ÷ 8 = 120KB.
  2. D_PPN table: 8K entries (2¹³). Each entry: D_PPN 19b + sctr 3b + valid 1b = 22b. 8K × 22b ÷ 8 = 32KB.
  3. Helper table: 128 entries/core, 4-way. Each entry: VPN 29b + PPN 32b + valid 1b + sctr 3b = 64b. 128 × 64b ÷ 8 = 1KB/core × 40 cores = 40KB.
  4. 1-bit I/D indicator: 30MB ÷ 64B = 480K cachelines. 480K bits ÷ 8 ÷ 1024 = 60KB.
  5. Total: 120 + 32 + 40 + 60 = 252KB for 40 cores = 252KB ÷ (30 × 1024KB) = 0.82% of LLC capacity.
💡
The paper reports 193.9KB (0.6%) for the Garibaldi module alone (pair table + D_PPN + helper), then 0.2% for the I/D bit, totaling ~0.8%. Our calculation matches. For a CXL memory system with a huge LLC (e.g., 100MB+), the overhead shrinks to <0.3%.

Problem 1 · easy
In SPEC workloads, instruction accesses are 0.26% of LLC traffic; in servers, 13.4%. Why does Garibaldi help servers but barely affects SPEC?
Show answer
SPEC: few-to-many (few hot instructions → many data); instructions rarely reach LLC, so instruction victims are negligible. Mockingjay's data caching is enough (8.4% speedup vs I-oracle's 9.2%). Servers: many-to-few (many cold instructions → few hot data); instructions flood the LLC (13.4% traffic, 95.9% miss rate), creating instruction victims. Mockingjay only achieves 6.3% speedup vs I-oracle's 42.5%, leaving huge headroom for instruction management.Misconception: that a smart LLC policy like Mockingjay automatically handles instructions. It doesn't — it treats instructions and data equally, missing the asymmetry in server workloads.
Problem 2 · medium
An instruction line I has miss_cost = 15, last color = 3. The system is at color 7, threshold = 20. Is I protected?
Show answer
Color transitions: 3→4, 4→5, 5→6, 6→7 = 4 transitions. aged_cost = 15 − 4 = 11. 11 < 20 → not protected. The entry has decayed below the threshold after 4 color periods (~400K LLC accesses if N=100K).Misconception: forgetting to subtract the color transitions. The miss_cost is aged — old entries decay to prevent stale high-cost entries from being over-protected.
Problem 3 · medium
Garibaldi prefetches paired data lines only on unprotected instruction misses. Why not prefetch on protected misses too?
Show answer
Protected instructions have high miss_cost → paired with hot data → that data is likely already cached in the LLC. Prefetching hot data (already present) wastes bandwidth and evicts other lines unnecessarily. Unprotected instructions have low miss_cost → paired with cold data → prefetching cold data during the instruction miss overlaps latencies without harming cache efficiency.Misconception: that prefetching is always beneficial. Prefetching hot data (already cached) pollutes the LLC; Garibaldi's prefetch is conservative, targeting only cold data indicated by low miss_cost.
Problem 4 · hard
The dynamic threshold adjusts by comparing P(D_miss | I_miss) to the overall LLC miss rate. Explain the logic: why decrease threshold if P(D_miss | I_miss) < overall, and increase if > overall?
Show answer
If P(D_miss | I_miss) < overall miss rate → data accesses after instruction misses are hitting more often than average → the paired data is hot, but instructions are being evicted (instruction victims) → decrease threshold to protect more instructions, conveying that data hotness. If P(D_miss | I_miss) > overall → data accesses after instruction misses are missing more often → instructions are being over-protected despite cold paired data → increase threshold to protect fewer, letting data caching dominate.Misconception: thinking the comparison is arbitrary. It's a feedback loop: if instruction misses lead to unexpectedly high data hits, protect more instructions; if low, relax protection.
Problem 5 · medium
Why does Garibaldi use a direct-mapped pair table (16K entries) instead of a set-associative structure, and why 16K specifically?
Show answer
Direct-mapped = no search, minimal lookup latency (1 cycle via CACTI7 at 22nm). Set-associative would add complexity and latency for query-based protection, which happens on the critical eviction path. 16K entries ≈ 2× the LLC's 8K sets (with oversampling margin), enough to track the working set of instructions. Smaller tables (e.g., 2¹⁰ = 1K entries, Fig. 14c) degrade performance (6.2% vs 10.1%); larger (2¹⁸ = 256K) improve marginally (11.1%) but cost >6% LLC capacity, impractical.Misconception: that set-associative is always better. For query-based protection, latency matters more than avoiding aliasing; direct-mapped is the right trade-off.
Problem 6 · hard
Static partitioning (reserving 2 ways for instructions, Fig. 14d) achieves 6.5% speedup; Garibaldi achieves 10.1%. Why does partitioning underperform despite guaranteeing 2 ways for instructions?
Show answer
Partitioning reduces effective associativity for data (12-way → 10-way), increasing conflict misses and hurting data caching — especially in a 40-core system where LLC pressure is extreme. Some workloads' data miss rate rises enough to offset instruction hit gains, even degrading below baseline (8-way partition in Fig. 14d). Garibaldi's selective protection preserves all 12 ways for data when instructions don't need protection, avoiding the capacity trade-off while still protecting high-cost instructions.Misconception: that reserving capacity is safer than dynamic management. Static partitioning wastes capacity when instructions don't need it and can't adapt to workload phases; Garibaldi's query-based approach is more efficient.
Problem 7 · easy
In verilator, Garibaldi+Mockingjay improves performance 65.2% over LRU (vs Mockingjay's 18.4%). Explain the huge gap using the instruction-victim concept.
Show answer
verilator has severe instruction victims: 73.7% of data lines that hit were shared by multiple instructions, but 90.5% of instructions triggering data hits were themselves missed. Mockingjay caches the hot data but evicts the cold instructions → frontend stalls → data sits idle. Garibaldi protects those instructions (miss_cost > threshold), reducing ifetch stalls by 42% (Fig. 13), unlocking the benefit of the cached data. The 65.2% speedup is the combined effect of data caching + instruction protection.Misconception: that verilator is instruction-heavy so it benefits from any instruction scheme. No — it's the instruction-victim pattern (hot data, cold instructions) that Garibaldi specifically targets.
Problem 8 · medium
Garibaldi's benefit peaks at high LLC associativity (48-way: 7.1% over Mockingjay) but is smaller at low associativity (6-way: 5.4%). Why?
Show answer
At low associativity (6-way), conflict misses dominate for both instructions and data — there's not much capacity to "protect." Mockingjay's data caching is limited by capacity, so instruction protection can't unlock much benefit (few data lines are cached anyway). At high associativity (48-way), conflict misses are rare; the LLC can cache substantial data, but instruction victims remain (instructions still have long reuse distances in multi-core). Garibaldi's protection prevents those instruction victims, fully exploiting the cached data → larger benefit.Misconception: that more associativity always favors data caching. High associativity helps both, but in servers, instruction victims persist even with ample capacity — Garibaldi captures that remaining headroom.
Quick answers

Frequently asked questions

Ten questions readers ask most. Tap any to expand.

Q1How is Garibaldi different from instruction prefetching (I-SPY, FDIP)?▾
Instruction prefetchers (I-SPY, FDIP) predict which instructions to fetch from DRAM into the cache, reducing ifetch latency by overlapping fetch with execution. Garibaldi retains instructions already fetched in the LLC, preventing eviction by data contention. They're orthogonal: prefetching brings instructions in; Garibaldi keeps them in. The paper uses I-SPY as baseline and shows Garibaldi provides additional benefit (42.5% I-oracle headroom vs 6.3% Mockingjay, even with I-SPY active).
Q2What's the connection to tiering/CXL work?▾
Speculative extrapolation (not demonstrated in the paper): CXL-attached memory (e.g., CXL-DRAM, CXL-NVM) relies on the LLC to hide latency. If the LLC stalls on instruction misses, tiering can't help — the CPU is waiting for code, not data. Garibaldi reduces ifetch stalls (18% reduction in servers), which could improve the effectiveness of CXL tiering by keeping the pipeline fed. For OS-managed tiering (e.g., page promotion/demotion), Garibaldi's pair table might inform which pages contain hot instruction-data pairs, guiding promotion to faster tiers — though the paper does not explore this.
Q3Does Garibaldi work with other LLC schemes (Hawkeye, DRRIP, your own policy)?▾
Yes — Garibaldi is orthogonal. The paper implements it on top of Hawkeye, Mockingjay, and DRRIP (Fig. 12). Garibaldi's pair table tracks hit/miss events from the existing policy and selectively protects instructions during eviction. Any policy that reports hit/miss and allows eviction priority resets can integrate Garibaldi. For your policy, hook the data access events to update miss_cost, and query the pair table on instruction eviction.
Q4Why not just increase the L2 cache size instead of managing the LLC?▾
The paper's sensitivity study (Fig. 15b) shows adding 200KB to the LLC improves performance only 0.21% over Mockingjay, and adding 5KB to L1I improves 0.48% — both far below Garibaldi's 5.25%. Reason: L2 is shared by 4 cores (Table 1: 4 MB, shared by 4 cores, 16-way); server workloads' instruction footprint exceeds L2 capacity, so misses still descend to the shared LLC. Garibaldi's benefit comes from managing the shared LLC where multi-core contention creates instruction victims. Increasing L2 doesn't address cross-core contention.
Q5What happens if miss_cost saturates (reaches 63 for a 6-bit counter)?▾
It stops incrementing (saturating counter). This is intentional: once an instruction's paired data is consistently hot, miss_cost maxes out and stays high until aged by the color timer. Saturation prevents overflow and ensures high-cost entries remain protected even if data continues to hit. The 6-bit range (0–63) is empirically sufficient; the paper doesn't report saturation as a problem. If needed, the counter could be widened, but that increases storage overhead.
Q6Does Garibaldi handle instruction prefetch (e.g., FDIP bringing speculative instructions)?▾
Yes. Instruction prefetchers (whether profile-guided like I-SPY or branch-driven like FDIP) generate virtual addresses and follow normal translation. The helper table observes the PC→P.A. mapping during LLC access, so prefetched instructions enter the pair table tracking just like demand-fetched instructions. Garibaldi protects prefetched instructions if their paired data is hot, and prefetches paired data on prefetched instruction misses — no special handling needed.
Q7Why does kafka degrade with Garibaldi (−18% speedup, Fig. 12)?▾
In kafka, both instructions and data are cold (longest reuse distance among all workloads, per the paper's analysis). Data reuse distance is not significantly lower than instruction reuse distance — there's no "hot data, cold instruction" imbalance to fix. Garibaldi's protection trades data caching for instruction caching, but when data is cold too, caching instructions doesn't help (they trigger misses anyway), and reduced data caching hurts. This is an honest limitation: Garibaldi helps workloads with many-to-few patterns (hot data, cold instructions), not uniformly cold workloads.
Q8How does Garibaldi interact with inclusive vs non-inclusive LLCs?▾
The paper uses a non-inclusive LLC (Table 1), where L1/L2 lines aren't required to be in the LLC. Garibaldi tracks pairs at the LLC level, independent of inclusion policy. In an inclusive LLC, protecting a line in the LLC does not automatically keep it in L1I; inclusion only requires that if a line is in an upper-level cache, it must also be in the LLC. LLC eviction may invalidate upper-level copies. The paper evaluates a non-inclusive LLC and does not establish inclusive LLC behavior.
Q9Can Garibaldi be extended to distinguish read vs write data accesses?▾
Currently, Garibaldi treats all data accesses (loads and stores) equally — miss_cost increments on any data hit, decrements on miss, regardless of access type. A refinement could weight read-hits more (since load latency directly affects execution) and write-hits less (stores often buffered). However, the paper's design is simpler and already effective. Such an extension would add complexity (tracking access type in the pair table) for likely marginal benefit; the existing scheme captures the core idea (data hotness) without distinguishing read/write.
Q10What's the latency impact of querying the pair table on the eviction path?▾
Pair table lookup is reported as 0.331 ns by the paper, which is 1 cycle at the 3 GHz clock (Table 1). QBS_MAX_ATTEMPTS=2 caps the worst-case at 2 queries (2 cycles) per eviction. The 2-cycle overhead is small compared to the LLC latency and the DRAM fetch latency (Table 1: 49 ns DRAM access latency, which is about 147 cycles at 3 GHz before queuing) that Garibaldi avoids. The paper reports no timing closure issues.

Glossary & further reading

Core terms
GaribaldiInstruction-data pairwise LLC management: pair table tracks (IL_P.A. → [miss_cost, DL_P.A.]), selectively protects high-cost instructions, prefetches paired cold data.
Instruction victimAn instruction line evicted by LLC data contention despite triggering hot data accesses, causing frontend stalls that delay cached data.
Many-to-fewServer access pattern: many cold instructions trigger a few hot data lines, vs SPEC's few-to-many (few hot instructions, many data lines).
Pair table16K-entry direct-mapped structure mapping IL_P.A. → (miss_cost 6b, color 3b, k DL_P.A. fields); core of Garibaldi's tracking.
miss_cost6-bit saturating counter per instruction line: incremented when paired data hits (hot), decremented when it misses (cold); quantifies opportunity cost of instruction miss.
Selective protectionQuery-based mechanism: on instruction eviction, query pair table; if aged_cost > dynamic threshold, reset eviction priority (protect); else evict.
Pair-wise prefetchOn unprotected instruction miss (low miss_cost), prefetch the k paired data lines recorded in the pair table to overlap data miss latency.
Dynamic thresholdProtection threshold adjusted every N LLC accesses (e.g., 100K) by comparing P(D_miss | I_miss) to overall LLC miss rate; decreases if P < overall (protect more), increases if P > overall.
Coloring (aging)3-bit timer (8 colors) cycling every N accesses; each pair table entry records last color; aged_cost = miss_cost − (current_color − entry_color) mod 8, decaying stale entries.
Helper table128-entry/core, 4-way ITLB-like structure in LLC: translates PC_VPN → I_PPN at page granularity for instruction-data pairing; decoupled from core ITLB to avoid pipeline interference.
Supporting terms
LLC (Last-Level Cache)Shared cache (30MB, 12-way in paper) among 40 cores; final on-chip stop before DRAM; where instruction victims occur due to multi-core contention.
Ifetch stallFrontend pipeline stall waiting for instruction fetch from LLC/DRAM; delays all subsequent execution, including cached data accesses.
Reuse distanceNumber of unique LLC accesses between consecutive accesses to the same line; longer distance = higher eviction risk; instructions have longer distances than data in servers (Fig. 3a).
MockingjayState-of-the-art LLC replacement policy approximating Belady's optimal by predicting reuse distance; achieves 90–99% hit-status similarity to the optimal hot set under modern LLC policies; Garibaldi integrates with it as baseline.
QBS (Query-Based Selection)Technique from inclusion victim avoidance (Jaleel 2010): query side table on eviction; if cost exceeds threshold, protect; caps queries at QBS_MAX_ATTEMPTS=2.
D_PPN table8K-entry decoupled storage holding data page frame numbers (D_PPN 19b + sctr 3b); shared by multiple DL_P.A. fields exploiting spatial locality, reducing pair table storage.
old_bit1-bit flag per DL_P.A. field; set on color update or instruction miss; cleared when data access matches; tracks first k data lines after instruction miss for conservative prefetch.
sctr (saturating counter)3-bit counter per DL_P.A. field: incremented on match (data hit), decremented on mismatch; if < threshold (e.g., 4), replace field with new DL_P.A.
I-SPYContext-driven instruction prefetcher (Khan 2020) using CFG + 80 recent basic blocks; baseline in paper; Garibaldi provides additional benefit beyond I-SPY's prefetch.
Weighted speedupPerformance metric: Σ (IPCshared / IPCsingle) across all cores; accounts for each program's single-core baseline; Garibaldi achieves 13.2% over LRU (vs Mockingjay's 6.1%).
Key results (40-core, 16 server workloads)
  • Speedup: Garibaldi+Mockingjay 13.2% over LRU (vs Mockingjay 6.1%, Hawkeye 1.9%, DRRIP 1.5%).
  • Best workload: verilator 65.2% speedup (vs Mockingjay 18.4%).
  • Ifetch stalls: 18% reduction (geomean, Fig. 13) — up to 42% in verilator.
  • Energy: 10.4% savings vs LRU (5.0% over Mockingjay); up to 42.3% in verilator.
  • Storage: 193.9KB Garibaldi module + 60KB I/D bit = 252KB (0.82% of 30MB LLC).
  • Latency: 1-cycle pair table lookup, 2-query max per eviction → negligible overhead.
  • Sensitivity: k=2 slightly higher (10.2%) than k=1 (10.1%); k=1 chosen as default for latency/storage trade-off; dynamic threshold outperforms fixed by 2.7pp; 16K entries best trade-off.
