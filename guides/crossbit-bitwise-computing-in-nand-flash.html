MICRO 2025 · A Field Guide

Talking across bitlines to unlock MLC and rich queries

Previous in-flash computing could only talk within one bitline—so it couldn't correct errors or merge data across bits. CrossBit fixes that with cheap dynamic circuits, enabling reliable MLC and efficient database operations.

📄 Kim et al., "CrossBit: Bitwise Computing in NAND Flash Memory with Inter-Bitline Data Communication" 🎓 Assumes storage and memory-tiering basics ⏱ ~30 min read
🔗

Give bitlines the ability to communicate — not just compute alone — and you can correct errors in MLC and run complex operations without moving data out.

Before you begin

Ten ideas the paper assumes you know

Each card stands alone. Read in order the first time, or jump to whatever looks unfamiliar.

intra-bitline (old IFP) inter-bitline (CrossBit) dynamic circuits error/unreliability
1NAND flash structure: planes, pages, bitlines▾

A NAND flash chip is organized into planes, each with a cell array and page buffer. A page (typically 16 KiB) is read or written at once, with each column of cells served by a bitline and its corresponding bitline buffer in the page buffer. Conventional operations sense data from the cell array into the page buffer, then transfer it out.

Now you're ready: CrossBit adds circuitry to the page buffer so bitline buffers can talk to each other—something that doesn't happen in conventional flash.
2In-flash processing (IFP) and its throughput advantage▾

In-flash processing means computing inside the NAND flash plane before transferring data out. Because the internal read bandwidth between cell array and page buffer far exceeds the channel bandwidth (the narrow pipe to the SSD controller), IFP can deliver throughputs much higher than external I/O—provided it can leverage massive parallelism across bitlines.

Now you're ready: CrossBit is an IFP architecture, so it inherits this throughput advantage—but only if its inter-bitline operations don't kill the parallelism.
3SLC vs MLC: capacity vs reliability tradeoff▾

Single-level cell (SLC) stores 1 bit per cell with two VTH levels; multi-level cell (MLC) stores 2 bits per cell with four levels. MLC doubles capacity but shrinks the voltage margin between levels, raising the bit error rate. Without error correction, MLC is too unreliable. But if you can correct errors inside the flash, you can use MLC for IFP and gain 1.8× bit density.

Now you're ready: CrossBit's IF-ECC (in-flash ECC) is what enables reliable MLC for IFP, differentiating it from all prior work that required SLC for reliable IFP.
4Why prior IFP couldn't do ECC: no inter-bitline ops▾

ECC algorithms (like Hamming code) XOR bits from different bit positions—but previous IFP architectures (ParaBit, Flash-Cosmos) could only operate on data from the same bitline (intra-bitline). To XOR bits from different columns, they'd have to read them out and write intermediate results back to cells, incurring huge latency and new errors. So MLC was off-limits.

Now you're ready: CrossBit adds inter-bitline communication inside the page buffer, so it can XOR across columns without cell writes, enabling cheap in-flash ECC.
5Dynamic circuits: holding charge instead of driving transistors▾

A dynamic circuit represents logic 1 by temporarily holding charge on a wire node (which eventually leaks), rather than continuously driving it with transistors like static circuits do. Dynamic circuits are 5× smaller in area than static equivalents but require periodic refresh. Modern page buffers already use dynamic circuits to save space, so CrossBit reuses the same technique for its inter-bitline connections.

Now you're ready: CrossBit's "dynamic circuit-based interconnect" (DC-interconnect) is what keeps the area overhead tiny (2.2% of die)—it's not a new idea, just reusing the page buffer's existing design.
6Data layout: row-major vs columnar, and transposition cost▾

In row-major layout, all bits of one word sit in the same row (page). In columnar layout (common in databases), each column is stored contiguously across pages. Prior IFP architectures that only supported intra-bitline operations needed data transposed (one word spread across bitlines), requiring multiple row accesses and cell writes. CrossBit works natively with columnar data because it can talk across bitlines.

Now you're ready: CrossBit avoids the transposition overhead that plagued earlier IFPs, making database queries much faster.
7Hierarchical structure: local groups + global modules▾

A page has ~128K bitlines. Connecting all of them to one shared node is impractical (huge wire capacitance). Instead, CrossBit bundles 32 bitlines into a local group with a small local module, then chains local groups together with even smaller global modules. This keeps parallelism high (4096 local groups per plane × 256 planes = 1M parallel groups) while keeping wires short.

Now you're ready: The hierarchical design is what lets CrossBit scale to full-page parallelism without ballooning area.
8Latches in a bitline buffer: L1 vs L2, and DC-interconnect▾

Each bitline buffer holds multiple latches (L1[0], L1[1], L1[2], L2) and a DC-interconnect (a dynamic-circuit-based wire connecting them). Conventional operations use Set/Reset signals to move data between latches. CrossBit repurposes these signals plus a shared "charge" node to implement logic operations (NOR) without adding gates—just by controlling which latch is selected to discharge the shared node.

Now you're ready: CrossBit's "computing latch scheme" is the intra-bitline foundation; the inter-bitline modules connect L2 latches across different buffers.
9Boolean-completeness: NOR is enough▾

A set of logic gates is Boolean-complete if you can build any logic function from them. NOR alone is Boolean-complete (you can construct AND/OR/NOT from NORs). CrossBit's two primitives (prim_OR and prim_AND) are built from init + d-NOR + store sequences, and together they can implement NOR—thus CrossBit is Boolean-complete.

Now you're ready: CrossBit can compute any logic function, not just hardwired operations—critical for general-purpose IFP.
10Hamming code: single-error correction with parity bits▾

Hamming code is a lightweight ECC that adds parity bits (P1, P2, ...) computed by XORing subsets of data bits. On read, recompute parity; if parity doesn't match, the syndrome (pattern of mismatches) uniquely identifies the erroneous bit, which you flip. CrossBit uses Hamming code for IF-ECC because it's cheap enough to run in parallel across all local groups.

Now you're ready: IF-ECC = three-phase Hamming code (error detection, error marking, error flip) implemented with CrossBit's inter-bitline XOR.
The whole story in one place

CrossBit in a nutshell

The limitation of prior IFP, the inter-bitline fix, and how it unlocks MLC and rich queries.

Previous in-flash processing architectures (ParaBit, Flash-Cosmos, Ares-Flash) could perform bulk bitwise operations on data sensed through the same bitline (intra-bitline), but lacked general inter-bitline communication. That restriction forced them to either write intermediate results back to cells (slow, induces new errors) or offload inter-bitline logic to the SSD controller (losing IFP's throughput advantage). As a result, they couldn't run ECC inside the flash and had to use SLC only, and complex database queries (pattern match, range, join) incurred massive overhead.

CrossBit adds inter-bitline operation capability by connecting L2 latches from different bitline buffers via a dynamic circuit-based local module (32 bitlines per group) and chaining local groups with small global modules. The entire design reuses the dynamic-circuit technique already present in modern page buffers, adding just 2.2% die area. With this, CrossBit implements in-flash error correction (IF-ECC) using Hamming code: three phases of inter-bitline XOR detect, mark, and flip erroneous bits, reducing MLC bit error rate by 1.1E+9× to 3.7E+10× and achieving 1.8× higher bit density than SLC-only IFPs.

For database workloads, CrossBit's inter-bitline capability avoids the transposition and cell-write overhead of prior IFPs. It processes pattern match (bitwise match → inter-bitline word merge) and range queries (4-phase numerical comparison with unidirectional OR across bitlines) efficiently, and supports join queries by broadcasting dimension-table keys across local groups. On fundamental database operations, CrossBit achieves 2.1× average speedup and 2.5× energy efficiency over the state-of-the-art IFP. For the full Star Schema Benchmark (SSB) with scale factor 1000, CrossBit delivers 1.7× geomean speedup over prior IFPs.

CrossBit remains compatible with Open NAND Flash Interface (ONFI), storing control signals in a 40 KiB SRAM within the flash die. It modifies the cache-read command to include a page-buffer computation phase (tRCBSY_CrossBit) before data transfer, enabling arbitrary logic—including IF-ECC, pattern match, range queries, and arithmetic—without changing the external protocol.

🍯 Key takeaways

  • Prior IFPs could only compute within one bitline; CrossBit adds inter-bitline communication via dynamic-circuit-based local and global modules.
  • IF-ECC (in-flash error correction) uses inter-bitline XOR for Hamming code, reducing MLC BER by 9–10 orders of magnitude and enabling 1.8× bit density gain.
  • CrossBit avoids cell writes for intermediate results, cutting overhead for pattern match, range, and join queries by 2–5×.
  • Boolean-complete via two primitives (prim_OR, prim_AND) that map to init + d-NOR + store sequences; optimization techniques (d-NOR-skip, subset-skip, parallel-prim) reduce steps further.
  • Hierarchical design (32-bitline local groups + global modules) keeps parallelism at 1M local groups (across 256 planes) while keeping area overhead at 2.2%.
  • Compatible with ONFI; control signals stored in 40 KiB SRAM; modified cache-read command for page-buffer computation phase.
  • On SSB (scale factor 1000), CrossBit: 1.7× geomean speedup; on pattern match/range/join, 2.1× avg speedup and 2.5× energy efficiency.
Six everyday pictures

How to think about CrossBit

Each picture stands alone—read them in any order. Every card ends with where the analogy stops being true.

The core idea · Intra- vs inter-bitline

Swimmers in lanes

A pool has four lanes (bitlines). Intra-bitline means each swimmer only talks to themself (computing on bits from one column). Inter-bitline means swimmers can pass a baton across lanes (merging bits from different columns). Previous IFPs had no cross-lane communication—so merging word-level results required getting out of the pool (transferring data out). CrossBit adds the cross-lane pass without slowing anyone down.

Where it breaks: Swimmers move; bitlines are fixed wires. The "baton pass" is actually a dynamic-circuit charge transfer through a shared node.
IF-ECC unlocks MLC · Error correction

The tightrope with a safety net

SLC is a wide beam—you rarely fall off. MLC is a tightrope with four positions—easy to slip. Prior IFPs had no safety net (no ECC inside flash), so they couldn't use the tightrope. CrossBit weaves a net by XORing parity bits across bitlines (Hamming code), catching single-bit errors. Now the tightrope is safe, so you can double your capacity (2 bits/cell instead of 1) with confidence.

Where it breaks: A safety net catches any fall; Hamming code only corrects one error per 32-bit word. Two errors = uncorrectable.
Hierarchical design · Local + global

City blocks and highways

32 houses (bitlines) share one neighborhood phone line (local module). Neighborhoods connect via highways (global modules). To call across the city, you first dial within your block (local inter-bitline), then the highway merges all blocks (global group AND/OR). This keeps wire lengths short and parallelism high—4096 neighborhoods operate at once, but you never need one giant cable connecting 128K houses.

Where it breaks: Phones are bidirectional; CrossBit's global modules are unidirectional (data flows sequentially through adjacent groups). Still, it's enough for AND/OR aggregation.
Dynamic circuits · Holding charge

A balloon vs a fan

A static circuit is a fan that continuously blows (transistors always driving). A dynamic circuit is a balloon inflated once—hold the knot (electrically isolated node) and it stays full for a while (until charge leaks). The balloon is 5× smaller than the fan and uses less power, but you must re-inflate it periodically. Modern page buffers already use balloons (dynamic logic) to save area; CrossBit just adds more balloons for inter-bitline connections.

Where it breaks: Balloons are visible; dynamic-circuit charge is invisible. And real circuits need careful timing—precharge before discharge, evaluate before store.
Database queries · Pattern match

Assembly line: inspect, then merge

A factory checks each part (bit position) of 32-bit widgets (words). Intra-bitline: each inspector marks their part pass/fail. Inter-bitline: a supervisor walks down the line and merges all 32 marks into one "good widget" verdict. Previous IFPs had no supervisor—they'd ship all 32 marks externally and let the CPU merge them, wasting bandwidth. CrossBit merges in-place, shipping only 1 bit per word (32× less data).

Where it breaks: Real assembly lines are serial; CrossBit's 4096 local groups operate in parallel, so it's 4096 supervisors working at once.
Range query · Numerical comparison

A thermometer: who's above the line?

You have three readings (stored data) and one threshold (range from host). Phase 1: eliminate identical digits (common 1s). Phase 2: unidirectional OR from MSB to LSB—like mercury rising until it hits the first difference. Phase 3: bitwise compare at that first difference. Phase 4: OR across all bits to get a final "above/below" answer. This 4-phase cascade is efficient because inter-bitline communication is cheap in CrossBit.

Where it breaks: A thermometer is continuous; CrossBit's comparison is discrete (bitwise). And "unidirectional OR" isn't a real thermometer operation—it's a digital trick.
The full explanation

How it actually works

Section by section, with the paper's key mechanisms rebuilt as clear visuals.

1 · The limitation of prior IFP: no inter-bitline ops

ParaBit and Flash-Cosmos perform bulk bitwise operations on data sensed through the same bitline (intra-bitline). To merge results from different bitlines, they must write intermediate data back to flash cells (latency = 200–500 µs per write, plus inducing new errors) or transfer data out to the SSD controller. Ares-Flash partially addresses this with a shift operation that moves data between adjacent bitlines in one direction (e.g., BL[n] → BL[n+1]) within a 32-BL window, but it's still unidirectional and limited in range. None of them can perform ECC (which requires XOR across arbitrary bit positions) efficiently inside flash.

🌱
Concretely: pattern match requires merging 32 bitwise-match results into one word-match verdict. ParaBit-LocFree ships all 32 bits externally (high bandwidth). Flash-Cosmos writes intermediate results to cells, then reads them back (high latency). CrossBit merges them in-place with inter-bitline AND, shipping 1 bit per word.

2 · CrossBit's inter-bitline solution: local + global modules

CrossBit introduces a hierarchical inter-bitline architecture. Each local group of 32 bitlines shares one local module—a dynamic-circuit-based DC-interconnect connecting the L2 latches of those 32 buffers. The local module reuses the same init + d-NOR + store operations used for intra-bitline computation (Section 4.2.1 of the paper), so it adds minimal circuitry: 2 transistors per bitline for the DC-interconnect, plus 2 logic gates for enable/output signals.

Adjacent local groups are chained by global modules—small tri-state inverter circuits (16 transistors each) that perform group AND or group OR. When GAND=1, the global module propagates a 0 if any upstream local group's result is 0 (AND). When GOR=1, it propagates a 1 if any upstream group's result is 1 (OR). This sequential cascade allows results from all 4096 local groups in a plane to be merged into one register in ~0.5 µs.

Why not one big shared node for all 128K bitlines? Wire capacitance would explode, and parallelism would collapse (only one operation per plane at a time). The hierarchy keeps wire lengths short and parallelism high.

3 · The computing latch scheme: init + d-NOR + store

CrossBit's intra-bitline and local inter-bitline operations both use the same three basic operations on the DC-interconnect:

  • init: Charge=0 precharges Shared_node to 1.
  • d-NOR: Select[X]=1 discharges Shared_node to 0 if Q[X]=1. Performing this across multiple latches implements NOR: Shared_node = NOR(Q[0] AND Select[0], ..., Q[N] AND Select[N]).
  • store: When Shared_node=1, Set=1 or Reset=1 overwrites Q to 1 or 0, respectively. When Shared_node=0, Set/Reset have no effect (discharge path disconnected).

From these, CrossBit defines two primitives: prim_OR (dst = dst + (src1 + ... + srcN)′) and prim_AND (dst = dst · (src1 + ... + srcN)), which together are Boolean-complete because they can synthesize NOR.

4 · In-flash error correction (IF-ECC): Hamming code

CrossBit applies Hamming code to each local group (32 bits) with 6 parity bits (P1–P6). The encoding is done at the SSD controller (so parity bits are stored alongside data). On read, CrossBit executes three phases inside the page buffer:

  1. Error detection: Recompute parity by inter-bitline XOR. E.g., P1 = XOR(P1, D1, D2, D4, D5). If P1–P6 are all 0, no error; otherwise, the syndrome (pattern of non-zero parity) uniquely identifies the erroneous bit.
  2. Error marking: For each data bit DN, compute mark bit MN = boolean expression of parity bits (e.g., M1 = P1 + P2 + P3′ + P4′). If MN=0, DN is erroneous.
  3. Error flip: Perform DN = MN · DN + MN′ · DN′ (intra-bitline). If MN=0, DN is flipped; otherwise DN is unchanged.

This three-phase cascade consumes 5.7 µs (783 computing steps after optimization, vs 2217 steps with AND/OR/NOT, 1485 with primitives). For context, a conventional ECC module in the SSD controller would require serially streaming 16 KiB through one pipeline (≈8 µs for read + ECC + write back), losing plane parallelism. CrossBit's IF-ECC runs in parallel across all 4096 local groups, so throughput scales with the number of planes (256 planes × 4096 groups = 1M parallel ECC operations).

🍯
The result: MLC with IF-ECC achieves BER of 10⁻¹⁶ at 1 month retention (2K P/E cycle), meeting JEDEC standard for enterprise SSDs. Without IF-ECC (EMP only), BER is ~10⁻⁷ after 1 day—unusable. BER reduction: 1.1E+9× to 3.7E+10×.

5 · Database queries: pattern match, range, join

Pattern match: Compare search data (S1, S2) with cell data (C1, C2) using 2-bit encoding (01=0, 10=1, 00=don't care). Intra-bitline computes Match_bit = S1·C2 + S2·C1 for each bit position (0=match). Inter-bitline computes Match_word = (Match_bit[0])′ · ... · (Match_bit[31])′ (1=full match). For "any match?" query, global module performs group OR across all local groups, storing 1-bit result in flag register (read via ONFI Read Status command 78h). This reduces data transfer from 16 KiB (full page) to 1 bit.

Range query: Four phases: (1) Eliminate common 1s: A1=A0·B0′, B1=A0′·B0. (2) Unidirectional OR from MSB to LSB: A2 accumulates first-1 from MSB. (3) Bitwise compare: C=A2·B2′ (for A>B?). (4) Inter-bitline OR across all C bits → final result. This 4-phase cascade avoids cell writes and exploits inter-bitline parallelism.

Join query: Adopt partitioned join [Balkesen et al., VLDB'13]. Broadcast dimension-table keys to all local groups by writing to shared control lines (prim_OR with N=0 → dst=1, or prim_AND with N=0 → dst=0). Then perform pattern match between fact-table data and broadcasted keys. CrossBit's inter-bitline capability makes broadcast cheap (no cell writes), so join overhead is dominated by pattern match, not data movement.

Follow the numbers, step by step

Worked examples

Four traces, each carried all the way through—from primitives to optimization to final speedup.

① Computing inter-bitline XOR with primitives

Phase 1 of IF-ECC requires XOR(A, B) across bitlines. How many computing steps?

  1. Conventional AND/OR/NOT: XOR = A′·B + A·B′ = (A′+B′)·(A+B). Implement as: NOT(A)=6 steps, NOT(B)=6 steps, OR(A′,B′)=6 steps, OR(A,B)=6 steps, AND=6 steps. Total: 18 steps (paper §7.1).
  2. With primitives: (A+B′)·(A′+B) = (0+A′)·(A+B′)·(0+B′)·(A′+B). Step 1: prim_AND (N=0) → dst=0. Step 2–5: four prims (A′, A+B′, B′, A′+B). Total: 1 + 4×3 = 12 steps (paper states primitives reduce basic XOR to 12 steps).
  3. With optimization: d-NOR-skip on step 1 (1 step), subset-skip on steps 2–5 (2 steps each). Total: 1 + 4×2 = 8 steps (paper §7.1 explicitly: "8 steps after optimization").
🍯
Insight: Primitive-based programming + optimization reduces XOR from 18 steps to 8 steps—2.25× improvement. This compounds across IF-ECC's 32-bit word (hundreds of XORs), making IF-ECC feasible.

② IF-ECC for a 32-bit word: parity computation

A local group has 26 data bits (D1–D26) and 6 parity bits (P1–P6). Phase 1 recomputes parity. How many XORs?

  1. Hamming code parity sets: Each parity bit covers a subset of data bits. E.g., P1 = XOR(D1, D2, D4, D5, D7, D9, D11, D13, D15, D17, D19, D21, D23, D25) (14 bits). Similarly, P2 covers 13 bits, P3 covers 13 bits, etc. Total: 6 parity bits.
  2. XOR tree: Recompute P1 = XOR(P1, D1, D2, ...) by XORing 14+1=15 bits. Use binary tree: 8 XOR2s + 4 XOR2s + 2 XOR2s + 1 XOR2 = 15 XORs. For 6 parity bits: ~90 XORs.
  3. With primitives + optimization: Each XOR2 = 8 steps (from example ①). But many XORs share operands, so subset-skip and parallel-prim apply. Paper reports phase 1 = 2217 steps (AND/OR/NOT) → 1485 steps (primitives) → 783 steps (optimized).
💡
Why not BCH or LDPC? Stronger codes require iterative decoding (hundreds of cycles per bit), and their parity-check matrices are sparse—so parallelizing across 4096 local groups would require enormous control logic. Hamming code is weak (1 error/32 bits) but cheap enough to run everywhere in parallel.

③ Pattern match: data-transfer reduction

A page has 4096 32-bit words. ParaBit-LocFree and CrossBit each search for one word. How much data is transferred out?

  1. ParaBit-LocFree (intra-bitline only): Computes bitwise match for each bit position (32 bits), but cannot merge across bitlines. Must transfer all 32 bits × 4096 words = 131,072 bits = 16 KiB to the controller for merging.
  2. CrossBit local inter-bitline word merge: Performs Match_word per local group (1 bit per 32-bit word), then global group OR (any-match flag). Transfer: 1 bit (flag only) for "any match?" query, or 4096 bits = 512 bytes (one bit per word) for "which words?" query.
🍯
Insight: CrossBit reduces data transfer by 32× (one bit per 32-bit word) for pattern match—this is why speedup is 4.2× over ParaBit-LocFree for pattern match (SLC). With MLC, CrossBit's IF-ECC overhead is hidden by cell-read latency, so speedup is even higher.

④ Bit density gain: SLC vs MLC with IF-ECC

A plane has 1024 blocks × 1024 pages/block × 16 KiB/page. SLC stores 1 bit/cell; MLC stores 2 bits/cell. CrossBit adds 6 parity bits per 26 data bits. What's the usable capacity?

  1. SLC: 1024 × 1024 × 16 KiB = 16 GiB per plane. No parity overhead for IFP (though the paper notes prior IFPs could only use SLC and still had limited reliability).
  2. MLC physical capacity: 2 bits/cell × 16 GiB = 32 GiB. With IF-ECC: 26 data bits per 32-bit codeword (26 data + 6 parity) = 26/32 = 81.25% coding rate. Usable capacity = 32 GiB × 0.8125 = 26 GiB per plane.
  3. Bit density gain: 26 GiB ÷ 16 GiB = 1.625×. Paper reports 1.8× (Fig. 13a)—the difference likely comes from slight adjustments in coding rate or metadata overhead not detailed in the text.
🍯
Insight: Even with IF-ECC parity overhead, MLC delivers 1.8× higher density than SLC. For data-intensive workloads (where capacity is the bottleneck), this is a major advantage.
Test yourself

Practice problems

Try each before opening the answer. Every one ends with the common misconception it's checking for.

Problem 1 · easy
ParaBit and Flash-Cosmos support only intra-bitline operations. Ares-Flash adds shift operations between adjacent bitlines. Why doesn't Ares-Flash's inter-bitline capability enable IF-ECC as efficiently as CrossBit?
Show answer
Ares-Flash's shift is unidirectional (e.g., BL[n] → BL[n+1]) and limited to adjacent bitlines. Hamming code requires XOR across arbitrary subsets of bits (e.g., P1 = XOR(D1, D2, D4, D5, ...)), which are not adjacent. To implement XOR with shift, Ares-Flash would need multiple shifts + cell writes to accumulate results—much slower than CrossBit's direct inter-bitline XOR via local modules. Paper shows: IF-ECC on Ares-Flash is 71× slower than CrossBit (Fig. 13b).Misconception: that any inter-bitline operation is sufficient. The flexibility of communication (arbitrary subsets, bidirectional) matters for ECC.
Problem 2 · medium
CrossBit's local module uses dynamic circuits (charge-holding nodes). Dynamic circuits leak charge over time. Why doesn't this cause errors in IF-ECC or database queries?
Show answer
CrossBit fixes one basic operation time to 7.3 ns (the max across all ops, validated by Monte Carlo simulation). Within this time window, charge leakage is negligible (HSPICE validation at 150nm CMOS). Each operation (init + d-NOR + store) completes in 3×7.3 = 21.9 ns, well within the safe margin. For longer sequences (e.g., IF-ECC's 783 steps = 5.7 µs), the system operates within validated timing margins established by the Monte Carlo analysis.Misconception: that dynamic circuits are inherently unreliable. They're fine for short-duration operations (nanoseconds to microseconds) with proper timing.
Problem 3 · medium
CrossBit bundles 32 bitlines per local group. Why not 64 or 128 to reduce the number of global modules?
Show answer
Larger local groups → longer DC-interconnect wire → higher capacitance → slower charge/discharge → longer operation time. Paper chose 32 BLs as a balance between parallelism (4096 groups/plane) and area (short wires). Also, Hamming code with 64 data bits would require 7 parity bits (coding rate 64/71 = 90.1%), vs 6 parity bits for 32 data bits (coding rate 26/32 = 81.25%)—the latter is easier to manage with simple XOR trees. The paper doesn't explicitly explore 64-BL groups, but their design choice (32 BLs, 6 parity) is standard in embedded Hamming implementations.Misconception: that bigger groups are always better. Wire capacitance and ECC complexity impose practical limits.
Problem 4 · hard
CrossBit's IF-ECC can correct 1 error per 32-bit word. Two errors in one word are uncorrectable. For a page with 4096 words, what's the expected number of uncorrectable errors if raw BER = 10⁻⁴?
Show answer
Probability of ≥2 errors in a 32-bit word: P(≥2) ≈ (32 choose 2) × (10⁻⁴)² = 496 × 10⁻⁸ = 4.96×10⁻⁶. Expected uncorrectable words per page: 4096 × 4.96×10⁻⁶ ≈ 0.02 words. So uncorrectable errors are rare even at high raw BER. However, if raw BER = 10⁻², then P(≥2) ≈ 496 × 10⁻⁴ = 0.0496, and expected uncorrectable = 4096 × 0.0496 ≈ 203 words—unacceptable. This is why the paper emphasizes that IF-ECC works only when enhanced MLC programming (EMP) keeps raw BER low enough (Fig. 12 shows BER after IF-ECC meets JEDEC standard only when raw BER is already ~10⁻⁷).Misconception: that ECC magically fixes any BER. Hamming code (and all bounded-distance codes) have hard limits—if raw BER is too high, you need stronger ECC (BCH, LDPC) or better cell programming.
Problem 5 · medium
CrossBit's global modules perform group AND or group OR sequentially across 4096 local groups. Why not a tree structure (log₂(4096) = 12 levels) to be faster?
Show answer
The paper notes that a tree structure could improve merging latency if area overhead is acceptable. CrossBit chose a sequential design to minimize area (each global module is 16 transistors). The sequential cascade takes ~0.5 µs for group operations, which is acceptable since global operations are rare (e.g., pattern match "any-match" flag is checked infrequently) and the bottleneck is typically cell-read latency (20–25 µs). The paper explicitly mentions the tree option as a potential future optimization but prioritizes area efficiency in the presented design.Misconception: that faster is always better. The paper prioritized area efficiency (2.2% overhead) over absolute latency for rarely-used operations.
Problem 6 · easy
CrossBit stores control signals in a 40 KiB SRAM within each flash die. IF-ECC consumes 8.5 KiB, pattern match 0.2 KiB, range 2.7 KiB, multiplication 16.8 KiB, addition 5.0 KiB (total 33.2 KiB). What happens if an application requires more than 40 KiB?
Show answer
Two options: (1) Stream control signals from the SSD controller DRAM during execution. The paper notes that for each basic op (7.3 ns), control signal transfer (5 ns via channel) can be hidden, so performance is unaffected. Energy increases by 12% on average (streaming overhead), which is acceptable. (2) Compress control signals using sparsity (e.g., if fewer than half of the 32 bitlines in a local group are active, omit their control signals and send a bitmask)—the paper reports 1.46× compression for IF-ECC. Either way, the 40 KiB SRAM is not a hard limit, just a practical default for the evaluated workloads.Misconception: that the SRAM size is a fundamental constraint. It's a design choice—real systems could use larger SRAM or streaming.
Problem 7 · hard
CrossBit's hierarchical design (32-BL local groups + sequential global modules) achieves 1M parallel operations across 256 planes. If each plane has 4096 local groups, why isn't the total parallelism 256×4096 = 1,048,576 independent operations?
Show answer
Within each plane, the 4096 local groups share control lines—i.e., bitline buffers at the same index in different local groups receive the same control signals (Select[0], Set[0], etc.). This is standard in NAND flash to reduce wiring overhead. So all 4096 groups perform the same operation in lockstep (SIMD parallelism). Across 256 planes, each plane can run a different operation (MIMD parallelism if the SSD controller issues different commands to each plane). In practice, for bulk operations (e.g., IF-ECC on a full SSD), all planes run the same operation, so the effective parallelism is 256×4096 = 1M data elements processed simultaneously, but only 256 independent programs. For database queries (e.g., pattern match on different tables), each plane can search a different table, giving true MIMD parallelism.Misconception: that SIMD and MIMD parallelism are the same. CrossBit has SIMD within each plane (all local groups do the same op) and MIMD across planes (each plane can do a different op).
Problem 8 · easy
CrossBit modifies the ONFI cache-read command to include a computation phase (tRCBSY_CrossBit). Does this break compatibility with existing SSDs?
Show answer
No, because ONFI defines vendor-specific opcodes (33h, 34h in the paper) for custom commands. CrossBit uses opcode 36h for the modified cache-read command. Existing SSD controllers that don't recognize 36h would treat it as an invalid command and return an error—but a CrossBit-aware controller (with updated firmware) would issue 36h to trigger IFP. The data interface (WE#, RE#, CE# signals) remains unchanged, so the physical protocol is compatible. However, CrossBit still requires modified NAND flash die circuitry (local modules, global modules, SRAM) and controller support, so it's not truly "drop-in" without hardware changes to the flash die itself.Misconception: that ONFI compatibility means no hardware changes. Vendor-specific opcodes maintain interface compatibility, but the flash die itself must be modified.
Quick answers

Frequently asked questions

Ten questions newcomers ask most. Tap any to expand.

Q1What does "inter-bitline" actually mean?▾
It means data from different bitlines (columns) can communicate. Concretely: bitline buffer #5's L2 latch can XOR with bitline buffer #18's L2 latch, without reading data out to the SSD controller. Prior IFPs could only operate on data from the same bitline (intra-bitline).
Q2Why can't prior IFPs just add an XOR gate between bitlines?▾
Adding static logic gates (AND, OR, XOR) across 128K bitlines would explode area and power. Commercial in-memory computing (HBM-PIM, GDDR6-AiM) uses this approach and incurs 10–20% area overhead for processing-in-memory logic. CrossBit avoids this by reusing the page buffer's existing dynamic circuits (DC-interconnect), adding only 2.2% area.
Q3How does IF-ECC compare to the SSD controller's ECC module?▾
Controller ECC (BCH, LDPC) is stronger (corrects 10–100 errors per 1 KiB) but serial (streams data through one pipeline, ~8 µs per page). IF-ECC is weaker (1 error per 32 bits) but parallel (runs across all 4096 local groups simultaneously, 5.7 µs per page). For IFP, IF-ECC is better because it doesn't lose plane parallelism and doesn't require data transfer out. For conventional memory operations, controller ECC is better (higher coding rate).
Q4Does CrossBit replace the SSD controller's ECC, or supplement it?▾
Supplement. CrossBit proposes separating the physical address space: IFP data uses MLC with IF-ECC (Hamming code inside flash), while conventional data uses MLC with controller ECC (BCH/LDPC outside flash). This is necessary because IF-ECC has higher parity overhead (18.75%) than controller ECC (~7%), so mixing them would waste capacity.
Q5What's the tradeoff between local group size (32 BLs) and performance?▾
Larger groups → fewer groups → less parallelism, but also longer wires → higher capacitance → slower ops. The paper chose 32 BLs empirically (no explicit exploration of alternatives). For Hamming code, 32 data bits + 6 parity = 38 bits per group is standard. Future work could explore adaptive group sizing (e.g., 64 BLs for addition/multiplication, 32 BLs for ECC).
Q6Why is CrossBit faster than Ares-Flash for pattern match (MLC) but slower for multiplication (SLC)?▾
For pattern match (MLC), CrossBit's IF-ECC overhead (5.7 µs) is hidden by cell-read latency (25 µs for MLC), so CrossBit's inter-bitline merge wins (3.9× speedup). For multiplication (SLC), Ares-Flash's shift-all-at-once is faster than CrossBit's sequential shift (1.4× slowdown). But with MLC, Ares-Flash's ECC overhead (using a separate controller module) dominates, so CrossBit wins (2.7× speedup). Trade: CrossBit is better for general-purpose (ECC + database), Ares-Flash is better for arithmetic-only (SLC).
Q7Can CrossBit run on existing NAND flash chips, or does it require new hardware?▾
New hardware. CrossBit requires modifying the page buffer (adding local modules) and control logic (adding global modules + 40 KiB SRAM). But the modifications are minimal (2.2% die area), and the interface (ONFI) remains compatible—so a CrossBit-enabled flash die could work with existing SSD interfaces (with firmware update to issue opcode 36h). However, the flash die itself still needs the added circuitry.
Q8Does CrossBit's hierarchical design (local + global) impose a bottleneck?▾
Only for global operations (group AND/OR across all 4096 local groups). The sequential cascade takes ~0.5 µs, vs local operations (~tens of nanoseconds). But global operations are rare (e.g., pattern match "any-match" flag is checked once per page, not per word). For most workloads, the bottleneck is cell-read latency (20–25 µs), so global overhead is negligible.
Q9Why didn't the paper evaluate TLC or QLC with IF-ECC?▾
TLC (3 bits/cell) and QLC (4 bits/cell) have even smaller VTH margins than MLC, so raw BER is higher. Hamming code (1 error per 32 bits) is insufficient for TLC/QLC—they need stronger ECC (e.g., BCH with t=8–16 errors per 1 KiB). But BCH is too complex to implement in parallel across 4096 local groups (it requires Galois field arithmetic, which needs large lookup tables). The paper does not explicitly flag TLC/QLC support as future work.
Q10What did the authors leave for future work?▾
The paper explicitly mentions: (1) developing a logic optimizer to compile high-level operations into primitives (currently manual), and (2) exploring better control-signal compression (beyond the 1.46× achieved for IF-ECC). The tree-based global module architecture is mentioned as a potential optimization if area overhead is acceptable. Other extensions (TLC/QLC, adaptive group sizing, TSV, CXL, NVMe-oF) are not flagged as future work in the paper.

Glossary & further reading

Every term in one place, then a path to go deeper.

Core terms
In-flash processing (IFP)Computing inside NAND flash planes (before data transfer to SSD controller), leveraging internal bandwidth and plane parallelism.
Intra-bitline operationComputation on bits from the same bitline (column). Supported by ParaBit, Flash-Cosmos, Ares-Flash—but insufficient for ECC or complex queries.
Inter-bitline operationComputation on bits from different bitlines. CrossBit's key innovation—enables XOR across columns for ECC and word-level merging for database queries.
Local moduleDynamic-circuit-based DC-interconnect connecting L2 latches of 32 bitlines. Performs init + d-NOR + store for inter-bitline ops within one local group.
Global moduleTri-state inverter (16 transistors) chaining adjacent local groups. Performs group AND or group OR across all 4096 groups in a plane.
IF-ECC (In-Flash ECC)Hamming code (6 parity bits per 32 data bits) implemented via inter-bitline XOR. Reduces MLC BER by 9–10 orders of magnitude, enabling 1.8× bit density gain.
Computing latch schemeRepurposing page buffer's latches (L1, L2) + DC-interconnect for logic operations via init + d-NOR + store sequences. Area-free for intra-bitline ops.
Primitives (prim_OR, prim_AND)Two circuitry-aware building blocks for Boolean logic: prim_OR = dst + (src1 + ... + srcN)′, prim_AND = dst · (src1 + ... + srcN). Together Boolean-complete.
Supporting terms
Bitline bufferPage-buffer circuit serving one bitline: holds latches (L1[0:2], L2) + DC-interconnect for data movement. CrossBit connects L2 across buffers for inter-bitline ops.
Dynamic circuitLogic circuit that holds charge on an isolated node (1) rather than continuously driving with transistors. 5× smaller area than static circuits, but leaks charge over time.
DC-interconnectDynamic circuit-based wire in page buffer connecting latches within one bitline buffer (conventional) or across buffers (CrossBit's local module).
SLC / MLC / TLC / QLCSingle- / Multi- / Triple- / Quad-level cell: store 1 / 2 / 3 / 4 bits per cell. Higher levels = higher capacity but lower reliability (smaller VTH margins).
Enhanced MLC Programming (EMP)Tighter ISPP voltage step (vs conventional MLC programming) to widen VTH margins. Alone insufficient for IFP—must combine with IF-ECC (paper Fig. 8a).
Hamming codeLinear block code that adds parity bits to detect and correct single-bit errors. CrossBit uses (38,32) Hamming: 32 data + 6 parity = 38 bits, corrects 1 error.
Pattern match querySearch for words matching input pattern. CrossBit: intra-bitline bitwise match → local inter-bitline word merge → global group OR (any-match flag). 32× data reduction.
Range queryCheck if data falls within [a, b]. CrossBit: 4-phase cascade (eliminate common 1s, unidirectional OR, bitwise compare, inter-bitline OR). Avoids cell writes.
Join queryMatch fact-table records with dimension-table keys. CrossBit: broadcast keys via shared control lines, then pattern match. Cheap broadcast = key advantage.
ONFI (Open NAND Flash Interface)Standard protocol for SSD controller ↔ NAND flash. CrossBit uses vendor-specific opcodes (33h, 34h, 36h) for IFP commands—compatible with existing interface.
d-NOR-skip / subset-skip / parallel-primOptimization techniques to reduce computing steps: skip d-NOR when N=0 (3→1 steps), reuse d-NOR when selection is superset (3→2 steps), run independent prims in parallel.

Related papers

ParaBit: Processing Parallel Bitwise Operations in NAND Flash Memory medium

Gao et al. · MICRO · 2021

First IFP architecture (to authors' knowledge). Applies special read voltages for in-cell AND/OR; ParaBit-LocFree variant uses page buffer for intra-bitline ops. CrossBit's baseline—no inter-bitline capability.

Flash-Cosmos: In-Flash Bulk Bitwise Operations Using Inherent Computation Capability medium

Park et al. · MICRO · 2022

IFP using parallel connection of cells inside array for bulk AND/OR. Demonstrated reliable IFP in SLC with Enhanced SLC Programming (ESP). CrossBit extends this approach to MLC with EMP + IF-ECC for higher density.

Ares-Flash: Efficient Parallel Integer Arithmetic Operations Using NAND Flash Memory medium

Chen et al. · MICRO · 2024

IFP with unidirectional shift (BL[n]→BL[n+1]) for addition/multiplication. Limited inter-bitline capability—CrossBit generalizes to arbitrary inter-bitline XOR/AND/OR. Paper shows: Ares-Flash + CrossBit hybrid achieves best arithmetic + ECC performance.

MQSim: A Framework for Enabling Realistic Studies of Modern Multi-Queue SSD Devices hard

Tavakkol et al. · FAST · 2018

SSD simulator used for CrossBit evaluation. Models plane-level parallelism, channel bandwidth, and command scheduling. Critical for understanding IFP's throughput advantage vs in-storage processing (ISP).

The Working Set Model for Program Behavior medium

Denning · CACM · 1968

Classic notion of "working set" (data actively used). Relevant for understanding why database queries (pattern match, range, join) are data-intensive—they scan large datasets that don't fit in DRAM, making IFP's capacity advantage critical.
