ASPLOS 2025 · A Field Guide

Clone processes across a cluster instantly, share state automatically

CXL shared memory lets you checkpoint once, restore everywhere — close to zero-serialization and zero-copy, just attach and run. This is the plain-English story of CXLfork, a remote fork primitive that dramatically reduces cold starts.

📄 "CXLfork: Fast Remote Fork over CXL Fabrics" · Alverti et al. 🎓 Some CXL & serverless basics assumed ⏱ ~25 min read

1What CXL shared memory means for cluster computing▾

Traditional clusters use RDMA (e.g., Infiniband) to move data node-to-node: fast, but still a copy over the network. CXL 3.0+ provides byte-addressable, cache-coherent shared memory across compute nodes — like a giant NUMA domain spanning the rack. A load instruction on Node A can fetch a cache line from CXL memory as if it were local (albeit slower: ~400 ns vs ~100 ns for DRAM). This obviates serialization and bulk transfers for state sharing.

Now you're ready: CXLfork checkpoints process state directly into CXL memory as native data structures, then remote nodes attach those structures without copying — shared read-only state is accessed on-demand via cache-line fetches.
2Serverless cold starts and why they hurt▾

In FaaS (e.g., AWS Lambda), functions are spawned on-demand to handle requests. A cold start = booting a new function instance from scratch: loading runtime, libraries, model weights — often 250–500 ms, plus ~130 ms for container setup. The mitigation is to keep idle functions "warm" (cached in memory) for a keep-alive window (minutes), at the cost of memory pressure. Load spikes necessitate spinning up many instances across nodes — expensive with traditional cold starts.

Now you're ready: CXLfork checkpoints a warmed-up function to CXL once, then instantly clones it across nodes on demand — cutting cold-start end-to-end execution by 2.26× on average over CRIU, while reclaiming 87% of local memory via shared state.
3Why existing remote fork (CRIU, Mitosis) is slow▾

CRIU serializes the entire process state (page tables, VMAs, registers, data pages) to files using Protocol Buffers, then deserializes on the target node. Even stored in CXL memory, de-serialization is costly. Mitosis (for RDMA) avoids serializing data by copying pages lazily via remote paging on faults, but still serializes OS metadata and copies every accessed page to local memory — penalizing functions with large working sets (e.g., BERT: 54% overhead just copying pages). Both also suffer from coupling or serialization issues detailed in §2.3.

Now you're ready: CXLfork skips serialization for almost everything (only lightweight global state like file paths is encoded), and skips copying read-only data — it stays in CXL, shared across all clones. CoW handles writes, and tiering policies control locality when needed.
4Private vs. global process state▾

Private state is wholly owned by one process: its memory pages, page tables, VMA tree, CPU registers. No external pointers — can be copied standalone. Global state is shared OS structures: open files (inodes), sockets, namespaces, mount points. These link to the file system layer and can't just be copied; they need explicit detach/attach to keep metadata consistent across OS instances.

Now you're ready: CXLfork copies private state as-is to CXL (then rebases internal pointers so other OSes can remap them). For global state, it serializes minimal info (e.g., file paths) and re-does operations (e.g., re-opens files) on restore to re-link the remote OS.
5The working-set split of serverless functions▾

The paper profiles 10 FaaS functions (Figure 1): on average, 72.2% is Init data (rarely accessed after startup), 23% is Read-only during execution, and only 4.8% is Read/Write. Init + Read-only dominate. This means that, after a function is initialized, most of its footprint can be shared read-only across sibling instances — and small working sets often fit in CPU caches, hiding CXL latency.

Now you're ready: CXLfork leverages this split by keeping Init and Read-only state in CXL (shared cluster-wide), and only copying R/W pages to local memory on writes via CoW. Result: 87% memory savings on average over cold starts.
6CXL latency and when it matters▾

CXL memory access is slower than local DRAM: in the paper's setup, round-trip is 391 ns (vs ~100 ns local). If a function's working set doesn't fit in the hardware caches and accesses spill to CXL frequently, execution slows down. However, many FaaS functions have small working sets (the paper shows most fit in caches). The trade-off: using CXL to store read-only state saves local memory but may penalize runtime if the working set is large (e.g., BFS, BERT).

Now you're ready: CXLfork exposes tiering policies (migrate-on-write, migrate-on-access, hybrid) to control which pages stay in CXL vs. copy to local memory, balancing memory savings against performance. The default (migrate-on-write) keeps most state in CXL and only copies on stores.
The whole story in one place

CXLfork in a nutshell

The problem, the CXL-enabled fix, the tiering policies, and the serverless autoscaler that uses it.

Existing remote fork (CRIU, Mitosis) is designed for disconnected memory: they serialize, transfer, and deserialize process state across nodes. CRIU checkpoints to files (huge overhead); Mitosis lazily copies pages over RDMA but still serializes OS metadata and copies all accessed data locally. On CXL, both miss the chance to checkpoint once and share state directly via the fabric.

CXLfork leverages byte-addressable shared CXL memory to checkpoint process state (data, page tables, VMA tree, registers) as native structures to CXL, rebasing internal pointers so any node can remap them. On restore, the remote node attaches the checkpointed page tables and VMAs in constant time (allocates upper levels, attaches CXL-checkpointed leaves), maps the data as read-only, and resumes. Copy-on-Write handles stores, copying pages to local memory on fault. Read-only state remains in CXL, automatically shared by all clones cluster-wide.

CXLfork distinguishes private state (copied as-is, then rebased) from global state (lightly serialized, e.g., file paths, then re-instantiated by redoing operations like re-opening files on restore). It also tracks access patterns via Access/Dirty bits in checkpointed PTEs and offers three tiering policies: migrate-on-write (default: keep read-only in CXL), migrate-on-access (copy all touched pages to local), and hybrid (copy only "hot" pages marked by A-bit). A user-space controller or the runtime can dynamically choose policies based on memory pressure and SLOs.

The paper builds CXLporter, a horizontal autoscaler for serverless functions. It maintains a pool of ghost containers (empty, 512 KB each) and uses CXLfork to instantly clone functions into them on demand. It dynamically adjusts tiering and keep-alive windows to balance latency vs. memory.

The payoff (vs. LocalFork baseline): end-to-end cold-start execution only 14% slower on average, but remote. Vs. CRIU-CXL: 2.26× faster execution, 87% less local memory. Vs. Mitosis-CXL: 1.40× faster, 61% less memory. On a memory-constrained cluster (25% mem), CXLfork cuts P99 latency by ~16× over both by enabling more concurrent instances.

🍯 Key takeaways

  • CXL shared memory enables checkpoint once, restore everywhere without serialization or bulk transfers.
  • Private state (data, page tables, VMAs) → copied to CXL as-is, rebased; global state → lightly serialized (file paths), re-instantiated on restore.
  • Restore = attach checkpointed page tables/VMAs in O(1), map data read-only, resume. CoW migrates writes to local memory.
  • Read-only state stays in CXL, shared cluster-wide — 72% of footprint (Init) + 23% (Read-only) = 95% shareable on average.
  • Tiering policies (migrate-on-write, migrate-on-access, hybrid) balance memory savings vs. CXL access latency.
  • CXLporter uses ghost containers (512 KB idle overhead) + CXLfork to scale functions instantly, dynamically shortening keep-alive to reclaim memory under pressure.
  • Result: 2.26× faster than CRIU, 87% memory savings; 1.40× faster than Mitosis, 61% savings; near-local-fork performance (14% slower) but remote.
Five everyday pictures

How to think about CXLfork

Each picture stands alone. Every card ends with an honest note about where the picture stops being true.

The core idea · Checkpoint once, restore many

A shared blueprint

Instead of photocopying a blueprint for each construction crew (CRIU) or letting them radio for pages of the plan one-by-one (Mitosis), you pin one master blueprint to a central board (CXL). All crews read from the same source. If a crew marks it up, they trace that page to their own clipboard (CoW). Most of the plan stays pristine and shared — faster setup, no wasted paper.

Where it breaks: real blueprints don't have latency tiers; reading from the central board is slightly slower than having a local copy. CXLfork's tiering policies decide what to cache locally vs. keep shared.
Constant-time restore · Attach, don't copy

LEGO brick slots

Imagine page table trees as LEGO towers. CRIU rebuilds the entire tower brick-by-brick on the new node (slow). Mitosis copies the instructions and builds lazily. CXLfork allocates just the top few layers locally, then snaps on the pre-built lower sections from CXL — instant assembly, same result. The CXL bricks stay shared; only writes force a local copy.

Where it breaks: real LEGO bricks don't have access latency. CXLfork's attached page table leaves point to CXL pages, so load misses that reach CXL pay ~400 ns instead of ~100 ns. Tiering mitigates this by copying hot pages.
Memory split · Init + Read-only dominate

A mostly-read library

Most of a function's footprint is like a reference library: hundreds of books (Init data, libraries) that are opened once and rarely touched again, plus a reading room (Read-only runtime state). Only a tiny corner is a notepad (R/W working set). CXLfork keeps the library in CXL (shared across all readers) and only the notepads locally. This cuts local memory by 87% on average.

Where it breaks: a few workloads (BFS, BERT) have large working sets that thrash the caches, turning "reference lookups" into slow CXL trips. Hybrid tiering fetches those hot books to local memory selectively.
Global state · Lightweight serialization + redo

A shared file cabinet

Private state (your desk drawers) can be moved as-is. Global state (the company file cabinet) can't — it's linked to the building's file system. CXLfork saves just the drawer labels (file paths, permissions) and, on restore, walks to the new building's file cabinet and re-opens the same folders. Minimal serialization, correct linkage.

Where it breaks: this assumes the "building" (root filesystem, container image) is identical across nodes — a standard assumption in serverless (same Docker image). If filesystems differ, restores fail.
CXLporter · Ghost containers + dynamic tiering

A warehouse with empty shells

Traditional autoscaling provisions full containers (heavy crates) on-demand — slow. CXLporter keeps a stock of ghost containers (empty shells, 512 KB each). On a request, it clones the function into a ghost via CXLfork (<1 ms), using CXL-shared state. Under memory pressure, it shortens keep-alive from minutes to 10 seconds and switches tiering to maximize sharing. More functions fit, cold starts vanish.

Where it breaks: ghost containers are Linux-specific (namespaces, cgroups). The analogy also hides that CXLporter's tiering decisions depend on profiled access patterns — it's not static warehouse logistics.
The full explanation

How it actually works

Section by section, checkpoint → restore → tiering → CXLporter, with key diagrams from the paper.

1 · Checkpoint to CXL memory

CXLfork's checkpoint phase copies the parent process's state to CXL memory. It distinguishes private and global state:

  • Private state (task struct, memory descriptor, VMA tree, page tables, CPU registers, process memory pages, private file mappings like libraries) → copied to CXL as native structures using memory copies.
  • Global state (open files, sockets, namespaces, mount points) → lightly serialized (e.g., file paths, permissions) and stored in CXL.

After copying private state, CXLfork rebases internal pointers: it walks the checkpointed OS structures (e.g., page table entries) and updates them to index the CXL physical address space instead of local memory. For example, a PTE that mapped virtual page V to local physical page F now maps V to CXL page X (where F's replica resides). CXLfork also preserves Access/Dirty bits in PTEs to track the parent's access pattern. The checkpointed page table is marked read-only. Crucially, the checkpoint is now decoupled from the parent node's OS — any other node can remap and use it.

🌱
Why rebase? Without rebasing, pointers in the checkpointed structures would index the parent node's local memory, making them useless to other nodes. Rebasing converts them to CXL-relative offsets that any node can interpret.

2 · Restore by attaching, not copying

CXLfork's restore phase creates a new process on the target node and attaches the CXL-checkpointed state without copying it to local memory. The steps:

  1. Allocate upper page table levels. CXLfork creates only the top few levels (PGD, PUD, PMD) of the child's page table tree in local memory.
  2. Attach CXL page table leaves. It directly links the local upper levels to the CXL-checkpointed last-level PTEs (the leaves). These PTEs already map CXL physical pages and are marked read-only. This is O(1) in the number of leaves — just pointer updates, no copies.
  3. Attach VMA tree leaves. Similarly, CXLfork allocates upper levels of the VMA tree locally and attaches CXL-checkpointed VMA leaves, which describe private file mappings (e.g., libraries).
  4. Restore global state. CXLfork deserializes the lightweight global state (file paths, etc.) and redoes operations to re-link the child to the local OS — e.g., re-opens files, re-attaches to namespaces.
  5. Restore registers and resume. The child's CPU state is loaded from the checkpoint, and execution resumes.

The child's virtual memory now maps to CXL physical pages, all marked read-only. Load instructions that miss in the CPU caches fetch data directly from CXL (no fault, no copy). Store instructions trigger CoW faults that copy the target page to local memory and update the local PTE. The checkpointed CXL page remains pristine and shared.

🍯
Restore latency is 1.2–6.1 ms (vs 16–423 ms for CRIU, up to 15 ms for Mitosis), and the child shares read-only state cluster-wide without any explicit coordination.

3 · Tiering policies: controlling CXL vs. local memory

By default, CXLfork keeps read-only data in CXL (migrate-on-write). But for functions with large working sets, repeated CXL accesses can hurt performance. CXLfork exposes three policies:

PolicyWhen to copy to localTrade-off
Migrate-on-writeOnly on stores (CoW faults)Max memory savings, may penalize loads if working set >> caches
Migrate-on-accessOn first access (read or write)Faster execution, but 250% more local memory (no sharing)
Hybrid tieringOn access, but only if A-bit set in checkpointed PTEMiddle ground: copy hot pages, share cold pages

CXLfork tracks access patterns via the Access (A) and Dirty (D) bits in the CXL-checkpointed PTEs. Hardware updates A-bits when a restored process accesses checkpointed pages (page-table walks write to CXL PTEs). User-space controllers can periodically reset A-bits to re-profile working sets, similar to MGLRU. Alternatively, profilers can explicitly mark hot pages by setting an unused PTE bit in the checkpoint.

💡
Why track A/D bits? The A-bit in the parent's page table (captured at checkpoint time) reflects the parent's access pattern. For serverless, functions are profiled after warmup (16th invocation), so A-bits capture steady-state hot pages. Hybrid tiering uses this to pre-fetch hot pages to local memory on access.

4 · CXLporter: horizontal autoscaling for FaaS

CXLporter is the serverless runtime that uses CXLfork. It maintains:

  • Ghost containers: empty containers (512 KB each) waiting for restore requests. CXLporter clones functions into ghosts via CXLfork, eliminating the ~130 ms container-setup overhead.
  • Checkpoint object store: maps ⟨user, function⟩ → CID (checkpoint ID in CXL memory). Checkpoints are taken after the 16th invocation (post-JIT warmup), with A/D bits cleared after the 1st invocation to capture steady-state.
  • Dynamic tiering: by default, migrate-on-write. If tail latency approaches SLOs or avg latency degrades, CXLporter switches the function to hybrid tiering (copy hot pages). Under high memory pressure (>90%), no more functions are promoted to hybrid.
  • Dynamic keep-alive: when memory usage is high, CXLporter shortens keep-alive windows from minutes to 10 seconds, reclaiming memory faster while relying on CXLfork's fast restore to handle near-future requests.
🍯
On a memory-constrained cluster (25% of baseline), CXLporter with CXLfork cuts P99 latency by ~16× over CRIU/Mitosis by enabling 2× more concurrent function instances (thanks to 87% memory savings).
Follow the numbers, step by step

Worked examples

Four traces using the paper's real measurements — restore latency, memory savings, tiering trade-offs, and CXLporter under load.

① Restore latency breakdown: BERT function

BERT (630 MB footprint) is a large ML inference function. How long does it take to clone it to a remote node and run one inference under CRIU-CXL, Mitosis-CXL, and CXLfork?

  1. CRIU-CXL: restore = 423 ms (deserialize OS state + copy all data from CXL files). Execution = 137 ms. Total = 560 ms.
  2. Mitosis-CXL: restore = 15 ms (transfer/deserialize OS state). Page faults = 178 ms (copy accessed pages from CXL). Execution = 137 ms. Total = 330 ms. (54% of latency is page faults!)
  3. CXLfork: restore = 6.1 ms (attach CXL PTs/VMAs, restore global state). Page faults = 18 ms (CoW on writes, opportunistic prefetch of dirty pages). Execution = 137 ms. Total = 161 ms.
  4. LocalFork baseline: 156 ms (restore ~0, page faults ~19 ms for lazy file mappings, execution 137 ms). CXLfork is only 3% slower than local fork!
🍯
Insight: CXLfork is 3.5× faster than CRIU-CXL and 2× faster than Mitosis-CXL. The page fault overhead is minimal because CXLfork (1) attaches read-only PTEs (no read faults) and (2) opportunistically prefetches dirty pages, avoiding TLB shootdowns on most CoW faults.

② Memory consumption: 10 FaaS functions

Compare local memory consumption when cloning a function to a remote node using CRIU-CXL, Mitosis-CXL, and CXLfork, normalized to a cold start (which copies everything locally).

  1. CRIU-CXL: ~100% of cold-start memory (deserializes/copies all state locally). Occasionally less for small functions due to not checkpointing clean file pages, but they fault in lazily.
  2. Mitosis-CXL: ~40% on average (copies only accessed pages via remote paging). But for large working sets (BFS, BERT), it's ~70–80% because many pages are accessed.
  3. CXLfork (migrate-on-write): ~13% on average (only CoW'd pages copied locally; Init + Read-only stay in CXL). For functions with large footprints, memory savings are substantial.
  4. Example breakdown: Functions vary widely. Some small workloads achieve 95%+ savings, while functions with larger working sets (though still dominated by read-only state) see lower but still significant savings.
💡
CXLfork's savings come from not copying Init (72%) or Read-only (23%) data — only the 4.8% R/W working set is copied on CoW. Mitosis copies everything accessed, so savings depend on working-set size relative to footprint.

③ Tiering trade-offs: BFS function (warm execution)

BFS has a large working set (125 MB). Compare warm execution time (already restored, running 2nd invocation) under the three tiering policies.

  1. Migrate-on-write (MoW): read-only data stays in CXL. Working set exceeds caches → many CXL accesses. Warm execution = 2.1× baseline (LocalFork).
  2. Migrate-on-access (MoA): all accessed pages copied to local memory on 1st invocation. Warm execution = 1.0× baseline (matches LocalFork). But cold execution was 14% slower (copying overhead), and local memory is 250% higher.
  3. Hybrid tiering (HT): copies only pages with A-bit set in checkpoint. Warm execution = 1.3× baseline. Cold execution = 1.02× MoW. Local memory = 1.6× MoW (middle ground).
  4. Recommended: if memory is tight, use MoW and tolerate the 2.1× slowdown. If latency SLOs are strict, use HT or MoA and accept higher memory usage.
🍯
Tiering is a control knob. CXLporter dynamically switches policies based on observed latency and memory pressure. For most functions (small working sets), MoW is fine; for BFS/BERT, HT or MoA is needed.

④ CXLporter under load spike (25% memory environment)

Run 10 FaaS functions at 150 RPS on a cluster with only 25% of the memory used in the baseline (abundant-memory) experiment. Compare P99 latency under CRIU-CXL, Mitosis-CXL, and CXLfork.

  1. CRIU-CXL: P99 = ~16× baseline (25% mem). Memory thrashing forces constant container recycling, and CRIU's slow restore (16–423 ms per function) causes queuing delays.
  2. Mitosis-CXL: P99 = ~16× baseline (25% mem). Same thrashing issue — Mitosis copies accessed pages locally, so memory fills up quickly despite being faster than CRIU.
  3. CXLfork (migrate-on-write): P99 = 1.0× baseline (25% mem). Why? 87% memory savings → 2× more function instances fit → no thrashing. Fast restore (1.2–6.1 ms) → no queuing. CXLporter also shortens keep-alive to 10 s under pressure.
  4. Throughput: CXLfork achieves ~2× throughput of CRIU/Mitosis at 25% memory by enabling more concurrent instances.
🍯
Under memory pressure, CXLfork's memory savings are the dominant factor. Ghost containers + fast restore + shared state in CXL enable CXLporter to scale functions without thrashing, cutting tail latency by ~16×.

Practice problems

Problem 1 · easy
A serverless function has a 200 MB footprint: 150 MB Init, 40 MB Read-only, 10 MB Read/Write. You clone it to 3 nodes using CXLfork (migrate-on-write). How much local memory is consumed in total across the 3 nodes, and how much CXL memory?
Show answer
CXL memory = 200 MB (the checkpoint, shared by all 3 nodes). Local memory = 3 × 10 MB = 30 MB total (only the R/W working set is CoW'd to local memory per node; Init and Read-only stay in CXL, shared). If you had used cold starts or CRIU, it would be 3 × 200 MB = 600 MB local.Misconception: that each node needs a full copy. CXLfork's whole point is that read-only state (Init + R/O = 95%) stays in CXL, shared cluster-wide.
Problem 2 · medium
Why does CXLfork checkpoint private file mappings (e.g., libraries) to CXL instead of letting the child fault them in lazily from the filesystem, like LocalFork does?
Show answer
Because faulting in file pages on a remote node (across the network/storage fabric) is much slower than fetching them from CXL memory. LocalFork can lazily fault libraries from the local filesystem (fast), but CXLfork restores on a remote node where file I/O would cross the network. So CXLfork trades checkpoint size (larger, includes clean file pages) for restore performance (faster, no remote file faults). This also explains why CXLfork's page fault overhead is sometimes lower than LocalFork's — LocalFork lazy-faults libraries, CXLfork doesn't.Misconception: that CXLfork should mimic LocalFork's lazy file faulting. The network hop makes that prohibitively expensive for remote restore.
Problem 3 · medium
CXLfork "rebases" checkpointed page table entries. What does this mean concretely, and why is it necessary?
Show answer
Rebasing = walking the checkpointed page tables and updating PTEs to map virtual pages to CXL physical addresses instead of the parent's local physical addresses. Concretely: if the parent's PTE for virtual page V mapped it to local phys page F, and F's replica is now at CXL phys page X, the checkpointed PTE is updated to map V → X. Why necessary? Without rebasing, the PTEs would still point to the parent node's local memory, making them unusable by other nodes. Rebasing decouples the checkpoint from the parent's OS instance.Misconception: that PTEs can be copied verbatim. They contain machine-dependent physical addresses, so they must be rebased to the CXL address space.
Problem 4 · hard
CXLfork attaches checkpointed page table leaves in O(1). But what about the first access to a page mapped by those leaves — does it trigger a fault? If so, what overhead is incurred? If not, how is that possible?
Show answer
No fault for read accesses. The attached CXL PTEs are valid and map CXL physical pages, so a load instruction that misses in the CPU caches walks the page table (local upper levels + CXL leaves), finds a valid PTE, and fetches the cache line from CXL memory (hardware handles it, ~400 ns latency). No OS intervention, no fault. Fault only on writes: the PTEs are marked read-only, so a store triggers a CoW fault. The OS copies the page to local memory, updates the local PTE to map local phys page + RW, and the store completes (~2.5 μs overhead). This is why CXLfork has near-zero read fault overhead but still pays for CoW on writes.Misconception: that attaching PTEs means accesses fault in pages like Mitosis. CXLfork's PTEs are valid and populated, so reads are just slow loads (CXL latency), not faults.
Problem 5 · medium
CXLporter checkpoints a function after its 16th invocation, not its 1st. Why? And why does it clear the A/D bits after the 1st invocation?
Show answer
16th invocation: Commonly used FaaS languages/runtimes may need a few invocations to reach steady state and optimize. Checkpointing at invocation 1 may capture cold, unoptimized state; checkpointing at 16 captures warmed-up, steady-state code. Clear A/D bits after 1st invocation: the 1st invocation includes initialization (loading model weights, etc.), which touches Init pages heavily but won't touch them again. Clearing A/D bits after the 1st invocation ensures that the A-bits captured at checkpoint time (16th invocation) reflect steady-state access patterns, not initialization noise. This makes hybrid tiering accurate — it copies hot pages, not initialization pages.Misconception: that checkpointing early is better. For workloads requiring warmup, you need to wait. The A/D bit clearing is a subtle detail that makes tiering effective.
Problem 6 · hard
CXLfork uses one checkpoint per function across the cluster. What happens if two nodes restore the same function simultaneously and both try to write to the same checkpointed page? Does this break CoW or cause coherence issues?
Show answer
No coherence issues. Each restored process has independent local page tables (only the leaves are attached from CXL, but the upper levels and the process's mm struct are local). When Node 1's process writes to a checkpointed page, it triggers a CoW fault in Node 1's OS, which copies the page to Node 1's local memory and updates Node 1's local PTE to point to the local copy. Node 2's process still maps the original CXL page (its PTE is unchanged), and if it writes, it triggers an independent CoW fault in Node 2's OS, getting its own local copy. The checkpointed CXL page itself is never written to (it's marked read-only in hardware), so no write conflicts. CXL 3.0's cache coherence ensures that if both nodes cache the CXL page, they see a consistent view before CoW.Misconception: that CoW breaks when multiple nodes share the same checkpoint. Each node's OS handles CoW independently; the checkpoint is immutable and protected by read-only PTEs.
Problem 7 · easy
What is a "ghost container" and why does it speed up function cold starts?
Show answer
A ghost container is an empty, pre-configured container (namespaces, cgroups, network set up) with no function loaded, consuming only 512 KB of memory. When a function needs to be deployed, CXLporter issues a restore request to a ghost container, which calls CXLfork-restore to attach the function's checkpointed state. This skips the ~130 ms overhead of container creation (setting up namespaces, cgroups, network), which is function-agnostic. Result: instant function deployment into a sandboxed container.Misconception: that container creation is negligible. The paper measures ~130 ms overhead, which is significant for latency-sensitive FaaS. Ghost containers amortize this by pre-creating the sandbox.
Problem 8 · medium
Under what workload conditions does CXLfork's migrate-on-write policy hurt performance the most, and which tiering policy should CXLporter switch to?
Show answer
Migrate-on-write hurts most when the function's working set is large and doesn't fit in the CPU caches, causing frequent load misses that fetch from CXL memory (~400 ns) instead of local DRAM (~100 ns). Examples: BFS (125 MB working set), BERT (630 MB). In the paper, BFS's warm execution is 2.1× slower than baseline with MoW. Solution: CXLporter should switch to hybrid tiering (copy pages with A-bit set) or migrate-on-access (copy all accessed pages). Hybrid is the middle ground: warm execution 1.3× baseline, local memory 1.6× MoW. If SLOs are strict, use MoA (1.0× baseline exec, but 2.5× MoW memory).Misconception: that MoW is always best. It maximizes memory savings but can penalize functions with large working sets. Tiering is a control knob that trades memory for performance.
Quick answers

Frequently asked questions

Ten questions newcomers ask most. Tap any to expand.

Q1How is CXLfork different from just using CXL as swap or remote paging?▾
Swap/remote paging copies pages on-demand to local memory on faults, one page at a time, and the remote copy is treated as backing store. CXLfork checkpoints the entire process state once to CXL (not per-page), then restores by attaching checkpointed page tables/VMAs so the child can directly access CXL pages without faults (for reads). Writes trigger CoW, but reads are just slow loads. Also, CXLfork shares read-only state cluster-wide automatically — swap doesn't deduplicate across nodes.
Q2Why not just use RDMA + Mitosis instead of CXL?▾
Mitosis over RDMA copies accessed pages via one-sided RDMA writes, which is fast but still a bulk copy per page. CXL shared memory allows byte-addressable, cache-coherent access — loads and stores fetch cache lines on-demand without software intervention. This enables CXLfork to keep read-only data in CXL (shared) and only copy writes via CoW, whereas Mitosis copies all accessed pages (reads and writes) locally. Also, CXL hardware can cache hot pages automatically; RDMA requires explicit prefetching logic.
Q3What happens if the CXL memory is full? Can CXLfork evict old checkpoints?▾
Yes. CXLporter is responsible for managing checkpoint lifecycle and can reclaim checkpoints under CXL memory pressure. The paper does not specify the exact eviction policy, but standard approaches would ensure no active clones are using a checkpoint before eviction.
Q4Does CXLfork work for multi-threaded processes or just single-threaded functions?▾
The paper targets single-process serverless functions (common in FaaS). The paper does not evaluate multi-threaded workloads. Shared-memory parallel workloads (e.g., MPI processes sharing anonymous memory via shm) are explicitly out of scope — CXLfork doesn't currently support shared anonymous mappings between processes.
Q5How does CXLfork handle open sockets or database connections?▾
Sockets and other file descriptors are part of global state. CXLfork handles them similar to files, using checkpointed paths and permissions. The paper does not detail socket-specific behavior such as re-establishing TCP connections or handling stateful database sessions.
Q6Why does CXLfork opportunistically prefetch dirty pages? Doesn't that negate the zero-copy claim?▾
CXLfork prefetches pages marked dirty in the parent's checkpointed PTEs, betting that if the parent wrote to them, the child will too (profiling shows >95% accuracy for FaaS). This avoids TLB shootdown overhead on CoW faults (~500 ns per fault). But it's a small fraction of the footprint — only 4.8% R/W on average. So CXLfork is still close to zero-copy (read-only state, 95%, is never copied). The prefetch is an optimization to reduce CoW latency, not a fundamental requirement.
Q7What is the overhead of CXLfork's checkpoint phase?▾
The paper says checkpointing is 1.5× slower than Mitosis (which checkpoints to local memory) because CXLfork copies to CXL memory (slower writes). But checkpoint is off the critical path — it's done once, after the 16th invocation, and then reused many times. The paper focuses on restore latency (on the critical path) and reports that CXLfork and Mitosis checkpoint one order of magnitude faster than CRIU.
Q8Can CXLfork checkpoint GPU state or accelerator state?▾
No, not in the current implementation. CXLfork only checkpoints CPU state (registers, page tables, process memory). GPU memory and accelerator state would require device-specific support (e.g., CUDA API for GPU checkpointing). The paper focuses on CPU-centric FaaS workloads (Python, Node.js).
Q9How does CXLfork ensure the checkpoint is "immutable"? What prevents a bug in one node from corrupting it?▾
CXLfork marks all checkpointed pages as read-only in the PTEs. Hardware enforces this — any write attempt triggers a CoW fault, and the OS copies the page locally before writing. So bugs in one node's process can't corrupt the CXL checkpoint (they'd fault and write to their local copy). However, a kernel bug (e.g., a rogue kernel write to CXL memory) could corrupt it. The paper assumes trusted OS instances and excludes CXL memory from kernel memory reclamation (LRU lists).
Q10What did the authors leave for future work?▾
Several items: (1) extending CXLfork to large distributed systems (the eval is on 2 VMs due to hardware limits); (2) handling shared anonymous memory between processes; (3) more sophisticated tiering policies (e.g., bandwidth-aware, not just latency-aware); (4) checkpoint eviction policies under CXL memory pressure; (5) workflows (multi-function FaaS apps) and inter-function communication over CXL.

Core terms
CXLforkRemote fork interface for CXL shared memory: checkpoint once to CXL, restore by attaching (not copying), CoW on writes.
CXL (Compute Express Link)Cache-coherent interconnect (v3.0+) enabling byte-addressable shared memory across compute nodes in a rack.
CRIUCheckpoint/Restore In Userspace — serializes process to files (Protocol Buffers), standard for process migration. Slow on restore.
MitosisRemote fork for RDMA: serializes OS metadata, lazily copies pages via remote paging on faults. Faster than CRIU, but copies all accessed data.
Private stateProcess-owned data/metadata (page tables, VMA tree, CPU registers, memory pages, private file mappings) — checkpointed as-is, then rebased.
Global stateOS-shared structures (open files, sockets, namespaces, mount points) — lightly serialized (file paths, etc.), re-instantiated on restore.
RebasingUpdating checkpointed PTEs/pointers to index CXL physical addresses (not local), so other OS instances can remap them.
Copy-on-Write (CoW)Pages mapped read-only; on store, OS copies page to local memory and updates PTE to RW. Checkpoint stays immutable.
Ghost containerEmpty pre-configured container (512 KB) waiting for CXLfork-restore. Skips ~130 ms container-setup overhead.
CXLporterHorizontal autoscaler for FaaS using CXLfork: maintains ghost containers, dynamic tiering, short keep-alive under memory pressure.
Supporting terms
PTE (Page Table Entry)Maps virtual page → physical page. CXLfork rebases checkpointed PTEs to map CXL phys addresses.
VMA (Virtual Memory Area)Describes a memory region (e.g., heap, stack, file mapping). CXLfork attaches checkpointed VMA tree leaves from CXL.
A/D bits (Access/Dirty)Hardware-maintained flags in PTEs. A=1 if accessed, D=1 if written. CXLfork uses A-bits to identify hot pages for hybrid tiering.
Migrate-on-writeCXLfork tiering policy: copy pages to local memory only on stores (CoW faults). Max memory savings, may penalize loads.
Migrate-on-accessTiering policy: copy pages to local memory on first access (read or write). Faster execution, but 250% more local memory.
Hybrid tieringTiering policy: copy pages to local memory on access, but only if A-bit set in checkpointed PTE. Middle ground.
FaaS (Function-as-a-Service)Serverless computing: deploy functions on-demand, pay per invocation. Cold start = boot from scratch (slow).
Init dataFunction state used during initialization (model weights, libraries), rarely accessed during execution. 72% of footprint on average.
Keep-alive windowTime idle function instances stay cached in memory to avoid near-future cold starts. Default: minutes. CXLporter shortens to 10s under pressure.

Wei et al. · OSDI 2023

State-of-the-art remote fork for RDMA. Lazily copies pages via remote paging. CXLfork builds on its ideas but uses CXL shared memory to avoid copying read-only data.

Open-source project, Linux Foundation

Standard process migration tool. Serializes state to files using Protocol Buffers. Baseline for the paper's comparison.

Li et al. · ASPLOS 2023

Explores CXL memory for cloud platforms. Discusses latency, bandwidth, and disaggregation. Complementary to CXLfork's process-cloning use case.

TPP: Transparent Page Placement for CXL hard

Maruf et al. · ASPLOS 2023

Memory tiering for CXL: identifies hot pages and migrates them to local DRAM. CXLfork's hybrid tiering uses similar ideas (A-bit tracking).
