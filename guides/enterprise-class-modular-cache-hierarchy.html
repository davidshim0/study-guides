HPCA 2025 · A Field Guide

Building 256 MB of L3 from eight 32 MB L2s without a separate array

IBM's Telum chip turned eight private caches into one big virtual tier — and did it algorithmically, with lateral eviction, dynamic activity counters, and adaptive LRU insertion — making idle caches useful and cache growth modular.

📄 "Enterprise Class Modular Cache Hierarchy" · Walters et al. 🎓 Assumed: CXL basics, tiering intuition ⏱ ~25 min read
🌊

Push evictions from a private cache sideways to the least-active peer cache (not down to a separate victim array). That peer becomes your virtual L3 — and the sum across all peers is 256 MB, built at zero extra die area.

Before you begin

Twelve ideas the paper assumes you know

Each card clears up one concept the Telum cache design builds on. They're collapsed — tap to open. Read in order the first time, or jump to whatever looks unfamiliar.

Virtual L3 / active core Primary castout (PCO) Lateral persistence machinery Traditional tier (z15 baseline)
1What a tiered cache hierarchy is▾

Classic design: each core gets a private L1 and maybe L2, then a shared L3 (sometimes a shared L4) sits farther out, physically separate, before memory. Each tier is inclusive — everything in L2 is also in L3 — so evicting from L3 means invalidating L2 entries, but lookup latency rises as you go outward.

IBM's z15 (the baseline in this paper) had: 128K L1 → 4 MB private L2 → 256 MB shared L3 (on-chip) → 960 MB shared L4 (off-chip, system controller) → memory. All physically separate arrays, tuned and built separately.

Now you're ready: Telum (z16) doesn't build a separate L3 array. It takes eight 32 MB L2s and makes them behave collectively as if there were a 256 MB L3, using eviction flow and activity tracking.
2Victim cache vs inclusive cache▾

An inclusive cache (classic L3) mirrors everything in lower tiers — hit L2, it's also in L3. A victim cache only holds what was evicted from somewhere else — disjoint content. Telum's L2s are victim caches for each other: your evictions land in a peer's L2, not in a separate L3 array, and that peer's L2 now holds your victims.

Victim caches avoid duplication (good for capacity) but require tracking per-tier state to know if a line is local or a victim (more complex directory).

Now you're ready: "Lateral cache persistence" is a victim-cache algorithm that treats peer L2s as eviction targets, creating a virtual L3 from disaggregated building blocks.
3SMP scaling and cache sharing▾

IBM Z systems run Symmetric Multi-Processor (SMP) workloads — dozens to hundreds of cores, all sharing coherent memory. Traditional designs face a dilemma: a bigger shared cache helps, but building it centrally becomes latency-bound and area-expensive. A distributed approach (many small private caches) wastes capacity if cores sit idle.

The paper tests 16-way, 28-way, and 32-way workloads — "way" here means active threads/cores, not cache associativity. These are large, transactional, database-heavy applications that scale across many cores and benefit from big caches.

Now you're ready: Telum's virtual caching lets idle L2s contribute to SMP performance as L3 capacity even when their cores aren't running, which traditional private L2 + shared L3 designs can't do.
4What "lateral" eviction means▾

In a traditional hierarchy, evictions go down: L1 → L2, L2 → L3, L3 → L4, L4 → memory. "Lateral" means evictions go sideways: when your L2 evicts a line, instead of sending it to a separate L3 array, it sends it to a peer L2 on the same chip ring, chosen by an activity counter.

This is the core trick: your victim becomes another core's L2 entry, marked as "persisted state" (virtual L3), not local. Repeat across all cores and you've built a distributed 256 MB L3 out of eight 32 MB L2s, with zero extra array.

Now you're ready: The paper's "Primary Castout (PCO)" is the sideways eviction event; "Secondary Castout (SCO)" is a cascaded eviction to the drawer scope (virtual L4).
5Activity counters and LRU at the scope level▾

To decide which peer cache receives your eviction, each L2 has an activity counter: incremented on every fetch miss + every eviction install, decremented globally when one counter approaches saturation (right-shift). The L2 with the lowest count is least-recently-used at the scope level (on-chip for L3, on-drawer for L4) and becomes the eviction target.

This is LRU applied to whole caches, not individual cache lines. The counter tracks "how busy is this L2?" and evictions flow to the quietest one.

Now you're ready: The "combined response" at the diagonal L2 (opposite the requester on the ring) merges all the activity counters and broadcasts the target back, enabling distributed, fair victim placement.
6Directory state tags for local vs persisted lines▾

A given L2 cache line can be in one of three logical states: local (installed by this core's fetch), on-chip persisted (evicted from a peer L2 on this chip → virtual L3), or on-drawer persisted (evicted from a peer chip → virtual L4). The directory tracks this with extra state bits.

When an L2 evicts a line, it checks the state: if it's local or on-chip persisted, it can be invalidated or pushed laterally; if it's the highest coherency copy globally, it must be written back.

Now you're ready: These state tags let Telum mimic a tiered hierarchy's eviction flow (local → L3 → L4 → memory) using only L2 arrays, by knowing what tier a given line "belongs to."
7Adaptive mid-LRU insertion policy▾

When a laterally persisted line (a peer's eviction) arrives at your L2, where in the LRU stack should it land? If always MRU, it will thrash out lines a running core actually uses. Telum inserts victims at mid-LRU (1/n-th position) to protect active working sets, but switches to MRU install when the lower LRU positions fill up, assuming the core is idle or under-utilizing the cache.

This adaptive policy balances protecting an active core's footprint (mid-LRU) with making full use of idle capacity (MRU fill when safe).

Now you're ready: Without this adaptive policy, lateral caching would either waste capacity (fixed mid-LRU) or thrash active cores (fixed MRU). The paper's design dynamically adapts per cache row.
8Ring bus topology and diagonal merging▾

Telum's eight L2s sit on a dual-concentric on-chip ring bus. A fetch request from one L2 launches in both directions (clockwise/counter-clockwise), interrogating all peer L2s. Directory lookup results are merged at the diagonal L2 (the one opposite the requester on the ring) and broadcast back.

This merging point is where activity counters are compared and the eviction target is chosen. The diagonal position minimizes the max latency for both ring directions to converge.

Now you're ready: The diagonal merge is a key implementation detail — it's where the "combined response" (chip-level coherent state + eviction target) is created, enabling scalable lateral eviction.
9Multi-Copy Scope (MCS) bits▾

A cache line can be shared at different scopes: only on-chip, only on-drawer, or globally (cross-drawer). Telum's directory includes MCS bits to track this, reducing the scope of coherence broadcasts. If a line is only shared on-chip, an exclusive fetch can be satisfied with an on-chip broadcast, not a global one.

This is a latency optimization for the modified-MOESI protocol: without MCS, every exclusive-fetch of a shared line would have to go global to be safe.

Now you're ready: MCS bits are part of Telum's enhanced Z-MOESI protocol (new to the paper), helping lateral persistence scale by limiting unnecessary drawer/global broadcasts.
10Speculative memory access▾

When a fetch request reaches the chip containing the memory port, it may speculatively access the DIMM in parallel with the drawer-scope cache interrogation, using partial coherency info. If the full coherent state later indicates a cache hit, the memory data is dropped.

This overlaps memory latency with coherence latency, substantially lowering effective memory access time when it's actually needed.

Now you're ready: Speculative memory is why Telum's ~60-cycle drawer miss latency isn't dominated by DRAM latency — the DIMM access runs concurrently with the drawer coherence broadcast.
11What RNI (Relative Nest Intensity) means▾

"Nest" = cache hierarchy. RNI classifies workloads by how much they stress the cache hierarchy vs the core. Low RNI = core-bound (e.g., app servers); High RNI = cache/memory-bound (e.g., large databases with random access). The paper's four workloads span Low to High RNI.

High-RNI workloads benefit most from bigger caches; Low-RNI from lower cache latency or core frequency. The sensitivity studies in §VI show High-RNI apps get +9–11% from doubling L2, while one Low-RNI app (32w-Mic-Tran) is cache-insensitive.

Now you're ready: The paper demonstrates cache size sensitivity across RNI categories to show lateral persistence helps broadly, but some apps (Low RNI) need other optimizations (frequency, prefetch) more than capacity.
12EDRAM loss and the 7nm transition▾

IBM's z15 (14nm) used EDRAM for cache arrays — denser than SRAM, enabling a 256 MB L3 and 960 MB L4 in prior generations. On 7nm, EDRAM became unviable (Lesson #10 in the paper). SRAM-only caches would have shrunk capacity dramatically.

Telum's lateral persistence was born from this constraint: instead of building a smaller separate L3, they kept L2s big and made them double as L3 via eviction flow, reclaiming the lost density through algorithmic reuse of the same silicon.

Now you're ready: The technology disruption (loss of EDRAM) is what motivated the architectural innovation. When the traditional path closed, IBM invented a new one.

The whole story in one place

Lateral cache persistence in a nutshell

The problem, the algorithmic fix, the performance win, and how it scales to Telum II.

Traditional tiered cache hierarchies waste die area: each tier (L1, L2, L3, L4) is a physically separate array, inclusive duplication reduces net capacity, and when a core goes idle its private L2 sits unused. Worse, technology shifts (losing EDRAM at 7nm) made the classic L3 array unaffordable in density.

IBM's Telum (z16) answer: don't build a separate L3. Take eight 32 MB L2-Caches, cluster them on a dual-ring bus, and push evictions sideways (not down) using lateral cache persistence. Each L2 tracks an activity counter (fetch misses + eviction installs); on a local eviction, the chip-wide combined response identifies the least-active peer L2 and sends the victim there (a Primary Castout, PCO). That peer marks the line as "on-chip persisted" (virtual L3 state) and installs it mid-LRU to protect its core's working set, switching to MRU if the core is idle. Repeat across all cores: you've built 256 MB of virtual L3 from the same silicon that was already there for L2s, at zero extra die area.

Scaling to the drawer: eight chips in a star mesh; each chip's evictions can cascade to a peer chip's L2 (a Secondary Castout, SCO) using the same activity-counter algorithm, creating 2 GB of virtual L4 from 64 on-drawer L2s. Final evictions (Tertiary Castout, TCO) go to memory. Directory state tags (local / on-chip persisted / on-drawer persisted) let the hardware track logical tier membership without a separate array, and Multi-Copy Scope (MCS) bits limit coherence broadcasts to the region of active sharing.

The payoff: z16 improved over z15 by +7.5% to +13.8% across four SMP workloads, with cache/memory gains (finite) contributing two-thirds to five-sixths of the total. Doubling L2 size (which also doubles L3 and L4 via persistence) improves High-RNI workloads by +9–11% at constant latency. And crucially, idle L2s contribute to SMP throughput instead of sitting dark — something no traditional private L2 + shared L3 design can match.

🍯 Key takeaways

  • Lateral persistence = push evictions sideways to the least-active peer L2, creating virtual L3/L4 from disaggregated L2 arrays.
  • Activity counters per L2 (fetch misses + eviction installs) identify the LRU cache at the scope level; the diagonal L2 merges and broadcasts the target.
  • Adaptive mid-LRU insertion: victims land mid-LRU to protect active cores, switch to MRU when cache rows fill (idle cores).
  • Directory state tags (local / on-chip persisted / on-drawer persisted) track logical tier membership without a separate L3 array.
  • MCS bits (Multi-Copy Scope) limit coherence broadcasts to on-chip, on-drawer, or global, reducing latency for exclusive fetches.
  • Zero extra die area: 256 MB L3 and 2 GB L4 built from the same L2 silicon, reclaiming capacity lost to EDRAM's demise.
  • Idle caches contribute: unlike traditional private L2s, Telum's L2s remain useful as virtual L3/L4 even when cores are offline.
  • Result: +7.5–13.8% over z15; High-RNI workloads get +9–11% from doubling L2 (which also doubles L3 and L4).
Six everyday pictures

How to think about lateral cache persistence

Each picture stands alone. Every card ends with an honest note about where the picture stops being true.

Core idea · Sideways eviction

Shared bookshelves

Four people share four bookshelves. The first person (active) fills theirs. When it's full, instead of dumping books on the floor (memory), they look at the other three shelves to see which is least-used and slide their overflow there. That shelf becomes their virtual extra row. Now multiply: everyone does this, and collectively they've built a library (virtual L3) from the same shelves, with no extra furniture.

Where it breaks: Real bookshelves don't track "activity counters" or merge at the "diagonal" shelf. The ring-bus merge is an implementation detail the analogy skips.
Adaptive insertion · Mid-LRU vs MRU

Desk drawers with priority rows

Your desk drawer has two sections: top row for your own stuff (MRU), bottom row for overflow from a colleague (mid-LRU). When a colleague's eviction arrives, it lands in the bottom row, protecting your top-row working set. But if you're away and the bottom fills up, the drawer switches to accepting colleague overflow in the top row too, making full use of idle space.

Where it breaks: Real drawers don't "switch policy" — the adaptive mid-LRU → MRU transition is a per-cache-row decision in hardware, checked every eviction install.
Scope-level LRU · Activity counters

Team meeting rooms with usage logs

Five meeting rooms on a floor. Each room has a log sheet: +1 tally for every meeting held, +1 for every overflow group sent here from a full room. When a room is full and needs to evict a group, the room with the fewest tallies (least-active) accepts the overflow. This is LRU applied to whole rooms, not individual chairs.

Where it breaks: Real logs don't auto-decrement when one approaches saturation. Telum's counters right-shift globally to keep relative magnitude meaningful without overflow.
Directory state · Logical tier membership

Luggage tags at the airport

A bag can have one of three tags: "local flight" (stays at this terminal), "domestic connection" (goes to another terminal at this airport), or "international" (goes to a different airport). The tag tells the system which scope it belongs to, even though all bags physically move on the same conveyor belt (the L2 arrays).

Where it breaks: Bags don't change tags mid-flight. In Telum, a line can be promoted/demoted between local and persisted states dynamically as cores share or evict.
MCS bits · Scoped broadcasts

Email distribution lists

A document is shared at three levels: "team only" (send to this floor), "department" (send to this building), or "company-wide" (send globally). The MCS bits are like the distribution list: when you need to mark a document exclusive, you only broadcast to the narrowest scope that's actively using it, not the whole company every time.

Where it breaks: Emails don't auto-upgrade scope when forwarded. MCS bits track observed sharing dynamically and can escalate a line from on-chip to on-drawer to global as sharing spreads.
Idle cache reuse · Zero wasted area

Hotel rooms in off-season

A hotel has ten guest rooms. In high season, all are booked (active cores). In low season, three are occupied and seven are empty. Traditional design: those seven rooms sit dark, wasted. Telum: the occupied guests' overflow luggage fills the empty rooms, so all ten rooms contribute to total capacity regardless of guest count. No wasted space.

Where it breaks: Hotel rooms don't track "activity counters" or adaptively accept luggage. The analogy skips the mid-LRU policy that prevents active guests from being thrashed out by luggage.
The full explanation

How it actually works

Section by section, with the paper's key mechanisms rebuilt as visuals you can read at a glance.

1 · The fetch request flow — on-chip interrogation

When a core's L1 misses, the request goes to its paired L2. On an L2 hit (nominal 19 cycles), data returns immediately. On an L2 miss, the request launches on the on-chip ring bus in both directions (clockwise and counter-clockwise). Each peer L2 does a directory lookup for the fetch address and the address of the local L2's eviction target (the LRU line in the set that will be replaced).

Directory results are merged at the diagonal L2 (the one opposite the requester on the ring). This combined response includes: (1) whether any peer has the fetch address (and its state), (2) the activity counter values from all L2s, and (3) which L2 has the lowest activity count — that one becomes the eviction target for the initiating L2's victim.

If a peer L2 has the fetch address, it returns the data immediately on the ring (using the shortest direction). If no on-chip L2 has it, the request escalates to the off-chip / on-drawer scope.

🌱
Key insight: The on-chip interrogation is serialized with the L2 lookup (L1 miss → L2 lookup → ring broadcast), so the base L2 latency (19 cycles) includes the time to reach the diagonal L2 and get the combined response. This is why the paper says average L3 latency is 63 cycles — it's L2 latency + ring traversal + merge, not a separate L3 lookup.

2 · The fetch request flow — on-drawer and global

If no on-chip L2 has the fetch address, the initiating chip's off-chip interface broadcasts the request to all seven peer chips in the drawer (star mesh). Each chip receives it, forwards it to its own on-chip ring (same diagonal merge as §1), and returns a remote-chip partial response.

These remote responses are merged into a drawer-level combined response at the initiating chip and broadcast back to all peer chips, enabling them to take coherent actions (e.g., invalidate or downgrade shared copies). If the drawer-level response indicates a miss, the request escalates to the off-drawer scope (cross-drawer broadcast to up to three other drawers in a max-config z16 system).

In parallel with scope-based interrogation, when the fetch request reaches the chip containing the memory port for the target address, it may speculatively access the DIMM using partial coherency info. If the final combined response indicates a cache hit, the memory data is dropped. This overlaps memory latency with coherence, substantially lowering effective DRAM access time.

3 · Lateral cache persistence — PCO, SCO, TCO

When L2-0 evicts a line, it checks the directory state. If the line is local or on-chip persisted and not the highest coherency copy globally, it can be invalidated or sent as a Primary Castout (PCO) to the least-active peer L2 (determined by the combined response).

The receiving L2 installs the PCO data with "on-chip persisted" directory state, marking it as a virtual L3 entry. It selects a victim using its own LRU algorithm. If that victim also requires write-back, the receiving L2 creates a Secondary Castout (SCO) to the least-active peer chip in the drawer (using drawer-level activity counters). The SCO is broadcast to all chips in the drawer; the target chip interrogates its on-chip L2s and installs the victim with "on-drawer persisted" state (virtual L4 entry).

If the SCO recipient's L2 evicts a line that also requires write-back, it creates a Tertiary Castout (TCO) to the memory port associated with the address, completing the eviction path from L2 → L3 → L4 → memory using only L2 arrays.

🍯
The insight: PCO/SCO/TCO mimic the eviction flow of a traditional L2 → L3 → L4 → memory hierarchy, but all three "tiers" are built from L2 victim installs, with directory state tags tracking logical membership. Zero separate L3/L4 arrays, zero wasted die area.

4 · Adaptive mid-LRU insertion policy

When a PCO (victim from a peer L2) arrives at your L2, where in the LRU stack does it land? If always MRU, it will thrash out lines your core is actively using. If always LRU, it will be evicted immediately, wasting the lateral persistence effort. Telum uses an adaptive mid-LRU policy:

# Per cache row, on PCO/SCO install:
if persisted entries have filled up to 1/n-th LRU position:
    install at MRU    # core is idle or under-utilizing, make full use
else:
    install at mid-LRU (1/n position)    # protect active core's working set

This policy dynamically balances two goals: (1) protect an active core's local working set by keeping victims in mid-LRU, and (2) make full use of idle capacity by switching to MRU install when the core isn't using the lower LRU positions.

💡
Why adaptive? A fixed mid-LRU policy would limit lateral persistence to 1/n of total capacity (wasting the rest). A fixed MRU policy would thrash active cores. The adaptive switch lets Telum use all L2 capacity when cores are idle (like a traditional shared L3) while protecting active cores' footprints (like a private L2).

5 · Activity counters and scope-level LRU

Each L2 has an activity counter: incremented on every fetch miss and every eviction install (both local fetches and lateral PCO/SCO installs). When one counter approaches saturation, all counters in the scope are right-shifted (multi-bit decrement) to maintain relative magnitude.

On a PCO eviction, the diagonal L2 reads all on-chip activity counters and identifies the lowest value — that L2 is the least-recently-used at the cache scope level and becomes the eviction target. This is LRU applied to whole caches, not individual lines, and ensures evictions flow to the quietest peer, balancing load.

For SCO (drawer scope), the off-chip interface tracks drawer-level activity counters per chip (fetch misses + SCO installs from all chips). The same lowest-count algorithm identifies the target chip for SCO install.

6 · MCS bits and scoped coherence

A cache line can be shared at three scopes: on-chip only (among L2s on one chip), on-drawer only (among chips in one drawer), or globally (across drawers). Telum's directory includes Multi-Copy Scope (MCS) bits to track this observed sharing, reducing the scope of coherence broadcasts.

For example, if a line's MCS bits indicate "on-chip only," an exclusive fetch of that line can be satisfied with an on-chip interrogation, not a drawer or global broadcast. If the line is later shared to a peer chip, the MCS bits are upgraded to "on-drawer." This is a latency optimization for the Z-MOESI protocol, avoiding unnecessary global broadcasts when sharing is local.

🌱
MCS bits are dynamic: they track observed sharing as the system runs and upgrade scope when necessary. This differs from static partitioning schemes where scope is fixed at allocation time.
Follow the numbers, step by step

Worked examples

Four traces using the paper's real data — eviction flow, cache size growth, latency impact, and Telum II scaling.

① Eviction cascade: PCO → SCO → TCO

Core-0's L2 evicts a modified line. Walk through the three castout events (PCO, SCO, TCO) using the paper's drawer topology (8 chips, 64 L2s).

  1. PCO (Primary Castout): Core-0's L2-0 evicts a modified line. Diagonal merge identifies L2-5 (on the same chip) as lowest activity count. PCO sends the victim to L2-5, which installs it with "on-chip persisted" state (virtual L3 entry).
  2. L2-5 cascades: L2-5's LRU algorithm selects a victim for replacement. That victim is in "on-chip persisted" state and modified, so a write-back is required.
  3. SCO (Secondary Castout): L2-5 sends an SCO to the off-chip interface. Drawer-level activity counters identify chip-3 (on a different DCM) as the least-active chip. SCO is broadcast to all eight chips; chip-3 receives the data.
  4. Chip-3 interrogates: SCO enters chip-3's ring bus, diagonal merge occurs, and one L2 on chip-3 (say L2-18) is selected for install with "on-drawer persisted" state (virtual L4 entry).
  5. L2-18 cascades: L2-18's victim is in "on-drawer persisted" state and modified. It creates a TCO (Tertiary Castout) to the memory port associated with the victim's address, writing the line back to DRAM. Done.
🍯
The point: This three-step cascade (PCO → SCO → TCO) mirrors the eviction flow of a traditional L2 → L3 → L4 → memory hierarchy, but all three "tiers" are installs into L2 arrays, with directory state tags tracking logical membership. (Paper Section II.D, Figure 5)

② Cache size growth: L2 → L3 → L4 scaling

Telum: 8 cores, 8×32 MB L2s. What are the L3 and L4 sizes? What if L2 size doubles to 64 MB?

  1. Base config: 8 L2s × 32 MB = 256 MB total on-chip L2 capacity. With lateral persistence, one active core sees 32 MB local L2 + 7×32 MB = 224 MB from peer L2s (virtual L3). Total L3 capacity = 256 MB (the sum of all on-chip L2s).
  2. Drawer config: 8 chips × 256 MB = 2048 MB total on-drawer capacity. One active core sees 32 MB local + 256 MB on-chip L3 + (7 chips × 256 MB) = 1792 MB from peer chips (virtual L4). Total L4 = 2 GB.
  3. Double L2 size: If each L2 is 64 MB, then L3 = 8×64 MB = 512 MB and L4 = 8 chips × 512 MB = 4 GB. Doubling L2 doubles all tiers simultaneously.
  4. Add L2s, not size: If L2 size stays 32 MB but you add 2 more L2s per chip (10 total), then L3 = 10×32 MB = 320 MB and L4 = 8×320 MB = 2.56 GB. This is exactly what Telum II does (see Example ④).
💡
Insight: In a traditional tiered design, doubling L2 size doesn't automatically double L3/L4 — you'd have to redesign each tier separately. With lateral persistence, L3 and L4 are derived from L2 capacity via the eviction flow, so growth is multiplicative and modular. (Paper §III, Figure 9–11)

③ Latency adder sensitivity: +2, +4, +6 cycles

The paper's Figure 12 shows performance impact of adding 2/4/6 cycles to L2 latency (which also impacts L3/L4 by the same amount, diluted). Use the High-RNI workload (28w-Rel-DB) from the graph.

  1. Base latency: L2 nominal = 19 cycles (3.6 ns @ 5.2 GHz). L3 nominal = 63 cycles (12 ns). These include ring traversal + diagonal merge.
  2. +2 cycle adder: L2 becomes 21 cycles. Figure 12 shows 28w-Rel-DB drops to ~0.985 relative performance (1.5% loss). High-RNI workloads are less sensitive to L2 latency (more L3/L4-bound) than Low-RNI.
  3. +4 cycle adder: L2 = 23 cycles. 28w-Rel-DB → ~0.97 relative (3% loss).
  4. +6 cycle adder: L2 = 25 cycles. 28w-Rel-DB → ~0.955 relative (4.5% loss). The Low-RNI workload (16w-Tran-Proc) drops more steeply (~6% at +6 cycles) because it's more L2-latency-sensitive.
  5. Telum II goal: Maintain the same 19-cycle nominal latency despite adding 2 more L2s to the ring (10 total), ensuring no latency regression. (Paper §VI.D, Figure 12)
🌱
Lower-RNI workloads (core-bound) suffer more from L2 latency adders. Higher-RNI workloads (cache-bound) benefit more from L2 capacity growth (Figure 9). This is why the paper tests both dimensions: latency and capacity are orthogonal levers.

④ Telum II scaling: 10 L2s → 360 MB L3, 2.88 GB L4

Telum II adds 2 L2s (one for the DPU, one isolated) and grows L2 size from 32 MB to 36 MB. What are the new L3/L4 sizes?

  1. L2 size growth: 32 MB → 36 MB is +4 MB (+12.5%). This alone would grow L3 by 12.5% (256 MB → 288 MB) and L4 by 12.5% (2 GB → 2.24 GB).
  2. Add 2 L2s per chip: 8 L2s → 10 L2s. Now L3 = 10×36 MB = 360 MB (+40% over Telum's 256 MB). L4 = 8 chips × 360 MB = 2.88 GB (+44% over Telum's 2 GB).
  3. The DPU's L2: When the DPU is active, its L2 acts as local cache for DPU workloads. When idle, it contributes to the lateral persistence pool as virtual L3/L4 for other cores. Same for the isolated L2 (used for cache expansion, not paired with a core).
  4. Zero wasted area: Traditional design: adding a DPU with its own L2 would compete with the L3 array for die space, requiring either a smaller L3 or a larger chip. Telum II: the DPU's L2 is the L3 growth, at zero extra die area. (Paper §VII, Figure 13)
🍯
The payoff: Telum to Telum II delivers +40% L3 and +44% L4 capacity in one generation, driven by a +12.5% L2 size bump and adding 2 L2s per chip. This would be nearly impossible in a traditional L2+L3+L4 tiered design without a massive die size increase. (Paper §VII–VIII)

Problem 1 · easy
An L2 has an activity counter of 1200. It receives a PCO (victim from a peer L2). Does the counter increment, and if so, by how much?
Show answer
Yes, the counter increments by 1. Activity counters track both fetch misses (demand from this core) and eviction installs (lateral persistence traffic). A PCO install is an eviction install, so the counter goes 1200 → 1201.Misconception: that only fetch misses increment the counter. Both fetch misses and lateral eviction installs increment it, so the counter reflects total "busyness" (demand + victim traffic).
Problem 2 · medium
A Telum chip has 8 L2s, each 32 MB. One core is running; seven cores are idle. What is the effective L3 capacity available to the running core?
Show answer
The running core sees its local 32 MB L2 + 7×32 MB = 224 MB from peer L2s (virtual L3). Total L3 capacity = 256 MB (the sum of all on-chip L2s). The idle cores' L2s contribute to L3 capacity via lateral persistence, unlike a traditional private L2 design where idle L2s sit unused.Misconception: that idle L2s are wasted. In Telum, they're not — they accept PCOs from active cores and act as virtual L3, so all 256 MB on-chip is useful regardless of how many cores are running.
Problem 3 · medium
A victim cache line arrives at L2-3 via PCO. The adaptive mid-LRU policy checks the cache row. If the lower LRU positions (say, below 1/4-th) are already full of persisted entries, where does the PCO install?
Show answer
The PCO installs at MRU (most-recently-used position). The adaptive policy switches from mid-LRU to MRU when the lower positions fill up, assuming the core is idle or under-utilizing that row. This lets the cache make full use of idle capacity.Misconception: that mid-LRU is always used for victims. The policy is adaptive: mid-LRU protects active cores, MRU makes full use of idle rows. The switch is per cache row, checked on every install.
Problem 4 · hard
Compare the eviction path of a traditional tiered cache (L2 → L3 → L4 → memory) vs Telum's lateral persistence (PCO → SCO → TCO). What's the key difference in where the "tiers" physically live?
Show answer
Traditional: L2, L3, L4 are physically separate arrays (different SRAM or EDRAM blocks on different parts of the chip/package). Telum: all three "tiers" (L2, virtual L3, virtual L4) are installs into L2 arrays, differentiated only by directory state tags (local / on-chip persisted / on-drawer persisted). The eviction cascade (PCO → SCO → TCO) moves data between L2s, not between separate arrays. Zero extra die area for L3/L4.Misconception: that Telum has hidden separate L3/L4 arrays. It doesn't — L3 and L4 are virtual (logical) tiers built from L2 victim installs, tracked by state tags. This is the core innovation: building a tiered hierarchy algorithmically from disaggregated building blocks.
Problem 5 · medium
The paper says MCS bits reduce coherence broadcast scope. Give a concrete example: a line is shared only between L2-0 and L2-3 (on-chip). Core-0 wants to fetch it exclusively. What scope does the coherence broadcast use?
Show answer
The MCS bits indicate "on-chip only" sharing. So the exclusive fetch broadcasts only on-chip (to the eight L2s on this chip), not to the drawer or globally. L2-3 downgrades or invalidates its copy; L2-0 gets exclusive. Without MCS, a traditional MOESI protocol would have to do a global broadcast to be safe (in case the line was shared off-chip), adding latency.Misconception: that all exclusive fetches go global. MCS bits let Telum limit the scope to where sharing is actually observed, reducing latency for local exclusive fetches. The bits upgrade dynamically if sharing spreads.
Problem 6 · hard
The paper says lateral persistence increases communication overhead (Lesson #3) but that L2 capacity growth mitigates it (Lesson #5). Explain the trade-off using the 28w-Rel-DB workload (High RNI).
Show answer
Lateral persistence adds overhead: each eviction is a PCO/SCO/TCO cascade (ring broadcast, diagonal merge, victim install) instead of a simple write-to-separate-L3. But High-RNI workloads are capacity-starved, so the +9–11% gain from doubling L2 size (which also doubles L3/L4) more than offsets the overhead. For Low-RNI workloads (less capacity-sensitive), the overhead might dominate, which is why one workload (32w-Mic-Tran) shows no gain from cache growth.Misconception: that bigger caches always help. They help High-RNI (cache-bound) workloads; Low-RNI (core-bound) workloads need other optimizations (frequency, latency, prefetch). The trade-off is: overhead vs capacity gain, and it depends on workload sensitivity.
Problem 7 · medium
Telum II has 10 L2s (8 cores + DPU + isolated). When the DPU is idle, does its L2 contribute to virtual L3? Explain how.
Show answer
Yes. When the DPU is idle, its L2 accepts PCOs from peer L2s (via the activity counter algorithm) and acts as virtual L3 capacity for other cores. The adaptive mid-LRU → MRU policy lets the DPU's L2 fill with victims when it's not being used locally, so all 36 MB contribute to on-chip capacity. When the DPU becomes active, it reclaims its L2 by evicting victims (which cascade to SCO/TCO), and the L2 reverts to local use.Misconception: that the DPU's L2 is wasted when the DPU is idle. In Telum, idle caches are always useful as virtual L3/L4, which is the key advantage over traditional private L2 designs.
Problem 8 · easy
If you double the number of L2s on a chip (8 → 16) but keep L2 size constant (32 MB), what happens to L3 and L4 capacity?
Show answer
L3 doubles: 16×32 MB = 512 MB (was 256 MB). If the drawer still has 8 chips, L4 doubles: 8×512 MB = 4 GB (was 2 GB). Doubling the number of L2s doubles all tiers, at constant L2 size. This is the growth path Telum II takes (10 L2s vs Telum's 8).Misconception: that you have to grow L2 size to grow L3/L4. You can also grow the number of L2s, which has the same multiplicative effect. Both levers (size and count) work independently.
Quick answers

Frequently asked questions

Twelve questions newcomers ask most. Tap any to expand.

Q1What does "lateral" persistence mean in this context?▾
"Lateral" means evictions flow sideways (to peer L2s on the same chip or drawer) rather than downward (to a separate L3 or L4 array). A PCO (Primary Castout) is a sideways eviction to a peer L2 on-chip; an SCO (Secondary Castout) is a sideways eviction to a peer chip on-drawer. This lateral flow builds virtual L3/L4 from L2 victim installs, with no separate arrays.
Q2Why doesn't Telum just build a traditional separate L3 array?▾
Two reasons: (1) Loss of EDRAM at 7nm made a dense L3 array unaffordable in die area (Lesson #10). (2) Even with EDRAM, a traditional L3 wastes area when cores are idle (their private L2s sit unused). Lateral persistence reclaims that idle capacity as virtual L3, making all L2s useful regardless of core activity. Plus, it's modular: doubling L2 size doubles all tiers, which a traditional design can't match.
Q3How is this different from a victim cache in a traditional design?▾
A traditional victim cache is a small buffer between L1 and L2 (or L2 and L3) that holds recent evictions from the lower tier. Telum's lateral persistence is a cluster of victim caches (the peer L2s) that collectively act as a larger tier (virtual L3/L4). The key difference: Telum's "victim cache" is 256 MB (sum of peer L2s), not a few KB, and it's built from the same L2 arrays already present for local use.
Q4What are the directory state tags (local / on-chip persisted / on-drawer persisted) for?▾
They track logical tier membership without a separate L3/L4 array. A line in "local" state was fetched by this core → L2. A line in "on-chip persisted" state is a peer's eviction (PCO) → virtual L3. A line in "on-drawer persisted" state is a cross-chip eviction (SCO) → virtual L4. The state determines eviction behavior: local/on-chip can be invalidated or sent sideways; on-drawer/modified must be written back (TCO to memory).
Q5Why use activity counters instead of just LRU on the cache lines themselves?▾
Activity counters implement LRU at the scope level (whole caches), not the line level. The goal is to send evictions to the least-active cache, balancing load across peer L2s. Line-level LRU (within each L2) still exists for choosing victims locally, but the cross-cache eviction target is chosen by comparing activity counters (fetch misses + eviction installs), which is cheaper than tracking global line-level LRU across all L2s.
Q6What is the "diagonal L2" and why does it matter?▾
The diagonal L2 is the cache opposite the requester on the ring (e.g., if L2-0 requests, L2-4 is diagonal in an 8-cache ring). It's the convergence point where directory lookups from both ring directions (clockwise/counter-clockwise) meet and are merged into a combined response. This combined response includes: fetch hit/miss state, activity counter values, and the chosen eviction target (lowest activity count). The diagonal position minimizes max latency for both directions to converge.
Q7Why adaptive mid-LRU insertion instead of always MRU or always mid-LRU?▾
Always MRU would thrash out an active core's working set (victims would displace hot lines). Always mid-LRU would waste idle capacity (if the core is idle, the lower LRU positions sit unused). Adaptive mid-LRU switches to MRU when the lower positions fill up, signaling the core is idle or under-utilizing that row. This balances two goals: protect active cores (mid-LRU) and make full use of idle capacity (MRU fill).
Q8How does lateral persistence scale to many cores (e.g., 64 in a drawer)?▾
Each L2's activity counter is local; the diagonal merge compares them all and picks the lowest in O(1) time (just a comparison tree). The ring bandwidth (160 GB/s per direction on-chip, 33 GB/s per link off-chip) is sized for peak lateral traffic. The paper shows the algorithm scales to eight chips (64 L2s) with acceptable overhead. Telum II extends to 10 L2s per chip (80 total on-drawer) with the same nominal latency, proving scalability.
Q9What happens if all activity counters approach saturation?▾
When one counter approaches its max value, all counters in the scope are right-shifted (multi-bit decrement). This preserves relative magnitude — if L2-0 had 1200 and L2-3 had 600, after right-shift they'd be 600 and 300 respectively (same 2:1 ratio). The absolute value doesn't matter; only the relative ordering for LRU. This prevents overflow while keeping the counters meaningful.
Q10Can Telum do lateral persistence across drawers (inter-drawer L5)?▾
Not in z16 or Telum II. The paper (Section II.D) explicitly says lateral caching is not extended across drawers, to avoid configuration-dependent behavior (a 1-drawer vs 4-drawer system would perform differently). TCO (Tertiary Castout) always targets memory, not a peer drawer's L2s. The paper notes this could be done if desired (Lesson #11), but it's not in the current design.
Q11How does MCS (Multi-Copy Scope) reduce coherence latency?▾
MCS bits track whether a line is shared only on-chip, only on-drawer, or globally. When an exclusive fetch needs to invalidate sharers, the hardware broadcasts only to the observed sharing scope, not globally every time. For example, if MCS says "on-chip only," the broadcast stays on-chip (eight L2s), not drawer-wide (64 L2s) or global (256 cores). This is a latency win for local exclusive fetches, common in SMP workloads.
Q12What's the practical impact of the 10–15% capacity loss from duplicate entries?▾
The paper (Section II.F) notes that line sharing across L2s reduces effective unique content by 10–15% at the L3 layer (less at L4 due to de-duplication). So 256 MB physical L3 might hold ~220–230 MB unique data. This is the cost of using a victim-cache architecture instead of an inclusive one. But the trade-off is worth it: the area savings (no separate L3 array) and idle-cache reuse more than offset the duplication loss. The performance gains (§V: +7.5–13.8%) prove it.

Lateral cache persistenceThe algorithm: push evictions sideways to peer L2s (not down to a separate L3 array), building virtual L3/L4 from L2 victim installs.

TCO (Tertiary Castout)Final eviction to memory from the on-drawer scope (write-back path).
Activity counterPer-L2 counter tracking fetch misses + eviction installs; lowest count = LRU target for lateral eviction.
Diagonal L2The cache opposite the requester on the ring; where combined response (fetch state + activity counters + eviction target) is merged.

Supporting terms
Virtual L3 / L4Logical cache tiers built from lateral eviction flow, not physically separate arrays. L3 = sum of on-chip L2s, L4 = sum of on-drawer L2s.
Victim cacheA cache that holds only evictions from another cache, not original fetches. Telum's peer L2s are victim caches for each other.
Combined responseThe merged coherency state + activity counters + eviction target, created at the diagonal L2 and broadcast back to all peers.

Star mesh (on-drawer)Fully interconnected eight-chip topology; each chip has off-chip links to all peers (33 GB/s per link).

Speculative memory accessAccessing DIMM in parallel with drawer coherence broadcast, using partial info; memory data dropped if cache hit occurs.
DPU (Data Processing Unit)On-chip accelerator in Telum II for I/O; gets its own L2, which contributes to L3 when idle.
Telum IISecond-gen Telum (z17): 10 L2s (8 cores + DPU + isolated), 36 MB L2, 360 MB L3, 2.88 GB L4, 5.5 GHz.
