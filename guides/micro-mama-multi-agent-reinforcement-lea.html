MICRO 2025 · A Field Guide

When every prefetcher shouts "more bandwidth," who listens?

Eight cores, one shared memory bus. Each runs an RL-trained prefetcher that learned "aggressive = fast" on a single core — but now they're competing, and what's good for one harms the rest. This is the story of µMama, a lightweight supervisor that teaches them to cooperate.

📄 "Micro-MAMA" · Block, Gerogiannis, Torrellas 🎓 Assumes RL basics + cache prefetching ⏱ ~25 min read
🤝

When bandwidth is shared, local rewards drive competition and agents converge to Nash equilibria that hurt the system. Solve it with a global supervisor that tracks high-performing joint actions and guides local agents toward cooperation.

Seven concepts the paper assumes

1What prefetching is and why it matters▾

Main memory is slow — hundreds of cycles away. A prefetcher guesses which data you'll need soon and fetches it early, so when the CPU asks for it, it's already in the fast L2 cache. Miss? 400 cycles. Hit? ~10 cycles. Good prefetching = big speedup.

Prefetchers have aggressiveness knobs: how many blocks to fetch (degree), which patterns to chase (stride, stream, next-line). An aggressive policy prefetches lots; a conservative one fetches little. On a single core, aggressive often wins. On a multicore sharing one memory bus, aggressive from everyone creates contention.

Now you're ready: µMama is about coordinating prefetchers so they don't all choose "aggressive" and jam the shared bus.
2Online RL and the explore/exploit tradeoff▾

Reinforcement Learning = an agent takes actions, observes rewards, and learns which actions maximize reward. Online RL = learning at runtime, without prior training. The agent must balance exploration (try new actions to learn) and exploitation (stick with known-good actions).

For prefetching: the agent's actions are prefetcher policies (e.g., "degree 4 streamer + stride off"), the reward is the core's normalized IPC, and the agent runs live, adapting to workload phases as they happen.

Now you're ready: the paper's baseline is Micro-Armed Bandit, an online RL prefetcher that works great on one core but competes when multiple cores run it.
3Multi-Armed Bandits and UCB▾

A Multi-Armed Bandit is the simplest RL model: rewards depend only on the action you take (the "arm" you pull), not on some underlying state. The Upper Confidence Bound (UCB) algorithm tracks each arm's average reward plus a bonus for infrequently-tried arms, choosing the arm with the highest sum.

For time-varying workloads, Discounted UCB (DUCB) adds a "forgetfulness" parameter γ that decays old rewards, so the agent adapts to phase changes. The Bandit prefetcher uses DUCB with 17 arms (combinations of next-line/stride/stream degrees).

Now you're ready: µMama keeps these local DUCB agents but adds a supervisor to coordinate them, solving the competition problem.
4Multi-Agent RL and Nash equilibria▾

Multi-Agent RL (MARL) = multiple agents in one environment; each agent's reward depends on the joint action (what all agents do together). In general-sum games, one agent can improve its reward at another's expense. A Nash equilibrium is a joint action where no single agent can unilaterally improve by changing only its own action.

The problem: Nash equilibria aren't always optimal for the system. Classic example (Prisoner's Dilemma): both agents defect even though both cooperating would yield higher total reward. In multicore prefetching, each agent prefers "aggressive" (good for its core), but all-aggressive creates contention and a worse system outcome than a mix of aggressive and conservative.

Now you're ready: uncoordinated Bandit agents converge to contentious, globally suboptimal equilibria. µMama breaks the equilibrium by guiding them toward cooperative policies.
5Credit assignment in MARL▾

The credit assignment problem: when an agent sees a reward, how much was due to its own action vs. the actions of others? Naïve fix: give every agent the same global reward (e.g., system throughput). But now an agent can't tell if a reward change came from switching its own policy or from another agent's shift — signal is diluted by noise from co-runners.

This gets worse with more agents and more actions. Trying to learn from a global reward without knowing others' actions is like diagnosing a car problem when five parts changed at once.

Now you're ready: µMama solves this by tracking joint actions (combinations of all agents' policies) explicitly in a small cache, so it knows which full configurations work, sidestepping the credit assignment mess.
6Weighted Speedup (throughput metric)▾

Weighted Speedup (WS) measures multicore throughput: WS = Σ (IPC_multicore / IPC_alone). Each core contributes its speedup relative to running solo without prefetching. Higher WS = better system throughput, but it doesn't care about fairness — one core can get a huge speedup while others suffer.

µMama can also optimize for Harmonic Mean Speedup (HS), which balances throughput and fairness (lower if any core is starved). By changing only the reward function, the same hardware targets different goals — showing µMama's flexibility.

Now you're ready: µMama estimates WS at runtime (using a heuristic for IPC_alone) and uses it as the global reward to guide the system toward high throughput. Switching to HS changes the tradeoff without changing the hardware.
7Temporal homogeneity and why Bandit fits▾

Temporal homogeneity = a workload's behavior is stable over short timescales (e.g., thousands of instructions), so the reward for a prefetch policy today is similar to tomorrow's. This justifies the Bandit model (rewards depend only on the action, not some hidden state). Workloads have phases, but within a phase, the same policy yields similar rewards.

That's why Discounted UCB works: it forgets old data (via γ) to adapt when phases shift, but exploits stability within a phase to quickly identify good policies without needing a huge state space.

Now you're ready: µMama inherits this property. Timesteps are defined by L2 accesses (not cycles) so all workloads see policy effects before re-deciding. The JAV cache tracks joint actions that work now, and decay handles phase changes.

The whole story in one place

µMama in a nutshell

The problem, the fix, and the results — all in under 500 words.

When multiple cores share a memory bus, their prefetchers compete for bandwidth. The Micro-Armed Bandit prefetcher uses online RL to pick policies (combinations of stride/stream/next-line degrees) that maximize its core's IPC. On a single core, this works brilliantly — Bandit learns to be aggressive when it helps. But on a multicore, every Bandit learns "aggressive = good reward" because each only sees its own core's IPC. The result: all cores prefetch aggressively, memory contention spikes, and the system converges to a Nash equilibrium where no single agent wants to back off (doing so would hurt its own core) — even though the system would be faster if some agents dialed back.

This is a classic MARL problem. Naïve fixes fail: giving all agents a shared global reward (e.g., system throughput) creates a credit assignment problem — agents can't tell if a reward change came from their own action or from another agent's shift, diluting the signal. Treating the whole system as one giant agent with 17^8 = 6 billion joint actions is intractable for initial exploration: 4 cores may take almost a second, 8 cores may take almost a day, and 16 cores millennia.

µMama's solution: keep the local Bandit agents (they're good at exploring promising policies for their own cores), but add a lightweight supervisor that tracks high-performing joint actions (combinations of all cores' policies) in a tiny JAV cache. An arbiter (itself a 2-action DUCB agent) decides once every T_arbit timesteps (evaluation uses T_arbit = 5): "let the local agents explore freely" or "enforce the best joint action from the JAV cache." This balances exploration (local agents find good individual policies) and exploitation (the JAV cache identifies which combinations work system-wide).

The JAV cache is small (2 entries in the paper's eval) because it only stores actions it's tried, not all possible combinations — and it's greedy (always picks the highest-reward entry), so it converges fast. To further help cooperation, µMama identifies "low-importance" cores (those contributing little to system throughput due to high contention) and gives them the global reward, encouraging them to adopt conservative policies that free bandwidth for others. This sidesteps credit assignment for the cores that matter most.

The payoff: on 52 eight-core workloads (SPEC, Ligra, PARSEC traces), µMama improves Weighted Speedup by 2.1% over uncoordinated Bandits when optimizing for throughput, and Harmonic Mean Speedup by 10.4% when optimizing for fairness. The gain grows on more bandwidth-constrained systems (up to 2.56% with tight bandwidth). Hardware cost: the 2-entry JAV cache is 42 bytes (storing 2 × (40-bit action + 64-bit reward + 64-bit counter)), separate from arbiter tables; ~2% of local agent storage. Communication: ~200-cycle round-trip latency to the central µMama unit, with timestep synchronization on the critical path, but most computation is hidden via pipelining.

By simply changing the reward function (WS → HS or a weighted mix), the same hardware explores the throughput/fairness tradeoff, making µMama flexible enough for both HPC (maximize throughput) and cloud (balance fairness) environments.

🍯 Key takeaways

  • Uncoordinated RL prefetchers on a multicore converge to contentious, globally suboptimal equilibria.
  • Shared rewards create credit assignment problems; treating the system as one agent explodes the initial exploration time to impractical durations (days for 8 cores, millennia for 16).
  • µMama uses local agents for exploration + a JAV cache for joint-action tracking + an arbiter to switch between them every T_arbit timesteps.
  • Low-importance cores get global rewards → encourages them to free bandwidth for others.
  • Result: +2.1% WS (throughput), +10.4% HS (fairness), 42 bytes JAV cache (2 entries), timestep synchronization on critical path.
  • Change the reward → change the tradeoff (WS vs HS) without changing hardware.
  • Gains are higher when bandwidth is tighter (up to +2.56% with DDR4-1866 single-channel).
Five everyday pictures

How to think about µMama

Each analogy stands alone. Every card ends with where it breaks.

The core problem · Nash equilibrium

Three taps, one pipe

Three faucets share one water main. Each wants maximum flow for its sink. Turn all taps fully open → pipe pressure drops, everyone gets less than if some throttled back. But nobody wants to throttle (unilaterally closing your tap just helps the others). So all stay open — a Nash equilibrium. System-wide optimal: two moderate, one closed. But no single tap volunteers to close because it only cares about its own sink.

Where it breaks: real taps can't sense pressure or adjust in milliseconds. µMama's agents adapt every ~150k cycles as workload phases shift.
Credit assignment · Shared reward

Two cooks, one dish

Two chefs share one pan. Taste the result: "pretty good." But was it chef A's spice or chef B's heat adjustment that helped? If you give both the same score (shared reward), they can't tell. Chef A switches spice; B simultaneously changes heat → new taste. A can't isolate whether its change mattered. This is the credit assignment problem. µMama solves it by tracking specific combinations (A's spice + B's heat) explicitly, so it knows which pairings work.

Where it breaks: real cooking has state (pan temperature). Bandit assumes rewards depend only on the action, not state — valid when workloads show temporal homogeneity.
JAV cache · Joint action tracking

Recipe cards for combos

You can't try every combination of eight agents' techniques (17^8 = 6,975,757,441 joint actions). But you can keep a small box of index cards with the best combos you've actually tried: "Agent 0 method 5 + Agent 1 method 2 + ... = delicious (score 9)." When you want to cook, check the cards for the highest-scoring recipe. That's the JAV cache: a tiny table (2 entries in paper) of joint actions you've tested, always picking the best. Local agents explore; the cache remembers what worked.

Where it breaks: real recipes have many ingredients. The JAV cache only stores actions tried by the local agents, so it relies on them to explore good regions of the space. Bad local exploration → bad JAV entries.
Arbiter · Explore/exploit switch

A two-sided coin

Every few rounds, flip a weighted coin: heads = let local agents try new things (explore); tails = use the best recipe from the cards (exploit). The coin is itself learning: if exploiting (using JAV entries) keeps giving high rewards, bias the coin toward tails. That's the arbiter, a 2-action DUCB agent that balances exploration (local agents) and exploitation (JAV cache). In the paper's traces, it picks JAV ~67% of the time on 8-core workloads.

Where it breaks: a coin doesn't learn from outcomes. The arbiter's "coin" is UCB-weighted by past rewards, so it adapts to whether exploration or exploitation has been paying off lately.
Global rewards · Low-importance cores

Supporting actors

In a play, the star gets spotlight-based direction ("your scene was strong"), but supporting actors are told how the whole show went. If their scene drags, they learn to trim it, not expand. µMama identifies low-importance cores (those contributing little to system speedup due to contention) and gives them the global reward. They learn "my aggressive prefetching hurts the system" and adopt conservative policies, freeing bandwidth for high-importance cores. Threshold: cores with ˜S_MP < θ_global.

Where it breaks: real actors have egos. µMama's agents are purely reward-maximizing, so giving them a global reward immediately shifts their incentive. But it only helps if the low-importance cores are correctly identified — the heuristic for ˜S_MP (Eq. 4) is approximate.
The full explanation

How µMama actually works

Section by section, with key equations and architecture diagrams.

1 · The problem — Bandit agents compete

On a single core, the Micro-Armed Bandit prefetcher [17] uses DUCB to pick from 17 policies (combinations of next-line on/off, stride degree 0/2/4/8/15, stream degree 0..15). It observes normalized IPC as reward and adapts to workload phases. It works great — often outperforms fixed heuristics.

But put multiple Bandits on a multicore sharing one memory channel → competition. Each agent only sees its own core's IPC, so each learns "aggressive prefetching = good reward" (because it individually speeds up that core). Result: all agents converge to contentious, globally suboptimal policies. Memory bandwidth saturates, and the system lands at a Nash equilibrium where no single agent wants to back off (doing so would hurt its own core's IPC).

📊
The data (Fig. 3): Bandit's total prefetch count grows almost 10× from 1-core to 8-core (vs. 8× expected), unlike Pythia/Bingo which stay flat. This indicates agents are increasing aggressiveness in the multicore environment, not dialing back. On the 4-core trace mix from Fig. 2, a better joint action (policy 1 for all cores) speeds up the system by 5% — but uncoordinated agents can't find it.

2 · Why naïve fixes fail

Shared reward: give every agent the same global reward (e.g., system WS). Now agents should cooperate. But the credit assignment problem wrecks this: when agent A sees a reward change, it can't tell if its own action caused it or if agent B's action did. The global reward is diluted by noise from other agents. On the paper's 52 4-core workloads, shared reward provides no meaningful gain (Fig. 4 shows agents fail to converge).

Single system-level agent: treat the whole multicore as one agent with a joint action space = all combinations of local policies. For 8 cores with 17 actions each, that's 17^8 = 6,975,757,441 joint actions. Just the initial UCB exploration step (trying each action once) would take impractically long: 4 cores may take almost a second, 8 cores may take almost a day, and 16 cores millennia at the paper's timestep rate (150k cycles) — far too long to exploit temporal homogeneity. A reward table for 6B actions is also intractable.

⚠
The tradeoff: local rewards drive competition; global rewards blur credit; joint-action enumeration explodes. µMama needs a middle path.

3 · µMama's architecture — three agents

µMama keeps the local Bandit agents but adds two lightweight RL components:

  1. Local agents (one per L2, unchanged from Bandit): DUCB with 17 arms, local IPC reward, explore promising policies. Step size = 800 L2 accesses.
  2. JAV cache (Joint Action-Value cache): small fully-associative table (2 entries in eval) storing joint actions (combinations of all cores' policies) and their average system-level rewards. Always greedy — picks the highest-reward entry. Updated after each timestep with the system reward.
  3. Arbiter: 2-action DUCB agent that decides once every T_arbit timesteps (evaluation uses T_arbit = 5): "let local agents explore freely" or "enforce the best joint action from the JAV cache." Observes system reward for its choice.

At each timestep (Algorithm 1), the arbiter picks. If "enforce JAV," µMama broadcasts the JAV's best joint action to all cores; if "local explore," each agent picks its own action via DUCB. The JAV cache is updated with the observed joint action and system reward; if full, it evicts the lowest-reward entry.

🍯
The division of labor: local agents narrow the search space (they explore ~17 actions each, not 6B); the JAV cache disambiguates credit (it tracks specific combinations); the arbiter balances exploration/exploitation globally. Together they sidestep both the explosion and the credit problem.

4 · Computing system-level rewards (WS estimation)

To optimize for throughput, µMama uses Weighted Speedup (Eq. 2): WS = Σ (IPC_opt,MP / IPC_base,SP), where IPC_opt,MP = IPC with prefetching in the multicore, and IPC_base,SP = IPC alone without prefetching. The numerator is measurable; the denominator isn't (you can't run a core alone at runtime).

µMama's heuristic (Eq. 5): approximate each core's speedup as ˜S_i = ˜S_MP_i × ˜S_opt_i, where ˜S_opt_i = r_i (the local normalized reward, already tracked by Bandit), and ˜S_MP_i ≈ (1 − δ_i / Σδ_j), where δ_i = L2 accesses that would have missed without prefetching, per instruction. Intuition: cores that miss more in L2 suffer more from multicore contention, so ˜S_MP_i is lower. The formula exploits WS's homogeneity: multiplicative constants cancel, so µMama only needs relative weights correct.

For Harmonic Mean Speedup (HS, Eq. 6), the same approximation works but the threshold for "low-importance" cores (§4.2.4) uses ∂HS/∂S_opt_i = (S_MP_i / n) · (HS/S_i)^2 (where the 1/n factor from the derivative is absorbed into θ_global).

💡
Why heuristic vs. profiling? Profiling (running each workload alone offline) gives ground-truth IPC_base,SP and boosts µMama's WS gain from +2.1% to +3.0% (Fig. 15a). But it requires software support + storage for profiles. The heuristic is pure hardware, no software involvement, and still delivers +2.1% — good enough for many cases.

5 · Global rewards for low-importance cores

Key insight: when ˜S_MP_i is very low (core i is heavily slowed by contention), increasing its S_opt_i (via aggressive prefetching) barely helps system WS, because the multiplicative weight is tiny. But that core's aggressive prefetching does create bandwidth contention for others.

Solution: if ˜S_MP_i < θ_global (threshold = 1 − 1.4/n for n cores), give agent i the system-level reward instead of its local reward. Now agent i learns to optimize for system throughput, not its own IPC. Since its own contribution is negligible, this shifts it toward conservative policies that free bandwidth for high-importance cores. On the 4-core traces, avg ~1.5 agents get global rewards; on 8-core, ~3.5.

🍯
This is a targeted fix for credit assignment: only low-importance cores get the shared reward, so the signal isn't diluted for cores that matter. High-importance cores keep local rewards and learn aggressively when it helps them, but the system as a whole converges to a cooperative policy.

6 · The JAV cache — tracking joint actions

The JAV cache is a small fully-associative table (2 entries in the paper's eval; larger for bigger systems). Each entry has three fields:

  • aField: the joint action (e.g., "core 0 policy 5, core 1 policy 10, ..."). Size = n_cores × log₂(17) ≈ 40 bits for 8 cores.
  • rField: average system reward (double float, 64 bits).
  • nField: number of times this joint action has been tried (double, 64 bits).

Total size for 2 entries: 2 × (40 + 64 + 64) = 336 bits = 42 bytes for the JAV cache (arbiter tables are separate). This is about 2% of the local agents' storage. For a 40-core system with a 64-entry JAV cache, area overhead is ~0.001% of a 628 mm² die (§4.4.1).

Operations: (1) lookup: return the action with max(rField) (Fig. 7a); (2) insert/update: check if the joint action is in the cache (fully associative lookup), update nField and rField with the new reward. If full, evict the entry with min(rField) (Fig. 7b). The cache is greedy — always exploits, never explores (exploration is the local agents' job).

🌱
Why so small? The JAV cache only stores actions that have actually been tried (not all 6B possibilities), and it evicts low-reward entries fast. This keeps it nimble: after a workload phase change, stale entries are quickly replaced. A larger cache would remember old high-reward actions longer, slowing adaptation.

7 · Hiding latency — planning one step ahead

Communication overhead: each local agent must (1) signal "ready for next timestep" to µMama; (2) send its r_i and δ_i so µMama can compute system reward; (3) receive the next timestep's instructions (local explore or joint action from JAV). This is two round-trips per timestep.

µMama hides most of this by planning one step ahead (Fig. 8). After agents finish step n, µMama immediately broadcasts "start step n+1" while agents send their r_i/δ_i from step n in parallel. µMama computes the system reward, updates JAV/arbiter, determines the policy for step n+2, and sends that to agents — all while step n+1 is running. The "critical path" (timestep synchronization: agent signals ready → µMama broadcasts start) includes the round-trip latency, modeled as 200 cycles. The rest is hidden behind the ongoing ~150k-cycle timestep.

🍯
Result: latency tolerance. Even with 200-cycle NoC round-trip, µMama adds < 0.2% to timestep length. Total NoC bandwidth: ~28 MB/s for 40 cores (27 bytes/agent/timestep) — negligible vs. modern NoC capacity (GB/s).

8 · Hyperparameters and design choices

Key parameters (Table 1):

  • Local agents: c = 0.01, γ = 0.995, step = 800 L2 accesses (same as Bandit).
  • Arbiter: c = 0.1 (more exploration), γ = 0.995, T_arbit = 5 (re-decide every 5 local timesteps).
  • JAV cache: 2 entries, γ = 0.999 (longer memory for stable joint actions).
  • Global reward threshold: θ_global = 1 − 1.4/n (more cores → more agents get global rewards).
  • Timestep sync: k_step = 5 (if one agent reaches 5× threshold before majority, force sync).

The paper scanned these in a limited range around Bandit's known-good values (c, γ within ~1 order of magnitude) on 25 4-core workloads. JAV size and T_arbit were chosen by testing feasible ranges. The final choices have intuitive justification: local agents and arbiter use lower γ (adapt faster) than the JAV cache (remember joint actions longer).

Follow the numbers, step by step

Worked examples

Four traces using the paper's real data.

① Computing system reward (WS) from local rewards

4-core system: r₀=1.0, r₁=1.2, r₂=0.9, r₃=1.1 (local normalized IPCs). δ₀=2, δ₁=5, δ₂=3, δ₃=4 (L2 misses per instruction, scaled). Estimate WS.

  1. Sum δ: Σδ = 2+5+3+4 = 14.
  2. Compute ˜S_MP for each core (Eq. 5): ˜S_MP₀ = 1 − 2/14 = 0.857; ˜S_MP₁ = 1 − 5/14 = 0.643; ˜S_MP₂ = 1 − 3/14 = 0.786; ˜S_MP₃ = 1 − 4/14 = 0.714.
  3. Compute ˜S for each core (˜S = ˜S_MP × r): ˜S₀ = 0.857×1.0 = 0.857; ˜S₁ = 0.643×1.2 = 0.772; ˜S₂ = 0.786×0.9 = 0.707; ˜S₃ = 0.714×1.1 = 0.785.
  4. Sum for WS: ˜WS = 0.857 + 0.772 + 0.707 + 0.785 = 3.121.
  5. Check threshold: θ_global = 1 − 1.4/4 = 0.65. Core 1 has ˜S_MP₁ = 0.643 < 0.65 → gets global reward next timestep.
🍯
Insight: core 1 (highest δ, most contention) contributes least to WS despite decent r₁. µMama gives it the global reward, encouraging it to adopt conservative policies that free bandwidth for cores 0/2/3.

② JAV cache insert/evict decision

JAV cache (size 2) currently holds: entry A (action [5,10,2,8], reward 3.1), entry B ([3,12,4,6], reward 2.9). New joint action [7,10,3,8] observed with reward 3.05. What happens?

  1. Check if in cache: [7,10,3,8] → fully associative lookup finds it's not present.
  2. Cache is full (2 entries). Evict the lowest-reward entry: min(3.1, 2.9) = 2.9 → evict B.
  3. Insert new entry: [7,10,3,8], reward = 3.05, n = 1.
  4. Cache now holds: A ([5,10,2,8], 3.1) and new ([7,10,3,8], 3.05).
  5. Next JAV lookup: arbiter queries → return action [5,10,2,8] (max reward 3.1).
💡
The JAV cache is greedy (always exploits best entry) and evicts aggressively (drop low-reward actions fast). This keeps it nimble: if the workload phase changes, stale entries are replaced quickly. The small size (2 entries) forces rapid turnover.

③ Arbiter decision: local vs. JAV

Arbiter has played "local explore" 100 times (avg reward 2.95) and "enforce JAV" 120 times (avg reward 3.08). T_arbit = 5 timesteps passed. Which action does the arbiter pick?

  1. Compute UCB value for "local": value_local = 2.95 + 0.1 × sqrt(ln(220)/100) ≈ 2.95 + 0.1×0.835 = 3.03.
  2. Compute UCB value for "JAV": value_JAV = 3.08 + 0.1 × sqrt(ln(220)/120) ≈ 3.08 + 0.1×0.777 = 3.16.
  3. Pick max: 3.16 > 3.03 → arbiter chooses "enforce JAV".
  4. µMama looks up JAV cache, gets best joint action (e.g., [5,10,2,8] from example ②), broadcasts it to all cores.
  5. After 5 timesteps, arbiter re-decides. If JAV's avg reward stays high, it'll keep picking JAV (~67% of the time in paper's traces).
🍯
The arbiter balances exploration (local agents try new things) and exploitation (use known-good joint actions). If JAV entries are working well, the arbiter gravitates toward them; if not (e.g., after a phase change), it lets local agents explore.

④ Communication overhead calculation

8-core system, 150k-cycle timesteps. Each agent sends 27 bytes per timestep to µMama. What's the total NoC data rate?

  1. Bytes per timestep (all agents): 8 cores × 27 B = 216 B.
  2. Timestep duration: 150k cycles @ 4 GHz = 150k / 4e9 = 37.5 μs.
  3. Data rate: 216 B / 37.5 μs = 5.76 MB/s.
  4. Scale to 40 cores: 40 × 27 B / 37.5 μs ≈ 28.8 MB/s.
  5. Compare to NoC capacity: modern NoCs (even cross-chiplet) do GB/s. 28 MB/s is < 3% of a 1 GB/s NoC — negligible.
💡
µMama's communication is lightweight because (1) messages are small (27 B = a few floats + metadata), (2) timesteps are long (150k cycles), and (3) most of the round-trip latency is hidden by pipelining (Fig. 8). Even with 200-cycle latency, it's < 0.2% overhead.

Problem 1 · easy
Four uncoordinated Bandit agents share one memory channel. Each learns "aggressive prefetching = high reward" on its own core. Explain why this is a Nash equilibrium and why it's suboptimal for the system.
Show answer
It's a Nash equilibrium because no single agent can improve its reward by unilaterally switching to a less aggressive policy — doing so would just reduce that core's IPC while the others keep consuming bandwidth. But it's suboptimal because all-aggressive creates contention; if some agents dialed back, the system WS would improve (as shown by the paper's 5% gain from switching to policy 1 for all on the Fig. 2 workload). The agents are stuck in a local optimum.Misconception: that Nash equilibria are globally optimal. They're only stable (no unilateral improvement), not necessarily best for the system.
Problem 2 · medium
Why does giving all agents a shared global reward (e.g., system WS) fail to solve the competition problem? Relate your answer to "credit assignment."
Show answer
With a shared reward, when agent A switches its policy and sees a reward change, it can't tell if the change came from its own action or from agent B (or C, D, ...) also changing. The signal is diluted by noise from other agents' simultaneous actions. This is the credit assignment problem: without knowing others' actions, A can't isolate the causal effect of its own policy. Result: agents fail to converge (Fig. 4) or converge to the wrong policy.Misconception: that shared rewards automatically induce cooperation. They do shift incentives, but blur causality, especially with multiple agents acting in parallel.
Problem 3 · medium
The JAV cache is only 2 entries (42 bytes) for an 8-core system. Why so small, and what would go wrong with a 64-entry JAV cache?
Show answer
Small size forces rapid turnover: low-reward entries are evicted fast, so when the workload phase changes, stale joint actions (that worked in the old phase) are quickly replaced. A 64-entry cache would remember old high-reward actions longer, slowing adaptation — the arbiter might keep picking a joint action that was great 10 million cycles ago but isn't anymore. The paper's γ_JAV = 0.999 helps (decay old rewards) but a huge cache still lags. Trade-off: bigger cache → more coverage of joint-action space, but slower phase adaptation.Misconception: that bigger is always better. For non-stationary environments (workloads with phases), a small fast-evicting cache can outperform a large slow one.
Problem 4 · hard
Explain the "one-step-ahead" pipelining trick (Fig. 8) and calculate: if a timestep is 150k cycles and the round-trip to µMama is 200 cycles, what fraction of the timestep is added by the critical path?
Show answer
After agents finish step n, µMama broadcasts "start n+1" immediately (critical path includes synchronization: 200 cycles). While step n+1 runs, agents send r_i/δ_i from step n (off the critical path) and µMama computes the policy for n+2 (also off critical path). By the time n+1 ends (~150k cycles), n+2's policy is ready. Only the synchronization overhead is on the critical path. Fraction: 200/150k = 0.00133 = 0.13%. The rest of the communication and computation is hidden.Misconception: that all communication latency linearly adds to timestep length. µMama overlaps communication with execution, so only the unavoidable synchronization sits on the critical path.
Problem 5 · medium
Core 0 has ˜S_MP = 0.60 (well below θ_global = 0.65). Core 1 has ˜S_MP = 0.85. µMama gives core 0 the global reward. Why, and what policy shift do we expect from core 0?
Show answer
Core 0's low ˜S_MP means it's heavily slowed by multicore contention → even with aggressive prefetching (high r₀), its contribution to WS is tiny (˜S₀ = 0.60 × r₀). But its aggressive prefetching does consume bandwidth that could help core 1 (which has higher ˜S_MP and thus higher weight in WS). Giving core 0 the global reward shifts its incentive: it now learns "my aggressive policy hurts the system" and adopts a conservative policy (lower degree). This frees bandwidth for core 1, increasing system WS.Misconception: that every core should always maximize its own IPC. In a bandwidth-constrained system, "supporting actors" (low-˜S_MP cores) should yield resources to stars (high-˜S_MP cores).
Problem 6 · hard
µMama's arbiter picks "enforce JAV" 67% of the time on 8-core workloads. Is this exploration or exploitation? Explain the explore/exploit split and why the arbiter gravitates toward JAV.
Show answer
"Enforce JAV" is exploitation (use the known-best joint action). "Local explore" is exploration (let agents try new policies). The arbiter is itself a 2-action DUCB agent: it picks the action with higher UCB value = avg_reward + c×sqrt(ln(T)/n). If JAV entries have been yielding high system rewards, the JAV action's avg_reward is high, so its UCB value dominates. The 67% figure means JAV entries are consistently good — the arbiter has learned that exploiting (using JAV) outperforms letting agents roam freely. But it still explores 33% of the time to let local agents discover new joint actions for the JAV cache to track.Misconception: that 67% JAV = too much exploitation. The arbiter's UCB formula balances this: if JAV rewards drop (e.g., phase change), the bonus term for the under-tried "local explore" action will grow, shifting the balance.
Problem 7 · medium
µMama improves WS by 2.1% (Fig. 9) but HS by 10.4% (Fig. 13b). Why is the HS gain so much larger?
Show answer
Weighted Speedup (WS) = Σ(IPC_multi/IPC_alone) rewards any core's speedup, even if others are starved. Harmonic Mean Speedup (HS) = n / Σ(1/S_i) penalizes imbalance — if one core is slowed way down, HS tanks. Uncoordinated Bandits create high unfairness (Fig. 13a): some cores grab bandwidth, others starve. µMama (especially µMama-Fair, which optimizes for HS) balances the speedups, reducing unfairness by ~30% → much larger HS gain. WS cares less about balance, so the gain is smaller.Misconception: that throughput and fairness are the same. WS maximizes total work; HS requires balanced speedups. µMama can target either by changing the reward, showing its flexibility.
Problem 8 · easy
Why can't you just run one giant Bandit agent with 17^8 = 6 billion arms covering all joint actions?
Show answer
Two reasons: (1) Initial exploration: UCB requires trying each arm once before exploitation. 6 billion arms × 150k cycles/timestep would take impractically long (4 cores may take almost a second, 8 cores may take almost a day, 16 cores millennia) — far too long to exploit temporal homogeneity (workloads change phases). (2) Storage: a table storing avg rewards for 6 billion entries is intractable (gigabytes). µMama sidesteps this by storing only 2 joint actions that have been tried, leveraging local agents to narrow the search space.Misconception: that combinatorial explosions can be brute-forced. 17^8 is feasible in theory but impractical in the timestep budget and storage constraints of real hardware.
Quick answers

Frequently asked questions

Ten questions newcomers ask. Tap to expand.

Q1What's the difference between µMama and just giving all agents a shared reward?▾
Shared reward dilutes the signal (credit assignment problem): agents can't tell if a reward change came from their own action or another agent's. µMama solves this by (1) tracking joint actions explicitly (JAV cache), so the system knows which combinations work, and (2) giving shared rewards only to low-importance cores, so high-importance cores keep clean local signals.
Q2Why does Bandit work great on a single core but compete on a multicore?▾
On a single core, "aggressive prefetching = high reward" is often true — there's no competition. On a multicore with shared bandwidth, aggressive from everyone creates contention, lowering system throughput. Each agent only sees its own core's IPC (local reward), so each learns to be aggressive (good for that core), converging to a contentious, globally suboptimal equilibrium.
Q3What's a Nash equilibrium and why does it trap the agents?▾
A joint action where no single agent can improve its reward by unilaterally changing its own action. Example: all cores aggressive. If any one agent switches to conservative, its own core slows down (lower IPC) while the others still hog bandwidth → no incentive to switch. But if some agents switched together, system WS would improve. The equilibrium is stable but suboptimal.
Q4How small is the JAV cache, really, and why not bigger?▾
2 entries in the paper's eval = 42 bytes (40-bit action tags + 64-bit rewards + 64-bit counters for the JAV cache; arbiter tables are separate). Tiny — about 2% of the local agents' storage. Why small? Forces rapid eviction of low-reward entries → fast adaptation to workload phase changes. A 64-entry cache would remember stale high-reward actions longer, slowing convergence after a phase shift. Trade-off: coverage vs. adaptability.
Q5What happens if local agents explore bad regions and never find good joint actions?▾
The JAV cache can only store joint actions that have been tried by the local agents. If the agents explore poorly (e.g., all avoid a certain region), the JAV cache won't discover optimal actions there. But the agents are DUCB with UCB's exploration bonus, so they'll eventually try undersampled actions. Also, the arbiter lets them explore 33% of the time, ensuring continued search.
Q6Does µMama help if workloads aren't bandwidth-constrained?▾
Less. Figure 11 shows µMama's gain shrinks as memory bandwidth increases (from +2.56% on DDR4-1866 single-channel to +1.5% on dual-channel DDR4-2400). When bandwidth is ample, agents' aggressive policies don't interfere as much → less competition → less room for µMama to coordinate. The gain is highest when contention is severe.
Q7How does µMama handle workload phase changes?▾
Three mechanisms: (1) Local agents use γ=0.995 (decay old rewards) → adapt to new phase within ~1000 timesteps. (2) JAV cache uses γ=0.999 but is small (2 entries) → stale entries evicted fast. (3) Arbiter re-decides every 5 timesteps, so if JAV entries become stale, it shifts toward "local explore." Avg timestep ~150k cycles → phases lasting millions of cycles are tracked.
Q8Can µMama target metrics other than WS or HS?▾
Yes — change the reward function. The paper shows WS (throughput), HS (fairness), and weighted mixes ((1−α)·WS + α·HS for α∈{0,0.25,0.5,0.75,1}). Figure 14 plots the throughput/fairness tradeoff for these configurations, forming a Pareto frontier. You could also target energy, latency, or custom SLOs — just define the reward accordingly. Same hardware, flexible goals.
Q9What's the area and power overhead of µMama?▾
Area: 42 bytes for the JAV cache (2 entries with action/reward/counter fields), plus arbiter tables (2% of local agents' storage total). For a 40-core system with 64-entry JAV cache, ~0.001% of a 628 mm² die (§4.4.1). Power: minimal — a few float ops per timestep (150k cycles) + lightweight NoC messages (27 B/core/timestep = 28 MB/s for 40 cores, negligible). Timestep synchronization is on the critical path. The paper doesn't report explicit power, but it's dominated by the L2 caches, not µMama.
Q10Does µMama work for non-Bandit RL prefetchers (e.g., Q-learning, deep RL)?▾
Directly, no — the architecture assumes local agents provide discrete actions and rewards that can be sampled. But the principles (local exploration + joint-action tracking + arbiter) could extend: replace local Bandits with other RL agents (e.g., Q-learning over states), replace the JAV cache with a similar joint-state-action table, and the arbiter still decides "local vs. joint." The paper notes this as future work (§7).

Glossary & further reading

Every term in one place, then paths to go deeper.

Core terms (alphabetical)
ArbiterThe 2-action DUCB agent that decides "let local agents explore" or "enforce JAV cache's best joint action" every T_arbit timesteps (evaluation uses T_arbit = 5).
Credit assignment problemWhen an agent sees a reward, it can't isolate how much was due to its own action vs. others' actions (especially with shared rewards).
DUCB (Discounted UCB)Upper Confidence Bound with exponential decay (γ) for time-varying rewards; used by Bandit and µMama's local agents.
Harmonic Mean Speedup (HS)n / Σ(1/S_i); fairness-aware metric that penalizes imbalance (if one core starves, HS tanks).
JAV cacheJoint Action-Value cache: small fully-associative table (2 entries in paper) storing joint actions (combinations of all cores' policies) and their avg system rewards. Always greedy.
Joint actionA combination of all agents' individual actions (e.g., core 0 policy 5, core 1 policy 10, ...). The JAV cache tracks these.
Local agentOne per L2 cache; uses DUCB (17 arms) to pick prefetch policies. Gets local IPC reward (or global reward if low-importance).
Low-importance coreA core with ˜S_MP < θ_global (heavily slowed by contention); µMama gives it global reward to encourage cooperation.
MARLMulti-Agent Reinforcement Learning: multiple agents in one environment; rewards depend on joint actions.
Micro-Armed Bandit (Bandit)Prior work [17]: online RL prefetcher using DUCB with 17 arms (combinations of stride/stream/next-line degrees). Works great on 1 core; competes on multicores.
Nash equilibriumA joint action where no single agent can unilaterally improve its reward by changing only its own action. Not necessarily globally optimal.
Temporal homogeneityProperty that workload behavior is stable over short timescales (e.g., thousands of instructions), justifying the Bandit model.
Weighted Speedup (WS)Σ(IPC_multi/IPC_alone); throughput metric that sums each core's speedup relative to running alone. Doesn't care about fairness.
µMamaMicro-MAMA = the paper's system: a lightweight supervisor of local Bandit agents, using JAV cache + arbiter to coordinate them.
Key equations (from the paper)

Weighted Speedup (Eq. 2): WS = Σ (IPC_opt,MP / IPC_base,SP), where IPC_opt,MP = with prefetching in multicore, IPC_base,SP = alone without prefetching.

µMama's WS approximation (Eq. 5): ˜S_i ≈ (1 − δ_i/Σδ_j) × r_i, where δ_i = L2 misses per instruction, r_i = local normalized IPC.

Global reward threshold (§4.2.4): θ_global = 1 − 1.4/n for n cores. If ˜S_MP_i < θ_global, agent i gets system reward instead of local reward.

DUCB value (Eq. 1, from UCB): value(a_i) = ˆr_i + c·sqrt(ln(T)/n_i), where ˆr_i = avg reward, n_i = times arm used, T = global timestep, c = exploration constant.
