When your GPU is drowning in KV cache, borrow your neighbor's memory

LLM inference hits a wall when the KV cache exceeds GPU memory. Aqa pages that state over NVLink to a neighbor GPU's spare HBM instead of crawling through PCIe to DRAM — turning a 9× bandwidth gap into a 20× responsiveness win.

Vijaya Kumar, Antichi & Singh · Cornell / Politecnico di Milano GPU memory tiering · NVLink paging · CFS for LLMs ~30 min read
🔗

Before you begin

Eight ideas the paper assumes you know

Each card clears up one concept Aqa builds on. Tap any to open. Read in order first, or jump to whatever is unfamiliar.

producer GPUs (spare HBM) consumer GPUs (memory-starved) Aqa machinery PCIe / baseline bottleneck
1GPU HBM and the memory wall▾

GPUs do ML inference using High Bandwidth Memory (HBM) — very fast DRAM stacked right on the GPU package. An H100 has 80 GB of HBM at ~3 TB/s bandwidth. But model weights alone can consume most of that: Llama-405B needs ~800 GB, spread across many GPUs.

Once you load the model, whatever HBM is left must hold the KV cache for every active prompt. When that remaining space runs out, inference stalls.

Now you're ready: Aqa's entire premise is that some GPUs have spare HBM while others are starved — so let the starved ones borrow from the rich ones.
2The KV cache — the growing memory hog▾

During autoregressive inference, each transformer layer caches the key and value vectors of every token seen so far. This KV cache grows with sequence length and batch size, and can consume many GB per prompt. For a 1M-token context, it dwarfs the model weights.

When multiple prompts are being served simultaneously, their KV caches compete for the same finite GPU memory. A sudden burst of requests can exhaust it entirely.

Now you're ready: the KV cache is the specific tensor Aqa pages out to reclaim GPU memory — it's the "inference state" being offloaded.
3PCIe vs NVLink — the bandwidth asymmetry▾

GPUs connect to host CPU/DRAM via PCIe (50 GB/s for Gen5, 25 GB/s for Gen4). But GPUs connect to each other via NVLink (450 GB/s on H100, 300 GB/s on A100) — roughly 9× more bandwidth. This asymmetry is the hardware fact Aqa exploits.

Now you're ready: when you need to page KV cache off a GPU, sending it to a neighbor GPU over NVLink is 9× faster than sending it to DRAM over PCIe. That's the whole trick.
4Paging & preemption in LLM serving▾

When GPU memory fills up, serving engines like vLLM can page out a prompt's KV cache to host DRAM, freeing GPU memory for another prompt. Preemption is the act of pausing one prompt and saving its state so another can run. This is the GPU analog of OS context-switching.

The problem: paging to DRAM over PCIe is slow, so preemptive scheduling tanks throughput. vLLM avoids preemption by using admission control (queueing), which tanks responsiveness instead.

Now you're ready: Aqa makes preemption cheap by paging over NVLink instead of PCIe, so you get both responsiveness and throughput.
5Scale-up domains and NVSwitch▾

A scale-up domain is the set of GPUs connected by high-bandwidth NVLink/NVSwitch on a single server (8 GPUs on DGX H100) or rack (72 GPUs on NVL72). Within this domain, any GPU can access any other's memory at full NVLink speed. Outside it, you're on the datacenter network.

Now you're ready: Aqa limits memory offloads to within a scale-up domain — it's fast precisely because it never crosses the slow boundary.
6TTFT and TPOT — the responsiveness metrics▾

TTFT = Time To First Token: how long until the model starts responding. TPOT = Time Per Output Token: how fast tokens stream once started. Users abandon services after ~3 seconds of unresponsiveness, so TTFT matters enormously.

Batch-processing (vLLM default) gives low TPOT but high TTFT during bursts. Fair scheduling gives low TTFT but high TPOT due to paging overhead. Aqa targets low on both.

Now you're ready: the paper's headline numbers — 20× TTFT improvement, 4× throughput — are measured in these metrics.
7Completely Fair Scheduling (CFS)▾

Linux's CFS gives every process fair CPU time by context-switching between them in time slices. Aqa adapts this to GPU inference: each prompt gets a time slice of GPU compute, with its KV cache paged in/out at slice boundaries. This prevents head-of-line blocking where one long prompt starves everything behind it.

Now you're ready: Aqa's CFS for GPUs is only practical because NVLink paging makes the context-switch overhead small enough to be worthwhile.
8Tensor parallelism and model sharding▾

Models too large for one GPU are split across multiple GPUs. Tensor parallelism (TP) splits individual layers across GPUs on the same server (they communicate per-layer over NVLink). Pipeline parallelism (PP) splits layers across servers.

TP shards share the NVLink fabric, so Aqa must be careful: consumer traffic on NVLink could interfere with TP all-reduce traffic. The paper shows the overhead is only ~5% because TP messages are small and NVLink is idle during computation.

Now you're ready: Aqa-placer explicitly avoids oversubscribing NVLink bandwidth when placing producers and consumers together.
The whole story in one place

Aqa in a nutshell

The problem, the insight, and how three components solve it.

LLM inference is memory-capacity-bound. When a burst of requests hits a GPU serving an LLM, the KV cache overflows HBM. Today's serving engines (vLLM, FlexGen) handle this by either queueing new requests (responsive TPOT, terrible TTFT) or paging to DRAM over PCIe (better TTFT, terrible TPOT). Both lose.

Aqa's insight: in a multi-GPU server, not all GPUs are memory-starved at the same time. Vision and audio models saturate on compute with tens of GB of HBM to spare. Even some LLMs under SLA have >25 GB free. These are producers. The memory-starved LLMs are consumers. Let consumers page their KV cache to a producer's spare HBM over NVLink (450 GB/s on H100) instead of to DRAM over PCIe (50 GB/s) — a 9× bandwidth advantage.

Three components make this work:

Key components

  • Aqa-profiler classifies each GPU as producer or consumer by profiling memory utilization at peak throughput (or SLA-constrained request rate).
  • Aqa-placer solves an ILP to co-locate producers and consumers on the same NVLink domain, balancing memory supply/demand per server.
  • Aqa-lib exposes elastic Aqa Tensors that transparently live on a neighbor GPU's HBM. Producers can reclaim memory when their load spikes.

On top of this, Aqa builds a preemptive CFS scheduler for prompts that time-slices GPU compute, paging KV cache in/out via Aqa Tensors at slice boundaries. Because the paging is now fast, the overhead is small.

The payoff: on 8 H100 GPUs, Aqa improves TTFT by 20× over vLLM's default scheduler, and generation throughput by 4× over FlexGen on memory-constrained long-prompt inference. TPOT improves 1.5× over CFS-with-DRAM-paging. Producer throughput drops <5%.

Seven everyday pictures

How to think about Aqa

Each picture stands alone. Every card ends with an honest note about where the picture stops being true.

The core idea · Borrow thy neighbor's memory

The shared fridge in a co-working space

Your office fridge is crammed, but the design studio next door barely uses theirs. Rather than hauling food to a cold-storage warehouse across town (DRAM via PCIe), you just use their empty shelf (NVLink to neighbor GPU). Aqa is the building manager who profiles which offices have fridge space (profiler), assigns desks so full-fridge tenants sit near empty-fridge ones (placer), and handles shelf-sharing logistics (lib).

Where it misleads: a fridge shelf doesn't need to be returned urgently. In Aqa, the producer can reclaim memory when its own load spikes — the "shelf" is elastic, not permanently lent.
CFS scheduling · Fair time-slicing

Round-robin at the doctor's office

Instead of one patient monopolizing the doctor until done (FCFS batch scheduling), each patient gets a 10-minute slot before yielding. Nobody waits hours for a first appointment (TTFT). The doctor's "context switch" — reading the patient's chart — is fast if the chart is in the next room (NVLink) but slow if it's in a warehouse (PCIe). Aqa makes the chart retrieval fast, so the round-robin overhead is bearable.

Where it misleads: a doctor's context-switch is constant. In Aqa, the paging cost depends on how much KV cache must move — long prompts have larger "charts."
The bottleneck · PCIe bandwidth wall

A highway with a one-lane bridge

Your city has an 8-lane highway (NVLink, 450 GB/s) connecting neighborhoods, but the only road to the suburbs (DRAM) is a single-lane bridge (PCIe, 50 GB/s). Evacuating people (KV cache) to the suburbs during a flood (memory pressure) is painfully slow. Aqa evacuates them to a neighbor's house instead, using the highway. The bridge still exists as a fallback.

Where it misleads: a highway connects everywhere. NVLink only connects GPUs within the same scale-up domain — typically one server (8 GPUs) or one NVL72 rack.
Profiling · Producer vs consumer

A dating app for GPUs

Aqa-profiler is like a matchmaking service that profiles each GPU's "personality" — does it have memory to share (producer) or does it need more (consumer)? Image generators (Stable Diffusion) are generous memory donors who plateau on compute. LLMs under heavy load are needy consumers. Aqa-placer then arranges dates: pairing complementary GPUs on the same server.

Where it misleads: matchmaking is one-time. Aqa's profiles can change as workloads shift — a producer can become a consumer if its traffic spikes, and Aqa-lib handles the breakup (memory reclamation) gracefully.
Elasticity · Aqa Tensors

An adjustable partition wall

Two offices share a wall that can slide. When one office is empty, the wall slides over to give you more room. When they need it back, the wall slides home. Aqa Tensors are that sliding wall: they're PyTorch tensors that transparently live on another GPU, and the producer can reclaim the space by sliding the wall back when it needs memory for its own inference load.

Where it misleads: a sliding wall is instant. Reclaiming Aqa Tensors requires copying data back to DRAM (if no other producer is available), which has latency.
Placement · Co-location constraint

Seating plan at a dinner party

You want lively conversation, so you alternate introverts and extroverts at each table. Aqa-placer does the same: it seats producers and consumers at the same table (server) so they can share resources over the fast NVLink "table talk." An ILP solver (Gurobi) finds seating that balances memory supply and demand per table, converging within 5 seconds for 128 GPUs.

Where it misleads: at a dinner party, everyone stays seated. In production, workloads change and Aqa-placer may need to re-run (though this migration cost is left to future work).
Optimization · Gather-copy-scatter

Packing a moving truck efficiently

Moving tiny boxes one by one is slow (NVLink is bad at small transfers: 4 MB = 50 GB/s, 64 MB = 200 GB/s). So you gather small boxes into one big crate, move the crate in one trip, then unpack at the destination. Aqa's CUDA kernel gathers scattered KV cache tensors into a contiguous buffer, does one large NVLink copy, then scatters back. This gets close to peak bandwidth.

Where it misleads: packing a truck is free. The gather/scatter kernel uses GPU compute cycles, though these are overlapped with the copy where possible.
The full explanation

How it actually works

Section by section, tracing data from profiling to runtime reclamation.

1 · The problem — memory contention kills responsiveness or throughput

When a burst of requests arrives at a GPU serving an LLM, the KV cache overflows HBM. Today's engines offer two bad options:

StrategyTTFTTPOT / ThroughputProblem
vLLM (FCFS + queue)20s spikeGoodHead-of-line blocking
vLLM + CFS (DRAM paging)~1s>2× worsePCIe paging overhead
Aqa~1sGoodNeeds a nearby producer
📊
Concretely (Fig. 1): on an A100 serving Llama with 25 prompts at steady state, doubling the request rate for 1 minute causes vLLM's TTFT to spike 4× (to ~20s). CFS+DRAM-paging fixes TTFT but more than doubles request completion time. Aqa fixes both.

2 · Aqa-profiler — who has memory to spare?

Different ML models bottleneck on different resources. Vision and audio models (Stable Diffusion, AudioGen) saturate GPU compute with 10–30+ GB of HBM free (Fig. 3). LLMs under SLA constraints also have spare memory: Llama 3.1 8B at 5 RPS within interactive SLA leaves >25 GB free on an H100 (Fig. 7b).

For non-LLMs: increase batch size until throughput saturates, record free memory. For LLMs: binary search + linear sweep over request rates to find the highest rate within SLA, then report free memory at that rate. If free memory exceeds a threshold (e.g., 10 GB), the GPU is a producer. Otherwise it's a consumer.

3 · Aqa-placer — put producers next to consumers

Offloading only works if the producer is on the same NVLink domain. Aqa-placer solves an ILP (via Gurobi) to map models to servers, subject to constraints:

  1. One model per GPU. Each model shard maps to exactly one GPU.
  2. TP shards co-located. Tensor-parallel shards go on the same server.
  3. GPU count constraint. Total shards per server ≤ number of GPUs.
  4. Balanced memory. Minimize the worst-case memory imbalance across servers (Eq. 4).
  5. Balanced producer/consumer count. Prevent one producer from serving multiple consumers, which would oversubscribe NVLink bandwidth (Eq. 5).
💡
Scale: Gurobi reaches solutions within 2% of optimal in 5 seconds for clusters up to 128 GPUs. After server-level assignment, simple stable matching pairs producers to consumers within each server.

4 · Aqa-lib — elastic Aqa Tensors

Aqa-lib runs on each GPU and exposes a simple API: convert a PyTorch CPU tensor into an Aqa Tensor that transparently lives on a neighbor GPU's HBM. A central coordinator (REST API) tracks offers (from producers) and requests (from consumers).

Key behaviors:

  • Allocation: consumer asks coordinator for space; coordinator returns a reference to the producer GPU's memory region.
  • Reclamation: producer's Aqa-lib polls its inference load; if load increases, it tells the coordinator to reclaim memory. Consumer migrates the Aqa Tensor to DRAM.
  • Fallback: if no producer exists, Aqa-lib falls back to DRAM (matching baseline behavior).

5 · CFS for GPU inference — fair scheduling with fast paging

Aqa builds a CFS scheduler (150 LOC in vLLM) that time-slices GPU compute across prompts. The key challenge: naively time-slicing by least-generated-tokens breaks chunked-prefill optimization (Sarathi-Serve), which batches compute-bound prefill with bandwidth-bound decode for free.

Aqa's batch partitioning: given batch size b, set decode slots d to the maximum prompts that fit in GPU memory, fill prefill slots p = b - d with prompts having the fewest prefill tokens computed, fill decode slots with prompts having fewest tokens generated, then overflow remaining d slots with prefill prompts. This preserves chunked-prefill benefits while ensuring fairness.

🔧
Gather-scatter optimization: vLLM stores all prompts' KV tensors for a layer as one big tensor. Paging one prompt means many small, scattered copies. NVLink bandwidth drops from 200 GB/s at 64 MB to just 50 GB/s at 4 MB. So Aqa gathers small pieces into a temp buffer (CUDA kernel, 200 LOC), does one big NVLink copy, then scatters at the destination.

6 · Elasticity — what happens when the producer gets busy?

Producer workloads can change. Aqa-lib monitors the producer's inference load via the northbound interface. When load increases (longer queue), Aqa-lib reclaims memory: the consumer must migrate its Aqa Tensors back to DRAM. When load drops, memory is offered again.

The paper demonstrates this with Llama 8B (producer) + OPT-30B (consumer): at 60s, consumer starts long-prompt inference at 8 tok/s; at 120s, producer traffic doubles, Aqa reclaims memory, consumer throughput drops; when traffic returns to normal, Aqa re-offers memory and throughput recovers (Fig. 12).

Follow the numbers, step by step

Worked examples

Four calculations using the paper's real numbers.

① The bandwidth advantage — NVLink vs PCIe

How much faster is paging KV cache over NVLink vs PCIe on an H100 server?

  1. H100 NVLink bandwidth: 450 GB/s (via NVSwitch connecting all 8 GPUs).
  2. H100 PCIe Gen5 bandwidth: 50 GB/s to host DRAM.
  3. Bandwidth ratio: 450 / 50 = 9×.
  4. Suppose you page 10 GB of KV cache. Over PCIe: 10 / 50 = 0.2s = 200ms. Over NVLink: 10 / 450 = 0.022s = 22ms.
  5. Savings: 178ms per page-in/out cycle. At decode speeds of ~30 tok/s, that's ~5 tokens of generation saved per context switch.
💡
Insight: the 9× bandwidth gap is the hardware fact that makes everything else work. On A100 (NVLink 300 GB/s vs PCIe 25 GB/s), the ratio is 12×. Future NVL72 racks extend this to 72 GPUs.

② Profiling Stable-Diffusion-3 as a producer

How does Aqa-profiler decide SD-3 is a producer on an H100?

  1. Increase batch size from 1 until throughput saturates. SD-3 peaks at batch size ~8 images.
  2. Record free GPU memory at peak throughput: ~30 GB free out of 80 GB (Fig. 7a).
  3. Check threshold: 30 GB > 10 GB threshold → classify as producer.
  4. Report to Aqa-placer: R_m = -30 GB (negative = producer with 30 GB to offer).
📊
Why it works: SD-3 is compute-bound — more images don't need proportionally more memory because the model processes each image independently. Increasing batch past saturation gives diminishing throughput returns with no memory benefit.

③ End-to-end CFS benefit for Llama 70B

Llama 3.1 70B on 2 GPUs (TP), paired with Llama 8B + AudioGen as producers. 25 prompts at stable rate, then 2× burst for 1 minute.

  1. vLLM FCFS baseline: TTFT spikes to ~20 seconds during the burst (Fig. 8a). Prompts 25–50 are queued behind the current batch.
  2. vLLM + CFS (DRAM paging): TTFT drops to ~1s (fair scheduling works), but TPOT rises to ~100ms due to PCIe paging overhead (Fig. 8b).
  3. Aqa + CFS (NVLink paging): TTFT ~1s (same fairness benefit), TPOT stays at ~65ms — 1.5× lower than CFS+DRAM.
  4. Burst duration impact: elevated TPOT lasts from prompt 25–48 with CFS+DRAM, but only 25–40 with Aqa — faster paging means faster recovery.
  5. TTFT improvement: 20s / 1s = 20× vs FCFS baseline.

④ Long-prompt throughput with FlexGen

OPT-30B on 1 GPU, 8192-token prompts (arxiv summarization median ~7000 tokens). Model weights + KV cache exceed GPU memory.

  1. FlexGen baseline: pages layer context to DRAM over PCIe. Average throughput: ~2.5 tokens/s (Fig. 9b).
  2. FlexGen + Aqa: pages to Llama 8B's spare HBM over NVLink. Average throughput: ~10 tokens/s.
  3. Improvement: 10 / 2.5 = 4× throughput.
  4. Producer impact: Llama 8B TPOT increases by only ~4.5ms (Fig. 10b) — a small overhead on a single-GPU producer whose NVLink bandwidth is otherwise idle during decode.
💡
Why 4× not 9×: the throughput gain is less than the raw bandwidth ratio because (a) FlexGen has compute time between page operations, and (b) the gather/scatter overhead and small-transfer penalties reduce effective bandwidth. Still, 4× is substantial for offline inference.
Test yourself

Practice problems

Try each before opening the answer. Every one ends with the common misconception it's testing.

Problem 1 · easy
An H100 server has 8 GPUs, each with 80 GB HBM. NVLink provides 450 GB/s between GPUs; PCIe provides 50 GB/s to DRAM. You need to page out 20 GB of KV cache. How long does each path take, and what's the speedup?
Show answer
PCIe: 20 GB / 50 GB/s = 0.4s = 400ms. NVLink: 20 GB / 450 GB/s = 0.044s = 44ms. Speedup: 400 / 44 = 9.1×. This is the raw bandwidth ratio; effective speedup may be less due to protocol overhead and small-transfer penalties.Misconception: that the 9× bandwidth advantage translates directly to 9× end-to-end throughput improvement. It doesn't — compute time, gather/scatter overhead, and other factors reduce the effective gain to ~4× in practice.
Problem 2 · medium
A server has 4 GPUs: GPU 0 runs Stable Diffusion (20 GB free), GPU 1 runs AudioGen (20 GB free), GPU 2 runs Llama 70B shard 0 (needs 15 GB swap), GPU 3 runs Llama 70B shard 1 (needs 15 GB swap). How should Aqa-placer pair them? Why not share one producer with both consumer shards?
Show answer
Pair GPU 0 (SD, 20 GB) with GPU 2 (shard 0, needs 15 GB) and GPU 1 (AudioGen, 20 GB) with GPU 3 (shard 1, needs 15 GB). Each producer serves exactly one consumer. Sharing one producer with both shards would oversubscribe NVLink bandwidth — both shards might page simultaneously, halving effective bandwidth per shard. Aqa-placer's Eq. 5 explicitly prevents this by balancing the producer-consumer count per server.Misconception: that a producer with enough total memory can serve multiple consumers. Memory capacity isn't the only constraint — NVLink bandwidth is shared, and concurrent access degrades performance for both consumers.
Problem 3 · medium
Why does Aqa-profiler use a different strategy for LLMs vs vision/audio models? What would go wrong if you just increased batch size for an LLM until throughput saturated?
Show answer
Vision/audio models produce complete outputs (images, clips) — increasing batch size until throughput saturates is fine because partial outputs are useless. LLMs produce tokens incrementally and have strict interactive SLAs (TTFT <1s, TBT <100ms). Increasing batch size indefinitely would violate the SLA — TPOT increases with batch size, and TTFT increases as queues grow. So Aqa-profiler instead sweeps request rates to find the highest rate within SLA constraints, then reports free memory at that operating point.Misconception: that free memory at peak throughput tells you the right number. For LLMs, "peak throughput" might mean unacceptable latency — you need to profile at SLA-constrained load, which leaves more memory free.
Problem 4 · hard
Aqa's CFS scheduler partitions each batch into p prefill tokens and d decode tokens. Why does it set d to the maximum number of prompts that fit in GPU memory, rather than splitting p and d evenly?
Show answer
The key is preserving chunked-prefill optimization (Sarathi-Serve). Prefill is compute-bound; decode is memory-bandwidth-bound. Batching them together means decode tokens execute "for free" during prefill's compute. Setting d to its maximum ensures you always batch as many decode tokens as possible alongside prefill, getting the most free decode work. An even split would waste GPU compute by leaving decode slots unfilled when there are decode prompts available to fill them. The remaining slots in d overflow to prefill prompts when no decodes are pending, ensuring full GPU utilization.Misconception: that fair scheduling means equal allocation of batch slots. Aqa's fairness comes from time-slicing (who gets to run in which iteration), not from equal batch composition — the batch composition is optimized for throughput.
Problem 5 · medium
NVLink bandwidth for a 4 MB buffer is only 50 GB/s on an A100, vs 200 GB/s for a 64 MB buffer. Why does this matter for KV cache paging, and how does Aqa address it?
Show answer
vLLM stores each layer's KV tensors for all prompts as one tensor. Paging one prompt's KV cache means extracting small, scattered pieces across many layer tensors — resulting in many small NVLink copies that achieve only a fraction of peak bandwidth. Aqa addresses this with a gather-copy-scatter pattern: a CUDA kernel gathers the scattered pieces into a contiguous temporary buffer on the GPU, performs one large NVLink copy (hitting 200+ GB/s), and scatters at the destination. This is worth the extra GPU compute because the bandwidth gain is 4×.Misconception: that NVLink bandwidth is a single number. Like any interconnect, NVLink has significant transfer-size-dependent behavior — small transfers waste bandwidth on protocol overhead.
Problem 6 · hard
Llama 70B uses tensor parallelism across 4 GPUs, which means inter-GPU NVLink traffic for all-reduce between layers. If each of those 4 GPUs also serves as a producer for an OPT-30B consumer, why is the interference only ~5% on TPOT?
Show answer
Two reasons from the paper (§9.3): (1) TP all-reduce messages are small — only a few megabytes per layer boundary, far below NVLink's capacity. (2) NVLink is idle during computation — all-reduce happens only between layers, so consumer paging traffic can use the interconnect during the compute phase without contention. The combination means consumer traffic barely competes with TP traffic. The authors note the overhead could be eliminated entirely if NVIDIA exposed packet-priority APIs on NVLink.Misconception: that sharing NVLink between TP and paging traffic must cause severe interference. In practice, TP communication is bursty and short, leaving most of the link idle for paging.
Problem 7 · easy
All 8 GPUs on a server are running memory-hungry LLMs with no spare memory. What does Aqa do?
Show answer
Aqa falls back to DRAM paging over PCIe, exactly matching the baseline behavior of vLLM or FlexGen. Aqa-lib checks the coordinator for available producers; finding none, it allocates swap space on host DRAM. This is by design — Aqa falls back gracefully, matching baseline behavior when no producers exist.Misconception: that Aqa requires a producer to function at all. It doesn't — it degrades gracefully to the status quo when no producers exist. Its benefit is opportunistic.
Problem 8 · medium
Connect Aqa to your tiering/CXL research: how is Aqa-lib's producer/consumer memory sharing analogous to OS-managed memory tiering between DRAM and CXL-attached memory?
Show answer
Both systems exploit a memory tier hierarchy with different bandwidth/latency characteristics. In OS memory tiering (your research), the kernel promotes hot pages from slow CXL/NVM to fast local DRAM and demotes cold pages the other way. In Aqa, KV cache tensors are "promoted" from slow DRAM (via PCIe) to fast neighbor-GPU HBM (via NVLink). Both rely on profiling access patterns to decide placement, and both face the elasticity challenge — the fast tier's capacity can change (CXL memory can be shared across hosts; producer GPU memory can be reclaimed). The key difference: OS tiering is page-granularity and transparent; Aqa operates at tensor granularity with application-level control over prefetch and eviction order, which avoids the CUDA unified memory problem of evicting model weights instead of KV cache.Misconception: that GPU memory offloading and CPU memory tiering are unrelated problems. They share the same core challenge — placing data on the right tier based on access patterns and bandwidth constraints — just in different parts of the system stack.
Quick answers

Frequently asked questions

Twelve questions, tap to expand.

Q1What exactly is an "Aqa Tensor"?▾
A PyTorch tensor whose backing storage transparently lives on a different GPU's HBM (or falls back to host DRAM). Aqa-lib provides functions to convert CPU tensors to Aqa Tensors and retrieve GPU/DRAM pointers; serving-engine integration is small (<50 LOC) but explicit. Think of it as a virtual-memory abstraction for GPU memory across GPUs.
Q2Why not just use CUDA Unified Virtual Memory?▾
CUDA UVM provides a single address space but its default prefetching is application-agnostic — it might evict model weights instead of KV cache, or page in data the model won't need next. Aqa gives the serving engine explicit control over what to page and in what order, plus it opportunistically migrates offloaded data to faster GPU memory rather than relying on the CUDA driver's policy (§10).
Q3How much code change does Aqa require in vLLM?▾
Aqa-lib integration: <50 LOC (swap PyTorch CPU tensors for Aqa Tensors). CFS scheduler: 150 LOC. Gather/scatter CUDA kernel: 200 LOC. Total <400 LOC in vLLM v0.5.3.
Q4What happens if the producer's load changes?▾
Aqa-lib polls the producer's northbound interface. When load increases (queue grows), it reclaims memory: consumer's Aqa Tensors are migrated to DRAM. When load drops, memory is re-offered. Fig. 12 shows this elasticity in action — consumer throughput drops during the spike but recovers fully after.
Q5Does Aqa hurt producer performance?▾
Minimally. Image/audio producers show <5% throughput difference (Fig. 10a). LLM producers see ~4.5ms TPOT increase (Fig. 10b). Distributed Llama 70B on 4 GPUs sees ~5% TPOT overhead even with 4 OPT-30B consumers actively paging (Fig. 11a).
Q6Can one producer serve multiple consumers?▾
Aqa-placer deliberately prevents this. Sharing a producer with multiple consumers could oversubscribe NVLink bandwidth, degrading both consumers. The design maps one producer to one consumer. Future work may explore controlled sharing with bandwidth isolation.
Q7How does Aqa handle distributed (TP) models as consumers?▾
Each TP shard is treated as a separate consumer and matched with its own producer. The paper evaluates Llama 70B on 2 GPUs (TP) paired with Llama 8B + AudioGen as producers. Both shards' Aqa-lib instances independently manage their swap space.
Q8What datasets are used in evaluation?▾
ShareGPT for LLMs (median ~2000 tokens), arxiv summarization (median ~7000 tokens) for longer prompts, parti-prompts for vision models, and AudioGen's default prompts for audio. The ShareGPT + interactive SLA setup follows Sarathi-Serve's methodology.
Q9Why 20× TTFT improvement but only 4× throughput?▾
Different experiments. The 20× TTFT comes from CFS vs FCFS during bursts — most of the gain is from not queueing (fair scheduling), with NVLink making the paging overhead small. The 4× throughput is from FlexGen long-prompt inference, where every decode iteration requires paging — here the gain is purely from faster paging (NVLink vs PCIe). Both are real but measure different things.
Q10Does Aqa work across servers?▾
Currently, no. Aqa limits offloads to within a single NVLink/NVSwitch domain (one server). Cross-server would mean using the datacenter network, which has lower bandwidth. However, NVL72 racks (72 GPUs connected by NVSwitch) extend the domain to an entire rack, increasing Aqa's reach.
Q11How is this related to Infiniswap / Fastswap?▾
Infiniswap and Fastswap accelerate OS paging by moving DRAM pages across servers over RDMA. Aqa is the GPU analog: moving GPU tensors across GPUs over NVLink. The key difference is that Aqa-placer uses 1:1 producer-consumer pairing to avoid likely NVLink oversubscription (formal bandwidth guarantees are future work), while Infiniswap/Fastswap allow sharing, which can cause contention. Also, Aqa operates at the application layer with explicit control over what to page.
Q12What did the authors leave for future work?▾
Three main items: (1) exploring the trade-off between maintaining suboptimal placement vs re-running Aqa-placer with migration; (2) modeling NVLink bandwidth contention to provide paging bandwidth guarantees; (3) extending to larger scale-up domains (NVL72) and disaggregated serving (DistServe/Splitwise).

Glossary, further reading & source notes

Every term in one place, then a path to go deeper.

Core terms
AqaThe paper's GPU memory management framework. Three components: Aqa-profiler, Aqa-placer, Aqa-lib. Enables fast paging of KV cache over NVLink to neighbor GPU's spare HBM.
Producer GPUA GPU with spare HBM capacity (e.g., running a vision model that saturates on compute). Can offer memory to a consumer via NVLink.
Consumer GPUA GPU bottlenecked on memory capacity (e.g., LLM under high load). Needs to page KV cache off-GPU.
Aqa TensorAn elastic PyTorch tensor whose backing store transparently lives on a neighbor GPU's HBM (or falls back to DRAM). The producer can reclaim the memory when needed.
Scale-up domainThe set of GPUs connected by high-bandwidth NVLink/NVSwitch on one server or rack. Aqa limits offloads to within this domain.
CFS (Completely Fair Scheduler)Aqa's preemptive scheduler for prompts. Time-slices GPU compute across prompts, paging KV cache via NVLink at slice boundaries. Adapted from Linux CFS.
Supporting terms
KV cacheCached key and value vectors from the attention mechanism. Grows with sequence length and batch size; the primary consumer of GPU memory during LLM inference.
HBM (High Bandwidth Memory)Stacked DRAM on the GPU package. H100: 80 GB at ~3 TB/s bandwidth. The fast tier in GPU memory hierarchy.
NVLink / NVSwitchNvidia's high-bandwidth GPU-GPU interconnect. H100: 450 GB/s (all-to-all via NVSwitch). A100: 300 GB/s (peer-to-peer).
TTFT / TPOTTime To First Token / Time Per Output Token. The responsiveness and throughput metrics for LLM serving.
Chunked prefillSarathi-Serve's optimization: batch compute-bound prefill tokens with bandwidth-bound decode tokens for free decode work.
Tensor parallelism (TP)Splitting individual model layers across GPUs on one server. Requires NVLink all-reduce between layers.
Admission control / FCFSvLLM's default strategy: batch-process admitted prompts, queue the rest. Good throughput, bad TTFT during bursts.
Gather-scatterAqa's CUDA kernel optimization: coalesce scattered small KV cache pieces into one large buffer before NVLink copy, to hit peak bandwidth.

Related papers

vLLM: Efficient Memory Management with PagedAttention medium

Kwon et al. · SOSP 2023

The serving engine Aqa builds on. Introduced paged KV cache management. Aqa replaces vLLM's DRAM swap space with NVLink-accessible Aqa Tensors.

Sarathi-Serve: Taming Throughput-Latency Tradeoff medium

Agrawal et al. · OSDI 2024

Introduced chunked prefill — batching prefill and decode tokens. Aqa's CFS scheduler preserves this optimization while adding fair time-slicing.

FlexGen: High-Throughput Generative Inference medium

Sheng et al. · ICML 2023

Offline inference engine that pages model context between GPU and DRAM. Aqa replaces its PCIe paging with NVLink for 4× throughput.

Memory Harvester: Hierarchical Unified Virtual Memory hard

Choi et al. · USENIX ATC 2022

Harvests idle GPU memory at the CUDA driver level. Application-agnostic, so it can't control eviction order (weights vs KV cache). Aqa's application-layer approach avoids this.

Infiniswap: Efficient Memory Disaggregation hard

Gu et al. · NSDI 2017

The CPU-side analog: harvests DRAM across servers over RDMA for OS paging. Aqa is its GPU counterpart, with the key difference of bandwidth-isolated 1:1 pairing.

HeteGen: CPU+GPU Computation for Inference medium

2024

Extends FlexGen by using both CPU and GPU for compute. Still relies on PCIe for CPU-GPU I/O, so Aqa is complementary.
