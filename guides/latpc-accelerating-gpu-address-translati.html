Prefetching translations together, not one by one

When 32 GPU threads all need different pages, the address-translation machinery drowns. LATPC spots the hidden pattern in their chaos and solves page walks as batches—under 1% hardware cost, 47% faster.

📄 "LATPC" · Ha et al. (MICRO'25) 🎓 Assumes TLBs, page tables, GPU warps ⏱ ~25 min read
🧵

When threads in a warp request translations, capture their regularity (the stride of VPNs), batch their page-table walks into one DRAM row hit, and compress up to 32 misses into one or a few TLB MSHR entries—not 32 separate entries burning PTW and MSHR slots one at a time.

Before you begin

Seven ideas the paper builds on

Expand each to clear up the assumed background. They're short — read in order the first time.

warp-level regularity (LATPC's insight) baseline contention (PTW/MSHR exhaustion) compression & prefetch machinery prior TLB prefetchers (fail on GPUs)
1GPU virtual memory & why translation is expensive▾

Modern GPUs virtualize memory, so every global-memory access triggers a TLB lookup that translates the Virtual Page Number (VPN) into a Physical Frame Number (PFN). A TLB miss launches a page table walk: 4 levels of indirection through DRAM (hundreds of cycles each) to fetch the PTE. GPU cores (SMs) have tiny private L1 TLBs (e.g., 32 entries in the paper's simulated configuration) and share a bigger L2 TLB (1,024 entries). Misses exhaust the limited page-table walkers (16 PTWs) and TLB MSHR entries (16 per L1 TLB), stalling the whole warp until translation completes.

Now you're ready: LATPC targets exactly this bottleneck—PTWs swamped by 32 threads each touching a different page, and TLB MSHRs running out before the warp even finishes issuing all its misses.
2Page divergence: one warp, many pages▾

GPU threads in a warp often access memory by their thread index (e.g., array[threadIdx]). When the array is large, consecutive threads land on different pages. The paper defines page divergence as the number of unique pages a warp instruction touches—up to 32 (one per thread). In their workloads, 75% of warp memory instructions require multiple translations; 29% hit the max of 32. That's 32 TLB misses from a single instruction, all fighting for the same 16 L1 TLB MSHR slots and 16 PTWs.

Now you're ready: Page divergence is the problem that breaks naïve TLB prefetchers—32 scattered VPNs look random in access order, but they're actually regular if you look at thread index instead.
3TLB MSHR: the reservation table for outstanding misses▾

A TLB Miss-Status Holding Register (MSHR) tracks outstanding TLB misses so the TLB doesn't re-fetch the same PTE. Each entry holds one VPN's miss state. When an instruction with high page divergence fires, it tries to allocate up to 32 entries—one per VPN. But the L1 TLB has only 16 MSHR slots. Once those fill, subsequent misses stall (MSHR reservation failure) until an earlier walk completes and frees a slot. The paper shows MSHR reservation failures account for 53% of total translation latency (Figure 1).

Now you're ready: LATPC's compressed MSHR (LATC) lets one entry represent up to 32 VPNs, freeing slots and reducing reservation failures from 68.84% to 43.02%.
4Page table locality: PTEs cluster in DRAM rows▾

A page table is a 4-level tree in DRAM (x86-64). The leaf level (L4 PT) stores 512 PTEs in a single 4 KB row—exactly one DRAM row-buffer. If two VPNs differ only in their bottom 9 bits (512-page window), their L4 PTEs sit in the same row. The paper finds 80.20% of translations within a warp share the same L4 PT base address (Figure 9). Opening the row once and fetching multiple PTEs from it is far cheaper than 32 separate DRAM accesses, each closing/opening rows.

Now you're ready: LATPC's prefetcher (LATP) batches walks that share an L4 PT, loading multiple PTEs in one pass and exploiting row-buffer hits.
5Why prior TLB prefetchers fail on GPUs▾

CPU TLB prefetchers (Sequential, Stride, Distance) learn patterns from the time-ordered stream of TLB accesses. On a GPU, thousands of threads interleave their accesses chaotically—no stable stride emerges. The paper shows a 3D plot: TLB accesses vs time are scattered (Figure 7b), but TLB accesses vs thread index reveal a clear line (Figure 7c). Existing prefetchers see scatter → predict poorly (accuracy <80%, Figure 13). Valkyrie (the GPU TLB prefetcher from PACT'20) only replicates one fetched PTE to likely L1 TLBs—it doesn't prefetch additional entries from the page table, so it achieves only 3% speedup.

Now you're ready: LATPC avoids prediction entirely by detecting the VPN stride directly from the warp's coalesced VPN list (determined by thread index), achieving near-100% accuracy.
6Regularity within a warp: the hidden stride▾

Even for "irregular" workloads (e.g., graph traversal), the paper finds that VPNs within one warp instruction exhibit almost-strided behavior (Figure 8): on average only 1.96 distinct stride values (irregular workloads average 2.49). "Regular" workloads (e.g., matrix ops) use 1.42. A VPN stride is (VPNi+1 − VPNi); if that stride stays constant or follows a few discrete values across threads 0..31, you can represent many VPNs compactly as {base, stride, count}. This regularity is invisible to time-ordered prefetchers but obvious when you look at the coalescer's output (which is determined by thread index).

Now you're ready: LATPC's Regularity Detector scans the coalescer's VPN list, detects the stride(s), and generates prefetch requests for the missing threads' VPNs—all in one pass, no prediction.
7The two bottlenecks: PTW contention & MSHR exhaustion▾

The paper's Figure 1 breakdown shows: 39% of translation latency = page table walks (limited to 16 PTWs); 53% = MSHR reservation failures (limited to 16 L1 TLB MSHR entries). Figure 6 demonstrates that even with 2 MB large pages (which reduce TLB pressure), increasing PTWs to 64 or MSHR entries to 1,024 yields 1.38× and 1.66× speedup, respectively—proving the bottlenecks persist regardless of page size. No single fix solves both: more PTWs help walks but not MSHR contention; more MSHR entries help reservations but not PTW throughput.

Now you're ready: LATPC attacks both: LATC (compressed MSHR) squeezes up to 32 misses into 1–2 entries → reduces MSHR exhaustion. LATP (batched walks) processes multiple translations with batched L4 PT fetches → reduces PTW contention. Together they achieve 1.47× GMean speedup.

The whole story in one place

LATPC in a nutshell

The problem, the two-part fix, and why it works when prior approaches don't.

Modern GPUs virtualize memory, but high page divergence (one warp instruction touching 32 different pages) overwhelms the limited address-translation machinery. In the baseline, 32 TLB misses exhaust the 16 L1 TLB MSHR entries (causing reservation failures that account for 53% of translation latency), and 32 page-table walks saturate the 16 page-table walkers (another 39% of latency). Existing TLB prefetchers learn patterns from time-ordered TLB accesses, which on GPUs is a chaotic interleaving of thousands of threads—no stable stride, low accuracy.

LATPC's insight: although TLB accesses are disordered in time, the VPNs within a single warp instruction exhibit strong regularity (thread-indexed VPNs form almost-strided sequences, averaging 1.96 distinct strides) and locality (80.20% of VPNs share the same L4 page table base address, meaning their PTEs can be fetched from one 4 KB DRAM row). LATPC exploits both properties with two mechanisms:

  • LATC (Locality-Aware TLB MSHR Compression): extends each L1 TLB MSHR entry with a 9-bit Stride field and a 32-bit Valid Mask, so one entry can represent up to 32 VPNs (base + stride×index). This compresses up to 32 misses into 1–2 entries instead of 32, freeing MSHR slots and reducing reservation failures.
  • LATP (Locality-Aware TLB Prefetching): when a demand miss arrives, LATP's Regularity Detector scans the coalescer's VPN list (determined by thread index), detects the stride, generates prefetch requests for the full warp, and batches their L4 page-table walks. Because many share an L4 PT, one DRAM row-buffer open can fetch multiple PTEs—reducing separate row opens and PTW invocations.

Regularity Detector is a tiny state machine (0.0009 mm² in 7 nm, 1-cycle latency modeled) that processes one VPN per cycle, tracks the previous stride, and outputs triples <VPN, Stride, Index>. Demand requests have stride=0, prefetch requests have non-zero stride. The extended PW Buffer coalesces multiple walks with the same base address, and the PTW fetches multiple PTEs in batched fashion.

The payoff: LATPC reduces address translation latency to 69% of baseline and achieves a 1.47× GMean speedup across 24 workloads (up to 3.6× for atax/bicg), for under 1% area overhead (0.26 mm² on a 445 mm² GPU). Valkyrie (the prior GPU TLB prefetcher) achieves only 1.03× because it replicates one entry without prefetching more. CPU prefetchers (Sequential/Stride/Distance) achieve ≤1.01× due to low accuracy on scattered GPU access patterns. Avatar (MICRO'24, the state-of-the-art in GPU translation via speculation) achieves 1.40×, but LATPC+Avatar reaches 1.64× by resolving the bottlenecks during Avatar's background walks.

🍯 Key takeaways

  • High page divergence (32 pages/warp) saturates 16 PTWs and 16 L1 TLB MSHR entries → 92% of translation latency.
  • VPNs are scattered in time but regular in thread-index order (1.96 distinct strides) and local in page-table space (80.20% share same L4 PT base address).
  • LATC compresses up to 32 TLB misses into 1–2 MSHR entries via {base, stride, 32-bit mask} → reduces reservation failures from 68.84% to 43.02%.
  • LATP batches multiple page-table walks' L4 fetches via detected stride → exploits DRAM row-buffer hits.
  • Regularity Detector detects stride by scanning coalescer output (not time-ordered accesses) → near-100% accuracy.
  • Result: 1.47× GMean speedup, 0.26 mm² (0.06% die), 36 mW (0.02% TDP), 1-cycle modeled latency with negligible impact.
  • Works with 4 KB or 2 MB pages; orthogonal to Avatar (combining yields 1.64×).
Five everyday pictures

How to think about LATPC

Each picture stands alone. Every card ends with where the analogy breaks.

The core problem · PTW contention

Five passengers, one ticket clerk

Five passengers line up at one ticket window (the baseline: 5 warps, 16 PTWs). Each passenger hands over 32 tickets to validate—one at a time, one clerk. The queue explodes. LATPC notices all 32 tickets for one passenger are consecutive seat numbers (the stride), bundles them, and the clerk validates the whole block with one stamp (batched L4 PT fetch). The next passenger's turn arrives faster.

Where it breaks: real tickets aren't laid out in memory rows; PTEs are. The analogy undersells the DRAM row-buffer locality—fetching multiple PTEs from one row is essentially low-latency after opening it once.
The compression · LATC

Compressing a sequence into a formula

Instead of storing [2, 4, 6, 8, …, 64] (32 numbers, 32 slots), write "base=2, stride=2, count=32" (one slot). LATC does this for VPNs: one L1 TLB MSHR entry holds a 9-bit stride and a 32-bit Valid Mask (one bit per thread). When thread 5's VPN arrives, LATC checks if base + 5×stride matches—if yes, set bit 5 of the mask. Up to 32 misses compress into 1–2 entries instead of exhausting all 16.

Where it breaks: real sequences aren't always perfectly strided. The paper finds 1.96 distinct strides on average, meaning you might need 2 entries instead of 1—still far better than 32.
The locality · page table clustering

Books on the same shelf

A library shelf holds 512 books (an L4 PT = 512 PTEs in one 4 KB DRAM row). You need books 100, 101, 102, …, 131—multiple on the same shelf. Instead of 32 trips (each opening the shelf, grabbing one book, closing it), you open the shelf once and grab multiple in one pass. LATP does this: if 80.20% of a warp's VPNs share the same L4 PT base address, batching their fetches exploits row-buffer hits.

Where it breaks: the remaining ~20% of VPNs span multiple shelves. LATP handles them with separate walks, so you get partial batching—still a win, just not 100%.
Why prior prefetchers fail · time vs thread order

Sorting a shuffled deck by suit vs by rank

A deck of cards arrives shuffled (TLB accesses in time order—thousands of threads interleaved). CPU prefetchers try to predict the next card by watching the shuffle: "hearts → spades → clubs → ?" (no pattern). LATPC instead looks at one hand (warp instruction): "3♠ 4♠ 5♠ 6♠" (consecutive ranks, same suit). The pattern is obvious—next is 7♠. Regularity Detector scans the coalescer's output (which is determined by thread index, like sorting cards by rank within a suit), not the time-ordered chaos.

Where it breaks: cards are independent; GPU threads aren't. The analogy oversimplifies the interleaving—real warps overlap in time, but LATPC works within each warp's coalesced VPN list, so the interleaving doesn't matter.
Detection vs prediction · no ML, no training

Measuring vs guessing

Prior TLB prefetchers predict: "I've seen VPNs 100, 104, 108, so the next is probably 112" (Stride prefetcher). If the pattern changes mid-stream, the prediction is wrong. LATPC detects: "This warp's coalesced VPN list is [100, 104, 108, 112, …, 224]—the stride is 4, so thread 5 needs VPN 120." No prediction, no training, no misprediction penalty. Accuracy is near-100% because you're reading the actual stride from the warp's data, not inferring it from history.

Where it breaks: "detection" sounds instant, but Regularity Detector is a 1-cycle state machine processing one VPN/cycle. For a warp with 32 unique VPNs, it takes 32 cycles to scan the full list—longer than an idealized "instant" measurement. Still, 32 cycles is negligible vs hundreds of cycles for a page-table walk.
The full explanation

How it actually works

Component by component, with the paper's key mechanisms rebuilt as visuals.

1 · Regularity Detector — spotting the stride

Regularity Detector sits after the TLB coalescer (which outputs unique VPNs determined by thread index). It's a tiny state machine: 5 registers (36-bit Base VPN, 36-bit Prev. VPN, 9-bit Prev. Stride, 5-bit Prev. Index, 1-bit Prev. Valid), a 9-bit subtractor, two comparators, and a few muxes. Each cycle it reads one VPN, computes Stride = (VPN - Prev.VPN) & 0x1FF (bottom 9 bits), checks if the stride matches Prev.Stride and if the MSBs match Base VPN. If yes, increment Index; if no, emit the previous sequence (if any), reset Base VPN to the current VPN, and start a new sequence.

Output is a triple <VPN, Stride, Index>. Demand requests (start of a new sequence) have Stride=0, Index=0. Prefetch requests have non-zero stride and index. Negative strides are allowed (subtractor underflows, but the underflown 9-bit value is used as-is; the base VPN calculation uses only the 9 LSBs, so the borrow bit is ignored). The hardware cost: 0.0009 mm² in 7 nm, 1-cycle latency modeled.

💡
Why 9 bits? An L4 PT holds 512 PTEs (9-bit index). Constraining the stride to 9 bits helps ensure VPNs in a detected sequence can fall within one L4 PT (same DRAM row), enabling batched walks. Larger strides break into multiple sequences, which is fine—each sequence still benefits from compression and batching within its own L4 PT.

2 · LATC — compressed L1 TLB MSHR

Each baseline L1 TLB MSHR entry holds one VPN (36-bit tag) and a 1-bit valid. LATC extends it: 32-bit Valid Mask (one bit per thread), 9-bit Stride, and 36-bit Base VPN. The comparison logic now checks (Base VPN + Stride × Index) == VPN using a 9-bit multiplier, an adder, and two comparators. If a match is found and the corresponding bit of Valid Mask is already set, that's an MSHR hit (hit-under-miss). If the bit is unset, set it and forward the request—an MSHR miss (miss-under-miss). If no entry matches and all entries are occupied, that's an MSHR reservation failure (reduced with LATC, since one entry can hold up to 32 VPNs).

When a PTE returns from the L2 TLB, the L1 TLB MSHR clears the corresponding bit of Valid Mask. When all bits are zero, the entry is freed. This allows gradual release: if only 10 of 32 PTEs have returned, those 10 threads can proceed while the other 22 wait—no all-or-nothing blocking.

🍯
The compression ratio: in the worst case (32 distinct strides, no regularity), LATC allocates 32 entries—same as baseline. In the common case (1.96 distinct strides), LATC allocates ~2 entries for 32 misses → up to 16:1 compression. The paper shows MSHR reservation failures drop from 68.84% (baseline) to 43.02% (LATPC).

3 · LATP — batched page-table walks

The Page Table Walker (PTW) normally processes one VPN at a time: walk L1→L2→L3 PTs (intermediate levels), fetch the L4 PTE, fill the L2 TLB. LATP extends the Page Walk Buffer (PW Buffer) similarly to LATC: add 9-bit Stride and 32-bit Valid Mask to each entry. When a translation request arrives, the PTW checks if it can be coalesced into an existing PW Buffer entry (same base address, same stride). If yes, set the corresponding bit of Valid Mask and reserve the L4 walk until the intermediate traversal (L1→L2→L3) completes. Once the L3 PTE is fetched, the PTW can issue the reserved L4 walks—the paper says they are issued one by one but typically hit the DRAM row buffer, so latency is low.

Because the VPNs in a sequence differ only in their bottom 9 bits (enforced by Regularity Detector), their L4 PTEs reside at L3_PTE.base + VPN[8:0] × 8 (8 bytes/PTE). Many addresses fall within a single 4 KB row. The first fetch opens the row; subsequent fetches can be row-buffer hits (latency ~4 cycles each instead of ~50 cycles for a row miss). The intermediate L1→L2→L3 walks are cached in the Page Walk Cache (PWC), which the paper reports hits 99.99%, 99.93%, and 94.72% for L1/L2/L3 PTs, so the common path is: L1 TLB miss → allocate compressed MSHR → probe L2 TLB (miss) → PTW fetches L3 PTE from PWC (hit) → issue batched L4 fetches.

🌱
Prefetch overhead: each L4 fetch is a separate memory request, but they're to consecutive addresses in the same row. Modern GDDR6 DRAM (used in the paper's GPU config) handles consecutive accesses with tCCDL=4 cycles (column-to-column delay, same row). The paper notes reserved walks are issued one by one and typically hit the row buffer.

4 · Working model — end-to-end flow

Follow one warp instruction with 32 unique VPNs through LATPC:

  1. Warp scheduler issues a load instruction → LD/ST units generate 32 addresses → coalescer extracts unique VPNs (determined by thread index) → forwards to Regularity Detector.
  2. Regularity Detector scans the VPN list, detects stride (e.g., stride=4), emits 32 triples: <VPN0, 0, 0> (demand), <VPN1, 4, 1> (prefetch), …, <VPN31, 4, 31> (prefetch) → forwards to L1 TLB.
  3. L1 TLB checks each triple. Demand VPN0 misses → allocates one LATC MSHR entry with Base=VPN0, Stride=4, Valid Mask bit 0 set. Prefetch VPN1 misses → coalesces into same entry (Base + 4×1 = VPN1), sets bit 1. Repeat for others → one MSHR entry can hold many.
  4. Misses forward to L2 TLB. Assume misses → forward to PW Queue. PTW dequeues demand request, starts L1→L2→L3 walk, allocates one PW Buffer entry with Base=VPN0, Stride=4, Valid Mask bit 0 set. Prefetch requests arrive → coalesce into same entry, set bits 1..31. Intermediate walk completes → PTW issues reserved L4 walks.
  5. L4 fetches to DRAM, with first access potentially a row miss and subsequent accesses row hits if they share the same row. PTEs return → fill L2 TLB → forward to L1 TLBs.
  6. L1 TLB receives PTEs, fills entries, clears corresponding bits in LATC MSHR's Valid Mask. Once all bits for an entry are cleared, MSHR entry is freed. L1 TLB signals LD/ST unit to replay the load → threads proceed.

The paper models 1-cycle added latency for Regularity Detector and LATC tagging, reducing GMean speedup by 0.45%—negligible performance impact.

Follow the numbers, step by step

Worked examples

Three traces from the paper's evaluation, carried all the way through.

① Detecting the stride for a warp with 8 unique VPNs

A warp instruction touches 8 threads, each on a different page. The coalescer outputs VPNs: [0x1000, 0x1002, 0x1004, 0x1006, 0x1008, 0x100a, 0x100c, 0x100e]. What does Regularity Detector emit?

  1. Cycle 0: Input VPN=0x1000, Prev.Valid=0 (uninitialized) → emit <0x1000, 0, 0> (demand). Set Base VPN=0x1000, Prev.VPN=0x1000, Prev.Stride=0, Prev.Index=0, Prev.Valid=1.
  2. Cycle 1: Input VPN=0x1002. Stride = 0x1002 - 0x1000 = 0x2. Prev.Stride=0 (first after demand), so start new sequence with stride=2. Emit <0x1002, 2, 1>. Set Prev.VPN=0x1002, Prev.Stride=2, Prev.Index=1.
  3. Cycle 2: Input VPN=0x1004. Stride = 0x1004 - 0x1002 = 0x2 (matches Prev.Stride). Emit <0x1004, 2, 2>. Increment Prev.Index to 2, set Prev.VPN=0x1004.
  4. Cycles 3–7: Repeat: each VPN = Prev.VPN + 2. Emit <0x1006, 2, 3>, …, <0x100e, 2, 7>. Final state: Prev.Index=7.
  5. Result: 1 demand request + 7 prefetch requests, all with stride=2. LATC compresses all 8 into one MSHR entry (Base=0x1000, Stride=2, Valid Mask bits 0..7 set). Accuracy = 100% (no misprediction).
🍯
Why it works: the coalescer output is determined by thread index, so consecutive threads with a memory access pattern like array[tid*2] produce consecutive VPNs with stride=2. Regularity Detector just reads that stride directly—no need to infer it from a chaotic time-ordered stream.

② Batching L4 page-table walks for 32 VPNs in the same L4 PT

A warp's 32 unique VPNs all fall within a 512-page window (e.g., VPNs 0x10000..0x1001f). How does LATP batch their walks?

  1. L1 TLB miss: demand VPN=0x10000 misses → LATC allocates one MSHR entry (Base=0x10000, Stride=1, bit 0 set). Prefetch VPNs 0x10001..0x1001f coalesce into same entry (bits 1..31 set). All forward to L2 TLB.
  2. L2 TLB miss: all 32 miss → forward to PW Queue. PTW dequeues demand VPN=0x10000, starts L1→L2→L3 walk. Allocates one PW Buffer entry (Base=0x10000, Stride=1, bit 0 set).
  3. Coalesce prefetches: prefetch VPNs 0x10001..0x1001f arrive at PTW → coalesce into same PW Buffer entry (bits 1..31 set). Walks are reserved until L3 walk completes.
  4. L3 walk completes: PTW fetches L3 PTE (from PWC, hit). The L3 PTE gives the base address of the L4 PT, e.g., PA=0x50000. PTW calculates L4 PTE addresses: 0x50000 + VPN[8:0] × 8 for each VPN. For VPN=0x10000, VPN[8:0]=0x00, so PA=0x50000. For VPN=0x10001, VPN[8:0]=0x01, so PA=0x50008. Addresses = 0x50000..0x500f8 (256 bytes = 32 PTEs × 8 bytes/PTE).
  5. L4 fetch: PTW issues memory requests to DRAM. The paper says reserved walks are issued one by one but typically hit the row buffer. First request (PA=0x50000) may be a row miss; subsequent requests to the same row are row hits (tCCDL=4 cycles each).
  6. PTEs return: fill L2 TLB, forward to L1 TLBs. L1 TLB fills entries, clears LATC MSHR bits 0..31, frees entry. Warp proceeds.
💡
The locality payoff: baseline would issue 32 separate walks. LATP's batching exploits row-buffer locality for the L4 fetches.

③ Measuring the compression ratio for atax workload

The paper reports atax achieves 3.6× speedup (Figure 17a). What's the compression ratio and prefetch coverage?

  1. Baseline: atax has L2 TLB MPKI = 78.46 (Figure 3, Table 3). High MPKI → high page divergence. Average warp instruction requires 16+ unique translations (Figure 3 shows atax in "Regular+High" class, meaning high MPKI). Baseline: each TLB miss allocates one L1 TLB MSHR entry → 16 misses exhaust all 16 entries → subsequent misses stall.
  2. LATPC: Regularity Detector scans coalescer output for atax's warp instructions. The workload is "regular" (matrix ops), so VPNs follow a near-perfect stride (Figure 8 shows regular workloads have 1.42 distinct strides on average). LATC compresses misses into 1–2 MSHR entries → compression ratio up to 8–16:1.
  3. Prefetch coverage: the paper reports LATPC achieves 54.78% prefetch coverage (Figure 19a). For atax, coverage is higher—LATP prefetches based on detected stride, and 93.38% of translations within "Regular" workloads share the same L4 PT base address (Figure 9). So coverage ≈ 90+%, meaning many TLB misses become TLB hits due to prefetching.
  4. Speedup breakdown: atax baseline IPC ≈ X. With LATPC, IPC ≈ 3.6X. Translation latency drops significantly. PTW occupancy: baseline 16/16 → LATPC lower. L1 TLB MSHR occupancy: baseline 16/16 → LATPC lower. Both bottlenecks reduced.
  5. Result: atax is a strong case for LATPC—highly regular VPNs, high locality, benefits from compression and batching. 3.6× speedup is near the maximum achievable without changing the page size or adding more hardware.
🍯
Insight: the paper's GMean speedup (1.47×) is pulled down by workloads with low MPKI or irregular access patterns. For high-MPKI, regular workloads (atax, bicg, fdtd2d), LATPC achieves 2–4× speedup. The key is not just the speedup but the consistency: LATPC never degrades performance (no workload below 1.00×), unlike prior prefetchers (Sequential drops to 0.82×).

Problem 1 · easy
A warp instruction touches 32 threads, each on a different page. The baseline has 16 L1 TLB MSHR entries. Without LATPC, what happens to MSHR reservation attempts?
Show answer
The first 16 TLB misses allocate all 16 MSHR entries. The next 16 misses find no free entries → MSHR reservation failures. Those 16 threads stall until one of the first 16 walks completes and frees an entry. With LATPC, up to 32 misses can compress into 1–2 entries (depending on stride regularity) → reduced reservation failures (from 68.84% to 43.02% globally).Misconception: that the failures are "only" 16/32 = 50%, so not a big deal. In reality, those 16 stalled threads block the entire warp, and the warp can't proceed until all 32 translations complete—so the 16 failures serialize the second half, increasing latency.
Problem 2 · medium
Regularity Detector sees VPNs [0x1000, 0x1005, 0x100a, 0x100f]. What triples does it emit, and why?

Cycle 2: Stride = 0x100a - 0x1005 = 0x5 (matches) → emit <0x100a, 5, 2>. Cycle 3: Stride = 0x100f - 0x100a = 0x5 → emit <0x100f, 5, 3>. Result: 1 demand + 3 prefetch, all with stride=5. Problem 3 · medium
Why does LATP constrain the stride to 9 bits, and what happens if two VPNs differ by 1,000 pages?
Show answer
An L4 PT holds 512 PTEs (9-bit index). Constraining stride to 9 bits helps ensure VPNs in a detected sequence can fall within one L4 PT, enabling batched L4 walks (exploiting DRAM row-buffer locality). If two VPNs differ by 1,000 pages (stride=1,000 > 511), Regularity Detector breaks the sequence: the first VPN starts one sequence (Base=VPN0, Stride=0, Index=0), the second starts a new sequence (Base=VPN1, Stride=0, Index=0). Result: two separate MSHR entries, walks handled separately. Still correct, just less batching benefit for that pair. The paper finds 80.20% of translations within a warp share the same L4 PT base address (Figure 9), so the 9-bit constraint is rarely a hard limitation.Misconception: that LATPC "fails" on large strides. It doesn't fail—it just doesn't batch those translations. They're still handled correctly, just without the row-buffer locality benefit.
Problem 4 · hard
The paper reports Valkyrie (the prior GPU TLB prefetcher) achieves only 1.03× GMean speedup vs 1.47× for LATPC. Valkyrie replicates PTEs to likely L1 TLBs when an L2 TLB miss is handled. Why doesn't this help as much as LATPC?
Show answer
Valkyrie's replication targets spatial locality across SMs: if SM 0 fetches a PTE, replicate it to SM 1–29 that might need it. But it doesn't prefetch additional PTEs from the page table—it only replicates the one PTE already fetched. So it saves L1→L2 TLB traffic but doesn't reduce L2 TLB misses or page-table walks. LATPC, in contrast, prefetches additional PTEs from the page table (LATP) when one demand PTE is fetched, reducing future L2 TLB misses and page-table walks. The paper shows L2 TLB MSHR subentries in the baseline already perform Valkyrie-like replication (Section 7 "Valkyrie's Effectiveness"), so Valkyrie's marginal benefit is small. Furthermore, the paper uses large memory footprints (29.50 MB average) with low inter-SM locality (e.g., KM workload: only 1.49 SMs share a page on average), reducing Valkyrie's replication effectiveness.Misconception: that TLB prefetching = replicating entries to other TLBs. Valkyrie does replication; LATPC does prefetching from the page table. The latter is far more effective because it prevents L2 TLB misses (53% of translation latency) and page-table walks (39% of latency).
Problem 5 · medium
The paper reports L1 TLB MSHR reservation failure rate drops from 68.84% (baseline) to 43.02% (LATPC). Why not 0%?
Show answer
LATC compresses up to 32 VPNs into one entry, assuming they form a strided sequence. If a warp's VPNs exhibit multiple distinct strides (e.g., irregular workload with 2.49 strides on average, Figure 8), LATC allocates one entry per stride group. For example, if a warp has 3 distinct strides covering 10, 12, and 10 threads respectively, LATC allocates 3 entries instead of 32—still compression, but not 32:1. The remaining failures occur when: (1) multiple warps' misses arrive concurrently and exhaust the 16 entries despite compression, or (2) irregular workloads with very high stride diversity (e.g., 5+ distinct strides) require more entries. The paper shows irregular workloads (DMR, HS, MRG, MST, PR, SSP) have lower failure-rate reduction (Figure 18b) than regular workloads (atax, bicg, fdtd2d) because they have less compressible VPN sequences.Misconception: that LATC is a silver bullet that eliminates all MSHR failures. It's a compression technique—highly effective for regular access patterns, less so for truly random patterns. But even irregular workloads benefit (e.g., PR: 68% → 43% failure rate).
Problem 6 · hard
The paper evaluates LATPC with 4 KB pages (Section 6) and also with 2 MB pages (Section 6.6), reporting 1.47× and 1.18× GMean speedup respectively. Why does LATPC still help with large pages, and why is the speedup lower?
Show answer
Large pages (2 MB) increase TLB reach by 512×, reducing TLB miss rate. But the paper uses enlarged memory footprints (1.14 GB average for "Regular+High" workloads, Figure 6) that still cause high page divergence. For example, a matrix larger than 2 MB still requires multiple 2 MB pages, and a warp accessing the matrix by rows can touch many different 2 MB pages (one per thread or groups of threads). So the bottlenecks (PTW and MSHR contention) persist, just with fewer misses. LATPC's mechanisms (compression, batching) still apply: LATC compresses misses into 1–2 entries; LATP batches their walks. The speedup is lower (1.18× vs 1.47×) because the absolute number of misses is lower with large pages, so the absolute benefit of LATPC is smaller. But the relative benefit (contention relief) remains. Figure 6 shows that even with 2 MB pages, increasing PTWs to 64 or MSHR entries to 1,024 yields 1.38× and 1.66× speedup, proving the bottlenecks persist. LATPC addresses those bottlenecks without adding more PTWs or MSHR entries.Misconception: that large pages "solve" the address-translation problem, so LATPC is unnecessary. Large pages help but don't eliminate the problem for workloads with huge memory footprints (which are common in HPC, ML, graph analytics). LATPC is orthogonal and complementary.
Problem 7 · easy
LATPC adds Regularity Detector (0.0009 mm² in 7 nm) and extends L1 TLB MSHR entries (9-bit Stride + 32-bit Valid Mask per entry × 16 entries × 30 SMs). What's the total on-chip storage overhead?
Show answer
L1 TLB MSHR extension: (9 + 32 - 1) bits × 16 entries × 30 SMs = 40 bits × 480 = 19,200 bits = 2,400 bytes. PW Buffer extension: (9 + 32 - 1) bits × 16 entries = 40 bits × 16 = 640 bits = 80 bytes. Total storage = 2,400 + 80 = 2,480 bytes ≈ 2.48 KB. The paper reports 2.48 KB storage overhead (Section 5.5). Logic area (Regularity Detector + LATC tagging) = 0.26 mm² (7 nm). Baseline GPU die = 445 mm² (NVIDIA TU106). Overhead = 0.26 / 445 = 0.058%. Power = 36 mW (baseline TDP = 175 W) = 0.02%. Latency = 1 cycle (both Regularity Detector and LATC tagging), which the paper models and finds reduces GMean speedup by only 0.45%.Misconception: that "compression" always means more complex hardware. LATC's compression is simple arithmetic (base + stride × index), implemented with a 9-bit multiplier and adder—tiny compared to a full ALU or FPU.
Problem 8 · medium
The paper compares LATPC to Avatar (MICRO'24), the state-of-the-art GPU address-translation technique. Avatar achieves 1.40× GMean speedup; LATPC achieves 1.47×. Why does Avatar+LATPC achieve 1.64× (Figure 20)?
Show answer
Avatar uses speculative address translation: upon an L1 TLB miss, it guesses the PFN from a learned offset and speculatively accesses the cache, while the real translation proceeds in the background. If the guess is correct, the cache access completes early; if wrong, discard and retry. The problem: even during speculation, the TLB MSHR entry and PTW are occupied for the background translation—so Avatar still suffers from MSHR/PTW contention (Figure 18b: 66.33% MSHR failure rate vs 68.84% baseline). LATPC's LATC+LATP reduce that contention by compressing the MSHR and batching the background walk. Avatar+LATPC combines speculation (Avatar) with contention relief (LATPC): the speculation often succeeds (hiding latency), and when it fails, the background translation is faster due to LATPC. The paper notes that "both TLB MSHRs and PTWs are occupied until the speculation is validated" (Section 6.5), so LATPC accelerates the validation. Result: 1.64× GMean speedup, a 17% improvement over Avatar alone. The techniques are orthogonal and synergistic.Misconception: that Avatar "solves" translation by speculation, so LATPC is redundant. Avatar hides translation latency; LATPC reduces translation latency. Both are valuable, and they stack.
Quick answers

Frequently asked questions

Ten questions newcomers ask most. Tap any to expand.

Q1What is "page divergence" and why does it matter?▾
Page divergence = the number of unique pages a warp instruction touches. In the worst case, each of 32 threads accesses a different page → 32 TLB lookups, potentially 32 TLB misses. The paper finds 75% of warp instructions require multiple translations, and 29% hit the max of 32 (Figure 3). High divergence exhausts the limited 16 L1 TLB MSHR entries and 16 PTWs, causing stalls. LATPC targets exactly this: compressing up to 32 misses into 1–2 MSHR entries and batching walks to exploit row-buffer locality.
Q2How does LATPC differ from CPU TLB prefetchers (Sequential/Stride/Distance)?▾
CPU prefetchers learn patterns from the time-ordered stream of TLB accesses (e.g., "VPNs 100, 104, 108 → stride=4, prefetch 112"). On GPUs, thousands of threads interleave chaotically, so the time-ordered stream is scattered (no stable stride). LATPC instead analyzes VPNs within a single warp instruction (the coalescer output, determined by thread index), detecting the stride directly from that small, ordered list. No prediction, no training—just a running diff. Accuracy near 100% (Figure 13). CPU prefetchers achieve <80% accuracy on GPU workloads and degrade performance (Sequential: 0.82× GMean speedup, Figure 4).
Q3Why does LATPC compress TLB misses instead of just adding more MSHR entries?▾
Adding MSHR entries costs on-chip SRAM (expensive, power-hungry). The paper shows increasing MSHR entries from 16 to 1,024 (64× increase) would yield ~2× speedup (Figure 6), but at huge area/power cost (64× more storage). LATPC achieves 1.47× speedup with only 2.48 KB storage overhead (0.06% of baseline GPU die area) by compressing misses into existing entries. Compression is cheaper than brute-force scaling. Also, compression is adaptive: regular workloads get up to 16:1 compression; irregular workloads get lower but still beneficial compression (e.g., 4:1). Fixed hardware (64× MSHR entries) doesn't adapt.
Q4Does LATPC work for irregular workloads (e.g., graph traversal)?▾
Yes, but with lower benefit. The paper classifies workloads into "Regular" (e.g., matrix ops, 1.42 distinct strides) and "Irregular" (e.g., graph, 2.49 distinct strides). Irregular workloads still exhibit some regularity within a warp (Figure 8): VPNs aren't perfectly strided, but they cluster into a few stride groups. LATC compresses each group into one MSHR entry → still beneficial, just not 32:1. For example, 3 stride groups → 3 MSHR entries instead of 32 → up to 10:1 compression. The paper shows irregular workloads (PR, SSP) achieve 1.2–1.3× speedup (Figure 17a), lower than regular workloads (atax: 3.6×) but still a win. Irregular workloads also have lower page-table locality (67.00% vs 93.38% share same L4 PT base, Figure 9), reducing LATP's batching benefit—but 67% is still substantial.
Q5What happens if a warp's VPNs don't form a strided sequence at all?▾
Regularity Detector breaks the sequence into singleton groups (each VPN is its own "sequence" with Index=0, Stride=0). LATC allocates one MSHR entry per VPN—same as the baseline. No harm, no benefit. The paper finds this is rare: even irregular workloads average 2.49 distinct strides (Figure 8), meaning most warp instructions have some grouping. The worst case (32 distinct VPNs, no grouping) is possible but uncommon. LATPC degrades to baseline behavior in that case, never worse.
Q6How does LATP know when to batch walks? What if some VPNs arrive before others?▾
LATP batches walks that share the same base address and stride in the PW Buffer. When a prefetch request arrives, the PTW checks if it can coalesce into an existing entry (base + stride × index matches). If yes, set the corresponding bit of Valid Mask and reserve the L4 walk. The PTW issues the batched L4 walks after the intermediate L1→L2→L3 walk completes, one by one but typically hitting the row buffer. Early-arriving prefetches wait for the demand walk's L3 fetch; late-arriving prefetches (after L3 completes) are issued as they arrive. The reservation mechanism ensures correctness even with out-of-order arrivals.
Q7Does LATPC require changes to the GPU programming model or runtime?▾
No. LATPC is a purely hardware mechanism—transparent to software. Programmers write CUDA/OpenCL as usual; the coalescer, Regularity Detector, L1 TLB, and PTW handle the rest automatically. No API changes, no kernel modifications, no recompilation. This is a key advantage over software-based approaches (e.g., page prefetching via OS hints), which require programmer intervention.
Q8Why does LATPC's speedup vary so much across workloads (1.0× to 3.6×)?▾
Speedup depends on three factors: (1) baseline bottleneck severity (high MPKI → high potential), (2) VPN regularity (low stride diversity → high compression), (3) page-table locality (high % share same L4 PT base → high batching). Workloads with all three (atax, bicg, fdtd2d) achieve 2–4× speedup. Workloads with low MPKI (e.g., vectorAdd: 3.84 MPKI) have little room for improvement → 1.0× speedup. Workloads with irregular VPNs (e.g., DMR: graph traversal) get lower compression/batching → 1.2× speedup. The paper's GMean (1.47×) balances across all workload types. The key is LATPC never degrades performance (no workload below 1.00×), unlike prior prefetchers (Sequential: 0.82×).
Q9Can LATPC be combined with other GPU address-translation optimizations?▾
Yes, LATPC is orthogonal to most techniques. The paper evaluates LATPC+Avatar (1.64× vs 1.47× for LATPC alone, Figure 20). LATPC is also compatible with: large pages (1.18× with 2 MB pages, Figure 21), shared L2 TLB across SMs (baseline already uses this). The paper does not substantiate broader compatibility claims with techniques like SnakeByte, UVM prefetching, direct segments, or multi-GPU/MCM configurations—those are mentioned as potential future work areas.
Q10What did the authors leave for future work?▾
The paper flags three areas: (1) extending LATPC to sorted VPN lists (Section 7 "Sorting Requirement")—sorting by VPN (not thread index) could increase regularity, but it adds latency (15 cycles for a bitonic sort) and area (454× the size of Regularity Detector), so the authors didn't pursue it. (2) Handling multi-stride sequences more efficiently—current LATC breaks sequences at stride changes; a smarter detector could learn multiple strides per warp (e.g., "stride alternates between 4 and 8"). (3) Applying LATPC to multi-GPU or MCM-GPU systems—the paper's workload suite is single-GPU; extending to multi-GPU would require inter-GPU TLB coherence, which is orthogonal to LATPC but could benefit from its compression/batching.

Core terms
LATPCLocality-Aware TLB Prefetching and MSHR Compression—the paper's mechanism combining LATC and LATP.
LATCLocality-Aware TLB MSHR Compression—extends each L1 TLB MSHR entry to represent up to 32 VPNs via {base, stride, 32-bit mask}.
LATPLocality-Aware TLB Prefetching—batches page-table walks sharing the same L4 PT, exploiting DRAM row-buffer locality.
Page divergenceThe number of unique pages a warp instruction touches (1–32). High divergence exhausts PTWs and TLB MSHRs.
Regularity DetectorA 1-cycle state machine that scans the coalescer's VPN list, detects VPN strides, and emits <VPN, Stride, Index> triples.
VPN stride(VPNi+1 − VPNi). The paper finds 1.96 distinct strides on average within a warp instruction.
Page-table locality80.20% of translations within a warp share the same L4 PT base address (512-page window), enabling batched L4 fetches from one DRAM row.
TLB MSHRMiss-Status Holding Register—tracks outstanding TLB misses. Baseline: 16 L1 TLB MSHR entries. LATPC: each entry can hold up to 32 VPNs.
Supporting terms
WarpA group of 32 GPU threads executing the same instruction in lockstep (SIMT). A warp instruction's memory accesses are coalesced by the hardware.
CoalescerHardware unit that merges a warp's 32 addresses into unique VPNs (page-size granularity). Output is determined by thread index.
VPN / PFNVirtual Page Number / Physical Frame Number. VPN is the input to translation; PFN is the output (from the PTE).
PTEPage Table Entry—stores the VPN→PFN mapping. Resides in the page table in DRAM; cached in TLBs.
PTWPage Table Walker—hardware unit that traverses the page table (L1→L2→L3→L4 PTs) to fetch a PTE. Baseline: 16 PTWs.
L4 PTLeaf-level page table (x86-64). Holds 512 PTEs in a single 4 KB DRAM row. The target of LATP's batched walks.
PWCPage Walk Cache—caches intermediate page-table levels (L1/L2/L3 PTEs). The paper reports 94–99% hit rates.
Row-buffer localityDRAM accesses to the same row are fast (tCCDL=4 cycles). LATP batches L4 fetches to exploit this.
MPKIMisses Per 1000 Instructions (L2 TLB). The paper uses this to classify workloads: Low (<5), High (≥5).
ValkyriePrior GPU TLB prefetcher (PACT'20). Replicates PTEs to L1 TLBs; doesn't prefetch from page table. Achieves 1.03× GMean speedup.
AvatarState-of-the-art GPU translation technique (MICRO'24). Speculates PFN; LATPC accelerates its background walks. Together: 1.64× speedup.
