MICRO 2006 · A Field Guide

Giving cache to whoever benefits, not whoever asks loudest

Two programs share one cache. The default rule quietly hands it to whoever touches the most data — which is often the wrong program. This is the plain-English story of UCP, a tiny circuit that fixes it.

📄 "Utility-Based Cache Partitioning" · Qureshi & Patt 🎓 Some cache basics assumed — the primers cover them ⏱ ~30 min read
🗄️

Give each slice of cache to the program that SAVES the most misses from it — not the one that DEMANDS the most space.
Measure the benefit first, then divide.

Before you begin

Ten ideas the paper assumes you know

Each card clears up one concept UCP builds on. They're collapsed — tap any open. Read in order the first time, or jump to whatever looks unfamiliar.

high-utility app (vpr) saturating app (equake) UCP's machinery LRU / demand (the baseline)
1What a CPU cache is, and why it exists▾

Main memory (RAM) is big but slow — hundreds of CPU cycles away. A cache is a small, very fast memory that sits right next to the processor and keeps copies of the data you've used recently, so most accesses are answered in a few cycles instead of hundreds.

This paper is about the last-level cache (here a 1 MB L2): the big shared one, the last stop before the long trip to main memory. Using it well is how you avoid that trip.

Now you're ready: UCP is about dividing that one big shared cache between two programs so the whole system makes fewer slow trips to memory.
2A cache hit, a miss, and the miss penalty▾

Ask the cache for data. If it's there, that's a hit (fast). If not, a miss — the CPU must fetch it from main memory and wait. That wait is the miss penalty (here ~400 cycles, vs a ~15-cycle cache hit).

Fewer misses = faster program. So the whole game is: keep the data that will be reused, evict the data that won't.

Now you're ready: UCP measures exactly one thing — how many misses each program would save with more cache — and gives cache to whoever saves more.
3Sets and ways (how a cache is carved up)▾

A cache isn't one big bucket. It's split into many sets, and each set has a few slots called ways. A given address can only live in one set, in any one of that set's ways. "16-way" means each set has 16 slots.

The key point for UCP: the cache is allocated in units of ways. Giving a program "9 ways" means it may use 9 of the 16 slots in every set. Turn off a way and that slice of cache is gone.

Now you're ready: "partitioning the cache" = deciding how many of the 16 ways each program gets. UCP's answer for the example apps is 13 ways to one, 3 to the other.
4LRU — the standard eviction rule▾

When a set is full and a new block arrives, something must be evicted. The usual choice is LRU: throw out the Least Recently Used block, betting that what you haven't touched in a while you won't need soon.

LRU is sensible — but notice it has no idea which program benefits more. It just tracks recency.

Now you're ready: UCP's villain is plain LRU on a shared cache. It quietly hands cache to whoever's been busiest, which isn't the same as whoever benefits.
5A shared cache on a multicore chip▾

Modern chips have several CPU cores running different programs at once. To save space they often share the big last-level cache. Now two programs' data live in the same cache, competing for the same ways.

Under plain LRU, that competition is settled by recency across both programs combined — so one program's flood of accesses can crowd out another's.

Now you're ready: UCP targets exactly this two-program shared cache, deciding the split deliberately instead of letting LRU decide by accident.
6"Demand" vs benefit (the crux)▾

Demand = how much cache a program tries to use — roughly, how many distinct blocks it touches. A program streaming through gigabytes touches a huge number of unique blocks, so it has enormous demand.

But touching a block once and never returning gains nothing from keeping it cached. High demand, zero benefit. The two come apart, and that gap is the entire paper.

Now you're ready: LRU rewards demand. UCP rewards benefit. The word the paper uses for benefit is utility.
7How we measure: MPKI and CPI▾

Two scorecards. MPKI = Misses Per 1000 Instructions (how often we take the slow path). CPI = Cycles Per Instruction (how slow the program runs overall — lower is faster).

Cutting misses (lower MPKI) usually lowers CPI too, so the paper uses "misses saved" as a stand-in for "speedup." It flags one exception (swim) where they don't track, but mostly they move together.

Now you're ready: when you see a curve of "misses vs ways," that's the raw material UCP reads to decide the split.
8Tags and a tag directory▾

Each cached block carries a small label — a tag — recording which memory address it holds, plus bookkeeping bits. The collection of all tags (without the actual data) is the tag directory.

You can answer "would this access hit or miss?" using only the tags — you don't need the data. That's how UCP can run a cheap side-experiment: a tiny tag-only copy that tracks hits and misses without storing any real data.

Now you're ready: UCP's monitor is basically a small extra tag directory plus a few counters — tags only, so it's cheap.
9Sampling — measure a few, infer the whole▾

To learn how a huge population behaves, you don't survey everyone — you poll a representative sample and accept a small, bounded error. A few thousand voters predict millions.

A cache has ~1000 sets. Watching all of them is expensive. Watching a well-chosen handful and inferring the rest is nearly as accurate, far cheaper. The math that says "how few is safe" is just a statistics bound.

Now you're ready: UCP watches only 32 of 1024 sets ("dynamic set sampling") and proves that's enough — which is what shrinks the monitor under 2 kB.
10Convex vs non-convex curves (and why search explodes)▾

A convex benefit curve has diminishing returns: the first slice helps most, each next slice a little less. Easy to optimize greedily — always take the next best slice. A non-convex curve has a delayed payoff: nothing helps until some threshold, then a sudden cliff of benefit.

Also: with many programs, the number of ways to split a cache grows astronomically (a 32-way cache among 8 programs has over 15 million splits). Trying them all is hopeless; finding the exact best is provably hard.

Now you're ready: UCP's "Lookahead" algorithm exists to handle non-convex curves and avoid the explosion — it's the last section of the paper.
The whole story in one place

UCP in a nutshell

The problem, the cheap measurement that fixes it, and how it all connects.

When two programs share a cache, the standard LRU rule partitions it by demand: whoever touches more blocks gets more cache. But demand isn't benefit. A program streaming through data it never reuses has huge demand and gains nothing from extra cache, while a neighbor that would have used that space goes short.

The paper's fix, Utility-Based Cache Partitioning (UCP), measures the thing that actually matters — utility, the number of misses a program would save with each additional way — and gives each way to whoever saves the most. To do it cheaply at runtime, each core gets a tiny monitor (UMON): a small tag-only directory plus a handful of counters that, thanks to a property of LRU, reads off the miss count for every possible cache size at once. A sampling trick shrinks it to under 2 kB — about 0.2% of the cache it manages.

A simple algorithm then picks the split that minimizes total misses, and a one-bit owner tag on each cache line enforces it. For many programs, where trying every split is impossible, a Lookahead algorithm finds a near-best split in a few hundred operations.

The payoff: on 20 two-program workloads, UCP improves weighted speedup by up to 23%, on average 11% over LRU, plus 16.8% throughput and 11% fairness — for under 1% extra area, with nothing on the critical path.

🍯 Key takeaways

  • LRU splits a shared cache by demand (blocks touched); UCP splits it by utility (misses saved).
  • A program's utility for cache can be low, high, or saturating — and high demand ≠ high utility.
  • The UMON monitor reads the full "misses vs ways" curve using one tag-only directory + recency counters (the LRU stack property).
  • Dynamic set sampling (32 of 1024 sets) keeps the monitor under 2 kB — about 0.2% of the cache.
  • Decide by maximizing combined utility; enforce with a 1-bit owner tag; re-decide every 5M cycles.
  • Lookahead handles many programs and non-convex curves in ~N²/2 operations.
  • Result: up to 23% (avg 11%) faster, <1% area, nothing on the critical path.
Seven everyday pictures

How to think about UCP

Each picture stands alone — read them in any order. Every card ends with an honest note about where the picture stops being true.

The core idea · Utility vs demand

Watering two plants

A thirsty fern and a small cactus share your watering can. "Fair by demand" means giving each as much as it tries to drink — but the cactus drowns past a few drops while the fern would happily use far more. Pour by benefit, not thirst: a little to the cactus, the rest to the fern, and both are better off. UCP waters vpr (the fern) with 13 ways and equake (the cactus) with just 3.

Where it misleads: a plant's need is steady; a program's "thirst" shifts over time (phases), which is why UCP re-decides periodically rather than once.
The monitor · Stack property

Nested measuring cups

One set of nested cups tells you every smaller volume at once — fill the big one and you can read off the 3-cup, 2-cup, and 1-cup lines without separate measurements. LRU has the same nesting: anything that fits in a small cache also fits in a bigger one. So one tag directory, plus a counter per recency position, reveals the miss count at every cache size simultaneously — no need for 16 separate experiments.

Where it breaks: the neat nesting is special to LRU (the "stack property"). A different eviction rule wouldn't nest so cleanly, and the one-directory trick would fail.
Cheap monitoring · Set sampling

Tasting the soup

You don't drink the whole pot to check the seasoning — you stir and taste one spoonful, trusting it represents the rest. UCP does this with cache sets: instead of monitoring all 1024, it watches 32 representative ones and infers the whole cache's behavior. A statistics bound (Chebyshev) guarantees the spoonful is close enough — which is what makes the monitor tiny.

Where it breaks: a spoonful only represents a well-stirred pot. If sets behaved wildly differently (high variance), 32 wouldn't be enough — the paper checks that real workloads are "well-stirred" (variance < 3).
Scaling · Marginal utility & Lookahead

Judging a long-term investment

Some investments lose money for a year, then pay off enormously. Judge them by the first month's return and you'd never invest. The greedy approach ("take whatever helps most right now") makes exactly that mistake on programs like art, which gains nothing until it has 8 ways, then drops sharply. UCP's Lookahead instead scans the whole return curve, spots the delayed payoff, and commits the lump sum to reach it.

Where it breaks: a real investor weighs risk; Lookahead just maximizes known benefit from a measured curve. There's no uncertainty in the curve — it's already been measured by the monitor.
Enforcement · The owner bit

Assigned parking spots

A shared lot where each car is stickered with its owner's color. The rule: if your team is under its quota, you may take a rival's spot when you arrive; if you're at quota, you bump one of your own. Cache does this with a single owner bit per line. And spots are only reclaimed when a car actually shows up (a miss) — so a rival keeps a useful spot until the moment you genuinely need it.

Where it breaks: real parking is per-lot; UCP enforces one quota across all sets uniformly, trading fine per-set control for a much cheaper, simpler scheme.
Staying current · Re-decide & decay

A dimmer, not a switch

You don't reset your sense of a song's volume every second, nor cling to how loud it was a minute ago. You keep a fading running impression. UCP re-runs its decision every 5 million cycles, and after each round halves the counters — so recent behavior dominates while older behavior fades smoothly. That lets it follow a program's shifting phases without lurching on every blip.

Where it breaks: a dimmer is continuous; UCP only adjusts at fixed intervals, so it can lag programs whose phases flip faster than 5M cycles (a few workloads where plain LRU actually wins).
Why a separate monitor · No pollution

Two thermometers

To learn how warm one room runs, you don't measure it while a second room's heater blasts through an open door — your reading is contaminated. UCP keeps each program's monitor separate from the shared cache, so it measures that program's true utility for all 16 ways, untouched by the co-runner. An earlier scheme that read utility from inside the shared (polluted) cache got only +4%; UCP's clean measurement gets +11%.

Where it breaks: a thermometer is free; the separate monitor costs real bits — which is exactly why the sampling trick (taste the soup) matters, to keep that cost tiny.
The full explanation

How it actually works

Section by section, with the paper's key diagrams rebuilt as visuals you can read at a glance.

1 · The problem — LRU rewards demand

On a shared last-level cache, plain LRU partitions by demand: over the last N unique accesses, a program that touched more unique blocks ends up holding more of the cache. Nobody chose this — it falls out of recency ordering. The trap: a streaming program touches a flood of unique blocks (huge demand) but reuses none of them, so the cache it's handed buys zero miss reduction while a neighbor starves.

🌱
Concretely: run vpr and equake together. equake stops benefiting after 3 ways; vpr keeps benefiting to 16. LRU gives equake ~7 and vpr ~9. Utility says give equake 3 and vpr 13 — and vpr's CPI falls 2.0 → 1.5 at no cost to equake.

2 · Defining utility — and its three shapes

Make "benefit" quantitative, per way. If a program misses miss_a times with a ways and miss_b with b ways, the utility of growing from a to b is simply the misses saved:

U(a→b) = miss_a − miss_b     # misses saved by adding those ways

Across programs, utility curves come in three shapes:

ShapeBehaviorCauseExample
Lowbarely drops 1→16 wayscompulsory-miss-bound, or dataset ≫ cachegap, mcf
Highkeeps dropping 1→16reuse-rich, working set > cachevpr
Saturatingdrops, then flattens (~3–8 ways)small working set that fits earlyequake
⚠
Edge case — step functions. Low utility isn't always permanent: art is flat for 8 ways, then its misses drop 5× once the dataset finally fits. These non-convex curves will wreck the simple partitioning algorithm — handled in section 5.

3 · The monitor (UMON) — reading the whole curve cheaply

To partition by utility you need each program's miss count at every way-count, measured without the co-runner polluting it. UMON does both. The key is the LRU stack property: a hit in an N-way cache is also a hit in any larger one. So one tag-only Auxiliary Tag Directory plus a counter per recency position (MRU, position 1, …, LRU) reveals the miss count for every cache size at once. On a hit, bump the counter for the recency position that hit; that counter is "misses saved by owning this many ways."

4 · Making it tiny — dynamic set sampling

A monitor with a tag entry for every line is still too big. The fix is sampling: keep entries for only a few sampled sets and infer the rest. How few is safe? A Chebyshev bound says that if the per-set variance σ² is small, a modest sample nails the average:

# probability the sampled estimate is off by ≥ 1 way
P(off by ≥ 1 way) ≤ σ² / n      # n = number of sampled sets

Most workloads have σ² < 3, so 32 sampled sets (of 1024) pin the estimate within one way with high probability. That single decision is what drops the monitor to 1920 bytes — about 0.2% of the 1 MB cache it manages. (Sample set 0, then every 33rd set.)

🍯
The storage, exactly: each tag entry is 29 bits (1 valid + 24 tag + 4 LRU); 16 ways/set × 29 bits = 58 B/set; × 32 sets = 1856 B; + 64 B of counters = 1920 B per monitor. Dual-core < 0.4% of cache area; quad-core < 1%. Nothing on the critical path.

5 · Decide & enforce

Minimizing total misses = maximizing total utility. For two programs over a 16-way cache, try each boundary and keep the best — then enforce it with a 1-bit owner tag.

# decide (every 5M cycles)
for i in 1..15:
    total[i] = U_vpr(0→i) + U_equake(0→16−i)
choose i that maximizes total      # guarantee ≥1 way each
halve all counters                       # fade old data

# enforce (on a miss by program X)
if X's lines in this set < X's quota:
    evict LRU among the other program's lines   # take a way back
else:
    evict LRU among X's own lines               # stay in budget
💡
Why reclaim only on a miss? When a quota grows, the program claims the new ways gradually, as it misses — so the other program keeps its hot blocks until they're actually needed. The partition slides into place instead of snapping.

6 · Scaling to many programs — Lookahead

"Try every split" is fine for two programs but explodes for many (a 32-way cache among 8 programs has over 15 million splits; the exact optimum is NP-hard). The obvious greedy fix — always give the next way to whoever benefits most from it right now — is optimal only when every curve is convex. Non-convex curves (art: nothing until 8 ways, then a cliff) break it: greedy sees zero gain from way 1 and never starts.

UCP's Lookahead algorithm fixes this with marginal utility = utility per way:

MU(a→b) = (miss_a − miss_b) / (b − a)

Each round, for every program it finds the maximum marginal utility over all the ways it could still receive (not just the next one), and the fewest ways to reach it; it hands that block of ways to the winner; repeat. On convex curves it reduces to greedy (which is optimal there). Worst case ≈ N²/2 operations — for a 32-way cache, ≤ 512, once per 5M cycles.

🍯
On a quad-core (2 MB, 32-way), Lookahead matches the exhaustive "try everything" optimum without its cost, while greedy lags on the non-convex mixes. That's what makes UCP practical beyond two cores.
Follow the numbers, step by step

Worked examples

Four traces, each carried all the way through — reading the miss curve, picking the split, defeating greedy, and tallying the hardware cost.

① Reading the miss curve from the counters

A 4-way monitor saw 100 accesses: 25 missed outright; of the hits, 30 landed in MRU, 20 in position 1, 15 in position 2, 10 in LRU. What's the miss count at every cache size?

  1. 4 ways = 25 misses. Everything that hit, hit — only the 25 true misses remain.
  2. Shrink to 3 ways: the deepest recency slot (LRU) disappears, so its 10 hits become misses. 25 + 10 = 35.
  3. Shrink to 2 ways: also lose position 2's 15 hits. 35 + 15 = 50.
  4. Shrink to 1 way: also lose position 1's 20 hits. 50 + 20 = 70. (MRU's 30 always hit.)
  5. Utility of going 1 → 4 ways = misses saved = 70 − 25 = 45.
🍯
Insight: the counters are the curve. One tag-only structure, filled by ordinary hits, yields the miss count at every size — no separate experiments, no stored data.

② Picking the best split for vpr + equake

Using the curve shapes from the figure (equake flat at 3.3 after 3 ways; vpr still falling), where's the minimum total?

  1. The cache has 16 ways. Give vpr v, equake 16 − v, both ≥ 1.
  2. equake contributes the same 3.3 for any allocation ≥ 3 ways. So as long as equake keeps ≥ 3 ways (i.e. v ≤ 13), only vpr's term changes.
  3. vpr keeps improving with more ways, so push v as high as possible while equake still has 3: v = 13, equake = 3.
  4. Check v = 14: equake drops to 2 ways and its misses tick up (3.3 → ~3.7), costing more than vpr's tiny gain. Worse.
  5. Winner: 13 ways to vpr, 3 to equake — exactly UCP's choice. LRU's demand-split (9/7) leaves 4 ways idle on equake.
💡
This is the combined-utility maximization, by hand. The interactive on the "How It Works" tab computes it live — slide to 13 and watch total misses bottom out.

③ Why greedy fails on art — and Lookahead wins

art saves no misses until it has 8 ways, then drops sharply. A second program offers a small but steady gain on its first way. Who should get the next way?

  1. Greedy asks: "best gain from one more way?" art's first way saves ~0; the rival's saves a little. Greedy gives it to the rival — every single round.
  2. art never reaches 8 ways, so its huge payoff is never unlocked. Greedy is stuck in the flat zone.
  3. Lookahead asks a better question: "best average gain over any number of ways I could get?" It evaluates marginal utility MU = misses-saved ÷ ways for art jumping straight to 8.
  4. That 8-way jump has high marginal utility (the cliff averaged over 8 ways still beats the rival's trickle), so Lookahead commits all 8 ways to art at once.
  5. Result: art's cliff is unlocked; Lookahead matches the exhaustive optimum, greedy doesn't.
🍯
Insight: the bug in greedy is looking only one way ahead. Lookahead's whole trick is scanning the entire remaining curve, so a delayed payoff isn't invisible.

④ Tallying the hardware cost

Is a per-core monitor actually cheap? Add it up (40-bit addresses, 32 sampled sets).

  1. One tag entry = 1 valid bit + 24 tag bits + 4 LRU bits = 29 bits.
  2. One sampled set holds 16 ways: 16 × 29 bits = 464 bits = 58 bytes.
  3. All 32 sampled sets: 32 × 58 B = 1856 bytes of tag directory.
  4. Add the counters: 16 counters × 4 B = 64 B. Total = 1856 + 64 = 1920 bytes per monitor.
  5. As a fraction: 1920 B ÷ 1088 kB of cache ≈ 0.17%. Two cores < 0.4%; four cores < 1%.
💡
That's the punchline: a measurement worth +11% performance costs about one-fifth of one percent of the cache — and sits entirely off the critical path.
Test yourself

Practice problems

Try each before opening the answer. Every one ends with the common misconception it's checking for.

Problem 1 · easy
A program streams through a 100 MB array once, touching every block exactly once, and shares a 1 MB cache with a neighbor. It has by far the highest demand. How many ways should UCP give it, and why?
Show answer
The minimum (UCP guarantees ≥1 way). Streaming once means no block is reused before eviction, so every extra way saves ~0 misses — utility ≈ 0 despite enormous demand. Hand the rest to the neighbor, who can actually use it.Misconception: that high demand deserves more cache. Demand is what LRU rewards; UCP rewards utility, and a stream's utility is near zero.
Problem 2 · medium
A 4-way monitor records, over 100 accesses: 22 misses, then hits of 40 (MRU), 18 (pos 1), 12 (pos 2), 8 (LRU). What is the miss count at 2 ways, and the utility of going from 2 → 4 ways?
Show answer
Peel from the bottom. 4 ways = 22. 3 ways = 22 + 8 (lose LRU) = 30. 2 ways = 30 + 12 (lose pos 2) = 42 misses. Utility(2→4) = miss₂ − miss₄ = 42 − 22 = 20 misses saved.Misconception: peeling from the MRU end. You lose the deepest (least recent) positions first when the cache shrinks; MRU hits survive longest.
Problem 3 · medium
The "tasting the soup" analogy: UCP samples 32 of 1024 sets. Complete and explain: "The spoonful only represents the pot if ______." What does UCP check to be sure?
Show answer
"...the pot is well-stirred" — i.e. the sets behave similarly, with low variance. UCP's Chebyshev bound says the sampling error shrinks with low variance (P(off ≥ 1 way) ≤ σ²/n), and the paper checks that real workloads have σ² < 3, so 32 sets suffice. If sets varied wildly, 32 wouldn't be representative.Misconception: that more samples are always needed. The needed count depends on variance, not cache size — for well-stirred workloads, 32 of 1024 is plenty.
Problem 4 · hard
Greedy gives the next way to whoever benefits most from it right now. For which utility-curve shape is greedy provably optimal, and construct a short example where it fails badly.
Show answer
Greedy is optimal when all curves are convex (diminishing returns), because the best next way is always the globally best move. It fails on non-convex curves: e.g. program art saves 0 misses for ways 1–7, then 100 misses at way 8. A rival saves 1 miss per way. Greedy always picks the rival (gain 1 > 0) and never gives art its 8th way, missing the 100-miss payoff. Lookahead sees art's 8-way jump has marginal utility 100/8 = 12.5 per way and grabs it.Misconception: that greedy is "good enough." It's exactly optimal for convex curves but can be arbitrarily bad on the step-function curves real benchmarks (art, galgel) exhibit.
Problem 5 · medium
An earlier scheme estimated utility by reading recency information from inside the live shared cache, and got +4%. UCP uses a separate monitor and gets +11%. What goes wrong with reading from the live cache?
Show answer
Pollution. Inside the shared cache, a program's recency positions — and how many of its lines even survive in each set — are disturbed by the co-runner's accesses. So the "utility" you measure is contaminated by the neighbor, not the program's true standalone utility for all 16 ways. UCP's separate monitor measures each program in isolation, so the curve is clean — worth roughly 3× the benefit (the "two thermometers" analogy).Misconception: that the separate monitor is just an implementation detail. The isolation is the whole point — it's what makes the measurement trustworthy.
Problem 6 · medium
UCP re-decides every 5M cycles and halves the counters after each interval. For a few workloads (twolf-vpr, gzip-twolf) plain LRU actually beats UCP. Explain why, using these two design choices.
Show answer
UCP's decision is coarse in time (every 5M cycles) and uniform across all sets. Workloads whose useful-vs-useless behavior flips faster than 5M cycles change phase between decisions, so UCP's stale, global split lags. Plain LRU adapts continuously — at every access and independently per set — so it can ride fast phase changes UCP can't see yet. The halving softens but doesn't eliminate the lag. It's the deliberate cost of being cheap and global.Misconception: that UCP always beats LRU. It wins on average (+11%) but trades fine-grained adaptivity for low overhead, so it can lose on fast-phasing workloads.
Problem 7 · hard
UCP enforces the partition with a single owner bit per line and reclaims ways only on a miss. Why "only on a miss," and what would go wrong if a grown quota seized its new ways immediately?
Show answer
Reclaiming on a miss makes the quota a soft cap that fills gradually as the winning program actually needs space. If a grown quota instead instantly evicted the loser's lines, it would throw out blocks the loser is still using and that the winner hasn't even asked for yet — pure loss on both sides until the winner's demand catches up. Demand-paced reclamation keeps the loser's hot blocks alive until the moment they're genuinely displaced, so no cache sits idle and nothing useful is evicted prematurely.Misconception: that a bigger quota should mean immediately more lines. The quota is a ceiling realized through misses, not a reservation that pre-empts the neighbor.
Problem 8 · easy
Two programs both have saturating utility, and the cache is large enough to fit both their working sets. How much does UCP help, and why?
Show answer
Barely at all. If both saturate and both fit, neither is starved under LRU — there's no misallocation to fix, so weighted speedup is already near ideal (the paper's Type E workloads). UCP helps most when utilities differ sharply (one high, one low) or when both are high and high-demand, forcing a real trade-off.Misconception: that smarter partitioning always helps. It only helps when LRU's demand-split actually misallocates; when nobody is starved, there's nothing to gain.
Quick answers

Frequently asked questions

Twelve questions newcomers ask most. Tap any to expand.

Q1What does "utility" actually mean here?▾
Concretely: the number of cache misses a program would save if you gave it more ways. Utility of going from a to b ways = miss_a − miss_b. It's "benefit from cache," measured in misses avoided — deliberately different from "demand," which is how much cache a program tries to use.
Q2Why does plain LRU make a "bad" split if LRU is a good policy?▾
LRU is great at deciding what to keep within a fixed amount of cache. But across two programs it implicitly hands more cache to whoever touches more blocks (demand), with no notion of who benefits. A streaming program with huge demand and zero reuse gets rewarded anyway, starving a neighbor who'd use the space. UCP keeps LRU inside each program's share but decides the shares by utility.
Q3How can one monitor know the miss count for all 16 cache sizes at once?▾
The LRU "stack property": anything that hits in an N-way cache also hits in any larger one. So if you track which recency position each hit lands in (MRU, position 1, …), you can compute the miss count for every smaller cache by peeling off the deepest positions. One tag directory plus a counter per position gives the whole curve.
Q4Why only 32 sampled sets — isn't that too few out of 1024?▾
A statistics bound (Chebyshev) says the sampling error depends on how much sets differ (variance), not on the total number of sets. Measured workloads have low variance (σ² < 3), so 32 randomly-chosen sets estimate the cache-wide decision to within one way with high probability. Sampling more barely helps; sampling fewer (≤ 8) starts to hurt on some workloads.
Q5How big is the hardware overhead, really?▾
About 1920 bytes per monitor (one per core) — roughly 0.2% of the 1 MB cache. A dual-core system spends under 0.4% of cache area; quad-core under 1%. Plus one owner bit per cache line and a small adder/shifter. Crucially, none of it sits on the critical path, so it doesn't slow normal hits.
Q6How does UCP physically enforce "program A gets 13 ways"?▾
Each cache line carries a 1-bit tag for which core installed it. On a miss, the replacement engine counts how many lines in that set belong to the missing program. If it's under quota, it evicts the LRU line among the other program's lines (taking a way back); otherwise it evicts its own LRU line. Quotas fill gradually, only as misses occur.
Q7What is "marginal utility" and why introduce it?▾
Marginal utility = utility per way = (misses saved) ÷ (ways added). Plain utility tells you total benefit; marginal utility tells you benefit per unit of cache, which is what you want when comparing programs for the next allocation. It's the key to Lookahead: a program with a delayed payoff can have low utility for one way but high marginal utility for a bigger jump.
Q8Why does the simple "greedy" algorithm fail?▾
Greedy gives the next way to whoever benefits most from that single way. That's optimal only when every program's benefit curve is convex (diminishing returns). Real programs (art, galgel) have non-convex, step-function curves: zero benefit until a threshold, then a cliff. Greedy, seeing zero gain from the first way, never invests enough to reach the cliff. Lookahead scans the whole curve and can commit to the jump.
Q9Does UCP always beat LRU?▾
No — on average it's +11% (up to +23%), but a few workloads (twolf-vpr, crafty-perl, gzip-twolf) do slightly worse. UCP re-decides only every 5M cycles and applies one split across all sets, so it lags workloads whose useful/useless phases flip quickly, where per-access, per-set LRU adapts faster. It's an honest trade for low overhead.
Q10Is this the same as cache "ways" being turned off to save power?▾
No. UCP doesn't turn ways off — it allocates all 16 ways between programs (e.g. 13 to one, 3 to the other). Every way stays active and useful; the question is only who gets to use each one. (The paper does vary ways to measure utility curves, but that's the experiment, not the mechanism.)
Q11Does UCP work for more than two programs?▾
Yes — that's what the Lookahead algorithm is for. Trying every split is impossible with many programs (a 32-way cache among 8 programs has >15 million splits, and the exact optimum is NP-hard). Lookahead finds a near-best split in about N²/2 operations (≤512 for a 32-way cache), and on a quad-core it matches the exhaustive optimum.
Q12What did the authors leave for future work?▾
They flag extending UCP to SMT (hyperthreaded) cores, partitioning between demand and prefetch streams, estimating CPI directly to provide quality-of-service guarantees, supporting execution-time fairness, and handling multithreaded workloads where threads share data. UCP as presented targets multiprogrammed (independent-program) workloads on CMPs.

Glossary & further reading

Every term in one place, then a path to go deeper.

Core terms
UCP (Utility-Based Cache Partitioning)The paper's mechanism: divide a shared cache between programs by their utility (misses saved per way), not by demand.
UtilityThe benefit a program gets from cache: U(a→b) = miss_a − miss_b, the misses saved by growing from a to b ways.
DemandHow much cache a program tries to use — roughly the number of unique blocks it touches. What LRU implicitly rewards, and not the same as utility.
UMON (Utility Monitor)A per-core circuit — a tag-only Auxiliary Tag Directory plus recency-position counters — that measures a program's full miss-vs-ways curve, isolated from the co-runner.
Stack propertyA property of LRU: a hit in an N-way cache is also a hit with more ways. Lets one directory reveal the miss count for every smaller cache size.
Dynamic Set Sampling (DSS)Monitoring only a few representative sets (32 of 1024) and inferring the whole cache — the trick that shrinks UMON under 2 kB.
Marginal utilityUtility per unit of cache: MU(a→b) = (miss_a − miss_b)/(b − a). The signal Lookahead uses to handle delayed (non-convex) payoffs.
Lookahead algorithmUCP's scalable partitioner: each round give a program the block of ways with the highest marginal utility. Handles non-convex curves; ~N²/2 operations.
Supporting terms
Way / setA cache is split into sets; each set has several slots (ways). UCP allocates cache in units of ways.
LRULeast-Recently-Used eviction — the standard policy. On a shared cache it partitions by demand.
MPKI / CPIMisses Per 1000 Instructions / Cycles Per Instruction — the miss-rate and speed scorecards. Lower is better; usually move together.
Auxiliary Tag Directory (ATD)A small copy of the tags (no data) the monitor maintains to track hits/misses for a single program.
Recency positionWhere a hit lands in the LRU order (MRU, position 1, …, LRU). One counter per position yields the whole miss curve.
Combined utilityThe sum of programs' utilities for a given split; UCP picks the split that maximizes it (= minimizes total misses).
Owner bitOne bit per cache line recording which core installed it, used to enforce the way quota on eviction.
Convex / non-convex curveConvex = diminishing returns (greedy-friendly). Non-convex = delayed payoff / step function (needs Lookahead).
Weighted speedup / throughput / fairnessThe three evaluation metrics: execution-time gain (Σ IPCᵢ/SingleIPCᵢ), raw throughput (Σ IPCᵢ), and a fairness-balancing harmonic mean.
Half-and-HalfA baseline that statically splits the cache 50/50 — gives isolation but can't adapt to demand.
Working setThe set of blocks a program actively reuses. If it fits in the cache, the program saturates; if not, it keeps benefiting.
