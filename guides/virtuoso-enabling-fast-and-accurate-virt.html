Simulating the kernel without booting the whole OS

Full-system simulators are slow and heavyweight; emulation-based simulators fake the OS with fixed-latency guesses. Virtuoso threads the needle: a lightweight userspace kernel that imitates only the memory-management code you care about — fast, flexible, and accurate.

📄 Kanellopoulos et al. · ETH Zürich 🎓 Some VM & simulator basics assumed ⏱ ~25 min read

1Why virtual memory matters for data-intensive apps▾

Virtual memory (VM) enables applications to work with a large address space while the OS allocates physical pages on-demand. Address translation (via TLBs and page tables) maps virtual → physical, and page faults allocate new physical pages. Long-running workloads (e.g., graph analytics) spend up to 40% of total time on translation, and short-running workloads (e.g., serverless functions) spend up to 95% on page fault handling. As memory grows (hybrid DRAM/NVM, CXL), these overheads worsen.

Now you're ready: Virtuoso is a simulator for VM research — it models both address translation and page fault handling with the accuracy of a real OS, at the speed of lightweight emulation.
2What a simulator needs to model VM accurately▾

VM spans the OS (page fault handler, physical memory allocator, page table updates) and hardware (TLB, page table walker, caches). The OS decides which pages to allocate and when; the hardware walks the page table and caches translations. Fixed-latency approximations (e.g., "PTW = 50 cycles") miss dynamic effects: page fault latency varies 1000× depending on fragmentation and huge-page policy; PTW latency varies 4× depending on cache hits and memory contention.

Now you're ready: Virtuoso models the OS side by executing a real (simplified) kernel that allocates pages, updates page tables, and injects its disassembled instruction stream into the simulator — so the simulator sees the latency, cache pollution, and memory contention the OS causes.
3Emulation-based vs full-system simulators▾

Emulation-based simulators (e.g., Sniper, ChampSim) emulate system calls with fixed latencies — fast (20 s to simulate 1B instructions) but inaccurate for VM (e.g., Sniper's fixed PTW latency leads to 35% IPC error). Full-system simulators (e.g., gem5-FS) boot a full Linux kernel on top of the simulator — accurate but slow (77% slower than gem5-SE, 1.69 GB RAM per job) and hard to modify (need to patch the kernel, rebuild, verify).

Now you're ready: Virtuoso sits between them: a userspace kernel (MimicOS) written in C++ that mimics only the memory-management routines, producing the same outcomes as Linux but 49% faster than gem5-FS and with only 20% overhead over emulation-based simulators.
4What "userspace kernel" means here▾

A userspace kernel is a standalone C++ program that runs outside the simulator, imitating the behavior of a subset of the OS kernel (e.g., page fault handler, THP allocator, page table manager). When the simulator encounters a page fault, it sends a message to the userspace kernel; the kernel executes the fault handler (allocating a page, updating the page table); a binary instrumentation tool (Intel Pin or DynamoRIO) captures the kernel's instructions and sends them to the simulator. The simulator models their execution (cache misses, memory latency, pipeline stalls), then resumes the application. The kernel never "boots" — it just handles requests on-demand.

Now you're ready: Virtuoso's userspace kernel (MimicOS) is modular — you pick which OS modules to include (e.g., THP, swap, khugepaged) to balance accuracy and simulation speed, and you can easily add new modules (e.g., a custom physical memory allocator) without touching a production kernel.
5Why injecting the kernel's instruction stream is key▾

If the simulator only receives the functional outcome of an OS routine (e.g., "page fault resolved → physical address 0x12345"), it can update the page table but can't estimate the performance impact: cache pollution, memory interference, pipeline stalls, or the wall-clock latency. By injecting the kernel's disassembled instructions (e.g., mov, add, load, store) into the simulator's core model, the simulator sees the kernel's behavior as it sees the application's — computing accurate latency, cache state, and memory contention.

Now you're ready: Virtuoso's "instruction stream channel" streams the kernel's instructions to the simulator in parallel with their execution, so the simulator can model them cycle-by-cycle.
6Validation against a real system▾

The paper validates Virtuoso+Sniper against an Intel Xeon Gold 6226R (2.9 GHz, 256 GB DDR4-2400, Linux 5.15). Key results: (1) L2 TLB MPKI estimated with 82% accuracy, (2) PTW latency with 85% accuracy, (3) page fault latency with 66% (up to 79%) accuracy, (4) IPC with 80% accuracy (vs 66% for baseline Sniper). The overhead is 35% simulation time vs baseline Sniper, and only 20% averaged across four simulators (Sniper, ChampSim, Ramulator, gem5-SE) — much faster than gem5-FS (77% overhead).

Now you're ready: Virtuoso achieves near-real-system accuracy at emulation-like speed, making it practical for rapid VM design-space exploration.

Virtuoso in a nutshell

The problem, the lightweight kernel that fixes it, and how it all connects.

Evaluating new VM schemes (e.g., hash-based page tables, huge-page allocators, contiguity-aware TLBs) requires modeling both OS routines (page fault handling, THP, physical memory allocation) and hardware (TLB, PTW, caches). Emulation-based simulators use fixed latencies — fast but inaccurate (e.g., Sniper's fixed PTW latency → 35% IPC error). Full-system simulators boot a real Linux kernel — accurate but slow (77% slower, 1.69× RAM), hard to modify, and include irrelevant kernel code (scheduler, file systems).

Virtuoso's key idea: run a lightweight userspace kernel (MimicOS) that mimics only the memory-management routines, written in C++, as a separate process. When the simulator triggers a page fault or system call, it sends a message to MimicOS via shared memory. MimicOS executes the handler (e.g., allocates a 2 MB page, updates the page table), and a binary instrumentation tool (e.g., Intel Pin) captures the disassembled instructions and streams them to the simulator's core model. The simulator models their execution (cache misses, memory latency, pipeline stalls), then resumes the application. This way, Virtuoso gets the accuracy of full-system simulation (the kernel's instruction stream is modeled cycle-by-cycle) at the speed of emulation-based simulation (only the VM-relevant code runs, no full OS boot).

The payoff: MimicOS achieves 80% IPC accuracy vs 66% for baseline Sniper, with only 20% simulation time overhead (averaged across Sniper, ChampSim, Ramulator, gem5-SE). Porting MimicOS to a new simulator requires modifications listed in Table 3: ChampSim 123 LoC, Sniper 261 LoC, Ramulator2 206 LoC, gem5-SE 265 LoC. The paper integrates MimicOS with five simulators and builds VirTool, a comprehensive VM toolset including Radix PT, ECH, HDC, Utopia, Midgard, RMM, and more.

  • Virtuoso's MimicOS is a lightweight userspace kernel (written in C++) that mimics only the memory-management routines (page fault handler, THP, page table manager).
  • MimicOS runs as a separate process and communicates with the simulator via shared memory. A binary instrumentation tool (Intel Pin) captures MimicOS's instruction stream and injects it into the simulator's core model.
  • The simulator models the kernel's instructions cycle-by-cycle → accurate latency, cache pollution, memory contention (same as full-system), but only the VM-relevant code runs (fast as emulation-based).

Five everyday pictures

How to think about Virtuoso

Each picture stands alone. Every card ends with an honest note about where the picture stops being true.

The core idea · Userspace kernel

A stunt double for the OS

A movie doesn't film every scene with the real actor — for the dangerous bits, a stunt double steps in, mimicking only the required actions. MimicOS is a stunt double for the Linux kernel: it mimics the page fault handler, THP allocator, and page table manager (the "dangerous" parts for VM research) without booting the full kernel (the "actor"). The simulator only sees the instructions MimicOS executes (the "stunt"), not the whole OS.

Where it breaks: a stunt double is indistinguishable from the actor on screen, but MimicOS is a simplified kernel — it lacks the scheduler, file systems, network stack, and other subsystems. It's accurate for VM research but not a drop-in replacement for a full OS.
Architecture · Separate processes

Two dancers in sync

In a synchronized dance, two performers mirror each other's moves while staying on separate stages. Virtuoso's simulator and MimicOS run as separate processes (separate stages) that communicate via shared memory (the choreography). When the simulator encounters a page fault, it sends a message; MimicOS executes the handler; the simulator models the instructions. They stay in sync without merging into one process.

Where it breaks: dancers can improvise, but Virtuoso's communication is strictly one-way: simulator → MimicOS → simulator. The kernel can't "query" the simulator's state mid-execution (e.g., ask how many cache misses occurred so far).
Communication · Two channels

Walkie-talkies vs a parcel

A construction crew uses two channels: walkie-talkies for quick commands ("start the crane") and parcels for detailed blueprints. Virtuoso's functional channel is the walkie-talkie (shared memory + magic ops) — fast messages like "page fault at 0xABC". The instruction stream channel is the parcel — Pin sends the kernel's disassembled instructions (the "blueprints") to the simulator's core model, which executes them cycle-by-cycle.

Where it breaks: walkie-talkies are instant, but Virtuoso's instruction stream channel has latency — the instrumentation tool must capture and send instructions. The paper hides this latency by running the simulator and Pin in parallel (production/consumption pipeline), but it's not zero-cost.
Modularity · Pick your modules

Lego bricks, not a monolith

You don't buy a whole Lego castle when you only need the drawbridge — you pick the pieces. MimicOS is modular: you select which kernel modules to include (e.g., page fault handler + THP allocator + khugepaged) based on what you're studying. If you're evaluating a new TLB design, you might skip swap and page cache to speed up simulation. If you're studying page fault latency, you include them. Each module is a "Lego brick" you add or remove.

Where it breaks: Lego bricks snap together without dependencies, but kernel modules have subtle interactions (e.g., khugepaged assumes THP is enabled). MimicOS enforces some dependencies, but researchers must understand which modules interact to avoid functional bugs.
Speed vs accuracy · The sweet spot

Express checkout vs full cart scan

A full grocery cart scan (full-system sim) is thorough but slow (77% overhead). Guessing the total from a few items (emulation-based sim) is fast but often wrong (35% IPC error). Virtuoso is the express checkout: it scans only the VM-relevant items (page fault handler, THP, page table), achieving 80% IPC accuracy with only 20% overhead. You get most of the accuracy of a full scan at express-lane speed.

Where it breaks: express checkout assumes you're only buying a few items. If your research needs all kernel subsystems (e.g., studying scheduler/VM interactions), MimicOS isn't enough — you need the full scan (gem5-FS).
The full explanation

How it actually works

Section by section, with the paper's key mechanisms rebuilt as visuals you can read at a glance.

1 · The problem — emulation vs full-system

Emulation-based simulators (e.g., Sniper) emulate system calls with fixed latencies. For example, Sniper assumes every page table walk (PTW) takes a fixed number of cycles (set to the average PTW latency measured on a real system). But the paper shows that PTW latency varies 4× (39–180 cycles) across workloads depending on memory intensity, cache hits, and page table structure. Similarly, page fault (PF) latency varies 1000× (0.5–500 μs) depending on fragmentation, huge-page availability, and fallback mechanisms. Fixed latencies miss these dynamics, leading to 35% IPC error.

Full-system simulators (e.g., gem5-FS) boot a full Linux kernel on top of the simulator. The kernel executes all OS routines (scheduler, file systems, network stack, VM), and the simulator models their instructions. This approach supports realistic memory management, but is slow (77% simulation time overhead) and heavyweight (1.69 GB RAM per job). Modifying the kernel requires patching the source, recompiling, and verifying — a high barrier for rapid prototyping.

🌱
Concretely: For an I/O-bound workload, PTW latency is about 39 cycles. For SSSP from GraphBIG, the real PTW latency is 180 cycles — a 4× difference. This variance is not captured by fixed latencies, leading to IPC estimation errors.

2 · MimicOS — a lightweight userspace kernel

MimicOS is a standalone C++ program that mimics the memory-management routines of the Linux kernel. It includes (but is not limited to): (1) page fault handler (do_page_fault), (2) physical memory allocator (buddy allocator, slab allocator), (3) THP allocator (transparent huge pages, khugepaged), (4) page table manager (4-level radix tree, updates PGD/PUD/PMD/PTE), (5) page cache (file-backed pages), (6) swap cache (swapped-out pages).

MimicOS is modular: you pick which modules to include. For example, if you're studying TLB prefetching, you might skip swap and page cache (they don't affect address translation). If you're studying page fault latency, you include them (they affect whether a fault is major or minor).

MimicOS is portable: it's written in standard C++ (no kernel-specific constructs), so it compiles on any platform. The paper integrates MimicOS with five simulators (Sniper, ChampSim, Ramulator, gem5-SE, MQSim) with modifications per Table 3: ChampSim 123 LoC, Sniper 261 LoC, Ramulator2 206 LoC, gem5-SE 265 LoC.

💡
Why userspace? Writing MimicOS in userspace (vs in-kernel) enables (1) easy debugging (printf, gdb), (2) easy development (C++ vs kernel C, no need to reboot), (3) easy instrumentation (Intel Pin works on userspace binaries), (4) easy portability (no kernel dependencies). The tradeoff is that MimicOS lacks some kernel features (e.g., real interrupts, preemption), but the paper shows this doesn't affect VM simulation accuracy.

3 · Communication channels — functional + instruction stream

Virtuoso establishes two communication channels between MimicOS and the simulator:

  1. Functional channel: shared memory (POSIX shm) + magic operations (e.g., xchg R1,R2 in Sniper). When the simulator triggers a page fault, it writes the virtual address to shared memory and executes a magic op to notify MimicOS. MimicOS reads the request, executes the fault handler, writes the outcome (e.g., physical address, page size) to shared memory, and executes a magic op to notify the simulator. The simulator reads the outcome and resumes the application.
  2. Instruction stream channel: MimicOS runs on top of a binary instrumentation tool (e.g., Intel Pin), which captures the disassembled instructions of the fault handler and streams them to the simulator. The simulator's core model executes the instructions (modeling cache misses, memory latency, pipeline stalls), computing the latency and microarchitectural effects of the fault handler.

The two channels run in parallel: while MimicOS executes the fault handler (producing the functional outcome), Pin streams the instructions to the simulator (which models their execution). This hides the instrumentation latency and keeps simulation speed high.

🍯
Insight: the functional channel ensures correctness (the simulator gets the right physical address, page size, page table updates), and the instruction stream channel ensures accuracy (the simulator models the latency, cache pollution, memory contention of the fault handler). Together, they bridge emulation-based and full-system simulation.

4 · Page fault handling — the full flow

Here's what happens when the simulator encounters a page fault at virtual address 0xABC:

  1. The simulator's MMU model triggers a page fault (TLB miss → PTW miss → PTE not present).
  2. The simulator writes 0xABC to the functional channel (shared memory) and executes a magic op to notify MimicOS.
  3. MimicOS wakes up, reads 0xABC from shared memory, and calls do_page_fault(0xABC).
  4. do_page_fault walks the page table (4-level radix tree), finds that the PTE is empty, and allocates a new physical page via the buddy allocator. It checks if THP is enabled and if a 2 MB page is available; if so, it allocates a 2 MB page; otherwise, a 4 KB page.
  5. While do_page_fault executes, Intel Pin instruments the instructions (mov, add, load, store) and sends them to the simulator via the instruction stream channel.
  6. The simulator's core model executes the instructions, modeling cache misses (e.g., page table entries miss in L2 cache), memory latency (e.g., load from buddy allocator's free list), and pipeline stalls. It computes the total latency of do_page_fault (e.g., 2.2 μs on average, but up to 500 μs for outliers).
  7. do_page_fault updates the page table (writes the physical address to the PTE) and writes the outcome (physical address 0x12345, page size 2 MB) to the functional channel.
  8. do_page_fault executes a magic op to notify the simulator.
  9. The simulator reads the outcome, updates the TLB with the new translation (0xABC → 0x12345, 2 MB), and resumes the application.
💡
Why inject instructions? If the simulator only received the functional outcome (0xABC → 0x12345, 2 MB), it could update the TLB but couldn't estimate the page fault latency, cache pollution, or memory contention. By injecting the fault handler's instructions into the core model, the simulator sees the fault handler's behavior as it sees the application's — cycle-by-cycle, with cache misses, memory accesses, and pipeline stalls. This is what makes Virtuoso accurate.

5 · Integration with simulators — trace-based vs execution-driven

Virtuoso integrates with five simulators: Sniper (execution-driven), ChampSim (trace-based), Ramulator (trace-based), gem5-SE (emulation-based), MQSim (SSD simulator). The integration requires modifications per Table 3: ChampSim 123 LoC, Sniper 261 LoC, Ramulator2 206 LoC, gem5-SE 265 LoC, divided into three steps:

  1. Establish functional channel: attach the MMU model to MimicOS via shared memory. When the MMU triggers a page fault, write the VA to shared memory and execute a magic op to notify MimicOS. When MimicOS resolves the fault, read the outcome from shared memory.
  2. Establish instruction stream channel: attach the core model to Pin's output. When Pin streams instructions, inject them into the core model (replacing the application's instruction stream temporarily). When MimicOS finishes, switch back to the application's instruction stream.
  3. Synchronize: use magic ops (e.g., xchg R1,R2 in Sniper, custom ISA extensions in gem5) to notify the simulator when MimicOS starts/finishes executing.

For trace-based simulators (ChampSim, Ramulator), the integration is straightforward: Pin generates a trace file for MimicOS (in the same format as the application trace), and the simulator switches between the application trace and the MimicOS trace. For execution-driven simulators (Sniper), Pin injects instructions on-the-fly (no trace file needed).

🍯
Insight: the integration is lightweight because Virtuoso's design is simulator-agnostic. The functional channel uses standard POSIX primitives (shm), the instruction stream channel reuses the simulator's existing frontend (trace parser or instrumentation tool), and the synchronization uses simulator-specific magic ops (a small API surface).
Follow the numbers, step by step

Worked examples

Four traces using the paper's real numbers.

① Computing page fault latency from instruction counts

MimicOS's page fault handler executes approximately 1200 instructions (estimated based on typical handler complexity). The simulator's core model runs at 2.9 GHz with an IPC of 1.5. What's a rough estimate of page fault latency?

  1. Cycles for 1200 instructions at IPC 1.5: 1200 instr ÷ 1.5 IPC = 800 cycles.
  2. Time at 2.9 GHz: 800 cycles ÷ 2.9 GHz = 275.9 ns.
  3. Add memory latency: the handler loads from the buddy allocator's free list (1 DRAM access, ~70 ns), writes the PTE (1 DRAM access, ~70 ns). Total memory latency = 140 ns.
  4. Add cache misses: the handler misses L2 cache 3 times (page table entries not in cache), each L2 miss costs ~16 cycles (gem5 model). 3 × 16 = 48 cycles = 16.6 ns.
  5. Total latency: 275.9 + 140 + 16.6 = 432.5 ns (≈ 0.43 μs). The paper reports an average of 2.2 μs (includes outliers with huge-page fallback, page zeroing, khugepaged).
🍯
Insight: this example shows the minimum page fault latency (no fallback, no zeroing). The paper's average (2.2 μs) is 5× higher because most faults trigger additional work (e.g., allocating a 2 MB page requires scanning the buddy allocator's free list, which can take 1000+ cycles if fragmented). Virtuoso captures this variability by modeling the actual instruction stream, not a fixed latency.

② Overhead of integrating MimicOS with Sniper

Baseline Sniper simulates 1B instructions in 20 seconds. MimicOS+Sniper takes 27 seconds. What's the overhead, and where does it come from?

  1. Overhead: (27 − 20) ÷ 20 = 35% (matches the paper's Fig. 11 for randacc, the worst-case workload with 1 page fault per kilo instruction).
  2. Sources of overhead: The overhead comes from simulating MimicOS instructions injected into the core model. The paper reports average overhead of 20% across four simulators, with variance depending on page fault frequency (PFKI).
  3. Why not higher? The instruction stream channel runs in parallel with MimicOS (production/consumption pipeline), so instrumentation latency is mostly hidden. The paper also uses offline instrumentation (pre-instrument MimicOS's binary) for Ramulator, reducing overhead to 2%.
💡
Why 35% for randacc? randacc (random access, from GUPS) triggers 1 page fault per 1000 instructions (PFKI = 1). Each fault injects MimicOS instructions into the simulator, so the simulator executes more total instructions. The overhead is sublinear because the simulator's frontend (Pin) caches MimicOS's instructions and reuses them across faults.

③ Memory overhead of MimicOS vs gem5-FS

gem5-SE uses 1.0 GB RAM per simulation job. gem5-FS uses 1.69 GB. MimicOS+gem5-SE uses 1.02 GB. Why does MimicOS add so little overhead?

  1. gem5-FS overhead: the full Linux kernel (binary + page cache + buffers) requires ~700 MB of RAM, plus the simulator's internal state (500 MB), totaling 1.69 GB.
  2. MimicOS overhead: MimicOS's binary is ~5 MB (only memory-management code), plus shared memory (10 MB for functional channel), plus Pin's instrumentation cache (~5 MB), totaling ~20 MB overhead → 1.02 GB total.
  3. Why so small? MimicOS doesn't boot a full kernel — it's a standalone program that only includes the VM subsystem. It doesn't allocate buffers, page caches, or file system metadata. It uses mmap (not malloc) for large allocations, so the OS can reclaim unused memory.
🍯
Insight: the memory overhead is critical for simulation throughput. If a machine has 64 GB RAM, gem5-FS can run 64 ÷ 1.69 = 37 jobs in parallel, while MimicOS+gem5-SE can run 64 ÷ 1.02 = 62 jobs — 1.68× more throughput. The paper also notes that Sniper+MimicOS uses 0.8 GB (using offline instrumentation, no online Pin), enabling 80 parallel jobs on the same machine.

④ Validation — IPC accuracy for SSSP

The real Xeon system runs SSSP (single-source shortest path from GraphBIG). Baseline Sniper (fixed PTW latency) has estimation error. MimicOS+Sniper improves accuracy. What's the difference?

  1. Baseline Sniper: Uses fixed PTW latency, leading to 35% IPC estimation error (reported in paper).
  2. MimicOS+Sniper: Achieves 80% IPC accuracy overall (vs 66% for baseline Sniper).
  3. Why Sniper underestimates: SSSP has very high L2 TLB MPKI (40 misses per kilo instructions, measured in Fig. 10). PTW latency varies significantly (39–180 cycles across workloads). Fixed latencies miss this dynamic behavior.
  4. Why MimicOS is closer: MimicOS models the actual PTW (walks the 4-level radix tree, models cache misses for page table entries). For SSSP, the page table entries are scattered in memory (poor spatial locality), so most PTWs miss in the cache hierarchy. MimicOS's model captures this behavior more accurately than fixed latencies.
💡
Remaining error sources: MimicOS still has some error because (1) the simulator's DRAM model is approximate (Ramulator-based, but not cycle-exact for the real Xeon's memory controller), (2) MimicOS doesn't model speculative PTWs (the real Xeon prefetches page table entries speculatively), (3) the real system has other VM overheads (TLB shootdowns, page reclamation) that MimicOS doesn't model. The paper notes that 80% average IPC accuracy (across all workloads) is state-of-the-art for emulation-based simulators.

Problem 1 · easy
Baseline Sniper uses a fixed PTW latency. The real system's PTW latency varies from 39 cycles (I/O workload) to 180 cycles (SSSP). For which type of workload does fixed latency produce larger estimation errors?
Show answer
Workloads with PTW latency far from the fixed value. If the fixed latency is set to an average, workloads at the extremes (very low or very high actual PTW latency) will have larger errors. The paper shows fixed latencies lead to 35% IPC error because they can't capture the 4× variance (39–180 cycles) across workloads.Misconception: that fixed-latency approximations are "conservative" (always underestimate performance). Fixed latencies can either underestimate or overestimate depending on the workload — they're not systematically biased, just inaccurate for dynamic behaviors.
Problem 2 · medium
MimicOS's page fault handler executes approximately 1200 instructions. The simulator runs at 2.9 GHz with IPC 1.5. The handler triggers 3 L2 cache misses (16 cycles each) and 2 DRAM accesses (70 ns each). What's a rough estimate of total page fault latency in nanoseconds?
Show answer
Cycles for 1200 instr at IPC 1.5: 1200 ÷ 1.5 = 800. Time at 2.9 GHz: 800 ÷ 2.9e9 = 275.9 ns. L2 misses: 3 × 16 = 48 cycles = 16.6 ns. DRAM accesses: 2 × 70 = 140 ns. Total: 275.9 + 16.6 + 140 = 432.5 ns (≈ 0.43 μs).Misconception: that cache misses and DRAM accesses can be added to instruction count before dividing by IPC. The IPC applies only to instructions, not to memory latency — memory latency is computed separately and added at the end.
Problem 3 · medium
MimicOS+Sniper has 35% simulation time overhead vs baseline Sniper for randacc (PFKI = 1). If a workload has PFKI = 0.1 (10× fewer page faults), roughly what overhead would you expect?
Show answer
The overhead scales roughly with PFKI (more faults → more MimicOS instructions injected). For randacc (PFKI = 1), overhead = 35%. For PFKI = 0.1, overhead ≈ 35% × 0.1 = 3.5%. The paper reports 20% average overhead across all workloads (which have PFKI between 0.01 and 1), consistent with this scaling.Misconception: that MimicOS has a fixed "base" overhead independent of the workload. The overhead is dominated by simulating MimicOS instructions, which scales with the number of page faults — so workloads with fewer faults (low PFKI) have lower overhead.
Problem 4 · hard
gem5-FS boots a full Linux kernel and simulates all OS routines (scheduler, file systems, network stack, VM). Virtuoso's MimicOS only mimics the VM subsystem. Name two scenarios where gem5-FS would be more accurate than Virtuoso, and one where Virtuoso would be faster without losing accuracy.
Show answer
gem5-FS more accurate: (1) Studying interactions between the scheduler and VM (e.g., how TLB shootdowns affect context-switch latency) — MimicOS doesn't model the scheduler. (2) Studying network-stack-induced memory pressure (e.g., how socket buffers compete for physical pages with the application) — MimicOS doesn't model the network stack. Virtuoso faster, same accuracy: Studying TLB prefetching (only affects address translation, not scheduler or network stack) — MimicOS models TLB/PTW/page faults, so it's as accurate as gem5-FS for VM-only studies, but 49% faster (no need to simulate irrelevant OS code).Misconception: that full-system simulation is always "better" than Virtuoso. Full-system is more complete (models all OS subsystems), but Virtuoso is more efficient for VM-focused studies — the key is matching the simulation methodology to the research question.

The integration is mostly "plumbing" — connecting existing components (MMU, core model, shared memory) via communication channels. The simulator's core logic is unchanged.
Problem 6 · easy
MimicOS is written in C++ and runs in userspace (not in-kernel). Name two advantages of this design choice for rapid prototyping of new VM schemes.
Show answer
(1) Easy debugging: you can use printf, gdb, valgrind, and other userspace tools (can't use these in-kernel without special setup). (2) Easy development: you write C++ (not kernel C), compile with g++/clang (not kernel build system), and test without rebooting (just rerun the MimicOS binary). This makes the edit-compile-test cycle much faster than modifying a production kernel.Misconception: that userspace kernels are "toy" implementations. MimicOS is simplified (lacks scheduler, file systems, network stack), but it's not a toy — it mimics the memory-management routines of Linux with high fidelity (66–79% page fault latency accuracy), using the same algorithms (buddy allocator, slab allocator, 4-level radix tree).
Problem 7 · hard
The paper reports that MimicOS achieves 66% (up to 79%) page fault latency accuracy vs the real Linux kernel. Why isn't it 100%? Name potential sources of error.
Show answer
Potential sources: (1) Approximate DRAM model: the simulator's DRAM model (Ramulator-based) is approximate — it doesn't capture all memory controller behaviors with the same fidelity as real hardware. (2) Simplified OS modules: MimicOS simplifies some OS routines or omits certain subsystems. (3) Measurement methodology: the validation compares simulated vs real system using cosine similarity, which measures distribution shape rather than point-by-point accuracy.Misconception: that 66% accuracy means MimicOS is "wrong" 34% of the time. The error is a cosine similarity metric (measures how well the latency distribution matches the real one), not a binary right/wrong. MimicOS captures the shape of the distribution well but may underestimate outliers due to simplified models.
Problem 8 · medium
The instruction stream channel runs "in parallel" with MimicOS's execution (production/consumption pipeline). Explain why this parallelism is critical for simulation speed (hint: what would happen if the simulator waited for Pin to finish instrumenting before resuming the application?).
Show answer
If the simulator waited for Pin to finish instrumenting all of MimicOS's instructions before resuming the application, the simulation would stall for the entire page fault handler execution. By running Pin and the simulator in parallel, the simulator can start executing MimicOS instructions while Pin is still instrumenting later instructions, hiding the instrumentation latency. This is why the paper reports only 20% overhead (instead of higher overhead if instrumentation was serialized).Misconception: that parallelism eliminates overhead entirely. The parallelism hides instrumentation latency, but the simulator still has to execute MimicOS's instructions (which adds overhead for high-PFKI workloads). The parallelism prevents instrumentation latency from adding another layer of overhead on top.
Quick answers

Frequently asked questions

Twelve questions newcomers ask most. Tap any to expand.

Q1What does "imitation-based OS simulation" mean?▾
Instead of emulating the OS with fixed latencies (emulation-based) or booting a full Linux kernel (full-system), Virtuoso runs a lightweight userspace kernel (MimicOS) that imitates only the memory-management routines (page fault handler, THP allocator, page table manager). MimicOS is written in C++, runs as a standalone process, and is instrumented by Intel Pin — so the simulator sees the kernel's instruction stream (just like the application's) and models it cycle-by-cycle. This is "imitation" because MimicOS mimics the behavior of Linux's VM subsystem without being a full kernel.
Q2Why not just use gem5-FS (full-system mode)?▾
gem5-FS boots a full Linux kernel, which is accurate but slow (77% overhead, 1.69 GB RAM per job). It also simulates irrelevant OS code (scheduler, file systems, network stack) even if you're only studying VM. Virtuoso is 49% faster than gem5-FS (only 20% overhead on average) and uses less RAM (1.02 GB for gem5-SE+MimicOS) because it only simulates the VM subsystem. The tradeoff is that Virtuoso doesn't model interactions between VM and other OS subsystems (e.g., scheduler/VM interactions) — but for pure VM research, this is acceptable.
Q3How does MimicOS communicate with the simulator?▾
Two channels: (1) Functional channel (shared memory + magic ops): the simulator sends requests (e.g., "page fault at 0xABC") to MimicOS, and MimicOS sends outcomes (e.g., "allocated PA 0x12345, 2 MB page") back. (2) Instruction stream channel: Intel Pin instruments MimicOS's instructions and streams them to the simulator's core model, which executes them cycle-by-cycle. The two channels run in parallel: while MimicOS produces the functional outcome, Pin streams the instructions to the simulator.
Q4What's the simulation time overhead of MimicOS?▾
20% on average across four simulators (Sniper, ChampSim, Ramulator, gem5-SE). The overhead varies by workload: 2% for low-PFKI workloads (few page faults) to 35% for high-PFKI workloads (1 page fault per 1000 instructions). The overhead comes from simulating MimicOS's instructions injected into the core model. The paper hides instrumentation latency via parallelism (production/consumption pipeline).
Q5How accurate is MimicOS vs a real Linux kernel?▾
Page fault latency: 66% (up to 79%) cosine similarity vs real Linux. L2 TLB MPKI: 82% accuracy. PTW latency: 85% accuracy. IPC: 80% accuracy (vs 66% for baseline Sniper). The errors come from approximate DRAM models and simplified OS routines in MimicOS compared to full Linux.
Q6Can MimicOS model huge pages (THP)?▾
Yes. MimicOS includes a Linux-like THP allocator (transparent huge pages) with 4 KB and 2 MB pages. It also supports khugepaged (background daemon that merges 4 KB pages into 2 MB pages), HugeTLBFS (explicit huge-page reservation), and reservation-based THP. The paper's use case 2 (§7.5) evaluates different THP allocation policies (BD, CR-THP, AR-THP, Utopia) for LLM inference workloads.
Q7How hard is it to port MimicOS to a new simulator?▾
Table 3 shows lines of code added: ChampSim 123, Sniper 261, Ramulator2 206, gem5-SE 265. The integration requires (1) establishing the functional channel (attach MMU model to shared memory), (2) establishing the instruction stream channel (attach core model to Pin's output), (3) synchronization (use magic ops to notify the simulator when MimicOS starts/finishes).
Q8Can Virtuoso model nested virtualization (VM on VM)?▾
Yes. Virtuoso spawns two MimicOSes: one acts as the hypervisor (e.g., KVM), one acts as the guest OS (e.g., Linux). When the guest OS triggers a page fault, the simulator sends a message to the guest MimicOS → guest MimicOS sends a message to the hypervisor MimicOS → hypervisor MimicOS allocates a physical page → outcome propagates back to the simulator. VirTool already includes nested address translation (AMD-V nested paging, Intel EPT) in its hardware model.
Q9What VM schemes are included in VirTool?▾
Table 2 lists 20+ schemes: Radix PT (4-level, 5-level), ECH (elastic cuckoo hashing), HDC (hash-based direct-cache PT), HT (hash table PT), Utopia (restrictive virtual-to-physical mapping), Midgard (intermediate address space), RMM (range-based TLB + eager paging), Direct Segments, THP variants (Linux-like, reservation-based), TLB prefetching, page-size prediction, software-managed TLBs, nested TLB/PTW, Mondrian memory protection, and more. The paper also includes five use cases (§7.4–7.6.3) demonstrating how to use VirTool for VM research.
Q10Can Virtuoso model major page faults (disk I/O)?▾
Yes. MimicOS includes a page cache (stores recently-accessed file-backed pages) and a swap cache (stores swapped-out pages). When a page fault misses in both caches, MimicOS triggers a disk access, and the simulator models the disk latency via MQSim (an SSD simulator). The paper's use case (§7.6.2) evaluates the effect of memory fragmentation on swapping activity and shows that large restrictive segments (Utopia) increase swapping by 203× due to hash collisions that prevent data from being stored in memory.
Q11Does MimicOS replace the need for full-system simulation?▾
No — Virtuoso complements full-system simulators, not replaces them. For VM-focused research (e.g., evaluating a new TLB design, testing a huge-page allocator), Virtuoso is faster and easier to use than gem5-FS. For system-wide research (e.g., studying scheduler/VM interactions, network-stack-induced memory pressure), gem5-FS is more appropriate because it models the full OS. The key is matching the simulation methodology to the research question.
Q12Where can I get Virtuoso's source code?▾
Freely available at https://github.com/CMU-SAFARI/Virtuoso. The repo includes MimicOS (C++ source), integration with five simulators (Sniper, ChampSim, Ramulator, gem5-SE, MQSim), VirTool (20+ VM schemes), scripts to reproduce the paper's results, and documentation. The authors welcome contributions (new VM schemes, integration with new simulators, bug fixes).

Glossary & further reading

Core terms
VirtuosoA simulation framework for VM research that employs a lightweight userspace kernel (MimicOS) to model OS routines and inject their instruction stream into an architectural simulator.
MimicOSA lightweight userspace kernel (written in C++) that mimics the memory-management routines of Linux (page fault handler, THP allocator, page table manager, page cache, swap cache).
Imitation-based simulationA methodology where a lightweight userspace kernel imitates only the desired OS routines (vs emulation with fixed latencies or full-system with a real kernel).
Functional channelA communication channel (shared memory + magic ops) where the simulator sends requests (e.g., "page fault at 0xABC") to MimicOS and receives outcomes (e.g., "allocated PA 0x12345, 2 MB").
Instruction stream channelA communication channel where Intel Pin instruments MimicOS's instructions and streams them to the simulator's core model, which executes them cycle-by-cycle.
VirToolA comprehensive VM toolset built on Virtuoso+Sniper, including 20+ state-of-the-art VM schemes (Radix, ECH, HDC, Utopia, Midgard, RMM, THP variants, etc.).
Page fault (PF)An event where the MMU can't find a translation (TLB miss → PTW miss → PTE not present) and triggers the OS to allocate a new physical page. Latency varies 0.5–500 μs depending on fragmentation, huge-page availability, and fallback mechanisms.
Transparent Huge Pages (THP)A Linux kernel feature that automatically promotes 4 KB pages to 2 MB pages to reduce TLB misses. MimicOS includes a Linux-like THP allocator and khugepaged (background daemon that merges 4 KB pages).
Validation metrics
L2 TLB MPKIMisses per kilo instructions (L2 TLB). MimicOS+Sniper estimates with 82% accuracy vs a real Xeon Gold 6226R.
PTW latencyCycles to walk the page table (4-level radix tree). MimicOS+Sniper estimates with 85% accuracy (varies 39–180 cycles across workloads).
Page fault latencyTime to allocate a new physical page and update the page table. MimicOS achieves 66% (up to 79%) cosine similarity vs real Linux (varies 0.5–500 μs across workloads).
IPC accuracyMimicOS+Sniper achieves 80% IPC accuracy vs a real Xeon (vs 66% for baseline Sniper with fixed PTW latency).
Simulation time overheadMimicOS adds 20% overhead on average (2–35% depending on PFKI) vs baseline emulation-based simulators. gem5-FS adds 77% overhead vs gem5-SE.
Memory overheadMimicOS+gem5-SE uses 1.02 GB RAM per job (vs 1.0 GB for gem5-SE, 1.69 GB for gem5-FS). Sniper+MimicOS uses 0.8 GB (with offline instrumentation).
VM schemes in VirTool
Radix PT4-level or 5-level radix tree (standard x86-64 page table). MimicOS supports 4 KB and 2 MB pages.
ECH (Elastic Cuckoo Hashing)A hash-based page table (Skarlatos et al., ASPLOS 2020) that reduces PTW latency via cuckoo hashing. VirTool includes the full ECH design (8K entries/way, 4-way, CITY hash).
HDC (Hash-based Direct-Cache PT)A 4 GB global hash table (Yaniv & Tsafrir, SIGMETRICS 2016) with open addressing. VirTool shows it reduces PTW latency but increases DRAM row-buffer conflicts.
UtopiaA hybrid virtual-to-physical mapping (Kanellopoulos et al., MICRO 2023) with restrictive segments (hash-based) and flexible segments (radix tree). VirTool includes the TAR cache, SF cache, and RestSeg allocator.
MidgardAn intermediate address space scheme (Gupta et al., ISCA 2021) that defers address translation until main memory access. VirTool includes the VMA lookaside buffer, B+ tree, and M→P radix tree.
RMM (Redundant Memory Mappings)A contiguity-aware TLB (Karakostas et al., ISCA 2015) with range translation and eager paging. VirTool includes the range lookaside buffer (RLB) and B+ tree to store ranges.
Simulators integrated with Virtuoso

Sniper execution-driven

Carlson et al., SC 2011

A fast parallel multi-core simulator (interval-based timing model). Virtuoso+Sniper is the baseline for VirTool and achieves 80% IPC accuracy with 35% simulation time overhead.

ChampSim trace-based

github.com/ChampSim/ChampSim

A trace-based simulator for cache/prefetcher research. Virtuoso+ChampSim adds 13% simulation time overhead and enables VM research on top of ChampSim's cache models.

Ramulator trace-based

Kim et al., CAL 2015

A DRAM simulator. Virtuoso+Ramulator adds 2% simulation time overhead (using offline instrumentation) and enables VM research with detailed DRAM models.

gem5-SE emulation-based

Binkert et al., 2011

An emulation-based execution mode of gem5. Virtuoso+gem5-SE adds 28% simulation time overhead and 1.02× RAM overhead (vs 77% time and 1.69× RAM for gem5-FS).

MQSim SSD simulator

Tavakkol et al., FAST 2018

An SSD simulator. Virtuoso+MQSim enables modeling of major page faults (disk I/O) and swapping activity (§7.6.2 shows 203× swapping overhead for large restrictive segments).

Related papers

gem5 full-system sim

Binkert et al., 2011

The standard full-system simulator. gem5-FS boots a full Linux kernel, achieving high accuracy but 77% simulation time overhead and 1.69 GB RAM per job.

QFlex full-system sim

EPFL PARSA, 2020

A QEMU-based full-system simulator. QFlex is more flexible than gem5-FS (supports ARM, RISC-V, x86) but has similar overhead (slow, high RAM usage).

FireSim FPGA-accelerated

Karandikar et al., ISCA 2018

An FPGA-accelerated full-system simulator. FireSim is faster than gem5-FS (10–100× speedup) but requires RTL synthesis (slow compilation, high barrier to entry).

ECH (Elastic Cuckoo Hashing) hash PT

Skarlatos et al., ASPLOS 2020

The hash-based page table that VirTool implements. The paper shows ECH reduces PTW latency by up to 40% but increases DRAM row-buffer conflicts by 52%.

Utopia hybrid mapping

Kanellopoulos et al., MICRO 2023

The hybrid virtual-to-physical mapping that VirTool implements. The paper shows Utopia reduces translation latency by up to 23% but increases swapping by 203× for very large segments.
