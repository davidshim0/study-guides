Eight primers on PIM, CXL, and LLM inference

1What PIM is and why it matters▾

Processing-In-Memory (PIM) places compute units inside DRAM chips, adjacent to memory banks, exploiting internal memory bandwidth (e.g., 16 TB/s in GDDR6-PIM vs 2 TB/s external HBM). For memory-bound tasks — like LLM decoding — this bypasses the off-chip bottleneck.

Trade-off: near-bank logic (in DRAM process) reduces memory density by ~25% because compute units occupy die area. That's fine if you need bandwidth more than raw capacity.

Now you're ready: CENT is built on GDDR6-PIM modules, using 512 TB/s internal BW to serve Llama2-70B at 2.3× the throughput of 4 A100 GPUs while costing 2.5× less (3-year TCO).
2CXL memory expansion and why it's key▾

Compute eXpress Link (CXL) is a low-latency PCIe-based protocol for attaching memory to a host CPU. CXL 3.0 supports switch-based fabric letting devices talk to each other (peer-to-peer), not just the host.

For LLMs with 70B parameters + large KV caches, CXL memory expansion scales capacity beyond what fits in a single box. Unlike NVLink (8 devices max), CXL can scale to 4,096 nodes.

Now you're ready: CENT uses a CXL switch to connect 32 PIM devices (512 GB total), running tensor-parallel and pipeline-parallel mappings across them without needing GPUs.
3Why LLM inference is memory-bound, not compute-bound▾

Decoder LLMs generate one token at a time (autoregressive). Each token is a matrix-vector multiply (GEMV) with no parameter reuse — you load the entire weight matrix from memory just to compute one output vector.

Operational intensity = (ops)/(bytes moved). GEMV is ≪1, so GPUs sit idle waiting for data. On an A100, Llama2-70B achieves only 21% GPU utilization (paper Fig. 2b) because memory BW is the bottleneck, not TFLOPs.

Now you're ready: CENT targets the 79% of GPU compute that sits idle — it bets on high memory BW (512 TB/s internal) rather than wasted TFLOPs.
4Prefill vs decoding — and which one dominates▾

Prefill encodes the input prompt in parallel (matrix-matrix, high ops/byte). Decoding generates output tokens sequentially (matrix-vector, low ops/byte). On Llama2-70B with 512 input / 3584 output tokens, prefill is only 2% of GPU time (paper §7.1).

Because decoding dominates, optimizing it yields the biggest speedup. GPUs excel at prefill but underutilize in decoding; PIM is the opposite (lower peak TFLOPs, but memory BW shines).

Now you're ready: CENT is 2.5× slower than GPU at prefill, 2.5× faster at decoding — and since decoding is 98% of runtime, CENT wins overall (2.3× throughput).
5KV caches and the batch-size trap▾

Modern LLMs support 128K–1M token contexts. KV (key-value) caches store past tokens' attention states, unique per prompt. Large contexts → large KV caches → high memory demand.

Batching multiple prompts together improves GPU utilization — but KV caches limit feasible batch size. On 4 A100s, a 32K context saturates at batch=16 (vs batch=128 at 4K), halving throughput (paper Fig. 1).

Now you're ready: CENT uses pipeline-parallel mapping (each stage processes one prompt) to keep memory full without needing large batches — 512 GB across 32 devices, each with ~16 GB.
6GDDR6-PIM channel architecture▾

Each GDDR6-PIM channel has 16 banks organized into 4 bank groups. Every bank has a near-bank processing unit (PU) — a 16-way MAC reduction tree (BF16) operating at 1 GHz → 32 GFLOPS per channel, 16 TB/s internal BW.

A 2 KB Global Buffer broadcasts vectors to all 16 PUs in parallel. PUs read rows locally from their bank and accumulate partial sums into 32 registers. Result: fully pipelined GEMV across all banks.

Now you're ready: CENT's 32 devices × 32 channels = 1024 PIM channels → 512 TFLOPS peak (PIM logic) + 96 TFLOPS (PNM accelerators), all near-memory.
7PNM (Processing-Near-Memory) vs PIM▾

PIM = compute inside DRAM chips (near banks). PNM = compute outside DRAM chips but near them (e.g., in the controller). PNM uses CMOS process → more area-efficient for complex ops (square root, division, softmax) but lower BW than PIM.

CENT uses both: MAC ops (99% of FLOPs) in PIM; normalization, activation, rotary embedding in PNM (8 RISC-V cores + accelerators per device). Hierarchical design avoids over-provisioning near-bank PUs.

Now you're ready: PIM handles GEMV; PNM handles RMSNorm, SoftMax, SiLU, RoPE. Together they execute an entire transformer block on-device, no host interaction.
8Tensor-parallel vs pipeline-parallel mapping▾

Tensor-Parallel (TP) splits a single transformer block across all devices — lowers latency for one query but requires broadcast/gather (expensive). Pipeline-Parallel (PP) assigns each block to one device, chains them — maximizes throughput (many queries in flight) but higher per-query latency.

CENT supports both: TP for latency-critical (real-time chatbots), PP for throughput-critical (batch serving). Hybrid TP-PP balances the two.

Now you're ready: On Llama2-70B, CENT uses PP (80 stages across 32 devices, batch=80 queries) to achieve 2.3× GPU throughput while fitting in 512 GB memory.
The whole story

CENT in a nutshell

The problem, the solution, the architecture, and why it works.

LLM inference has two phases: prefill (parallel encoding of input tokens, matrix-matrix, high ops/byte) and decoding (sequential generation, matrix-vector, low ops/byte). Decoding dominates runtime (98% on Llama2-70B) and is memory-bound — you load the full weight matrix once per token with no reuse. GPUs are built for compute throughput (312 TFLOPS on A100) but deliver only 2 TB/s external BW, bottlenecking decoding and leaving 79% of compute idle.

Meanwhile, model size + KV caches demand huge memory (Llama2-70B needs 320 GB at batch=128, 4K context). As context length grows (128K–1M tokens in modern LLMs), GPU memory saturates at smaller batches, crippling throughput. You're paying for TFLOPs you can't use and memory you run out of.

CENT (CXL-Enabled GPU-Free sysTem) fixes this by moving compute into memory via GDDR6-PIM modules (16 TB/s internal BW per chip, 512 TB/s total). A hierarchical PIM-PNM design places MAC units near DRAM banks (99% of ops) and PNM accelerators + RISC-V cores for the rest (RMSNorm, SoftMax, RoPE). Each of 32 CXL devices integrates 16 PIM chips + PNM logic; a CXL switch connects them with peer-to-peer primitives (send/recv, broadcast, gather). Tensor-parallel and pipeline-parallel mappings distribute Llama2 models across devices, eliminating the need for GPUs.

The payoff: on Llama2 7B/13B/70B, CENT achieves 2.3× higher throughput than 4 A100 80GB GPUs (at max supported batch sizes), 2.9× better energy efficiency, and 5.2× more tokens per dollar (3-year owned TCO). The CXL controller (19 mm² @ 7nm) costs $11.9 at 3M volume; total system: $14,873 vs $42,128 for 4 GPUs.

🍯 Key takeaways

  • LLM decoding is memory-bound (GEMV with no reuse) — GPUs achieve 21% utilization because they're built for compute, not BW.
  • PIM's internal BW (16 TB/s/chip, 512 TB/s total) lets CENT achieve 2.5× higher decoding throughput than GPU despite lower peak TFLOPs.
  • CXL memory expansion (512 GB across 32 devices) scales capacity without GPUs; CXL switch enables TP/PP parallelism via broadcast/gather.
  • Hierarchical PIM-PNM design: 99% of FLOPs (GEMV) in near-bank PUs; RMSNorm/SoftMax/RoPE in PNM (RISC-V + accelerators).
  • Pipeline-parallel mapping (80 stages, batch=80) maximizes throughput; tensor-parallel (all devices per block) minimizes latency.
  • CENT is 2.5× slower at prefill but 2.5× faster at decoding — since decoding is 98% of time, end-to-end throughput beats GPU by 2.3×.
  • Cost: $14,873 (CENT) vs $42,128 (4×A100) → 5.2× more tokens/$ over 3 years; 2.9× better energy efficiency (tokens/J).

How to think about CENT

The core insight · Bandwidth beats idle TFLOPs

An overpowered sports car on a dirt road

A GPU is a 300 mph Ferrari on a gravel path (2 TB/s memory BW) — the engine (312 TFLOPS) can't help if the road is the limit. LLM decoding loads the full weight matrix once per token with no reuse (matrix-vector), so you're IO-bound, not compute-bound. CENT is a pickup truck with a highway (512 TB/s internal BW) — less peak speed, but the road capacity matches the task.

Where it breaks: the Ferrari does shine on smooth highways (prefill's matrix-matrix ops). CENT is 2.5× slower there — but prefill is only 2% of runtime.
PIM architecture · Near-bank processing

Workers in the warehouse vs truckers

Traditional GPU: data travels long distances (off-chip to HBM, then to compute units) like trucking goods across the country. PIM: place workers inside the warehouse (near DRAM banks) so they walk to the shelf instead of driving. Internal DRAM bandwidth (16 TB/s per chip) is orders of magnitude higher than the external link (2 TB/s HBM).

Where it breaks: workers in the warehouse take up storage space (PIM reduces memory density by ~25%). You accept this if throughput matters more than raw capacity.
CXL network · Scalable expansion

A conference call vs a chain of phone calls

Pre-CXL: devices talk to the host one-by-one (like calling the CEO, who relays to others). CXL switch: devices broadcast/multicast to each other directly (conference call). CENT uses this for tensor-parallel (broadcast embedding vector to all devices) and pipeline-parallel (send result device-to-device). CXL 3.0 supports up to 4,096 nodes; NVLink caps at 8.

Where it breaks: NVLink has higher BW (900 GB/s vs CXL's ~64 GB/s per x4 link) — but CENT's inter-device transfers are small (16 KB per transformer block in TP), so CXL BW suffices.
Pipeline-parallel · Maximize throughput

An assembly line for mass production

Each transformer block is a station on the line. Multiple prompts flow through simultaneously: prompt 1 finishes block 0 and enters block 1 while prompt 2 starts block 0. CENT assigns 80 blocks across 32 devices (2–3 blocks/device), achieving batch=80 throughput without needing 80× memory for one prompt's KV cache.

Where it breaks: an assembly line has fixed per-item latency (sum of all stages). Pipeline-parallel is best for throughput-critical serving, not single-query latency.
Tensor-parallel · Low latency, high BW cost

Parallel workers on one task

Split a single transformer block's weight matrix across all devices — each computes a slice of the output vector in parallel. Lower latency (all devices work together) but requires broadcasting the input vector (16 KB) and gathering partial results (16 KB) per block. On Llama2-70B, 80 blocks × 135 KB/block ≈ 11 MB CXL traffic per token — small enough that CXL BW isn't a bottleneck.

Where it breaks: unlike parallel workers who can share a whiteboard, CXL devices must serialize broadcast/gather through the switch. Still fast enough for LLM inference (ms-scale latency).
Hierarchical PIM-PNM · Division of labor

Specialized roles in a factory

PIM workers (near banks) do the repetitive heavy lifting (GEMV = 99% of FLOPs). PNM units (in the controller, CMOS process) handle delicate operations (RMSNorm, SoftMax, square root). You wouldn't put a RISC-V core in every bank (area overhead) when 99% of work is MACs. CENT uses near-bank PUs (domain-specific) for MACs and 8 RISC-V cores + accelerators per device for everything else.

Where it breaks: in a factory, workers can be reassigned. PIM units are hard-wired MAC trees; PNM has general-purpose cores but less BW. The split is optimized for transformer workloads.
Trade-off · Compute vs bandwidth

Sprinter vs marathoner

Prefill = a sprint (short, high ops/byte, GPU's strength). Decoding = a marathon (long, low ops/byte, PIM's strength). CENT is slower at sprinting but dominates the marathon — and the marathon is 98% of the race. On Llama2-70B, GPU wins prefill (2.5×) but CENT wins decoding (2.5×), and since decoding is 49× longer, CENT wins overall (2.3× end-to-end throughput).

Where it breaks: if you had a short race (prefill-heavy workload like BERT inference), the sprinter wins. CENT is optimized for decoder LLMs where decoding dominates.
The full explanation

How it actually works

Section by section, with key diagrams and numbers from the paper.

1 · The problem — GPUs are built for the wrong bottleneck

Decoder LLMs generate one token at a time. Each token requires loading the entire weight matrix from memory to compute a single output vector (GEMV). Unlike earlier ML models (BERT, ResNet) where data is reused across a batch (high ops/byte), GEMV has no parameter reuse — operational intensity ≪1.

On an A100 GPU (312 TFLOPS, 2 TB/s HBM2E), Llama2-70B achieves only 21% GPU utilization (paper Fig. 2b) because the compute units sit idle waiting for memory. Meanwhile, large KV caches (unique per prompt) limit batch size: at 32K context, the GPU saturates at batch=16 vs batch=128 at 4K (paper Fig. 1), halving throughput.

🔍
Concretely: decoding a token on Llama2-70B takes 3.4× longer than encoding a token in prefill (paper §2) because GEMV is memory-bound. You're paying for 312 TFLOPS but using only 65 effective TFLOPS.

2 · PIM solution — move compute to memory, not memory to compute

GDDR6-PIM places near-bank processing units inside each DRAM bank, exploiting internal memory bandwidth (row buffer to PU). One GDDR6-PIM channel has 16 banks, each with a 16-way MAC reduction tree (BF16) operating at 1 GHz → 32 GFLOPS per channel. A 2 KB Global Buffer broadcasts vector segments to all 16 PUs in parallel.

Key advantage: internal BW is 16 TB/s per chip (vs 2 TB/s external for A100 HBM2E). CENT uses 32 devices × 32 channels = 1024 PIM channels → 512 TB/s internal BW total. This is 256× the external BW of a single A100 GPU.

Why 512 TB/s matters: GEMV loads the matrix once with no reuse. For a 70B model (140 GB at BF16), one token decoding requires ~140 GB of memory traffic. At 512 TB/s internal BW, this takes 0.27 ms; at 2 TB/s external BW (GPU), it takes 70 ms.

3 · Hierarchical PIM-PNM — domain-specific near-bank + general PNM

A transformer block contains more than just GEMV: RMSNorm, RoPE (rotary embedding), SoftMax, SiLU activation. Putting general-purpose cores in every bank would waste area (only 1% of FLOPs are non-MAC). CENT uses a two-tier design:

  1. PIM (near-bank): 16-way MAC reduction tree per bank → handles GEMV (99% of arithmetic ops). Includes lookup tables for activation functions (SiLU via sigmoid + multiply).
  2. PNM (near-controller): 8 RISC-V cores + accelerators (exponent, reduction, accumulation) per device → handles RMSNorm (square root, division), SoftMax (exp, division), RoPE (complex multiply). Manufactured in CMOS process (not DRAM) for area efficiency.

This division keeps PIM area overhead low (~25% memory density loss, vs 50% if all ops were in near-bank PUs) while still executing the entire transformer block on-device — no host CPU involvement per token.

4 · CXL network — scalable memory expansion + peer-to-peer communication

CXL 3.0 (PCIe 6.0 physical layer) connects 32 CXL devices via a switch. Each device has x4 lanes → 64 GB/s bidirectional BW to the switch. The switch supports broadcast (one device writes to multiple devices in one transaction) by reserving a header code in the CXL.mem flit format.

CENT implements three primitives:

  1. SEND_CXL / RECV_CXL: Peer-to-peer write transaction. Used in pipeline-parallel to pass embedding vectors (16 KB) between transformer blocks on different devices.
  2. BCAST_CXL: Broadcast to multiple devices specified by an 8-bit device count. Used in tensor-parallel to distribute the input vector to all devices before each FC layer.
  3. Gather (multiple RECV_CXL): Master device collects partial results from all other devices. Used in tensor-parallel after each FC layer to assemble the output vector.

Each CXL device has a 64 KB Shared Buffer (256-bit granularity) that PIM channels and PNM units both access — intra-device data movement between PIM and PNM goes through this buffer via WR_SBK / RD_SBK instructions.

🍯
CXL communication cost: In tensor-parallel, each transformer block requires 135 KB of CXL transfers (broadcast 16 KB input, gather 16 KB partial results, repeated for 7 FC layers + attention). On Llama2-70B (80 blocks), total CXL traffic = 10.8 MB per token. At 64 GB/s, this takes 0.17 ms — negligible vs PIM compute (ms-scale).

5 · Parallelism strategies — TP for latency, PP for throughput

Tensor-Parallel (TP): Split each transformer block across all 32 devices. Broadcast input vector (16 KB), compute partial GEMV on each device, gather results. Minimizes per-query latency (all devices work in parallel) but requires CXL traffic per FC layer. On Llama2-70B, CENT achieves 4.6× lower latency than GPU at batch=1 (paper Fig. 13a).

Pipeline-Parallel (PP): Assign each transformer block to one device (or a few consecutive devices). Process multiple queries simultaneously in different pipeline stages. Maximizes throughput (80 queries in flight for 80-block Llama2-70B) with minimal CXL traffic (only 16 KB per block boundary). CENT achieves 2.3× higher throughput than GPU at max batch sizes (paper Fig. 13b).

Hybrid TP-PP: Example — split 32 devices into 4 groups of 8. Each group uses TP (8-way split) for one transformer block; pipeline 20 blocks across 4 groups. Balances latency (8× parallelism) and throughput (batch=20). CXL traffic = (8-way TP) × (20 blocks) = moderate.

📊
QoS trade-off (Fig. 14b): At similar throughput (~600 tokens/s), CENT provides 3.4–7.6× lower query latency than GPU. This is because GPU must batch=128 to saturate throughput (high per-query latency), while CENT uses PP (many queries in flight, each flows through pipeline quickly).

6 · Hardware cost — die area, packaging, TCO

Each CXL device contains: (1) 16 PIM chips (512 MB each, 8 GB total), (2) CXL controller (19 mm² @ 7nm: instruction buffer, Shared Buffer, PNM accelerators, RISC-V cores, CXL port logic), (3) 96-lane switch port (shared across 32 devices).

Component costs (per device):

  • GDDR6-PIM: ~$371 (10× the cost of commodity GDDR6, based on UPMEM DDR4-PIM pricing model)
  • CXL controller die: $4.2 @ 3M volume (300mm 7nm wafer = $9,346, yield-adjusted for 19 mm² die)
  • 2D packaging: $1.2 (29% of die cost)
  • NRE amortized: $6.5 (frontend labor $5.2M, backend $3.1M, masks $11.6M, spread over 3M units)
  • Switch (shared): $15.3/device (96-lane 48-port PCIe 6.0 switch ≈ $490)

Total per device: $11.9 (PIM) + $11.9 (controller+packaging) ≈ $23.8 per device. 32 devices + CPU = $14,873 total. Compare: 4× A100 80GB + CPU = $42,128.

🍯
3-year TCO (owned): CENT = $0.73/hr (hardware $14,873, power 32.4W/dev × 32 = 1037W avg, electricity $0.139/kWh). GPU = $1.76/hr (hardware $42,128, power ~300W/GPU × 4 = 1200W). CENT generates 5.2× more tokens per dollar. (Paper Table 4, Fig. 12)

Worked examples

Four traces with exact calculations — memory BW, CXL traffic, throughput, cost.

① GEMV performance on PIM vs GPU

Llama2-70B has 140 GB of weights (BF16). How long does one token's GEMV take on CENT vs A100?

  1. A100 external BW: 2 TB/s (HBM2E). Load 140 GB → 140 GB / 2 TB/s = 70 ms.
  2. CENT internal BW: 512 TB/s (32 devices × 32 channels × 16 banks × 2 bytes × 1 GHz). Load 140 GB → 140 GB / 512 TB/s = 0.27 ms.
  3. Speedup: 70 ms / 0.27 ms = 259× on memory load time.
  4. But: PIM also needs time for MAC operations. 70B params × 2 ops (multiply-add) / 608 TFLOPS peak = 0.23 ms. Total ≈ 0.5 ms.
  5. Real speedup: GPU takes 70 ms (memory-bound, compute idle). CENT takes 0.5 ms (memory + compute overlapped). 140× faster on GEMV.
💡
Why overlap? PIM near-bank PUs read from local banks and compute in parallel — the 0.27 ms memory time and 0.23 ms compute time happen simultaneously across banks, not sequentially.

② CXL communication overhead in tensor-parallel

Llama2-70B has 80 transformer blocks. In TP mode (all 32 devices per block), how much CXL traffic per token?

  1. Per FC layer: Broadcast 8K embedding vector (16 KB at BF16) to 32 devices. Gather 32 partial results (16 KB total). Total = 32 KB.
  2. Per transformer block: Self-attention has 4 FC layers (Q/K/V projections + output). FFN has 3 FC layers (W1/W2/W3). Total 7 layers × 32 KB = 224 KB.
  3. Attention score: Also requires broadcast/gather (Q·K^T score vector, 4K context → 8 KB). Add 16 KB.
  4. Per block total: 224 KB + 16 KB ≈ 240 KB (paper says 135 KB — likely grouped-query attention reduces some transfers).
  5. Per token (80 blocks): 135 KB × 80 = 10.8 MB. At 64 GB/s CXL BW, this takes 10.8 MB / 64 GB/s = 0.17 ms.
  6. Conclusion: CXL traffic is 0.17 ms out of ~3 ms total token latency → 5.7% of time. Not a bottleneck.

③ Throughput calculation — CENT vs GPU at max batch

Llama2-70B, 512 input + 3584 output tokens. What's the end-to-end throughput?

  1. GPU (4× A100, batch=128): Prefill 512 tokens × 128 queries = 65,536 tokens in 39 seconds (paper Fig. 15a). Decoding 3584 tokens × 128 queries = 458,752 tokens in 246 seconds. Total 524,288 tokens / 285 s = 1840 tokens/s.
  2. CENT (32 devices, PP batch=80): Prefill 512 × 80 = 40,960 tokens in 98 seconds. Decoding 3584 × 80 = 286,720 tokens in 73 seconds. Total 327,680 tokens / 171 s = 1916 tokens/s.
  3. CENT speedup: 1916 / 1840 = 1.04× on Llama2-70B (paper Fig. 13b shows 1.2× average due to grouped-query attention reducing PIM efficiency).
  4. Across 7B/13B/70B: geomean speedup = 2.3× because smaller models (no grouped-query attention) see larger gains (4.6× on Llama2-7B).
🍯
Why CENT wins despite slower prefill? Decoding is 73 s out of 171 s (43% of CENT time) but 246 s out of 285 s (86% of GPU time). CENT's 2.5× decoding speedup dominates the 2.5× prefill slowdown.

④ Cost efficiency — tokens per dollar over 3 years

CENT vs GPU: which generates more tokens per dollar of TCO?

  1. GPU 3-year owned TCO: Hardware $42,128 + electricity (1200W avg × 3 years × 8760 hr/yr × $0.139/kWh) = $42,128 + $4,377 = $46,505. Hourly cost = $46,505 / (3 × 8760) = $1.76/hr.
  2. CENT 3-year owned TCO: Hardware $14,873 + electricity (1037W × 3 yr × 8760 hr × $0.139/kWh) = $14,873 + $4,338 = $19,211. Hourly cost = $0.73/hr.
  3. GPU throughput: 1840 tokens/s (from example ③). Tokens per dollar = (1840 tokens/s × 3600 s/hr) / $1.76 = 3.76M tokens/$.
  4. CENT throughput: 1916 tokens/s × 2.3× avg speedup = 4407 tokens/s. Tokens per dollar = (4407 × 3600) / $0.73 = 21.74M tokens/$.
  5. CENT advantage: 21.74M / 3.76M = 5.78× more tokens per dollar. (Paper Fig. 13c shows 5.2× — slight variance due to rounding.)

Practice problems

Problem 1 · easy
A GPU has 300 TFLOPS and 2 TB/s external BW. A PIM system has 600 TFLOPS and 512 TB/s internal BW. For a GEMV operation loading 100 GB with 200 GFLOPS of compute, which is faster and by how much?
Show answer
GPU: memory time = 100 GB / 2 TB/s = 50 ms. Compute time = 200 GFLOPS / 300 TFLOPS = 0.67 ms. Total = 50 ms (memory-bound). PIM: memory and compute overlap (near-bank). Max(100 GB / 512 TB/s, 200 GFLOPS / 600 TFLOPS) = max(0.20 ms, 0.33 ms) = 0.33 ms. Speedup: 50 / 0.33 = 152× faster.Misconception: that higher peak TFLOPs makes GPU faster. For low ops/byte (GEMV), memory BW dominates — PIM's internal BW wins.
Problem 2 · medium
CENT uses 32 devices in tensor-parallel mode. Each transformer block requires broadcasting a 16 KB vector and gathering 16 KB of results. At 64 GB/s CXL BW, how long does one block's communication take?
Show answer
Broadcast 16 KB to 32 devices: 16 KB × 32 = 512 KB out (but CXL switch can multicast — still limited by output BW, so ~512 KB / 64 GB/s = 8 μs). Gather 16 KB from 32 devices: 16 KB × 32 = 512 KB in → 8 μs. Total ≈ 16 μs. (Real paper value is higher due to CXL protocol overhead and multiple FC layers — 7 layers × 16 μs ≈ 112 μs per block.)Misconception: that broadcast is free. The switch must serialize writes to all devices, consuming BW proportional to (data size × device count).
Problem 3 · medium
Why does CENT achieve only 1.2× speedup on Llama2-70B but 4.6× on Llama2-7B, despite the same architecture?
Show answer
Llama2-70B uses grouped-query attention (merges multiple query heads to share K/V projections), which converts some GEMV into narrow GEMM → higher ops/byte → better GPU utilization, reducing CENT's advantage. Llama2-7B uses standard multi-head attention (all GEMV) → lower ops/byte → GPU utilization stays low → CENT's memory BW wins by more. (Paper §7.1 notes this explicitly.)Misconception: that CENT always beats GPU by the same factor. Speedup depends on how memory-bound the workload is — grouped-query attention makes Llama2-70B less memory-bound.
Problem 4 · hard
CENT uses hierarchical PIM-PNM: 99% of FLOPs in near-bank MACs, 1% in PNM (RISC-V + accelerators). Why not put all ops in near-bank PUs (like UPMEM does)?
Show answer
Near-bank PUs are fabricated in DRAM process (1Y nm, ~14–16 nm equivalent) → low transistor density, high area overhead. A general-purpose core (RISC-V) in every bank would reduce memory density by 50%+ (vs CENT's 25%). Since 99% of ops are MACs (domain-specific), over-provisioning near-bank with full cores wastes area. CENT's PNM units (8 RISC-V cores per device, not per bank) are in CMOS process → higher density, shared across 32 PIM channels.Misconception: that PIM should handle all ops near-memory. Area overhead forces a trade-off — domain-specific near-bank (MACs) + general PNM (rare ops) is optimal for transformers.
Problem 5 · medium
Pipeline-parallel achieves 2.3× throughput on Llama2-70B (batch=80 CENT vs batch=128 GPU). Why doesn't CENT just use batch=128 in PP to match GPU's batch size?
Show answer
PP assigns each transformer block to one device (or a few). Llama2-70B has 80 blocks → 80 pipeline stages. Batch size = number of stages (one query per stage). To get batch=128, you'd need 128 blocks or split blocks across devices (increasing CXL traffic). CENT's 32 devices × 16 GB = 512 GB can fit 80 stages comfortably; 128 stages would require either more devices or smaller per-stage memory (limiting KV cache size).Misconception: that batch size is independent of pipeline stages. In PP, batch = stages, bounded by (device count) × (blocks per device) and memory capacity per device.
Problem 6 · hard
CENT is 2.5× slower than GPU at prefill. Why not use GPU for prefill and CENT for decoding (disaggregated serving)?
Show answer
You could do this — paper §1 mentions prefill/decode disaggregation as a future direction. But: (1) Prefill is only 2% of GPU time (low impact). (2) Transferring KV cache from GPU to CENT after prefill adds latency and complexity. (3) For latency-critical apps, the extra hop hurts. (4) For throughput-critical apps, CENT's PP already achieves 2.3× higher throughput end-to-end, so the 2.5× prefill loss is worth it. Disaggregation makes sense for very long contexts (128K+) where prefill becomes a bigger fraction.Misconception: that you should always use the best hardware for each phase. System-level latency, data movement costs, and operational simplicity matter — CENT's end-to-end win justifies slower prefill.
Problem 7 · easy
Why does CENT's cost advantage (5.2× tokens/$) grow larger than its throughput advantage (2.3×)?
Show answer
CENT is 2.5× cheaper to build ($14,873 vs $42,128 for 4 A100s). Over 3 years, it's also more energy-efficient (2.9× tokens/J), lowering operational cost. So: cost advantage = (throughput ratio) × (hardware cost ratio) × (energy ratio) → 2.3 × 2.5 × (energy factor) ≈ 5.2×. The hardware cost difference amplifies the throughput gain.Misconception: that cost efficiency = throughput. Cost also includes capital expenditure (hardware) and opex (electricity) — CENT wins on all three (throughput, hardware cost, energy).
Problem 8 · medium
CENT uses CXL 3.0 (up to 4,096 nodes). NVLink caps at 8 GPUs. Does CXL's lower per-link BW (64 GB/s x4 vs NVLink's 900 GB/s) become a bottleneck?
Show answer
For LLM inference (not training), inter-device transfers are small. Tensor-parallel: 135 KB per block. Pipeline-parallel: 16 KB per stage. Even at 80 blocks × 135 KB = 10.8 MB per token, 64 GB/s handles this in 0.17 ms (5.7% of token latency). NVLink's 900 GB/s is overkill. For training (gradients + optimizer states), NVLink's BW is necessary. CENT targets inference, where CXL's lower BW + higher scalability (4,096 nodes) is the right trade-off.Misconception: that higher BW is always better. For inference, communication volume is low — scalability (device count) matters more than per-link BW.
Quick answers

Frequently asked questions

Twelve questions about CENT's design and limitations. Tap to expand.

Q1Why does LLM decoding have low operational intensity?▾
Decoder LLMs generate one token at a time (autoregressive). Each token requires matrix-vector multiply (GEMV) — you load the entire weight matrix from memory to compute one output vector with no parameter reuse. Ops/byte = (2 × params) / (params × 2 bytes) = 1. Compare: GEMM (batch of vectors) can reuse the matrix across rows → ops/byte = batch size. For Llama2-70B at batch=1, ops/byte ≪ 1, leaving GPU compute idle.
Q2What's the fundamental trade-off between PIM and GPU?▾
PIM: high memory BW (512 TB/s internal), lower peak TFLOPs (608 total), area overhead reduces memory density 25%. GPU: lower memory BW (2 TB/s external), high peak TFLOPs (312 per GPU), no density penalty. Winner depends on workload: memory-bound (LLM decoding) → PIM. Compute-bound (training, prefill) → GPU. CENT targets memory-bound inference where GPU's TFLOPs sit idle.
Q3Why use CXL instead of NVLink or Ethernet RDMA?▾
CXL: latency ~8× lower than RDMA, PCIe-based (commodity), scales to 4,096 nodes, sufficient BW for inference (64 GB/s per x4 link). NVLink: higher BW (900 GB/s) but expensive, caps at 8 devices (18 with NVSwitch), overkill for inference (small inter-device transfers). RDMA: high latency, complex stack. For LLM inference, CXL's low latency + scalability + cost is the sweet spot.
Q4Does CENT work for models larger than Llama2-70B?▾
Yes — even better. Larger models (Llama3-405B, DeepSeek-V3-671B) have more parameters → more memory traffic → higher memory-bound fraction → CENT's BW advantage grows. The paper notes that CENT's cost efficiency improves with model size because you'd need even more GPUs (e.g., 8× A100 for 405B), while CENT scales linearly with CXL devices. Limitation: very large models may exceed 512 GB; use 16 Gb GDDR6-PIM (1 TB capacity).
Q5What about long-context scenarios (128K–1M tokens)?▾
CENT's advantage increases. Long contexts → large KV caches → GPU batch size collapses (Fig. 1: batch=16 at 32K context vs batch=128 at 4K). Meanwhile, attention becomes more memory-bound (Q·K^T is larger). CENT achieves 3.3× decoding speedup at 32K context (Fig. 14a) vs 2.3× at 4K. At 128K+, prefill also becomes memory-bound (KV cache dominates), potentially justifying GPU/CENT disaggregation (GPU prefill, CENT decode).
Q6Why not just use more GPUs to match CENT's throughput?▾
You could — but cost explodes. CENT's 2.3× throughput at $14,873 (32 devices) vs 4 A100s at $42,128. To match CENT's throughput, you'd need 4 × 2.3 = 9.2 ≈ 10 A100s → ~$105K hardware + higher power. CENT's cost advantage (5.2× tokens/$) comes from using cheaper memory-centric hardware instead of expensive GPU compute. You're paying for TFLOPs you can't use.
Q7What happens to CENT if future LLMs use more compute-heavy layers?▾
CENT's advantage shrinks. If a new architecture has higher ops/byte (e.g., sparse MoE with gating networks, or recurrent layers with state reuse), GPUs utilize their TFLOPs better. CENT is optimized for current decoder transformers (GEMV-dominated). The paper's RISC-V cores provide flexibility for new ops, but PIM's lower peak TFLOPs can't match GPU on compute-bound tasks. Future: hybrid systems (GPU prefill + CENT decode) or algorithmic changes (quantization to reduce memory traffic).
Q8How does CENT handle model updates (fine-tuning)?▾
CENT targets inference, not training. Fine-tuning requires backpropagation (gradients), weight updates, optimizer states — all memory-intensive but also compute-heavy (∇W is GEMM-like). GPUs are better suited. For inference-only serving, model weights are loaded once and read-only. If you need fine-tuning, use GPUs for training/fine-tuning, then deploy the updated model to CENT for inference.
Q9What's the latency breakdown in tensor-parallel mode?▾
On Llama2-70B (Fig. 14c, TP=32): PIM compute dominates (~2.5 ms per token). CXL communication is ~0.2 ms (broadcast/gather 135 KB × 80 blocks = 10.8 MB at 64 GB/s). PNM (RMSNorm, SoftMax) is ~0.1 ms. Host CPU (top-k sampling) is ~0.05 ms. Total ≈ 2.85 ms per token. Compare GPU: ~13 ms per token at batch=1 (memory-bound) → CENT is 4.6× faster in TP mode.
Q10Why 32 devices? Why not 64 or 128?▾
Scalability limits: (1) CXL switch supports 72 ports (144 lanes); 32 devices × x4 = 128 lanes fits. (2) Power: 32 devices × 32.4W/dev = 1037W avg (fits in single-server PSU). 64 devices would exceed typical server power budgets (need multi-socket CPU or dual-level switches). (3) Llama2-70B needs ~32 devices for PP (80 blocks / 2.5 blocks per device). Larger models scale linearly (e.g., 405B → 64 devices). The paper's design targets a sweet spot for 70B–100B models.
Q11Does CENT support multi-modal models (vision + language)?▾
Partially. Vision encoders (ViT, CLIP) are CNN- or transformer-based → high ops/byte in convolutions or self-attention over image patches → GPU-friendly. The language decoder (autoregressive text generation) is where CENT shines. For multi-modal inference: use GPU for vision encoding, CENT for text decoding. Alternatively, if the vision encoder is prefill-like (one-time encoding), CENT can handle it at 2.5× slower than GPU — still acceptable if text generation dominates runtime.
Q12What are CENT's failure modes?▾
CENT underperforms when: (1) Workload is compute-bound (e.g., training, dense GEMM-heavy models). (2) Very short sequences where prefill dominates (CENT is 2.5× slower at prefill). (3) Phase changes: workloads like twolf-vpr have fast phase shifts; LRU/demand-driven GPU batching adapts faster than CENT's coarse 5M-cycle reconfiguration (though this is a minor issue). (4) Need for on-chip SRAM capacity (GPU's ~60 MB L2 cache helps with intermediate results; CENT has only 64 KB Shared Buffer per device).

Glossary & further reading

Every term in one place, then a path to go deeper.

Core terms
CENT (CXL-Enabled GPU-Free sysTem)The paper's architecture: 32 CXL devices (GDDR6-PIM + PNM units) for LLM inference without GPUs.
PIM (Processing-In-Memory)Compute units inside DRAM chips (near banks) → high internal BW (16 TB/s per chip) at cost of lower memory density.
PNM (Processing-Near-Memory)Compute units near DRAM (in controller, CMOS process) → area-efficient for complex ops but lower BW than PIM.
CXL (Compute eXpress Link)PCIe-based protocol for memory expansion + peer-to-peer communication. CXL 3.0 supports switch fabric, up to 4,096 nodes.
GEMV (matrix-vector multiply)Load full matrix, compute one output vector → no parameter reuse → low ops/byte → memory-bound. Dominates LLM decoding.
Operational intensity(Ops performed) / (bytes moved from memory). GEMV ≈ 1 op/byte. GEMM (matrix-matrix) ≈ batch size ops/byte.
Tensor-Parallel (TP)Split each transformer block across all devices → low latency (parallel compute) but high CXL traffic (broadcast/gather).
Pipeline-Parallel (PP)Assign each block to one device, chain them → high throughput (many queries in flight) but higher per-query latency.
Architecture terms
GDDR6-PIM channel16 banks, each with near-bank PU (16-way MAC tree, 32 GFLOPS). 2 KB Global Buffer broadcasts to all PUs.
Near-bank PU (processing unit)Domain-specific MAC reduction tree in DRAM process (1Y nm). 32 accumulation registers per PU.
Shared Buffer64 KB on-device SRAM (256-bit granularity) for intra-device data movement between PIM channels and PNM units.
CXL controller19 mm² @ 7nm. Integrates CXL port, instruction buffer (2 MB), Shared Buffer, PNM accelerators, RISC-V cores.
PNM accelerators32× Exponent (10-order Taylor series), 32× Reduction, 32× Accumulation units for RMSNorm/SoftMax.
RISC-V cores8× BOOM-2wide per device for complex/rare ops (sqrt, division, RoPE). 64 KB instruction buffer each.
KV (key-value) cacheStored attention states for past tokens, unique per prompt. Large contexts → large KV cache → limits GPU batch size.
Grouped-query attentionTechnique in Llama2-70B: merge multiple query heads → narrow GEMM instead of GEMV → higher ops/byte → reduces CENT's advantage.

Related papers

AiM: GDDR6-based Accelerator-in-Memory medium

Kwon et al. · IEEE JSSC · 2023

The GDDR6-PIM architecture CENT is based on. Near-bank 16-way MAC trees, Global Buffer, 16 TB/s internal BW. CENT extends AiM with CXL networking + PNM units.

UPMEM: Commercial PIM with DDR4 DIMMs easy

Devaux · Hot Chips · 2019

The first commercial PIM (general-purpose DPUs in every bank). CENT contrasts with UPMEM's approach — domain-specific near-bank MACs vs general cores.

CXL-PNM: Processing-Near-Memory for LLMs medium

Park et al. · HPCA · 2024

Samsung's LPDDR5X-based CXL system (PNM only, no near-bank PUs). CENT compares against CXL-PNM and shows 4.5× higher throughput due to PIM's internal BW.

AttAcc: PIM for Batched Transformer Inference hard

Park et al. · ASPLOS · 2024

HBM-PIM + GPU heterogeneous system. Prefill on GPU, decoding on PIM. CENT is GPU-free and shows 1.8–3.7× better cost efficiency (tokens/$) than AttAcc.

vLLM: Efficient LLM Inference Serving medium

Kwon et al. · SOSP · 2023

PagedAttention for KV cache management, continuous batching. The GPU baseline CENT compares against. CENT's pipeline-parallel achieves higher throughput at lower cost.

Megatron-LM: Model Parallelism for Training hard

Shoeybi et al. · arXiv · 2019

Tensor-parallel and pipeline-parallel for LLM training on GPUs. CENT adapts these strategies for inference on CXL-PIM devices.
