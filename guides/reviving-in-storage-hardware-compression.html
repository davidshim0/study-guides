Reviving in-storage compression on ZNS SSDs through collaboration

ZNS SSDs eliminate device-level garbage collection and reduce cost — but host-transparent compression fails there. CCZNS offloads compression execution to the SSD yet responds post-compression metadata to the host, bridging cross-layer management with a novel NVMe command set.

📄 "Reviving In-Storage Hardware Compression on ZNS SSDs through Host-SSD Collaboration" · Wang et al. 🎓 Assumes ZNS basics, compression fundamentals ⏱ ~25 min read
🤝

Decouple compression execution (SSD-side hardware) from indexing (host-side metadata) — let the SSD compress and report chunk sizes, while the host manages fine-grained addresses.

Before you begin

Eight ideas the paper builds on

Each primer unfolds one piece of background. Read them in order or skip to what you need.

host-side (collaborative) SSD-side (CCZNS) baseline (host-transparent/CPU)
1Zoned Namespace (ZNS) SSD — what and why▾

Traditional block-interface SSDs hide flash erase-before-write complexity via a Flash Translation Layer (FTL), requiring large on-board DRAM (~0.1% of capacity) for page-level mapping and over-provisioned flash (up to 50% in Meta's CacheLib) to counteract garbage collection. ZNS divides storage into zones — each writable sequentially and mapped to a physical superblock. Sequential writes couple logical and physical addresses within a zone, so ZNS uses only a coarse zone-level table (10–20× less DRAM). Host software manages GC holistically, reducing write amplification and OP space.

You're ready: CCZNS revives in-storage compression for ZNS by exploiting this coupled address space — host software directly indexes compressed chunks inside the SSD without device-level fine-grained mapping.
2In-storage hardware compression — the promise and the problem▾

FPGA/ASIC compression engines integrated inside the SSD deliver high throughput (e.g., 11 GB/s compress, 14 GB/s decompress) with ~5% extra area, offload CPU, and avoid round-trip data movement between host and external accelerators. Current in-storage compression SSDs (TCSSDs) operate host-transparently: the SSD compresses data invisibly and exposes a magnified logical volume to the host (e.g., 16 TB logical from 4 TB physical at 4× compression). But this requires the SSD to maintain fine-grained indexes of compressed chunks and reintroduces DRAM overhead, conflicting with ZNS's cost goal.

You're ready: CCZNS breaks host transparency to preserve ZNS's DRAM-less design while keeping in-storage compression benefits.
3Why host-transparent compression fails on ZNS▾

Challenge #1 (chunk indexing): To locate compressed chunks whose addresses and sizes are altered invisibly, the SSD needs fine-grained indexes — contradicting ZNS's coarse zone-level table. A coarse-grained workaround (Balloon-ZNS) pre-allocates space proportionally, but most chunks don't match pre-allocated sizes, causing space waste and double reads for oversized chunks. Challenge #2 (space harvesting): ZNS's explicit zone boundaries complicate presenting space savings. Enlarging logical zones causes overflow; shrinking physical flash per zone limits parallelism and throughput (Balloon-ZNS groups zones, reducing per-zone bandwidth to 60.5% of CCZNS).

You're ready: The paper's key insight — since ZNS couples logical/physical addresses per zone, host software's existing fine-grained indexes can directly manage compressed data if the SSD reports post-compression metadata.
4NVMe metadata field and bi-directional transfer▾

NVMe commands include a metadata field (commonly for hints or data protection; size flexible) transferred alongside data. Standard Linux NVMe driver transfers metadata only in I/O direction (host→SSD on write, SSD→host on read). CCZNS modifies the driver (a few LoC) to enable bi-directional transfer: on a compressed-write (CW), metadata goes host→SSD with compression hints, then SSD→host with post-compression chunk size. On a decompressed-read (DR), host→SSD with chunk location, SSD→host with decompressed data.

You're ready: This bi-directional metadata transfer is how CCZNS integrates compression semantics into traditional read/write without extra I/O operations.
5Byte-addressable storage management on flash▾

Flash writes are page-granular (e.g., 16 KB), but compressed chunks rarely align. CCZNS exposes zones and write pointers in bytes. Data less than a page is buffered (capacitor-protected), merged with subsequent writes. Post-compression chunk metadata (offset, size, feedback size) is stored in the flash Out-Of-Band (OOB) area (128–256 B per 4 KB flash page, trivial overhead: ~40 B for 4 chunks). This allows harvesting space savings maximally and checking DR address legality.

You're ready: Byte granularity bypasses the logical block tax, making CCZNS's space savings approach host-side compression (ZSTD) while keeping SSD-side execution speed.
6Multi-chunk I/O aggregation and prefetching▾

Single-chunk CW/DR (one chunk per I/O) suits large files but tanks performance when chunks are small (RocksDB's default 4 KB blocks). Multi-chunk CW batches chunks in write buffer flushes, passing NCK (number of chunks) and pre-compression sizes in metadata; SSD compresses each, writes sequentially, responds post-compression sizes. Multi-chunk DR prefetches a range, using the first chunk's compression ratio to forecast NLB (buffer size). SSD locates chunks via OOB, decompresses, returns NCK + all sizes. This cuts I/O count and overhead.

You're ready: Multi-chunk commands are why CCZNS achieves 3.6× throughput on RocksDB YCSB Load (write-only) vs. CPU-based ZSTD.
7Padding zones for CW/traditional write compatibility▾

Different file parts (e.g., RocksDB data blocks vs. index blocks) need separate compression policies. Mixing CW and traditional writes in one zone is tricky due to byte vs. block granularity. CCZNS introduces padding zones: after each CW, the last chunk is automatically padded to the logical block boundary. The SSD reports the padded "feedback size" (for host block-level addressing) but stores the actual post-compression size in OOB (for future DR). Space overhead is negligible: ~0.8% for 256 KB compressed data (2 KB padding / 256 KB).

You're ready: This lets RocksDB's data blocks use CW and index blocks use traditional write in the same zone without fragmentation.
8Quick glossary of key acronyms▾

CCZNS = Collaborative Compression ZNS. CW = Compressed-Write command. DR = Decompressed-Read command. NCK = Number of Chunks in an I/O. CNLB = Compressed Number of Logical Blocks. OOB = Out-Of-Band area (flash page metadata). TCSSD = Transparent Compression SSD (baseline, host-transparent). Balloon-ZNS = state-of-the-art host-transparent approach. ZenFS = RocksDB's ZNS filesystem backend. ZSTD = standard compression algorithm (level 1 in paper). Snappy/LZ4 = lightweight algorithms.

You're ready: These acronyms recur throughout — they're now at your fingertips.
The entire story in one place

CCZNS in a nutshell

Problem, measurement, cross-layer design, and results — all connected.

ZNS SSDs eliminate device GC and reduce on-board DRAM by coupling logical/physical addresses within zones. Enabling in-storage hardware compression on ZNS is promising for further cost reduction, but the standard host-transparent approach reintroduces fine-grained indexes inside the SSD, contradicting ZNS's goal. Coarse workarounds (Balloon-ZNS) sacrifice write amplification (wasted space, double reads) and performance (limited flash parallelism per zone).

The paper's key insight: host software already maintains fine-grained chunk indexes (e.g., RocksDB's SST index blocks). If the SSD reports post-compression metadata, the host can directly manage compressed chunks inside the SSD without SSD-side indexing. This decouples compression execution (SSD-side) from indexing (host-side).

CCZNS implements this via two NVMe commands (CW, DR) integrated with the metadata field. On CW, the host passes compression hints (e.g., chunk count, pre-compression sizes), the SSD compresses and writes, then responds post-compression sizes via bi-directional metadata transfer. On DR, the host passes chunk location and size, the SSD reads, decompresses, and returns data. Multi-chunk modes batch small chunks to reduce I/O count. Byte-addressable storage management harvests space savings maximally (no logical-block alignment tax). Padding zones harmonize CW with traditional writes.

Results on RocksDB/ZenFS (YCSB): CCZNS reduces write volume by 44.5% vs. no compression, 25.2% vs. Snappy, and 12.6% vs. Balloon-ZNS while improving write throughput by 65.3% vs. Balloon and 3.6× vs. ZSTD (best CPU compression ratio). Read-intensive workloads: +10–32% throughput vs. all schemes. CPU usage drops 73.1% vs. ZSTD-PC. Hardware cost: <1% SSD area, no critical path impact. Multi-chunk commands are essential: without them, throughput drops to 55.2% or space allocation fails entirely.

🍯 Core takeaways

  • Host-transparent compression on ZNS reintroduces DRAM overhead (fine-grained indexes) and either wastes space or limits parallelism.
  • CCZNS decouples execution from indexing: SSD compresses, host indexes (leveraging existing metadata structures).
  • CW/DR commands integrate compression semantics via NVMe metadata field with bi-directional transfer (few LoC Linux driver mod).
  • Multi-chunk modes batch small chunks (e.g., RocksDB's 4 KB blocks) to reduce I/O count — critical for performance.
  • Byte-addressable storage (OOB metadata in flash pages) harvests space savings maximally, bypassing logical block granularity.
  • Padding zones align last chunks to logical blocks, enabling CW + traditional write in one zone (0.8% overhead).
  • Results: 12.6% less write volume vs. Balloon-ZNS, 65.3% higher write throughput, 3.6× vs. ZSTD (CPU), <1% area cost.
Six everyday pictures

How to think about CCZNS

Each analogy stands alone. Every card ends with where the picture breaks.

The core idea · Decoupling

Division of labor

A factory separates heavy lifting (warehouse robots) from inventory tracking (office database). CCZNS does the same: the SSD is the warehouse robot (fast ASIC compression engines handle execution), the host is the office database (RocksDB's index blocks track where chunks live and their sizes). The SSD compresses, reports "I stored chunk X at byte Y with size Z," and the host updates its index. Neither duplicates the other's job.

Where it breaks: real factories have physical proximity; CCZNS requires NVMe metadata transfer (bi-directional, few LoC driver mod) to bridge host and SSD. The analogy skips that plumbing.
The DRAM tax · Host-transparent vs collaborative

Paying once vs. twice

A city charges a toll to enter (DRAM for SSD indexes). Host-transparent compression makes you pay twice: once for the SSD's fine-grained chunk map (reintroduced DRAM despite ZNS's coarse zone-level table), once for host indexes (RocksDB's SST metadata). CCZNS pays once: only host indexes exist. The SSD uses cheap flash OOB metadata (~40 B per 4 chunks) for chunk boundaries, not expensive DRAM.

Where it breaks: tolls are per-trip; DRAM is per-capacity and persistent. The analogy simplifies that ZNS's whole point is to avoid scaling DRAM with capacity (density gap: DRAM/flash >1000× by 2030).
Space harvesting · Coarse vs byte-addressable

Tetris vs. bin packing

Balloon-ZNS pre-allocates space per chunk proportionally (like Tetris: fixed slots). Most chunks don't fit perfectly — smaller-than-expected chunks leave gaps (wasted space), larger ones are split (tail stored separately → double read). CCZNS is bin packing: chunks are stored byte-consecutively, no pre-allocation. The SSD's byte-addressable write pointer advances exactly the compressed size, and OOB metadata records chunk boundaries. Maximum space utilization.

Where it breaks: Tetris lines clear; CCZNS zones don't reorganize mid-write. The analogy also skips flash page granularity (data <16 KB buffered, merged with next write).
Multi-chunk I/O · Batching

Bus vs. taxi

RocksDB's data blocks are 4 KB. One single-chunk I/O per block = calling a taxi for each person (high overhead). Multi-chunk CW/DR = a bus: batch 64 blocks (256 KB) in one I/O. The metadata field includes NCK (number of chunks), pre-compression sizes; SSD compresses each independently, returns post-compression sizes. Without multi-chunk, CCZNS throughput drops to 55.2% or space allocation fails (every chunk padded to logical block boundary → 100% overhead).

Where it breaks: buses have fixed routes; multi-chunk prefetching uses adaptive NLB sizing (first chunk's compression ratio forecasts total buffer need, may undershoot if data compressibility varies).
Compatibility · Padding zones

Adapter plug

RocksDB's SST has data blocks (compress via CW) and index blocks (no compression, traditional write). Mixing CW (byte write pointer) and traditional write (logical block pointer) in one zone = plugging 220V into 110V. Padding zones are the adapter: after each CW, the last chunk is auto-padded to the logical block boundary. SSD reports "feedback size" (padded) for host block alignment but stores actual post-compression size in OOB for DR. Space overhead: ~0.8% for 256 KB compressed data (2 KB padding).

Where it breaks: adapters are passive; padding zones require zone-type bit in NVMe open command and SSD logic to detect/pad. Also, the analogy skips that this enables avoiding file fragmentation (no need to separate data/index into different zones).
Flash parallelism · Zone-to-superblock mapping

Lanes on a highway

ZNS zones map 1:1 to flash superblocks (e.g., 32 blocks across 32 dies = full parallelism). Balloon-ZNS groups zones to shrink physical allocation, assigning each zone to a subset of dies — like closing half the highway lanes. Per-zone bandwidth drops (experiments: 60.5% of CCZNS). CCZNS keeps 1:1 mapping: every zone uses all dies, maximum throughput. Byte-addressable storage harvests space savings without restricting parallelism.

Where it breaks: highways have traffic jams; flash parallelism is deterministic (each die processes requests independently, no contention within a superblock). The analogy also simplifies that Balloon's grouping is a workaround for harvesting space, not a design goal.
The full explanation

How it actually works

Commands, metadata transfer, storage management, compatibility — step by step.

1 · CW/DR command formats — single-chunk mode

CW and DR extend NVMe read/write by one reserved bit (identifies compression) + metadata field. Single-chunk CW (one chunk per I/O): host provides LBA, NLB (≥ pre-compression size for buffer), intra-block offset (byte position within the starting logical block). SSD checks offset matches byte write pointer, compresses, writes, and overwrites metadata with post-compression size (O), then DMAs metadata back to host. Single-chunk DR: host provides LBA, NLB (≥ pre-compression size), intra-block offset, and post-compression size (O). SSD locates chunk, reads, decompresses, returns data.

🔧
NVMe driver mod (Linux): vanilla driver transfers metadata only in I/O direction (host→SSD on write). CCZNS adds bi-directional transfer (few LoC) so CW can respond metadata SSD→host after write completes. This is the plumbing that makes cross-layer management work.

2 · Multi-chunk CW/DR — batching for performance

When chunks are small (RocksDB's 4 KB blocks), single-chunk mode tanks performance. Multi-chunk CW (identified by second reserved bit): metadata includes intra-block offset, NCK (number of chunks), and array of pre-compression sizes (R₁, R₂, ...). SSD partitions the transferred data, compresses each chunk independently, writes sequentially (byte write pointer advances sum of post-compression sizes), and responds array of post-compression sizes (O₁, O₂, ...). Host updates all chunk indexes in one batch.

Multi-chunk DR (prefetching): metadata includes intra-block offset, CNLB (compressed number of logical blocks = read size), PDSZ (padding in last block). SSD reads CNLB bytes starting at LBA + offset, uses OOB metadata to locate chunk boundaries, decompresses as many chunks as fit in NLB buffer (forecast via first chunk's compression ratio), returns NCK + sizes. If NLB underestimated, SSD returns maximum fitting chunks (host can issue another DR for remainder).

🍯
Impact on RocksDB: multi-chunk CW cuts I/O count 64× (64 blocks in one 256 KB flush). Without it, throughput drops to 55.2% (inefficient prefetching) or space allocation fails (padding overhead). This is why the paper emphasizes multi-chunk as essential, not optional.

3 · Byte-addressable storage management + OOB metadata

To harvest compression space maximally, CCZNS exposes zones and write pointers in bytes. Data less than a flash page (16 KB) is buffered (capacitor-protected [Chen'19]), merged with subsequent writes. Write pointer advances exactly the compressed chunk size. For each chunk, the SSD stores lightweight metadata in the flash page's Out-Of-Band (OOB) area (128–256 B per 4 KB page): (1) offset within page, (2) post-compression size, (3) feedback size (may differ from post-compression if padding applied, see §3.5). When the page is programmed, OOB persists alongside data.

OOB overhead is trivial: assuming extreme case (4 KB chunks, 4× compression ratio → 4 chunks per 4 KB flash page), metadata is (2 + 4 + 4) × 4 = 40 B out of 128–256 B available. DR uses OOB to (1) check address legality (does the read range preserve chunk integrity?), (2) locate chunk boundaries in multi-chunk DR (decompress independently), (3) distinguish actual vs. feedback size (for compatibility, see below).

💡
Why OOB instead of DRAM? OOB is free (embedded in flash pages), doesn't scale with SSD capacity, and suffices for byte-level chunk management. Storing metadata in DRAM would reintroduce the cost CCZNS aims to avoid.

4 · Padding zones — harmonizing CW with traditional writes

RocksDB's SST files mix compressed data blocks (CW) and uncompressed index/meta blocks (traditional write). Mixing them in one zone is tricky: CW advances byte write pointer, traditional write expects logical block alignment. CCZNS introduces padding zones (specified via reserved bit in NVMe zone open command): after each CW, the last chunk is automatically padded to the next logical block boundary. SSD reports feedback size (post-compression + padding) to host for block-level addressing but stores actual post-compression size in OOB for future DR.

Space overhead is negligible for batched writes: assuming 256 KB compressed data (1 MB raw at 4× compression), average padding = 4 KB / 2 = 2 KB → 0.8% overhead (2 / 256). For single large chunks, overhead is even smaller. This small cost enables flexible compression policies per file part without fragmenting zones (RocksDB's 14 open zone limit).

5 · RocksDB integration — case study

RocksDB organizes data in LSM tree levels (L0 → Ln). Each SST file uses block-based format: data blocks (4 KB default, compressed), index blocks (locate data blocks), meta blocks. CCZNS integration (~800 LoC RocksDB, ~300 LoC ZenFS):

  1. Data blocks → multi-chunk CW. During flush/compaction, data blocks are batched in write buffer (1 MB default), sent via one multi-chunk CW. SSD compresses each block, responds post-compression sizes. Index entries are constructed after all data blocks are persisted (deferred indexing, similar to Zone Append requirement).
  2. Index entries extended. Original: (max_key, offset, post-compression size O). CCZNS adds: pre-compression size R. This enables DR to set NLB correctly (allocate buffer ≥ R). Memory overhead: ~3% in YCSB workloads (Table V: 3.24 GB vs. 3.15 GB for ZSTDPC).
  3. Index/meta blocks → traditional write. Stored in padding zones (same zones as data blocks), no compression. Zones opened in padding mode harmonize byte/block pointers.
  4. Reads → multi-chunk DR. Point queries use single-chunk DR (one index entry → one chunk). Compaction and iterator (scan) prefetch via multi-chunk DR, using first chunk's compression ratio to forecast NLB.

Crash consistency is unaffected: SST construction is atomic in RocksDB. If crash occurs mid-flush, RocksDB rebuilds the SST post-recovery. CCZNS follows the principle that indexes persist after data (same as Zone Append), so no stale pointers.

🍯
Why deferred indexing works: RocksDB already constructs index blocks incrementally during SST write. CCZNS postpones the index construction until after all data blocks are written (so post-compression sizes are known), then writes index blocks. This is a natural fit for LSM's append-only nature.
Follow the numbers, step by step

Worked examples

Four traces using the paper's real measurements and configurations.

① Multi-chunk CW arithmetic (RocksDB flush)

RocksDB flush writes 64 data blocks (4 KB each, 256 KB total) in one multi-chunk CW. Compression ratio (ZSTD level 1) is 2.0. How much space is actually written, and what's the space saving?

  1. Pre-compression size: 64 × 4096 = 262,144 bytes.
  2. Post-compression (ideal): 262,144 / 2.0 = 131,072 bytes.
  3. Padding overhead (padding zone): last chunk padded to logical block boundary. Average padding = 4 KB / 2 = 2 KB. Total: 131,072 + 2,048 = 133,120 bytes.
  4. Space saving: (262,144 − 133,120) / 262,144 ≈ 49.2%.
  5. I/O count: 1 multi-chunk CW (vs 64 single-chunk I/Os → 64× reduction).
🍯
Contrast with Balloon-ZNS: Balloon pre-allocates proportionally (131,072 bytes at 2.0 ratio), but individual blocks rarely match. Experiments show CCZNS reduces write volume 12.6% more than Balloon (paper §V-C, Figure 8).

② Bi-directional metadata transfer overhead

CCZNS transfers metadata bi-directionally in CW. For 64 chunks, metadata size is 64 × (2 B offset + 4 B size) = 384 B. How much latency does this add vs. data transfer?

  1. Metadata size: 384 bytes (NCK + array of sizes). Host→SSD: pre-compression sizes. SSD→host: post-compression sizes (overwritten in same field).
  2. Data DMA time (256 KB at 14 GB/s): 256,000 / 14,000,000,000 ≈ 18.3 µs.
  3. Metadata DMA time (384 B at 14 GB/s): 384 / 14,000,000,000 ≈ 0.027 µs (round-trip: 2× = 0.054 µs).
  4. Compression time (11 GB/s bandwidth): 256,000 / 11,000,000,000 ≈ 23.3 µs.
  5. Flash write time (assume ~700 µs TLC): dominates. Metadata overhead: 0.054 µs / (18.3 + 23.3 + 700) ≈ 0.007% of total I/O time.
💡
The paper reports metadata transfer accounts for <1% of SSD processing time across benchmarks (§V-B). This confirms bi-directional transfer is a trivial cost enabler, not a bottleneck.

③ OOB metadata overhead calculation

Extreme case: 4 KB chunks, 4× compression ratio → 4 chunks per 4 KB flash page. OOB area is 128 B per 4 KB page. What's the metadata overhead?

  1. Per-chunk metadata: 2 B (offset) + 4 B (post-compression size) + 4 B (feedback size) = 10 B.
  2. 4 chunks: 4 × 10 = 40 B.
  3. Available OOB: 128 B per 4 KB page (modern SSDs: 128–256 B [paper citation 45, 58, 89]).
  4. Overhead: 40 / 128 = 31.25% of OOB space used.
  5. Context: OOB is separate from data (doesn't reduce user capacity). Total SSD overhead: compression engines (~5% area) + OOB metadata (embedded in flash, free). No DRAM added.
📝
Why this is acceptable: OOB is per-page (fixed cost in flash design), not per-capacity. The 40 B stores crucial metadata for 4 independent chunks (16 KB raw data), enabling byte-level space harvesting without SSD DRAM. Trade-off heavily favors CCZNS.

④ YCSB Load throughput calculation

RocksDB loads 20M key-value pairs (Amazon dataset, average 1174 B/value). CCZNS achieves 3.6× throughput vs. ZSTD (CPU). Why the huge gap?

  1. CPU ZSTD compression bandwidth: ~800 MB/s per core (single-threaded, level 1 [paper citation 42]). RocksDB flush/compaction is mostly single-threaded per operation.
  2. CCZNS compression bandwidth: 11 GB/s (ASIC engines, paper §V-A). No CPU bottleneck.
  3. Write bandwidth gain: 11,000 / 800 ≈ 13.75×. But observed throughput gain is only 3.6×. Why?
  4. Other bottlenecks: (a) RocksDB's LSM compaction involves merge + sort (not pure compression), (b) YCSB Load also reads (for duplicate key checks), (c) flush interval pacing. Compression time is ~30% of total flush/compaction time in ZSTD case, so 13.75× on 30% → ~4× overall, close to observed 3.6×.
  5. CPU usage: ZSTD consumes 8.0% average CPU (Table IV). CCZNS: 5.8% (offloading saves 2.2 percentage points, or 27.5% relative reduction).
🍯
The real win: CCZNS matches ZSTD's compression ratio (same algorithm/level, 8.8% better write volume than no compression, Table in paper) while achieving 3.6× throughput and freeing up CPU for other tasks. This is the practical payoff of in-storage compression.
Test yourself

Practice problems

Try before opening answers. Each ends with the misconception it checks.

Problem 1 · easy
Why can't CCZNS use the standard Linux NVMe driver's metadata transfer as-is (without modification)?
Show answer
Vanilla Linux NVMe driver transfers metadata only in I/O direction: host→SSD on write, SSD→host on read. CW needs bi-directional transfer on write: host→SSD with pre-compression sizes/hints, then SSD→host with post-compression sizes (overwritten in the same metadata field). This requires a few LoC driver mod to enable reverse DMA after write completion. Without it, the host can't extract post-compression metadata to update indexes.Misconception: that NVMe metadata is inherently bi-directional. It's not — the standard only supports one direction per I/O type. CCZNS's cross-layer design requires the reverse channel.
Problem 2 · medium
Balloon-ZNS groups 4 zones into one superblock (each zone gets 8 of 32 dies). CCZNS maps 1 zone to 1 superblock (all 32 dies). If both write to one zone at QD=16, which has higher bandwidth and why?
Show answer
CCZNS has higher bandwidth. Each zone's write operations can parallelize across 32 dies (full superblock), while Balloon's zones are restricted to 8 dies each. At QD=16, CCZNS can dispatch up to 16 writes across 32 dies (full utilization if requests are spread), while Balloon is limited to 8 dies per zone. Experiments show Balloon's write throughput is 60.5% of CCZNS (paper §V-C). The grouping strategy (Challenge #2 in paper) is Balloon's workaround for harvesting space savings without enlarging logical zones, but it sacrifices flash parallelism.Misconception: that zone grouping is a neutral trade-off. It's a forced choice due to host-transparent compression's space harvesting problem — CCZNS avoids this entirely via byte-addressable management.
Problem 3 · medium
A 4 KB data block compresses to 1.8 KB. In padding zone mode, what is the feedback size reported to the host, and why is it different from the post-compression size stored in OOB?
Show answer
Post-compression size: 1.8 KB (actual). Feedback size (reported to host): 4 KB (padded to logical block boundary). The difference enables compatibility: the host's index uses 4 KB for block-level addressing (so traditional writes can coexist), while the SSD's OOB stores 1.8 KB for accurate DR (locating the chunk's actual end). When DR reads this chunk, the SSD uses the 1.8 KB from OOB to decompress correctly, not the 4 KB feedback. The 2.2 KB gap is padded space (wasted, but overhead is small for batched writes: 0.8% for 256 KB compressed data).Misconception: that padding means the compressed chunk is actually 4 KB. No — it's still 1.8 KB on flash; the padding is just logical alignment for host compatibility.
Problem 4 · hard
Why does CCZNS's multi-chunk DR use the first chunk's compression ratio to forecast NLB (buffer size), and what happens if the forecast is wrong?
Show answer
Multi-chunk DR prefetches a contiguous range (e.g., 1 MB compressed data). The host doesn't know individual chunk sizes upfront (only total CNLB in compressed space). To allocate enough buffer (NLB ≥ total pre-compression size), CCZNS uses the first chunk's compression ratio as a proxy for the entire range, exploiting data compressibility locality (nearby chunks likely have similar compressibility). If wrong (e.g., first chunk compresses 2×, later chunks 4×), the forecast underestimates. SSD decompresses as many chunks as fit in NLB buffer, returns NCK + sizes for those chunks only. Host can issue another DR for remainder. This is a practical trade-off: exact forecasting requires traversing all chunk indexes (high overhead), while prefix-based prediction is fast and usually accurate.Misconception: that multi-chunk DR always returns all chunks in the read range. No — it's bounded by NLB. The paper's design prioritizes low overhead (no index traversal) over perfect prefetching, accepting rare re-fetch if compressibility varies sharply.
Problem 5 · medium
CCZNS's memory usage is 2.9% higher than ZSTDPC in YCSB Load (3.24 GB vs 3.15 GB, Table V). Where does the extra memory come from, and is this a fair trade?
Show answer
Extra memory: RocksDB's index blocks are expanded to store pre-compression size (R) in addition to post-compression size (O). Each index entry grows by ~4 B (assuming 32-bit size). With millions of 4 KB data blocks in 20M key-value pairs (Amazon dataset), index blocks grow ~2.9%. This memory is host DRAM, not SSD DRAM. Fair trade? Yes, because: (1) CCZNS eliminates all SSD-side fine-grained indexes (saving scarce on-board DRAM), (2) host DRAM is scalable (CXL memory disaggregation [paper citation 43, 44]), (3) the 2.9% is negligible in cloud servers (typical: hundreds of GB DRAM), (4) CPU usage drops 73.1% vs ZSTDPC (freeing cores is valuable). The paper acknowledges this as the one side effect but argues host memory is the right place to pay.Misconception: that CCZNS's "no DRAM" claim is absolute. It's "no SSD DRAM" — host DRAM usage grows slightly, which is acceptable given host scalability and the density gap (DRAM/flash >1000× by 2030).
Problem 6 · hard
Why does CCZNS require indexes to be persisted after chunks/data (same principle as Zone Append), and what happens if the system crashes mid-CW before metadata is returned?
Show answer
Principle: if indexes persist before data, and crash occurs after index write but before data write, indexes point to garbage (or stale data). CCZNS follows RocksDB's atomic SST construction: (1) all data blocks are written via multi-chunk CWs, (2) index blocks are constructed after all data blocks succeed, (3) index blocks are written last. If crash occurs mid-CW (before metadata response), the CW is incomplete — RocksDB discards the partial SST and rebuilds it post-recovery (flush from memtable or re-run compaction). The key is that CCZNS never updates host indexes until the SSD confirms completion (metadata DMA back). This is why deferred indexing works: RocksDB's SST construction is already atomic (all-or-nothing), so deferring index construction until after data is a natural fit. Contrast with Zone Append: similar — Zone Append reports actual LBA after write, so host can't update mappings until completion. CCZNS is the compression-aware analog.Misconception: that CCZNS's deferred indexing is a hack. It's actually the correct recovery-safe design, aligned with RocksDB's atomic SST semantics and Zone Append's principles.
Problem 7 · easy
Why does removing multi-chunk CW (using only single-chunk CW) cause CCZNS to fail space allocation when loading 20M key-value pairs?
Show answer
Single-chunk CW in padding zone mode: each chunk (even 1.8 KB compressed) is padded to 4 KB logical block boundary. For 20M key-value pairs → millions of data blocks. If every block is padded, space overhead is ~100% (1.8 KB data + 2.2 KB padding = 4 KB per chunk, vs 1.8 KB in multi-chunk mode where only the last chunk per I/O is padded). The SSD runs out of space before loading all data. Multi-chunk CW batches 64 blocks in one I/O, so only the final chunk is padded (2 KB for 256 KB compressed → 0.8% overhead). This is why the paper emphasizes multi-chunk as essential: without it, padding overhead is catastrophic.Misconception: that single-chunk CW is a viable alternative. For small chunks in padding zones, it's unusable due to space explosion. Multi-chunk is not an optimization — it's a necessity for workloads like RocksDB.
Problem 8 · medium
CCZNS achieves 12.6% less write volume than Balloon-ZNS but only 8.8% less than ZSTD (CPU). Why isn't the gap larger, given CCZNS uses byte-addressable storage vs ZSTD's byte-perfect compression?
Show answer
CCZNS and ZSTD both use ZSTD level 1 algorithm on the same data, so their compression ratios are identical. The difference is padding overhead. ZSTD (host-side) stores compressed chunks byte-perfectly in the filesystem (no padding). CCZNS uses padding zones: the last chunk in each multi-chunk CW is padded to logical block boundary (~2 KB for 256 KB compressed data → 0.8% overhead). Over the full YCSB Load (20M key-value pairs), this accumulates to ~8.8% write volume for ZSTD vs ~12.6% for Balloon (which has coarse-grained pre-allocation waste). CCZNS sits in between: better than Balloon (byte-addressable vs proportional slots), slightly worse than ZSTD (padding overhead). The paper accepts this trade-off because: (1) 0.8% is small, (2) it enables CW/traditional write coexistence (avoiding zone fragmentation), (3) performance gains (3.6× throughput) far outweigh the minor write volume loss.Misconception: that byte-addressable management guarantees zero overhead. Padding zones introduce a small overhead for compatibility, but it's a deliberate design choice with strong net benefit.
Quick answers

Frequently asked questions

Ten questions newcomers ask most. Tap any to expand.

Q1What's the single biggest advantage of CCZNS over host-transparent compression (TCSSDs/Balloon)?▾
Eliminating SSD-side fine-grained indexes. Host-transparent requires the SSD to track every compressed chunk's address/size (reintroducing DRAM overhead, contradicting ZNS's cost goal). CCZNS decouples execution from indexing: SSD compresses, host indexes (leveraging existing RocksDB metadata). This preserves ZNS's DRAM-less design while keeping in-storage compression's performance/CPU benefits. The paper's measurements confirm <1% SSD area cost (compression engines + OOB metadata only, no DRAM).
Q2Why can't the host just pre-allocate space proportionally like Balloon-ZNS does?▾
Individual chunks rarely match pre-allocated sizes (compression ratio varies per chunk). Balloon's proportional allocation → smaller-than-expected chunks waste space, larger-than-expected chunks must be split (tail stored separately, double read). CCZNS's byte-addressable storage writes chunks consecutively without pre-allocation, eliminating both problems. Experiments: CCZNS reduces write volume 12.6% vs Balloon, average read latency 16.1% lower (paper §V-C).
Q3How much engineering effort is required to integrate CCZNS into an existing ZNS-aware application?▾
Moderate. RocksDB + ZenFS: ~1100 LoC total (~800 RocksDB, ~300 ZenFS). Linux NVMe driver: few LoC for bi-directional metadata transfer. The changes are focused: (1) replace compression library calls with CW/DR, (2) extend index entries to store pre-compression size, (3) defer index construction until after data blocks written. Crash consistency and core LSM logic are unaffected. The paper emphasizes this is achievable because RocksDB already maintains fine-grained chunk indexes (the key enabler for CCZNS's approach).
Q4Does CCZNS work with Zone Append, or only traditional sequential writes?▾
The current design uses write-based CW (host specifies LBA + byte offset). Zone Append-based CW is feasible: the SSD would determine the actual write address (as Zone Append does) and report it in byte granularity (not logical block) along with post-compression size. The deferred indexing principle is identical. However, no existing ZNS-compatible host software supports Zone Append in the compressed I/O path (e.g., BtrFS uses Zone Append only for non-compressed data/metadata [paper citation 25]). The paper leaves Append-based CW evaluation as future work.
Q5What happens if the data is incompressible (e.g., already-compressed images)?▾
Host software should switch to traditional read/write commands for incompressible data (avoiding unnecessary compression overhead). This decision can be made per-file or per-dataset (e.g., RocksDB's SST files storing compressed images use traditional write, text files use CW). The SSD can also skip compression via sampling-based prediction [paper citation 47] or entropy estimation [paper citation 34]. If predicted compression ratio is too poor (e.g., <1.1×), the FTL writes data without compression. CCZNS's software-defined nature (host controls CW vs traditional write) aligns with ZNS's philosophy.
Q6Why does CCZNS use ZSTD level 1 instead of higher levels for better compression ratio?▾
Trade-off between compression ratio and throughput. ZSTD level 1 is the fastest (sacrifices some ratio for speed). Higher levels (e.g., level 3) achieve ~10% better compression ratio but run 2–3× slower. Since CCZNS already offloads compression to hardware (11 GB/s), the bottleneck shifts to other parts of RocksDB (merge/sort in compaction). Using a slower algorithm would degrade throughput for marginal space savings. The paper uses level 1 to match the common deployment choice (e.g., Facebook's RocksDB default). Future work could explore adaptive level selection.
Q7How does CCZNS compare to external hardware accelerators like Intel QAT?▾
Both offload compression, but QAT suffers from round-trip data movement (host ↔ PCIe card). For small chunks (RocksDB's 4 KB blocks), QAT's throughput is unsatisfactory (Table II: 1.26 GB/s on 4 KB chunks vs 11 GB/s SSD engines). CCZNS compresses on the device-side I/O path (no extra round-trip), so small-chunk performance is high. Also, QAT occupies PCIe lanes and adds device cost, while CCZNS scales with number of SSDs (each has its own engines). Experiments: CCZNS achieves 3.58× throughput vs QAT in YCSB Load (paper §V-C, Figure 9a).
Q8What's the storage overhead of OOB metadata, and does it reduce user capacity?▾
OOB is embedded in flash pages (128–256 B per 4 KB page), separate from user data area. It doesn't reduce user capacity. CCZNS uses ~40 B per 4 chunks (worst case: 4 KB chunks at 4× compression). OOB is a fixed flash design cost (not unique to CCZNS), so the incremental overhead is zero. Contrast with SSD DRAM: host-transparent compression requires DRAM that scales with capacity (expensive, scarce). CCZNS replaces scalable DRAM with fixed OOB.
Q9Can CCZNS be applied to Flexible Data Placement (FDP) instead of ZNS?▾
FDP retains block interface but allows host hints for data placement (similar goal: reduce device write amplification). Since FDP still uses fine-grained mapping inside the SSD, logical/physical addresses are not coupled like ZNS. This prevents host-side cross-layer management (the key enabler for CCZNS). However, FDP could leverage CCZNS's host-SSD collaboration concept to suppress DRAM overhead (proportional increase with compression ratio in host-transparent schemes). The paper flags this as future work (§III-F), noting FDP's easier deployment (backward compatible) vs ZNS's deeper optimization potential.
Q10What are the main limitations or future work directions?▾
(1) Zone Append support (Append-based CW not yet implemented/evaluated). (2) Extending to SMT (hyperthreaded) cores, demand/prefetch stream partitioning, QoS guarantees, multithreaded workloads (currently targets multiprogrammed independent-program workloads). (3) Adaptive compression level selection (currently fixed at ZSTD level 1). (4) Handling workloads with fast phase changes (CCZNS re-decides compression policy every 5M cycles; a few workloads where LRU beats UCP due to per-access adaptivity could be analogous here, though not explored in the paper). (5) Generalizing to other ZNS-compatible host software (F2FS, BtrFS) beyond RocksDB.

Glossary & further reading

Core terms, supporting concepts, then papers to go deeper.

Core terms (CCZNS-specific)
CCZNS (Collaborative Compression ZNS)An advanced ZNS interface that revives in-storage hardware compression through host-SSD collaboration, decoupling compression execution (SSD-side) from indexing (host-side).
CW (Compressed-Write)NVMe command (extends traditional write) that transfers data + metadata to SSD, compresses, writes, and responds post-compression sizes via bi-directional metadata transfer.
DR (Decompressed-Read)NVMe command (extends traditional read) that locates compressed chunks via host-provided metadata, reads, decompresses, and returns data.
Multi-chunk CW/DRBatched versions of CW/DR (identified by second reserved bit) that handle multiple chunks per I/O, reducing I/O count for small-chunk workloads (e.g., RocksDB's 4 KB blocks).
Bi-directional metadata transferLinux NVMe driver modification (few LoC) enabling metadata to flow host→SSD (compression hints) then SSD→host (post-compression sizes) in one I/O, integrating compression semantics without extra round-trips.
Byte-addressable storageCCZNS's zone/write pointer management in bytes (not logical blocks), enabling maximum space harvesting. Data <16 KB buffered (capacitor-protected), merged with subsequent writes.
OOB (Out-Of-Band) metadataFlash page metadata area (128–256 B per 4 KB page) storing per-chunk info: offset, post-compression size, feedback size. Enables DR address legality checks and chunk boundary location.
Padding zoneZNS zone type (specified via reserved bit in open command) where last chunk in each CW is auto-padded to logical block boundary, enabling CW + traditional write coexistence (0.8% overhead for 256 KB compressed).
Supporting terms (ZNS / compression background)
ZNS (Zoned Namespace)SSD interface dividing storage into zones (sequential write only), eliminating fine-grained FTL mapping (10–20× less DRAM) and device GC (up to 50% less flash OP space).
TCSSD (Transparent Compression SSD)SSD with in-storage hardware compression operating host-transparently (e.g., ScaleFlux CSD, Samsung SmartSSD). Compresses invisibly, exposes magnified logical volume, requires SSD-side fine-grained indexes.
Balloon-ZNSState-of-the-art host-transparent in-storage compression on ZNS (HPCA'25 prior work). Pre-allocates space proportionally, groups zones to shrink physical allocation (limits parallelism to 60.5% of CCZNS).
NCK (Number of Chunks)Metadata field in multi-chunk CW/DR specifying how many independent compression units are in the I/O (e.g., 64 for a 256 KB flush of 4 KB blocks).
CNLB (Compressed Number of Logical Blocks)Multi-chunk DR metadata: the read size in compressed space. SSD uses this + OOB to locate chunk boundaries.
PDSZ (Padding Size)Multi-chunk DR metadata: padding in the last logical block of the compressed read range, enabling exact address calculation.
Feedback sizeSize reported to host in padding zone mode (post-compression + padding to logical block), enabling block-level addressing. Actual post-compression size stored in OOB for DR.
RocksDB / ZenFSPopular LSM-tree key-value store (Facebook) / ZNS filesystem backend. CCZNS case study: ~1100 LoC integration, multi-chunk CW for data blocks, deferred indexing.
ZSTD / Snappy / LZ4Compression algorithms. ZSTD level 1: standard algorithm, ~2× compression ratio, fast. Snappy/LZ4: lightweight, ~1.4× ratio, faster but lower ratio.

ZNS: Avoiding the Block Interface Tax for Flash-based SSD easy

Bjørling et al. · USENIX ATC 2021

The ZNS paper. Explains zones, sequential write constraint, coupled logical/physical addresses, zone-level mapping table, device GC elimination. Essential background for understanding why CCZNS's host-side indexing works.

The CASE of FEMU: Cheap, Accurate, Scalable and Extensible Flash Emulator easy

Li et al. · USENIX FAST 2018

The SSD emulator used in CCZNS evaluation. Explains how to emulate flash latency, parallelism, and extend for compression engines (parallel threads in FEMU).

Related work on in-storage compression

zFTL: Power-efficient Data Compression Support for NAND Flash-based Consumer Electronics Devices medium

Park & Kim · IEEE Trans. Consumer Electronics 2011

Early FPGA-based in-storage compression. Introduces compression-aware FTL but targets block interface (fine-grained mapping). CCZNS differs by targeting ZNS and decoupling indexing.

POLARDB Meets Computational Storage medium

Cao et al. · USENIX FAST 2020

Alibaba's relational database on compression-enabled SSD. Uses host-transparent approach (SSD-side indexes). CCZNS's host-side indexing is a fundamental departure.

KallaxDB: A Table-less Hash-based Key-Value Store on Storage Hardware with Built-in Transparent Compression hard

Chen et al. · DaMoN 2021

Hash-based KV store on TCSSD. Redesigns data structures to exploit compression. CCZNS focuses on LSM-tree (RocksDB) and cross-layer collaboration instead.

Balloon-ZNS: Constructing High-Capacity and Low-Cost ZNS SSDs with Built-in Compression hard

Wang et al. · DAC 2024

State-of-the-art host-transparent compression on ZNS. CCZNS's main comparison baseline. Key differences: CCZNS decouples indexing (no SSD DRAM), byte-addressable storage (no proportional pre-allocation), 1:1 zone-superblock mapping (full parallelism).
